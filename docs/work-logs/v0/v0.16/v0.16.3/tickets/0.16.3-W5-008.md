## Problem Analysis

## 現況分析

### 1. 派發流程中的日誌填寫要求

**tdd-flow.md（第 242 行）**：代理人完成後自行執行步驟列表中，第 1 步就是 ticket track append-log。但這是「完成後」的收尾步驟，不是「過程中」的指示。

**commit-and-phase-responsibility.md（第 24 行）**：要求 commit 後更新 Ticket，同樣是事後行為。

**reporting-and-review-standards.md**：明確規定「執行過程細節」應寫入 Ticket Execution Log，但未說明填寫時機。

**parallel-dispatch.md**：完全無提及 append-log 或執行日誌填寫。

### 2. Ticket 範本分析

**ticket_builder.py（create_ticket_body 函式）**：範本中 Solution 和 Test Results 區段使用 `<!-- To be filled by executing agent -->` 佔位符。Problem Analysis 區段則在建立時由 PM 預填。

問題：佔位符只說明「由執行代理人填寫」，未說明「何時填寫」。代理人容易理解為「完成時補充」而非「過程中即時記錄」。

### 3. 驗證機制分析

**lifecycle.py（第 432-441 行）**：complete 時有 validate_execution_log 軟檢查，偵測佔位符是否仍存在，但只是 WARNING 不阻止完成。

**acceptance_auditor.py**：有 validate_execution_log_completeness 函式檢查三區段（Problem Analysis、Solution、Test Results），但同樣是軟檢查。

### 根因總結

| 問題層 | 具體問題 |
|--------|---------|
| 範本層 | 佔位符只說 who（executing agent），不說 when（過程中 vs 事後） |
| 流程層 | tdd-flow 和 commit 規則都把 append-log 放在「完成後」步驟 |
| 驗證層 | complete 時只 WARNING 不 BLOCK，無強制力 |
| 派發層 | parallel-dispatch.md 無日誌填寫提醒 |

## 建議方案

### 方案 A：範本佔位符加入填寫時機提示（低成本，中效果）

**修改位置**：ticket_builder.py 的 create_ticket_body 函式

**修改內容**：將佔位符從：
```
<!-- To be filled by executing agent -->
```
改為：
```
<!-- [強制] 實作過程中即時填寫，非完成後補充。complete 前必須填寫。 -->
```

**優點**：修改量極小（改 2 行字串），代理人每次讀 Ticket 都能看到提示。
**缺點**：仍依賴代理人自律，無技術強制力。

---

### 方案 B：ticket track complete 改為 BLOCK（中成本，高效果）

**修改位置**：lifecycle.py 的 complete 流程（第 432-441 行）

**修改內容**：將 validate_execution_log 的結果從 WARNING 改為 BLOCK——未填寫時 exit 1，阻止 complete。

**優點**：技術強制，代理人無法跳過。
**缺點**：需要修改 lifecycle.py 程式碼和對應測試。可能影響某些不需要 Execution Log 的 Ticket 類型（如 DOC、ANA）。需考慮類型豁免機制。

**實作注意**：
- DOC 類型 Ticket 可能不需要 Solution/Test Results，需加類型判斷豁免
- ANA 類型 Ticket 的 Solution 可能是分析結論而非實作方案，需確認是否適用

---

### 方案 C：tdd-flow.md 調整 append-log 的位置和措辭（低成本，中效果）

**修改位置**：tdd-flow.md 第 240-244 行

**修改內容**：將 append-log 從「完成後步驟」移至「執行中強制行為」，並在代理人自治規則中明確：
- Solution 區段：實作完成、commit 前填寫
- Test Results 區段：測試執行後立即填寫

同時在 commit-and-phase-responsibility.md 強調「commit 前必須已填寫 Execution Log」（而非 commit 後）。

**優點**：從流程定義層修正，所有代理人都會看到。
**缺點**：仍依賴代理人遵守流程文件。

---

### 方案 D（推薦）：A + B 組合

**範本提示（A）+ 技術強制（B）雙管齊下**。

- 第一步做 A（改佔位符文字），立即生效，成本接近零
- 第二步做 B（BLOCK 機制），提供技術強制力
- 同時做 C（調整流程文件措辭），確保流程定義準確

**實作優先級**：A（立即）> C（同步）> B（需評估類型豁免後實作）

---

### 優先級建議

**投入最小、效果最大：方案 A**（改 2 行佔位符文字）。

**長期最佳解：方案 D**（A+B+C 組合），但 B 的類型豁免邏輯需要額外設計。
