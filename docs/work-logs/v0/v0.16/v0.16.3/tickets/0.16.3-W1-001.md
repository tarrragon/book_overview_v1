## Problem Analysis

### 問題根因

v0.15.0 重啟開發時，面對停滯半年的 legacy code，缺乏標準化的接手流程。
導致團隊在修復 512 個測試失敗後才發現根本設計問題（ErrorCodes 系統架構缺陷）。

### 影響範圍

- `.claude/skills/doc/` — doc skill 框架，需新增 legacy code 處理流程
- 未來所有接手 legacy code 的場景

### 需求來源（v0.15.0 ~ v0.16.2，226 commits）

| 版本 | 關鍵教訓 | 對應步驟 |
|------|---------|---------|
| v0.15.0 | 512 個測試失敗（87.6%），ErrorCodes 架構缺陷 | 步驟 1, 5 |
| v0.15.1 | Build 失敗（content.js 缺失、ES module 宣告缺失） | 步驟 0 |
| v0.15.2 | STORAGE_KEYS 常數缺失、runtime 錯誤 3 類、健康檢查誤判 | 步驟 1 |
| v0.15.3 | Popup var hoisting 衝突、Chrome Extension 平台陷阱 | 步驟 6 |
| v0.15.4 | 提取流程靜默失敗（無日誌）、SW error handler 延遲註冊 | 步驟 6 |
| v0.16.0 | SPA 渲染等待、選擇器在實際網頁失效 | 步驟 5 |
| v0.16.1 | 191 個 skip 測試逐批啟用、E2E 需獨立配置 | 步驟 2 |
| v0.16.2 | 提案驅動需求追蹤、UseCase 遷移、逐 UC 整合測試 | 步驟 3, 4, 5 |

關鍵 Ticket：
- `0.15.0-W1-001`: ErrorCodes 系統設計分析（盤點現狀）
- `0.15.0-W1-002`: 全面修復 512 個測試失敗（87.6% → 100%）
- `0.15.0-W2-001`: 多視角分析測試失敗模式（歸納根因）
- `0.15.1-W1-001`: content.js 缺失修復（build 系統過時引用）
- `0.15.4-W2`: 補齊提取流程日誌缺口（靜默失敗問題）
- `0.16.1-W1-001`: skip 測試重新啟用（191 → 65）

### 七步驟流程摘要

0. **編譯與建置驗證** — 先確認專案能編譯通過；修復過時引用、模組宣告缺失等基礎問題（來源：v0.15.1）
1. **Spec 標準化與功能盤點** — 有 spec 則標準化並比對實作（含常數/配置完整性）；無 spec 則閱讀程式碼產生 domain/subdomain 規格（來源：v0.15.0, v0.15.2）
2. **測試程式碼審查（非執行）** — 審查測試是否符合 usecase 描述；盤點 skip 測試數量和原因；無測試或無 usecase 則跳過進入步驟 3（來源：v0.16.1）
3. **提案與標準流程** — 釐清未實現部分或新需求，建立提案 → spec → usecase → ticket（來源：v0.16.2）
4. **補充測試** — 無測試的程式碼，補充 usecase 後再開始寫測試
5. **逐 UC 整合測試驗證** — 依 usecase 個別執行整合測試，禁止一次全跑；區分測試環境 vs 實際環境差異（來源：v0.15.0, v0.15.3-0.16.0）
6. **可觀測性審計** — 偵測靜默失敗（無日誌輸出）、error handler 未註冊、平台特有陷阱；確保問題可被發現（來源：v0.15.3, v0.15.4）
