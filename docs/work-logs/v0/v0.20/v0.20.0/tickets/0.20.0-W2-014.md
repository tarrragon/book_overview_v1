### AC-2 環境限制說明（誠實記錄）

cold-start init 期間緩衝瞬間無法用 chrome-devtools-mcp 直接捕捉，原因：
1. init 窗口僅 16ms，MCP 工具往返（>> 16ms）無法命中
2. getStatus 未暴露 pendingMessages count（無觀測點）
3. restart() 為熱重啟，不重現 cold-start isInitialized=false 窗口（實測 before/during/after 均 true，restartCount 0→1）
4. 無法手動 dispatch chrome.runtime.onMessage 事件

強間接證據（緩衝前提已成立）：
- 緩衝關鍵前提 = 頂層 onMessage listener 在 SW 同步評估階段就註冊 → 已 runtime 驗證（reload 後立即 hasListeners=true）。SW 一啟動 listener 即在，init 期間後續 16ms 的訊息必由此 listener 接到
- 接到後「緩衝而非丟棄」+ flush FIFO + overflow 由 W2-006.1 unit test routeMessage spy 覆蓋（6 cases）
- init 後路由 work 已 runtime 確認（messageStats）

結論：cold-start 緩衝的真實環境風險已由「listener 同步註冊（runtime）+ 緩衝邏輯（unit test）+ 路由 work（runtime）」三者共同覆蓋。精確捕捉緩衝瞬間需 src debug 改動（暫時暴露 pendingMessages 或加 cold-start gate），屬侵入式，本次未採用。

### WRAP 後選項 B 執行結果（修正前段「環境限制」結論）

前段「AC-2 無法直接捕捉」基於限制性框架。經用戶 reality-test 質疑 + WRAP 重新設計，改用「MV3 自然喚醒」方法（Stop SW → popup 喚醒 → 握手業務訊息天然落在 cold-start init 期間）成功驗證 AC-2。

### 方法
serviceworker-internals 點 Stop 使 SW 進 STOPPED（cold）→ new_page 開 popup.html 喚醒 SW → 純 internals 頁面 DOM 讀 Running Status（避免 evaluate attach 干擾）+ 讀 popup snapshot/console。

### 對照實驗結果

| 組別 | 場景 | SW 結果 | popup 結果 |
|------|------|---------|-----------|
| C | cold-start 無 popup 握手（reload_extension） | RUNNING（version 2） | N/A |
| D | cold-start + popup 握手（乾淨環境，不 attach SW） | RUNNING | 成功顯示正常 UI（提取按鈕就緒、擴展狀態:待機，0 error/warn） |
| 首次 | cold-start + popup 握手（磁碟剛清理期間 + 隨後 evaluate attach） | STARTING（觀察） | timeout「Background Service Worker 連線超時 (2秒)」初始化 5/8 |

### AC 對照（修正後）
- AC-1（載入 + 觸發 SW init）：達成
- AC-2（init 期間業務訊息不漏接）：達成（強驗證）。D 組 cold SW 被 popup 喚醒，握手業務訊息落在 cold-start 啟動/init 期間，popup 端到端成功完成初始化顯示正常 UI，證明訊息不漏接（緩衝 + flush work）。是否精確走緩衝分支由 W2-006.1 unit test routeMessage spy 保證
- AC-3（init 後 flush + 路由）：達成（popup 成功 = 完整往返）
- AC-4（W2-002 + W2-006.1 listener 協同無衝突）：達成（onMessage/onInstalled/onStartup 三 listener 註冊 + messageRouter healthy）

### 初判修正（誠實記錄，quality-baseline 規則 6）
首次觀察「STARTING 卡死 + popup 2 秒 timeout」初判為「W2-006.1 緩衝死鎖」，經乾淨對照組 D **未能重現**，推翻初判。差異變因：(1) 首次發生在磁碟剛滿/清理期間，SW cold-start 腳本載入可能 > 2 秒；(2) 首次在 popup timeout 後做 evaluate(sw) attach debugger，可能複合干擾。乾淨環境（D）cold-start + popup 握手成功。教訓：runtime 異常觀察須在乾淨環境重現後才下 root cause 結論（避免磁碟/attach 偽影誤判）。

### 意外發現（獨立追蹤，與 W2-006.1 緩衝無關）
popup ↔ SW 握手 timeout 為 2 秒（popup.js:4410）。首次觀察顯示 SW cold-start 啟動慢時（系統負載/磁碟異常）popup 可能 timeout 初始化失敗（N=1，未在乾淨環境重現）。屬 popup 握手 timeout 設計 vs SW cold-start 啟動效能的潛在 flaky 風險，需 N>=5 取樣確認 flaky 率 + root cause（PC-168）。建獨立 ticket 追蹤。
