## 1. 測試策略總覽

### 1.1 分層測試決策

依 SPEC-010 §2 API 契約分層，套用 hybrid-testing-strategy 決策樹：

| 受測對象 | SPEC 章節 | 層級判定 | 測試目錄 | 理由 |
|---------|----------|---------|---------|------|
| `makeCategoryKey` 純函式 | §2.2 | 單元 | `tests/unit/data-management/` | 無外部依賴的鍵生成邏輯 |
| `validateTagCategory` 驗證 | §2.3 | 單元 | `tests/unit/data-management/` | 純函式（傳入兄弟集合），無 I/O |
| Storage Adapter（create/update/delete/cascade/樹查詢/preset） | §2.4 | 整合 | `tests/integration/` | 經 `chrome.storage.local` mock，跨 schema + adapter 模組協作（Sociable Unit） |
| `checkReferentialIntegrity` 自癒 | §2.4 | 整合 | `tests/integration/` | 依賴 storage 狀態與多節點關聯 |
| 預裝載入（onInstalled/onStartup） | §3 D | 整合 | `tests/integration/` | 經 `chrome.runtime` + `storage.local` mock，跨 background + adapter |
| 三層同步策略（LWW/tombstone/version flag） | §3 E | 整合 | `tests/integration/` | computeMergeResult 跨模組行為 |
| 資料檔打包（build.js COPY_PATHS） | §3 E3 | 整合 | `tests/integration/` | build 配置斷言 |
| Tag 管理 UI 服務契約（rename/move/batch/search） | §2.5 §3 F | 整合 | `tests/integration/` | 服務層協調多 adapter 操作 |

**計時測試一律放 `tests/perf/`**（`test-assertion-design-rules.md` 規則 1/2）。本 ticket 的場景無計時 SLA 需求（§5「環境假設」明示 in-memory 樹操作無效能瓶頸），故**無 tests/perf/ 測試**。若 Phase 3 量測需要大幅退化防護，另開 ticket。

### 1.2 Mock 策略（Sociable Unit Tests）

| 依賴 | Mock 策略 | 理由 |
|------|----------|------|
| `chrome.storage.local` | Mock（jsdom 無 Chrome API） | 外部世界（持久層），測試不關心實作 |
| `chrome.runtime.onInstalled/onStartup` | Mock（事件觸發器） | 外部世界（SW 生命週期） |
| `TagSchema` / `makeCategoryKey` | 不 Mock（真實物件） | 內層邏輯，Sociable Unit 直接使用 |
| `OperationResult` / `ErrorCodes` | 不 Mock | 內層 value object |
| `chinese-classification.json` 預裝資料 | 視 Phase 3 接線；測試以小型 fixture 樹替代 | 避免 110 節點真實資料耦合測試 |

**斷言原則**（`test-assertion-design-rules.md`）：
- 禁 `tests/unit|integration` 計時硬門檻（`toBeLessThan(Nms)`）
- `toBeCloseTo` 精度 ≤ 2（本設計無浮點斷言需求）
- 快取/單例驗證用參考比較（`toBe`）或命中率，非計時比較
- 錯誤路徑斷言 `error` code 字面（如 `expect(result.error).toBe('duplicate_name')`），對齊 SPEC §4 error code 表

### 1.3 拆分友善性（Phase 3b 可拆分）

各場景組對應獨立功能職責，可獨立派發：

| 測試群組 | 功能職責 | 跨群組依賴 |
|---------|---------|-----------|
| A 唯一鍵語意 | `makeCategoryKey` + scoped dup 校正 | 無（B-F 共用 makeCategoryKey 但各自獨立測） |
| B parentId 防護 | 循環/深度/引用完整性驗證 | 依賴 A 的 makeCategoryKey（唯讀，不共享 mutable 狀態） |

| D 預裝載入 | onInstalled/onStartup upsert | 無 |
| E 同步與打包 | 三層同步 + build 配置 | 無 |
| F Tag 管理 UI | CRUD/批量/搜尋服務 | 依賴 A/B 驗證（唯讀） |

共用 fixture（小型樹建構 helper）抽為 `tests/integration/fixtures/tag-tree-fixture.js`，各群組獨立 import。

## 2. 場景組 A：唯一鍵語意（unit + integration）

### 行為鏈推演

行為鏈起點：建立 category 時系統以 `(parentId, name)` 生成唯一鍵。前置條件 → 鍵生成 → dup 比對 → 寫入/拒絕。

| # | 測試案例 | 維度 | 層級 | 前置驗證（A） | When（B） | Then（C） |
|---|---------|------|------|------------|----------|----------|
| A-U1 | `makeCategoryKey(null, '總論')` 與 `makeCategoryKey('cat_phil', '總論')` 不相等 | 正常 | unit | 兩 parentId 不同 | 呼叫 makeCategoryKey | 兩鍵字串不相等 |
| A-U2 | `makeCategoryKey` 大小寫正規化：`'特藏'` vs `'特藏'`（含半形空白 trim 前置由 caller 負責） | 邊界 | unit | name 相同 | 同 parentId 呼叫 | 兩鍵相等 |
| A-U3 | `makeCategoryKey(null, name)` 用 'ROOT' 前綴（null parentId） | 邊界 | unit | parentId=null | 呼叫 | 鍵含 'ROOT' sentinel，與 `'ROOT'` 字串 parentId 不混淆（見 §4 邊界澄清） |
| A-V1 | `validateTagCategory` 兄弟集合內同鍵衝突 → valid=false | 異常 | unit | siblingCategories 含同 (parentId,name) | validate | errors 含 duplicate；valid=false |
| A-V2 | `validateTagCategory` 兄弟集合內不同 parent 同名 → valid=true | 正常 | unit | sibling 為同 parent 集合（不含跨 parent 節點） | validate | valid=true |
| A1 | 跨主類同名次類並存 | 正常 | integration | 預裝樹含「0 總類」「1 哲學類」兩根 | 各自下建同名「總論」 | 二者並存，無 duplicate_name |
| A2 | 同父下同名次類被拒 | 異常 | integration | 「0 總類」下已有「00 特藏」 | 再建「00 特藏」 | `{success:false, error:'duplicate_name'}` |
| A3-1 | createTagCategory 改 scoped（adapter:256-258） | 正常 | integration | 不同 parent 同名 | create 第二個 | 不觸發 duplicate_name |
| A3-2 | computeMergeResult 改 scoped（merge:917-918） | 正常 | integration | 匯入含不同 parent 同名 category | merge | 二者皆建立，不被誤併 |
| A3-3 | validateTagCategory 改 scoped（TagSchema:81-83） | 正常 | unit | 全域有同名但不同 parent | validate（sibling 不含他者） | valid=true（不殘留全域比對） |

**A3 殘留防護斷言**：A3-1/2/3 各驗「不同 parent 同名不被誤擋」，三處任一殘留全域 name 比對即測試失敗（場景 A3「不得殘留任一處全域 name 比對」）。

## 7. 邊界澄清與設計決策

| # | 議題 | 決策 |
|---|------|------|
| 1 | `makeCategoryKey(null, ...)` vs `makeCategoryKey('ROOT', ...)` | SPEC §2.2 用 `${parentId ?? 'ROOT'}` sentinel。理論上用戶建立 id 為字串 'ROOT' 的 category 會與根衝突，但 adapter 生成 ID 為 `cat_<timestamp>` 格式，'ROOT' 不可能被生成，視為安全 sentinel。測試 A-U3 標註此假設，留 Phase 3a 評估是否需更嚴格 sentinel（如 Symbol 或不可碰撞前綴）。 |
| 2 | cascade_partial 回滾測試 | 需 storage mock 支援「第 N 次 set 失敗」注入，列為 Phase 3a fixture 需求 |
| 3 | 預裝 fixture | 測試用小型樹（2 根 + 4 次類）替代 110 節點，避免真實資料耦合；驗「機制」非「資料完整性」（資料完整性由 W2-003 資料 ticket 負責） |
| 4 | D2 listener 註冊時機斷言方式 | 透過 mock `addListener` 並斷言在 module require 同步階段已呼叫；無法直接斷言「無 await」，改以「require 後 listener 已註冊」間接驗證 |
