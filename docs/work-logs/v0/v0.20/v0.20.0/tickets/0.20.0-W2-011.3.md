### 問題根因

IMP .1（strip_code_regions 修復根因 A 4-space 縮排 + 根因 B 跨行 backtick）後，仍殘留一類誤報：**行首裸 invoke/parameter/close-invoke 標記字面出現在純散文**（無 fenced / inline backtick / 4-space 縮排包覆）。最典型場景為「meta 自我引用」——正常討論本 hook 偵測的失敗形態本身（hook docstring 解說、ticket 工作流、PC error-pattern 撰寫），散文句中嵌入裸標記字面作說明。

strip code 區塊無法解決此類（作者刻意不包 code 以利閱讀），且 linux taste veto 禁止再疊 signature regex 特例。

### 殘餘評估（evidence-first，acceptance 1）

實機跑 detect() 構造三個殘留案例，全部誤命中（確認殘餘存在，走 acceptance 2a 路徑）：

| prose_stray_token_invoke | 游離 token + 裸 invoke | signature 1 |
| prose_bare_close | 裸 close-invoke 在散文 | signature 3 |

### 影響範圍

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 治本方案（PC-099 對齊，linux taste 合規）

採**顯式 meta-context 豁免標記**（非 signature regex 特例，符合 linux taste veto）：

訊息任意位置含 `<!-- malformed-detector-exempt: <reason> -->` 時，detect() 優先回傳空字串整段豁免。

實作要點：
1. 新增 `EXEMPT_MARKER` regex（HTML 註解形式，IGNORECASE | DOTALL）。
2. `detect()` 在 strip / signature 比對**之前**先檢查 marker（針對原始 text，避免 marker 被 strip）。
3. 新增真陰 fixture `meta_context_exempt`（自動納入 _self_test + 外部 pytest 單一事實來源）。
4. 新增 3 個專屬測試：殘留散文無 marker 仍命中 / 有 marker 豁免 / marker 不削弱真實壞呼叫。
5. 更新 docstring Action 段說明豁免機制。

### 為何不削弱 true-positive（與 signature 4 防削弱要求一致）

真正寫壞的工具呼叫由 harness 把 tool-call 結構當文字渲染而成，模型當下意圖是「執行工具」而非「撰寫 meta 註解」，產出絕不含此 HTML 註解。本 session 6 次真實攔截皆純壞標記，無一含 marker。實機驗證：3 個 SELF_TEST_TRUE_POSITIVES + 真實壞呼叫案例在豁免機制下仍全數命中。

### 設計選型理由（為何不用關鍵字密度啟發式）

關鍵字密度 / 標記字元佔比閾值屬啟發式，brittle 且可能在真陽訊息（恰含解說性文字）誤豁免，削弱攔截力。顯式 marker 為確定性訊號，與 PC-099 既有 marker 慣例（PC-093-exempt）同構，且作者於 meta 散文加一行成本極低。

### 不新增 signature regex 特例（acceptance 4）

本變更僅動「豁免層」（EXEMPT_MARKER + detect 前置檢查），SIGNATURE_PATTERNS 4 條簽章原封不動，signature 4 防削弱不變。symbolically 符合 linux 治本（修剝除/豁免正確性）而非疊特例。
