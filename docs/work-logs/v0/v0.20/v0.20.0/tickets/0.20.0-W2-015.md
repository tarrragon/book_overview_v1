## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

popup 在 SW cold-start 期間的握手（checkBackgroundStatus）採「單次嘗試 + 固定 2 秒 timeout + 無重試」設計。正常情況下 cold-start 遠快於 2 秒（實測 ~104ms），握手穩定成功；但在 SW 腳本載入受 I/O 拖累（磁碟異常/系統高負載）時，cold-start 可能超過 2 秒，導致 popup 顯示「離線/背景服務未回應」且不自動恢復（需使用者關閉重開 popup）。

握手機制（src/popup/popup.js）：
- `checkBackgroundStatus()`（line 695）：`Promise.race([chrome.runtime.sendMessage({type:'GET_STATUS'}), createBackgroundTimeoutPromise()])`
- `createBackgroundTimeoutPromise()`（line 617）：固定 `setTimeout(reject, 2000)`（建置後對應 build popup.js:4410）
- `initialize()`（line 1041）步驟 6：背景檢查失敗即 `errorHandler.handleInitializationError` + `return`，無重試（line 1100-1116）

SW 端 cold-start 回應路徑（src/background/background.js）：
- 頂層 onMessage listener 同步註冊（W2-002，line 250）
- init 進行中（isInitialized=false）popup 的 `GET_STATUS` 不匹配 baseline 分支（line 257 僅特判 `GET_SYSTEM_STATUS`），落入 `bufferMessageDuringInit`（line 282/311）緩衝，保持 sendResponse 通道開啟
- init 完成後 `flushPendingMessages`（line 421）將緩衝的 `GET_STATUS` 交 MessageRouter 路由（message-router.js:261 → handleGetStatusMessage）回應原 caller

關鍵：popup 的 `GET_STATUS` 在 cold-start 期間「被緩衝」而非「立即回 baseline」。因此握手能否在 2 秒內回應，完全取決於 SW init 是否在 2 秒內完成（包含 951KB bundle 載入+parse + 8 模組 import + coordinator init）。

### 影響範圍

- src/popup/popup.js：`createBackgroundTimeoutPromise`（2 秒 timeout）、`checkBackgroundStatus`（單次無重試）、`initialize` 步驟 6（失敗即終止）
- src/background/background.js：cold-start init 路徑（決定握手回應延遲）；GET_STATUS 在 init 期間走緩衝分支而非 baseline 即時回應
- 使用者體驗：cold-start 慢於 2 秒時 popup 顯示離線且不自動恢復（rare，需重開 popup）

### 相關 Error Pattern

- PC-168（Flaky Baseline 少量 sample 推導 stable 錯覺）：本 ANA 直接適用。W2-014 首次 N=1 失敗不足以判定 flaky，故 N>=5 取樣 + GREEN/RED 分佈紀錄。本次 N=6 全 GREEN（乾淨環境）。

### 實驗發現

已驗證的事實：
1. 乾淨環境 cold-start popup 握手 N=6 全 GREEN，0 timeout。cold-start 總時間（含 951KB bundle parse + 8 模組 import + coordinator init）遠低於 2 秒 timeout（裕度 > 10x）。
2. popup 的 GET_STATUS 在 SW init 期間走 bufferMessageDuringInit 緩衝分支（非 baseline 即時回應），init 完成後 flushPendingMessages 經 MessageRouter 正確往返回應。
3. W2-014 首次 N=1 失敗為環境偽影：發生於磁碟剛清理期間（SW 951KB bundle 從 thrashing 磁碟載入可能 > 2 秒，屬 I/O-bound 異常）+ 隨後在 popup timeout 後做 evaluate(sw) attach debugger（複合干擾）。乾淨環境對照組 D（W2-014）與本次 N=6 均未重現，推翻「真實 code-level flaky」假設。
4. chrome-devtools-mcp 限制（已知並記錄）：Stop SW 後 MCP debugger auto-attach 使 SW 進入 STARTING/thread-id=-1 limbo（非乾淨 STOPPED）；list_pages 的 SW handle（sw-1/sw-4）在 Stop/Start cycle 後 stale，直接 evaluate(serviceWorkerId) timeout。改用 new_page 喚醒 + popup console 判定（W2-014 同方法）規避此限制。

仍存在的設計脆弱性（非 flaky，但屬 robustness gap）：
- popup 握手為「單次嘗試 + 固定 2 秒 + 無重試」。極端 I/O 異常下 cold-start > 2 秒時 popup 永久顯示離線、不自動恢復（需使用者手動重開 popup）。此為設計層面的潛在風險，非當前環境可重現的 flaky。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### Root Cause 結論

popup↔SW cold-start 握手「偶發 timeout」的 root cause 為：**popup 握手採單次嘗試 + 固定 2 秒 timeout + 無重試，且 GET_STATUS 在 SW init 期間走緩衝分支（其回應延遲 = SW init 完成時間）**。乾淨環境下 SW init 遠快於 2 秒（實測總時間 < 200ms，裕度 > 10x），故握手穩定成功（N=6 全 GREEN）。W2-014 首次 N=1 失敗為**環境偽影**（磁碟異常期間 951KB bundle I/O 載入 > 2 秒 + timeout 後 attach SW debugger 複合干擾），非 code-level flaky；乾淨環境（W2-014 對照組 D + 本次 N=6）均未重現。

### flaky 率判定

乾淨環境 flaky 率 = 0%（6/6 GREEN，N >= 5 滿足 PC-168）。無真實 flaky condition。W2-014 的 N=1 失敗歸因於外部 I/O 異常（磁碟剛清理）疊加觀測干擾（debugger attach），不可重現。

### 修復必要性評估（WRAP: Prepare to be wrong）

雖無可重現 flaky，但靜態分析確認存在**設計層面 robustness gap**：握手單次無重試，極端 I/O 異常（磁碟壓力 / 系統高負載 / 大型 bundle 在低階機器）下 cold-start > 2 秒時 popup 永久顯示「離線」且不自動恢復（需使用者手動重開）。此為真實但低頻的 UX 脆弱性，依 quality-baseline 規則 5（所有發現必須追蹤）應建 follow-up IMP 追蹤，但**優先級 P2/P3**（非 v1.0 內測 blocker，因乾淨環境穩定成功）。

### 候選修復方向（供 IMP 評估，本 ANA 不選定實作）

| 方向 | 內容 | 優點 | 缺點 |
|------|------|------|------|
| A. 握手重試 | checkBackgroundStatus 失敗後重試 1-2 次（指數退避） | 直接消除「單次失敗即永久離線」；改動局部 | 總等待時間延長（2s → 4-6s） |
| B. 延長/動態 timeout | 固定 2s → 3-5s，或首次 cold-start 用較長 timeout | 改動最小 | 治標；極端 I/O 仍可能超時；延長使用者等待 |
| C. init 期間 GET_STATUS 即時回 baseline | 頂層 listener 將 popup GET_STATUS（如 GET_SYSTEM_STATUS 般）即時回 initializing baseline，popup 顯示「初始化中」並輪詢 | 握手不再阻塞於 init 完成；UX 明確 | 需 popup 配合「初始化中」狀態 + 輪詢/重試邏輯，改動較大 |
| D. SW 啟動最佳化 | 縮減 951KB bundle / 延後非關鍵模組 import | 治本降低 cold-start 時間 | 範圍大、效益對乾淨環境邊際（已 < 200ms） |

推薦組合：**A（重試）+ C（init 期間即時 baseline + popup 輪詢）** 為 robustness 治本方向；B 為最小成本止血。D 不建議（乾淨環境已快，投報率低）。最終方案由 IMP ticket 經 design-decision-framework 評估後選定。
