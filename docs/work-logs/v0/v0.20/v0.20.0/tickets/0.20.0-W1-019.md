### 0.19.1 發布實證（2026-06-05）

### 缺陷 A：CHANGELOG 空殼插入
- update_changelog（version_release.py:1313-1358）：偵測既有 "## [version]" 區段則跳過插入（line 1336），但版本區段完全不存在時走 line 1340 插入空殼 placeholder 模板（含 "UC-XX 功能名稱"、"新增功能項目" 等佔位符）。

- 修復方向：版本區段不存在時改 fail-loud（exit non-zero + 提示手動撰寫），不靜默插空殼。

### 缺陷 B：Mark Completed 步驟順序
- release 流程 Step 3 Git Operations（git push origin main）排在 Step "Mark Completed"（todolist active→completed）之前。
- 後果：todolist completed 標記落在已推送 commit 之外，成孤兒未提交變更。0.19.1 發布後遠端 todolist 仍為 active，靠手動補 commit 261c0a56 + push 對齊。
- 修復方向：Mark Completed 移至 push 之前納入同一推送，或標記後自動 commit + push。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 缺陷 A：CHANGELOG 空殼插入 → fail-loud

`update_changelog`（version_release.py）原 fallback 路徑：版本區段不存在且無 In-Development 區段時，插入 UC-XX 空殼 placeholder 模板。改為 fail-loud：

- 移除 placeholder 插入分支，改 `print_error` + `return False`，提示手動撰寫 `## [version] - date` 區段或建立 `## [v{version}] - In Development` 開發期區段供自動 finalize。
- 保留 W1-004 既有 finalize 跳過邏輯（finalize 既有 In-Development 區段 + 冪等性檢查既有發布區段），未破壞。

### 缺陷 B：Mark Completed 步驟順序 → 移至 push 之前

原流程：main() 在 `git_merge_and_push`（含 commit+push）完成「之後」才呼叫 `mark_version_completed`，導致 todolist completed 標記成 push 後孤兒未提交。

修復：將 `mark_version_completed` 呼叫移入 `git_merge_and_push` 內 step 3.1 commit_changes 之前（新增 step 3.0）。因 `commit_changes` 已 `git add docs/todolist.yaml`，標記變更隨 3.1 commit + 3.6 push 一併推送，不再成孤兒。main() 內原 post-push 標記區塊移除（改為說明註解）。

### 測試契約變更

原 W1-004 測試 `test_no_in_development_section_falls_back_to_template` 斷言舊 placeholder 插入行為，與缺陷 A 修復直接衝突（同一 update_changelog 行為的目標變更）。已更新為 `test_no_in_development_section_fails_loud`，斷言 fail-loud（return False + 檔案不變）。
