## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`src/background/background.js` 的 onInstalled / onStartup listener 過去包在 `registerServiceWorkerEvents()` 內，而該函式於 `initializeBackgroundSystem()` 的 `await coordinator.initialize()`（行 157）、`await coordinator.start()`（行 161）等多個 await **之後**（行 188）才被呼叫。onMessage listener 更僅在 `activateEmergencyMode()`（init 失敗時）才註冊，正常路徑下喚醒期間完全沒有頂層 onMessage listener。

MV3 service worker 非持久（約 30 秒無活動即終止）。Chrome 要求事件 listener 在 SW 腳本「初始評估的同步階段」就 `addListener`。現況下 SW 在某 await 期間被喚醒/重啟時，事件可能在 listener 註冊前派發 → 靜默漏接。

獨立驗證（RED 測試）：以「永不 resolve 的 coordinator.initialize」模擬 SW 卡在 await 期間，require background.js 後同步階段檢查三個 `addListener` 呼叫次數，修復前皆為 0，確認缺陷存在。

### 影響範圍

- `src/background/background.js`（唯一 where.files）：listener 註冊時機。
- 連帶發現（已建 0.20.0-W2-006 ANA 追蹤，本 ticket 不修）：真正的業務訊息路由由 coordinator 內 `MessageRouter.setupMessageListener`（`src/background/messaging/message-router.js:166`）在 async init 之後 `chrome.runtime.onMessage.addListener`，同屬 MV3 listener 延後註冊缺陷家族。

### 相關 Error Pattern

CLAUDE.md 第 7 節「事件監聽器必須在 SW 頂層同步註冊，禁止 async 延遲註冊」。無對應 PC error-pattern，屬 MV3 平台鐵則直接違反。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 修復摘要

1. 新增模組頂層同步函式 `registerLifecycleListeners()`，於 background.js 模組求值同步階段（緊接 error/unhandledrejection handler 之後）立即呼叫，同步註冊 onInstalled / onStartup / onMessage 三個 listener。callback 內部的 async 工作（eventBus.emit / baseline 回應）不變，於事件實際觸發時才執行，此時 `backgroundCoordinator` 多已就緒（callback 內保留既有 null guard）。
2. 移除 `initializeBackgroundSystem()` 內 `await registerServiceWorkerEvents()`（行 188）與整個 `registerServiceWorkerEvents()` async 函式。
3. 移除 `activateEmergencyMode()` 內的 `onMessage.addListener`（延後註冊 + 與頂層重複）。emergency baseline 回應改由頂層 onMessage handler 依 `emergencyMode` 旗標切換。

### onMessage 設計決策（避免破壞既有路由）

頂層 onMessage handler 僅在系統未就緒（init 進行中）或 emergency 模式時，對 `GET_SYSTEM_STATUS` 提供 baseline 回應；系統正常就緒（`isInitialized && !emergencyMode`）時回傳 `undefined` 不攔截，交由 coordinator 內 MessageRouter 註冊的 listener 處理，避免 sendResponse 通道雙重佔用、不破壞既有訊息路由行為。

### 不在本 ticket 範圍

MessageRouter listener 本身的延後註冊（message-router.js:166）屬不同檔案的深層架構問題，已建 0.20.0-W2-006 ANA 獨立追蹤（quality-baseline 規則 5）。

### 本 ticket 精確缺陷（MV3 多視角審查 Q1 發現，2026-06-05）

**缺陷**：`src/background/background.js` 的 `registerServiceWorkerEvents()` 在 `initializeBackgroundSystem()` 內，於 `await coordinator.initialize()`（約行 157）、`await coordinator.start()`（約行 161）等多個 await **之後**（約行 188）才註冊 onInstalled/onMessage/onStartup listener。

**修正方向**：將 onInstalled/onMessage/onStartup 的 `addListener` 移至 background.js 模組頂層同步階段執行；callback 內部再做 async 工作（callback 可呼叫既有的 coordinator/handler，但「註冊」本身要在頂層）。

**驗證要點**：
- RED 測試：斷言 SW 模組求值同步階段（import/require background.js 後、未等任何 async）三個 listener 已註冊。現有測試環境用 jsdom + chrome API mock（見 tests/ 既有 background 測試慣例）。
