### 問題根因

現行 TagSchema 採扁平二層結構（TagCategory → Tag），設計時明確標註「樹狀延至 v0.20.0」。v0.20.0 W2 需預裝賴永祥《中文圖書分類法》（二層以上、數百節點），但 TagCategory 無 parentId，無法表達分類法的階層關係（如「0 總類 → 00 特藏 → 001 善本」）。

### 影響範圍

| 模組 | 檔案 | 影響 |
|------|------|------|
| Schema 定義 | `src/data-management/TagSchema.js` | TAG_CATEGORY_SCHEMA 需新增 parentId 欄位 |
| Storage adapter | `src/storage/adapters/tag-storage-adapter.js` | CRUD / 合併 / 覆蓋 / 完整性檢查需處理 parentId |
| UI 渲染 | `src/overview/tag-cell-renderer.js` | tag chip 需顯示階層路徑（低影響，取決於 UI spec） |
| 搜尋解析 | `src/ui/search/tag-resolver.js` | resolveTag 需攜帶階層資訊供搜尋篩選（低影響） |
| 匯出/匯入 | `src/export/`, `src/data-management/migration/` | IF-v2 需序列化/反序列化 parentId |
| BookSchemaV2 | `src/data-management/BookSchemaV2.js` | Book.tagIds 不變（仍指向葉節點 Tag），無直接影響 |

### 相關 Error Pattern

無直接相關 error pattern。注意 PC-068（Phase 3a 規劃新建前必掃描既有）：本分析已確認無既有 parentId 或樹狀實作。

### 實驗方法

### 實驗執行

1. `rg "parentId|parent_id" src/ --type js` — 零命中，確認 codebase 無任何 parentId 欄位
2. 讀取 `TagSchema.js` TAG_CATEGORY_SCHEMA — 確認欄位為 `{id, name, description, color, isSystem, sortOrder, createdAt, updatedAt}`，無 parentId
3. 讀取 `tag-storage-adapter.js` — 所有 CRUD 與合併邏輯僅處理 `categoryId`（Tag → TagCategory 單層引用），無遞迴走訪
4. `BookSchemaV2.js` Book 欄位含 `tagIds: string[]`，指向 Tag.id（葉節點），不直接引用 TagCategory

### 實驗發現

**已驗證事實**：
- TagCategory 為扁平結構，無 parentId、無 level、無 path 欄位

- tag-storage-adapter 的合併邏輯（computeMergeResult）依賴 categoryId 同名比對，不涉及階層

**未驗證假設**：
- 賴永祥分類法具體節點數量（主類 10 + 次類約 100 + 三級約 800-1000）——已知為公開標準但未引入實際資料檔
- Chrome Storage 5MB 配額對數百預裝節點的容量影響——需 Phase 1 估算

### SA 前置審查結論

### AC-1：樹狀化方案評估（3 方案）

#### 方案 A：TagCategory 加 parentId（遞迴分類）——建議採用

**做法**：在 TAG_CATEGORY_SCHEMA 新增 `parentId: { type: 'string', required: false, default: null }` 欄位。TagCategory 形成遞迴樹，Tag 仍掛載於葉節點 TagCategory。

**利弊**：

| 維度 | 評估 |
|------|------|
| 改動幅度 | 最小。僅 TagSchema 加 1 欄位 + storage adapter 增補 parentId 處理 |
| 既有資料相容 | 完全相容。現有 TagCategory 的 parentId 預設 null = 根節點，零遷移 |
| 語意一致性 | TagCategory 本就代表「分類」，加 parentId 成遞迴分類是自然延伸 |
| PROP-007 對齊 | Tag-based model 的 category 階層化，不違反跨專案規格 |
| 合併邏輯影響 | computeMergeResult 的 category 同名比對需加入 parentId 維度（同名 + 同 parent 才合併），中等複雜度 |
| 查詢複雜度 | 需遞迴查父鏈（getAncestors）或子樹（getDescendants），但 Chrome Storage 全量載入後 in-memory 操作，無效能問題 |
| 深度限制 | 建議設 MAX_DEPTH = 4（賴永祥分類法實際最深 3 層），防止無限遞迴 |

#### 方案 B：新增 TagNode 階層實體

**做法**：引入獨立 `TagNode` entity（id, name, parentId, level, path），與 TagCategory/Tag 並存。TagCategory 退化為「分類法來源」標記，TagNode 承載階層。

**利弊**：

| 維度 | 評估 |
|------|------|
| 改動幅度 | 大。新增 entity + storage key + CRUD + 合併邏輯 + 所有引用點 |
| 既有資料相容 | 需遷移：現有 TagCategory 轉為 TagNode，或保留 TagCategory 做平行結構 |
| 語意一致性 | 語意最清晰，但與現有 CategoryId 引用鏈斷裂（Tag.categoryId → 何者？）|
| PROP-007 對齊 | 需要跨專案規格同步變更，Flutter App 端也需對應調整 |
| 合併邏輯影響 | 新增一整套合併邏輯（TagNode 同名比對 + 階層對齊），高複雜度 |
| 風險 | 雙模型共存（TagCategory + TagNode）增加認知負擔與一致性維護成本 |

**結論**：不建議。改動幅度與認知負擔不成比例，且 TagCategory 本身語意已足以承載階層。

#### 方案 C：Tag 加 path 欄位（物化路徑）

**做法**：不改 TagCategory，而是在 Tag 加 `path: string`（如 `"/0/00/001"`）扁平化儲存階層資訊。

**利弊**：

| 維度 | 評估 |
|------|------|
| 改動幅度 | 中。Tag schema 加 1 欄位，但查詢需 regex/prefix 比對 |
| 既有資料相容 | 相容（path 預設空字串），但既有 Tag 無 path 語意 |
| 語意一致性 | 違反現有設計：Tag 是葉節點（書的標籤），不應承載分類階層；階層是 category 的職責 |
| 合併邏輯影響 | path 字串比對脆弱，分類法結構變更時所有 path 需重算 |

**結論**：不建議。語意混淆，且 path 維護成本高於 parentId 遞迴。

### AC-2：賴永祥中文圖書分類法二層以上預裝資料影響

#### 資料規模估算

| 層級 | 節點數（約） | 範例 |
|------|------------|------|
| 第一層（主類） | 10 | 0 總類、1 哲學類、2 宗教類、...、9 藝術類 |
| 第二層（次類） | 100 | 00 特藏、01 目錄學、02 資訊科學、... |
| 第三層（細類） | 800-1000 | 001 善本、002 地方文獻、... |

用戶決策為「二層以上」，建議預裝至第二層（約 110 個 TagCategory 節點），第三層以 lazy-load 或用戶按需展開。

#### Model 影響

- **TagCategory 數量**：從用戶自建的個位數增長至 110+。TAG_CATEGORY_SCHEMA 的 `isSystem: true` 標記可區分預裝 vs 用戶自建。
- **預裝資料格式**：需新增預裝資料檔（如 `src/data-management/presets/chinese-classification.json`），格式為 `[{ id, name, parentId, isSystem: true, sortOrder }]`。
- **ID 穩定性**：預裝節點 ID 必須為確定性（如 `sys_cat_0`、`sys_cat_00`），不可用 `Date.now()` 隨機 ID，否則合併/同步時無法比對。

#### Storage 影響

- **容量**：110 個 TagCategory 物件（每個約 150 bytes JSON）約 16.5 KB，佔 Chrome Storage 5MB 配額 0.33%，可忽略。即使預裝三層（1000 節點），也僅約 150 KB（3%）。
- **tag-storage-adapter 變更**：
  - `createTagCategory`：預裝節點 `isSystem=true` 不可被用戶刪除（既有邏輯已有 `isSystem` 保護）
  - `loadCategories`：無需變更（全量載入，in-memory 篩選）
  - `computeMergeResult`：category 同名比對需加入 parentId 維度
  - 新增 `initializePresets()` 函式：首次安裝或版本升級時注入預裝資料

#### 載入機制

建議採用「首次安裝 / 版本升級時一次性注入」：

| 機制 | 說明 |
|------|------|
| 觸發時機 | `chrome.runtime.onInstalled`（install / update 事件）|
| 執行者 | background service worker |
| 邏輯 | 讀取 `schema_version`，若版本 < 預裝版本，呼叫 `initializePresets()` |
| 冪等性 | 以 `isSystem=true` + 確定性 ID 做 upsert（存在則跳過/更新，不重複建立）|
| 用戶自建保護 | 只操作 `isSystem=true` 節點，用戶自建節點不受影響 |

### AC-3：與 v0.17 model、PROP-007、既有資產的一致性檢查

#### 系統一致性 checklist

| 檢查項 | 結果 |
|--------|------|
| 命名規範 | parentId 遵循既有 camelCase（categoryId, tagIds, sortOrder）|
| 架構模式 | 遞迴 parentId 模式與既有 Tag→TagCategory 引用模式一致（外鍵引用同表）|
| 依賴方向 | TagCategory 自引用，不引入新依賴方向，正確 |
| BookSchemaV2 相容 | Book.tagIds 仍指向 Tag.id，TagCategory 階層化不影響 Book 結構 |
| PROP-007 對齊 | Tag-based model 的 category 階層化屬「Extension 端實作細節」，不違反跨專案規格；IF-v2 需序列化 parentId（Phase 1 spec 處理）|
| tag-cell-renderer | 依賴注入 `getCategoryById`，新增 parentId 後只要 category 物件多一個欄位，無 breaking change |
| tag-resolver | `resolveTag` 回傳 `categoryName` / `categoryColor`，階層資訊屬擴展（可加 `categoryPath`），非 breaking |
| tag-storage-adapter | `checkReferentialIntegrity` 需擴展：驗證 parentId 引用有效性（若 parentId 非 null 必須指向存在的 category）|

#### 系統衝突檢查（強制章節）

**跨模組依賴衝突**：無衝突。parentId 為 TagCategory 自引用欄位，不引入跨模組新依賴。

**既有 UC 變動影響**：無 UC 變動。UC-01~UC-07 的 tag 操作語意不變（Tag 仍掛載於 TagCategory，Book.tagIds 不變）。預裝分類法屬新增功能，不修改既有 UC 行為。

**跨版本相容性**：無相容性影響。parentId 預設 null，既有資料自動成為根節點，零遷移。IF-v2 匯出格式需在 Phase 1 spec 處理 parentId 序列化（schema_version bump）。

**衝突檢查結論**：無 critical 衝突，可進入 Phase 1 spec。

### AC-4：建議方案與 Phase 1 spec spawn 結論

#### 建議方案

**採用方案 A：TagCategory 加 parentId**。

理由：
1. 最小改動幅度——僅 TagSchema 加 1 欄位 + storage adapter 增補
2. 完全向後相容——既有資料零遷移
3. 語意自然——TagCategory 本就代表分類，遞迴階層是自然延伸
4. 合併邏輯可控——在既有 computeMergeResult 擴展 parentId 維度即可

#### Phase 1 spec 需涵蓋範圍

| 項目 | 說明 |
|------|------|
| TAG_CATEGORY_SCHEMA 擴展 | 新增 parentId 欄位定義 + MAX_DEPTH 常數 |
| 預裝資料規格 | 賴永祥分類法 JSON 格式、ID 命名規範、預裝深度（二層 or 三層）|
| 載入機制 | initializePresets() 觸發條件 + 冪等設計 |
| Storage adapter 擴展 | parentId 處理（CRUD + 合併 + 完整性）|
| 驗證函式擴展 | validateTagCategory 加 parentId 型別 + 循環引用防護 |
| IF-v2 序列化 | parentId 欄位在匯出 JSON 的位置 |

#### 結論

**建議 spawn Phase 1 spec ticket**。方案 A 方向明確、風險低，Phase 1 spec 可直接基於本審查結論設計詳細規格。
