## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 技術可行性調查（PM 前台評估，2026-06-06，acceptance 1）

**1. 資料可取得性：可行**
transcript jsonl 的 assistant message 含 `usage.cache_read_input_tokens`（W2-012 即由此取得 ~230K onset 數據）。hook 可透過 stdin payload 的 `transcript_path` 讀取，技術上可行。

**2. 掛載點修正（原構想缺陷）**
W2-013 原 where.files 寫 `branch-status-reminder.py`（SessionStart hook）。但 SessionStart 僅在 session 啟動觸發，cache_read 處於低點（剛 /clear），無法在 context 累積到 ~200K 時偵測。正確掛載點為 **Stop event**（每回合結束觸發，transcript 含最新 usage）。原構想掛載點錯誤，需修正。

**3. 既有可複用基礎設施**
- `lib/transcript_tail_reader.py`：反向讀 transcript + offset cache（W11-004，~ms 級成本）。目前只抽 text content，需擴充抽 usage 或新增讀 usage 函式。
- `malformed-tool-call-detector-hook.py`（W2-011）：已是 Stop hook，已讀 transcript 尾部。

**4. 風險**
- 閾值穩定性未驗證：230K 為單一 session 觀察值（W2-012 已標註），不同任務 context 增長速度不同，固定閾值有誤判風險。
- 提示疲勞：超閾值後每回合觸發，需去重機制（state file，提示一次）。
- 非阻擋設計：須 exit 0 + stderr（提示性質），不可如 malformed-detector exit 2 強制打斷對話。

### 效益再評估（修正 W2-012 自評）

W2-012 自評「marginal（W2-011 已足夠）」。技術調查發現此評估漏算一項：W2-011 消除的是「malformed 靜默失敗」**危害**，但未消除 **retry 的 token 開銷**。~200K 後 41.5% 復發率 → 每 2.4 個 tool-call turn 有 1 次 retry，每次浪費一回合 + 重發 token。context-depth-warning 的真實效益是提示用戶在進入高 retry 窗口前 /clear，省下累積 retry 開銷——非 marginal，屬中等（依 session 長度而定）。

## Solution

### 三方案評估（acceptance 2/3 決策依據）

| 方案 | 描述 | 成本 | 主要風險 | 效益 |
|------|------|------|---------|------|
| A. 獨立 Stop hook | 新增 context-depth-warning Stop hook，讀 cache_read，超閾值 + 去重提示 | 第 89 個 hook + 閾值/去重/測試 | 閾值穩定性未驗證 + 提示疲勞 | 主動覆蓋所有高 context 場景 |
| B. 不實作 | 採 W2-012 結論結案，W2-011 已攔危害，retry 開銷可接受 | 零 | 無 | 無（保留 retry 開銷） |
| C. 附帶於既有 malformed-detector | 不新增 hook；malformed-detector 偵測到 malformed 時，於重發指令 stderr 後附一句「context 已深，建議 /clear」 | 改現有 Stop hook ~10 行 + 測試 | 低 | 在真實高風險點提示，消除閾值/疲勞風險 |

### C 方案關鍵洞察

malformed 事件本身就是「context 已深」的高特異性訊號（W2-012 證據：onset 後才出現，~230K 前 0 次）。因此提示只在 malformed 實際發生時附帶 → **自動解決閾值穩定性 + 提示疲勞兩大風險**（不需猜 cache_read 閾值、不會在無事時打擾），且零新增 hook（不增加 88→89 維護負擔，file-size-guardian 已顯示 27 超標 / 42 警戒）。

代價：C 不覆蓋「context 深但尚未發生 malformed」的提前提示窗口。但考量 (1) malformed 未發生時 retry 開銷為零；(2) 一旦發生即附帶提示，用戶仍能在進入高頻 retry 窗口初期 /clear，C 捕捉了主要效益。

### PM 評估傾向

**C（最低成本 + 消除主要風險 + 捕捉主要效益）**。此傾向基於技術證據（malformed = context 深的代理訊號），非估時或便利。屬 P3，最終由用戶決定走 acceptance 2（實作 A 或 C）或 acceptance 3（結案 B）。
