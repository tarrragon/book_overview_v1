## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### Phase 1 功能規格產出

產出文件：`docs/spec/data-management/tag-tree-model-spec.md`（SPEC-010，Full 模式 6 區段）。

依 W2-001 SA 方案 A + 四視角校正 + 4 用戶決策 + W2-003 授權調查設計，涵蓋：

1. **Schema 規格**：TAG_CATEGORY_SCHEMA 加 parentId（nullable/default null）+ TAG_TREE_MAX_DEPTH=4。唯一鍵語意由全域 name 改為 scoped (parentId, name)，列為首要決策項。明列 3 處全域 dup 檢查改 scoped 的精確位置：TagSchema.js validateTagCategory:81-83、tag-storage-adapter.js createTagCategory:256-258、computeMergeResult:917-918。經 makeCategoryKey 抽象共用（沿用既有 makeTagKey:842 pattern）避免 ARCH-020。

2. **cascade 分層語意**（決策 2 WRAP）：刪非葉禁止（has_children）/ 刪葉轉 Uncategorized / 子樹 opt-in + 確認對話框 / isSystem 保護。parentId 防護（循環/MAX_DEPTH/引用存在性）落 create + update 雙寫入路徑（修 linux#2：update 既有不呼叫 validate 缺陷）。GWT 場景組 A-F 共 20 三元組。

3. **載入機制**（MV3 校正）：onInstalled 頂層同步註冊 + onStartup 補償冪等 + 整批原子 storage.local.set；明文 storage.local（非 sync，避 8KB/item reject）；預裝走獨立 upsert 確定性 ID（sys_cat_*）不經 createTagCategory 隨機 ID。

4. **同步與打包**：三層同步策略（預裝本地重建不 sync / 用戶差異 LWW+tombstone / version flag local）；資料檔 bundle 打包決策 + build.js COPY_PATHS 影響。

5. **決策固化表**（7 項）：任意層級掛載、metadata.formatVersion（非 schema_version）、預裝二層 + 第三層 lazy-load 移出 scope、bundle 打包、makeCategoryKey 共用。Tag 管理 UI 服務契約（rename/move/batchDelete/batchMove/searchTags）。
