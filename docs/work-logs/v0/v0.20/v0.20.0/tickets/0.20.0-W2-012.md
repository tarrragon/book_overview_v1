## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

**現象定性**：模型在生成 tool-call XML 標記前，100% 一致地前綴一個游離 token `count`（17/17 次，零變異），隨後輸出缺少 `antml:` 命名空間前綴的裸 `<invoke name="...">` 標記。harness 無法解析為 tool call，將整段視為純文字渲染。

**根因假設（evidence-backed，非定論）**：

**H4（採信，新假設）：token-level generation artifact — `count` 是 tool-call 啟動序列的 misfired prefix token**

證據鏈：
1. `count` 在所有 17 次事件中 100% 一致出現，無任何變異（不是 `counter`、`counting`、`account`），且始終緊接 `\n<invoke`
2. 舊 session transcript 中，**onset 之前（127 個 assistant turns）context 中無任何 `count` token 出現**——排除 priming/pollution 假說
3. `count` 的出現與模型嘗試輸出 `<function_calls>` 結構的時機完全同步——它不出現在 prose 中，只出現在 tool-call boundary
4. 模型本應生成 `<function_calls>\n<invoke name="...">` 序列，但生成路徑偏移為 `count\n<invoke name="...">`——`count` 可能是 tool-call 啟動 token 序列中一個被 sampling 過程異常暴露的 internal token（正常情況下被 harness 的結構化輸出機制攔截或替換）

**H1（priming）：反證**
- onset 前 127 turns 無 `count` 出現，排除 context priming
- session 中無 `wc -l` 類計數操作或 `count` 討論

**H2（邊界）：部分支持，但不足以解釋 `count` 的一致性**
- 17 次均發生在 prose→tool-call 切換邊界，支持邊界假說
- 但邊界假說無法解釋為何是 `count` 而非其他 token——必須是更深層的 token generation 機制

**H3（depth/attention）：部分支持**
- onset 發生在 cache_read_input_tokens = 230,609（~230K tokens 累積）
- 之前 127 turns / 73 次正常 tool call 無事件
- onset 後 malformed rate = 17/41 = 41.5%（tool-call turns）
- 但 onset 後正常 tool call 仍可成功（25/41），顯示非完全退化，更像是 sampling 不穩定性增加

### 影響範圍

- 影響層：PM 主線程 tool-call 生成（模型層面）
- 影響檔案：無 src 檔案受影響（純 model output 層面問題）
- 影響程度：tool call 靜默失敗需 retry，W2-011 hook 已有效攔截

## Solution

### 根因結論（假設層級，非定論）

**主要假說 H4**：`count` 是模型在生成 tool-call XML 啟動序列時，因 context 深度累積（~230K+ tokens）導致 sampling 不穩定，將 internal tool-call 啟動 token 異常暴露到文字輸出。證據：(1) 100% 一致的 `count` token 且無 context priming 來源；(2) 與 `antml:` 命名空間前綴遺漏共現（同一 generation 路徑偏移）；(3) 閾值效應（230K 前 0%，230K 後 41.5%）。

**H3 為 contributing factor**（非獨立根因）：context 深度與事件頻率正相關，但 onset 後仍有 58.5% 正常 tool call，顯示深度增加的是 sampling 偏移概率而非確定性觸發。

| PC-177 | W2-011 的文字品質修正（W2-011.5） | 偵測面的衍生維護 |
