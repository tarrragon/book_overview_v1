## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

W2-006 方案 A：在 src/background/background.js 頂層 onMessage listener 補洞，使 init await 期間漏接的業務訊息得以緩衝並於 init 完成後 flush 至 MessageRouter。src/background/messaging/message-router.js 不改（只透過 background.js 補洞）。

### 變更內容（src/background/background.js）

1. 新增 import：OperationResult、ErrorCodes（src/core/errors/），用於 overflow / flush 失敗回應，遵專案錯誤體系（禁 throw 字串字面）。
2. 新增頂層狀態：const pendingMessages = [] + const MAX_PENDING_MESSAGES = 50。
3. 新增三個 local dict 訊息 key：INIT_BUFFER_FLUSH / INIT_BUFFER_OVERFLOW / INIT_BUFFER_FLUSH_ERROR（含 placeholder，避免 Logger 渲染 [Missing: KEY]）。
4. 改寫頂層 onMessage listener 分支邏輯（三種未就緒情境）：
   - 系統已就緒（isInitialized && !emergencyMode）→ 回 undefined 不攔截，交 MessageRouter listener。
   - GET_SYSTEM_STATUS（未就緒）→ 回 baseline 狀態（不緩衝），保留 W2-002 行為。
   - emergency 模式 → 業務訊息回受限提示（無 coordinator 可 flush，故不緩衝）。
   - init 進行中 → 呼叫 bufferMessageDuringInit 緩衝三元組，return true 保通道。
5. 新增 bufferMessageDuringInit(message, sender, sendResponse)：未滿則 push 三元組 + return true；達上限 50 則回 OperationResult.failure(...).toJSON()（不入緩衝）+ return true。
6. 新增 flushPendingMessages()：splice 全部緩衝清空後 FIFO 逐一 await messageRouter.routeMessage；單條失敗以 OperationResult 回應 + 記日誌，不中斷後續（錯誤隔離）；messageRouter 缺失時降級回受限提示不靜默吞掉。
7. initializeBackgroundSystem 於 isInitialized = true 後加 await flushPendingMessages()（此時 coordinator 已 start，MessageRouter listener 已註冊；init 後新訊息由 MessageRouter 直接接手）。

### 設計決策

- sendResponse 通道：頂層 listener 緩衝分支 return true，保持 MV3 異步通道開啟，使 flush 時的 routeMessage 回應能送回原 caller。
- flush 時機：選在 isInitialized = true 之後（coordinator.start 完成），確保 MessageRouter 已就緒。
- overflow 第 51 條不入緩衝：避免 SW 喚醒風暴下無上限記憶體成長；回 OperationResult 失敗讓 caller 可重試。
- 錯誤隔離：flush 迴圈單條 try/catch，sendResponse 二次失敗（通道已關）僅記日誌不拋出。

### 不在本 ticket 範圍

- MessageRouter.setupMessageListener 保留不變（約束）。
- coordinator 訊息佇列（messageRouter.messageQueue）機制不動，本 ticket 僅補 background.js 頂層 init 期間漏洞。
