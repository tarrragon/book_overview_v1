## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

MessageRouter.setupMessageListener()（message-router.js:151-173）在 `_doInitialize()` 內呼叫，而 `_doInitialize()` 是 coordinator async init 鏈的一環：

```
background.js 頂層同步
  → registerLifecycleListeners()  // 頂層 baseline onMessage listener 已註冊
  → initializeBackgroundSystem()  // async，不阻塞頂層
    → coordinator.initialize()    // await
      → initializeCoreDependencies()  // await（EventCoordinator 等）
      → createFunctionalModules()     // await（建 MessageRouter 實例）
      → initializeAllModules()        // await
        → messageRouter.initialize()  // await
          → _doInitialize()           // await
            → setupMessageListener()  // 此處才 addListener（message-router.js:166）
    → coordinator.start()         // await
      → messageRouter.start()     // processQueuedMessages()
```

MessageRouter 的 `chrome.runtime.onMessage.addListener` 在 coordinator 多層 await 之後才執行，違反 MV3「listener 須在 SW 模組頂層同步階段註冊」鐵則。

### 影響範圍

**檔案**：`src/background/messaging/message-router.js`（主要）、`src/background/background-coordinator.js`（init 時序）、`src/background/background.js`（頂層 baseline listener 交互）

**功能影響**：

W2-002 已在 background.js 加頂層 baseline onMessage listener（行 222-250），提供 init 期間的緩解：
- `isInitialized && !emergencyMode` 時回傳 `undefined`（不攔截），交 MessageRouter 處理
- init 進行中時：GET_SYSTEM_STATUS 回傳狀態，其餘訊息回傳「系統初始化中」錯誤

**實際漏接風險評估**：

| 時間窗口 | 行為 | 風險 |
|---------|------|------|
| SW 頂層同步階段（registerLifecycleListeners 後、initializeBackgroundSystem await 期間） | baseline listener 攔截所有訊息，回覆「初始化中」 | 訊息不漏接但功能不可用（popup/content-script 收到錯誤回應） |
| coordinator init 完成、isInitialized=true 後 | baseline listener 回傳 undefined，MessageRouter listener 接手 | 正常路由，無風險 |
| SW 被終止後重新喚醒 | MV3 重新求值整個模組 → registerLifecycleListeners 同步註冊 → async init 重走 | 同第一行，init 期間功能不可用 |

**結論**：W2-002 的頂層 baseline listener 已消除「靜默漏接」風險（訊息不會被 Chrome 丟棄），但 init 期間正常業務訊息（PING、GET_STATUS、content-script 提取指令、popup 操作）全部收到錯誤回應，使用者體驗為「擴充功能短暫無回應」。

### 相關 Error Pattern

無直接對應的已知 error pattern。屬 MV3 listener 延後註冊缺陷家族（W2-002 source ticket 同類）。

## 重現實驗結果
### 實驗方法

靜態 code-trace 分析（非 runtime 重現）：追蹤 `chrome.runtime.onMessage.addListener` 在 SW 模組求值期間的註冊時機，對照 MV3 規範要求。

### 實驗執行

**步驟 1：確認 MessageRouter listener 註冊時機**

message-router.js:166 `chrome.runtime.onMessage.addListener(this.chromeMessageListener)` 位於 `setupMessageListener()` 內，被 `_doInitialize()` :73 呼叫。`_doInitialize` 是 BaseModule async 生命週期方法，由 `initialize()` await 呼叫。

呼叫鏈（全 await）：
```
background-coordinator._doInitialize() :106
  → initializeCoreDependencies() :112 [await — EventCoordinator init/start]
  → createFunctionalModules() :115 [await — 建 MessageRouter 實例，不註冊 listener]
  → initializeAllModules() :124 [await]
    → messageRouter.initialize() :358 [await — 呼叫 _doInitialize]
      → setupMessageListener() :73 [await — 此處 addListener]
```

**步驟 2：確認頂層 baseline listener 覆蓋範圍**

background.js:222-250 頂層 listener 行為：
- `isInitialized && !emergencyMode`（正常就緒）→ `return undefined`（不攔截）
- `message.type === 'GET_SYSTEM_STATUS'` → 回傳狀態（baseline 回應）
- 其餘訊息 → `sendResponse({ success: false, error: '系統初始化中...' })` + `return true`

init 期間（isInitialized=false），baseline listener 對**所有**訊息 `return true`（佔用 sendResponse 通道），MessageRouter 即使提前註冊也無法回應（Chrome 僅允許一個 listener 呼叫 sendResponse）。

**步驟 3：確認雙 listener 共存行為（init 完成後）**

init 完成後 `isInitialized=true`：
- baseline listener 回傳 `undefined`（不佔用通道）
- MessageRouter listener 正常路由

Chrome MV3 規範：多個 onMessage listener 依註冊順序逐一呼叫，第一個 `return true` 或 `sendResponse` 的佔用通道。baseline 回傳 `undefined` 等效「不處理」，後續 MessageRouter listener 可正常接手。

### 實驗發現

**已驗證事實**：
1. MessageRouter 的 onMessage listener 在 coordinator 多層 await（至少 4 層 async）之後才註冊——屬 MV3 延後註冊缺陷
2. W2-002 頂層 baseline listener 已消除靜默漏接：init 期間所有訊息都有回應（雖為錯誤回應）
3. init 完成後 baseline + MessageRouter 雙 listener 共存，行為正確（baseline 不攔截，MessageRouter 接手）
4. MessageRouter 已有 messageQueue 機制（行 38-41、571-593），但佇列僅在 `_doStart` 時 `processQueuedMessages`，且無人往佇列推訊息（現有 code 中 messageQueue 始終為空）

**仍未驗證假設**：
- init 耗時實際值（估計 50-200ms，取決於 EventCoordinator、I18nManager、各 domain coordinator 初始化速度）
- 真實使用場景中 init 期間收到業務訊息的頻率（popup 開啟觸發 GET_STATUS/PING 最常見）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 風險等級判定

**中低風險**。W2-002 頂層 baseline listener 已消除靜默漏接，init 期間訊息有錯誤回應而非被 Chrome 丟棄。殘餘問題是 init 視窗內（估計 50-200ms）業務功能不可用——但此視窗極短，且 popup/content-script 端已有 retry 機制。

### 修正方案

### 方案 A：頂層 baseline listener 擴展為完整路由（推薦）

**做法**：將 background.js 頂層 onMessage listener 從「baseline 回應」擴展為「完整訊息緩衝 + 延後路由」。init 期間收到的訊息暫存，coordinator 就緒後批次交給 MessageRouter 路由。

**修改範圍**：
- background.js 頂層 listener：init 期間將訊息（message, sender, sendResponse 三元組）推入緩衝陣列
- initializeBackgroundSystem 完成後：將緩衝的訊息逐一交給 messageRouter.routeMessage 處理
- MessageRouter.setupMessageListener 保留不變（init 完成後正常接手新訊息）

**利**：
- 最小改動：僅改 background.js，不動 MessageRouter 架構
- 利用既有 MessageRouter 路由邏輯，不重複實作
- init 期間訊息不遺失、不回錯誤，延後處理

**弊**：
- sendResponse 通道有 timeout 限制（Chrome 約 5 分鐘），init 若極慢，緩衝的 sendResponse 可能失效
- 需管理緩衝陣列上限，防止 init 卡死時無限堆積

### 方案 B：MessageRouter listener 提至 background.js 頂層同步註冊

**做法**：在 background.js 頂層建立一個同步 onMessage listener，內部委派給 MessageRouter 實例（待 init 完成後可用）。

**修改範圍**：
- background.js 頂層新增同步 listener，取代現有 baseline listener
- 需暴露 messageRouter 引用或建立全域 getter
- MessageRouter._doInitialize 移除 setupMessageListener 呼叫

**利**：
- listener 在 SW 頂層同步註冊，完全符合 MV3 規範
- init 完成後零延遲切換到正常路由

**弊**：
- 破壞 MessageRouter 封裝性（listener 管理從類別內部移到外部）
- background.js 需知道 MessageRouter 路由細節（或透過 getter 間接存取）
- coordinator 重啟時 listener 管理變複雜（頂層 listener 不隨 coordinator 重啟而移除/重建）

### 方案 C：維持現狀 + 加強 popup/content-script retry

**做法**：不改 background 端。在 popup 和 content-script 端加 retry 邏輯：收到「系統初始化中」回應時自動延遲重試。

**修改範圍**：popup 訊息發送層、content-script 訊息發送層

**利**：
- 不動 background 架構
- 符合分散式系統「caller 端 retry」慣例

**弊**：
- 每個 caller 都要加 retry，維護成本分散
- init 期間使用者仍看到短暫「無法連線」狀態
- 不解決 MV3 規範違反的根本問題

### 建議方案

**建議採方案 A（頂層 listener 緩衝 + 延後路由）**。

理由：
1. 改動最小（僅 background.js 約 20 行），不破壞 MessageRouter 封裝
2. init 視窗內訊息從「回錯誤」升級為「延後處理」，使用者無感
3. MessageRouter 既有 messageQueue 機制（行 38-41）可復用概念
4. 方案 B 破壞封裝且增加 coordinator 重啟複雜度，收益不對稱
5. 方案 C 不解決根因，只是分散成本

### Spawn 結論

**建議 spawn 1 個 IMP ticket**：實作方案 A（background.js 頂層 listener 緩衝 + 延後路由）。

範圍：
- 修改 background.js 頂層 onMessage listener：init 期間緩衝訊息
- initializeBackgroundSystem 完成後 flush 緩衝到 messageRouter
- 加 RED 測試：驗證 init 期間訊息被緩衝且 init 後正確路由
- 緩衝上限（建議 50 條）+ overflow 時回傳錯誤回應

優先級：P2（現有 baseline 已防靜默漏接，緩衝是體驗改善非緊急修復）。
