## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### false-positive 驗證實證（2026-06-06 PM session，retry-continuation 待補項範疇）

發現：本 hook 對「描述自身偵測目標的純文字訊息」會誤報。

情境：PM 在 session 結尾撰寫純文字 handoff（該訊息無任何工具調用），內文說明本 session 工具標記寫壞的成因時，引述了偵測器的觸發關鍵字（以 inline code 形式）。Stop hook 掃描該純文字訊息，將引述字面判為未解析工具標記，exit 2 要求重發——但該訊息本無工具可重發，形成偵測器與 meta 描述文字的遞迴誤判。

根因：PC-099 模式（偵測某反模式的機制在討論該反模式時自我誤報）。本 hook 缺對 meta 描述文字的豁免。

建議修復方向（下 session 評估）：
- 對 fenced code block / inline code（backtick 包覆）內的標記字面豁免
- 參考 PC-099 既有的 ticket 級 / 自我引用豁免機制設計

對照（true positive 有效性）：本 session 真實寫壞的工具調用共 6 次，hook 全部正確攔截並成功觸發 retry-continuation（PM 依提示重發後成功）；誤報僅發生於 meta 描述純文字，邊界明確。此實證同時驗證 retry-continuation 機制有效 + 暴露 false positive 邊界。

## 重現實驗結果

### 實驗方法

重現由兩條證據鏈構成：(1) 本 session 真實 malformed tool-call 觸發觀察（真陽）；(2) thyme 多視角審查以 importlib 載入 hook、直呼 `detect` / `strip_code_regions` 的 /tmp 重現 script（false-positive 根因定位）。

### 實驗執行

- 真陽重現：0.20.0 W2 session PM 工具調用於 invoke 標記前混入游離 count token 或漏 antml 前綴，6+ 次；hook 全部 exit 2 攔截並觸發 retry-continuation（PM 依提示重發後成功）。審查階段 thyme/basil 委員討論偵測器字面時，其 Stop event 亦實際觸發本 hook（活活印證 false-positive）。
- false-positive 重現：thyme 構造「4-space 縮排引述標記」（根因 A）與「跨行 backtick 引述標記」（根因 B）案例，實機呼叫 detect 確認命中 signature 1/2（誤報）；對照單行 inline 引述正確被剝除不誤報。

### 實驗發現

- 已驗證：malformed 標記在 tool 事件生成前發生，PreToolUse 結構上接不到；Stop hook 可攔（真陽 6/6）。
- 已驗證根因（共振）：`strip_code_regions` 漏剝 4-space 縮排 block（A）+ inline regex 拒跨行（B）→ 引述標記命中行首簽章誤報。
- 證偽：signature 4 非誤報源，為「游離 token 接 invoke」真陽偵測形態，禁削弱。
- 落地驗證：.1 修 strip（真陰不誤觸）、.2 內嵌 self-test、.3 meta-context 豁免（PC-099 marker）、.4 簽章補強（成對閉合）、.5 PC-177 文字。最終 self-test 真陰 6 + 真陽 4 全綠。
