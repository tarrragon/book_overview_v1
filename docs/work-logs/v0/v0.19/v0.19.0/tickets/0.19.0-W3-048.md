## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

PC-078（並行 session ticket 誤判）與 PC-141（並行 claim AC verification collision）暴露：當多個同 wave ticket 在不同 terminal / session 幾乎同時 claim 時，缺乏「claim 當下的同 wave in_progress 快照」，事後無法歸因衝突來源（誰先 claim、claim 時還有哪些同 wave ticket 處於 in_progress）。

W3-045 ANA 結論將阻擋型 hook（L1 PreToolUse:Bash 阻擋）排除（流程鎖補設計，非根本解），但 basil 視角補充：非阻擋型 observability hook 仍有獨立價值——只記錄不阻擋、零 race condition，作為事後歸因依據。符合 memory「Ticket Guidance Over Hooks」（hook 為防護手段非根本解；此處 hook 為記錄非阻擋）。

### 影響範圍

- 新增 `.claude/hooks/parallel-claim-audit-hook.py`（PostToolUse:Bash）
- 新增 `.claude/hooks/tests/test_parallel_claim_audit_hook.py`
- 寫入 `.claude/settings.json` PostToolUse:Bash 區塊
- 產出 audit log：`.claude/hook-logs/parallel-claim-audit.log`（JSONL）

### 相關 Error Pattern

- PC-078 並行 session ticket 誤判
- PC-141 監測 ANA 訊號分解 / 並行 claim AC verification collision
- 設計遵循 PC-068（新建資產前 grep 既有 hook，沿用 changelog-update-hook.py 的 PostToolUse:Bash 慣例與 hook_utils API）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 設計決策

| 決策點 | 選擇 | 理由 |
|-------|------|------|
| 快照來源 | 直接讀 ticket 檔案 frontmatter（`find_ticket_files` + `parse_ticket_frontmatter`） | 不執行 `ticket track list` 子程序，避免與並行 claim 競爭 → 零 race condition |
| 阻擋性 | 一律 `exit 0` | 純 observability，任何失敗只 log 不影響 claim（acceptance 3） |
| 失敗可觀測 | `try/except` 包裹快照與寫檔，例外同時寫 `logger.info` + `sys.stderr` | quality-baseline 規則 4 雙通道（stderr + log） |
| claim 成功判定 | 掃描 stdout/stderr 失敗訊號（already claimed / error / failed / blocked / traceback） | observability 寧可多記，明確失敗才跳過 |
| log 格式 | JSONL append（每行一筆 JSON） | 便於事後 grep / jq 歸因，append 模式不覆蓋歷史 |
| wave 過濾 | 以被 claim 的 ticket_id 推導 wave，過濾跨 wave 與非 in_progress | 快照精準對應同 wave 並行情境 |

### 實作要點

- 命令偵測：`\bticket\s+track\s+claim\s+(\S+)`，排除 `-` 開頭選項（如 `claim --help`）誤判
- 快照欄位：`{ ticket_id, wave, timestamp, same_wave_in_progress_count, same_wave_in_progress: [{id, started_at, who}] }`
- 被 claim 的 ticket 自身納入快照（claim 後即為 in_progress）
- 沿用 `changelog-update-hook.py` 的 PEP 723 / UV single-file header 與 `hook_utils` 統一 API（`setup_hook_logging` / `run_hook_safely` / `read_json_from_stdin`）

### 不在本 ticket 範圍

- L1 阻擋 hook（W3-045 ANA 已排除）
- L3-b CLI AC verification 行為（根本解，獨立追蹤）

### multi_view_status

非阻擋純 observability hook，P3 範圍單一（檔案範圍 = 1 新 hook + 1 測試 + settings 註冊），不觸發強制多視角審查門檻。設計沿用既有 PostToolUse:Bash 慣例，已 dogfooding 驗證真實 trigger 邏輯。
