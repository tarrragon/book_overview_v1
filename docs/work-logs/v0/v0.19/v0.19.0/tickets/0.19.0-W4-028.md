### v2.1.157 七項更新逐項評估

來源：Claude Code v2.1.157 release notes（用戶 /release-notes 查閱）。評估維度：是否改變本專案既有框架流程、價值、衝突風險。

| # | 更新項 | 與本專案關聯 | 初判 |
|---|--------|------------|------|
| 1 | `.claude/skills` plugin 自動載入 | 本專案 44 skill 屬 standalone 機制，運作正常；plugin 機制帶 namespace（`/x:y`），重構成本高且無立即需求 | 不採用（記為未來選項） |
| 2 | `claude plugin init` scaffold | 同上，無 plugin 化開發需求 | 不採用 |
| 3 | `/plugin` autocomplete | 純互動 UX，無框架調整空間 | 不採用 |
| 4 | settings.json `agent` 欄位（dispatched session 預設 agent，`--agent` 覆蓋） | 須先釐清：本專案以 `Agent(subagent_type=...)` 派發，未必走 dispatched session 路徑；且易與 PC-059（subagent permissions.allow 無效）混淆 | 待驗證 |
| 5 | `EnterWorktree` mid-session 切換 | 本專案 worktree 工作流現靠 runtime 自動切 cwd（見記憶 PM cwd auto-switch）；此工具讓切換更明確，但也使「切換後查 branch+pwd」更必要 | 部分採用（DOC 補註） |
| 6 | OTEL `tool_decision` 新增 `tool_parameters`（需 `OTEL_LOG_TOOL_DETAILS=1`） | 可作 dispatch_stats.py 外部遙測來源；但開啟後 bash 全文進遙測有隱私成本 | 部分採用（可選增強，低優先） |
| 7 | Claude-managed worktree 完成後解鎖 | 直接解決孤兒 worktree 清理痛點（release note 開頭即有 orphaned worktree 警告）；現可直接 `git worktree prune` 免 unlock 前置 | 採用（DOC 更新清理 SOP） |

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 決策（採用範圍）

**W — Widen options**：採用粒度非「全採/全不採」二元，分四級：立即採用（改規範）/ 待驗證後決定 / 部分採用（補註能力）/ 不採用（記未來選項）。worktree 第 5、7 項同領域合併一個 DOC。

**R — Reality-test**：
- 第 7 項（worktree 解鎖）：確實移除 `git worktree unlock` 前置，清理為高頻動作 → 價值確定，採用。
- 第 5 項（mid-session 切換）：本專案目前未顯式呼叫 EnterWorktree（靠 runtime 自動切 cwd），價值在「補註能力 + 強化查核」而非引入新流程 → 輕量補註。
- 第 4 項（agent 欄位）：未驗證本專案 `Agent(subagent_type=...)` 派發是否走 dispatched-session 路徑，貿然寫規則有與 PC-059 混淆的高風險 → 須先驗證，不直接改規則。
- 第 6 項（OTEL）：開啟後 bash 全文進遙測有隱私成本，非剛需 → 補記為可選能力，低優先。

**A — Attain distance**：機會成本排序——worktree 清理 SOP 對每次 ticket 收尾都有收益（高）；agent 欄位驗證收益不確定（中低）；plugin 重構收益遠低於重構成本（不做）。

**P — Prepare to be wrong（絆腳索）**：
- 若驗證發現本專案不走 dispatched-session 路徑 → 第 4 項降級為「於 PC-059 附近補一句澄清，避免後人誤用」，不開新規則。
- 若 `git worktree prune` 實測仍遇 lock → 回退保留現有手動 unlock 說明。
