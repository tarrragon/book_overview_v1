### 實作摘要

新增獨立 E2E 測試檔 `tests/e2e/browser/uc03-export.e2e.test.js`（8 個 test case），重用 W1-008 框架（ExtensionTestSetup + helpers/storage-reader + helpers/extraction-flow + helpers/timeouts），未重建基礎設施、未改既有 2 測試檔。

### 測試設計

完整經過真實匯出鏈：提取 fixture 5 本書 -> 開啟 overview 頁（event-driven 自動讀 storage 渲染清單）-> 真實 `page.click('#exportJSONBtn' / '#exportCSVBtn')` -> overview-page-controller `handleExportJSONv2` / `handleExportCSVv2` -> BookDataExporter v2 -> blob 下載。

blob 攔截：page context 內 override `URL.createObjectURL`，於 FileReader.onload callback 將已解析字串 push 至 `window.__capturedExports`，測試端 poll 純資料陣列（不跨 Puppeteer bridge await page 內 Promise，避免 FileReader 未 resolve 時 evaluate 無 timeout hang）。僅替換「blob -> object URL」這一步取得內容，不繞過任何匯出業務邏輯。

### 斷言分組

- G1（[SETUP]）：提取 5 本書 + overview 渲染 5 列
- G2（[JSON]，acceptance #1）：root 含 Interchange Format v2 四區段（metadata/tagCategories/tags/books）；metadata.formatVersion='2.0.0'、source='readmoo-book-extractor'、totalBooks 與 books.length 交叉驗證；每本書 readingStatus 為 6 狀態合法值、tagIds 為陣列（spec §3.5 強制規範）
- G3（[CSV]，acceptance #2）：MIME=text/csv；header（跳過 # source-limitation 註解列）含 readingStatus + tagIds + tagNames + tagCategories；資料列數=5 且每列 readingStatus 為 6 狀態合法值

### 規格依據

- JSON：docs/spec/export-interchange-format-v2.md §3
- CSV：docs/spec/export-interchange-format-v2.md §5（overview 採 COMPLETE_V2 preset；陣列以 '; ' 分隔）

### spike 解決的兩個 hang 根因（W4-001.1 實證）

1. 跨 bridge await page 內 Promise -> 改為 poll 純資料陣列
2. browser.close() 在多 Chrome instance 連續啟動的全套件情境偶爾 hang -> afterAll 以 Promise.race(cleanup, 15s timeout) bounded，程序退出時 Chrome 仍被回收

### 受測 src 唯讀

未修改任何 src。受測入口：src/overview/overview.html（#exportCSVBtn L30 / #exportJSONBtn L31）、src/overview/overview-page-controller.js（handleExport*v2 / _triggerExportDownload）、src/export/book-data-exporter.js（v2 匯出）。

### Regression 發現（移交 PM，見 Exit Status）

csv-export-spec.md 與 export-interchange-format-v2.md 對 CSV 陣列分隔符描述衝突（前者 '|'，後者 '; '）；實作遵循後者（approved interchange spec）。屬文件一致性問題，非功能 regression，建議 PM 建 DOC follow-up。

### 移交 PM：文件一致性發現（quality-baseline 規則 5，建議建 DOC follow-up）

**發現**：CSV 陣列欄位分隔符在兩份規格文件描述衝突。

| 來源 | 對 CSV 陣列（authors/tagIds）分隔符的描述 |
|------|------------------------------------------|
| docs/spec/csv-export-spec.md（v1.0，2026-05-17）§陣列序列化策略 | 管道符 `\|`（含 escape 規則 `\\\|`） |
| docs/spec/export-interchange-format-v2.md（v2.0.0，approved）§4.2 / §5.1 | 分號加空格 `; ` |

**實作現況**：`src/export/book-data-exporter.js` `_processFieldValue`（L178-180）與 `_resolveTagNames` / `_resolveTagCategoryNames` 一律用 `'; '`，遵循 approved interchange spec（export-interchange-format-v2.md）。

**判定**：非功能 regression（實作與 approved spec 一致），屬 csv-export-spec.md 未對齊 approved interchange spec 的文件一致性問題。本 E2E 測試斷言對齊實作與 approved spec，未驗證分隔符字面（避免綁定有爭議的描述）。

**建議**：PM 建 DOC follow-up，更新 csv-export-spec.md §陣列序列化策略對齊 export-interchange-format-v2.md 的 `; ` 分隔，或顯性標註兩者版本適用範圍。本 ticket 範圍唯讀 src，未擅改 src 或 spec。
