### 問題根因

PC-159 識別出 IMP ticket 安裝指令在 PM/agent 既有環境通過驗證但 fresh shell 失敗的系統性風險。W3-050（codegraph placeholder package）和 W3-051（sys.path hack 在 uv tool install 後失效）為實際案例。

**根因鏈**：IMP ticket 撰寫安裝指令時缺乏三道防線：

| 防線 | 現況 | 缺口 |
|------|------|------|
| Hook 層（自動偵測） | 無任何 hook 偵測安裝指南檔案修改 | 編輯 development-setup.md / environment-recovery-guide.md 時無 reminder |
| Acceptance schema 層 | IMP acceptance 模板無 fresh shell 驗證條件 | 含安裝指令的 IMP 可跳過 fresh shell 驗證 |
| 文件層（自律） | development-setup.md 無 scoped name 規範聲明 | 短名假設可無阻擋地寫入 |

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 三層防護方案評估

| 防線 | 方案 | 成本 | 收益 | 建議 |
|------|------|------|------|------|
| (a) PostToolUse hook reminder | 新建 install-guide-edit-reminder-hook.py，PostToolUse:Edit/Write 偵測 file_path 含 `development-setup` / `environment-recovery`，輸出 INFO reminder | 低（30-50 行 Python，hook 基礎設施成熟） | 中（提醒效果，不阻擋；PM/agent 可忽略但有訊號） | 建議實作 |
| (b) ticket-body-schema IMP acceptance 模板 | IMP acceptance 模板新增條件欄位：「含安裝指令時須附 fresh shell 驗證結果」 | 低（schema 文件修改） | 高（acceptance-gate-hook 在 complete 時強制檢查） | 建議實作 |
| (c) development-setup.md 開頭聲明 | 新增 scoped name 規範聲明 + 短名禁止 | 極低（文件修改） | 中（自律層，降低短名假設機率） | 建議實作（與修正 L54 合併） |

### 方案 (a) Hook 設計細節

**觸發時機**：PostToolUse:Edit / PostToolUse:Write

**偵測條件**：stdin JSON 的 `tool_input.file_path` 含以下 pattern 之一：
- `development-setup`
- `environment-recovery`
- `docs/` 下含 `install` / `setup` / `recovery` 關鍵字的 `.md` 檔

**輸出**：INFO 級 reminder（不阻擋，exit 0）：
```
[install-guide-reminder] 您正在編輯安裝指南檔案。請確認：
1. 安裝指令使用 scoped package name（如 @scope/pkg-name），非短名
2. 指令已在 fresh shell 驗證（非既有環境）
3. 若為 IMP ticket，acceptance 含 fresh shell 驗證條件
```

**與既有 hook 的衝突評估**：
- 不與 PC-093 hook 豁免機制（phase4-decision-enforcement）衝突（不同偵測對象）
- 不與 style-guardian-hook 衝突（不同觸發路徑，互補）
- 純 INFO 輸出，無 exit 2 deny，無誤判風險

### 方案 (b) Acceptance Schema 補強細節

在 ticket-body-schema.md IMP section 新增條件式 acceptance 指引：

```markdown
#### 安裝指令 IMP 額外 acceptance（PC-159 防護）

IMP ticket 含以下特徵時，acceptance 必須包含 fresh shell 驗證條件：
- ticket what/how 含 `npm install` / `pip install` / `brew install` / `uv tool install` 等安裝動詞
- ticket where.files 含 `development-setup.md` / `environment-recovery-guide.md`

必須至少一項：
- [ ] 安裝指令在 fresh shell（新 terminal、無 .bashrc/.zshrc 以外環境變數）執行通過
- [ ] package name 為完整 scoped name（含 @scope/ 前綴或完整 registry URL）
- [ ] 附 package registry 驗證截圖/輸出（npm info / pip show）
```

**implementation 路徑**：此為文件層指引，非 hook 自動偵測。acceptance-gate-hook 不直接 parse 安裝指令 pattern（複雜度過高、誤判率高）。依賴 PM 在建 ticket 時手動加入此 acceptance 條件。

### 方案 (c) 文件層聲明

development-setup.md 開頭新增：

```markdown
> **安裝指令規範（PC-159）**：本文件所有安裝指令必須使用完整 scoped package name + version range，
> 禁止未驗證的短名假設。每次修改安裝指令後須在 fresh shell 驗證。
```

同時修正 L54 `npm install -g codegraph` → `npm install -g @astudioplus/codegraph-mcp`。
