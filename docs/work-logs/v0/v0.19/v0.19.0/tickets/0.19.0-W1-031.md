## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

tests/integration/cross-module/event-system-integration.test.js 有兩處絕對記憶體門檻斷言，屬 test-assertion-design-rules 規則 1 反模式（絕對門檻當 pass-fail）：

| 行號 | 斷言 | 指標來源 |
|------|------|---------|
| 413 | `expect(subscriptionAnalysis.memoryFootprint).toBeLessThan(10 * 1024 * 1024)` | event-system-analyzer.js `_measureDeliveryPerformance`（第 1293 行）：`baseMemoryFootprint + memoryDelta`，memoryDelta = `finalMemory.heapUsed - initialMemory.heapUsed` |
| 787 | `expect(performanceAnalysis.peakMemoryUsage).toBeLessThan(150 * 1024 * 1024)` | event-system-analyzer.js 第 2740 行：執行期間取樣 `Math.max(peakMemoryUsage, currentMemory.heapUsed)` |

兩處指標皆由真實 `process.memoryUsage().heapUsed` 計算，非 mock 固定回傳值，**不適用** test-assertion-design-rules 規則 1 的 mock 豁免。全套件並行（195 suites）下記憶體負載與 GC 壓力使 heapUsed 波動，line 787 實測約 154MB 超過 150MB 門檻而 flaky。

### 影響範圍

- 直接影響：event-system-integration.test.js 兩個 test case（「應該支援事件廣播和選擇性訂閱」、「應該提供事件系統的效能監控和分析」）。
- 間接影響：npm test 全套件通過率不穩定，違反 quality-baseline 規則 1。
- 非 W1-030 引入：斷言早於 commit e0bba4a8。

### 相關 Error Pattern

無直接對應 PC error-pattern。屬 test-assertion-design-rules 規則 1（W1-017 / W1-018 已建立的規則）反模式的延續案例。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

依 test-assertion-design-rules 規則 1（移除絕對門檻、保留功能正確性驗證）處理兩處斷言。未採規則 2（移至 tests/perf/），因兩個 test case 除記憶體門檻外另含多項功能驗證（投遞率、過濾開銷、延遲、CPU、告警處理），整體搬移會讓功能驗證脫離 npm test 主套件；只移除記憶體門檻行符合規則 1「計時門檻混在功能測試檔 → 只移除計時行，該檔其他功能斷言保留原位」。

| 行號 | 原斷言 | 新斷言 |
|------|--------|--------|
| 413 | `expect(subscriptionAnalysis.memoryFootprint).toBeLessThan(10 * 1024 * 1024)` | `expect(typeof subscriptionAnalysis.memoryFootprint).toBe('number')` + `expect(subscriptionAnalysis.memoryFootprint).toBeGreaterThan(0)` |
| 787 | `expect(performanceAnalysis.peakMemoryUsage).toBeLessThan(150 * 1024 * 1024)` | `expect(typeof performanceAnalysis.peakMemoryUsage).toBe('number')` + `expect(performanceAnalysis.peakMemoryUsage).toBeGreaterThan(0)` |

### 設計決策

移除絕對門檻後改驗證「指標為有效正數」，確認分析器確實回傳記憶體足跡 / 峰值記憶體欄位（防止欄位漏算或回傳 undefined 的回歸），同時與環境記憶體負載解耦，消除 flaky。每處新斷言上方加 4-6 行註解說明指標來源、flaky 根因與規則依據。
