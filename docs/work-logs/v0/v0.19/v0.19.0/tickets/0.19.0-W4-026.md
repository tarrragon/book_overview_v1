## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

command-entrance-gate-hook 的認領閘門在「doc 編輯 prompt + 描述含開發詞 + 存在無關 pending ticket」情境下誤擋合法操作。本 ANA 以 WRAP 框架檢討其是否過時，結論為設計缺陷需修（非退役）。

### 問題根因

`command-entrance-gate-hook.py` 認領閘門誤擋合法 doc 編輯 prompt。三層共振（詳細逐層分析見 Solution）：

1. L1 `is_development_command`（L382）：以 substring 比對開發關鍵字，prompt「反映**新增**的監測波次」中描述性「新增」命中 IMPLEMENT_KEYWORDS，誤判為開發命令（無 word-boundary、無語境）。
2. L2 `get_latest_pending_ticket`（L607）：對全版本 ticket 按 `st_mtime` 回傳最新 pending，與 prompt/版本無關。
3. L3 `check_ticket_status`（L649-665）：pending 直接阻擋，relevance 檢查只對 in_progress 生效。

### 影響範圍

`.claude/hooks/command-entrance-gate-hook.py`（UserPromptSubmit 閘門）。影響所有「含描述性開發詞 + 存在無關 pending ticket」的 PM prompt，尤以 doc/worklog 編輯意圖為甚。

### 相關 Error Pattern

PC-113（短英文標記缺字邊界）、PC-138（表格內合法標記誤判）、PC-144（Solution 內合法待辦標記誤判）、ARCH-018（hook 單維度強制與真實意圖衝突）——同屬 hook/validator 關鍵字 substring false-positive 家族。

> 註：本 ticket complete 過程二次實證此家族——body-check validator 因本節原引用待辦標記字面（描述性引用而非佔位符）而誤判 Problem Analysis 為空，改寫為中文描述後解除。屬 PC-144 既有範疇。

## 重現實驗結果

### 實驗方法

本次誤判即為現成 live 重現：UserPromptSubmit 提交 prompt「更新 v0.20.0-main.md 反映新增的監測波次」，觀察 hook 是否阻擋及阻擋對象。對照 hook 源碼三層函式行為。

### 實驗執行

- 輸入 prompt：「更新 v0.20.0-main.md 反映新增的監測波次」（純 doc 編輯意圖，未提任何 ticket ID）。
- hook 實際輸出：`錯誤：Ticket 0.20.0-W1-013 尚未認領`，要求 claim。
- 背景：W1-013 為數分鐘前 migrate 遷入的 ticket（mtime 最新），prompt 與其毫無關聯。
- 源碼追蹤：`is_development_command` 對「新增」回 True（L382-385）→ `check_ticket_status` → `get_latest_pending_ticket` 按 mtime 回 W1-013（L607-613）→ status=pending 直接 block（L649-653），未經 relevance 檢查。

### 實驗發現

已驗證事實：(1) 誤判來自「新增」substring 命中而非「更新」（「更新」不在 keyword 集）；(2) 阻擋對象 W1-013 純由 mtime 排序決定，與 prompt 無關；(3) pending 阻擋路徑無 relevance 閘門。已驗證為三層共振設計缺陷，非單點 bug。修復方向（組合 F）見 Solution。
