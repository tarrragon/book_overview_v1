## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

兩失敗為同一行號漂移的一體兩面（pre-existing 測試碼問題，非 runtime regression）。

#### 根因
`ALLOWED_FILTER_SITES` 以 `commands/resume.py:195` 為 allowlist key，但 W3-018.1 在前方插入 target_id 反向查找邏輯（is_task_chain_direction 分支），使 `from_status != "completed"` 過濾行由 195 漂移至 201。
- `test_no_bare_status_completed_negation_filter`：lint 掃到 201 不在 allowlist（key 仍為 195）→ 誤報為新違規。
- `test_allowlist_sites_still_exist`：allowlist key 195 對應行已不含 `!= "completed"` → 報 allowlist 漂移。

實際開檔確認當前正確行號為 **201**（`if record.from_status != "completed":`），182 為註解（lint 已 skip）。

### 去耦合決策（脆弱設計評估）

**結論：採用 content-anchored 去耦合並實作（低成本，範圍限 lint 測試檔）。**

原設計以 `檔名:行號` 為 allowlist key，任何上游插入行都會破壞，屬 quality-baseline 規則 3 設計問題。評估後採低成本去耦合：

- 將 `ALLOWED_FILTER_SITES` 改為 `{ relative_file: [(content_substring, 註記), ...] }`，以「檔名 + 行內容子串」為錨點。
- 新增 `_is_allowed_site(rel_file, line_text)` helper；`test_no_bare_status_completed_negation_filter` 改以內容比對放行；`test_allowlist_sites_still_exist` 改以「子串行實際存在且仍匹配 pattern」驗證。
- 效果：站點只要語意行存在即視為合法，與物理行號無關，徹底消除行號耦合，未來上游插入行不再破壞 lint。
- 成本：僅 lint 測試檔（約 30 行），無 runtime 變更，落在 ticket 約束允許範圍（「lint 測試的錨點邏輯」）。無需 follow-up ticket。

### 影響範圍
僅 `tests/test_no_bare_status_completed_filter.py`。未改 runtime / resume.py。
