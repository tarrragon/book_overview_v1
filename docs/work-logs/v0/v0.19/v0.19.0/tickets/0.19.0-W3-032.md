## Task Summary

評估 /goal 指令（設定完成條件持續執行）與現有 ticket acceptance / Stop hook 機制的整合機會與衝突點

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

Claude Code v2.1.139 新增 `/goal` 指令，允許用戶設定 session-level 目標並持續執行直到達成。本專案已有 ticket-level acceptance 機制（frontmatter acceptance 欄位 + check-acceptance CLI + acceptance-gate-hook）。兩者在「完成條件定義」上有概念重疊，需釐清是否整合、如何整合、或維持獨立。

### 影響範圍

- `.claude/skills/ticket/` — ticket CLI（check-acceptance / complete 指令）
- `.claude/hooks/acceptance_checkers/` — acceptance-gate-hook 驗收閘門
- `.claude/pm-rules/ticket-lifecycle.md` — 驗收流程定義

### 相關 Error Pattern

無直接相關。間接相關：PC-055（AC 漂移）、PC-093（無 trigger 延後決策）

### 實驗方法

1. 盤點現有 ticket acceptance 機制的完整鏈路（frontmatter → CLI → hook → lifecycle）
2. 分析 `/goal` 指令的運作模型（session-level goal + evaluator + 持續執行）
3. 比對兩者的抽象層級、觸發時機、持久性、驗證方式

### 實驗執行

**現有 acceptance 機制鏈路盤點**：

| 元件 | 層級 | 持久性 | 觸發時機 |
|------|------|--------|---------|
| frontmatter `acceptance` 欄位 | ticket-level | 持久（.md 檔） | ticket 建立時定義 |
| `ticket track check-acceptance` | CLI 驗證 | 無（即時執行） | complete 前手動/自動 |
| `acceptance-gate-hook` | Hook 閘門 | 無（即時攔截） | complete 觸發時 |
| `claim_verification.py` | CLI 前置 | 無（即時執行） | claim 時自動 |
| `ticket-lifecycle.md` 驗收流程 | 規則定義 | 持久（規則檔） | PM 遵循 |

**`/goal` 指令運作模型**（基於 Claude Code v2.1.139 release notes + v2.1.143 修復）：

| 屬性 | `/goal` | ticket acceptance |
|------|---------|-------------------|
| 層級 | session-level | ticket-level |
| 持久性 | session 結束即消失 | .md 檔持久存在 |
| 定義者 | 用戶即時輸入 | PM 建立 ticket 時 |
| 驗證者 | Claude Code evaluator（自動） | hook + CLI（半自動） |
| 執行模式 | 持續執行直到 goal 達成 | 手動觸發 check/complete |
| 可追蹤性 | 無（session 內） | 有（ticket history + git） |
| 多步驟支援 | 單一 goal | 多條 acceptance 條件 |

### 實驗發現

**已驗證事實**：

1. `/goal` 與 ticket acceptance 運作在**完全不同的抽象層級**：session vs ticket。session 是暫態、不可追蹤；ticket 是持久、可追蹤、可審計。
2. `/goal` 的 evaluator 機制與 acceptance-gate-hook 的觸發機制**無直接交互**：/goal 是 Claude Code runtime 內部機制，不觸發 PreToolUse/PostToolUse hook；acceptance-gate-hook 是 ticket CLI complete 指令觸發。
3. `/goal` 的價值在於**自動化持續執行**（agentic loop），而非**品質閘門**（acceptance 的核心用途）。
4. v2.1.143 修復 /goal evaluator 在 background shell 誤觸發的 bug，表明此功能仍在早期穩定化階段。

**未驗證假設**：
- /goal 是否能讀取 ticket frontmatter 作為 goal 定義（需實際測試 /goal 的輸入格式）
- /goal evaluator 的失敗模式是否會與 Stop hook 產生衝突（需測試環境）

## WRAP 框架分析

### W — Widen（擴增選項）

| # | 整合方式 | 說明 |
|---|---------|------|
| A | /goal 取代 acceptance | 用 /goal 設定完成條件取代 ticket frontmatter acceptance |
| B | /goal 補強 acceptance 自動化 | PM 派發代理人時，用 /goal 自動化「持續執行直到 acceptance 滿足」的 agentic loop |
| C | /goal 補強 5W1H why | 用 /goal 設定 session-level 目標對齊 ticket why，作為方向指引而非驗收 |
| D | 不整合，平行存在 | /goal 作為用戶自選的 session 工具，ticket acceptance 維持現狀，互不干涉 |

### R — Reality Test（現實檢驗）

| 方案 | 與現有結構的衝擊 | 可行性 |
|------|----------------|--------|
| A（取代） | **破壞性**：/goal 為 session-level 暫態，無持久性、無 git 追蹤、無 hook 閘門。取代 acceptance 等同放棄品質閘門。acceptance-gate-hook 的 7 個 checker 全部失效。 | 不可行 |
| B（自動化補強） | **中度**：可在 PM 派發 prompt 加入「/goal: 完成 ticket AC[0]-AC[N]」。但 /goal evaluator 的判定邏輯不可客製化（由 Claude Code runtime 控制），與 check-acceptance CLI 的精確驗證（正則、指令執行）不同層級。且 v2.1.143 尚在修 bug，穩定度存疑。 | 有限可行，但收益低於風險 |
| C（方向指引） | **低度**：/goal 作為 session 方向提示，不觸碰 acceptance 機制。但 5W1H why 已有 ticket frontmatter 記錄，/goal 只是暫態重複。 | 可行但冗餘 |
| D（不整合） | **零衝擊**：兩者各自運作。用戶若想用 /goal 做 session 管理，不影響 ticket lifecycle。 | 完全可行 |

### A — Attain Distance（收益/風險評估）

**收益分析**：

| 方案 | 收益 | 風險 |
|------|------|------|
| A | 無（反向收益：品質下降） | 品質閘門全失、追蹤性歸零 |
| B | 代理人自動持續執行直到 AC 滿足，減少 PM 手動 check-acceptance 次數 | /goal evaluator 誤判（v2.1.143 bug 佐證）→ 代理人無限循環或過早停止；兩套 source of truth（/goal 判定 vs hook 判定）不一致時無調解機制 |
| C | 代理人有 session-level 方向感 | 與 ticket why 冗餘；/goal 無法引用 ticket frontmatter |
| D | 零風險、零成本；用戶可自行選擇是否使用 /goal | 無主動收益（但也無機會成本，/goal 是平台功能，不需專案整合即可使用） |

**核心判斷**：/goal 是 **session-level agentic execution 工具**，ticket acceptance 是 **ticket-level 品質閘門**。兩者解決不同問題：

- /goal 回答「何時停止執行」（execution boundary）
- acceptance 回答「產出是否合格」（quality gate）

將 execution boundary 與 quality gate 混合會導致：若 /goal evaluator 認為「完成了」但 acceptance-gate-hook 認為「未完成」，agent 會停止但 ticket 無法 complete，產生死鎖或混淆。

### P — Prepare to be Wrong（失敗回退）

| 方案 | 失敗回退 |
|------|---------|
| B（若未來嘗試） | 移除 /goal 設定，回歸純 acceptance 流程。成本：一個 DOC ticket 更新派發 prompt 範本。 |
| D（建議方案） | 本身即為最安全選項，無需回退。若未來 /goal 穩定化且支援 structured goal definition，可重啟評估（建 ANA ticket 追蹤）。 |

## 結論

**建議方案：D — 不整合，平行存在**

理由：
1. `/goal` 與 ticket acceptance 運作層級根本不同（session vs ticket），強制整合的收益不抵風險
2. `/goal` 尚在早期穩定化階段（v2.1.143 修 bug），不適合作為品質閘門依賴
3. 現有 acceptance 機制已有完整鏈路（frontmatter → CLI → hook → lifecycle），/goal 無法取代任何環節
4. 用戶若需 /goal 的 agentic loop 功能，可自行使用，無需專案框架介入

**附帶建議**：在 PM 派發 prompt 範本中加一句提示——「若代理人使用 /goal，goal 定義應與 ticket acceptance 對齊，但 /goal 不取代 acceptance 驗收」。此為 DOC 層級變更，非結構性整合。
