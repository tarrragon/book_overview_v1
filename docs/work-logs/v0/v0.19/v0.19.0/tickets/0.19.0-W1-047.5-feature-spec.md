# 0.19.0-W1-047.5 功能規格

> TDD Phase 1 功能規格（/spec Full 模式）— UC-04 匯入模式選擇 UI（IMP-E）

## 1. Purpose（目的）

### 問題背景

UC-04 匯入功能在 IMP-C/D 完成後，`tag-storage-adapter` 已具備兩種寫入策略：

- `replaceAllData`（覆蓋）：清空 storage 三 key 後完全載入匯入資料。
- `mergeAllData`（合併）：同 id 書籍 tagIds 聯集、同名 tag/category 映射、其餘新建。

但 `handleFileLoad` 目前硬編碼只走 `replaceAllData`（覆蓋），使用者無法選擇合併。SPEC-EXPORT-V2 §8 定義兩種載入模式皆為正式需求，匯入端缺少「讓使用者選模式」的決策入口。

### 目標用戶

overview 書庫頁面的使用者——已有本地書庫，準備匯入一份外部 JSON 檔案，需要決定「取代現有書庫」還是「與現有書庫合併」。

### 核心價值

匯入是破壞性操作。覆蓋模式會清空現有 tag/書籍，合併模式保留現有資料。讓使用者在資料被寫入前明確選擇模式，避免誤覆蓋整個書庫造成不可逆資料遺失。

### 範圍邊界

| 範圍內 | 範圍外 |
|--------|--------|
| 匯入時彈出模式選擇 modal | `replaceAllData` 實作（IMP-C 已完成） |
| modal 的覆蓋/合併兩個選項 + 取消 | `mergeAllData` 實作（IMP-D 已完成） |
| `handleFileLoad` 依使用者選擇分流至對應 adapter API | 版本偵測 / v1 轉換（IMP-A 已完成） |
| modal 取消時中止匯入、不寫 storage | tagCategories/tags 提取（IMP-B 已完成） |
| modal 無障礙屬性與鍵盤操作 | 匯入結果摘要報表（非本 ticket） |

## 2. API Signatures（介面定義）

### 2.1 模式選擇 modal

於 `overview-page-controller.js` 新增方法（命名為建議，實作時可微調但語義須一致）：

```js
/**
 * 顯示匯入模式選擇 modal，回傳使用者選擇的模式。
 * @returns {Promise<'overwrite' | 'merge' | null>}
 *   - 'overwrite'：使用者選擇覆蓋模式
 *   - 'merge'：使用者選擇合併模式
 *   - null：使用者取消（點取消鈕 / 關閉鈕 / Esc / 點遮罩）
 */
async promptImportMode ()
```

**回傳值語義**：

| 回傳值 | 觸發來源 | 後續行為 |
|--------|---------|---------|
| `'overwrite'` | 點「覆蓋」按鈕 | `handleFileLoad` 走 `replaceAllData` |
| `'merge'` | 點「合併」按鈕 | `handleFileLoad` 走 `mergeAllData` |
| `null` | 取消鈕 / 關閉鈕 / Esc / 點遮罩 | `handleFileLoad` 中止，不讀檔不寫 storage |

**Promise 解析時機**：使用者做出任一選擇（含取消）後才 resolve；resolve 前 modal 持續顯示，不自動關閉。modal 一次只允許一個實例。

### 2.2 handleFileLoad 分流改造

`handleFileLoad(file)` 改造後流程：

```
handleFileLoad(file)
  1. _validateFileBasics / _validateFileSize（不變）
  2. mode = await promptImportMode()        ← 新增
  3. if (mode === null) → return（中止，不顯示 loading、不讀檔）
  4. showLoading('正在讀取檔案...')
  5. importResult = await _readFileWithReader(file)
  6. writeResult = mode === 'overwrite'
       ? await TagStorageAdapter.replaceAllData({...})
       : await TagStorageAdapter.mergeAllData({...})   ← 分流
  7. writeResult 成功 → _updateUIWithBooks；失敗 → showError（不變）
```

**`mergeAllData` 入參**：與 `replaceAllData` 同形 `{ books, tags, tagCategories }`，三欄位皆為陣列（importer INV-1 保證）。

**`mergeAllData` 回傳值**：與 `replaceAllData` 對稱 `{ success, error }`；`error` 可能為 `'quota_exceeded'` 或其他。錯誤分支沿用既有 `showError` 文案邏輯。

### 2.3 modal DOM 結構（overview.html）

於 `overview.html` 新增 modal 容器（預設隱藏）：

| 元素 | id / class（建議） | 用途 |
|------|-------------------|------|
| 遮罩層 | `importModeOverlay` | 半透明背景，點擊視為取消 |
| modal 容器 | `importModeModal` | `role="dialog"` `aria-modal="true"` `aria-labelledby` |
| 標題 | `importModeTitle` | 「選擇匯入模式」 |
| 說明文字 | — | 簡述兩模式差異 |
| 覆蓋按鈕 | `importModeOverwriteBtn` | 文案含「覆蓋（清空現有書庫）」 |
| 合併按鈕 | `importModeMergeBtn` | 文案含「合併（保留現有書庫）」 |
| 取消按鈕 | `importModeCancelBtn` | 中止匯入 |

## 3. GWT Scenarios（行為場景）

### 場景 1: 使用者選擇覆蓋模式（正常流程）

- **Given**: overview 頁面已載入，使用者點「匯入 JSON」並選好一個有效檔案
- **When**: 模式選擇 modal 彈出，使用者點「覆蓋」按鈕
- **Then**: modal 關閉；`handleFileLoad` 呼叫 `TagStorageAdapter.replaceAllData`（非 `mergeAllData`）；成功後以匯入 books 更新 UI

### 場景 2: 使用者選擇合併模式（正常流程）

- **Given**: overview 頁面已載入，使用者點「匯入 JSON」並選好一個有效檔案
- **When**: 模式選擇 modal 彈出，使用者點「合併」按鈕
- **Then**: modal 關閉；`handleFileLoad` 呼叫 `TagStorageAdapter.mergeAllData`（非 `replaceAllData`）；成功後以合併後 books 更新 UI

### 場景 3: 使用者取消匯入（取消鈕）

- **Given**: 模式選擇 modal 已彈出
- **When**: 使用者點「取消」按鈕
- **Then**: modal 關閉；`handleFileLoad` 中止；不讀取檔案、不呼叫任何 `*AllData`、不顯示 loading、storage 與 UI 維持原狀

### 場景 4: 使用者以 Esc 鍵取消

- **Given**: 模式選擇 modal 已彈出且取得焦點
- **When**: 使用者按 Esc 鍵
- **Then**: 等同場景 3——modal 關閉、匯入中止、無副作用

### 場景 5: 使用者點遮罩取消

- **Given**: 模式選擇 modal 已彈出
- **When**: 使用者點 modal 外的半透明遮罩區域
- **Then**: 等同場景 3——modal 關閉、匯入中止、無副作用

### 場景 6: 合併模式 storage 寫入失敗

- **Given**: 使用者於 modal 選「合併」，檔案讀取成功
- **When**: `mergeAllData` 回傳 `{ success: false, error: 'quota_exceeded' }`
- **Then**: 呼叫 `showError('儲存空間不足，匯入未完成')`；不更新 UI；UI 維持匯入前狀態（與覆蓋模式失敗處理對稱）

### 場景 7: 鍵盤導覽 modal

- **Given**: 模式選擇 modal 彈出
- **When**: 使用者用 Tab 鍵切換焦點
- **Then**: 焦點在 modal 內三個按鈕間循環（focus trap），不跑到 modal 外的頁面元素；modal 開啟時初始焦點落在預設選項按鈕

### 場景 8: 重複觸發匯入

- **Given**: 模式選擇 modal 已開啟且尚未做出選擇
- **When**: 使用者再次觸發「匯入 JSON」（例如重複點擊按鈕）
- **Then**: 不彈出第二個 modal；維持單一 modal 實例，等待當前選擇完成

## 4. Error Handling（錯誤處理）

| 錯誤情境 | 偵測點 | 處理策略 | 回傳/使用者反饋 |
|---------|--------|---------|----------------|
| 使用者取消（任何途徑） | `promptImportMode` resolve `null` | `handleFileLoad` 提前 return | 無錯誤訊息（取消是正常操作），無副作用 |
| 檔案基本驗證失敗 | `_validateFileBasics` / `_validateFileSize` | 沿用既有：importer 呼叫 showError 並 throw | 既有錯誤文案 |
| 檔案讀取/解析失敗 | `_readFileWithReader` | 沿用既有 importer 錯誤處理 | 既有錯誤文案 |
| 合併寫入 quota 超限 | `mergeAllData` 回傳 `error === 'quota_exceeded'` | `showError('儲存空間不足，匯入未完成')` | 既有覆蓋模式同文案 |
| 合併寫入其他失敗 | `mergeAllData` 回傳 `success === false` 且非 quota | `showError('儲存失敗，已還原原有資料')` | 既有覆蓋模式同文案 |
| modal 元素缺失（DOM 未注入） | `promptImportMode` 取不到 modal 元素 | 視為設計缺陷，記錄 console.error，`handleFileLoad` 中止匯入並 showError | `showError('匯入功能初始化失敗')` |

**錯誤處理原則**：

- 取消不是錯誤，不顯示錯誤訊息。
- 驗證順序：檔案基本驗證在 modal 之前——無效檔案不應讓使用者先選模式再被拒。
- modal 在「驗證通過後、讀檔前」彈出：避免使用者選完模式才發現檔案無效。

## /spec validate 結果

### 結構檢查：通過

Full 模式 6 區段（Purpose / API Signatures / GWT Scenarios / Error Handling / Dependencies / Acceptance）皆存在且非空；Scenarios 含 8 組完整 GWT 三元組；Acceptance 全部以 `- [ ]` 開頭。

### 語義推演：第 1 輪 — 4 個未回答問題

#### 維度 1: 邊界完整性

- Q1: 匯入檔案內容為空（books/tags/tagCategories 皆空陣列）時，覆蓋模式會清空整個書庫——是否需要在 modal 或 modal 後追加二次確認？
- Q2: 本地書庫本來就為空時，覆蓋與合併結果等價——是否仍需顯示 modal，或可略過直接走覆蓋？

#### 維度 2: 錯誤路徑

- Q3: `promptImportMode` 期間頁面被關閉或重整，pending Promise 永不 resolve——是否需要清理機制？

#### 維度 3: 狀態完整性

- Q4: modal 開啟期間，使用者觸發其他破壞性操作（如「重新載入」按鈕）——是否需要在 modal 開啟時禁用其他操作入口？

### 第 1 輪回答（lavender 決議）

| Q | 回答 | 歸屬 |
|---|------|------|
| Q1 | 不在本 ticket 範圍。空檔案二次確認屬「匯入安全強化」，與模式選擇 UI 職責分離。IMP-E 聚焦模式選擇分流；若需空檔案防護應另開 ticket。已於範圍邊界明示「匯入結果摘要報表」類強化非本 ticket。 | 範圍外，建議後續 ticket |
| Q2 | 維持一律顯示 modal。理由：(1) 行為一致性，使用者預期匯入必選模式；(2)「本地為空」判斷需先讀 storage，增加分支複雜度；(3) 覆蓋與合併在空書庫下結果等價，略過 modal 無功能損失但破壞可預期性。決議納入規格——`handleFileLoad` 不做「書庫為空則略過 modal」最佳化。 | 已決議，補入規格 |
| Q3 | 接受風險，不額外處理。理由：頁面關閉/重整時整個 controller 實例銷毀，pending Promise 隨之回收，無記憶體洩漏或資料損壞風險（modal 在 resolve 前不寫 storage）。屬瀏覽器生命週期自然回收。 | 已決議，無需動作 |
| Q4 | 採 focus trap 已部分緩解（場景 7）。modal 開啟期間其他按鈕雖未顯式禁用，但 focus trap 使鍵盤使用者無法觸及；滑鼠點擊其他破壞性操作（如「重新載入」）為獨立流程，不與匯入競爭同一 storage 寫入時序（匯入在 modal resolve 後才寫）。決議：不顯式禁用其他入口，但補入場景說明此設計取捨。 | 已決議，補入規格 |

### 補入規格的決議

- **Q2 決議**（補入 §2.2）：`handleFileLoad` 一律呼叫 `promptImportMode`，不因本地書庫為空而略過 modal，確保行為可預期。
- **Q4 決議**（補入 §3 場景 7 註記）：modal 開啟期間不顯式禁用頁面其他按鈕；focus trap 已阻擋鍵盤誤觸，滑鼠觸發的其他操作與匯入寫入時序不重疊（匯入於 modal resolve 後才寫 storage）。
