## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W4-005 重構（timeouts.js + reExtract helper + 4 檔字面值替換）**不是** beforeAll Extension ID timeout 10000ms regression 的根因。

**二分定位實證**（W4-005.1 執行）：

| 狀態 | 樣本 | RED 次數 | 失敗模式 |
|------|------|---------|---------|
| Refactored (W4-005 完整應用) | 9 runs | 3 RED（2 個 1/9 beforeAll timeout、1 個 8/9 G4-1 chrome.storage.local） | 1/9 + 8/9 |
| Pristine baseline (W4-005 revert) | 5 runs | 2 RED（皆 1/9 beforeAll timeout） | 1/9 |

兩種狀態**同失敗模式、同失敗訊息、同 timeout 數值（10000ms）**，但前次 thyme 的「pristine baseline 9/9 GREEN 兩次連續」是 lucky 2-run streak，廣樣本下兩者皆呈現 ~33-40% flakiness。

**真實根因**（兩個 race condition 疊加）：

1. **SW launch timing race**：`tests/e2e/setup/extension-setup.js:138` 與 `tests/e2e/browser/helpers/storage-reader.js:37` 硬編碼 `SW_WAIT_TIMEOUT = 10000`，在 cold-start 或系統負載情境下不足，導致「取得 Extension ID 失敗 Timed out after waiting 10000ms」。
2. **chrome.storage.local 初始化 race**：即使 SW target 取得成功，worker context 內 `chrome.storage.local` 仍可能短暫 undefined（worker 啟動完成 ≠ chrome API 注入完成），導致 `clearBooksStorage` 拋 `TypeError: Cannot read properties of undefined (reading 'local')`。

### 影響範圍

- `tests/e2e/setup/extension-setup.js`：`getExtensionId` / `getBackgroundPage` 硬編碼 10s timeout、`setup()` 無 retry 機制
- `tests/e2e/browser/helpers/storage-reader.js`：`getServiceWorker` 硬編碼 10s default、`clearBooksStorage` / `readBooksFromStorage` 未防護 chrome.storage API race
- `tests/e2e/browser/helpers/timeouts.js`：需新增 SW_LAUNCH_TIMEOUT 常數（與 W4-005 SSOT 一致）

### 相關 Error Pattern

無直接對應 PC。此屬 E2E 測試基礎設施對 cold-start 缺乏防護的常見模式。

### 二分定位證據

詳見 `tests/e2e/` 測試 log（W4-005.1 階段 5 + 10 round runs）：
- /tmp/w4-005.1-repro-{1..9}.log（refactored state 9 runs：3 RED）
- /tmp/w4-005.1-pristine-{1..5}.log（pristine state 5 runs：2 RED，**證實 W4-005 重構非根因**）
- /tmp/w4-005.1-after-fix-{1..5}.log（僅 timeout 延長 30s：1/5 RED）

- /tmp/w4-005.1-final2-{1..10}.log（加入 chrome.storage.local race 防護：**10/10 GREEN**）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

採三層防護解決 SW launch 與 chrome.storage.local 兩個 race condition，**完全不撤回 W4-005 重構**（timeouts.js + reExtract helper + 5 個原有常數全部保留）：

**第 1 層：SW launch timeout 對齊**

`tests/e2e/browser/helpers/timeouts.js` 新增 `SW_LAUNCH_TIMEOUT = 30000`（對齊 DEFAULT_NAV_TIMEOUT）：

- `tests/e2e/setup/extension-setup.js`：`getExtensionId` / `getBackgroundPage` 改用 `SW_LAUNCH_TIMEOUT`（原硬編碼 10000ms）
- `tests/e2e/browser/helpers/storage-reader.js`：`getServiceWorker` default timeout 改用 `SW_LAUNCH_TIMEOUT`（原 10000ms）

**第 2 層：Chrome instance 級別 retry**

`tests/e2e/setup/extension-setup.js` `setup()` 改為 retry loop（SW_LAUNCH_RETRY_MAX = 3）：

- 抽出 `_setupSingleAttempt()` 包含原 launch/getExtensionId/newPage/configureTestPage 流程
- 失敗時 cleanup 當前 browser 再重試，繞過單次 Chrome cold-start 失敗

**第 3 層：chrome.storage.local 初始化等待**

`tests/e2e/browser/helpers/storage-reader.js` 新增 `waitForChromeStorageReady(worker, timeout, interval)`：

- Node 側 polling worker.evaluate 檢查 `chrome.storage.local` 可用性（避免 evaluate 內 setTimeout undefined 問題）
- `clearBooksStorage` / `readBooksFromStorage` 在 evaluate 前先呼叫此 helper

### 設計決策

| 候選 | 採用？ | 理由 |
|------|-------|------|
| A. 僅延長 timeout（10s → 30s） | 否 | 實測 5 runs 仍 1 RED（30s 不足 cold-start 邊緣情況） |
| B. 加入 Chrome retry + timeout 延長 | 部分 | 解決 1/9 beforeAll timeout 但 G4-1 仍有 storage race |
| C. **A + B + chrome.storage race 防護** | 是 | 三層防護後 10/10 GREEN，含 2 次自動 retry 成功 |
| D. 撤回 W4-005 重構 | 否 | 違反 ticket 強制條件「禁撤回 timeouts.js / reExtract / 5 常數」 |

### 不在本 ticket 範圍

- service-worker-target.js 仍保留 default timeout 10000ms（caller 傳值優先，未影響行為）
- 完全消除 SW launch flakiness：屬 Chrome runtime cold-start 行為，超出測試基礎設施範圍

### 對 W4-005 重構的影響

- **零撤回**：W4-005 的 5 個 timeout 常數、reExtract helper、4 檔字面值替換全部保留
- **新增 1 個常數**：`SW_LAUNCH_TIMEOUT = 30000` 加入 timeouts.js 共同入口（與 W4-005 SSOT 一致）
- **重構完全保留可達成 9/9**：W4-005 達成 AC#3「9/9 GREEN」原本被 baseline flakiness 阻擋，本 ticket 修復 baseline 後可順利 complete
