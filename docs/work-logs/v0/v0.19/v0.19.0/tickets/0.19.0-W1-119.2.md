## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

MessageDictionary 是 Logger 的依賴（Logger.js:21 require MessageDictionary 取 GlobalMessages），但 MessageDictionary.js:1 也反向 require Logger，形成循環依賴。

當前 commonjs 容忍此狀況（partial exports）+ Logger.js try/catch 保護 require fail，但違反依賴方向原則（messages 是更底層 primitive，不應依賴 logging）。esbuild bundle 階段順序敏感，潛在 break 風險。

### 影響範圍

**MessageDictionary.js 當前狀態（W1-119.1 merge 後）**：

| 位置 | 內容 | 處理 |
|------|------|------|
| `src/core/messages/MessageDictionary.js:1` | `const { Logger } = require('src/core/logging/Logger')` | **移除** |
| `src/core/messages/MessageDictionary.js:232` | `Logger.warn(`MessageDictionary: Adding "${key}" would exceed cache limit...`)` | **替換 console.warn 或 throw** |
| `src/core/messages/MessageDictionary.js:227` | `_checkCacheSize` method | 保留（caller 是內部） |

**Logger.js 不變**：

| 位置 | 內容 | 處理 |
|------|------|------|
| `src/core/logging/Logger.js:21` | `GlobalMessages = require('src/core/messages/MessageDictionary').GlobalMessages` | **保留**（單向依賴：Logger 依賴 messages 合理） |

### 相關 Error Pattern

依賴方向原則：messages → log？實際上 messages 不該依賴 logging（messages 是低階 primitive）。Logger.warn 在 _checkCacheSize 的使用是「實作便利」而非「設計需求」，可改用 console.warn。

### 設計決策

**選項 A：console.warn**（推薦）
```js
console.warn(`MessageDictionary: Adding "${key}" would exceed cache limit (${this._maxCacheSize / 1024}KB)`)
```
- 優點：簡單、無依賴、瀏覽器/node 通用
- 缺點：繞過 Logger 統一管道（但 cache warning 是極少觸發的內部 warning，可接受）

**選項 B：throw Error**
- 優點：強制 caller 處理
- 缺點：cache size warning 是 informational，不應 throw

**選項 C：移除 _checkCacheSize warning**
- 評估：cache 警告未使用價值低，但移除是 ANA 範圍外的決策

**裁定**：採選項 A（console.warn）。

### Test 策略

1. 移除 require Logger 後 MessageDictionary.test.js 應仍綠（測試不應依賴此 require）
2. _checkCacheSize cache 限制行為若有測試，確認 console.warn spy 替代 Logger.warn spy
3. npm test 全綠（baseline 5284 passed after W1-119.1）
4. 建置 dev bundle 驗證 Chrome Extension runtime 無回歸（chrome-devtools-mcp 觀察 console）
