### 問題根因

Claude Code v2.1.139-143 連續引入 6 項 hook 行為變更，專案 124 個已註冊 hook command（79 個 .py 檔案）的設計假設可能與新行為不一致。需系統性盤點影響。

### 影響範圍

- 79 個 hook .py 檔案（.claude/hooks/）

- hook_utils 共用模組（hook_logging.py 含 stderr 寫入邏輯）
- 規則 4（Hook 失敗必須可見）— quality-baseline.md

- PC-086: subagent hook exec bit
- PC-135: subagent pytest vs hook subprocess env

- IMP-055: hook stdout must be JSON

### 實驗執行

**變更 1 (args exec form 不需引號)**:
- settings.json 中唯一使用 args 的 hook: style-guardian-hook.py，用法為 `$CLAUDE_PROJECT_DIR/.claude/hooks/style-guardian-hook.py $CLAUDE_FILE_PATH`（PostToolUse Edit/Write 各一處）
- 此為 exec form（空格分隔），v2.1.139 前需要引號，之後不需
- 影響：2 個註冊項。目前用法為 exec form 無引號，**與新行為一致，無需修改**

**變更 2 (PostToolUse continueOnBlock)**:
- grep 結果：0 個 hook 使用 continueOnBlock
- 這是新能力（PostToolUse hook 可回饋訊息讓 Claude 繼續），非破壞性變更
- 影響：0 個 hook 受衝擊。**可整合機會**：6 個 PostToolUse hook 可評估是否利用此能力

**變更 3 (hooks 無 terminal access)**:
- 44 個 hook 檔案含 sys.stderr.write 或 print(file=sys.stderr)
- hook_utils/hook_logging.py 的 _log_exception() 和 setup_hook_logging() 都寫 stderr
- 「無 terminal access」意味 stderr 輸出**不會顯示在用戶終端**
- 但 Claude Code 仍會讀取 stderr 內容作為 hook 執行結果的一部分
- 影響：**核心問題**。規則 4 要求「Hook 失敗必須在 stderr 輸出以確保用戶可見」，若 stderr 不再到達終端，此要求的假設基礎動搖

**變更 4 (terminalSequence 取代 terminal write)**:
- grep 結果：hooks 中無任何直接 terminal write 使用
- terminalSequence 是新的 hookResponse 欄位，用於在終端顯示資訊
- 影響：0 個 hook 受衝擊。**可整合機會**：若要恢復 stderr 等效的用戶可見性，可改用 terminalSequence

**變更 5 (Stop 8 consecutive blocks cap)**:
- Stop event 註冊 5 個 hook command：
  1. evaluate-session.py（continuous-learning）
  2. handoff-auto-resume-stop-hook.py（ticket）
  3. session-experience-persistence-reminder-hook.py
  4. worktree-auto-commit-hook.py（timeout 15s）
  5. stop-worklog-handoff-sync-check-hook.py（timeout 10s）
- v2.1.143 限制 Stop hook 最多連續 8 blocks 輸出
- 影響：需評估各 hook 輸出量是否超過 8 blocks。大部分 hook 輸出簡短（JSON + 少量提示），**低風險但需確認**

**變更 6 (MCP stdio servers 收到 CLAUDE_PROJECT_DIR)**:
- 15 個 hook 檔案在 Python 中讀取 CLAUDE_PROJECT_DIR env var
- settings.json 中所有 command 路徑以 $CLAUDE_PROJECT_DIR 開頭（124 個）
- 此變更影響 MCP stdio servers 而非 hooks 本身
- 影響：**hooks 不受影響**。但若專案有自訂 MCP server 使用 project dir，需確認

### WRAP 框架分析

**W (Widen) — 6 項變更各自影響範圍**:

| 變更 | 版本 | 性質 | 受影響 hook 數 | 嚴重度 |
|------|------|------|--------------|--------|
| 1. args exec form 不需引號 | v2.1.139 | API 簡化 | 2（style-guardian PostToolUse Edit/Write） | info — 現有寫法已兼容 |
| 2. PostToolUse continueOnBlock | v2.1.139 | 新能力 | 0（新 API，無現有使用） | info — 純機會 |
| 3. hooks 無 terminal access | v2.1.139 | 行為限制 | 44（含 stderr 寫入的 hook）+ hook_utils 核心 | critical — 規則 4 假設動搖 |
| 4. terminalSequence 取代 terminal write | v2.1.141 | 新 API | 0（無現有 terminal write） | info — 替代方案 |
| 5. Stop 8-block cap | v2.1.143 | 行為限制 | 5（Stop event 註冊 hook） | warning — 需確認輸出量 |
| 6. MCP stdio CLAUDE_PROJECT_DIR | v2.1.139 | 配置擴充 | 0（hooks 不受影響） | info — MCP server 層面 |

**R (Reality-test) — 對照現有 hook 行為與規則 4**:

規則 4 原文要求：「Hook 異常時必須寫入 stderr，確保用戶端可見」。hook_utils/hook_logging.py 的 _log_exception() 在 L237 明確寫 `sys.stderr.write("[Hook Error] ... failed unexpectedly...")`，setup_hook_logging() 在 L110 建立 StreamHandler(sys.stderr)。

變更 3「hooks 無 terminal access」的語意需精確理解：
- **「無 terminal access」≠「stderr 被丟棄」**。Claude Code 仍讀取 hook 的 stderr，用於 (a) 判定 hook 是否失敗 (b) 在 UI 顯示 hook error 訊息（IMP-048 已記錄此行為）
- 「無 terminal access」= hook process 無法直接寫入宿主終端的 raw fd（如 /dev/tty），僅影響試圖繞過 stdout/stderr 直接操控終端的場景
- **本專案所有 hook 均透過 sys.stderr 寫入**，不直接操控終端 fd，因此 stderr 可見性機制**未受破壞**

結論：規則 4 的 stderr 雙通道要求在「無 terminal access」變更下**仍然有效**。

**A (Attain distance) — 規則 4 是否需修訂**:

不需修訂規則 4 本體。但建議：
1. 在規則 4 或 hook-architect-technical-reference.md 補充說明：「hooks 無法直接存取宿主終端（v2.1.139+），stderr 寫入仍被 Claude Code 讀取並呈現為 hook error」
2. 新增 terminalSequence 作為「非錯誤但需用戶注意」的可見性通道（與 stderr 的 error-level 區分）

**P (Prove) — 失敗回退**:

若 H1 假設錯誤（stderr 確實不再被讀取）：
- 回退方案：所有 hook 改用 stdout JSON 的 hookResponse.terminalSequence 或 hookResponse.decision.reason 欄位傳遞可見訊息
- 影響範圍：44 個 hook + hook_utils 核心模組（成本高，需確認後再啟動）
