## Solution

### 第 1 輪審查（compositional-writing frame：對意圖 + 一致性 + grep-ability + focus 完整度）

審查標的：SKILL.md + web/app/backend-testing-context.md。發現 7 項。

| # | 嚴重度 | 發現 | 修正 |
|---|-------|------|------|
| F1 | Critical | 類型編號全面不一致：SKILL.md 用類型 1-9，三個 references 用舊的 5-12 編號（非同步時序在 references 標「類型 8」、SKILL.md 是「類型 5」）。grep-ability 完全失效，讀者無法從 SKILL.md 跳到 reference。 | references 依類型名稱對應 SKILL.md 編號：非同步時序=5、亂數輸出=6、測試隔離=7、快照過度覆蓋=8、斷言過度集中=9；references 的「類型 5、6（計時相關）」section 改為「類型 1-3（計時相關）」。 |
| F2 | Critical | SKILL.md 族範圍錯誤：「環境依賴 flaky 族（類型 1–3）」「斷言設計族（類型 4–9）」。類型 4（記憶體絕對值）識別信號為 GC 時機影響、來源欄標 W1-017/018 實證，屬環境依賴族。 | 改為 環境依賴 flaky 族 1–4、斷言設計族 5–9。 |
| F3 | High | SKILL.md 三問表「確定性 否 → 類型 1–3」遺漏。記憶體絕對值（4）、非同步時序（5）、亂數輸出（6）皆為確定性失效。 | 確定性「否」對應改為「類型 1–6」。 |
| F4 | High | 三問 routing 不完整：SKILL.md line 12 宣稱「任一問題答否即進入對應的類型判斷」，但類型 5、6、8 無入口。 | 三問 routing 改為：確定性→1-6、聚焦性→8-9、隔離性→7，涵蓋全 9 類。 |
| F5 | Medium-High | SKILL.md「判斷軸」段概念混淆：宣稱「兩個正交軸」但只描述「兩族」（單一分類的兩個值）；「正交」「同時落入兩族」語意不成立（1-4/5-9 是 partition）。 | 明示為兩族 partition；與「斷言品質三問」區分為不同視角（三問是快速分流、分族是依問題本質歸類）。 |
| F6 | High | references Node.js 雙重歸類：web-testing-context.md 將「Node.js 服務（含 REST API、GraphQL）」歸 WEB 類，backend-testing-context.md 將「Node.js 服務」歸後端類。同一專案被歸兩類，違反 W1-024 專案類型分類軸。 | Node.js 後端服務屬後端類，從 web-testing-context.md 適用清單移除。 |
| F7 | Medium | references 覆蓋類型不一致：web/backend 有計時 section 無「斷言過度集中」；app 有「斷言過度集中」無計時。focus 完整度不足——references 聲稱「各斷言類型的 X 脈絡」但只覆蓋部分。 | 三個 references 統一覆蓋相關類型：web/backend 補「斷言過度集中」section，app 補「計時相關」section；內容種子取自 W1-024 AC1 與重現實驗結果。 |

修正方式：派 thyme-documentation-integrator 套用 F1-F7，PM 執行第 2 輪複審。
