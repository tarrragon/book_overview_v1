## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 決策摘要

**決策 1（配置層級）**：採用「專案層級 `.mcp.json` 配置 + binary user-level 安裝」混合架構。

| 維度 | 選擇 | 理由 |
|------|------|------|
| Binary 安裝 | user-level | 跨專案共用，避免每專案重複下載；既有 zhtw-mcp / cbm 已在 ~/.local/bin/、codegraph 為 npm global，符合工具部署慣例 |
| 配置寫入 | 專案 `.mcp.json` | clone 後 session 啟動即就緒、跨團隊一致性、版本控制可追蹤；與既有 chrome-devtools 同 pattern |

**決策 2（首次 index 時機）**：暫不在本 ticket 執行 index_repository / codegraph index。
- 兩 MCP 在 client 端有 lazy init 機制（codegraph 錯誤訊息直接給「需 init」指引）
- 開發者首次需要時觸發即可，避免在配置 ticket 中綁定大型耗時操作
- 在 docs/development-setup.md 步驟 3/4 提供 init 流程供開發者按需執行
- acceptance #2/#3 改以「驗證 MCP 載入成功 + Tool schema 可解析 + 提供 init 指令」為通過依據

### 技術設計細節

**`.mcp.json` 新增區塊**：

```json
"codebase-memory-mcp": {
  "type": "stdio",
  "command": "codebase-memory-mcp",
  "args": [],
  "env": {}
},
"codegraph": {
  "type": "stdio",
  "command": "codegraph",
  "args": ["serve", "--mcp", "--path", "."],
  "env": {}
}
```

設計考量：
- cbm 用 `command: "codebase-memory-mcp"` 走 PATH 解析，跨開發者環境（brew/pipx/手動下載）皆支援；放絕對路徑會綁死 user 安裝位置
- codegraph 加 `--path .` 解決首次撞牆「No CodeGraph project is loaded」（錯誤訊息明示兩個 fix 之一，此為專案層級配置正確做法）
- 兩者皆採 stdio type，與既有 chrome-devtools npx pattern 並存無衝突

**`.gitignore` 新增**：`.codegraph/`（per-developer local index，不污染 repo）
cbm 索引存 `~/.cache/codebase-memory-mcp/`，無需 gitignore。

**文件責任分離**：
- `docs/development-setup.md`（新建）：MCP 配置 + 安裝驗證 + 工具選擇策略（Claude Code session 層）
- `docs/environment-recovery-guide.md`（不動）：npm install / build artifact 層
- `CLAUDE.md §1`：摘要列出四 MCP + 指向 development-setup.md 詳細指引
