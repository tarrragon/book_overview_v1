## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 修復摘要

採用 W1-090/094/095 同模式：移除 Rule 1 違規的 `toBeLessThan` 計時門檻，補 `// W1-096 移除 Rule 1 違規: <欄位> 由 Date.now() 差值計算，效能驗證移至 npm run test:perf` 註解。功能性斷言（success/responseType/recipientsReached 等）全數保留。

### 三檔處理明細

| 檔案 | 真實 Rule 1 違規 | 業務不變式保留 | 其他保留 |
|------|---------------|-------------|---------|
| content-script-injection.test.js | 18 行全移除 | - | - |
| storage-api-integration.test.js | 4 行 (writeTime/readTime/totalBatchTime/updateTime) | - | line 300 local.used 比較（儲存空間，非計時） |
| runtime-messaging-integration.test.js | 18 行 (responseTime/avgResponseTime/executionTime/avgDeliveryTime/reportingTime/avgProcessingTime/popupUpdateLatency/routingOverhead/broadcastTime/p2pTime/avgQueueTime/totalProcessingTime/avgProcessingTime/handlingTime/avgHandlingTime/retryTime x3/avgDeliveryTime) | 5 行: averageHops(計數)/bottleneck.impact(比率)/urgentMessageIndex(索引)/normalIndicesAfter(計數)/avgRetryAttempts(計數) | - |

### 設計決策亮點

1. **phases.*.duration 改 toBeDefined**：content-script-injection.test.js 第 358-361 行 4 個階段時間斷言（injection/initialization/execution/cleanup）改為 `expect(phases.X).toBeDefined()` 存在性驗證。lifecycle phases 結構完整性 vs 計時量測責任拆分，結構驗證仍保留在主套件，計時量測移至 perf。

2. **業務不變式 vs 計時違規區分**：runtime-messaging 5 個 toBeLessThan 留下：
   - `averageHops < 2.5`：跳躍次數計數，路由拓樸效率指標（非計時）
   - `bottleneck.impact < 0.3`：影響比率 0-1 範圍（非計時）
   - `urgentMessageIndex < 5`：訊息處理順序索引（非計時）
   - `normalIndicesAfter < 8`：佇列中正常訊息計數（非計時）
   - `avgRetryAttempts <= 3`：重試次數計數（非計時，且為 toBeLessThanOrEqual）
   均補上「為 X 計數/比率/索引，非計時」豁免註解。

3. **storage-api line 300 保留**：`postCleanupUsage.local.used < preCleanupUsage.local.used` 為儲存空間 byte 數比較（驗證清理後空間下降），非計時，保留作功能性業務驗證。
