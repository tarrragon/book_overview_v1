### 問題根因

1. ticket frontmatter 無 `trigger_bound` 標記欄位，dashboard scheduler 無從區分一般 pending 與 trigger-bound pending

### Phase 2: dashboard scheduler 三方案利弊評估（主議題）

源碼確認（`.claude/skills/ticket/ticket_system/commands/track_dashboard.py:80-111` `load_top_ready`）：
- 只依 `_is_unblocked_pending`（status=pending + blockedBy=[]）+ `_compute_readiness == "READY"` 篩選
- 排序純 `_priority_rank` + id；**無任何 trigger-bound 偵測機制**
- Ticket frontmatter 無 `trigger_bound` 標記欄位

| 方案 | 實作 | 優點 | 缺點 | 估時/風險 |
|------|------|------|------|----------|
| A. 完全過濾 trigger-bound | dashboard 排除 `trigger_bound: true` 的 ticket | Ready 100% 可 claim，0 摩擦 | trigger 真實成立時無提示，易遺忘 | 低風險（過濾邏輯簡單）|
| B. 醒目標記 trigger-bound | Ready 行加 `[T]` 前綴或排在最後 | 保留可見性 + 明示風險 | PM 仍需手動驗證 trigger 狀態 | 低風險 |
| C. runqueue 預檢 trigger | dashboard 對 trigger_bound ticket 執行 trigger_check_command 判定 | trigger 成立才列入 Ready，全自動 | 每張 ticket 需寫 check command，dashboard 變慢 + shell injection 風險 | 高風險（安全性 + 效能）|

**選擇方向**：方案 B（短期）+ 方案 C（中長期評估）

- 短期：方案 B 4 個 acceptance（frontmatter 加 `trigger_bound` boolean 欄位 + dashboard 醒目標記 + 排序權重降低 + `--exclude-trigger-bound` 旗標可選）
- 中長期：方案 C 需獨立 ANA 評估（trigger check command DSL + 安全沙箱 + 效能基準）

**不採方案 A 理由**：完全隱藏會造成 trigger 成立時系統性遺忘，違反 quality-baseline 規則 5「所有發現必須追蹤」。

### Phase 3: NeedsContext hook 來源區分子議題

源碼確認（`.claude/skills/ticket/hooks/needs-context-listener-hook.py:86-89`）：

```python
message = (
    f"[NeedsContext] 代理人已於 {ticket_id} 回報資料缺口；"
    f"請 PM 執行 `ticket track full {ticket_id}` 檢視 NeedsContext section 並補料"
)
```

問題：訊息預設 caller=代理人，但 hook 僅檢查 command 是否含 `--section NeedsContext`，**不區分 PM 自填 vs 代理人回報**。

| 方案 | 實作 | 優點 | 缺點 |
|------|------|------|------|
| X. transcript 上下文判斷 | 依 transcript 內容判 caller | 無 CLI 改動 | hook 內無法存取 transcript，不可靠 |
| Y. CLI 加 `--source pm/agent` | append-log 新增 source flag，hook 智能切換訊息 | caller 顯式宣告 | 需 CLI 改 + 所有 caller 配合 |
| Z. CLI 加 `--silent` 旗標 | 旗標靜音 hook | 實作簡單 | silent 反語意，PM 需記得加 |
| V. 訊息中性化 | hook 訊息改為「X 的 NeedsContext section 已更新」 | 0 CLI 改動，立即可做 | 訊息頻率不變，PM 仍可能困惑 |

**選擇方向**：方案 V（短期）+ 方案 Y（中期）

- 短期：方案 V 訊息改中性語意（如「NeedsContext 已更新於 X，請 PM 確認是否需補料」），對 PM 自填情境不再誤導
- 中期：方案 Y 提供 `--source` flag 讓 caller 可顯式宣告，hook 能精準切換訊息
