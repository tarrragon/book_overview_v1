## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-030 feature-spec 7 D6 明確標記「SPA 路由切換返回時重新提取觸發」不在 W1-030 範圍，待 PM 評估。本 ANA 需判斷：使用者在 read.readmoo.com SPA hash 路由間切換（如 `#/library` -> `#/reading` -> 返回 `#/library`）後，書庫資料是否需要重新提取。

### 影響範圍

- `src/content/content-modular.js` — URL 變更監聽與 `reportReadyStatus()` 邏輯
- `src/content/detectors/page-detector.js` — `onUrlChange` MutationObserver 機制
- `src/content/utils/event-utils.js` — `onURLChange` popstate 監聽

### 相關 Error Pattern

無直接相關 error pattern。

### 調查過程記錄

1. `content-modular.js:172` — `pageDetector.onUrlChange` 回呼僅呼叫 `reportReadyStatus()`，不觸發提取
2. `content-modular.js:296-297` — 提取由 `START_EXTRACTION` 訊息觸發（使用者在 popup 點按鈕）
3. `reportReadyStatus()` 只更新 popup 顯示的 bookCount 與 extractable 狀態
4. `page-detector.js:168` — `onUrlChange` 用 MutationObserver 偵測 URL 變化，觸發 `detectReadmooPage()` 重新辨識頁面類型
5. Readmoo SPA 為 hash-based 路由（`read.readmoo.com/#/library`），docs/bookstores/readmoo.md 已記載

## 重現實驗結果
### 實驗方法

靜態程式碼分析：追蹤 SPA 路由切換時的完整事件鏈，確認提取觸發機制與 URL 變更處理的交互。

### 實驗執行

**事件鏈追蹤（URL 變更 -> 提取觸發）**：

1. URL 變更偵測路徑：`page-detector.js onUrlChange` (MutationObserver) -> callback in `content-modular.js:172`
2. callback 行為：`changeInfo.changed == true` -> `setTimeout(() => reportReadyStatus(), 1000)`
3. `reportReadyStatus()` 行為：讀取 pageStatus + bookCount -> `sendToBackground({ type: 'CONTENT.STATUS.READY', data: {...} })`
4. 提取觸發路徑（獨立）：`chrome.runtime.onMessage` -> `case 'START_EXTRACTION'` -> `bookDataExtractor.startExtractionFlow()`

**關鍵發現**：URL 變更事件鏈終止於 `reportReadyStatus()`（狀態上報），與提取觸發路徑（`START_EXTRACTION` 訊息）完全獨立。兩條路徑無交叉。

**SPA 路由切換場景分析**：

| 場景 | 使用者動作 | 系統行為 | 提取是否觸發 |
|------|-----------|---------|------------|
| 首次進入 library | 導航至 `#/library` | content script 注入 + `reportReadyStatus()` | 否（等用戶點按鈕） |
| library -> reading | 點選書籍 | URL 變更偵測 -> `reportReadyStatus()` 更新狀態 | 否 |
| reading -> library | 返回書庫 | URL 變更偵測 -> `reportReadyStatus()` 更新狀態 | 否 |
| 返回後手動提取 | 點 popup 按鈕 | `START_EXTRACTION` -> `startExtractionFlow()` | 是 |

### 實驗發現

**已驗證事實**：
1. 提取為使用者主動觸發（popup 按鈕 -> `START_EXTRACTION`），非自動觸發
2. SPA 路由切換返回 library 時，`reportReadyStatus()` 會正確更新 popup 狀態（bookCount、extractable）
3. 使用者返回 library 後可再次手動點按鈕觸發提取，W1-030 的捲動載入邏輯會在 `extractAllBooks` 內部重新執行
4. 現行架構不存在「返回後資料過期」問題——每次提取都是完整流程（捲動 + 提取全部）

**結論**：現行設計已足夠。不存在需要自動重新提取的技術缺口。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 分析結論：標記不需要實作

**決策（狀態 a）：不建立 SPA 路由切換重新提取觸發的實作 ticket。**

### 理由

1. **提取為使用者主動觸發**：v1.0 設計為使用者在 popup 點按鈕觸發提取，非自動背景提取。SPA 路由切換返回 library 後，使用者若需更新資料，再點一次按鈕即可。每次提取都執行完整流程（W1-030 捲動載入 + 全量提取），不依賴前次結果快取。

2. **現行 URL 變更處理已足夠**：`content-modular.js` 的 `onUrlChange` 回呼正確呼叫 `reportReadyStatus()`，返回 library 後 popup 會顯示正確的 bookCount 和 extractable 狀態，使用者知道可以再次提取。

3. **自動重新提取反而有害**：
   - 使用者可能只是瀏覽經過 library，不需要每次都觸發耗時的捲動載入 + 提取流程
   - 自動提取會消耗 CPU/記憶體資源（W1-030 的捲動載入涉及多次 DOM 操作）
   - 違反使用者預期——Chrome Extension 背景自動執行重量操作屬 anti-pattern

4. **v1.0 內測範圍無此需求**：v0.19.0 目標為「可分發打包 + 核心流程驗證」，使用者手動提取已滿足所有 UC（UC-01 書庫提取為手動觸發）。

### 未來考量（設計筆記，非延後決策）

若 v2.0+ 加入「自動同步」功能（定時背景提取），屆時需重新評估 SPA 路由切換偵測作為觸發條件之一。此為架構擴展點記錄，不建立 ticket。

### Spawn 落地確認

無需建立 spawn ticket。本 ANA 結論為「不需要實作」，無衍生工作項。
