## Solution

### Stage A 驗證結論：mergeAllData 對 duplicate-handling-v2 案例覆蓋分析

#### 結論

無需補測試。duplicate-handling-v2.test.js 測試的 _handleDuplicateBooks 三策略（skip/override/merge）與 v2 欄位合併邏輯為從未在生產路徑啟用的孤兒實作。生產匯入路徑（W1-047.5 起）為 handleFileLoad 直接呼叫 TagStorageAdapter.mergeAllData，該函式採取簡化的合併語意（同 id 用匯入版本覆蓋 + tagIds 取聯集 + 新 id 加入 + 永不清空本地），與 DuplicateBookMerger 的 v2 欄位細部規則為不同設計。

#### 驗證證據

##### Evidence 1：src/ 無任何呼叫 _handleDuplicateBooks 的程式碼

grep -rn 結果：僅 controller proxy 方法（L1486-1487）與 duplicate-book-merger.js 自身定義出現，無生產呼叫路徑。

##### Evidence 2：handleFileLoad 直接呼叫 mergeAllData，未經 DuplicateBookMerger

overview-page-controller.js L1243-1245 顯示無 strategy 參數，無 DuplicateBookMerger 呼叫：

const writeResult = mode === CONSTANTS.IMPORT_MODE.OVERWRITE
  ? await TagStorageAdapter.replaceAllData(payload)
  : await TagStorageAdapter.mergeAllData(payload)

##### Evidence 3：mergeAllData 已有獨立測試覆蓋（11 個 TC）

tests/unit/storage/adapters/tag-storage-merge-all-data.test.js 涵蓋 TC-M1~M11：原子寫入三 key、wrapper 結構保持、同 id 更新、配額攔截、原子回滾、空匯入、operationLock 序列化。

##### Evidence 4：DuplicateBookMerger vs mergeAllData 語意對照（不對等屬設計差異）

DuplicateBookMerger 同 id 依 strategy 分流（skip/override/merge），merge 策略下 readingStatus isManualStatus=true 優先、progress 取較大、tagIds 聯集、跨平台 (title+authors) 視為重複。mergeAllData 一律以匯入版本覆蓋（同 id），tagIds 一律聯集，不偵測跨平台同書，永不清空本地。

關鍵觀察：UI 只暴露 OVERWRITE / MERGE 兩模式。OVERWRITE 走 replaceAllData，MERGE 走 mergeAllData 簡化合併。DuplicateBookMerger 的 skip/override/merge 三策略與 v2 欄位細部規則未連接任何使用者可觸發路徑。

#### 規範性結論

1. duplicate-handling-v2.test.js（600 行，20+ 案例）為孤兒測試，測試從未在生產啟用的 v2 細部合併規則。
2. overview-import-duplicate-handling.test.js（439 行，UC-01 三策略）亦為孤兒測試。ANA Problem Analysis §2 已識別「測試需遷移」但 Solution 章節漏列入刪除清單。本 IMP 一併刪除以保持一致性，避免 Stage B 後 controller proxy 不存在時殘留 39 個失敗測試。
3. mergeAllData 既有 11 個 TC 對生產路徑覆蓋足夠，無覆蓋缺口需補測試。
4. Stage B 可進行：刪 DuplicateBookMerger 整檔 + controller 三處引用 + 兩個孤兒測試檔。

#### Acceptance #1 狀態

Stage A 完成：mergeAllData 對應測試覆蓋驗證為無需補測試，因孤兒測試的標的（v2 細部合併策略）從未在生產路徑啟用，覆蓋層次不對等屬設計差異而非覆蓋缺口。
