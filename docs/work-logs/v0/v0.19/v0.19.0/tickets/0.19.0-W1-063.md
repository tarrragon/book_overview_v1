### 變更摘要

採選項 A：移除 line 504 `expect(stats.buildDuration).toBeGreaterThan(0)` 計時硬門檻，改為 `toBeGreaterThanOrEqual(0)`。

### 選用方案理由

評估三個選項後選定 A：

| 選項 | 評估 | 採用 |
|------|------|------|
| A（移除 line 504 計時門檻） | 風險最低，與 line 251 既有處理一致；保留 buildDuration 欄位驗證；scope 鎖定單行 | 採用 |
| B（整段移至 tests/perf/） | 該 test 標題雖含「效能指標」但實際斷言含 `totalBooks === 1000`（功能驗證），整段搬移會丟功能斷言或重複定義 | 不採用 |
| C（mock performance.now()） | 需改 src 端（SearchIndexManager 注入 timer），改動範圍超出 test 檔；scope 蔓延 | 不採用 |

### 實作細節

修改 `tests/unit/ui/search/core/search-index-manager.test.js` line 504：

```diff
- expect(stats.buildDuration).toBeGreaterThan(0)
+ // 移除 buildDuration > 0 計時斷言：JIT 暖機 + Jest jsdom mock 下實測可能收到 0ms（W1-063）。
+ // 改為驗證 buildDuration 存在且為非負數，符合 test-assertion-design-rules.md 規則 1，
+ // 與 line 251 toBeGreaterThanOrEqual(0) 一致。上界沿用既有 wall-clock + 50ms 容忍作為大幅退化防護。
+ expect(stats.buildDuration).toBeGreaterThanOrEqual(0)
```

### 不在本 ticket 範圍

- Line 505 `toBeLessThan(endTime - startTime + 50)`：屬上界 sanity 斷言，`buildDuration` 從 `performance.now()` 差值取得且必然 ≤ `Date.now()` 差值（內含於外），+50ms 容忍下結構安全。若未來實證 flaky 另開 ticket。
- Line 251 既有 `toBeGreaterThanOrEqual(0)`：本來就安全，無需動。
