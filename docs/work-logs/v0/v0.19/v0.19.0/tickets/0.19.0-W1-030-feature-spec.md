# 0.19.0-W1-030 功能規格

> TDD Phase 1 功能規格 — Readmoo 書庫提取前自動捲動載入全部書籍。
> 模式：Full（IMP 新增行為，跨 3 個 content script 檔案，演算法含停止條件設計）。
> 修訂：2026-05-22 依 4 視角審查（R1-R10）整合修訂。詳見 §8 修訂記錄。

## 1. Purpose（目的）

### 背景

Readmoo 書庫頁（`read.readmoo.com/#/library`）是 Vue SPA，採虛擬捲動（virtual scroll）lazy-load。
任一時刻 DOM 初次僅渲染約 96 個 `.library-item`，尚未捲動到的書籍不進入 DOM。
W6-012 實機取證確認 Readmoo 目前的虛擬捲動為**累積式**——捲動到底後已載入的書籍項目持續保留在 DOM
（實測捲動到底後 928 個 `.library-item` 同時存在），而非回收式（捲出視窗即移除）。
因此「捲動到底再一次提取」的策略目前成立。

現行提取流程（`extractAllBooks` → `waitForBookElements`）的 resolve 條件是「`length > 0`」——
首批 96 本渲染後即消費，不驗證是否達書庫總數。
W1-001.2 實機驗證確認：用戶帳號 944 本書，extension 單次提取僅取得 96 本（涵蓋率約 10%）。
所有內測使用者書庫提取嚴重不完整，違反 v1.0 內測「核心功能端到端可運作」要求。

### 書庫數量術語（R7 修正）

Readmoo header 文字格式為「擁有 N 本書，其中封存 X 本，借出 Y 本」。本規格區分兩個數量：

| 術語 | 定義 | 範例（W1-001.2 帳號） |
|------|------|----------------------|
| 書庫總數 | header 的 N（含封存、借出） | 944 |
| 可見書目數 | N - X - Y，library 列表實際渲染為 `.library-item` 的書籍數 | 約 928 |

**提取涵蓋率的比對基準是「可見書目數」**，非書庫總數。封存與借出書籍不渲染為 `.library-item`，
故 `loadedCount` 的目標值與 `coverageComplete` 判定皆以可見書目數為準。
本規格其餘章節凡涉及目標數，一律指可見書目數（`expectedTotal`）。

### 目標

在提取前自動捲動 library 容器，重複「捲動 → 等待新項目渲染 → 量測已載入書籍數」直到達可見書目數或穩定，
再執行提取。涵蓋率從約 10%（96/928）提升至 100%，或於不一致時明確記錄「無法載入數 + 原因」。

`docs/bookstores/readmoo.md` §虛擬 scroll（W6-021 ANA 結論）已書面化此演算法，
本 ticket 將其落地至 production extraction code。

### 非目標

- 不修改 Book ID 生成策略、parser 容錯邏輯（W6-012.2.x 已完成，本 ticket 不動）。
- 不清理過時 `readmoo.com/library` URL pattern（屬 W1-029.1 / W1-029.2 範圍）。
- 不實作背景增量同步 / 提取後續抓（一次提取即載入全部）。
- 不處理 SPA 路由切換返回時的重新提取觸發（W1-030 只負責「提取被觸發後」載入全部）。
- 不改動 background SW / storage / overview UI（本 ticket 僅 content script 提取 pipeline）。

## 2. API Signatures（介面定義）

### ReadmooAdapter — 捲動載入主方法（新增）

於 `src/content/adapters/readmoo-adapter.js` 的 adapter 物件新增方法：

```javascript
/**
 * 程式化捲動 library 容器，反覆觸發虛擬捲動 lazy-load，
 * 直到累積已載入的 unique 書籍數達可見書目數、連續數輪穩定、或達上限/逾時。
 *
 * @param {Object} [scrollOptions={}]
 * @returns {Promise<LoadAllBooksResult>}
 */
async loadAllBooksLazy (scrollOptions = {})
```

**LoadAllBooksResult 結構**：

```javascript
{
  loadedCount: number,        // 捲動載入後累積收集到的 unique 書籍數（見下方「量測方式」）
  expectedTotal: number|null, // 從頁面解析的可見書目數；解析失敗為 null
  coverageComplete: boolean,  // loadedCount >= expectedTotal（expectedTotal 為 null 時恆 false）
  missingCount: number,       // expectedTotal - loadedCount（>= 0；expectedTotal 為 null 時為 0）
  stopReason: string,         // 枚舉，見下表
  iterations: number,         // 實際執行的捲動輪數
  durationMs: number          // 捲動載入耗時
}
```

**loadedCount 量測方式（R1 設計決策，採 unique book ID Set）**：

`loadedCount` 不直接取「當下 DOM 的 `.library-item` 數量」，而是維護一個累積 `Set<bookId>`：
每輪捲動後對當下 DOM 內所有 `.library-item` 解析其書籍識別符（沿用 adapter 既有的 book ID 解析邏輯），
加入 Set；`loadedCount` 即 Set 的大小。

決策理由見 §7 設計決策 D1。要點：累積 Set 對「DOM 回收式虛擬捲動」天然免疫（即使 Readmoo 未來改版
啟用 DOM 回收，捲出視窗的書籍仍已被收進 Set），且 Set 天然去重，避免重複 ID 灌大 count。
代價為每輪對 DOM item 多一次 ID 解析，屬輕量讀取操作。

**stopReason 枚舉**（共 6 個值）：

| stopReason | 觸發條件 | iterations |
|-----------|---------|-----------|
| `reached_total` | 捲動後 `loadedCount >= expectedTotal` | >= 1 |
| `already_complete` | 首次量測即 `loadedCount >= expectedTotal`，未進入捲動迴圈 | 0 |
| `count_stable` | 連續 `stableRounds` 輪 `loadedCount` 不變 | >= stableRounds |
| `max_iterations` | 達 `maxIterations` 輪上限 | == maxIterations |
| `timeout` | 累計耗時達 `overallTimeoutMs` | >= 1 |
| `container_not_found` | 捲動容器所有 fallback 皆失敗 | 0 |
| `error` | 捲動或量測過程拋出例外被捕捉 | >= 0 |

> 實際為 7 個枚舉值。`reached_total` 與 `already_complete` 維持區分（R3 設計決策 D3）：
> 兩者語義不同（有無實際捲動），對涵蓋率診斷日誌有操作意義，`iterations` 欄位亦同步區分。

**scrollOptions 參數說明**（R3 設計決策：移除 `skipWhenComplete`，保留 `overallTimeoutMs`）：

| 參數 | 型別 | 說明 | 有效範圍 / 預設 |
|------|------|------|---------|
| maxIterations | number | 捲動輪數上限（防無限迴圈） | 1-100，預設 30 |
| stableRounds | number | 書籍數連續不變達此輪數即停止 | 2-5，預設 3 |
| renderWaitMs | number | 每輪捲動後等待新項目渲染的毫秒數上限 | 200-3000，預設 800 |
| overallTimeoutMs | number | 整體捲動載入逾時上限（單輪 hang 的硬上限） | 5000-120000，預設 60000 |

`skipWhenComplete` 已移除：首次量測即達標由停止條件 (a) 直接判定為 `already_complete`，無需獨立旗標。

**回傳值**：永遠 resolve（不 reject）。捲動容器找不到時回傳 `stopReason: 'container_not_found'` 且 `loadedCount` 為當下 DOM 解析出的書籍數（降級為現行行為）。

### ReadmooAdapter — 捲動容器辨識（新增私有輔助）

```javascript
/**
 * 多層 fallback 辨識虛擬捲動容器。
 * @returns {{ container: HTMLElement|null, strategy: string }}
 */
findScrollContainer ()
```

`strategy` 枚舉：`selector` / `scrollable-ancestor` / `load-more-button` / `document` / `none`。

### ReadmooAdapter — 書庫總數解析（新增私有輔助）

```javascript
/**
 * 從頁面 header 文字解析「可見書目數」。
 * 來源文字格式：「擁有 N 本書，其中封存 X 本，借出 Y 本」→ 可見書目 = N - X - Y。
 * @returns {{ total: number|null, raw: string }}
 */
parseLibraryTotal ()
```

`total` 為可見書目數（N - X - Y）；文字格式變更或元素不存在時為 `null`。

### extractAllBooks 整合（修改既有）

`extractAllBooks` 在呼叫 `waitForBookElements` 之前先呼叫 `loadAllBooksLazy`，
並將 `LoadAllBooksResult` 的涵蓋率資訊併入既有 `EXTRACTION_COMPLETED` 日誌。
`extractAllBooks` 回傳型別不變（仍為 `Promise<Object[]>`），維持呼叫者相容。

**`waitForBookElements` 的角色（R6 修正）**：`waitForBookElements` 設計為「等待書籍元素出現」
（resolve 條件 `length > 0`），不是「等待 DOM 穩定」。捲動載入成功後 DOM 必已有 `> 0` 個 item，
故成功路徑中 `waitForBookElements` 僅作為**降級路徑的安全網**——當 `loadAllBooksLazy` 回傳
`container_not_found` 或 `error`、未能保證 DOM 狀態時，`waitForBookElements` 確保至少有書籍元素
可供提取。成功路徑（`reached_total` / `already_complete` / `count_stable`）下 DOM 已就緒，
`waitForBookElements` 會立即 resolve，不造成額外延遲。

## 3. GWT Scenarios（行為場景）

### 正常流程

#### 場景 1: 大型書庫完整捲動載入
- **Given**: 書庫頁書庫總數 944 本，header 可解析出可見書目數約 928，首批 DOM 僅渲染 96 個 `.library-item`
- **When**: `extractAllBooks` 觸發提取
- **Then**: `loadAllBooksLazy` 反覆捲動至累積 unique 書籍數達 928，`stopReason='reached_total'`，`coverageComplete=true`，後續提取產出約 928 筆書籍資料

#### 場景 2: 小型書庫首批即完整
- **Given**: 書庫帳號可見書目數 < 96，首批渲染即包含全部書籍
- **When**: `loadAllBooksLazy` 首次量測 `loadedCount >= expectedTotal`
- **Then**: 略過捲動迴圈，`stopReason='already_complete'`，`iterations=0`

#### 場景 3: 書籍數連續穩定後停止
- **Given**: 書庫已捲動到底，但累積 `loadedCount` 仍小於解析出的 `expectedTotal`（部分書籍如已下架無法渲染）
- **When**: 連續 `stableRounds`（預設 3）輪 `loadedCount` 不再增加
- **Then**: 停止捲動，`stopReason='count_stable'`，記錄 `missingCount` 與原因「捲動到底後書籍數穩定但未達可見書目數」

### 邊界條件

#### 場景 4: 達捲動輪數上限
- **Given**: 書籍數每輪持續微幅增加但未達可見書目數，已執行 `maxIterations`（預設 30）輪
- **When**: 第 30 輪完成
- **Then**: 強制停止，`stopReason='max_iterations'`，回傳當下 `loadedCount`，記錄 `missingCount`

#### 場景 5: 整體逾時
- **Given**: 單輪渲染緩慢（網路節流）或 MutationObserver 久未觸發，累計耗時達 `overallTimeoutMs`（預設 60000ms）
- **When**: 任一輪結束後（或輪內等待時）檢測到逾時
- **Then**: 停止捲動，`stopReason='timeout'`，回傳當下 `loadedCount`，記錄 `missingCount`

#### 場景 6: 頁面總數無法解析
- **Given**: header 文字格式變更或元素不存在，`parseLibraryTotal` 回傳 `total: null`
- **When**: `loadAllBooksLazy` 執行
- **Then**: 捲動迴圈仍執行（依「連續穩定」停止），`expectedTotal=null`、`coverageComplete=false`、`missingCount=0`，日誌記錄「無法取得可見書目數，依穩定條件停止」

### 異常流程

#### 場景 7: 捲動容器找不到（降級）
- **Given**: Readmoo DOM 結構變更，`findScrollContainer` 所有 fallback 策略皆回傳 `none`
- **When**: `loadAllBooksLazy` 執行
- **Then**: 不執行捲動迴圈，回傳 `stopReason='container_not_found'`、`loadedCount` 為當下 DOM 解析出的書籍數；`extractAllBooks` 降級為現行提取行為（僅取已渲染書籍），輸出 `warn` 級 log `SCROLL_CONTAINER_NOT_FOUND`

#### 場景 8: 捲動過程拋出例外
- **Given**: 捲動或量測過程中 DOM 操作拋出例外
- **When**: 例外於 `loadAllBooksLazy` 內被捕捉
- **Then**: 終止捲動迴圈，回傳 `stopReason='error'` + 當下已累積的 `loadedCount`，記錄 `error` 級 log（含錯誤訊息與元件名稱）；提取流程以已載入書籍繼續，不整體失敗

#### 場景 9: 書庫為空（0 本）
- **Given**: 空書庫帳號，DOM 無 `.library-item`，`parseLibraryTotal` 解析出 0 或 null
- **When**: `loadAllBooksLazy` 執行
- **Then**:
  - `parseLibraryTotal` 回傳 0 → 首輪量測 `loadedCount=0 >= expectedTotal=0`，`stopReason='already_complete'`，`iterations=0`
  - `parseLibraryTotal` 回傳 null → 進入迴圈，因 `loadedCount` 恆為 0 連續穩定，`stopReason='count_stable'`
  - 兩路徑皆不無限捲動，提取結果為空陣列（沿用現行 `NO_BOOK_ELEMENTS_FOUND` 行為）

## 4. Error Handling（錯誤處理）

| 錯誤情境 | 處理策略 | 回傳/拋出 | 需要回滾？ |
|---------|---------|----------|-----------|
| 捲動容器找不到 | 降級為現行提取 | `LoadAllBooksResult{stopReason:'container_not_found'}` + warn log | 否 |
| 頁面總數無法解析 | 改依連續穩定條件停止 | `expectedTotal=null`，捲動仍執行 | 否 |
| 捲動/量測拋出例外 | 終止迴圈，以已載入書籍繼續 | `stopReason:'error'` + error log | 否 |
| 達 maxIterations | 強制停止防無限迴圈 | `stopReason:'max_iterations'` + 記錄 missingCount | 否 |
| 達 overallTimeoutMs | 強制停止防單輪 hang | `stopReason:'timeout'` + 記錄 missingCount | 否 |
| 捲動無效（捲動位置未變化） | 早期降級下一 fallback 策略（R2） | 換策略；全失敗則 `container_not_found` | 否 |
| 涵蓋率不一致（loadedCount < expectedTotal） | 記錄而非失敗 | `coverageComplete=false` + warn log（含 missingCount + stopReason） | 否 |

**捲動有效性探測（R2 整合）**：每輪捲動後，除量測書籍數外，另檢查捲動是否實際生效——
比對捲動前後的容器 `scrollTop`（或 window `scrollY`）。若連續輪捲動位置完全未變化且
`loadedCount` 亦未增加，判定當前 fallback 策略對此頁面無效，早期切換至下一順位 fallback 策略
（見 §5 捲動容器辨識策略表），避免耗盡 `maxIterations` 才發現策略失效。所有策略皆無效時回傳
`container_not_found`。

**渲染等待策略（R4 整合，混合式）**：每輪捲動後的「等待新項目渲染」採固定上限 + 提早結束混合——
在容器上掛 `MutationObserver` 監聽子節點新增，一旦偵測到新 `.library-item` 即提早結束本輪等待；
若 `renderWaitMs`（預設 800ms）內無新增則逾時結束本輪。此設計避免固定 800ms 在慢網路下
「等不夠」誤觸發 `count_stable` 提早停止，也避免快網路下無謂等滿 800ms。
`expectedTotal` 可解析時，停止條件 (a)`loadedCount >= expectedTotal` 為主要判定，
`count_stable` 僅作為 expectedTotal 不可達時的退路，進一步降低慢網路誤判風險。

**涵蓋率回報日誌契約**（acceptance #2）：

- `loadAllBooksLazy` 結束後輸出 `info` 級 `SCROLL_LOAD_COMPLETED`，欄位含 `loadedCount` / `expectedTotal` / `coverageComplete` / `stopReason` / `iterations` / `durationMs`。
- `coverageComplete === false` 時額外輸出 `warn` 級 `COVERAGE_INCOMPLETE`，欄位含 `missingCount` 與停止原因（即 `stopReason` 對應的可讀說明）。
- `extractAllBooks` 既有 `EXTRACTION_COMPLETED` 日誌併入 `expectedTotal` / `coverageComplete` 欄位。

**防無限迴圈契約**（acceptance #3）：捲動迴圈每輪結束必檢查四個停止條件（任一成立即停止）：
(a) `loadedCount >= expectedTotal`（expectedTotal 非 null）；(b) 連續 `stableRounds` 輪 `loadedCount` 不變；
(c) `iterations >= maxIterations`；(d) 累計耗時 `>= overallTimeoutMs`。
四條件構成獨立於 DOM 行為的硬上限。其中 (c) 限制輪數、(d) 限制單輪 hang（如 MutationObserver
永不觸發），兩者覆蓋不同失效模式，不冗餘（R3 設計決策 D2）。DOM 永不穩定亦保證終止。

## 5. Dependencies（依賴）

| 依賴 | 類型 | 用途 | 不可用時 |
|------|------|------|---------|
| `window` / `document` DOM API | 瀏覽器內建 | `scrollTo` / `scrollTop` / `scrollHeight` / `querySelectorAll` 捲動與量測 | content script 環境保證存在；非瀏覽器（測試）由 jsdom mock |
| `MutationObserver` | 瀏覽器內建 | 偵測新書籍項目渲染，提早結束輪內等待 | jsdom 支援；不可用時退回固定 `renderWaitMs` 等待 |
| `Logger`（`src/core/logging/Logger`） | 專案模組 | 涵蓋率與停止原因可觀測性日誌 | 已為 adapter 既有依賴 |
| `waitForBookElements`（同檔既有方法） | 同模組方法 | 降級路徑安全網（見 §2） | 既有，無新增風險 |
| Readmoo DOM 結構（`.library-item`、捲動容器、「更多...」按鈕、header 總數文字） | 外部頁面契約 | 元素辨識與總數解析 | 多層 fallback；最終降級為現行提取行為 |

**整合點**（acceptance #1，注入位置）：

| 檔案 | 修改內容 |
|------|---------|

| `src/content/content-modular.js` | 本 ticket 不修改（R9 決策，見 §7 設計決策 D4） |

**捲動容器辨識策略（多層 fallback，R2 整合「更多...」按鈕路徑）**：

| 優先級 | 策略 (`strategy`) | 說明 |
|--------|------|------|
| 1 | `selector` | 嘗試 Readmoo library 捲動容器 class（實作時以 Phase 3a 實機 DOM 取證為準，列入 `SELECTORS`） |
| 2 | `load-more-button` | 偵測並點擊底部「更多...」按鈕觸發 lazy-load（readmoo.md §虛擬 scroll 明列此為官方載入路徑之一） |
| 3 | `scrollable-ancestor` | 從 `.library-item` 向上找 `scrollHeight > clientHeight` 且 `overflow` 允許捲動的祖先元素 |
| 4 | `document` | 退回 `window.scrollTo(0, document.body.scrollHeight)` 捲動整頁 |
| 5 | `none` | 以上皆失敗 → `strategy='none'`，觸發場景 7 降級 |

> R2 設計考量：原規格的 `window.scrollTo` 對 Vue 內層捲動容器可能無效。新增「更多...」按鈕
> 路徑（優先級 2）作為對內層容器不依賴捲動事件的可靠替代；策略順序與 §4「捲動有效性探測」

## 6. Acceptance（驗收條件）

## 7. 設計決策記錄（R1/R3/R8/R9 權衡）

本章記錄多視角審查中屬「設計決策」的取捨，依 design-decision-framework 權衡，禁止盲套單一視角建議。

### D1（R1）：loadedCount 量測方式 — 採累積 unique book ID Set

| 方案 | 優點 | 缺點 |
|------|------|------|
| A. 當下 DOM `.library-item` 數量 | 實作最簡，零額外解析 | 隱含「累積式虛擬捲動」前提；Readmoo 改版啟用 DOM 回收即靜默失準；DOM 重複/重排時 count 不可靠 |
| B. 累積 unique book ID Set 大小（採用） | 對回收式虛擬捲動天然免疫；天然去重；直接服務 AC#4「提取數與書庫總數一致」 | 每輪對 DOM item 多一次 ID 解析 |

**決策：採方案 B。** 理由：thyme 以 W6-012 證據確認「目前」為累積式，但此為**未標示的隱性前提**
（R1 重要級發現）——前提失效時方案 A 無任何訊號即靜默產出錯誤涵蓋率。方案 B 的代價（每輪一次
輕量 ID 解析，沿用 adapter 既有 book ID 解析邏輯）相對於「robustness + 天然去重 + 直接支撐 AC#4」
的收益明顯偏低。非盲採 thyme 建議：已確認 ID 解析邏輯已存在（Explore 重用清單），無新造輪子成本。

### D2（R3）：停止條件 (c)(d) 是否冗餘 — 保留 overallTimeoutMs

linux L2 建議刪 `overallTimeoutMs`/`timeout`，理由「`maxIterations` 已限制輪數，逾時冗餘」。

**決策：拒絕簡化，保留 overallTimeoutMs。** 理由：`maxIterations` 只限制**輪數**，不限制**單輪耗時**。
本規格 R4 整合後每輪含 `MutationObserver` 等待——若 observer 因 DOM 異常永不觸發，單一輪可能
無限 hang，`maxIterations` 完全無法兜底。`overallTimeoutMs` 是針對「單輪 hang」的獨立硬上限，
與 `maxIterations`（針對「輪數爆炸」）覆蓋不同失效模式，不冗餘。盲套 linux 簡化會留下單輪 hang
的無防護缺口。

### D3（R3）：reached_total / already_complete 是否合併 — 維持區分

linux L3 建議合併兩枚舉以壓縮 stopReason 數量。

**決策：維持區分。** 理由：兩者語義不同——`already_complete` 表示「首批即完整、未發生任何捲動」，
`reached_total` 表示「捲動後達標」。此差異對涵蓋率診斷有操作意義（運維可據此判斷捲動機制是否
被觸發）。`iterations` 欄位雖也能區分（0 vs >0），但 stopReason 作為單一可讀訊號直接表達意圖，
壓縮後反而需交叉 `iterations` 才能還原語義。枚舉數從 7 壓到 6 的收益小於診斷清晰度損失。

### D4（R3）：scrollOptions 參數收斂 — 移除 skipWhenComplete

**決策：移除 `skipWhenComplete`（5 參數收斂為 4）。** 理由：此旗標的功能與停止條件 (a)
`loadedCount >= expectedTotal` 在首次量測時的判定完全重疊——首次量測達標即 `already_complete`，
無需獨立旗標控制。移除消除「旗標 true 但條件 (a) 為 false」的不一致可能。其餘 4 參數
（maxIterations / stableRounds / renderWaitMs / overallTimeoutMs）各對應獨立的停止/節流維度，
無進一步可刪者。

### D5（R8）：程式化捲動對使用者可見頁面的副作用

**決策（狀態 a）：捲動載入結束後，將捲動位置還原至提取開始前的位置。** 理由：提取為背景行為，
使用者未預期頁面被自動捲到底。`loadAllBooksLazy` 開始前記錄容器 `scrollTop`（或 window `scrollY`），
結束時（無論何種 stopReason）還原。此還原為 best-effort——還原失敗不影響提取結果，僅記錄 debug log。
此項列入 §4 錯誤處理隱含的收尾步驟，Phase 3b 實作時於 `loadAllBooksLazy` 的 finally 區塊處理。

### D6（R9）：content-modular.js 是否需修改

原規格 §5 寫「content-modular.js 評估是否需要，初版可不動」——屬無 trigger 延後決策（R9 / basil 指出）。

**決策（狀態 a）：本 ticket 不修改 content-modular.js。** 理由：捲動載入邏輯完全由
`extractAllBooks` 內部承載（`loadAllBooksLazy` 在 `extractAllBooks` 內呼叫），`content-modular.js`
的 `reportReadyStatus` / `getPageStatus` 取得的 `bookCount` 來自提取結果，提取結果在捲動載入後
自然反映完整數，無需 `content-modular.js` 端任何改動。Phase 3b 實作完成後須驗證
`content-modular.js` 相關測試無回歸（已列入 §6 acceptance 末條）。

**SPA 路由切換返回時的重新提取觸發**（`content-modular.js` 串接 `event-utils.onURLChange`）
不在本 ticket 範圍（§1 非目標已明列）。此為獨立功能需求，若需推進屬**待 PM 評估建立 follow-up
ticket**——本規格不自行建立，標記交 PM 決策。
