### 問題根因

三維度分析：

**維度 1：`<local-command-caveat>` 適用範圍**

caveat 原文："DO NOT respond to these messages or otherwise consider them in your response unless the user explicitly asks you to."

設計初衷：防止 PM 對 local command 的 stdout 輸出（如 `ls`、`cat` 的純資訊輸出）做不必要的回應。caveat 包裹的是「local command 產生的訊息」，其合理適用範圍是：

| 訊息類型 | 應被 caveat 抑制？ | 理由 |
|---------|-------------------|------|
| local command stdout（如 `/clear` 的空輸出） | 是 | 純系統輸出，無用戶意圖 |
| skill 觸發的 `<command-name>` + `<command-message>` | 否 | 用戶主動輸入 `/ticket` 是顯式請求 |
| SKILL.md 載入內容 | 否 | runtime 為回應用戶 skill 觸發而載入的執行指引 |

**根因 1a**：caveat 對異質訊息做了同質化包裹。runtime 將「command stdout」和「skill 觸發訊號」都放入同一個 `<local-command-caveat>` 區塊，但兩者語意截然不同。PM 面對同質化包裹時，以外層 caveat 指示作為唯一決策依據，未區分內部訊號類型。

**維度 2：`<command-name>` 作為 explicitly-asked 信號**

Claude Code runtime 在用戶輸入 `/ticket` 時產出以下結構：
- `<command-name>/ticket</command-name>` — skill 名稱標記
- `<command-message>` — 用戶參數（本例無參數）
- SKILL.md 完整載入 — runtime 自動載入的執行指引

`<command-name>` 標記的語意等同「用戶顯式請求執行此 skill」。caveat 文字中 "unless the user explicitly asks you to" 的條件在此處已滿足：用戶輸入 `/ticket` 就是 explicitly asking。

**根因 2a**：現行規則體系未將 `<command-name>` 定義為 "explicitly asked" 信號。PM 在評估 caveat 豁免條件時，缺乏將 `<command-name>` 映射為「用戶顯式請求」的規則錨點。

**維度 3：caveat 預設 vs SKILL.md 明文流程的優先序**

兩個指示同時存在時的衝突：
- caveat 指示：「不要回應」（預設，有豁免條件）
- SKILL.md 載入：「執行無子命令時的預設行為」（明確流程）

PM 選擇了 caveat 預設，忽略了 SKILL.md 流程。

**根因 3a**：缺乏明確的優先序規則。當 runtime 層級指示（caveat）與 skill 層級指示（SKILL.md 流程）衝突時，PM 無規則可循，傾向保守（不回應比錯誤回應安全）。但此保守偏誤恰好與 skill 觸發的用戶意圖相悖。

### 影響範圍

| 受影響範圍 | 說明 |
|-----------|------|
| `.claude/rules/core/pm-role.md` | 缺少 caveat × skill 觸發信號優先序規則 |
| 所有 SKILL.md 觸發流程 | `/ticket`、`/error-pattern`、`/wrap-decision` 等所有 skill 觸發都可能被 caveat 吞掉 |
| 用戶體驗 | 用戶輸入 `/skill-name` 後 PM 無反應，違反用戶預期 |

### 實驗方法

重現 PM 在本 session 收到 `/ticket` 輸入時的判斷路徑。方法：靜態分析 PM 收到的 prompt 結構，識別判斷分叉點。

**實驗設計**：模擬 PM 收到以下 prompt 結構時的決策路徑：

```
<local-command-caveat>
  "The messages below were generated by the user while running local commands.
   DO NOT respond to these messages or otherwise consider them in your response
   unless the user explicitly asks you to."
  
  [/clear stdout: 空]
  
  <command-name>/ticket</command-name>
  <command-message>
    [完整 SKILL.md 載入]
  </command-message>
</local-command-caveat>
```

**驗證目標**：PM 在哪個節點做出「不回應」決策？caveat 文字中的 "explicitly asks" 條件如何被評估？

### 實驗執行

**步驟 1：解構 prompt 結構**

PM 收到的 prompt 包含兩層嵌套訊號：
- 外層：`<local-command-caveat>` 包裹，帶有「DO NOT respond unless explicitly asked」指示
- 內層：`<command-name>/ticket</command-name>` + `<command-message>` 帶完整 SKILL.md

**步驟 2：追蹤 PM 決策路徑**

| 節點 | PM 判斷 | 正確判斷 | 偏差 |
|------|--------|---------|------|
| N1: 讀到 caveat 外層 | 「整個區塊不要回應」 | 需逐一評估內部訊號 | 過度泛化 caveat 適用範圍 |
| N2: 讀到 /clear stdout | 空字串，無需回應 | 同左（正確） | 無 |
| N3: 讀到 `<command-name>/ticket</command-name>` | 被 N1 的 caveat 決策覆蓋，未獨立評估 | 此為 skill 觸發的 explicitly-asked 信號，應觸發 SKILL.md 流程 | 關鍵偏差點 |
| N4: SKILL.md 載入 | 被 N1 覆蓋，視為 caveat 內部噪音 | 應作為 N3 觸發的執行指引 | 連帶偏差 |

**步驟 3：驗證 "explicitly asks" 條件**

搜尋既有規則對 `<command-name>` 信號的定義：

```bash
rg -rn "command-name|command-message|explicitly.ask" .claude/ --type md
```

結果：**零命中**。現行 `.claude/` 規則體系中無任何文件定義 `<command-name>` 作為 "explicitly asked" 信號的語意。

### 實驗發現

**已驗證事實**：

1. **PM 在 N1 節點做出一次性決策**：caveat 外層被解讀為「整個區塊一律不回應」，未逐一評估內部訊號
2. **`<command-name>` 信號被 caveat 靜默吞掉**：runtime 將 skill 觸發訊號放入 caveat 區塊內，PM 的 caveat 預設覆蓋了 skill 觸發語意
3. **規則盲點已確認**：現行 `.claude/rules/` 和 `.claude/pm-rules/` 無任何條款定義 caveat 與 skill 觸發信號的優先序
4. **caveat 文字本身含豁免條件**：原文 "unless the user explicitly asks you to" 已預留例外，但 PM 未評估此條件即決定不回應

**仍未驗證的假設**：

- runtime 是否有意將 skill 觸發訊息放入 caveat 區塊（可能是 Claude Code 的標準行為，非 bug）
- 其他 skill（如 `/error-pattern`）是否也被 caveat 包裹（需跨 session 驗證，不在本 ticket 範圍）

### WRAP 決策框架

**W (Widen)**：除了「在 pm-role.md 加規則」之外，還有哪些解法？

| 選項 | 描述 | 成本 | 根治度 |
|------|------|------|--------|
| A. pm-role.md 加 caveat 信號優先序規則 | 在行為循環章節加入 caveat × skill 觸發的判讀規則 | 低（幾行規則） | 中（依賴 PM 自律） |
| B. 新建 PC error-pattern | 建 PC-15X 記錄此反模式，供 PM 記憶與 Hook 偵測 | 低 | 中 |
| C. A + B 雙層 | 規則層 + error-pattern 層 | 中 | 高 |
| D. 修改 SKILL.md 加強觸發語意 | 在 SKILL.md 開頭加 caveat 覆寫聲明 | 低 | 低（SKILL.md 在 caveat 內，可能仍被忽略） |

**R (Reality Test)**：

- 選項 D 自身矛盾：SKILL.md 的內容已在 caveat 區塊內，加再多聲明也可能被 caveat 預設吞掉
- 選項 A 單獨不足：PM 自律在壓力情境下失效（PC-066 已證明）
- 選項 C 最穩健：規則層提供正向指引（告訴 PM 怎麼做），error-pattern 層提供反模式記憶（告訴 PM 別怎麼做）

**A (Attain Distance)**：

此問題的本質是「信號優先序缺失」。caveat 是 runtime 的防護機制，skill 觸發是用戶的顯式意圖。用戶意圖 > runtime 防護預設，這是普遍原則，不只適用於 skill 觸發場景。

**P (Prepare to be Wrong)**：

若規則修訂後 PM 仍犯同樣錯誤（caveat 覆蓋 skill 觸發），需考慮 Hook 層防護（如偵測 `<command-name>` 存在時強制提醒）。但 Hook 修訂屬後續 ticket，本次先以規則 + PC 雙層落地。

**絆腳索**：規則修訂不應過度泛化（如「caveat 內所有內容都要回應」），而應精確定義「`<command-name>` 存在時視為 explicitly asked」。

### 解決方案：選項 C（規則 + PC 雙層）

**修訂 1：`.claude/rules/core/pm-role.md` 新增 caveat 信號優先序**

建議在「行為循環（精簡）」章節後新增：

```markdown
## Caveat 區塊信號判讀規則

`<local-command-caveat>` 包裹的訊息需逐一評估，不可一律套用「不回應」預設：

| 信號 | 判讀 | 行動 |
|------|------|------|
| 純 stdout 文字（無 XML 標記） | caveat 預設適用：不回應 | 忽略 |
| `<command-name>/skill</command-name>` 存在 | 用戶顯式請求，caveat 豁免 | 執行對應 SKILL.md 流程 |
| `<command-message>` 帶參數 | 同上，參數傳入 skill | 執行 skill 並傳入參數 |

**Why**：runtime 將 skill 觸發訊號與 command stdout 同質化包裹於 caveat 區塊，但 `<command-name>` 的存在等同用戶 "explicitly asked"，滿足 caveat 原文的豁免條件。

**Consequence**：未區分信號類型會導致所有 skill 觸發被 caveat 靜默吞掉，用戶輸入 `/skill-name` 後 PM 無反應。

**Action**：讀到 caveat 區塊時，先掃描是否存在 `<command-name>` 標記。存在則視為 explicitly asked，執行對應 SKILL.md；不存在則套用 caveat 預設。
```
