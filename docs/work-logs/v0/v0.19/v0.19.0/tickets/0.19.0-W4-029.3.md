## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
評估 CC v2.1.154 dynamic workflow（/workflows + Workflow 工具，背景編排數十至數百 agent）是否納入本專案 PM 派發策略。

## 重現實驗結果
機制比對（workflow 工具能力 vs 既有 skill）：

| 能力 | Workflow 工具 | 既有機制 |
|------|--------------|---------|
| 批量拆分 | fan-out（parallel/pipeline） | bulk-evaluate（拆 N 子 ticket 寫回） |
| 並行派發 | parallel() barrier | parallel-dispatch 規則 |
| 多視角審查 | dimension + verify pattern | parallel-evaluation（三人組 + linux） |
| deterministic 控制流 | 獨有（loop/conditional/pipeline 無 barrier） | 無對等 |
| 大規模（數百 agent） | 獨有 | 無（專案規模不需） |
| 與 ticket/hook 整合 | 否（用 script，不寫 ticket / 不觸發驗收 hook） | bulk-evaluate 整合（寫回 ticket + acceptance gate） |
| 啟用 | 需 opt-in（「workflow」關鍵字 / ultracode） | 預設可用 |

關鍵差異：workflow 是獨立編排層（script-based，token 密集，不整合專案 ticket/hook）；既有機制整合 ticket 生命週期與驗收 hook。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP

- **W**：(a) 引入 workflow 到派發規則 / (b) 不引入，記錄適用場景 / (c) 僅 bulk-evaluate 場景補 workflow 選項。
- **R**：(a) 屬 YAGNI——本專案為中小型 Chrome extension，無「數百 agent / 全 repo 大規模編排」需求，且 workflow 不整合 ticket/hook（喪失驗收防護）；(b) 符合現況；(c) bulk-evaluate 已覆蓋專案規模批量需求。
- **A**：引入規則增加 PM 認知負擔卻無對應需求；不引入零成本。既有 bulk-evaluate + parallel-dispatch + parallel-evaluation 已滿足現有規模。
- **P 絆腳索**：條件式重評——若未來出現「單批 > 20 子任務」或「全 repo 掃描類」需求，屆時新建 ANA 評估 workflow 引入。此為量化條件觸發，非無 trigger 延後。

### 結論（已決策）

**不引入 workflow 到 PM 派發策略規則**（方案 b）。理由：YAGNI——專案規模不需大規模編排，既有機制（bulk-evaluate 整合 ticket/hook）更適合本專案。workflow 列為平台可選工具，遇上述量化條件時再評估。

**無需建 ticket**：本結論為「不引入」明確決策，不改派發規則，無落地 child。重評由量化條件觸發（非延後）。
