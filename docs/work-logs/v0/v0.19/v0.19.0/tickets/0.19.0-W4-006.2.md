### PM 認領盤點（WRAP 三問 + 派發決策，2026-05-31）

#### 問題根因（繼承 W4-006.1 ANA 結論）

popup-error-handler.js 第 3 行 `let Logger, ErrorCodes, MessageDictionary`（W1-117 引入）將 Logger 放入 HTML page 共享的 classic script global lexical record；popup-diagnostic-enhancer.js / popup-initialization-tracker.js 後續再宣告 `var Logger` 時 parse 階段即拋 `SyntaxError: Identifier 'Logger' has already been declared`，導致 tracker.js 整檔不執行 → popup.js 內 `new PopupInitializationTracker()` 拋 ReferenceError。spike v7/v8 已重現並驗證 stack trace。

#### 影響範圍

- popup 載入：3 筆 pageerror（Logger SyntaxError x2 + Tracker ReferenceError x1）
- popup UI button 觸發層 E2E 覆蓋缺口（W1-008 Phase 3b 遺留）
- 整合測試（popup-interface.test.js）能通過但屬 mock 環境，未覆蓋真實 popup.html 載入鏈

#### 相關 Error Pattern

無直接命中既有 PC pattern；W4-006.1 ANA 結論已記載為跨檔 classic script global scope 汙染反模式。

#### W（Widen）候選方案

繼承 W4-006.1 ANA 4 方案，已 parallel-evaluation 確定推薦方案 B：
- 方案 B（已選）：合併 popup 4 檔為單一 popup.js entry，esbuild 自動 bundle 解決 Logger 衝突
- 方案 A（後續 TD）：popup 全面 ESM 化（超本 ticket 範疇）
- 方案 C/D（已否決）：let→var 補丁 / build.js IIFE 包覆

#### A（Attain distance）機會成本

P2 ticket 涉及 popup 4 檔 + 1 個新 E2E test，估時相對長。完成後解鎖 W4-006.3（Phase 4a 3 項 TD 重構），且收尾 W4-006 整條鏈。並行可選 W3-094 (P2) 但屬被動觀察任務，W4-006 鏈是主動修復。優先序合理。

#### P（Prepare to be wrong）失敗預想

- **預想 1**（bay 已標記，medium-risk）：bundled popup.js 含 esbuild 的 `__commonJS` / `__require` runtime helpers，jest jsdom 環境 eval(popupScript) 執行時 require 解析路徑可能與 esbuild `__require` shim 衝突 → integration test RED。**防護**：實作後第一步跑 integration test 驗證，若 RED 於 Problem Analysis 標記並調整 mock。
- **預想 2**（linux 已標記）：實作者順手清理三個 helper 檔末尾 `window.PopupErrorHandler = X` 顯式注入，破壞 popup-controller / popup-ui-manager / popup-event-controller 對 `window.Popup*` 的訪問 → popup runtime 拋 ReferenceError。**防護**：AC 7 + AC 8 強制保留 + grep 驗證。
- **預想 3**：方案 B require 三檔被 esbuild 處理時，三檔的 `typeof require !== 'undefined'` guard 不執行（bundled 環境 require 已是內部 `__require`）→ Logger 取得路徑變更。**防護**：保留三檔內既有 fallback 鏈 `require(...).Logger || window.Logger || {...}`（W4-006.1 linux 讓步建議 3）。
