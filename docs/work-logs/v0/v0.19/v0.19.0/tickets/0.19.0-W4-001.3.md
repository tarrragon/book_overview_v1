## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

UC-07 錯誤場景（DOM 變更降級 / 匯入格式錯誤 / 配額不足）無自動化 regression 防護。錯誤處理優雅降級若回歸（如改成崩潰或錯誤訊息消失），無測試攔截（quality-baseline 規則 5）。

### 影響範圍

- 新增：`tests/e2e/browser/uc07-error-handling.e2e.test.js`（獨立測試檔，不改既有 4 檔含 uc03/uc04）
- 新增錯誤場景 fixtures：`tests/e2e/fixtures/`（如 mutated DOM HTML、malformed JSON）
- 受測 src（只讀，禁改）：頁面偵測 `src/content/detectors/page-detector.js`、提取 `src/extractors/`、匯入驗證 `src/overview/import/file-validator.js`、錯誤工廠 `src/core/errors/UC0XError*.js`

### 框架慣例（重用，禁重建基礎設施）

與 W4-001.1/.2 同一框架（W1-008 建立）：
- `tests/e2e/setup/extension-setup.js` — `ExtensionTestSetup`
- `tests/e2e/browser/helpers/storage-reader.js` — `getServiceWorker` / `clearBooksStorage`
- `tests/e2e/browser/helpers/timeouts.js` — `DEFAULT_NAV_TIMEOUT` / `JEST_TEST_TIMEOUT`
- 參考範本：`uc03-export.e2e.test.js` / `uc04-import.e2e.test.js`（同 wave 已完成，setup + describe 分組 + poll 慣例）

### UC-07 規格要點（docs/use-cases.md L365-446）

| 場景 | 觸發 | 預期系統行為 | E2E 可測性 |
|------|------|------------|-----------|
| B Readmoo DOM 結構變更 | 提取時 DOM 結構不符 | 「無法找到書籍資料，可能是網站結構已更新」優雅降級不崩潰 | 可測（mutated fixture HTML）|
| C 檔案匯入格式錯誤 | 選錯格式檔案 | 「檔案格式不正確 / 內容不符合書籍資料格式」 | 可測（malformed JSON fixture）|
| D/G 儲存配額不足 | chrome.storage 接近 5MB | 配額警告 + 清理建議 | 需 mock storage（可測性評估，若無法 E2E 化須顯性標註理由）|

### E2E 設計策略

1. 場景 B：建 mutated DOM fixture（移除/改名書籍 DOM 結構，仿 readmoo-mock-page.html 但破壞 selector），導航後觸發提取，斷言系統優雅降級（錯誤訊息存在 + 無 pageerror 崩潰）
2. 場景 C：建 malformed JSON fixture（壞 JSON / 結構不符），經 `#loadFileBtn` 注入匯入，斷言顯示格式錯誤訊息（不崩潰、storage 未被污染）
3. 場景 D/G：評估 chrome.storage 配額 mock 可行性；可測則斷言警告訊息，不可 E2E 化則在 Solution 顯性標註理由（acceptance 第 3 項允許此豁免路徑）
4. 新增獨立 test 檔 + 錯誤 fixtures，不改既有測試檔

### 執行前置與指令

- 前置：`npm run build:dev`
- 執行：`npm run test:e2e:browser`（須全綠，含既有 uc03/uc04 + 新增 uc07）；E2E 環境敏感，N>=5 取樣（PC-168）

### 邊界與防護

- 部分 UC-07 場景（A 網路中斷 / E 遷移失敗 / F tag 修復）E2E 化困難，本 ticket 聚焦 B/C/D-G；若某場景無法 E2E 化，在 Solution 顯性標註理由（規則 5：發現即標註，禁無 trigger 延後）
- 受測 src 唯讀。若發現錯誤處理實作有缺陷（如未優雅降級反而崩潰）→ 屬 regression 發現，在 Exit Status 回報建議 PM 建 follow-up，**勿擅改 src**
- 接手者應獨立重新驗證入口/錯誤訊息文字/error factory 結構（PC-162）
