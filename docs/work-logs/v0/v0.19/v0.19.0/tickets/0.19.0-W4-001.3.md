## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 認領三問

- **W（Widen）**：候選方案——(a) 真實塞滿 5MB 觸發配額（脆弱、慢、不可控）；(b) override 用量量測來源命中真實配額閘（採用）；(c) 直接 mock TagStorageAdapter.replaceAllData 回 quota_exceeded（繞過真實配額決策邏輯，降低 regression 價值）。場景 B 候選——(a) 修改既有 extraction-flow helper 服務 mutated fixture（違反「禁改既有」約束）；(b) 在本檔自帶 mutated fixture 的 request interception（採用）。選 (b) 對兩者皆成立：保留真實生產路徑、不改既有資產。
- **A（Attain distance）**：機會成本——E2E 框架已穩定（W1-008 / W4-001.1/.2），此 ticket 補的是錯誤路徑 regression 缺口，與成功路徑互補，不擠壓其他 W4 任務。
- **P（Prepare to be wrong）**：最可能失敗——E2E 環境敏感（headed Chrome / SW cold-start），可能 flaky。防護：N>=5 取樣記錄 GREEN/RED 分佈（PC-168）；所有等待用 bounded poll 取代固定 sleep；dialog auto-dismiss 安全網防 alert 阻塞 hang。

### 實作摘要

新增獨立測試檔 `tests/e2e/browser/uc07-error-handling.e2e.test.js` + 4 個錯誤場景 fixtures（`tests/e2e/fixtures/uc07-error/`），重用 W1-008 框架（ExtensionTestSetup / storage-reader / extraction-flow / timeouts），未改任何 src 與既有測試/helper。

#### 場景 B（DOM 結構變更優雅降級）— acceptance #1

- fixture `readmoo-mutated-page.html`：5 個書卡全用 v2 命名（.shelf-card-v2 / .book-name-v2 / .thumb-v2 / .reading-meter-v2，reader href 改 /v2/reader-app/），使 adapter 三層提取策略（主選擇器 .library-item / alternativeContainers / readerLink 祖先回溯）全失效。
- 本檔自帶 mutated fixture 的 request interception（既有 helper 硬編碼正常 fixture，不可改），SW 直送 START_EXTRACTION，採 bounded poll（6s 窗）確認 storage 維持 0 本，並以 pageerror 監聽 + SW PING 雙重斷言「不崩潰」。
- 斷言：START_EXTRACTION 即時回 success（flow 啟動）、提取 0 本且 storage 未被寫垃圾、無 pageerror、content script PING 存活。

#### 場景 C（檔案匯入格式錯誤）— acceptance #2

- 兩 fixture 覆蓋兩條錯誤路徑（皆 .json 副檔名，通過 FileValidator.validate，錯誤精準發生在 read/parse 階段）：
  - `malformed-broken-syntax.json`：JSON.parse 失敗 → ContentParser._parseJSONContent SyntaxError → PARSE_ERROR（訊息含「格式」）。
  - `structurally-invalid.json`：JSON 合法但非陣列且無 books → _extractBooksFromData path E → VALIDATION_ERROR（訊息含「books」）。
- 真實匯入鏈：注入 #jsonFileInput → click #loadFileBtn → handleFileLoad → ImportFlowController.execute → modal 選模式 → bookFileImporter.read → FileContentReader._handleSuccess 對 parse 拋錯先 showError 再 reject。showError 接線 controller.showError → #errorContainer 顯示 + #errorMessage 文字。
- 斷言：錯誤訊息顯示且文字明確（含「格式」/「books」）、storage 未被污染（books 維持 0）。

#### 場景 D/G（儲存配額不足）— acceptance #3（已 E2E 化，非豁免）

- 經評估場景 D/G **可 E2E 化**（不採豁免路徑）：配額閘為真實生產碼——TagStorageAdapter.replaceAllData/mergeAllData 寫入前呼叫 checkQuotaLevel()，讀 chrome.storage.local.getBytesInUse(null) / MAX_STORAGE_SIZE(5MB)，usageRatio >= 0.95 回 { success:false, error:'quota_exceeded' }，ImportFlowController.execute 轉為 showError('儲存空間不足，匯入未完成')。
- 在 overview page context 覆寫 getBytesInUse 回傳 96% 用量（命中真實 blocked 分支），匯入合法 v2 fixture `valid-v2-for-quota.json`。覆寫僅替換「用量量測來源」一步，不繞過配額決策業務邏輯。測試後還原 getBytesInUse（含 best-effort 失敗還原）避免污染。
- 斷言：顯示「空間不足」訊息、storage 未被寫入（配額閘在持久化前攔截）。

### 設計決策

- 兩層 describe 結構：G1（場景 B）用獨立 ExtensionTestSetup（mutated fixture + storage 隔離）；G2/G3（場景 C / D-G）共用一個 overview-page setup（仿 uc04 beforeAll/afterAll + bounded cleanup race 慣例）。
- 失敗性質前綴：[SETUP]（環境前置）/ [DEGRADE]（場景 B）/ [IMPORT]（場景 C）/ [QUOTA]（場景 D/G），RED 時可快速區分環境故障與 regression。
- 所有等待用 bounded poll（waitForErrorDisplayed / observeStorageStaysEmpty），無固定 sleep（test-assertion-design-rules）。

### 不在本 ticket 範圍（規格 §UC-07 其他場景）
