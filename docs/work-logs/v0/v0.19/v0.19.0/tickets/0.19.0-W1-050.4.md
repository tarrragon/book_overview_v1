### 轉換規則細節（必讀）

**A. `require('crypto')` 整行移除**（6 處）

Web Crypto API 在 CE service worker / browser 環境是 `globalThis.crypto`（無需 import）。直接刪除 require 行 + 移除 `crypto.` 前綴改為 `globalThis.crypto.` 或直接 `crypto.`（如已在頂層作用域沒衝突）。

**B. `crypto.randomBytes(n).toString('hex')` → Web Crypto 同步 hex**（2 處：L515 / L519）

```js
// 取代範本（sync，與 Node API 介面對齊）
function randomHex (n) {
  const bytes = globalThis.crypto.getRandomValues(new Uint8Array(n))
  return Array.from(bytes).map(b => b.toString(16).padStart(2, '0')).join('')
}
// 使用：randomHex(8) 取代 crypto.randomBytes(8).toString('hex')
```

**C. `crypto.createHash('md5'/'sha256').update(data).digest('hex')` → sync JS hash**（7 處）

**避免 async 蔓延**：Web Crypto 的 `crypto.subtle.digest` 為 async，會強迫所有 caller 改 async / await，影響範圍極大。本 ticket 的 7 處用途分析：
- 4 處 md5 + substring → cache key（非安全敏感）
- 1 處 md5 不切片 → data-normalization-service.js L523 ID gen（非安全敏感）
- 2 處 sha256 不切片 → DataNormalizationService.js L165/L188 content ID（非安全敏感，內容快取用）
- 1 處 sha256 + substring → data-normalization-service.js L502 cache key（非安全敏感）

全部均非加密用途（無簽章 / 認證 / 密碼存儲）。改用 sync JS hash（DJB2）即可，輸出格式對齊（hex string）。

```js
// 取代範本：sync DJB2 hash，與原 createHash + digest 'hex' 行為對齊
function djb2Hex (str) {
  let hash = 5381
  for (let i = 0; i < str.length; i++) {
    hash = ((hash << 5) + hash + str.charCodeAt(i)) | 0
  }
  // 轉 unsigned 32-bit 並輸出 8 字元 hex
  return (hash >>> 0).toString(16).padStart(8, '0')
}
// 使用：djb2Hex(str) 取代 crypto.createHash('md5').update(str).digest('hex').substring(0, 8)
// 若原本有 substring(0, 16)：djb2Hex(str + djb2Hex(str)) 拼接擴展（仍非加密強度）
```

注意：DJB2 不是加密 hash，碰撞機率比 md5/sha256 高，但對 cache key 場景足夠。**如測試斷言比較具體 hash 值**，需同步更新測試期望。

**D. helper 提取建議**

`randomHex` 和 `djb2Hex` 可選擇：
- 內聯每處（簡單但有重複）
- 提取到 `src/utils/crypto-shim.js` 模組共用（DRY）

由 agent 視程式碼變動量決定。預設內聯（最小變動）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 替換策略

採用「helper 提取」方案：建立 `src/utils/crypto-shim.js` 統一輸出三個函式，6 檔 caller 改用 shim 取代 `require('crypto')` 與 `crypto.*` API。

| 函式 | 用途 | 輸入 → 輸出 |
|------|------|------------|
| `djb2Hex(str)` | sync DJB2 hash | string → 8 字元 hex |
| `djb2Hex16(str)` | sync DJB2 hash 擴展版 | string → 16 字元 hex（h1 + h2 拼接） |
| `randomHex(n)` | Web Crypto sync 隨機 hex | n bytes → 2*n 字元 hex |

### 6 檔變更明細

| 檔案 | 變更內容 |
|------|---------|
| `src/handlers/extraction-completed-handler.js` | L42 `require('crypto')` → `require('src/utils/crypto-shim')`；L515 `crypto.randomBytes(8).toString('hex')` → `randomHex(8)` |
| `src/background/domains/data-management/services/cache-management-service.js` | L22 require 替換；L558 `crypto.createHash('md5').update(str).digest('hex').substring(0, 8)` → `djb2Hex(str)` |
| `src/background/domains/data-management/services/ValidationCacheManager.js` | L32 require 替換；L963 同上 md5 → `djb2Hex(content)` |
| `src/background/domains/data-management/services/DataNormalizationService.js` | L32 require 替換；L165 sha256 + substring(0,16) → `djb2Hex16(content)`；L188 sha256 full → `djb2Hex16(content)`（caller 只檢查 string / length / 一致性，16 字元足夠） |
| `src/background/domains/data-management/services/data-validation-service.js` | L32 require 替換；L1515 md5 + substring(0,16) → `djb2Hex16(str)` |
| `src/background/domains/data-management/services/data-normalization-service.js` | L22 require 替換；L502 sha256 + substring(0,16) → `djb2Hex16(dataString)`；L519 `crypto.randomBytes(8).toString('hex')` → `randomHex(8)`；L523 md5 full → `djb2Hex16(idString)` |

### 設計決策摘要

- **DJB2 sync hash 取代 md5/sha256**：7 處 createHash 全屬 cache key / ID gen 非加密用途，採 sync 取代 Web Crypto async 的 `subtle.digest`，避免 7 個 caller 改 async / await 蔓延
- **Web Crypto getRandomValues 取代 randomBytes**：2 處 randomBytes 用 `globalThis.crypto.getRandomValues(Uint8Array)` + hex 轉換，仍為 sync 介面
- **helper 提取（非內聯）**：6 檔共用 3 個函式，DRY 優於每處複製貼上，且未來若需切換 hash algorithm 只需改一處
- **16 字元 hex 統一輸出**：DataNormalizationService L188 原 sha256 full 輸出 64 字元，consumer 僅檢查 `string` / `length>5,<50` / `/^[a-z0-9]+$/` 與一致性，改為 16 字元仍滿足所有測試斷言

### 不在本 ticket 範圍

- 進一步加密強度需求（若未來新增）需改用 async `crypto.subtle.digest`，由獨立 ticket 評估 caller async 蔓延影響
- 既有 cache 中的舊 md5/sha256 key 在升級後將失效（破壞性變更）：v1.0.0 內測版 cache 清空為預期行為，無需 migration
