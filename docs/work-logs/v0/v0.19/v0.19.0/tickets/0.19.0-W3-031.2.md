### 5 個 Stop hook 阻擋機制盤點

| Hook | 行數 | sys.exit(2) | decision:"block" | continue:False | 阻擋語意 |
|------|-----|-----------|------------------|----------------|---------|
| evaluate-session.py | 75 | 0 | 0 | 0 | 純 stderr 訊號（不阻擋） |
| session-experience-persistence-reminder-hook.py | 249 | 0 | 0 | 0 | 純 stderr/stdout 訊號（不阻擋） |
| worktree-auto-commit-hook.py | 196 | 0 | 0 | 0 | 純 stderr 訊號 + timeout 15s（不阻擋） |
| handoff-auto-resume-stop-hook.py | 747 | 0 | **2 處** | 0 | **會阻擋**（偵測到未完成 handoff 時） |
| stop-worklog-handoff-sync-check-hook.py | 580 | 0 | 0 | 0 | hookSpecificOutput 訊號（不阻擋） |

**結論**：5 個 Stop hook 中**只有 1 個會實際 block**（handoff-auto-resume），其餘 4 個為純訊號式。單次 Stop event 最大 block 數 = 1（非 ANA 推測的 5）。

### handoff hook 阻擋路徑分析

`.claude/skills/ticket/hooks/handoff-auto-resume-stop-hook.py` L643-666 兩處 `decision: "block"`：
- L644-654：偵測到單一未完成 handoff 任務 → block + 提示用戶 /clear + /ticket resume
- L663-666：多個待恢復任務 → block + 提示選擇

兩條件互斥（單任務 vs 多任務分支）。每次 Stop event 最多觸發其中 1 次。

### 「8 consecutive blocks cap」最可能語意推斷

基於阻擋機制盤點 + Hook 設計慣例（防 hook 無限循環）：

- **block 計算單位**：同一 hook 透過 `decision: "block"` 阻擋主線程結束的「跨 Stop event 累積次數」
- **觸發場景**：用戶反覆觸發 Stop（按 Esc 或 finish）但不處理 handoff，handoff hook 每次都 block → 累積至 8 次後 cap → 第 9 次起 Stop 不再被阻擋
- **單次 Stop event 內**：5 hook 並行執行，最多 1 個產生 block，故單次 Stop 不會直接撞 cap

**注意**：以上為推斷，因官方無公開資料無法 100% 確認。**禁止編造機制細節**（PC-111 防護）。

### 風險評估

| 場景 | 觸發機率 | 衝擊 | 等級 |
|------|---------|------|------|
| 用戶反覆觸發 Stop 8 次而不 resume handoff | 極低（通常 1-2 次後會 resume） | handoff 提示靜默失效 | 低 |
| 跨 session 累積（若 cap 跨 session 持久化） | 極低 | 同上 | 低 |
| 其他 4 hook 升級為 blocking 後撞 cap | 低（目前無此設計） | 取決於新阻擋 hook 行為 | 中（未來風險） |

**結論：風險低，無需立即補救 spawn ticket**。
