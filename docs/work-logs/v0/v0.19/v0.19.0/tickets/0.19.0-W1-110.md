依 W1-107 議題 B 方案 3 落地 GlobalMessages 物理防護：

1. 在 `const GlobalMessages = new MessageDictionary()` 後新增 `Object.freeze(GlobalMessages.messages)`，附 22 行說明註解（設計依據、阻止的四種寫入嘗試、shallow vs deep freeze 選擇、與 local dict 實例的範圍區隔）。
2. 更新 line 63 過時註解：`background ... 仍走 GlobalMessages.addMessages() 注入（W1-110 freeze 規劃時需評估遷移為 local dict）` → `background ... 已遷移為 backgroundMessages local dict（W1-110.1）`。

**`tests/unit/core/messages/MessageDictionary.test.js`**：
1. require 加上 `GlobalMessages` 解構。
2. 檔尾新增 `describe('W1-110: GlobalMessages.messages 物理 freeze 防護')` 含 9 個 test case：
   - `Object.isFrozen(GlobalMessages.messages) === true` 直接驗證 freeze 狀態
   - strict mode 下直接賦值新 key / 覆寫既有 key / delete 既有 key 三種寫入嘗試皆拋 TypeError
   - `GlobalMessages.set` 對新 key 與既有 key 兩種情境皆拋 TypeError
   - `GlobalMessages.addMessages` 走 `Object.assign` 路徑亦被 freeze 阻擋拋 TypeError
   - 既有 key 仍可正常讀取（freeze 只阻寫不阻讀）
   - local dict 實例（`new MessageDictionary({...})`）未受 freeze 影響，可正常 set 寫入

### 設計決策

| 決策點 | 選擇 | 依據 |
|--------|------|------|
| freeze 深度 | shallow `Object.freeze(GlobalMessages.messages)` | messages 結構為 `{ key: string }`，無 nested object，shallow 足夠 |
| freeze 範圍 | 僅 GlobalMessages 全域實例 | local dict 實例仍需支援 set/addMessages 動態註冊（caller 透過建構子注入 messages） |
| 寫入嘗試行為 | TypeError（strict mode）| ES Module source 預設 strict，違規寫入立即拋錯，符合 W1-107 「物理防護」設計目標 |
| 註解定位 | source 內聯 22 行 | 後人讀程式碼即見 freeze 設計依據，無需翻 ticket 歷史 |
