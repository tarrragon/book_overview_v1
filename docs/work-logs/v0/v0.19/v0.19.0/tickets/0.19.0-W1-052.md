## Problem Analysis
<!-- Schema[DOC/Problem Analysis]: 選填（若 DOC 起因於缺陷或盤點結論可填）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

ticket 建立時（2026-05-23）所描述的「hook 要 marker、YAML linter 卻刪 marker」不一致，**在 W1-092（2026-05-27）已被根本性消解**，本 ticket 屬事後釐清。調查發現：

1. **W1-048.3 事件時序**（2026-05-23）：當時 phase4-decision-enforcement-hook 仍會掃描 frontmatter，sage complete 時 hook 命中 `when` 欄位的「Phase 4 評估」歷史字面而阻擋；sage 加 `PC-093-exempt: history` marker 後才 complete 成功。complete 後 ticket CLI 的 metadata sync（PyYAML round-trip 重新序列化 frontmatter）移除了該 marker——因 markdown 註解 `<!-- -->` 在 YAML 中非註解而是 scalar 值的一部分，序列化後位置漂移/移除。

2. **W1-092 修復**（2026-05-27，commit de2f82c3，PC-142 case 5）：phase4-hook 新增 `compute_frontmatter_lines`，將整個 frontmatter 區塊（含起訖 `---`）排除於 phrase 掃描與 marker 蒐集範圍。frontmatter 屬結構化元資料，欄位常含 source ticket history 引用字面，與 Context Bundle auto-extracted（W1-120）、Schema placeholder（W10-130）、fenced code block（W11-018）同類整段跳過。

3. **驗證**：構造測試確認 hook 對 frontmatter 內「Phase 4 評估」「視結果再決定」「之後再說」命中數為 0。當前 W1-048.3 frontmatter 的 `when` 欄位仍含「Phase 4 評估」字面、且**無 marker**，ticket 已正常 complete——反證 frontmatter 不需要 marker。

**結論**：當前一致行為等同方案 A（hook 排除 frontmatter 掃描）已落地。「YAML linter 移除 marker」不再有害，因 frontmatter 本就不需 marker。無需改 hook 程式碼。

### 影響範圍

- `.claude/hooks/phase4-decision-enforcement-hook.py`：已含 `compute_frontmatter_lines`（W1-092），無需修改。
- `.claude/rules/core/decision-trigger-binding.md`：缺 frontmatter 場景說明，本 ticket 補充。
- 無獨立 YAML linter hook；「移除 marker」的機制是 ticket CLI complete 後的 metadata sync（PyYAML frontmatter 序列化），非專門 linter。

### 相關 Error Pattern

PC-142（phase4-hook 誤判家族，case 5 = frontmatter 場景，W1-092 修復）、PC-146（exempt marker 位置誤用）、PC-093（YAGNI 延後累積，本 hook 防護的根模式）。

## Solution
<!-- Schema[DOC/Solution]: 免填（DOC 類型以變更摘要取代）（.claude/pm-rules/ticket-body-schema.md） -->

### 設計決策

採方案 A（phase4 hook 排除 frontmatter 掃描範圍）——但此方案在 W1-092 已落地，本 ticket 無需改 hook 程式碼或測試。理由：

- 方案 A 與既有 W1-092 的 `compute_frontmatter_lines` 設計完全一致，且與其他三類整段豁免區塊（fenced code / Schema placeholder / Context Bundle）同精神，架構連貫。
- 方案 B（要求 YAML linter 保留 marker）反而較差：marker 在 frontmatter 內對 hook 無作用（冗餘），保留它只會誤導後人以為 frontmatter 需要 marker；且 PyYAML 序列化保留行內 markdown 註解需引入 ruamel.yaml 等 round-trip preserving 套件，成本高於收益。

### 變更摘要

| 檔案 | 章節 | 變更 |

| `.claude/rules/core/decision-trigger-binding.md` | 新增「Frontmatter 場景：不需 exempt marker（W1-092 起）」子章節（位於「多命中行情境」與「與其他規則的邊界」之間） | 三明示（Why/Consequence/Action）說明 frontmatter 內字面不需 marker，附「字面所在位置 vs 是否需 marker」對照表與判別準則 |
| 同上 | 版本 footer | 1.3.0 → 1.4.0，Source 追加 W1-092 |

無程式碼變更，無新增/修改測試。
