### Phase 1 規格產出（thyme-extension-engineer, 2026-05-31）

### 1. 技術可行性 spike 結論

執行 5 個 prototype（`scripts/experiments/w4-006-popup-launch-spike{,-v2,-v3,-v4,-v5,-v6,-v7}.js`，主 repo cwd），對 W1-008 既有描述「popup 為獨立分頁致 checkCurrentTab 誤判」進行重新驗證並深度探索三方案。

**重大發現 1：W1-008 ticket description 的根因描述部分不準確**

W1-008 ticket why 寫：「popup 的 checkCurrentTab() 會把 popup 自身判定為 active tab，導致 isReadmoo=false、#extractBtn 保持 disabled」。

實測（spike v1 Probe A + v6 攔截）顯示，popup 在 Puppeteer 環境下：

- popup tab 確實被瀏覽器標為 `active: true`
- 但 popup 內 `chrome.tabs.query({active:true, currentWindow:true})` 從 popup 自身上下文呼叫時，回傳 tab 的 `url` 為 `undefined`（chrome 安全機制：popup 不能讀自己的 url 屬性）
- 因此 `const [tab] = ...; const isReadmoo = tab.url && tab.url.includes(...)` 走 false 分支，pageInfo 顯示「非 Readmoo 頁面」，extractBtn.disabled = true

修正後的根因：「popup 為新分頁時，瀏覽器自動標為 active，且 popup 自身上下文取不到自己的 url，使 isReadmoo 永遠為 false」。

**重大發現 2：popup 在 Puppeteer 載入時有 pageerror（pre-existing latent bug）**

spike v6/v7 在 popup page console 捕捉到三個 pageerror：

```
Identifier 'Logger' has already been declared (x2)
PopupInitializationTracker is not defined
```

根因：`popup.html` 載入順序為 popup-ui-components.js → popup-error-handler.js → popup-diagnostic-enhancer.js → popup-initialization-tracker.js → popup.js。其中 popup-error-handler.js 與 popup.js（bundled）都宣告 `Logger`，造成重複宣告。在真實 Chrome action popup overlay 不會觸發（不同生命週期），但在 Puppeteer `page.goto('chrome-extension://.../popup.html')` 把 popup 當一般 tab 載入時必發。

實際影響：popup.js 的 `const Logger` 行被 SyntaxError 中斷，導致整個 popup.js 不執行 → `setupEventListeners()` 永遠不被呼叫 → `#extractBtn` 的 click 事件監聽器永遠不被綁定。實測：spike v6 攔截 `chrome.tabs.sendMessage` 紀錄中，click 後完全沒有 START_EXTRACTION 呼叫，只有 PING（PING 是別處呼叫，與 click 無關）。

註：spike v7 看到 `typeof window.startExtraction === 'function'` 為 true，是因為 popup.js 後段 `window.startExtraction = startExtraction` 仍可執行（top-level statement），但 `setupEventListeners()` 在 DOMContentLoaded handler `initialize()` 內，整個函式因 `const Logger` 重複宣告未定義，導致 click 不會觸發 startExtraction。

**spike 探索的三條觸發路徑**

| 路徑 | 描述 | 結果 |
|------|------|------|
| 1. SW `chrome.action.openPopup()` | spike v1 Probe B | API 可呼叫且回傳 success，但無法在 Puppeteer headless/visible 環境驗證真實 popup overlay 是否顯示；即便顯示也無法以 Puppeteer Page handle 操作（chrome.action popup 不暴露為 page target） |
| 2. popup context 直接 sendMessage | spike v4 | 完全可行：開 popup 分頁 → popup 內 `chrome.tabs.update(fixtureTab.id, {active:true})` → popup 內 `chrome.tabs.sendMessage(fixtureTab.id, {type:'START_EXTRACTION'})` → storage 寫入 5 本書，包含 v1.1 model 欄位（readingStatus, tagIds） |
| 3. 真實 page.click('#extractBtn') | spike v5/v6/v7 | 受 Logger 重複宣告 pageerror 影響，click event listener 未綁定，click 後無動作。修復該 pageerror 後可預期路徑 3 可行 |

### 2. 三方案利弊分析（spike 結果驅動）

| 方案 | 描述 | 可行性 | 工程成本 | UI button 觸發層覆蓋 | 風險 |
|------|------|--------|----------|------------------|------|
| 方案 1：Puppeteer 真實 popup overlay + click extractBtn | SW 呼叫 `chrome.action.openPopup()` 或 Puppeteer 觸發 action click | 不可行 | 高 | 完整 | Puppeteer Page handle 無法操作 action popup overlay（chrome 限制）；即便 openPopup() 成功，仍無法以 page.click 驅動 |
| 方案 2：integration test 層 mock chrome.tabs API | jsdom + mock 環境覆蓋 checkCurrentTab/extractBtn.disabled/click → startExtraction → chrome.tabs.sendMessage 邏輯 | 完全可行 | 低 | 邏輯層完整，但 SW/content script pipeline 不真實 | 既有 `tests/integration/chrome-extension/popup-interface.test.js` 已部分覆蓋（行 283-350），但未覆蓋 click → startExtraction → 真實提取完整鏈 |
| 方案 3：注入 popup.html 為 page + 真實 Chrome + 修復 Logger 重複宣告 | 開 fixture → 開 popup tab → popup 內 `chrome.tabs.update(fixture.id, {active:true})` 切換 active → 修復 Logger 重複宣告 → click extractBtn | spike 證實 popup→content script→storage 通訊路徑可走（v4），但完整 click 路徑需先修 popup pageerror | 中（含 popup pageerror 修復） | 完整（涵蓋 button 觸發 + UI handler + SW + content script + storage） | popup 為新分頁的 active tab 邏輯與真實 popup overlay 有差異（spike 中需 `chrome.tabs.update` 切回 fixture），測試行為與生產 UX 不完全一致 |

### 3. 推薦實作路徑

**推薦：方案 2 為主 + 方案 3 變體為輔的混合策略**

- **方案 2（主）：擴充既有 integration test**，補測試覆蓋 popup button 觸發 startExtraction 完整邏輯
  - 在 `tests/integration/chrome-extension/popup-interface.test.js` 既有 mock 基礎上補：
    1. `extractBtn` 啟用條件斷言（checkCurrentTab 回傳 readmoo tab 後 extractBtn.disabled = false）
    2. `extractBtn.click()` 觸發 startExtraction，驗證 chrome.tabs.sendMessage 被以 `{type:'START_EXTRACTION'}` 呼叫且 tab.id 正確
    3. mock 回傳 success 後 popup UI 進入「提取中 → 完成」狀態流轉
  - 估時：1-2 hr（含測試撰寫與 CI 驗證）
  - 工程價值：UI button 觸發層的「邏輯正確性」regression 防線；與既有 jest 單測共用 jsdom + chrome mock 基礎設施

- **方案 3 變體（輔）：補 E2E 但不阻塞 main acceptance**
  - 修復 popup-error-handler.js 與 popup.js 的 Logger 重複宣告（建獨立 IMP follow-up）
  - 之後可在 `tests/e2e/browser/` 補測試覆蓋真實 click（同時驗證 popup pageerror 不存在的 regression）
  - 估時：先派 ANA 評估 Logger 重複宣告修法影響範圍，再決定 IMP scope（可能涉及 ESM/CJS 載入策略調整）

**選擇理由**

1. **方案 1 物理不可行**：spike v1 Probe B + chrome 官方文件確認 Puppeteer 無法操作 action popup overlay。排除。
2. **方案 2 是 cost/benefit 最佳的快速覆蓋路徑**：既有測試已部分覆蓋（行 283-350），擴充成本低，覆蓋邏輯層完整。
3. **方案 3 真實 E2E 受 popup pageerror 阻擋**：在不修 popup 結構性 bug 的前提下不可行。修 Logger 重複宣告超出 W4-006 的「補 E2E 覆蓋」範圍，是獨立的 IMP；強行納入會讓 W4-006 工程成本飆升且風險不可控。
4. **方案 3 變體（修 pageerror 後再補 E2E）保留為 follow-up**：可作為 W4-006 後續延伸 ticket，但 ticket acceptance 不依賴此項。
