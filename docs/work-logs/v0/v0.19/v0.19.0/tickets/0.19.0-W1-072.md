### 問題根因

`package.json` 與 `manifest.json` 仍為 v0.18.0，但專案已進入 v0.19.0 wave（10 ticket complete，main worklog 標 v0.19.0）。`build.js:190` 從 `package.json` 注入 `manifest.version`，`package.js` 從 build 後 manifest 讀版號命名 ZIP，導致：
- dist ZIP 命名 `readmoo-book-extractor-v0.18.0.zip`
- 載入 unpacked / unzip 後 manifest 顯示 v0.18.0
- 內測使用者看到舊版本號無法對應 v0.19.0 內測 milestone

根因為 version bump 時機策略未明示：
- `version-release skill` 設計流程是「發布前在 feature/v{VERSION} 分支 bump」（前置條件章節），屬於發布閘門前 bump
- 但 v0.19.0 內測 milestone 在「發布」前就需要打包 ZIP 分發（W1-002 group），形成時機落差

### 影響範圍

- `scripts/package.js` (ZIP 命名邏輯)
- `.claude/skills/version-release/SKILL.md` (bump 時機策略)

## 重現實驗結果
### 實驗方法

直接讀檔案驗證版號狀態與 build 注入邏輯，比對 W1-002.2 觀察到的 ZIP 命名現象。實驗指令：


### 實驗執行

1. **package.json 版號**：`grep -E '"version"' package.json` → `"version": "0.18.0"` (line 3)
2. **manifest.json 版號**：`grep -E '"version"' manifest.json` → `"version": "0.18.0"` (line 4)
3. **build.js 注入邏輯**：`scripts/build.js:190` `manifest.version = packageJson.version`；line 191 console.log `[VERSION] 版本注入: ${packageJson.version} (來源: package.json)`
4. **package.js ZIP 命名**：`scripts/package.js:146-153` `readVersion()` 讀 build/production/manifest.json → ZIP 命名 `readmoo-book-extractor-v${version}.zip`
5. **W1-002.2 觀察證據**：ticket why 欄位明示「dist ZIP 命名為 readmoo-book-extractor-v0.18.0.zip 且 manifest 顯示 v0.18.0」
6. **v0.18.0 vs v0.19.0 狀態**：CHANGELOG.md line 5 「## [v0.18.0] - 2026-05-20」（已發布）；v0.19.0 main worklog line 5 `**版本號**: v0.19.0`，line 40-61 已 complete 10+ ticket

### 實驗發現

**已驗證的事實**：
1. `package.json` 是 version SSOT，`build.js` 自動同步到 `manifest.json`，`package.js` 從 build manifest 讀版號命名 ZIP——整條 pipeline 一致
2. 當前 SSOT (package.json) 為 0.18.0，但專案實際進入 v0.19.0 開發 wave（CHANGELOG 已標 v0.18.0 發布，v0.19.0 main worklog 進行中）
3. version-release skill 設計時機（feature/v{VERSION} 分支發布前 bump）與 v0.19.0 內測場景（wave 中需打包分發）有時機落差，此落差未被既有規範覆蓋
4. W1-002.2 觸發案例為「內測使用者載入會看到舊版本號」，屬使用者體驗問題，非功能 bug

**仍未驗證的假設**：
- 內測使用者實際對舊版本號的混淆程度（無使用者反饋資料）→ 不影響本 ANA 結論，bump 收益僅次於正確性
- 是否有其他 Chrome Extension 專案採「wave 內 bump」vs「發布前 bump」的最佳實踐統計 → 不在本 ANA 範圍

**結論定性**：版號 bump 時機策略缺失，需在「立即 bump」「保持現狀」「內測前 bump」三方向中明確決策並落地。

### 三方向 Trade-off

| 維度 | A. 保持現狀 | B. 立即 bump 至 0.19.0 | C. 內測前 bump（推薦） |
|------|------------|-----------------------|---------------------|
| 操作 | 不動，發布前才 bump | 現在執行 version-release start --version 0.19.0 | wave 內 dev 階段不動，內測 ZIP 正式產出前 bump |
| 內測 UX | 內測 ZIP 顯示 v0.18.0，與 wave 不一致 | 內測 ZIP 顯示 v0.19.0，與 wave 一致 | 內測 ZIP 顯示 v0.19.0，與 wave 一致 |
| 符合 version-release skill 原意 | 是（發布前 bump） | 否（提早於發布前 bump） | 是（內測 ZIP 即 release artifact，仍屬發布閘門前 bump） |
| 開發期 noise | 無 | git diff 出現 package.json/manifest.json 變動 baseline | 無（dev 期維持） |
| 與 check 一致性驗證 | 維持一致 | 維持一致 | 維持一致 |
| 後續處理成本 | 發布時 bump 一次 | wave 內後續 commit 以新版號為 baseline | bump 後 dev 可繼續，無額外管理成本 |
| 風險點 | 內測使用者反饋疑慮 | 違反 skill 設計原意，無 sanity check 仍可能跳號 | 需建立「內測 ZIP 產出前 bump」流程 checkpoint |

### 推薦方向：C（內測前 bump）

**核心理由**：
1. **解決 W1-002.2 觀察的 UX 問題**：內測使用者載入 ZIP 看到 v0.19.0 與 wave 對齊
2. **不違反 skill 設計原意**：v0.19.0 milestone 本身是「內測版本」（CLAUDE.md §8），內測 ZIP 即等同 release artifact，bump 仍在「打包前」事件邊界內
3. **dev 期影響最小**：W1-001/W1-009 等 model gap 修復 commit 不需混入版號變動
4. **可借鑑 IMP-068 sanity check**：build:prod 時 assert 版號合理性，避免靜默漂移

### Dissenting View（規則 5 §5.6 反諂媚必填）

方向 A 支持者觀點：避免 wave 中途版號混亂；發布前才 bump 是業界 release management 常見做法。

**反駁**：v0.19.0 milestone 本身定義為「內測版本」，內測 ZIP 即等同 release artifact；W1-002 group 已實質執行「內測前打包驗證」流程，bump 觸發點移到 W1-002.3 安裝指南撰寫前（或 W1-003 跨裝置同步驗證前）並未違反「發布前 bump」原則，僅是將「發布」邊界從 Chrome Web Store 上架擴展到內測 ZIP 分發。
