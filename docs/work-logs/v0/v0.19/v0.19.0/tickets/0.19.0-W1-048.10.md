## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-048 Phase 4 三方視角共識：底線私有方法（`_validateFileBasics` / `_validateFileSize` / `_readFileWithReader` / `_handleFileContent`）原本被 `overview-page-controller` 跨檔呼叫造成封裝洩漏。W1-048.1 已修補 controller 端（Stage A：建立 public API、Stage C：測試遷移、Stage C.7：移除 controller proxy）。

當前殘留問題：importer 內部 4 個底線方法仍為實作骨架（不可單純移除），但 8 處 jest mock 直接賦值 `importer._method = jest.fn()` 延續「測試耦合到實作」反模式。

### 影響範圍

- **src 層**（不可變更實作結構，僅評估）：`src/overview/book-file-importer.js`（4 個底線方法 + 3 個 public API 為 thin orchestrator）
- **test 層**（潛在改寫對象）：
  - `tests/unit/overview/import-mode-modal.test.js`：5 處直接 mock（TC-C5 / TC-C5+ 系列）
  - `tests/unit/overview/overview-page-controller.test.js`：3 處直接 mock（TC-12 / TC-15 / TC-16 系列）
- **守門測試**（保留）：`tests/unit/overview/overview-import-private-methods.test.js:772-775` 反向斷言 controller side 無 proxy

### 相關 Error Pattern

- PC-055（AC 漂移偵測）：本 ticket pending 1 天，AC 未漂移；when 條件「W1-048.1 Stage C 完成」已於 2026-05-23 達成（前置 ticket completed 23h58m 前）
- W1-048 Phase 4 多視角審查 TD1+TD2：「測試耦合到實作」反模式（本 ticket 為其延伸 follow-up）

### 決策方向（結論）

**推薦方案 C（保留底線 + 改寫 mock）**，但需 PoC 驗證可行性後再全面落地。

**理由**：

1. **不破壞既有實作結構**：4 個底線方法為 importer 內部 SRP 切分（validation / IO / content parsing 三責分離），public API 為 thin orchestrator，符合 W1-048 Phase 4 多視角共識「保留作為內部實作細節」
2. **解決核心反模式**：消除 8 處「測試耦合到 implementation detail」（W1-048 Phase 4 TD1+TD2 延伸）
3. **變更面可控**：僅 2 個測試檔（`import-mode-modal.test.js` + `overview-page-controller.test.js`），8 處 mock 點
4. **退路明確**：若 PoC 失敗則退回方案 A，本評估 ticket 不會卡關

### 用戶最終決策（覆蓋原推薦）

**採用方案 D：拆分 FileValidator / FileReader / ContentParser 三個獨立 helper class**

**用戶覆蓋理由**：
- 4 個底線方法可清楚對應 3 個獨立職責邊界（validation x2 / IO x1 / parsing x1）
- 顯性化 importer 內部 SRP 切分，從 private method 提升為獨立 class
- 接受變更面擴大，換取強封裝（function reference 鎖定 + DI 注入點）

**方案 D 落地架構**：

| 新檔案 | 職責 | 封裝原底線方法 | 邊界 |
|--------|------|--------------|------|
| `src/overview/import/file-validator.js` | 業務規則：file 格式 / 大小驗證 | `_validateFileBasics` + `_validateFileSize` | 純函式驗證，無 IO |
| `src/overview/import/file-reader.js` | IO 邊界：FileReader API 封裝 | `_readFileWithReader` | 包裹 FileReader API（測試可 mock 此 class） |
| `src/overview/import/content-parser.js` | 純函式邊界：content 解析 | `_handleFileContent` | 純函式，無 IO 無 state |
| `src/overview/book-file-importer.js`（改寫） | Orchestrator：協調 validator / reader / parser | 移除 4 個底線方法 | 透過 constructor DI 注入 helper instance |

**DI 設計**：
- `BookFileImporter` constructor 接受 `{ validator, reader, parser }` 選用參數
- 預設值由 importer 自建（生產情境零變更）
- 測試時可注入 stub helper 取代 `importer._method = jest.fn()` 模式

**測試端影響**：
- 8 處原 mock 改寫為注入 stub helper（或保留 importer constructor 等效行為）
- 新增 helper class 單元測試（覆蓋 validator / reader / parser 各自邏輯）
- `overview-import-private-methods.test.js` 反向斷言需更新（檢查 importer 不再有 `_handleFileContent` 等底線方法）

**接受面**：

- 對齊用戶「強封裝」優先於「最小變更」的取捨
- 機會成本：W1-001.3 P0 仍 pending，但用戶顯式選擇先落地方案 D
- 退路：若 PoC 階段發現 helper 拆分破壞核心測試意圖，回退至方案 C 或方案 A
