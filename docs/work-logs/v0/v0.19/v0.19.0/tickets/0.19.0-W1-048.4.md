## Task Summary

釐清 IMP-A 補回 — cover 型別契約矛盾（required 寬鬆 vs typed 嚴格）並落地

---

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

`_validateRequiredFields` 與 `_validateFieldTypes` 對 cover 欄位的契約矛盾：

| 函式 | 行為 | 行號 |
|------|------|------|
| `_validateRequiredFields` | 僅驗證 `id` 與 `title`，註解明示「cover 為選填」 | L740-746 |
| `_validateFieldTypes` | 要求 `typeof book.cover === 'string'`，cover 為 `undefined` 時回傳 `false` | L753-756 |

**矛盾效果**：宣稱 cover 選填，但缺 cover 的書籍會被 `_validateFieldTypes` 過濾——selectively-loose validation 反模式。

### 影響範圍

- `src/overview/book-file-importer.js`：`_isValidBook`（L724-725）串接兩個驗證函式
- UC-04 匯入流程：缺 cover 的合法 v2 JSON 書籍會被靜默丟棄（不報錯，只是 filter 過濾）

### 相關 Error Pattern

無既有 error pattern。本矛盾屬 W1-047.1 修復時引入的局部不一致（修了 required 但漏改 types）。

## 重現實驗結果
### 實驗方法

靜態程式碼分析：讀取 `_validateRequiredFields` 與 `_validateFieldTypes` 原始碼，對照 SPEC-EXPORT-V2 cover 欄位定義。

### 實驗執行

1. `_validateRequiredFields`（L744-746）：`return Boolean(book.id) && Boolean(book.title)` — 不檢查 cover
2. `_validateFieldTypes`（L753-756）：`typeof book.cover === 'string'` — cover 為 undefined 時回傳 false
3. SPEC-EXPORT-V2（L209）：`cover | string | 否 | "" | 封面圖片 URL` — 明確標示**非必填**，預設空字串
4. `data-management.md`（L201）：`cover: { type: 'string', required: false, default: '' }` — 再次確認非必填

### 實驗發現

**已驗證事實**：
- SPEC-EXPORT-V2 與 data-management 規格一致：cover 為選填，預設 `""`
- `_validateRequiredFields` 與規格一致（不要求 cover）
- `_validateFieldTypes` 與規格矛盾（要求 cover 為 string，undefined 不通過）
- W1-047.1 註解（L740-742）已意識到此設計但認為「v1 轉換產出 cover 為 ''，型別檢查通過」——**僅考慮 v1 轉換路徑，未考慮外部匯入檔案可能缺 cover 的情境**

**結論**：矛盾確認。修復方向為「真寬鬆」——`_validateFieldTypes` 應容許 cover 為 undefined。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 分析

**W（Widen）**：三個候選方案

| 方案 | 描述 | 優劣 |
|------|------|------|
| A: 真寬鬆 | `_validateFieldTypes` 改為 `typeof book.cover === 'string' \|\| book.cover === undefined` | 與規格一致；最小改動 |
| B: 真嚴格 | `_validateRequiredFields` 加回 cover required | 違反 SPEC-EXPORT-V2 §3.5 明文「否」；不採 |
| C: 正規化層補 default | 在驗證前統一補 `cover: ''`（若 undefined） | 改動更大，但也解決問題 |

**R（Reality-test）**：方案 A 改 1 行、方案 C 需新增正規化邏輯且可能影響其他欄位策略。方案 B 直接違反規格。

**A（Attain distance）**：方案 A 最小改動、與規格完全對齊、不影響既有 v1 轉換路徑（v1 轉換已產出 `cover: ''`，string 型別檢查仍通過）。

**P（Prepare to be wrong）**：若未來有更多選填欄位加入 `_validateFieldTypes`，同樣需要 `|| === undefined` 處理。可在 IMP ticket 中一併檢查其他欄位是否有同類問題。

### 設計決策

**採方案 A：真寬鬆**。

修改 `_validateFieldTypes`（L756）：

```javascript
// 現行
typeof book.cover === 'string'
// 改為
(typeof book.cover === 'string' || book.cover === undefined)
```

理由：SPEC-EXPORT-V2 與 data-management 規格皆定義 cover 為選填（`required: false, default: ''`）。驗證層應與規格一致，允許 cover 缺失。
