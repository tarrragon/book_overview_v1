## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-047.1 (IMP-A) 接線 BookFileImporter 版本偵測閘門時，在 `_extractBooksFromData` (src/overview/book-file-importer.js 行 607) 加入 `fileFormat === 'json'` 條件，使 CSV 路徑刻意 bypass 版本偵測。

設計理由（行 580-599 註解）：
- `detectFormatVersion` 對純陣列一律判為 'v1'，但 CSV 解析產出的書籍物件結構（status / tags）與 v1 JSON 不同，不應走 v1→v2 轉換
- CSV 自有 `detectCsvFormatVersion` 偵測機制
- CSV 解析結果（陣列）應落入 v2 路徑的 `_isDirectArrayFormat` 分支，tag 區段經 `_extractTagSection` 對非物件安全回傳 []

風險：此設計決策僅在註解說明，無 regression test 鎖定。若未來重構誤刪 `fileFormat === 'json'` 條件，CSV 會被 `detectFormatVersion` 判為 'v1' → 進入 `convertV1ToV2Data` → 資料結構錯位（status / tags 欄位不對齊）。靜默回歸，現有測試無法捕捉。

W1-048 Phase 4 評估 F11 識別此缺口為 IMP-A 跳過 Phase 4 的補回項，列為 P1 立即執行（Worth-It Filter）。

### 影響範圍

- **設計位置**: `src/overview/book-file-importer.js` 行 605-650 `_extractBooksFromData`（fileFormat 分支邏輯）
- **註解錨點**: `src/overview/book-file-importer.js` 行 580-599（CSV bypass 版本偵測的設計理由）
- **測試新增位置**: `tests/unit/overview/book-file-importer-version-detection.test.js`（既有 regression test 檔，補 CSV bypass case）

### 相關 Error Pattern

### 測試策略（sage 落地時細化）

- 在現有 `book-file-importer-version-detection.test.js` 內新增 describe block：「CSV 路徑不走版本偵測閘門」
- mock `detectFormatVersion` 與 `convertV1ToV2Data`（spy 觀察呼叫次數）
- 案例 1：CSV input 進入 `_extractBooksFromData('csv')` → 斷言 `detectFormatVersion` 未被呼叫、`convertV1ToV2Data` 未被呼叫
- 案例 2：JSON v1 input 對照組 → 斷言兩 spy 都被呼叫（既有路徑不回歸）
- 鎖定行為：未來移除 `fileFormat === 'json'` 條件時，案例 1 必失敗

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 實作摘要

於 `tests/unit/overview/book-file-importer-version-detection.test.js` 新增 Group F「CSV 路徑不走版本偵測閘門」describe，補 3 個 TC：

| TC | 目的 | 核心斷言 |
|----|------|---------|
| TC-18 | CSV 路徑 bypass 版本偵測（核心鎖定） | `detectSpy.not.toHaveBeenCalled()` + `convertSpy.not.toHaveBeenCalled()` |
| TC-19 | JSON v1 對照組（避免過度收斂） | `detectSpy.toHaveBeenCalledTimes(1)` + `convertSpy.toHaveBeenCalledTimes(1)` |
| TC-20 | 預設值（未傳 fileFormat）仍走 JSON 路徑 | `detectSpy.toHaveBeenCalledTimes(1)` |

### 設計決策

- **Spy 注入機制**：採 `jest.isolateModules` + `jest.doMock` 雙路徑 mock（src/ alias + 相對路徑），原因為 `BookFileImporter` 模組頂層以解構 require 持有 `detectFormatVersion` / `convertV1ToV2Data` 的閉包引用，`jest.spyOn(module, 'fn')` 無法攔截，必須在 require 前以 doMock 替換模組導出
- **不污染其他 TC**：spy 限定於 Group F 內部 `loadImporterWithSpies` helper，透過 isolateModules 隔離；既有 17 個 TC 維持原 sociable test 設計（不 mock 內層）
- **鎖定行為**：若未來重構誤刪 `src/overview/book-file-importer.js:607` 的 `fileFormat === 'json'` 條件，TC-18 必失敗（CSV 會觸發 detectSpy）

### 不在本 ticket 範圍

- 不改 `src/overview/book-file-importer.js`（純測試補回）
- 不調整既有 17 個 TC 結構
