## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W1-018 ANA 盤點發現 `npm test` 範圍內 5 個測試檔含計時硬門檻與高精度浮點脆弱斷言。Jest jsdom 環境非有效效能量測環境，計時受全套件機器負載、GC、JIT 暖機影響，在完整 `npm test` 下 flaky；`toBeCloseTo` 5 位小數精度在 IEEE 754 下對浮點平均值脆弱。設計原則（W1-017 用戶確認）：計時不應作為測試驗收條件，效能差是設計問題而非功能問題。

### 影響範圍

5 個測試檔，依 W1-018 Solution 三組處理：

- A1：`tests/integration/performance/performance-monitoring-integration.test.js`（3 個計時斷言）
- A2：`tests/unit/handlers/extraction-completed-handler.test.js`（1 個計時斷言）
- A3：`tests/unit/extractors/readmoo-data-validator.test.js`（2 個：1 絕對計時門檻 + 1 相對計時比較）
- B1：`tests/integration/workflows/cross-device-sync-workflow.test.js`（1 個 toBeCloseTo 5 位精度）
- B2：`tests/unit/background/domains/platform/services/platform-detection-service.test.js`（2 個 toBeCloseTo 5 位精度）

### 相關 Error Pattern

無既有 error-pattern 精確匹配。延續 W1-017 / W1-018 的測試設計問題（計時硬門檻當驗收 + 浮點高精度脆弱），W1-018 Solution 已建議升級為測試設計規則（屬後續 DOC 範圍，非本 IMP）。

## Solution

### 變更摘要

依 W1-018 Solution 三組處理 5 個測試檔的脆弱斷言。

| 組別 | 檔案 | 變更 |
|------|------|------|
| A1 | `tests/perf/performance-monitoring-integration.test.js`（由 `tests/integration/performance/` 移入） | `git mv` 整檔搬至 `tests/perf/`（與 W1-017 方案 A 物理拆檔一致）。`npm test`（`jest tests/unit tests/integration`）路徑 regex 不含 `tests/perf` 子字串，故自動排除；`npm run test:perf`（`jest tests/perf`）涵蓋。檔頭補註 W1-019 移檔理由與量測環境限制。3 個計時硬門檻斷言（avgCreationTime / avgTimePerError < 0.5ms、scalabilityRatio < 2）不再進 npm test |
| A2 | `tests/unit/handlers/extraction-completed-handler.test.js` | 行 724 `processingTime < 1000ms` 計時硬門檻移除。改為驗證 1000 本書完整且正確處理：`result.success === true`、`result.processedBooks === 1000`、`completionStats.totalCompletions === 1`、`successfulSaves === 1`。test 名稱由「保持效能」改為「正確處理大量資料的完成事件」 |
| A3 | `tests/unit/extractors/readmoo-data-validator.test.js` | 行 485 `processingTime < 1000ms` 計時門檻移除，保留 `result.validBooks === 100` 功能驗證。行 641 `secondRunTime < firstRunTime * 0.8` 相對計時比較改為 `getCacheHitRate() > 0`——直接驗證快取建立並命中，而非以計時間接推論。`getCacheHitRate()` 在 `getCacheSize() > 0` 時回傳 0.8，故兩次驗證後快取非空即 > 0 |
| B1 | `tests/integration/workflows/cross-device-sync-workflow.test.js` | 行 145 `toBeCloseTo(scenario.averageProgress, 5)` 精度由 5 降為 2（W1-017 已實證此行 intermittent flaky） |
| B2 | `tests/unit/background/domains/platform/services/platform-detection-service.test.js` | 行 518 `toBeCloseTo(0.9, 5)`、行 983 `toBeCloseTo(1.0, 5)` 精度由 5 降為 2，與同檔其他 toBeCloseTo 斷言一致 |

### 設計決策

- A2/A3 計時斷言移除後並非「刪除測試」：每處皆改為等價或更強的功能正確性驗證（A2 從「處理夠快」升級為「處理結果正確且完整」；A3 行 641 從「以計時推論快取生效」升級為「直接驗證快取命中」）。符合 W1-017 三類處理原則「移除計時門檻須補功能驗證」。
- A1 採物理拆檔而非移除斷言：整檔位於 `tests/integration/performance/` 子目錄，定位即效能測試，移至 `tests/perf/` 與 W1-017 event-system 處理一致，計時斷言保留作大幅退化防護但不汙染 npm test。
- `moduleNameMapper` 的 `^src/(.*)$` 為 `<rootDir>` 相對，移檔不影響 `require('src/...')` 解析。

### 不在本 ticket 範圍

- 升級為測試設計規則（W1-018 Solution 建議的 DOC 產出）：屬後續 DOC ticket，非本 IMP。
- A4 `content-script-injection.test.js` 的 10 個 mock 計時斷言：W1-018 已判定為低風險保留（mock 固定回傳值，不 flaky），無需處理。
