### W3-018 ANA 修復 2 設計策略

#### 現有邏輯

`handoff_gc.py:37-76 _collect_stale_handoffs()` 已 delegate 到 `handoff_utils.is_handoff_stale(record.data)`（W17-163 L1-A）。對 task-chain (to-sibling/to-parent/to-child) handoff，需目標 ticket 已 in_progress 才標 stale；對 context-refresh 等非 task-chain，來源 ticket completed 即標 stale。

#### --force 行為設計

- `force=False`（預設）：保持現有行為（delegate to is_handoff_stale）
- `force=True`：跳過 is_handoff_stale 改用 `is_ticket_completed(ticket_id)` 統一判定，task-chain handoff 不再受保護

#### 實作要點

| 檔案 | 修改 |
|------|------|
| `handoff_gc.py:_collect_stale_handoffs` | 新增 `force: bool = False` 參數；force=True 時走新 branch（is_ticket_completed），reason 含 「(--force 模式跳過 task-chain 保護)」字面 |
| `handoff_gc.py:execute_gc` | 新增 `force: bool = False` 參數，傳遞到 `_collect_stale_handoffs(force=force)` |
| `handoff.py` CLI register（gc 子命令） | 新增 `--force` argparse argument，預設 False；傳遞到 `execute_gc(force=args.force)` |
| Markdown 格式 handoff | 不受 force 影響（原本就用 completed 判定） |

#### 測試覆蓋要求

新增 pytest 測試（位置：`.claude/skills/ticket/tests/`）：

1. `test_gc_force_clears_task_chain`：含 task-chain handoff（to-sibling），來源 ticket completed，force=True 時應被標 stale
2. `test_gc_no_force_preserves_task_chain`：相同情境 force=False 時不標 stale（向後相容驗證）
3. `test_gc_force_no_op_when_source_not_completed`：force=True 但來源 ticket 仍 pending/in_progress 時不標 stale（force 不打破「來源 completed」前提）
4. CLI 端對端：`ticket handoff gc --force --dry-run` 輸出含 task-chain handoff

### 設計決策

- **複用既有 utility（PC-068）**：`is_ticket_completed` 已存在於 `handoff_utils`，直接 import；不重新實作判定邏輯，避免 ARCH-020 漂移。
- **預設行為不變（向後相容）**：`force=False` 走原 `is_handoff_stale` 路徑，既有 19 個測試全綠驗證。
- **Markdown 格式豁免**：Markdown handoff 無 direction 資訊，原本即用 `is_ticket_completed` 判定，故 `--force` 對其無實質差異，邏輯路徑共用無需分支。
