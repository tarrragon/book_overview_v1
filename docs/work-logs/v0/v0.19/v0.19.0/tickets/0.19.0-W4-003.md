## Solution
### 變更摘要

修復 `tests/integration/cross-module/event-system-integration.test.js` 第 799-802 行 `averageCpuUsage` 的絕對門檻斷言。

**修改前**：
- `expect(performanceAnalysis.averageCpuUsage).toBeLessThan(0.7)` — 在 `if (averageCpuUsage !== undefined)` 守衛內，對 CPU 使用率使用 0.7 絕對門檻當 pass-fail 斷言。

**修改後**：
- 移除 `toBeLessThan(0.7)` 絕對門檻。
- 保留 `if (performanceAnalysis.averageCpuUsage !== undefined)` 守衛（基於真實測量，該值可能為 undefined）。
- 改為指標有效性驗證：`toBe('number')` + `toBeGreaterThanOrEqual(0)`。
- 補上比照 W1-031 的說明註解，標明 W4-003 修復出處、根因（環境依賴非確定性）、規則依據（test-assertion-design-rules 規則 1）。

### 設計決策

- **比照 W1-031 範本**：同檔第 792-798 行 W1-031 對 `peakMemoryUsage` 的修法（移除絕對門檻、保留 `typeof number` + 正數驗證）是同性質先例，本修復直接沿用一致風格。
- **CPU 使用率與記憶體門檻同屬規則 1 反模式**：`averageCpuUsage` 由執行期環境取樣，受全套件並行測試的機器負載、GC、JIT 暖機影響，是非確定性數值。絕對門檻 `0.7` 在完整 `npm test` 負載下可能超標而 flaky，屬 test-assertion-design-rules 規則 1 明文禁止的「絕對資源門檻當 pass-fail 斷言」。
- **CPU 用 `toBeGreaterThanOrEqual(0)` 而非 `toBeGreaterThan(0)`**：與 W1-031 記憶體（必為正數）不同，CPU 使用率合法值可為 0（空閒取樣），故用 `>= 0` 避免引入新的零值 flaky 風險。
