### 問題根因

`handleFileLoad`（overview-page-controller.js:1050）在 IMP-C 後硬編碼只走 `replaceAllData`（覆蓋模式）。SPEC-EXPORT-V2 §8 定義覆蓋/合併兩種載入模式皆為正式需求，IMP-D 已完成 `mergeAllData`，但匯入端缺少讓使用者選擇模式的決策入口，合併模式無法被觸發。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `src/overview/overview-page-controller.js` | `handleFileLoad` 加 `promptImportMode` 呼叫 + 依選擇分流；新增 modal 控制方法 |
| `src/overview/overview.html` | 新增模式選擇 modal DOM 容器（預設隱藏） |

### 相關 Error Pattern

無直接對應 PC error pattern。屬功能缺口（adapter API 已建但缺 UI 入口），非流程或實作錯誤。

### TDD Phase 1 產出

功能規格已產出於 `0.19.0-W1-047.5-feature-spec.md`（/spec Full 模式，6 區段）。
- /spec validate 第 1 輪 4 問已決議（Q1 範圍外、Q2/Q4 補入規格、Q3 接受風險），第 2 輪無新問題。
- 8 組 GWT 場景、原 3 條 acceptance 細化為 17 條可測試驗收條件。
- 範圍邊界：限模式選擇 modal + handleFileLoad 分流；`replaceAllData`/`mergeAllData` 已存在不在範圍。

### 二、行為鏈推演（前置條件分析）

handleFileLoad 行為鏈：`觸發匯入 → 檔案基本驗證 → promptImportMode（modal 彈出）→ 使用者選擇 → 分流寫入 → UI 更新`

每個 When 的前置條件斷言：
- promptImportMode 前置：modal DOM 容器存在（overlay/modal/三按鈕）、modal 初始隱藏
- 使用者選擇前置：modal 已顯示（visible）、按鈕可點擊、初始焦點已設定
- 分流寫入前置：promptImportMode 已 resolve 非 null、importResult 三欄位為陣列
- UI 更新前置：writeResult.success === true

### 三、測試案例清單

#### Group A：promptImportMode modal 基礎行為（場景 1/2 modal 端）

- TC-A1 promptImportMode 呼叫後 modal 由隱藏轉為顯示（前置：modal 初始 display:none → 斷言顯示）
- TC-A2 modal 顯示時三按鈕（覆蓋/合併/取消）皆存在且可見且未 disabled
- TC-A3 modal 顯示說明文字含「覆蓋」「清空」與「合併」「保留」語意關鍵字
- TC-A4 點「覆蓋」按鈕 → promptImportMode Promise resolve 為 `'overwrite'`，modal 關閉（display:none）
- TC-A5 點「合併」按鈕 → Promise resolve 為 `'merge'`，modal 關閉
- TC-A6 resolve 前 modal 持續顯示（點擊前 await 未完成、modal 仍 visible）

#### Group B：取消途徑（場景 3/4/5）

- TC-B1 點「取消」按鈕 → Promise resolve `null`，modal 關閉
- TC-B2 按 Esc 鍵（modal 取得焦點）→ Promise resolve `null`，modal 關閉
- TC-B3 點遮罩層（importModeOverlay）→ Promise resolve `null`，modal 關閉
- TC-B4 點 modal 容器內部（非遮罩）不觸發取消（事件不冒泡誤判，Promise 未 resolve）

#### Group C：handleFileLoad 分流（場景 1/2/3 流程端）

- TC-C1 選覆蓋 → handleFileLoad 呼叫 `replaceAllData` 一次，不呼叫 `mergeAllData`；成功後 _updateUIWithBooks 收 importResult.books
- TC-C2 選合併 → handleFileLoad 呼叫 `mergeAllData` 一次，不呼叫 `replaceAllData`；成功後 _updateUIWithBooks 收合併後 books
- TC-C3 取消（promptImportMode resolve null）→ handleFileLoad 提前 return：不呼叫 _readFileWithReader、不呼叫任何 *AllData、不呼叫 showLoading、currentBooks 維持原狀
- TC-C4 modal 在檔案基本驗證之後彈出：_validateFileBasics/_validateFileSize 失敗時 promptImportMode 不被呼叫（驗證先於 modal）
- TC-C5 modal 在讀檔之前彈出：promptImportMode 的 resolve 早於 _readFileWithReader 被呼叫（順序斷言，用 mock call order）

#### Group D：合併模式錯誤處理（場景 6，與覆蓋對稱）

- TC-D1 mergeAllData 回傳 `{success:false, error:'quota_exceeded'}` → showError 訊息含「儲存空間不足」，不更新 UI，currentBooks 維持原狀
- TC-D2 mergeAllData 回傳 `{success:false, error:'storage_error'}` → showError 訊息含「儲存失敗」，不更新 UI
- TC-D3 mergeAllData 回傳 `{success:true}` → _updateUIWithBooks 被呼叫一次

#### Group E：無障礙與鍵盤導覽（場景 7）

- TC-E1 modal 容器具 `role="dialog"`、`aria-modal="true"`、`aria-labelledby` 指向標題 id
- TC-E2 modal 開啟時初始焦點落於預設選項按鈕（document.activeElement === 預設按鈕）
- TC-E3 focus trap：焦點在最後一個按鈕按 Tab → 回到第一個按鈕（document.activeElement 斷言）
- TC-E4 focus trap：焦點在第一個按鈕按 Shift+Tab → 跳到最後一個按鈕
- TC-E5 modal 開啟期間 Tab 不跑到 modal 外頁面元素（焦點仍在三按鈕集合內）

#### Group F：單一 modal 實例（場景 8）

- TC-F1 modal 開啟中再次呼叫 promptImportMode（或再次觸發匯入）→ DOM 中 modal 仍只有一個實例（不新增第二個 modal node）
- TC-F2 重複觸發後第一個 modal 的 pending Promise 仍有效，做出選擇後正常 resolve

#### Group G：modal DOM 缺失防禦（Error Handling 表第 6 列）

- TC-G1 modal 元素缺失（DOM 未注入）→ promptImportMode 記錄 console.error，handleFileLoad 中止並 showError 訊息含「匯入功能初始化失敗」，不呼叫任何 *AllData

### 四、預期斷言重點

| 類型 | 斷言方式 |
|------|---------|
| modal 顯示/隱藏 | `element.style.display` 或 class 切換（visible/hidden） |
| Promise resolve 值 | `await promptImportMode()` 結果 `toBe('overwrite'|'merge'|null)` |
| 分流正確性 | `replaceAllDataMock` / `mergeAllDataMock` 的 `toHaveBeenCalledTimes` + `not.toHaveBeenCalled` 成對 |
| 呼叫順序 | `mock.invocationCallOrder` 比較 promptImportMode vs _readFileWithReader |
| focus trap | `document.activeElement` 比對；keydown 事件用 `new KeyboardEvent('keydown',{key:'Tab',...})` |
| 無障礙屬性 | `getAttribute('role'|'aria-modal'|'aria-labelledby')` |
| 單一實例 | `document.querySelectorAll('#importModeModal').length === 1` |
