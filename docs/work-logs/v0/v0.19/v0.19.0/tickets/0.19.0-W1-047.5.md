### 1. 實作策略設計

#### 1.1 promptImportMode modal 策略

modal 採「DOM 既有 modal 慣例 + Promise 包裝事件監聽器」。核心是把一次性 modal 互動轉為單一 Promise：modal 顯示後掛上各按鈕監聽器，任一選擇途徑觸發時 resolve 對應值並清理。

虛擬碼：

```
promptImportMode():
    # 取得 modal DOM 元素群
    overlay   = getElement('importModeOverlay')
    modal     = getElement('importModeModal')
    overwrite = getElement('importModeOverwriteBtn')
    merge     = getElement('importModeMergeBtn')
    cancel    = getElement('importModeCancelBtn')

    # DOM 缺失防禦（對應 TC-G1 / Error Handling 表第 6 列）
    IF 任一元素為 null:
        console.error('匯入模式 modal 元素缺失')
        RETURN Promise.resolve(MODAL_MISSING_SENTINEL)
        # 註：以特殊 sentinel 區分「DOM 缺失」與「使用者取消(null)」，
        #     使 handleFileLoad 能對兩者分流（缺失→showError，取消→靜默）

    # 單一實例保護（對應 TC-F1/F2 / 場景 8）
    IF modal 已顯示 (this._importModePending 存在):
        RETURN this._importModePending   # 回傳既有 pending Promise，不開第二個

    this._importModePending = new Promise(resolve):
        # 統一收斂出口：解除所有監聽器 + 隱藏 modal + resolve + 清 pending 旗標
        DEFINE settle(result):
            removeListeners()
            hideModal()                  # overlay/modal display = none
            this._importModePending = null
            restoreFocus()               # 焦點還原至觸發前元素
            resolve(result)

        # 綁定四種出口
        onClick(overwrite, () => settle('overwrite'))
        onClick(merge,     () => settle('merge'))
        onClick(cancel,    () => settle(null))
        onClick(overlay,   (e) => IF e.target === overlay: settle(null))
            # e.target === overlay 確保只有點遮罩本身才取消，
            # 點 modal 內部冒泡到 overlay 不誤判（對應 TC-B4）
        onKeydown(modal, (e) =>
            IF e.key === 'Escape': settle(null)
            IF e.key === 'Tab':    handleFocusTrap(e))

        showModal()                      # overlay/modal display = block/flex
        setInitialFocus(預設按鈕)          # 對應 TC-E2

    RETURN this._importModePending
```

focus trap 虛擬碼（對應 TC-E3/E4/E5）：

```
handleFocusTrap(event):
    focusable = [overwriteBtn, mergeBtn, cancelBtn]   # modal 內可聚焦元素有序集合
    first = focusable[0]
    last  = focusable[last index]

    IF event.shift 為 true AND activeElement === first:
        event.preventDefault()
        last.focus()
    ELSE IF event.shift 為 false AND activeElement === last:
        event.preventDefault()
        first.focus()
    # 其餘情況讓瀏覽器預設 Tab 行為在三按鈕間移動
```

#### 1.2 handleFileLoad 分流策略

modal 在「檔案基本驗證之後、讀檔之前」彈出。取消（resolve null）提前 return；DOM 缺失（sentinel）showError 後 return；正常選擇分流至對應 adapter API。

虛擬碼：

```
handleFileLoad(file):
    # 1. 檔案基本驗證（不變，先於 modal — 對應 TC-C4）
    importer._validateFileBasics(file)     # 失敗時 throw，modal 永不彈出
    importer._validateFileSize(file)

    # 2. 模式選擇（新增）
    mode = AWAIT promptImportMode()

    # 3. DOM 缺失分流（對應 TC-G1）
    IF mode === MODAL_MISSING_SENTINEL:
        showError('匯入功能初始化失敗')
        RETURN

    # 4. 取消分流（對應 TC-C3 / 場景 3/4/5）
    IF mode === null:
        RETURN          # 不 showLoading、不讀檔、不寫 storage、currentBooks 不變

    # 5. 讀檔（modal resolve 之後 — 對應 TC-C5 順序斷言）
    showLoading('正在讀取檔案...')
    importResult = AWAIT importer._readFileWithReader(file)

    # 6. 依模式分流寫入（對應 TC-C1/C2 / 場景 1/2）
    payload = { books, tags, tagCategories } FROM importResult
    IF mode === 'overwrite':
        writeResult = AWAIT TagStorageAdapter.replaceAllData(payload)
    ELSE:   # mode === 'merge'
        writeResult = AWAIT TagStorageAdapter.mergeAllData(payload)

    # 7. 寫入結果處理（不變，覆蓋/合併對稱 — 對應 TC-D1/D2/D3）
    IF writeResult.success === true:
        _updateUIWithBooks(importResult.books)
    ELSE IF writeResult.error === 'quota_exceeded':
        showError('儲存空間不足，匯入未完成')
    ELSE:
        showError('儲存失敗，已還原原有資料')
```

關鍵設計：步驟 7 的三分支對 `replaceAllData` 與 `mergeAllData` 完全共用，因兩者回傳契約對稱（`{success, error}`）。合併模式錯誤處理（TC-D1/D2）因此自動對稱，無需額外分支。

### 2. 資料結構選擇

| 結構 | 選擇 | 理由 |
|------|------|------|
| modal 回傳值 | 字串列舉 `'overwrite'` / `'merge'` / `null` | 規格 §2.1 已定義；字串列舉比布林對更可讀，未來可擴充第三模式 |
| DOM 缺失標記 | 獨立 sentinel（非 `null`） | `null` 已被「使用者取消」佔用；若 DOM 缺失也回 `null`，handleFileLoad 無法區分「靜默中止」與「showError 中止」。建議用 module 私有常數（如 `Symbol` 或字串 `'__modal_missing__'`），確保不與合法模式值碰撞 |
| pending Promise 旗標 | `this._importModePending`（實例屬性） | 單一實例保護需跨呼叫共享狀態；實例屬性生命週期與 controller 一致，頁面銷毀時隨之回收（符合 /spec Q3 決議「接受瀏覽器生命週期自然回收」） |
| focusable 元素集合 | 固定有序陣列 `[overwrite, merge, cancel]` | modal 內可聚焦元素固定三個，硬編碼順序比 `querySelectorAll('[tabindex]')` 動態查詢更明確、更易測 |
| 監聽器引用 | 區域變數持有 handler 參考 | settle 時需精準 `removeEventListener`；匿名函式無法移除，必須具名持有 |

### 7. 語言特定實作注意事項（JavaScript / Chrome Extension）

- **jsdom 測試環境**：modal DOM 必須在 `beforeEach` 注入 body；`document.activeElement` 在 jsdom 可用但元素需在 document 樹內且可聚焦（button 預設可聚焦）。focus trap 測試用 `new KeyboardEvent('keydown', {key:'Tab', shiftKey:bool})` dispatch。
- **事件冒泡**：遮罩取消判斷必須 `e.target === overlay`，否則點 modal 內部會冒泡到 overlay 監聽器誤觸取消（TC-B4 即驗此）。
- **監聽器移除**：jsdom 與瀏覽器一致，`removeEventListener` 需傳入與 `addEventListener` 同一函式參考；匿名箭頭函式無法移除，必須具名持有。
- **CSP 與 inline 樣式**：modal 顯示/隱藏用 `element.style.display` 切換即可，無需 inline `onclick`（Manifest V3 CSP 禁 inline handler；本策略全用 `addEventListener`，合規）。
- **`getElementById` 容錯**：取不到回傳 `null` 不 throw，DOM 缺失防禦依賴此行為——`promptImportMode` 主動檢查 null 而非依賴 try-catch。
- **無 Chrome API 依賴**：modal 純前端 DOM，不碰 `chrome.*`，測試不需 mock Chrome API。

#### Group A：promptImportMode modal 基礎行為（場景 1/2 modal 端）

- TC-A1 promptImportMode 呼叫後 modal 由隱藏轉為顯示（前置：modal 初始 display:none → 斷言顯示）

- TC-A5 點「合併」按鈕 → Promise resolve 為 `'merge'`，modal 關閉
- TC-A6 resolve 前 modal 持續顯示（點擊前 await 未完成、modal 仍 visible）

#### Group B：取消途徑（場景 3/4/5）

- TC-B1 點「取消」按鈕 → Promise resolve `null`，modal 關閉
- TC-B2 按 Esc 鍵（modal 取得焦點）→ Promise resolve `null`，modal 關閉
- TC-B3 點遮罩層（importModeOverlay）→ Promise resolve `null`，modal 關閉
- TC-B4 點 modal 容器內部（非遮罩）不觸發取消（事件不冒泡誤判，Promise 未 resolve）

#### Group C：handleFileLoad 分流（場景 1/2/3 流程端）

- TC-C1 選覆蓋 → handleFileLoad 呼叫 `replaceAllData` 一次，不呼叫 `mergeAllData`；成功後 _updateUIWithBooks 收 importResult.books
- TC-C2 選合併 → handleFileLoad 呼叫 `mergeAllData` 一次，不呼叫 `replaceAllData`；成功後 _updateUIWithBooks 收合併後 books
- TC-C3 取消（promptImportMode resolve null）→ handleFileLoad 提前 return：不呼叫 _readFileWithReader、不呼叫任何 *AllData、不呼叫 showLoading、currentBooks 維持原狀

- TC-C5 modal 在讀檔之前彈出：promptImportMode 的 resolve 早於 _readFileWithReader 被呼叫（順序斷言，用 mock call order）

#### Group D：合併模式錯誤處理（場景 6，與覆蓋對稱）

- TC-D1 mergeAllData 回傳 `{success:false, error:'quota_exceeded'}` → showError 訊息含「儲存空間不足」，不更新 UI，currentBooks 維持原狀
- TC-D2 mergeAllData 回傳 `{success:false, error:'storage_error'}` → showError 訊息含「儲存失敗」，不更新 UI
- TC-D3 mergeAllData 回傳 `{success:true}` → _updateUIWithBooks 被呼叫一次

#### Group E：無障礙與鍵盤導覽（場景 7）

- TC-E1 modal 容器具 `role="dialog"`、`aria-modal="true"`、`aria-labelledby` 指向標題 id
- TC-E2 modal 開啟時初始焦點落於預設選項按鈕（document.activeElement === 預設按鈕）
- TC-E3 focus trap：焦點在最後一個按鈕按 Tab → 回到第一個按鈕（document.activeElement 斷言）
- TC-E4 focus trap：焦點在第一個按鈕按 Shift+Tab → 跳到最後一個按鈕
- TC-E5 modal 開啟期間 Tab 不跑到 modal 外頁面元素（焦點仍在三按鈕集合內）

#### Group F：單一 modal 實例（場景 8）

- TC-F1 modal 開啟中再次呼叫 promptImportMode（或再次觸發匯入）→ DOM 中 modal 仍只有一個實例（不新增第二個 modal node）
