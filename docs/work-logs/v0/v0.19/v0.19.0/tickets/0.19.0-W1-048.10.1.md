#### B. BookFileImporter Constructor DI Signature

```js
class BookFileImporter {
  /**
   * @param {Object} deps
   * @param {Document} deps.document - DOM 文檔（既有，保留）
   * @param {Function} deps.showError - 錯誤訊息回呼（既有，保留）
   * @param {FileValidator} [deps.validator] - 選用，預設由 importer 自建
   * @param {FileContentReader} [deps.reader] - 選用，預設由 importer 自建
   * @param {ContentParser} [deps.parser] - 選用，預設由 importer 自建
   */
  constructor ({ document, showError, validator, reader, parser }) {
    this.document = document
    this.showError = showError
    this.parser = parser ?? new ContentParser()
    this.validator = validator ?? new FileValidator({ showError })
    this.reader = reader ?? new FileContentReader({
      parser: this.parser,
      showError,
      detectFormat: (file) => this.validator.detectFormat(file)
    })
  }
}
```

**設計要點**：
- 三 helper 皆選用，預設由 importer 自建（生產情境零變更，向後相容）
- reader 預設取 importer 內部 parser 實例，確保 read → parse 鏈一致
- 測試注入 stub helper 取代 `importer._method = jest.fn()` 反模式

#### C. 既有 Public Method 兼容策略

**保留全部 4 個 public method 作 thin wrapper delegate**，維持向後相容：

| Public Method | Phase 3 改寫後實作 | 兼容性 |
|--------------|------------------|--------|
| `handleFileLoad(file)` | `this.validator.validate(file); return this.reader.read(file)` | 完全相容 |
| `validate(file)` | `this.validator.validate(file)` | 完全相容 |
| `read(file)` | `this.reader.read(file)` | 完全相容 |
| `parseContent(content, fileFormat)` | `this.parser.parse(content, fileFormat)` | 完全相容（parser.parse 內含既有 TypeError / PARSE_ERROR 檢查） |

**移除對象**：4 個底線方法（`_validateFileBasics` / `_validateFileSize` / `_readFileWithReader` / `_handleFileContent`）及其僅供內部使用的所有子 helper（共 ~20 個 `_xxx` method），全數搬遷至對應 helper class。

**反向斷言守門**（`overview-import-private-methods.test.js`）：
- 既有：檢查 controller 不再暴露底線 proxy
