## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

content script bundle 內註冊了多個 `chrome.runtime.onMessage.addListener`：

1. `src/content/content-modular.js:275` — 處理 `START_EXTRACTION`，回應 `{ success: true, flowId, message }`
2. `src/content/bridge/chrome-event-bridge.js:360` — 原為 `async` 函式，僅處理 `CROSS_CONTEXT_EVENT`

直接根因：`chrome-event-bridge.js` 的 `messageListener` 宣告為 `async`。當 popup 透過 `chrome.tabs.sendMessage({ type: 'START_EXTRACTION' })` 觸發提取時，Chrome 會把該訊息分派給「所有」已註冊監聽器。`messageListener` 收到 `START_EXTRACTION` 不符合 `CROSS_CONTEXT_EVENT` 分支，函式體直接結束——但因為是 `async` 函式，回傳的是 `Promise.resolve(undefined)`。

Manifest V3 行為：監聽器回傳 Promise 會被 Chrome 視為「此監聽器負責回應」，並把 Promise resolve 的值（`undefined`）送回 sender。此回應與 `content-modular.js` 中真正處理 `START_EXTRACTION` 的監聽器競爭，導致 popup 端 `chrome.tabs.sendMessage` 的 await 取得 `undefined`。

popup 端 `src/popup/popup.js:765-781` `startExtraction`：`response` 為 falsy → 走 else 分支 `new Error(response?.error || '未知錯誤')` → catch 區塊 `updateStatus('失敗', '提取失敗', ...)`。提取本身（content script + storage 寫入）已成功，但 popup UI 誤顯「提取失敗 / 未知錯誤」。

此根因與 `docs/data-flow-architecture.md` 已驗證陷阱 #1「async listener 導致訊息通道關閉」同源。

### 影響範圍

- `src/content/bridge/chrome-event-bridge.js` — `messageListener` 改為同步函式（修復點）
- `tests/unit/core/chrome-event-bridge.test.js` — 既有測試依賴 buggy async 行為，需對齊新契約
- 行為影響：popup 提取結果回饋；不影響 content script 提取邏輯與 storage 寫入

### 相關 Error Pattern

無既有 error-pattern 直接對應。根因吻合 `docs/data-flow-architecture.md` §已驗證陷阱 #1（async listener 回傳 Promise）。建議後續評估是否升級為 error-pattern（MV3 多監聽器回應契約）。

調查過程：
- `grep -rn "onMessage.addListener" src/content/` 定位三個監聽器（content-modular / chrome-event-bridge / event-utils，後者未在 content bundle 路徑）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 修復摘要

將 `src/content/bridge/chrome-event-bridge.js` 的 `messageListener` 從 `async` 函式改為「同步」函式，遵循 Manifest V3 多監聽器回應契約：

- 非處理的訊息類型（如 `START_EXTRACTION`）→ 同步回傳 `undefined`（非 Promise），Chrome 忽略此監聽器，不搶通道。
- 處理的訊息類型（`CROSS_CONTEXT_EVENT`）→ 將 async 邏輯抽出為獨立的 `handleCrossContextEvent` fire-and-forget 函式，監聽器同步回傳 `true` 保持通道開啟，由 `handleCrossContextEvent` 非同步呼叫 `sendResponse`。

此修復讓 popup `START_EXTRACTION` 的回應僅由 `content-modular.js` 中真正處理該訊息的監聽器負責，popup 取得正確的 `{ success: true, ... }`，提取成功時正確顯示。

### 設計決策

- 不重構 popup 端等待邏輯（popup.js startExtraction 仍 await content script 回應）：root cause 在監聽器回應契約，最小修復即修正監聽器，不擴大範圍。
- 監聽器函式禁止宣告為 `async`：已在程式碼補充註解說明（MV3 多監聽器回應契約 + 0.19.0-W1-007 BUG-A 來源），防止後續回歸。
- 既有測試 `tests/unit/core/chrome-event-bridge.test.js` 對 CROSS_CONTEXT_EVENT 流程的 `await messageListener` 斷言依賴 buggy async 行為；新增 `flushMicrotasks` helper 並改為「同步取回傳值 + flush microtask 等非同步鏈完成」，對齊新契約。

### 變更檔案

- `src/content/bridge/chrome-event-bridge.js` — messageListener 改同步 + 抽出 handleCrossContextEvent
- `tests/integration/chrome-extension/popup-extraction-result-feedback.test.js` — 新增 RED 測試（BUG-A 重現與防護）
- `tests/unit/core/chrome-event-bridge.test.js` — 對齊新契約（flushMicrotasks helper + 同步回傳斷言）

- 實機重測 UC-01（acceptance #3）：subagent 環境無 chrome-devtools-mcp，以整合測試驗證為主，實機部分見 NeedsContext。
