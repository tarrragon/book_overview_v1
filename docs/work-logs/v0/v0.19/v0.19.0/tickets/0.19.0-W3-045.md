## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### Context Bundle（PM 派發前置）

**動機觸發**：
- 用戶 memory「Parallel Claim AC Verification Collision」：並行 claim 時 AC verification 跑全套件測試會撞其他 agent 未 commit 中間狀態
- 用戶在 W6 wave session 觀察「同 wave 平行 claim 風險」並提出 hook 化建議
- 屬 PC-078 並行 session 同源衝突問題的子場景

**三層方案**（待評估定位優先序）：
- L1: PreToolUse:Bash hook 偵測 ticket track claim 命令，查 dispatch-active.json 或 ticket track list --status in_progress 同 wave，若同 wave 有正在跑全套件測試的 ticket 則阻擋 claim（exit 2）
- L2: acceptance 設計約定：規範 acceptance 禁寫「跑全套件測試」字面（如 npm test / pytest 無 path），改為「跑相關檔案測試 N 個」；linter 偵測既有違反案例
- L3: CLI AC verification 自身行為修改：ticket track claim 的 AC verification 邏輯只跑 acceptance 提到的相關檔案，不全套件（從根源消除衝突）

**既有相關位置**：
- .claude/skills/ticket/ticket_system/ — CLI AC verification 邏輯所在
- .claude/hooks/ — 既有 PreToolUse:Bash hook 範本
- .claude/pm-rules/ticket-lifecycle.md — acceptance 設計規範
- 用戶 memory feedback_parallel_claim_ac_verification_collision.md

### 問題根因

AC verification 衝突的直接原因是 validation_templates.py 中 npm_test_pass 模板硬編碼 command="npm test 2>&1 | tail -5"。當 acceptance 含「npm test」「測試全部通過」「測試通過率 100」等關鍵字時，match_template() 匹配此模板，execute_verification() 以 subprocess.Popen(shell=True) 在專案根目錄執行全套件測試。

全套件測試是全域操作：Jest 掃描工作樹中所有檔案（含其他 agent 未 commit 的中間狀態），導致 false failure。這不是 claim 的邏輯錯誤，而是 AC verification 的測試範圍過寬。

### 影響範圍

| 檔案 | 影響 |

| .claude/skills/ticket/ticket_system/lib/validation_templates.py | npm_test_pass 模板的 command 欄位（L3 修改點） |

## 重現實驗結果

### 實驗方法

靜態分析 validation_templates.py + claim_verification.py + lifecycle.py 三個模組的呼叫鏈，確認 claim AC verification 的完整執行路徑和測試命令來源。

### 實驗執行

1. validation_templates.py:65-76 — npm_test_pass 模板 patterns 為 ("npm test", "測試全部通過", "測試通過率 100", "0 個失敗", "0 failed")，command 為 "npm test 2>&1 | tail -5"
2. claim_verification.py:285-363 — execute_verification() 直接用 subprocess.Popen(vc.command, shell=True, cwd=cwd) 執行模板 command，cwd 為 resolve_project_cwd() 回傳的專案根
3. lifecycle.py:570-573 — claim_with_verification() 呼叫 run_all_verifications(pairs, cwd) 循序執行所有可驗證 AC
4. lifecycle.py:1979-2000 — execute_claim() 統一走 claim_with_verification 入口

關鍵發現：整條鏈路中沒有任何機制將測試範圍縮限到 ticket 相關檔案。npm_test_pass 模板的 command 是固定字串，不接受參數化。

### 實驗發現

**已驗證事實**：
- AC verification 的測試命令來自 validation_templates.py 的硬編碼 command 欄位，不可參數化
- execute_verification() 在專案根執行 npm test，掃描全工作樹
- 並行 agent 未 commit 的中間狀態必然被 Jest 掃到
- 目前無任何 hook 或機制在 claim 前檢查同 wave 是否有其他 in_progress ticket

**仍未驗證假設**：
- L3 修改 validation_templates.py 支援參數化 command 的實作複雜度（需設計 command template + ticket where.files 注入機制）
- L1 hook 跨 session 通訊的 race condition 風險（dispatch-active.json 的 atomic write 可靠性）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### WRAP 決策框架執行紀錄

**錨點**：客戶是專案開發團隊（PM + agent），核心目標是並行派發時 claim AC verification 不產生 false failure。

**Step 0（資料充足度）**：已確認 validation_templates.py 源碼、claim_verification.py 執行路徑、lifecycle.py 串接方式、歷史觸發案例。資料充足，進入 W 階段。

#### W（擴增選項）

| 選項 | 描述 | 修改範圍 |
|------|------|---------|
| L3-a: command 參數化 | validation_templates.py 的 command 改為 template string，claim 時注入 ticket where.files 對應的測試路徑 | validation_templates.py + claim_verification.py + lifecycle.py |
| L3-b: claim 時 skip AC verification | 並行情境下自動加 --skip-verify，complete 時再跑驗證 | lifecycle.py（判斷同 wave in_progress 數量） |
| L2: acceptance 規範 + linter | 規範 acceptance 禁寫全套件測試字面，linter 偵測違反 | pm-rules/ + validators/ |
| L1: PreToolUse:Bash hook | claim 前偵測同 wave in_progress 並阻擋 | hooks/ |
| L0（未列入原方案）: PM 流程約束 | PM 手動在所有 agent commit 後再 claim 下一批 | 無程式碼修改，純流程 |

**Consider the Opposite 檢驗**：L3-a 看似最完整但修改範圍最大（3 個模組 + 需要 ticket metadata 注入測試路徑映射邏輯）。L3-b 更簡單但犧牲了 claim 時的早期回饋。L0 成本最低但依賴 PM 自律（PC-066 已證明自律機制在壓力下失效）。

#### R（現實檢驗）

| 選項 | 實作複雜度 | 防護完整性 | 副作用 |
|------|-----------|-----------|--------|
| L3-a | 高（需設計 command template 語法 + file-to-test 映射） | 最高（根源消除） | 映射邏輯可能不準確（哪些測試對應哪些 src 檔？） |
| L3-b | 低（加一個條件判斷） | 中（延遲驗證到 complete） | claim 失去早期回饋功能 |
| L2 | 中（linter 規則 + 文件更新） | 低（僅防新 acceptance，既有不受保護） | 限制 acceptance 撰寫自由度 |
| L1 | 中（hook + 跨 session 狀態查詢） | 低（補丁解，不解決根因；race condition 風險） | hook 誤阻擋合法 claim |
| L0 | 零 | 低（依賴 PM 自律） | 降低並行效率 |

**基本率**：歷史案例中觸發頻率為「並行 2+ agent 且其中至少一個 ticket acceptance 含全套件測試」。目前專案大多數 ticket 的 acceptance 含「npm test 100%」類字面，因此幾乎每次並行派發都會觸發。

#### A（拉開距離）

**機會成本**：不修 L3 只做 L1，每次並行派發 PM 仍需手動排序 claim 時序或事後處理 false failure，長期成本高。

**10/10/10 框架**：
- 10 分鐘後：L3-b 最快落地
- 10 天後：L3-a 若映射邏輯成熟，體驗最佳
- 10 個月後：L3-a 的 file-to-test 映射需持續維護，維護成本未知

#### P（準備好犯錯）

**絆腳索**：若 L3-b 落地後發現 complete 階段 false failure 同樣頻繁（因 complete 也跑全套件），需回退到 L3-a。

### 結論：Layer 定位與優先序

| Layer | 定位 | 優先序 | 理由 |
|-------|------|--------|------|
| **L3-b（主解）** | claim 時跳過全套件 AC verification；驗證延遲到 complete 或 agent 自行跑測試 | P1 — 首先實作 | 修改範圍最小（lifecycle.py 一處條件）、根源消除並行衝突、不影響 complete 驗證 |
| L2（補位） | acceptance 設計約定 + linter | P2 — L3-b 後 | 從源頭規範新 acceptance 不觸發全套件測試；但既有 acceptance 無法改（向後相容） |
| L1（暫緩） | PreToolUse:Bash hook | 無需建 ticket | L3-b 已從根源消除衝突，hook 層補丁不再必要 |

**L1 豁免理由**：L3-b 修改 claim_with_verification 使其在偵測到同 wave 有其他 in_progress ticket 時自動 skip-verify（或全面改為 claim 預設不跑 AC verification、僅 complete 時跑）。此修改從根源消除了「claim 時跑全套件測試撞並行 agent」的問題，L1 hook 的補丁定位完全被 L3-b 涵蓋。

### L3-b 實作方向建議（供 spawn IMP 參考）

兩個候選實作策略（IMP ticket 的 Phase 1 應評估取捨）：

**策略 A：claim 偵測同 wave in_progress 數量**
- claim_with_verification() 在執行 run_all_verifications() 前，查詢同 wave 是否有其他 status: in_progress 的 ticket
- 若有 >= 1 個同 wave in_progress 則自動降級為 skip_verify=True
- 優點：保留單獨 claim 時的早期回饋
- 缺點：需要在 CLI 中加入 ticket 查詢邏輯

**策略 B：claim 全面移除 AC verification（僅 complete 時跑）**
- 移除 claim_with_verification 中的測試執行邏輯，claim 只做狀態切換
- complete 時（acceptance-gate-hook）執行 AC verification
- 優點：最簡單、徹底消除衝突
- 缺點：失去 claim 時的早期回饋（「AC 已全部達成，建議直接 complete」的提示）
