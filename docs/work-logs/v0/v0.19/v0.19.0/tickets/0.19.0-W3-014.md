## Problem Analysis

### PC-074 現況

PC-074 記錄了字元集守衛 Hook 實作時繁簡共用字 false positive 問題。防護措施目前為「行為準則」層級（非自動載入），核心知識包含：

1. **靜態清單驗證流程**：擴充 SIMPLIFIED_CHARS 前逐字查繁簡對照字典
2. **動態驗證根本性解法**（v1.2.0 新增）：用 OpenCC `s2t(X) != X` 動態建構黑名單 + self-test
3. **推廣原則**：「依賴維護者記憶力區分相似實體的清單，必須改用工具自動驗證」

### 升級動機

- PC-074 已被**二度自證**（W17-144.1）：即使讀過 PC-074 警告，維護者仍在新增清單時混入共用字。證明行為準則層級不足
- 核心原則「動態驗證取代靜態維護」具跨場景通用性（簡體字 / 日漢字 PC-084 / 異體字 / 任何字元集子集），適合升級為自動載入規則
- 參考先例：PC-079 backtick 問題已升級為 `bash-tool-usage-rules.md` 規則四

### 影響範圍

- `.claude/hooks/askuserquestion-charset-guard-hook.py` — 主要受益 Hook
- `.claude/error-patterns/process-compliance/PC-074*.md` — 需補「已升級」備註
- `.claude/error-patterns/process-compliance/PC-084*.md` — 姊妹模式，交叉引用

### 相關 Error Pattern

## Analysis Result

### (a) 升級目標位置評估

| 候選位置 | 適合度 | 理由 |
|---------|--------|------|
| `rules/core/language-constraints.md` 新增規則 5 | 最佳 | PC-074 核心是「字元集清單維護」，直接歸屬語言約束規則；language-constraints.md 已有繁體/簡體/emoji 規則，新增「字元集子集清單動態驗證」規則是自然延伸。自動載入，所有角色可見 |
| `methodologies/` 新建方法論 | 次佳 | 「動態驗證取代靜態維護」有方法論特質，但內容量不足以獨立成方法論（核心原則僅一條 + 一個流程）；且方法論需按需讀取，無法自動載入 |
| `skills/` 新建 skill | 不適合 | PC-074 是防護規則，非互動式工具；skill 定位為按需觸發的工作流，不符合「每次維護清單都要遵守」的強制性需求 |

**結論：升級至 `rules/core/language-constraints.md` 新增規則 5**。

### (b) 規則內容骨架

```markdown
### 規則 5：字元集子集清單必須動態驗證

**維護任何「字元集子集」清單（簡體字黑名單、日漢字清單、異體字白名單等）時，
禁止僅依人工判斷，必須用工具動態驗證清單純度。**

**Why**：人工撰寫字元清單時無法可靠區分繁簡共用字與簡體專屬字。
PC-074 已被二度自證（W17-144.1）：即使讀過警告，維護者仍在新增清單時混入共用字。
Token 生成過程不會主動檢查文件中的警告規則。

**Consequence**：共用字混入黑名單會產生 false positive，
打斷正常繁體文本流程，比 false negative 更影響信任。

**Action**：

| 場景 | 反模式 | 正確做法 |
|------|-------|---------|
| 簡體字黑名單 | 人工列簡體字清單 | OpenCC `s2t(X) != X` 動態過濾 |
| 日漢字黑名單 | 人工列日漢字清單 | unicodedata + JIS 表動態過濾 |
| 異體字白名單 | 人工列異體字 | OpenCC round-trip 動態判定 |
| 任何字元集子集 | 靜態維護 | 依規則動態建構 + 啟動時 self-test |

> 來源：PC-074（v1.2.0 動態驗證章節）。完整案例與已知共用字清單見 PC-074。
```
