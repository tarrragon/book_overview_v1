## Task Summary

評估 v2.1.139–143 引入的 plugin 管理變更對 references/plugin-management.md 規範與專案 skill 設計慣例的影響

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

Claude Code v2.1.139–143 引入多項 plugin 管理機制變更，專案 `references/plugin-management.md`（v1.0.0, 2026-04-20）尚未反映這些新能力，可能導致規範與平台實際行為脫節。

### 影響範圍

- `.claude/references/plugin-management.md` — 安裝前評估清單、context 成本對照表、審查週期
- `.claude/plugins/` — 專案級 LSP plugin（dart-lsp, marksman-lsp, yaml-lsp）
- `~/.claude/plugins/installed_plugins.json` — 使用者級 12 個 plugin

### 相關 Error Pattern

無直接相關 PC。間接相關：PC-088（LLM 工具選擇偏誤）— projected token cost 功能可補強評估清單第 4 題的可執行性。

### 實驗方法

1. 盤點專案 plugin 使用現況（`.claude/plugins/` 目錄 + `~/.claude/plugins/installed_plugins.json`）
2. 確認當前 CC 版本（v2.1.145，已超過目標範圍 v2.1.139–143）
3. 對照 v2.1.139–143 各版本 changelog 描述的 plugin 變更，逐一檢查對專案的影響
4. 檢查專案是否存在 root-level SKILL.md（v2.1.142 會將其曝光為 skill）

### 實驗執行

**專案 plugin 現況**：

- 專案級 `.claude/plugins/`：3 個 LSP plugin（dart-lsp, marksman-lsp, yaml-lsp）
- 使用者級：12 個 plugin（github, commit-commands, linear, playwright, pyright-lsp, typescript-lsp, doc-structure-plugin, gopls-lsp, ralph-wiggum, plugin-dev, hookify, greptile）
- 無根目錄 SKILL.md 檔案
- 無 `$CLAUDE_PLUGIN_ROOT` 使用
- 無 plugin 間 dependency 宣告

**v2.1.139–143 變更逐項檢查**：

| 版本 | 變更 | 專案影響 |
|------|------|---------|
| v2.1.143 | plugin dependency enforcement（disable 拒絕有 dependent / enable force-enable transitive） | 低：當前 12 個 plugin 無 dependency 鏈 |
| v2.1.143 | projected context cost 顯示 | 正面：可量化第 1 章第 4 題「Token 成本評估」 |
| v2.1.142 | root-level SKILL.md 曝光為 skill | 無影響：專案無根目錄 SKILL.md |
| v2.1.142 | LSP servers 顯示 | 正面：可改善第 5 章查詢指令 |
| v2.1.141 | config 變數錯誤訊息改善 | 低：純 DX 改善，無規範影響 |

### 實驗發現

**已驗證事實**：

1. 專案當前 12 個 plugin 無 dependency 關係，dependency enforcement 不會造成破壞
2. 專案無根目錄 SKILL.md，root SKILL.md 曝光機制無衝擊
3. `plugin-management.md` v1.0.0 缺乏 dependency enforcement 概念，未提及 projected cost 工具
4. 第 1 章評估清單第 4 題「Token 成本評估」目前靠人工估算，projected cost 可量化

**仍未驗證假設**：

- projected context cost 的具體 UI/CLI 介面尚未實測（需升級後觀察）
- dependency enforcement 的具體行為（disable 時是否有 confirmation prompt）尚未實測

## WRAP 框架分析

### W — Widen（擴增選項）

列舉 5 項變更對規範的影響面：

| # | 影響面 | 現狀 | 變更後 | 規範修訂需求 |
|---|--------|------|--------|------------|
| 1 | dependency enforcement | plugin-management.md 未提及 plugin 間依賴 | v2.1.143 引入 disable 拒絕有 dependent、enable force-enable transitive | 第 1 章新增第 7 題「依賴性檢查」；第 4 章卸載流程新增 dependency 警告 |
| 2 | projected context cost | 第 1 章第 4 題「Token 成本評估」靠人工估算 ~110 tokens/skill | v2.1.143 顯示 projected cost | 第 1 章第 4 題改用 projected cost 作判定工具；第 2 章補充 projected cost 查詢指令 |
| 3 | root-level SKILL.md 曝光 | 無提及 | v2.1.142 root SKILL.md 自動成為 skill | 第 7 章反模式新增「意外的根目錄 SKILL.md 曝光」條目 |
| 4 | LSP servers 顯示 | 第 2 章 LSP 列為「無 context 成本」但無查詢方式 | v2.1.142 可查詢 LSP servers | 第 5 章查詢指令補充 LSP 查詢方法 |
| 5 | config 變數錯誤訊息改善 | 無提及 | v2.1.141 改善錯誤訊息 | 無規範修訂需求（純 DX） |

### R — Reality-test（現實測試）

對照現有 plugin 用法：

- 12 個已安裝 plugin 無 dependency 鏈 → dependency enforcement 短期無破壞風險
- 專案無根目錄 SKILL.md → root SKILL.md 機制無即時衝擊
- `plugin-management.md` 第 1 章評估清單是專案 plugin 篩選的唯一規範入口 → 必須更新以反映新工具

### A — Attain distance（評估升級成本）

| 方案 | 成本 | 收益 |
|------|------|------|
| A: 不更新規範 | 零 | 規範與平台脫節，新功能（projected cost）未被利用 |
| B: 最小更新（僅第 1 章 + 第 7 章） | DOC ticket 1 個，約 30 分鐘 | 覆蓋 dependency enforcement 和 root SKILL.md 防護 |
| C: 完整更新（第 1/2/4/5/7 章） | DOC ticket 1 個，約 1 小時 | 全面對齊 v2.1.143，projected cost 可量化評估 |

**結論：採方案 C**。plugin-management.md 是防止 context 膨脹的關鍵規範，完整更新成本低且收益明確。

### P — Prepare to be wrong（失敗回退）

- projected cost CLI 介面若與預期不同 → DOC ticket 完成後實測，不符則再修
- dependency enforcement 行為若有變 → 觀察下次 plugin 安裝/卸載時的實際提示
