## 1. Purpose（目的）

**問題**：IMP-A（W1-047.1）已讓 `BookFileImporter` 接入版本偵測與 v1→v2 轉換，但回傳值仍為純 `books` 陣列。v2 JSON 檔案的頂層 `tagCategories` 與 `tags` 區段在 `_extractBooksFromData` 中被丟棄（程式碼註解明示「tagCategories/tags 暫時丟棄，回傳介面變更屬 IMP-B」）。v1 路徑經 `convertV1ToV2Data` 產生的 `tagCategories`/`tags`（由 v1 category 自動建立）同樣被丟棄。

**目標用戶**：UC-04 匯入功能的下游消費者（overview-page-controller，以及後續 IMP-C 的 storage 持久化）。

**核心價值**：讓 `BookFileImporter` 回傳完整的 v2 interchange 三區段結構 `{ books, tagCategories, tags }`，使 tag 資料不在匯入第一站即遺失。本任務只負責「提取並回傳」，tag 的持久化與覆蓋/合併屬 IMP-C~E，不在範圍。

**範圍邊界**：

| 在範圍 | 不在範圍 |
|--------|---------|
| v2 路徑提取頂層 `tagCategories`/`tags` 區段 | 覆蓋/合併模式選擇（IMP-C/IMP-D/IMP-E） |
| 回傳介面由純陣列改為 `{ books, tagCategories, tags }` 物件 | `chrome.storage.local` 持久化（IMP-C） |
| v1 路徑回傳 `convertV1ToV2Data` 產出的 `tagCategories`/`tags` | tag id 重映射（IMP-D） |
| overview-page-controller `handleFileLoad` 與 `_handleFileContent` 代理方法適配 | 模式選擇 UI（IMP-E） |
| 既有所有非匯入呼叫端不破壞 | 修改 `detectFormatVersion` / `convertV1ToV2Data` 行為 |

## 2. API Signatures（介面定義）

### 2.1 核心型別：ImportResult

```
ImportResult = {
  books: Book[],              // 必有，可能為空陣列 []
  tagCategories: TagCategory[],// 必有，可能為空陣列 []
  tags: Tag[]                  // 必有，可能為空陣列 []
}
```

三個欄位**永遠存在且型別恆為陣列**（依 SPEC-EXPORT-V2 §2.3「`tagCategories` 必須為陣列，否則初始化為空陣列」；`tags` 同）。消費端可無條件 `.length` / 解構，無需 null 檢查。

`Book` / `TagCategory` / `Tag` 結構依 SPEC-EXPORT-V2 §3.3/§3.4/§3.5，本任務不驗證 tag/category 內部欄位（僅提取，不做 schema 驗證——欄位驗證屬後續 ticket 或既有 schema 機制）。

### 2.2 BookFileImporter 變更後簽名

| 方法 | 變更前回傳 | 變更後回傳 | 說明 |
|------|-----------|-----------|------|
| `handleFileLoad(file)` | `Promise<Book[]>` | `Promise<ImportResult>` | 公開入口，匯入主路徑 |
| `_readFileWithReader(file)` | `Promise<Book[]>` | `Promise<ImportResult>` | 被 controller `handleFileLoad` 直接呼叫 |
| `_handleFileContent(content, fileFormat)` | `Book[]` | `ImportResult` | 被 controller `_handleFileContent` 代理方法呼叫 |
| `_processBookData(data, fileFormat)` | `Book[]` | `ImportResult` | 組裝 ImportResult 的匯聚點 |
| `_extractBooksFromData(data, fileFormat)` | `Book[]` | `{ books, tagCategories, tags }` | 提取點，新增 tag 區段提取 |

`_filterValidBooks` / `_isValidBook` / `_validateRequiredFields` / `_checkLargeDataset` 簽名不變（仍只作用於 books 陣列）。

### 2.3 各路徑的 tagCategories/tags 來源

| 路徑 | books 來源 | tagCategories 來源 | tags 來源 |
|------|-----------|-------------------|----------|
| v1 JSON | `convertV1ToV2Data(data).books` | `convertV1ToV2Data(data).tagCategories` | `convertV1ToV2Data(data).tags` |
| v2 JSON | 既有四形狀辨識（直接陣列 / `data.books` / `data.data`） | `data.tagCategories`（缺失或非陣列 → `[]`） | `data.tags`（缺失或非陣列 → `[]`） |
| CSV | 既有 `_parseCSVContent` 結果 | `[]`（CSV 無頂層 tag 區段） | `[]` |
| 空物件 `{}` | `[]` | `[]` | `[]` |

### 2.4 overview-page-controller 適配

| 方法 | 適配內容 |
|------|---------|
| `handleFileLoad(file)` | `_readFileWithReader` 回傳由陣列改為物件，需解構出 `books` 傳給 `_updateUIWithBooks`；`tagCategories`/`tags` 本 ticket 暫不使用（IMP-C 接手） |
| `_handleFileContent(content)` | 同上，解構 `books` 傳給 `_updateUIWithBooks` |

`_updateUIWithBooks` / `_updateBooksData` / `updateDisplay` 簽名與行為不變，仍只接收 books 陣列。本 ticket 不讓 tagCategories/tags 流入 UI 層。

## 3. GWT Scenarios（行為場景）

### 場景 1：v2 JSON 含 tagCategories/tags — 完整提取
- **Given**：使用者匯入一個 v2 JSON 檔案，根物件含 `metadata.formatVersion: "2.x"`、非空 `tagCategories` 陣列、非空 `tags` 陣列、`books` 陣列
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：回傳 `ImportResult`，`books` 為驗證後書籍陣列，`tagCategories` 等於檔案頂層 `tagCategories`，`tags` 等於檔案頂層 `tags`

### 場景 2：v2 JSON 無 tagCategories/tags 欄位 — 降級為空陣列
- **Given**：使用者匯入一個被偵測為 v2 的 JSON，但根物件不含 `tagCategories` 與 `tags` 欄位
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：回傳 `ImportResult`，`books` 正常提取，`tagCategories` 為 `[]`，`tags` 為 `[]`（依 SPEC §2.3）

### 場景 3：v2 JSON 的 tagCategories/tags 型別非陣列 — 降級為空陣列
- **Given**：v2 JSON 的 `tagCategories` 為物件或字串等非陣列值
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：該欄位降級為 `[]`，不拋錯，不中斷匯入（依 SPEC §2.3「必須為陣列，否則初始化為空陣列」）

### 場景 4：v1 JSON 含 category — 回傳轉換產生的 tag 結構
- **Given**：使用者匯入 v1 JSON（純陣列或 `data.books` 形狀），書籍含 `category` 欄位
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：回傳 `ImportResult`，`books` 為 v1→v2 轉換後書籍，`tagCategories` 與 `tags` 為 `convertV1ToV2Data` 自動建立的 tag 結構（「匯入分類」category + 各 category 對應 tag）

### 場景 5：v1 JSON 無 category — 回傳空 tag 結構
- **Given**：使用者匯入 v1 JSON，書籍皆無 `category` 欄位（或值為空/「未分類」）
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：`books` 正常轉換，`tagCategories` 為 `[]`，`tags` 為 `[]`（`convertV1ToV2Data` 在無有效 category 時回傳空陣列）

### 場景 6：CSV 匯入 — tag 區段為空陣列
- **Given**：使用者匯入 CSV 檔案
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：`books` 為 CSV 解析結果，`tagCategories` 為 `[]`，`tags` 為 `[]`（CSV 無頂層 tag 區段；CSV 既有 detectCsvFormatVersion 路徑不受本變更影響）

### 場景 7：空物件 JSON — 三區段皆空
- **Given**：使用者匯入內容為 `{}` 的 JSON
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：回傳 `{ books: [], tagCategories: [], tags: [] }`，不拋錯（保留既有「空對象回傳空陣列」契約）

### 場景 8：無法辨識的 JSON 結構 — 維持既有錯誤契約
- **Given**：使用者匯入既非陣列、亦無 `books`/`data` 屬性、亦非空物件的 JSON
- **When**：呼叫 `handleFileLoad(file)`
- **Then**：拋出既有 `VALIDATION_ERROR`（訊息「JSON 檔案應該包含一個陣列或包含books屬性的物件」），錯誤契約零回歸

### 場景 9：controller handleFileLoad 適配新回傳結構
- **Given**：`BookFileImporter._readFileWithReader` 回傳 `ImportResult` 物件
- **When**：overview-page-controller 的 `handleFileLoad` 被呼叫
- **Then**：controller 從回傳物件解構 `books`，傳入 `_updateUIWithBooks`，UI 顯示書籍數量與變更前一致；不因回傳結構變更而 UI 破圖或顯示 `undefined`

### 場景 10：既有非匯入呼叫端不破壞
- **Given**：`_updateBooksData` 仍被 `STORAGE.LOAD.COMPLETED` / `EXTRACTION.COMPLETED` / `UI.BOOKS.UPDATE` 等事件處理路徑呼叫，這些路徑傳入純陣列
- **When**：執行非匯入路徑
- **Then**：行為完全不變（本 ticket 不改 `_updateBooksData` 簽名，僅改匯入路徑的呼叫方式）

## 4. Error Handling（錯誤處理）

| 錯誤情境 | 處理策略 | 回傳/拋出 |
|---------|---------|----------|
| v2 JSON `tagCategories` 缺失 | 降級 | `tagCategories: []`，不拋錯 |
| v2 JSON `tagCategories` 非陣列 | 降級 | `tagCategories: []`，不拋錯（SPEC §2.3） |
| v2 JSON `tags` 缺失或非陣列 | 降級 | `tags: []`，不拋錯 |
| JSON 無法辨識結構 | 維持既有契約 | 拋 `VALIDATION_ERROR`（不變） |
| JSON 解析失敗 | 維持既有契約 | 拋 `PARSE_ERROR`（不變） |
| 檔案為空 / 過大 / 格式錯 | 維持既有契約 | 拋 `VALIDATION_ERROR`（不變） |
| v1 轉換中個別書籍轉換失敗 | 既有行為 | `convertV1ToV2Data` 內部跳過該書，不影響本 ticket |

**核心原則**：tag 區段提取採「寬鬆降級」（缺失/型別錯一律降為 `[]`），books 提取維持「既有嚴格契約」（無法辨識結構仍拋錯）。本 ticket 不新增任何拋錯路徑。

**可觀測性**：v2 JSON 的 `tagCategories`/`tags` 因型別非陣列而降級時，建議以 `console.warn` 記錄（對齊 SPEC §2.3「初始化為空陣列」並讓開發者可見降級事件）。此為建議非強制，由 Phase 2 測試與 Phase 3b 實作斟酌；若記錄，須含欄位名與「型別非陣列，降級為空陣列」說明（observability-rules 規則 1）。

### 迭代 1 — 回答

- **Q1（已回答）**：維持 IMP-A 既有四形狀辨識順序——`_isDirectArrayFormat` → `_isWrappedBooksFormat`（`data.books`）→ `_isMetadataWrapFormat`（`data.data`）。本 ticket 不改 books 提取順序，`tagCategories`/`tags` 一律從根物件 `data.tagCategories`/`data.tags` 讀取，與 books 取自哪個形狀無關。已補入 §2.3 表格。
- **Q2（已回答）**：空陣列與缺失一視同仁——兩者回傳結果都是 `[]`，消費端無法也無需區分。場景 1（非空）與場景 2（缺失）已涵蓋語義端點，空陣列為場景 2 的等價子集，無需額外場景。
- **Q3（已回答）**：不在本 ticket 範圍。本 ticket 只「提取並回傳」原始 tag 區段，不做 referential integrity 驗證。`categoryId` 懸空引用的處理屬 IMP-C/IMP-D（持久化與合併時的 id 重映射，SPEC §8.3 與 §9.2「匯入 tag 的 categoryId 在檔案內不存在 → 建立『未分類』category」）。已在 §2.1 明示「本任務不驗證 tag/category 內部欄位」。
- **Q4（已回答）**：無影響。版本判定完全由 IMP-A 已接入的 `detectFormatVersion` 決定。被判為 v1 的「舊格式」走 `convertV1ToV2Data` 路徑（場景 4/5 涵蓋）；被判為 v2 走頂層提取路徑（場景 1-3 涵蓋）。本 ticket 不碰版本判定邏輯，兩條路徑的 tag 來源已在 §2.3 分別定義。
- **Q5（已回答，標記為後續 ticket 範圍）**：本 ticket CSV 一律回傳 `tags: []`。v2 CSV 的 `tagNames`/`tagCategories` 衍生欄位反向解析（SPEC §5.1）屬獨立功能，不在 W1-047 ANA 的 IMP-A~E 拆分範圍內。回傳 `[]` 是型別安全的預設值，未來新增 CSV tag 解析時擴充該路徑即可，不造成相容破壞（消費端契約「型別恆為陣列」不變）。
