### 設計決策亮點

1. **Rule 4 高密度檔（data-validation-service）**：原斷言 `duration2 < duration1` 屬規則 4 反模式（計時比較驗證快取）。改用 `validationCache.size` 比對：
   - 首次呼叫後 size > 0（建立快取項）
   - 第二次呼叫後 size 不增加（命中快取）
   - 比計時更強的驗證（cache 失效時 size 增加會立即失敗，與計時完全無關）
   - 設計考量：`_setCachedValidation` 用 spread copy 儲存，無法用 `toBe(firstResult)` 參考相等驗證（與 W1-095 ErrorAdapter 模式不同）

2. **Rule 4 platform-detection-service**：同模式採用 `detectionCache.size` 命中率驗證，搭配 `service.clearCache()` 確保起點乾淨。

3. **shutdown-handler 區間斷言**：原 `duration > 29000 && duration < 32000` 為計時區間斷言，依規則 1 整段移除；保留 `error.code === TIMEOUT_ERROR` 已涵蓋超時功能。

4. **改寫驗證方向**：多檔將計時斷言改為 `not.toThrow()` / 結果結構驗證（render/event handler/批次執行），保留功能正確性同時消除計時依賴。

### 保留斷言（非違規）

依規則 1 例外或非計時類保留：
- confidence/qualityScore/score 等業務值範圍（toBeLessThan(0.8) 等）
- memoryIncrease/memoryGrowth/dataSize 資源預算斷言
- fingerprint.length/cache.size/result.length 等容器體積斷言
- toBeCloseTo 精度 ≤ 2 的浮點比較（規則 3 合規）
