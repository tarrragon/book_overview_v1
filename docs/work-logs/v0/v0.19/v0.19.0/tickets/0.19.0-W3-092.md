## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 結論：延後至 W4-015 第二輪重評（trigger 條件未達成）

#### 為何延後而非達成 trigger

W3-092 設計目標：判定「--skip-verify 是否可移除」。判定條件：
1. 外部 / 用戶腳本對 --skip-verify 依賴 = 0 → **已驗證**（grep baseline）
2. 真實 ticket track claim 觸發 --skip-verify deprecation warning 案例 = 0 → **觀察期不足無法定論**

條件 1 已滿足，但條件 2 的「觀察期」必須足夠長才能排除「使用者偶發使用」的可能。本次觀察期僅 ~25 分鐘，期間 ticket track claim 呼叫 3 次（皆無 --skip-verify），樣本太小無代表性。

#### 為何不直接 complete + spawn W4-015 + 不另行 release

按 ticket trigger 設計（decision-trigger-binding 規則 2 + PC-093）：
- 本 ticket 已完成本輪可執行範圍（baseline + 第一輪觀察）
- 下一輪觀察期 trigger 應綁定新 ticket（W4-015），由 W4-015 承擔
- 本 ticket complete 後，monitoring 責任交棒，不形成「無 trigger 延後」陷阱

#### Spawned Tickets

- **0.19.0-W4-015 (ANA, P3)**：第二輪 reevaluation，trigger 條件 = W4 wave 完成 3+ 個 ticket（含實際使用 ticket track claim 的開發週期）後啟動
  - 若達移除 trigger → 直接建落地 IMP（拆 verify 子命令 + 移除 --skip-verify + 條件分支簡化）
  - 若未達 → spawn 第三輪 W4-XXX reevaluation 並交棒
  - 持續觀察鏈設計：每輪 ANA 都有明確 trigger 落點，避免 PC-093 陷阱

#### 監測機制改善建議（給 W4-015 第二輪參考）

當前 hook-logs 無法精確區分「真實 claim 觸發 [Deprecation]」vs「Bash 命令字串副作用」。建議第二輪 ANA 執行時：
- 改用 ticket CLI 內建 telemetry（若未來實作）
- 或檢查 stderr 輸出有 [Deprecation] 標頭的實際 claim event（需 hook 配合記錄）
- 短期方案：手動詢問用戶 / 抽樣最近 3 個 wave 是否有 --skip-verify 使用記憶
