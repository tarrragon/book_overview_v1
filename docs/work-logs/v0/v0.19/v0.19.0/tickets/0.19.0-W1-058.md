## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 實作摘要

新建 `tests/unit/overview/import-flow-controller.test.js`，10 個測試覆蓋 ImportFlowController.execute() 4 條路徑 + DI 注入驗證。

### 設計決策

**1. 斷言觀測角度：side effect 而非回傳值**

execute() 回傳 `Promise<void>`，無 `reason` / `books` 回傳欄位（與 ticket how 描述的 "assert reason 與 books" 不完全對應，採實作真實契約為準）。4 條路徑差異透過 mock 呼叫觀察：

| 路徑 | 觀測點 |
|------|--------|
| 主路徑（overwrite） | replaceAllData 被呼叫 + onImportSuccess(books) 被呼叫 |
| 主路徑（merge） | mergeAllData 被呼叫 + onImportSuccess(books) 被呼叫 |
| 取消（null） | read / storage / onImportSuccess 皆未被呼叫，無 showError |
| modal_missing | showError 被呼叫（匯入功能初始化失敗），後續流程全中止 |
| storage_error | showError 被呼叫對應訊息（quota_exceeded 或一般 storage error） |

**2. Mock 策略：純 jest.fn() 不依賴 jsdom**

execute() 流程透過 promptImportModeFn callback 注入解耦 modal 互動，本測試不涉及真實 DOM，elements / document 給空物件即可。modal UX 測試由 import-mode-modal.test.js / empty-file-confirm-modal.test.js 覆蓋（範圍邊界）。

**3. DI 注入驗證採 3 個面向**

- constructor 接受 5 個必要依賴並綁定於實例屬性（黑箱驗證）
- 未注入 promptImportModeFn 時 fallback 至 this.promptImportMode（測試 fallback 路徑可達）
- validate throw 時流程立即中止（測試早期錯誤路徑）

### 範圍邊界（不在本 ticket）

- W1-049 空檔案二次確認分支（Modal B）→ empty-file-confirm-modal.test.js 已覆蓋
- Modal A / B 真實 DOM 互動細節 → 對應 modal 測試檔已覆蓋
- BookFileImporter 內部 validate / read 細節 → book-file-importer*.test.js 已覆蓋
