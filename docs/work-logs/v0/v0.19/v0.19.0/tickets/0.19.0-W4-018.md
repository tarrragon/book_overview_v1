### 候選方案（W 階段擴增）

#### 方案 P1：純清理（不調整規則）

| 動作 | 範圍 | 預期 |
|------|------|------|
| 不改 `package.json` `eslintConfig` | — | warnings 規則維持原狀 |
| Spawn cleanup IMP ticket | G1（src/ 14 console）+ G4（3 unused vars）| 預期 49 → 32 warnings |
| G2 / G3 不處理（保留 warn） | benchmark + Chrome mock | 永久顯示 32 warnings |

優點：規則最嚴格、不增加 overrides 維護負擔
缺點：G2 / G3 warnings 永遠存在，每次 `npm run lint` 都看到「假警報」，long-term 鈍化 warnings 訊號

#### 方案 P2：混合策略（合理 override + 真實清理）

| 動作 | 範圍 | 預期 |
|------|------|------|
| `package.json` `eslintConfig.overrides` 加兩條：(a) `tests/e2e/performance/**` disable `no-console`；(b) `tests/mocks/**` + `tests/unit/storage/adapters/**` disable `n/no-callback-literal` | G2 + G3 | 32 warnings 消除 |
| Spawn cleanup IMP ticket | G1（src/ 14 console → AppLogger）+ G4（3 unused vars 刪除） | 17 warnings 消除 |
| 結果 | 49 → 0 warnings |

優點：差異化規則合理（benchmark 本來就要 console、mock callback literal 是 API 簽名強制）、warnings 歸零、pre-commit hook 可 enforce 新警告
缺點：overrides 範圍需精準（路徑 glob 寫錯會誤豁免；應 IMP ticket 內含驗證步驟）

#### 方案 P3：嚴格化（all warn → error）

| 動作 | 範圍 | 預期 |
|------|------|------|
| `eslintConfig.rules` 將 23 條 warn 改 error | 整體 | 規則違反一律阻擋 |
| 同 P2 加 overrides | G2 + G3 | 豁免合理場景 |
| Spawn cleanup IMP ticket | G1 + G4 | 17 warnings 消除 |
| W4-023 `--max-warnings=0` 自然落地 | — | pre-commit hook 額外嚴格 |

優點：技術債紀律最強、未來新增違反立即阻擋（不等 W4-023 spike）
缺點：W4-023 spike 結論被預先決定（剝奪 W4-023 評估空間，違反 decision-trigger-binding）、整體開發體驗變嚴、若團隊未共識會引發 `--no-verify` 濫用（W4-017 邊界）
