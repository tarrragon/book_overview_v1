## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 實作摘要

依父 ticket 0.19.0-W4-031.2 §Phase 3a §2 控制流虛擬碼擴充 `src/export/format-version-detector.js`，新增 `detectInterchangeSource` 四來源 detector，並使既有 `detectFormatVersion` 委派收斂（H1 / ADR-2）。

| 變更 | 內容 |
|------|------|
| 新增 `detectInterchangeSource(data)` | 四來源優先序短路互斥（spec §8）：canonical(format===) → 內部 v2(metadata.formatVersion 2.) → app-legacy(backup_info/export_info wrapper + books) → flat v1(純陣列 / {books:[]}) → null |
| H1（ADR-2）委派收斂 | `detectFormatVersion` 委派 `detectInterchangeSource` 後 switch 收斂：'v2'/'v1' 原樣回傳；'canonical'/'app-legacy' → null（既有二值消費端不消費新來源，接線走 .2.2 新 detector） |
| H2 移除冗餘 `!format` | Rule 2（v2 判定）靠 return 順序保證互斥（Rule 1 命中 canonical 已 short-circuit return），移除原死條件，降認知負擔 |
| 抽出 `isLookupableObject` / `hasImplicitV2Signal` | 型別守衛與 v2 隱含偵測啟發式抽為具名子函式 |

### B5 回歸的設計決策（重要）

新 `detectInterchangeSource` 四來源**不含**既有 `detectFormatVersion` 的「v2 隱含偵測」啟發式（metadata + books 含 readingStatus → v2）。若採純委派，既有測試 fixture `{metadata:{source:'test'}, books:[{readingStatus:'reading'}]}` 將回 'v1' 而破壞 B5。

**解法**：將「v2 隱含偵測」定位為 `detectFormatVersion` 專屬啟發式（非四來源 canonical 辨識的一部分），保留為委派回 'v1'/null 後的 fallback（`hasImplicitV2Signal`）。此同時滿足：
- H1 / ARCH-020：四來源 canonical 判定邏輯單一真值來源於 `detectInterchangeSource`，無漂移
- B5：既有 `detectFormatVersion` 對 v1/v2/null 三類 fixture（含 v2 隱含）行為完全不變

### 邊界

- 僅改 detector 檔 + 自己的測試檔；未碰 adapter 檔（3b-B/.2.4 負責）與既有 `format-version-detector.test.js`（既有 describe 不動，B5 設計要求）。
- 計時斷言：無計時硬門檻（純判定函式值比對，遵循 test-assertion-design-rules）。
