### 修復摘要

W1-083 反轉 build.js sanity check SSOT，落實 W1-077 ANA 推薦方向 A：
- **舊邏輯**（W1-074）：worklog `**狀態**: 開發中` 標記為 SSOT → 推斷期望版號 → 對比 package.json
- **新邏輯**（W1-083）：package.json 為唯一版本 source of truth → 確認對應 worklog 目錄與 main.md 存在 → 自動 sync 標記為「開發中」

落實 Linus 原則「Don't add a check for the thing humans forget to do. Make it impossible to forget.」

### 實作細節

| 函式 | 變更性質 | 行為 |
|------|---------|------|
| `resolveWorklogPath` | 新增 | 純函式推斷 `0.19.0` -> `v0/v0.19/v0.19.0/v0.19.0-main.md` |
| `validateVersionAlignment` | 反轉斷言方向 | 以 package.json 為源 → 確認 worklog 目錄與 main.md 存在，不存在 fail-fast |
| `syncWorklogStatus` | 新增 | 自動將對應 main.md 的「**狀態**:」標記同步為「開發中」（changed=true/false 區分行為） |
| `detectActiveWorklogVersion` | 保留 | 公用工具，供 version-release skill 偵測使用，不再為 build sanity check 主入口 |
| `defaultFsWriter` | 新增 | 寫入注入介面（與 fsAdapter 讀寫分離） |

### Build flow 整合

production mode 兩階段：
1. `validateVersionAlignment`：結構性存在斷言（patch dir + main.md 都要存在）
2. `syncWorklogStatus`：自動 sync 標記，補救「狀態漂移」（changed=true 時 log 變更）

`--skip-version-check` 同時略過兩階段，dev 期短暫不一致時使用。

### Version-release skill 整合評估

**結論**：不擴展 `version-release` skill。

**理由**：
- `version-release start` 子命令已負責「建立 worklog 目錄 + bump package.json + 寫入新 main.md」的正向流程，會帶入正確的「**狀態**: 開發中」標記
- 本 ticket 的 `syncWorklogStatus` 是「補救路徑」——當開發者跳過 skill（如 W1-073 已手動 bump 過）時的兜底機制，職責與 skill 的「建立 / 發布」流程語意不同
- 兩者形成互補：skill 處理正向 lifecycle，build.js 處理 build 階段的最後對齊保證
- 強制整合會使 build.js 依賴外部 skill 執行環境（Python 3.10+ / uv），破壞 build 純 Node.js 的部署簡潔性

### 既有 bug 註記（W1-083 範圍外）

`detectActiveWorklogVersion` 的 `[::]` regex 註解聲稱「容忍全形冒號」但實際只有兩個半形冒號（U+003A），屬 W1-074 implementation bug。本 ticket 在 `syncWorklogStatus` 採顯式 `[:：]`（U+003A + U+FF1A）真正支援全形冒號，並在註解標記既有 detect 函式的 bug 待後續 ticket 處理（避免擴大本 ticket 範圍超過 SSOT 反轉）。

### 設計決策：fsAdapter / fsWriter 讀寫分離

寫入介面獨立 (`defaultFsWriter`) 而非沿用 `fsAdapter` 加 `writeFile` 方法的理由：
- 讀寫責任分離：驗證測試無需 mock 寫入，sync 測試無需 mock 讀取的全部介面
- 認知負擔降低：閱讀 `validateVersionAlignment` 簽名不會誤以為它會寫檔
- 與 `script-test-helpers.js` 設計哲學一致（最小介面 vs 完整介面分工）
