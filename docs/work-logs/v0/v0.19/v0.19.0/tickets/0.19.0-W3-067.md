### 問題根因

W3-041 接手過程的關鍵 turn point 分析：

| 步驟 | 動作 | 觸發來源 |
|------|------|---------|
| 1 | PM 切換到 W3-041，read ticket 規格 | PM 自發 |
| 2 | PM 注意 where.files = `.claude/hooks`（目錄級而非具體檔案）| PM 警覺 |
| 3 | PM 執行 find `*pre-fix-eval*` → 找不到 .py（只剩 .pyc）| PM 自發 |
| 4 | **用戶提示「查 commit 確認何時/為何移除」** | **用戶介入** |
| 5 | PM 執行 git log --diff-filter=D → 找到 W10-001 (2026-04-07) | 用戶提示後 |

**關鍵發現**：步驟 4 是用戶介入。沒有此提示，PM 可能停在「假設 hook 被移除 → 關閉 ticket」或「直接派 agent 自己想辦法」。

### 影響範圍

PM 接手所有「描述含 hook / 模組 / 檔案」的 ticket，特別是：
- where.files 為目錄級而非具體檔案
- ticket created 距今 > 30 天
- ticket 描述跨月份引用既有環境狀態

### 接手考古 SOP 設計（草案，落地需 IMP/DOC 子任務）

#### 觸發識別（PM 接手 ticket 時自問）

任一成立即進入 SOP：

- where.files 是目錄級（如 `.claude/hooks`）而非具體檔案？
- ticket 描述包含具體 hook / 模組 / 檔案名？
- ticket created 距今 > 30 天且引用既有環境狀態？
- 接手第一次 ls / grep 驗證 where.files 發現不存在或不一致？

#### SOP 三階段

**階段 1: 考古（最多 5 個指令）**

```bash
# 1.1 驗證 where.files 實際存在
ls -la <ticket where.files 列出的每個路徑>

# 1.2 若描述含 hook，驗證註冊狀態
grep "<hook-name>" .claude/settings.json

# 1.3 不存在 → git log 考古
git log --all --diff-filter=D --name-only --pretty=format:"%h %ad %s" --date=short -- <檔名>

# 1.4 找出邏輯實際位置（grep）
grep -l "<關鍵字>" .claude/ -r --include="*.py"

# 1.5 看相關 commit 完整訊息（理解 root cause）
git show <commit-sha> -- <檔名>
```

**階段 2: 釐清三問**

| 問題 | 行動 |
|------|------|
| 何時改變？ | 確認 commit 日期 + 對應 ticket |
| 為何改變？ | 讀 commit message + 對應 ticket 結論 |
| 邏輯實際在哪？ | 新位置（檔案 + 行範圍 + 註解標記） |

**階段 3: 需求重評**

| 評估維度 | 判定 | 行動 |
|---------|------|------|
| 原需求是否仍有效？ | 是 → 修正 where.files | 階段 4 派發 |
| 原需求是否已被解？ | 是 → 關閉 ticket | 補 Problem Analysis 說明 + close |
| 原需求是否需重寫？ | 是 → 重寫 acceptance | set-acceptance + 補 strategy |

**階段 4: 落地**

- 修 where.files（具體檔案路徑非目錄）
- ticket Problem Analysis 補「前提驗證考古結果」段（含三問答案 + git log 證據）
- 派發或關閉

#### 規則層級的「最低要求」（PM 自我約束）

PM 接手任一 ticket 強制四問（即使 SOP 觸發條件未明確命中）：

1. ticket where.files 列出的所有路徑當下都存在嗎？（ls 驗證）
2. ticket 描述提到的具體名稱（hook/檔案/模組）當下還在嗎？（grep / find 驗證）
3. ticket created 距今超過 30 天嗎？若是，引用的環境是否仍一致？
4. 我是否依「過去經驗 / memory 中既有引用」而非當下驗證在判斷？

任一答「否」或「不確定」→ 進入 SOP 階段 1。
