```markdown
## Acceptance 設計規範

### 核心原則：Complete-Time Verification Semantics

Acceptance 欄位表達的是 **complete 時的驗收條件**，而非 claim 時的早期回饋。

**為何**：L3-b 實作後，claim 階段移除 AC verification，所有驗收測試延遲到 complete。若 acceptance 寫「npm test 100%」，應理解為「complete 時驗收 npm test 100% 通過」，而非「claim 時驗證」。

**邊界**：
- 包含測試驗收的 acceptance（如「npm test 全綠」「pytest 通過」）→ **完整-time** 驗收條件
- 包含中間檢查點的 acceptance（如「檔案結構驗證」「linter 無錯誤」）→ 可在 claim 驗證，亦可延遲至 complete（由 acceptance 字面決定）
- 包含功能行為的 acceptance（如「3 個檔案的測試通過」「特定模組未回歸」）→ 建議明示驗收範圍（相對檔案測試 vs 全套件）

### 撰寫建議

避免歧義表述，遵循以下範例：

| 反模式 | 建議改為 | 說明 |
|-------|--------|------|
| `npm test 100% 通過` | `complete 時驗收：npm test 100% 通過` | 明示驗收時機 |
| `全套件測試無回歸` | `相關檔案測試（5 個 test.js）通過` | 明示驗收範圍（避免全套件衝突） |
| `測試通過率 100%` | `complete 時驗收：npm test 綠燈（0 failed）` | 使用具體指標 |
```

### 撰寫原則

| 原則 | 示例 |
