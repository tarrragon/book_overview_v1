### 問題根因

W4-017 phase4 Plan 視角發現：CLAUDE.md §5 宣稱「pre-commit hook 為 local first line，CI 為 safety net」，但 repo 無 `.github/` 目錄、無任何 workflow。雙重防線目前單靠本機 pre-commit hook（lint-staged --max-warnings=0），`--no-verify` 繞過後無遠端防線接住，違規可直接流入 main。

### 影響範圍

- 新增：`.github/workflows/lint.yml`（全新建立，無既有 ci.yml 可拆分——PM 已驗證前提修正）
- 修改：`CLAUDE.md` §5（補 CI Lint safety net 段落，指向 workflow 檔名）
- 修改：`docs/development-setup.md`（補 CI Lint 行為說明 + required status check 啟用步驟）

### 相關 Error Pattern

W4-017 phase4 P-1「文件宣稱但實作不存在」模式：避免文件描述 safety net 卻無對應實作。本 ticket 同時建立實作（workflow）與文件（CLAUDE.md / dev-setup）以閉合此 gap。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 變更摘要

| 檔案 | 變更 |
|------|------|
| `.github/workflows/lint.yml`（新增） | GitHub Actions lint workflow：trigger `push`（main / feat/**）+ `pull_request`；單一 `lint` job：checkout → setup-node（lts/* + npm cache）→ `npm ci --legacy-peer-deps` → `npm run lint`。含 `concurrency` 取消同 ref 進行中舊 run |
| `CLAUDE.md` §5 | 新增「CI Lint（GitHub Actions safety net）」段落，指向 `.github/workflows/lint.yml` 檔名（閉合 W4-017 phase4 P-1「文件宣稱但實作不存在」）|
| `docs/development-setup.md` | 新增「CI Lint（GitHub Actions）」章節：workflow 行為表、PR 期望、required status check 的 repo admin 啟用步驟 |

### 設計決策

- **全新建立 workflow**（非拆既有 ci.yml）：PM 已驗證 repo 無 `.github/` 目錄、無 ci.yml，strategy 原「ci.yml 拆 lint job」前提不成立，改建全新 `lint.yml`。
- **Node 版本 `lts/*`**：專案無 `.nvmrc`、package.json 無 `engines`，依 ticket 指引採 LTS。
- **`npm ci --legacy-peer-deps`**：對齊專案環境恢復標準（CLAUDE.md §4.1）的 peer deps 處理。
- **`lint` 為獨立 job**：未與 build/test 耦合（且本 repo 尚無 build/test workflow），符合 acceptance 2「lint 為獨立 job」。

### 邊界（明示，勿誤判為已完成）

- **required status check 無法由 workflow 檔本身啟用**：屬 GitHub branch protection 設定，需 repo admin 於 GitHub Settings → Branches 操作。本 ticket acceptance 1 的「required status check」以「workflow 建立 + docs 記錄啟用步驟」達成；**實際 enable 屬 PM/用戶後續操作，本 ticket 未啟用**。

### 驗證方式與結果

本 ticket 無法在本地實際觸發 GitHub Actions run，驗證限於 YAML 語法正確性 + workflow 邏輯合理性。

**1. YAML 語法驗證**（actionlint 環境不可用，改用 python yaml）

```bash
python3 -c "import yaml; d=yaml.safe_load(open('.github/workflows/lint.yml')); print('YAML OK'); print('jobs:', list(d['jobs'].keys()))"
```

結果：
```
YAML OK
jobs: ['lint']
triggers: ['push', 'pull_request']
```

說明：PyYAML 將 `on:` 解析為 boolean key `True`（YAML 1.1 已知 quirk，無害——GitHub Actions 解析器正確識別為 `on`）。job 名稱 `lint`、trigger `push` + `pull_request` 均符合預期。

- trigger：push（main / feat/**）+ pull_request — 符合 acceptance 2

- 安全性：無 untrusted event input 流入 run:/ref:（僅 github.ref 用於 concurrency group），無 command injection 風險
