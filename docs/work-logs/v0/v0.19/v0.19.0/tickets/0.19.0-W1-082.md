## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

兩處 tag ID 產生邏輯採用不同格式：

| 來源 | 格式 | 範例 |
|------|------|------|
| tag-storage-adapter.js:406 (互動建立) | `tag_{timestamp}-{random6}` | `tag_1716648000000-a3f8k2` |
| tag-storage-adapter.js:1046 (合併匯入) | 同上（nextTagId） | 同上 |
| v1-to-v2-converter.js:110 (批次匯入) | `tag_{tsMs}-{paddedIndex}` | `tag_1716648000000-003` |

後綴語意不同：random6 提供全域唯一性；paddedIndex 提供同批次內排序但跨批次可能碰撞。

### 影響範圍

- `src/storage/adapters/tag-storage-adapter.js`：互動建立 + 合併匯入（2 處）
- `src/export/v1-to-v2-converter.js`：v1 資料批次轉換（1 處）
- 消費者：所有讀取 tag.id 的模組（overview-page-controller、匯出、搜尋）僅做 exact match，不解析格式

### 相關 Error Pattern

無直接相關 error-pattern。屬跨模組設計不一致（低風險），非 bug。

## 重現實驗結果
### 實驗方法

靜態分析 src/ 中所有 tag ID 產生點與消費點，確認格式差異與碰撞風險。

### 實驗執行

1. `rg "tag_.*Date\.now|nextTagId" src/` → 3 處產生點（storage-adapter 2 處 + converter 1 處）
2. `rg "tag_" src/` → 消費者僅做 exact match（map lookup / filter by id），不解析後綴格式
3. 碰撞分析：converter 使用 `tsMs-paddedIndex`，同一毫秒內匯入兩批（不同檔案）理論上可產生相同 ID（如 `tag_1716648000000-000`），但實務上 v1→v2 轉換只執行一次

### 實驗發現

已驗證事實：
- 兩種格式共存但消費者不依賴格式結構（opaque ID）
- converter 的 paddedIndex 在單次執行安全，跨次碰撞需同毫秒（極低機率）
- storage-adapter random6 碰撞機率：~1/2.18B（36^6），實務無風險

未驗證假設：
- 未來是否有 tag ID sort-by-creation 需求（目前無，sortOrder 獨立欄位）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
multi_view_status: single_analyst_with_dissenting

### 場景需求差異

| 維度 | 互動建立 (storage-adapter) | 批次匯入 (converter) |
|------|--------------------------|---------------------|
| 唯一性保證 | 強（random6 = 2.18B 空間） | 弱（同 ms 跨批可碰撞） |
| 排序語意 | 無（靠 sortOrder 欄位） | 隱含批次內順序（paddedIndex） |
| 除錯辨識 | 可由 timestamp 推建立時間 | 可由 index 推原始順序 |
| 執行頻率 | 高（用戶每次手動建 tag） | 極低（v1→v2 一次性遷移） |

### 三方向 Trade-off 表格

| 維度 | A: 保持現狀 | B: 統一為 random6 格式 | C: 加 source-marker 前綴 |
|------|------------|----------------------|------------------------|
| 唯一性 | 互動強/匯入弱 | 全強 | 全強（random6 + marker） |
| 既有資料相容 | 無破壞 | 需 migration 改既有匯入 ID | 需 migration |
| 實作成本 | 零 | 低（改 converter 1 行） | 中（改格式 + migration） |
| 業務語意 | 無差異（opaque ID） | 無差異 | 可辨來源但無人消費 |
| 碰撞實際風險 | 極低（converter 一次性） | 零 | 零 |
| 維護負擔 | 兩套格式但無人解析 | 單一格式 | 增加格式複雜度 |

### 推薦方向

**推薦 A：保持現狀。**

理由：
1. 消費者對 tag ID 為 opaque 使用（exact match only），格式不一致不影響功能
2. converter 為一次性 v1→v2 遷移，不會重複執行，碰撞風險趨近於零
3. 統一格式需 migration 處理既有資料，ROI 極低（解決不存在的問題）
4. v0.19.0 處於內測準備階段，不應引入不必要的 migration 風險

### Dissenting View

統一格式（方案 B）的支持論點：
- 程式碼一致性是長期維護的基本衛生；現在不統一，未來有人抄 converter 格式用於其他場景時可能引入碰撞
- 實作成本低（converter 改 1 行 + 無需 migration 因為既有 ID 不改，只改未來新產生的 ID）

反駁：converter 是 dead path（v1→v2 一次性），不會被複製為新場景模板。若未來新增批次產生場景，應在該時點直接使用 storage-adapter 的 nextTagId 工廠函式（已提供 L1046），而非從 converter 複製。

### Spawn 規劃

推薦 A（保持現狀），**無需 spawn IMP ticket**。格式不一致為低風險設計 note，不構成需修復的技術債。
