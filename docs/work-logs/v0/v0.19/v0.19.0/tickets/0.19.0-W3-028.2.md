## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W3-028 ANA 結論識別出「SessionStart event 是否在 /resume bg session 觸發」屬未驗證假設（W3-028.md WRAP A 階段「待驗證」項目）。靜態分析顯示：

- Claude Code v2.1.144+ 引入對 bg session 的 `/resume` 支援，但其 SessionStart event 觸發行為未在 Claude Code 文件中明示
- 既有 SessionStart hooks（session-start-scheduler-hint-hook 等）無條件執行，不感知 source 欄位（startup / clear / resume）
- 當前 session 由 `/clear` 觸發，hook 收到 source="clear"（從 system-reminder「SessionStart:clear hook success」推論）
- /resume bg session 後 SessionStart event 是否觸發 source="resume"，須實機觀察

### 影響範圍

| 檔案 / 元件 | 用途 | 與本 ticket 關聯 |
|------------|------|----------------|
| .claude/hooks/ | SessionStart hook 群（19 個） | 觀察 hook 在 /resume 後是否觸發是核心 |
| .claude/hook-logs/ | hook 觸發紀錄 | 證據來源（log file mtime + content） |
| .claude/settings.json | SessionStart event 註冊 | 新 diagnostic hook 需註冊 |
| .claude/pm-rules/session-switching-sop.md | bg session resume 場景章節（W3-028.1） | 待驗證標記依本 ticket 結論更新 |

### Context Bundle

#### Hook event payload 規格（Claude Code 平台）

SessionStart event 的 stdin payload 結構（依 Claude Code hooks 規範推論）：

```json
{
  "session_id": "<uuid>",
  "hook_event_name": "SessionStart",
  "source": "startup" | "resume" | "clear",
  "transcript_path": "<path>",
  "cwd": "<path>"
}
```

關鍵欄位：`source` 區分三種 session 啟動類型。

#### 既有 hook 對 source 的處理

執行 `grep -rn '"source"' .claude/hooks/` 結果：所有既有 SessionStart hook 皆**不讀取 source 欄位**。Hook 無條件執行業務邏輯（runqueue 提示、依賴檢查等），故無法從現有 hook log 倒推 source 來源。

#### 驗證障礙

PM / subagent 無法在當前 session 內觸發 `/resume bg session`：
- `/resume` 是 Claude Code runtime 用戶層指令
- bg session 是用 `run_in_background: true` 啟動 Agent，subagent 本身結束時 session 即結束
- 須由用戶在新 Claude Code session 中執行 `/resume <bg_session_id>` 才會觸發 resume 路徑

#### 實驗設計

| 步驟 | 動作 | 觀察點 |
|------|------|--------|
| 1 | 撰寫 diagnostic hook 記錄 source 到 log | hook 註冊 + log 機制就緒 |
| 2 | 在 settings.json SessionStart event 註冊該 hook | hook 在 next SessionStart 啟動 |
| 3 | 用戶啟動一個 bg session（透過 spawned background agent） | bg session 進入背景 |
| 4 | bg session 結束後，用戶用 `/resume <session-id>` 恢復該 bg session | 觸發 SessionStart event（或不觸發） |
| 5 | 觀察 diagnostic hook log | 若有紀錄且 source="resume" → 觸發；若 log 無新紀錄 → 不觸發 |
| 6 | 結論寫入 ticket Solution + 更新 session-switching-sop.md | 將「待驗證」標記改為確定狀態 |

#### Diagnostic hook 規格（≤ 30 行 PEP 723 UV single-file script）

- 路徑：`.claude/hooks/session-source-diagnostic-hook.py`
- 觸發：SessionStart event
- 邏輯：讀 stdin JSON → 取 `source` 欄位 → append 一行 `[timestamp] source=<value> session_id=<truncated>` 到 `.claude/hook-logs/session-source-diagnostic/diagnostic.log`
- 失敗模式：靜默（依規則 4 雙通道，stderr + log；但本 hook 為純觀察，無業務阻擋）
- 移除時機：W3-028.2 結論落地後可移除（屬一次性實驗，非長期 framework 資產）

#### 預期結論分支

| 觀察結果 | 結論 | 後續動作 |
|---------|------|---------|
| /resume bg session 觸發 SessionStart 且 source="resume" | 行為符合預期；runqueue hint 在 resume 後自動出現 | 更新 session-switching-sop.md：移除「待驗證」標記 |
| /resume bg session 不觸發 SessionStart | runqueue hint 在 resume 後缺失 | 建議補救：(a) spawn IMP 為 PostToolUse 或其他 event 補觸發；(b) 更新 SOP 強調手動執行 `ticket track runqueue` |
| 觸發但 source="clear" 或其他值 | Claude Code 行為與假設不同 | 詳細分析並建議補救 |

實作 SessionStart source diagnostic hook 用於觀察 /resume bg session 是否觸發 SessionStart event。

**新增檔案**：

### 實機驗證結果（2026-05-26，Claude Code v2.1.150）

完成多路徑實機驗證實驗，收集 8 筆 SessionStart event 紀錄，涵蓋 cold start / clear / attach / resume 全部已知進入路徑。

**實驗流程與觀察**：

| Step | 動作 | 觀察結果 |
|------|------|---------|
| 1 | `claude --background "echo hello"` 啟動 bg session（id `7af4155b-bd8d-4008-b331-1389b5cf307e`） | log 寫入 `source=startup session_id=7af4155b` |
| 2 | `/resume 7af4155b`（短 ID） | runtime 回應 `Session 7af4155b was not found.`，不觸發 event |
| 3 | `/resume 7af4155b-bd8d-...`（全 UUID，bg running） | runtime 回應 `That session is still running as a background agent. Open claude agents to attach to it, or stop it there first to resume here.`，不觸發 event |
| 4 | `claude attach 7af4155b` | log 寫入 `source=clear session_id=e8141123`（**新子 UUID，不繼承 bg UUID**） |
| 5 | `claude stop 7af4155b` → `claude` → `/resume 7af4155b-bd8d-...` | log 兩行：`source=startup session_id=56b71125`（claude 啟動）與 **`source=resume session_id=7af4155b`**（/resume 觸發，**繼承 bg UUID**） |

**完整 SessionStart source 對照表**（v2.1.150 實證）：

| 進入路徑 | source 值 | session_id 行為 |
|---------|----------|---------------|
| `claude` cold start | `startup` | 新 UUID |
| `claude --background "..."` 建立 bg | `startup` | 新 UUID |
| `/clear` 在現有 session | `clear` | 維持原 UUID |
| `claude attach <bg-id>` | `clear` | 新子 UUID（不繼承 bg UUID） |
| `/resume <running-bg-id>` 全 UUID | runtime 拒絕（無 event） | N/A |
| `/resume <stopped-bg-id>` 全 UUID | `resume` | 繼承原 bg UUID |
| `/resume` 短 ID（任何狀態） | runtime 拒絕（無 event） | N/A |

**對 runqueue hint 系統的影響評估**：

所有實際可進入 session 的路徑（startup / clear / resume），SessionStart hooks 都會跑（含 `session-start-scheduler-hint-hook`），因此 runqueue hint 在所有可行接手場景均會出現。

**Acceptance #3「若不觸發則建議補救方案」條件不成立**——hooks 在所有可進入路徑均觸發，無需補救方案。

**attach vs /resume 設計差異揭露**：

| 維度 | `claude attach` | `/resume` |
|------|----------------|-----------|
| 適用 bg 狀態 | running | stopped（runtime 拒絕 running） |
| source 值 | `clear` | `resume` |
| session_id | 新生 child UUID | 繼承原 bg UUID |
| 語義 | 「進來看看」（不繼承狀態） | 「接管原狀態」 |
| 短 ID 支援 | 是 | 否（需全 UUID） |

### 實驗工具自指涉觀察（設計洞察）

W3-028.2 實驗本身產生了一筆**非實驗目的**的 log 紀錄：

- `[2026-05-26T11:44:25] source=clear session_id=60a6a197` ← 本 PM session 在 11:44 `/clear` 啟動時觸發。此時 diagnostic hook 已部署但**尚未進入實驗模式**，純粹被「正常工作流」觸發

此自指涉觀察證實兩件設計上的事實：

1. **Diagnostic hook 的可重入性**：hook 在日常使用情境（非實驗）下也會被觸發。實驗工具不只在實驗時運作，也持續觀察工作流——延伸了實驗證據基底，並驗證 hook 不干擾正常工作流（11:44 之後 PM 繼續所有 Edit/Bash/Read 操作未受影響）
2. **觀察工具的雙重身份**：diagnostic hook 既是「實驗工具」也是「長期 telemetry」。本 ticket「後續動作」決定保留 hook 為長期資產，此自指涉觀察為該決定提供了具體證據——hook 在實驗結束後仍持續產出可用資料（未來 source 值演化可被偵測）

**可推廣設計原則**：設計觀察類工具時，主動讓工具可觀察自身的「日常使用」，能在零額外成本下取得正常情境對照基底，並驗證工具本身對工作流的零侵入性。
