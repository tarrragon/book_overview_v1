# Execution Log

## Task Summary

評估 worktree.bgIsolation: "none" 新設定與 background sessions 自動 worktree 行為對專案 worktree skill / branch-worktree-guardian 規則的影響

---

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

Claude Code v2.1.143 新增 `worktree.bgIsolation` 設定，允許值為 `"worktree"`（預設）或 `"none"`。設為 `"none"` 時 background sessions（即 subagent）直接在 working copy 上操作，不建立 git worktree。這改變了專案長期仰賴的隔離假設。

### 影響範圍

| 受影響模組 | 影響程度 |
|-----------|---------|
| `.claude/pm-rules/worktree-operations.md` | 高——SOP 預設 `isolation: worktree` 為唯一合法隔離方式 |
| `.claude/skills/worktree/SKILL.md`（agent isolation 章節） | 中——殭屍 GC hook 邏輯仍有效但覆蓋範圍縮小 |
| `.claude/skills/branch-worktree-guardian/SKILL.md` | 低——保護分支規則不依賴 worktree 機制 |
| `.claude/hooks/worktree-zombie-cleanup-hook.py` | 低——bgIsolation none 時不產生 agent worktree，hook 自然無目標 |
| `.claude/pm-rules/parallel-dispatch.md` | 高——並行安全檢查假設 worktree 隔離 git index |
| ARCH-015 / PC-137 互動 | 關鍵——bgIsolation none 可能繞過 .claude/ Edit deny（待驗證） |

### 相關 Error Pattern

- PC-137：並行 3+ subagent 對 .claude/ Edit deny。bgIsolation none 讓 subagent 直接在主 repo working copy，可能改變 deny 行為（未驗證）。
- ARCH-015：subagent .claude/ hardcoded 寫入保護。保護是否綁 worktree cwd 還是 subagent 身份，是本 ANA 關鍵未知。
- PC-092：並行 commit 邊界混亂。bgIsolation none 下多 subagent 共享 git index，PC-092 風險從「理論」升級為「必然」。
- PC-019：worktree 合併流程。bgIsolation none 下不需合併，PC-019 SOP 不適用。
## 重現實驗結果

### 實驗方法

本 ANA 為「平台升級影響評估」，非 bug 重現。實驗方法為：

1. **文件盤點**：Read 現有 worktree 相關模組（worktree skill、branch-worktree-guardian skill、worktree-operations SOP、parallel-dispatch 規則、PC-137/ARCH-015 error patterns），提取所有「假設 worktree 為隔離機制」的斷言
2. **設定空間分析**：對照 v2.1.143 changelog 的 bgIsolation 選項（worktree / none），逐一評估每個斷言在 none 模式下是否仍成立
3. **交叉比對**：將 ARCH-015（.claude/ Edit deny 綁 worktree cwd 假設）與 bgIsolation none（subagent 在主 repo cwd）交叉，推導可能行為變化

### 實驗執行

**盤點結果——現有 worktree 隔離假設清單**：

| 假設位置 | 假設內容 | bgIsolation none 時是否成立 |
|---------|---------|---------------------------|
| worktree-operations.md §核心原則 | 「先 commit 再派發」防 stash 丟失 | 不適用——none 模式不做 stash/checkout |
| worktree-operations.md §入口選擇 | 非 .claude/ 實作預設 isolation: worktree | 需改為條件式——none 模式下無 worktree |
| worktree-operations.md §ARCH-015 | .claude/ Edit 不在 worktree 進行 | 可能失效——none 模式 subagent 已在主 repo |
| parallel-dispatch.md §安全檢查 | 實作代理人使用 isolation: worktree 派發 | 需改為條件式 |
| parallel-dispatch.md §PC-092 防護 | 禁用 git add ./git add -A | none 模式下更加關鍵（共享 index） |
| worktree skill §殭屍問題 | agent worktree 累積需 GC | none 模式下無此問題 |
| PC-137 §防護 | .claude/ 並行 ≤ 2 避免 deny | none 模式下 deny 行為未知 |

**ARCH-015 交叉分析**：

ARCH-015 的 hardcoded deny 根因未完全確認（PC-115 調查中）。兩個假設：
- 假設 A：deny 綁定 worktree cwd（subagent cwd 非主 repo 時觸發）→ bgIsolation none 可能解除 deny
- 假設 B：deny 綁定 subagent 身份（任何 subagent 對 .claude/ Edit 都受限）→ bgIsolation none 不改變行為

W17-108/109 實驗顯示「冷快取下 subagent .claude/ Edit 也 success」（非並行），支持假設 A 但不排除假設 B 在並行場景的作用。

### 實驗發現

**已驗證事實**：
1. 專案現有 6+ 個文件/規則假設 worktree 為唯一隔離方式
2. bgIsolation none 消除殭屍 worktree 問題（收益）
3. bgIsolation none 使並行 subagent 共享 git index，PC-092 風險從「需防護」升級為「必然發生」（風險）
4. 專案目前未在 settings.json 設定 bgIsolation（使用預設值 worktree）

**未驗證假設**：
- ARCH-015 deny 在 bgIsolation none 下的行為（需實驗，本 ANA 為 read-only 不執行）
- bgIsolation none + 並行派發的 git index 競爭是否導致 data corruption（需實驗）
## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### WRAP 框架分析

#### W — Widen Options

三種 bgIsolation 策略：

| 策略 | 設定值 | 說明 |
|------|-------|------|
| A. 維持現狀（嚴格 worktree） | 不設定（預設 worktree） | 所有 subagent 繼續用 worktree 隔離 |
| B. 全面切換 none | bgIsolation: none | 所有 subagent 直接操作 working copy |
| C. 條件式——依任務類型決定 | 不改預設；特定任務 per-dispatch override | .claude/ 修改用 none（可能解除 ARCH-015 deny）；src/ 並行用 worktree（保留 index 隔離） |

影響面展開：

**影響面 1：並行派發 isolation 預設值**
- 策略 A：現行 parallel-dispatch 安全檢查不變
- 策略 B：並行 subagent 共享 git index，PC-092 風險必然化；需強化 git staging 精準規則或改為序列
- 策略 C：分場景處理，複雜度最高但風險最低

**影響面 2：branch-worktree-guardian hook 邏輯**
- 策略 A/C：hook 邏輯不變
- 策略 B：殭屍 GC hook 無目標（正面）；branch-verify-hook 需確認 subagent 在 main 上操作時的分支保護是否仍有效

**影響面 3：ARCH-015 .claude/ Edit 邊界**
- 策略 A：ARCH-015 限制不變，.claude/ 修改仍需主 repo 前台
- 策略 B/C：若 ARCH-015 deny 綁定 worktree cwd（假設 A），bgIsolation none 可能解鎖 subagent .claude/ Edit 能力，這將根本改變 PC-137 防護策略
- 此為最大未知：需實驗驗證

#### R — Reality-test

| 維度 | 現狀 | bgIsolation none 後 |
|------|------|---------------------|
| 殭屍 worktree | 每 session 累積，需 GC hook | 無此問題 |
| 並行 src/ 修改 | worktree 隔離 git index，安全 | 共享 index，git add 競爭 |
| .claude/ Edit（序列） | 非並行時 success（W17-108/109） | 預期不變 |
| .claude/ Edit（並行 3+） | deny（PC-137 7/7） | 未知——可能解鎖或不變 |
| 合併流程成本 | 每次 subagent 完成需 merge/cp | 不需合併（直接在 working copy） |
| PC-019 stash 風險 | 需「先 commit 再派發」 | 不適用 |

#### A — Attain distance

**策略 A 收益/風險**：
- 收益：零變更成本；所有既有規則/hook 不需修改
- 風險：持續承受殭屍 worktree 管理成本（已有 GC hook 緩解）；ARCH-015 限制仍存在

**策略 B 收益/風險**：
- 收益：消除殭屍 worktree；消除合併流程成本；可能解鎖 .claude/ Edit（巨大收益——PC-137 防護可簡化）
- 風險：並行 git index 競爭（高）；PC-092 風險必然化（高）；需大量規則更新（中）

**策略 C 收益/風險**：
- 收益：.claude/ 修改場景可能解鎖（若假設 A 成立）；src/ 並行保留隔離
- 風險：規則複雜度增加（PM 需判斷每次派發用哪種模式）；需實驗確認 per-dispatch override 是否可行

#### P — Prepare to be wrong

| 失敗情境 | 回退方案 |
|---------|---------|
| 策略 B 導致 git index corruption | 立即移除 bgIsolation 設定，回到預設 worktree |
| 策略 C 的 per-dispatch override 不存在 | 回退到策略 A |
| ARCH-015 deny 在 none 模式下仍存在 | 策略 B/C 的 .claude/ 解鎖收益不成立，回到策略 A |

### 策略對照表

| 維度 | A. 維持 worktree | B. 全面 none | C. 條件式 |
|------|-----------------|-------------|----------|
| 變更成本 | 零 | 高（6+ 文件需更新） | 中（新增判斷邏輯） |
| 殭屍管理 | 需 GC hook | 無需 | 部分需 |
| 並行 src/ 安全性 | 高（index 隔離） | 低（共享 index） | 高 |
| .claude/ Edit 解鎖 | 否 | 可能（需驗證） | 可能（需驗證） |
| 合併成本 | 每次需合併 | 無 | 部分需 |
| 規則複雜度 | 不變 | 需全面更新 | 增加判斷邏輯 |
| 回退難度 | N/A | 低（移除設定） | 低 |

### 建議

**短期（v0.19.x）：採策略 A（維持現狀）**

Why：bgIsolation none 的最大收益（.clone/ Edit 解鎖）取決於 ARCH-015 deny 根因假設 A 是否成立，這是未驗證假設。在未驗證前切換會引入並行 git index 競爭風險，且需更新 6+ 個規則文件。

**中期：建 IMP 實驗 ticket 驗證 ARCH-015 假設**

設計受控實驗：bgIsolation none + 單一 subagent .claude/ Edit，對比現行 worktree 模式，確認 deny 行為是否改變。若假設 A 成立（deny 消失），則策略 C 成為最佳選擇。

**長期（假設 A 成立後）：採策略 C（條件式）**

.claude/ 修改派發用 bgIsolation none（解鎖 Edit）；src/ 並行派發維持 worktree（保留 index 隔離）。需更新 worktree-operations.md 和 parallel-dispatch.md。

### branch-worktree-guardian skill 評估

branch-worktree-guardian skill 的核心邏輯（保護分支禁止直接編輯、命名規範）不依賴 bgIsolation 設定。無論採哪種策略，該 skill 不需更新。
