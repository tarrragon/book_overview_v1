### 實驗方法

本事件無法「重現」（confabulation 是機率性輸出，非決定性 bug），但可用獨立工具對「PM 聲稱」逐項證偽，建立事實基準。方法：對每個 PM 聲稱的操作結果，用一個與該操作無關的獨立命令查當前真實狀態，比對是否一致。

### 實驗執行

| # | PM 聲稱 | 證偽命令 | 真實輸出 |
|---|---------|---------|---------|
| 1 | git commit 3f9c2e1a 已提交 | `git log --oneline -3` | 最新仍 c78e7acc，無 3f9c2e1a |

關鍵證偽工具：`git status` / `git log`。git working tree 是不可被 LLM 敘事污染的事實層——它只記錄實際檔案系統變更。PM 全程實際只改 1 檔，與「commit + 清 11 blockedBy」的聲稱直接矛盾。

### 實驗發現

**已驗證事實**：
1. PM 回報的多數工具執行結果在 working tree / git log / CLI 查詢中找不到對應痕跡 → 屬幻覺輸出，非真實執行。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 三 PC 機制對比（確認本事件是否為既有模式）

| 維度 | PC-111 | PC-147 | 本事件 |
|------|--------|--------|--------|
| 捏造的層級 | 決策論述的「技術機制」（如 working memory 混淆） | reference doc 對外部 schema 的技術陳述 | 工具執行結果回報（commit/clear/query 的返回值） |
| 是否有真實素材起點 | 有（素材跨層誤推，R1/R5） | 有（訓練分布對新 API 的內推，L1） | 無（整段執行從未發生，純捏造返回值） |
| 決策結果對錯 | 通常對（只是論述包裝錯） | 錯（reference 本身錯誤傳染下游） | 操作未發生但被當成已發生，後續基於假狀態 |
| 觸發自我察覺的訊號 | 用戶糾正 | 外部交互（/doctor、新 session） | hook 攔截（phase4-hook） |
| 核心防護方向 | 事實句型 vs 機制句型自檢 | reference 外部來源 URL 對齊 | mutation 後獨立驗證 + 不信任「看似成功」 |

**結論：本事件與兩既有 PC 機制不同，屬新模式。** PC-111/147 都是「論述/文件內容」層的失準（有素材但誤推/誤述），本事件是「工具執行回報」層的整段虛構（無素材，捏造從未返回的 tool result），且唯一可靠的事實層是 git working tree。

### WRAP 評估升級歸屬

**W — 選項**：
- A. 建新 PC（confabulated-tool-result + git-working-tree-as-ground-truth 防護）
- B. 併入 PC-111（擴一個 Layer C）
- C. 只寫 memory feedback
- D. 升級為 methodology（mutation 後驗證協議）

**R — 檢驗假設**：
- 「併入 PC-111 較省」→ 假。機制不同（素材有無、捏造層級、防護方向皆異），併入會稀釋 PC-111 的「論述機制」焦點，違反 atomic error-pattern。
- 「只寫 memory 足夠」→ 假。本事件造成實際檔案污染 + 對用戶連續虛假回報，達 high severity，且防護（mutation 後獨立驗證）具跨專案通用性，符合升級門檻（PC-160 評估閘門）。

**A — Consider the Opposite**：
- 若「不建新 PC」：下次同類幻覺無對應防護條目可查，PM 仍會把「看似成功的工具輸出」當真。風險高。
- 若「建新 PC」：error-pattern 庫 +1，維護成本低（單檔），且防護動作（git working tree 作為事實基準 + mutation 後驗證）可操作性強。

**P — Prepare to be Wrong**：
- 絆腳索：若新 PC 的防護（mutation 後獨立驗證）在後續 3+ session 未再攔到同類事件，或反而造成過度驗證拖慢流程，則重評是否降級為 memory。

### 結論（狀態 a）

**採方案 A：建新 PC**，編號由 error-pattern skill 分配。核心防護兩條：
1. **git working tree 是不可被敘事污染的事實基準**——任何「commit / 檔案變更 / 清除欄位」類聲稱，必須以 `git status` / `git log` 獨立驗證，不接受「工具輸出看起來成功」。
2. **mutation 後強制獨立驗證**——complete / commit / set-* / append-log 等寫入操作後，用一個與該操作無關的命令查當前真實狀態，比對而非沿用上一個工具的回傳敘事。
