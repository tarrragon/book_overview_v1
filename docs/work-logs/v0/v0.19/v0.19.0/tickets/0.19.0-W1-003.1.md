### PM Context Bundle (2026-05-27)

#### 驗證策略決策

用戶決策 (2026-05-27)：採 **Fixture-based 程式驗證**。理由：

- 原 strategy 要求兩個 Chrome profile + Readmoo 真實帳號提取 10+ 本書，涉及 SSO 登入無法 subagent 自動化
- 既有 src/export/ 與 src/data-management/ 模組已完整實作 v2 匯出入流程（W1-001.2 UC-03 已驗證 JSON v2 結構合規）
- Fixture-based 驗證能直接覆蓋 round-trip 邏輯而免實機操作
- W1-001.2 UC-04（JSON 匯入）仍 NOT TESTED，本 ticket 同時補上匯入路徑覆蓋

#### 任務範圍

撰寫 integration test 模擬「裝置 A 匯出 -> 雲端傳輸 -> 裝置 B 匯入」round-trip，**不依賴 Readmoo 真實帳號或實機瀏覽器**。

- `src/export/book-data-exporter.js` — JSON v2 匯出主入口

- `src/export/format-version-detector.js` — v1/v2 格式自動偵測
- `src/export/v1-to-v2-converter.js` — v1 -> v2 轉換器

#### Test 撰寫策略

##### 測試檔案

新增 `tests/integration/cross-device-round-trip.test.js`

##### Fixture 設計

代表性 fixture：12 本書（避免過大但涵蓋所有狀態與 tag 組合）

- 6 種 readingStatus 各 2 本：unread / reading / finished / queued / abandoned / reference
- 3 種 tag_categories：例「文學」「技術」「未分類」
- tag_categories 含 4-5 個 tagIds，每本書 0-3 個 tag references
- 含 isManualStatus = true 的書 2 本（為 W1-003.2 鋪墊但本 ticket 也驗證保留）
- v2 schema 完整：id / title / authors / cover / progress / readingStatus / tagIds / lastReadAt / updatedAt 等

##### Round-trip 流程

```
Step 1: 建立「裝置 A」storage 狀態（mock chrome.storage.local 含 12 本 fixture）
Step 2: 呼叫 book-data-exporter.js 匯出 JSON 字串
Step 3: 驗證 JSON 結構合規 v2（formatVersion / schemaVersion / 四區段）
Step 4: 建立「裝置 B」storage 狀態（clean，空 chrome.storage.local）
Step 5: 呼叫 import handler（覆蓋模式）匯入 Step 2 的 JSON
Step 6: 讀「裝置 B」storage 比對 round-trip 一致性
Step 7: 重複 Step 4-6 改用「合併模式」（B 先有 5 本不同書）
```
