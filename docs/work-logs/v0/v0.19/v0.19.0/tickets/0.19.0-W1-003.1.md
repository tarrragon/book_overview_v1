### 完成內容

新增 1 個 integration test 檔（10 個 test case）+ 2 個 fixture：

| 檔案 | 用途 | 摘要 |
|------|------|------|
| tests/integration/cross-device-round-trip.test.js | 主測試檔 | 3 個 group（覆蓋 / 合併 / 摘要對比）共 10 個 test case |
| tests/integration/fixtures/device-a-library.json | 裝置 A fixture | 10 本書 / 5 tag / 2 category / 6 種閱讀狀態（unread/reading/finished/queued/abandoned/reference） |
| tests/integration/fixtures/device-b-existing.json | 裝置 B fixture（合併模式） | 3 本書，其中 1 本（rt-book-01）與 A 同 id 但 tag 不同，用於驗證合併 tagIds 聯集 |

### 測試流程設計

[裝置 A storage] 預置 fixture（10 本書 + tags + categories）
    -> BookDataExporter.exportToJSON（fieldPreset=COMPLETE_V2）產出 v2 JSON 字串
    -> simulateCloudTransport（JSON.parse -> JSON.stringify）模擬雲端傳輸
    -> ContentParser.parse（fileFormat='json'，不指定版本自動偵測）抽出 ImportResult
    -> TagStorageAdapter.replaceAllData / mergeAllData 寫入裝置 B storage mock
    -> 與 A 端 fixture 對比驗證

### 設計決策

1. chrome.storage mock 分離：使用 bindStorageMock(store) 工廠在 A 與 B 之間切換 chrome.storage.local 的 mock implementation，模擬兩個獨立裝置 storage。複用 W1-047.3 測試的 mock 模式（同 store 物件 + get/set 攔截）。
2. fixture 涵蓋 6 種 readingStatus：10 本書分配為 finished×2 / reading×2 / unread×2 / queued×2 / abandoned×1 / reference×1，並含 isManualStatus 兩種狀態樣本，確保 AC-3 全狀態保留可驗證。
3. 不測試 modal / DOM 元件：本任務範圍為純資料 round-trip，模式選擇 modal 由 W1-049 + W1-047.5 單元測試覆蓋。本檔直接呼叫 TagStorageAdapter.replaceAllData / mergeAllData，繞過 ImportFlowController.execute 的 UI 編排層。
4. 合併模式 tag id 重映射處理：B 端原有 tag 與 A 匯入 tag name 不同，預期經 computeMergeResult 後產生新 id；驗證透過 name 反查（而非硬編碼 id），避開 id 重映射細節，符合 SPEC §8.2/§8.3。
