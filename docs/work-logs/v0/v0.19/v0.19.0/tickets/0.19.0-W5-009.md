### 問題根因

W5-003 parallel-evaluation 三人組審查（saffron-system-analyst）發現完整性缺口：

#### 已契約化 vs 未契約化邊界

**已契約化**（散落於現有 6 章）：
- §2.4 配額管理規則（checkQuotaLevel 4 層級 + 5MB 上限）
- §2.5 Migration backup + rollback 機制（已含 5 案例規則）
- §3.4 結構化 log 前綴（`[SW]` 全域 handler 已列）
- §3.5 status string 集合（含 `failed` / `error` / `timeout` / `cancelled`）
- §4.7 SW 關閉清理（§Phase 6 含 SYSTEM.SHUTDOWN 廣播）

**未契約化**（saffron 識別缺口）：
- RetryCoordinator 完整策略（maxRetryAttempts / backoffDelay 演算法 / circuitBreaker 觸發條件 / 狀態持久化恢復）
- CS 注入失敗 recovery（emergency mode + SW 重啟流程）
- Error-collector 累積策略與彙整時機
- Tag operation 衝突回滾（withAtomicRollback）

### 三方案利弊評估

#### 方案 A：擴 §7 Error Recovery 完整契約章節

| 維度 | 評估 |
|------|------|
| 契約完整性 | 高（集中所有 error recovery） |
| 規格篇幅 | 已 2200 行，再加 §7 預估 +400-500 行 |
| 規格穩定性 | 低（RetryCoordinator 策略與 emergency mode 仍演進中） |
| 落地成本 | 高（盤點 + 撰寫 + grep 驗證） |
| 維護成本 | 高（演進時雙重維護） |

#### 方案 B：於 §4.8 並發約束內補 Error Recovery 邊界 short section

| 維度 | 評估 |
|------|------|
| 契約完整性 | 中（記錄關鍵 invariant，細節留實作層） |
| 規格篇幅 | +30-50 行（控制） |
| 規格穩定性 | 高（只記錄已穩定的不變式） |
| 落地成本 | 低（單章補述） |
| 維護成本 | 低 |

#### 方案 C：維持現狀，不擴章節

| 維度 | 評估 |
|------|------|
| 契約完整性 | 低（saffron 識別缺口不解） |
| 規格篇幅 | 不變 |
| 規格穩定性 | 高 |
| 落地成本 | 0 |
| 維護成本 | 0 |

### 結論

**採方案 B**：於 §4.8 並發約束內補 Error Recovery 邊界 short section。

**Why**：
- 方案 A 過早契約化：RetryCoordinator 策略（次數 / 間隔 / circuit breaker）與 emergency mode 仍在演進中，現在契約化會在 production 變動時造成 spec / code 漂移成本
- 方案 C 留缺口太大：saffron 識別的「retry 策略未契約化」會被新人誤判為「無 retry 機制」，影響後續實作決策
- 方案 B 折衷：記錄已穩定的不變式（如「失敗的 storage write 必須由 RetryCoordinator 接管」、「emergency mode 不繞過 §2.4 配額檢查」），細節留實作層 SoT 引用

**Consequence（採 B）**：
- §4.8 Error Recovery 邊界子節包含 5-8 條 invariant + 對應實作層 SoT 引用（RetryCoordinator / checkQuotaLevel / activateEmergencyMode / error-collector）
- 未來 retry 策略演進不破壞規格（只要不變式仍成立）
- saffron 識別缺口收斂：新人讀規格能識別 retry / quota / emergency 機制存在

**Action**：spawn 一個 DOC ticket W5-009.1 落地方案 B 補述。
