## Problem Analysis
<!-- Schema[DOC/Problem Analysis]: 選填（若 DOC 起因於缺陷或盤點結論可填）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

E2E 提取流程涉及四類契約（URL/Storage/Console message/Lifecycle），目前全硬編碼散落於 tests/e2e、src/background、src/content、src/storage、src/event-system，無集中規格文件。新人接手時無從參考契約原貌；W1-001.2 SPA hash 路由誤判事件已證明缺乏契約規格導致誤判故障的成本極高。

### 影響範圍

- tests/e2e/browser/ - 所有 E2E 測試的契約硬編碼來源
- src/background/service-worker - SW lifecycle、訊息處理契約
- src/content/ - content script 注入、提取觸發契約
- src/storage/ - chrome.storage key naming 與 value schema
- src/event-system/ - event message structure、status 集合
- docs/use-cases.md - UC 層描述但未契約化
- docs/bookstores/readmoo.md - 平台 URL 與 DOM 結構（已含部分契約）

### 相關 Error Pattern

- W1-001.2 案例（feedback_verify_spa_hash_routes_before_concluding_broken）：缺契約規格導致 SPA hash 路由誤判
- W1-050 ANA 致命 21 處：缺契約導致 CE-Node 環境前提誤判

### WRAP 認領 Checkpoint

**W（Widen）替代方案**：
- 方案 A（選定）：PM 親自盤點 + 撰寫，多重契約交叉驗證
- 方案 B（否決）：派 saffron 全程做 - 單一代理人視角易遺漏跨契約引用一致性
- 方案 C（否決）：先派 saffron 盤點再 PM 撰寫 - 盤點本身就需要交叉驗證判斷，PM 親做最低誤差

**A（Attain distance）機會成本**：
- 擠壓 v0.19.0 其他 P2 ticket（W1-049/W1-050.5/W3-016 等）的 PM context
- 但 W5-003 契約規格錯誤成本高（後人依規格實作會偏離實況），優先級高於 P3 任務；且 W1-001.2 與 W1-050 已證明缺契約成本

**P（Prepare to be wrong）最可能失敗**：
- 風險 1：盤點遺漏隱性契約（如 ext_dynamic_id、PROP-007 tag schema 細節） → 防護：完成盤點後 spawn ANA 派 parallel-evaluation 三人組（linux/saffron/basil）審查契約完整性
- 風險 2：source code 與 tests/e2e 兩者不一致時 PM 主觀判斷錯誤 → 防護：交叉驗證階段 explicit 列出差異點，由用戶決策真實契約
- 風險 3：規格寫得過抽象失去契約強制力 → 防護：每個契約都附 source of truth 檔案:行號引用，可 grep 驗證
