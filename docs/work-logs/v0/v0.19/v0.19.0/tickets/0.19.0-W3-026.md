盤點專案內所有 Stop/SubagentStop hook，評估 v2.1.145 新增的 background_tasks 與 session_crons 欄位可整合到哪些 hook 以提升判斷準確度

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

Claude Code v2.1.145 為 Stop/SubagentStop hook input 新增 background_tasks（背景代理人清單及狀態）與 session_crons（排程任務清單）兩個欄位。專案現有 5 個 Stop hook 和 2 個 SubagentStop hook 皆未讀取這兩個欄位，可能在背景任務未完成時做出不精確的 session 結束判斷。

### 影響範圍

SubagentStop hooks（2 個）：
- subagent-stop-dispatch-cleanup-hook.py：清理 dispatch-active.json，已用 agent_id 精準清理。新欄位價值低。
- agent-commit-verification-hook.py：驗證代理人 commit 狀態。新欄位價值低。

Stop hooks（4 個，settings.json）：
- evaluate-session.py：長 session 學習摘取。純觀察性，新欄位價值低。
- handoff-auto-resume-stop-hook.py：偵測未完成 handoff 任務，決定是否阻止退出。新欄位價值高——可用 background_tasks 判斷是否有代理人仍在執行。
- session-experience-persistence-reminder-hook.py：提醒 context 敏感 Ticket。新欄位價值中。
- stop-worklog-handoff-sync-check-hook.py：偵測 worklog handoff 與 CLI 不同步。新欄位價值中——可用 background_tasks 避免在代理人執行中誤報。

Stop hooks（1 個，settings.local.json 重複註冊）：handoff-auto-resume-stop-hook（PC-148 已知問題）

### 相關 Error Pattern

無直接對應 error-pattern。間接相關：PC-148（Hook 雙重註冊 shebang bypass）。

### 實驗方法

1. grep settings.json 和 settings.local.json 盤點所有 Stop/SubagentStop 事件註冊的 hook
2. 逐一 Read 每個 hook 原始碼前 80-120 行，確認 input_data 使用的欄位

### 實驗執行

步驟 1：settings.json 含 SubagentStop 2 個 hook、Stop 4 個 hook；settings.local.json 含 Stop 1 個 hook（與 settings.json 重複的 handoff-auto-resume-stop-hook）

步驟 2：逐一讀取 6 個 hook 原始碼：
- subagent-stop-dispatch-cleanup-hook.py：讀取 input_data.get("agent_id")，不讀其他欄位
- agent-commit-verification-hook.py：讀取 stdin JSON，主要用 git status 判斷
- evaluate-session.py：讀取 CLAUDE_TRANSCRIPT_PATH 環境變數，不讀 stdin
- handoff-auto-resume-stop-hook.py：讀取 stdin JSON 但主要掃描 .claude/handoff/pending/ 目錄和 ticket frontmatter，未讀 background_tasks
- session-experience-persistence-reminder-hook.py：掃描 ticket 檔案，未讀 stdin 的 background_tasks

步驟 3：grep -rn "background_tasks|session_crons" .claude/ 結果為空——專案內零引用

### 實驗發現

已驗證事實：
- 專案有 6 個不重複的 Stop/SubagentStop hook，無一使用 v2.1.145 新欄位
- handoff-auto-resume-stop-hook 現用 started_at 時間戳判斷「最近任務」（RECENT_TASK_THRESHOLD_MINUTES=30），這是間接判斷背景代理人狀態的 workaround；background_tasks 欄位可提供直接判斷
- stop-worklog-handoff-sync-check-hook 現用 in_progress 狀態判斷代理人是否執行中，同屬間接推斷
- evaluate-session.py 完全不讀 stdin，改造成本最高但價值最低

仍未驗證假設：
- background_tasks 欄位的具體 schema（欄位名稱、狀態枚舉值）需實際攔截 Stop event 確認
- session_crons 在本專案是否有實際用途（本專案未使用 CC session cron 功能）

### WRAP 分析

#### W（Widen）：整合方向列舉

方向 A — handoff-auto-resume-stop-hook 整合 background_tasks：用 background_tasks 直接判斷是否有代理人仍在執行，取代現行 started_at 時間戳推斷（RECENT_TASK_THRESHOLD_MINUTES=30 的 workaround）。收益：消除 30 分鐘閾值的 false positive/negative。

方向 B — stop-worklog-handoff-sync-check-hook 整合 background_tasks：代理人仍在背景執行時跳過 worklog-handoff 同步檢查，避免在代理人尚未寫 handoff 的情況下誤報「worklog 有 / pending 無」。

方向 C — hook-architect-technical-reference.md 文件更新：將 background_tasks 和 session_crons 新欄位的 schema 與用途寫入技術參考文件，供後續 hook 開發者查閱。

方向 D — session-experience-persistence-reminder-hook 整合：背景代理人仍在執行時可調整提醒策略。收益低：此 hook 僅輸出 stderr 提醒，不阻止退出。

#### R（Reality-Test）：既有 hook 行為對照

handoff-auto-resume-stop-hook 現行判斷邏輯：掃描 .claude/handoff/pending/ 目錄 → 對每個 pending JSON 查 ticket frontmatter status 與 started_at → 若 in_progress 且 started_at 在 30 分鐘內則視為「最近任務」不阻止退出。此邏輯有兩個弱點：(1) started_at 是 ticket claim 時間，非代理人實際啟動時間；(2) 30 分鐘閾值是硬編碼魔術數字。background_tasks 可直接回答「此刻是否有背景代理人在跑」，精確度遠高於時間戳推斷。

stop-worklog-handoff-sync-check 現行判斷邏輯：掃描 worklog handoff 關鍵字 vs pending 目錄。當代理人仍在背景執行時，worklog 尚未寫入 handoff 段落，此時報「worklog 無 / pending 有」屬正常而非異常。讀取 background_tasks 可在有背景任務時跳過檢查或降級為 INFO。

#### A（Attain distance）：收益與風險評估

收益排序：
1. 方向 A（handoff hook）收益最高——直接消除 30 分鐘魔術數字，精確判斷阻止/放行
2. 方向 C（文件更新）成本最低——純 DOC，無程式碼風險
3. 方向 B（sync-check hook）收益中等——減少誤報但該 hook 本身不阻止退出
4. 方向 D 收益最低——skip

風險：
- background_tasks schema 尚未實際驗證（需先攔截一次 Stop event 確認欄位結構）
- 若 CC 未來版本改動 schema，hook 需跟著改（低風險，JSON get 有 default）

#### P（Prepare to be wrong）：失敗回退

- 所有整合點皆為 additive（讀新欄位 → 有則用，無則 fallback 到現行邏輯）
- 若 background_tasks 為空或格式不符預期，fallback 到現行 started_at 推斷
- 回退成本：刪除新增的 if 分支即可
