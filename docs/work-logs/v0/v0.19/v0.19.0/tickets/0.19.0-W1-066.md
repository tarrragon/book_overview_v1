## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 設計決策：新建 vs 擴展既有腳本

評估 `scripts/validate-build.js`（既有 388 行 CLI 腳本）後決定**新建獨立腳本**，理由：

| 維度 | 既有 validate-build.js | 新建 validate-manifest.js |
|------|----------------------|-------------------------|
| 職責 | 完整建置健康度（含 icons、檔案大小、目錄結構） | 聚焦 manifest 結構 + 路徑 + 權限白名單 |
| 權限白名單 | 軟性警告（logWarning） | 硬性錯誤（exit 1） |
| 可測試性 | 純程序式無法 unit test | 純函式 + fsAdapter 注入可測 |
| 風險 | 修改可能破壞 `validate:build:prod` | 新建零風險 |

擴展既有腳本會違反「白名單嚴格性」acceptance（既有為警告語意），且既有腳本已 388 行無 unit test 難以重構。新建腳本作為前置 gate，未來可與 `validate-build.js` 組合（先 manifest gate，後完整健康度檢查）。

### 實作架構

**純函式驗證層**（可單元測試）：

- `validateManifestVersionAndServiceWorker(manifest, buildDir, fsAdapter)` — manifest_version=3 + SW 路徑存在
- `validateContentScripts(manifest, buildDir, fsAdapter)` — content_scripts[].js 路徑全部存在
- `validatePermissionsWhitelist(manifest)` — permissions 嚴格等於 [storage, activeTab]
- `validateHostPermissions(manifest)` — host_permissions 限 *.readmoo.com（正規式 `/^\*:\/\/(\*\.)?readmoo\.com\/\*$/`）
- `validateAll(manifest, buildDir, fsAdapter)` — 彙總全部驗證

**I/O 分離設計**：`fsAdapter.fileExists(path)` 介面注入；預設使用 Node.js `fs.existsSync`，測試用 mock。所有驗證函式回傳 `{ ok: boolean, errors: string[] }`，避免 `process.exit` 干擾測試。

**CLI 層**：`main()` 解析 `--dir` 參數（預設 `build/production`），讀取 `manifest.json`，呼叫 `validateAll`，依結果 exit 0/1。

### npm script 整合

```json
"validate:manifest": "node scripts/validate-manifest.js",
"validate:manifest:prod": "node scripts/validate-manifest.js --dir build/production"
```

### 測試覆蓋（25 個 test cases）

- `validateManifestVersionAndServiceWorker`：5 cases（成功 + 4 失敗情境：版本錯、缺 background、缺 SW、SW 檔案不存在）
- `validateContentScripts`：6 cases（成功 + 5 失敗情境：缺欄位、非陣列、檔案不存在、部分檔案不存在、缺 js）
- `validatePermissionsWhitelist`：6 cases（順序不變、順序顛倒、含多餘、缺項、欄位缺、空陣列）
- `validateHostPermissions`：5 cases（成功 + 4 失敗：含非 readmoo、缺欄位、空陣列、`<all_urls>`）
- `validateAll` 整合：3 cases（全通過、多重失敗彙總、缺檔彙總）

### 端到端驗證

實際對 `build/production/manifest.json` 執行：
