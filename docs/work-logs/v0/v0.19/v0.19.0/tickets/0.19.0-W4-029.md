### 核心發現

1. **AUQ 規則對齊（最高價值）**：CC v2.1.154 行為改變使「強制列選項即用 AUQ」可能與平台「少問」傾向衝突，導致過度提問。需評估專案 AUQ 規則是否從「列選項必用 AUQ」精煉為「真正需用戶決策時才問，且問則用 AUQ 而非純文字」。此改動敏感（涉及 PC-064 + hook 強制），方向交用戶決策。
2. **dynamic workflow 納入派發策略（中價值）**：評估是否在 bulk-evaluate / 大規模批量場景引入 Workflow 工具。
3. **release-note 監測 skill（用戶主訴求）**：將本次 + W4-028 的「評估 CC 更新對框架影響」流程固化為可重複 skill。

## Solution

### WRAP 決策

**W — Widen**：採用分級——skill 建立（用戶主訴求，建）/ AUQ 對齊（敏感，交用戶決策方向再落地）/ workflow 評估（中價值，建評估 child）/ MCP env 補記（低優先）/ 其餘無影響或不採用。

**R — Reality-test**：
- skill：用戶明確要求，價值確定（自動化未來 CC 更新評估，複用 W4-028/W4-029 流程）。
- AUQ 對齊：CC「少問」與專案「列選項用 AUQ」並非全衝突——專案規則核心是「要問就用工具不用純文字」（PC-064 根因：PM 純文字列選項繞過）；CC 新傾向是「先判斷是否真需要問」。**可調和**：保留「列選項用 AUQ」，前置補「先判斷是否 genuinely cannot decide」。但此改動觸及 hook 強制層，方向須用戶拍板。
- workflow：opt-in（關鍵字/ultracode），對 bulk-evaluate 等大規模批量是潛在增強，但 token 成本高。

**A — Attain distance**：機會成本——skill 價值高且複用現成流程；AUQ 對齊價值高但敏感；workflow 中價值需先評估再決定引入。

**P — Prepare to be wrong（絆腳索）**：
- 若 AUQ 規則放寬後 PM 重現純文字繞過（PC-064）→ 回退強制。
- 若 workflow 引入造成 token 爆量 → 限制為明確 opt-in 場景。
- 若 skill 維護成本 > 手動評估成本 → 降級為純流程文件（checklist）。
