### 三決策點裁決

### 決策 1：Focus Trap Utility 提取 → **不提取（狀態 a）**

**結論**：採 linux 立場，不提取 Focus Trap Utility。

**理由**：
1. 「跨域 reuse」前提不成立（focus trap 僅存在於單一檔案兩處，無其他消費者）
2. 「第三 modal」前提不成立（docs 中無規劃）
3. 可共用邏輯僅 24 LOC / 479 行 = 5%，建新模組的固定成本（80-120 行）超過收益
4. 決策可逆——未來有第三 modal 時可低成本 Extract Method
5. handleFocusTrap 自 W1-049 以來零 bug，穩定性風險極低

**Explore #2 的 Redundancy HIGH 判定是正確的**（33-40% 結構重複確實存在），但「重複」不等於「應立即抽象」。兩處複寫位於同一檔案、同一開發者維護範圍、同一 grep 可達——這是 linux 所說的 Good taste：在只有兩個實例且無第三消費者時，對稱複製比過早抽象更可讀。

**延後 trigger**：第三個 modal 需求出現在 spec 或 proposal 時重新評估（YAGNI tripwire）。此為條件 trigger 而非時間 trigger——若第三 modal 永不出現，本決策永遠有效，無需重評。

### 決策 2：Sentinel vs Boolean 統一 → **不統一（狀態 a）**

**結論**：維持現狀，Modal A 用 `IMPORT_MODE_MODAL_MISSING` Symbol，Modal B 用 `false`。

**理由**：
1. 語意差異有業務理由（L259-261 註解已解釋）：Modal A 下游需區分「DOM 缺失 → showError」與「取消 → 靜默中止」；Modal B 兩者處理相同（不寫 storage）
2. 強制統一需為 Modal B 增加一個無用的 sentinel + 下游分支，增加複雜度而無業務收益
3. 這不是不一致，而是有意的語意分流設計

### 決策 3：spy hook 接縫消除 → **不消除（狀態 a）**

**結論**：維持 `promptImportModeFn` / `confirmEmptyFileOverwriteFn` 可選 callback 模式。

**理由**：
1. 消除 spy hook 需改為共用 modal 基類，但決策 1 已裁定不抽象
2. 即使不提取 utility，可選 callback 模式提供 controller-level 測試可觀測性（L62-70 doc comment 已解釋設計用途）
3. 消除 callback 需改動 `overview-page-controller.js` 的委派方式 + 對應測試，成本高於收益
4. 兩 callback 對稱設計（L88 / L96），模式一致，不增認知負擔
