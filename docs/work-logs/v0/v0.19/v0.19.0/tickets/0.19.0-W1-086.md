### 設計決策

1. **混雜處理選擇移除而非搬遷**：行 390 / 721 / 736 三處計時斷言混在功能測試 test 內，其周圍仍有具實質驗證價值的功能斷言（totalAssignments 計數、conversionErrors 為 0 等）。若整 test 搬遷會把功能驗證一併移出 npm test，違反「移除計時門檻須補功能驗證」原則。故只移除單行計時斷言，保留功能斷言原位（符合 W1-019 A2/A3 處理一致性）。

2. **效能驗證 describe 整段拆檔**：原 4 個 test 全為計時/記憶體基準，無實質功能驗證（功能正確性已由前 3 個整合 describe 涵蓋）。整段移至 tests/perf/ 與 W1-017 / W1-019 A1 處理一致。

3. **perf 檔門檻放寬到「大幅退化防護」**：原 < 5ms / < 1ms / < 0.1ms 等門檻在 jsdom 環境下對機器負載敏感，新檔放寬至 < 500ms（單次）/ < 200ms（平均），記憶體增長從 < 15% 放寬至 < 200%。理由：jsdom 非有效效能量測環境，計時斷言定位為「攔截數量級災難性退化」而非 SLA；真實效能評估須在實機環境用 profiling 工具。檔頭已明確標注此限制。

4. **保留高負載 promise loop**：第 692 行的「應該在高負載下保持系統穩定性」test 雖移除計時門檻，但保留 200 並發 promise loop——這是並發穩定性測試的執行路徑，移除會弱化「無錯誤」斷言的覆蓋強度。

### 不在本 ticket 範圍

- 第 2 次完整 npm test 偶發失敗：`tests/unit/popup/diagnostic-module-enhancement.test.js`（單檔 13/13 通過、第 1 與第 3 次完整 npm test 都通過），是另一個既有 flaky test，與本 ticket 修改無關。建議 PM 另建追蹤 ticket 處理（屬 quality-baseline 規則 5 範圍）。
