### 設計摘要

新增 `uv-tool-ownership-guard-hook.py`（PreToolUse: Bash，UV single-file 模式），落地 W3-087 方案 A。

**核心流程**：
1. `read_json_from_stdin` 取 `tool_input.command`（tool_name != Bash 立即 exit 0）。
2. **迴圈防護**：command 含 `uv tool install` 字面 → 放行（reinstall 命令本身不再觸發）。
3. **O(1) fast-path**：先廉價子字串掃描 command 是否含任一受管理 exe 名（EXE_SET 7 個）；無則立即返回，不做任何 IO / regex。
4. 命中時以 `_extract_invoked_exes` 解析命令首 token（處理 `(cd x && ticket ...)` / `A && B` / `A; B` / `A | B`），僅匹配首 token 位置，避免參數位誤判（`echo ticket` 不命中）。
5. 對每個命中 exe：讀 `~/.local/share/uv/tools/<package>/uv-receipt.toml` 的 `directory` 欄位，`Path.resolve()` 後與 `<CLAUDE_PROJECT_DIR>/<source_subpath>` 比對。match → 不動；mismatch / receipt 缺失 → 在 source 目錄執行 `uv tool install . --reinstall`（timeout 60s）。
6. 可觀測性（quality-baseline 規則 4）：reinstall 動作雙通道（logger.info + sys.stderr）。
7. Exit code 永遠 0（不阻塊使用者命令；reinstall 失敗僅 warn）。

**對照表**：重用既有 staleness hook 的 SkillEntry 結構（DRY，避免 ARCH-020 平行實作漂移），EXE_TO_SKILL 涵蓋 7 個 uv-tool skill（ticket/doc/version-release/mermaid-ascii/worktree/branch-worktree-guardian/project-init）。

### 與既有 hook 關係（不重複 / 不衝突）

- ticket-reinstall-hook（SessionStart, 單 skill, SHA 比對自動修）+ uv-tool-staleness-check-hook（SessionStart, 7 skill, 僅提示）= 啟動期防線。
- 本 hook = mid-session 防線：event 不同（PreToolUse vs SessionStart），判據不同（ownership directory 比對 vs SHA 比對）。

### 多視角自評結論：無重大架構取捨，設計直接

W3-087 ANA 已完成方案 A vs B/C/D 取捨並鎖定 A，架構選擇已定。本 IMP 剩餘決策點（fast-path 策略 / 迴圈防護 / 對照表複用 / ownership 比對）均為機械性實作選擇，無互斥架構分歧。cognitive-load 判據：功能職責數=1、修改檔案數=3、無跨架構層分歧 → 不需 parallel-evaluation。

### Dogfooding（trigger 邏輯實機驗證）

對當前專案真實 receipt 跑 hook 邏輯（mock _reinstall 觀察觸發判斷）：
- `ticket track list`（OWNED）→ 不觸發
- `mermaid-ascii render x`（receipt 指向 ccsession）→ 觸發 reinstall
- `git status`（非工具）→ fast-path 不觸發
- `(cd .claude/skills/worktree && worktree list)`（指向 ccsession）→ 命令辨識正確 + 觸發

暴露既有污染：mermaid-ascii / worktree 全域工具當前指向 ccsession（與 ANA W3-087 盤點一致），branch-worktree-guardian 無 receipt。此為設計預期之 mid-session 防護目標，hook 上線後會在相關命令前自動對齊，無需本 ticket 手動 reinstall（避免影響其他並行 session 工作狀態）。
