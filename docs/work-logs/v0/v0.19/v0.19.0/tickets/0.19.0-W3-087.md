## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

`uv tool install` 以 package name 為全域唯一 key（裝至 ~/.local/share/uv/tools/<package>/，executable 連 ~/.local/bin/）。多專案的 skill 副本 package name 相同（ticket-system 等）→ 共用單一全域 slot → 任一專案 reinstall 即覆蓋全域 executable。最後 reinstall 者勝。

### 影響範圍

本專案 7 個 uv-tool skill（ticket / doc / mermaid-ascii / project-init / version-release / worktree / branch-worktree-guardian）。實測 3 專案（book_overview_v1 / screen_clock / ccsession）共享，6 工具散落指向 3 專案（詳見重現實驗結果盤點表）。

### 相關 Error Pattern

PC-145（Stale CLI Install）的升級版：PC-145 是「改 source 忘 reinstall」單專案問題；本案是「多專案同名工具互搶全域 slot」跨專案問題，SessionStart hook 不防 mid-session 覆蓋。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### Root Cause（機制）

`uv tool install` 以 **package name** 為全域 key，安裝到 `~/.local/share/uv/tools/<package>/`，executable 連到 `~/.local/bin/<exe>`。多專案的 skill 副本 package name 相同（ticket-system 等）→ 共用單一全域 slot → 任一專案 reinstall 即覆蓋全域 executable。SessionStart hook 只在啟動時對齊，無法防 mid-session 被並行專案 session 覆蓋。

### 解法選項（acceptance 3）

| 選項 | 做法 | 利 | 弊 |
|------|------|----|----|
| A 每命令 ownership guard | PreToolUse(Bash) hook：偵測呼叫 uv-tool skill CLI 時，讀 receipt directory，≠ $CLAUDE_PROJECT_DIR 則先 reinstall | 防 mid-session clobber；reinstall cached ~3ms；不破壞統一命令名 | 每次相關 Bash 命令加 hook 開銷；需維護 exe→package 對照表；跨專案 sync |
| B source 執行（uv run） | 廢全域 install，改 `uv run <tool>`（各專案跑自己 source） | 根治（無共享 slot） | 全 framework call site 遷移（hooks/SKILL/PM 習慣）；啟動 env 解析較慢 |
| C package 命名空間化 | 各專案 package name 加專案後綴 | 各專案獨立 slot | executable 名衝突（都叫 ticket）或命令名變專案專屬，破壞統一 |
| D SessionStart 強化 + 接受 | 維持現狀 + 文件記錄 mid-session 限制 | 成本最低 | 不防 mid-session（本 session 已實證失效） |

### 建議

主推 **A（per-command ownership guard）**：以 PreToolUse hook 做廉價 ownership 檢查（讀 receipt directory 字串比對 $CLAUDE_PROJECT_DIR），mismatch 才 reinstall（罕見、cached 快）。保留既有 SessionStart auto-reinstall 作首道防線。A 兼顧「不破壞統一命令名」與「mid-session 防護」。

**設計風險**：A 是新 PreToolUse hook，影響 7 個 uv-tool skill 的所有命令、需 sync 至 3+ 專案的共享 .claude/。實作前的 hook 設計（命令偵測、對照表、開銷最小化）有多個取捨點，建議 spawn IMP 時納入 basil-hook-architect 設計或多視角審查。
