## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-057 修復「同次 conversion 兩次取時間」設計 bug 時，對 `Date.parse(timestamp)` 回 NaN 的 fallback 處理不完備：

- 修補前：`tsMs = Number.isFinite(parsedMs) ? parsedMs : Date.now()`
- 但同時 `ts = timestamp`（保留原始無效字串，如 `'invalid-date'`）
- 結果：tag id 中的 ms 用了 `Date.now()`，但 createdAt/updatedAt 保留原始無效字串，**指向不同時刻**

正是 W1-057 想消除的「雙時間」問題的變體——只是觸發條件從「同次兩次取值」變成「無效輸入 fallback」。

### 影響範圍

- `src/export/v1-to-v2-converter.js:83-94`（`convertV1CategoryToTag` Date.parse NaN 分支）
- 實際觸發機率極低（caller 通常傳入合法 ISO 字串），屬邏輯不完備而非運行時 bug
- 不影響既有 5168 個 passing test

### 相關 Error Pattern

無直接對應的 error-pattern；屬 W1-057 同家族設計 bug（fallback 路徑未維持 invariant）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

`convertV1CategoryToTag` 的 Date.parse NaN 分支改為「tsMs 與 ts 一起回退至 `Date.now()` 同源衍生」，與既有 else 分支（無 timestamp 輸入）的邏輯保持一致。

### 程式碼變更

**修改前**（`src/export/v1-to-v2-converter.js`）：

```js
if (timestamp) {
  ts = timestamp                              // 保留原始字串
  const parsedMs = Date.parse(timestamp)
  tsMs = Number.isFinite(parsedMs) ? parsedMs : Date.now()  // NaN 時與 ts 脫鉤
}
```

**修改後**：

```js
if (timestamp) {
  const parsedMs = Date.parse(timestamp)
  if (Number.isFinite(parsedMs)) {
    ts = timestamp
    tsMs = parsedMs
  } else {
    // NaN 時 tsMs 與 ts 一起回退並從同一 tsMs 衍生，保證同源
    tsMs = Date.now()
    ts = new Date(tsMs).toISOString()
  }
}
```

### 設計決策

- **採方案**：把 NaN 視為「等同 caller 未傳 timestamp」，走與 else 分支相同的衍生路徑
- **為何不直接 throw**：caller（如 `convertV1ToV2Data`）有自己的 catch-and-skip 邏輯，若 throw 會擴大行為變更面；fallback 維持「永不產出無效 tag」的不變式更符合 graceful degradation 原則
- **註解**：標 W1-078 與 W1-057 雙引用，後人可循 commit history 理解設計脈絡

### 不在本 ticket 範圍

- 不引入新的 caller 端 timestamp 驗證（屬 caller 責任）
- 不修改 `convertV1ToV2Book` 的 timestamp 處理（不同函式邊界）
