### 實作摘要

新增 `tests/integration/cross-device-conflict-resolution.test.js` 整合測試檔，覆蓋 UC-05 v1.1 跨裝置衝突解決 4 場景，並建立 4 個獨立 fixtures：

- `conflict-scenario-a-tag-union.json` — 場景 A 雙裝置 fixture（含 deviceA + deviceB 兩段）
- `conflict-scenario-b-manual-status.json` — 場景 B 雙裝置 fixture
- `conflict-scenario-c-v1-format.json` — 真實 v1 schema 純陣列（id, title, author, cover, isNew, isFinished, progress, category, type, publisher, extractedAt），無造假
- `conflict-scenario-d-missing-category.json` — 已序列化的 v2 interchange JSON（直接作為 cloud blob 餵 ContentParser）

測試結構沿用 W1-003.1 風格（bindStorageMock 兩 store 工廠 + simulateCloudTransport + parseImportContent），無計時硬門檻，符合 test-assertion-design-rules。

### 設計決策

| 決策 | 理由 |
|------|------|
| 4 場景拆 4 個獨立 describe | 衝突類別語意上互不相關，獨立 describe 便於後續 spawn IMP 時鎖定範圍 |
| 場景 D fixture 為「裝置 A 匯出後的 v2 interchange JSON」（裸字串） | 無法以 BookDataExporter 產出「tag 引用已刪除 categoryId」狀態（exporter 不會故意製造孤兒 tag），改用直接 fixture 模擬該邊界情境，呼叫端仍走 ContentParser.parse 正規路徑 |
| 場景 A 同名 tag 收斂測試獨立成第 2 個 test case | 「tag 聯集」與「同名 category/tag 收斂」是兩個獨立保證，混在同一 test case 會造成單一 assertion 推論多個語意 |
| 場景 B 擴充第 2 個 test case 覆蓋 queued / abandoned / reference 三種手動狀態 | 單一 abandoned case 無法驗證 isManualStatus 對全部 manual-only 狀態的一致行為 |
| 中文 sort 改用 Set 比對 | Array.sort 對中文採 UTF-16 code unit 序，依賴排序語意脆弱；以 Set 等價比對即可 |

### 既有實作行為與 PCB 期望對齊驗證

| 行為 | 實作位置 | PCB 期望 | 對齊狀態 |
|------|---------|---------|---------|
| tag 聯集 | tag-storage-adapter.js:973 `[...new Set([...local, ...remapped])]` | 場景 A | 完全對齊 |
| isManualStatus 保留（incoming → local） | tag-storage-adapter.js:980 `{ ...localBook, ...impBook, tagIds }` | 場景 B（A 為 incoming） | 完全對齊（A 端 manual=true 寫入 B）|
| v1→v2 轉換 | content-parser.js → format-version-detector → v1-to-v2-converter | 場景 C | 完全對齊 |
| 孤兒 tag 歸入「未分類」 | tag-storage-adapter.js:925-931 `ensureUncategorized()` | 場景 D | 完全對齊 |

### 不在本 ticket 範圍

- 未驗證「反向場景 B」：若 B 端為 manual=true、A 端為 auto=false，A 匯入後 B 端 manual 狀態會被 incoming 覆蓋。當前 `{ ...localBook, ...impBook }` 為「incoming 永遠勝出」語意，**並非真正的「手動狀態保留優先級」邏輯**。本 ticket AC 只驗證 PCB 描述的單向場景，PCB 期望已對齊。若未來需要「雙向手動狀態優先級」（無論哪端 manual=true 皆保留），屬設計缺陷層級，需另開 ANA/IMP follow-up（quality-baseline 規則 5）。
