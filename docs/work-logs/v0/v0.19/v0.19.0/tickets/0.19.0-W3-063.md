### 問題根因

W3-031.1 規劃實作 hook_utils terminalSequence helper，但觸發 YAGNI 疑慮：

1. **零現有使用者**：grep -rn terminalSequence .claude/hooks/ 結果為 0

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 分析

### W（Widen）：潛在 terminalSequence 使用情境分類

基於 56 個使用 stderr 的 hook，按語意分 3 類評估 terminalSequence 適用性：

| 類別 | 代表 hook | 數量 | terminalSequence 適用？ | 理由 |
|------|----------|------|----------------------|------|
| 1. 閘門類（deny/block） | file-type-permission, branch-verify, process-skip-guard | ~30 | 否 | 閘門結果由 Claude Code 顯示為工具拒絕 UI，stderr 是 error-level 正確通道 |
| 2. 提醒/建議類（exit 0 + stderr INFO） | session-start-scheduler-hint, memory-upgrade-reminder, parallel-suggestion | ~15 | 可能 | 非錯誤的提示資訊，語意上 terminalSequence notification 更貼合 |
| 3. 診斷/觀測類（日誌記錄） | hook-health-monitor, dispatch-stats, session-source-diagnostic | ~10 | 否 | 輸出對象是日誌檔非終端用戶，terminalSequence 定位為終端通知 |

結論：僅類別 2（提醒/建議類，約 15 個 hook）在語意上可能受益於 terminalSequence。

### R（Reality Test）：類別 2 是否真的需要 terminalSequence？

類別 2 的 hook 目前用 stderr + exit 0 輸出 INFO 提示，Claude Code 會將 stderr 內容呈現給 PM/代理人。terminalSequence 的差異在於：

| 維度 | stderr（現狀） | terminalSequence |
|------|--------------|-----------------|
| 可見對象 | Claude Code 對話 UI（PM/代理人可見） | 終端視窗（桌面通知/視窗標題/bell） |
| 適合場景 | 需要 Claude Code 讀取並處理的資訊 | 需要人類用戶直接注意的資訊 |
| 本專案現況 | 56 個 hook 已採用，穩定運作 | 0 個 hook 採用 |

關鍵洞察：本專案的 hook 提示對象是 Claude Code PM/代理人（AI），不是直接面對人類用戶的終端通知。stderr 提示被 Claude Code 讀取後融入對話流程，這正是設計意圖。terminalSequence 的桌面通知/bell 定位為「讓人類注意」，但本專案 hook 的讀者是 AI。

**類別 2 中唯一可能受益的情境**：長時 hook（如 pre-push test runner）完成後用 bell 提醒人類用戶切回終端。但本專案目前無此類長時 hook。

### A（Attain distance）：helper 設計成本 vs 直接寫的成本

| 方案 | 成本 | 適用前提 |
|------|------|---------|
| 方案 A：helper（thin wrapper） | hook_io.py 新增函式 + 測試 + 文件 ~2h | 有 3+ 個 hook 確定要用 |
| 方案 B：helper（domain-specific） | 封裝 progress/warning/completion 三種模式 ~4h | 有明確的語意分類需求 |
| 方案 C：各 hook 直接寫 | 每個 hook 約 5 行 JSON 建構 | hook 數量少（1-2 個） |
| 方案 D：豁免實作 | 0h | 無真實使用者 |

已有前車之鑑：generate_hook_output / emit_hook_output 設計成本已付出，但 0 個呼叫者。新增 terminalSequence helper 大概率重蹈覆轍。

### P（Prepare to be wrong）：失敗回退

若豁免實作後未來出現需求：直接在需要的 hook 內寫 5 行 JSON 建構（方案 C），成本極低。helper 的價值在於 3+ 個 hook 共用時 DRY，但從 generate_hook_output 的前例看，即使 helper 存在也不會被呼叫。

### 評估必答問題回答

### Q1：hook 體系內哪些情境真的需要 terminalSequence？

短期（現在）：0 個。83 個 hook 全部使用 stderr + exit code，無一使用 JSON stdout。即使語意上類別 2（提醒類）可受益，但提示對象是 AI 非人類，stderr 已足夠。

中期（可預見）：僅「長時 hook 完成 bell 通知」有微弱需求，但本專案無長時 hook。

### Q2：helper 是否應豁免實作？

**建議豁免**。理由：
1. YAGNI：0 個現有使用者，0 個可預見的近期使用者
2. 前車之鑑：同層級的 generate_hook_output / emit_hook_output 已有 0 呼叫者
3. 語意不匹配：本專案 hook 提示對象是 AI，terminalSequence 定位為人類終端通知

### Q3：若有清晰對象，API 形態？

不適用（建議豁免）。但若未來需要：

| 方案 | 代表性 case | 優點 | 缺點 |
|------|------------|------|------|
| Thin wrapper | `emit_terminal_sequence(title=..., notification=..., bell=True)` | 簡單、低維護、與 spec 1:1 | 無語意抽象，每個 hook 需自行組織內容 |
| Domain-specific | `emit_hook_completion(hook_name, duration)` / `emit_hook_warning(msg)` | 語意清晰、呼叫者零認知負擔 | 預設了使用模式，可能與實際需求不符；generate_hook_output 前例證明預設模式易猜錯 |

dissenting view：thin wrapper 風險低但價值也低（5 行 JSON 省為 1 行呼叫）；domain-specific 價值高但前提是使用模式可預測——而從 generate_hook_output 前例看，預測準確度為 0%。

### Q4：helper 設計成本 vs 直接寫的對比

| 指標 | helper | 直接寫 |
|------|--------|--------|
| 一次性成本 | ~2-4h（實作+測試+文件） | 0h |
| 每個 hook 使用成本 | 1 行呼叫 | 5 行 JSON |
| 損益平衡點 | 3+ 個 hook 使用 | 不適用 |
| 前例（generate_hook_output） | 已付一次性成本，0 個呼叫者 | 不適用 |
