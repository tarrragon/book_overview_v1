# 0.19.0-W1-047.3 功能規格

> TDD Phase 1 功能設計產出物。模式：Full（6 區段）。
> 範圍：UC-04 匯入覆蓋模式 + storage 持久化。合併模式（IMP-D）、模式選擇 UI（IMP-E）不在本規格範圍。

---

## 1. Purpose（目的）

**問題背景**：UC-04 匯入功能（`overview-page-controller.handleFileLoad`）目前只將解析後的書籍寫入記憶體（`currentBooks` / `filteredBooks`）並更新 UI，不寫入 `chrome.storage.local`。IMP-A/B 已讓 `BookFileImporter` 回傳 `ImportResult { books, tagCategories, tags }`，但 `tagCategories` 與 `tags` 在 controller 邊界被丟棄（見 `handleFileLoad` 註解「IMP-C 接手 storage 持久化」）。結果：匯入後重新整理 overview 頁面，匯入的書籍與標籤全部遺失；匯出端產生的 v2 JSON 無法被自身持久化匯入。

**目標用戶**：使用 Readmoo 書庫管理器、執行「匯入 JSON」操作以恢復備份的使用者。

**核心價值**：讓匯入的 `books` / `tags` / `tagCategories` 完整持久化至 `chrome.storage.local`，使匯入結果跨頁面重整存活，達成 SPEC-EXPORT-V2 §8.1「覆蓋模式 = 清空後載入」契約，補齊 v1.0.0 內測「匯出入正常」硬需求的匯入端。

**本 ticket 行為定位**：本 ticket 不提供模式選擇 UI，**預設一律以覆蓋模式執行**。模式選擇 modal 由 IMP-E 疊加；合併邏輯由 IMP-D 疊加。

## 2. API Signatures（介面定義）

### 2.1 新增：tag-storage-adapter 覆蓋模式 API

> **設計決策（須由 Phase 2/3b 落實）**：`tag-storage-adapter.js` 目前 **未匯出**任何「整批覆寫」公開 API。`saveToStorage`、`saveBooksWrapper`、`loadBooks` 皆為模組內私有函式，`module.exports` 僅含逐項 CRUD（`createTag` / `createTagCategory` 等）與讀取 API。覆蓋模式需要「清空後整批寫入」語意，無法用逐項 CRUD 表達（逐項 create 會與既有資料疊加，且 `createTag` 含 duplicate_name 檢查會拒絕）。因此本規格定義在 tag-storage-adapter 新增一個公開 API。

```javascript
/**
 * 覆蓋模式整批寫入：以匯入資料完全取代 storage 中的 books / tags / tag_categories。
 * 語意 = 清空後載入（SPEC-EXPORT-V2 §8.1）。三個 key 一併取代。
 *
 * @param {Object} data
 * @param {Array<Object>} data.books          - v2 書籍陣列
 * @param {Array<Object>} data.tags           - v2 tag 陣列
 * @param {Array<Object>} data.tagCategories  - v2 tag category 陣列
 * @returns {Promise<{ success: boolean, error?: string,
 *                      counts?: { books: number, tags: number, tagCategories: number } }>}
 *   success=true：三 key 寫入完成，counts 回報寫入筆數
 *   success=false：error 為 'quota_exceeded' | 'storage_error'
 */
async function replaceAllData ({ books, tags, tagCategories })
```

**設計約束**：

| 約束 | 說明 |
|------|------|
| 原子性 | 三 key 寫入須走 `withAtomicRollback`（既有機制），任一 key 寫入失敗時回滾全部三 key 至寫入前快照 |
| 並發保護 | 經 `operationLock.run` 序列化，與其他 storage 操作互斥 |
| key 對應 | books → `readmoo_books`、tags → `tags`、tagCategories → `tag_categories`（`STORAGE_KEYS` 既有常數） |
| books 結構 | 透過 `saveBooksWrapper` 寫入，保持既有 `readmoo_books` 結構慣例（陣列或 `{ books: [...] }`）|
| 配額檢查 | 寫入前呼叫 `checkQuotaLevel`，level 為 `blocked` 時回傳 `{ success: false, error: 'quota_exceeded' }`，不執行寫入 |
| 命名為新增方法非修改 | 不更動既有 CRUD API 簽名，僅在 `module.exports` 追加 `replaceAllData` |

> 若 Phase 3b 評估認為應拆為三個獨立 `replaceBooks` / `replaceTags` / `replaceTagCategories`，須在 ticket Solution 記錄理由；本規格以單一 `replaceAllData` 為基準（覆蓋模式三 key 為一個語意單元，原子性需求使單一 API 更安全）。

### 2.2 修改：overview-page-controller.handleFileLoad

```javascript
/**
 * UC-04 匯入入口。讀取檔案 → 解析為 ImportResult → 覆蓋模式持久化 → 更新 UI。
 *
 * 變更（IMP-C）：解構 importResult 後，先呼叫 TagStorageAdapter.replaceAllData
 * 持久化三區段，成功後再以 books 更新 UI。
 *
 * @param {File} file - 使用者選擇的匯入檔案
 * @returns {Promise<void>}
 * @throws 檔案驗證失敗時由 importer throw（既有行為，不變）
 */
async handleFileLoad (file)
```

**回傳值語義**：`Promise<void>`。成功路徑無回傳值；持久化失敗時呼叫 `showError` 並中止 UI 更新（見 §4）。

## 3. GWT Scenarios（行為場景）

### 場景 1: 覆蓋模式持久化成功（正常流程）

- **Given**: storage 中已有舊書籍與舊標籤資料；使用者選擇一個合法的 v2 JSON 匯入檔案（含 books / tags / tagCategories）
- **When**: 執行 `handleFileLoad`，importer 回傳 `ImportResult { books, tagCategories, tags }`
- **Then**:
  - `chrome.storage.local` 的 `readmoo_books` / `tags` / `tag_categories` 三 key 被完全取代為匯入資料（舊資料不殘留）
  - `replaceAllData` 回傳 `{ success: true, counts: {...} }`
  - overview UI 顯示匯入後的書籍（`currentBooks` / `filteredBooks` 更新、`updateDisplay` 執行）

### 場景 2: 匯入檔案僅含 books、無 tags / tagCategories

- **Given**: 匯入檔案為 v1 格式或 v2 但 tags / tagCategories 為空（IMP-B 的 INV-1 保證 ImportResult 三欄位恆為陣列，此時為空陣列）
- **When**: 執行 `handleFileLoad`
- **Then**:
  - `tags` 與 `tag_categories` key 被覆蓋為空陣列 `[]`（覆蓋語意：清空後載入空集合）
  - `readmoo_books` 被覆蓋為匯入書籍
  - UI 正常顯示書籍

### 場景 3: 匯入空書庫（books 為空陣列）

- **Given**: 匯入檔案合法但 `books` 為空陣列（SPEC §9 允許書籍數為 0）
- **When**: 執行 `handleFileLoad`
- **Then**:
  - 三 key 皆被覆蓋為 `[]`
  - UI 顯示空書庫狀態（既有空表格行行為）
  - 不視為錯誤

### 場景 4: storage 配額不足

- **Given**: `chrome.storage.local` 配額已達 `blocked` 門檻
- **When**: 執行 `handleFileLoad`，`replaceAllData` 內 `checkQuotaLevel` 回報 `blocked`
- **Then**:
  - 不執行任何寫入，storage 維持寫入前狀態
  - `replaceAllData` 回傳 `{ success: false, error: 'quota_exceeded' }`
  - controller 呼叫 `showError`，**不更新 UI**（避免 UI 與 storage 不一致）

### 場景 5: storage 寫入中途失敗（原子回滾）

- **Given**: 三 key 寫入過程中第二或第三個 `chrome.storage.local.set` 失敗（`chrome.runtime.lastError`）
- **When**: 執行 `replaceAllData`
- **Then**:
  - `withAtomicRollback` 將已寫入的 key 回滾至寫入前快照
  - storage 三 key 維持寫入前一致狀態（不出現「books 已換、tags 未換」的部分更新）
  - `replaceAllData` 回傳 `{ success: false, error: 'storage_error' }`
  - controller 呼叫 `showError`，不更新 UI

### 場景 6: 檔案驗證失敗（既有行為，不變）

- **Given**: 使用者選擇非法檔案（格式錯誤、超過大小上限）
- **When**: 執行 `handleFileLoad`
- **Then**: importer 在解析前 throw 並呼叫 `showError`；不進入持久化流程（既有行為，本 ticket 不變更）

## 4. Error Handling（錯誤處理）

| 錯誤情境 | 偵測點 | 處理策略 | 回傳值 / UI |
|---------|--------|---------|-------------|
| 配額不足（blocked） | `replaceAllData` 內 `checkQuotaLevel` | 寫入前攔截，不寫入 | `{ success: false, error: 'quota_exceeded' }`；controller `showError('儲存空間不足，匯入未完成')` |
| storage 寫入失敗 | `chrome.runtime.lastError` | `withAtomicRollback` 回滾三 key | `{ success: false, error: 'storage_error' }`；controller `showError('儲存失敗，已還原原有資料')` |
| 檔案 / 解析失敗 | importer（既有） | importer throw + `showError` | 不進入持久化；既有行為 |
| `replaceAllData` 回傳 success=false | `handleFileLoad` 解構回傳值後 | 中止 UI 更新，呼叫 `showError` | 不呼叫 `_updateUIWithBooks` |

**錯誤處理一致性原則**：

- 持久化失敗時 **不更新 UI**。UI 反映 storage 真實狀態，避免使用者看到「已匯入」假象但重整後消失。
- 回滾後 storage 維持寫入前狀態，使用者原有資料不受損（覆蓋模式失敗 ≠ 資料遺失）。
- 所有 catch 區塊須記錄日誌（專案 observability 規則 1）：`replaceAllData` 失敗時記錄 `error.message` + 元件名 `tag-storage-adapter`。
