#### cache 作用域分析

`frontmatter_cache` 是 `scan_pending_handoff_tasks` 函式第 606 行建立的 local dict，生命週期 = 單次 scan 呼叫。目前只有 `is_ticket_recently_started` 透過參數傳入使用它（經 `_load_frontmatter_cached` helper）。`is_ticket_completed` 和 `is_handoff_stale` **不經過** cache——它們走外部 lib 路徑（`_lib_is_ticket_terminal` / `handoff_utils.is_handoff_stale`），各自內部讀檔。

**關鍵發現**：cache 的實際覆蓋範圍比想像中窄。只有 `is_ticket_recently_started` 一條路徑受 cache 保護；`is_ticket_completed` 和 `is_handoff_stale` 走 lib 各自讀檔，不經 cache。這表示拆分時 cache 遷移的複雜度**低於預期**。

### 實驗發現

**已驗證事實**：

1. **939 行含 11 個職責域**，其中 6 個職責域可歸為 3 大 domain：(a) session 管理 (stop-flag + session state, 88 行), (b) ticket/handoff 判定 (版本偵測 + cache + ticket 查詢 + handoff 判定, 302 行), (c) scan + 決策 + output (scan 主迴圈 + generate + format + main, 369 行)
2. **單檔承載 domain 數 = 3**（session 管理 / ticket-handoff 判定 / scan-決策-output），超過 cognitive-load.md「> 2 必須拆分」閾值
3. **cache 實際覆蓋範圍窄**：frontmatter_cache 僅被 is_ticket_recently_started 一條路徑使用，拆分後遷移成本低
4. **版本偵測 get_active_version (62 行) 在本檔 939 行中未被任何函式呼叫**——搜尋全檔 `get_active_version` 僅出現於定義處，屬 dead code 或外部引用（需確認）
5. **scan_in_progress_tickets (DEPRECATED, 447-464) 為 dead code**，return [] 空實作

### 多視角分歧處理（dissenting_view, PC-101）

| 視角 | 立場 | 核心論據 | 本 ANA 獨立評估 |
|------|------|---------|----------------|
| linux | 拆分 | 939 行 > 300 行閾值；domain 數 > 2 | **部分認同**：domain 數確實 = 3，超過閾值。但 300 行閾值是「考慮拆分」非「必須拆分」；真正觸發強制拆分的是 domain 數 > 2 |
| ginger | 反對 | 拆分破壞 PC-097 cache 共用作用域 | **不認同**：實際驗證 cache 僅被 1 條路徑使用（is_ticket_recently_started），覆蓋範圍遠窄於預期。拆分後 cache 遷移成本低，不構成拆分阻力 |

**獨立結論**：ginger 的擔憂基於「cache 在拆分後難以維持一致性」的假設，但實驗證明 cache 只覆蓋 1 條路徑，拆分不會破壞 PC-097 保護。linux 的觀察正確但理由需修正——觸發拆分的不是行數而是 domain 數。

### cache 替代方案評估表

| 方案 | 機制 | PC-097 相容性 | 複雜度 | 建議 |
|------|------|-------------|--------|------|
| A. DI（參數傳入） | scan 函式建 cache dict，傳入各 predicate 函式（現行做法） | 完全相容，cache 生命週期 = 單次 scan | 低（已實作） | 推薦：拆分後各 predicate 移入獨立模組，scan 仍建 cache 傳入，零改動 |
| B. module-level singleton | 模組全域 dict，scan 開始前清空 | 相容但有 stale 風險（忘記清空） | 中 | 不推薦：引入全域狀態 |
| C. pass cache around (context object) | 建 ScanContext dataclass 封裝 cache + project_root + logger | 相容 | 中偏高（新增抽象層） | 過度設計：目前 cache 只服務 1 條路徑 |

**結論**：方案 A（現行 DI）在拆分後零改動即可運作，無需替代方案。

### 3 方案權衡表

| 方案 | 認知負擔降幅 | cache 複雜度增量 | 回歸風險 | 維護成本 | 建議 |
|------|------------|----------------|---------|---------|------|
| **1. 保留現狀** | 無（domain=3 持續違反閾值） | 無 | 無 | 隨功能增長持續惡化 | 不推薦 |
| **2. 部分拆分**：抽出 session 管理 + 版本偵測為獨立模組；scan/判定/output 留原檔 | 中（domain 降為 2，消除獨立子系統；原檔降至 ~700 行） | 零（cache 不涉及被抽出的函式） | 低（被抽出函式為葉節點，無交叉依賴） | 改善（session 管理可獨立測試） | **推薦** |
| **3. 完全拆分**：session_mgmt.py + ticket_predicates.py + scan_engine.py + output_format.py + main.py | 高（每個模組 < 200 行，domain=1） | 零（方案 A DI 不變） | 中（5 個模組間介面需定義；測試 fixture 需調整） | 長期最佳但當前 ROI 不足 | 未來版本可漸進演化 |
