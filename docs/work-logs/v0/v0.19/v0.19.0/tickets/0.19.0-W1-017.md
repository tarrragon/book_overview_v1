### 問題根因

`npm test`（解析為 `jest tests/unit tests/integration`）下，`event-system-v2-performance-stability.test.js` 與 `UC07ErrorAdapter.test.js` 以計時硬門檻（`toBeLessThan` 計時類斷言）作為 pass-fail 驗收條件。Jest 在 jsdom 下執行且大量使用 mock，並非有效效能量測環境；計時斷言受全套件機器負載與 JIT 暖機影響，在完整 npm test 下 flaky（實測 `avgSuggestionTime` 2.48ms 超過 `<2ms` 門檻、UC07 `duration` 超過 `<10ms` 門檻）。

根本問題：計時不應作為測試驗收條件。效能差是設計問題，不是功能問題（用戶確認的設計原則）。

### 三視角審查彙整（parallel-evaluation 情境 A）

W1-017 首版修復（計時硬門檻放寬為「大幅退化防護」門檻 + 補功能正確性驗證）經 linux / code-reviewer / code-explorer 三視角審查：

**共識**：

- 修復方向正確（計時不該當驗收 gate，補功能驗證是對的）
- UC07 `toEqual(firstMapping)` 應改 `toBe(firstMapping)`（三方一致；`getErrorMapping` 為 `Object.freeze` 單例，`toBe` 才真正驗證快取不重建）
- `listeners.size` 基線比對是最優質的新增驗證（good taste 範本）

**衝突與仲裁**：

### 移出機制驗證

`npm test` 解析為 `jest tests/unit tests/integration`，路徑參數為 regex；`tests/perf/...` 不含 `tests/unit` 或 `tests/integration` 子字串，故自動排除。`npm run test:perf`（`jest tests/perf`）獨立執行，因 regex 子字串匹配亦涵蓋既有 `tests/performance/` 目錄（同屬效能測試，分組合理且不影響主套件）。

### baselineListenerCount 保護斷言

ticket 原規劃 `expect(baselineListenerCount).toBe(0)`，但實測 `baselineListenerCount === 5`。根因查證：`ReadmooPlatformMigrationValidator` 建構子呼叫 `registerEventListeners()`，在共用 `eventBus` 上註冊 5 個監聽器（`PLATFORM.VALIDATION.REQUESTED`、`MIGRATION.VALIDATION.REQUESTED`、`VALIDATION.READMOO.{START,VERIFY,COMPLETE}.REQUESTED`）。`EventBus` 與 `EventNamingUpgradeCoordinator` 建構子本身不註冊監聽器。

保護斷言的真實意圖是「顯性化 beforeEach 注入的監聽器數量為已知固定值，偵測未來初始化邏輯漂移」。故改為 `expect(baselineListenerCount).toBe(5)` 並補完整來源註解。`toBe(0)` 為基於錯誤前提的假設值，採用會使 `npm test` 永久紅燈，違反 acceptance 4。

### 範圍外發現（轉 W1-018 / 待 PM 決策）

完整 `npm test` 首次執行時 `tests/integration/workflows/cross-device-sync-workflow.test.js:145` 的 `expect(avgProgress).toBeCloseTo(scenario.averageProgress, 5)` 失敗一次；該檔單獨執行 13/13 通過，後續兩次完整 `npm test` 亦通過，確認為 cross-suite 環境負載下的 intermittent flaky。此檔不在 W1-017 file scope（W1-017 限 event-system 與 UC07 兩檔），失敗原因為浮點平均值高精度（5 位）斷言而非純計時硬門檻。屬 W1-018「全專案計時 / 脆弱驗收測試盤點」精神範圍，建議 W1-018 盤點時納入「浮點精度脆弱斷言」一併規劃。詳見 NeedsContext。
