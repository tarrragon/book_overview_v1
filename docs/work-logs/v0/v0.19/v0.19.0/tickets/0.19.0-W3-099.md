## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W10-107 對齊 ticket-body-schema.md 時將 VALID_SECTIONS 從原始清單擴充至 8 個章節，補入「重現實驗結果」（PC-063 ANA 必填）。但同一次對齊遺漏了 `Completion Info` 與 `Task Summary` 兩個 schema 三類型皆必填的章節。遺漏原因：W10-107 聚焦 ANA 專屬章節（重現實驗結果），未系統性比對 schema 全量必填清單。

直接後果：agent 嘗試 `ticket track append-log --section "Completion Info"` 時被 `track_acceptance.py` line 647 白名單驗證拒絕（return 1），被迫降級為 Edit 直接修改 ticket md。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `.claude/skills/ticket/ticket_system/lib/command_tracking_messages.py` line 160-169 | VALID_SECTIONS 定義缺 `Completion Info` 和 `Task Summary` |
| `.claude/skills/ticket/ticket_system/commands/track_acceptance.py` line 644-650 | section 白名單檢查邏輯（消費 VALID_SECTIONS） |
| `.claude/pm-rules/ticket-body-schema.md` | 權威 Schema：Completion Info 三類型皆必填；Task Summary 三類型皆必填 |

所有 IMP/ANA/DOC ticket 的執行代理人在 complete 前嘗試用 append-log 填寫 Completion Info 時皆會被拒。

### 相關 Error Pattern

無直接對應的 PC error-pattern。MEMORY feedback `feedback_append_log_completion_info_not_accepted.md` 已記錄此問題但未建 ticket 追蹤（直到 W3-097 第 N+1 次重現才建本 ANA）。

## 重現實驗結果
### 實驗方法

靜態分析 VALID_SECTIONS 定義（`command_tracking_messages.py` line 160-169）與 ticket-body-schema.md Schema 對照表，列出所有 gap。

### 實驗執行

VALID_SECTIONS 現有 8 個值：

```
Problem Analysis, Context Bundle, 重現實驗結果, Solution,
Test Results, Execution Log, NeedsContext, Exit Status
```

ticket-body-schema.md 三類型必填章節對照：

| Schema 必填章節 | ANA | IMP | DOC | 在 VALID_SECTIONS 中？ |
|----------------|-----|-----|-----|----------------------|
| Task Summary | 必填 | 必填 | 必填 | MISSING |
| Problem Analysis | 必填 | 選填 | 選填 | IN |
| 重現實驗結果 | 必填 | 免填 | 免填 | IN |
| Solution | 必填 | 選填 | 免填 | IN |
| Test Results | 選填 | 必填 | 免填 | IN |
| Completion Info | 必填 | 必填 | 必填 | MISSING |

Gap：`Completion Info` 和 `Task Summary` 兩個三類型皆必填章節不在 VALID_SECTIONS 中。

### 實驗發現

已驗證事實：

1. VALID_SECTIONS 缺 `Completion Info` 和 `Task Summary`，與 schema 不一致
2. `track_acceptance.py` line 647 `if section not in valid_sections: return 1` 會拒絕這兩個章節的 append-log
3. W10-107 註解明確說「對齊 ticket-body-schema.md」，但對齊不完整
4. W3-097 source ticket 記錄 thyme agent 實際遭遇此拒絕並降級 Edit

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 三方向利弊評估

#### 方向 A：擴 VALID_SECTIONS 加入 Completion Info 和 Task Summary

| 面向 | 評估 |
|------|------|
| 實作成本 | 極低（加 2 行字串到 list） |
| 語意合理性 | Task Summary 合理。Completion Info 存在語意疑慮（「完成終態」vs「遞增式日誌」），但實際上 append-log 語意是「將內容追加到指定章節」，章節名稱本身決定內容語意，工具層不應強加語意限制 |
| W10-107 相容性 | 完全相容，是 W10-107 對齊的延伸補完 |
| 風險 | 低 |

#### 方向 B：schema 改 Completion Info 非 append-log 必填

| 面向 | 評估 |
|------|------|
| 實作成本 | 中（修改 schema + 可能影響 acceptance-gate-hook） |
| 語意合理性 | 不合理——Completion Info 三類型皆必填，降級削弱 ticket 完整性 |
| 風險 | 高。schema 遷就 CLI 限制是本末倒置 |

結論：方向 B 不可行。

#### 方向 C：雙軌（append-log 全開 + complete 前 schema 強制檢查）

| 面向 | 評估 |
|------|------|
| 實作成本 | 高（需在 complete 流程加入 type-aware section 完整性檢查） |
| 語意合理性 | 最完整 |
| 風險 | 過度工程。acceptance-gate-hook 已在 complete 時驗證 body，擴白名單（方向 A）即可解決 |

### 結論

採方向 A：擴 VALID_SECTIONS 加入 Completion Info 和 Task Summary。

理由：
1. 實作成本最低（2 行字串），風險最低
2. agent 在 complete 前填寫 Completion Info 是正常工作流程（agent-definition-standard.md 明確規定），append-log 是填寫工具手段，不應阻擋合法填寫目標
3. W10-107 對齊的自然延伸補完，無相容性衝突
4. 方向 B 不可行（本末倒置），方向 C 過度工程（已有 acceptance-gate-hook 閘門）

W10-107 相容性確認：W10-107 設計意圖是「VALID_SECTIONS 與 ticket-body-schema.md 對齊」。本修復是補完該對齊的遺漏項，完全符合 W10-107 設計方向。
