## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

**共同根因（單一）**：Claude Code runtime 的 `isolation: "worktree"` 在派發 subagent 時，以**當下 main HEAD** 作為 `git worktree add` 的 start point，之後不再同步。若 PM 主線程在派發後繼續 commit（或合併其他 agent 產出），worktree 的 base 即與 main 產生落差，且落差只增不減。

**機制拆解**：

| 步驟 | 時序 | main HEAD | worktree base |
|------|------|-----------|--------------|
| PM 派發 W1-031/032/033 | T0 | de8e6261 | de8e6261（快照） |
| PM 繼續 commit（W1-030 Phase 4、W5-003/004 等） | T1~T5 | 逐步推進至 ab233fc5+ | de8e6261（凍結） |
| Agent 在 worktree 內執行 ticket create / 讀 ticket | T2+ | 2cd516f7 | de8e6261（落後 20+ commits） |

**根因屬 cc runtime 層級**，非專案 skill 或 hook 可直接修改的範圍。專案只能在派發前/中/後三個時機點加入防護。

### 四症狀歸因

| 症狀 | 直接成因 | 歸因至根因 |
|------|---------|-----------|
| (1) cinnamon 需手動 git merge main | worktree base 落後，新 ticket 檔不存在於 de8e6261 | base 凍結 |
| (2) saffron stale 分支無法 FF | worktree 基於 de8e6261 commit，main 已前進，分支歷史分叉 | base 凍結 |
| (3) thyme ticket create ID 碰撞 | ticket create 掃描 worktree 本地檔案系統，de8e6261 無 W1-032/033 檔案 | base 凍結 + ticket CLI 掃描路徑為 cwd |
| (4) PM Bash cwd 漂移進 worktree | cc runtime 副作用（非 base 問題，但在同一派發事件觸發） | 獨立子問題，與 base 無因果關係 |

症狀 (1)(2)(3) 共享單一根因；症狀 (4) 為獨立問題。

### 影響範圍

- `.claude/skills/worktree/`（殭屍清理已有，但無 base freshness 防護）
- `.claude/skills/ticket/ticket_system/`（ticket create 的 ID 掃描範圍為 cwd，受 worktree base 影響）
- PM 派發流程（缺乏 pre-dispatch commit gate）

### 相關 Error Pattern

- 無完全匹配的既有 PC/IMP pattern
- 最近似：PC-076（跨 session 未 commit 遺留）、PC-078（並行 session ticket 誤判）——但本問題是 worktree base 時間凍結，非遺留或並行

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 根因結論

cc runtime `isolation: "worktree"` 以派發瞬間 main HEAD 作為 worktree start point，不做後續同步。PM 主線程在 agent 執行期間持續 commit 導致 base 落差遞增，引發症狀 (1)(2)(3)。症狀 (4) 為獨立的 cwd 漂移問題。

### 防護方案（派發前 / 中 / 後三階段）

#### 階段一：派發前（Pre-dispatch）

| 防護 | 描述 | 可行性 | 建 IMP？ |
|------|------|--------|---------|
| A1. PM commit gate | PM 在 `Agent(isolation:"worktree")` 派發前，確認所有 pending changes 已 commit 到 main，使 agent worktree base = 最新 main | 高（純流程規範，加入 pm-rules 或 dispatch prompt） | 不需要（DOC 級，寫入 pm-rules/behavior-loop-details.md 或 agent-dispatch-template.md） |
| A2. 避免在 agent 執行期間 commit 到 main | PM 若有多個背景 agent，等所有 agent 完成後再統一 commit | 低（限制 PM 生產力，違反 async-mindset） | 不需要（不建議採用） |

#### 階段二：執行中（During execution）

| 防護 | 描述 | 可行性 | 建 IMP？ |
|------|------|--------|---------|
| B1. Agent prompt 加 `git merge main` 指引 | 派發 prompt 明示 agent 開始工作前先執行 `git fetch && git merge origin/main`（worktree 共享 object store 可直接 merge） | 高（prompt 層面，零成本） | 不需要（寫入 agent-dispatch-template.md） |
| B2. worktree-base-distance-check hook 升級為阻擋 | 現有 hook 僅 approve+warn；升級為 exit 2 deny 當距離 > MAX_BASE_DISTANCE | 中（需驗證 cc runtime 對 deny 的行為） | 建 IMP（如決定實作） |
| B3. ticket create 改用 main ref 掃描 | ticket_system 的 ID 掃描改為 `git ls-tree main docs/work-logs/` 而非掃描 cwd 本地檔案 | 中（需修改 ticket_system，有跨專案影響） | 建 IMP（最根本解法） |

#### 階段三：合併後（Post-merge）

| 防護 | 描述 | 可行性 | 建 IMP？ |
|------|------|--------|---------|
| C1. 合併前 FF 檢查 | `/worktree merge` 已有 behind 檢查（messages.py BRANCH_BEHIND_BASE），但 agent worktree 不走 `/worktree merge` 而是 PM 手動 `git merge` | 高（既有機制，但覆蓋路徑不同） | 不需要（PM 意識即可） |
| C2. Agent 完成後自動 rebase onto main | Agent complete 前自動 `git rebase main` 確保 FF | 低（rebase 有衝突風險，agent 無法處理） | 不需要（風險大於收益） |

### 建議採用方案

**最小成本 / 最高 ROI**：A1 + B1（純流程 + prompt 層面，零程式碼修改）

- A1：PM 派發 `isolation:worktree` agent 前，先 commit 所有 pending changes
- B1：派發 prompt 加入「開始前先 `git merge main`」指引

**進階防護（視需求建 IMP）**：

- B3（ticket create 改用 main ref 掃描）是最根本的防護，但修改範圍大，建議獨立 IMP 評估
