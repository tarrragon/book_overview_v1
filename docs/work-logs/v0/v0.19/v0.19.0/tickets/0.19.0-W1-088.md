## Problem Analysis

### 問題根因

W1-087 ANA 確認 `tests/unit/popup/diagnostic-module-enhancement.test.js` 行 47:

```javascript
expect(diagnostic.initializationMetrics.initTime).toBeCloseTo(initTime, 0)
```

屬「真實計時量測值相互比較」斷言：

- 右值 `initTime` = `performance.now() - startTime`（測試端外部量測）
- 左值 `diagnostic.initializationMetrics.initTime`（被測模組內部量測）

兩者皆為真實 `performance.now()` 差值，受 JIT 暖機 / GC 觸發 / 全套件並行負載影響。

`toBeCloseTo(v, 0)` 容差 ±0.5ms，於完整 `npm test`（215 suites 並行）下兩個獨立量測點時間差可能超過 0.5ms，於 W1-086 完整跑第 2 次首跑時偶發失敗、單檔 13/13 通過、第 1/3 次完整跑通過 — 典型計時 flaky 訊號。

### 影響範圍

- 檔案：`tests/unit/popup/diagnostic-module-enhancement.test.js` 行 47（單行修改）
- 涵蓋範圍：W1-087 ANA 已盤點本檔內僅此一處違規（其他 performance.now 為 unused 變數，未進入斷言）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

將 `tests/unit/popup/diagnostic-module-enhancement.test.js` 行 47 計時比較斷言改為存在性 + 範圍驗證：

**Before**:
```javascript
expect(diagnostic.initializationMetrics.initTime).toBeCloseTo(initTime, 0)
```

**After**:
```javascript
expect(diagnostic.initializationMetrics.initTime).toBeDefined()
expect(diagnostic.initializationMetrics.initTime).toBeGreaterThanOrEqual(0)
```

### 設計決策

1. **保留 initTime 計算**（行 44 `const initTime = performance.now() - startTime`）：原有 `eslint-disable-next-line no-unused-vars` 註解仍覆蓋此變數，未產生 lint 警告。保留原 code shape 降低 diff noise，未來若需重新引入比較可直接套用。
2. **斷言設計**：
   - `toBeDefined()`：驗證被測模組確實設定 `initializationMetrics.initTime`（功能正確性）
   - `toBeGreaterThanOrEqual(0)`：驗證該值合理（時間量測必為非負），但不約束絕對值（不再受環境影響）
3. **未拆檔至 tests/perf/**：依 W1-087 ANA Solution 建議，本 test case 主體驗證的是「lazy initialization with performance tracking」功能存在性，非效能 SLA；移除計時比較後即恢復為純功能測試，無需移檔。
