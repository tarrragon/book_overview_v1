## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 變更摘要

於 `package.json` 的 `eslintConfig.overrides` 陣列追加兩個 override 物件：

1. **Benchmark 報告 console 豁免**
   - `files: ["tests/e2e/performance/**"]`
   - `rules: {"no-console": "off"}`
   - 設計理由：`tests/e2e/performance/benchmark-tests.test.js` 內 19 個 `console.log` 為效能基準報告輸出（測試完成後 console.log 量測結果），屬報告本質而非 debug 殘留；以 override 標示「此目錄 console 為設計意圖」，避免每個 console 位置加 `/* eslint-disable no-console */` 的散落豁免。

2. **Chrome mock callback literal 豁免**
   - `files: ["tests/mocks/**", "tests/unit/storage/adapters/**"]`
   - `rules: {"n/no-callback-literal": "off"}`
   - 設計理由：Chrome runtime / storage API 的 callback 簽名為 `(err: null | Error, value)`，mock 必須以 literal `null` 模擬「無錯誤」狀態（如 `cb(null, mockData)`）。`n/no-callback-literal` 規則設計用於偵測「應傳 Error 物件卻傳字串」反模式，對 Chrome API mock 屬規則設計錯位；override 標示「此目錄為 Chrome API mock 簽名強制」。

### Override 設計原則

| 原則 | 落地 |
|------|------|
| 範圍最小化 | 兩個 override 各自精準對齊 W4-018 分類的 G1 / G2 路徑，未泛化至整個 `tests/` |
| 規則對場景過嚴 | 兩個豁免規則皆屬「規則設計與場景錯位」，非「容忍 anti-pattern」 |
| Inline comment 註明（採方案）| eslintConfig 為 JSON，無法寫 inline comment；改以 ticket Solution 章節記錄豁免理由作 SSOT |

### 不在本 ticket 範圍

- `tests/test-setup.js:212` 殘留 1 個 `n/no-callback-literal` warning：該檔不在 `tests/mocks/**` / `tests/unit/storage/adapters/**` 路徑內，依規則 5 屬「所有發現必須追蹤」（PM 後續決策是否擴大 override 範圍或修代碼）。
