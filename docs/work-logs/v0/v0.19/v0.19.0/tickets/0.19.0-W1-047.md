## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

UC-04 匯入功能（BookFileImporter + overview-page-controller handleFileLoad）未依 SPEC-EXPORT-V2 實作。現行匯入流程只做三件事：(1) 檔案驗證/讀取 (2) 從 JSON/CSV 提取 books 陣列 (3) 更新記憶體 UI（`_updateBooksData` + `updateDisplay`）。缺失五項 SPEC 要求：

| 缺失項 | SPEC 章節 | 現行狀態 |
|--------|----------|---------|
| 版本偵測 | §2 | `_extractBooksFromData` 只辨識陣列形狀，不呼叫 `detectFormatVersion` |
| v1 轉換 | §7 | 無 isNew/isFinished/category → readingStatus/tag 轉換，不呼叫 `convertV1ToV2Data` |
| tagCategories+tags 處理 | §3 | 匯入時完全忽略頂層 `tagCategories`/`tags` 區段 |
| 合併/覆蓋模式 | §8 | 無模式選擇 UI，無合併邏輯；`_handleDuplicateBooks`（DuplicateBookMerger）存在但零呼叫者接入匯入流程 |
| Storage 持久化 | §8.1 | 匯入只更新記憶體 `currentBooks`/`filteredBooks`，不寫 `chrome.storage.local`（`readmoo_books`/`tags`/`tag_categories`） |

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `src/overview/book-file-importer.js`（592 行） | 需重構 `_extractBooksFromData` + `_validateRequiredFields` + 新增版本偵測/轉換流程 |
| `src/overview/overview-page-controller.js` | `handleFileLoad` 需接入模式選擇 UI + storage 持久化 |
| `src/export/format-version-detector.js`（70 行） | 已完整實作 SPEC §2，可直接重用 |
| `src/export/v1-to-v2-converter.js`（175 行） | 已完整實作 SPEC §7（含 category→tag 轉換），可直接重用 |
| `src/storage/adapters/tag-storage-adapter.js` | 已有 `saveToStorage` for tags/tag_categories/readmoo_books，可直接重用 |
| `src/overview/duplicate-book-merger.js` | 已存在但未接入匯入流程，需評估是否可重用於合併模式 |

### 相關 Error Pattern

無直接對應的 PC error pattern。此為功能缺口（SPEC 模組已建但未接線），非流程或實作錯誤。

### WRAP 三問

**W（Widen）**：
- 方案 A：在 BookFileImporter 內部接線（版本偵測→轉換→回傳 v2 結構），overview-page-controller 負責模式選擇+storage 持久化
- 方案 B：建立新的 ImportManager 模組統一協調（類似 ExportManager），BookFileImporter 退化為純檔案讀取
- 選擇方案 A：BookFileImporter 已承擔「檔案→書籍資料」職責，版本偵測/轉換屬同一 pipeline；新建 ImportManager 過度設計（UC-04 為單一匯入入口）

**A（Attain distance）**：
- 執行 UC-04 修復會佔用 Wave 容量，但 UC-04 匯入為 v1.0.0 內測「匯出入正常」硬需求，且 UC-03 匯出端已 PASS，匯入端缺失造成功能不對稱

**P（Prepare to be wrong）**：
- 最可能失敗原因：合併模式的 tag id 重映射邏輯（SPEC §8.3）複雜度高，與既有 DuplicateBookMerger 介面可能不匹配
- 防護：將合併模式拆為獨立子任務，先實作覆蓋模式（簡單清空+載入），再疊加合併模式

### 版本歸屬建議

**建議歸入 v0.19.0**。理由：

1. **v1.0.0 硬需求**：v1.0.0 內測要求「匯出入正常」。UC-03 匯出已驗證 PASS（W1-001.2 commit 0ffe31a1），UC-04 匯入端缺失造成功能不對稱——匯出產生的 v2 JSON 無法被自身匯入。
2. **v0.19.0 定位**：v0.19.0 為「端到端驗證、打包、內測準備」版本。匯入功能屬端到端驗證必要環節。
3. **實作成本可控**：五項 SPEC 模組中三項（版本偵測、v1 轉換、storage adapter）已完整實作可直接重用，僅需接線+補覆蓋/合併邏輯。
4. **v0.20.0 為 Tag 管理功能**：UC-04 匯入的 tag 處理是「匯入時被動接收 tag」，非「主動管理 tag」（v0.20.0 範圍），兩者職責不重疊。
