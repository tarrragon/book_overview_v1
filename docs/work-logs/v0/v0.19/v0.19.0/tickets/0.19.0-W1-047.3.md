### 問題根因

UC-04 匯入功能 `overview-page-controller.handleFileLoad` 只將解析後書籍寫入記憶體（`currentBooks` / `filteredBooks`）並更新 UI，不寫入 `chrome.storage.local`。IMP-A/B 已讓 `BookFileImporter` 回傳 `ImportResult { books, tagCategories, tags }`，但 `tagCategories` / `tags` 在 controller 邊界被丟棄（見 `handleFileLoad` 既有註解「IMP-C 接手 storage 持久化」）。結果：匯入後重整 overview 頁面，匯入的書籍與標籤全部遺失。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `src/overview/overview-page-controller.js` | `handleFileLoad` 接入覆蓋模式持久化，成功才更新 UI |
| `src/storage/adapters/tag-storage-adapter.js` | 需新增 `replaceAllData` 公開 API（既有無整批覆寫 API） |

**關鍵設計發現**：`tag-storage-adapter.js` 目前 `module.exports` **未匯出**任何整批覆寫 API。`saveToStorage` / `saveBooksWrapper` / `loadBooks` 皆為私有函式。覆蓋模式（清空後載入）無法用逐項 CRUD 表達（`createTag` 含 duplicate_name 檢查、會與既有資料疊加），故規格定義新增 `replaceAllData`。

### 相關 Error Pattern

無直接對應 PC error pattern。屬功能缺口（IMP-B 已預留接線點）。

### Phase 1 功能規格摘要

完整規格見 `0.19.0-W1-047.3-feature-spec.md`。實作要點：

1. **tag-storage-adapter 新增 `replaceAllData({ books, tags, tagCategories })`**：覆蓋模式整批寫入，三 key（`readmoo_books` / `tags` / `tag_categories`）一併取代；經 `withAtomicRollback` 保證原子性、`operationLock` 序列化、寫入前 `checkQuotaLevel` 攔截配額不足。回傳 `{ success, error?, counts? }`。

3. **覆蓋語意**：空 tags / tagCategories 也覆蓋為 `[]`（清空後載入）；空書庫不視為錯誤。
4. **範圍邊界**：本 ticket 預設一律覆蓋模式，不含模式選擇 UI（IMP-E）與合併邏輯（IMP-D）。

#### 1.1 storage 層：新增 replaceAllData 公開 API

虛擬碼（語言無關邏輯流程）：

```
函式 replaceAllData(入參: { books, tags, tagCategories })
  回傳: { success, error?, counts? }

  以 序列鎖.執行(內層任務) 包裝整個流程   // 確保與其他 storage 操作互斥

  內層任務:
    // --- 步驟 A：配額前置攔截 ---
    配額 ← 查詢配額狀態()
    若 配額.等級 = 'blocked':
        回傳 { success: false, error: 'quota_exceeded' }    // 不寫入任何 key

    // --- 步驟 B：建立寫入前快照（三 key）---
    舊書籍   ← 讀取書籍()           // 用既有 loadBooks（解 wrapper）
    舊標籤   ← 讀取標籤()
    舊類別   ← 讀取類別()

    // --- 步驟 C：原子寫入（包入回滾機制）---
    結果 ← 原子回滾.執行(
              快照集合 = { books: 舊書籍, tags: 舊標籤, categories: 舊類別 },
              操作 = 依序寫入三 key:
                       寫入書籍包裝(入參.books)                  // saveBooksWrapper
                       寫入儲存({ TAGS: 入參.tags })
                       寫入儲存({ TAG_CATEGORIES: 入參.tagCategories })
                       回傳 { success: true },
              操作名稱 = 'replaceAllData')

    // --- 步驟 D：判定終點 ---
    若 結果.success = true:
        回傳 { success: true,
               counts: { books:入參.books.length,
                         tags:入參.tags.length,
                         tagCategories:入參.tagCategories.length } }
    否則:   // withAtomicRollback 已完成回滾，回傳 { success:false, error:'rollback', ... }
        回傳 { success: false, error: 'storage_error' }
```

### 2. 關鍵架構決策

| 決策 | 內容 | 理由 |
|------|------|------|
| ADR-1 包裝既有機制非重寫 | replaceAllData 內層直接複用 operationLock.run / withAtomicRollback / checkQuotaLevel / saveBooksWrapper / saveToStorage，零新增基礎設施 | 既有機制已被 deleteTag / deleteTagCategory 驗證；新 API 僅組合，降低回歸風險 |
| ADR-2 快照 key 名稱對齊既有契約 | withAtomicRollback 的 snapshotKeys 必須用 `books` / `tags` / `categories` 三鍵（**非 tagCategories**） | SNAPSHOT_KEY_TO_STORAGE_KEY 既有對映：`categories→TAG_CATEGORIES`、`tags→TAGS`，`books` 特殊走 saveBooksWrapper。用錯 key（如 `tagCategories`）會讓回滾迴圈靜默跳過該 key（對映表查無 → storageKey undefined → 不還原），破壞原子性 |
| ADR-3 寫入順序 books→tags→tagCategories | 內層操作依序寫三 key | 對應 TC-S8（第二 key=tags 失敗）/ TC-S9（第三 key=tagCategories 失敗）的回滾驗證；任一失敗 withAtomicRollback 還原全部三 key |
| ADR-4 error code 正規化 | withAtomicRollback 回滾後回傳 `{ success:false, error:'rollback' }`；replaceAllData 對外統一轉為 `error:'storage_error'` | feature-spec §4 與 TC-S8/S9 要求對外 error 為 `'storage_error'`，內部 `'rollback'` 為實作細節不外洩 |
| ADR-5 配額檢查在鎖內、快照前 | checkQuotaLevel 於 operationLock.run 內、建立快照前執行 | 配額 blocked 時提早回傳，不做無謂的快照與寫入（TC-S6 要求 store 完全未被修改）|
| ADR-6 空集合語意 = 覆蓋為 [] | 空 tags / tagCategories / books 直接寫入 `[]`，不跳過、不保留舊值 | SPEC §8.1 覆蓋模式 = 清空後載入；TC-S4 / TC-S5 / TC-G2 / TC-G3 驗證 |
| ADR-7 books wrapper 結構保持 | 透過 saveBooksWrapper 寫入，沿用既有「陣列 or { books:[...] }」結構偵測 | TC-S3 要求既有 `{ books, extractionTimestamp }` 物件形式寫入後仍為物件形式 |
| ADR-8 module.exports 僅追加 | TagStorageAdapter 物件追加 `replaceAllData`，不更動既有 CRUD 簽名 | feature-spec §2.1 約束「命名為新增方法非修改」；TC 不觸及既有 CRUD（chrome-storage-tag-operations.test.js 無影響）|

- **CJS 雙模式**：tag-storage-adapter.js 為 `module.exports` 物件，`replaceAllData` 須同時加入函式定義與 `TagStorageAdapter` 物件成員。

### 2. 行為鏈推演

#### 2.1 replaceAllData 行為鏈（storage 層）

```
前置：chrome.storage.local mock 就緒、store 有舊資料
  → A. checkQuotaLevel 查詢配額（前置驗證：配額未 blocked）
  → B. 建立三 key 寫入前快照（withAtomicRollback snapshot）
  → C. 依序寫入 readmoo_books（saveBooksWrapper）/ tags / tag_categories
  → D. 中間狀態：任一 set 失敗 → withAtomicRollback 回滾三 key
  → E. 終點：success=true + counts，或 success=false + error
```

#### 2.2 handleFileLoad 行為鏈（controller 層）

```
前置：importer mock 回傳 ImportResult、replaceAllData mock 就緒
  → A. importer 解析回傳 ImportResult { books, tags, tagCategories }（前置驗證：三欄位為陣列）
  → B. 呼叫 replaceAllData(books, tags, tagCategories)
  → C. 分支判定 replaceAllData 回傳 success
       success=true  → D1. _updateUIWithBooks(books)，UI 更新
       success=false → D2. showError(...)，中止 UI 更新（不呼叫 _updateUIWithBooks）
```
