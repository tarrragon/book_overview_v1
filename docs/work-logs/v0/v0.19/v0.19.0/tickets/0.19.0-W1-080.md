### 實作摘要

依 PM 預填 Action 嚴格執行三項變更：

1. **新建 `tests/unit/scripts/script-test-helpers.js`**
   - 提供 `createMinimalFsAdapter(paths[])`：僅 fileExists，用於純路徑存在性測試
   - 提供 `createFullFsAdapter(fileMap)`：fileExists + readFile + readdir + stat，用於需遍歷/讀取的測試
   - 兩個工廠命名明確表達語意差異，消除 TD-1 同名異介面 naming-confusion

2. **遷移既有測試至共用 helper**
   - `tests/unit/scripts/build-version-check.test.js`：原 `createMockFsAdapter(files)`（Map 介面）→ `createFullFsAdapter(files)`，10 處呼叫全替換
   - `tests/unit/scripts/validate-manifest.test.js`：原 `createMockFsAdapter(existingFiles)`（陣列介面）→ `createMinimalFsAdapter(existingFiles)`，10 處呼叫全替換
   - 兩檔本地 `createMockFsAdapter` 函式定義移除

3. **`scripts/build.js` esbuild early validation**
   - 新增 `ensureEsbuildAvailable()` 純函式：強制呼叫 `getEsbuild()` + 驗證 `mod.build` 是 function
   - 失敗時拋出帶 3 條修復建議的 Error（node_modules 重裝 / platform binary 確認）
   - 在 `build()` 開頭呼叫一次（在 version check 之前），確保副作用（rmSync/mkdirSync/copyRecursive）發生前 fail-fast
   - 嚴格限縮在 `getEsbuild()` 子系統，不觸碰 W1-083 範圍

### W1-083 隔離度自檢（PM 預填強制要求）

| 子系統 | W1-080 是否觸碰 | 驗證 |
|--------|---------------|------|

| `validateVersionAlignment` | 否 | `git diff` 確認該函式 0 行變更 |
| `detectActiveWorklogVersion` | 否 | `git diff` 確認該函式 0 行變更 |
| `WORK_LOGS_ROOT` 常數 | 否 | `git diff` 確認該常數 0 行變更 |

| `module.exports` | 否 | 仍為 `{ detectActiveWorklogVersion, validateVersionAlignment, defaultFsAdapter }` |

`ensureEsbuildAvailable()` 與版號驗證子系統獨立呼叫，不互動。註解內已標記範圍邊界，W1-083 可獨立 reshape 版號驗證而不影響 esbuild 驗證。

### 設計決策

- **early validation 位置**：放在 `build()` 開頭、version check 之前。理由：esbuild 載入失敗的修復方法（npm install）與版號不一致無關，先驗證最基礎的工具鏈再驗證版號契約，錯誤訊息更聚焦。
- **不抽 esbuild 驗證到 module top-level**：保留 lazy require 設計（避免 jsdom 測試載入崩潰，原註解已說明），驗證僅在 `build()` runtime 觸發。
- **helper 命名為 W1-083 預留擴展**：`createFullFsAdapter` 支援 readdir/stat/readFile，W1-083 若需新增 worklog 同步測試可直接複用，無需第三個 helper。

- esbuild 模組層級 require（保留 lazy 設計）
