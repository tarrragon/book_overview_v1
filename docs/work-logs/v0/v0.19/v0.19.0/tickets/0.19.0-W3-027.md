### 問題根因

Claude Code v2.1.145 新增 `claude agents --json` 指令，輸出當前機器上所有活躍 Claude Code sessions 的 JSON 資訊。專案現有 agent 觀察體系（dispatch-active.json + TaskOutput + transcript_tail_reader + SubagentStop hooks）依賴多個自建機制追蹤代理人狀態，部分依賴 transcript JSONL 解析（PC-009 等已知盲點）。需評估新指令能否補強或簡化現有體系。

### 影響範圍

- `.claude/hooks/dispatch-record-hook.py` — 派發記錄
- `.claude/hooks/subagent-stop-dispatch-cleanup-hook.py` — 完成清理
- `.claude/hooks/lib/dispatch_tracker.py` — dispatch-active.json 狀態管理
- `.claude/hooks/lib/transcript_tail_reader.py` — transcript JSONL 解析
- `.claude/references/pm-agent-observability.md` — PM 觀察指南（五工具體系）
- `.claude/skills/agent-team/` — agent-team skill

### 相關 Error Pattern

- PC-009：handoff 標記錯誤（transcript 解析盲點之一）
- PC-050：PM 在代理人仍工作時誤判完成（dispatch-active.json + TaskOutput 為防護）
- PC-070：PM 以 Hook 訊號推論 agent 失敗（Hook 廣播可靠度問題）
## 重現實驗結果


### 實驗方法

在有活躍 subagent 派發的 session 中執行 `claude agents --json`，對比 dispatch-active.json 內容，驗證兩者資訊粒度差異。

### 實驗執行

1. 當前 session 有 3 個背景代理人（W3-026, W3-027, W3-031）
2. 執行 `claude agents --json` 輸出：

```json
[
  {
    "pid": 30405,
    "cwd": "/Users/tarragon/Projects/book_overview_v1",
    "kind": "interactive",
    "startedAt": 1779287955125,
    "sessionId": "44a7e18f-6aaa-4c16-b584-9791a4f8f25f",
    "status": "busy"
  }
]
```

3. 同時 dispatch-active.json 含 3 筆派發記錄，每筆有 agent_description、tool_use_id、agent_id、ticket_id、files、branch_name、dispatched_at

### 實驗發現

**已驗證事實**：

1. `claude agents --json` 輸出粒度為 **session 層級**（一個 interactive session = 一筆），不區分該 session 內的多個 subagent
2. 輸出欄位：pid, cwd, kind, startedAt, sessionId, status — **無 agent_id、無 ticket_id、無 files、無 branch**
3. subagent 不作為獨立 session 出現在 `claude agents --json` 輸出中（只有 parent interactive session 出現）
4. `status` 欄位值為 `busy`（正在處理中）— 這是 session 級別狀態，非 agent 級別

**仍未驗證的假設**：

- `claude agents --json` 在 headless/agent mode 下是否會列出 subagent sessions（目前環境為 interactive mode，subagent 未獨立列出）
- `status` 的所有可能值（目前只觀察到 `busy`）
- 未來版本是否會擴充欄位（如加入 agent_id 或 task 級別資訊）
## Solution

### WRAP 框架分析

### W — Widen（擴增選項）

四種整合方式對照：

| 方案 | 描述 | 整合成本 | 價值 | 風險 |
|------|------|---------|------|------|
| A. 純取代 | 用 `claude agents --json` 取代 dispatch-active.json + TaskOutput | 高 | 低 | **不可行**：輸出粒度為 session 級，無 agent_id/ticket_id/files，無法區分同一 session 內的多個 subagent |
| B. 補強（session 健康檢查） | 在 agent-failure-sop 的 Step 0.5 新增 `claude agents --json` 作為 session 存活性檢查（pid alive + status） | 低 | 中 | 低：僅新增一個輔助訊號，不改變既有體系 |
| C. 並存（orphan 偵測） | 在 dispatch_tracker.py 的 `cleanup_expired` 新增交叉驗證：若 dispatch-active.json 有記錄但 `claude agents --json` 無對應 session（pid 不存在），主動清理 orphan | 中 | 中 | 中：需處理 pid 比對邏輯，跨平台差異 |
| D. 不整合（持續觀察） | 暫不整合，等 Claude Code 擴充 subagent 級別 JSON 輸出後再評估 | 零 | 零 | 零 |

### R — Reality-test（現實檢驗）

對照現有 PC-009/PC-050/PC-070 盲點：

| 盲點 | `claude agents --json` 能否解決？ | 說明 |
|------|--------------------------------|------|
| PC-050：PM 誤判代理人完成 | **否** | 該指令不列出 subagent，無法判斷特定 agent 是否仍在執行。TaskOutput 仍是唯一可信的 agent 級狀態來源 |
| PC-070：Hook 廣播訊號與真實狀態不同步 | **部分** | 可用 pid 存活性作為「session 是否還活著」的外部驗證，但仍無法區分 session 內多個 agent |
| transcript 解析盲點 | **否** | 該指令不提供 transcript 內容，transcript_tail_reader 的用途（讀取 assistant 最後訊息）無法被取代 |
| orphan dispatch 記錄 | **是** | 若 session pid 已不存在，dispatch-active.json 記錄即為 orphan，可主動清理 |

**核心結論**：`claude agents --json` 的資訊粒度為 session 級，與專案需要的 agent/dispatch 級粒度不匹配。無法取代現有五工具體系中的任何一個。唯一有價值的整合點是 **orphan 偵測**（方案 C）和 **session 存活性輔助檢查**（方案 B）。

### A — Attain distance（評估衝擊）

| 既有元件 | 方案 B 衝擊 | 方案 C 衝擊 |
|---------|------------|------------|
| dispatch-active.json | 無變更 | cleanup_expired 新增 pid 驗證邏輯 |
| TaskOutput 使用流程 | 無變更 | 無變更 |
| SubagentStop hooks | 無變更 | 無變更 |
| transcript_tail_reader | 無變更 | 無變更 |
| pm-agent-observability.md | 新增一段「session 存活性檢查」說明 | 同左 + orphan 偵測說明 |
| agent-failure-sop.md | Step 0.5 新增可選步驟 | 無變更 |

衝擊範圍小，不破壞既有五工具互補架構。

### P — Prepare（失敗回退）

- 方案 B/C 均為增量式，回退成本 = 移除新增程式碼，不影響既有流程
- `claude agents --json` 若未來版本移除或格式變更，影響範圍限於新增的輔助功能
- 建議先實作方案 B（最低成本），觀察實際使用率後再決定方案 C

### 結論

**建議方案：D（不整合，持續觀察）**

理由：
1. 實驗證實 `claude agents --json` 粒度為 session 級，不列出 subagent，與專案 agent/dispatch 級需求不匹配
2. 方案 B（session 存活性）和方案 C（orphan 偵測）有邊際價值，但現有 SubagentStop hook + TaskOutput 已覆蓋主要場景
3. Claude Code 仍在快速迭代，後續版本可能擴充 subagent 級輸出，屆時整合價值將大幅提升
4. 當前投入整合的 ROI 不足以建 IMP ticket
