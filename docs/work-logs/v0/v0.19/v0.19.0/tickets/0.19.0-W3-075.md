### 根因分析（PM 前台分析，2026-05-28）

**Bug 重現流程**（2026-05-28 實證 v1.35.1 → v1.35.2）：

1. 用戶觸發 `/sync-push`，自上次推送後無新 .claude/ commit
2. `main()` line 783 `collect_claude_commits` 回傳空 list
3. line 793-794 fallback：`commit_message = "sync .claude configuration"`, `bump_suggestion = "patch"`
4. line 821 `bump_version()` 無條件 bump（v1.35.1 → v1.35.2）
5. line 827 **VERSION 檔被寫入新版號**
6. line 830 **CHANGELOG 被寫入新 entry（含當前日期）**
7. line 848 `git diff --cached --quiet` 偵測到 VERSION/CHANGELOG 有變動 → 不阻擋
8. line 853 commit + line 879 push 成功
9. 結果：遠端多一個內容相同（hash `191d124ba0687655`）但版號 +1 的空殼版本

### 既有阻擋機制失效原因

line 848 的 `diff --cached --quiet` early-exit 是「post-modification」檢查，VERSION/CHANGELOG 已被改寫；該檢查只能擋「實際檔案 diff 為空」但 VERSION 一定有 diff。

### 既有可重用工具

- `_compute_content_hash(claude_dir)` line 627-645：純函式，計算 .claude/ 內容指紋（前 16 字元 sha256）
- `.sync-state.json` line 895-905：已存 `last_push_hash` / `last_push_version` / `last_push_time`

### Early-exit 設計

**位置**：main() Step 2 之後（檢查 .claude 狀態完成），Step 3 clone 之前
- 不依賴 clone，純本地檔案系統操作
- 失敗 fast：節省 clone 時間（~5 秒）+ 避免後續 VERSION/CHANGELOG 寫入

**判斷邏輯**：
```
if .sync-state.json 存在 AND 可解析:
    last_hash = state["last_push_hash"]
    current_hash = _compute_content_hash(claude_dir)
    if last_hash == current_hash AND 無新 .claude/ commit since last_push_time:
        early-exit（除非 --force 旗標）
```

**雙重訊號設計理由**：
- 單獨 hash 比對：可能因 .sync-state.json 過時誤判
- 單獨 commit 計數：未必反映實質內容（例如 commit 後又 revert）
- 兩者同時為「無變更」才 abort，保守設計

**邊界處理**：
- `.sync-state.json` 不存在 → 首次推送，正常流程
- `.sync-state.json` 解析失敗（檔案損壞） → 正常流程（不 abort）
- user 提供 commit message 參數 → 跳過 early-exit（明確意圖）
- `--force` 旗標 → 跳過 early-exit（明確意圖）

### 實作摘要

1. 新增 `check_no_change_early_exit(claude_dir, project_root) -> tuple[bool, str]` 純函式
   - 雙重訊號設計：hash 比對 + commit 計數，兩者同時為「無變更」才回傳 `(True, reason)`
   - 邊界處理：state 檔不存在 / JSON 解析失敗 / 缺欄位 → 一律 fail-safe 回傳 `(False, ...)` 走正常流程
   - 回傳 `(should_exit, reason)` tuple 含人類可讀說明（含當前 hash + last hash 診斷資訊）

2. `main()` 加入 `--force` 旗標解析（與 `--clean` / `--dry-run` 同樣 sys.argv.remove 處理）

3. `main()` Step 2.5（檢查 .claude 狀態後、Step 3 clone 之前）呼叫 early-exit 檢查
   - 跳過條件：user 提供 commit message（明確意圖）OR `--force` 旗標
   - abort 路徑：print 黃色提示 + 提示 `--force` 用法 + `return`（不 clone、不 bump、不 commit、不 push）

4. 測試：`.claude/scripts/tests/test_sync_claude_push_early_exit.py` 共 6 個案例
   - test_first_push_no_state_file（首次推送）
   - test_corrupted_state_file（state JSON 損壞）
   - test_state_missing_fields（state 缺欄位）
   - test_hash_match_no_new_commit_aborts（核心 abort 路徑）
   - test_hash_match_but_has_new_commit_continues（罕見 commit 後 revert 案例）
   - test_hash_mismatch_continues（一般正常流程）

### 設計決策

- **位置選在 Step 3 clone 之前**：節省 clone 時間（~5 秒），且避免 VERSION/CHANGELOG 被寫入後再回滾的複雜度
- **雙重訊號保守設計**：單一 hash 比對可能因 .sync-state.json 過時誤判；單一 commit 計數可能因 commit-後-revert 無法反映實質內容；兩者交集才 abort
- **`--force` 而非 `--check-only`**：違反「最小驚奇」原則的是「無變更時繼續推送」，預設 abort 才符合直覺；想強制推送透過明確旗標
- **fail-safe over fail-fast**：state 檔損壞或缺欄位時走正常流程而非 abort，避免因狀態檔問題阻塞正常推送
