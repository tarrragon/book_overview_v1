## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

book interchange id/title 驗證邏輯在三處字面複寫（ARCH-020 duplicate validation logic 家族）：

| 位置 | 函式 | 非物件判斷 | 行為 |
|------|------|-----------|------|
| src/export/book-interchange-v1-adapter.js | assertRequiredBook（被 mapV1BookToCanonical / mapCanonicalToV1Book 呼叫） | `!isPlainObject(book)`（排除 null/陣列/原始型別） | throw BookValidationError |
| src/export/v1-to-v2-converter.js | convertV1ToV2Book | `!v1Book \|\| typeof !== 'object'`（**不排除陣列**） | throw BookValidationError |
| src/export/v1-to-v2-converter.js | convertAppLegacyToV2Book | `typeof !== 'object' \|\| Array.isArray`（排除陣列） | throw BookValidationError |

三處 missing 欄位收集邏輯（id/title）字面完全相同，皆依賴 BookValidationError。複寫導致驗證行為漂移風險：任一處修正其他兩處可能遺漏（W4-031.2 漏修同構即 ARCH-020 反覆發生模式）。

### 影響範圍

- src/export/book-interchange-v1-adapter.js（移除 local isPlainObject + assertRequiredBook，改 import）
- src/export/v1-to-v2-converter.js（移除兩處 inline 驗證 block，改 import）
- 新建 src/export/book-validation-helpers.js（共用 helper）
- consumer 透傳測試：book-data-exporter-canonical / overview-import-private-methods / content-parser-interchange（皆未直接斷驗證 block，等價透傳）

### 相關 Error Pattern

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 重構摘要

抽取共用 `assertBookHasIdTitle(book)` + `isPlainObject(value)` 至新模組 `src/export/book-validation-helpers.js`（兩消費端皆在 src/export/，依既有 co-location 慣例就近放置，非憑空新建抽象層）。

- adapter：移除 local `isPlainObject` + `assertRequiredBook`，改 `require('./book-validation-helpers')`，兩 call site（mapV1BookToCanonical / mapCanonicalToV1Book）改呼叫 `assertBookHasIdTitle`。adapter 其餘 6 處 `isPlainObject` 使用改用 import 版本（同實作）。
- converter：移除 convertV1ToV2Book / convertAppLegacyToV2Book 兩處 inline 驗證 block，改呼叫 `assertBookHasIdTitle`；`BookValidationError` import 移除（已無直接使用，僅存於 doc comment）。

### 零行為改變等價性論證

helper 採 `isPlainObject`（排除 null/陣列/原始型別）+ 共同 id/title missing 收集邏輯，作為三處單一真值來源：

| call site | 原非物件行為 | helper 後行為 | 等價性 |
|-----------|------------|--------------|--------|
| adapter（assertRequiredBook） | `!isPlainObject` 排除陣列 | `!isPlainObject` 排除陣列 | **完全相同**（helper 直接搬移原碼） |
| convertAppLegacyToV2Book | `Array.isArray` 排除陣列 | `!isPlainObject` 排除陣列 | **完全相同**（兩種寫法對陣列/null/原始型別判定一致）；測試 line 60 `convertAppLegacyToV2Book([]) → throw` 維持綠 |
| convertV1ToV2Book | `typeof !== 'object'` **不排除陣列** | `!isPlainObject` 排除陣列 | 唯一差異：陣列輸入由「進入後續 `![].id` → missing 欄位錯誤」改為「`must be an object` 錯誤」。兩者皆 throw BookValidationError，且：(a) convertV1ToV2Book **無外部 caller**（僅 convertV1ToV2Data 內部逐本迭代，傳入的是 book element 非陣列本身），(b) **無陣列輸入測試/consumer 依賴該錯誤 payload**（grep 確認），(c) convertV1ToV2Data 對任一錯誤皆 try/catch 跳過。故差異**不可觀測**，且 helper 形式為更正確的 array-rejecting 語意。 |

missing 欄位 throw 行為（id/title 收集結構、BookValidationError 型別）三處原本字面相同，helper 後完全保留。

### 次要項（acceptance 3）豁免決策

`accumulateTree` 內 nodeId/parentId 推導**豁免抽取 computeNodeIds**。理由（狀態 a 明確結論，非延後）：
1. inline 形式為 2 個 ternary 派生（`isLeaf ? node.id : deriveStableId(...)` / `i===0 ? null : deriveStableId(...)`），命名自說明，認知密度已低（指數約 5）。
2. 抽取需傳 (node, segments, i, isLeaf) 4 參數，indirection 成本高於可讀性收益。
3. id 派生是 round-trip 行為關鍵邏輯；零行為改變重構下保留 inline 降低風險面。
4. 主目標（ARCH-020 三處驗證複寫）已完整消除，為本 Phase 4b 高價值產出。
