### 問題根因

**Why（建立此 error-pattern 的動機）**：Jest 測試環境（Node.js + jsdom，內建 `global`/`process`/`require` 等 Node-only globals）與 Chrome Extension runtime（esbuild IIFE bundle + 瀏覽器，無 Node.js polyfill）的 API 可用性差異未被防護。當 `src/` 程式碼直接使用 Node-only API 且無 `typeof` guard 時，Jest 單元測試 100% 通過、acceptance 勾選 + commit，但 CE runtime 立即 `ReferenceError` 崩潰——這是「測試綠燈但實機失敗」的典型 anti-pattern。

**直接觸發案例**：0.19.0-W1-001.2.1 實機驗證 UC-04 JSON 匯入流程時，`src/utils/file-reader-factory.js:24,37` 使用 `global.FileReader` 在 CE overview runtime 立即崩潰，匯入功能完全失效；前置的 0.19.0-W1-047.1~.5 五個 IMP（BookFileImporter 版本偵測、v2 結構提取、覆蓋/合併模式、UI 接線）全部 acceptance 勾選 + commit 但實機 100% 不可用，根因即此類環境前提誤判。

**系統根因**：esbuild build.js 配置 `platform: browser, bundle: true, format: iife` 但**未配置** `define: { 'process.env.NODE_ENV': ... }` 或 Node.js polyfill；Jest jsdom 環境提供 Node.js globals 遮蔽問題；單元測試斷言聚焦邏輯正確性而非環境前提，三層共振讓修復鏈每步綠燈但端到端 runtime 失效。

### 影響範圍

**程式碼層級**（依 0.19.0-W1-050 ANA 全 src 盤點）：

| 類別 | 致命級（runtime 100% 崩潰） | CE Runtime 影響 |
|------|--------------------------|----------------|
| `global.*` 裸用 | F1-F3, F5, F9, F10（6 處 / 5 檔） | overview / SW / content / popup |
| `process.env.NODE_ENV` | P7-P16（13 處 / 11 檔） | 全部 4 個 runtime |
| `process.hrtime/nextTick/memoryUsage/on` | F4, F6-F8, F17-F21（多處 / 8 檔） | SW / overview |
| `require('crypto')` | F11-F16（6 處 / 6 檔） | SW |

**合計**：致命 21 處 + 潛在 17 處（含 `window.matchMedia` 在 SW、`window.ClassName` 暴露無效等）。盤點完整表格見 `docs/work-logs/v0/v0.19/v0.19.0/tickets/0.19.0-W1-050.md` §Solution。

**流程層級**：影響所有依賴 Jest jsdom 驗證但部署至 CE runtime 的 ticket 完成度判定——任何「acceptance 全綠 + commit」的 IMP 都可能在實機 100% 失效，造成後續 ticket 基於失敗修復推進，連鎖回溯成本高。

**Consequence**：未建立 error-pattern 防護會讓此類「測試綠燈但實機失敗」事件反覆發生；W1-047.1~.5 → W1-001.2.1 阻塞 → W1-050 全域盤點 → W1-050.1~.6 修復鏈是此 anti-pattern 缺乏防護的具體代價（單一根因引發 8+ ticket 連鎖修復）。
