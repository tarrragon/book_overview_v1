### 問題根因

W1-002.2 由 PM 規劃為「驗證 ZIP 安裝至 3 個 Chrome Profile」的 IMP ticket，who=thyme-extension-engineer，但 5 個 acceptance 條目本質含 **OS 層級的用戶手動操作**，agent 無法獨立完成：

| Acceptance 條目 | 可執行性分析 |
|---|---|
| (1) Profile A (clean) 安裝 Service Worker 0 error + content script 注入 | thyme 可透過 chrome-devtools MCP 操作當前 profile，但無法建立「全新 clean profile」 |
| (2) Profile B (multiple extensions) 安裝後無衝突 | 需事先準備「裝 5+ 個其他 extension 的 profile」——user 環境準備，agent 不可控 |
| (3) Profile C (dev mode off) 顯示明確提示 | 需 toggle Chrome 設定 chrome://extensions/ 開發者模式 —— OS 層級設定 |
| (4) 3 個 profile 安裝時間 < 30 秒 | 需手動量測 user-perceived 安裝體感時間 |
| (5) 每個 profile 截圖或文字描述記錄到 Test Results | 截圖需 user 操作 OS screenshot 工具 |

**設計層面問題**：

1. **agent vs user 分工不清**：ticket who=thyme，但實質執行者必為 user；who 欄位失真，handoff 接手代理人無從推進
2. **acceptance 顆粒度過大**：5 個條目綁定在單一 ticket，違反 Atomic Ticket 原則（atomic-ticket-methodology），且各條目失敗時無法獨立追蹤
3. **內測 gate 必要性未經評估**：「3 個 profile」是 PM 規劃時的保守設計，是否真為內測 v1.0 release 阻擋條件？對照 use-cases.md v1.1 內測需求未必需要 3 profile

**視角 2：Atomic Ticket 合規性**

**視角 3：執行者匹配**

| Acceptance | 可自動化部分 | 必須手動部分 |

| (2) Profile B multi-ext 無衝突 | permissions 分析（storage/activeTab 無衝突風險） | 準備 5+ extension 環境、實際安裝測試 |
| (3) Profile C dev mode off 提示 | 無（Chrome 原生行為，非 extension 控制） | toggle dev mode、觀察 Chrome 提示文字 |
| (4) 安裝時間 < 30 秒 | 無 | 手動計時 |

自動化驗證已執行結果：

- SW/CS 檔案均存在於 build/production/
- permissions=[storage, activeTab]，host_permissions=[*://*.readmoo.com/*]——無 broad permissions 衝突風險
- tests/ 目錄無 manifest/SW/CS 專項自動化測試

**視角 4：減裁可行性**

- Profile C (dev mode off) 測試的是 Chrome 原生行為（unpacked extension 需開啟 dev mode），非本 extension 功能——Chrome 已有明確 UI 提示，無需 extension 額外處理
- Profile B (multi-extension) 風險低：permissions 僅 storage + activeTab，無 webRequest/tabs 等高衝突 API
- 安裝時間 < 30 秒：Load Unpacked 是檔案系統操作，秒級完成，無需量測
- 最小必要驗證：1 個 clean profile + Load Unpacked + SW DevTools 0 error + 訪問 Readmoo 確認 CS 注入

### 實驗發現

**已驗證事實**：
1. 多 profile 驗證非 use-cases.md v1.1 內測 gate 要求
2. W1-002.2 acceptance 認知負擔過高（指數 ~15）
3. 5 項中 0 項可由 thyme agent 獨立完成（全需用戶手動）
4. manifest 結構正確，SW/CS 檔案存在，permissions 無衝突風險
5. Profile C 測試的是 Chrome 原生行為，與 extension 功能無關

**結論**：W1-002.2 設計過度（3 profile + 計時 + 截圖），建議大幅減裁為 1 profile 基本驗證 + 自動化前置檢查。

### (a) W1-002.2 五項 Acceptance 可執行性分類表

**採方案 B：精簡為 1 profile + 自動化前置檢查**

| 條目 | 決策 | 理由 |

| Profile B multi-ext | 移除 | permissions 無衝突風險，內測階段無需驗證 |
| Profile C dev mode off | 移除 | Chrome 原生行為，非 extension 功能測試 |
| 安裝時間 < 30 秒 | 移除 | 無鑑別力，Load Unpacked 必然秒級 |
| 截圖/文字記錄 | 降級為選填 | Test Results 章節以文字描述為主 |
| 新增：自動化前置檢查 | 新增 | manifest 結構驗證 + SW/CS 檔案存在性 + permissions 合理性——agent 可執行 |

**方案比較**：

| 方案 | 內容 | 評估 |
|---|---|---|
| A 維持原設計 | 3 profile + 計時 + 截圖 | 認知負擔過高，who 欄位失真，gate 卡在非必要驗證 |
| **B 精簡（採用）** | 1 clean profile 手動 + 自動化前置檢查 | 覆蓋內測真實需求，agent/user 分工清晰 |
| C 全減裁 | 僅自動化檢查，無手動驗證 | 風險：跳過實機安裝驗證，內測信心不足 |
