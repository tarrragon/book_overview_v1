## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

**單一根因，雙重症狀**：worktree skill 安裝為獨立 uv tool 後，`worktree_manager.py:48-51` 用 `Path(__file__).resolve().parent.parent.parent.parent.parent` 推導專案根目錄，但安裝後 `__file__` 位於 uv tool venv（`~/.local/share/uv/tools/worktree-skill/lib/python3.13/site-packages/scripts/worktree_manager.py`），往上 5 層解析到 `~/.local/share/uv/tools/worktree-skill`（uv tool root），與專案根目錄 `/Users/tarragon/Projects/book_overview_v1` 完全無關。

連帶：sys.path 插入錯誤路徑 → `from git_utils import ...` 失敗 → 觸發 `worktree_manager.py:62-96` fallback → `run_git_command` 永遠返回 `(False, "git_utils unavailable")` → 任何 git 分支驗證（含 main 存在性檢查）必失敗 → 誤判「基礎分支不存在 main」。

### 重現驗證

執行 `worktree status` 後輸出：
- `[Warning] Failed to import git_utils: No module named 'git_utils'`
- `[Warning] Worktree SKILL may not function properly`

路徑計算驗證：
- `__file__`: `~/.local/share/uv/tools/worktree-skill/lib/python3.13/site-packages/scripts/worktree_manager.py`
- `project_root` (5 levels up): `~/.local/share/uv/tools/worktree-skill`
- 計算出的 git_utils 目標: `~/.local/share/uv/tools/worktree-skill/.claude/lib/git_utils.py` (exists: False)
- 實際 git_utils 位置: `/Users/tarragon/Projects/book_overview_v1/.claude/lib/git_utils.py`

### 影響範圍

所有 worktree 子命令（create / status / merge / cleanup）皆走 fallback 模式，導致：
- create: 任何 git 操作失敗 → 「基礎分支不存在 main」誤判
- status: 顯示「目前沒有任何 worktree」（fallback get_worktree_list 返回 []）
- merge / cleanup: 前置驗證失敗

W1-110 agent 已透過手動 git worktree add 繞過，但所有後續 IMP 派發若用 worktree skill 都會中招。

### 相關 Error Pattern

- IMP-074（W11-031）Skill scripts package vs sys.path import 衝突 - 同根因家族
- PC-135 修共用 lib 後 caller sync - git_utils.py 屬 hooks/lib 共用資產
- PC-159（W3-050）安裝指令 fresh shell 驗證 - 安裝時未驗證從專案外執行 entry point 的行為

### 修復方向（建議方案 D：雙策略 fallback）

候選方案：

| 方案 | 原理 | 風險 |
|------|------|------|
| A. git rev-parse --show-toplevel 動態定位 | 從 worktree cwd 上溯到 git root | worktree 命令在非 git 目錄執行時失敗（但本就應失敗） |
| B. CLAUDE_PROJECT_DIR 環境變數 | Claude Code 慣例，hooks 普遍使用 | 環境變數未設時需 fallback |
| C. 打包 git_utils.py 進 worktree-skill | 避開 sys.path 依賴 | 違反 DRY（IMP-074 同類陷阱） |
| D. A + B 雙策略 fallback | 環境變數優先 + git rev-parse 兜底 | 邏輯較複雜但最穩 |

建議方案 D：CLAUDE_PROJECT_DIR 優先 → fallback 至 git rev-parse --show-toplevel → 雙雙失敗才報錯。Phase 3a (pepper) 可調整方案選擇。

不在範圍：
- 不修改 git_utils.py 本身（屬 .claude/lib 共用資產）
- 不重新設計 worktree skill 安裝模式（仍走 uv tool）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 採用方案：D（雙策略 fallback）

修復 `.claude/skills/worktree/scripts/worktree_manager.py` 第 48-51 行 project_root 推導邏輯。

### 執行步驟

1. **修改 project_root 推導**：取代 `Path(__file__).resolve().parent.parent.parent.parent.parent` 為：
   - 優先讀環境變數 `CLAUDE_PROJECT_DIR`（Claude Code 慣例）
   - Fallback：`subprocess.run(["git", "rev-parse", "--show-toplevel"], cwd=os.getcwd())` 從 cwd 上溯
   - 雙雙失敗才報錯（含明確錯誤訊息指引修復）
   - Pepper Phase 3a 可調整方案

2. **重新安裝**：`uv tool install --reinstall .claude/skills/worktree/`

3. **重現驗證**（主 repo cwd）：
   - `worktree status` 不再出現 `[Warning] Failed to import git_utils`
   - `worktree status` 不再出現「目前沒有任何 worktree」（除非確實沒有）
   - 建測試 worktree 不再誤報「基礎分支不存在 main」

4. **新增 regression test** 至 `.claude/skills/worktree/tests/test_path_resolution.py`：
   - 模擬 `__file__` 在 uv tool 安裝路徑下（mock Path）
   - 驗證 project_root 透過 `CLAUDE_PROJECT_DIR` 環境變數能正確定位
   - 驗證 fallback 至 `git rev-parse --show-toplevel` 能正確定位
   - 驗證兩者皆失敗時報明確錯誤

5. **跑全套 worktree skill test**：`(cd .claude/skills/worktree && uv run pytest tests/ -v)`

6. acceptance 全通過後 commit 並 complete ticket。

### 注意事項

- 禁止用 `cd`（用子 shell 或 `uv -d`，bash-tool-usage-rules 規則一）
- commit 範圍精確：只 `git add .claude/skills/worktree/` 內檔案
- 不修改 `.claude/lib/git_utils.py`（屬共用資產）
- 不重新設計安裝模式（仍走 uv tool）
- W1-110 agent 之前用手動 `git worktree add` 繞過，本修復目標是讓 worktree skill 恢復可用
