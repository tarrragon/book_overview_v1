## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

MessageDictionary 系統存在兩個架構層級問題：

**議題 A（作用域邊界缺失）**：GlobalMessages（`_loadDefaultMessages()`）混入大量 module-specific key。三個模組（popup、search-filter、validator）各自以 `new MessageDictionary({...})` 建立 local dict 並註冊 module-specific key，但 GlobalMessages 的 `_loadDefaultMessages()` 同時也包含這些模組的 key（如 `VALIDATOR_INIT`、`SEARCH_CLEARED`、`COMPONENT_INIT`），造成 key 雙重存在。沒有規則定義 GlobalMessages 的納入標準。

**議題 B（命名規範缺失）**：messageKey 命名無前綴規範強制模組歸屬。`POPUP_OPENED` 看似 popup-specific 但在 GlobalMessages，`COMPONENT_INIT` 看似通用但只被 search-filter 使用。開發者無法從 key 名稱判斷「該 key 應屬哪個 dict」。

### 影響範圍

| 檔案 | 角色 |
|------|------|
| `src/core/messages/MessageDictionary.js` | GlobalMessages 定義 + `_loadDefaultMessages()` |
| `src/popup/popup.js` | local dict: 22 個 popup-specific key |
| `src/ui/book-search-filter-integrated.js` | local dict: 12 個 search-UI-specific key |
| `src/platform/readmoo-platform-migration-validator.js` | local dict: 22 個 validator-specific key |

無直接對應的 PC error-pattern。本議題屬架構設計缺口（無規範導致污染），非實作錯誤。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 議題 A 診斷：GlobalMessages 作用域邊界

### 納入標準定義

GlobalMessages 應僅包含**跨模組共用 key**，定義為：

| 條件 | 說明 |
|------|------|
| 被 2+ 獨立模組引用 | 同一 src/ 子目錄視為同一模組 |
| 屬通用詞彙 | 錯誤碼（NETWORK_ERROR）、操作狀態（SUCCESS/FAILED）、系統生命週期（SYSTEM_READY） |
| 無模組前綴 | key 名稱不含 POPUP_/SEARCH_/VALIDATOR_ 等模組識別詞 |

### 合格 GlobalMessages key 清單（約 20 個）

錯誤類：`VALIDATION_FAILED`, `NETWORK_ERROR`, `STORAGE_ERROR`, `PERMISSION_DENIED`, `UNKNOWN_ERROR`
操作類：`OPERATION_START`, `OPERATION_COMPLETE`, `OPERATION_CANCELLED`, `OPERATION_TIMEOUT`, `OPERATION_RETRY`
系統類：`SYSTEM_READY`, `SYSTEM_SHUTDOWN`, `LOADING`, `PROCESSING`
通用 UI：`SUCCESS`, `FAILED`, `RETRY`, `CANCEL`, `CONFIRM`
測試專用：`TEST_MESSAGE`, `TEST_WITH_PARAMS`

### 應移除的 key（約 60+ 個）

- search-filter 專屬 8 個（已有 local dict）
- validator 專屬 15 個（已有 local dict）
- extractor/content 專屬 ~18 個（需建 local dict）
- Chrome Extension 專屬 4 個（`POPUP_OPENED` 等，未被引用可直接移除或歸 popup local dict）
- 日誌模板 4 個（`DEBUG_MESSAGE` 等，屬 Logger 模組 specific）
- 書庫相關 5 個（多數僅被 1 個模組引用）
- 中文 key 3 個（`未知的篩選條件` 等，屬遺留，應移至對應 local dict 或移除）

---

### 議題 B 診斷：防止未來污染的方案

### 方案 1：命名前綴 lint（輕量）

**做法**：建立 messageKey 命名規範文件 + ESLint custom rule 或 code review checklist，規定含模組前綴（如 `POPUP_`、`VALIDATOR_`）的 key 禁止出現在 `_loadDefaultMessages()` 中。

| 優點 | 缺點 |
|------|------|
| 零 runtime 成本 | 依賴人工遵守或 lint 維護成本 |
| 不改既有 API | 無法阻止命名不含前綴的 module-specific key（如 `CACHE_CLEANUP`） |
| 可立即實施 | lint 規則需要維護白名單 |

### 方案 2：register API 加 scope 參數（中量）

**做法**：`MessageDictionary` constructor 或 `addMessages()` 加 `scope` 參數（如 `'global'` / `'popup'` / `'validator'`），GlobalMessages 只接受 `scope='global'` 的 key。

| 優點 | 缺點 |
|------|------|
| runtime 強制阻擋污染 | 需修改 constructor API（影響所有 caller） |
| scope 可作日誌分類依據 | 增加 API 複雜度 |
| 可自動化驗證 | 既有測試需大量更新 |

### 方案 3：拆分 GlobalMessages 為 readonly（推薦）

**做法**：`_loadDefaultMessages()` 回傳 `Object.freeze()` 的物件，GlobalMessages 建立後 freeze，禁止後續 `addMessages()`。各模組 local dict 獨立運作，不依賴 GlobalMessages 繼承。

| 優點 | 缺點 |
|------|------|
| 最強防護：freeze 後物理不可變 | 需確認無模組對 GlobalMessages 做 set/addMessages |
| 簡化心智模型：GlobalMessages = 唯讀常數表 | 需一次性清理 module-specific key |
| 零額外 API | 若有模組依賴 GlobalMessages 動態新增，需重構 |

### WRAP 框架審查

### W（Widen）：擴增方案

三方案已涵蓋光譜（lint 靜態 → API 動態 → freeze 物理）。第四選項「不做任何事」不可行——W1-004/W1-105/W1-106 連鎖事件已證明無邊界的 GlobalMessages 造成實際 bug。

### R（Reality-test）：現實檢驗

- 方案 1（lint）：專案目前無 custom ESLint rule 基礎設施，建立成本中等但長期維護高
- 方案 2（scope API）：影響 3 個 caller + GlobalMessages + 測試，變更面大
- 方案 3（freeze）：需先驗證是否有模組對 GlobalMessages 做 set()，grep 快速確認即可

### A（Attain distance）：機會成本

方案 3 的一次性清理成本最高（移除 60+ key），但完成後維護成本最低（freeze 後不可能再污染）。方案 1 成本最低但效果最弱。建議：方案 3 為主 + 方案 1 作輔助文件。

### P（Prepare to be wrong）：失敗預想

方案 3 風險：若有模組在 runtime 對 GlobalMessages 做 `set()` 或 `addMessages()`，freeze 會 throw。**防護**：實作前先 grep `GlobalMessages.set` 和 `GlobalMessages.addMessages` 確認無此用法。
