### 問題根因

W4-009 暴露專案 ESLint 債務累積問題：9 天內從 194 個違規增至 220 個（+26）。根因為 `git commit` 流程缺少 lint 把關機制，新引入的違規可無阻礙進入 main 分支。歷史違規清理（W4-018）屬「下游清理」，本 ticket 屬「源頭擋住」，優先級在歷史清理之前。

### 相關 Error Pattern

- 無新 error-pattern；本任務目的為建立預防機制避免 PC-XXX 類「ESLint 違規無人擋住」模式擴大。

### Phase 1 功能規格

**核心行為**：
1. `git commit` 觸發 `.husky/pre-commit` → 呼叫 `npx lint-staged`
2. `lint-staged` 對 staged `*.{js,jsx,ts,tsx,mjs,cjs}` 檔執行 `eslint`（不開 `--fix`，避免改動 user diff）
3. ESLint errors > 0 → 阻擋 commit（exit non-zero）+ 印出 errors 詳情
4. ESLint warnings 不阻擋（與既有 220 違規分開處理）
5. `git commit --no-verify` → 跳過 hook（git 內建機制，無需額外配置）

**非範圍（明示）**：
- 不修復既存 220 個違規（W4-018）
- 不對 staged 自動 `--fix`（後續可評估）
- 不檢查非 JS/TS 檔（.md / .yaml / .json 不涵蓋）
- 不替代 CI lint job（hook 為 local first line，CI 為 safety net）

**AC 對應**：
- AC1（hook 對 errors 阻擋 commit）→ `.husky/pre-commit` + `lint-staged` config
- AC2（--no-verify 繞過）→ git 內建行為，文件化即可
- AC3（CLAUDE.md 文件化）→ §5 補 `pre-commit hook 自動執行 lint`、`--no-verify` 緊急豁免使用情境
- AC4（README/CONTRIBUTING 提及 dev 環境）→ 補 `husky install` 由 `npm install` 自動觸發的說明

### WRAP 三問

**W（Widen）—— 候選方案**:
1. husky + lint-staged（推薦，業界 JS/Node 專案 90%+ 採用）
2. native git pre-commit shell（無 npm 依賴但 cross-platform 與 staged-only 邏輯需自寫）
3. simple-git-hooks（lightweight 但生態小、文件少）
選擇 1：契合既有 npm 工作流、新成員熟悉度高、cross-platform 由 husky 處理。

**A（Attain distance）—— 機會成本**:
擠壓的目標：W4-018（ESLint 規則調整 + 46 warnings 分類）。判斷：W4-017 屬「源頭擋」優先級高於 W4-018「歷史清理」；先擋住才不會邊清邊累積。

**P（Prepare to be wrong）—— 失敗預想**:
最可能失敗：husky 的 `prepare` script 與既有 `package.json` install 流程或 CI build 衝突。
防護：(1) commit 前用 `git diff package.json` 確認改動範圍最小（只 `prepare` + `lint-staged` config + devDependencies）；(2) 在 CI 環境驗證 `npm ci` + `npm run build:dev` 仍通過；(3) `.husky/_/` 目錄處理採 husky v9 預設（自動 gitignore）。

### 設計決策

| 決策點 | 採行 | 理由 |
|--------|------|------|
| husky 版本 | v9.1.7 | 最新穩定版；移除 v8 必須的 `husky.sh` source line，hook 為純 shell script |
| lint-staged 版本 | v17.0.7 | 對應 husky v9 + node 18+ 環境 |
| lint-staged glob | `*.{js,jsx,ts,tsx,mjs,cjs}` | 涵蓋本專案所有 JS/TS 源檔副檔名；不包含 .md / .yaml / .json |
| ESLint 執行方式 | `eslint`（不加 `--fix`） | 避免 hook 自動改動 user diff，errors 由開發者主動處理 |

| `.husky/_/` 處理 | 由 husky 自動產生 `_/. gitignore` 內含 `*` 自我忽略 | 無需手動加入專案 .gitignore |
| `--no-verify` 機制 | git 內建，文件化即可 | 不需額外配置；CLAUDE.md §5 明示使用情境與反濫用守則 |
