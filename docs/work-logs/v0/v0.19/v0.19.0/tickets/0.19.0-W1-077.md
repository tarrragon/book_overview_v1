## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-074 sanity check 的 SSOT 鏈為「worklog markdown 狀態標記 → 推斷預期版號 → assert package.json 一致」。語意上將人工手動維護的 markdown 標記置為 source of truth，而將機器可讀的 package.json 降為 derived value。這在版本管理語意中是倒置的——package.json 是 npm/build 生態唯一認定的版本來源。

### 影響範圍

- `scripts/build.js`（detectActiveWorklogVersion + validateVersionAlignment）
- `docs/work-logs/v0/` 下所有 worklog main.md 的「**狀態**: 開發中」標記
- `.claude/skills/version-release/` check 子命令（已以 package.json 為偵測來源之一）

### 相關 Error Pattern

- IMP-068：版號 bump sanity check 缺失導致靜默錯誤推送
- PC-093：無 trigger 延後決策累積（若保持現狀不明確 trigger）

## 重現實驗結果
### 實驗方法

靜態分析 W1-074 commits（2109a1bc, dc7fcd7e）的 SSOT 鏈 + 比對 version-release skill 的版本偵測優先序。

### 實驗執行

1. `detectActiveWorklogVersion` 掃描 worklog markdown `**狀態**: 開發中` 標記 → 推斷版號
2. `validateVersionAlignment` 將推斷版號與 package.json 比對 → 不一致 fail-fast
3. version-release skill SKILL.md 明確記載偵測優先序：`--version 參數 > git branch > package.json > git tag`

觀察：version-release skill 已將 package.json 定位為第三優先偵測源（高於 git tag），但 build.js sanity check 卻將 worklog 定位為 SSOT。兩處語意不一致。

### 實驗發現

已驗證事實：
- build.js sanity check 以 worklog 標記為 SSOT（W1-074 設計）
- version-release skill 以 package.json 為主要版本源（SKILL.md L61-70）
- 兩者語意方向相反，但目前不衝突（version-release 用於 release 階段，build.js 用於 build:prod 階段）

仍未驗證假設：
- 反轉後 worklog 自動 sync 的穩定性（需實作驗證）

## Solution

### Trade-off 表格

| 維度 | 方向 A：反轉 SSOT（package.json → worklog 自動 sync） | 方向 B：雙向同步 + pre-commit hook 防偏差 | 方向 C：保持現狀（worklog = SSOT） |
|------|------|------|------|
| 自動化程度 | 高：bump package.json 後 script 自動更新 worklog 標記 | 中：手動操作但 hook 攔截偏差 | 低：完全手動維護 worklog 標記 |
| 維護成本 | 低：單一 SSOT 無漂移風險 | 高：雙向同步邏輯複雜且 hook 需持續維護 | 中：現有 regex 容錯已足夠，但人工記憶負擔 |
| 風險 | 中：需修改 build.js 反轉斷言方向 + 與 version-release skill 整合 | 高：雙向同步引入新故障模式（hook 靜默失敗） | 低：已上線運行，已知穩定 |
| 語意正確性 | 高：package.json 是 npm 生態唯一版本源，語意自然 | 中：無明確 SSOT，兩者對等 | 低：markdown 標記作 SSOT 違反生態慣例 |
| 與 version-release skill 一致性 | 高：skill 已以 package.json 為主，方向統一 | 中：skill 仍以 package.json 為主，hook 是額外層 | 低：build.js 與 skill 語意矛盾 |

### 推薦方向

**推薦方向 A：反轉 SSOT（package.json 為唯一版本 source of truth，worklog 標記由 script 自動 sync）**

理由：
1. **語意正確**：package.json 是 npm/Chrome Extension 生態唯一認定的版本來源，將其作為 SSOT 符合 Linus 原則「Make it impossible to forget」
2. **與 version-release skill 一致**：skill 偵測優先序已以 package.json 為主，反轉後全系統版本語意統一
3. **降低人工記憶負擔**：開發者 bump package.json 後 worklog 標記自動更新，無需記得手動維護兩處

實作策略：
- build.js `validateVersionAlignment` 改為：讀 package.json 版號 → 確認對應 worklog 目錄存在（不再要求 worklog 標記作為斷言來源）
- 新增 `syncWorklogStatus` 函式：package.json bump 時自動將對應 worklog 標記為「開發中」
- version-release `start` 子命令已有類似功能（建立 worklog），可整合

### Dissenting View（方向 C 保持現狀）

方向 C 的合理論據：
1. **已驗證穩定**：W1-074 已通過測試並上線，現有 sanity check 在 build:prod 有效防護
2. **改動風險**：反轉需修改已通過 parallel-evaluation 的程式碼，引入新 bug 的風險存在
3. **實際問題頻率低**：版號不一致問題在整個開發週期僅發生 1 次（v0.18.0 → v0.19.0 轉換），投入反轉的 ROI 可能不高

反駁：
- 穩定不等於正確；語意倒置在系統規模擴大時會成為認知負擔
- version-release skill 與 build.js 的 SSOT 矛盾會在 v1.0 多書城/多版本時造成維護困惑
