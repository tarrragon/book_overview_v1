## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`create.py` 原 `_parse_cli_args_to_config` 內對每個 --acceptance 值無條件
`item.split("|")`，將分隔符（vertical bar）視為多條拆分字元，但無逃逸機制。
acceptance 內文本身合法使用該字元（如描述 shell pipe「-q | tail」）時，會被靜默
拆成多條 acceptance，且無任何提示。

### 影響範圍

- `.claude/skills/ticket/ticket_system/commands/create.py`（acceptance 解析）
- 所有透過 `ticket create --acceptance` 建立含分隔符內文的 ticket。

### 相關 Error Pattern

PC-079（backtick command substitution in CLI args）同家族：CLI 參數內含工具特殊
字元被誤解析。本案為分隔符特殊字元被無條件拆條。

### 方案選擇：(a) + (b) 並用

選 **(a) 逃逸機制 + (b) 偵測 warn 並用**。

- 理由：(b) 單獨可提醒但無法讓使用者「正確表達內文含分隔符」；(a) 單獨提供逃逸
  但使用者未必知道何時需要。兩者互補：未跳脫時拆條並 emit warn 提示可用反斜線
  跳脫（指向 (a)），跳脫後保留字面不拆條也不警告。
- 風險低：跳脫採逐字元掃描的 `_split_unescaped`，僅 `\\|` 還原為字面 `|`，其餘
  行為與原 `split("|")` 等價，向後相容（既有不含跳脫序列的 acceptance 行為不變）。

---

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 變更摘要

1. `commands/create.py`
   - 新增 `_split_unescaped(text, sep)`：逐字元掃描，`\\sep` 還原為字面 sep
     不拆，其餘等價於 split。
   - 新增 `_parse_acceptance_items(raw_items)`：套用 `_split_unescaped` + 去
     空白/去空項；單一值未跳脫拆出 > 1 段時回傳拆條警告。
   - `_parse_cli_args_to_config` 改呼叫 helper，將警告以 `print(..., file=sys.stderr)`
     輸出（規則 4 雙通道：對使用者可見）。
2. `lib/command_lifecycle_messages.py`：新增 `CreateMessages.ACCEPTANCE_PIPE_SPLIT_WARNING`
   訊息模板（含拆條預覽與跳脫指引）。
3. `SKILL.md`：create-command 區補註分隔符限制與反斜線跳脫用法 + warn 行為。

### 不在本 ticket 範圍

- 其他多值參數（--where / --blocked-by / --related-to）採逗號分隔，內文含逗號
  的需求未在本 ticket 觀察到，未一併處理。
