#### 1. `require_in_progress` helper 函式規格

**檔案位置**：`.claude/skills/ticket/ticket_system/lib/ticket_lifecycle.py`（新增於 TicketLifecycle 類別上方的 module-level helpers 區域，緊鄰 `sync_completion_body_fields` 之後）

**函式簽章**：

```python
def require_in_progress(
    ticket: Dict[str, Any],
    ticket_id: str,
    operation: str,
    *,
    allow_completed: bool = False,
    force: bool = False,
) -> Tuple[bool, Optional[str]]:
    """檢查 body 操作的 status precondition。

    Args:
        ticket: 已 load 的 ticket dict（含 status 欄位）
        ticket_id: ticket ID，用於錯誤訊息
        operation: 呼叫端操作名稱（"append-log" / "set-acceptance" / "complete"），
            用於錯誤訊息與 hook-logs 記錄
        allow_completed: 是否允許 status=completed（True 用於 append-log 補 review）
        force: 是否旁路檢查（記錄到 hook-logs，returns (True, None)）

    Returns:
        (ok, error_msg)
        - ok=True, error_msg=None：通過檢查（或 --force 旁路）
        - ok=False, error_msg=str：阻擋，error_msg 含建議的後續指令
    """
```

**行為定義**：

| status | allow_completed=False | allow_completed=True | force=True |
|--------|---------------------|---------------------|-----------|
| pending | 拒絕，建議 `ticket track claim <id>` | 拒絕，建議 claim | 通過 + log |
| in_progress | 通過 | 通過 | 通過 |
| completed | 拒絕，建議 `ticket track reopen <id>` 或 `--force` | 通過（補 review 場景） | 通過 + log |
| blocked | 拒絕，建議「先解除阻塞或 release」 | 拒絕 | 通過 + log |
| closed | 拒絕，建議「該 ticket 已 closed，不可修改」 | 拒絕 | 通過 + log |

**重要**：helper **不直接 print 或 exit**，僅回傳結果。由 caller 決定如何輸出與退出（保持單一職責，便於測試）。

**`force=True` 副作用**：寫入 `.claude/hook-logs/cli-force-usage.jsonl`（append-only JSONL），每行格式：

```json
{"timestamp": "2026-05-25T14:30:00", "ticket_id": "0.19.0-W3-044", "operation": "append-log", "status_at_time": "pending", "reason": "force-flag"}
```

> **Why hook-logs 不是 ticket md**：force 使用是元事件（meta-event），不屬 ticket 內容；且若記入 ticket md，force 旁路 status 檢查的同時又寫 ticket md，自我矛盾。

```python
ok, error_msg = require_in_progress(
    ticket, ticket_id, "append-log",

)

    print(error_msg, file=sys.stderr)
    return 2  # exit code 2 = precondition violation
```

#### 3. `--force` flag 設計

**CLI 表面**：每個 entry point 的 argparse 新增 `--force` flag（`action='store_true'`，預設 False）。

| Entry Point | Flag 名稱 | 衝突檢查 |
|-------------|---------|---------|
| `append-log` | `--force` | 與既有 flag 無衝突 |
| `set-acceptance` | `--force` | 與既有 flag 無衝突 |
| `complete` | `--force` | **與既有 `force` 參數合併使用**（既有 force 用於 pending children 旁路；本 force 用於 status precondition 旁路；語義相容可合併） |

**使用追蹤**：

- 寫入路徑：`.claude/hook-logs/cli-force-usage.jsonl`（已存在則 append，不存在則建立）
- 由 `require_in_progress` 在 `force=True` 路徑內寫入
- 統計用途：後續若 `--force` 使用 > 0，PM 可定期 review 是否為誤用模式（記入 PC error-pattern）

**stderr 提示**：使用 `--force` 時 stderr 輸出警告：

```
[Warning] --force 旁路 status precondition 檢查；使用已記入 hook-logs
```

---

#### 4. completed 狀態的特例處理

**設計決策**：`append-log` 允許 `status=completed` 通過（`allow_completed=True`），其他兩個入口拒絕。

**Why**：

- `append-log`：completed 後仍可能需補 review 評論、後續發現的 follow-up 記錄、auditor 結論。禁止會迫使用戶 `reopen` 後 `complete`，產生不必要的 status 漂移。
- `set-acceptance`：completed 後改 AC 違反語義（AC 已驗證完畢，不應改變）。如需修改，先 `reopen`。
- `complete`：既有 `is_already_complete` 邏輯已處理（顯示友好訊息 return 0），無需重複阻擋。

**與既有 close 命令的關係**：本規格不影響 `close` 命令（close 對 completed → closed 已有明確路徑，無 body 操作語義）。

#### 5. 錯誤訊息格式

**統一格式**（所有 status 違規）：

```
[Error] <ticket_id> 無法執行 <operation>：當前 status=<status>，需要 status=in_progress
        建議：<具體後續指令>
        逃生閥：--force（記入 hook-logs，需於後續 review 中說明理由）
```

**具體建議對照表**：

| status | 建議指令 |
|--------|---------|
| pending | `ticket track claim <ticket_id>` |
| completed（非 allow_completed 入口）| `ticket track reopen <ticket_id>`（若需修改）或檢視是否誤呼叫 |
| blocked | `ticket track release <ticket_id>` 或解除 blockedBy 依賴後 claim |
| closed | 該 ticket 已 closed；如需修改請建立新 ticket 處理 |

**訊息常數位置**：新增至 `.claude/skills/ticket/ticket_system/lib/messages.py` 的 `ErrorMessages` class，建議名稱 `BODY_OP_REQUIRES_IN_PROGRESS`，支援 `.format(ticket_id=, operation=, status=, suggestion=)` 佔位符。
