# 0.19.0-W1-009 功能規格

> Mode: Lite（單檔 IMP，修改 1 檔 `src/background/events/event-coordinator.js`，task_type=Implementation，不符 Full 觸發條件）

## 1. Purpose（目的）

修復 runtime 提取 pipeline 的 model 斷層：SW `event-coordinator` 的 `EXTRACTION.COMPLETED` handler 在存入 `chrome.storage.local` 前，未對提取的 book 物件套用 BookSchemaV2 model 轉換，導致 96 本書 `readingStatus` 全 undefined、無 `tagIds` 欄位。本 ticket 在 handler 內既有 `books.map`（tags 正規化處）加入 model 轉換層，使存入 storage 的每本書符合 BookSchemaV2（含 `readingStatus`、`tagIds`、`isManualStatus`、`schema_version`），讓 Overview 能正確顯示閱讀狀態與分類。

服務對象：v0.19.0 內測版本的提取 → 顯示 pipeline；下游 UC-01/03/04/06 的 6 狀態與 tag 系統。

## 2. 設計決策定案

### D1. readingStatus derivation 規則與邊界

採用 `BookSchemaV2.mapV1StatusToV2(book)` 作為 derivation 唯一實作（不另寫 progress→status 邏輯，遵守 DRY、ARCH-010 既有機制優先）。

理由：提取的 book 物件無 `isFinished`/`isNew`（ReadmooAdapter 不產出），`mapV1StatusToV2()` 對「無 isFinished」情形即退化為純 progress derivation，且內建 `normalizeV1Progress()` 已處理 null/undefined/NaN/字串/物件等邊界。

| progress 輸入 | 經 normalizeV1Progress | derive 結果 | 說明 |
|--------------|----------------------|------------|------|
| `0` | 0 | `unread` | progress=0 |
| `45`（0<p<100） | 45 | `reading` | 閱讀中 |
| `100` | 100 | `finished` | p>=100 視為完成 |
| `null` / `undefined` | 0 | `unread` | 缺值視為未讀 |
| `NaN` | 0 | `unread` | 非法數字視為未讀 |
| 負值（如 `-5`） | -5 | `unread` | p<=0 落入「其餘」分支 → unread（負值與 0 同義，保守視為未讀） |
| `>100`（如 `150`） | 150 | `finished` | `progress >= 100` → finished（超界視為完成，保守不丟錯） |
| 字串 `"45"` | 45 | `reading` | normalizeV1Progress 解析字串 |

邊界決策依據：runtime pipeline 不應因單一異常 progress 值中斷 96 本書的整批儲存（規則 4 可觀測性 — 異常以日誌記錄、不丟錯）。負值/超界視為「最接近的合理狀態」（負→unread、超界→finished），符合保守降級原則。

> 注意：`mapV1StatusToV2` 僅產出 `unread`/`reading`/`finished` 三種自動可追蹤狀態。`queued`/`abandoned`/`reference` 為手動專用狀態，首次提取不可能 derive 出，由使用者後續手動設定（v0.20.x tag 管理功能範圍）。

### D2. applyDefaults() 接線點

接線點 = 既有 `EXTRACTION.COMPLETED` handler 內 `books.map(b => {...})` 區塊（event-coordinator.js 約 L543-549，目前做 tags 正規化處）。

每本書的轉換順序（單一 map callback 內）：

```
1. tags 正規化（既有邏輯，保留 — 確保含 'readmoo'）
2. derived = BookSchemaV2.mapV1StatusToV2(b)        // 由 progress derive readingStatus
3. withStatus = { ...b, tags, readingStatus: derived }
4. final = BookSchemaV2.applyDefaults(withStatus)   // 填入 tagIds:[]、isManualStatus:false、其餘預設
```

順序理由：先 derive `readingStatus` 再 `applyDefaults`。因 `applyDefaults` 只在欄位 `=== undefined` 時填預設（會填 `unread`），必須先放入 derive 結果，`applyDefaults` 才不會覆蓋為預設 `unread`。

模組引入：`event-coordinator.js` 頂層 `const BookSchemaV2 = require('src/data-management/BookSchemaV2')`（比照既有 `require('src/...')` bare-specifier 風格，由 esbuild alias 解析）。

### D3. 既有 `tags:["readmoo"]` 欄位的去留

保留既有 `tags` 欄位，與 `tagIds:[]` 並存，互不取代。

依據 BookSchemaV2.BOOK_SCHEMA_V2.fields：schema 定義 `tagIds`（tag 系統參照 id 陣列）但**未定義 `tags`**。`tags:["readmoo"]` 是書城來源標記（既有 runtime 慣例），非 v2 tag 系統欄位。`applyDefaults` 只處理 schema 已定義欄位，不會刪除 `tags`。

- `tags`：保留（書城來源名，既有 handler 正規化邏輯不動）
- `tagIds`：由 `applyDefaults` 初始化為 `[]`（v2 tag 系統，首次提取尚無使用者標籤）

兩者語義不同（來源標記 vs 使用者分類 tag），共存不衝突。tag 系統正式消費 `tagIds` 為 v0.20.x 範圍。

### D4. isManualStatus 初值與 schema_version

| 欄位 | 初值 | 來源 |
|------|------|------|
| `isManualStatus` | `false` | `applyDefaults` 套用 schema default（首次提取為自動追蹤狀態，未經使用者手動指定） |
| `readingStatus` | derive 結果（D1） | `mapV1StatusToV2` |
| `tagIds` | `[]` | `applyDefaults` schema default |
| schema 版本欄位 | `'3.0.0'` | 見下方說明 |

schema 版本欄位命名說明：ticket acceptance 與 W1-006 Solution 用詞為 `schema_version`，但 `BOOK_SCHEMA_V2.fields` **未定義 `schema_version` 欄位**，`applyDefaults` 不會自動填入。`BookSchemaV2.SCHEMA_VERSION` 常數值為 `'3.0.0'`。

定案：在 map callback 內顯式設定 `schemaVersion: BookSchemaV2.SCHEMA_VERSION`（採 camelCase，與 book 物件既有欄位 `extractedAt`/`coverInfo` 命名風格一致；`schema_version` snake_case 與 runtime book 物件慣例不符）。此為 book 物件層級的版本標記。

> 待 Phase 2 確認：consumer（Overview / 匯出）是否已預期某特定欄位名讀取 schema 版本。若有既有讀取點，以該名稱為準。Phase 2 測試需明確斷言欄位名。

### D5. ticket where.layer

定案為 `Application`。

依據：`event-coordinator.js` 屬 SW 層資料協調入口（接收事件 → model 轉換 → 寫 storage），對應架構層級執行順序表的 Application 層（UseCase/協調，依賴 Domain model `BookSchemaV2`）。與 source ANA W1-006 的 `where.layer: Application` 一致。

## 3. GWT 行為場景

### 場景 1：正常提取 — progress 涵蓋三種自動狀態（正常流程）
- **Given**: `EXTRACTION.COMPLETED` 事件帶 books 陣列，含 progress=0、progress=45、progress=100 三本書
- **When**: handler 處理事件並寫入 storage
- **Then**: 寫入 `readmoo_books.books` 對應為 `readingStatus` = `unread`/`reading`/`finished`；每本含 `tagIds:[]`、`isManualStatus:false`、`schemaVersion:'3.0.0'`、保留 `tags`（含 `'readmoo'`）

### 場景 2：progress 異常值不中斷儲存（邊界條件）
- **Given**: books 陣列含 progress 為 `null`、`undefined`、`NaN`、`-5`、`150` 的書
- **When**: handler 處理事件
- **Then**: 全數成功寫入；`null/undefined/NaN/-5` → `readingStatus:'unread'`，`150` → `readingStatus:'finished'`；無例外拋出、無書遺漏

### 場景 3：空 books 陣列（邊界條件）
- **Given**: `EXTRACTION.COMPLETED` 事件 books 為 `[]`
- **When**: handler 處理事件
- **Then**: 寫入 `readmoo_books.books = []`，map 不執行；維持既有行為，不丟錯

### 場景 4：books 缺失或非陣列（異常流程）
- **Given**: 事件 data 無 `books`/`booksData`，或值非陣列
- **When**: handler 處理事件
- **Then**: 走既有 `else` 分支記錄 warning「沒有有效的書籍資料」，不寫 storage；本 ticket 不改變此行為

### 場景 5：既有 tags 欄位保留與 tagIds 並存（正常流程）
- **Given**: 提取的 book 物件已含 `tags:["readmoo"]`，無 `tagIds`
- **When**: handler 處理事件
- **Then**: 寫入後 `tags` 仍為 `["readmoo"]`（未被刪除/覆蓋），`tagIds` 為新增的 `[]`

### 場景 6：已有 readingStatus 的書不被預設覆蓋（邊界條件）
- **Given**: 極少數 book 物件已帶非空 `readingStatus`（如未來 adapter 補上）
- **When**: handler derive + applyDefaults
- **Then**: derive 結果取代輸入值（derive 為單一事實來源），`applyDefaults` 不再覆寫；確認 derive 先於 applyDefaults 的順序正確

## 4. 行為單元清單

| 行為單元 | Given-When-Then | 邊界條件 | 獨立可測？ |
|---------|----------------|---------|-----------|
| U1 progress→readingStatus derivation | Given progress 值 → When 經 mapV1StatusToV2 → Then 對應 unread/reading/finished | progress=0/45/100、null、undefined、NaN、負值、>100、字串 | 是 |
| U2 model 預設填充 | Given derive 後 book → When applyDefaults → Then tagIds:[]、isManualStatus:false 填入 | 欄位已存在不覆蓋、tags 欄位保留 | 是 |
| U3 handler 整合（map 接線） | Given EXTRACTION.COMPLETED 事件 → When handler 處理 → Then storage book 符合 BookSchemaV2 | 空陣列、非陣列、缺 books | 是 |

> U1 邏輯本身屬 BookSchemaV2（已有單元測試覆蓋），本 ticket Phase 2 測試聚焦 U2/U3 — 即「event-coordinator 正確接線並產出合規 book 物件」。

## 6. Acceptance（驗收條件）

- [ ] `event-coordinator` 在 `EXTRACTION.COMPLETED` handler 寫 storage 前，對每本 book 呼叫 `BookSchemaV2.mapV1StatusToV2()` derive `readingStatus`，再呼叫 `BookSchemaV2.applyDefaults()`

- [ ] 既有 `tags`（含 `'readmoo'`）欄位保留，與 `tagIds` 並存
- [ ] 空 books 陣列、非陣列、缺 books 情形維持既有行為，不丟錯
- [ ] progress 異常值不中斷整批儲存（96 本書全數寫入）
- [ ] `npm test` 通過（測試通過率 100%）
