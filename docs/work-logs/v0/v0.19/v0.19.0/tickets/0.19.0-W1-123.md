### 實驗發現

**已驗證的事實**：

1. PC-165 觸發條件聯集含「修復鏈 + mock 替代 + 後續信任」三項，IMP-078 觸發條件聯集不含此三項（PC-165 line 182 邊界判定 vs IMP-078 line 80 觸發條件聯集）
2. 兩者共有的「測試綠燈但 runtime 失效」是表層症狀（IMP-078 line 19 與 PC-165 line 21-28 描述同類）
3. 根因領域正交（流程層 vs 實作層），防護機制完全不同
4. 案例交集：W1-047.1~.5 具備兩者部分特徵（IMP-078 API 誤用觸發 + PC-165 acceptance 全勾 commit 後續推進）

**仍未驗證的假設**：

- 是否存在「Test/Production Environment Divergence」上位 pattern 收編兩者：目前案例 2 個，不足以證明需要上位 pattern；若未來新增第 3 個跨環境差異案例，可重評

**結論**：

「特化」表述不成立。「並列」表述更精確：兩者為獨立 anti-pattern，共享表層症狀，根因領域正交。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 判定結論

**PC-165 vs IMP-078 為並列關係（非特化關係）**

兩者為獨立 anti-pattern，共享表層症狀「測試綠燈但 runtime 失效」，但根因領域正交、觸發條件聯集不重疊、防護機制完全不同。詳見「重現實驗結果」對照表。

### IMP-078 line 40（「與其他 error-pattern 關係」section）修訂建議

當前表述：

> | PC-165（false-positive-fix-chain） | 強相關（上位 pattern） | PC-165 描述「測試綠燈但 runtime 無效」的通用機制（mock 自洽 + 斷言不覆蓋 runtime 路徑 + 後續 ticket 信任前序綠燈）；IMP-078 是 PC-165 在「跨環境部署（Node.js 測試環境 vs 瀏覽器 CE runtime）」維度的特化版本 |

建議改為：

> | PC-165（false-positive-fix-chain） | 並列關係（共通表層症狀）| 共享「測試綠燈但 runtime 失效」表層症狀，但根因領域正交：PC-165 聚焦「修復鏈 + mock 替代 + 後續信任」三層共振（流程層）；IMP-078 聚焦「Jest jsdom Node globals vs 瀏覽器 esbuild bundle API 可用性差異」（實作層）。交集案例為 W1-047.1~.5（既觸發 IMP-078 API 誤用，也具備 PC-165 acceptance 全勾 + commit + 後續推進的修復鏈特徵）|

### PC-165 line 90（「與 PC-028 / PC-082 / PC-131 / PC-136 的關係」section）修訂建議

當前表述未提及 IMP-078（因 IMP-078 是 W1-122 後才完整存在）。建議新增一列：

> | IMP-078 | runtime API mismatch | 共通「測試環境 vs 目標環境差異」上位概念；IMP-078 聚焦跨環境部署（Node.js vs Browser API），與本 PC「修復鏈 false positive」並列存在，交集案例為 W1-047.1~.5 |

### 為何採「並列」而非「新建上位 pattern」

| 選項 | 評估 |
|------|------|
| A. 改「並列」（推薦） | 精確描述當前事實；雙向引用建立交叉索引；維護成本低 |
| B. 保留「特化」 | 否決：非真子集關係，誤導語意 |
| C. 新建上位 pattern「Test/Production Environment Divergence」 | 目前案例僅 2 個，不足以證明上位 pattern 必要性；若未來累積第 3 個跨環境差異案例（如 Service Worker context vs 主執行緒 vs OffscreenDocument），可重評 |

採選項 A。未來若新增第 3 個跨環境案例，再考慮選項 C。
