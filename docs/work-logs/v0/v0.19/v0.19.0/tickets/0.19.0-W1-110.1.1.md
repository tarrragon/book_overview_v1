### 問題根因（PM 補強，13 處精確定位）

### 實作摘要

W1-110.1.1（W1-110.1 PC-165 收尾）將 background.js 14 處 log call 從「描述性中文字串字面」遷移為 backgroundMessages local dict messageKey 常數,補齊 W1-110.1 切換 local dict 後 messageKey 機制的「半套」狀態。

### 範圍實際 14 處(PA 表 13 + EMERGENCY_ERROR)

PA 表精確列出 13 處(line 193 MAX_RETRIES_REACHED / 210 REGISTER_LIFECYCLE / 215 EXTENSION_INSTALLED / 230 EXTENSION_STARTUP / 243 LIFECYCLE_COMPLETE / 245 LIFECYCLE_FAILED / 259 EMERGENCY_MODE / 269 EMERGENCY_MESSAGE / 293 EMERGENCY_COMPLETE / 295 EMERGENCY_FAILED / 369 INIT_FLOW_START / 372 INIT_FLOW_SUCCESS / 375 INIT_FLOW_FAILED)。

實作時在 grep 全量盤點發現第 14 處:line 322 log.error 模板字串 EMERGENCY_ERROR 描述。屬同一 PC-165 反模式(描述性字串字面繞過 messageKey),且 backgroundMessages dict 已含 EMERGENCY_ERROR 帶 {eventType} placeholder 結構。一併修復符合本 ticket why(messageKey 機制完整套用)+ quality-baseline 規則 5(發現即建立,但此處為同類反模式同一 commit 內修復更符合認知負擔最小化)。

### 程式碼變更

1. src/background/background.js (+27 -10):
   - line 82-92:log.error wrapper 升級為 union 型別 (Error | Object)。傳 Error 時自動萃取 { error: error.message };傳 Object 時直傳 data 給 logger,保留 messageKey placeholder 所需 context 欄位(如 EMERGENCY_ERROR 需 { eventType, error } 同時提供)。
   - line 202 MAX_RETRIES_REACHED / 219 REGISTER_LIFECYCLE / 224 EXTENSION_INSTALLED (with { reason }) / 239 EXTENSION_STARTUP / 252 LIFECYCLE_COMPLETE / 254 LIFECYCLE_FAILED / 268 EMERGENCY_MODE / 278 EMERGENCY_MESSAGE (with { message }) / 302 EMERGENCY_COMPLETE / 304 EMERGENCY_FAILED / 336 EMERGENCY_ERROR (with { eventType, error }) / 383 INIT_FLOW_START / 386 INIT_FLOW_SUCCESS / 389 INIT_FLOW_FAILED:14 處改用 messageKey 常數。
   - line 327-335:EMERGENCY_ERROR 處加業務情境註解(W1-110.1.1 補強標記 + PA 表外但同類反模式說明)。

2. tests/unit/background/background-local-dict.test.js (+419 -22,含原有 W1-110.1 三個 test case 保留):
   - 新增 describe 區塊「W1-110.1.1:PC-165 收尾 — 14 keys 全 runtime 渲染驗證」
   - 成功路徑(6 test):載入 background.js + 等 microtask + 手動觸發 onInstalled / onStartup listener,覆蓋 REGISTER_LIFECYCLE / LIFECYCLE_COMPLETE / INIT_FLOW_START / INIT_FLOW_SUCCESS / EXTENSION_INSTALLED / EXTENSION_STARTUP
   - 失敗路徑(3 test):coordinator mockRejectedValue + fake timers 加速 3 次 retry → 觸發 MAX_RETRIES_REACHED / INIT_FLOW_FAILED / EMERGENCY_MODE / EMERGENCY_COMPLETE,緊急模式 onMessage listener 觸發 EMERGENCY_MESSAGE
   - LIFECYCLE_FAILED 獨立 describe:chrome.runtime.onInstalled.addListener throw 觸發 registerServiceWorkerEvents catch 路徑
   - EMERGENCY_FAILED 獨立 describe:chrome.runtime.onMessage.addListener throw + retry → activateEmergencyMode catch 路徑
   - EMERGENCY_ERROR 雙保險 describe:dict 結構驗證(placeholder 渲染)+ src grep 防護(messageKey 字面斷言 + 反模式字面禁止)
   - 完整性掃描(2 test):14 keys 必須在 src 同時以 dict key 與 log call messageKey 形式出現 + 反模式 emoji prefix 字串 log call 全面禁止

### 設計決策

- log.error wrapper 升級而非新增函式:保留現有 caller 寫法(傳 Error 時自動萃取 message),新增 Object 直傳能力,零回歸。原 wrapper 對 line 192 log.error('INIT_FAILED', { attempt, error }) 的行為(將整個 data 物件包裹為 { error: { attempt, error } })反而是 bug,新版本將 { attempt, error } 直傳 logger 保留 top-level attempt 欄位,屬行為改善。
- EMERGENCY_ERROR 與 PA 表外發現:在 commit message 與本 Solution 顯性標註此為 PA 漏列項,避免後人考古困惑「為何修了 14 處而 ticket 標題寫 13 處」。grep 防護測試(src 字面禁止 emoji prefix log call)將 14 keys 同等強度防護。
- 失敗路徑用 fake timers:背景 retry 邏輯 await new Promise(setTimeout, 2000) 與三次重試共 6 秒實時等待會拖慢測試。jest.useFakeTimers() + advanceTimersByTimeAsync(2000) × 3 將時間壓到毫秒級,三個失敗路徑測試合計 < 40ms。
- EMERGENCY_ERROR runtime trigger 取捨:實際 emit handler 拋錯路徑需構造異步 listener throw,測試骨架複雜度高且斷言價值低(dict 完整性 + src grep 已足夠防護)。改採等價驗證:(a) dict 結構驗證 placeholder 渲染正確;(b) src grep 字面斷言 log call 使用 messageKey。雙保險覆蓋 PC-165 核心防護面(既有 bug 不會悄悄退回字串字面)。
- setImmediate → setTimeout(_, 0):jsdom 環境無 setImmediate,改用 jsdom-compatible microtask flush 模式。
