### 問題根因

現行 `handoff-auto-resume-stop-hook.py` 用 `is_ticket_recently_started()`（line 359-420）判斷「最近任務」：

1. 讀 ticket frontmatter 的 `started_at` 時間戳
2. 比對 `RECENT_TASK_THRESHOLD_MINUTES = 30`（line 136）
3. 在 30 分鐘內視為「最近任務」（可能有代理人在跑）→ 不阻止退出
4. 超過 30 分鐘視為「遺留任務」→ 阻止退出

弱點：
- `started_at` 是 ticket **claim 時間**，非代理人實際啟動時間。PM claim 後若 30 分鐘內未派發，會誤判為「最近任務」。
- 30 分鐘閾值是硬編碼魔術數字，與代理人實際生命週期解耦（小任務 5 分鐘完成、大任務 1 小時未完皆無法精準對齊）。
- 推斷 → 「代理人狀態」（間接），而 v2.1.145 的 `background_tasks` 直接提供答案。

### 影響範圍

**主檔案**：`.claude/skills/ticket/hooks/handoff-auto-resume-stop-hook.py`（865 行，已超 file-size-guardian 閾值 300 行——非本 ticket 範圍）

#### Step 2：新增 `has_background_agents()` helper

```python
def has_background_agents(input_data: Dict[str, Any], logger) -> bool:
    """
    判斷 input_data 是否有背景代理人正在執行。

    W3-026.1：用 v2.1.145 新欄位 background_tasks 取代 started_at 30 分鐘閾值推斷。
    採「弱依賴策略」——只判斷 list 非空，不依賴 task object 內部結構，
    避免 schema 不確定風險。

    Args:
        input_data: Stop hook stdin JSON
        logger: 日誌記錄器

    Returns:
        bool - 是否有背景代理人執行中。欄位不存在 / 非 list / 空 list 皆回 False。
    """
    bg_tasks = input_data.get("background_tasks") if isinstance(input_data, dict) else None
    if not isinstance(bg_tasks, list):
        # 欄位不存在或型別異常 → 視為「無 schema 資訊」，由 caller fallback
        return False
    has_active = len(bg_tasks) > 0
    if has_active:
        logger.info(f"input_data.background_tasks 非空（{len(bg_tasks)} 項），偵測到背景代理人")
    else:
        logger.debug("input_data.background_tasks 為空 list，無背景代理人")
    return has_active
```

### Fallback 行為保證

| 情境 | input_data | 行為 |
|------|-----------|------|
| 新版 CC，hook 收到 background_tasks=非空 list | `{"background_tasks": [...]}` | 用新邏輯 → recent_tasks |
| 新版 CC，背景無代理人 | `{"background_tasks": []}` | has_background_agents 回 False → fallback 到 started_at |
| 舊版 CC，欄位不存在 | `{}` | has_background_agents 回 False → fallback 到 started_at |
| stdin 解析失敗 | input_data = {} | 同上 fallback |
| stdin 為 tty（手動測試） | input_data = {} | 同上 fallback |
