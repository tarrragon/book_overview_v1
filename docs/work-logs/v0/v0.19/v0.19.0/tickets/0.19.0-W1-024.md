## 重現實驗結果
### 實驗方法

本 ANA 的「重現實驗」不是重現 bug，而是驗證 5 個新增斷言類型的跨語言通用性假設。方法：逐一分析每個類型在 WEB（JS/TS）、APP（Flutter/Dart）、後端（Go/Python）三類專案中是否存在同質問題，判斷其是否為語言無關的測試設計概念。

### 實驗執行

**逐類型跨專案類型分析**

#### 類型 1：非同步斷言時序

問題本質：測試依賴非同步操作完成的時間順序，而非操作結果。

| 專案類型 | 典型表現 | 語言無關？ |
|---------|---------|----------|
| WEB | setTimeout / Promise 後斷言 DOM 狀態，未等待渲染完成 | 是 |
| APP | Widget rebuild 後斷言 UI 狀態，未 pumpAndSettle | 是 |
| 後端 | goroutine / async handler 回傳後斷言副作用，未等待 channel/callback | 是 |

通用概念：斷言對象必須是操作的最終狀態，不是中間狀態。等待機制因框架而異（await/pump/channel），但「等待完成再斷言」是語言無關原則。

#### 類型 2：亂數種子與隨機性

問題本質：測試結果依賴隨機數產生器狀態，不可重現。

| 專案類型 | 典型表現 | 語言無關？ |
|---------|---------|----------|
| WEB | Math.random() 驅動 UI 排序，斷言固定順序 | 是 |
| APP | 隨機 ID 生成、A/B test 分組，斷言特定分組 | 是 |
| 後端 | 負載均衡 hash、加密 salt、取樣演算法 | 是 |

通用概念：測試中的隨機源必須可控（固定種子或注入 mock）。斷言驗證演算法行為而非特定隨機輸出。所有語言都有此問題。

#### 類型 3：測試隔離與順序依賴

問題本質：測試案例間共享狀態，執行順序影響結果。

| 專案類型 | 典型表現 | 語言無關？ |
|---------|---------|----------|
| WEB | 全域 DOM 狀態、localStorage 未清、模組快取 | 是 |
| APP | 單例 service、shared preferences、靜態變數 | 是 |
| 後端 | 資料庫狀態、全域 registry、環境變數 | 是 |

通用概念：每個測試必須建立自己的前置條件，不依賴其他測試的副作用。teardown 必須還原共享狀態。所有語言都有此問題，且 flaky 症狀一致（隨機失敗、順序敏感）。

#### 類型 4：快照測試脆弱性

問題本質：快照斷言比對整體輸出結構，微小格式變更觸發 false positive failure。

| 專案類型 | 典型表現 | 語言無關？ |
|---------|---------|----------|
| WEB | Jest snapshot 比對整個 React component tree，加一個 CSS class 全紅 | 是 |
| APP | Golden test 比對整個 Widget tree / 像素 | 是 |
| 後端 | API response snapshot，加一個欄位全紅 | 是 |

通用概念：快照粒度必須與驗證意圖匹配。全結構快照適合偵測非預期變更，但不適合作功能驗收——功能驗收應斷言特定欄位/屬性。所有有快照機制的框架都有此問題。

#### 類型 5：斷言粒度與過度斷言

問題本質：單一測試案例塞入過多斷言，前面的斷言失敗遮蔽後面的，且任一微小變更觸發大量失敗。

| 專案類型 | 典型表現 | 語言無關？ |
|---------|---------|----------|
| WEB | 一個 test case 驗 20 個 DOM 屬性 | 是 |
| APP | 一個 test 驗整個頁面所有 widget 狀態 | 是 |
| 後端 | 一個 test 驗整個 API response 所有欄位 | 是 |

通用概念：一個測試案例驗證一個行為面向。斷言數量與被測行為複雜度成正比，而非與輸出結構大小成正比。所有語言皆適用。

### 實驗發現

**已驗證事實**：
1. 全部 5 個新增類型都是語言無關的測試設計概念。問題根因在測試設計邏輯而非語言特性。
2. 每個類型的等待/修正機制因框架而異，但判斷概念（何時該注意、為何是問題）跨語言一致。
3. 本專案目前無這 5 類的實證案例（W1-017/W1-018 只涵蓋計時與浮點）。

**仍未驗證的假設**：
1. 各專案類型（WEB/APP/後端）對這 5 類問題的嚴重度排序是否不同（例如 WEB 可能快照問題更嚴重，後端可能隔離問題更嚴重）。這需要實際專案經驗累積才能驗證，目前只能標記為合理推測。
2. 5 個類型的完整性——是否還有其他跨語言通用的斷言設計問題類型未列入。本 ANA 範圍限於用戶指定的 5 個類型，完整性留待實踐補充。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### AC1：五個新增斷言類型跨語言通用性研究結論

全部 5 個類型均為語言無關的測試設計概念，可納入 skill 概念框架。詳見「重現實驗結果」章節的逐類型分析表。

**為何此判斷重要**：若任一類型實為語言相關卻被納入通用 skill，會在 WEB / APP / 後端套用時產生誤導，使用者依不適用的原則設計測試。研究逐類型確認語言無關性，正是為消除此誤導風險。

總結判斷準則：

| 斷言類型 | 核心概念（語言無關） | 專案類型差異（WEB/APP/後端） |
|---------|-------------------|--------------------------|
| 非同步時序 | 斷言最終狀態，不斷言中間狀態 | 等待機制不同（await vs pump vs channel），但原則相同 |
| 亂數種子 | 隨機源必須可控，斷言行為不斷言輸出 | 無顯著差異，三類專案皆適用 |
| 測試隔離 | 每測試自建前置條件，不依賴執行順序 | 共享狀態類型不同（DOM vs singleton vs DB），但隔離原則相同 |
| 快照脆弱性 | 快照粒度匹配驗證意圖，功能驗收用欄位斷言 | WEB 和 APP 快照更常見（UI rendering），後端較少但 API response snapshot 同理 |
| 斷言粒度 | 一測試一行為面向，斷言數與行為複雜度成比例 | 無顯著差異 |

### AC2：SKILL.md 概念框架內容架構

SKILL.md 嚴格限於判斷概念，不含測試寫法與重構方法。架構如下：

```
.claude/skills/test-assertion-design/
  SKILL.md                          # 概念框架主文
  references/                       # 按需讀取
    web-testing-context.md          # WEB 類專案考量脈絡（若需要）
    app-testing-context.md          # APP 類專案考量脈絡（若需要）
    backend-testing-context.md      # 後端類專案考量脈絡（若需要）
```

**SKILL.md 內容架構**：

1. **觸發條件**：何時應啟用此 skill（撰寫新測試、審查測試、測試 flaky 診斷）
2. **斷言分類決策表**（從 methodology 提煉，擴充 5 個新增類型）：
   - 確定性判斷：此斷言的 pass/fail 是否受環境影響？
   - 9 個斷言類型的判斷概念（既有 4 個 + 新增 5 個），每個只寫：問題本質、識別信號、設計原則
3. **斷言品質三問**（從既有核心原則抽象）：
   - 確定性：同一程式碼在任何環境下結果一致嗎？
   - 聚焦性：此斷言驗證一個行為面向嗎？
   - 隔離性：此斷言不依賴其他測試的副作用嗎？
4. **不在範圍**聲明：明示 skill 不處理測試寫法、重構方法、框架 API 用法

**刻意不放入 SKILL.md 的內容**：
- Jest/Flutter/Go 的具體 API（toBeLessThan、pumpAndSettle、testing.T）
- 重構步驟（如何從計時斷言改為功能斷言）
- 目錄結構規定（tests/perf/ 等，屬專案 rules 層）
- 精度數字（numDigits <= 2 等，屬專案 rules 層）
