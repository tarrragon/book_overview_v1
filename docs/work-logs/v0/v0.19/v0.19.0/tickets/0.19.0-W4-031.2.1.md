### 問題根因

V1（Extension）匯出器 `BookDataExporter.exportToJSON` 既有兩條路徑：v1 預設（直接序列化 books 陣列）與 v2（`formatVersion === '2.0.0'`，metadata/tagCategories/tags/books 四區段）。皆無法產出跨專案 canonical `book-interchange-v1` v3.0.0（everything-as-tags）格式。本 ticket 補上 canonical write 路徑。

### 影響範圍

- `src/export/book-data-exporter.js`：新增 `formatVersion === '3.0.0'` 分支 + `_exportToJSONCanonical` 內部方法；既有 v1/v2 路徑不動。
- 消費 foundation（W4-031.2）`src/export/book-interchange-v1-adapter.js` 的 `mapV1BookToCanonical`（逐書 write 映射）與 `buildTagTree`（root tagTree 聚合），不重新實作。

### 相關 Error Pattern

- PC-162：源檔行為以實際碼為準。已 sanity 驗證 foundation export（含 mapV1BookToCanonical / buildTagTree），base 非 stale。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP（claim 三問）

- W：(a) 在 `exportToJSON` 新增 `formatVersion === '3.0.0'` dispatch 分支（採用，仿既有 v2 @ line 412 慣例）；(b) 獨立 public `exportToCanonical()` 方法（棄用，破壞既有 options.formatVersion dispatch 一致性）。
- A：本 ticket 為 foundation（.2）收斂的 leaf，位於關鍵路徑，無更高優先目標被擠壓。
- P：最可能失敗＝round-trip 斷言對 foundation 產出的空 `_passthrough:{}`/空 tag 陣列做 deep-equal 不符。防護：採結構性斷言（property/toEqual 個別欄位）而非整物件 deep-equal，並保 v2 回歸綠燈。

### 變更摘要

1. require foundation：`const { mapV1BookToCanonical, buildTagTree } = require('src/export/book-interchange-v1-adapter')`。
2. `exportToJSON`：v2 分支前插入 `formatVersion === '3.0.0'` → `_exportToJSONCanonical(options, startTime)`。
3. 新增 `_exportToJSONCanonical`：
   - 過濾非物件書項 → 逐書 `mapV1BookToCanonical` 得 canonicalBooks（everything-as-tags，spec §4）。
   - `buildTagTree(canonicalBooks)` 組 root tagTree（ccl/custom，spec §6）。
   - 組裝 root（spec §3）：`{format:'book-interchange-v1', formatVersion:'3.0.0', metadata{exportedAt,sourceApp,totalBooks=canonicalBooks.length,...options.metadata}, books, tagTree}`。
   - `totalBooks` 由實際匯出書數推導（交叉驗證 = books.length）。
   - pretty 控制 + recordExport 沿用既有慣例。

### 設計決策

- 不破壞既有 v1/v2 路徑（向後相容靠 `options.formatVersion` dispatch）。
- 不重新實作映射純函式（foundation 已交付）；root 組裝與接線屬本 ticket 範圍（spec §1 範圍邊界表「.2.1」列）。
