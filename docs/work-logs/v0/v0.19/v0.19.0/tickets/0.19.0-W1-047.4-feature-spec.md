## 1. Purpose（目的）

### 問題背景

IMP-C 已完成「覆蓋模式」匯入：`TagStorageAdapter.replaceAllData` 清空 storage 三 key（`readmoo_books` / `tags` / `tag_categories`）後整批寫入。但 SPEC-EXPORT-V2 §8.1 定義兩種載入模式，覆蓋只是其一。覆蓋模式會丟棄使用者本地既有書庫與標籤——若使用者想把備份檔的書籍「加進」現有書庫而非「取代」，目前無對應功能。

### 目標用戶

使用 UC-04 匯入功能、且本地已有書庫資料的 Readmoo 書庫管理器使用者（例如跨裝置同步、合併多份備份）。

### 核心價值

提供 SPEC-EXPORT-V2 §8.2「合併模式」的資料合併邏輯與 §8.3 的 tag/category id 重映射，使匯入資料能與本地資料無損疊加：

- 書籍同 id 時 `tagIds` 取聯集，不丟失任一側標籤關聯。
- tag / category 同名（大小寫不敏感）視為相同，保留本地版本，避免重複建立。
- 匯入檔案內的 tag / category id 與本地不同時，透過映射表轉換書籍 `tagIds` 至本地 id，維持引用完整性。

### 範圍邊界

| 範圍 | 歸屬 |
|------|------|
| 合併資料計算邏輯（books 聯集 / tag·category 同名比對 / id 重映射） | 本 ticket（IMP-D） |
| 合併結果的 storage 持久化 | 本 ticket（IMP-D）—— 設計見 §2.2 |
| 模式選擇 UI（覆蓋 vs 合併 modal） | IMP-E，不在本 ticket |
| `handleFileLoad` 依使用者選擇分流呼叫覆蓋或合併 | IMP-E，不在本 ticket（本 ticket 僅提供可被 IMP-E 呼叫的合併 API） |
| 版本偵測 / v1 轉換 / tagCategories·tags 提取 | IMP-A/B 已完成，本 ticket 假設輸入已是標準 v2 結構 |

## 2. API Signatures（介面定義）

### 2.1 架構決策：storage 層新增 `mergeAllData`

與 IMP-C 覆蓋模式對稱：IMP-C 在 `tag-storage-adapter.js` 新增 `replaceAllData`（覆蓋）。本 ticket 在同一檔案新增 `mergeAllData`（合併），語意對稱、呼叫端介面一致。

**決策理由**：

| 候選方案 | 評估 |
|---------|------|
| A. 合併邏輯置於 controller，controller 讀本地資料後算好再呼叫 `replaceAllData` | 否決。合併需讀取本地三 key、做 id 重映射、寫回三 key，屬 storage 層原子操作範疇；放 controller 會讓 controller 直接觸碰 storage 私有讀取，且無法複用 `operationLock` / `withAtomicRollback` 原子保證 |
| B. storage 層新增 `mergeAllData`，與 `replaceAllData` 對稱（**採用**） | 合併的「讀本地 → 計算 → 原子寫回」全程在鎖內，與覆蓋模式共用 `operationLock` / `withAtomicRollback` / `checkQuotaLevel`；controller 只需依使用者選擇呼叫 `replaceAllData` 或 `mergeAllData`，介面對稱 |

**合併計算的純函式可測性**：id 重映射與 books 聯集為純資料轉換，與 storage I/O 無關。`mergeAllData` 內部呼叫一個純函式 `computeMergeResult`（見 §2.3），使核心合併邏輯可獨立於 chrome.storage mock 進行單元測試。

### 2.2 `mergeAllData`（公開 API，新增於 `tag-storage-adapter.js`）

```
mergeAllData({ books, tags, tagCategories })
  → Promise<{ success: boolean,
              error?: 'quota_exceeded' | 'storage_error',
              counts?: { books: number, tags: number, tagCategories: number },
              remap?: { categories: number, tags: number } }>
```

| 參數 | 型別 | 說明 |
|------|------|------|
| `books` | `Array<Object>` | 匯入的 v2 書籍陣列（ImportResult.books），恆為陣列（INV-1） |
| `tags` | `Array<Object>` | 匯入的 v2 tag 陣列，恆為陣列 |
| `tagCategories` | `Array<Object>` | 匯入的 v2 tag category 陣列，恆為陣列 |

| 回傳欄位 | 說明 |
|---------|------|
| `success` | `true`：合併結果已原子寫回三 key。`false`：未寫入（配額不足或寫入失敗已回滾） |
| `error` | `'quota_exceeded'`：寫入前配額 blocked。`'storage_error'`：原子寫入失敗已回滾 |
| `counts` | success 時回報合併後三 key 的最終筆數（本地 + 新增） |
| `remap` | success 時回報重映射統計：`categories` / `tags` 為「匯入 id 被映射到不同本地 id」的筆數（含同名映射至既有本地，不含新建）。供觀測與測試斷言 |

**語意**：合併模式「同 id 更新、新 id 新增」，永不清空本地資料。空匯入（三者皆 `[]`）為合法輸入，結果等於本地資料原樣，`success: true`。

### 2.3 `computeMergeResult`（內部純函式，可獨立測試）

合併計算與 storage I/O 解耦的核心。`mergeAllData` 讀取本地三 key 後，呼叫此純函式取得待寫回的合併結果。

```
computeMergeResult(local, incoming)
  local    = { books, tags, tagCategories }   // 本地現況
  incoming = { books, tags, tagCategories }   // 匯入資料
  →  { books, tags, tagCategories,            // 合併後待寫回三集合
       remap: { categoryIdMap, tagIdMap } }   // 匯入 id → 本地 id 映射表（供觀測/測試）
```

純函式無副作用、不觸碰 storage、不產生時間戳以外的不確定性（新建 id 由注入的 id 產生器提供，見 §5）。

### 2.4 匯出變更

`tag-storage-adapter.js` 的 `TagStorageAdapter` 物件追加 `mergeAllData` 成員，不更動既有 CRUD 與 `replaceAllData` 簽名（僅追加，對齊 IMP-C ADR-8）。`computeMergeResult` 視實作需要決定是否匯出（建議匯出以利獨立單元測試）。

## 3. GWT Scenarios（行為場景）

### 場景 1: 書籍同 id tagIds 取聯集

- **Given**: 本地有書籍 `book-001`（`tagIds: [t_local_a, t_local_b]`）；匯入檔案有書籍 `book-001`（`tagIds: [t_imp_x, t_imp_y]`，且 `t_imp_x` 經重映射對應本地 `t_local_b`、`t_imp_y` 為新 tag）
- **When**: 以合併模式匯入（`mergeAllData`）
- **Then**: 合併後 `book-001` 的 `tagIds` 為 `t_local_a`、`t_local_b`、`t_imp_y`（聯集且去重，`t_imp_x` 已重映射為 `t_local_b` 故不重複）；`book-001` 其餘欄位以匯入資料覆蓋

### 場景 2: 書籍新 id 直接加入

- **Given**: 本地有書籍 `book-001`；匯入檔案有 `book-001` 與 `book-999`（本地不存在）
- **When**: 合併模式匯入
- **Then**: 合併後書庫含 `book-001`（合併）與 `book-999`（新增）；本地原有其他書籍保留不刪除

### 場景 3: tag 同名（大小寫不敏感）保留本地版本

- **Given**: 本地有 category `cat_local_1`（name `"自訂標籤"`）下 tag `t_local_sci`（name `"科幻"`）；匯入檔案有 category（name `"自訂標籤"`）下 tag（name `"科幻"`，id `t_imp_sci`）
- **When**: 合併模式匯入
- **Then**: 合併後 tag 集合中該 tag 仍為本地 `t_local_sci`（id、createdAt、sortOrder 等屬性保留本地版本），不新建第二個「科幻」tag；映射表記錄 `t_imp_sci → t_local_sci`

### 場景 4: tag 名稱大小寫差異視為相同

- **Given**: 本地 category 下有 tag name `"SciFi"`；匯入同 category 下 tag name `"scifi"`
- **When**: 合併模式匯入
- **Then**: 兩者視為同名（`toLowerCase` 比對，與 `TagSchema.validateTag` 一致），保留本地 `"SciFi"`，不新建

### 場景 5: tag 不存在則新建並重映射

- **Given**: 本地無名為 `"歷史"` 的 tag；匯入檔案有 tag（name `"歷史"`，id `t_imp_his`），其 category 經重映射對應某本地 category
- **When**: 合併模式匯入
- **Then**: 合併後 tag 集合新增一個 `"歷史"` tag，使用新建本地 id `t_local_new`（非沿用匯入 id）；`categoryId` 指向重映射後的本地 category id；映射表記錄 `t_imp_his → t_local_new`；引用 `t_imp_his` 的匯入書籍 `tagIds` 轉換為 `t_local_new`

### 場景 6: category 同名保留本地、不存在則新建

- **Given**: 本地有 category name `"書籍類型"`；匯入有 category name `"書籍類型"`（id 不同）與 category name `"閱讀清單"`（本地不存在）
- **When**: 合併模式匯入
- **Then**: 合併後 category 集合中 `"書籍類型"` 為本地版本（id、屬性保留）；新增 `"閱讀清單"`（新建本地 id）；映射表記錄 `匯入"書籍類型"id → 本地id`、`匯入"閱讀清單"id → 新建id`

### 場景 7: tag id 重映射貫穿 category → tag → book

- **Given**: 匯入檔案 category `c_imp`、tag `t_imp`（`categoryId: c_imp`）、書籍 `book-001`（`tagIds: [t_imp]`）；本地皆不存在同名
- **When**: 合併模式匯入
- **Then**: 新建本地 category `c_local`、新建本地 tag `t_local`（`categoryId` 為 `c_local`，非 `c_imp`）；`book-001` 的 `tagIds` 為 `[t_local]`（非 `t_imp`）—— 三層 id 一致重映射

### 場景 8: 合併結果原子持久化

- **Given**: 本地有書籍與 tag；合法匯入資料
- **When**: 呼叫 `mergeAllData`
- **Then**: 合併後三集合經 `withAtomicRollback` 原子寫回三 key；`success: true`，`counts` 為合併後筆數

### 場景 9: 配額不足攔截

- **Given**: storage 配額達 BLOCK 門檻
- **When**: 呼叫 `mergeAllData`
- **Then**: 寫入前 `checkQuotaLevel` 攔截，回傳 `{ success: false, error: 'quota_exceeded' }`；本地三 key 完全未被修改

### 場景 10: 寫入中途失敗回滾

- **Given**: 本地有資料；合併計算完成後寫第二 key（tags）時 `chrome.storage.local.set` 失敗
- **When**: 呼叫 `mergeAllData`
- **Then**: `withAtomicRollback` 將三 key 全數回滾至合併前快照；回傳 `{ success: false, error: 'storage_error' }`；本地資料維持合併前狀態

### 場景 11: 空匯入合法

- **Given**: 本地有書籍與 tag；匯入 `{ books: [], tags: [], tagCategories: [] }`
- **When**: 合併模式匯入
- **Then**: 合併結果等於本地資料原樣（無新增、無刪除）；`success: true`，`counts` 為本地原筆數

### 場景 12: 本地空、匯入非空

- **Given**: 本地三 key 皆空；匯入含書籍與 tag
- **When**: 合併模式匯入
- **Then**: 合併結果等於匯入資料；所有匯入 tag/category 皆走「新建」路徑（無同名可映射），但仍須重映射（匯入 id 仍轉為新建本地 id，保持與場景 7 一致的 id 規範）。**設計註**：本地空時是否沿用匯入 id 或新建——統一採新建本地 id，使 `mergeAllData` 行為對輸入無分支差異，降低複雜度（見 §4 邊界）

---

### 場景 13: 兩個匯入 category 收斂至同一本地 category 時 tag 不重複

- **Given**: 本地有一個 category name `"自訂標籤"`；匯入檔案有兩個 category 皆 name `"自訂標籤"`（id `c_imp_1`、`c_imp_2`），`c_imp_1` 下有 tag name `"科幻"`、`c_imp_2` 下也有 tag name `"科幻"`
- **When**: 合併模式匯入
- **Then**: `categoryIdMap` 將 `c_imp_1`、`c_imp_2` 皆映射至同一本地 category id；tag 階段以「重映射後本地 categoryId + name(toLowerCase)」為鍵，兩個匯入 `"科幻"` tag 收斂至同一本地 tag（同名映射或單次新建），合併後不出現重複 `"科幻"` tag

### 場景 14: 合併不修復本地既有孤兒引用

- **Given**: 本地書籍 `book-001` 的 `tagIds` 含 `t_orphan`（指向已不存在的本地 tag）；匯入 `book-001`（`tagIds: [t_imp_a]`）
- **When**: 合併模式匯入
- **Then**: 合併後 `book-001` 的 `tagIds` 為 `[t_orphan, <重映射後 t_imp_a>]`（聯集，`t_orphan` 原樣保留不被移除）—— 合併僅做資料疊加，本地引用完整性修復由 `checkReferentialIntegrity` 負責，不在合併範圍

## 4. Error Handling（錯誤處理）

| 情境 | 偵測點 | 處理策略 | 回傳值 |
|------|-------|---------|--------|
| 配額 blocked | `mergeAllData` 寫入前 `checkQuotaLevel` | 提早返回，不讀本地、不計算、不寫入 | `{ success: false, error: 'quota_exceeded' }` |
| 原子寫入失敗 | `withAtomicRollback` 內任一 `saveToStorage` reject | 回滾三 key 至合併前快照 | `{ success: false, error: 'storage_error' }` |
| 匯入 book 缺 `id` | `computeMergeResult` books 合併 | 跳過該書籍並記錄 warning（與 SPEC §9.2「v1 書籍缺 id 跳過」一致） | 不影響 `success`，該書不進結果 |
| 匯入 book `tagIds` 含無法重映射的 id | `computeMergeResult` book tagIds 轉換 | 過濾掉該 id 並記錄 warning（孤兒 tag 引用，匯入檔案不一致時的防禦） | 不影響 `success` |
| 匯入 tag 的 `categoryId` 在匯入檔案內找不到對應 category | `computeMergeResult` tag 處理 | 建立名為 `"未分類"` 的 category 並歸入（與 SPEC §9.2 一致），該 `"未分類"` 同樣參與同名比對 | 不影響 `success` |
| 匯入 tag/category `name` 超過 50 字元 | `computeMergeResult` | 截斷至 50 字元（與 SPEC §9.2、`TagSchema` 長度上限一致） | 不影響 `success` |
| 單本書 `tagIds` 聯集後超過 100 個 | `computeMergeResult` books 聯集 | 截斷至 100 個並記錄 warning（`TagSchema.MAX_TAGS_PER_BOOK`） | 不影響 `success` |
| 本地書籍既有孤兒 `tagId`（指向已不存在的本地 tag） | `computeMergeResult` books 聯集 | **不主動修復**：合併僅做「資料疊加」，不承擔引用完整性修復職責（該職責屬既有 `checkReferentialIntegrity`）。孤兒 tagId 原樣保留在聯集結果中 | 不影響 `success` |

**設計澄清（/spec validate 第 1 次迭代回答）**：

- **Q1（重映射收斂）**：id 重映射採「先 category 後 tag」兩階段。category 階段先建立完整 `categoryIdMap`（匯入 category id → 本地 category id），同名收斂至同一本地 id。tag 階段以「重映射後的本地 categoryId + tag name（toLowerCase）」為比對鍵——故即使兩個匯入 category 收斂到同一本地 category，其下同名 tag 也會收斂至同一本地 tag，不重複映射。
- **Q2 / Q3（合併不修復本地破損）**：`computeMergeResult` 對本地資料只讀不校驗。本地既有孤兒 tagId 在 books 聯集時原樣保留，合併不順帶修復。理由：(1) 合併職責單一（資料疊加），混入修復會擴大複雜度與測試面；(2) 引用完整性修復已有專責 `checkReferentialIntegrity` API。§4 表中「匯入 book tagIds 含無法重映射的 id」的過濾僅針對**匯入側**孤兒（匯入檔案自身不一致），不觸碰本地側。

**觀測性（規則 1 / observability-rules）**：`mergeAllData` 配額攔截分支、回滾分支須 `console.error` 含元件名 `[tag-storage-adapter]`；`computeMergeResult` 的跳過 / 截斷 / 過濾分支須 `console.warn` 含元件名與被影響的 id/name，與既有檔案風格一致。

- 執行於 Chrome Extension（Manifest V3），`chrome.storage.local` 可用。
- 測試於 Jest jsdom，`chrome.storage.local` 以有狀態 mock store 模擬（沿用 `chrome-storage-tag-operations.test.js` 的 `createStorageMock`）。

### 5.4 id 產生器注入

新建本地 tag/category 需產生本地 id。`computeMergeResult` 為純函式，新建 id 不可在函式內直接呼叫 `Date.now()` / `Math.random()`（破壞可測性）。設計：

- `computeMergeResult` 接受一個 id 產生器（或由 `mergeAllData` 在呼叫前準備好新建 id，注入為參數）。
- id 格式須與既有一致：category 為 `cat_` 前綴、tag 為 `tag_` 前綴（對齊 `createTagCategory` / `createTag` 的 `cat_${Date.now()}-${rand}` / `tag_${Date.now()}-${rand}` 格式）。
- 測試時注入可預測的 id 產生器，斷言重映射表內容。

> **待 Phase 3a 決策**：id 產生器是「函式參數」或「`mergeAllData` 預先算好 id 陣列傳入」——屬實作策略，Phase 1 僅約束「不可在純函式內呼叫不確定來源」。
