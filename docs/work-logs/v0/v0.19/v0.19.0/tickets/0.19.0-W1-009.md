## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

直接原因：SW `event-coordinator.js` 的 `EXTRACTION.COMPLETED` handler（約 L536-575）收到 content script 傳來的 books 陣列後，僅做 tags 正規化（確保含 `'readmoo'`），即直接包成 `readmoo_books` 寫入 `chrome.storage.local`，未呼叫任何 BookSchemaV2 model 轉換。

來源 ANA 0.19.0-W1-006 確認：`BookSchemaV2.applyDefaults()`、`mapV1StatusToV2()` 已實作且 193 單元測試通過，但 runtime 提取 pipeline 從未呼叫。v0.17.x PROP-007 重構範圍限於 model 定義 + 匯出入 + 搜尋適配，未涵蓋 runtime 提取路徑。結果：96 本書 `readingStatus` 全 undefined、無 `tagIds`，Overview 狀態欄/分類欄全顯示「未知」「未分類」。

### 影響範圍

| 檔案/模組 | 影響 |
|----------|------|
| `src/background/events/event-coordinator.js` | 唯一修改檔。`EXTRACTION.COMPLETED` handler 內 `books.map` 加入 model 轉換層 |
| `src/data-management/BookSchemaV2.js` | 被呼叫（`mapV1StatusToV2` + `applyDefaults` + `SCHEMA_VERSION`），不修改 |
| 下游 Overview / UC-01/03/04/06 | 修復後可正確顯示 6 種閱讀狀態與 tag 系統 |

### 修復範圍邊界

- 本 ticket：event-coordinator 接線 model 轉換（W1-006 IMP-1）
- 不在本 ticket：ReadmooAdapter cover URL 過濾（W1-006 IMP-2 = 0.19.0-W1-010）
- 不在本 ticket：tag_categories 初始化（v0.20.x tag 管理功能範圍）

### 相關 Error Pattern

無完全對應 error-pattern。問題本質為「重構範圍遺漏」— model 層重構完成但 runtime 消費端未同步接線。

### Phase 1 設計決策（詳見 feature-spec）

功能規格：`0.19.0-W1-009-feature-spec.md`（同目錄）。關鍵定案：

1. readingStatus derivation 採 `mapV1StatusToV2()` 單一實作（不另寫邏輯）；progress null/undefined/NaN/負值→`unread`，>=100→`finished`
2. 接線於既有 `books.map` 內，順序：tags 正規化 → derive readingStatus → applyDefaults（先 derive 後 applyDefaults，避免預設值覆蓋）
3. 既有 `tags:["readmoo"]`（書城來源名）保留，與 `tagIds:[]`（v2 tag 系統）並存
4. `isManualStatus` 初值 `false`（applyDefaults schema default）；`schemaVersion:'3.0.0'` 顯式設定（schema fields 未定義此欄位，applyDefaults 不會填）
5. where.layer 定案 = `Application`

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### Phase 2 測試設計（RED）

測試檔：tests/unit/background/events/event-coordinator-schema-v2.test.js（新建，20 測試案例）

測試策略：
- U1 derivation 契約鎖定（9 案例，test.each）— 對 BookSchemaV2.mapV1StatusToV2 鎖定 Phase 1 D1 邊界表（0/45/100/null/undefined/NaN/-5/150/字串）。BookSchemaV2 已實作，此 9 案例 GREEN 屬契約驗證（非待修目標）。
- U3 handler 整合（11 案例）— 透過真實 EventCoordinator + EventBus 路徑驅動 EXTRACTION.COMPLETED，斷言 chrome.storage.local.set 收到的 readmoo_books.books 合規。U2（applyDefaults 填充）效果由 U3 一併斷言。

場景覆蓋（GWT）：
- 場景 1：progress=0/45/100 derive unread/reading/finished；每本含 tagIds:[]/isManualStatus:false/schemaVersion；tagIds 獨立陣列參照（Q9 防碰巧通過）
- 場景 2：progress 異常值 null/undefined/NaN/-5/150 全數寫入不漏書、不丟錯
- 場景 3：空陣列寫入 books=[]（既有行為不回歸）
- 場景 4：缺 books / 非陣列不寫 storage（既有行為不回歸）
- 場景 5：既有 tags 保留與 tagIds 並存
- 場景 6：derive 先於 applyDefaults 順序正確（選 progress=100 使正確 finished vs 錯誤預設 unread 結果不同）
- 整合：每本書通過 BookSchemaV2.validateBook

RED 確認：npx jest event-coordinator-schema-v2 → 20 測試，12 passed / 8 failed。
- 8 failed = U3 整合測試（model gap 未修），失敗訊息：expect(stored[0].readingStatus).toBe(unread) 收到 undefined、validateBook valid=false
- 9 U1 + 3 regression-guard（場景 3/4）passed = 設計正確（無 false RED）

第 8 項欄位名鎖定結論：per-book schema 版本欄位採 camelCase schemaVersion，值 3.0.0。
依據 consumer 掃描（src/overview/、src/export/、src/storage/）：
- export 模組 schemaVersion 位於 metadata 物件（非逐本書）
- tag-storage-adapter schema_version 是 storage 頂層 key（STORAGE_KEYS.SCHEMA_VERSION，非逐本書欄位）
- 無任何逐本書讀取 schema 版本的 consumer，故採與 book 既有欄位（extractedAt/coverInfo）一致的 camelCase，對齊 Phase 1 D4。
