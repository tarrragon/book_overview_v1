## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

長背景任務（如 ticket_system pytest 約 9 分鐘）的輸出在 buffer 機制下憋到結束才出現，導致用戶與 PM 無法即時觀察。直接原因見「重現實驗結果」（-q + | tail 雙層 buffer）。

### 影響範圍

所有 PM/Claude 用 run_in_background 執行的長時間命令（測試、build、批量腳本）。非單一指令問題，屬通用慣例缺口。

### 相關規則

- bash-tool-usage-rules.md 已有「大輸出防護」（| tail / | head 防淹沒），但該防護與「可觀測性」目標衝突——防淹沒用的 tail 正是殺死即時進度的元兇。兩需求需在規則層調和。
- quality-baseline 規則 4（異常可觀測性）精神延伸：任務監控也需可觀測。

## 重現實驗結果
### 本 session buffering 實證（2026-05-29）

指令：`pytest tests/ -q -p no:cacheprovider 2>&1 | tail -5`（run_in_background）
觀察：執行中多次讀取輸出檔 `tasks/<id>.output` 均為 **0 行（全空）**，直到指令結束才一次性出現 `1770 passed in 525.07s`。

根因兩層：
1. `| tail -5`：tail 只在輸入 EOF（pytest 全跑完）後才吐最後 5 行 → 憋住全部輸出到最後。
2. `-q` + 非 TTY：pytest 偵測 stdout 非終端機時進度點不即時 flush（fully-buffered）。

對照：`stdio` 三態 = 終端機 line-buffered（即時）/ 檔案 pipe fully-buffered（4-8KB 才 flush）/ tail 額外等 EOF。

### Spike 驗證結果（acceptance 3，2026-05-29）

**受控 emitter 對比**（python 每秒印 1 行，共 8 行，經 tee 寫 log）：

| 時間 | buffered（預設） | unbuffered（PYTHONUNBUFFERED=1） |
|------|------------------|----------------------------------|
| t+4s | 0 行 | 3 行 |
| t+6s | 0 行 | 5 行 |
| t+8s | 0 行 | 7 行 |
| t+10s | 8 行（結束一次倒出） | 8 行 |

**結論**：
1. `PYTHONUNBUFFERED=1 <cmd> \| tee logfile`（不接 `\| tail`、不用 `-q`）→ log 逐行即時成長，`!tail -f logfile` 可觀察。
2. buffered（預設）→ 全程 0 行、結束才一次吐出（= 本 session `-q \| tail` 病根）。
3. **`PYTHONUNBUFFERED=1` 單獨即足**，不需 `stdbuf`（且 macOS stdbuf LD_PRELOAD 可靠性存疑）→ WRAP B 方案（複雜 wrapper）非必要。
4. pytest 子集驗證：`-v + tee`（非 -q）log 得 250 行逐測試輸出（對比 -q|tail = 0 行）；惟子集 1.68s 太快無法觀察 mid-flight，故 streaming 增量由 emitter 對比證實。

**對解法的修正**：A 規則為主已足夠，B wrapper 降級為非必要（一個環境變數即解）。pytest streamable 慣例 = `PYTHONUNBUFFERED=1 ... pytest -v 2>&1 | tee <logfile>`，告知用戶 logfile path 供 `!tail -f`。
## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 分析

**錨點（流程決策）**：影響開發效率與用戶信任；不做的代價是每次長任務黑箱、用戶焦慮、PM 無法早期偵測卡死。一次性投入 → 持續回報。

**W — 擴增選項**：

| 方案 | 層級 | 做法 |
|------|------|------|
| A 規則 | 同層 | bash-tool-usage-rules 加「長背景任務可觀測性」條：需觀察的長任務禁 `-q \| tail`，改 unbuffered（PYTHONUNBUFFERED=1 / stdbuf -oL）+ `-v`，主動告知輸出檔 path 供 `!tail -f` |
| B wrapper | 跨領域 | scripts/ 建 run-observable.sh 封裝 unbuffered + tee 固定 log + 印 tail 指令 |
| C Monitor 工具 | 同領域 | Claude Code Monitor 串流回 Claude（串給 Claude 非用戶，定位不同） |
| D 純自律 | — | 現狀，PM 每次手動記得 |

**Consider the Opposite（重定義需求）**：逐行進度非真需求（1770 個快速通過測試人不會逐行看）。真需求 = **存活性（任務沒卡死）+ 失敗早現（即時知哪個 fail，非等 9 分鐘）**。故方案應含 pytest-instafail（失敗即現）+ 心跳式進度，優於純 verbose。

**R — 實境檢驗**：
- 基本率：pytest 慣例 = -v / --tb=short / pytest-instafail / -x / PYTHONUNBUFFERED；tee 到 log 為 unix 慣例。
- 試水溫（spike，acceptance 3）：驗證 unbuffered + tee 是否讓輸出檔即時長大。
- 鋼鐵人：B(wrapper) 優點=一次封裝不靠記憶；A(規則) 缺點=靠自律會忘 / 增 context / 跨框架（jest vs pytest）unbuffered 寫法不同 → 暗示 A+B 結合最穩。

**A — 拉開距離**：低成本、持續回報的 P2/P3 流程改善，不擋核心開發；排 pending，不需立即做。10/10/10：長期影響為「PM 跑長任務有一致可觀測習慣」。

**P — 準備犯錯**（premortem）：最可能失敗 =（1）規則靠自律仍忘；（2）跨框架 unbuffered 行為不同。防護 = spike 先驗證 + 規則配輕量 wrapper，回退成本低（規則可刪）。

### 結論

重新定義需求核心為「存活性 + 失敗早現」，採 **A 規則為主 + B 輕量 wrapper 為輔**，spike 驗證後落地。
