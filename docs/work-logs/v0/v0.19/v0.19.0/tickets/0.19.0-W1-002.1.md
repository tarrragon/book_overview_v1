## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 實作摘要

建立 `npm run build:prod` -> ZIP 打包流程，產出可分發的 Chrome Extension 壓縮檔。

### 新增 / 修改檔案

| 檔案 | 變更 |
|------|------|
| `scripts/package.js` | 新增打包腳本（Node script，與既有 `scripts/*.js` 一致） |
| `package.json` | 新增 `package`（一鍵 build:prod + 打包）與 `package:zip`（僅打包）兩個 npm script |
| `.gitignore` | 新增 `dist/` 排除打包產物 |

### 設計決策

1. 版本號動態讀取：`scripts/package.js` 從 `build/production/manifest.json` 的 `version` 欄位讀取版本號作為 ZIP 檔名（`readmoo-book-extractor-v{version}.zip`），禁止硬編碼。目前產出 `v0.18.0`，符合 Context Bundle 說明——版本 bump 至 0.19.0 由 version-release 流程負責，不在本 ticket 範圍。未修改 manifest.json / package.json 的 version 欄位。

2. manifest.json 根層：透過 `execFileSync` 的 `cwd` 選項在 `build/production/` 目錄內執行 `zip`，使壓縮路徑相對於該目錄，確保 manifest.json 落在 ZIP 根層（符合 Chrome unpacked 載入要求）。

3. 不引入新 npm 依賴：採用系統 `zip` 命令而非 `archiver` 套件，避免為內測打包工具增加 supply chain 表面積。`zip` 在 macOS/Linux 與 CI 均預裝；不可用時腳本給出明確錯誤指引與安裝命令（符合可觀測性規則 4）。

4. 排除 .backup 殘留檔：`zip -x "*.backup*"` 排除 build 產物中的 4 個開發殘留檔（`.backup` / `.backup-*`），避免污染分發產物。

5. 產物驗證內建：`scripts/package.js` 打包後自動驗證（a）ZIP 大小 < 5 MB，（b）manifest.json 位於根層；任一不符即 `process.exit(1)` 中止。

### 未動範圍說明

未精簡 `build/production/src/` 散落檔案。雖然 esbuild 已將依賴 bundle 進 4 個 entry bundle，但 `popup.html` 額外引用 4 個非 esbuild 入口的 js（`popup-ui-components.js` 等），精準判定可刪除檔案需深度依賴分析，風險高且超出「建立打包流程」範圍。ZIP 壓縮後僅 1.21 MB，遠低於 5 MB 上限，無體積問題。
