## Feature Spec
<!-- TDD Phase 1 功能規格（/spec Full 模式：聚焦接線介面契約）。撰寫者：lavender-interface-designer -->

### 1. Purpose（目的）

讓 `BookFileImporter` 在解析匯入檔案後，自動偵測 v1/v2 格式並將 v1 資料轉換為統一的 v2 interchange 結構，使 UC-04 匯入流程能正確處理 Readmoo 書庫匯出的 v1 舊格式檔案。

- **目標用戶**：使用 UC-04「匯入 JSON」功能、持有 v1 格式備份檔的使用者。
- **核心價值**：消除匯出/匯入功能不對稱（UC-03 匯出已 PASS，UC-04 匯入端缺版本偵測），達成 v1.0.0 內測「匯出入正常」硬需求。
- **範圍邊界**：本 ticket 限「版本偵測分流 + v1 轉換接線 + 驗證放寬」三項；v2 路徑 `tagCategories`/`tags` 結構提取與回傳介面變更屬 IMP-B（W1-047.2），不在範圍。

### 2. API Signatures（介面定義）

接線涉及的既有模組為外部依賴，介面契約如下（不修改其實作）：

| 模組函式 | 簽名 | 回傳語義 |
|---------|------|---------|
| `detectFormatVersion(data)` | `(data: any) => 'v1' \| 'v2' \| null` | 依 SPEC §2 四條規則判定；無法辨識回 `null` |
| `convertV1ToV2Data(v1Data)` | `(v1Data: Object\|Array) => v2Interchange` | 回傳 `{ metadata, tagCategories, tags, books }`；逐本轉換，失敗書籍跳過 |

`BookFileImporter` 內部變更的介面契約：

| 方法 | 變更前 | 變更後 |
|------|--------|--------|
| `_extractBooksFromData(data)` | 回傳 `Array`（books 陣列），不辨識版本 | 回傳 `Array`（books 陣列）；內部先呼叫 `detectFormatVersion`，v1 → 經 `convertV1ToV2Data` 轉換後取 `.books`；v2 / 三種既有陣列形狀維持原行為。**回傳型別不變**（仍為純 books 陣列），確保 IMP-B 之前下游不受影響 |
| `_validateRequiredFields(book)` | `Boolean(id) && Boolean(title) && Boolean(cover)` | `Boolean(id) && Boolean(title)`（移除 cover 強制，依 SPEC §3.5 cover 選填） |

**回傳結構決策**：IMP-A 維持 `_extractBooksFromData` 回傳純 books 陣列（v1 轉換後僅取 `convertV1ToV2Data(...).books`），`tagCategories`/`tags` 暫時丟棄。理由：回傳介面變更為 `{ books, tagCategories, tags }` 屬 IMP-B 範圍，IMP-A 先完成版本偵測 + 書籍欄位轉換，避免跨 ticket 介面破壞。

**版本偵測與既有陣列形狀辨識的關係**：`detectFormatVersion` 涵蓋三種 v1 結構（純陣列 / `data.books` 無 formatVersion / metadata 包裝）。`_extractBooksFromData` 既有的 `_isMetadataWrapFormat`（`data.data` 陣列）不在 `detectFormatVersion` 規則內——此形狀偵測結果為 `null`（無法辨識），須維持既有 fallback 行為直接取 `data.data`，不可因偵測回 `null` 而拋錯。

### 3. GWT Scenarios（行為場景）

**場景 1: v1 純陣列格式匯入並轉換**
- **Given**: 匯入檔案解析後為純物件陣列（含 `isNew`/`isFinished`/`category` 欄位、無 `metadata`）
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `'v1'`；資料經 `convertV1ToV2Data` 轉換；回傳的 books 每本含 `readingStatus`（依 `mapV1StatusToV2`）、`authors` 陣列、`tagIds`、`isManualStatus`，不含 `isNew`/`isFinished`

**場景 2: v1 books 包裝格式（無 formatVersion）匯入並轉換**
- **Given**: 匯入檔案為 `{ books: [...] }` 結構、無 `metadata.formatVersion`、books 不含 `readingStatus`
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `'v1'`；經 `convertV1ToV2Data` 轉換；回傳轉換後 books 陣列

**場景 3: v2 格式匯入維持原行為**
- **Given**: 匯入檔案含 `metadata.formatVersion` 以 `"2."` 開頭，或 `metadata` + books 含 `readingStatus`
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `'v2'`；不呼叫 `convertV1ToV2Data`；直接取 `data.books` 回傳（v2 結構提取留待 IMP-B）

**場景 4: 既有 metadata 包裝（`data.data` 陣列）維持 fallback**
- **Given**: 匯入檔案為 `{ data: [...] }` 結構（`detectFormatVersion` 規則不涵蓋）
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `null`；不拋錯；維持既有行為取 `data.data` 回傳

**場景 5: v1 書籍缺 cover 不被過濾**
- **Given**: v1 格式書籍含合法 `id` 與 `title`、無 `cover` 欄位
- **When**: 經版本偵測轉換後進入 `_filterValidBooks` → `_validateRequiredFields`
- **Then**: `_validateRequiredFields` 回傳 `true`（僅驗證 id+title），該書籍保留於結果

**場景 6: 無法辨識且非既有陣列形狀**
- **Given**: 匯入檔案為非陣列、無 `books`、無 `data` 陣列的物件（且非空物件）
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: 維持既有行為拋出 `VALIDATION_ERROR`「JSON 檔案應該包含一個陣列或包含books屬性的物件」

**場景 7: 空 JSON 物件**
- **Given**: 匯入檔案為空物件 `{}`
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: 維持既有行為回傳空陣列 `[]`

### 4. Error Handling（錯誤處理）

| 錯誤情境 | 處理策略 | 回傳/拋出 |
|---------|---------|----------|
| `detectFormatVersion` 回傳 `null` 但符合既有 `_isMetadataWrapFormat`（`data.data`） | fallback 取 `data.data` | 回傳 `data.data` 陣列 |
| `detectFormatVersion` 回傳 `null` 且為空物件 | 既有行為 | 回傳 `[]` |
| `detectFormatVersion` 回傳 `null` 且為其他無法辨識結構 | 既有行為 | 拋 `VALIDATION_ERROR` |
| v1 轉換中個別書籍缺 `id`/`title` | `convertV1ToV2Data` 內部逐本 try-catch，失敗書籍跳過（不影響整批） | 該書籍不出現於回傳陣列 |
| v1 `category` 為空字串或 `"未分類"` | `convertV1CategoryToTag` 內部過濾，不建立 tag | 該書籍 `tagIds` 為 `[]` |

**設計約束**：版本偵測接線不得改變既有 `_extractBooksFromData` 的錯誤拋出契約（場景 6/7）。新增分支僅在 `detectFormatVersion` 明確回 `'v1'` 時介入轉換。
