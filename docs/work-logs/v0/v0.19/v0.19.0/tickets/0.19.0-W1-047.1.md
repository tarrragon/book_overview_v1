## Feature Spec
<!-- TDD Phase 1 功能規格（/spec Full 模式：聚焦接線介面契約）。撰寫者：lavender-interface-designer -->

### 1. Purpose（目的）

讓 `BookFileImporter` 在解析匯入檔案後，自動偵測 v1/v2 格式並將 v1 資料轉換為統一的 v2 interchange 結構，使 UC-04 匯入流程能正確處理 Readmoo 書庫匯出的 v1 舊格式檔案。

- **目標用戶**：使用 UC-04「匯入 JSON」功能、持有 v1 格式備份檔的使用者。
- **核心價值**：消除匯出/匯入功能不對稱（UC-03 匯出已 PASS，UC-04 匯入端缺版本偵測），達成 v1.0.0 內測「匯出入正常」硬需求。
- **範圍邊界**：本 ticket 限「版本偵測分流 + v1 轉換接線 + 驗證放寬」三項；v2 路徑 `tagCategories`/`tags` 結構提取與回傳介面變更屬 IMP-B（W1-047.2），不在範圍。

### 2. API Signatures（介面定義）

接線涉及的既有模組為外部依賴，介面契約如下（不修改其實作）：

| 模組函式 | 簽名 | 回傳語義 |
|---------|------|---------|
| `detectFormatVersion(data)` | `(data: any) => 'v1' \| 'v2' \| null` | 依 SPEC §2 四條規則判定；無法辨識回 `null` |
| `convertV1ToV2Data(v1Data)` | `(v1Data: Object\|Array) => v2Interchange` | 回傳 `{ metadata, tagCategories, tags, books }`；逐本轉換，失敗書籍跳過 |

`BookFileImporter` 內部變更的介面契約：

| 方法 | 變更前 | 變更後 |
|------|--------|--------|
| `_extractBooksFromData(data)` | 回傳 `Array`（books 陣列），不辨識版本 | 回傳 `Array`（books 陣列）；內部先呼叫 `detectFormatVersion`，v1 → 經 `convertV1ToV2Data` 轉換後取 `.books`；v2 / 三種既有陣列形狀維持原行為。**回傳型別不變**（仍為純 books 陣列），確保 IMP-B 之前下游不受影響 |
| `_validateRequiredFields(book)` | `Boolean(id) && Boolean(title) && Boolean(cover)` | `Boolean(id) && Boolean(title)`（移除 cover 強制，依 SPEC §3.5 cover 選填） |

**回傳結構決策**：IMP-A 維持 `_extractBooksFromData` 回傳純 books 陣列（v1 轉換後僅取 `convertV1ToV2Data(...).books`），`tagCategories`/`tags` 暫時丟棄。理由：回傳介面變更為 `{ books, tagCategories, tags }` 屬 IMP-B 範圍，IMP-A 先完成版本偵測 + 書籍欄位轉換，避免跨 ticket 介面破壞。

**版本偵測與既有陣列形狀辨識的關係**：`detectFormatVersion` 涵蓋三種 v1 結構（純陣列 / `data.books` 無 formatVersion / metadata 包裝）。`_extractBooksFromData` 既有的 `_isMetadataWrapFormat`（`data.data` 陣列）不在 `detectFormatVersion` 規則內——此形狀偵測結果為 `null`（無法辨識），須維持既有 fallback 行為直接取 `data.data`，不可因偵測回 `null` 而拋錯。

### 3. GWT Scenarios（行為場景）

**場景 1: v1 純陣列格式匯入並轉換**
- **Given**: 匯入檔案解析後為純物件陣列（含 `isNew`/`isFinished`/`category` 欄位、無 `metadata`）
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `'v1'`；資料經 `convertV1ToV2Data` 轉換；回傳的 books 每本含 `readingStatus`（依 `mapV1StatusToV2`）、`authors` 陣列、`tagIds`、`isManualStatus`，不含 `isNew`/`isFinished`

**場景 2: v1 books 包裝格式（無 formatVersion）匯入並轉換**
- **Given**: 匯入檔案為 `{ books: [...] }` 結構、無 `metadata.formatVersion`、books 不含 `readingStatus`
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `'v1'`；經 `convertV1ToV2Data` 轉換；回傳轉換後 books 陣列

**場景 3: v2 格式匯入維持原行為**
- **Given**: 匯入檔案含 `metadata.formatVersion` 以 `"2."` 開頭，或 `metadata` + books 含 `readingStatus`
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `'v2'`；不呼叫 `convertV1ToV2Data`；直接取 `data.books` 回傳（v2 結構提取留待 IMP-B）

**場景 4: 既有 metadata 包裝（`data.data` 陣列）維持 fallback**
- **Given**: 匯入檔案為 `{ data: [...] }` 結構（`detectFormatVersion` 規則不涵蓋）
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: `detectFormatVersion` 回傳 `null`；不拋錯；維持既有行為取 `data.data` 回傳

**場景 5: v1 書籍缺 cover 不被過濾**
- **Given**: v1 格式書籍含合法 `id` 與 `title`、無 `cover` 欄位
- **When**: 經版本偵測轉換後進入 `_filterValidBooks` → `_validateRequiredFields`
- **Then**: `_validateRequiredFields` 回傳 `true`（僅驗證 id+title），該書籍保留於結果

**場景 6: 無法辨識且非既有陣列形狀**
- **Given**: 匯入檔案為非陣列、無 `books`、無 `data` 陣列的物件（且非空物件）
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: 維持既有行為拋出 `VALIDATION_ERROR`「JSON 檔案應該包含一個陣列或包含books屬性的物件」

**場景 7: 空 JSON 物件**
- **Given**: 匯入檔案為空物件 `{}`
- **When**: `_extractBooksFromData` 處理該資料
- **Then**: 維持既有行為回傳空陣列 `[]`

### 4. Error Handling（錯誤處理）

| 錯誤情境 | 處理策略 | 回傳/拋出 |
|---------|---------|----------|
| `detectFormatVersion` 回傳 `null` 但符合既有 `_isMetadataWrapFormat`（`data.data`） | fallback 取 `data.data` | 回傳 `data.data` 陣列 |
| `detectFormatVersion` 回傳 `null` 且為空物件 | 既有行為 | 回傳 `[]` |
| `detectFormatVersion` 回傳 `null` 且為其他無法辨識結構 | 既有行為 | 拋 `VALIDATION_ERROR` |
| v1 轉換中個別書籍缺 `id`/`title` | `convertV1ToV2Data` 內部逐本 try-catch，失敗書籍跳過（不影響整批） | 該書籍不出現於回傳陣列 |
| v1 `category` 為空字串或 `"未分類"` | `convertV1CategoryToTag` 內部過濾，不建立 tag | 該書籍 `tagIds` 為 `[]` |

**設計約束**：版本偵測接線不得改變既有 `_extractBooksFromData` 的錯誤拋出契約（場景 6/7）。新增分支僅在 `detectFormatVersion` 明確回 `'v1'` 時介入轉換。

#### 3. 測試案例清單

##### Group A：版本偵測分流（對應 AC-1、場景 1-4）

**TC-01 v1 純陣列格式偵測並轉換**
- Given：`data` = 純物件陣列 `[{ id:'b1', title:'書一', isNew:false, isFinished:true, progress:100, author:'作者甲', category:'科幻' }]`（無 metadata）
- 前置驗證：`Array.isArray(data) === true`
- When：`importer._extractBooksFromData(data)`
- Then 斷言：
  - 回傳值為陣列，`length === 1`
  - `result[0].readingStatus === 'finished'`（isFinished=true 優先）
  - `result[0].authors` 為陣列且 `toEqual(['作者甲'])`
  - `result[0]` 不含 `isNew` 屬性（`not.toHaveProperty('isNew')`）
  - `result[0]` 不含 `isFinished` 屬性
  - `result[0].isManualStatus === false`
  - `result[0].tagIds` 為陣列且 `length === 1`（category 科幻已轉 tag）
  - `result[0].source === 'readmoo'`

**TC-02 v1 books 包裝格式（無 formatVersion）偵測並轉換**
- Given：`data` = `{ books: [{ id:'b1', title:'書一', progress:50 }] }`，無 `metadata`，books 不含 `readingStatus`
- 前置驗證：`data.metadata === undefined` 且 `Array.isArray(data.books) === true`
- When：`importer._extractBooksFromData(data)`
- Then 斷言：
  - `detectFormatVersion(data)` 回 `'v1'`（可獨立斷言驗證偵測，或僅驗證轉換結果）
  - 回傳值 `length === 1`
  - `result[0].readingStatus === 'reading'`（progress=50 > 0）
  - `result[0]` 為 v2 結構（含 `tagIds`/`isManualStatus`/`authors`）

**TC-03 v2 格式（明確 formatVersion）維持原行為**
- Given：`data` = `{ metadata: { formatVersion:'2.0.0' }, books: [{ id:'b1', title:'書一', readingStatus:'finished', cover:'http://x/c.jpg' }] }`
- 前置驗證：`data.metadata.formatVersion` 以 `'2.'` 開頭

#### 1. 實作策略設計

本 ticket 為純接線（wiring），既有 SPEC 模組 100% 可重用，策略聚焦「在何處插入版本偵測分流」與「v1 分支如何銜接轉換結果」。共三處改動，相互獨立，可並行實作。

##### 改動 1：模組引用（檔案頂部）

在 `_extractBooksFromData` 取得分流能力前，需引用兩個既有模組。既有檔案已用 `require('src/...')` alias 形式（見 `_loadFileReaderFactory` 與檔頭 `require('src/core/errors/ErrorCodes')`），新增引用沿用同一風格，置於檔頭 `ErrorCodes` 引用之後：

```
引用 detectFormatVersion  自 版本偵測模組（src/export/format-version-detector）
引用 convertV1ToV2Data    自 v1轉v2轉換模組（src/export/v1-to-v2-converter）
```

##### 改動 2：`_extractBooksFromData` 接入版本偵測分流（核心）

設計決策：版本偵測作為「前置分流閘門」插入既有四種形狀辨識之前，且僅在偵測明確回傳 `'v1'` 時介入。`'v2'` 與 `null` 一律落回既有形狀辨識邏輯（不改動既有四分支）。此設計確保既有錯誤契約（場景 6/7）與 `data.data` fallback（場景 4）零回歸。

```
函式 _extractBooksFromData(data):
    版本 ← detectFormatVersion(data)

    若 版本 == 'v1':
        # v1 分支：呼叫轉換，僅取 books 陣列（tagCategories/tags 暫丟棄，屬 IMP-B）
        轉換結果 ← convertV1ToV2Data(data)
        回傳 轉換結果.books

    # 版本為 'v2' 或 null：完全沿用既有四分支邏輯，不改動
    若 _isDirectArrayFormat(data):     回傳 data
    若 _isWrappedBooksFormat(data):    回傳 data.books
    若 _isMetadataWrapFormat(data):    回傳 data.data
    若 data 為空物件:                  回傳 []
    拋出 VALIDATION_ERROR（既有訊息與 code 不變）
```

關鍵設計理由：

| 決策點 | 設計選擇 | 理由 |
|--------|---------|------|
| 偵測閘門位置 | 置於四分支之前 | v1 純陣列 / v1 books 包裝會同時命中 `_isDirectArrayFormat` / `_isWrappedBooksFormat`，必須在既有分支前攔截，否則 v1 資料以原始欄位繞過轉換 |
| 僅 `'v1'` 介入 | `'v2'` 與 `null` 都 fall through | `'v2'` 直接取 `data.books`（IMP-A 不做 v2 結構提取）；`null` 涵蓋 `data.data` 包裝與空物件與無法辨識，須交既有邏輯處理 |
| v1 取 `.books` | 丟棄 `tagCategories`/`tags` | 回傳型別維持純 books 陣列，回傳介面變更屬 IMP-B；IMP-A 不破壞下游 |
| 不在 v1 分支重複辨識陣列 | 直接傳整個 `data` 給 `convertV1ToV2Data` | `convertV1ToV2Data` 內部已處理 Array 與 `{books:[]}` 兩種輸入（見其 L119-125），無需在 importer 端再拆 |

##### 改動 3：`_validateRequiredFields` 放寬

```
函式 _validateRequiredFields(book):
    回傳 truthy(book.id) AND truthy(book.title)
    # 移除原 AND truthy(book.cover)
```

注意：`_validateFieldTypes`（L582）仍檢查 `typeof book.cover === 'string'`。v1 轉換產出的 book 其 `cover` 為 `''`（`convertV1ToV2Book` L41 `cover: v1Book.cover || ''`），型別為 string，通過型別檢查。TC-12 前置驗證已確認此鏈路。**`_validateFieldTypes` 不在本 ticket 改動範圍**——v1 轉換保證 cover 為字串，型別檢查無需放寬。

#### 6. 權宜方案與技術債務

- **最小可行實作**：本 ticket 為純接線，無權宜簡化。三處改動皆為最終形態。
- **已知限制（非技術債，屬 IMP-B 範圍）**：
  - v1 轉換產出的 `tagCategories` / `tags` 於 IMP-A 被丟棄。`_extractBooksFromData` 維持回傳純 books 陣列。**這是刻意的範圍切割**（§2 回傳結構決策），IMP-B（W1-047.2）將變更回傳介面為 `{ books, tagCategories, tags }` 並完成 v2 路徑的 `tagCategories`/`tags` 提取。已有 follow-up ticket（IMP-B / W1-047.2）承接，符合決策 trigger 綁定。
- **架構決策紀錄**：

- **無技術債標記**：本 ticket 不產生需 Phase 4 重構的技術債。
