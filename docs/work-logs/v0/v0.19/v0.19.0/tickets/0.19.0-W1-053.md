### Problem Analysis

### 問題根因

W1-048.4.1 派發 thyme（isolation=worktree）時，cc runtime 自建的 agent worktree base 停在 W4-002（未含 main 最新的 W1-047.1 / W1-048.x commits）。thyme 以舊版檔案為基礎工作，需自行執行 stash → merge main → stash pop 才能對齊。

**cc runtime worktree base 選擇邏輯（實證歸納）**：

cc runtime `isolation:worktree` 以 PM 觸發派發的瞬間 main HEAD commit 為 worktree base，此後不後續同步。base 建立後主 repo 新增的 commit 不自動反映到 worktree，agent 以快照時間點的檔案為基礎工作。worktree 與主 repo 共享 git object store，可在 worktree 內直接執行 `git merge main` 取得主 repo 已 commit 的內容。

此行為為系統性（每次 worktree 派發都可能遇到），落差大小取決於派發前主 repo 的 commit 活躍度。

### 影響範圍

每次使用 isolation=worktree 的背景代理人派發（parallel-dispatch.md 規定所有實作代理人強制使用）都可能遇到 stale base，未處理時 agent 以過時版本工作，產出與 main 不相容需手動整合。

## Solution
<!-- Schema[DOC/Solution]: 免填（DOC 類型以變更摘要取代）（.claude/pm-rules/ticket-body-schema.md） -->
### Solution

### 三方案評估

| 方案 | 說明 | 優點 | 缺點 |
|------|------|------|------|
| A：PM 派發前 commit gate | PM 派發前確認 main HEAD 已 commit | 縮小初始落差 | 無法防止派發後主 repo 新增 commit；agent 不受保護 |
| B：prompt 中加 base-stale 處理 step | prompt 開頭加 git merge main 指引 | 覆蓋全部落差（初始 + 派發後）；agent 端自律；成本低 | 每次 prompt 需加入，漏加失保護 |
| C：hook 在 worktree 建立後自動 merge | PostToolUse hook 偵測 worktree 建立 | 全自動無需修改 prompt | cc runtime 建 worktree 屬內部機制，hook 難以可靠偵測時機 |

### 選定方案：B（prompt 加 base-stale 處理 step）

B 覆蓋範圍最完整（初始落差 + 派發後 main 新增 commit 均覆蓋），成本為每次 prompt 多一行 git merge 指引。A1（PM 派發前 commit gate）作為輔助防護與 B 並用。

### 文件變更摘要

1. `.claude/references/agent-dispatch-template.md`：W1-035 章節新增「cc runtime worktree base 選擇邏輯（實證歸納）」（行為表 + 結論）與「三方案評估與選定理由」（三方案比較表 + 選定理由），並補充 prompt 插入範本的 Why 說明。
