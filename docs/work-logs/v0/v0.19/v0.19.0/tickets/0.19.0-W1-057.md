### 採用方案

**方案 B：產品碼 timestamp 一致化**（PM 預填首選）

### Root Cause 確認

`src/export/v1-to-v2-converter.js` 內部 `convertV1CategoryToTag` 取兩次時間：
- L74 `const ts = timestamp || new Date().toISOString()`（ISO 字串，用於 createdAt / updatedAt）
- L75 `const tsMs = Date.now()`（毫秒整數，用於 tag id `tag_<tsMs>-NNN`）

兩次取值間 sub-millisecond 內通常一致，但 GC / scheduler 切換造成 > 1ms 間隔時，tag id 中的 tsMs 與 createdAt / updatedAt 的 ms 部分不一致——這違反「同次 conversion 的所有 timestamp 應源自單一時刻」的業務語意，且是測試 ms-race flaky 的真正根因。

W1-060 雖已從測試端用 inline structural assertions 規避 flaky（測試已穩定），但 src 端的設計 bug 仍存在。本 ticket 採方案 B 修 src 端，根除設計 bug。

### 變更摘要

| 檔案 | 變更 | 行數 |
|------|------|------|
| `src/export/v1-to-v2-converter.js` | `convertV1CategoryToTag` 內單次時間取值 + 雙向衍生（傳入 timestamp 時用 Date.parse 解出 tsMs；未傳入時 Date.now() 後衍生 ISO） | -2 / +12（含 doc comment） |
| 測試檔 | 不修改 | - |

### 設計決策

1. **不修改 caller signature**：`convertV1CategoryToTag` 簽章保持不變，向後相容
2. **不修改 `convertV1ToV2Data`**：原本就一次取 `timestamp = new Date().toISOString()` 傳入 `convertV1CategoryToTag`，現在能正確衍生 tsMs
3. **Date.parse 失敗回退**：若 caller 傳入非 ISO 字串導致 Date.parse 回 NaN，回退 `Date.now()` 避免產出無效 tag id（`tag_NaN-000`）
4. **未選 A（測試端 toMatchObject）**：W1-060 已採方案 A 解決測試 flaky，但留下 src 設計 bug 未解；本 ticket 補修根本問題
5. **未選 C（fakeTimers）**：只動測試不改 src，無法修復「同次 conversion 兩個 timestamp」的設計 bug
