### 規格補充（PM 預填）

**參考檔案**：

**三方案偏好順序評估**（依「修復根本設計問題優先」）：

| 方向 | 性質 | 評估 |
|------|------|------|
| B：產品碼 timestamp 一致化（推薦首選） | 修 converter 內 `new Date().toISOString()` 兩次呼叫，改為單次取值多處引用 | 修復真實設計 bug——同次 conversion 不應產生兩個 timestamp，這本身是 src 設計問題；測試會自然穩定；scope 僅 converter 內局部，非架構級改動 |
| C：Jest fakeTimers（次選） | 在測試內用 `jest.useFakeTimers()` 控制時間，凍結 `Date.now()` | 只動測試，不動產品碼；適用於確實需要 timestamp 唯一性但 src 設計合理的情境（本案 src 設計本身有 bug，C 是 fallback） |
| A：toMatchObject 排除 timestamp（末選） | 改用 `toMatchObject` 或顯式排除 `updatedAt` 欄位 | 失去 timestamp 驗證能力；接受設計問題不修；最低成本但留下技術債 |

**首選 B 的理由**：
- 根因是 converter 設計問題（同次 conversion 兩個 timestamp）不是測試問題
- 修產品碼後測試斷言可保持 toEqual 完整覆蓋
- W1-075 採方向 C（mock data 重設計）是因為 mock data 設計問題；本案是 src 設計問題，對應修在 src 端
- 影響範圍局部（converter 內變數提取），低風險

**驗證要求**：
- 連跑 `npm test` 全套件 3 次穩定（規則 1）
- 隔離執行 `npx jest tests/unit/overview/book-file-importer-import-result.test.js` 仍全綠（無 regression）
