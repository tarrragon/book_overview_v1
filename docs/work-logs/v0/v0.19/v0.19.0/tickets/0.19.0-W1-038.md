## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

雙因素共振：(1) Agent 寫作慣性——LLM 自然以 `## Phase X` 組織多 Phase 產出，自律層規則約束力不足；(2) append-log CLI 無規範化——偵測到 H2 後僅 stderr warning 但照原樣寫入，工具層未阻斷違規內容進入 ticket md。

### 影響範圍

- `track_acceptance.py`：append-log 寫入管道（改動點）
- 所有透過 append-log 寫入的 ticket md：受益於源頭規範化
- acceptance-gate-hook 的「### 自檢結果」偵測：間接受益（H2 不再進入 → 不再需要 PM 批次降級 → 不再觸發 false negative）

### 相關 Error Pattern

- PC-110（body-check false negative via schema separator）：自定義 H2 切斷 Schema 章節範圍的原始案例
- PC-123（rule documented agent not complying execution gap）：規則已寫但 agent 不遵循的通用模式

### 實驗發現

**已驗證事實**：
1. W17-208 stderr warning 在三個 agent 執行時已部署且會觸發
2. Subagent stderr warning 不阻擋寫入，agent 無義務處理 stderr warning
3. custom_h2_checker 僅在 complete 時觸發（事後偵測），不阻止寫入時的違規
4. append-log CLI 不修改寫入內容（照原樣插入），無標題層級規範化機制
5. 防護鏈的兩個 warning（寫入時 W17-208 + complete 時 W17-072）皆為非阻擋，形成連續 warning 無阻力的落差

**根因分類**：雙因素共振
- 因素 A（agent 寫作慣性）：agent 自然以 `## Phase X` 組織多 Phase 產出，因 AGENT_PRELOAD 章節結構規則是「自律層」，無強制機制
- 因素 B（append-log 無規範化）：CLI 照原樣寫入，不檢查或轉換標題層級，即使偵測到 H2 也僅 stderr warning

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 根因分類表

| 因素 | 類型 | 證據 | 可控性 |
|------|------|------|--------|
| Agent 寫作慣性 | 行為層（LLM 輸出傾向） | W1-037 三個 agent（sage/thyme/cinnamon）各自獨立產出 `## Phase X` 標題 | 低：自律層規則（AGENT_PRELOAD）對 LLM 輸出的約束力有限，W1-037 三連違規已實證 |
| append-log CLI 無規範化 | 工具層（寫入管道） | track_acceptance.py:639-651 偵測 H2 後僅 stderr warning，照原樣寫入 | 高：工具層改動確定性最強，一次修改覆蓋所有 agent |
| Warning 不阻擋 | 機制層（回饋強度） | W17-208（寫入時）+ W17-072（complete 時）皆為 warning | 中：升級為阻擋需評估 false positive 風險 |

### 方案比較表

| 方案 | 描述 | 成本 | 效益 | 能否阻止 W1-037 模式 |
|------|------|------|------|---------------------|
| A：W17-072 warning 升級為阻擋 | complete 時偵測到自定義 H2 → exit 2 阻擋 complete | 低（custom_h2_checker return 改為阻擋信號）| 事後阻擋：agent 必須修正後才能 complete | 部分：阻止 complete 但不阻止寫入，PM 仍需手動修正 |
| B：append-log CLI 自動降級 H2 → H3 | append-log 寫入時 `re.sub(r'^## ', '### ', content)` | 低（track_acceptance.py 加 5 行） | 源頭阻斷：H2 永遠不進入 ticket md | 完全阻止：三個 agent 的 H2 在寫入時即被降級，無後續連鎖 |
| C：hook 偵測改用 H3/H4 通配 | acceptance-gate-hook 的「### 自檢結果」改為 `#{3,4} 自檢結果` regex | 低（regex 修改） | 降低 false negative | 不阻止：只修復下游偵測容錯，H2 仍進入 ticket md |
| D：agent definition 補章節結構範例 | 在 AGENT_PRELOAD 加 append-log 標題層級範例 | 極低（文件修改） | 自律層強化，效果不確定 | 不可靠：W1-037 已證明自律層失效 |
| E：B + A 組合 | append-log 自動降級（源頭）+ complete 阻擋（安全網） | 低（B + A 各自獨立） | 雙層防護：源頭規範化 + 事後安全網 | 完全阻止 + 安全網兜底 |

### 採用結論

**採方案 B（append-log CLI 自動降級 H2 → H3）作為唯一落地方案。**

理由：

1. **源頭阻斷 > 事後偵測**：W1-037 暴露的核心問題是「H2 進入 ticket md 後的連鎖效應」（PM 降級 H2 時連帶降級 H3 子標題 → acceptance-gate-hook false negative）。方案 B 在源頭阻斷 H2 進入，連鎖效應不會發生。

2. **確定性最高**：工具層改動對所有 agent 生效，不依賴 agent 自律或 PM 手動修正。`re.sub(r'(?m)^## ', '### ', content)` 為純字串操作，無 false positive 風險（append-log 內容本就不應含 H2）。

3. **不需方案 A（complete 阻擋）**：方案 B 已從源頭消除問題，complete 時的 W17-072 warning 維持現況即可（作為防禦性觀測，偵測非 append-log 途徑的 H2 寫入）。升級為阻擋會增加 false positive 風險（例如 PM 手動 Edit ticket md 時誤寫 H2）。

4. **不需方案 C**：H2 不再進入 ticket md 後，PM 不需批次降級 H2，`### 自檢結果` 不會被連帶降為 H4，acceptance-gate-hook 的固定 H3 偵測維持正確。

5. **不需方案 D**：自律層強化的邊際效益在工具層已阻斷的前提下極低。

**既有 ticket md migration**：不需要。方案 B 僅影響未來 append-log 寫入，既有 ticket md（含已修正的 W1-037）不需 batch migration。
