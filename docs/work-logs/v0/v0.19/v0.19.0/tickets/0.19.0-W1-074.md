## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

`scripts/build.js` 缺乏版號 sanity check：build:prod 流程從 `package.json` 注入 `manifest.version`，但未驗證該版號是否與當前 active worklog 一致。當 wave 已進入下一版（如 v0.19.0）但 package.json 未 bump 時，build 流程靜默產出舊版號 ZIP（W1-002.2 觀察：dist 命名為 `readmoo-book-extractor-v0.18.0.zip` 但 wave 已 10+ ticket complete 於 v0.19.0）。

### 影響範圍

| 檔案 | 修改類型 |
|------|---------|
| scripts/build.js | 新增 validateVersionAlignment + detectActiveWorklogVersion + lazy esbuild require + 移除模組頂層 side effect |
| tests/unit/scripts/build-version-check.test.js | 新增（10 個測試覆蓋一致/不一致/skip/掃描容錯四類） |

### 相關 Error Pattern

- **IMP-068**（sync-push 版號 bump 缺 sanity check）：本 ticket 借鑑同思路（assert 版號合理性、寫入前驗證、fail-fast）遷移至 build:prod 流程
- **PC-085** 同家族：避免靜默漂移的 sanity check 設計原則

### 設計決策

| 決策 | 結論 | 理由 |
|------|------|------|
| 純函式 + I/O 分離 | 採用 | 對齊 scripts/validate-manifest.js 既有模式，便於單元測試 mock |
| dev mode 是否強制檢查 | 否 | 避免 dev 期短暫不一致干擾日常開發（W1-072 ANA 推薦方向 C：內測前 bump） |
| prod mode 強制檢查 | 是 | build:prod 產出即潛在 release artifact，必須 fail-fast |
| --skip-version-check 逃生閥 | 提供 | dev 期特殊情境（如手動測試 build flow）使用 |
| esbuild lazy require | 採用 | jsdom 環境 require esbuild 觸發 TextEncoder/Uint8Array invariant 失敗，lazy 確保單元測試可 require build.js |
| 模組頂層 mkdir side effect | 移除 | 避免 require build.js 即建立 build/ 目錄污染測試環境，build() 內部仍負責建立 |

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 實作概要

在 `scripts/build.js` 加入兩支純函式 + build flow 整合 + 一支單元測試檔，落地 W1-072 ANA spawn IMP-B：

#### 新增純函式（scripts/build.js）

1. **`detectActiveWorklogVersion(workLogsRoot, fsAdapter)`**
   - 掃描 `docs/work-logs/v0/` 下所有 `v*.*/v*.*.*/v*.*.*-main.md`
   - 偵測 `**狀態**: 開發中` 標記
   - 回傳 `{ ok, version | reason, candidates? }`
   - 容錯：忽略不符 `^v\d+\.\d+(\.\d+)?$` 命名的目錄與檔案（如 `i18n-backlog.md` / `tickets/` / `README.md`）

2. **`validateVersionAlignment(packageVersion, workLogsRoot, fsAdapter)`**
   - 呼叫 `detectActiveWorklogVersion` 取得預期版號
   - 比對 `packageVersion` 字串相等
   - 回傳 `{ ok, error? expectedVersion, actualVersion }`

#### Build flow 整合

```js
// 進入 build() 後第一步：版號 sanity check
if (MODE === 'production' && !SKIP_VERSION_CHECK) {
  const result = validateVersionAlignment(packageJson.version, WORK_LOGS_ROOT);
  if (!result.ok) {
    console.error(result.error);
    // 三條修復建議
    process.exit(1);
  }
} else if (MODE === 'production' && SKIP_VERSION_CHECK) {
  console.warn('[VERSION CHECK] 已透過 --skip-version-check 略過版號 sanity check');
}
```

#### 副作用清理（為測試可 require）

- `esbuild` 從頂層 `require` 改為 `getEsbuild()` lazy require（避免 jsdom TextEncoder invariant 失敗）
- 移除模組頂層 `mkdirSync` side effect（`build()` 內部 line 320-324 仍負責清理 + 重建 BUILD_DIR）
- 移除頂層 `console.log` `🔨 開始編譯 ...`，改為 `build()` 函式內第一行（避免測試 require 輸出污染）

#### 測試檔（tests/unit/scripts/build-version-check.test.js）

10 個測試覆蓋四類情境：

| describe | 測試項 | 涵蓋 acceptance |
|----------|--------|----------------|
| detectActiveWorklogVersion | 單一 active / 全部已完成 / 多 active / 根目錄不存在 / 干擾目錄容錯 | AC2/AC4 掃描容錯 |
| validateVersionAlignment | 一致 / 不一致 / 無法偵測 | AC1/AC2/AC4 |
| --skip-version-check 契約宣告 | 純函式單一職責 + flag 由 build flow 控制 | AC3 |

mock fsAdapter 設計：記憶體 `{ path: content | null }` 對應表，`null` 表目錄、字串表檔案內容；`readdir` 列舉直接子項；`stat().isDirectory()` 依 content 型別判定。
