## Problem Analysis

### 問題根因

IMP-A（W1-047.1）已讓 `BookFileImporter` 接入版本偵測與 v1→v2 轉換，但回傳值仍為純 `books` 陣列。`_extractBooksFromData` 在 v1 路徑只取 `convertV1ToV2Data(data).books`、在 v2 路徑完全忽略頂層 `tagCategories`/`tags`（程式碼註解明示「tagCategories/tags 暫時丟棄，回傳介面變更屬 IMP-B」）。tag 資料在匯入第一站即遺失，後續 IMP-C 的 storage 持久化無資料可寫。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `src/overview/book-file-importer.js` | `_extractBooksFromData` 新增 tag 區段提取；`_processBookData` / `_handleFileContent` / `_readFileWithReader` / `handleFileLoad` 回傳結構由陣列改為 `{ books, tagCategories, tags }` |
| `src/overview/overview-page-controller.js` | `handleFileLoad` 與 `_handleFileContent` 代理方法解構 `books` 後傳給 `_updateUIWithBooks`；`_updateBooksData` 不變 |

### 相關 Error Pattern

無直接對應的 PC error pattern。此為介面契約擴充（IMP-A 已預留銜接點），非流程或實作錯誤。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### Phase 1 設計摘要（lavender-interface-designer）

**核心介面變更**：定義 `ImportResult = { books: Book[], tagCategories: TagCategory[], tags: Tag[] }`，三欄位型別恆為陣列。

**回傳鏈變更**：`_extractBooksFromData` → `_processBookData` → `_handleFileContent` → `_readFileWithReader` → `handleFileLoad` 五處回傳由 `Book[]` 改為 `ImportResult`。

**各路徑 tag 來源**：
- v1 JSON：取 `convertV1ToV2Data(data)` 的 `.tagCategories`/`.tags`（converter 回傳已含三區段，IMP-A 僅取 `.books`，本 ticket 補取其餘兩段）。
- v2 JSON：從根物件讀 `data.tagCategories`/`data.tags`，缺失或非陣列降級為 `[]`（SPEC §2.3）。
- CSV / 空物件：`tagCategories: []`、`tags: []`。

**controller 適配**：`handleFileLoad` 與 `_handleFileContent` 解構 `books` 傳給 `_updateUIWithBooks`；`tagCategories`/`tags` 本 ticket 暫不流入 UI 層（IMP-C 接手 storage 持久化）。`_updateBooksData` 簽名與非匯入呼叫端不變。

**範圍邊界**：僅 v2 結構提取 + 回傳介面變更；覆蓋/合併模式、storage 持久化、tag id 重映射、模式選擇 UI 屬 IMP-C~E。

**錯誤處理原則**：tag 區段「寬鬆降級」（缺失/型別錯 → `[]`，不拋錯）；books 提取「維持既有嚴格契約」（無法辨識結構仍拋 `VALIDATION_ERROR`）。本 ticket 不新增拋錯路徑。

詳見 `0.19.0-W1-047.2-feature-spec.md`。Phase 2（sage 測試設計）依該規格 10 個場景與 AC-1~AC-4 設計 RED 測試。

### Phase 3a 實作策略規劃（語言無關，pepper-test-implementer）

**執行時間**: 2026-05-23

#### 1. 策略總覽

回傳介面由「純 books 集合」升級為「三欄位結果結構 ImportResult」。核心是一條 5 段回傳鏈的逐段透傳改造，加上提取點新增 tag 區段擷取，加上消費端（controller）解構適配。整條鏈遵循一個不變式：

> 不變式 INV-1：回傳鏈上任一段的回傳值，三欄位 `books` / `tagCategories` / `tags` 永遠存在且型別恆為陣列。任何路徑不得回傳 `undefined` / `null`。

策略分三個改造區塊，彼此無循環依賴，可依序實作：

| 區塊 | 範圍 | 對應 TC |
|------|------|--------|
| 區塊 A：提取點 | `_extractBooksFromData` 改為回傳 `{books,tagCategories,tags}` | TC-01~05、07、08、10 |
| 區塊 B：回傳鏈透傳 | `_processBookData` → `_handleFileContent` → `_readFileWithReader` → `handleFileLoad` 透傳 ImportResult | TC-06、09、11 |
| 區塊 C：消費端適配 | controller `handleFileLoad`、`_handleFileContent` 解構 books | TC-12~14 |

#### 2. 資料結構選擇

| 結構 | 形態 | 理由 |
|------|------|------|
| ImportResult | 三鍵物件 `{books:[], tagCategories:[], tags:[]}` | 依 feature-spec §2.1；三鍵恆陣列使消費端可無條件解構，免 null 檢查 |
| tag 區段提取結果 | 子函式回傳陣列（合法）或 `[]`（非法/缺失） | 寬鬆降級原則，不拋錯 |

不新增類別或型別宣告檔（JS 無型別系統，ImportResult 為文件層約定，靠 INV-1 與測試保證）。

#### 3. 區塊 A：提取點 `_extractBooksFromData` 改造

現況：回傳純陣列（v1 路徑取 `convertV1ToV2Data(data).books`；v2 四形狀辨識回傳對應陣列；空物件回傳 `[]`；無法辨識拋 `VALIDATION_ERROR`）。

改造後虛擬碼：

```
函式 _extractBooksFromData(data, fileFormat = 'json'):
    # 路徑 1：v1 JSON — 取 converter 完整三區段
    若 fileFormat == 'json' 且 detectFormatVersion(data) == 'v1':
        converted = convertV1ToV2Data(data)
        回傳 {
          books: converted.books,
          tagCategories: 正規化為陣列(converted.tagCategories),
          tags: 正規化為陣列(converted.tags)
        }

    # 路徑 2：v2 JSON 四形狀辨識 — books 取既有順序，tag 從根物件提取
    booksArray = null
    若 _isDirectArrayFormat(data):      booksArray = data
    否則若 _isWrappedBooksFormat(data): booksArray = data.books
    否則若 _isMetadataWrapFormat(data): booksArray = data.data

    若 booksArray 非 null:
        回傳 {
          books: booksArray,
          tagCategories: 提取tag區段(data, 'tagCategories'),
          tags: 提取tag區段(data, 'tags')
        }

    # 路徑 3：空物件
    若 data 為物件 且 鍵數量 == 0:
        回傳 { books: [], tagCategories: [], tags: [] }

    # 路徑 4：無法辨識 — 維持既有錯誤契約（零回歸）
    拋出 VALIDATION_ERROR(
      message='JSON 檔案應該包含一個陣列或包含books屬性的物件',
      code=VALIDATION_ERROR, details={category:'validation'})
```

關鍵設計點：

- **books 提取順序完全不動**：`_isDirectArray` → `_isWrappedBooks` → `_isMetadataWrap` 三辨識函式與順序維持原樣（feature-spec Q1 已定案），避免回歸。
- **tag 提取與 books 形狀脫鉤**：v2 路徑 `tagCategories`/`tags` 一律從根物件 `data.tagCategories` / `data.tags` 讀取，與 books 取自哪個形狀無關。
- **拋錯路徑不變**：路徑 4 拋錯時不回傳值，訊息/code/details 一字不改（TC-08 驗此契約）。
- **CSV 路徑**：CSV 不經 `_extractBooksFromData` 的 v1 分支（`fileFormat==='csv'` 時跳過版本偵測），會落入「路徑 2」的 `_isDirectArrayFormat`（CSV 解析結果為陣列），故 tag 區段會走 `提取tag區段(陣列, 'tagCategories')`。輔助函式需對「輸入非物件」也安全回傳 `[]`（見下節）。

#### 4. 輔助函式：tag 區段提取（新增）

```
函式 提取tag區段(data, 欄位名):
    候選 = (data 為物件 非 null) ? data[欄位名] : undefined
    若 Array.isArray(候選):
        回傳 候選
    否則:
        # 缺失 / 物件 / 字串 / null 一律降級
        若 候選 不為 undefined 且 候選 不為 null:
            記錄 console.warn(欄位名 + ' 型別非陣列，降級為空陣列')   # 可觀測性，feature-spec §4 建議
        回傳 []
```

```
函式 正規化為陣列(值):
    回傳 Array.isArray(值) ? 值 : []
```

設計理由：

- 單一函式涵蓋「缺失」「非陣列」「null」三種降級情境（TC-02、TC-03），滿足 INV-1。
- `console.warn` 僅在「值存在但型別錯」時觸發（缺失屬正常情況不告警），符合 observability-rules 規則 1「降級需含欄位名與原因」。
- `正規化為陣列` 用於 v1 converter 輸出防禦——converter 契約已回傳陣列，但加防禦使 INV-1 在所有路徑成立，且免去對 converter 內部實作的耦合假設。

#### 5. 區塊 B：回傳鏈透傳改造

四個方法的改造皆為「機械式透傳」，不含邏輯分支變更：

```
_processBookData(data, fileFormat):
    result = _extractBooksFromData(data, fileFormat)      # 現為 ImportResult
    validBooks = _filterValidBooks(result.books)          # 過濾只作用 books
    _checkLargeDataset(validBooks)
    回傳 { books: validBooks,
           tagCategories: result.tagCategories,           # 透傳，不過濾
           tags: result.tags }

_handleFileContent(content, fileFormat='json'):
    cleanContent = _validateAndCleanContent(content)
    data = (fileFormat=='csv') ? _parseCSVContent(...) : _parseJSONContent(...)
    回傳 _processBookData(data, fileFormat)               # 現為 ImportResult，直接回傳

_handleReaderSuccess(e, resolve, reject, fileFormat):
    嘗試:
        result = _handleFileContent(e.target.result, fileFormat)   # ImportResult

# _readFileWithReader / handleFileLoad 本身不解構，
# Promise resolve 值即為 ImportResult，無需改 code 主體（型別語意改變，註解需更新）
```

關鍵設計點：

- **`_filterValidBooks` / `_isValidBook` / `_checkLargeDataset` 簽名與行為完全不動**：仍只接收並處理 books 陣列。`_processBookData` 中明確只把 `result.books` 餵給過濾器，`tagCategories`/`tags` 原樣透傳未過濾（TC-11 驗此）。
- **`_readFileWithReader` / `handleFileLoad`**：程式碼主體無需改動（它們只是傳遞 Promise / 呼叫鏈），但 JSDoc 的 `@returns {Promise<Array>}` 須改為 `Promise<ImportResult>`，`_handleReaderSuccess` 內變數名 `books` 宜改為 `result`（語意正確性）。
- **錯誤路徑零回歸**：`_handleReaderSuccess` 的 try/catch、`_handleFileProcessError`、`_parseJSONContent` 的 PARSE_ERROR 全部不動（TC-09 驗此）。

#### 8. 權宜方案與技術債務

- **最小可行實作**：本 ticket 只「提取並回傳」tag 區段，不做 referential integrity 驗證（tag 的 `categoryId` 懸空引用不處理）、不做 tag schema 欄位驗證。屬刻意範圍切割，非技術債。
- **已知限制**：
  - CSV 路徑恆回傳 `tags:[]`、`tagCategories:[]`；v2 CSV 衍生 tag 欄位反向解析（SPEC §5.1）為後續獨立功能（feature-spec Q5 已標記範圍外）。
  - `tagCategories`/`tags` 在本 ticket 流到 controller 後即被丟棄（未流入 UI 層 / storage），等 IMP-C 接手持久化。此為刻意 ticket 鏈設計，非債務。
- **技術債務標記**：無新增技術債。本 ticket 為 IMP-A 已預留銜接點的介面契約完成，提取點 + 透傳 + 解構皆為低風險機械改造。
- **架構決策記錄**：
  - ADR-1：tag 區段「寬鬆降級」vs books「嚴格契約」並存——降級邏輯集中於單一輔助函式 `提取tag區段`，不污染既有 books 辨識/拋錯邏輯。理由：兩種錯誤策略本質不同（tag 缺失可容忍、books 結構錯誤不可容忍），分離可避免互相干擾且利於未來各自演進。
  - ADR-2：INV-1（三欄位恆陣列）以「提取端保證 + 輔助函式 `正規化為陣列` 防禦」雙層落實，而非依賴消費端 null 檢查。理由：契約保證應在生產端（對齊 memory「修復優先生產端」），消費端可無條件解構，IMP-C storage 寫入不需額外防禦。

#### 9. 語言特定實作注意事項（JavaScript / Chrome Extension）

- **CJS 模組**：`book-file-importer.js` 為 `require` 模組，不新增 import；`detectFormatVersion`/`convertV1ToV2Data` 已於 IMP-A 引入。
- **無型別系統**：ImportResult 為文件約定，靠 INV-1 + 測試保證；Phase 3b 在 JSDoc 中以 `{Object}` 加屬性說明描述回傳結構（或定義 `@typedef ImportResult`）。
- **`console.warn` 降級告警**：沿用檔案既有 `// eslint-disable-next-line no-console` 模式（見 `_checkLargeDataset` L495），避免 lint 失敗。
- **物件防禦**：`提取tag區段` 須對 `data` 為非物件（CSV 路徑傳入陣列、或 null）安全——先判 `data 為物件 非 null` 再讀屬性，避免 `null.tagCategories` 拋 TypeError。
