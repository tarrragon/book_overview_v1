#### 2. 三個待決策（產出 2）

##### 決策 A：Fixture DOM 對齊

**問題**：`tests/e2e/fixtures/readmoo-mock-page.html` 目前用 `.book-item` + `.book-title` + `.book-progress` + `data-book-id`，與真實 Content Script 提取邏輯不符。

**實機 selector 清單**（grep `src/content/adapters/readmoo-adapter.js` SELECTORS 確認）：

| 用途 | 真實 selector | fixture 現況 | 須修正 |
|------|--------------|-------------|--------|
| 書籍容器 | `.library-item`（主）；備用 `.book-item` / `.book-card` / `.library-book` | `.book-item` | 是 — 改為 `.library-item` |
| reader 連結（ID 來源） | `a[href*="/api/reader/"]` | 無 | 是 — 每本書補 `<a href="/api/reader/{id}">` |
| 封面圖 | `.cover-img`（`img` 元素） | `.book-cover`（純 div 文字） | 是 — 改為 `<img class="cover-img">` |
| 書名 | `.title` | `.book-title` | 是 — 改為 `.title` |
| 進度條 | `.progress-bar` | `.book-progress`（文字「已讀 45%」） | 是 — 補 `.progress-bar` 元素 |
| 書籍類型 | `.label.rendition` | 無 | 是 — 補 rendition label |
| privacy 元素 | `[id^="privacy-"]` | 無 | 選填 — privacyBookId 為輔助 ID 來源，可省略 |

**決策結論（A 已決策）**：Phase 3a 須**重寫 `readmoo-mock-page.html`**，使 DOM 結構對齊上表「真實 selector」欄。fixture 須含**正好 5 本書**（L1：與去重斷言「= 5」一致，非「至少 5 本」），每本明確指定 progress 值以驗證 readingStatus 自動判斷：

| 書序 | progress | 期望 readingStatus | 用途 |
|------|----------|-------------------|------|
| 書 1 | 0 | unread | 邊界：未讀 |
| 書 2 | 1 | reading | 邊界：剛開始（驗證 0<p 即 reading） |
| 書 3 | 45 | reading | 中段 |
| 書 4 | 99 | reading | 邊界：接近完成（驗證 p<100 仍 reading） |
| 書 5 | 100 | finished | 邊界：完成 |

**progress-bar DOM 寫法（M2 已寫死）**：`.progress-bar` 元素的百分比**必須以 inline `style="width:N%"` 表達**。`extractProgressFromContainer`（`readmoo-adapter.js:776`）只解析 inline style 的 `width:X%`，**不讀 `aria-valuenow`**。fixture 範例（Phase 3a 落地）：

```html
<div class="progress-bar" style="width:45%"></div>
```

不得使用 `aria-valuenow` 或文字「已讀 45%」表達進度——前者不被解析，後者是舊 fixture 寫法。

##### 決策 B：CI headed 模式方案

**問題**：Manifest V3 Extension 不支援 headless 載入（`extension-setup.js` 已設 `headless: false`）；CI 環境無顯示伺服器。

**現況確認**：專案**目前無 CI pipeline**（`.github/workflows/` 不存在）。

**決策結論（B 已決策）**：
- 本 ticket **不建立 CI pipeline**（超出範圍，且無 CI 為既成事實）。
- acceptance 第 1 條「可在 CI 以 npm script 執行」重新詮釋為：**提供可在 CI 環境執行的 npm script**，測試本身須以「CI-ready」方式設計：
  - 新增 npm script（建議 `test:e2e:browser`）執行新測試檔，以**路徑參數限定掃描 `tests/e2e/browser/`**（不影響既有 e2e script）。
  - 文件（Solution 或 README）註明 CI 整合方式：`xvfb-run -a npm run test:e2e:browser`。
- 實際 CI pipeline 建立另立 ticket（OQ-2）。

**環境變數行為澄清（L4）**：MV3 Extension 不支援 headless 載入，`extension-setup.js` 已硬編碼 `headless: false`。因此：

| 變數 | 對 browser 測試的行為 |
|------|---------------------|
| `HEADLESS` | **不適用**——MV3 限制下 headed 恆為必要，browser 測試不提供 headless 切換，不讀此變數 |
| `CI` | 可用於調整 timeout / 截圖路徑等周邊行為，但**不改變 headed 必要性** |

CI 環境執行 browser 測試的**唯一手段為 `xvfb-run`**（提供虛擬顯示伺服器，讓 headed Chrome 在無實體顯示的 CI 上運行）。不存在「CI 上改用 headless」的替代路徑。

#### 3. E2E 測試契約（產出 3）

定義「提取書物件符合 v1.1 model」的可驗證斷言規格。Phase 2 sage 據此設計 RED 測試。

##### 3.1 測試流程契約（UC-01 主流程）

| 步驟 | 動作 | 驗證點 |
|------|------|--------|
| S1 | `ExtensionTestSetup` 啟動 Puppeteer + 載入 `build/development` Extension | Extension ID 成功解析、SW 註冊 |
| S2 | 導航至 `file://` 協定的 `readmoo-mock-page.html` fixture | 頁面載入完成 |
| S3 | 觸發提取：開啟 popup（`chrome-extension://{id}/src/popup/popup.html`）→ 點擊 `#extractBtn` | popup 對 background 送出 `START_EXTRACTION` 訊息，提取流程啟動無 JS error |
| S4 | 等待提取完成、讀取 SW 端 `chrome.storage.local` 的 `readmoo_books` | storage 內書籍數 = fixture 書籍數（5） |
| S5 | 開啟 Overview 頁面 | Overview 顯示書籍清單，書名與 fixture 一致 |

**S3 觸發方式（M1 收斂）**：唯一方式為 popup 點擊 `#extractBtn`（`src/popup/popup-event-controller.js:216,268` 確認此 selector 與 click handler）。點擊後 controller 送出 `START_EXTRACTION` 訊息（`popup-event-controller.js:709`，`MESSAGE_TYPES.START_EXTRACTION`）。此方式覆蓋完整 UC-01 pipeline（popup → background → content script → storage），不採「注入訊息」捷徑——注入訊息會略過 popup 互動層，無法驗證 UC-01 的使用者操作入口。

**S4 storage 讀取契約（M4）**：

| 項目 | 規格 |
|------|------|
| storage key | `readmoo_books`（`event-coordinator.js:586` 確認 `chrome.storage.local.set({ readmoo_books: storageData })`） |
| 書籍陣列位置 | 巢狀 `.books`；`readmoo_books` 可能為 `{ books: [...], extractionCount: N }` 物件，亦可能為直接陣列（`tag-storage-adapter.js:129` 確認雙形態），讀取時須兩種皆容錯 |
| 書數來源 | `readmoo_books.books.length`（主）；`readmoo_books.extractionCount`（輔，`event-coordinator.js:580`） |
| Puppeteer 讀取手段 | `chrome.storage.local` 由 Service Worker 持有。Puppeteer 須取得 SW 的 Worker target（`extension-setup.js` 已偵測 `target.type() === 'service_worker'`），透過 `worker.evaluate(() => chrome.storage.local.get(['readmoo_books']))` 讀取。不可在 content/popup page context 讀（權限與時序不一致） |

**setup 前置失敗與 model 斷言分離（M6）**：S1–S3 屬「測試環境前置」（Extension 載入、SW 註冊、fixture 導航、提取觸發）。此階段失敗代表 setup 問題，**非 model regression**。Phase 2 sage 設計時須使前置失敗的錯誤訊息與 §3.2 model 斷言失敗在輸出上可區分（例如前置以獨立 `beforeAll` 或前置 expect 呈現，model 斷言集中於獨立 test case），避免 setup 故障被誤判為 model regression。

##### 3.2 書物件 model 斷言（核心，對應 acceptance 2）

對 storage 中每一本提取書物件斷言：

| 欄位 | 斷言規格 | 說明 |
|------|---------|------|
| `id` | 非空字串 | 穩定 ID（reader-{id} 等策略） |
| `title` | 非空字串，等於 fixture 對應書名 | |
| `readingStatus` | 必存在，值為 6 狀態 enum `{unread, reading, finished, queued, abandoned, reference}` 的合法成員 | v1.1 model 必填欄位 |
| `tagIds` | 必存在，型別為陣列（首次提取為 `[]`） | v1.1 model 欄位 |
| `progress` | 數字，0–100 | |

**readingStatus 斷言層次（M5 澄清）**：readingStatus enum 共 6 狀態，但**首次提取只能自動 derive 3 種**（`unread` / `reading` / `finished`，由 progress 經 `BookSchemaV2.mapV1StatusToV2` 推導）。`queued` / `abandoned` / `reference` 為使用者手動設定狀態（v0.20.x Tag 管理功能範圍），首次提取流程不會也無法產生。因此斷言分兩層：

1. **enum 合法性斷言**：每本書 `readingStatus` 是 6 狀態 enum 的合法成員（防止出現 undefined / 空值 / 非法字串）。
2. **自動 derive 正確性斷言**：fixture 由 progress 驅動，derive 結果預期只出現 3 種；fixture 不需也無法覆蓋手動狀態。

**readingStatus 自動 derive 正確性斷言**（依 progress→status 規則）：

| fixture 書籍 progress | 期望 readingStatus |
|----------------------|-------------------|
| 0 | `unread` |
| 0 < p < 100 | `reading` |
| 100 | `finished` |

> 斷言設計遵循 `test-assertion-design-rules.md`：使用結構/列舉/精確值斷言，**禁止計時硬門檻**（`toBeLessThan(Nms)`）。若需驗證 pipeline 穩定性，用「零 JS error」「書籍數正確」等確定性斷言，不用執行時間。Puppeteer 等待用 `waitForSelector` / `waitForFunction`，不用固定 sleep。

##### 3.3 去重斷言（UC-02 部分納入）

| 步驟 | 斷言 |
|------|------|
| 同一 fixture 連續觸發提取 2 次 | 第 2 次後 storage 書籍數仍 = 5（無重複記錄） |
| 同上 | 第 2 次後每本書 `id` 唯一：`new Set(ids).size === ids.length` |

**斷言設計（L3）**：單純 `count === 5` 無法區分「正確去重」與「整批覆寫」兩種情況——若每次提取整批覆寫舊資料，count 也會維持 5 但非去重行為。故須額外斷言 id 唯一性（`new Set(ids).size === ids.length`），確認 storage 中無重複 id 記錄。禁止用計時比較（`secondRunTime < firstRunTime` 等）。

##### 3.4 顯示斷言（UC-06 部分納入）

| 斷言 | 驗證點 |
|------|--------|
| Overview 頁面 DOM 含 5 本書的書名 | 透過 `page.$$eval` 取書名清單，與 fixture 比對 |

**Overview 資料載入路徑與時序（L5）**：

| 項目 | 規格 |
|------|------|
| Overview URL | `chrome-extension://{id}/src/overview/overview.html`（Phase 3a 須對照實際 overview 入口檔名確認，若非此名以實際為準） |
| 資料來源 | Overview 開啟後從 `chrome.storage.local` 的 `readmoo_books` 讀取並渲染（與 S4 同一 storage key） |
| 書名 selector | Phase 3a 須 grep `src/overview/` 確認書籍清單 item 的書名 selector（暫記為待查；不可猜測） |
| 時序處理 | Overview 渲染為非同步，斷言前須以 `page.waitForSelector(書籍清單容器 selector)` 等待渲染完成，禁止固定 sleep |

##### 3.5 測試定位：regression 鎖定 + 一次性 RED 驗證（B1 重新定位）

##### 3.6 model 轉換 pipeline 現況（B1/M3/L2 更正）

#### 2. 測試結構分層（M6 物理分離 — 關鍵）

setup 前置失敗必須與 model 斷言失敗在輸出上可區分，確保一次性 RED 驗證時 RED 能明確歸因 model gap：

| 層 | 實作位置 | 失敗語意 | 失敗訊息前綴建議 |
|----|---------|---------|----------------|
| L-A setup 前置 | `beforeAll` / `beforeEach` | 環境問題（Extension 載入失敗、SW 未註冊、fixture 導航失敗、提取觸發無回應） | `[SETUP]` |
| L-B model 斷言 | 獨立 `test()` case（G2/G3） | model regression（readingStatus/tagIds 缺失） | `[MODEL]` |
| L-C 行為斷言 | 獨立 `test()` case（G4/G5） | 去重 / 顯示行為 regression | `[DEDUP]` / `[DISPLAY]` |

判定點：一次性 RED 驗證時，若失敗訊息含 `[SETUP]` → setup 故障須先排除；若含 `[MODEL]` → 確認為 model gap RED（符合預期）。

#### 3. U2：helpers/storage-reader.js 策略

**職責**：封裝「取得 SW worker target → 讀 `chrome.storage.local['readmoo_books']` → 雙形態容錯解析書籍陣列」。

**虛擬碼**：

```
FUNCTION readBooksFromStorage(browser):
  // 取得 Service Worker worker context
  swTarget ← browser.waitForTarget(t => t.type()==='service_worker'
                                     AND t.url().startsWith('chrome-extension://'),
                                     { timeout: 10000 })
  worker ← await swTarget.worker()

  // 在 SW context 內讀 storage（content/popup page 無此權限與時序）
  raw ← worker.evaluate(() =>
          new Promise(resolve =>
            chrome.storage.local.get(['readmoo_books'], r => resolve(r.readmoo_books))))

  // 雙形態容錯：物件含 .books / 直接陣列
  IF raw IS NULL OR undefined: RETURN []
  IF Array.isArray(raw): RETURN raw
  IF Array.isArray(raw.books): RETURN raw.books
  RETURN []

FUNCTION waitForBooksStable(browser, expectedCount, timeout):
  // 用 polling 等待書數穩定，禁止固定 sleep
  POLL every 500ms UNTIL timeout:
    books ← readBooksFromStorage(browser)
    IF books.length === expectedCount: RETURN books
  THROW '[SETUP] storage 未在時限內達 expectedCount 本'
```

**約束**：SW worker.evaluate 內只能用 Chrome API 與純 JS（無 Node 模組）；callback 形式的 `chrome.storage.local.get` 須包成 Promise。

#### 4. U3：helpers/extraction-flow.js 策略

**職責**：封裝完整提取前置流程，供 G2/G4/G5 共用。每個 test case 重啟 browser，helper 不持有 mutable 狀態。

**虛擬碼**：

```
FUNCTION runExtraction(setup):
  // 前置：setup 已 setup() 完成（Extension 載入、page 建立）
  // 1. 導航 fixture
  setup.navigateToReadmoo()              // 內部用 getTestReadmooUrl() → file:// fixture
  ASSERT page 含 .library-item × 5  ELSE THROW '[SETUP] fixture 書數異常'

  // 2. 開啟 popup
  popupPage ← setup.openExtensionPopup() // chrome-extension://{id}/src/popup/popup.html
  popupPage.waitForSelector('#extractBtn', { timeout })
  ASSERT #extractBtn 未 disabled  ELSE THROW '[SETUP] extractBtn 不可用'

  // 3. 掛 console error 監聽（page + SW 兩端）
  errors ← []
  page.on('console', m => IF m.type()==='error': errors.push(m.text()))
  popupPage.on('console', ...) 同上

  // 4. 觸發提取
  popupPage.click('#extractBtn')         // 送出 START_EXTRACTION 訊息

  // 5. 等待提取完成（用 storage-reader.waitForBooksStable，禁止 sleep）
  books ← waitForBooksStable(setup.browser, 5, timeout)

  RETURN { books, jsErrors: errors }
```

**約束**：失敗於導航/popup/按鈕階段一律以 `[SETUP]` 前綴拋錯（M6 物理分離）；提取後 storage 斷言失敗才屬 `[MODEL]` / pipeline regression。
