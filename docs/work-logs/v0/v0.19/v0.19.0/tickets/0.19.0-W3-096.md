### Phase 1 規格

### 1. trigger_bound frontmatter 欄位

| 欄位 | 型別 | 預設 | 用途 |
|------|------|------|------|
| `trigger_bound` | boolean | false（缺欄位視為 false） | 標記此 ticket 受外部 trigger 條件限制 |

**判定準則**：ticket when 含「Trigger:」前綴明確列出 trigger 條件，且該條件無法由 PM 在 claim 當下立即驗證（需依賴外部事件或長期觀察）。

向後相容：缺欄位等同 false，既有 67 張 ticket 不受影響。

### 2. dashboard 顯示策略

**Text 格式**：在 `[ready]` 標籤後插入 `[T]` 標籤（trigger-bound）

格式對照：

| 狀態 | 渲染 |
|------|------|
| 一般 ticket | `[N] [P3] [ready] {id}  {title}` |
| trigger-bound ticket | `[N] [P2] [ready] [T] {id}  {title}` |

選用 `[T]` 而非 `[trigger-bound]`：對齊既有 `[N]` `[P2]` `[ready]` 的單字元/縮寫慣例，視覺對齊整齊。

**JSON 格式**：ready 陣列每個物件加 `trigger_bound: bool` 欄位（hook/自動化消費）

### 3. 排序策略

`load_top_ready` sort key 變更：

```python
# Before (track_dashboard.py:99)
candidates.sort(key=lambda pair: (_priority_rank(pair[0]), str(pair[0].get(\"id\") or \"\")))

# After
candidates.sort(key=lambda pair: (
    _priority_rank(pair[0]),
    1 if pair[0].get(\"trigger_bound\") else 0,  # 同 priority 內 trigger-bound 排後段
    str(pair[0].get(\"id\") or \"\")
))
```

效果（以當前 dashboard 為例）：

| Before | After |
|--------|-------|
| [1] [P2] [ready] W3-094 (trigger-bound P2) | [1] [P3] [ready] W4-005 (一般 P3) |
| [2] [P3] [ready] W3-021 (trigger-bound P3) | [2] [P3] [ready] W3-096 (一般 P3, 但本 ticket 已 claim) |
| [3] [P3] [ready] W3-096 (一般 P3) | [3] [P2] [ready] [T] W3-094 (trigger-bound P2, 排到 P3 後是因為同 priority 才比 trigger_bound) |

**注意**：跨 priority 仍依 priority 排序——P2 trigger-bound 不會被 P3 一般 ticket 跨越。原因是 sort key 第一項仍是 `_priority_rank`。

**選擇方案 B 不選方案 A 理由**：完全過濾會造成 trigger 成立時系統性遺忘，違反 quality-baseline 規則 5「所有發現必須追蹤」。
