## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

TC-04 在 `tests/unit/overview/book-file-importer-import-result.test.js:125-145` 採用「兩次呼叫 `convertV1ToV2Data` 後比對輸出」反模式：

```javascript
const result = importer.extractBooksFromData(data, 'json')  // 內部呼 convertV1ToV2Data → timestamp_A
const converted = convertV1ToV2Data(data)                   // 外部再呼 convertV1ToV2Data → timestamp_B
expect(result.tagCategories).toEqual(converted.tagCategories)
expect(result.tags).toEqual(converted.tags)
```

`src/export/v1-to-v2-converter.js` 兩處關鍵點：
- line 74: `const ts = timestamp || new Date().toISOString()` — 每次呼叫取新 ISO timestamp
- line 75: `const tsMs = Date.now()` — 每次取新 ms timestamp，組成 tag id 字串 `'tag_' + tsMs + '-' + ...`

兩次呼叫間隔通常 sub-millisecond → timestamps 相同 → pass。但 GC / scheduler 切換造成 > 1ms 間隔時，timestamps / tag id 不同 → `toEqual` 比較 `tagCategories.createdAt/updatedAt` 與 `tags.id/createdAt/updatedAt` 失敗。

baseline 重現實證（worktree post-symlink，10 跑）：fail 1/10（10% rate）。觀察到的 diff 範例：

```
- createdAt: "2026-05-24T09:56:41.077Z"
+ createdAt: "2026-05-24T09:56:41.076Z"
```

ms 差異 1，符合 PM root cause 定位（baseline 40% fail 估算為高負載 CI 機器；本次 10% 為開發機 + 單檔測試低負載結果，race 性質同源）。

### 影響範圍

- 直接：`tests/unit/overview/book-file-importer-import-result.test.js` TC-04 case
- 間接：`npm test` 全套件穩定性（違反 `.claude/rules/core/test-assertion-design-rules.md` 規則 1）
- 不修改：`src/export/v1-to-v2-converter.js`（行為正確，問題在測試斷言設計）

### 相關 Error Pattern

對齊 `.claude/rules/core/test-assertion-design-rules.md` 規則 1（npm test 主套件禁止計時依賴的 pass-fail 斷言）+ 規則 4（驗證一致性禁用兩次取樣 + 比對的反模式）。

## Solution

### 採用方案

**方案 A：inline expected values + 移除外部 `convertV1ToV2Data` 呼叫**（PM 推薦方案）

### Root Cause 確認

PM 定位 100% 命中：
- baseline 10 跑實證 fail 1/10，diff 顯示 `createdAt: "2026-05-24T09:56:41.077Z"` vs `2026-05-24T09:56:41.076Z`（ms 差 1）
- 與 `src/export/v1-to-v2-converter.js:74-75` `Date.now()` / `new Date().toISOString()` 每次取值的行為一致
- `convertV1ToV2Data` 兩次呼叫造成 timestamps_A != timestamps_B 是唯一根因，無其他次要因素

### 變更摘要

| 檔案 | 變更 | 行數 |
|------|------|------|
| `tests/unit/overview/book-file-importer-import-result.test.js` | 移除 `convertV1ToV2Data` import；TC-04 改用 inline 結構斷言（`toMatchObject` + `stringMatching` 容許動態 timestamp 與 tag id） | -1 import / TC-04 重寫（+30 行 +14 行說明註解 / -3 行原斷言） |
| `src/export/v1-to-v2-converter.js` | 不修改 | - |

### 設計決策

1. **不修改 src**：converter 行為本身正確（每次匯入產生新 timestamp 是合理業務語意），race 屬於測試斷言設計問題
2. **保留語意覆蓋**：原測試核心語意「v1 含 category 確實轉換為 tag 結構」由以下 inline 斷言守住：
   - `tagCategories[0]` 含 `id: 'cat_imported'` + `name: '匯入分類'`（IMPORT_CATEGORY_ID / IMPORT_CATEGORY_NAME 常數契約）
   - `tags[0]` 含 `name: 'V1匯入分類'` + `categoryId: 'cat_imported'`（v1 category → v2 tag 轉換契約）
   - `tags[0].id` matches `/^tag_\d+-000$/`（tag id 格式契約）
3. **timestamp 欄位用 regex `/^\d{4}-\d{2}-\d{2}T/`** 容許動態 ISO 字串，不再硬比對具體值
4. **未選 B**（`toMatchObject` 過濾 timestamp）：仍需引入 converter 二次呼叫產生 expected baseline，未根除 race
5. **未選 C**（mock Date.now）：影響其他 TC（TC-11/12/13 內部也呼 convertV1ToV2Data），全測試上下文 mock 風險過高
6. **未選 D**（converter 接 timestamp 注入）：改 src API 過度設計，影響範圍最廣
