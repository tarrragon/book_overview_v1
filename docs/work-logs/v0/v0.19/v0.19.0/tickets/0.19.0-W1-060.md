### Root Cause（PM 已定位）

TC-04 line 125-144 `tests/unit/overview/book-file-importer-import-result.test.js`：

```javascript
const result = importer.extractBooksFromData(data, 'json')  // 內部呼 convertV1ToV2Data 取 timestamp_A
const converted = convertV1ToV2Data(data)                   // 外部再呼 convertV1ToV2Data 取 timestamp_B
expect(result.tagCategories).toEqual(converted.tagCategories)  // 若 ts_A != ts_B 則 fail
expect(result.tags).toEqual(converted.tags)
```

`src/export/v1-to-v2-converter.js`：
- line 74: `const tsMs = Date.now()` ← 每次呼叫取新 ms timestamp
- line 115: `const timestamp = new Date().toISOString()` ← 每次取新 ISO timestamp

兩次呼叫間隔通常 sub-millisecond（timestamps 相同 → pass），但若 GC / scheduler 切換造成 > 1ms 間隔，timestamps 不同 → `toEqual` 比較 tagCategories/tags 含 timestamp 欄位 fail。**baseline 40% fail rate** 與此 race 一致。

對齊 `.claude/rules/core/test-assertion-design-rules.md` 規則 1（npm test 主套件禁止計時依賴的 pass-fail 斷言）—— TC-04 屬於 timestamp 直接比較的反模式。

### 修復選項（agent 自選）

| 方案 | 做法 | 利弊 |
|------|------|------|
| A. inline expected | 移除外部 `convertV1ToV2Data(data)` 呼叫；直接斷言 `result.tagCategories[0].name === 'V1匯入分類'` 等具體欄位（不比 timestamp） | 最低侵入；明確語意；但需手寫期望值 |
| B. 比較時忽略 timestamp | 用 `toMatchObject` 或自訂 helper 過濾 timestamp 欄位後再 `toEqual` | 保持「比較 converter 一致性」語意；helper 抽象成本 |
| C. mock 時間 | `jest.spyOn(Date, 'now').mockReturnValue(固定值)` + mock `new Date().toISOString()` | 全測試上下文 mock 風險（影響 TC-05 / 其他） |
| D. converter 接受 timestamp 注入 | `convertV1ToV2Data(data, { timestamp })` 讓 caller 傳入；兩次呼叫傳同值 | 改 src API；影響範圍最廣；過度設計 |

**推薦方案 A**（最低侵入 + 明確語意）。agent 若選 B/C/D 請在 Solution 說明取捨。
