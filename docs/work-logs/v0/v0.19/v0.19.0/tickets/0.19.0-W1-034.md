### 問題根因

`tests/unit/overview/overview-import-function.test.js` 的「應該處理大型資料集的效能」test case 使用 `Date.now()` 差值（`startTime` / `endTime`）計算載入耗時，並以 `expect(endTime - startTime).toBeLessThan(3000)` 作為 pass-fail 斷言。`Date.now()` 差值是真實計時，受全套件並行負載、GC、JIT 暖機影響，屬非確定性數值。隔離執行通過（26/26），全套件負載下 flaky 失敗。

### 影響範圍

單一測試檔 `tests/unit/overview/overview-import-function.test.js` 的單一 test case。屬 `test-assertion-design-rules` 規則 1 反模式（絕對計時門檻當 pass-fail）。斷言早於 source ticket W1-031（git blame：2025-08-23 commit 1627fcd0），非 W1-031 引入。違反 `quality-baseline` 規則 1（測試通過率 100%）。

`test-assertion-design-rules` 規則 1（npm test 主套件禁止絕對計時門檻當 pass-fail 斷言）。同型修復：source ticket 0.19.0-W1-031（event-system-integration 絕對記憶體門檻）、0.19.0-W1-017 / W1-018（計時斷言全專案盤點）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

依 `test-assertion-design-rules` 規則 1 Action 表，此 case 屬「計時門檻混在功能測試檔」（含 `toHaveLength(5000)`、id 比對、tags 陣列檢查等功能斷言），採「只移除計時行，功能斷言保留原位」，不整體移至 `tests/perf/`。

變更內容（`tests/unit/overview/overview-import-function.test.js`）：

- 移除計時量測：刪除 `const startTime = Date.now()`、`const endTime = Date.now()` 及對應 `// eslint-disable-next-line` 註解。
- 移除絕對計時門檻斷言：刪除 `expect(endTime - startTime).toBeLessThan(3000)`。
- 保留資料完整性驗證：`toHaveLength(5000)`、`currentBooks[0].id`、`currentBooks[4999].id`、`tags` 陣列型別檢查全數保留。
- test 名稱由「應該處理大型資料集的效能」改為「應該正確載入大型資料集」，使名稱與移除計時後的實際斷言意圖一致。
- 註解同步調整（「執行檔案載入並測量時間」改為「執行檔案載入」、「驗證效能要求」改為「驗證大型資料集正確載入與資料完整性」）。
