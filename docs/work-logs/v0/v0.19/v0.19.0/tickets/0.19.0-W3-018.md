## Problem Analysis
### 問題根因

W14-016 resume 流程暴露兩個 UX 缺口，源自 handoff/resume 設計早期只考慮 context-refresh 場景：

**缺口 1：GC 對任務鏈 handoff 的清理不足**

`is_handoff_stale`（handoff_utils.py:292-348）判定規則：
- 任務鏈方向（to-sibling/to-parent/to-child）：僅在目標 ticket 已 in_progress/completed 時標為 stale
- 非任務鏈方向：來源 ticket completed 即 stale

問題：PM 手動處理完 to-sibling 交接後（例如直接 claim 目標 ticket），若目標 ticket 尚未轉為 in_progress，handoff JSON 永不被 gc 清理。PM 需要手動 `ticket handoff gc --execute` 或刪檔。

**缺口 2：resume 按目標 ticket 查找的部分失效**

`_find_handoff_file`（resume.py:80-128）支援三層查找：
1. 直接匹配檔名 `{ticket_id}.json`（來源 ticket）
2. direction 後綴反向匹配 `extract_direction_target_id(direction) == ticket_id`
3. `ticket_id` 欄位匹配

問題場景：
- `handoff --next TARGET --from-ticket-id SOURCE` 產出的 JSON：direction="context-refresh"（無後綴），target 存於 `target_ticket_id` 頂層欄位（handoff.py:1564）。resume 的反向匹配只檢查 direction 後綴，不檢查 `target_ticket_id` 欄位，導致 `resume TARGET` 找不到。
- `handoff --auto --direction to-sibling:TARGET --from-ticket-id SOURCE`：direction 後綴含 target，反向匹配可成功。但 `target_ticket_id` 頂層欄位（handoff.py:1478）未被 resume 讀取，語意不一致。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `.claude/skills/ticket/ticket_system/commands/resume.py` | `_find_handoff_file` 反向匹配需擴充讀取 `target_ticket_id` 欄位 |
| `.claude/skills/ticket/ticket_system/lib/handoff_utils.py` | `is_handoff_stale` 可能需新增「已 resume 但未歸檔」清理路徑 |
| `.claude/skills/ticket/ticket_system/commands/handoff_gc.py` | 呼叫端，若 stale 判定邏輯變更則自動受益 |

## 重現實驗結果
### 實驗方法

靜態程式碼分析：追蹤 handoff --next / --auto 產出的 JSON 結構與 resume _find_handoff_file 的查找路徑。

### 實驗執行

1. handoff --next TARGET --from-ticket-id SOURCE 產出 JSON：
   - direction: "context-refresh"（無 :TARGET 後綴）
   - target_ticket_id: TARGET（頂層欄位，handoff.py:1567）

2. resume TARGET 查找路徑：
   - 直接匹配 `TARGET.json`：不存在（檔名是 SOURCE.json）
   - direction 反向匹配：direction="context-refresh"，extract_direction_target_id 回傳 None
   - ticket_id 欄位匹配：data["ticket_id"]=SOURCE != TARGET
   - 結果：回傳 None → resume 失敗，exit 1

3. handoff --auto --direction to-sibling:TARGET --from-ticket-id SOURCE：
   - direction: "to-sibling:TARGET"
   - target_ticket_id: TARGET
   - resume TARGET 的 direction 反向匹配：extract_direction_target_id("to-sibling:TARGET") == TARGET → 成功

4. gc 清理：task-chain handoff 在目標 ticket 未 in_progress 時不被清理，正確但缺少 PM 手動強制 gc 的機制。

### 實驗發現

已驗證事實：
- `--next` 模式的 handoff 無法透過 `resume TARGET` 查找（缺口確認）
- `--auto` to-sibling/to-child 模式的 handoff 可透過 direction 後綴反向匹配（部分正常）
- gc 對 task-chain handoff 的判斷邏輯正確（設計意圖是保留未啟動的任務鏈交接），但缺少 PM 手動強制清理選項
- `target_ticket_id` 頂層欄位在 resume 端完全未被讀取

## Solution
### 方案設計

### 修復 1：resume 擴充 target_ticket_id 查找（P1）

在 `_find_handoff_file` 的 fallback 迴圈（resume.py:110-127）新增一層匹配：

```python
# 新增：target_ticket_id 頂層欄位匹配（支援 --next 模式）
target_tid = data.get("target_ticket_id")
if target_tid and target_tid == ticket_id:
    return (json_candidate, "json")
```

位置：在 direction 反向匹配之後、ticket_id 欄位匹配之前。

影響範圍：僅 resume.py `_find_handoff_file`，不影響 gc 或 handoff 建立邏輯。

### 修復 2：gc 新增 --force 旗標（P2）

新增 `ticket handoff gc --force --execute` 選項，忽略 task-chain 保護，清理所有來源 ticket 已 completed 的 handoff（含 to-sibling/to-parent/to-child）。

實作方式：在 `_collect_stale_handoffs` 新增 `force` 參數，force=True 時跳過 `is_task_chain_direction` 保護分支，統一以「來源 ticket 已 completed」為 stale 判定。

影響範圍：handoff_gc.py + handoff.py register 新增 CLI 參數。`is_handoff_stale` 本體不變（保持預設保守行為）。

### 不做的事

- 不修改 `is_handoff_stale` 核心邏輯：現有的保守 stale 判定（任務鏈保留）是正確設計，--force 作為 opt-in 覆蓋即可。
- 不修改 handoff JSON schema：`target_ticket_id` 欄位已存在，resume 端讀取即可。

| 類型 | 標題 | 優先級 | 說明 |
|------|------|--------|------|
| IMP | resume 擴充 target_ticket_id 反向查找 | P1 | 修復 _find_handoff_file + 測試 |
| IMP | gc 新增 --force 旗標 | P2 | handoff_gc.py + CLI 參數 + 測試 |
