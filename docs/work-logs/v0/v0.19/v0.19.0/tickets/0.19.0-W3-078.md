### 洞察 1：不對稱設計原則

**核心主張**：雙向操作（sync-push/pull, lock/unlock, create/destroy 等）的防呆設計不該因「對稱性誘惑」而強制等價；應依「副作用嚴重度」與「可逆性」差異化設計。

**Why**：人類大腦傾向尋找對稱以降低認知負擔，但工程系統的對稱性不是設計目標而是描述工具。雙向操作實際副作用嚴重度往往不對稱：寫公開狀態 vs 本地操作、永久污染歷史 vs 可備份還原、影響下游 vs 僅影響自己。

**Consequence**：
- 過度防護輕量端：sync-pull 加 hash early-exit 增加複雜度與誤判風險（state 過時可能漏 pull 真實更新）
- 欠保護重型端：只做 sync-pull 防呆但 sync-push 仍會空 commit 進公開歷史

**案例來源**（W3-075 + W3-076 連續設計）：

| 評估維度 | sync-push（W3-075） | sync-pull（W3-076） |
|---------|-------------------|-------------------|
| 主要副作用 | 推 commit 到公開 repo（不可逆） | 覆寫本地 .claude/（已有 backup） |
| 可逆性 | 低（git 歷史永久） | 高（_sync_with_backup 自動建立） |
| 影響下游 | 是 | 無 |
| 適合方案 | hash + commit 雙重訊號 early-exit | 順手清理累積 backup_dir |

**判別流程**：
1. 列出兩端副作用
2. 評估可逆性差異（一端比另一端低 ≥ 2 個數量級即不對稱）
3. 不對稱 → 方案差異化

**反模式信號**：對稱方案的理由若是「另一端做了所以這端也應該做」、「為了完整性」、「邊界一致」，必先回答副作用嚴重度差異問題。

**歸屬候選評估**：

| 候選 | 適用性 | 評估 |
|------|-------|------|
| rule | 中 | 屬「設計判斷原則」不是「禁令」 |
| methodology | 高 | 跨類型可重用（hook/CLI/API 設計皆適用） |
| skill（design-decision-framework 補充） | 高 | 該 skill 已負責多方案評估 |
| memory | 低 | 跨 session 適用性高，限 memory 違反 PC-160 |

**初步建議**：升級到 design-decision-framework skill（補充「對稱性誘惑檢查」段落）。

### 洞察 2：Hook Dogfooding 設計即驗證

**核心主張**：設計診斷類 / 檢查類 hook 時，完成實作後立即用「當前專案」執行一次（dogfooding），通常能立即暴露 hook 預期偵測類型的既有殘留問題。

**Why**：診斷 hook 本職是「偵測某類問題」，專案在 hook 不存在的歷史時期已可能累積該類問題。Hook 第一次跑是專案歷史的一次體檢，立即驗證有雙重價值：(1) 確認 hook 功能性 (2) 自動發現殘留 case。

**Consequence**：
- 不知道 hook 是否真能偵測（unit test 過但 trigger 條件可能與設計不同）
- 殘留問題被新 hook 證實存在卻未處理 → 每次 session-start 反覆 WARN 累積雜訊

**案例來源**（W3-077 session-start-gitignore-check-hook）：
- 第一次跑：抓到 `.claude/PM_INTERVENTION_REQUIRED` 被 tracked（2026-03-30 起）
- 處理：同 session `git rm --cached` + commit 4fbaca9f
- 結果：Hook 設計目的閉環完成

**判別何時 dogfooding**：

| Hook 類型 | 是否強制 dogfooding |
|---------|------------------|
| 診斷類（偵測既有問題） | 是 |
| 強制類（阻擋未來操作） | 否 |
| 提示類（提醒下一步） | 否 |

**反模式**：完成 hook 後只跑 pytest（綠燈通過）就 close，缺對「當前真實環境」的調用。Pytest 用 mock state 無法暴露專案歷史累積。

**歸屬候選評估**：

| 候選 | 適用性 | 評估 |
|------|-------|------|
| rule | 中 | 屬「開發工法」不是普世強制 |
| methodology | 中 | 可寫但較零散 |
| skill（basil-hook-architect 補充） | 高 | basil 負責 hook 設計，補充 dogfooding 步驟自然 |
| memory | 低 | 同洞察 1 |

**初步建議**：升級到 basil-hook-architect agent definition「適用情境」或新增「實作完成檢查」段落。
