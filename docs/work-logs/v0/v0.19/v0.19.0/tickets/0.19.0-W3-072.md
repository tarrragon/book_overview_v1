### 問題根因

W3-071 session 中 PM 執行 `ticket track set-where 0.19.0-W3-071 --layer 'Framework Rules'` 失敗（unrecognized arguments: --layer）。根因：`set-where` argparse 註冊僅接受 positional `value` 參數（track.py L620-623），無任何 optional flag。`execute_set_field` 內部透過 `DICT_FIELD_SUBKEY` 映射表（fields.py L49-53）將 positional value 寫入 `where.layer`，但 CLI 層面不提供直接更新 `where.files` 或 `who.history` 等其他子欄位的路徑。

同類問題存在於三個 dict 欄位：

| dict 欄位 | 子欄位 | DICT_FIELD_SUBKEY 映射（CLI 可寫） | CLI 不可寫 |
|-----------|--------|----------------------------------|-----------|
| who | current, history | current | history |
| where | layer, files | layer | files |
| how | task_type, strategy | strategy | task_type |

### 影響範圍

- `.claude/skills/ticket/ticket_system/commands/fields.py` — DICT_FIELD_SUBKEY + execute_set_field
- `.claude/skills/ticket/ticket_system/commands/track.py` — argparse 子命令註冊（L600-635）
- `.claude/skills/ticket/SKILL.md` — 文件描述

無直接對應的 PC error-pattern。最近似者為 PC-040（context 應寫入 ticket 而非 prompt），但本案為 CLI 能力缺口而非流程違規。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 方案評估

### 子欄位更新需求頻率分析

判斷方案前先釐清各子欄位的更新頻率與使用者：

| 子欄位 | 更新者 | 更新時機 | 頻率 | 現行路徑 |
|--------|--------|---------|------|---------|
| who.current | PM/CLI | claim/派發 | 高（每 ticket） | set-who（已可用） |
| who.history | PM/CLI | 極少手動更新 | 極低 | Edit frontmatter |
| where.layer | PM | ticket 建立/修正 | 中 | set-where（已可用） |
| where.files | PM/create | ticket 建立/修正 | 中 | create 時自動填；後續 Edit |
| how.strategy | PM | ticket 建立/修正 | 中 | set-how（已可用） |
| how.task_type | PM | ticket 建立時 | 低（通常不變） | create 時填；極少修正 |

### 方案 A：補 --layer / --files / --task-type 子欄位 flag

**做法**：為 set-where 加 `--layer`、`--files`（nargs='+'）flag；為 set-how 加 `--task-type` flag；set-who 加 `--history` flag（或不加，因 history 極少手動寫）。positional value 改為 optional（向後不相容）或保留為 --layer 的 shorthand。

**優點**：
- 與 set-decision-tree 已有先例模式一致
- CLI 操作完整，PM 不需 Edit frontmatter
- 單一命令更新多個子欄位（如 `set-where ID --layer X --files a.py b.py`）

**缺點**：
- 向後不相容：現行 `set-where ID "value"` 的 positional value 語意為寫入 layer，加 --layer flag 後 positional 與 --layer 衝突，需決定遷移策略
- where.files 為 list 型態，需 nargs='+' 且需決定是「覆寫」還是「追加」語意（追加需額外 --add-file / --remove-file）
- 實作量中等：3 個命令 x 2-4 個 flag = 6-12 個新 flag 註冊 + execute_set_field 邏輯分支

**變體 A'**：保留 positional value 作為主子欄位 shorthand（向後相容），僅新增 optional flag 處理次要子欄位。positional 與 --layer 同時存在時以 --layer 為優先。

### 方案 B：子命令拆分（set-where-layer / set-where-files）

**做法**：為每個子欄位建獨立子命令。

**優點**：
- 語意最清晰，無 positional vs flag 混淆
- 每個子命令職責單一

**缺點**：
- 子命令爆炸：現有 6 個 set-* 命令可能膨脹為 12-15 個
- 違反現有 CLI 設計慣例（set-decision-tree 用 flag 不用子命令拆分）
- 對 PM 記憶負擔高

### 方案 C：維持現狀（PM 直接 Edit frontmatter）

**做法**：不改 CLI，PM 用 Edit 工具直接修改 ticket YAML frontmatter 中的子欄位。

**優點**：
- 零實作成本
- Edit 工具對 YAML 操作已成熟（PM 日常使用）
- who.history / how.task_type 更新頻率極低，CLI 支援的 ROI 低

**缺點**：
- where.files 更新為 list 操作，Edit YAML list 語法易錯（缺 `-` 前綴、縮排錯誤）
- 破壞 CLI-first 原則的一致性（部分子欄位可 CLI，部分需 Edit）
- 若未來 hook 依賴 CLI 寫入路徑做追蹤（如 audit），Edit 繞過 CLI 無法被追蹤

### 方案評估總結

| 維度 | A（補 flag） | A'（flag + 向後相容） | B（子命令拆分） | C（維持現狀） |
|------|-------------|---------------------|----------------|-------------|
| 實作成本 | 中 | 中 | 高 | 零 |
| 向後相容 | 否 | 是 | 是 | 是 |
| CLI 完整性 | 高 | 高 | 高 | 低 |
| 設計一致性 | 高（對齊 set-decision-tree） | 高 | 低（違反現有慣例） | 低 |
| PM 認知負擔 | 低 | 低 | 高 | 中（需記住何時用 CLI 何時用 Edit） |
| 高頻需求覆蓋 | 全覆蓋 | 全覆蓋 | 全覆蓋 | 部分覆蓋 |
| 低頻需求（history/task_type） | 覆蓋 | 覆蓋 | 覆蓋 | Edit 可達 |

### 證據與數據

1. **先例**：set-decision-tree（track.py L662-667）已使用 optional flag 模式更新 dict 子欄位，驗證此模式在本專案可行
2. **觸發場景**：W3-071 session PM 嘗試 --layer 失敗，為實際使用需求而非假設需求
3. **子欄位更新頻率**：where.layer 和 how.strategy 為中頻（每 ticket 可能更新），where.files 為中頻（create 後偶爾修正），who.history / how.task_type 為極低頻
4. **where.files 的 list 特殊性**：files 為 YAML list，CLI 操作語意需區分覆寫 vs 追加 vs 移除，複雜度高於 string 子欄位
