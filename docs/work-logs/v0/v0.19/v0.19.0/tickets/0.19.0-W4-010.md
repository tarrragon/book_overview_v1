| `'大量錯誤轉換應該保持效能'` | 396 | `expect(duration).toBeLessThan(50)` | latent flaky（目前過關，隨負載可能失敗） |

### 變更摘要

| 檔案 | 變更 |

| `tests/perf/uc04-error-adapter-performance.test.js` | 新增（遷移自原效能測試的批量轉換 case） |

### Test case 個別處理

**1. `'大量錯誤轉換應該保持效能'`（原 `toBeLessThan(50)`）**
純計時、無功能斷言。依 test-assertion-design-rules 規則 1/2 遷移至 `tests/perf/uc04-error-adapter-performance.test.js`，重命名為 `'大量錯誤轉換不應出現數量級退化'`，門檻放寬至 `toBeLessThan(10000)` 作為大幅退化防護（非效能 SLA）。新檔依規則 2 加上 `tests/perf/` 標準檔頭（量測環境限制、執行方式、來源說明）。

**2. `'映射表快取應該減少重複計算'`（原 `toBeLessThan(10)`，完整套件下實測 11ms 失敗）**
意圖為驗證快取生效。檢視 `UC04ErrorAdapter.getErrorMapping()`（src/core/errors/UC04ErrorAdapter.js:30-47）確認其為 memoized 單例：`if (!this._errorMapping)` 守衛 + `Object.freeze`，重複呼叫回傳同一參考。依 test-assertion-design-rules 規則 4 選項 C（`toBe` 參考比較），改寫為 `'多次存取映射表均回傳同一參考（快取不重建）'`：迴圈 1000 次斷言每次回傳 === 首次參考。若快取失效改為每次重建物件，`toBe` 必然失敗，與環境計時完全無關。放置於 `getErrorMapping` describe（與既有 `'應該快取映射表'` 同組，語意聚合）。

### 設計決策

- 規則 4 提供三個選項（命中率 / size / 參考比較）。`getErrorMapping()` 無命中率 API，選項 C（`toBe`）是最強驗證——直接證明「快取不重建物件」，且涵蓋原測試「減少重複計算」的意圖。
- 既有 `'應該快取映射表'` test 已做單次 `toBe` 比較；新增 test 補強「高頻多次呼叫」維度（原計時測試的 1000 次迴圈規模），兩者互補不重複。
- 效能回歸防護未直接捨棄：批量轉換 case 保留於 `tests/perf/`，以數量級門檻攔截災難性退化。
