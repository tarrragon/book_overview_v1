## Task Summary

評估 /resume bg sessions 新功能對既有 ticket handoff 機制（handoff-checker-hook、handoff resume 流程）的影響

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

Claude Code v2.1.144 引入 `/resume` 支援 background sessions（subagent 結束後 session 仍可被 `/resume` 恢復）。既有 ticket handoff 機制設計時假設：
1. subagent 結束 = session 終止，不可恢復
2. `/resume` 僅用於前台（interactive）session 的恢復
3. handoff JSON 的消費者是「下一個前台 session」

bg session 可 resume 的新行為可能改變上述假設，但需評估實際影響程度。

### 影響範圍

| 元件 | 路徑 | 潛在影響 |
|------|------|---------|
| commit-handoff-hook | `.claude/hooks/commit-handoff-hook.py` | 已有 is_subagent_environment guard，subagent 內不觸發 handoff 提醒 |
| session-start-scheduler-hint-hook | `.claude/hooks/session-start-scheduler-hint-hook.py` | SessionStart 顯示 runqueue，bg session resume 是否觸發 SessionStart 待驗證 |
| ticket handoff CLI | `.claude/skills/ticket/` | handoff JSON 結構與 session 類型無關 |
| ticket resume CLI | `.claude/skills/ticket/` | resume 從 pending JSON 讀取，不區分來源 session 類型 |
| session-switching-sop | `.claude/pm-rules/session-switching-sop.md` | SOP 假設 /clear 前做 handoff，bg session resume 不經此 SOP |

### 相關 Error Pattern

- PC-018: Resume 後必須檢查 5W1H
- PC-104: Handoff 提醒機制（commit-handoff-hook 的設計動機）

### 實驗方法

1. 靜態分析：盤點所有 handoff 相關 hook 與 CLI 的程式碼，檢查是否有 session 類型（前台/背景）區分邏輯
2. 行為推演：基於 Claude Code v2.1.144 changelog 描述，推演 /resume bg session 時各 hook 的觸發行為
3. 交叉比對：對照 handoff JSON 結構、GC 邏輯、session-start hook 的觸發條件

### 實驗執行

**步驟 1：盤點 hook 的 session 類型感知能力**

掃描所有 handoff 相關 hook（commit-handoff-hook、session-start-scheduler-hint-hook、stop-worklog-handoff-sync-check-hook）：
- `commit-handoff-hook.py`：有 `is_subagent_environment` guard（L306），subagent 內跳過。不區分「前台 session」與「resume 的 bg session」。
- 其餘 hook：無 `is_background` / `bg_session` 相關判斷。hooks 不感知 session 是新建還是 resume。

**步驟 2：推演 /resume bg session 行為**

Claude Code `/resume` 恢復 bg session 時：
- 該 session 恢復為前台 interactive session
- SessionStart event 是否觸發：**不觸發**（resume 不是新 session，是既有 session 的 reattach）
- Stop event：session 結束時正常觸發
- PostToolUse event：正常觸發（含 commit-handoff-hook）

**步驟 3：handoff JSON 結構檢查**

handoff pending JSON 結構（target_ticket_id / direction / auto_generated）與 session 類型完全無關。JSON 是檔案系統持久化，不綁定 session lifecycle。

### 實驗發現

**已驗證事實**：
1. 既有 hook 不感知 session 是 fresh start 還是 /resume（無 session 類型判斷碼）
2. handoff JSON 結構是 session-agnostic，不受 bg session 影響
3. commit-handoff-hook 在 subagent 內已有 guard，不會因 bg session resume 誤觸發
4. `/resume` bg session 後，session 變為前台模式，所有 PostToolUse/Stop hook 正常運作

**仍未驗證的假設**：
- `/resume` bg session 是否觸發 SessionStart event（需實機測試，靜態分析無法確認）
- 若 SessionStart 不觸發，session-start-scheduler-hint-hook 的 runqueue 提示不會出現（可能是 feature 而非 bug）

**結論**：bg session resume 對既有 handoff 機制的**直接衝擊低**。主要風險點在 SessionStart event 是否觸發的不確定性，以及 session-switching-sop 未涵蓋 bg session resume 場景的文件缺口。

## WRAP 框架分析

### W — Widen：bg session 對 handoff 流程影響的所有可能性

| 影響面 | 描述 | 嚴重度 |
|--------|------|--------|
| 1. SessionStart hook 觸發不確定性 | /resume bg session 是否觸發 SessionStart event？若不觸發，runqueue 提示缺失 | warning |
| 2. handoff GC 條件 | handoff pending JSON 無 TTL/過期機制，bg session 長期存活不影響 GC（因為 GC 不存在） | info |
| 3. target_ticket_id 解析 | handoff JSON 的 target_ticket_id 是檔案系統持久化，與 session 類型無關，解析不受影響 | info |
| 4. session-switching-sop 文件缺口 | SOP 未涵蓋「bg session resume 後的 handoff 流程」場景 | warning |
| 5. subagent guard 正確性 | commit-handoff-hook 的 is_subagent_environment guard 在 bg session resume 後不適用（已變前台），行為正確 | info |

### R — Reality-Test：對照現有 error patterns

| Error Pattern | 與 bg session 的關聯 | 風險 |
|--------------|---------------------|------|
| PC-018（Resume 後檢查 5W1H） | bg session resume 同樣適用，但 PC-018 的觸發依賴人工自律，非 hook 強制 | 低 |
| PC-104（Handoff 提醒） | commit-handoff-hook 在 resume 後的前台 session 正常觸發，無缺口 | 無 |

### A — Attain Distance：衝擊評估

**整體衝擊：低**

核心原因：既有 handoff 機制的設計恰好是 session-agnostic 的：
1. handoff JSON 是檔案系統持久化，不綁 session
2. resume CLI 讀檔案、不讀 session metadata
3. commit-handoff-hook 的 subagent guard 在 resume 後自動解除（正確行為）
4. 唯一不確定點是 SessionStart event 是否觸發，但即使不觸發，用戶可手動 `ticket track runqueue`

**需要行動的項目**：
- 文件更新：session-switching-sop 補充 bg session resume 場景
- 實機驗證：確認 /resume bg session 是否觸發 SessionStart event

### P — Proof：失敗回退

若未來 Claude Code 改變 /resume 行為（如 resume 觸發不同 event type）：
- handoff JSON 結構不受影響（檔案系統層）
- hook 層可透過 event type 判斷新增分支，成本低
- 回退方案：無需回退，當前機制天然相容

## bg session 與既有 handoff 機制互動矩陣表

| 機制 | 新 session（前台） | /resume 前台 session | /resume bg session | subagent 內 |
|------|-------------------|---------------------|-------------------|------------|
| commit-handoff-hook 觸發 | 是 | 是 | 是（resume 後變前台） | 否（guard） |
| SessionStart hint 觸發 | 是 | 待驗證 | 待驗證 | N/A |
| Stop hook 觸發 | 是 | 是 | 是 | 是 |
| handoff JSON 可讀 | 是 | 是 | 是 | 是 |
| ticket resume CLI 可用 | 是 | 是 | 是 | N/A |
| session-switching-sop 涵蓋 | 是 | 是 | **否（文件缺口）** | N/A |

## 風險點識別

1. **SessionStart event 不確定性**（warning）：/resume bg session 是否觸發 SessionStart event 尚未實機驗證。若不觸發，用戶 resume bg session 後不會自動看到 runqueue 建議。建議建 IMP ticket 做實機測試並視結果決定是否需要補償邏輯。
2. **session-switching-sop 文件缺口**（warning）：SOP 未描述 bg session resume 後的 handoff 流程。建議建 DOC ticket 補充。
