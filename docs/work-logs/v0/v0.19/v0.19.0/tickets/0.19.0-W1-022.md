## Problem Analysis

### 問題根因

用戶有多個專案（網頁 npm test、Flutter app test、後端多語言 test），W1-017/W1-018/W1-020 確立的測試斷言設計判斷（確定性 vs 環境依賴、計時不當驗收、退化防護、浮點精度）目前以兩個檔案存在：

1. `.claude/rules/core/test-assertion-design-rules.md` — 4 條規則，JavaScript/Jest 專屬語言
2. `.claude/methodologies/test-assertion-design-methodology.md` — 決策樹 + 精度表 + 反模式，同樣 JavaScript 專屬

兩檔內容高度重疊（決策樹、精度表、反模式表幾乎相同），且綁定單一語言生態。跨專案 sync 時這兩個檔案會被帶到 Flutter/Go 專案但內容不適用，造成 context 浪費且誤導。

### 影響範圍

- .claude/rules/core/test-assertion-design-rules.md（W1-020 產出）
- .claude/methodologies/test-assertion-design-methodology.md（未追蹤既有檔）
- 未來所有使用 .claude/ sync 機制的專案

### 實驗方法

### 實驗執行

**1. 內容重疊度分析**

比對 test-assertion-design-rules.md（172 行）與 test-assertion-design-methodology.md（122 行）：

- 斷言分類決策樹：methodology 有完整決策樹，rules 規則 1/2 是決策樹的展開版
- 浮點精度表：兩檔幾乎相同（場景/精度/理由三欄）
- 反模式表：methodology 5 行反模式表涵蓋 rules 4 條規則的所有禁止行為
- 退化防護設計：兩檔內容一致（10x 門檻、標註要求）
- 結論：重疊率約 70%，methodology 是 rules 的濃縮版，rules 是 methodology 的展開+三明示版

**2. 跨語言適用性分析**

rules 檔中的專案特定元素：
- Jest API：toBeLessThan、toBeCloseTo(v, numDigits)、toBe
- 路徑：tests/perf/、tests/unit/、tests/integration/
- 指令：npm test、npm run test:perf
- 環境：jsdom、JIT、GC

methodology 檔中的專案特定元素：
- 指令：npm test、npm run test:perf
- 路徑：tests/perf/

抽象後的語言無關概念（適用 Jest/Flutter test/Go test/pytest）：
- 確定性 vs 環境依賴的二分法
- 功能測試與效能測試物理隔離
- 浮點精度上限（IEEE 754 通用）
- 快取驗證用狀態而非計時
- 退化防護門檻 >= 10x

**3. 既有 skill 定位比較**

- test-async-guardian：Flutter/Dart 專屬，處理異步資源清理。與斷言設計正交（資源管理 vs 斷言品質）。不適合併入。

### 用戶補充修正：分類軸與職責邊界（2026-05-21）

用戶對 skill 設計提出兩項根本修正，凌駕先前 saffron ANA 與三視角審查中的 references 結構規劃。

**修正 1：分類軸為專案類型，而非語言**

- saffron 原設計 references/ 按語言分（js-jest、flutter、go）為錯誤的分類維度。
- 應按專案類型分：WEB 類、APP 類、後端類。
- 理由：語言不決定測試設計考量。決定考量的是專案類型的架構模式與運作邏輯：
  - WEB 類：渲染方式與運作邏輯
  - APP 類：widget 渲染與呼叫後端再處理的邏輯
  - 後端類：查詢再回應的邏輯與順序
- 三者效能瓶頸本質不同，無法用語言維度涵蓋（同一語言可橫跨多種專案類型，如 JavaScript 可用於 WEB 前端與 Node 後端，測試考量完全不同）。

**修正 2：SKILL 職責邊界為概念，不是方法**

- SKILL 處理的是一個概念——測試斷言設計的判斷理念（什麼該測、什麼不該當驗收、確定性對比環境依賴等抽象原則）。
- SKILL 不應定義實際的測試設計與調整重構方法——那些因專案類型架構而異，屬各專案自身範疇。
- references（若保留）只承載不同專案類型的測試考量脈絡差異，不規定測試該怎麼寫或怎麼重構。

**對下 session W1-022 重新評估的影響**：

- references/ 結構從語言適配改為專案類型考量脈絡（WEB / APP / 後端），或重新評估是否需要 references/ 這一層。
- SKILL.md 嚴格限於概念框架，剔除任何具體測試寫法或重構方法內容。
- 擴大範圍決策（補非同步、隔離、快照等）的對象是判斷概念，不是各專案類型的測試方法。
- 此修正同時化解三視角審查中 flutter/go 骨架的爭議——問題不在於骨架延後與否，而在於語言維度本身即錯誤分類軸。
