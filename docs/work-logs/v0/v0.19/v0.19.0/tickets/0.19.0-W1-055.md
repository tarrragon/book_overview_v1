## Problem Analysis
<!-- Schema[DOC/Problem Analysis]: 選填（若 DOC 起因於缺陷或盤點結論可填）（.claude/pm-rules/ticket-body-schema.md） -->
### PM 前置分析（2026-05-30）

### 問題根因

`.claude/hooks/post-test-hook.py` 的 `TEST_FAILURE_PATTERNS` (L127-132) 用單純字面 regex 匹配：

```python
TEST_FAILURE_PATTERNS = [
    (r"Expected:\s*(.+?)\s*Actual:", "斷言失敗: Expected {0}"),
    (r"(\d+)\s+tests?\s+failed", "{0} 個測試失敗"),
    (r"FAILED", "測試失敗"),
    (r"AssertionError", "斷言錯誤"),
]
```

任何 tool_response 含上述字面（包括綠燈測試 summary 中提及「N tests failed」其中 N=0）都會誤判。

### 影響範圍

- 主檔：`.claude/hooks/post-test-hook.py`
- 影響：所有 PostToolUse:Bash + PostToolUse:mcp__dart__run_tests 場景
- 連帶：誤判會寫入 `.claude/hook-logs/pre-fix-evaluation-*.json` + 觸發 PM/agent 額外 ticket 流程

### 既有豁免機制狀態

W3-041 / W3-066 已實作 `_is_ticket_body_write()` 早期 gate (L398)，覆蓋：

| 條件 | 處理 |
|------|------|
| Bash command 含 `ticket track append-log` / `ticket create` 等 | 跳過所有子邏輯 |
| file_path 匹配 ticket md 路徑 pattern | 跳過（防御性，目前 hook 僅 Bash 註冊） |

**仍誤判的原因**：豁免只覆蓋「ticket CLI 命令」的 tool_response，不覆蓋「實際跑 npm test 但結果含 `0 tests failed`」等 summary 字面。最新 log（2026-05-28-20:47）顯示 3 次 `FAILED` 字面命中，需追溯該次 Bash command 為何未被豁免。

### 真根因 vs ticket 描述差異

Ticket why 描述：「thyme append-log Test Results 含 0 failed 字面被誤判」——此情境理論上已被 W3-066 豁免擋下。

**待 hook-architect 驗證**：
1. 2026-05-28-20:47 log 是否真為 ticket body 寫入場景？若是→豁免未生效；若否→是真實測試 summary 含 `FAILED` 字面（如 Jest `Tests: 0 failed, N passed`）
2. Pattern `FAILED` 過寬，會命中 `0 failed` / `not failed` / `FAILED-CASES` 等綠燈 summary

### 修復方向建議

| 方向 | 描述 | 風險 |
|------|------|------|
| A | 縮窄 `FAILED` regex 為 `\bFAILED\b(?!\s*[:=]\s*0)` 等 word boundary + 排除「0」 | regex 易出漏網 |
| B | 改為上下文識別：先檢查 output 是否含「all tests passed」「Tests: N passed」綠燈 summary，有則 skip | 已有部分實作（L348）但只覆蓋 lowercase 字面 |
| C | 在每個 PATTERN 比對前先抽取 Jest/Mocha/pytest summary 結構化欄位（passed / failed 數），用結構化判斷取代字面比對 | 須處理多 framework summary 格式 |
| D | 上述方向組合（B 強化 + A 縮窄 regex） | 風險最低，覆蓋面最廣 |

### 雙向回歸驗證需求

### 方向選擇

採方向 D（B 強化 + A 縮窄）。

### 真根因驗證

讀 `.claude/hook-logs/pre-fix-evaluation-20260528-204742.json` 確認：3 次 `FAILED` 字面命中，pattern 為裸 `r"FAILED"`，與 `0 failed` 或 hyphen 識別符如 `FAILED-CASES` 共振。豁免 `_is_ticket_body_write` 未生效原因不可考（log 未保留原始 command），但 root pattern 過寬為事實。

### 實作摘要

`.claude/hooks/post-test-hook.py`：

1. **TEST_FAILURE_PATTERNS 縮窄（方向 A）**：
   - `(\d+) tests? failed` → `(?<![\d])([1-9]\d*) tests? failed\b`：排除「0 tests failed」
   - `FAILED` → `(?<![\w-])FAILED(?![\w-])(?!\s*[:=]\s*0\b)`：word boundary 排除 hyphen 識別符 / 識別符前後綴 / `FAILED: 0` 樣態

2. **新增 `_is_green_summary()` helper（方向 B 強化）**：
   - Jest/Vitest: `Tests: N passed, 0 failed` / `Tests: N passed, N total`
   - pytest: `===== N passed in X.XXs =====`
   - Mocha: `N passing`
   - Go test: `ok <pkg> X.XXs`
   - 通用「0 failed」字面
   - 既有「all tests passed」「no issues found」保留相容

3. **整合至 `evaluate_test_failure`**：替換原 lowercase 字面檢查為 `_is_green_summary()` 呼叫。

### 不在本 ticket 範圍

- `_is_ticket_body_write` 豁免機制不動（W3-041/W3-066 已穩定）。
- 升級為 PC error-pattern 留待後續評估（PM 建議方向，本 ticket 不執行）。
