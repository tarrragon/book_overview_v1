## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W1-008 Phase 4 Redundancy 視角發現 E2E 測試基礎設施有 3 處重複的「取得 Extension Service Worker target」邏輯，核心皆為同一段 `browser.waitForTarget(target => target.type() === 'service_worker' && target.url().startsWith('chrome-extension://'), { timeout })`：

- `tests/e2e/browser/helpers/storage-reader.js` `getServiceWorker`（31-42 行）→ 取 target 後 `.worker()`
- `tests/e2e/setup/extension-setup.js` `getExtensionId`（133-155 行）→ 取 target 後 `.url()` 解析 extensionId（不呼叫 `.worker()`）
- `tests/e2e/setup/extension-setup.js` `getBackgroundPage`（243-264 行）→ 取 target 後 `.worker()` 並指派 `this.backgroundPage`（instance 副作用）

重複的是「等待並取得 SW target」這段；3 處 caller 在取得 target 之後各自分歧（worker / url 解析 / 副作用指派）。

### 影響範圍

- 檔案：`tests/e2e/browser/helpers/storage-reader.js`、`tests/e2e/setup/extension-setup.js`（皆 E2E 測試基礎設施，非 `src/` 產品碼）。
- 三處 caller 的錯誤包裝各不相同，重構必須原樣保留（屬可觀測契約）：
  - `getServiceWorker`：`throw new Error('[SETUP] 取得 Service Worker 失敗: ...')`
  - `getExtensionId`：custom Error，`.code = ErrorCodes.OPERATION_FAILED`、`.details.originalCode = 'E2E_EXTENSION_ID_RETRIEVAL_FAILED'`
  - `getBackgroundPage`：custom Error，`.details.originalCode = 'E2E_BACKGROUND_SCRIPT_RETRIEVAL_FAILED'`

### 相關 Error Pattern

- quality-baseline 規則 5（W1-008 Phase 4 發現轉 P2 follow-up ticket 追蹤）。
- DRY 違反（quality-common §1.4）；同源 Phase 4 收斂先例：W1-033（loadAllBooksLazy 區域收斂）。

### PM 設計建議（供執行代理人 Phase 3a 參考）

1. **SSOT 回傳型別**：回傳 Puppeteer **Target**（非 worker）。target 是 3 caller 的最小共同單位；`getExtensionId` 只需 `.url()`，強制 `.worker()` 會多建立不必要的 CDP worker session。
2. **命名**：建議 `acquireServiceWorkerTarget(browser, timeout = 10000)`。
3. **放置位置**：優先依 ticket strategy 放 `extension-setup.js` 靜態方法（storage-reader 以 `ExtensionSetup.acquireServiceWorkerTarget(browser)` 呼叫）；若 storage-reader 因此引入過重依賴（path / ErrorCodes 等），改放共用 leaf module，由代理人依實際 import 成本判定並於 Solution 記錄。
4. **錯誤處理邊界**：SSOT 只收斂 `waitForTarget` predicate + timeout，**不含 try/catch**；3 處各自的 try/catch 與專屬錯誤契約原樣保留。
5. **副作用拆分**：`getBackgroundPage` 的 `this.backgroundPage = ...` 副作用留在 caller，不進 SSOT。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 變更摘要

抽出單一 SSOT 函式 `acquireServiceWorkerTarget(browser, timeout = 10000)`，回傳 Puppeteer **Target**（依 PM 設計建議 1：target 為 3 caller 最小共同單位，回傳 worker 會對只需 `.url()` 的 `getExtensionId` 多建立不必要的 CDP worker session）。函式只收斂 `waitForTarget` predicate（`type() === 'service_worker'` AND `url().startsWith('chrome-extension://')`）+ timeout，**不含 try/catch**（PM 設計建議 4）。

3 處 caller 改用共用函式，各自的 try/catch 與專屬錯誤契約原樣保留：

| Caller | 取得 target 後的後續處理 | 保留的錯誤契約 |
|--------|-----------------------|---------------|
| `storage-reader.js` `getServiceWorker` | `.worker()` | `throw new Error('[SETUP] 取得 Service Worker 失敗: ...')` |
| `extension-setup.js` `getExtensionId` | `.url()` 解析 extensionId | custom Error，`.code = ErrorCodes.OPERATION_FAILED`、`.details.originalCode = 'E2E_EXTENSION_ID_RETRIEVAL_FAILED'` |
| `extension-setup.js` `getBackgroundPage` | `.worker()` + `this.backgroundPage = ...`（副作用留在 caller，PM 設計建議 5） | custom Error，`.details.originalCode = 'E2E_BACKGROUND_SCRIPT_RETRIEVAL_FAILED'` |

`getExtensionId` / `getBackgroundPage` 原本以 `SW_WAIT_TIMEOUT = 10000` 常數顯式傳入，重構後保留該常數顯式傳入 SSOT，timeout 行為不變。

### 放置位置決策

新建 leaf module `tests/e2e/setup/service-worker-target.js`，未採 ticket strategy 原建議的「extension-setup.js 靜態方法」。

理由（依 PM 設計建議 3 的 import 成本判定）：`storage-reader.js` 重構前是**零 import 的純 leaf module**（只用全域 `setTimeout`）。若 SSOT 放 `extension-setup.js` 靜態方法，`storage-reader.js` 須 `require('../../setup/extension-setup')`，會把整個 `ExtensionTestSetup` class 及其依賴鏈（`puppeteer`、`path`、`fs`、`src/core/errors/ErrorCodes`）拉進這個輕量 storage helper —— 屬 PM 設計建議 3 所述「引入過重依賴」情形。獨立 leaf module 只依賴 `puppeteer`（型別文件用途，無實際 require），兩 caller 各以相對路徑引用（`./service-worker-target`、`../../setup/service-worker-target`），依賴成本最小。

### Phase 4 評估報告

- **重構需要**: 否（本 ticket 本身即為 W1-008 Phase 4 發現的 DRY 收斂，已完成收斂）
- **技術債務**: 發現 1 項，不在本 ticket 範圍 —— `extension-setup.js` 既有 ESLint `comma-dangle` 錯誤（`launchOptions` 物件 `protocolTimeout` 後 trailing comma，重構後位於 line 87）。經 `git stash` 驗證為重構前既有問題，依「scope expansion 應獨立 commit」原則未在本 DRY 重構處理。
- **程式碼品質**: A（SSOT 收斂後三處 caller 各保留專屬錯誤契約與副作用，職責分界清晰；新 leaf module 依賴最小化）

### 結論

純 DRY 重構，無新增技術債務。SW target 取得邏輯已收斂為單一 SSOT，3 處 caller 可觀測行為（回傳值、錯誤碼、錯誤訊息前綴、副作用）完全不變，E2E browser 測試 9/9 維持 GREEN。既有 `comma-dangle` lint 問題建議由後續 lint 清理 ticket 處理。
