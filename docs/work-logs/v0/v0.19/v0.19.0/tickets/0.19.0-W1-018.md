## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

多個測試以絕對計時門檻（`toBeLessThan(Nms)`）或相對計時比較（`secondRunTime < firstRunTime * 0.8`）作為 pass-fail 斷言。Jest jsdom 環境非有效效能量測環境，計時受全套件機器負載、GC、JIT 暖機影響，在完整 `npm test` 下 flaky。此外，部分 `toBeCloseTo` 使用 5 位小數精度驗證浮點平均值，在跨 suite 負載下亦 intermittent flaky（W1-017 實證）。

根本問題（W1-017 用戶確認的設計原則）：計時不應作為測試驗收條件，效能差是設計問題而非功能問題。

### 影響範圍

**`npm test` 範圍內（`tests/unit` + `tests/integration`）**，排除 W1-017 已處理的兩檔：

| 類別 | 檔案數 | 斷言數 |
|------|--------|--------|
| A: 計時硬門檻（設計不良） | 4 檔 | 17 個 |
| B: 高精度浮點（flaky） | 2 檔 | 4 個 |
| C: 合理斷言（保留） | 多檔 | 不計 |

詳細清單見重現實驗結果。

### 相關 Error Pattern

無既有 error-pattern 精確匹配。Solution 評估是否升級為測試設計規則。

## 重現實驗結果
### 實驗方法

使用 ripgrep 掃描 `tests/unit/` 與 `tests/integration/`（即 `npm test` 範圍）中的三類模式：
1. `toBeLessThan` / `toBeLessThanOrEqual` 搭配計時變數
2. `Date.now()` / `performance.now()` 差值用於斷言
3. `toBeCloseTo` 高精度（numDigits >= 5）

排除 W1-017 已處理的 `event-system-v2-performance-stability.test.js` 與 `UC07ErrorAdapter.test.js`。

### 實驗執行

搜尋指令：
- `rg -n 'toBeLessThan|toBeLessThanOrEqual' tests/unit/ tests/integration/ -g '*.test.js'`
- `rg -n 'toBeCloseTo' tests/unit/ tests/integration/ -g '*.test.js'`
- `rg -n 'Date\.now|performance\.now' tests/unit/ tests/integration/ -g '*.test.js'`

逐一讀取上下文判斷每個斷言是計時硬門檻、浮點精度脆弱、記憶體保護、邏輯不變式。

### 實驗發現

#### 類別 A：計時硬門檻（設計不良，需重新設計）

**A1. tests/integration/performance/performance-monitoring-integration.test.js**
- :464 `expect(avgCreationTime).toBeLessThan(0.5)` — 平均建立時間 < 0.5ms
- :524 `expect(avgTimePerError).toBeLessThan(0.5)` — 每個錯誤平均建立時間
- :533 `expect(scalabilityRatio).toBeLessThan(2)` — 擴展性比率
- 判斷：三個斷言皆為 Date.now/performance.now 差值硬門檻。0.5ms 門檻在 GC 觸發時極易超過。整檔位於 `tests/integration/performance/` 子目錄，定位即效能測試。
- 建議：整檔移至 `tests/perf/`（與 W1-017 方案 A 一致）。

**A2. tests/unit/handlers/extraction-completed-handler.test.js**
- :724 `expect(processingTime).toBeLessThan(1000)` — 處理 1000 本書 < 1 秒
- 判斷：單一計時斷言混在功能測試檔中。1000ms 門檻較寬鬆但仍為環境依賴。
- 建議：移除計時斷言，保留功能驗證（該 test case 其餘斷言皆為功能正確性）。

**A3. tests/unit/extractors/readmoo-data-validator.test.js**
- :485 `expect(processingTime).toBeLessThan(1000)` — 驗證 100 本 < 1 秒
- :641 `expect(secondRunTime).toBeLessThan(firstRunTime * 0.8)` — 快取加速 >= 20%
- 判斷：:485 為絕對計時門檻；:641 為相對計時比較，受 GC/JIT 雜訊影響同樣 flaky（mock 環境下兩次呼叫差值可能在雜訊範圍內）。
- 建議：:485 移除計時斷言（保留 `result.validBooks` 功能驗證）；:641 改為驗證快取命中（`getCacheHitRate() > 0`）而非計時比較。

**A4. tests/integration/chrome-apis/content-script-injection.test.js**
- 10 個計時斷言：:128, :142, :201, :298, :356, :361-364, :444
- 門檻範圍 500ms ~ 15000ms，皆為模擬環境中 mock 回傳值的計時驗證
- 判斷：整檔為 Chrome API 整合測試，計時值來自 mock。mock 計時門檻無實際效能量測意義，但因值由 mock 控制故實務上不 flaky（mock 回傳固定值）。
- 建議：保留但降為 info 級（不需 spawn IMP）。若未來 mock 策略變更再評估。歸類為「低風險保留」。

#### 類別 B：高精度浮點斷言（flaky，需降精度）

**B1. tests/integration/workflows/cross-device-sync-workflow.test.js**
- :145 `expect(avgProgress).toBeCloseTo(scenario.averageProgress, 5)` — 5 位小數精度
- W1-017 已實證此行 intermittent flaky（完整 npm test 首次失敗，單獨執行通過）。浮點平均值在 IEEE 754 下 5 位精度過高。
- 建議：降為 `toBeCloseTo(..., 1)` 或 `toBeCloseTo(..., 2)`。

**B2. tests/unit/background/domains/platform/services/platform-detection-service.test.js**
- :518 `expect(result.confidence).toBeCloseTo(0.9, 5)` — 5 位小數
- :983 `expect(confidence).toBeCloseTo(1.0, 5)` — 5 位小數
- 判斷：confidence 為浮點計算結果，5 位精度在 IEEE 754 下脆弱。同檔其他 toBeCloseTo 使用 1-2 位精度（合理）。
- 建議：降為 `toBeCloseTo(..., 2)`，與同檔其他斷言一致。

#### 類別 C：合理斷言（保留）

以下斷言經判斷為合理，不需重新設計：

- `event-system-v2-performance-stability.test.js` 中所有 `memoryGrowth < 2.0`、`cacheSize <= maxCacheSize`、`activeHandlerCount < 10`：記憶體洩漏與資源清理保護，非計時門檻。W1-017 已處理純計時部分，保留的為穩定性/功能類。
- `readmoo-data-validator.test.js:535` timestamp 順序比較：邏輯不變式（先建立的時間戳 < 後建立的）。
- `readmoo-data-validator.test.js:656,673` 快取比率與記憶體邏輯：`getCacheHitRate() < 1`、`currentUsage < peakUsage` 為邏輯不變式。
- 所有 `toBeCloseTo(..., 0-2)` 精度斷言：2 位以下精度對確定性浮點計算合理。
- `content-script-injection.test.js` 的 mock 計時斷言（A4 結論：低風險保留）。

### 盤點結論與 Spawn 規劃

#### 需 Spawn IMP 的修復項目

| Spawn IMP | 範圍 | 處理方式 | 預估規模 |
|-----------|------|---------|---------|
| IMP-1 | A1: performance-monitoring-integration.test.js（3 個計時斷言） | 整檔移至 tests/perf/（與 W1-017 方案 A 一致） | S（單檔搬移 + 驗證 npm test 不含） |
| IMP-2 | A2+A3: extraction-completed-handler.test.js（1 個） + readmoo-data-validator.test.js（2 個） | 移除計時斷言保留功能驗證；:641 改為快取命中率驗證 | S（兩檔各改 1-2 行） |
| IMP-3 | B1+B2: cross-device-sync-workflow.test.js（1 個） + platform-detection-service.test.js（2 個） | toBeCloseTo 精度從 5 降為 2 | XS（三處數字修改） |

三個 IMP 可合併為一個 ticket（總修改量 < 10 處，皆為測試碼，無跨模組依賴）。

#### 豁免項目（無需 Spawn）

- A4 content-script-injection.test.js（10 個 mock 計時斷言）：計時值由 mock 固定回傳，實務上不 flaky。無需建 ticket：mock 控制的計時斷言不受環境負載影響，修改無實際收益。
- 類別 C 全部：合理斷言，設計無問題。

#### 是否升級為 Error Pattern / 測試設計規則

建議建立測試設計規則（非 error-pattern），核心原則：

1. `npm test` 主套件禁止以絕對計時門檻作為 pass-fail 斷言
2. 效能測試放 `tests/perf/`，透過 `npm run test:perf` 獨立執行
3. `toBeCloseTo` 精度不超過 2 位（除非驗證確定性整數計算）
4. 快取加速驗證用命中率而非計時比較

此規則可作為 DOC ticket spawn 或併入 IMP ticket 的 Phase 4 產出。
