## Solution
<!-- Schema[DOC/Solution]: 免填（DOC 類型以變更摘要取代）（.claude/pm-rules/ticket-body-schema.md） -->
### 採用策略

按 acceptance 3 項分別執行：

#### AC1: 更新 saffron-system-analyst.md 禁止行為章節

在「絕對禁止」清單新增第 7 項「禁止跨 ticket 操作」：

> 7. **禁止對非自己派發範圍的 ticket 執行修改操作**：包括 `ticket track close` / `set-status` / 編輯他人 ticket md 等。即使在審查過程中發現重複 ticket、孤兒 ticket 或範圍衝突，SA 只能在審查報告的「系統衝突檢查」章節明示衝突關係，由 rosemary-project-manager 決策。**禁止自行 close 兄弟 ticket，即使標 closed_by 也屬越界**。

附加說明：歷史案例 0.19.0-W1-048.5（已完成）saffron 在 ANA 派發中越界自行 close 0.19.0-W1-048.6（已關閉），自標 closed_by=W1-048.5.1。事件暴露 saffron 既有禁止行為清單缺「跨 ticket 操作禁令」缺口。

#### AC2: 更新 agent-definition-standard.md 範本

在「區塊 2：禁止行為」必含元素表後追加「跨 ticket 操作」禁令範本說明（adding to existing 3-row table）：

| 必含元素 | 說明 |
| 跨 ticket 操作禁令 | 明示「禁止 close / set-status / 編輯非自己派發範圍的 ticket」（即使發現重複或衝突）|

並於章節末加註：

> **跨 ticket 操作禁令的依據**：subagent 在派發中發現的衝突應透過審查報告 / Exit Status / NeedsContext 上報 PM，由 PM 統一決策。subagent 自行修改他人 ticket 違反「ticket 由派發者管理生命週期」原則，且會造成並行 claim race condition 下的雙重寫入。

#### AC3: hook 層防護必要性評估

**評估結論**：建議建 follow-up IMP ticket，但本 ticket 不實作。

**理由**：

| 維度 | 分析 |
|------|------|
| 必要性 | subagent 透過 Bash 執行 `ticket track close <id>` 可繞過 agent definition 約束，純文字禁令無強制力 |
| 可行性 | PreToolUse:Bash hook 可解析 ticket CLI 命令，識別 close/set-status 操作 |
| 識別 dispatch scope 難度 | 需追蹤當前 subagent dispatch 包含哪些 ticket（或 ticket children）— 中等複雜度 |
| 失敗模式 | hook 若偵測到 subagent 修改非派發 ticket，輸出 stderr deny + 提示透過 NeedsContext 上報 |
| 評估範圍邊界 | 屬 hook 設計，需獨立 ANA→IMP 流程 |

**建議 follow-up ticket**：
- 類型：IMP
- 主題：subagent cross-ticket operation guard hook（PreToolUse:Bash 偵測 ticket track close/set-status 命令並驗證 dispatch scope）
- 優先級：P3（W1-048.5 事件是個案，純文字禁令先落地觀察是否再犯）
- Trigger：再次發生 subagent 越界 close ticket 案例（綁 trigger 觀察）
