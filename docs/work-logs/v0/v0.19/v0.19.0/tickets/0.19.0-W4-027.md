## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

command-entrance-gate-hook 認領閘門對合法 doc 編輯 prompt 三層共振誤判（W4-026 已逐層對照源碼確認）：

1. L1 `is_development_command`：以 substring 比對開發關鍵字，prompt「反映新增的監測波次」中描述性「新增」命中 IMPLEMENT_KEYWORDS，被誤判為開發命令（無 word-boundary、無語境）。
2. L2 `get_latest_pending_ticket`：對全版本 ticket 按 `st_mtime` 回傳最新 pending（剛遷移的 W1-013），與 prompt/版本無關。
3. L3 `check_ticket_status`：pending 直接阻擋，relevance 檢查只對 in_progress 生效，doc 編輯意圖無任何過濾閘門。

### 影響範圍

`.claude/hooks/command-entrance-gate-hook.py`（UserPromptSubmit 閘門）。影響所有「含描述性開發詞 + 存在無關 pending ticket」的 PM prompt，尤以 doc/worklog 編輯意圖為甚。

### 相關 Error Pattern

ARCH-018（hook 單維度強制與真實意圖衝突）、PC-113 / PC-138 / PC-144（hook/validator 關鍵字 substring false-positive 家族）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 組合 F 三層修復摘要

採 W4-026 WRAP 結論的組合 F，僅改 hook + 其測試（version 3.4.0 → 3.5.0）。

L1 `is_development_command`（語境判斷）：
- 新增 helper `_is_command_keyword_occurrence`：開發詞「句首出現」視為命令；「緊接描述性後綴（的/過）」視為名詞性修飾不判命令；其餘位置保守視為命令（防護不喪失）。
- 新增常數 `_DESCRIPTIVE_SUFFIXES`。效果：「新增的監測波次」中「新增」不再誤觸。

L2 `get_latest_pending_ticket`（觸發限縮）：
- 新增 `prompt` 參數 + helper `extract_ticket_id_from_prompt`（regex `\d+\.\d+\.\d+-W\d+-\d+`）。
- 觸發規則改為：(1) prompt 顯式引用 Ticket ID → 精準匹配；(2) 存在 in_progress → 回傳（既有 gate 行為）；(3) 否則不再以 mtime-latest pending 盲選。
- 向後相容：`prompt=None` 時 fallback 回舊行為，維持既有呼叫端/測試。

L3 `check_ticket_status`（pending relevance 閘門）：
- pending 阻擋前加 `check_ticket_relevance`（mirror in_progress 既有路徑）：無關放行、相關才阻擋並要求 claim。

### 設計決策

- relevance 閘門對 pending 採「無關放行」而非 soft warning（與 in_progress 的 soft warning 不同）：因 pending 未認領時若無關，無認領必要，放行符合真實意圖。
- L2 保留 `prompt=None` 相容分支，避免破壞既有測試與其他潛在呼叫端。

### 不在本 ticket 範圍

- management 白名單（如 prompt 含「hook」「commit」走 bypass）為既有行為，未調整。
