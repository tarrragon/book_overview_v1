### 互動分支調查（acceptance 1）

調查目標：釐清 `claim` 的 `--yes` / `--skip-verify` 互動分支、TTY 偵測行為與 partial metadata 寫入點。實際 source tree 與 ticket where.files 路徑不符（無 `commands_claim.py`），claim 邏輯分散於以下實際檔案：

- `commands/lifecycle.py` — `claim()`、`claim_with_verification()`、`execute_claim()` dispatcher
- `commands/claim_verification.py` — `prompt_user_decision()`（互動層）

**TTY 偵測與互動分支現況**：

| 路徑 | TTY 偵測 | 互動行為 | metadata 寫入 |
|------|---------|---------|--------------|
| 裸 `claim`（預設，無 `--verify`） | 無 | 不讀 stdin、不 prompt | `file_lock` 內單一原子 load→modify→save（lifecycle.py:454-498）|
| `claim --verify`（明示啟用，除錯用） | `sys.stdin.isatty()`（lifecycle.py:539 / claim_verification.py:530） | 非 TTY 且無 `--yes` → fail-closed return 1（寫入前攔截）；`--yes` 短路 prompt 為 y | 驗證通過/y 後才呼叫 `claim()` 原子寫入 |

**partial metadata 寫入點分析**：現行程式碼**不存在** partial 寫入點。狀態變更（status / assigned / started_at）三欄位在同一 `file_lock` 區塊內一次 `save_ticket` 寫入（lifecycle.py:493-498），fail-closed 攔截發生在寫入之前（line 539），不會產生中間態。

**`--skip-verify` 現況**：已於 W4-019 移除（兩輪觀察期 24hr / 75 commits / 5 ticket cycles 驗證外部依賴=0）。`claim --help` 實測旗標僅剩 `--verify` / `--yes`，無 `--skip-verify`。ticket 描述「第二次用 `--skip-verify` 確認」屬 W1-048.4.1 觀察當時（W4-019 前）的歷史行為。

### code-fix 評估結論（acceptance 2）

**結論：不需要 code fix，文件化推薦旗標即足夠。本票直接於 DOC 範圍完成，不 spawn IMP follow-up。**

評估依據：

1. **半成功 root cause 已被既有修正消除**。ticket why 描述的「`claim --yes` 互動環境受限導致 metadata 部分寫入」屬 W1-048.4.1 觀察當時行為。現行程式碼已由兩階段修正治本：
   - W3-046：claim 預設**不執行** AC 驗證（裸 `claim` 不讀 stdin、不偵測 TTY），互動受限對預設路徑完全無影響。
   - W4-019：移除 `--skip-verify` 旗標，AC 驗證與 claim 解耦至 `ticket track verify` 子命令。
2. **裸 `claim` 寫入為原子操作**。三欄位在單一 `file_lock` 區塊內一次寫入，不存在 partial 寫入點。
3. **`--verify` 路徑的 fail-closed 已是清楚錯誤訊號**。非 TTY 無 `--yes` 時 return 1 並印明示訊息（寫入前攔截），不會造成「無清楚錯誤訊號的半成功」。

因 root cause 已消除，再加「subagent 環境自動切換 non-interactive」的 code fix 屬重複防護（裸 claim 本就不互動），不值得觸發 reinstall + ticket-skill-sync-check + pytest 成本。剩餘風險純為「subagent 不知道該用裸 claim 而誤用 `--verify`」，屬文件化即可解決的認知缺口。
