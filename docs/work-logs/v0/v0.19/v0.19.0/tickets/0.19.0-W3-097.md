## Problem Analysis

### 問題根因

needs-context-listener-hook.py line 86-89 message 字串預設「代理人已於 X 回報資料缺口」假設 caller 為代理人。實際上 PM 自填 NeedsContext section（例如 trigger 觀察期 re-check 記錄）也會觸發此 hook，造成 false positive 提示（W3-095 session 三次重現）。

### 影響範圍

- 修改：.claude/skills/ticket/hooks/needs-context-listener-hook.py（line 86-89 訊息字串）
- 新增：.claude/skills/ticket/tests/test_needs_context_listener_hook.py（4 案例）
- 不影響：hook 觸發條件、section 過濾、失敗指示判斷邏輯皆維持原狀

### 相關 Error Pattern

W3-095 Phase 3 ANA 結論方案 V（字串中性化）。不引入 caller 偵測邏輯（runtime 無可靠 caller 識別訊號，且中性化已解決 false positive）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### TDD 實作策略

#### Phase 1 (規格)
**中性化邊界**：
- 修改點：needs-context-listener-hook.py line 86-89 message 字串
- 新訊息：`[NeedsContext] 已更新於 {ticket_id}，請 PM 確認是否需補料或評估後續動作`
- 不再預設 caller=代理人；對 PM 自填與代理人回報情境都適用
- 不引入 caller 偵測邏輯（runtime 無可靠 caller 識別訊號，且中性化已解決 false positive）

#### Phase 2 (RED)
**新建測試檔**：`.claude/skills/ticket/tests/test_needs_context_listener_hook.py`

至少 3 個測試案例（acceptance 要求 ≥ 2，第 3 個為迴歸保護）：

1. `test_message_is_neutral_no_agent_assumption`：驗證 message 不含「代理人」「agent」「已回報」等預設 caller 詞彙
2. `test_message_format_matches_spec`：驗證 message 含 ticket_id 與「請 PM 確認是否需補料或評估後續動作」
3. `test_non_needscontext_section_no_message`：迴歸保護——command --section 非 NeedsContext 時不輸出 systemMessage

**測試手法**（參考 test_track_hook_health.py pattern）：
- 動態 import hook module（hook 檔名含 dash，用 importlib.util.spec_from_file_location）
- monkey-patch `read_json_from_stdin` 注入 fake payload
- 捕獲 stdout 解析 JSON 驗證 additionalContext

#### Phase 3 (GREEN)
修改 hook line 86-89：
```python
message = (
    f"[NeedsContext] 已更新於 {ticket_id}，"
    f"請 PM 確認是否需補料或評估後續動作"
)
```

簡單字串修改，預期評估結論：無需重構，A+ 品質。產出 Phase 4 評估報告寫入 Completion Info。
