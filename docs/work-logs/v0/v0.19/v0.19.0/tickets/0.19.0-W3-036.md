## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`has_background_agents(input_data or {}, logger)` 屬於 loop-invariant（迴圈不變式）：其唯一輸入 `input_data` 在 `scan_pending_handoff_tasks` 整個 `for file_path in sorted(pending_dir.glob("*.json"))` 迴圈中固定不變，但呼叫點位於迴圈內每筆 pending JSON 的分類分支。每筆未完成、非建議性方向的 pending JSON 都會重算一次，造成 O(n) 重複呼叫。

`has_background_agents` 並非純函式——它每次呼叫都會 `logger.info`（背景代理人存在時）或 `logger.debug`（不存在時）。因此重複呼叫除了 CPU 浪費，也產生 O(n) 重複日誌雜訊。Python 直譯器不會自動外提帶副作用的函式呼叫，必須手動處理。

### 影響範圍

| 項目 | 位置 |
|------|------|
| 目標檔案 | `.claude/skills/ticket/hooks/handoff-auto-resume-stop-hook.py` |
| `has_background_agents` 定義 | 第 532-562 行（不需改動，僅被呼叫端調整） |
| `scan_pending_handoff_tasks` 定義 | 第 565 行起 |
| 待外提的迴圈內呼叫點 | 第 690 行 `elif has_background_agents(input_data or {}, logger):` |
| 對應測試檔 | `.claude/hooks/tests/test_handoff_auto_resume_stop_hook.py`（現有 39 個 test，連字號檔名以 importlib 載入為 `hook` 模組） |

實際 n（pending JSON 數）通常 < 5，效能收益 < 1ms，屬微優化；主要價值在設計乾淨度與日誌雜訊消除。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 變更摘要

`has_background_agents` 屬 loop-invariant（迴圈不變式）——唯一輸入 `input_data` 在整個 `scan_pending_handoff_tasks` 迴圈中固定不變，但原呼叫點位於迴圈內每筆 pending JSON 的分類分支（第 690 行 `elif has_background_agents(...)`）。

實作（loop-invariant code motion）：

| 變更點 | 內容 |
|--------|------|
| `scan_pending_handoff_tasks` 開頭（pending_dir 存在性檢查後、`for` 迴圈前） | 新增 `bg_active = has_background_agents(input_data or {}, logger)`，一次性計算 |
| 迴圈內第 690 行 | `elif has_background_agents(input_data or {}, logger):` 改為 `elif bg_active:` |
| `has_background_agents` 函式本體（532-562 行） | 不改動 |

`has_background_agents` 並非純函式——每次呼叫都會 `logger.info` / `logger.debug`。外提後該函式內的日誌從 O(n) 降為 1 次。迴圈內第 698 行 `logger.info("background_tasks 非空，視為最近任務: {ticket_id}")` 為 per-ticket 訊息（含 ticket_id），屬正確語意保留不動。

### 設計決策

- 變數命名 `bg_active`：與既有 ticket `how.strategy` 一致，bool 語意自說明。
- 放置位置選在 `pending_dir.exists()` 早退之後：目錄不存在時直接 return，無需計算 bg_active，避免無謂呼叫與日誌。
