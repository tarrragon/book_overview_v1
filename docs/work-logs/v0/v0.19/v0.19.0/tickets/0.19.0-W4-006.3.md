### TD-1 修法（Case C 寬鬆斷言收斂）

採方案 A 變體：保留單一 case，將三組 OR 斷言收斂為精確終態斷言。可行性依據：

- mock 為 deterministic：`chrome.tabs.sendMessage.mockResolvedValue({ success: true })` 永遠走 popup.js startExtraction 成功分支
- `flushMicrotasks(n=3)` 已覆蓋全部 chained await（query + PING + START_EXTRACTION），終態 `updateStatus('完成', '資料提取完成', ..., STATUS_TYPES.READY)` 必達
- 過渡狀態（'提取中'）於 await sendMessage 前後被 race 取代，最終可觀察狀態唯一

斷言收斂為：

```js
expect(statusText.textContent).toBe('資料提取完成')
expect(extensionStatus.textContent).toBe('完成')
expect(statusDot.className).toBe('status-dot ready')
```

設計依據對應 popup.js:284-289 `updateStatus(status, text, info, type)`：
- `statusText.textContent ← text` 參數
- `extensionStatus.textContent ← status` 參數
- `statusDot.className ← \`status-dot ${type}\``，type=READY=`'ready'`

未來若 updateStatus 誤傳含「資料」字眼的錯誤訊息，TD-1 修復後的精確斷言必然 RED，達到 TD-1 設計目的（回歸偵測）。

### TD-2 修法（setTimeout(0) helper 抽取）

抽出 `flushMicrotasks(n = 3)` local helper（不升級為 tests/helpers/ shared helper，避免 CJS/ESM 邊界 regression 風險），位置在 W4-006 describe 區內 beforeEach 之後、test case 之前。

JSDoc 說明：
- chained await 深度對應（query + PING + START_EXTRACTION 三層）
- n 提高無害，但 startExtraction 新增 await 層時需同步調整
- 為何採 setTimeout(0) 而非 jest.useFakeTimers：避免與既有 chrome.storage onChanged 等 callback mock 衝突

Case B 與 Case C 由各 3 行 setTimeout 改為單行 `await flushMicrotasks()`，每 case 淨減 2 行。

### TD-3 修法（行號註解修正）

驗證確認：
- `grep "line 84[0-9]\|line 85[0-9]" src/popup/popup.js tests/integration/chrome-extension/popup-interface.test.js` 修改前僅命中 test 檔的 3 處過時行號，src/popup/popup.js 本身無相關註解
- W4-006.2 merge 後 popup.js bundle 從 81KB → 187.63KB，startExtraction 實際位於 line 869、updateStatus 終態呼叫位於 line 889，原 line 845/852/855 全部失效

修法：移除 `(line 845)` / `(line 852)` / `(line 855)` 註解，改為邏輯步驟描述：

```
// popup.js startExtraction 成功路徑邏輯步驟：
//   1. updateStatus 過渡呼叫：('提取中', EXTRACTION_IN_PROGRESS, ..., LOADING)
//   2. sendMessage(START_EXTRACTION) 等待回應
//   3. response.success=true 後 updateStatus 終態呼叫：('完成', '資料提取完成', ..., READY)
```

popup.js 本身不需修改（行號註解只在 test 檔內）。
