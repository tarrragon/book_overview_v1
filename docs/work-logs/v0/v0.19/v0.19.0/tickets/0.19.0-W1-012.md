## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

依 W1-010 同源 pattern 修復 `extractHrefFromElement` 對相對路徑 reader link 的 `isUnsafeUrl` 誤判：

| 位置 | 修改前 | 修改後 |
|------|--------|--------|
| `src/content/adapters/readmoo-adapter.js:1004` | `this.isUnsafeUrl(rawHref)` | `this.isUnsafeUrl(rawHref, hrefBase)`（`hrefBase = getLocation().origin`） |

註：ticket 5W1H 標示 `:361` 為已過時的行號，實際呼叫點位於目前檔案的 `:1004`（`isUnsafeUrl` 函式本體已在 W1-010 移至 `:1361`）。

### 根因鏈

1. `extractHrefFromElement` 取得 `rawHref` 後呼叫 `isUnsafeUrl(rawHref)` 未傳 base
2. `isUnsafeUrl` 內部 `new URL(rawHref)` 對相對路徑（如 `/api/reader/abc123`）throw
3. catch 區塊回傳 `true` → 判定為 unsafe → `extractHrefFromElement` 回傳 `{ href: '', readerId: '' }`
4. `readerId` 退化為 fallback（privacy ID 或硬編碼），書籍主 ID 不再來自真實 reader link

### 安全保證（測試覆蓋）

`isUnsafeUrl` 設計保證仍生效，下列攻擊向量在「傳入 base」情境下仍會被過濾：

- `javascript:alert(...)` → `protocol !== https/http` 判定 unsafe
- `data:image/png;base64,...` → 同上
- `/api/reader/../../../etc/passwd` → 路徑遍歷檢查（W1-010 已改為對原始字串檢查，避免 `new URL` 正規化掩蓋）
- `/api/reader/%2e%2e/%2e%2e/etc/passwd` → 編碼路徑遍歷同樣攔截

### 不在本 ticket 範圍

W1-013（`validateCoverUrlInput:1017` + `validateReadmooDomain:1029` 雙處 URL 解析）為 PM 派發指引的 Phase B，將在 W1-012 complete 後接續處理。
