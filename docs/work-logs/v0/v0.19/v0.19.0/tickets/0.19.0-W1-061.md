# Execution Log

## Task Summary

分析 Readmoo library 頁 DOM 結構是否提供作者欄位，評估 authors 全空的修復策略。

---

## Problem Analysis

W1-001.2 實機驗證（2026-05-23 第三次恢復驗證）發現 928/928 提取書物件 authors:[] 全空。readmoo.md 記載 library 頁 DOM 結構僅 .title 無作者欄位，疑為來源頁面限制。需確認是提取邏輯遺漏還是 DOM 確實無此資訊。

---

## 重現實驗結果

### R（Reality Test）—— 真根因驗證

### 接受的根因假設

W1-001.2 實機驗證發現 928/928 提取書物件 `authors: []` 全空。原始假設「DOM 確實無作者欄位（來源頁面限制）」與「提取邏輯遺漏」兩種可能並存。

### 重現實驗

| # | 驗證項 | 工具 | 結果 |
|---|--------|------|------|
| 1 | W1-001.2 實機提取 928 本書 | chrome-devtools-mcp | 928/928 authors:[] 全空（已記錄於 W1-001.2） |
| 2 | readmoo-adapter.js 是否提取 author | grep -n author src/content/adapters/readmoo-adapter.js | **0 結果**（無 author 提取邏輯） |
| 3 | parseBookElement() bookData schema 是否含 author/authors | Read line 1101-1130 | **schema 無 author/authors 欄位**（只有 id/title/cover/progress/type/url/source/identifiers/coverInfo/progressInfo/extractedFrom） |
| 4 | 下游 normalize 取值來源 | grep -rn authors src/ | data-normalization-service.js:152 用 book.authors \|\| book.author \|\| book.contributors——全 undefined |
| 5 | normalizeAuthors(undefined) 行為 | Read line 214-216 | if (!authors) return []——空陣列預設值 |
| 6 | readmoo.md DOM 結構文件 | Read line 95-109 | .library-item.info 只列 .title，無 author 子節點（W6-012 時期記錄） |

### 已驗證的事實

1. **提取端缺口**：readmoo-adapter.js#parseBookElement 完全未嘗試提取 author 欄位（既無 selector 也無 fallback）
2. **正規化默契**：下游 normalizeBook 接受 undefined 並轉成 []，掩蓋了提取端缺口（無告警 log）
3. **文件記載**：W6-012 時期 readmoo.md DOM 結構記錄 .info 子節點只有 .title

### 仍未驗證的假設（需實機 DOM snippet）

1. **library 頁 .openbook-overlay**（hover 後顯示的浮層）內部是否有作者資訊？readmoo.md 只記載 div.privacy[id="privacy-{8位數字}"]，未說明其他子節點。
2. **library 頁 SPA 動態渲染**：DOM 是否在 v0.19.0 時期已演進，新增 .author、.creator、.book-author 等子節點？（readmoo.md 為 2026-05-17 更新但未針對 author 重新驗證）
3. **書籍詳情頁 / Reader API**（替代來源）是否提供作者？

### 結論

提取端與正規化端**結構性不對齊**：提取端產出無 author 欄位，正規化端假設來源欄位存在。即使 DOM 完全無作者，目前架構也無法產生告警讓開發者發現缺口（類似 PC-066 靜默失敗）。

acceptance 第一條（DOM snippet 證據）需實機驗證 .openbook-overlay 內容後才能定論。

## Solution
### DOM 實機驗證結論（2026-05-24，v0.19.0 時期）

**驗證環境**：chrome-devtools-mcp + Readmoo 真實帳號（總書庫 945 本，首批 render 96 個 .library-item）

### .library-item 完整 DOM 結構（v0.19.0 確認版）

```

├── .cover-outer
```

### Selector 命中結果（96 個 .library-item 樣本第一個）

| Selector | 命中數 | 結論 |
|----------|--------|------|
| .author | 0 | 不存在 |
| .creator | 0 | 不存在 |
| .book-author | 0 | 不存在 |
| [class*="author"] | 0 | 不存在 |
| [class*="creator"] | 0 | 不存在 |
| [class*="writer"] | 0 | 不存在 |

**結論**：Readmoo library 頁（`https://read.readmoo.com/#/library`）DOM **不提供任何作者資訊**，無論在 `.info` 區（progress / title / star-rating 三項）或 `.openbook-overlay` 區（detail / privacy / menu-status 三項）。

這是 **source data limitation**，非 extractor 邏輯遺漏。即使現在補上 `.author, .creator` selector 也無法取得作者（與 dom-utils.js extractBookInfo 通用邏輯結果一致：empty string）。

### 替代來源可行性評估

對應 acceptance 第三條「若 DOM 無作者欄位：評估替代來源可行性並記錄結論」。

| # | 來源 | 可行性 | 成本 | 推薦度 |
|---|------|--------|------|--------|
| 1 | Reader 個別書頁面 `/api/reader/{id}` | 可能（reader 內部有 metadata） | 高（928 本書需 928 個額外請求，且需 SPA 載入解析） | 低（會嚴重拖慢提取，可能觸發 rate limit） |
| 2 | Readmoo 商品頁公開 API（如 `/api/book/{id}`） | 未知（需反向工程確認端點） | 中（若端點存在，可批次拉取） | 中（需先 spike 確認端點） |
| 3 | 書名 → 第三方 API（Google Books / Open Library） | 高（title 查作者準確度 70-90%） | 低（外部 API 免費，需用戶 opt-in） | 中（match 準確度需驗證，且引入第三方依賴） |
| 4 | 用戶手動補作者（UI 表單） | 高（直接，用戶體驗為輔助而非自動） | 低 | 高（短期最務實） |
| 5 | CSV/JSON 匯入既有 metadata（用戶從其他來源） | 高（適合進階用戶） | 低 | 中（適合配合 v0.20.0 tag 管理） |

### 短期方案（v0.19.0 內測階段，推薦）

**接受 source limitation，在 UI / 匯出明示「作者欄位由用戶手動補寫」**：
- 提取邏輯維持現狀（authors:[]）
- UI overview 頁顯示作者欄位為空時提示「未從來源取得，可手動編輯」
- CSV 匯出 author 欄位空白，並在欄位說明附註「source-limited」
- 配合 v0.20.0 tag 管理功能，可引入「手動補作者」工作流

### 中期方案（v0.20.0+）

探索 Readmoo API 端點（方案 2）：投入小型 spike ticket 用 chrome-devtools network monitor 觀察 reader 載入時的 XHR / Fetch 流量，確認是否有可重用的 catalog/metadata API。

### 長期方案（v2.x 多書城支援後）

引入第三方 metadata enrichment（方案 3）：當多書城已支援後，建立統一的 `MetadataEnrichmentService`（例如 Google Books API）作為跨書城共用補強，但需用戶 opt-in 設定。
