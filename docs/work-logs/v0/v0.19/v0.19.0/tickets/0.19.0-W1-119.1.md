## Solution

### 修復摘要

W1-107 議題 A 收斂時保留的 3 個中文 legacy keys 已隨對應 caller 建立 local dict 完成清理。GlobalMessages 進入「僅 21 個跨模組共用 key」的最終穩態。

### 三 caller local dict 對照

| Caller | 新 messageKey | 中文渲染文字 | local dict 命名 | 注入方式 |
|--------|--------------|------------|---------------|---------|
| filter-engine | FILTER_UNKNOWN_CONDITION | 未知的篩選條件 | filterEngineMessages | constructor `_registerLocalMessages` → 外部 logger.messages.addMessages |
| search-engine | SEARCH_INDEX_FALLBACK_TO_LINEAR | 索引搜尋失敗，回退到線性搜尋 | searchEngineMessages | 同上 |
| platform-detection-service | EVENT_LISTENER_REGISTRATION_FAILED | 事件監聽器註冊失敗 | platformDetectionMessages | `new Logger(name, level, platformDetectionMessages)`（W1-108 readmoo-adapter pattern） |

### 設計決策說明

#### 1. 兩種注入模式並存

filter-engine 與 search-engine 採用「constructor-time 註冊到外部注入的 logger.messages」模式，platform-detection-service 採用「internal Logger 注入 local dict」模式。差異源於 engine 強制依賴注入 Logger（外部來源），service 在內部 `new Logger`（內部來源）。

#### 2. Mock logger 相容性（既有測試零破壞）

`_registerLocalMessages` 採 defensive check：`logger.messages && addMessages instanceof function` 才註冊，mock logger（plain `{ info, warn, error, debug }` 物件）靜默略過。既有 mock-based 測試完全不需改 mock 結構。

#### 3. PC-165 防護：渲染層級驗證

為避免「unit test 綠燈但 runtime [Missing: KEY]」的 false positive 修復鏈（PC-165），每個 caller 都新增「真實 Logger + 真實 MessageDictionary」整合測試，斷言 `logger.messages.get(KEY)` 渲染為預期中文字面（非僅 messageKey 字面）。

### 不在本 ticket 範圍

- platform-detection-service 其他 `new Logger('[PlatformDetectionService]')` 後備呼叫（line 94 / 775 / 1005）：使用其他 messageKey（如「Service initialization partially failed」/「Failed to emit detection events」）。這些 keys 本來就不在 GlobalMessages（W1-109 清理後），亦不在本 ticket 的 3 keys 清單，維持原狀為已知 `[Missing: KEY]` runtime 顯示，屬其他 ticket 處理範圍。

- searchUIMessages（src/ui/book-search-filter-integrated.js）：未變動。W1-119.1 採「engine 自有 local dict 註冊到外部 logger.messages」模式，不擴張 searchUIMessages 的 SSOT 邊界。
