### 問題根因（PM 補強，W1-117 後現況）

`src/content/adapters/readmoo-adapter.js` 仍直接引用 GlobalMessages 的 extractor 專屬 key（NO_BOOK_ELEMENTS_FOUND / BOOK_CONTAINERS_FOUND / GET_BOOK_ELEMENTS_CALLED / WAIT_FOR_BOOK_ELEMENTS / FALLBACK_SELECTOR_ATTEMPT / LAST_RESORT_STRATEGY / UNSAFE_COVER_URL_FILTERED / PLACEHOLDER_URL_REPLACED 等），尚無自有 local MessageDictionary。

### 影響範圍

| 檔案 | 狀態 |
|------|------|
| `src/content/adapters/readmoo-adapter.js` | 仍依賴 GlobalMessages（無 local dict） |
| `src/popup/popup.js` | 已有 local dict（popupMessages） |
| `src/ui/book-search-filter-integrated.js` | 已有 local dict（searchUIMessages） |
| `src/platform/readmoo-platform-migration-validator.js` | 已有 local dict（validatorMessages） |
| `src/popup/popup-error-handler.js` | 已有 local dict |

### 相關 Error Pattern

PC-165 False Positive 修復鏈（W1-114 撰寫）— 切換 local dict 後須含 runtime 驗證（不可僅靠 unit test mock 綠燈）。

### Stale Status 解除依據

- 原 blocked 因 W1-105 MessageDictionary constructor 未支援 union signature（new MessageDictionary({...}) 會把物件當 language 字串）
- W1-105 (commit 4ee6c62c) 已改為 union signature
- W1-115 (Logger constructor 加第三參數 messages = GlobalMessages) 已 completed
- W1-117 重做 W1-106 清理 5 popup key 已 completed
- 結論：所有 prerequisite IMP 完成，extractor 建 local dict 技術路徑清通

### 簡化 WRAP 三問

**W（Widen）候選方案**：
- 方案 1（本案）：readmoo-adapter 建 local dict，仿 popup/search-filter 模式
- 方案 2：直接從 GlobalMessages 移除 extractor key 合併到 W1-109 — 不可行（會破壞 adapter runtime）
- 方案 3：保持現狀 extractor 永用 GlobalMessages — 違反 W1-107 議題 A 納入標準（W1-111 文件明示）

**A（Attain distance）機會成本**：W1-108 完成後解鎖 W1-109 → W1-110 spawn 鏈；不做整鏈卡住

**P（Prepare to be wrong）失敗預想**：替換 caller 時遺漏某處或測試 mock 不同步。對應防護：acceptance 3 強制 tests/unit/extractors/ + tests/integration/ 100% 通過；參照 PC-165 加 runtime 層級驗證

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修改摘要

**核心變更（`src/content/adapters/readmoo-adapter.js`）**:

1. 模組 import 新增 `MessageDictionary`（src/core/messages/MessageDictionary）
2. 新增 module-level `readmooAdapterMessages` local MessageDictionary,涵蓋 36 個 extractor / content layer 專屬訊息 key:
   - 18 個原 GlobalMessages 已含的 key（沿用原文字以維持 grep 對齊）
   - 18 個原 GlobalMessages 缺失、過去 runtime 命中 `[Missing: KEY]` 的 key（首次補上具業務情境的中文文字）
3. `createReadmooAdapter()` 內 default Logger 實例化改為 `new Logger('ReadmooAdapter', 'INFO', readmooAdapterMessages)`,以第三參數注入 local dict（仿 popup.js:103 / book-search-filter-integrated.js:54 模式,依賴 W1-115 修復的 Logger constructor union signature）

**新增測試（`tests/unit/adapters/readmoo-adapter-local-dict.test.js`,5 個 test）**:

- PC-165 防護 runtime 層驗證,刻意「不」使用 mockLogger,改用真實 Logger + spy console,觀察渲染後輸出的訊息字串
- 核心 test 3 驗證 `PARSE_LIBRARY_TOTAL_FAILED`(原 GlobalMessages 缺失此 key)在 runtime 渲染為「解析書庫總數失敗」而非 `[Missing: KEY]`——若 local dict 未真正生效必然失敗
- test 5 驗證 local dict 不污染 GlobalMessages(隔離性)

### 設計決策

- **不修改 `_loadDefaultMessages`**: 嚴守 ticket 範圍邊界,GlobalMessages 清理為 W1-109 範圍
- **不實作 freeze 機制**: W1-110 範圍,本 ticket 僅建 local dict 與切換 caller
- **文字保留原版**: 18 個 GlobalMessages 已有的 key,文字直接沿用,使 grep 字面與既有日誌契約一致
- **`options.logger` 注入路徑保留**: 既有 mockLogger 注入機制(W1-030 涵蓋率日誌契約測試需求)未動,新測試也驗證向後相容

### 不在本 ticket 範圍

- W1-109: GlobalMessages._loadDefaultMessages 清理 extractor key
- W1-110: MessageDictionary freeze 機制實作
- chrome-devtools-mcp 實機 runtime 驗證(可規劃為後續 IMP 或 W1-117 後續端到端驗證)
