### 採用方案 A：新增 `validateRequiredFields` / `filterValidBooks` public delegate（委派模式）

**選擇理由**（沿用 .5.1 / .5.2 已驗證的委派模式）：

| 方案 | 設計成本 | 風險 | 採用？ |
|------|---------|------|--------|
| A: 新增 public delegate（純委派 `_validateRequiredFields` / `_filterValidBooks`） | 最低（src 新增約 42 行 + JSDoc） | 零（內部呼叫鏈 `_isValidBook → _validateRequiredFields`、`_processBookData → _filterValidBooks` 完全不變） | 採用 |
| B: 注入 stub validator（透過 `import-stub-helpers.js`） | 中（測試需建 stub 與 inject 點，與 .5.1 / .5.2 委派風格不一致） | 中（測試與 SUT 介面分歧） | 否 |
| C: 改測高層 entry（`importer.import()` / `parseContent`） | 高（30 處全改，斷言重寫） | 中（測試意圖偏離單元職責，無法直接驗證純函式邏輯） | 否 |

方案 A 與 .5.1 `isCSVFile`、.5.2 `extractBooksFromData` / `processBookData` 同採委派模式，並與既有公開 API（`validate` / `read` / `parseContent`）設計風格一致。
