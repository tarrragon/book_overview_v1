## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`src/` 目錄殘留 4 個工具產生的 backup 檔（時間戳多對應 2025-09 專案暫停前）：

- `src/overview/overview.js.backup-1758413171003`
- `src/ui/handlers/ui-progress-handler.js.backup-1758413171004`
- `src/core/logging/Logger.js.backup`
- `src/ui/search/ui-controller/search-ui-controller.js.backup-1758413171004`

根因為 `scripts/build.js` 複製 src 樹時未過濾 `.backup` 檔（第 59 行 `fs.copyFileSync` 所在的複製流程不分副檔名），4 檔全數進入 `build/development/` 與 `build/production/` 產物，會混入 W1-002 內測分發 ZIP。

### 影響範圍

- 刪除目標：上列 4 個 `.backup` 檔。PM 前台已 grep 確認專案內無任何程式碼或測試引用 `.backup` 檔名或時間戳。
- 修改目標：`scripts/build.js` 複製邏輯，新增 `.backup` 與 `.backup-<時間戳>` pattern 排除。
- 各 backup 檔旁皆存在對應正式 `.js`（如 `ui-progress-handler.js` 16431 bytes vs backup 16387 bytes），刪除 backup 不影響功能。
- `.backup` 副檔名無法被 JS 模組解析（`import` / `require` 只解析 `.js` 等），無執行期引用風險；唯一問題是 build 過度複製，由 build.js 修正根除。

### 相關 Error Pattern

- quality-baseline 規則 5（發現即追蹤）：W4-007 原僅追蹤 1 檔，PM 派發前驗證發現實為 4 檔 + build.js 根因，經用戶決策擴大為單一 ticket 包辦（刪檔 + 修 build 腳本）。
- 同類議題：src/ 目錄衛生、build 產物純淨度（內測打包前置）。

### 重新驗證（PC-007）

開工前重新 grep 確認專案內無任何程式碼/測試引用這 4 個 backup 檔：

- `grep -rn "\.backup" src/ tests/ scripts/`：所有命中皆為無關業務概念（`backupManager` / `backupDir` / `.migration-backups` 等）或 `scripts/package.js` 既有的 `*.backup*` ZIP 排除規則，無一引用具體 4 個 backup 檔名。
- `grep -rn "1758413171003|1758413171004"`：時間戳零引用。
- 確認結論：4 檔皆為孤立殘留，刪除不影響功能（`.backup` 副檔名本就無法被 JS 模組解析器 `import` / `require`）。

### build.js 修改方式

根因位於 `scripts/build.js` 的 `copyRecursive`（遞迴複製 src 樹）。原檔案過濾僅跳過 `.test.` 與 `.spec.`，對其他副檔名（含 `.backup`）無條件 `fs.copyFileSync`。

修正：在 `items.forEach` 過濾區塊新增一段判斷，檔名含 `.backup` 子字串即 `return` 跳過。

```javascript
// 跳過 .backup 與 .backup-<時間戳> 開發殘留檔。
if (item.includes('.backup')) {
  return;
}
```

`item.includes('.backup')` 同時涵蓋 `.backup`（如 `Logger.js.backup`）與 `.backup-<時間戳>`（如 `overview.js.backup-1758413171003`）兩種 pattern，語意與 `scripts/package.js` 既有的 `*.backup*` glob 排除一致，形成 build 與 package 兩層一致的純淨度防護。
