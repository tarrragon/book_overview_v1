## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-001.2.1 實機驗證暴露 `src/utils/file-reader-factory.js` 使用 Node.js `global` 變數導致 CE runtime 100% 崩潰。初步 grep 命中 25 行 / 9 檔，但 `global` 只是已知 Node-only anti-pattern 之一。本 ANA 系統性審查全 src 所有 CE 環境不相容模式。

### 影響範圍

全 src/ 目錄，涵蓋四個 CE runtime entry point：
- SW（background.js）
- Content Script（content-modular.js）
- Popup（popup.js）
- Overview（overview.js）

### 相關 Error Pattern

目前無專屬 CE-Node 環境前提誤判 error-pattern，本 ANA 將評估是否需新建。

### 審查方法

1. 對照 CLAUDE.md §7 + chrome-extension-quickref.md 列出禁用 API
2. rg 搜尋危險關鍵字：global / process.* / require('crypto') / require('fs') / __dirname / Buffer / window（SW context）
3. 確認 esbuild build 配置（platform: browser, bundle: true, 4 entry points, 無 global/process polyfill）
4. 區分三類：致命 / 潛在 / 安全

## 重現實驗結果
### 實驗方法

以 rg 對全 src/ 搜尋 CE 環境不相容關鍵字，交叉比對 esbuild build 配置（scripts/build.js）確認哪些模式會進入 CE runtime bundle。

### 實驗執行

1. `rg -n '\bglobal\b' src/ -g '*.js'` — 31 命中（含 URL regex 2 條誤報）
2. `rg -n 'process\.' src/ -g '*.js'` — process.nextTick(4)、process.hrtime(4)、process.memoryUsage(12)、process.on(2)、process.cwd(3)、process.env.NODE_ENV(13)、process.platform(1)
3. `rg -n "require\('crypto'\)" src/` — 6 檔
4. `rg -n '\bwindow\.' src/background/` — 17 命中（8 為 window.ClassName = 暴露、9 為 window.matchMedia typeof guard）
5. 確認 build.js：esbuild platform:browser, bundle:true, 無 define/inject/polyfill
6. 確認 core/migration/ 的 fs/path/child_process require 不在任何 entry point 依賴鏈中

### 實驗發現

**已驗證事實**：
- esbuild bundle 會解析 require() 但不 polyfill Node.js API（global/process/crypto）
- `global` 裸用（無 typeof guard）在 CE runtime 立即 ReferenceError（W1-001.2.1 已實證）
- `process.nextTick` / `process.hrtime` / `process.memoryUsage` / `process.on` 在瀏覽器環境無定義
- `require('crypto')` esbuild bundle 時若找不到會 build error 或 runtime 空模組
- `window.*` 在 SW context 無定義，但 `typeof window !== 'undefined'` guard 可安全 fallback
- core/migration/ 使用 fs/path/child_process 但不在 CE entry point 依賴鏈中，為 dev-only 工具

**仍未驗證的假設**：
- esbuild 對 `require('crypto')` 的實際 bundle 行為需確認（可能 build warning 或 empty module）
- process.env.NODE_ENV 是否被 esbuild define 替換（build.js 未見 define 配置，可能保留原樣）

## Error Pattern 評估

**建議新建**：CE-Node 環境前提誤判 anti-pattern（暫定 IMP-CE-001 或 PC-xxx）

**模式描述**：開發者在 CE src/ 中使用 Node.js 專屬 API（global / process / require('crypto') / Buffer / fs 等），Jest 單元測試在 Node.js 環境下全綠，但 CE runtime 崩潰。根因是 Jest 測試環境（jsdom + Node.js）與 CE runtime（瀏覽器 + esbuild IIFE bundle）的 API 可用性差異。

**觸發條件**：src/ 下使用 Node.js API 且無 typeof guard，esbuild 未配置 polyfill/define。

**防護方向**：
1. ESLint no-restricted-globals 規則（禁 global、process、Buffer、__dirname、__filename）
2. esbuild define 配置 process.env.NODE_ENV
3. Build-time 驗證腳本掃描 bundle 產物是否含 Node.js API 殘留
