## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-039 thyme 完成 commit b7956e14 後執行 `ticket track complete 0.19.0-W1-039`，觸發 `phase4-decision-enforcement-hook.py`（PreToolUse 殘留二次掃描）。hook 掃描 W1-039.md frontmatter `why` 欄位：

```
why: W1-029.1 Phase 4 評估發現 tab-state-tracking-service.js ...
```

M1 regex `Phase\s*[0-9]+[^\n]{0,30}?(?:再|在)?(?:決定|決斷|判斷|評估)` 命中 "Phase 4 評估"（line 31），觸發 BLOCK。

此處 "Phase 4 評估發現" 語意為「W1-029.1 的 Phase 4 階段中發現的問題」，是 **source ticket 的歷史脈絡引用**，非本 ticket 的延後決策。屬 PC-142 case 3（frontmatter 合法欄位含階段名稱）與新 case：**source ticket history 引用中的階段名稱**。

代理人臨時在 main repo W1-039.md 加 `<!-- PC-093-exempt: history:... -->` marker 通過 hook，complete 後 `git restore` 還原。

### 影響範圍

- **直接影響**：任何 ticket 的 frontmatter `why` 含 "Phase N 評估/決定" 等歷史脈絡引用都會被攔截
- **受影響檔案**：`.claude/hooks/phase4-decision-enforcement-hook.py`（hook 邏輯）
- **觸發頻率**：中等。source_ticket 引用 Phase 4 發現是常見模式（Phase 4 產出 spawned ticket 時 why 自然描述「Phase 4 發現 X」）

### 相關 Error Pattern

- **PC-142**（Phase 4 Hook 字面誤判規則引用）：本事件為 PC-142 的新變體（frontmatter why 中的 source ticket 歷史引用，與原 case 3 frontmatter `how.strategy` 類似但位於不同欄位）
- **PC-077**（Hook worktree vs .claude/ 保護死結）：本事件次要面向。thyme 在 worktree 工作但 ticket md 位於 `docs/`（非 `.claude/`），hook 讀檔路徑取決於 `git rev-parse --show-toplevel` 回傳值（worktree 或 main repo）。W1-039 事件中代理人在 main repo 加 marker 才通過，暗示 hook 讀的是 main repo 的 ticket md

### 兩問題區分

| 維度 | PC-142（字面誤判） | PC-077（跨邊界讀檔） |
|------|-------------------|---------------------|
| 根因 | M1 regex 無法區分「歷史引用」vs「延後決策」 | hook 的 `find_ticket_file` 從 `get_project_root()` 解析路徑，worktree 環境下可能讀到 main repo 版本 |
| 觸發頻率 | 高（每次 frontmatter why 含 Phase N 相關詞） | 低（僅 worktree 派發 + ticket md 在兩處不同步時） |
| 嚴重度 | 中（false positive 阻擋合法 complete） | 低（大部分場景 ticket md 在 main repo 和 worktree 內容一致） |
| 獨立性 | 獨立問題，即使 main repo 執行也觸發 | 依附於 PC-142：只有 PC-142 命中時才暴露 worktree 讀檔問題 |

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 分析結論

### 問題優先級判定

PC-142（字面誤判）是主問題，PC-077（跨邊界讀檔）是次要問題且依附於 PC-142。修復 PC-142 即可消除 W1-039 事件的根因；PC-077 在本事件中的貢獻低（ticket md 位於 docs/ 非 .claude/，worktree 與 main repo 共享 docs/ 路徑結構）。

### 三方案評估

| 方案 | 描述 | 優點 | 缺點 | 建議 |
|------|------|------|------|------|
| (a) Hook 改讀 worktree 本地 ticket md | 修改 `find_ticket_file` 或 `get_project_root` 使 hook 感知 worktree 環境 | 解決 PC-077 跨邊界一致性 | 本事件 PC-077 貢獻低；docs/ 在 worktree 中通常與 main repo 同步；改動影響所有使用 `find_ticket_file` 的 hook | 不優先 |
| (b) Frontmatter YAML 區域豁免 | hook 掃描時跳過 frontmatter（`---` 到第二個 `---` 之間），不對 YAML 欄位做 phrase 偵測 | 根除 frontmatter why/how.strategy 觸發 M1 的問題；零 false positive 風險；實作簡單（類似既有 Schema placeholder 區塊跳過邏輯） | 若 frontmatter 真的含延後決策（極低機率），會漏偵測 | 強烈建議 |
| (c) Hook 改用 git show 取對應 commit 版本 | `git show HEAD:path/to/ticket.md` 讀已 commit 版本 | 確保讀到最新 commit 的穩定版本 | 增加 git 子進程開銷；未 commit 的 exempt marker 無法被讀取；複雜度高 | 不建議 |

### 建議組合

**採方案 (b)**：在 `scan_lines_for_phrases` 和 `collect_exempt_markers` 函式中新增 frontmatter 區塊偵測（`---` 起始到第二個 `---`），整段跳過 phrase 掃描。

理由：
1. Frontmatter 是 ticket 的結構化元資料（ticket CLI 管理），非人類撰寫的決策論述
2. 既有先例：hook 已跳過 Schema placeholder 區塊（W10-130）和 fenced code block（W11-018），frontmatter 跳過是同一設計模式的自然延伸
3. 實作量小：新增 `compute_frontmatter_lines(lines)` 函式（約 15 行），在 `scan_lines_for_phrases` 與 `collect_exempt_markers` 中加入 `if idx in frontmatter_lines: continue`
4. PC-077 跨邊界問題在本事件影響低，不需額外處理；若未來出現 PC-077 獨立觸發案例再建 ticket

### PC-142 error pattern 更新

新增 case 5：frontmatter `why` 欄位含 source ticket 歷史引用中的 Phase N 階段名稱。防護措施更新為方案 (b)。
