## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

**根因：v0.17.x PROP-007 重構只覆蓋了 model 層與匯出入層，未觸及 runtime 提取 pipeline。**

具體而言，存在三個斷層：

**斷層 1：ReadmooAdapter 不產出 readingStatus**
`src/content/adapters/readmoo-adapter.js` 的 `parseBookElement()` 建構 book 物件時（L530-559），只設定了 `id, title, cover, progress, type, extractedAt, url, source, identifiers, coverInfo, progressInfo, extractedFrom`。完全沒有 `readingStatus` 欄位。BookSchemaV2 的 `mapV1StatusToV2()` 和 `computeAutoStatusTransition()` 都是現成可用的，但 ReadmooAdapter 從未呼叫。

**斷層 2：event-coordinator 直接透傳存入 storage，無 model 轉換**
`src/background/events/event-coordinator.js` L536-575 的 `EXTRACTION.COMPLETED` 監聽器收到 content script 傳來的 books 陣列後，只做了 tags 正規化（確保含 'readmoo'），然後直接包成 `{ books, extractionTimestamp, ... }` 寫入 `chrome.storage.local`。沒有呼叫 `BookSchemaV2.applyDefaults()` 或任何 v1→v2 轉換。

**斷層 3：tagIds 與 tag_categories 初始化缺失**
BookSchemaV2 定義了 `tagIds: { type: 'array', default: [] }` 但 ReadmooAdapter 完全不產出此欄位。event-coordinator 也不初始化。Overview 層（`overview-page-controller.js`）直接讀 `book.tagIds`，結果是 undefined。

**為何 v0.17.x PROP-007 重構未涵蓋：**
PROP-007 的範圍是 Schema/Storage/匯出/搜尋適配——即「model 定義 + 匯出入格式 + 搜尋篩選」。193 單元測試全部通過，但這些測試只驗證 model 層邏輯（applyDefaults、validateBook、mapV1StatusToV2 等），從未整合測試 runtime 提取 pipeline（content script → SW → storage）。runtime pipeline 是既有程式碼（v0.12.x 時代），PROP-007 未將其納入重構範圍。

### 影響範圍

| 影響模組 | 具體問題 |
|---------|---------|
| `src/content/adapters/readmoo-adapter.js` | 不產出 readingStatus / tagIds / schema_version |
| `src/background/events/event-coordinator.js` | 直接透傳 book 物件入 storage，無 model 轉換 |
| `src/overview/overview-page-controller.js` | 讀取 undefined readingStatus → 顯示「未知」；讀取 undefined tagIds → 顯示「未分類」 |
| UC-01/03/04/06 | 6 種閱讀狀態完全失效、tag 系統完全失效 |

### 相關 Error Pattern

無既有 error-pattern 完全對應。此問題本質是「重構範圍遺漏」——model 層重構完成但 runtime 消費端未同步。

**Cover 過濾追蹤**：
- `extractCoverAndTitle()` L436 呼叫 `isUnsafeUrl(cover)`

- Readmoo 的 cover img src 若為相對路徑（如 `/cover/xxx.jpg`）或 data URI，`new URL()` 會 throw → catch 區塊 return true（判定為 unsafe）
- 結果：78/96 封面被錯誤過濾

### 實驗發現

**已驗證事實**：
1. ReadmooAdapter 完全不產出 readingStatus — 程式碼中無此欄位設定（L530-559 確認）
2. ReadmooAdapter 完全不產出 tagIds — 同上
3. event-coordinator 無 model 轉換層 — 直接透傳存入 storage（L536-575 確認）
4. BookSchemaV2 的 `applyDefaults()` 可正確填入 readingStatus 預設值 'unread'、tagIds 預設值 []，但 pipeline 中無人呼叫
5. BookSchemaV2 的 `mapV1StatusToV2()` / `computeAutoStatusTransition()` 可由 progress 推導 readingStatus，但 pipeline 中無人呼叫
6. cover 過濾的 `isUnsafeUrl()` 對相對路徑 URL 會 throw（因 `new URL()` 需要 base URL），導致大量合法封面被過濾

**仍未驗證假設**：
- cover URL 被過濾的 78 個具體原因（是相對路徑還是 data URI），需實機 console 確認實際 img src 值

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復策略

**推薦方案：在 event-coordinator（SW 層）加入 model 轉換層**

理由：
- ReadmooAdapter 是 DOM 提取層，職責是從 HTML 提取原始欄位，不應承擔 model 語意轉換
- event-coordinator 是 SW 層的資料入口，在存入 storage 前做 model 轉換（applyDefaults + readingStatus derivation）是最自然的位置
- 已有 BookSchemaV2.applyDefaults() 和 mapV1StatusToV2() 可直接使用

### readingStatus 應在哪一層 derive

在 **event-coordinator**（SW 層）存入 storage 前：
1. 呼叫 `BookSchemaV2.applyDefaults(book)` 填入 readingStatus 預設值 'unread'、tagIds 預設值 [] 等
2. 根據 progress 值呼叫 derive 邏輯：progress=0 → unread、0<progress<100 → reading、progress=100 → finished
3. 設定 schema_version

### tagIds / tag_categories 應在哪一層初始化

在 **event-coordinator**（SW 層）存入 storage 前：
- `applyDefaults()` 會將 tagIds 初始化為 []
- tag_categories 是獨立的 storage key（非 book 欄位），由 tag 管理功能（v0.20.x）初始化，目前不需處理

### cover 過濾問題

`isUnsafeUrl()` 使用 `new URL(url)` 解析，對相對路徑（無 protocol/host）會 throw。修復方向：為 content script 環境提供 base URL（`location.origin`），使 `new URL(cover, baseUrl)` 能正確解析相對路徑。
