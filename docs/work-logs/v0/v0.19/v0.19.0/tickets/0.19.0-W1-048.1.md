### Phase 1 功能設計（重構規格：行為保留、介面外露）

#### 設計核心原則

本 ticket 屬**重構規格**：行為不變，將 importer 既有的「驗證 + 讀檔 + 內容處理」三組職責外露為 public API，使 controller 不再呼叫底線私有方法，且消除 controller 三個 test-only proxy。新 API 是現有私有方法的契約包裝，內部實作可不變。

---

### 一、新 Public API 簽名

對 `src/overview/book-file-importer.js` 新增三個 public method，作為現有底線私有方法的對外契約：

| Public API | 對應私有方法 | 簽名 | 副作用 | 拋出 |
|-----------|-------------|------|--------|------|
| `validate(file)` | `_validateFileBasics(file)` + `_validateFileSize(file)` 合併 | `(file: File) => void` | 失敗時呼叫 `this.showError(msg)` | `Error{ code: VALIDATION_ERROR, details: { category: 'validation' } }` |
| `read(file)` | `_readFileWithReader(file)` | `(file: File) => Promise<ImportResult>` | FileReader 讀檔；失敗時呼叫 `this.showError(msg)` | `Error{ code: UNKNOWN_ERROR \| PARSE_ERROR, details }` |
| `parseContent(content, fileFormat)` | `_handleFileContent(content, fileFormat)` | `(content: string, fileFormat: 'json'\|'csv') => ImportResult` | 無（純函式） | `Error{ code: PARSE_ERROR, details: { category: 'parsing' } }` |

**設計選擇說明**：

- **`validate` 合併 basics + size 為單一進入點**：controller 端 handleFileLoad 行 1217-1218 連續呼叫兩者，外露為單一 `validate` 符合 controller 視角的「驗證階段」抽象；私有的 `_validateFileBasics` / `_validateFileSize` 保留為內部實作細節（後續若有差異化需求可獨立呼叫）。
- **`read` 單純委派至 `_readFileWithReader`**：簽名與行為一致，僅是「移除底線」的對外契約宣告。
- **`parseContent` 暴露 `fileFormat` 為必填參數**（解決 F16）：現行 `_handleFileContent` 預設 `fileFormat='json'`，controller proxy 行 1493-1496 未傳遞造成 CSV 訊號遺失。新 public API 強制 `fileFormat` 為必填，從型別契約消除遺漏。

**保留現有 `handleFileLoad(file)` public API 不變**：作為「驗證 + 讀檔」端到端的便利方法，內部改為呼叫新 `validate` + `read`（行為等價）。

---

### 二、語意契約

#### `validate(file)`

| 項目 | 內容 |
|------|------|
| 職責 | 完整驗證 file 是否符合匯入前置條件（存在性 + 格式 + 大小） |
| 前置 | 無 |
| 後置（成功） | file 通過所有驗證，可安全進入後續流程 |
| 後置（失敗） | 已呼叫 `showError` 並 throw；caller 收到 Error 後應中止流程 |
| 副作用 | UI showError（失敗時） |
| 對應失敗碼 | `VALIDATION_ERROR` |

#### `read(file)`

| 項目 | 內容 |
|------|------|
| 職責 | 從已驗證的 file 讀取內容、解析、提取 ImportResult |
| 前置 | file 已通過 `validate`（caller 責任；read 不重複驗證） |
| 後置（成功） | resolve 為符合 INV-1 不變式的 `ImportResult` |
| 後置（失敗） | reject 為含 code 的 Error，已呼叫 `showError` |
| 副作用 | FileReader 讀取 + UI showError（失敗時） |
| 對應失敗碼 | `UNKNOWN_ERROR`（reader 故障）/ `PARSE_ERROR`（解析失敗） |

#### `parseContent(content, fileFormat)`

| 項目 | 內容 |
|------|------|
| 職責 | 純函式：將字串內容解析為 ImportResult |
| 前置 | `content` 為字串；`fileFormat` 為 `'json'` 或 `'csv'` |
| 後置（成功） | 回傳符合 INV-1 的 `ImportResult` |
| 後置（失敗） | throw（不呼叫 showError，由 caller 決定 UI 處理） |
| 副作用 | 無（純函式，測試友善） |
| 對應失敗碼 | `PARSE_ERROR` |

> **注意**：`parseContent` 與 `validate` / `read` 的副作用語意不同——`parseContent` 不觸發 showError。此差異反映既有 `_handleFileContent` 行為（無 showError），並讓 `parseContent` 成為**純測試友善**的入口。

### 三、fileFormat 訊號修復（F16）

**問題**：controller `_handleFileContent` proxy（行 1493-1496）呼叫 `this.bookFileImporter._handleFileContent(content)` 未傳 `fileFormat`，CSV 路徑訊號遺失。

**修復路徑**：

1. 新 public `parseContent(content, fileFormat)` 將 `fileFormat` **設為必填**（無預設值）。
2. controller test-only proxy `_handleFileContent` 將於本 ticket 移除（見下節遷移路徑），caller 不再經此路徑。
3. 既有 `handleFileLoad` 流程已透過 `_readFileWithReader` → `_setupReaderHandlers` → `_handleReaderSuccess` 正確傳遞 fileFormat（行 163, 200-202, 213-215），無需改動。

**驗證點**：移除 proxy 後，任何進入 `parseContent` 的 caller 都必須顯式傳遞 fileFormat，否則 ESLint / runtime 立即可見。

#### 遷移風險

1. **行為等價性驗證**：spy on private method 改為呼叫 public API 後，部分測試斷言「方法被呼叫次數 / 順序」需轉為斷言「行為結果」（例如：showError 被呼叫的訊息內容 + throw 的 error.code，而非「`_validateFileBasics` 被呼叫一次」）。
2. **CSV 路徑覆蓋**：原 `_handleFileContent` 預設 json 的測試案例需顯式分流 json vs csv（同時也是 F16 防護的副產品）。
3. **mock importer 的測試**：原以 `jest.spyOn(controller, '_validateFileBasics')` 攔截的測試，若 controller 已直接呼叫 `importer.validate`，需改為 `jest.spyOn(importer, 'validate')` 或 inject 完整 mock importer。

### 十、Phase 3a 實作策略（語言無關）

本 Phase 3a 不寫程式碼，僅產出虛擬碼 / 流程圖 / 步驟順序,作為 Phase 3b（cinnamon-refactor-owl）的執行指引。

#### 1. 實作順序總策略：Producer-First, Consumer-Then, Cleanup-Last

依「依賴方向」與「RED→GREEN 最小化」原則排序：

```
[Stage A] Producer 端（importer）新增 public API
   ↓ （新 API 存在後 consumer 才能呼叫）
[Stage B] Consumer 端（controller）切換呼叫對象
   ↓ （consumer 不再依賴 proxy 後才能刪 proxy）
[Stage C] Cleanup（移除 controller proxy + 遷移測試）
```

**核心理由**：
- Producer 先動 → 新 API 加入時既有測試仍綠（純擴增,無破壞）
- Consumer 後動 → 切換呼叫對象時 producer 已就緒,單點切換可立即驗證
- Cleanup 最後 → 確認所有呼叫已遷移後才刪 proxy,避免 cascade 紅燈

**測試先後策略**：採「**實作先行、測試後遷移**」的 refactor 安全網模式。理由：本 ticket 屬重構（行為保留）,既有測試是 regression 安全網,不能在 Stage A 就改測試（會失去保護）。實作完成後才遷移測試,期間以**既有測試 + Stage A 新增的 contract test** 雙重保護。

#### 2. Stage A：Producer 端新增 Public API（importer）

**目標**：BookFileImporter 暴露 validate / read / parseContent 三個 public method,內部 delegate 至既有底線私有方法。

##### 2.1 新增 validate(file) 虛擬碼

```
method validate(file):
  // delegate 合併兩個既有私有驗證
  this._validateFileBasics(file)   // throws on invalid
  this._validateFileSize(file)     // throws on oversized
  // 無回傳值；失敗時兩個私有方法各自呼 showError + throw
  return
```

##### 2.2 新增 read(file) 虛擬碼

```
method async read(file):
  // 純 delegation,僅去底線
  return this._readFileWithReader(file)
  // _readFileWithReader 內部已正確處理 FileReader + fileFormat + showError
```

##### 2.3 新增 parseContent(content, fileFormat) 虛擬碼

```
method parseContent(content, fileFormat):
  // 強制 fileFormat 必填（F16 修復核心）
  if fileFormat is undefined or null:
    throw Error({ code: PARSE_ERROR, message: 'fileFormat is required' })
  if fileFormat not in ['json', 'csv']:
    throw Error({ code: PARSE_ERROR, message: 'fileFormat must be json or csv' })

  // delegate 至既有私有方法（傳遞 fileFormat,不 fallback）
  return this._handleFileContent(content, fileFormat)
```

**關鍵設計決策**：
- validate / read 為**純 thin wrapper**（零邏輯,僅 delegate）,避免行為飄移
- parseContent 加入 **必填 guard**（pre-condition check）,是唯一含新邏輯處
- 既有底線方法**不改名、不刪除**（Phase 4 再評估）,降低本 ticket 風險

##### 2.4 Stage A 測試先行（contract test）

在動實作前先寫 4 個 contract test（RED）：
- `validate(file)` 存在且 callable
- `read(file)` 回傳 Promise
- `parseContent(c, fmt)` 存在
- `parseContent(c)` 不傳 fileFormat 時 throw

放置於 `book-file-importer.test.js` 新 describe `Public API contract`。這些測試在實作前 RED,實作後 GREEN,作為 Stage A 完成判定。
