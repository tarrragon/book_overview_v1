## Problem Analysis
### W14-016 教訓摘要

W14-004 整併 TERMINAL_STATUSES 為 hook+skill 共用常數時，讓 hook（children_checker.py）直接 `from ticket_system.lib.constants import TERMINAL_STATUSES`。但 `ticket_system.lib.__init__` 會 eager-import 整個 package（ticket_loader -> parser -> yaml），而 hook 以系統 Python 執行（無 yaml）。結果：session 累積 97 次 non-blocking Traceback，污染 PM attention。

**根因**：hook（穩定層、系統 Python）直接依賴 skill 內部 package（功能層、uv 管理 deps），違反層級隔離。

W14-016 修復方案：將 constants 上移至 `ticket_system/constants.py`（package 頂層、無 eager-import 鏈），hook 改用此 canonical path。

### 當前 hook x skill 依賴盤點（2026-05-26）

**生產 hook 對 ticket_system 的直接 import**：

| Hook 檔案 | Import | 風險等級 |

**總計**：僅 1 處生產 hook import ticket_system，且已在 W14-016 修復為安全路徑。

**間接依賴（sys.path 設定）**：多個 hook 透過 `hook_utils` 間接依賴 `.claude/hooks/` 內部模組，但不觸及 skill package 內部。

## Solution
### Analysis Result: Hook-Skill 隔離原則規範

### 架構層級定義

| 層級 | 執行環境 | 依賴管理 | 範例 |
|------|---------|---------|------|
| Hook 層（穩定層） | 系統 Python（無 venv） | 僅 stdlib + hook_utils | `.claude/hooks/*.py` |
| Skill 層（功能層） | uv managed venv | pyproject.toml 宣告 | `.claude/skills/*/` |

### 隔離原則

#### 允許的依賴模式

| 模式 | 條件 | 範例 |
|------|------|------|
| Hook import skill package 頂層純常數模組 | 該模組無第三方 dep、無 eager-import 鏈 | `from ticket_system.constants import TERMINAL_STATUSES` |
| Hook 透過 subprocess 呼叫 skill CLI | 完全隔離，skill 在 uv 環境執行 | `subprocess.run(["uv", "run", "ticket", ...])` |
| Hook 讀取 skill 產出的檔案（JSON/YAML） | 檔案格式穩定、用 stdlib 解析 | `json.load(open(ticket_md_path))` |

#### 禁止的依賴模式

| 模式 | 風險 | W14-016 案例 |
|------|------|-------------|
| Hook import skill 內部子 package（`lib.*`） | __init__ eager-import 觸發第三方 dep 鏈 | `from ticket_system.lib.constants import X` → lib/__init__ → yaml |
| Hook import skill 需第三方 lib 的模組 | 系統 Python 無該 lib | 任何需 yaml/requests/etc 的模組 |
| Hook 依賴 skill __init__.py 的 re-export | __init__ 常做 eager-import | `from ticket_system import X`（若 __init__ import 重模組） |

### 安全判定規則

新增 hook 對 skill 的 import 前，必須驗證：

1. **目標模組是否為純 stdlib**：`grep -n "^import\|^from" <target_module>` 確認無第三方 dep
2. **目標模組的 __init__ 鏈是否觸發 eager-import**：從 import path 逐層檢查每個 `__init__.py`
3. **若不確定，改用 subprocess 呼叫 skill CLI**：完全隔離，零耦合風險

### 升級建議

| 選項 | 優先級 | 理由 |
|------|--------|------|
| 記錄為 error-pattern（PC-XXX） | P2（建議） | W14-016 是具體案例，error-pattern 格式適合記錄根因+防護 |
| 升級為 architecture rule | P3（可選） | 目前僅 1 處生產依賴且已修復，rule 的 ROI 低 |
| 升級為 methodology | 不建議 | 問題範圍太窄，不足以構成方法論 |

**結論**：建議建立 error-pattern（PC-XXX: hook-eager-import-chain-from-skill-package），記錄 W14-004→W14-016 鏈為案例。目前無需升級為 architecture rule（只有 1 處依賴且已安全）。若未來 hook 對 skill 的依賴增加超過 3 處，再升級。
