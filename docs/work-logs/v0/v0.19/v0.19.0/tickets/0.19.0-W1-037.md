### 問題根因

**直接原因**：`ticket create` 分配新 ID 時，序號掃描完全基於 cwd 本地檔案系統，看不到主 repo（main ref）上已存在但尚未 merge 進當前 worktree 的 ticket 檔。

**掃描邏輯定位**：

| 函式 | 檔案 | 行為 |
|------|------|------|
| `get_next_seq(version, wave)` | `lib/ticket_builder.py:206` | `glob("*-W{wave}-*.md")` 掃描 `tickets_dir`，取最大 seq + 1 |
| `_scan_child_files_max_seq(tickets_dir, parent_id)` | `lib/ticket_builder.py:290` | `glob("{parent_id}.*.md")` 掃描子 ticket 檔，取最大子 seq |
| `get_next_child_seq(parent_id)` | `lib/ticket_builder.py:335` | 合併 children 欄位 + 上述檔案掃描結果 |
| `get_tickets_dir(version)` | `lib/paths.py:69` | 回傳 `<project_root>/docs/work-logs/v{M}/v{M.m}/v{version}/tickets/` |
| `get_project_root()` | `lib/paths.py:18` | 優先 `CLAUDE_PROJECT_DIR` → `git rev-parse --show-toplevel` → marker 檔 → cwd |

呼叫鏈：`commands/create.py:_resolve_ticket_id_and_wave()`（行 601-602）與 `commands/generate.py:generate()`（行 216-221）→ `get_next_seq` / `get_next_child_seq`。

**根因鏈**：worktree 從 stale base（W1-035 案例 de8e6261）分叉 → worktree 內 `tickets/` 目錄缺少 main 上的新 ticket → `get_next_seq` glob 掃不到 → 回傳已被佔用的 seq → ID 碰撞。`get_project_root` 透過 `git rev-parse --show-toplevel` 正確解析到 worktree 根，但 worktree 的工作樹內容本身就是 stale，問題不在 root 解析而在「掃描的是工作樹快照而非 main ref」。

### 影響範圍

- `lib/ticket_builder.py`：`get_next_seq`、`_scan_child_files_max_seq`（核心改動點）
- `lib/paths.py`：可能需新增 main-ref 掃描輔助函式
- `commands/create.py`、`commands/generate.py`、`commands/bulk_create.py`：呼叫端，預期介面不變
- `lib/ticket_validator.py` 的 `_detect_duplicate_tickets`：既有的碰撞偵測安全網，與本改動互補

### 相關 Error Pattern

- IMP-072（`ticket create` 並行 race condition，W10-105 撞 W10-107 ID）— 同屬 ID 分配未涵蓋完整來源
- feedback: Worktree Agent Issues 子項 4（stale base 連鎖）— 本 ticket 的 source

### B3 方案跨專案影響與邊界評估（acceptance 1）

設計目標：`get_next_seq` / `_scan_child_files_max_seq` 改為「本地工作樹 glob ∪ `git ls-tree -r main` 路徑列舉」取聯集後再算 max seq。**採聯集而非取代**，是邊界安全的關鍵——下列邊界情境中本地掃描始終是可靠 fallback。

| 邊界情境 | `git ls-tree main` 行為 | 設計對策 |
|---------|------------------------|---------|
| main ref 不存在（新 repo / 預設分支非 main） | 命令 returncode != 0 | 偵測失敗 → 退回純本地掃描；不阻斷 create |
| detached HEAD | `main` 仍可被 `git ls-tree` 解析（ref 名查找，與 HEAD 無關） | 正常運作 |
| 非 git 環境（sync 到無 git 的專案） | `git` 命令不存在 / 非 repo → FileNotFoundError / returncode != 0 | 退回純本地掃描 |
| sync 到其他專案（預設分支為 master / develop） | `main` 不存在 → 失敗 | 退回純本地掃描；**或**設計為依序嘗試 `main`/`master`，仍失敗則純本地 |
| 未 commit 的新 ticket（剛建好還沒進 git） | `git ls-tree main` 看不到未 commit 檔 | 本地 glob 補足；聯集設計確保涵蓋 |
| worktree 內，main 落後於遠端 | `git ls-tree main` 取本地 main，可能仍 stale | 屬本 ticket 範圍外（B3 解決「worktree base stale」，非「本地 main stale vs origin」）；可在 Solution 標示為已知限制 |
| 子 ticket 掃描（`{parent_id}.*.md`） | `git ls-tree -r main` 可列全路徑後以 basename pattern 比對 | `_scan_child_files_max_seq` 同步改造 |

**跨專案風險評估結論（供 PM 決策）**：

- **無重大阻斷性風險**。聯集 + 失敗 fallback 設計使非 git / 無 main 環境降級為現行行為（純本地掃描），不會比現況更差。
- **中等注意點**：預設分支名硬編碼為 `main`。本專案預設分支為 main（已確認），但框架 sync 到 master-based 專案會使 B3 失效（降級為本地掃描）。建議 Solution 採「ref 名可設定 / 依序嘗試 main→master」而非寫死。
- **效能注意點**：每次 `create` 多一次 `git ls-tree` subprocess（約數十 ms），須加 timeout（比照 `GIT_TOPLEVEL_TIMEOUT`），逾時即 fallback。
- B3 不取代 `_detect_duplicate_tickets` 安全網，兩者並存。

### 新掃描行為規格

1. 新增輔助函式（建議置於 `lib/paths.py` 或 `lib/ticket_builder.py`）：`list_ticket_files_from_main(version, ref="main") -> Optional[List[str]]`
   - 執行 `git ls-tree -r --name-only <ref> -- <tickets_dir 相對路徑>`，cwd 設為 project root。
   - 成功：回傳路徑 basename 清單；失敗（returncode != 0 / git 不存在 / timeout）：回傳 `None`。
2. `get_next_seq`：max seq = max(本地 glob 結果, main ref 結果)。main ref 結果為 `None` 時只用本地。
3. `_scan_child_files_max_seq`：同上聯集邏輯，pattern 比對 basename。
4. 既有 seq 解析邏輯（`split("-W")` / `split(".")[0]`）不變，僅資料來源擴增。
5. 介面不變：`get_next_seq` / `get_next_child_seq` 簽名與回傳型別不變，呼叫端零改動。

### 驗收標準（供 Phase 2 測試設計）

- AC1：`main` 存在時，`get_next_seq` 回傳值 = max(本地檔案 seq, main ref 上 seq) + 1。
- AC2：`main` 不存在 / 非 git 環境時，`get_next_seq` 行為等同現行（純本地掃描），不拋例外、不阻斷 create。
- AC3：detached HEAD 下 `get_next_seq` 仍能解析 main ref。
- AC4：未 commit 的本地新 ticket 仍被本地 glob 涵蓋（聯集不遺漏）。
- AC5：stale base worktree 重現情境下，create 分配的 ID 不與 main 上既有 ticket 碰撞（對應 acceptance 3 重現測試）。
- AC6：`git ls-tree` 逾時觸發 fallback，create 不被卡住。
- AC7：子 ticket 序號（`get_next_child_seq`）同樣涵蓋 main ref 上的子 ticket 檔。

### Phase 1 結論

B3 方向經評估**無重大跨專案風險，可進入實作**。唯一需 PM 留意的決策點：預設分支名是否硬編碼 `main`——建議實作時採「main→master 依序嘗試」或可設定，避免框架 sync 到非 main 專案時靜默失效。其餘邊界情境均由「聯集 + 失敗 fallback」設計安全覆蓋。

### PM 決策（2026-05-22）

**分支名策略：main → master 依序嘗試**。`list_ticket_files_from_main` 掃描時先試 `git ls-tree main`，returncode != 0 再試 `master`，兩者皆失敗則 fallback 純本地掃描。Phase 2 測試需涵蓋 master fallback 情境（AC2 延伸），Phase 3b 實作依此。選此策略而非「可設定 ref 名」：涵蓋 main/master 兩大宗預設分支已足夠，且免去設定檔複雜度。

### 設計決策

- **聯集而非取代**：main ref 掃描失敗（非 git / 無 main+master / git 不存在 / ls-tree 逾時）一律回 None，caller fallback 純本地掃描，行為不比現況差（AC2/AC6）。
- **timeout 重用 `GIT_TOPLEVEL_TIMEOUT`（5s）**：比照 paths.py 既有 git 呼叫慣例。
- **異常可觀測（quality-baseline 規則 4）**：git 不存在 / 逾時時 `sys.stderr.write` 輸出 WARNING 後 fallback。
- **`get_project_root` 經由 ticket_builder 模組命名空間引用**：測試以 `monkeypatch.setattr(ticket_builder, "get_project_root", ...)` patch，import 進模組命名空間後 patch 生效。
- **`_parse_root_seq` 新增 wave 過濾**：原 `get_next_seq` 以 `glob("*-W{wave}-*.md")` 隱含過濾 wave；改聯集後 ls-tree 回傳全目錄檔，故解析時顯式比對 wave。

無變數作用域變更（全域→區域或模組級→函式級）。新增模組級 `_MAIN_REF_CANDIDATES` 常數與兩個新函式，既有函式簽名不變，無 helper 漏更新風險。
