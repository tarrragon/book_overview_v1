## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-049 Phase 4 Explore（Coupling+Quality）視角審查發現兩處測試覆蓋與測試隔離缺口：

**Finding #1（焦點還原 graceful fallback 缺測）**：
ImportFlowController.promptImportMode / confirmEmptyFileOverwrite 兩處 settle 內，焦點還原使用 `previousFocus && typeof .focus === 'function'` 守護，但缺少對應 TC 驗證「modal 開啟期間原焦點元素被 DOM 移除」的邊界。實際使用場景：SPA 路由切換、非同步資料載入完成後重新渲染都可能讓原焦點元素脫離 DOM。

**Finding #3（Group F 跨檔 mock 隔離強化）**：
empty-file-confirm-modal.test.js Group F 整合測試 TC 對 TagStorageAdapter 模組屬性（replaceAllData / mergeAllData）做 module-level 覆寫。雖然頂層 beforeEach 第 155 行已 `jest.resetModules()`，但 Group F 內未額外明確清除 TagStorageAdapter mock；理論上 Jest 並行執行下若 require cache 未完全清理，後續 TC 可能讀到前一 TC 的 mock 殘留。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修改摘要

**src/overview/import-flow-controller.js**（2 處 settle）：
- Modal A settle（line ~200-204）：`previousFocus && typeof .focus === 'function'` 改為 `?.focus?.()`
- Modal B settle（line ~340-344）：對稱套用，加註與 Modal A 對稱性說明

**tests/unit/overview/empty-file-confirm-modal.test.js**：
- 新增 TC-C7：modal 開啟前 previousFocus 元素被 `tempElement.remove()` 後 settle 不 throw + activeElement 不為 null
- 新增 Group F describe-level beforeEach：明確 jest.resetModules() + TagStorageAdapter.replaceAllData/mergeAllData mockReset（保險強化頂層 beforeEach）

### 設計決策

| 決策 | 理由 |
|------|------|
| `?.focus?.()` 取代 `&& typeof ===` 守護 | 兩層 optional chaining 同時涵蓋「物件存在」與「方法可呼叫」；語法更簡潔，行為一致 |
| TC-C7 用 `document.createElement('button')` + `remove()` 模擬 | 簡單可控；觸發 jsdom 真實 DOM 移除情境，無需 mock 整個 element 物件 |
| Group F beforeEach try-catch 包覆 mockReset | 首個 TC 執行前 adapter 可能尚未載入（require 在 createController 內才觸發），用 try-catch 容忍不存在 |

### 不在本 ticket 範圍

- Modal A（import-mode-modal.test.js）對稱 TC：W1-049 既有結構未含 focus-restoration 測試，補強對稱性屬獨立議題；本 ticket acceptance 僅要求 empty-file-confirm-modal.test.js 內補測試
