### WRAP 答案（PM 自處理 ANA）

**W（Widen）— 候選方案**：
1. **刪除 DuplicateBookMerger 模組**（dead code 清理）
2. 保留作未來合併功能備用，補對 mergeAllData 的整合測試
3. 將 mergeAllData 內部委派改用 DuplicateBookMerger（統一合併邏輯）

**A（Attain distance）— 機會成本**：
- 刪除：減少 cognitive load、避免「為什麼有此孤兒模組」反復疑問、tests 也是孤兒可一併清
- 保留：增加維護負擔、tests 變孤兒、未來每次重構都要考慮其是否該啟用
- 整合：mergeAllData 已對稱 replaceAllData 設計良好（純函式 computeMergeResult + storage 編排），無需重做

**P（Prepare to be wrong）— 最可能失敗**：
- 假設 mergeAllData 已涵蓋所有合併情境（含 strategy='skip'/'override'/'merge' 三策略）
- 風險點：duplicate-handling-v2.test.js 含的合併規則（20+ 案例）是否在 mergeAllData 都驗證？
- **防護**：spawn IMP ticket 時 acceptance 加「對 mergeAllData 整合測試覆蓋率不低於既有 _handleDuplicateBooks tests 範圍」

**R（Reality Test）— 真根因驗證（grep 證據）**：

### 問題根因

W1-048 Phase 4 評估 F3：DuplicateBookMerger 模組於 OverviewPageController 構造時注入（行 172 `this.duplicateBookMerger = new DuplicateBookMerger()`），controller 定義 `_handleDuplicateBooks` 委派方法（行 1486-1488）。

但 UC-04 新匯入流程（W1-047.5 後）改走 `handleFileLoad` → `TagStorageAdapter.mergeAllData / replaceAllData`，不經 `_handleDuplicateBooks`。grep 證實 `_handleDuplicateBooks` 在 src/ 內**無任何 caller**（僅 tests/unit/overview/duplicate-handling-v2.test.js 呼叫 20+ 次）。
