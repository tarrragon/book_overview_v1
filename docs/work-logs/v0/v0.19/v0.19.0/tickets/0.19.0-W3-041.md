### 前提驗證考古結果（PC-007）

接手時發現 ticket 描述的 `pre-fix-eval hook` 已不存在為獨立檔案，但邏輯仍運作。完整考古：

| 項目 | 結論 |
|------|------|
| 原 hook 檔位置 | `.claude/hooks/pre-fix-evaluation-hook.py`（已刪除） |
| 移除時間 | 2026-04-07 commit `7a22f9f0`（W10-001） |
| 移除原因 | W9 審查清理「刪除 9 個已合併的原始 Hook 檔案」 |
| 邏輯實際位置 | `.claude/hooks/post-test-hook.py` L191-296「子邏輯 2: 測試失敗評估」 |
| 註解標記 | L191 明確標註「來自 pre-fix-evaluation-hook.py」 |

### 問題根因

post-test-hook.py L91-92 的 regex pattern：

```python
(r"(\d+)\s+tests?\s+failed", "{0} 個測試失敗"),
(r"FAILED", "測試失敗"),
```

於 PostToolUse:Edit/Write 觸發。當 thyme 透過 Edit/Write 在 ticket body Test Results 章節寫入「修復場景描述」或「pre-existing skipped 測試說明」含「FAILED」/「N tests failed」字面時，regex 比對命中，誤觸發 pre-fix-eval 子邏輯。

### 影響範圍

- 唯一檔案：`.claude/hooks/post-test-hook.py`（W10-001 合併後唯一實作位置）
- 觸發場景：thyme 寫 ticket Test Results / Solution / Problem Analysis 含測試失敗描述字面
- 衝擊：未阻擋實際任務（hook 為提示性），但產生噪音 + 派 pre-fix-eval skill 引導誤觸發

### 相關 Error Pattern

- PC-007: 接手者獨立驗證 Ticket 描述（本次觸發，發現 hook 改名 + 邏輯合併）
- PC-111: PM narrative fabrication 防護（接手時禁止依 ticket 描述直接派發，必須先驗證）

### 修復方向（給接手 agent）

於 post-test-hook.py 子邏輯 2 加入 ticket body 寫入豁免：

1. 偵測 PostToolUse 的 `tool_input.file_path` 是否屬 ticket md（典型 path: `docs/work-logs/v*/v*/tickets/*.md` 或 `.claude/skills/*/tickets/*.md`）
2. 若是 ticket md → 跳過 pre-fix-eval 子邏輯（不執行 regex 掃描）
3. 確認既有真實測試失敗偵測（npm test / pytest 失敗輸出）不受影響

### 不在本 ticket 範圍

- 清理 `.claude/hooks/__pycache__/pre-fix-evaluation-hook.cpython-311.pyc` 殘留快取
- 更新 `.claude/sync-preserve.yaml` 移除已刪除 hook 的保留路徑項
- 上述兩項應建獨立 DOC ticket 追蹤（quality-baseline 規則 5）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修改摘要

於 `.claude/hooks/post-test-hook.py` 子邏輯 2「測試失敗評估」加入 ticket body 寫入豁免：

1. **新增常數**（L52-69）：
   - `TICKET_WRITE_COMMAND_KEYWORDS`：ticket CLI 寫入子命令清單（append-log / set-acceptance / check-acceptance / complete / claim / create / update）
   - `TICKET_MD_PATH_PATTERNS`：ticket md 路徑 regex（`docs/work-logs/v*/.../tickets/*.md`、`.claude/skills/*/tickets/*.md`）

2. **新增 `_is_ticket_body_write()` helper**（L286-310）：
   - Bash 命令含 ticket CLI 寫入子命令 → True
   - `tool_input.file_path` 匹配 ticket md 路徑 → True（防御性，目前 hook 僅 Bash/mcp__dart__run_tests 註冊）

3. **`evaluate_test_failure()` 入口處早退**（L315-318）：
   - 命中 `_is_ticket_body_write` 即 return None，跳過 regex 掃描
   - logger debug 記錄豁免原因（W3-041）

### 設計決策

- 採「白名單命令 + 路徑樣式」雙重判定，避免單一條件失效
- 不修改 `_classify_errors` regex 本身，保留既有真實測試失敗偵測能力
- `_is_test_command` 仍維持原狀（避免擴大改動範圍，需要時可在後續 ticket 收斂）

### 自檢結果

- 認知負擔指數：低（helper 6 行 + 早退 3 行）
- 函式長度：`_is_ticket_body_write` 16 行（含 docstring），符合 <=30 行
- 巢狀深度：2 層
- 無魔法數字、無重複程式碼
- 命名自說明（`TICKET_WRITE_COMMAND_KEYWORDS` / `_is_ticket_body_write`）
- catch 區塊：無新增 try/except（沿用既有 main 的保護）

### 不在本 ticket 範圍

- 清理 `__pycache__/pre-fix-evaluation-hook.cpython-311.pyc` 殘留（Problem Analysis 已標記獨立 DOC ticket）
- 更新 `.claude/sync-preserve.yaml` 移除已刪除 hook 路徑（同上）
- 收斂 `_is_test_command` 防止 substring 誤判（如 `echo "npm test"`）
