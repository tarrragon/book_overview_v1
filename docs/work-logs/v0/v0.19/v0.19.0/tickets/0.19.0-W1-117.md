## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 採方案 B（mock-based 機制驗證）+ 連帶修復 popup-error-handler.js

#### 改動 1：MessageDictionary.js（W1-106 重做清理）

從 `_loadDefaultMessages()` 移除 5 個 popup-specific key：
- POPUP_INTERFACE_LOADED
- POPUP_SCRIPT_LOADED
- POPUP_INIT_START
- POPUP_INIT_COMPLETE
- INITIALIZATION_COMPLETE

替換為註解說明（移至 popup local dict / popup-error-handler local dict）。

#### 改動 2：tests/integration/chrome-extension/popup-interface.test.js（方案 B 改寫）

原 test case `'無 local dict 第三參數時 popup-specific key 退回 GlobalMessages 解析'`（line 851-863）依賴 production state「GlobalMessages 含 popup-specific 5 key」，與 W1-117 清理目標衝突。

改寫為 `'Logger 不傳第三參數 messages 時應 fallback 到 GlobalMessages 機制'`：
- **驗證對象**：Logger constructor 第三參數省略時走 GlobalMessages 預設值的 fallback 機制本身
- **手法**：jest.spyOn(GlobalMessages, 'get').mockReturnValue('mocked-fallback-message')
- **斷言 1**：GlobalMessages.get 被呼叫，證明 fallback 機制觸發
- **斷言 2**：Logger 輸出採用 GlobalMessages.get 的回傳值
- **防護目標**：未來若 Logger constructor 簽章變更（messages 必填 / 刪除預設值），此 test 立即紅燈
- **獨立性**：即使 GlobalMessages 內容後續又被清理，本 test 仍能獨立驗證 fallback 路徑

#### 改動 3：src/popup/popup-error-handler.js（執行時發現的連帶修復）

執行過程中發現 popup-error-handler.js 的 Logger 不注入 local dict（`new Logger('PopupErrorHandler')`），會在 W1-117 移除 GlobalMessages INITIALIZATION_COMPLETE 後輸出 `[Missing: INITIALIZATION_COMPLETE]`。

採方案 B（模組自包含 local dict）：
- 新增 MessageDictionary import
- 創建 popupErrorHandlerMessages local dict（含 INITIALIZATION_COMPLETE: '初始化流程完成'）
- Logger constructor 改注入：`new Logger('PopupErrorHandler', 'INFO', popupErrorHandlerMessages)`
- 保留 fallback：若 MessageDictionary 未定義（測試環境），仍走無 local dict 構造

此修復屬「重做 W1-106 清理」mandate 範圍內，並非範圍蔓延。
