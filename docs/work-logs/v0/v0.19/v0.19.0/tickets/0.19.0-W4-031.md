## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

本專案（Chrome Extension）與配套 Flutter APP（`/Users/tarragon/Projects/book_overview_app`）需雙向匯入匯出做內容同步，但兩專案各自演進、無共同 interchange contract（SSOT）。我們的格式隨 v0.17+ tag-based model 升級到 v2（含 tags/tagCategories/readingStatus），APP 端 import 模型仍停在最初 `{id,title,cover}` 級別（連 author 為單數）。

> **PM 已蒐集初步證據（2026-06-02，以實作碼為準，非 stale 規格；接手者應獨立複驗——PC-162）**。詳見「重現實驗結果」章節。本 ANA 須在此基礎上完成完整欄位矩陣 + contract 提案。

### 設計意圖與契約目標（用戶澄清 2026-06-02，最高優先框架）

兩端定位**刻意不同**，欄位對不上是多次需求迭代的合理結果，**非 bug**。本 ANA 目標不是「強制兩端欄位相同」，而是設計達成無損互通的 superset interchange contract。

| 端 | 定位 | 欄位特性 |
|----|------|---------|
| APP（Flutter, book_overview_app） | 重端：複雜分類功能 + 額外書籍資料查詢（Google Books enrichment 補作者/書籍資訊） | 欄位豐富（分類體系、ISBN、publisher、rating、enrichment 等） |
| Chrome Extension（本專案） | 輕端：各書城原始資料蒐集為主，簡潔展示，多書城集合書櫃 | 欄位精簡（書城原始資料 + tag-based model） |

**核心需求（不可違反）**：
1. **同一份資料雙向互通且無損**：一端匯出經另一端再匯回，不得遺失任一端的資料（superset 攜帶，非交集裁切）。
2. **Extension 必須能「展示」APP 的分類設定**：即使 Extension 不做複雜分類編輯，匯入 APP 資料時必須 carry + display APP 的分類，禁止 drop。
   - 對照當前缺口：APP importer 丟 Extension 的 tags（Dir1），Extension importer 對 APP 格式判定不明（Dir2）——兩端都未達無損攜帶。
3. **以 APP 實際功能規劃為欄位權威，非 stale SPEC**：APP 原始 SPEC 欄位可能過時，本 ANA 須先確認 APP 端「實際/規劃」的完整欄位集（讀 APP domain Book model + 功能規劃文件 + enrichment 欄位），再設計 contract。

**契約設計方向（待 WRAP 評估）**：superset schema + 各端對未使用欄位採 pass-through 保留（carry-through）+ 版本協商 + 缺欄降級。Extension 對 APP-only 欄位（複雜分類/enrichment）至少「保留 + 唯讀展示」，不要求編輯能力。

#### 假設 #1：APP 匯出 JSON 被我方誤判為 v1（確認的實際互通失敗）

我方 `src/export/format-version-detector.js` `detectFormatVersion` 對 APP `{backup_info|export_info, books:[]}`：Rule 1/2 需 `data.metadata`（APP 用 `backup_info`/`export_info`）→ 跳過；Rule 4（有 `books` 無 formatVersion）→ **回傳 'v1'**。

後果：APP 匯出走 content-parser path A（v1→v2 converter）。v1 路徑從**不存在的** isNew/isFinished 重新推導 readingStatus（**忽略 APP 實際 emit 的 readingStatus**，全降為 unread/notStarted）；APP `tags`(名稱陣列) 與 v1 converter 期望的 `category`/`tags` 映射須複驗；`author`(單) / `coverImageUrl`(vs 我方 `cover`) 欄位名不符 → 大概率遺失。**這是當前實際 round-trip 的資料損壞點，非理論。**

#### 假設 #3：APP CSV import 僅自身格式對稱

APP `data_export_service.dart` `_parseCsvFields` + 12 欄 Book 建構解析自身格式（fields[0..11]，tags split `';'` 無空格，與自身匯出對稱 ✓）。**無法正確解析我方 CSV**（欄位佈局不同 → 欄位錯位：fields[2]=author 拿到我方 authors、fields[3]=isbn 拿到我方 source...）。chrome_extension import 路徑為 JSON-only。

#### 假設 #4：id scheme 不同但雙向保留

- Extension `generateStableBookId(readerId,title,cover)` → readmoo stable id（如 `210327003000101`）
- APP `BookId.generate()` → `book_{timestamp}_{counter}`（自建/實體書）；`fromJson`/CSV import 保留傳入 id
- round-trip：Extension 源書 id 雙向保留 ✓（dedup 前提=匯入不重生 id）；APP 源書（實體書）用不同 scheme，Extension 可 carry 但無法重新推導其 readmoo id。contract 須明定「匯入保留原 id，禁重生」。

#### 止血決策（用戶 2026-06-02）：誤判 v1 缺口併入本 contract，不單獨點修

評估「只修 version-detector 把 APP 判 v2」後確認其為 false-positive fix（PC-165）：v2 path 仍讀我方欄位名（`cover`/`tagIds`/`authors`），APP 的 `coverImageUrl` / 名稱 tags / `author`(單) 仍對不上，只是換一種壞法。真正修復需 APP↔Extension 欄位 adapter（含分類模型雙向映射），與 contract 設計不可分。

**決策**：不現在做任何點修（detector 或 fail-loud 皆不單獨出手），整個 APP↔Extension 匯入互通缺口由本 ANA（W4-031）落地的 contract IMP 統一處理。trigger = 本 ANA contract 提案經用戶拍板後 spawn 的 IMP。

**已知風險（contract 落地前的現狀，明示供知情）**：用戶今日若手動把 APP 匯出 JSON 載入 Extension，會靜默損壞（readingStatus 全變 unread、cover/分類/多數欄位遺失）。此為 latent footgun；W4-031 contract IMP 須含 fail-loud 或 adapter 任一以消除。當前判斷此路徑非 v1.0 活躍流程（v1.0 為 Extension 自身 round-trip，UC-04 已驗），故接受延後至 contract 落地，不另開緊急 IMP。

### PM 補讀成果：APP 完整欄位集 + 雙向對應矩陣（2026-06-02）

**APP `Book` entity（`lib/domains/library/entities/book.dart`）`toJson()` emit 欄位**：
`id / title / author(單數 string) / source(物件 source.toJson()) / status(BookStatus.name，獨立於 readingStatus) / isbn / publisher / publishDate(ms epoch) / description / coverImageUrl / rating(double) / tags(tag 名稱陣列 t.name) / readingStatus(name) / addedDate(ms epoch) / importanceLevel(value) / genre / apiEnriched(1/0) / activeLoan(物件，實體書借閱)`

**雙向欄位對應矩陣（book-level，以實作碼為準）**：

| 概念 | Extension v2 | APP Book.toJson | 落差 |
|------|-------------|-----------------|------|
| id | `id` (string) | `id` (string) | 一致（id 生成規則一致性待驗，假設 #4） |
| 書名 | `title` | `title` | 一致 |
| 作者 | `authors` (陣列) | `author` (單數 string) | 名稱 + 型別不一致 |
| 封面 | `cover` | `coverImageUrl` | 名稱不一致 |
| 書城來源 | `source` (string?) | `source` (物件) | 型別不一致 |
| 閱讀狀態 | `readingStatus` (6 態) | `readingStatus` (name) **+ `status` (BookStatus 另一維)** | APP 多 status 維度 |
| 進度 | `progress` (number) + `progressInfo` | （無；APP 用 reading_info VO，待確認） | Extension-only |
| 分類/tag | `tagIds[]` + root `tags[]` + `tagCategories[]`（正規化圖） | `tags`(名稱陣列) + classification(`importanceLevel`/`category`/`genre`) | **模型根本不同**：正規化 ID 圖 vs 名稱陣列 + 分類 metadata |
