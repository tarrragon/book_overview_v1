## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

本專案（Chrome Extension）與配套 Flutter APP（`/Users/tarragon/Projects/book_overview_app`）需雙向匯入匯出做內容同步，但兩專案各自演進、無共同 interchange contract（SSOT）。我們的格式隨 v0.17+ tag-based model 升級到 v2（含 tags/tagCategories/readingStatus），APP 端 import 模型仍停在最初 `{id,title,cover}` 級別（連 author 為單數）。

> **PM 已蒐集初步證據（2026-06-02，以實作碼為準，非 stale 規格；接手者應獨立複驗——PC-162）**。詳見「重現實驗結果」章節。本 ANA 須在此基礎上完成完整欄位矩陣 + contract 提案。

### 影響範圍

**本專案（book_overview_v1）**：
- `docs/spec/export-interchange-format-v2.md`（approved v2.0.0，我方 SSOT）
- `docs/spec/csv-export-spec.md`（W4-030 剛對齊，tags 分隔符 `; `）
- `src/export/book-data-exporter.js`、`src/overview/import/content-parser.js`（format-version-detector + v1-to-v2-converter）

**配套 APP（book_overview_app）**：
- `lib/domains/import/parsers/chrome_extension_parser.dart`（期望 root `{books:[...]}`）
- `lib/domains/import/value_objects/chrome_extension_book_data.dart`（只取 id/title/author/publisher/cover）
- `lib/infrastructure/export/data_export_service.dart`（JSON `{backup_info/export_info, books}` + CSV 12 欄）
- `docs/spec/import/chrome-extension-import.md`、`docs/spec/export/library-export.md`（皆 stub）
- `docs/app-requirements-spec.md` §JSON匯入格式（stale，範例扁平陣列與實際 parser 矛盾）

### 相關 Error Pattern

- 規格 vs 實作脫鉤（stale spec）：APP app-requirements-spec.md §JSON匯入格式 範例（扁平陣列）與實際 parser（期望 `{books:[]}`）矛盾，故本 ANA 一律以實作碼為準（PC-162 獨立複驗）。
- W4-030（CSV 分隔符 SSOT）為本 ANA 前置脈絡：我方 `; `（有空格）vs APP `;`（無空格）。

## 重現實驗結果

### 實驗方法

讀取兩專案實作碼（非 stale 規格），逐檔比對雙向匯入匯出格式：
- 我方：`docs/spec/export-interchange-format-v2.md`、`src/export/book-data-exporter.js`、`src/overview/import/content-parser.js`
- APP：`chrome_extension_parser.dart`、`chrome_extension_book_data.dart`、`data_export_service.dart`、`app-requirements-spec.md`

### 實驗執行（PM 2026-06-02 已蒐證）

**Direction 1：我方匯出 → APP 匯入**
- APP `chrome_extension_parser.dart:139-160` `_parseJsonInIsolate`：期望 root 為 Map 且含 `books` 陣列（`decoded is! Map` 拋錯、`!containsKey('books')` 拋錯）→ 與我方 v2 root `{metadata,tagCategories,tags,books:[...]}` 結構**相容**（讀得到 books）。
- APP `chrome_extension_book_data.dart:24-35` `fromJson`：僅取 `id / title / author(單數) / publisher / cover|coverUrl`。**未取** readingStatus / tagIds / tags / tagCategories / progress / authors(陣列)。

**Direction 2：APP 匯出 → 我方匯入**
- APP `data_export_service.dart:354-361 / 431` JSON：root `{backup_info|export_info:{version,...,format:'book_overview_app_export'}, books:[book.toJson()]}`（wrapper 非我方 `metadata.formatVersion`）。
- APP `data_export_service.dart:100-120` CSV 12 欄：`id,title,author,isbn,publisher,publishDate,description,coverImageUrl,rating,tags(以 ';' 無空格 join),readingStatus,addedDate`。
- 我方 `content-parser.js:420-460` 接受形狀：v2(`data.books`)/v1/metadata-wrap(`{data:[]}`)/空；APP 的 `{backup_info,books:[]}` 缺 `metadata.formatVersion`，version-detector 實際判定**待複驗**（高風險落 v1 或 path E throw）。

### 實驗發現（已驗證事實）

| 比對項 | 結果 |
|--------|------|
| Dir1 JSON root 結構 | 相容（APP 讀我方 books 陣列） |
| Dir1 per-book 欄位 | 不一致：APP 丟 readingStatus/tagIds/tags/tagCategories/progress；author(單) vs authors(陣列) |
| Dir2 JSON wrapper | 不一致：APP backup_info/export_info vs 我方 metadata；我方 version-detector 對 APP 格式判定待複驗 |
| Dir2 CSV 欄位集 | 不一致：欄位集完全不同 |
| Dir2 CSV tags 分隔符 | 不一致：APP `;`(無空格) vs 我方 `; `(有空格) |
| 雙方規格書 | 皆不可靠：APP import/export spec 為 stub，app-requirements §JSON匯入格式 stale；我方 export-interchange-format-v2 完整 |

**未驗證假設（本 ANA 須補完）**：
1. 我方 version-detector 對 APP `{backup_info,books:[]}` 的實際判定（v1？throw？）
2. APP 領域 `Book.toJson()` 完整欄位清單（本次未讀，需確認 tagIds/tags 是否出現在 JSON 而非僅 CSV）
3. APP CSV importer（若有）是否對稱解析自身 CSV 匯出
4. 兩專案 book id 生成規則是否一致（同步去重的前提）
