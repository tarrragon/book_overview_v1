## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`isUnsafeUrl(url)`（`src/content/adapters/readmoo-adapter.js:695`）以單參數形式呼叫 `new URL(url)`。WHATWG URL 規範要求單參數 `new URL()` 的輸入必須是**絕對 URL**；當 `url` 為相對路徑（例如 `/cover/abc/xyz_210x315.jpg`）時 `new URL()` 會 throw `TypeError`。該例外被 `:715` 的 catch 區塊吞掉並 `return true`（判定為 unsafe），導致所有相對路徑封面 URL 一律被誤判為不安全並清空。

W1-006 量測：96 本書中 78 本封面因此被錯誤過濾為 `UNSAFE_COVER_URL_FILTERED`。

### 影響範圍

- 檔案：`src/content/adapters/readmoo-adapter.js`（本 ticket `where.files` 唯一檔案）
- `isUnsafeUrl()` 共 3 個呼叫點：
  - `:436` `extractCoverAndTitle()` 對 cover `img src` — **本 ticket 修復目標**
  - `:361` `extractReaderLink()` 對 reader link `rawHref` — 潛在同源 bug，需調查
  - `:1017` `validateCoverUrlInput()` 對 cover URL — 潛在同源 bug，需調查
- 測試檔：`tests/unit/adapters/readmoo-adapter.test.js`（新增 RED 測試於此）

### 相關 Error Pattern

無直接對應 PC 編號。屬「URL API 契約誤用」——單參數 `new URL()` 隱含「輸入須為絕對 URL」前提，相對路徑須以 `new URL(url, base)` 雙參數形式解析。執行時可 `/error-pattern query` 再確認。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 修復摘要

`isUnsafeUrl()`（`src/content/adapters/readmoo-adapter.js`）新增選用 `base` 參數，並重新排列檢查順序：

1. **新增 `base` 參數**：`isUnsafeUrl(url, base)`。傳入 `base` 時改用 `new URL(url, base)` 解析，使相對路徑（如 `/cover/abc/xyz.jpg`）能正確解析協議；未傳 `base` 時維持單參數 `new URL(url)`，相對路徑仍 throw 後歸為 unsafe（向後相容舊行為）。
2. **路徑遍歷檢查改對原始輸入字串進行**：原本檢查解析後的 `urlObj.pathname.includes('..')`。但 `new URL(url, base)` 會將 `..` 正規化消除（`/a/../b` -> `/b`），若沿用解析後 pathname 將遺漏遍歷攻擊。改為在解析前對 `url.toLowerCase()` 檢查 `..` 與 `%2e%2e`，安全性反而比原本更強（原本對絕對 URL 也已被 `new URL` 正規化掉 `..`）。
3. **協議白名單不變**：仍只允許 `http:` / `https:`，`javascript:` / `data:` 等仍被擋。

`extractCoverAndTitle()`（`:436` 呼叫點）改為傳入 `getLocation().origin` 作 base。`getLocation()` 為既有輔助函數（`globalThis.location || window?.location || {}`）；若環境無 `location`（回傳 `{}`），`origin` 為 `undefined`，`isUnsafeUrl` 退回無 base 行為（相對路徑判 unsafe），屬可接受降級。

### 設計決策

- **為何用既有 `getLocation()` 而非直接 `location.origin`**：Service Worker 與測試環境 `window` / `globalThis.location` 不一定存在，`getLocation()` 已封裝多層 fallback，沿用一致。
- **為何路徑遍歷檢查對原始字串而非 pathname**：見上述第 2 點，正規化會掩蓋遍歷攻擊。`%2e%2e`（`..` 的 URL 編碼）一併在原始字串攔截。
- **安全性未弱化**：javascript:/data: 協議過濾、路徑遍歷過濾、空值/非字串過濾全部保留並由 RED 測試覆蓋。

### 同源 bug 1：`extractHrefFromElement()` 相對 reader link 誤判

- **觸發位置**: `src/content/adapters/readmoo-adapter.js:361`（`extractHrefFromElement()` 內 `this.isUnsafeUrl(rawHref)`）
- **問題**: `rawHref` 來自 reader link `a[href*="/api/reader/"]` 的 `getAttribute('href')`，可能為相對路徑（如 `/api/reader/210327003000101`）。`isUnsafeUrl(rawHref)` 以單參數呼叫，相對路徑 `new URL()` throw -> 判定 unsafe -> `href` 與 `readerId` 被清空（`return { href: '', readerId: '' }`）。
- **實證**: `new URL('/api/reader/210327003000101')` throw `TypeError`；舊版 `isUnsafeUrl` 對此回傳 `true`。
- **影響**: Readmoo 頁面若以相對路徑提供 reader link，`identifiers.readerLinkId` 會遺失，書籍主 ID 退化為 cover-based / title-based fallback。書籍記錄本身不會被丟棄（privacy ID 與其他 fallback 仍在），但 ID 穩定性下降。
- **建議修復**: 同本 ticket 模式 — `extractHrefFromElement()` 改傳 `getLocation().origin` 作 base：`this.isUnsafeUrl(rawHref, getLocation().origin)`。

### 同源 bug 2：`validateCoverUrlInput()` / `validateReadmooDomain()` 相對 cover URL 誤判

- **觸發位置**: `src/content/adapters/readmoo-adapter.js:1017`（`validateCoverUrlInput()` 內 `this.isUnsafeUrl(trimmed)`）+ 連帶 `:1029` `validateReadmooDomain()` 內單參數 `new URL(url)`
- **問題**: `validateCoverUrlInput()` 以單參數呼叫 `isUnsafeUrl()`，相對 cover URL 被判 `null`。連帶 `extractCoverIdFromUrl()`（`:1062`）對相對 cover URL 回傳 `null`，影響 `hasRequiredFields()`（`:457`）與 `:544` `identifiers.coverId`。`validateReadmooDomain()` 的單參數 `new URL()` 對相對路徑也 throw -> false，coverId 提取被雙重阻擋。
- **影響**: 相對路徑 cover URL 雖在本 ticket 修復後不會被 `extractCoverAndTitle` 清空（封面圖可正常顯示），但 `coverId` 仍無法提取，cover-based ID fallback 對相對路徑封面失效。
- **建議修復**: `validateCoverUrlInput()` 與 `validateReadmooDomain()` 需一併接受 base 參數（或內部使用 `getLocation().origin`）。此修復牽涉 `extractCoverIdFromUrl` 呼叫鏈，範圍較大，建議獨立 ticket 評估。

### 說明

兩處 bug 與本 ticket 同根因（單參數 `new URL()` 對相對路徑 throw），但修復牽涉不同呼叫鏈與測試，依 quality-baseline 規則 5 + scope-expansion 獨立 commit 原則，不在本 ticket 修復。
