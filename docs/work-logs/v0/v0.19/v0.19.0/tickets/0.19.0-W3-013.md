#### 步驟 5：重複次數與類型分析

| 重複類型 | 來源 | 影響 |
|---------|------|------|
| SessionStart hook runqueue × 1-2 | hook 自動執行 | 不計入 PM tool call，但仍消耗 ticket CLI 時間 |
| `/ticket` 流程 runqueue × 1 | SKILL.md step 1 | 與 hook 結果重複（hook output 已含相同資訊但 markdown 字串無法直接 reuse） |
| `/ticket` 流程 list × 1 | SKILL.md step 2 | 與 dashboard 部分重複（dashboard 已含 ready 清單） |

**淨重複（可被 cache 消除部分）**：1 次 runqueue 呼叫（hook 與 `/ticket` step 1 之間）。

### 實驗發現

**結論判定**：當前 SKILL.md `/ticket` 流程 = **3-4 tool call**，落入判斷標準的「3-4（略有重複但可接受）」區間 → **豁免方向**。

**核心發現**：
1. dashboard（W10-114）已將多次查詢聚合為單次呼叫，PM 主動使用 dashboard 時為 2 tool call（豁免區間最佳）
2. SKILL.md `/ticket` 預設流程仍走 runqueue + list 雙呼叫（3-4 tool call），與 hook 有 1 次 runqueue 重複
3. Hook 輸出已含 runqueue 結果但為 rendered markdown，PM 無結構化方式 reuse → 此為「hook output 與 /ticket 斷裂」根因（W10-113 識別），cache 設計可解但成本高
4. spawned pending 與 needs_context 兩個小節已由 hook 一次性掃描提供，PM 不需再查詢

**.claude/cache/ 失效策略成本效益分析**：

| 維度 | 評估 |
|------|------|
| 收益 | 節省 1 次 runqueue 呼叫（hook 與 `/ticket` step 1 之間） |
| 成本 | TTL 設計 + claim/complete/release 事件失效鉤子 + 跨 session 一致性保證 + cache schema 維護 |
| 風險 | false hit（cache 未及時失效顯示舊資料 → PM 接手錯誤 ticket） |
| 替代方案收益更高者 | (a) SKILL.md `/ticket` 預設流程改為直接呼叫 dashboard，無需 cache 即可達 2 tool call；(b) hook 可額外輸出結構化 JSON 至 stdin 供 SKILL.md reuse（需 SKILL.md 與 hook 雙向協議設計，但無 cache 失效風險） |

**結論**：cache 設計成本 > 收益，且有更安全的替代方向（dashboard-first SKILL.md 流程）。豁免本 ANA 進入 IMP，記錄當前 baseline 與替代方向建議。

### 結論：豁免（dashboard 已解決，無需 .claude/cache/ 失效策略）

**不建 IMP 落地 cache 機制**。理由：

1. dashboard 命令（W10-114）已將多次查詢聚合為單次呼叫，PM 主動使用時可達 2 tool call（豁免區間最佳）
2. SKILL.md `/ticket` 預設流程當前為 3-4 tool call（落入「3-4 略有重複但可接受」豁免區間）
3. Cache 設計成本（TTL + 事件失效 + false hit 風險）> 節省收益（1 次 runqueue 呼叫）
4. 存在更安全的替代方向（dashboard-first SKILL.md 流程 / hook stdin 結構化協議），無 cache 失效風險
