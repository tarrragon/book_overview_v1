## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

`popup.js:1363` 註冊 `setInterval(periodicStatusUpdate, 3000)`，內部呼叫 `checkCurrentTab()`，
其中對 Readmoo 頁面成功 ping 時會執行 `updateStatus('就緒', 'Content Script 連線正常', ...)`（popup.js:712），
覆寫 `setupExtractionCompletionListener`（popup.js:771）寫入的「提取成功 N 本書籍」終態。
結果終態僅顯示 < 3 秒就被回復為「就緒」。

### 影響範圍

- `src/popup/popup.js`
  - 模組 scope 變數新增 `isFinalStatus`
  - `setupExtractionCompletionListener` 監聽器內（line 771 鄰）
  - `startExtraction` 成功 response（line 825 鄰）與 catch 區塊（line 841 鄰）
  - `startExtraction` 函式開頭（重置 `isFinalStatus` 旗標）
  - `periodicStatusUpdate`（line 1113-1117）偵測旗標決定 skip checkCurrentTab
  - `window.*` 暴露（測試需可控制旗標）
- `tests/unit/popup/popup-final-status-sticky.test.js`（新增）

### 相關 Error Pattern

無直接對應的 PC error-pattern。本問題屬「UI 終態被 polling 機制覆寫」單一邏輯缺陷，
非反覆出現的 framework-level 模式。

### 設計決策

| 設計選項 | 採用 | 理由 |
|---------|------|------|
| 修改 updateStatus 內判定 status 名稱 | 否 | 耦合終態判定到 updateStatus，所有呼叫者皆受影響，違反 SRP |
| 終態 site 顯式設 isFinalStatus=true | 是 | 三處終態 site 共識明確，週期 polling site 單一守門點 |
| periodicStatusUpdate 完全停止 | 否 | 違反 AC #2「仍持續執行」+ 失去 tabs API 健康訊號 |
| periodicStatusUpdate 跳過 checkCurrentTab + 保留 health probe | 是 | 滿足 AC #2，週期日誌 + tabs.query 仍持續執行 |
| 旗標重置時機：startExtraction 開頭 | 是 | 用戶再次點擊提取，所有 status 應重置回 polling 行為 |

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

在 src/popup/popup.js 引入 `isFinalStatus` 模組 scope 旗標，於三處終態 site 設為 true、startExtraction 開頭重置為 false，並修改 periodicStatusUpdate 在終態時跳過 checkCurrentTab 改執行 chrome.tabs.query 健康探測。
