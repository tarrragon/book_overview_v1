### 設計分析（PM 前台，2026-05-28）

**背景**：W3-075 修復 sync-push 空 commit 後，用戶提問 sync-pull 是否需要對稱防呆。經對稱性分析（見 ticket why），決定不做 hash early-exit，改解決 backup_dir 累積這個真實小痛點。

**既有實作參考**（`.claude/scripts/sync-claude-pull.py` line 532）：
```python
backup_dir = Path(tempfile.mkdtemp(prefix="claude-backup-"))
```
- 位置：`tempfile.gettempdir()` 回傳的系統 temp 目錄
- 命名：`claude-backup-<random8>`（mkdtemp 預設）
- 每次 sync-pull 必建（無 short-circuit）

**清理邏輯**：
1. 取 `tempfile.gettempdir()` 為掃描根
2. 列出該目錄下符合 `claude-backup-*` glob 的子目錄
3. 對每個子目錄取 `mtime`，若 `now - mtime > retention_days` 則刪除
4. 統計刪除數量並輸出

**邊界處理**：
- 系統 temp 不存在或無權限 → log warning 不 abort
- 目錄非 `claude-backup-` 前綴 → 跳過（即使含 `claude` 字串）
- 目錄為符號連結 → 安全起見不刪
- shutil.rmtree 失敗（檔案被佔用）→ log warning 繼續

**整合位置**：
- 在 `_complete_sync` 結束後呼叫，或直接加在 `_finalize_sync` 內
- 不影響 backup_dir 的「本次備份保留」承諾（剛建立的 mtime = now，不會被誤刪）

**測試策略**（pytest，新增 `.claude/scripts/tests/test_sync_claude_pull_cleanup.py`）：
- `test_cleanup_old_backups_removes_expired` — 建 3 個 backup，2 個超期 1 個保留期內，驗證僅 2 個被刪
- `test_cleanup_no_match_prefix` — 建立非 claude-backup- 前綴的 temp dir，驗證不被刪
- `test_cleanup_empty_temp` — 系統 temp 無任何 claude-backup-，驗證 return 0
- `test_cleanup_respects_retention_days_param` — 不同 retention_days 參數影響刪除範圍
