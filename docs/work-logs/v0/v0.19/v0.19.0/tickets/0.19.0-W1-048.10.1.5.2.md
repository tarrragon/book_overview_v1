## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 採用方案 A：公開 `_extractBooksFromData` / `_processBookData` 為 `extractBooksFromData` / `processBookData`（委派模式）

**選擇理由**（沿用 .5.1 已驗證的委派模式）：

| 方案 | 設計成本 | 風險 | 採用？ |
|------|---------|------|--------|
| A: 新增 public delegate（純委派 `_extractBooksFromData` / `_processBookData`） | 最低（src 新增約 30 行 + JSDoc） | 零（內部呼叫鏈 `_handleFileContent → _processBookData → _extractBooksFromData` 完全不變） | 採用 |
| B: 透過 stub helper / dependency injection 注入 extractor | 中（需建立 stub 與 inject 點，與既有 isCSVFile 不一致） | 中（測試與 SUT 介面分歧） | 否 |
| C: 改測 `parseContent` / `handleFileLoad` 副作用 | 高（17 處全改為更高層 entry，斷言重寫） | 中（測試意圖偏離單元職責） | 否 |

方案 A 與 .5.1 `isCSVFile` 採用相同設計，並與既有公開 API（validate / read / parseContent / isCSVFile）設計風格一致。

### 變更摘要

- 新增 public method `extractBooksFromData(data, fileFormat)`（line 194-213），純委派 `_extractBooksFromData`

- 兩處皆含完整 JSDoc 註解（含委派理由、與既有 public API 並列說明、@throws 契約）
- 保留既有 `_extractBooksFromData` / `_processBookData` / `_handleFileContent` 內部呼叫鏈完全不變
- 38 行新增（含 JSDoc + delegate body）

**tests/unit/overview/book-file-importer-import-result.test.js**：

### 不在本 ticket 範圍

- `_extractBooksFromData` / `_processBookData` 私有 method 本身不移除（屬 W1-048.10.1.4 importer migration 範圍）
- `validateRequiredFields` / `filterValidBooks` 等其他底線方法的 public delegate（屬 .5.3 範圍，**明確排除**）
- TC-04 timestamp millisecond race 屬 pre-existing flaky 測試設計問題（`convertV1ToV2Data` 兩次呼叫 1ms 時差導致 timestamp 欄位 `toEqual` 失敗），baseline 已存在相同行為（20% fail 率），非本遷移引入。修復屬 test-assertion-design-rules.md 規則層獨立 ticket 範疇

### 設計考量

- **介面一致性**：與 .5.1 的 `isCSVFile` 同採委派模式，未來 .5.3 可沿用相同 pattern
- **零行為變更**：內部呼叫鏈、錯誤契約、回傳結構（ImportResult INV-1 三欄位恆為陣列）皆不變
- **可觀測性**：JSDoc 標註委派來源與設計脈絡，避免後人誤改私有 method 而漏改 public delegate
