## Problem Analysis
### 問題根因

W14-002 linux 視角指出 stale-check 邏輯散落於多個命令，每處獨立 try/except 降級。本分析盤點實際散落位置與模式差異。

### 影響範圍

#### 盤點：stale-check 消費者一覽（6 處）

| # | 檔案 | 函式/位置 | 呼叫的 staleness API | try/except 降級 | 用途差異 |
|---|------|----------|---------------------|----------------|---------|
| 1 | lifecycle.py:1988-1994 | execute_claim | format_stale_warning | 是（stderr） | claim 前提示，所有等級 |
| 2 | track_query.py:241-248 | query 單一 ticket | format_stale_warning（僅 WARNING/CRITICAL） | 是（stderr） | query 只顯示 >= 14 天 |
| 3 | track_query.py:903-910 | list 命令 | format_stale_list_summary | 是（stderr） | list 顯示數量摘要 |
| 4 | track_runqueue.py:109 | _is_eligible_for_list | is_stale_in_progress | 否（無 try/except） | 篩選 stale in_progress 進入 runqueue |
| 5 | track_runqueue.py:351 | 列表格式化 | is_stale_in_progress | 否（無 try/except） | 顯示 [STALE-IP] tag |
| 6 | track_dashboard.py:114-135 | load_stale_warning | compute_stale_minutes | 否（無 try/except） | dashboard stale 章節 |

#### 額外相關但不同 domain 的 stale 機制

| 檔案 | 用途 | 與 ticket stale-check 關係 |
|------|------|--------------------------|
| resume.py:135-214 | handoff stale 過濾 | 不同 domain（handoff lifecycle），已有 is_handoff_stale 統一判定 |
| handoff_gc.py | 清理 stale handoff | 同上，delegate 至 is_handoff_stale |
| track_stale_list.py | 專用 stale 清單命令 | 直接使用 staleness lib，無散落問題 |
| build-staleness-check-hook.py | build 產物過期 | 完全不同 domain |
| uv-tool-staleness-check-hook.py | uv 工具過期 | 完全不同 domain |

### 現況架構評估

**已有的集中化**：staleness.py 已作為純函式庫，6 個消費者都正確呼叫 lib 函式，**計算邏輯沒有重複**。

**實際散落的是什麼**：
1. **try/except 降級模式**：3 處（#1 #2 #3）各自包裹 try/except + stderr 輸出，模式相同但手寫
2. **顯示閾值篩選**：claim 顯示所有等級，query 只顯示 WARNING+，list 用摘要——這是業務差異，非重複
3. **runqueue/dashboard 無 try/except**：與 #1-3 不一致，但這些是純篩選/格式化，非命令入口

### 相關 Error Pattern

無直接相關 PC。DRY 違反程度輕微（僅 try/except 模板重複 3 處，每處 4-5 行）。

## Solution
### 方案評估

#### 方案 A：Decorator Pattern（函式裝飾器包裹 stale 提示）

```python
@with_stale_check(level_filter=[LEVEL_WARNING, LEVEL_CRITICAL])
def execute_query(args, version):
    ...
```

| 維度 | 評估 |
|------|------|
| 成本 | 中。需設計 decorator 簽名（ticket 從哪個參數取？）、處理 level_filter / format 差異 |
| 收益 | 消除 3 處 try/except 模板重複（共 ~15 行） |
| 風險 | decorator 隱藏控制流，降低可讀性；claim/query/list 的 ticket 取得方式不同（有的從 args，有的從已載入 dict），decorator 需多態適配 |
| 適用性 | 低。三個消費者的 stale 顯示邏輯各不相同（全等級 vs WARNING+ vs 摘要），decorator 需大量參數化，抽象洩漏 |

#### 方案 B：Pre-hook Pattern（命令執行前統一攔截）

```python
# 在命令分發層統一插入 stale check
def dispatch_command(cmd, args):
    pre_check_stale(cmd, args)
    cmd.execute(args)
```

| 維度 | 評估 |
|------|------|
| 成本 | 高。需重構命令分發架構，目前各命令是獨立函式非 Command 物件 |
| 收益 | 同方案 A，消除 ~15 行重複 |
| 風險 | 架構侵入性大；ticket CLI 命令分發目前是 argparse subparser 直接綁函式，改 pre-hook 需引入命令物件或中間層 |
| 適用性 | 低。投入遠超收益 |

#### 方案 C：Helper 函式提取（最小改動）

```python
def safe_stale_check(ticket, formatter, label="stale"):
    try:
        msg = formatter(ticket)
        if msg:
            print(msg)
    except Exception as exc:
        sys.stderr.write(f"[staleness] {label} 檢查異常：{exc}\n")
```

| 維度 | 評估 |
|------|------|
| 成本 | 極低。在 staleness.py 新增一個 helper，3 處呼叫者各改 1 行 |
| 收益 | 消除 try/except 模板重複，統一 stderr 輸出格式 |
| 風險 | 極低。不改架構，只提取重複模板 |
| 適用性 | 高。精準解決實際問題（try/except 重複），不過度抽象 |

#### 方案 D：不重構（維持現狀）

| 維度 | 評估 |
|------|------|
| 成本 | 零 |
| 收益 | 零 |
| 風險 | 零。現有 3 處重複各 4-5 行，總量 ~15 行，維護成本可忽略 |
| 適用性 | 合理。staleness.py 已是良好的 lib 抽象，消費者差異是業務需求而非架構問題 |

### 結論

**推薦方案 D（不重構），備選方案 C。**

理由：
1. **原始問題被高估**：W14-002 linux 視角指出「stale-check 邏輯散落」，但盤點後發現 staleness.py 已集中化計算邏輯，散落的僅是 3 處 try/except 降級模板（每處 4-5 行）
2. **業務差異不是重複**：claim（全等級）、query（WARNING+）、list（摘要）的顯示邏輯差異是業務需求，強行統一反而引入不必要的參數化複雜度
3. **投入產出比**：方案 A/B 的架構改動成本遠超 ~15 行模板重複的維護成本；方案 C 可行但收益僅消除 15 行重複，改善幅度不值得建 IMP ticket
4. **runqueue/dashboard 無 try/except 是合理的**：這些是內部篩選/格式化，不面對用戶命令入口，降級需求不同

### Spawn 規劃

無需建 follow-up ticket。結論為「不重構」，理由充分。若未來 stale-check 消費者增加到 6+ 處且 try/except 模式持續重複，可重新評估方案 C。
