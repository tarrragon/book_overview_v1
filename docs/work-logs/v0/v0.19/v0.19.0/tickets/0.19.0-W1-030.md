### 問題根因

W1-001.2 實機驗證（read.readmoo.com/#/library，Vue SPA hash 路由）確認：
- 用戶帳號書庫共 944 本，extension 單次提取僅取得 96 本（涵蓋率約 10%）。
- 根因：Readmoo library 為 Vue SPA 虛擬捲動（virtual scroll）lazy-load，DOM 任一時刻僅渲染約 96 個書籍項目；尚未捲動到的書籍未進入 DOM。
- 提取邏輯（book-data-extractor.js）對「當下 DOM 內的書籍元素」做 selector 查詢，因此只取到已渲染的 96 本。

來源鏈：ANA W1-028（已關閉，「extension 全面故障」為建立前提錯誤，已澄清）→ spawn W1-030。
W1-028/029 已確認正確提取頁為 read.readmoo.com/#/library，extension 於該頁可正常運作（96 本均帶 readingStatus + tagIds + cover，W1-009/W1-010 修復已驗證）。

### 影響範圍

src/content/ 提取 pipeline 3 檔（where.files）：
- content-modular.js：content script 提取流程協調入口
- adapters/readmoo-adapter.js：Readmoo DOM selector 與書籍元素定位
- extractors/book-data-extractor.js：實際逐本書籍資料抽取

功能影響：所有內測使用者書庫提取嚴重不完整，違反 v1.0 內測「核心功能端到端可運作」要求。

### 實機取證結果（2026-05-22，PM chrome-devtools-mcp，§8 Phase 3a 前置完成）

於用戶登入的 read.readmoo.com/#/library（書庫 944 本）實機探測，§8 四項全數取證：

1. 捲動容器：document/window（html 元素捲動）。.library-item 往上 12 層祖先全部 overflowY:visible 不可捲動；唯 html 元素 scrollHeight(4647) > clientHeight(761) 可捲動，無內層 scroll 容器。→ 規格 §5 策略 4 document（window.scrollTo(0, documentElement.scrollHeight)）為實際有效路徑；策略 1 selector / 策略 3 scrollable-ancestor 在此頁找不到容器會 fallthrough 至策略 4。thyme F2「window.scrollTo 對內層容器無效」風險在此頁不成立。

2. 載入更多按鈕：存在。初始 button.btn-outline-primary 文字「更多...」；首次載入後文字變為「載入更多...」；全部載入完成（928）後按鈕消失。→ 規格 §5 策略 2 load-more-button 可行。按鈕選擇器建議：button.btn-outline-primary + 文字含「更多」（需同時匹配「更多...」與「載入更多...」）。新增 stop 訊號：按鈕消失 = 全部載入完成，可作 stopReason 判定輔助。

3. DOM 累積 vs 回收：累積式確認。計數單調遞增 96→192→288→384→480→576→672→768→864→928，全部 item 留存 DOM，docScrollHeight 同步增長至約 27000+。無視窗回收，每批 +96 本。→ 規格 D1 隱性前提「累積式」實機證實；unique ID Set 仍保留作未來改版防護。達 928 後續捲動計數穩定（928→928→928）。

4. header 文字格式：div.item-list-state.container-fluid 文字「擁有 944 本書，其中封存 15 本，借出 1 本」。→ parseLibraryTotal：可見書目數 = 944 - 15 - 1 = 928。regex 建議：擁有 (數字) 本書，其中封存 (數字) 本，借出 (數字) 本。expectedTotal = 928，與規格 §1 術語一致。

框架更正：DOM 容器 id 為 react-container，此頁為 React SPA 非 Vue（規格/ANA 沿用「Vue SPA」術語不精確，惟虛擬捲動行為已完整刻畫，不影響設計）。

載入機制總結（Phase 3a 設計輸入）：捲動到底與點「載入更多」按鈕皆觸發批次載入（每批 +96）；約需 10 批達 928；每批在約 2.5s 內完成；達可見書目數後按鈕消失、計數穩定。規格三停止訊號（達 expectedTotal / 連續穩定 / 按鈕消失）在此頁可互相佐證。

### Phase 1 功能規格（lavender-interface-designer）

**產出文件**：`docs/work-logs/v0/v0.19/v0.19.0/tickets/0.19.0-W1-030-feature-spec.md`（Full 模式，6 區段）

**規格摘要**：

提取前自動捲動載入全部書籍，將 docs/bookstores/readmoo.md 虛擬 scroll 章節（W6-021 ANA 結論）的演算法落地到 production extraction code。

核心 API（新增於 src/content/adapters/readmoo-adapter.js）：
- loadAllBooksLazy(scrollOptions) -> Promise<LoadAllBooksResult>：捲動主迴圈，永遠 resolve 不 reject。
- findScrollContainer()：捲動容器辨識，4 層 fallback（指定 selector -> 可捲動祖先 -> document/window -> none）。
- parseLibraryTotal()：從 header「擁有 N 本書，其中封存 X 本，借出 Y 本」解析可見書目總數。

LoadAllBooksResult 含 loadedCount / expectedTotal / coverageComplete / missingCount / stopReason / iterations / durationMs。

**防無限迴圈（acceptance #3）**：4 個獨立停止條件，任一成立即停止——
(a) loadedCount >= expectedTotal；(b) 連續 stableRounds（預設 3）輪不變；
(c) iterations >= maxIterations（預設 30）；(d) 累計耗時 >= overallTimeoutMs（預設 60000ms）。
(c)(d) 為獨立於 DOM 行為的硬上限，DOM 永不穩定亦保證終止。

**涵蓋率回報契約（acceptance #2）**：結束輸出 SCROLL_LOAD_COMPLETED info log；不一致時額外 COVERAGE_INCOMPLETE warn log（含 missingCount + 停止原因）。

**失敗降級**：捲動容器找不到 -> stopReason=container_not_found，降級為現行提取行為 + SCROLL_CONTAINER_NOT_FOUND warn log，提取不整體失敗。

**整合點（acceptance #1）**：
- readmoo-adapter.js：新增 3 方法 + extractAllBooks 於 waitForBookElements 前呼叫 loadAllBooksLazy；SELECTORS 加捲動容器/header 選擇器。extractAllBooks 回傳型別不變（Promise<Object[]>），呼叫者相容。
- book-data-extractor.js：performActualExtraction 進度回報串接捲動載入階段。
- content-modular.js：初版可不動（捲動邏輯由 extractAllBooks 內部承載），確認無回歸即可。

**GWT 場景**：9 個（正常 3 / 邊界 3 / 異常 3）。

**Acceptance**：11 條可驗證條件，1:1 對應 ticket frontmatter 4 條 AC。

**spec validate**：結構檢查通過；語義推演 3 維度無阻擋性未回答問題（Q3 renderWaitMs vs 慢網路標記 Phase 2 以 fake timer 驗證，非規格缺口）。規格可進入 Phase 2。

### Phase 1 iteration 修訂（2026-05-22，依 4 視角審查 R1-R10）

**修訂標的**：`0.19.0-W1-030-feature-spec.md`。性質為整合修訂（thyme 已以 W6-012 實機證據確認 virtual scroll 為累積式，核心策略成立），非重寫。修訂後規格新增 §7 設計決策記錄、§8 實機取證需求、§9 修訂記錄三章。

**R1-R10 逐條處理**：

- **R1（重要，設計決策）**：採 design-decision-framework 權衡（§7 D1）。決策採方案 B——`loadedCount` 改用累積 unique book ID Set 大小，取代「當下 DOM .library-item 數」。理由：方案 A（DOM count）隱含「累積式虛擬捲動」未標示前提，Readmoo 改版啟用 DOM 回收即靜默失準；方案 B 對回收式天然免疫、天然去重、直接支撐 AC#4。代價（每輪一次 ID 解析）相對收益偏低，且 ID 解析邏輯已存在可重用（Explore 清單），非盲採 thyme 建議。

- **R2（重要）**：§5 捲動容器策略表新增優先級 2「load-more-button」（點擊「更多...」按鈕，readmoo.md 明列為官方載入路徑）；§4 新增「捲動有效性探測」——比對捲動前後 scrollTop，無效則早期降級下一 fallback，避免耗盡 maxIterations 才發現策略失效。`findScrollContainer` strategy 枚舉新增 `load-more-button`。

- **R3（中，設計決策）**：採 design-decision-framework 逐項權衡（§7 D2/D3/D4），未盲套 linux 簡化。D2：拒絕刪 `overallTimeoutMs`——maxIterations 只限輪數不限單輪耗時，MutationObserver 永不觸發時單輪可無限 hang，需獨立硬上限，不冗餘。D3：維持 `reached_total`/`already_complete` 區分——兩者語義（有無實際捲動）對涵蓋率診斷有操作意義。D4：採 linux 建議移除 `skipWhenComplete`——功能與停止條件 (a) 首次量測判定完全重疊，5 參數收斂為 4。

#### 1. 三項 Phase 3a 待釐清回應（Phase 2 §7）

**待釐清 1：注入介面具體形態 — 採「抽私有方法 + 測試 spyOn」**

| 方案 | 評估 |
|------|------|
| A. 建構參數注入（adapter 工廠帶 deps） | adapter 為既有物件字面值單例，改建構簽名侵入大、影響所有既有測試 |
| B. 方法參數注入（loadAllBooksLazy 收 scrollFn/measureFn） | 污染公開 API 簽名；正式呼叫端需傳 production 實作，違反「呼叫者相容」 |
| C. 抽私有方法 + 測試 spyOn（採用） | 與既有 findScrollContainer / parseLibraryTotal / waitForBookElements 的 spy 慣例一致；零公開簽名變更；最小侵入 |

決策：採方案 C。loadAllBooksLazy 內部所有 DOM 接觸點抽為 adapter 私有方法，Phase 2 測試以 jest.spyOn(adapter, '_xxx') 注入序列。抽出的私有方法清單：

| 私有方法 | 職責 | 回傳 |
| `_scrollStep(container, strategy)` | 執行單輪捲動動作（依 strategy 分派：捲容器 / 點按鈕 / 捲 document） | void（同步，可拋例外） |
| `_measureBooks()` | 查當下 DOM 全部 .library-item，逐一解析 book ID | string[]（當輪 book ID 陣列，可含重複） |
| `_getScrollPosition(container, strategy)` | 讀當下捲動位移（容器 scrollTop 或 window scrollY） | number |
| `_waitForRender(container, renderWaitMs)` | 混合等待：MutationObserver 提早結束 + renderWaitMs 上限 | Promise<void> |

findScrollContainer / parseLibraryTotal 已是規格定義的私有輔助，同樣以 spyOn 注入。_measureBooks 內部沿用既有 extractBookId / parseBookElement 的 ID 解析邏輯（Phase 2 §1.2 明列「不 Mock，使用真實」），故 _measureBooks 本身被 spy 時直接回傳注入序列，不觸及真實解析。

**待釐清 2：TC-7.6 N-X-Y 負數邊界 — 回傳 null（視為解析異常）**

| 方案 | 評估 |
|------|------|
| 夾為 Math.max(0, N-X-Y) | 把異常資料偽裝成「0 本可見」，與場景 9「真正空書庫」語義混淆；下游無法區分 |
| 回傳 null（採用） | N-X-Y < 0 代表 header 文字與假設模型矛盾（封存+借出 > 總數不可能），屬「格式不符預期」，與 TC-7.5 同類；回 null 觸發場景 6 依穩定條件停止，保守且不無限捲動 |

決策：parseLibraryTotal 計算 N-X-Y 結果 < 0 時回傳 total=null。理由：負數是邏輯矛盾資料，非合法數量；回 null 使其與「格式變更無法解析」共用場景 6 降級路徑，行為一致且可預測。Phase 3b 須在 spec §2 parseLibraryTotal 註解補一行：「N-X-Y < 0 視為解析異常，回 null」。Phase 2 TC-7.6 斷言對象更新為 total=null。

#### 2. 實作策略：loadAllBooksLazy 主迴圈虛擬碼

```
函式 loadAllBooksLazy(scrollOptions):
    參數預設套用 → maxIterations=30, stableRounds=3, renderWaitMs=800, overallTimeoutMs=60000
    startTime ← 現在時刻
    bookIdSet ← 空 Set                      // 累積 unique ID（D1）
    iterations ← 0, stopReason ← null
    stableCount ← 0, prevCount ← -1

    // 階段 A：前置量測（不在 try 內的純查詢可拋例外，整體仍 try 包覆）
    嘗試:
        expectedTotal ← parseLibraryTotal().total          // number | null
        {container, strategy} ← findScrollContainer()
        startScrollPos ← strategy='none' ? 0 : _getScrollPosition(container, strategy)

        // 容器找不到 → 降級
        若 strategy='none':
            loadedCount ← _measureBooks() 解析後加入 bookIdSet → bookIdSet.size
            stopReason ← 'container_not_found'
            跳至 階段 C

        // 首次量測（iteration=0）
        當輪IDs ← _measureBooks(); 全部加入 bookIdSet
        loadedCount ← bookIdSet.size

        // 停止條件 (a) 於 iteration=0 → already_complete
        若 expectedTotal 非 null 且 loadedCount >= expectedTotal:
            stopReason ← 'already_complete'
            跳至 階段 C

        // 階段 B：捲動迴圈
        重複:
            // 進入前先檢 (c)(d) 硬上限（保證即使 DOM 永不變仍終止）
            若 iterations >= maxIterations: stopReason ← 'max_iterations'; 跳出
            若 (現在 - startTime) >= overallTimeoutMs: stopReason ← 'timeout'; 跳出

            posBefore ← _getScrollPosition(container, strategy)
            _scrollStep(container, strategy)               // 捲動 / 點按鈕（可拋例外）
            等待 _waitForRender(container, renderWaitMs)    // MutationObserver | renderWaitMs 上限
            iterations ← iterations + 1

            當輪IDs ← _measureBooks(); 全部加入 bookIdSet
            loadedCount ← bookIdSet.size
            posAfter ← _getScrollPosition(container, strategy)

            // 捲動有效性探測（R2）：位置未動 且 數量未增 → 當前策略無效
            若 posAfter == posBefore 且 loadedCount == prevCount:
                若 還有下一順位 fallback 策略: 切換 strategy（重置 stableCount）; continue
                否則: stopReason ← 'container_not_found'; 跳出

            // 停止條件 (a)
            若 expectedTotal 非 null 且 loadedCount >= expectedTotal:
                stopReason ← 'reached_total'; 跳出
            // 停止條件 (b)
            若 loadedCount == prevCount:
                stableCount ← stableCount + 1
                若 stableCount >= stableRounds: stopReason ← 'count_stable'; 跳出
            否則:
                stableCount ← 0
            prevCount ← loadedCount
        結束重複

    捕捉 例外 err:
        stopReason ← 'error'
        logger.error('SCROLL_LOAD_ERROR', {error: err.message, component: 'ReadmooAdapter'})

    最終 finally:
        嘗試 還原捲動位置至 startScrollPos    // best-effort（D5）
        捕捉 → logger.debug('SCROLL_RESTORE_FAILED', {...})

    // 階段 C：組裝結果 + 日誌契約
    loadedCount ← bookIdSet.size（container_not_found 降級時為當下解析數）
    coverageComplete ← (expectedTotal 非 null) 且 (loadedCount >= expectedTotal)
    missingCount ← (expectedTotal 非 null) ? max(0, expectedTotal - loadedCount) : 0
    durationMs ← 現在 - startTime
    result ← {loadedCount, expectedTotal, coverageComplete, missingCount, stopReason, iterations, durationMs}

    logger.info('SCROLL_LOAD_COMPLETED', result 的 6 個欄位)
    若 coverageComplete == false:
        logger.warn('COVERAGE_INCOMPLETE', {missingCount, stopReason, reasonText: 可讀說明(stopReason)})
    若 stopReason == 'container_not_found':
        logger.warn('SCROLL_CONTAINER_NOT_FOUND', {...})

    回傳 result      // 永遠 resolve，永不 reject
```

關鍵設計點：
- 整個階段 A+B 包在單一 try，所有 DOM 接觸點（parseLibraryTotal / findScrollContainer / _scrollStep / _measureBooks / _getScrollPosition）拋例外皆落入 catch → stopReason='error'，bookIdSet 保留已累積值（對應 TC-4.2/4.3）。
- (c)(d) 檢查置於迴圈體最前端，確保 DOM 永不穩定（量測序列無限微增）時 iterations 與 durationMs 仍能觸發跳出（對應 TC-2.1/2.2/2.3/2.6 終止性）。
- 單輪 hang 的兜底：_waitForRender 內 MutationObserver 永不觸發時靠 renderWaitMs 上限結束本輪；若連 renderWaitMs setTimeout 都因環境異常 hang，overallTimeoutMs 在下一輪入口判定——這是 D2 保留 overallTimeoutMs 的設計依據。
- already_complete 與 container_not_found 不進迴圈，iterations=0（對應 stopReason 表）。

#### 3. 資料結構選擇

| 結構 | 選擇 | 理由 |
| 累積去重 | Set<string>（book ID） | O(1) 查重；天然去重（TC-5.1/5.2）；對 DOM 回收式免疫（TC-5.3）；size 直接為 loadedCount |
| LoadAllBooksResult | 純物件（7 欄位） | 規格 §2 定義；無方法，純資料載體 |

#### 4. 控制流程圖（停止條件判定）

```
迴圈每輪入口
   │
   ├─ iterations >= maxIterations ? ──是──→ max_iterations ─┐
   ├─ 已耗時 >= overallTimeoutMs ? ──是──→ timeout ────────┤
   │  否                                                     │
   ▼                                                          │
 捲動 → 等待渲染 → 量測（加入 Set）→ 探測捲動有效性          │
   │                                                          │
   ├─ 位置未動 且 count 未增 ? ─是→ 切換策略 / 耗盡→container_not_found
   │  否                                                     │
   ├─ loadedCount >= expectedTotal ? ─是→ reached_total ─────┤
   ├─ loadedCount == prevCount ? ─是→ stableCount++          │
   │     └─ stableCount >= stableRounds ? ─是→ count_stable ─┤
   │  否 → stableCount=0                                      │
   ▼                                                          │
 下一輪 ──────────────────────────────────────────────────────┘
                                                              ▼
                                              finally 還原捲動位置 → 組裝結果 → 日誌
```

#### 5. findScrollContainer / parseLibraryTotal / _waitForRender 策略

**findScrollContainer**（依序，命中即回）：
```
策略 selector:            document.querySelector(SELECTORS.scrollContainer)
                          → 取證確認 Readmoo 此頁無內層容器，預期 fallthrough
策略 load-more-button:    document.querySelector('button.btn-outline-primary')
                          且 textContent 含「更多」→ 命中回 {container:按鈕, strategy:'load-more-button'}
策略 scrollable-ancestor: 從任一 .library-item 向上找 scrollHeight>clientHeight 的祖先
策略 document:            document.scrollingElement || document.documentElement（恆有 → 取證確認此為實際路徑）
策略 none:                以上皆無（含無 .library-item 的空頁）
```
注意：_scrollStep 依 strategy 分派——load-more-button → container.click()；其餘 → window.scrollTo(0, documentElement.scrollHeight)（document）或 container.scrollTop=scrollHeight（內層容器）。

**parseLibraryTotal**：
```
el ← document.querySelector('.item-list-state')
若 el 不存在: 回 {total:null, raw:''}
raw ← el.textContent
m ← raw.match(/擁有\s*(\d+)\s*本書(?:，其中封存\s*(\d+)\s*本)?(?:，借出\s*(\d+)\s*本)?/)
若 m 為 null: 回 {total:null, raw}
N ← Number(m[1]); X ← Number(m[2]||0); Y ← Number(m[3]||0)
visible ← N - X - Y
回 {total: visible < 0 ? null : visible, raw}      // 負數視為解析異常（待釐清 2）
```

**_waitForRender(container, renderWaitMs)**（混合等待，R4）：
```
回傳 Promise:
    settled ← false
    observer ← MutationObserver 監聽 container 子節點新增:
        偵測到新 .library-item → settled=true → disconnect → resolve
    若 MutationObserver 不可用（jsdom 部分環境）: 略過 observer
    timer ← setTimeout(renderWaitMs): 若未 settled → disconnect → resolve
```
Phase 2 以 fake timer + mock MutationObserver（提供 triggerMutation）驗證（TC-9.1/9.2/9.3）。

#### 8. 語言特定實作注意事項（JavaScript / Chrome Extension Content Script）

- adapter 為物件字面值單例，方法間以 this. 互呼；_scrollStep 等私有方法掛在 adapter 上（非閉包函式），確保 jest.spyOn(adapter, '_xxx') 可攔截。
