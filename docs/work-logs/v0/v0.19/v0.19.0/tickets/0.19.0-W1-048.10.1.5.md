### 問題根因

W1-048.10.1.4 thyme 在 Phase 3b STOP 識別規格盲點：BookFileImporter 移除底線方法的 migration 與既有 3 個 scope 外測試檔 51 處底線方法直呼直接衝突。pepper Phase 3a 未掃描既有測試使用情境，sage Phase 2 假設「已用 public API」未實證——屬規則 6 失敗案例學習實例。

### 影響範圍

- tests/unit/overview/book-file-importer.test.js（`_isCSVFile` ×4）
- tests/unit/overview/book-file-importer-import-result.test.js（`_extractBooksFromData` ×14 + `_processBookData` ×3，共 17 處）
- tests/unit/overview/book-file-importer-version-detection.test.js（`_extractBooksFromData` ×26 + `_validateRequiredFields` ×3 + `_filterValidBooks` ×1，共 30 處）

實測對齊 frontmatter 51 處統計。

### 相關 Error Pattern

- 規則 6 失敗案例學習（quality-baseline.md）
- PC-040 Context 寫入 Ticket 而非 prompt

### 拆分決策（dispatch-readiness 軟警告應對）

dispatch-readiness 顯示 acceptance 4 > 2 軟警告。經 PM 覆核確認真實職責數 = 3 檔遷移 + 1 驗證 = 4 個獨立可派發單元，符合「拆分降低 cognitive load 且利於並行」。**拆分方案 A（依檔案）已採納**：

### WRAP 三問（認領 Checkpoint）

- **W（Widen）候選方案**：(a) 單一 thyme 一次性派發 51 處；(b) 拆 3 子 ticket 依檔案【採用】；(c) 先讀 SSOT 工廠再決定。(b) 勝出理由：降低單一 agent cognitive load + 利於模式先驗證再並行 + 與 .5.4 驗證單元分離避免實作與驗證混在同一 commit。
- **A（Attain distance）機會成本**：擠壓 W1-001.3「彙整 W1-001.2 實機驗證結果並建立 follow-up tickets」的處理時程。用戶若需切換可中斷此 ticket（parent in_progress 可釋放）。
- **P（Prepare to be wrong）最可能失敗原因**：SSOT 工廠 `tests/helpers/import-stub-helpers.js` 未涵蓋全部 5 個底線方法的對應 public path（例如 `_isCSVFile` 可能無對應 public API），導致 .5.1 模式驗證即失敗。**防護**：.5.1 排在最前先做 4 處（最小 blast radius），若失敗則停 .5.2/.5.3 派發、回頭設計 stub helper API。
