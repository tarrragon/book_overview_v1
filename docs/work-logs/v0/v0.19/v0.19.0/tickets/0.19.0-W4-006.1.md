### 實驗發現

**已驗證的根因（高信心）**：

1. **跨 classic script 同名 `let X` + `var X` 在同一 HTML page 是 SyntaxError**：
   - popup-error-handler.js 第 3 行 `let Logger, ErrorCodes, MessageDictionary` 將 Logger 放入 script 共享的 global lexical record
   - popup-diagnostic-enhancer.js 第 6-19 行的 `if (typeof Logger === 'undefined') var Logger = ...` 在 **parse 階段就 SyntaxError**（不是 runtime）
   - 關鍵：`typeof X` runtime guard 救不了 parse-time SyntaxError——SyntaxError 在 JS engine 看見 `var Logger` token 時即拋
   - popup-initialization-tracker.js 同樣模式，同樣 SyntaxError

2. **連鎖效應**：
   - tracker.js parse 失敗 → 整檔不執行 → 不定義 `PopupInitializationTracker` class
   - popup.js IIFE 正常執行（內部 var Logger 在 IIFE scope，不衝突），但呼叫 `new PopupInitializationTracker()`（line 1645）時 ReferenceError
   - popup.js 主要功能（startExtraction / checkCurrentTab / 點 extractBtn）**仍可運作**——因為這些不依賴 tracker

3. **esbuild bundle 行為**（驗證之前推測）：
   - build.js BUNDLE_ENTRY_POINTS 只列 popup.js 一個 popup 入口（line 422-425）
   - 其他三檔（error-handler / diagnostic-enhancer / initialization-tracker）走 `copyRecursive`（line 371-402），保留原始檔
   - 結果：popup.js 被 wrap 為 IIFE，但其他三檔保留 classic script + 頂層 let/var 宣告

**未驗證但高可信度的推論**：

- W1-117 將 popup-specific message keys 從 GlobalMessages 移除時，把 popup-error-handler.js 的 `var Logger` typeof guard 改為 `let Logger`（為了支援 local dict 模組獨立性）。這個改動讓 Logger 從「var with typeof guard 共享 hoisting」變為「let 進入 script global lexical record」，觸發本問題。
- 即無此問題長期存在於 production——若用戶從 Chrome Extensions 介面安裝 unpacked extension 並打開 popup，也會看到同樣三筆 pageerror。
- W4-006 commit 86d17db0 的 integration test 用 `eval(popupScript) + setupEventListeners()` 直接模擬 popup.js 載入，繞過了 popup.html 的多檔載入順序，所以 integration test 不會撞到此問題（屬合理的 test scope decision，但 latent bug 被遮蔽）。

**ESM 方案可行性確認**：

- manifest.json line 13-16 `background.service_worker + "type": "module"` 證實 Chrome Extension Manifest V3 已支援 ESM SW
- Chrome `target: chrome100`（build.js line 456）支援 `<script type="module">`（自 Chrome 89 起即 stable）
- popup.html `chrome-extension://<id>/src/popup/popup.html` 環境下 `<script type="module">` 可正常解析相對路徑 import
- 唯一已知限制：esbuild bundle 為 IIFE 後，import/export syntax 已被剝離；若改用 ESM 載入，需要：
  - (a) 改 build.js 將 4 popup 檔都列為 entry point + `format: 'esm'`，或
  - (b) 把 4 檔合併為單一 popup.js entry point，使 esbuild 自動處理依賴

**ESM POC 限制**：

v8 中的 ESM inline test 因 SW timeout 多次失敗（Chrome 環境清理問題），但已透過以下證據佐證 ESM 可行性：

1. background SW 本身已是 ESM（manifest type=module）
2. esbuild 對 popup.js 已能正確處理跨檔 require/import（觀察 bundled popup.js 含 `__commonJS` / `__esm` runtime helpers）
3. 改造為 ESM 不需要新增依賴，只需調整 build.js entry points 與 popup.html script tag

### 修法多方案分析（4 方案）

#### 方案 A：popup.html 改 `<script type="module">` + 4 檔重構為 ESM

**實作概要**：

1. popup.html 5 個 script tag 改為 `<script type="module" src="...">`（or 只保留 popup.js 一個 entry，內部 import 其他 3 個）
2. popup-error-handler.js / popup-diagnostic-enhancer.js / popup-initialization-tracker.js 改寫為 ESM：
   - 移除頂層 `let Logger` / `var Logger` 宣告 + typeof guard
   - 改 `import { Logger } from '../core/logging/Logger.js'`
   - 改 `export class PopupErrorHandler { ... }` / `export class PopupInitializationTracker { ... }`
3. popup.js 改為 `import { PopupErrorHandler, PopupDiagnosticEnhancer, PopupInitializationTracker } from './*.js'`
4. build.js BUNDLE_ENTRY_POINTS：popup 入口仍只列 popup.js，esbuild 自動 bundle 全部依賴；或 format 改 'esm' 保留 import 結構

**跨檔影響清單**：

| 檔案 | 修改項 |
|------|-------|
| src/popup/popup.html | 5 個 script tag 改為 `type="module"`（或合併為 1 個 entry） |
| src/popup/popup-error-handler.js | 移除 `let Logger` typeof guard，改 import；class 改 export |
| src/popup/popup-diagnostic-enhancer.js | 同上 |
| src/popup/popup-initialization-tracker.js | 同上 |
| src/popup/popup.js | 改 import 三個 class；移除 typeof guard |
| build.js | 確認 esbuild format 對 popup.js 處理（IIFE vs ESM） |
| tests/integration/chrome-extension/popup-interface.test.js | eval(popupScript) 模式可能無法處理 ESM；需改 import 或保留 mock |

**風險評估**：

| 風險 | 機率 | 影響 | 緩解 |
|------|------|------|------|
| ESM dynamic import 路徑解析失敗 | 低 | 高 | 已驗證 background SW 同模式正常運作 |
| 既有 integration test eval(popupScript) 機制失效 | 高 | 中 | W4-006 已新增 integration test 用 setupEventListeners() 直呼，可保留 mock 路徑 |
| jest CJS 環境與 src ESM 衝突 | 中 | 中 | 既有 src code 已大量混用 require + module.exports，需保留 dual export |
| Chrome 91 以下不支援 ESM dynamic import | 低 | 低 | build.js target=chrome100，已排除舊版本 |

**build.js bundling 影響**：是。需新增 popup-error-handler.js / popup-diagnostic-enhancer.js / popup-initialization-tracker.js 為 BUNDLE_ENTRY_POINTS（或讓 popup.js 統一 import）。

#### 方案 B：合併 4 檔為單一 popup.js entry，依賴 esbuild 自動 bundle

**實作概要**：

1. popup.html 移除 popup-error-handler.js / popup-diagnostic-enhancer.js / popup-initialization-tracker.js 三個 script tag
2. 只保留 `<script src="popup.js"></script>`
3. popup.js 開頭新增 `require('./popup-error-handler.js')` + `require('./popup-diagnostic-enhancer.js')` + `require('./popup-initialization-tracker.js')`（或 `import`）
4. 3 個檔案的 class 改 `module.exports = { PopupErrorHandler }` / `export class`
5. esbuild bundle popup.js 時自動將 3 檔內聯到 popup.js IIFE 內，所有 Logger 宣告封閉在 IIFE scope—— 不會洩漏到 global scope，自然解決衝突

**跨檔影響清單**：

| 檔案 | 修改項 |
|------|-------|
| src/popup/popup.html | 移除 3 個 script tag，只保留 popup.js |
| src/popup/popup-error-handler.js | 頂層 `let Logger` 改 `const { Logger } = require(...)`；class 加 `module.exports` |
| src/popup/popup-diagnostic-enhancer.js | 同上 |
| src/popup/popup-initialization-tracker.js | 同上 |

### 推薦方案：B（合併為單一 entry，依賴 esbuild 自動 bundle）

**推薦理由**：

1. **解 popup pageerror 全部三筆**：3 個 helper 檔合併到 popup.js IIFE scope，Logger 衝突 + Tracker undefined 同時消解
2. **與既有 build.js / esbuild 機制最對齊**：popup.js 已是 entry point，BUNDLE_ENTRY_POINTS 無需新增條目；require 圖譜自動展開符合 esbuild 主要工作模式
3. **最小破壞性**：popup.html 只移除 3 個 script tag；既有 W4-006 integration test 用 `eval(popupScript)` 仍可運作（bundled popup.js 已含全部依賴）
4. **規格契約穩定**：tests/integration/chrome-extension/popup-interface.test.js 的 mock 路徑不需大改
5. **方案 C 雖更小但是反模式**：let→var 是補丁，未解架構問題（script tag 順序依賴 + global scope 汙染長期累積）；方案 D 過度工程；方案 A 雖最現代但 test 衝擊大、需評估 jest ESM 相容性

**方案 B 的合理疑慮 + 應對**：

- Q: 為何不用方案 A 走 ESM？A: 方案 A 需評估 jest 跨 CJS/ESM 相容性、test mock 機制改寫成本，超出 W4-006.2 single-ticket 範疇，可作為後續獨立技術債（W4-006.3 或 W5+）。
- Q: 為何不用方案 C 最小修法？A: 規則 5（所有發現必須追蹤）+ 規則 6（失敗案例學習原則）—— `let Logger`（W1-117 引入）→ classic script global record 衝突 是已被驗證的反模式，補丁式修法（let→var）等同持續累積技術債。方案 B 是「最小可徹底解 + 不改架構」的甜蜜點。
- Q: bundle 後 popup.js 變大會否影響載入？A: 81KB → 預估 130KB，遠低於 Chrome Extension 24MB 限制；Manifest V3 popup 不存在「快速顯示」的硬性 SLA，CE 安裝後本地讀取，無網路延遲考量。
