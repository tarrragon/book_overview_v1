## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

兩失敗皆 pre-existing 測試碼問題（非 runtime regression），依約束僅改 `tests/`。

#### 失敗 1：test_scenario_3_section_not_found
- 根因：`fake_ticket` 缺 `status` 欄位。W3-044 引入的 `require_in_progress` precondition 在 SECTION_NOT_FOUND 邏輯之前檢查 status；`require_in_progress` 對缺 status 預設視為 `pending`（最嚴格），直接寫 stderr 並 return 2，導致測試捕捉的 stdout 為空。
- 修法：`fake_ticket` 補 `status: in_progress`，使其通過 precondition 進到 SECTION_NOT_FOUND 渲染。

#### 失敗 2：test_table_format_renders_status_column
- 根因（雙重）：(a) `_fake_stats` 的 `per_day` 寫死 2026-05-13~15 日期，但 `_evaluate_all` 以「執行當日」（`datetime.now()`）取 recent，寫死日期永遠 recent=0；(b) 25/22/13 在 high_freq 閾值（baseline×3=25.7）以下，三連日 critical 路徑也無法觸發。
- 數學限制：high_freq multiplier=3 下，僅 3 個（或任意數量）資料日的 critical 路徑在數學上不可達（任意 N 天值之和 v 需每日 > 3×(v/window)，自我矛盾）。故採 recent-spike warning 路徑。
- 修法：spike 落在動態 `_today`（`datetime.now()`），count=300 遠高於 baseline×3≈128.6，使 recent > threshold 穩定觸發 warning，徹底去除寫死日期的時間依賴。fixture 仍維持兩 hook（其他 test 依賴 results>=2 與兩 hook 名稱），不破壞既有斷言。

### 影響範圍
- 僅 `tests/test_error_channel_integration.py`、`tests/test_track_hook_health.py` 兩測試檔。
- 未改任何 runtime / 產品邏輯。
