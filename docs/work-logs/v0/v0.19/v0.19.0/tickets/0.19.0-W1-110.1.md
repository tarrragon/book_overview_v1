### 問題根因（PM 補強，W1-110 freeze 預檢發現）

### 實作摘要

仿 W1-108 readmoo-adapter local dict 模式（commit c5c5f21a）將 background.js 的 27 個 background-specific message key 從 GlobalMessages 解耦：

1. **src/background/background.js**（+41 -12）：
   - import 從 `GlobalMessages` 改為 `MessageDictionary` class
   - 新增 module-level `backgroundMessages = new MessageDictionary({...27 keys...})` 並含完整業務情境註解（Why W1-110 freeze 預檢 / PC-165 防護 / 範圍邊界）
   - 移除 `GlobalMessages.addMessages(...)` 動態註冊呼叫
   - Logger 實例化加第三參數：`new Logger('BackgroundService', 'INFO', backgroundMessages)`
   - 更新原 line 22-23 過渡註解（W1-115 後 Logger 可注入 local dict）

2. **tests/unit/background/background-local-dict.test.js**（新增 +146 行）：
   - 3 個 PC-165 runtime 驗證 test case
   - test 1：BACKGROUND_STARTUP 渲染後字串含「Readmoo 書庫提取器」非 [Missing: KEY]
   - test 2：INIT_ATTEMPT 渲染含 placeholder（嘗試 1/3）非 {attempt}/{max}
   - test 3：隔離性驗證 GlobalMessages.has('BACKGROUND_STARTUP') === false，證實 addMessages 已移除
   - 採真實 Logger + spy console 模式（仿 readmoo-adapter-local-dict.test.js），mock chrome.runtime / addEventListener / ./background-coordinator 處理 SW entry script 載入

- **為何不修 background.js 其他 log call 傳文字字串問題**：PC-165 runtime test 暴露 13 處 `log.info('🏁 開始...')` 直接傳中文字串而非 messageKey 的既有 bug（如 line 369 INIT_FLOW_START 對應的 log call）。屬本 ticket 範圍外（4 acceptance 聚焦 local dict 切換 + Logger 第三參數注入 + 1 個 PC-165 test）。已建追蹤 ticket **0.19.0-W1-110.1.1** 處理（規則 5）。
