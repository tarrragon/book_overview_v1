### 問題根因

Claude Code v2.1.145 status line JSON 新增 GitHub repo 和 PR 資訊欄位（如 repo name、PR number、PR status、PR URL 等）。專案目前無任何 GitHub PR 整合——沒有 PR 工作流 hook、沒有 status line 自訂配置、沒有 ticket-PR 連動機制。需評估新欄位是否為專案帶來整合價值。

### 影響範圍

- `.claude/settings.json` — statusline 配置（目前為空 `{}`）
- `.claude/hooks/branch-status-reminder.py` — SessionStart 分支狀態提醒（僅 git branch/worktree，無 GitHub 資訊）
- 專案整體 PR 工作流 — 目前專案以 main 單分支開發為主，無常規 PR 流程

### 相關 Error Pattern

無直接相關 error pattern。

### 實驗方法

1. 檢視 `.claude/settings.json` 的 `statusline` 配置現況
2. 檢視所有已註冊 hooks，搜尋 GitHub/PR 相關整合
3. 搜尋 `.claude/` 目錄下所有 `github`、`pull request`、`gh pr` 引用
4. 檢視 `branch-status-reminder.py` 了解現有分支狀態顯示機制

### 實驗執行

**statusline 配置**：`settings.json` 中 `statusline` 為空物件 `{}`，未啟用任何自訂 status line。

**Hook 盤點**：
- SessionStart: `branch-status-reminder.py`（v1.3.0）— 顯示 git branch、worktree 列表、uncommitted files 分組。無 GitHub API 呼叫。
- 其他 hooks（共 30+ 個）：無任何 hook 引用 `github`、`pull_request`、`pr_number`、`gh pr` 等關鍵字。

**PR 工作流現況**：
- 專案以 main 單分支開發，無常規 feature branch → PR → merge 流程
- git status 顯示目前在 main 分支，無 remote tracking 的 PR
- `.claude/` 下無任何 PR 相關規則或參考文件

### 實驗發現

**已驗證事實**：
1. 專案完全沒有 GitHub PR 整合基礎設施
2. `statusline` 配置為空，status line JSON 新欄位不會自動影響任何現有行為
3. 專案不使用 PR 工作流（main 直接開發）
4. `branch-status-reminder.py` 僅用 git CLI，不讀取 status line JSON

**關鍵推論**：
v2.1.145 status line JSON 新增的 GitHub repo/PR 欄位對本專案**零影響**——既無破壞風險，也無自動收益。整合價值取決於專案是否有意引入 PR 工作流。

## WRAP 框架分析

### W — Widen（擴增選項）

| # | 整合情境 | 描述 |
|---|---------|------|
| A | Status line 顯示 PR 狀態 | 在 status line 自訂配置中加入 PR number/status，讓開發者看到當前分支的 PR 狀態 |
| B | PR 狀態檢查 Hook | SessionStart 或 PreToolUse hook 讀取 status line JSON 的 PR 欄位，在有 open PR 時提醒或阻擋特定操作 |
| C | Ticket-PR 連動 | ticket metadata 新增 `pr_url` 欄位，complete 時自動檢查 PR merged 狀態 |
| D | 不整合（觀望） | 維持現狀，不做任何整合。等專案引入 PR 工作流後再評估 |

### R — Reality Test（現況檢驗）

| 檢驗項 | 現況 |
|--------|------|
| 專案是否使用 PR 工作流？ | 否。main 單分支直接開發，無 feature branch → PR → merge 流程 |
| 是否有多人協作需要 PR review？ | 否。單人開發 + AI 代理人，無 code review 需求 |
| status line JSON 新欄位是否自動影響既有行為？ | 否。`statusline` 配置為空，hooks 不讀取 status line JSON |
| 引入 PR 工作流的計畫？ | 無。里程碑 v0.19.x~v1.0.0 聚焦端到端驗證與內測，未規劃 PR 流程 |

### A — Attain Distance（收益/風險評估）

| 情境 | 收益 | 成本 | 風險 | 淨評估 |
|------|------|------|------|--------|
| A: Status line 顯示 | 低（無 PR 可顯示） | 低（配置改動） | 低 | 無實質收益 |
| B: PR 狀態 Hook | 中（若有 PR 流程） | 中（開發 + 維護 hook） | 中（假設依賴不存在的工作流） | 前提不成立 |
| C: Ticket-PR 連動 | 高（若有 PR 流程） | 高（ticket schema 變更 + hook 開發） | 高（架構改動大） | 前提不成立 |
| D: 不整合 | 零成本 | 零 | 零（新欄位不影響既有行為） | 最佳選項 |

### P — Prepare to Be Wrong（失敗回退）

若選 D（不整合），失敗情境為「專案未來引入 PR 工作流後需要整合」。回退成本極低——屆時 status line JSON 欄位仍在，可按需建 IMP ticket 整合。無需預留架構。

## WRAP 結論

**選擇方案 D：不整合。**

Why：專案無 PR 工作流，情境 A/B/C 的前提條件均不成立。為不存在的需求建基礎設施屬 YAGNI 反模式。

Consequence：若專案在 v1.0+ 引入 PR 工作流（如多人協作、Chrome Web Store 發布前 review），屆時需重新評估。但回退成本極低（status line JSON 欄位持續可用）。

Action：本 ANA 結案，不 spawn IMP/DOC。

## 整合情境對照表（完整版）

| 情境 | 前提條件 | 前提成立？ | 建議 |
|------|---------|-----------|------|
| A: Status line 顯示 PR 狀態 | 專案有活躍 PR | 否 | 不整合 |
| B: PR 狀態檢查 Hook | 專案有 PR 工作流 | 否 | 不整合 |
| C: Ticket-PR 連動 | 專案有 PR + ticket 需連動 | 否 | 不整合 |
| D: 不整合 | 無 | 永遠成立 | 採用 |

## Spawn 規劃

無需建 ticket：專案無 PR 工作流，四個整合情境的前提均不成立。方案 D（不整合）為零成本最佳選項，不產生任何落地任務。若未來引入 PR 工作流，由該工作流的規劃 ticket 評估是否需要 status line 整合。

```yaml
multi_view_status: single_view_sufficient
multi_view_reason: 評估對象為平台新功能與專案契合度，非架構衝突或多方案權衡。WRAP 四階段已充分覆蓋所有情境，無需額外視角
```
