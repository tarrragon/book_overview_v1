## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

Logger.js:17-38 條件性 require fallback 包含 4 個 hardcoded message keys（DEBUG_MESSAGE / INFO_MESSAGE / WARN_MESSAGE / ERROR_MESSAGE），等同「影子 GlobalMessages」，與 W1-110 freeze 設計理念矛盾（source of truth 唯一性）。

### 影響範圍

**Logger.js:17-38 當前完整 fallback 區塊**：

```js
let GlobalMessages
if (typeof require !== 'undefined') {
  try {
    GlobalMessages = require('src/core/messages/MessageDictionary').GlobalMessages
  } catch (e) {
    // 瀏覽器環境或引入失敗時，使用後備方案
    GlobalMessages = {
      get: (key, params = {}) => {
        const fallbackMessages = {
          DEBUG_MESSAGE: 'Debug: {message}',
          INFO_MESSAGE: 'Info: {message}',
          WARN_MESSAGE: 'Warning: {message}',
          ERROR_MESSAGE: 'Error: {message}'
        }
        return fallbackMessages[key] || key
      }
    }
  }
}
```

### Fallback 觸發路徑分析

| 環境 | typeof require | require 結果 | 走 fallback？ |
|------|---------------|------------|--------------|
| Node.js + Jest | defined | 成功 | 否 |
| Chrome Extension bundled (esbuild) | defined | 成功（內聯） | 否 |
| Raw browser load (未 bundled) | undefined | N/A | 整個 if block 跳過，GlobalMessages 為 undefined |
| Node.js + require fail（極少見：路徑壞） | defined | fail | **走 fallback** |

**結論**：fallback dict 僅在「Node.js 環境 + require fail」極少見路徑觸發。屬防禦性程式碼，但 4 keys 過度（與 _loadDefaultMessages 24 keys 不一致）。

### 設計決策

**選項 A：移除影子 dict，保留 stub fallback**（裁定採用）
```js
let GlobalMessages
if (typeof require !== 'undefined') {
  try {
    GlobalMessages = require('src/core/messages/MessageDictionary').GlobalMessages
  } catch (e) {
    // 後備：require fail 極少觸發，提供最小 stub
    GlobalMessages = { get: (key) => '[Missing: ' + key + ']' }
  }
}
```

優點：
- 移除 4 個 hardcoded fallback keys（與 _loadDefaultMessages SSOT 解耦）
- 保留防禦性 require fail 處理（避免 Logger init crash）
- stub 行為清楚（'[Missing: KEY]' 訊號）

**選項 B：拋錯**
- require fail 時直接 throw，但 Logger init 階段 crash 難 debug，否決

**選項 C：保留 fallback dict 但移到常數**
- 違反「移除影子 dict」目標，否決

### Test 策略

1. 移除 4 個 hardcoded keys 後既有測試應仍綠
2. 若有 Logger fallback 路徑測試，斷言 stub 行為（'[Missing: KEY]' 格式）
3. npm test 全綠（baseline 5284 passed）
4. Chrome Extension dev build 確認 console 無異常訊息
