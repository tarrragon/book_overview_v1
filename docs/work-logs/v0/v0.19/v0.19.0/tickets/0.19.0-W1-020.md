### 問題根因

W1-017 修復 event-system 與 UC07 計時 flaky 時，用戶確認「計時不應作為測試驗收條件，效能差是設計問題而非功能問題」為系統性設計原則。W1-018 ANA 盤點全專案，在 7 個測試檔發現 21 個問題斷言（計時硬門檻 17 個 + 高精度浮點 4 個），確認為可重現的跨模組設計反模式。需固化為規則防止再發。

### 影響範圍

新建規則檔：`.claude/rules/core/test-assertion-design-rules.md`

### 相關 Error Pattern

無既有 error-pattern 精確匹配（W1-018 ANA 確認）。本 ticket 產出規則文件取代 error-pattern，適用範圍更廣（設計指導 vs 事後追蹤）。

## Solution
<!-- Schema[DOC/Solution]: 免填（DOC 類型以變更摘要取代）（.claude/pm-rules/ticket-body-schema.md） -->

### 變更摘要

新建規則檔 `.claude/rules/core/test-assertion-design-rules.md`，含 4 條核心原則（三明示格式，每條附 Why/Consequence/Action）。

| 規則 | 核心原則 | 觸發案例 |
|------|---------|---------|
| 規則 1 | npm test 主套件禁止絕對計時門檻當 pass-fail 斷言 | W1-017：avgSuggestionTime<2ms 在完整套件下 2.48ms 失敗 |
| 規則 2 | 效能測試放 tests/perf/，透過 npm run test:perf 獨立執行 | W1-017：方案 A 物理拆檔，npm test 恢復 100% |
| 規則 3 | toBeCloseTo 精度不超過 2 位（除非驗證確定性整數計算） | W1-018：cross-device-sync toBeCloseTo(...,5) intermittent flaky |
| 規則 4 | 快取加速驗證用命中率而非計時比較 | W1-018：readmoo-data-validator secondRunTime < firstRunTime * 0.8 |

### 放置位置決策

評估新檔 vs 併入既有規則：

- `quality-baseline.md` 已有規則 1（通過率 100%），但不含斷言設計細節；本規則為其設計前提，放在不同層次（目標 vs 設計約束），避免 quality-baseline 過度膨脹
- `quality-common.md` 含命名/函式設計/DRY 等通用品質指標，不含測試斷言設計，放入會造成混淆
- 結論：新建獨立檔案最清晰，交叉引用章節已明確標注與 quality-baseline 的邊界

### 與 quality-baseline 邊界

規則 1（測試通過率 100%）是目標；本規則是達成目標的設計約束前提。兩者互補不重複。
