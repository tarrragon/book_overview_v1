### 問題根因

框架文件是靜態快照，MCP server 實機暴露是動態 runtime，兩者無自動同步機制，必然累積三層漂移（PC-173 已定義）：

| 層 | 漂移來源 | 本次盤點實例 |
|----|---------|-------------|
| 工具存廢 | server 跨版本改名 / 移除 | codegraph 舊速查表名（`get_callers` / `symbol_search` / `analyze_impact` / `reindex_workspace`） |
| 工具命名 | server 改 `<prefix>_<verb>` 命名 | `callers` → `codegraph_callers`（短名 vs 全名） |
| server 前綴 | 安裝方式決定 server id | user-level `mcp__serena__` vs plugin `mcp__plugin_serena_serena__`；cbm `mcp__codebase-memory-mcp__` |

### 影響範圍

盤點對象：`.claude/` 全樹（skills / templates / references / agents / methodologies / hooks）。`mcp__` 引用依**危害分級**分三類：

| 類別 | 危害 | 涉及檔案（代表） | 是否需修 |
|------|------|----------------|---------|
| HIGH（指引性引用，讀者會實際呼叫失效名） | 浪費回合 / 誤判平台不支援 | `search-tools-guide/SKILL.md` 速查表（codegraph 5 名 + cbm + serena 前綴）、`project-init/.../messages.py` 索引驗證提示 | 已由 W1-033/W1-034 修 |
| LOW（allowlist dead entries，不 match 即靜默不生效） | permission 授予失效但無誤導 | `templates/settings-local-template.json`（serena 7 名 + dart 名） | serena 已由 W1-034 修；dart 屬下一類 |
| NOT-DRIFT（project-type-generic 框架資產） | 無——對目標專案類型是正確名 | Flutter agents（parsley/basil/ginger/bay/cinnamon/pepper/linux/mint/sage 的 `mcp__dart__*`）、dart-test hooks、`lsp-first` / `test-async-guardian` / `acceptance-criteria` methodologies | 保留（acceptance 第 5 條：不假設單一 MCP 安裝方式） |

**關鍵約束發現**：本 session 工具（Bash grep / Read）對「可呼叫 MCP 工具名」做**顯示層 redaction**（`mcp__dart__X`→`lX`、`mcp__codegraph__codegraph_X`→`mcp__codegraph__n`、`ToolSearch`→`ln`），導致 PM 無法可靠逐名 audit。redaction 僅在顯示層，不影響檔案真實內容——Python hook 以 `open()` 讀檔不受影響。

### 常駐 hook 決策（對應 acceptance 第 4 條）

**結論：不新增即時 guardian hook。** 理由（決策三明示）：

- **Why**：guardian hook 的「文件引用 vs 實機暴露」交叉驗證需 runtime deferred tools 清單作 ground truth，但該清單僅在 session context（system-reminder）出現，standalone hook 無法存取。退而求其次的「hook 內維護一份前綴 allowlist」會讓 allowlist 本身成為新的靜態快照——複製了原問題（PC-173 根因）。
- **Consequence**：強行做 hook 會得到一個需人工維護、且自身會漂移的偵測器，增加維護面卻不解決根因；違反 quality-baseline 對「限制性解法 vs 探索性解法」的取捨。
- **Action**：改採三項低成本韌性化措施取代 hook：
  1. **文件韌性化**（spawn DOC）：將 `search-tools-guide/SKILL.md` 的硬編碼工具名改為「ToolSearch 發現 + server 前綴對照」描述，從源頭消除速查表漂移。
  2. **流程防護**（已落地）：PC-173 防護流程（以 deferred list 為 ground truth、用 Read 驗證、危害分級修）已是知識庫常駐規則，無需 hook。
  3. **可選定期掃描**（不 spawn，記錄於 PC-173）：若未來漂移復發頻率上升，可比照 `broken-link-check` 做 `/loop` 定期 audit skill；當前頻率（3 檔/單次）不足以證成此投入。
