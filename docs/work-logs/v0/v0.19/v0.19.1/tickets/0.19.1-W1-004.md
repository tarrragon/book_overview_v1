## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### Gap 1：CHANGELOG finalize In-Development 區段

**根因確認**（PC-162 重新驗證行號）：
- `update_changelog`（`scripts/version_release.py:1281`）冪等檢查為 `f"## [{version}]" in changelog_content`（原 1314）。
- 開發期 In-Development header 慣例為 `## [v{version}] - In Development`（W1-073 實證，header 帶 `v` 前綴）。version 參數為 `0.19.0` 時組出 `## [0.19.0]`，與帶 `v` 前綴的 In-Development header 不符 → 冪等檢查 miss → 插入第二個 placeholder 區段，造成雙區段需手動 consolidate。

**修正**：在冪等檢查前新增 finalize 偵測。
- regex `^## \[v?{version}\] - In Development\s*$`（MULTILINE）同時匹配帶/不帶 `v` 前綴的 In-Development header。
- 命中時：將該 header 行替換為 `## [{version}] - {today}`，保留其後人寫內容，**不插入** UC-XX placeholder 空殼。
- 未命中 In-Development 但版本已 finalize（header 帶日期）：冪等跳過（冪等檢查同時涵蓋 `## [{version}]` 與 `## [v{version}]`）。
- 無 In-Development 且版本不存在：維持原有插入模板行為（向後相容）。

### Gap 2：release 後 todolist active → completed

**根因確認**：release 主流程（`main()` release 分支，原 2364）於 `git_merge_and_push` 成功後直接 `print_summary`，缺少 todolist 狀態更新步驟。後果為後續 `start` 下一版被 `find_last_completed_version`（原 1470）阻擋。

**修正**：
- 新增 `mark_version_completed(todolist_path, version, dry_run)` 函式（仿 `insert_version_to_todolist` 字串操作模式，保留 YAML 格式）：定位 `- version: "X"` 條目範圍內第一個 `status:` 行，將值改為 `completed`，同步更新 `last_updated`。冪等（已 completed 回 True）；版本不存在回 False；dry_run 不寫檔。
- release 主流程於 `git_merge_and_push` 成功後新增「Step: Mark Version Completed」，呼叫 `mark_version_completed`（todolist 路徑 = `get_project_root() / "docs" / "todolist.yaml"`）；失敗僅 warning 不中止發布（git 已 push，不應因 todolist 更新失敗回滾）。

### 不在本 ticket 範圍

- `tests/test_monorepo_version_sync.py` 既有 collection error（import `get_monorepo_version`，該函式從未定義，count=0）為先前遺留問題，非本次變更引入。
