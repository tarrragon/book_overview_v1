## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 事件起點

PM 執行 `/sync-pull`，預期只拉入用戶在另一專案剛 push 的一個 error-pattern。實際 sync 產生 49 檔變更（+13376 行），含 27 個新 hook。用戶質疑後逐層比對，確認同步機制在跨框架分歧下有設計瑕疵（push 與 pull 雙向）。

### 三類設計瑕疵（根因）

### 瑕疵 A：pull 為全佈局覆蓋，對本地結構分歧無感知

- sync-claude-pull.py 將上游 repo 根目錄整體覆蓋到本地 `.claude/`，不知道本專案做過的結構遷移。
- 本專案 W10-092（commit 7ffba6d3）把 20+ 個 ticket 相關 hook 從 `.claude/hooks/` 遷移到 `.claude/skills/ticket/hooks/` 並刪除舊位置。
- 上游 claude.git 兩處都保留（`hooks/` 與 `skills/ticket/hooks/` 各一份，hooks/ 達 130 個）。
- 結果：pull 把上游 `hooks/` 的 21+ 個 ticket/worktree hook 又塞回本地 `.claude/hooks/`，與本地 `skills/ticket/hooks/` 形成重複（兩份檔案大小還不同，如 28456 vs 28480 bytes）。

### 瑕疵 B：版號基底失效，無法作為增量判斷依據

- 本專案曾 sync 到 v1.36.3（commit 868f1b44），隨後 7701d29e「版號基底修正」把 `.claude/VERSION` 刻意設回 v1.18.0。
- 上游維護自己的 v1.39.x 連續序列；本地 VERSION 與上游序列脫鉤。
- pull 比對時本地 v1.18.0、上游 v1.39.2，落差 21 個 minor，導致全量差異一次攤出，而非「上次 push 後的真實增量」。

### 瑕疵 C：push 內容與本地 git 追蹤狀態不一致

- 本專案上次 push（64bafb5 = 上游 v1.39.0，今天 11:01）帶著 130 個 hook（含 hooks/ 那 27 個）。
- 但本專案 git 只追蹤 102 個 hook（W10-092 已從 git 移除那 27 個）。
- 代表 push 出去的磁碟內容不等於本專案 git 追蹤的狀態。push 把磁碟全量推上去，未尊重本地 git 的結構決策，等於把本地已清理的重複又散播回上游。

### 衍生問題：跨框架 PC 編號撞號

- 上游 PC-165 = auq-dispatch-option-without-blockedby-readiness-check（用戶在另一專案新增）。
- 本專案 PC-165 = false-positive-fix-chain（既有，被 test-assertion-design-rules.md 引用）。
- 同編號、不同 pattern。上游本身就容忍兩個 PC-165 並存。pull 會把 auq-dispatch 版帶入造成本地撞號。
- 根因：PC 編號在上游框架線與各專案線各自遞增，無跨框架編號治理機制。

### 本 session 的處置（非根本解，僅止血）

1. 誤拉的 sync commit c8cbb71c 已 git reset --hard HEAD~1 還原。
2. 用戶要的 error-pattern 以重編號方式引入：本地建 PC-171（auq-dispatch），檔內加溯源註記「上游 PC-165」，commit b09c8e90。
3. hook 重複、版號重設均已隨 reset 消除。

### 待高 effort model 評估的核心問題

1. pull 如何感知並保留本地結構分歧（如 W10-092 遷移）？目前 preserve 機制僅 3 檔，是否足夠？
2. 版號基底是否該與上游序列重新對齊，或設計獨立的「本地 fork 版號 + 上游 base 版號」雙軌？
3. push 是否應以 git 追蹤狀態為準（而非磁碟全量），避免把本地未追蹤/已刪除內容散播回上游？
4. 跨框架 PC 編號治理：是否需要 namespace（framework-PC vs project-PC）或編號保留區段？

### 機制修正（讀 source 後對 Problem Analysis 的兩處更正）

**修正 1：瑕疵 B 不是 49 檔爆量的成因。** Problem Analysis 述「版號基底失效導致全量差異一次攤出」。但讀 `sync-claude-pull.py` 確認：pull 全程**不使用 VERSION 做增量判斷**，永遠 `clone --depth 1` + `sync_directory` 全量 overlay + `cleanup_stale_files`。49 檔爆量 100% 來自瑕疵 A（全量 overlay 把本地已遷移走的 hook 從上游 `hooks/` 抄回）。VERSION 1.18.0 vs 1.39.2 是獨立潛在不一致，**修 B 不會減少爆量**。此修正影響修法優先序。

**修正 2：瑕疵 A 與 C 構成惡性循環（feedback loop），非兩個獨立問題。**
- pull：`sync_directory(temp_dir, claude_dir)` 從上游磁碟全量抄回本地磁碟。
- push：`copy_filtered(claude_dir, temp_dir)` 從本地**磁碟**（filesystem walk，非 git 追蹤態）全量推回上游。
- 循環：pull 把 migrated-away hook 抄回本地磁碟 → push 把磁碟全量推回上游 → 下次 pull 再散播到所有專案。垃圾在兩端自我繁殖，這是 49 檔中 27 hook 重複的傳播機制。

### 根本根因（統一視角）

四個瑕疵收斂到**單一架構缺陷：同步機制沒有 base / 共同祖先（common ancestor）概念**。

現行 push/pull 都是**無狀態全量 overlay**——「讓本地 = 上游（扣除 preserve/skip）」。此模型無法表達兩件事：(1) 本地**刻意刪除**上游存在的檔案（結構遷移，如 W10-092）；(2)「上次同步到哪個上游版本」的錨點。`sync-preserve.yaml` 只能保護「保留但客製」的檔案（正向 preserve），無法表達「刻意不要」（負向 preserve / 刪除意圖）。

引入 base 後，A/B/C 收斂為同一個「帶 base 的三方合併」：
| 瑕疵 | 無 base 現況 | 有 base 後 |
|------|------------|-----------|
| A | 全量 overlay 無視本地刪除 | base→上游 delta 三方合併，本地刪除保留 |
| B | VERSION 裝飾性、與上游脫鉤 | base = 上次同步的上游 commit SHA（語意明確） |
| C | push 從磁碟全量 | push 以 git 追蹤態為 base |

瑕疵 D（PC 編號撞號）與 base 模型正交，屬獨立編號治理問題。

### 修法選項評估（每瑕疵 ≥ 2 選項）

#### 瑕疵 A：本地結構分歧保留

| 選項 | 做法 | 利 | 弊 | 評估 |
|------|------|-----|-----|------|
| A1 負向 preserve（刪除清單） | 新增 `sync-delete.yaml` 列「上游存在但本地拒絕再抄回」的 glob（如 `hooks/ticket-*.py`），pull 的 `sync_directory` 跳過 | 改動小、外科手術式、立即可用 | 需人工維護 denylist，遷移多了會 drift | 採（interim 止血） |
| A2 改用真 git subtree | pull/push 改 `git subtree`，靠 git merge 語意尊重本地刪除 | 語意正確、自動 | README 自述 subtree「歷史處理可能失敗」、churn 大、推翻現行設計初衷 | 不採（風險>價值） |
| A3 base snapshot 三方合併 | `.sync-state.json` 記上次同步的上游 commit SHA；pull 只套用 base→新 的上游 delta，與本地三方合併 | 根本解、本地刪除天然保留、與 B/C 共構 | 工程量最大、需設計合併衝突處理 | 採（根本解，需先 spike + 多視角） |

#### 瑕疵 B：版號基底

| 選項 | 做法 | 利 | 弊 | 評估 |
|------|------|-----|-----|------|
