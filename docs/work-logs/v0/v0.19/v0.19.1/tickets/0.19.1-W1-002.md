## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

**直接根因：表頭 `<th>` 數（7）與列渲染 `<td>` 數（8）不一致——thead 缺「標籤」欄表頭。**

ticket 草稿描述「5 欄但標題列只顯示 3 欄」與實況不符（PC-162：草稿寫於 2026-05-27，之後 W2/W3 視覺重塑已改動 overview）。PM 前台勘查（2026-06-05，code reading）確認真實狀態為 **7 th vs 8 td**：

| 欄位數來源 | 值 | 位置 | 內容 |
|-----------|-----|------|------|
| thead `<th>` | **7** | `src/overview/overview.html:64-74` | select / 封面 / 書名 / 作者 / 書城來源 / 進度 / 狀態 |
| createBookRow `<td>` | **8** | `src/overview/overview-page-controller.js:763-848` | select / 封面 / 書名 / 作者 / 書城來源 / 進度 / 狀態 / **標籤(tagCell)** |
| `CONSTANTS.TABLE.COLUMNS` | **8** | `overview-page-controller.js:59`（用於 empty-state colspan:1172） | 數值常數 |

根因鏈：tagCell（`overview-page-controller.js:843-845`，由 `tagCellRenderer` 建立）被加入列渲染時，HTML 表頭未同步新增 `<th>標籤</th>`。`overview.html` grep `標籤|tag` 零命中，確認表頭缺標籤欄。

**架構根因：欄位定義散落 3 處硬編碼、彼此獨立漂移。** 每新增一欄需手動同步（a）thead HTML、（b）createBookRow td 序列、（c）COLUMNS 常數，違反 DRY/SSOT，本次 tag 欄即漏改 (a)。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `src/overview/overview.html` | thead 缺標籤 th（直接症狀，標籤欄無表頭 + 資料列與表頭錯位） |
| `src/overview/overview-page-controller.js` | createBookRow 8 td 與 COLUMNS:8（資料來源散落點 2、3） |
| `src/overview/overview.css` | 表格 layout（需確認是否有 `display:none` 隱藏欄或 grid 欄數定義影響對齊；本次勘查未深入 CSS） |

附帶發現（非本 ticket 範圍，另記）：`overview-page-controller.js:1172` empty-state row 含硬編碼 inline `color: #666`，屬 W2-001 design-system 收斂殘留，建議後續 follow-up（quality-baseline 規則 5）。

### 相關 Error Pattern

- PC-162（接手者須獨立重新驗證範圍）：本 ticket 為正面案例，PM 重掃推翻草稿「3 vs 5」描述，確認真實為「7 vs 8」。
- DRY/SSOT 違反：多處硬編碼同一資料（欄位數）導致漂移，為本 ticket 動態化設計的動機。

## 重現實驗結果

### 實驗方法

以 code reading + grep 靜態驗證欄位數三來源（非 runtime，因根因為靜態結構不一致，靜態勘查已足以定位）：

```bash
# 1. thead th 數
rg -n "<th" src/overview/overview.html
# 2. createBookRow td 數（appendChild(...Cell)）
sed -n '763,848p' src/overview/overview-page-controller.js
# 3. COLUMNS 常數與使用點
rg -n "COLUMNS|colspan" src/overview/overview-page-controller.js src/overview/overview.html
# 4. 表頭是否有標籤欄
rg -ni "標籤|tag" src/overview/overview.html
```

### 實驗執行

| 步驟 | 指令 | 觀察 |
|------|------|------|
| 1 | grep thead th | 7 個 th：select / 封面 / 書名 / 作者 / 書城來源 / 進度 / 狀態 |
| 2 | 讀 createBookRow | 8 次 `row.appendChild`：selectCell / coverCell / titleCell / authorCell / sourceCell / progressCell / statusCell / **tagCell** |
| 3 | grep COLUMNS | `COLUMNS: 8`（行 59）；唯一使用點 empty-state colspan（行 1172）|
| 4 | grep 標籤/tag in html | **零命中** → 表頭確無標籤欄 |

### 實驗發現

**已驗證事實**：
1. thead 7 th、列 8 td、COLUMNS 常數 8——三來源不一致，thead 少 1（缺標籤欄表頭）。
2. 根因為 tag 欄加入列渲染時表頭漏同步；架構根因為欄位定義 3 處硬編碼漂移。
3. ticket 草稿「3 vs 5」描述與實況（7 vs 8）不符，已於 Problem Analysis 更正（PC-162）。

**未驗證假設**（留待 IMP 實作時確認）：
- CSS（overview.css）是否另有 grid template 欄數或 `display:none` 影響視覺對齊（本次未深入；靜態結構已足以解釋表頭缺欄，CSS 為次要驗證項）。
- 標籤欄是否「應顯示」（產品意圖）：tagCellRenderer 已實作且列已渲染 tagCell，推定標籤欄應可見、表頭應補；若產品意圖為隱藏標籤欄，則方向相反（移除 tagCell）。**此產品意圖須向用戶確認**。

### 產品意圖確認

### 三方案 ROI 對照

| 方案 | 擴充成本 | 可讀性 | 測試難度 | 影響範圍 | 達成 SSOT |
|------|---------|--------|---------|---------|----------|
| A 單一 column config 陣列驅動 thead+td+colspan | 中-高 | 長期佳/短期密 | 中-高 | 大（重構 createBookRow + thead 生成 + 移除 COLUMNS） | 是（結構層） |
| **B 補缺 th + 一致性測試守衛**（選定） | 低 | 不變 | 低 | 小（1 th + 1 測試） | 部分（測試強制防漂移） |
| C 從 Book schema 動態反射 | 高 | 中 | 高 | 大 | 是但 leaky（checkbox/cover/tag 非 data field） |

### 選定方案：B（用戶決策 2026-06-05）

**理由**：8 欄各有異質 render 邏輯（checkbox 事件監聽、status innerHTML、author 條件 tooltip、tag 獨立 renderer），方案 A 將其泛化進 config render callback 的複雜度與風險，對 P2、8 欄穩定的情境不成比例。B 立即修正 thead 缺欄 bug，並用測試守衛斷言「thead th 數 === CONSTANTS.TABLE.COLUMNS === createBookRow td 數」，使任何未來漂移在 CI 紅燈——達成用戶 `why` 真正關切的「防止兩處手動同步遺漏」的實際目標。

### Dissenting View（PC-101）

**反對立場**：偏好嚴格 SSOT 的審查者會選 A——B 仍保留 3 處硬編碼，「測試守衛」是事後攔截而非結構消除，未根除漂移可能性，且不符 `why` 字面「欄位定義成為單一資料來源」。

**回應與 trigger**：B 的測試守衛將「漂移」從 silent runtime bug 降級為 CI 即時紅燈，實務上等價於防漂移；A 的結構 SSOT 在 8 欄異質渲染下複雜度不划算。**重評 trigger**：若未來欄位增刪頻繁（建一個新 ticket 計數，累積 3+ 次欄位變更），則重新評估方案 A 的結構 SSOT（屆時擴充頻率使 A 的前期成本被攤平）。此為 decision-trigger-binding 合法狀態（b）——綁 ticket 計數而非時間。
