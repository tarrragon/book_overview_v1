## Context Bundle

### PM 已完成的前置取證（saffron 起步點，須獨立複驗 PC-162）

**完整發現**：見 `0.19.1-W1-001` Problem Analysis「PM 前置分析發現」章節。重點摘要：

1. **真 emoji 範圍**（ticket 定義 1F000-1FFFF + 2600-27BF）：src/overview 2 檔、src/popup 8 檔，~81 個。箭頭 `→`（2190-21FF, 91 個）非 emoji，排除。
2. **兩類 emoji**：UI-facing（html）vs console.log message 前綴（popup.js 等，如 `'✅ Popup Script 載入完成'`）。
3. **契約衝突**：`docs/spec/extraction/e2e-contract.md` §3.4（約 L709-734）結構化前綴表含 emoji（如 `❌ <名詞>失敗:`），明定「完全相等才算合法 E2E 觀察點」「改字面須同步本表 + 所有 E2E 觀察測試」。
4. **測試依賴**：`'Popup Script 載入完成'`、`'Popup Interface 初始化完成'` 各被 1 測試檔引用。

### 可複用掃描指令

- 真 emoji 逐檔：`rg -P -c '[\x{1F000}-\x{1FFFF}\x{2600}-\x{27BF}]' src/`
- 字元抽樣：`rg -P -o '[\x{1F000}-\x{1FFFF}\x{2600}-\x{27BF}]' src/ | sed 's/.*://' | sort | uniq -c | sort -rn`
- 契約前綴表：`sed -n '709,735p' docs/spec/extraction/e2e-contract.md`

### 關鍵決策點（須在 Solution 給結論）

- **範圍**：popup/overview-only（留 content/background emoji，文件化不一致）vs 全專案（一致但範圍大）。建議基於「契約一致性」評估，不以估時為依據。
- **替換規範**：emoji→ASCII（✅→[OK] / ❌→[FAIL] / 🔍→[CHECK]...）逐一決策表，或純移除。須與既有 ASCII 前綴慣例（`[SW]` / `[DIAG]` / `[ERROR]` / `[Logger Fallback]`）對齊。
- **spawn 規劃**：DOC（contract §3.4 修訂）/ IMP（src 移除）/ IMP（E2E 測試同步），含裁定 W1-001 最終 where.files。落地一律用 `--parent 0.19.1-W1-005`（PC-091）。
