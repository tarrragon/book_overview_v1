### 分析結論

**建議方向：方案 C（維持現狀），原因如下：**

1. **收益有限**：當前 .claude/ 修改頻率 PM 前台可承受（PC-114 記載小修 <30min PM 前台即可），主 repo cwd 不加 isolation 已支援 ≤ 2 並行
2. **merge 衝突成本不可忽略**：.claude/ 規則/方法論檔案為長文 markdown，並行修改同檔的衝突率遠高於 src/ 程式碼（程式碼以函式為單位切割，markdown 以段落為單位）
3. **平台依賴風險**：#15 是 CC 2.1.161 的行為變更，無 API 穩定性保證；若後續版本回退，已鬆動的流程需再次收緊
4. **H1/H2/H3 未驗證**：鬆動前應補全矩陣，但補測本身就需投入；若結論傾向不鬆動則補測 ROI 低

**方向交用戶決策**：本分析傾向「維持現狀 + 更新文件標註 #15 已解鎖但未採用」。若用戶決定鬆動，spawn IMP 改 PC-114 / behavior-loop-details / dispatch hook；若維持，spawn DOC 記錄本決策。

### linux 多視角審查結論（acceptance 3，S4）

linux 同意方案 C（維持現狀），但論證根基比 saffron 更穩固，並指出 saffron 利弊表的致命漏洞：

**關鍵發現：方案 A 的並行收益是假象**

| 事實 | 來源 |
|------|------|
| #15 只解鎖「主 repo 樹內 CC-managed agent worktree」（.claude/worktrees/agent-xxx） | W1-011 探針 SUCCESS 場景 |
| 外部 worktree（/tmp）的 .claude/ 在 2.1.161 仍被拒 | ARCH-015 line 89（050.1/050.3 矩陣） |
| worktree 內外 .claude/ 視為同一份樹 | ARCH-015 line 46 |
| 真正分界是「target 是否在主 session repo 樹內」，非 worktree vs 非 worktree | ARCH-015 line 96 |

**推論**：透過 worktree 並行寫 .claude/ = 寫同一份檔案，無真隔離。方案 A 相對方案 C 的並行性增量遠小於 saffron 利弊表宣稱的「高 vs 低」。鬆動不會帶來真正並行收益，只增加 hook 改造 + 3 文件變更 + 回退成本。

**saffron vs linux 論證差異（dissenting view 已仲裁）**：
- saffron：方案 A 可行但不划算（收益有限 + merge 成本 + 平台風險 + ROI 低）
- linux：方案 A 核心收益不存在（同一份樹 + 外部 worktree 仍被拒），技術前提已被 ARCH-015 line 89 證偽
- PM 仲裁：採 linux 框架。理由更穩固，且防止未來 CC 版本誤判重啟——除非 CC 改變「.claude/ worktree 內外視為同一份」runtime 設計，否則 worktree 並行寫 .claude/ 永遠無真隔離

**最終建議：維持現狀（方案 C），不鬆動 ARCH-015/PC-114。spawn S3 DOC 採 linux 框架記錄決策；S1/S2 IMP 豁免（理由更正為「方案 A 技術前提已被證偽」，非「結論傾向不鬆動」）。**
