### 分析結論

**建議方向：方案 C（維持現狀），原因如下：**

1. **收益有限**：當前 .claude/ 修改頻率 PM 前台可承受（PC-114 記載小修 <30min PM 前台即可），主 repo cwd 不加 isolation 已支援 ≤ 2 並行
2. **merge 衝突成本不可忽略**：.claude/ 規則/方法論檔案為長文 markdown，並行修改同檔的衝突率遠高於 src/ 程式碼（程式碼以函式為單位切割，markdown 以段落為單位）
3. **平台依賴風險**：#15 是 CC 2.1.161 的行為變更，無 API 穩定性保證；若後續版本回退，已鬆動的流程需再次收緊
4. **H1/H2/H3 未驗證**：鬆動前應補全矩陣，但補測本身就需投入；若結論傾向不鬆動則補測 ROI 低

**方向交用戶決策**：本分析傾向「維持現狀 + 更新文件標註 #15 已解鎖但未採用」。若用戶決定鬆動，spawn IMP 改 PC-114 / behavior-loop-details / dispatch hook；若維持，spawn DOC 記錄本決策。
