## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 背景

source: 0.19.1-W1-043（CC 2.1.162/2.1.163 release 影響評估）採用項 #4。CC 2.1.163 #4 讓 Stop/SubagentStop hook 可回傳 hookSpecificOutput.additionalContext 給 Claude 回饋且不算 hook error。

### 待答問題

本框架 7 個 Stop/SubagentStop hook（5 Stop + 2 SubagentStop）中，哪些應改用 additionalContext？需界定：
1. 各 hook 現有輸出機制與意圖對象
2. additionalContext（keep turn going）vs systemMessage（allow stop）的語意差異對各 hook 的影響
3. unwanted turn continuation 風險（Stop hook 採用會阻止用戶意圖的 session 停止）
4. #1 版本釘選（requiredMinimumVersion）是否依附採用

### 框架歷史關聯

W17-158/159/160：框架曾 EMPIRICALLY 撞「Stop/SubagentStop event schema 不允許 additionalContext」限制，被迫改用 systemMessage（3 個 hook header 明載）。2.1.163 #4 正是解除此限制——被迫 workaround 的 SubagentStop hook 為主要回收對象。

詳見「重現實驗結果」與「Solution」章節的逐 hook 判定。

## 重現實驗結果
### 實驗方法

驗證 CC 2.1.163 #4（Stop/SubagentStop hook 可回傳 hookSpecificOutput.additionalContext）對本框架 7 個 hook 的適用性，並界定 unwanted turn continuation 風險。方法：
1. 確認本 session CC 版本支援 #4
2. 程式碼考古：讀 7 個 hook 現有輸出機制（systemMessage / stderr / decision:block / suppressOutput）
3. 比對 additionalContext 與 systemMessage 的語意差異對各 hook 的影響
4. 佐證：grep 框架歷史（W17-158/159/160）對 Stop/SubagentStop schema 限制的原始發現

### 實驗執行

- `claude --version` = 2.1.163 → #4 可用（已確認）
- 歷史佐證：subagent-stop-dispatch-cleanup-hook.py:12、agent-commit-verification-hook.py:19、stop-worklog-handoff-sync-check-hook.py:22 三檔 header 明載「Stop/SubagentStop event schema 不允許 hookSpecificOutput.additionalContext，改用 systemMessage（W17-158/159/160）」→ 證明框架曾 EMPIRICALLY 撞此限制並 workaround
- 7 hook 現有機制盤點：
  | hook | event | 現機制 | 意圖對象 |
  |------|-------|--------|---------|
  | subagent-stop-dispatch-cleanup | SubagentStop | systemMessage `[OK] 可開始驗收` / `[WAIT]` | PM 主線程（驅動驗收） |
  | agent-commit-verification | SubagentStop | systemMessage（驗證 agent commit） | PM 主線程（驅動後續） |
  | continuous-learning evaluate-session | Stop | stderr `evaluate for patterns` | Claude（要求評估） |
  | session-experience-persistence-reminder | Stop | stderr 提醒 + suppressOutput | Claude（持久化提醒） |
  | stop-worklog-handoff-sync-check | Stop | systemMessage（worklog sync） | Claude（提醒同步） |
  | handoff-auto-resume | Stop | 4a decision:block（蓄意阻擋）/ 4b stderr+allow | Claude |
  | worktree-auto-commit | Stop | 無 Claude 溝通（靜默 auto-commit） | 無 |

### 實驗發現

**已驗證事實**：
1. additionalContext 語意 = 注入 context + **keep turn going（繼續，不停止）**；systemMessage 語意 = 通知 + **allow stop（允許停止）**。兩者非等價替換。
2. SubagentStop 時主線程仍執行 → additionalContext 注入主線程 context 推動繼續，無「阻止用戶停止」副作用 → 適合。
3. Stop 時主線程正要停止 → additionalContext 會阻止用戶意圖的停止 = unwanted turn continuation → 不適合（除非如 handoff 4a 蓄意 block）。
4. 被 W17-159 限制逼成 systemMessage 的 2 個 SubagentStop hook（dispatch-cleanup、commit-verification）正是 #4 解禁後的正確採用對象。

**未驗證假設（延後 IMP runtime 確認）**：
- 「CC 2.1.163 SubagentStop schema 實際接受 additionalContext 且不報 hook error」——release note 為權威 + 版本已確認，但 live SubagentStop cycle 觀察需實際派發 subagent 才能觀測，延後至 IMP rollout（含 rollback trail）驗證。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 7 個 Stop/SubagentStop hook 逐一適用性判定

| hook | event | 現機制 | 採用 additionalContext？ | 理由 |
|------|-------|--------|------------------------|------|
| subagent-stop-dispatch-cleanup | SubagentStop | systemMessage | **採用** | `[OK] 可開始驗收`/`[WAIT]` 是 PM 主線程應行動內容（PC-104）；SubagentStop 主線程仍執行，注入無停止副作用；正是 W17-159 被迫放棄的機制 |
| agent-commit-verification | SubagentStop | systemMessage | **採用** | 驗證 agent commit 結果應驅動 PM 後續決策；同上 SubagentStop 語境；W17-160 被迫 systemMessage |
| continuous-learning evaluate-session | Stop | stderr | 不採用 | Stop 時主線程正停止，additionalContext 會阻止用戶意圖的 session 結束（unwanted turn continuation）；保留 stderr notify |
| session-experience-persistence-reminder | Stop | stderr+suppressOutput | 不採用 | 同上 unwanted turn continuation；persistence 提醒應 notify-and-allow-stop |
| stop-worklog-handoff-sync-check | Stop | systemMessage | 不採用 | 同上；worklog sync 提醒不應阻止用戶停止 |
| handoff-auto-resume | Stop | 4a decision:block / 4b stderr | 不採用 | 4a 蓄意 block（語意正確，非 error workaround）；4b notify-allow-stop。additionalContext 非更優 |
| worktree-auto-commit | Stop | 靜默 auto-commit | 不適用 | 無 Claude 溝通需求 |

**採用 = 2（皆 SubagentStop）；不採用 4 + 不適用 1 = 5（皆 Stop）。**

### unwanted turn continuation 風險評估

| 維度 | 結論 |
|------|------|
| 核心風險 | additionalContext「keep turn going」會阻止用戶意圖的 session 停止 |
| SubagentStop | 無此風險（主線程本就繼續）→ 採用安全 |
| Stop | 有此風險（用戶要停卻被續行）→ 5 個 Stop hook 全不採用，保留 systemMessage/stderr/蓄意 block |
| 判別準則 | 「hook 是否要 Claude 在 stop 後**繼續做事**」→ 是且 event=SubagentStop 才採用；event=Stop 則僅蓄意阻擋（handoff 4a 既有）才合理續行 |

### #1 版本釘選（requiredMinimumVersion）依附判定

- 採用 #4（2 hook 改 additionalContext）後，該 hook 在 < 2.1.163 的 CC 上輸出 additionalContext 會被舊 schema 拒（W17-159 即此症狀，恐報 hook error）。
- 判定：**IMP 同步處理相容性**——二擇一交 IMP 決策：(a) 設 requiredMinimumVersion=2.1.163（保證能力存在，但會擋舊 CC 的 teammate）；(b) hook 內 graceful fallback（偵測版本，舊版降級 systemMessage）。
- 傾向 (b) graceful fallback（不強制全員升級，跨專案 sync 友善），由 IMP 評估實作成本後定案。
