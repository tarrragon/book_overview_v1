### 問題根因

本 ANA 統合分析 W1-005 emoji 清理鏈執行期暴露的三個流程/工具教訓，各自有獨立直接因與共享系統因。

**教訓 1 直接因**：W1-005 ANA 由 saffron 單獨執行，multi_view_status = skipped（非主動評估後跳過，是流程缺失）。emoji 偵測範圍靠手工列舉 Unicode 區段（U+1F000-1FFFF + U+2600-27BF），遺漏 U+2139-25B6 區段 9 種 + FE0F 336 處。AC-4「直接斷言依賴=0」僅用 `rg 'Popup Script 載入完成' tests/` 單一關鍵字驗證，未全面掃描 tests/ emoji 字面，致判定「無斷言依賴」（實際 12 檔 48+ 處）。

**教訓 2 直接因**：W1-005.2 acceptance 驗證項僅 `rg emoji=0` + `npm run build:dev`（exit 0），未含 `npm test`。build 綠燈掩蓋了 src emoji→[KEYWORD] 後 tests/ 斷言期待值仍含舊 emoji 字面的不一致（12 檔 48+ 處隱性回歸），直到 W1-005.3 PC-162 複驗才暴露。

**教訓 3 直接因**：scripts/remove-emoji.js 設計為「對指定目錄全量替換」，無檔案/語境排除機制。W1-005.3 對 tests/ 執行時，盲目轉換 background-local-dict.test.js L508-518 antiPatterns 陣列中的 regex 內嵌 emoji（偵測目標字面）。映射內 emoji 變 [KEYWORD] 導致 regex 字元類別 [FAIL] 誤匹配 F/A/I/L 四字元；映射外 emoji 直接移除致 regex 塌縮過寬。已手動還原並加註解警示（L499-500）。

**系統因（三教訓共享）**：
1. ANA 多視角審查為建議而非強制——PM 派發時若未明示，代理人預設跳過
2. 「改變 src 字串輸出字面」類 ticket 缺專門 acceptance 設計規則——build 通過不等於 test 通過（斷言期待值與 src 同源）

### WRAP 決策框架

**W（Widen）**：三教訓各有獨立根因，不應合併為單一防護——分別需要 (a) ANA grep 驗證完整性規範 (b) src 字串變更 acceptance 設計規則 (c) 機械替換偵測目標 error-pattern。

**R（Reality-Test）**：獨立 grep/git 驗證三教訓證據全數屬實（見「重現實驗結果」章節）。既有資產對照：PC-165 部分重疊教訓 2 但聚焦 mock 層非字串字面層；PC-158 聚焦 agent 產出非工具側；language-constraints 規則 3 聚焦禁令非工具排除。三個缺口確認無既有涵蓋。

**A（Attain Distance）**：反向思考——若不建 spawn，這三類問題是否會自然消解？答：否。(a) ANA 多視角仍為建議非強制，下次類似全量 grep 仍可能遺漏區段；(b) 下次 src 字串變更 ticket 仍可能只驗 build；(c) remove-emoji.js 或類似工具仍無排除機制。三缺口均屬系統性而非個案。

**P（Prepare）**：產出三項 spawn 規劃（見下方表格），由 PM 建立對應 ticket。

### 根因分析摘要

| 教訓 | 直接因 | 系統因 | 防護方向 |
|------|--------|--------|---------|
| 1. ANA 多視角缺失 → 範圍二度誤判 | saffron 跳過多視角 + grep 區段列舉不完整 + AC-4 單關鍵字驗證 | ANA 多視角為建議非強制；grep 驗證方法無完整性規範 | grep 驗證完整性規範（全字元掃描 vs 區段列舉） |
| 2. acceptance 只驗 build → 隱性回歸 | W1-005.2 acceptance 無 npm test | 「src 字串輸出變更」類 ticket 缺 acceptance 設計規則 | src 字串變更 acceptance 必含 npm test |
| 3. 機械替換誤傷偵測目標 | remove-emoji.js 無排除機制 | 批次工具缺「偵測目標字面」概念 | error-pattern 化 + 工具排除清單防護 |

### 防護設計

**教訓 1 防護**：建議在 tdd-flow 或 ANA 相關規則中補強——ANA 涉及「全量 grep/regex 範圍判定」時，驗證方法必須說明涵蓋完整性（列舉區段 vs 全字元集掃描 vs Python unicodedata）。非強制多視角（多視角是否強制屬 PM 層決策），而是要求 ANA Solution 明示 grep 範圍涵蓋度及已知盲區。

**教訓 2 防護**：建議新增 acceptance 設計規則——凡 ticket 修改 src/ 字串輸出字面（log 前綴、錯誤訊息、UI 文案等），acceptance 必含 `npm test`（或對應測試套件），理由：tests/ 斷言期待值與 src 字串同源，build 綠燈不覆蓋斷言一致性。可放入 ticket-body-schema 或 tdd-flow 的 acceptance 設計指引。

**教訓 3 防護**：建議新增 error-pattern（PC-176 或下一可用編號）——「機械式批次替換工具對偵測目標字面的破壞模式」。症狀：regex/meta-test 內嵌的待測字元被替換後語意塌縮。根因：工具無語境感知，無法區分「被替換的目標」與「用於偵測的目標」。防護：(a) 工具應支援排除清單（檔案路徑或行號標記）(b) 對 tests/ 執行批次替換前必須 diff 審查含 regex 字面的檔案。
