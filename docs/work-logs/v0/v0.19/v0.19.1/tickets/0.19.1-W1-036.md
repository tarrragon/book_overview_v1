## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 觸發背景

用戶 prompt「這個分支現在已經修復，可以合併」被 `command-entrance-gate-hook` 硬阻擋（exit 2），連續觸發三次。根因：「修復」命中 `ADJUSTMENT_KEYWORDS`，而 `_is_command_keyword_occurrence` 僅涵蓋描述性**後綴**（修復「的」/「過」），未涵蓋描述性**前綴**（「已經」修復），故保守規則 3 誤判為命令意圖。用戶提案：無 ticket 時應引導（開 ticket / 認領 / 確認）而非單純阻止。

### 技術邊界釐清

`UserPromptSubmit` hook 是 Python 子行程，**無法直接呼叫 AskUserQuestion tool**，只能 `exit` 與注入 `additionalContext`。故「引導式」以「放行（exit 0）+ 注入引導指令」間接達成——注入的 context 指示主線程改用 AskUserQuestion。

### 變更摘要（A+B 引導式，方向經 AskUserQuestion 確認）

| # | 變更 | 位置 |
|---|------|------|
| A1 | 新增 `_DESCRIPTIVE_PREFIXES = (已經/已/現已/剛剛/剛/都已)`，於 `_is_command_keyword_occurrence` 加規則 1.5：關鍵字緊接完成態前綴 → 狀態陳述非命令 | `command-entrance-gate-hook.py` |
| A2 | `MANAGEMENT_PATTERNS` 補 `合併 / merge / pull / rebase`（git 操作白名單，永遠放行） | 同上 |
| B1 | 新增 `GUIDANCE_NO_TICKET` 常數（引導訊息，指示主線程用 AskUserQuestion 提供 建立/認領/放行 三選項） | 同上 |
| B2 | `generate_hook_output` 增 `is_guidance` / `guidance_msg` 參數；引導式情境 `should_block=False` 並注入 guidance | 同上 |
| B3 | `main()` 加步驟 5.5：`is_guidance = is_dev_cmd and not is_valid and ticket_id is None`；step 8 改為僅 `should_block` 才 exit 2 | 同上 |
| - | HOOK_METADATA version 3.5.0 → 3.6.0 | 同上 |

### 設計決策（block vs guide 判別放 main()）

block vs guide 的判別用既有回傳值即可推導（`ticket_id is None` = 無 ticket → 引導；`ticket_id 非 None` = ticket 畸形/未認領 → 阻擋），**不改動 `check_ticket_status` 的 4-tuple 契約**，將既有測試破壞面降到最小（僅 2 個 main() 測試從「無 ticket 阻擋」改為「畸形 ticket 阻擋」語意）。

### 範圍邊界

- 「未認領」與「決策樹缺失」「未知狀態」情境（ticket_id 非 None）**維持硬阻擋**，不在本次引導範圍（acceptance 僅指定「無對應 ticket」情境）。
- 未改 `lib/hook_messages.py`（GUIDANCE 用本檔 local 常數，減少跨檔變更）。
