### 評估結論：B2 已被 W1-029 (B3) 完整涵蓋，無實質殘留

採 focused 評估（讀現行 sync-claude-push.py / sync-claude-pull.py VERSION 處理 + sync_exclude_manifest + git log），逐一比對 B2 兩個 acceptance 與 W1-029 B3 的涵蓋關係：

**acceptance 1（VERSION 上游單一權威：pull 後等於上游、本地永不手改）— 已涵蓋**
- push 端（W1-029 B3，docstring 行 35-37 明標「W1-016 B2」）：本地 .claude/VERSION 純鏡像上游、永不手動修改；push 流程在 clone 上 bump 遠端 VERSION，base 錨點由 .sync-state.json 的 last_synced_base_sha 承擔。
- pull 端（W1-028）：.claude/VERSION 走 apply_upstream_delta 三方合併。因本地永不手改（local==base），合併 fast-forward 到上游 → pull 後本地 VERSION = 上游。

**acceptance 2（移除「版號基底修正」反模式）— 已涵蓋**
- grep 確認 pull 端（sync-claude-pull.py）無任何手動 VERSION 修改 / 版號基底修正邏輯（零命中）。
- W1-029 B3 已將 base 錨點從 VERSION 遷移至 .sync-state.json（last_synced_base_sha），結構上消除「VERSION 當 base 而手動設回舊值」的反模式。
- push 端 validate_version_bump 為健全性檢查（防本地污染傳播至遠端），非「把 VERSION 設回舊值」的反模式，屬正向防護應保留。

**強制層（commit-hook 阻擋手改 VERSION）成本效益評估 — 不新增**
- 已有雙層被動防護：(a) pull 三方合併下，誤改 VERSION 最壞為產生衝突寫入 .sync-conflicts/（可見、本地原檔保留、可恢復），不靜默損壞；(b) push validate_version_bump 在版號跳躍異常時 abort，阻止污染傳播至遠端。
- 新增 commit-hook 需維護 + 跨框架 sync 成本，收益（防本地誤改）已由上述被動防護覆蓋，收益 < 成本。

**裁決**：B2 設計意圖（VERSION 上游權威、移除手動基底修正反模式）已於 W1-029 實作 B3 時直接落地並交叉引用本票（docstring「W1-016 B2」）。無未涵蓋的實質殘留，無值得新增的強制層。close as superseded_by 0.19.1-W1-029。
