### 問題根因

同步機制無 base / 共同祖先概念，push/pull 皆無狀態全量 overlay。四個瑕疵（A: pull 全量覆蓋無視本地刪除、B: VERSION 與上游脫鉤、C: push 從磁碟非 git 追蹤態、K/L/M: push 不傳播刪除 / pull 無原子回滾 / pull 不記 sync-state）收斂為單一架構缺陷。

## 重現實驗結果
### 實驗方法

在 /tmp 建立模擬 upstream repo，驗證 git diff --name-status 能否正確表達 base→new delta（Add/Modify/Delete），並在「本地已刪除上游仍存在的檔案」場景下驗證三方合併邏輯。另驗證 shallow clone（--depth 1）能否存取歷史 base commit SHA。

### 實驗執行

**Spike 1**：建 upstream repo 兩個 commit（base→new），`git diff --name-status BASE NEW` 正確輸出 `M VERSION`、`A error-patterns/PC-165.md`、`M rules/core.md`。

**Spike 2**：模擬三方合併——upstream 修改 `hooks/ticket-hook.py`，local 已刪除該檔（遷移）。合併邏輯正確判定：
- `hooks/ticket-hook.py`：upstream modified + local deleted → PRESERVE-LOCAL-DELETE
- `rules/core.md`：both modified → CONFLICT
- `error-patterns/PC-165.md`：upstream added → ADD
- VERSION：upstream modified + local unmodified → TAKE-UPSTREAM

**Spike 3**：shallow clone（--depth 1）無法 `git diff BASE HEAD`（fatal: bad object）。`git fetch --depth=100` 後成功。full clone 直接成功。

### 實驗發現

**已驗證事實**：
1. `git diff --name-status BASE NEW` 完整表達檔案級 A/M/D delta，足以驅動三方合併
2. 三方合併邏輯（base vs local vs upstream）正確處理四種情境：add / take-upstream / conflict / preserve-local-delete
3. shallow clone 無法存取歷史 base SHA；需改用 full clone 或 `fetch --depth=N`

**設計決策（由實驗推導）**：
- pull 必須改用 full clone（非 --depth 1），或 fetch --deepen 取得 base commit
- 推薦 full clone：upstream repo 極小（~500 個檔案），full clone 成本可忽略；避免 depth 猜測
- 合併衝突策略需明確定義（見 Solution）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 統一設計：base snapshot 三方合併模型（A3+B3+C1+K+L+M）

#### 核心概念

引入 `.sync-state.json` 記錄上次同步的上游 commit SHA 作為 base（共同祖先）。pull 和 push 都從 base 出發計算 delta，以三方合併取代無狀態全量 overlay。

#### .sync-state.json 擴充 Schema

```json
{
  "last_push_hash": "23810ebb48e5b85e",
  "last_push_version": "1.39.0",
  "last_push_time": "2026-06-03T11:01:12",
  "upstream_base_sha": "64bafb5...",
  "last_pull_base_sha": "64bafb5...",
  "last_pull_time": "2026-06-03T12:30:00"
}
```

新增欄位：
- `upstream_base_sha`：push 成功後記錄上游 commit SHA（取代 VERSION 作為 base 錨點，B3）
- `last_pull_base_sha`：pull 成功後記錄同步到的上游 commit SHA（M：pull 也記 sync-state）
- `last_pull_time`：pull 成功時間戳

base 取值邏輯：`max(upstream_base_sha, last_pull_base_sha)`（取最近一次同步的 SHA，無論 push 或 pull）。首次同步（.sync-state.json 不存在）fallback 為全量 overlay（向後相容）。

#### Pull 改造（A3 + L + M）

**現行流程**：`clone --depth 1` → `sync_directory`（全量 overlay）→ `cleanup_stale_files`

**新流程**：

1. `git clone`（full clone，非 --depth 1）上游 repo 到 temp_dir
2. 讀取 `.sync-state.json` 取 base SHA（`upstream_base_sha` 或 `last_pull_base_sha`）
3. 若無 base SHA（首次）→ fallback 全量 overlay（現行行為，向後相容）
4. 若有 base SHA → 計算 `git diff --name-status BASE HEAD` 取得上游 delta
5. 對每個 delta 項目執行三方合併：

| 上游 delta | 本地狀態 | 動作 |
|-----------|---------|------|
| A(dd) | 不存在 | 複製上游新檔到本地 |
| A(dd) | 已存在（撞號） | 報告衝突，保留本地，將上游版放 `.conflict/` |
| M(odify) | 存在且內容=base 版 | 取上游版（本地未改） |
| M(odify) | 存在且內容≠base 版 | 衝突：preserve 本地，報告衝突（W7） |
| M(odify) | 不存在（本地刪除） | 保留本地刪除（A3 核心：本地結構遷移不被覆蓋） |
| D(elete) | 存在 | 刪除本地檔（K：上游刪除傳播到本地） |
| D(elete) | 不存在 | 無動作 |

6. **原子套用 + 回滾（L）**：
   - 整個 pull 操作在 temp 複本上完成合併，全部成功後 `shutil.copytree` 原子替換到真正的 `.claude/`
   - 若中途失敗，已有的 backup_dir 即為回滾來源（現行 backup 機制已存在，只需加 auto-restore）
   - 失敗時自動 `shutil.copytree(backup_dir/'.claude', claude_dir)` 還原

7. 成功後寫入 `.sync-state.json`（M）：`last_pull_base_sha = HEAD_SHA`、`last_pull_time = now`

#### Push 改造（C1 + K）

**現行流程**：`copy_filtered`（磁碟 walk）→ `git add -A` → commit → push

**新流程**：

1. `git clone` 上游 repo 到 temp_dir（現行已是 full clone，不變）
2. **C1 核心改動**：以 `git archive HEAD -- .claude` 取本地 git 追蹤態的 `.claude/` 樹，解壓到 temp staging 區，取代 `copy_filtered` 磁碟 walk
   - 具體：`git -C <project_root> archive HEAD -- .claude | tar -x -C <temp_staging>`
   - 再從 temp_staging 複製到 temp_dir（套用 EXCLUDE_PATTERNS 過濾）
3. **K（刪除傳播）**：`git archive` 只包含 tracked 檔案，本地 `git rm` 的檔案自然不在 archive 中 → temp_dir 的 `git diff --cached` 會顯示 D(elete) → `git add -A` → commit 帶刪除
4. 保留現行 `--clean` 模式語意但改為預設行為（因 git archive 已是 tracked-only，clean 等同內建）
5. 成功後寫入 `.sync-state.json`：`upstream_base_sha = 新 push 的 commit SHA`

#### VERSION 處理（B3）

- VERSION 改為純鏡像上游（pull 時直接取上游值，不做三方合併的 conflict 判定）
- 本地禁止手動修改 VERSION（移除「版號基底修正」反模式）
- base 錨點完全由 `.sync-state.json` 的 commit SHA 承擔

#### 合併衝突處理策略

衝突定義：上游修改 + 本地也修改（相對 base）的同一檔案。

處理方式：
1. **檔案級衝突偵測**（非行級）：用 `base content == local content?` 判斷本地是否修改過，簡單明確
2. **衝突報告**：pull 結束時列出所有衝突檔案，標明「upstream version saved to .claude/.sync-conflicts/<path>」
3. **衝突暫存**：上游版本複製到 `.claude/.sync-conflicts/` 目錄，由用戶手動解決
4. **本地優先策略**：衝突時預設保留本地版本（preserve 清單的 preserve 行為已是此策略，擴展到所有衝突）
5. `.sync-conflicts/` 加入 `.gitignore` 和 LOCAL_ONLY / EXCLUDE_PATTERNS（不同步、不 push）

#### D2 PC 編號區段治理 Policy

| 區段 | 範圍 | 用途 |
|------|------|------|
| framework | PC-001 ~ PC-499 | 上游 framework 共用 error-pattern |
| project | PC-500+ | 各專案獨立 error-pattern |

Policy 規則：
- push 時若本地 `.claude/error-patterns/` 含 PC-001~499 且上游不存在 → 警告但不阻擋（可能是 framework 新增）
- pull 時若上游 PC-NNN 與本地 PC-NNN 檔名不同 → 撞號偵測（D3 已在 W1-017 實作）
- 新建 error-pattern 時，專案線從 PC-500 起編（本專案目前最大 PC-171，遠低於 500，安全）

#### 向後相容

- 首次使用（無 .sync-state.json 或無 upstream_base_sha）→ pull fallback 全量 overlay + push fallback copy_filtered，行為與現行完全一致
- 遷移路徑：第一次 push 成功後寫入 upstream_base_sha，之後的 pull 即啟用三方合併
- 不需要一次性 migration 腳本
