### WRAP 四階段分析

#### W — Widen（擴增選項）

問題框架：「PM 用 append-log --section Solution 寫入預寫策略時產生 H2 自訂章節且與 agent 填寫重複」。

擴增視角：
1. **引導層缺口**：append-log CLI 是否提示使用者內容應為 H3？
2. **職責邊界缺口**：PM 預寫 Solution vs agent 填寫 Solution 的分工是否定義？
3. **PC-040 副作用**：prompt ≤30 行限制是否催生 PM 預寫 Solution 反模式？
4. **工具層缺口**：CLI 是否應自動降級 H2→H3 或拒絕含 H2 的內容？

#### R — Reality-test（實證檢驗）

**假設 (a) 已驗證**：execute_append_log 原始碼（track_acceptance.py:576）對非 Execution Log section 直接 new_entry = f\"\\n{content}\"——零引導、零校驗。CLI help、Schema 註解、SKILL.md 均無「內容應為 H3 子節」提示。

**假設 (b) 已驗證**：rg 搜尋 PM.*Solution / PM.*預寫 於 .claude/pm-rules/ 零命中。PM 規則未定義 PM 何時可/不可預寫 Solution，也未定義 PM 預寫與 agent 填寫的合併策略。

**假設 (c) 已驗證**：W17-205 動作鏈顯示 PC-040 擋住 38 行 prompt → PM 被迫將實作策略外推到 ticket md Solution 區。PC-040 本意壓縮 prompt 體量，但未配套定義「外推內容應放哪個 section、用什麼層級」。

#### A — Attain distance（拉開距離）

核心問題不是 PM「寫錯層級」——而是系統在三個層面都沒有引導：
- CLI 工具層：無 heading 校驗
- Schema/規則層：W17-072 章節結構規則只約束 agent，未約束 PM
- 流程層：PM 預寫 Solution 是 PC-040 副作用，但無流程處理此副作用

#### P — Prepare to be wrong（預備錯誤）

最大風險：過度防護（如 CLI 禁止 H2）可能阻礙合法用途。應選擇引導優先、阻擋為次的策略。

### 問題根因

三層根因並存：

| 層級 | 根因 | 證據 |
|------|------|------|
| (a) CLI 引導不足 | append-log 對 Solution 內容零校驗零提示 | track_acceptance.py:576 raw append |
| (b) 職責邊界缺失 | PM 預寫 vs agent 填寫 Solution 無分工定義 | pm-rules/ 零命中 |
| (c) PC-040 副作用 | prompt 長度限制催生 PM 預寫 Solution 反模式 | W17-205 動作鏈 |

### 候選防護方案比較

#### 方案收集（Stage 1）

| 方案 | 層級 | 描述 |
|------|------|------|
| A. CLI auto-prefix H3 | 工具層 | append-log 寫入非 Execution Log section 時，偵測內容首行若為 ## 自動降級為 ### |
| B. CLI H2 warning | 工具層 | append-log 偵測內容含 ## 時輸出 stderr warning（不阻擋），引導使用者改用 H3 |
| C. PM 規則明示 | 規則層 | 在 pm-rules/ 新增條款：PM 預寫 Solution 必須使用 H3 子標題，並定義 PM 預寫 vs agent 填寫的職責邊界 |
| D. Context Bundle 引導 | 流程層 | PM 預寫實作策略應放 Context Bundle 而非 Solution，Solution 留給 agent 填寫 |

#### 評估維度與評分（Stage 2-3）

| 維度 | A. CLI auto-prefix | B. CLI warning | C. PM 規則明示 | D. CB 引導 |
|------|-------------------|---------------|---------------|------------|
| 防護強度 | 高（自動修正） | 中（提示但不擋） | 低（自律層） | 高（職責分離） |
| 實作成本 | 中（改 CLI + 測試） | 低（加 stderr） | 低（寫規則） | 低（寫規則） |
| 誤殺風險 | 高（合法 H2 被降級） | 低（僅提示） | 無 | 無 |
| 根因覆蓋 | (a) 部分 | (a) 部分 | (a)(b) | (a)(b)(c) 全部 |
| 副作用 | 改變內容語意 | 無 | 無 | 改變 PM 工作流程 |

#### 風險分析（Stage 4）

| 方案 | 主要風險 |
|------|---------|
| A | 過度防護：某些場景 PM 可能合法需要 H2（如 Context Bundle 重建），auto-prefix 會破壞內容 |
| B | 防護力弱：PM 可忽略 warning |
| C | 自律依賴：PM 未讀規則時不生效，但配合 custom_h2_checker hook 已有 complete 時 warning |
| D | 流程變更：需要 PM 改變既有習慣（實作策略放 CB 而非 Solution） |

#### 決策建議（Stage 5）

**推薦組合：D（主）+ C（補）+ B（安全網）**

理由：
1. **D 解決根因 (c)**：PC-040 催生的「PM 預寫 Solution」反模式，正確做法是 PM 預寫實作策略放 Context Bundle（已有 PCB 規範基礎），Solution 留給 agent
2. **C 解決根因 (b)**：明示 PM 與 agent 對 Solution 章節的職責邊界
3. **B 解決根因 (a)**：CLI 層加 warning 作為安全網，不阻擋但可見

不推薦 A：auto-prefix H2→H3 改變內容語意，誤殺風險高
