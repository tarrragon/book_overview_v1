### 問題根因

PC-143 原本只記錄 W10-115（lavender Phase 1 spec 對 CLI flag 假設未驗證）。W14-036 執行過程中發現同一模式的二度重現：basil 在 W14-032 ANA 規劃時，依「內化知識」（訓練習得的 hook 語意名稱）將 `decision-quality-guard-hook.py` 列入類別 A 候選清單，但此檔案實際不存在，對應 hook 為 `wrap-decision-tripwire-hook.py`。

根因結構與 PC-143 完全相同：
1. 規劃者以「語意推測」取代「grep/ls 驗證」
2. 偏差對象是「既有資源名稱」（事實陳述，非設計選擇）
3. 偏差在後續 Phase 實作者嘗試讀取或 ls 時才顯現

達到 framework 沉澱門檻（≥2 次不同 agent 跨不同資源類型重現）。

### 影響範圍

- `.claude/error-patterns/process-compliance/PC-143-lavender-cli-assumption-not-verified.md`：需擴充標題、加入 W14-036 案例、更新防護措施、加入決策記錄
- `.claude/agents/AGENT_PRELOAD.md`：需加入規則 8（規劃既有資源名稱前必須驗證存在性）

### 決策：選 A（擴充 PC-143）

W10-115 與 W14-036 共享相同根因結構，防護措施（grep/ls 驗證 + 來源標註）完全相同，僅差別在呼叫者角色和資源類型。合併在 PC-143 泛化標題下，讀者無需查兩個 PC 即能定位，認知負擔最低。
