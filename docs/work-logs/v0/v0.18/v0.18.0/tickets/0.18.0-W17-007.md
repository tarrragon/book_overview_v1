### 重要前提：失配 3 分析部分過時

W17-001 聲稱「沒有 waitpid，parent 靠 polling 猜 exit code」——但 Claude Code runtime **已提供** deferred tools：

| Runtime Tool | 用途 | PM 是否善用 |
|-------------|------|-----------|
| `TaskOutput(task_id, block=false)` | 查 agent running/completed/error | 部分用（PC-050 Step 0.5） |
| `TaskList` | 列所有 active agent | 很少用 |
| `TaskGet` | 取 agent 詳情 | 很少用 |
| `TaskStop` | 強制停 agent | 偶用 |
| `task-notification` 事件 | agent 完成/失敗自動通知 PM | **本 session 正在用**（3 個 Explore 完成通知） |
| `agent-commit-verification-hook` | agent commit 時驗證 | 已配置 |
| `dispatch-stats.py` / `dispatch-active.json` | 持久化 active agent 狀態 | 已配置 |

### W9-002 歷史教訓（Memory 引用）

W9-002 案例：兩天前 PM 處理 PC-050 時結論「缺乏主動查詢代理人狀態的工具」；當天稍後 ToolSearch 立即載入 TaskOutput 等工具 schema 實證可用。教訓：**PM 常誤判「缺工具」實則「未用 ToolSearch 發現」**。本 ticket 必須避免重蹈。

### 本 Session Live 觀察

3 個 Explore 背景派發後：
- PM 沒 polling，靠 `task-notification` 被動接通知——**這已是某種程度的 signal 機制**
- 通知含 `<status>completed</status>`, `<result>...</result>`, `<usage>...</usage>` YAML 結構
- 但 `<result>` 是自由文字，PM 仍必須 Read ticket Solution 確認真實狀態

### 真缺口候選（本 ticket 要驗證）

### 設計目標（若真缺口確認）

**結構化 exit code schema（append-log 新 section）**：
```yaml
## Exit Status
status: success | needs_context | blocked | failed
reason: "一句話狀態說明"
artifacts:
  - type: solution_written
    location: "Solution 區段"
  - type: needs_context_raised
    ref: "append-log NeedsContext 區段"
acceptance_met: [1, 2, 3]
acceptance_unmet: [4, 5]
```

**child-raised signal 協議**：
- Agent 執行中可 `append-log --section "NeedsContext"` 寫入需求
- Hook 監聽該 section 新增事件 → PM 收 PushNotification
- PM 非阻塞回補 Context → agent 繼續
