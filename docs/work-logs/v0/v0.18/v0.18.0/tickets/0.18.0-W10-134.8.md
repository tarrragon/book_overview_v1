### 1. Token 結果對比

| 方案 | 全載入 token | 情境載入 token | 節省量 | 前置成本 |
|------|------------|--------------|-------|---------|
| 現狀（L1 全載入） | ~29K | N/A | 0 | 0 |
| 純重構（X 方向：刪低價值 + 壓縮） | ~10-15K | N/A | 14-19K | 中（審計+重寫） |
| 純 L3（hook-driven 情境載入） | ~7K 核心 | +5-10K 情境 | 12-17K | 高（trigger map 設計+維護） |
| 先重構再 L3 | ~5K 核心 | +3-5K 情境 | 19-21K | 最高（但兩步各自更簡單） |

關鍵發現：純重構（10-15K）與純 L3（7K+情境 5-10K = 12-17K）的最終 token 結果高度重疊。差異不在終態而在路徑成本。

### 2. 三種順序策略評估

| 順序 | 優勢 | 劣勢 | 風險 |
|------|------|------|------|
| A. 先重構再 L3 | 重構縮小 L3 設計空間；trigger map 更簡單（fewer rules to route） | 前置成本最高；重構本身有破壞風險 | 重構過程可能刪錯 rule |
| B. 先 L3 再重構 | L3 立即有收益；重構可在 L3 框架內漸進 | trigger map 基於 15 檔設計，重構後需重新設計 | trigger map 設計兩次 |
| C. 兩者擇一 | 成本最低 | 放棄其中一個的收益 | 無 |

**推薦：先重構，再評估 L3 是否仍有必要（策略 A 的變體）。**

理由：重構將 29K 壓縮至 10-15K 後，L3 的邊際收益從「節省 22K」降至「節省 5-10K」。此時 L3 的 trigger map 維護成本（107 hooks 已存在，再加 rule-routing 層）可能超過 5-10K token 的節省效益。重構後應重新評估 L3 的 ROI。

### 3. 衝突點識別

| 衝突點 | 說明 | 嚴重度 |
|-------|------|--------|
| Trigger map 設計基礎 | L3 trigger map 依賴「哪些 rules 存在」設計路由；重構刪/合併 rules 後 map 失效 | critical |
| CLAUDE.md @ 引用結構 | 重構改變 rules 目錄結構 → CLAUDE.md 入口需同步 → L3 hook 的載入路徑也需同步 | warning |
| Hook 數量負擔 | 107 hooks 已是維護負擔；L3 再加 rule-routing hooks 惡化 | warning |
| 測試覆蓋 | 重構 rules 後需驗證 hook 行為不變；L3 新增 hooks 需新測試 | info |

最嚴重衝突：若先做 L3（設計 trigger map），再做重構（刪/合併 rules），trigger map 需重新設計。這是策略 B 的核心風險。

### 4. 我可能錯在哪裡

1. **低估重構風險**：我假設「刪低價值 rules」是安全的，但每條 rule 背後都有一個 PC pattern 教訓。80/20 審計可能發現「低頻但高價值」的 rules 不應刪除。（W10-134.7 devil's advocate 視角應驗證此點）

2. **高估重構收益**：「115K 壓到 10-15K」假設大量內容可移至 references/。但若 auto-load 是設計意圖（確保 PM 不漏讀），移至 references/ 等於承認「PM 會主動 Read」——歷史證明 PM 不會。

3. **第一輪結論「沿 L3 擴充」可能仍然正確**：即使 rules 可重構，L3 的價值不僅是 token 節省，還有「角色隔離」——代理人不需讀 PM rules，PM 不需讀實作 rules。重構無法達成此目標。

4. **兩者可能不衝突**：重構（精簡內容）和 L3（精準路由）操作不同維度。重構改 rules 內容品質；L3 改 rules 載入時機。兩者可獨立進行，衝突只在「先後順序」而非「互斥」。

### 5. 最終整合策略

**結論：先重構，後條件性 L3。兩者不互斥但有先後依賴。**

分階段執行：

**Phase 1：Rules 審計與重構（P0）**
- 80/20 違規頻率審計（W10-134.6 產出）
- 目標：115K → 40-50K chars（保留核心+高頻，移 references/ 低頻）
- 驗收：PM session token 占比從 15% 降至 7-8%
- 風險防護：每條被移除/移動的 rule 必須有對應 hook 強制執行（不靠 PM 閱讀）

<!-- PC-093-exempt: baseline-gated:15K tokens threshold as Phase 2 gate -->
**Phase 2：重新評估 L3 必要性（Phase 1 完成後）**
- 重構後 rules 若 <= 15K tokens → L3 邊際收益過低（節省 5-8K），不實施
- 重構後 rules 若 > 15K tokens → 實施 L3，但 trigger map 基於重構後的精簡 rule set 設計（更簡單）
- 決策閾值：15K tokens 為分水嶺

**Phase 3（條件觸發）：L3 實施**
<!-- PC-093-exempt: baseline-gated:Phase 2 evaluation gate with 15K threshold -->
- 僅在 Phase 2 評估結果為「實施」時進入
- trigger map 基於重構後 rule set（fewer rules = simpler map）
- 預期 trigger map 複雜度：5-8 個 rule 檔 x 3 情境 = 15-24 條路由（可管理）

**Tripwire**：
- Phase 1 重構後 3 個月內若新增 PM 違規 pattern > 3 條 → 回滾被移除的 rules
<!-- PC-093-exempt: baseline-gated:Phase 2 15K threshold determines L3 necessity -->
- Phase 2 評估若顯示 L3 不必要 → 直接結案，不進 Phase 3
