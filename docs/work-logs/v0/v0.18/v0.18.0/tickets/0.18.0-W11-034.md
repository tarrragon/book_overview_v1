## Problem Analysis

### 問題根因

ticket CLI `complete` 命令會自動 update：
- ticket md frontmatter（status: completed, completed_at, who.history）
- worklog md（append 完成項目到追蹤章節）
- acceptance 勾選狀態

但這些 file modifications **不會自動 staged 也不會自動 commit**。PM/agent 必須手動執行 `git add ... && git commit -m "chore(...): metadata sync post-completion"`。

### 本 session 實證

3 次 ticket complete（W14-046.2 / W11-016 / W11-023），3 次手動 metadata sync commit（0920a72d / 37269dee / 20ec9d7d）。每次重複動作：
1. `git status` 查 modified
2. `git add docs/work-logs/.../tickets/<id>.md docs/work-logs/.../v0.18.0-main.md`
3. `git commit -m "chore(<id>): metadata sync post-completion"`

### 量化頻率

- 本 session: 3/3 ticket complete 都需要手動 metadata commit
- 歷史 commit log grep `chore.*metadata sync` 可量化更長期樣本

### 設計取捨

| 方案 | 自動化程度 | 副作用風險 | git workflow 衝擊 |
|------|----------|---------|---------------|
| A. 保持現狀（手動） | 0% | 無 | 無，但 PM 重複動作多 |
| B. 自動 `git add`（不 commit） | 50% | 可能 stage 不應該的檔案；用戶 commit 時混入未注意的變更 | 低，僅改 staging |
| C. 全自動 add + commit | 100% | 誤 commit 中間狀態；用戶 WIP 變更被夾帶；commit 訊息僵化 | 中，多一個自動 commit；需處理 detached/dirty 狀態 |
| D. 自動 add + PM 互動提示 commit | 75% | 同 B + 互動成本 | 低 |

### 影響範圍

- `.claude/skills/ticket/ticket_system/commands/complete.py`
- `.claude/skills/ticket/SKILL.md`（文件描述）
- `.claude/pm-rules/ticket-lifecycle.md`（流程描述）

**complete 方法 file write 點分析**（lifecycle.py）：

| 寫入點 | 檔案 | 內容 |
|--------|------|------|
| L746 `save_ticket(ticket, ticket_path)` | ticket md | frontmatter（status/completed_at/who.history）+ body（Completion Info） |

4. complete 方法內部已有 `save_ticket` 和 `append_worklog_progress` 的精確呼叫位置，可在此後插入 git add 邏輯

### 方案深度分析

#### 四方案實作路徑與 Edge Case

| 方案 | 實作改動點 | Edge Case 處理 |
|------|-----------|---------------|
| A. 手動 | 無 | 無 |
| B. 自動 add | complete 方法 L752 後插入 `subprocess.run(["git", "add"] + modified_paths)` | detached HEAD：add 無影響（staging 仍可用）；dirty WIP：只 add complete 產生的檔案（精確路徑），不影響其他 unstaged 變更 |
| C. 全自動 commit | B + `subprocess.run(["git", "commit", "-m", msg])` | detached HEAD：commit 會成功但非任何 branch；dirty WIP：若用戶有其他 staged 變更會被夾帶進 commit；merge in progress：commit 會失敗；未設 user.email：commit 會失敗；index.lock 競爭（PC-092） |
| D. 自動 add + 提示 | B + stdout 提示 `git commit -m "chore(<id>): metadata sync"` | 同 B，提示訊息不執行 commit |

#### 同類工具設計對照

| 工具 | 設計 | 理由 |
|------|------|------|
| pre-commit hook | 自動 stage 修改後的檔案（如 formatters） | 確定性：hook 產物可精確預測 |
| editor on-save autosave | 寫檔但不 git add/commit | 寫檔 != 版控決策 |
| npm version | 自動 commit + tag | 版本動作語意明確，一對一 |
| GitHub CLI `gh pr merge` | 不自動 commit（遠端操作） | 操作者已在遠端 |

**關鍵觀察**：pre-commit hook 只做 git add（stage），不做 commit。npm version 做 commit 但語意非常明確（一個版本 = 一個 commit）。ticket complete 更接近 pre-commit 的場景——產物可預測，但 commit 時機應由用戶決定。

#### 風險加權評分

| 風險因子 | 權重 | A（手動） | B（auto add） | C（auto commit） | D（add+提示） |
|---------|------|---------|-------------|----------------|-------------|
| 誤 commit 中間狀態 | 高 | 0 | 0 | 3 | 0 |
| 用戶 WIP 夾帶 | 高 | 0 | 1（僅 staging） | 3 | 1 |
| 改變 git workflow | 中 | 0 | 1 | 3 | 1 |
| PM 重複操作成本 | 中 | 3 | 1 | 0 | 0.5 |
| 實作複雜度 | 低 | 0 | 1 | 3 | 1.5 |
| **加權總分（越低越好）** | — | **4.5** | **3** | **10.5** | **2.75** |

### 推薦方案

**推薦：方案 D（自動 git add + stdout 提示 commit 指令）**

**理由**：
1. **精確 staging**：complete 方法內部已知所有 modified 檔案路徑（ticket md + worklog md + cascade children），可精確 `git add`，不會夾帶其他變更
2. **不改變 commit 決策權**：用戶仍自行決定何時 commit、是否與其他變更合併 commit
3. **消除機械式重複**：PM 不再需要手動 `git status` + `git add <paths>`
4. **最低風險**：不觸碰 commit/merge/detached HEAD 等 git 寫入操作
5. **提示即可複製**：stdout 輸出完整 `git commit -m "chore(<id>): metadata sync post-completion"` 指令，PM 可直接複製貼上

**實作要點**：
- 在 `complete` 方法 L789（return 0 前）插入 git add 邏輯
- 收集 modified_paths：ticket_path + worklog_path + cascade 產物路徑
- `subprocess.run(["git", "add"] + modified_paths, capture_output=True)` — 靜默執行，失敗時 stderr 提示
- stdout 輸出建議 commit 指令
- 新增 `--no-stage` flag 作為 opt-out（PM 不想自動 staging 時使用）
