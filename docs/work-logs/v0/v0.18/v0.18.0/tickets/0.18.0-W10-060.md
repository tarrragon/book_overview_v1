### 根本架構問題

**CC runtime 限制**: PostToolUse(Agent) 在 `run_in_background=true` 下於 **代理人啟動完成** 時觸發（非工作完成）。所有 PostToolUse matcher=Agent 的 Hook 都在**啟動時**跑一次，無法區分「啟動 vs 完成」。

**真完成訊號**: 只有 `task-notification` 事件（CC runtime 內部）才在代理人真完成時觸發。但此事件目前無對應的 Hook 接入點（需確認 CC 是否有 TaskCompleted 類 Hook 事件）。

**現有 Hook 系統可用訊號**:
- PostToolUse:Agent: 代理人啟動時觸發（不可靠判完成）
- agent-commit-verification: 間接訊號（看 git log 是否有新 commit）— 但代理人可能工作但不 commit
- task-notification 事件: 真完成訊號（但 Hook event 名稱未確認）

### 修正方向（全域原則）

1. **識別 background 模式**: Hook 讀 event JSON 的 `tool_input.run_in_background`，若為 true 則推遲邏輯
2. **時機訊號重選**: 改用 `task-notification` / `TaskCompleted` 事件（需 CC Hook event 調研）
3. **內容訊號**: 讀 `tool_response` 的結構化完成標記（但啟動時可能已有 agentId 卻尚無 commit）
4. **保守策略**: PostToolUse(Agent) 不做「完成」判定，改為「派發成功確認」；完成判定留給 task-notification

## Acceptance 措辭修正（驗收後補正）

acceptance-auditor 審查發現：acceptance 第 4 項「字串截斷 bug 修復」勾選但實際未處理。**實際處理**：經 W10-024 basil 於 commit 049105f1 時 grep 確認「`ocs/work-logs/`」在 hook 原始碼中不存在（所有字串皆為完整 `docs/work-logs/`）— 判定為 CC runtime 顯示層問題，非 Hook 原始碼 bug，因此本 ticket 不處理該項。

**正確措辭**應為：「確認字串截斷非原始碼問題（W10-024 grep 驗證），標示 scope 外不處理」

此為事實性修正，Hook 修復本身無缺漏。
### 1. 共用 helper：`is_background_dispatch(tool_input)`

位置：`.claude/hooks/hook_utils/hook_io.py`（並經 `hook_utils/__init__.py` re-export）

語意：讀取 `tool_input.run_in_background`，回傳 bool。任何依賴「代理人工作完成」語意的 PostToolUse(Agent) Hook 都可在 main 早期以此 early-return 跳過完成判定邏輯。附完整 docstring 指向 W10-024 / W10-060 來源。

### 2. P-2 `agent-commit-verification-hook.py`（高風險修復）

在 `main()` 取得 `tool_input` 後、執行任何 git 檢查前，加入 `is_background_dispatch` 判斷；若為背景派發 → 輸出 `DEFAULT_OUTPUT` 並 `sys.exit(0)`。

**理由**：代理人啟動時尚未 commit、未合併、未產生任何變更。執行 uncommitted / unmerged worktree / unmerged feature branch 檢查會產生誤報，並產出錯誤的「PM 立即動作」摘要（PC-070 誘因）。housekeeping 類的 `scan_hook_errors` 同樣只在完成時才有意義（因為它報的是「代理人執行期間」的 Hook 錯誤），統一跳過。

### 3. P-3 `agent-dispatch-logger-hook.py`（低風險修復）

在 `main()` 取得 `tool_input` 後、呼叫 `extract_tool_response` 前加入 early-return。

**理由**：response 在啟動時為空字串，記錄會產生 `response_length=0` 的誤導性 JSONL entry，污染事後失敗追溯。真正完成時機應由 task-notification event 處理。

### 4. P-1 `dispatch-count-guard-hook.py`（不改邏輯，僅加語意備註）

本 Hook 計數的是「PM 已派發的 Agent 數」，非「已完成的 Agent 數」。PostToolUse 在背景派發的「啟動完成」觸發 ≡ 派發完成，計數時機正確。Docstring 加入備註說明為何不套用 `is_background_dispatch` early-return（避免未來維護者誤加）。

### 字串截斷 bug

W10-024 已確認「ocs/work-logs」非原始碼問題（屬 Hook output terminal 渲染截斷），本 ticket 不處理。
