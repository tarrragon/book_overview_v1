### PostToolUse matcher=Agent（派發「完成」後觸發，4 個，本 bug 焦點）

| # | Hook | 時機 bug 風險 | 備註 |
|---|------|-------------|------|
| P-1 | dispatch-count-guard-hook.py | **中** | 限制派發數量，若誤以為「已完成」可能放過超限 |
| P-2 | agent-commit-verification-hook.py | **高** | 驗證代理人 commit。代理人啟動時尚無 commit，會誤報；同時產生「ocs/work-logs/」字串截斷 bug |
| P-3 | agent-dispatch-logger-hook.py | 低 | 派發日誌記錄，誤觸僅影響日誌噪音 |
| P-4 | active-dispatch-tracker-hook.py | **高** | **W10-024 已知 bug**，清除記錄 + 廣播「完成」。正在由 basil 修中 |

### 根本架構問題

**CC runtime 限制**: PostToolUse(Agent) 在 `run_in_background=true` 下於 **代理人啟動完成** 時觸發（非工作完成）。所有 PostToolUse matcher=Agent 的 Hook 都在**啟動時**跑一次，無法區分「啟動 vs 完成」。

**真完成訊號**: 只有 `task-notification` 事件（CC runtime 內部）才在代理人真完成時觸發。但此事件目前無對應的 Hook 接入點（需確認 CC 是否有 TaskCompleted 類 Hook 事件）。

**現有 Hook 系統可用訊號**:
- PostToolUse:Agent: 代理人啟動時觸發（不可靠判完成）
- agent-commit-verification: 間接訊號（看 git log 是否有新 commit）— 但代理人可能工作但不 commit
- task-notification 事件: 真完成訊號（但 Hook event 名稱未確認）

### 修正方向（全域原則）

1. **識別 background 模式**: Hook 讀 event JSON 的 `tool_input.run_in_background`，若為 true 則推遲邏輯
2. **時機訊號重選**: 改用 `task-notification` / `TaskCompleted` 事件（需 CC Hook event 調研）
3. **內容訊號**: 讀 `tool_response` 的結構化完成標記（但啟動時可能已有 agentId 卻尚無 commit）
4. **保守策略**: PostToolUse(Agent) 不做「完成」判定，改為「派發成功確認」；完成判定留給 task-notification

### 建議實作順序

1. **W10-024 basil 修 active-dispatch-tracker**（進行中，模板參考）
2. **P-2 agent-commit-verification**: 等 basil 修完後套用同模板 + 字串截斷 bug
3. **P-1 dispatch-count-guard**: 審查是否有誤觸超限
4. **P-3 agent-dispatch-logger**: 低優先，日誌噪音問題
5. **PreToolUse 群組**: 本次 scope 外，不動
