修復 ReadmooAdapter LAST_RESORT_STRATEGY 頻繁觸發（primary selector 失效）

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

**直接原因**：`getBookElements()` 在 SPA 渲染完成前被多個同步呼叫路徑反覆觸發，每次都走完「primary → fallback → last_resort」三層失敗鏈並輸出 warning，造成 console 看似「LAST_RESORT_STRATEGY 頻繁觸發」。

**深層原因（兩條交互作用）**：

1. **同步 vs 非同步呼叫路徑混用**：
   - `extractAllBooks()`（src/content/adapters/readmoo-adapter.js:576）使用 `waitForBookElements({ timeoutMs: 5000 })`——正確
   - `reportReadyStatus()`（src/content/content-modular.js:200）使用 `waitForBookElements({ timeoutMs: 3000 })`——正確
   - `getPageStatus()`（src/content/content-modular.js:355）使用 `getBookElements()` 即時查詢，**無 wait**
   - `checkPageReady()`（src/content/extractors/book-data-extractor.js:123）使用 `getBookElements()` 即時查詢，**無 wait**

   後兩者在 SPA 載入早期（DOM 未 hydrate）必然落空，每次落空都走完三層選擇器鏈，產出 `FALLBACK_SELECTOR_ATTEMPT` + `LAST_RESORT_STRATEGY` 兩條 warning。

2. **SPA 動態渲染時序**：Readmoo 書庫頁是 Hash-based SPA（`read.readmoo.com/#/library`），DOM 主容器 `.library-item` 由前端 framework 在 `readyState === 'interactive'` 之後才注入。在 hydration 完成前：
   - `.library-item` 不存在 → primary 落空
   - `.book-item / .book-card / .library-book` 不存在（Readmoo 不使用這些 class）→ fallback 落空
   - `a[href*="/api/reader/"]` 也不存在 → last_resort 也找不到容器，最終回傳 `[]`

   這解釋了 W6-012.5 觀察到的 `WAIT_FOR_BOOK_ELEMENTS_TIMEOUT finalCount:0`：即使 5000ms timeout 後 DOM 仍未就緒。

**為何 primary selector 看似「失效」**：實機觀察 `.library-item` 並未真正「失效」——當 `extractAllBooks()` 經 `waitForBookElements` 適當等待後可成功取得 96 個元素（W6-012.5 Step 5 驗證書目存入 Storage 成功）。「失效」假象來自：
- (a) 兩個同步呼叫路徑在早期時序窗口連續觸發三層失敗鏈
- (b) MutationObserver timeout 場景（推測：頁面尚未登入完成或路由切換中時 DOM 從未產生 `.library-item`）

### 影響範圍

| 元件 / 檔案 | 影響 |
|------------|------|
| `src/content/adapters/readmoo-adapter.js` getBookElements()（L125-226） | 三層選擇器鏈無條件觸發 warning，造成 console 噪音 |
| `src/content/content-modular.js` getPageStatus()（L339-364） | 同步呼叫 getBookElements，pre-hydration 必觸發 warning |

不影響 `extractAllBooks()` 主流程（已透過 `waitForBookElements` 防護），不阻擋資料提取功能。

### 相關 Error Pattern

無直接對應 PC error-pattern。本案歸類為「diagnostic noise from premature DOM query」——SPA 環境下對同一 query 設計多條時序敏感度不同的 caller，但底層查詢函式對所有 caller 一視同仁地輸出 warning。屬於 logger level 設計問題，非 selector 設計問題。

可建議建立 PC：「SPA-aware DOM query 不應在同步 caller 中觸發 fallback warning」。

### 實驗執行

#### 步驟 1：定位 getBookElements 三層選擇器邏輯

readmoo-adapter.js L152-204 三層邏輯：

```
Tier 1 (primary)：document.querySelectorAll('.library-item')
   ├─ L156-166: 失敗 → logger.warn('FALLBACK_SELECTOR_ATTEMPT')
   │
Tier 2 (fallback)：嘗試 ['.book-item', '.book-card', '.library-book']
   ├─ L170-204: 失敗 → logger.warn('LAST_RESORT_STRATEGY')
   │
Tier 3 (last_resort)：querySelectorAll('a[href*="/api/reader/"]')
                       → walk up parent (max 10 levels)
                       → 找 classList.contains('library-item')
   ├─ 失敗：return []（finalCount:0）
   └─ 成功：若 directQuery 為 0 但 last_resort > 0 → 'SELECTOR_PARADOX' warn
```

#### 步驟 2：列出 getBookElements 所有呼叫點

```
src/content/adapters/readmoo-adapter.js:
  - L246 waitForBookElements 立即嘗試
  - L282 waitForBookElements tryResolve (mutation + interval)
  - L322 waitForBookElements timeout 最後嘗試
  - L1264 parseDocument
  - L1283 findBookContainers

src/content/content-modular.js:
  - L200 reportReadyStatus → waitForBookElements (async, 3000ms)
  - L355 getPageStatus → getBookElements 即時 (PING handler 呼叫)

src/content/extractors/book-data-extractor.js:
  - L123 checkPageReady → getBookElements 即時
```

#### 步驟 3：對照 SPA 時序

Readmoo 書庫頁載入時序（依 docs/bookstores/readmoo.md + SPA 通則）：

```
t=0    URL hash 改變 → page load 開始
t=t1   readyState=loading
t=t2   readyState=interactive（HTML 解析完）→ content script 注入
t=t3   framework JS 執行（Vue / React hydration）
t=t4   API XHR 取回書目資料
t=t5   .library-item 渲染進 DOM
t=t6   readyState=complete
```

PageDetector 在 t2-t3 之間多次發出 PING / 觸發 reportReadyStatus / checkPageReady。每次同步呼叫 `getBookElements()`（t < t5）→ Tier1 落空 → warn → Tier2 落空 → warn → Tier3 落空 → return []。

每次呼叫產出 2 條 warning（FALLBACK + LAST_RESORT）。若有 5 個同步呼叫點，console 累積 10 條 warn。

#### 步驟 4：解釋 WAIT_FOR_BOOK_ELEMENTS_TIMEOUT finalCount:0

`waitForBookElements` 使用 MutationObserver + interval polling 等到元素出現。timeout=5000ms 後仍 finalCount:0，意味：
- 5 秒內 DOM 始終未注入 `.library-item`，且
- `a[href*="/api/reader/"]` 也始終不存在

可能情境：
1. **頁面尚未完成登入**（redirect 至 SSO 中），content script 在登入頁注入 → 無書目 DOM 可等
2. **路由切換中觸發**：用戶從非 `/library` 路徑切換至 library，但提取被前一頁觸發
3. **書庫為空**（新註冊帳號，0 本書）→ Readmoo 顯示空狀態頁面，無 `.library-item`
4. **網路慢 / API 失敗**：5 秒內 XHR 未完成

W6-012.5 同時觀察到 Step 5 Storage 內有書目（提取成功），表示主流程有時可成功，但仍會在 hydration 早期窗口產出大量 warning（用戶實際看到的 console 噪音）。

### 實驗發現

**已驗證的事實**：

1. **`.library-item` selector 本身正確**（W6-012.5 Step 5 驗證提取成功，書目存入 Storage）
2. **三層選擇器邏輯設計合理**，但 logger 行為過度敏感：每次同步 caller 在 SPA hydration 前呼叫即觸發 warn
3. **至少存在兩個同步無 wait 的呼叫路徑**：`getPageStatus` (PING handler) 和 `checkPageReady` (BookDataExtractor)
4. **每次 getBookElements 在 pre-hydration 觸發兩條 warn**（FALLBACK_SELECTOR_ATTEMPT + LAST_RESORT_STRATEGY）

**仍未驗證的假設**（需 chrome-devtools-mcp 實機補驗，留作 spawned ANA 或 IMP 驗證）：

1. `WAIT_FOR_BOOK_ELEMENTS_TIMEOUT finalCount:0` 的具體 trigger 場景（登入中 / 路由切換 / 空書庫 / 慢網路 哪一個）
2. 是否存在書庫頁有書但 5 秒 wait 仍 timeout 的真實場景（若無，問題純屬 logger 噪音；若有，需重設 wait 策略）
3. `.library-item` 是否在某些 SPA 狀態下變更 class name（Readmoo 改版風險，目前無證據顯示）

**結論**：本問題 80% 為「logger 噪音 + 同步 caller pre-hydration query」設計問題，20% 為「特定情境下 wait 策略不足」。primary selector 本身沒有失效。

### 三層選擇器邏輯流程圖（文字版）

```
┌─────────────────────────────────────────────────────────────┐
│ getBookElements()                                            │
│                                                              │
│ ┌─ Tier 1 (Primary) ──────────────────────────────────────┐ │
│ │ querySelectorAll('.library-item')                       │ │
│ │   ↓ 成功 → return elements                              │ │
│ │   ↓ 失敗 → log.warn('FALLBACK_SELECTOR_ATTEMPT')        │ │
│ └─────────────────────────────────────────────────────────┘ │
│                          ↓                                   │
│ ┌─ Tier 2 (Alternative) ──────────────────────────────────┐ │
│ │ for selector in ['.book-item','.book-card',             │ │
│ │                  '.library-book']:                      │ │
│ │   querySelectorAll(selector)                            │ │
│ │     ↓ 任一成功 → log.info('FALLBACK_SUCCESS') → return  │ │
│ │ 全失敗 → log.warn('LAST_RESORT_STRATEGY')               │ │
│ └─────────────────────────────────────────────────────────┘ │
│                          ↓                                   │
│ ┌─ Tier 3 (Last Resort) ──────────────────────────────────┐ │
│ │ readerLinks = querySelectorAll('a[href*="/api/reader/"]')│ │
│ │ for each link:                                          │ │
│ │   walk up parent (max 10 levels)                        │ │
│ │   if parent.classList.contains('library-item'):         │ │
│ │     containers.add(parent)                              │ │
│ │ ↓ 找到（且 directQuery=0）→ log.warn('SELECTOR_PARADOX')│ │
│ │ ↓ 找不到 → return []                                    │ │
│ └─────────────────────────────────────────────────────────┘ │
└─────────────────────────────────────────────────────────────┘

同步呼叫路徑（無 wait，pre-hydration 必觸發三層失敗）：
  - content-modular.js:355 getPageStatus → PING handler
  - book-data-extractor.js:123 checkPageReady

非同步呼叫路徑（有 MutationObserver wait，正確）：
  - content-modular.js:200 reportReadyStatus (3000ms)
  - readmoo-adapter.js:576 extractAllBooks (5000ms)
```

### Primary Selector 失效根因（重述）

**Primary selector `.library-item` 並未真正失效**。實機觀察 `extractAllBooks` 路徑可成功取得 96 個元素並存入 Storage（W6-012.5 Step 5）。

「LAST_RESORT_STRATEGY 頻繁觸發」是兩個 effect 疊加的假象：

| 假象來源 | 機制 | 觸發頻率 |
|---------|------|---------|
| (1) 同步 caller 無 wait | `getPageStatus` / `checkPageReady` 在 SPA hydration 前同步查詢，三層必失敗 | 每次 PING / 頁面就緒檢查 |
| (2) Logger 級別過敏 | 三層失敗鏈中 fallback / last_resort 均使用 `warn` 級別，正常 SPA 早期時序也會觸發 | 每次 getBookElements pre-hydration |
| (3) WAIT_FOR_BOOK_ELEMENTS_TIMEOUT | 特殊情境（登入中 / 空書庫 / 慢網路）下 5 秒 wait 仍未渲染，最終呼叫 getBookElements finalCount:0 | 偶發，未驗證觸發場景 |

### 改善方案（含 trade-off）

#### 方案 A：降低 logger 噪音（最小改動，推薦先做）

**做法**：
1. `FALLBACK_SELECTOR_ATTEMPT` 從 `warn` 降為 `debug`（首次嘗試失敗為 SPA 載入早期正常現象）
2. `LAST_RESORT_STRATEGY` 從 `warn` 降為 `info`（最後手段觸發為診斷資訊，非錯誤）
3. 保留 `SELECTOR_PARADOX`（L195）為 `warn` ——此為真正異常（last_resort 找到但 primary 沒找到，可能 SPA class 名變更或 DOM 不對稱）
4. `WAIT_FOR_BOOK_ELEMENTS_TIMEOUT` 從 `warn` 降為 `info`（若 finalCount > 0 仍代表提取成功）

**Trade-off**：
- 好處：consol UX 大幅改善，用戶不再誤判系統故障；不改動 selector 邏輯，零回歸風險
- 壞處：真實 selector 失效會更難發現（需依賴 SELECTOR_PARADOX 與 extractAllBooks 最終結果）

#### 方案 B：同步呼叫路徑加入「靜默模式」（中等改動）

**做法**：給 `getBookElements({ silent: true })` 加參數，當 caller 確知自己在 pre-hydration 階段時不輸出 fallback/last_resort warn。

修改點：
- readmoo-adapter.js `getBookElements(options = {})` 加 silent option
- content-modular.js:355 `getPageStatus` 改用 `getBookElements({ silent: true })`
- book-data-extractor.js:123 `checkPageReady` 改用 `getBookElements({ silent: true })`

**Trade-off**：
- 好處：保留 logger 嚴格性（warn 仍代表異常），同時消除已知無害噪音
- 壞處：caller 需理解時序語意；新增 option 增加 API 表面積

#### 方案 C：統一同步 caller 改為非同步 + wait（大改動）

**做法**：讓 `getPageStatus` / `checkPageReady` 也改用 `waitForBookElements`（較短 timeout，如 500ms）。

**Trade-off**：
- 好處：徹底消除 pre-hydration query
- 壞處：影響 PING handler 同步回應語意（需重構 message 流程）；checkPageReady 在 extractAllBooks 上游也需重構；風險高、改動範圍大

#### 方案 D：調整 wait 策略（針對 finalCount:0）

**做法**：在 waitForBookElements timeout 後加入「頁面類型判斷」——若非書庫頁則靜默；若是書庫頁但無書目則回報空書庫（distinct from error）。

**Trade-off**：
- 好處：精準分辨「真實空書庫」vs「載入失敗」
- 壞處：需 PageDetector 提供更可靠的頁面類型判斷；可能與 W6-012.10 race condition 修復重疊

### 建議

**Spawn 三個 follow-up ticket**，依優先級執行：

| Ticket | 類型 | 內容 | 優先級 |
|--------|------|------|--------|
| 1 | IMP | 方案 A：降 logger 級別（FALLBACK→debug, LAST_RESORT→info, WAIT_TIMEOUT 條件式 info/warn） | P2（立即可做，零回歸風險） |
| 2 | IMP | 方案 B：同步 caller 加 `silent` 參數，搭配方案 A | P3（依方案 A 結果決定） |
