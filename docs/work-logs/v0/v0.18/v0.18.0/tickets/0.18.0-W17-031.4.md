### PM 評估結論（2026-05-03，W17-033 完成後）

**結論**：.4 **不能 no-op**，需補強 runqueue 對 stale in_progress / commit-without-complete 的標註。

#### W17-033 實際覆蓋範圍

W17-033 已完成（commit 期間 2026-04-30），落地兩檔：
- `.claude/agents/AGENT_PRELOAD.md` v1.7.0 §2.4「收尾責任」段落
- `.claude/rules/core/agent-definition-standard.md` v1.3.0「執行責任：Ticket 完成（收尾）」章節

效果：agent 自律於 commit 後執行 `check-acceptance --all + complete`；acceptance-gate-hook 為安全網。

#### 殘餘風險矩陣

| 情境 | W17-033 是否覆蓋 | scheduler 補強需求 |
|------|-----------------|------------------|
| Agent 完成後 self-complete（正常路徑） | 是（自律 + hook 安全網） | 不需要 |
| Agent crash / timeout / 未 commit | 否 | 是 — stale in_progress |
| Agent commit 但未 complete（自律規則執行失敗） | 部分（規則已寫，但 hook 不能保證 commit 後一定觸發 complete） | 是 — commit-without-complete |
| Agent 寫 NeedsContext 但未 release | 否 | 是 — needs_context 標註（W17-031.5 處理 hint 層；runqueue 仍應感知） |

#### 既有 staleness 機制盲區

`.claude/skills/ticket/ticket_system/lib/staleness.py` 用 `created` 日期算 stale level（INFO=7d / WARNING=14d / CRITICAL=30d），但：
- 以 `created` 計時，與 in_progress 狀態無關
- 不識別「started_at 很久之前但 completed_at 為 null」的 stale in_progress
- `track_runqueue.py` 完全未引用 staleness（`grep stale` 0 命中）

#### 補強範圍（.4 應做）

1. **runqueue 標註 stale in_progress**：對 `status=in_progress` 且 `started_at` > 閾值（建議 4-8h）的 ticket 加 `[stale]` 標籤
2. **runqueue 標註 commit-without-complete**（弱風險）：偵測 git log 中 ticket ID 已 commit 但 ticket 仍 in_progress 的情況；實作成本較高，建議列為延伸（不在 .4 必做範圍）
3. **新增測試**：`.claude/skills/ticket/tests/test_track_runqueue.py` 補 stale in_progress 顯示測試

#### 與 .1 / .3 的關聯

- .1（讀 Exit Status）：runqueue 讀 handoff JSON 的 needs_context / blocked / failed 標記
- .3（readiness tag）：runqueue 顯示 ready / no-CB 等可派發訊號
- .4（completion 感知）：runqueue 標 stale in_progress / commit-without-complete

三者都動 track_runqueue.py，且都是「runqueue 顯示 ticket 狀態警示」邏輯，存在合併重構機會（.4 完成後可由 thyme 評估是否抽 unified `tag_ticket_status()` helper）。

#### 派發順序確認

維持原計畫：.5 → .2 → .1 → .3 → .4
- .5（thyme 跑中）
- .2 等 .5 完成
- .1 等 .2 完成
- .3 等 .1 完成
- .4 在 .3 之後（共動 track_runqueue.py 序列）

#### 更新後實作策略

從原 strategy 的兩分支（補強 vs no-op）收斂為**補強路徑**：
1. 在 `track_runqueue.py` 加 `_compute_status_tags(ticket)` helper：依 `status` + `started_at` + `completed_at` + 當前時間判斷標籤
2. stale in_progress 閾值常數（建議 `STALE_IN_PROGRESS_HOURS = 6`）抽到 `staleness.py` 統一管理
3. 顯示格式：在 ticket id 後追加 `[stale-in-progress]` 標籤（與既有 `[INFO]` `[WARNING]` 對齊）
4. commit-without-complete 偵測延後（.4 範圍）— 列為 follow-up（如需另建 ticket）
