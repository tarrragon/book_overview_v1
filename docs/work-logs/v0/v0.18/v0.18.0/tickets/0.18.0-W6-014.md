### 選用方案

方案 B（降級兩個 plugin）。

**評估依據**：
- 方案 A（升級 eslint-config-standard）：17.1.0 已是最新版，npm registry 無更新版，無解
- 方案 C（改 neostandard）：neostandard@0.13.0 要求 eslint@^9，但專案固定 eslint@^8.50.0，改動過大
- 方案 D（自訂規則）：工作量最大，排除
- 方案 B：降級 plugin 至 standard 支援範圍，成本最低，規則集無破壞性差異

**修改內容**（package.json devDependencies）：
- `eslint-plugin-n`: `^17.21.3` → `^16.6.2`（對齊 standard 要求的 `^15||^16`）
- `eslint-plugin-promise`: `^7.2.1` → `^6.6.0`（對齊 standard 要求的 `^6.0.0`）

### 額外發現

`jest-chrome@0.8.0` 有獨立的 peer dep 衝突（要求 `jest@^26||^27`，專案用 `jest@29`），且官方無更新版（dist-tags.latest = 0.8.0）。這是既有問題，與 eslint 無關，導致 AC 1「npm install 無 --legacy-peer-deps 完全成功」無法達成。

AC 1 評估：eslint 衝突已修復（install 後無任何 eslint peer dep 警告），但 jest-chrome 衝突仍需 --legacy-peer-deps。建議 PM 重新評估 AC 1 定義，或將 jest-chrome 衝突建獨立 ticket 追蹤。
