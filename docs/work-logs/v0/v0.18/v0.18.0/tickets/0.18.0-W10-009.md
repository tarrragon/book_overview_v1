### Source-of-Truth 約束（W10-052 設計理由）

W10-008 多視角審查（linux 視角）發現「文檔引用」只是表面 DRY 修正——若 Hook 落地時硬編碼觸發條件清單，DRY 違反會從文檔層轉移到實作層（同樣的觸發條件出現在 SKILL.md 與 Hook 程式碼兩處，未來修改 SKILL 時 Hook 不會自動同步）。

**約束內容**：

1. **Single Source of Truth**：觸發條件的權威來源為 `.claude/skills/wrap-decision/references/tripwire-catalog.md`（人類可讀）或對應的 `.claude/config/wrap-triggers.yaml`（機器可讀）
2. **Hook 讀取規則**：`wrap-decision-tripwire-hook.py` 必須在 runtime 載入上述來源，禁止把觸發條件（關鍵字清單、失敗次數閾值、訊號類型）寫死在 Python 程式碼中
3. **格式選擇**：
   - 若 markdown 表格能被 parse 出穩定結構，Hook 可直接讀 markdown
   - 若解析複雜或脆弱，提取為 `.claude/config/wrap-triggers.yaml`，並在 SKILL.md 中註明「機器讀取版本見 .claude/config/wrap-triggers.yaml，兩者必須同步」
   - YAML 提取後需要 CI 或 hook 檢查兩者一致性（避免人類更新 markdown 卻忘記更新 YAML）
4. **acceptance-auditor 驗證**：本 Ticket 完成驗收時，需檢查 Hook 程式碼確實有 `open()` / `yaml.load()` / `parse_markdown()` 等讀取動作，而不是 `TRIGGERS = ["做不到", "沒辦法", ...]` 之類的硬編碼

**為何重要**：

絆腳索（tripwire）的設計重點是「容易被觸發」。如果觸發條件硬編碼在 Hook 內，未來新增訊號（如 W9-004 的 ANA claim 訊號）需要修改 Python 程式碼，而非更新 SKILL 文件。這違反「文檔即配置」原則，也讓 SKILL.md 變成「需要對照程式碼才知道實際生效規則」的失真文件。

### 2. YAML Schema 設計（.claude/config/wrap-triggers.yaml）

本檔案為 Hook 的**唯一觸發條件資料來源**（W10-052 約束）。SKILL.md 與 tripwire-catalog.md 為人類可讀來源，需與本 YAML 雙向同步。

#### 2.1 Schema 結構

```yaml
# .claude/config/wrap-triggers.yaml
# WRAP 絆腳索 Hook 的觸發條件配置
# 權威來源：本檔案（機器可讀）
# 人類可讀對應：.claude/skills/wrap-decision/SKILL.md（description + 觸發條件章節）
# 同步檢查：CI / consistency-check hook（參考 doc-sync-check-hook pattern）

version: "1.0.0"
last_updated: "2026-04-15"
source_of_truth_note: |
  本檔案為 wrap-decision-tripwire-hook.py 的唯一觸發條件來源。
  禁止在 Hook Python 程式碼中硬編碼 triggers / keywords / thresholds。
  若新增訊號，必須同步更新 SKILL.md 與 tripwire-catalog.md。

# 全域設定
settings:
  state_file: ".claude/hook-state/wrap-tripwire-state.json"
  warn_cooldown_seconds: 300  # 同一訊號類型 5 分鐘內不重複提醒
  hook_mode: "advisory"       # advisory: 只提醒不阻擋（exit 0 + stderr）

# 訊號定義
signals:
  - id: consecutive_failures
    enabled: true
    event_sources: ["PostToolUse"]
    tool_matcher: "Task"       # 僅在代理人派發工具觸發
    threshold: 2               # 連續失敗次數門檻
    failure_detection:
      # 「失敗」的認定方式（Phase 3b 須實作其中一項）
      primary: "agent_commit_verification"  # 讀取 agent-commit-verification-hook 的判定結果（若有）
      fallback: "task_output_error_marker"  # fallback：stdin 的 tool_response 包含 error/exception/failed 關鍵字
    ticket_context_key: "current_ticket"    # 跨 ticket 切換重置用
    reset_conditions:
      - agent_success           # 代理人成功完成 → 歸零
      - ticket_switch           # 切換到不同 Ticket → 歸零
      - manual_wrap_invocation  # 手動執行 /wrap-decision → 歸零
    message_template: |
      [WRAP 絆腳索] 連續 {count} 次代理人失敗（Ticket: {ticket_id}）。
      你是有選擇的：
        /wrap-decision        — 系統性擴增選項
        搜尋社群             — 看看有沒有人解決過
        建 Ticket 延後       — 回到核心任務

  - id: restrictive_keywords
    enabled: true
    event_sources: ["UserPromptSubmit"]
    # 關鍵字清單：由 SKILL description Use when 欄位擷取，擴充自 tripwire-catalog
    keywords:
      - "做不到"
      - "沒辦法"
      - "無法"
      - "不支援"
      - "CLI 不支援"
      - "禁止"
      - "不可能"
      - "impossible"
      - "限制性解法"
    match_mode: "substring"        # substring / regex / word_boundary
    case_sensitive: false
    min_prompt_length: 20          # 過短 prompt（如「無法」單詞）不觸發，避免誤報
    reset_conditions:
      - manual_wrap_invocation
    message_template: |
      [WRAP 絆腳索] 偵測到限制性結論（關鍵字：{matched_keyword}）。
      你是有選擇的：
        /wrap-decision        — 搜尋間接方案
        窮盡五問檢查         — tool-discovery.md 規則 1
        確認後再下結論       — 至少執行 ToolSearch

  - id: ana_claim
    enabled: true
    # 偵測來源（Phase 3b 選擇其一作為 primary）：
    #   a) PostToolUse(Bash) 且 command 匹配 `ticket.*claim.*ANA` pattern
    #   b) UserPromptSubmit 含 claim 動作描述 + ANA ticket id
    # 建議採 (a)：偵測精準且 side-effect 清楚
    event_sources: ["PostToolUse"]
    tool_matcher: "Bash"
    command_pattern: "ticket\\s+(track\\s+)?claim.*\\bANA\\b|ana[-_]?claim"
    ticket_type_filter: "ANA"      # 讀取 ticket frontmatter type 欄位確認
    reset_conditions:
      - wrap_section_written       # ticket Solution 章節已有 WRAP 三問內容
      - manual_wrap_invocation
    message_template: |
      [WRAP 絆腳索] 你正在 claim ANA Ticket（{ticket_id}）。
      ANA 分析過程必須執行 WRAP（至少快速模式）。
      你是有選擇的：
        /wrap-decision        — 執行完整 WRAP 流程
        在 Solution 寫三問     — W/A/P 三問作為 claim checkpoint
        降級為快速模式         — 錨點 + W + 基本率 + 決定

# 提醒訊息通用前綴
output:
  stderr_prefix: "[WRAP Tripwire]"
  include_signal_id: true
  include_state_snapshot: false  # 不外洩 state 內容
```

#### 2.2 Schema 契約

| 欄位 | 型別 | 必填 | 說明 |
|------|------|------|------|
| `version` | string | 是 | YAML schema 版本，Hook 啟動時驗證 |
| `signals[].id` | string | 是 | 訊號唯一識別符（對應 state.json 內的 counter key） |
| `signals[].enabled` | bool | 是 | 單一訊號開關，便於分階段上線 |
| `signals[].event_sources` | string[] | 是 | Hook 事件來源（PostToolUse / UserPromptSubmit） |
| `signals[].reset_conditions` | string[] | 是 | 合法值：agent_success / ticket_switch / manual_wrap_invocation / wrap_section_written |
| `signals[].message_template` | string | 是 | 絆腳索訊息，必須包含「你是有選擇的」並列 ≥ 2 選項（哲學約束） |

#### 2.3 Fallback 規則

| 情境 | Hook 行為 |
|------|---------|
| YAML 檔案不存在 | stderr 輸出「[WRAP Tripwire] config missing: {path}」，exit 0（不阻擋） |
| YAML parse 失敗 | stderr 輸出錯誤訊息 + traceback 摘要，exit 0 |
| signal.enabled = false | 該訊號跳過，其他訊號照常運作 |
| schema version 不相符 | stderr warning，嘗試以 best-effort 解析 |

### 3. Hook API 介面

#### 3.1 Input（stdin JSON）

Hook 接收 Claude Code runtime 傳入的 event JSON。兩種事件格式：

**PostToolUse 事件**：

```json
{
  "hook_event_name": "PostToolUse",
  "tool_name": "Task" | "Bash" | ...,
  "tool_input": { "...": "..." },
  "tool_response": { "...": "..." },
  "cwd": "/path/to/project",
  "session_id": "...",
  "transcript_path": "..."
}
```

**UserPromptSubmit 事件**：

```json
{
  "hook_event_name": "UserPromptSubmit",
  "prompt": "user input text",
  "cwd": "/path/to/project",
  "session_id": "..."
}
```

#### 3.2 Output（stderr + exit code）

遵循 `.claude/rules/core/observability-rules.md` 規則 4 雙通道要求。

| 情境 | stderr | 檔案日誌 | exit code |
|------|--------|---------|-----------|
| 無訊號觸發 | 無 | info 記錄事件受理 | 0 |
| 訊號觸發（在 cooldown 內） | 無 | info 記錄命中但抑制 | 0 |
| 訊號觸發（已提醒） | message_template 渲染結果 | info 記錄命中與提醒內容 | 0 |
| 讀取 YAML 失敗 | 錯誤摘要 | error 記錄 traceback | 0（不阻擋） |
| Hook 內部例外 | 錯誤摘要 | error 記錄 traceback | 0（遵循 hook_utils._log_exception 雙通道） |

**重要**：所有訊號型 Hook 統一 `exit 0`（advisory 模式），不阻擋用戶操作。絆腳索哲學 = 提醒「你是有選擇的」而非強制。

#### 3.3 Side Effects

| 副作用 | 說明 |
|--------|------|
| 讀 `.claude/config/wrap-triggers.yaml` | 每次呼叫重讀（避免 stale cache；檔案小） |
| 讀寫 `.claude/hook-state/wrap-tripwire-state.json` | 詳見第 4 節 |
| 讀當前 ticket frontmatter（S3 訊號） | 用 `ticket track` CLI 或直接 parse `docs/work-logs/**/{id}.md` |
| 寫 hook-logs | 透過 `hook_utils.setup_hook_logging` |

#### 3.4 Source 讀取流程

```
Hook 啟動
  → 讀 .claude/config/wrap-triggers.yaml
    ├─ 不存在 → stderr warning + exit 0
    ├─ parse 失敗 → stderr error + exit 0
    └─ 成功 → 取 settings + signals[]
  → 根據 hook_event_name 過濾 signals（event_sources 匹配）
  → 逐訊號偵測（enabled=true 才跑）
  → 命中訊號 → 更新 state.json → 渲染 message_template → stderr 輸出
  → exit 0
```

### 4. State JSON Schema（.claude/hook-state/wrap-tripwire-state.json）

```json
{
  "version": "1.0.0",
  "current_ticket": "0.18.0-W10-009",
  "last_updated": "2026-04-15T12:34:56",
  "signals": {
    "consecutive_failures": {
      "count": 2,
      "last_failure_time": "2026-04-15T12:30:00",
      "last_warned_at": "2026-04-15T12:30:01",
      "last_signal_ticket": "0.18.0-W10-009"
    },
    "restrictive_keywords": {
      "last_matched_keyword": "做不到",
      "last_warned_at": "2026-04-15T12:20:00"
    },
    "ana_claim": {
      "last_claimed_ticket": "0.18.0-W9-004",
      "last_warned_at": "2026-04-15T10:00:00"
    }
  }
}
```

#### 欄位說明

| 欄位 | 型別 | 用途 |
|------|------|------|
| `version` | string | State schema 版本，與 YAML 版本解耦 |
| `current_ticket` | string | 用於 ticket_switch 重置偵測 |
| `signals.{id}.count` | int | 連續失敗計數器（僅 consecutive_failures 使用） |
| `signals.{id}.last_warned_at` | ISO8601 | cooldown 計算基準（避免連續噪音） |
| `signals.{id}.last_signal_ticket` | string | 記錄觸發時的 ticket，切換時歸零 |

#### 不存在時的處理

State 檔不存在 → 視為初始狀態，所有 counter = 0，lastWarnedAt = null。首次寫入時自動建立。

### 5. 三訊號偵測規格

#### 5.1 S1 連續失敗（consecutive_failures）

**「失敗」的認定**（Phase 3b 擇一實作，primary 失效時降級為 fallback）：

| 方式 | 來源 | 優先級 |
|------|------|-------|
| primary | `agent-commit-verification-hook` 的 PostToolUse 判定結果（若輸出到 stdin 可讀的結構） | 高 |
| fallback | `tool_response` 字串中含 `error` / `exception` / `failed` / `timeout` 關鍵字（不區分大小寫） | 低 |

**跨 ticket 切換行為**：

```
讀當前 ticket（從環境或 git branch 或最近 claim 記錄推導）
  → 與 state.current_ticket 比較
    ├─ 相同 → 可累加 count
    └─ 不同 → 寫入 new ticket，count 歸零（ticket_switch 重置）
```

**觸發門檻**：`count >= threshold (=2)` → 渲染 message → 提醒。

**cooldown**：提醒後 `warn_cooldown_seconds` 內不再提醒，避免每次 Task 都觸發。

#### 5.2 S2 限制性關鍵字（restrictive_keywords）

**比對方式**：

```
取 UserPromptSubmit.prompt
  → 若 len(prompt) < min_prompt_length → 跳過
  → 對 keywords 清單逐項比對（match_mode=substring, case_sensitive=false）
  → 命中任一 → 記錄 last_matched_keyword → 渲染 message
```

**關鍵字清單來源**：`.claude/config/wrap-triggers.yaml` 的 `signals[id=restrictive_keywords].keywords`。**禁止在 Python 中硬編碼**（W10-052 約束）。

**誤報防護**：

| 情境 | 處理 |
|------|------|
| 引用他人文件（prompt 含「SKILL 說做不到時要...」） | 本 ticket 不做語義區分，交由 cooldown + PM 判斷（未來可升級） |
| 過短 prompt | `min_prompt_length` 過濾 |
| 中英混用（「impossible 啦」） | substring 比對仍能命中 |

#### 5.3 S3 ANA claim（ana_claim）

**偵測來源（建議 primary = PostToolUse Bash + command pattern）**：

```
hook_event_name == "PostToolUse" AND tool_name == "Bash"
  → 取 tool_input.command
  → regex 匹配 command_pattern (ticket claim ANA 或 ana-claim)
  → 匹配成功 → 提取 ticket_id
  → 讀 ticket frontmatter 的 type 欄位
    ├─ type == "ANA" → 觸發訊號
    └─ type != "ANA" → 忽略（避免 IMP/DOC 誤觸發）
```

**為何不採 UserPromptSubmit**：claim 動作的文字表述變異大（「我來做 W9-004」「claim W9-004」），PostToolUse(Bash) 偵測精準且對應實際系統動作。

**提醒抑制條件**：若 ticket Solution 章節已含 `WRAP 三問` 或 `W（Widen）` 章節標題，則視為已執行 WRAP → 不重複提醒（對應 `wrap_section_written` 重置條件）。

### 6. 重置條件規格

三個重置條件的偵測方式：

| 重置條件 | 偵測方式 | 實作位置 |
|---------|---------|---------|
| `agent_success` | PostToolUse(Task) 且無失敗訊號 → 重置對應 ticket 的 consecutive_failures.count = 0 | S1 訊號處理邏輯內 |
| `ticket_switch` | 每次 Hook 執行開始時，讀當前 ticket 與 state.current_ticket 比對，不同則全訊號 count 歸零 | Hook 主流程前置檢查 |
| `manual_wrap_invocation` | PostToolUse(SlashCommand) 或 UserPromptSubmit 含 `/wrap-decision` → 所有訊號 counter 歸零 | Hook 主流程 |

**額外**：`wrap_section_written`（S3 專用）— 偵測 ticket Solution 章節文字，用於避免重複提醒已執行 WRAP 的 ANA claim。

**當前 ticket 推導順序**（fallback chain）：

1. 環境變數 `TICKET_ID`（若 runtime 有提供）
2. `.claude/dispatch-active.json` 最近一筆派發的 ticket
3. git branch 名稱匹配 `feat/{ticket-id}` pattern
4. 無法推導 → state.current_ticket 暫不更新（維持上次值），count 不增不減
