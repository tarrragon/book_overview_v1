#### [Acceptable] 1. Strategy Pattern 對只有 3 個訊號的場景是合理但邊緣

**現狀**：S1/S2/S3 各自一個類別 + SIGNAL_STRATEGIES dict，main() 用 for-loop 分派。

**位置**：§1.5 derive_ticket（環境變數 → dispatch-active.json → git branch → None）

**Linus 視角**：4 層 fallback 本身是「特殊情況堆疊」訊號。Good taste 問法是「能不能讓 dispatch-active.json 永遠是 source of truth，讓其他 3 層消失？」但這裡實際反映**真實環境的不確定性**（subagent 環境無 TICKET_ID、worktree git branch 可能才是權威）——是「外部世界混亂」而非「設計混亂」，接受。

#### 1.3 Signal Strategy Pattern（每訊號獨立類別）

```
class SignalStrategy(Protocol):
    signal_id: str
    def detect(event, state, signal_def, current_ticket) -> DetectResult
    def apply(state, result) -> State            # 更新 counter / last_warned_at

class ConsecutiveFailuresStrategy:      # S1
    def detect(...):
        if event.tool_name != signal_def.tool_matcher:          # Task
            return DetectResult(hit=False)
        is_failure = primary_detect_failure(event) \
                     or fallback_keyword_detect_failure(event)  # 見 §5.1
        if is_failure:
            new_count = state.s1.count + 1
            return DetectResult(hit=True, count=new_count,
                                should_warn=(new_count >= threshold))
        else:
            return DetectResult(hit=False, reset=True)          # agent_success

class RestrictiveKeywordsStrategy:      # S2
    def detect(...):
        prompt = event.prompt
        if len(prompt) < signal_def.min_prompt_length:
            return DetectResult(hit=False)
        matched = find_first_keyword(prompt, signal_def.keywords,
                                     case_sensitive=False)
        return DetectResult(hit=bool(matched), matched_keyword=matched,
                            should_warn=bool(matched))

class AnaClaimStrategy:                 # S3
    def detect(...):
        if event.tool_name != "Bash":
            return DetectResult(hit=False)
        m = regex(signal_def.command_pattern).search(event.command)
        if not m:
            return DetectResult(hit=False)
        ticket_id = extract_ticket_id(event.command)
        fm = read_ticket_frontmatter(ticket_id)     # 見 §3 工具重用策略
        if fm is None or fm.type != signal_def.ticket_type_filter:
            return DetectResult(hit=False)
        if wrap_section_already_written(ticket_id):    # 抑制
            logger.info("wrap_section_written suppresses S3")
            return DetectResult(hit=False, log_reason="wrap_section_written")
        return DetectResult(hit=True, ticket_id=ticket_id, should_warn=True)

SIGNAL_STRATEGIES = {
    "consecutive_failures": ConsecutiveFailuresStrategy(),
    "restrictive_keywords": RestrictiveKeywordsStrategy(),
    "ana_claim": AnaClaimStrategy(),
}
```

#### 1.4 save_state_atomic 虛擬碼

```
save_state_atomic(path, state):
    parent = path.parent
    parent.mkdir(parents=True, exist_ok=True)
    # 使用同目錄下 NamedTemporaryFile，確保 os.replace 在同 filesystem（原子性前提）
    with NamedTemporaryFile("w", dir=parent, delete=False,
                            prefix=".state-", suffix=".tmp") as tmp:
        json.dump(state.to_dict(), tmp, ensure_ascii=False, indent=2)
        tmp.flush()
        os.fsync(tmp.fileno())                      # 確保寫入磁碟
        tmp_path = tmp.name
    os.replace(tmp_path, path)                      # 跨平台原子 rename
    # 失敗回滾：若 os.replace 失敗，tmp 檔保留於 parent 下以利 debug；下次 hook 會重建
```

#### 1.5 derive_ticket fallback chain

```
derive_ticket(event, cwd):
    # 層 1：環境變數
    t = os.environ.get("TICKET_ID")
    if t:
        return t
    # 層 2：dispatch-active.json（最近一筆派發）
    try:
        data = json.loads(Path(cwd, ".claude/dispatch-active.json").read_text())
        if isinstance(data, list) and data:
            latest = max(data, key=lambda x: x.get("dispatched_at", ""))
            t = latest.get("ticket_id")
            if t: return t
    except (FileNotFoundError, json.JSONDecodeError) as e:
        logger.warning(f"dispatch-active.json unreadable: {e}")  # 不 raise，進下一層
    # 層 3：git branch 匹配 feat/{ticket-id}
    try:
        branch = subprocess.check_output(
            ["git", "branch", "--show-current"], cwd=cwd, text=True, timeout=2
        ).strip()
        m = re.match(r"feat/(?P<id>[0-9.]+-W\d+-\d+)", branch)
        if m:
            return m.group("id")
    except (subprocess.SubprocessError, FileNotFoundError):
        logger.info("git branch derivation failed; non-git repo?")
    # 層 4：全部失敗 → None → caller 維持 state.current_ticket
    return None
```

```
1. PEP 723 metadata header (# /// script ... dependencies = ["pyyaml"])
2. Imports + path constants (CONFIG_PATH, STATE_PATH)
3. _now() helper  ←  時間注入點（§2 #5）
4. Data classes:  Config / SignalDef / State / DetectResult (dataclass)
5. load_config()
6. load_state() / save_state_atomic()
7. derive_ticket()
8. is_manual_wrap_invocation()
9. in_cooldown() / mark_warned()
10. render_message()  ←  YAML message_template format
11. Signal strategies (S1 / S2 / S3)   ← open-closed 擴充點
12. SIGNAL_STRATEGIES dict
13. main()  ← entry point，< 50 行
14. if __name__ == "__main__": main()
```

單檔目標行數：**~350-450 行**（含 docstring / type hints）。若超過 500 行，拆分 strategies 到 `lib/wrap_signals.py`。

### 4. State Write 並發策略

| 威脅 | 策略 | 理由 |
|------|------|------|
| 部分寫入 corrupt JSON | tmp file + `os.replace` | POSIX 保證 rename 原子性；Windows `os.replace` 同等 |
| 寫入中斷（power fail） | `os.fsync(tmp.fileno())` 後才 replace | 確保 tmp 檔已落盤 |
| 多 Hook 並發寫入 | **last-write-wins** | multi-hook 併發場景罕見；若衝突代價僅「某次 count 遺失」而非 corruption — acceptable |
| 讀到寫入中的 state | 不會發生 | 讀端看到的要嘛是舊檔、要嘛是完整新檔（rename 原子性） |
| tmp 檔洩漏 | 失敗路徑保留 tmp 檔於同目錄，下次執行時 best-effort 清理 `.state-*.tmp` 老檔（> 1 小時） | 不阻擋運行；debug 友善 |

**跨平台驗證點**（Phase 3b 測試）：H5 並發測試在 Linux / macOS 驗證；Windows 走 CI best-effort（專案未承諾 Windows 支援）。

**拒絕 flock 理由**：`fcntl` 僅 Unix；引入 `portalocker` 增加依賴複雜度；絆腳索哲學容許 last-write-wins。

### 5. Ticket 推導 Fallback Chain 實作步驟

| 層 | 來源 | 失敗行為 | 日誌級別 | 是否阻擋 |
|---|------|---------|---------|---------|
| 1 | `os.environ["TICKET_ID"]` | KeyError → 進層 2 | debug（日常無值） | 否 |
| 2 | `.claude/dispatch-active.json` | FileNotFoundError / JSONDecodeError / 空 list → 進層 3 | warning（parse 失敗）；info（不存在） | 否 |
| 3 | `git branch --show-current` + regex `feat/{ticket-id}` | SubprocessError / timeout / 無匹配 → 進層 4 | info（非 git repo / 無匹配） | 否 |
| 4 | 全部失敗 | `return None` → caller 維持 `state.current_ticket`，counter 不變 | info（「unable to derive ticket」） | 否 |

**關鍵設計**：所有層 fallback 失敗**均不 raise**，Hook 永遠 exit 0。層 4 的 `None` 讓 state.current_ticket 維持舊值（避免切換誤觸發 reset）。

### 6. YAML Parse 與快取策略

| 決策 | 選擇 | 理由 |
|------|------|------|
| 每次呼叫重讀 vs session cache | **每次重讀** | (a) YAML 檔 < 2KB，I/O 成本可忽略；(b) session cache 需共享記憶體（Hook 每次獨立 process，無法共用）；(c) 避免 stale cache |
| Parse 失敗 fallback | `return None` → main() 提前退出（advisory） | 絆腳索哲學：失敗不阻擋 |
| Schema 驗證 | 輕量自驗（dataclass 預設值補齊缺欄位）+ version check | 否決 `jsonschema` 依賴：增加 dependency；輕量驗證已足夠 |
| 缺欄位處理 | dataclass 預設值；signal 缺 `message_template` → 該 signal 跳過 + log warning | 容錯優先 |
| Version 不符 | warning + best-effort（仍嘗試讀 signals） | sage I3 測試期望 |

#### 1.3 測試組織原則（Sociable Unit Tests）

- **單元邊界**：以「Hook 主流程函式」為單元（接收 event JSON + state + yaml → 產出 stderr + state mutation + exit code），不切到 individual function 層級
- **不 Mock**：Domain 邏輯（YAML parse、訊號比對、message rendering）使用真實實作
- **只 Mock**：外部依賴（filesystem 路徑、stdin、subprocess、datetime）
- **行為導向**：測試 Hook 對外可觀察行為（stderr 內容、state.json 變化、exit code），不測試內部函式呼叫順序

### 4. Edge Cases 補充

#### 4.1 YAML 讀取多種失敗情境

#### 4.2 state.json 競爭條件（多 Hook 並發）

| 情境 | 測試 ID | 期望行為 |
|------|--------|---------|
| 兩個 Hook 同時讀取相同 state | 補 H5 | last-write-wins 但無 corrupted JSON（建議實作 atomic write：寫 .tmp 後 rename） |
| state 檔損毀（JSON parse 失敗） | 補 H6 | 視為初始狀態重建；hook-logs 記錄重建事件 |
| state 寫入時 disk full | 補 H7 | stderr error + exit 0；不影響後續事件 |

**設計建議（給 Phase 3a/3b）**：state 寫入採 `fcntl.flock` 或 atomic rename pattern，避免並發 corruption。

#### 4.3 cooldown 邊界

| 情境 | 測試 ID | 期望行為 |
|------|--------|---------|

| last_warned_at 為 null（首次） | C2 預設 | 直接提醒 |

#### 4.4 跨 ticket 切換 state 推導失敗 fallback chain

| 情境 | 測試 ID | 期望行為 |
|------|--------|---------|

| 情境 | 測試 ID | 期望行為 |

### 5. 測試檔案結構建議

#### 5.1 路徑與分檔策略

**主檔案**：`.claude/hooks/tests/test_wrap_decision_tripwire_hook.py`

**是否拆分**：建議**單檔**。理由：

| 考量 | 結論 |
|------|------|
| 測試總數約 50+（含補強） | pytest 對單檔可良好運作 |

### 3. Hook API 介面

#### 3.1 Input（stdin JSON）

Hook 接收 Claude Code runtime 傳入的 event JSON。兩種事件格式：

**PostToolUse 事件**：

```json

  "tool_response": { "...": "..." },
  "cwd": "/path/to/project",
  "session_id": "...",
  "transcript_path": "..."
}
```

**UserPromptSubmit 事件**：

```json
{
  "hook_event_name": "UserPromptSubmit",
  "prompt": "user input text",
  "cwd": "/path/to/project",
  "session_id": "..."
}
```

#### 3.2 Output（stderr + exit code）

遵循 `.claude/rules/core/observability-rules.md` 規則 4 雙通道要求。

| 情境 | stderr | 檔案日誌 | exit code |
|------|--------|---------|-----------|
| 無訊號觸發 | 無 | info 記錄事件受理 | 0 |
| 訊號觸發（在 cooldown 內） | 無 | info 記錄命中但抑制 | 0 |
| 訊號觸發（已提醒） | message_template 渲染結果 | info 記錄命中與提醒內容 | 0 |
| 讀取 YAML 失敗 | 錯誤摘要 | error 記錄 traceback | 0（不阻擋） |
| Hook 內部例外 | 錯誤摘要 | error 記錄 traceback | 0（遵循 hook_utils._log_exception 雙通道） |

**重要**：所有訊號型 Hook 統一 `exit 0`（advisory 模式），不阻擋用戶操作。絆腳索哲學 = 提醒「你是有選擇的」而非強制。

#### 3.3 Side Effects

| 副作用 | 說明 |
|--------|------|
| 讀 `.claude/config/wrap-triggers.yaml` | 每次呼叫重讀（避免 stale cache；檔案小） |
| 讀寫 `.claude/hook-state/wrap-tripwire-state.json` | 詳見第 4 節 |
| 讀當前 ticket frontmatter（S3 訊號） | 用 `ticket track` CLI 或直接 parse `docs/work-logs/**/{id}.md` |
| 寫 hook-logs | 透過 `hook_utils.setup_hook_logging` |

#### 3.4 Source 讀取流程

```
Hook 啟動
  → 讀 .claude/config/wrap-triggers.yaml
    ├─ 不存在 → stderr warning + exit 0
    ├─ parse 失敗 → stderr error + exit 0
    └─ 成功 → 取 settings + signals[]
  → 根據 hook_event_name 過濾 signals（event_sources 匹配）
  → 逐訊號偵測（enabled=true 才跑）
  → 命中訊號 → 更新 state.json → 渲染 message_template → stderr 輸出
  → exit 0
```

| 方式 | 來源 | 優先級 |
|------|------|-------|
| primary | `agent-commit-verification-hook` 的 PostToolUse 判定結果（若輸出到 stdin 可讀的結構） | 高 |
| fallback | `tool_response` 字串中含 `error` / `exception` / `failed` / `timeout` 關鍵字（不區分大小寫） | 低 |

**跨 ticket 切換行為**：

```
讀當前 ticket（從環境或 git branch 或最近 claim 記錄推導）
  → 與 state.current_ticket 比較
    ├─ 相同 → 可累加 count
    └─ 不同 → 寫入 new ticket，count 歸零（ticket_switch 重置）
```

**觸發門檻**：`count >= threshold (=2)` → 渲染 message → 提醒。

**cooldown**：提醒後 `warn_cooldown_seconds` 內不再提醒，避免每次 Task 都觸發。

#### 5.2 S2 限制性關鍵字（restrictive_keywords）

**比對方式**：

```
取 UserPromptSubmit.prompt
  → 若 len(prompt) < min_prompt_length → 跳過
  → 對 keywords 清單逐項比對（match_mode=substring, case_sensitive=false）
  → 命中任一 → 記錄 last_matched_keyword → 渲染 message
```

**關鍵字清單來源**：`.claude/config/wrap-triggers.yaml` 的 `signals[id=restrictive_keywords].keywords`。**禁止在 Python 中硬編碼**（W10-052 約束）。

**誤報防護**：

| 情境 | 處理 |
|------|------|
| 引用他人文件（prompt 含「SKILL 說做不到時要...」） | 本 ticket 不做語義區分，交由 cooldown + PM 判斷（未來可升級） |
| 過短 prompt | `min_prompt_length` 過濾 |
| 中英混用（「impossible 啦」） | substring 比對仍能命中 |

#### 5.3 S3 ANA claim（ana_claim）

**偵測來源（建議 primary = PostToolUse Bash + command pattern）**：

```
hook_event_name == "PostToolUse" AND tool_name == "Bash"
  → 取 tool_input.command
  → regex 匹配 command_pattern (ticket claim ANA 或 ana-claim)
  → 匹配成功 → 提取 ticket_id
  → 讀 ticket frontmatter 的 type 欄位
    ├─ type == "ANA" → 觸發訊號
    └─ type != "ANA" → 忽略（避免 IMP/DOC 誤觸發）
```

**為何不採 UserPromptSubmit**：claim 動作的文字表述變異大（「我來做 W9-004」「claim W9-004」），PostToolUse(Bash) 偵測精準且對應實際系統動作。

**提醒抑制條件**：若 ticket Solution 章節已含 `WRAP 三問` 或 `W（Widen）` 章節標題，則視為已執行 WRAP → 不重複提醒（對應 `wrap_section_written` 重置條件）。

### 6. 重置條件規格

三個重置條件的偵測方式：

| 重置條件 | 偵測方式 | 實作位置 |
|---------|---------|---------|
| `agent_success` | PostToolUse(Task) 且無失敗訊號 → 重置對應 ticket 的 consecutive_failures.count = 0 | S1 訊號處理邏輯內 |

| `manual_wrap_invocation` | PostToolUse(SlashCommand) 或 UserPromptSubmit 含 `/wrap-decision` → 所有訊號 counter 歸零 | Hook 主流程 |

**額外**：`wrap_section_written`（S3 專用）— 偵測 ticket Solution 章節文字，用於避免重複提醒已執行 WRAP 的 ANA claim。

**當前 ticket 推導順序**（fallback chain）：

1. 環境變數 `TICKET_ID`（若 runtime 有提供）
2. `.claude/dispatch-active.json` 最近一筆派發的 ticket
3. git branch 名稱匹配 `feat/{ticket-id}` pattern
4. 無法推導 → state.current_ticket 暫不更新（維持上次值），count 不增不減
