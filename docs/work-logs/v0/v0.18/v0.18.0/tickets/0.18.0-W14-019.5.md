## Problem Analysis

### 問題根因

W14-019 §5.11 定義規則 5 監測要求（每月檢視、每季檢視、3 個月條件驗證、失敗退出條件），但全為**抽象條款**：

- 「PM 記錄觸發頻率與情境」—— 記錄在哪？什麼格式？
- 「用戶主動評估是否保護 autonomy」—— 透過什麼介面？
- 「介入是否防止傷害」—— 如何量化「傷害」？
- 「利益是否大於風險」—— 如何量化「利益/風險」？

核心問題：**benevolent paternalism 條件 2/3 的可量化判據缺失**。若 3 個月後仍無可操作指標，規則 5 無法進入實證驗證，§5.11 的退出條件形同虛設（永遠無法證否）。

### 實驗執行

**步驟 1：既有記錄通道盤點**

| 通道 | 用途 | 規則 5 相關記錄可行性 |
|------|------|---------------------|
| auto-memory | 跨 session 用戶偏好與學習 | 高（已有 feedback/project 分類，可新增 rule5 類） |
| ticket 日誌 | 任務執行記錄 | 中（可在 ANA/IMP ticket 提及規則 5 觸發，但需手動） |
| worklog | 版本主線日誌 | 低（粒度過粗，不適合高頻記錄） |

**結論**：既有通道可支撐半自動記錄（memory + ticket），但**完全自動化需新 Hook**。

**步驟 2：候選指標可量化性試算**

| 指標類別 | 候選指標 | 資料來源 | 自動化程度 |
|---------|---------|---------|-----------|
| Layer 1-4 觸發頻率 | 反射檢查觸發次數/月 | PM 手動記錄或 Hook | 低→中 |
| Layer 2 指標分布 | 8 項動作觸發點命中次數 | PM 記錄 AUQ 前自檢結果 | 低 |
| Layer 3 訊號累計 | 多訊號累計判斷觸發次數 | PM 記錄訊號偵測 | 低 |
| Layer 4 依賴監測 | 跨 session 依賴偵測觸發次數 | PM 每月檢視歷史 session | 中 |
| 用戶主動授權 | §5.7 active consent 聲明次數 | memory 或 ticket 記錄 | 中 |
| 用戶否決規則 5 | 規則 5 條款被用戶推翻次數 | memory 或 PC pattern | 高 |
| 疲勞訊號 → 降權 | PM 偵測疲勞並降權次數 | 手動記錄 | 低 |
| 保護案例 | 規則 5 防止具體傷害的案例 | 回溯分析 session | 低 |
| 摩擦案例 | 規則 5 造成的不必要阻礙案例 | 用戶抱怨或 retrospective | 中 |

**步驟 3：紙筆模擬 1 次月度自檢**

假設 2026-05-17 執行首次月度自檢（規則 5 落地滿 1 個月）：

模擬流程：
1. 翻閱本月 session memory → 找出 rule5 相關記錄 → 統計觸發次數、情境分布
2. 對照 Layer 1-4 自問：「本月最明顯的權力不對等情境是什麼？規則 5 啟動了嗎？」
3. 用戶訪談（3-5 題）：「本月你是否感到 Claude 過度強勢？何時？規則 5 是否有實際保護？」
4. 產出月度 ANA ticket：`0.18.0-W17-001` 或類似，記錄指標值 + 質化觀察

**模擬結果**：流程可執行，但依賴 PM 養成「每次觸發規則 5 就寫 memory」的習慣。**失敗風險高**（PM 忘記記錄）。

**已驗證事實**：
1. 既有通道（memory + ticket）可支撐手動記錄，但需 PM 養成習慣
2. Layer 4 跨 session 依賴是最重要但最難量化的指標
3. 「防止傷害案例」vs「摩擦案例」的對比是 benevolent paternalism 條件 2/3 的核心量化來源

**未驗證假設**：
1. PM 會穩定執行月度記錄（驗證需 1 個月實測）
2. 用戶會配合季度訪談（驗證需 1 個季度實測）
3. 3 個月週期足夠累積顯著案例（樣本量未知）

## WRAP 四階段分析

### W（Widen Options）候選方案擴增

| 方案 | 描述 | 可操作性 | 自動化程度 | 認知負擔 |
|------|------|---------|-----------|---------|
| A. 純手動記錄 | PM 每次觸發規則 5 機制時寫 memory | 高 | 零 | 高（依賴習慣） |
| B. Hook 自動偵測 | 開發 rule5-trigger-hook 自動記錄觸發 | 低（技術上難偵測「反射檢查」這種主觀行為） | 高 | 低 |
| C. 事後歸納 | 每月 PM 回顧 session memory 抽取觸發情境 | 中 | 低 | 中（月度集中） |
| D. 用戶回饋導向 | 主依賴用戶季度訪談，PM 記錄為輔 | 中 | 零 | 中（用戶參與） |
| E. 對比案例分析 | 每月找 2-3 個「規則 5 有觸發」vs「類似但未觸發」案例對比 | 高（直接對應條件 2/3） | 低 | 中 |
| **F**. 組合方案（A+C+D+E） | 立即手動記錄 + 月度歸納 + 季度訪談 + 對比案例 | 最高 | 低-中 | 高但分散 |

### R（Reality Test）真根因驗證

| 假設 | 驗證狀態 | 證據 |
|------|---------|------|
| PM 觸發規則 5 機制可被自動偵測 | **部分反駁** | Layer 1 反射檢查是主觀自問，Hook 無法偵測；Layer 2 部分動作（AUQ 前自檢）技術上可偵測但需 PM 主動標記 |
| 「防止傷害」可量化 | **部分驗證** | 對比案例法（方案 E）提供半量化路徑，但樣本需 10+ 才有統計意義 |
| 3 個月週期足夠 | **未驗證** | §5.11 定義但無依據；多數 benevolent paternalism 研究採 6 個月以上 |
| 用戶會配合季度訪談 | **未驗證** | 依賴用戶意願，存在 response bias（用戶可能報喜不報憂） |
| auto-memory 可承載規則 5 記錄 | **已驗證** | 既有 feedback/project 分類可擴充，已有 memory 寫入習慣 |

### A（Attain Distance）機會成本

**方案 F（組合）的機會成本**：
- 直接成本：PM 每月花 30-60 分鐘歸納；每季安排用戶訪談；新增 auto-memory 分類
- 延遲成本：3 個月後才能產出首份條件驗證報告
- **收益**：規則 5 能進入實證驗證，不流於形式主義

**方案 B（Hook 自動）的機會成本**：
- 開發成本：~1 週設計 Hook + 定義觸發規則
- 誤報成本：Layer 1/3 主觀判斷難偵測，Hook 可能觸發率低
- **機會成本**：擠壓 v0.18.0 其他 ticket 進度

**結論**：方案 F 總成本低於方案 B，且覆蓋率更高（涵蓋所有 4 層）。

### P（Prepare to be Wrong）失敗預想

12 小時後方案 F 最可能失敗的原因：

1. **PM 忘記月度執行自檢**：無 Hook 自動提醒，PM 過忙跳過 → 防護：每月 1 號自動建立「規則 5 月度自檢」ticket 模板（如 W17-rule5-check），逾期 7 天觸發 handoff 提醒
2. **用戶訪談流於敷衍**：季度訪談問題設計不當，用戶隨便答 → 防護：訪談問題聚焦**具體 session 案例**（「上季度 X session 你覺得 Claude 太強勢嗎？」），而非抽象評價
3. **對比案例樣本不足**：3 個月樣本過少 → 防護：對比案例法將樣本門檻設為 6 個月 + 10 案例，3 個月僅做初步趨勢判斷
