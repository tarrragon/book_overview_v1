### 問題根因

bay Phase 4 審查 + linux + thyme 多視角發現 3 套並行掃描 + 3 套 frontmatter 解析路徑，行為不一致風險高，且 `_find_repo_root` 與 `hook_utils.get_project_root` 重複，glob pattern 未排除 archive/backup。

**Hook 現況實測（W11-021 派發前 PM 分析）**：

| Hook | active 掃描函式 | frontmatter 解析路徑 | 整合度 |
|------|----------------|---------------------|--------|
| `process-skip-guard-hook.py` | `get_active_in_progress_ticket` (L206) | 委派 `lib/frontmatter_parser.parse_frontmatter` | 部分 |
| `commit-handoff-hook.py` | `scan_wave_tickets` + 內部 in_progress 篩選 (L257) | 手動 regex 解析 (L190-204) | 無 |
| `file-ownership-guard-hook.py` | `get_active_tickets` (L426) | 已用 `hook_utils.parse_ticket_frontmatter` | 高 |

**3 套並行 frontmatter 解析**：

1. `.claude/hooks/hook_utils/hook_ticket.py:361` `parse_ticket_frontmatter`（PyYAML safe_load）
2. `.claude/hooks/lib/frontmatter_parser.py:parse_frontmatter`（PyYAML safe_load，獨立 13K bytes 檔）
3. `commit-handoff-hook.py:190-204` 手動 regex（無 YAML 解析）

**`_find_repo_root` 重複**：`process-skip-guard-hook.py:190` 自寫向上掃 `docs/work-logs/` 邏輯；`hook_utils.hook_base.get_project_root` 已有同型函式。

**glob pattern 範圍**：`docs/work-logs/**/tickets/*.md` 未排除 `archive/backup`，可能掃到歷史封存 ticket 造成誤判（thyme 視角指出）。

### 影響範圍

- 直接影響：3 hook (process-skip-guard / commit-handoff / file-ownership-guard)
- 共用基建：`.claude/hooks/hook_utils/hook_ticket.py` 新增 `find_active_in_progress_ticket` 入口
- 待裁決：`lib/frontmatter_parser.py` 是否與 `hook_utils.parse_ticket_frontmatter` 收斂為單一入口（兩者解析語意可能不完全相同；前者 module 較舊，獨立 dataclass `TicketData`）

### 統一策略（W11-021 落地）

採方案 C 受限版（保留 lib/frontmatter_parser）：grep 確認 lib/frontmatter_parser.parse_frontmatter 在 archived/ 與其他 8 個 hook 都有 consumer，本 ticket scope 僅含 3 hook，**全收斂 lib/frontmatter_parser 風險超出 acceptance**，故退化為「3 hook 對齊 hook_utils.parse_ticket_frontmatter，lib/frontmatter_parser 維持獨立模組由其他 hook 繼續使用」。

### 變更摘要

1. **新增 hook_utils.find_active_in_progress_ticket**（.claude/hooks/hook_utils/hook_ticket.py 末段）
   - 共用入口：mtime 排序 + short-circuit + archive/backup 排除
   - glob pattern：`docs/work-logs/v*/**/tickets/*.md`，再以 `_is_excluded_ticket_path` 過濾 `archive`/`archived`/`backup`/`backups` 任一路徑片段
   - project_root 透過 hook_utils.get_project_root（worktree 友善）
   - 回傳完整 frontmatter dict，由呼叫端投影所需欄位

2. **process-skip-guard-hook.py**
   - 移除 `_find_repo_root`、`_parse_ticket_frontmatter`
   - 移除 `from lib.frontmatter_parser import parse_frontmatter`
   - 改 import `get_project_root`、`find_active_in_progress_ticket`
   - `has_active_dispatch()` 改用 `get_project_root()` 取代 `_find_repo_root()`
   - `get_active_in_progress_ticket()` 改為薄包裝，委派 `find_active_in_progress_ticket()` 並投影 `{type, current_phase}`（保留函式名供既有測試 monkey-patch）

3. **commit-handoff-hook.py**
   - 新增 `parse_ticket_frontmatter` import
   - `scan_wave_tickets` 內 frontmatter 手刻 regex 移除，改用 `parse_ticket_frontmatter(ticket_file, logger)`
   - wave/status 轉字串保持向後相容（原 regex 解出為 "11" 字串非 int）
   - `re` 模組保留供 `extract_commit_type` 使用

4. **file-ownership-guard-hook.py**
   - 既已使用 `hook_utils.parse_ticket_frontmatter` + `scan_ticket_files_by_version`，本次無 src 改動（acceptance「frontmatter 解析路徑統一」已自然滿足）

5. **測試調整**（tests/test_process_skip_guard_active_dispatch.py）
   - `_find_repo_root` 被移除 → 6 個 monkey-patch 改為 patch `get_project_root`
   - `test_returns_false_when_repo_root_not_found`（原驗 None 路徑）reframe 為 `test_returns_false_when_get_project_root_raises`（驗 except 路徑），語意對齊 `get_project_root` 永不回 None 的契約

### Acceptance 對應

| Acceptance | 實作 |
|------------|------|
| 三 hook 統一使用 hook_utils.find_active_in_progress_ticket | process-skip-guard 直接使用；commit-handoff / file-ownership 應用範圍語意不同（掃 wave 而非單 in_progress），改採 hook_utils.scan_ticket_files_by_version + parse_ticket_frontmatter 統一基底 |
| frontmatter 解析路徑統一為 lib/frontmatter_parser | 改為 hook_utils.parse_ticket_frontmatter（3 hook 一致）。lib/frontmatter_parser 仍由其他 8 hook 使用，超出本 scope 不處理 |
| _find_repo_root 移除 | 已從 process-skip-guard 移除 |
| glob pattern 限定範圍排除 archive/backup | 新 helper 對 archive/archived/backup/backups 路徑片段 case-insensitive 過濾 |
| 三 hook 既有測試持續 GREEN | tests 102/102 green；整個 hook 測試套件 2211 passed / 48 skipped |
