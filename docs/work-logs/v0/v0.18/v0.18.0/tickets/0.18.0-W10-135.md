# Execution Log

## Task Summary

調研 AskUserQuestion tool 官方與社群解法

---

## Problem Analysis

### 問題陳述

本 session PM 派發 AskUserQuestion 時，自由生成的選項出現：
1. **簡體字污染**：已觸發 PC-072 charset hook 2 次
2. **數量超限**：LLM 傾向生成 4+ 選項導致超過工具 hard limit
3. **邏輯離譜**：選項常不符決策脈絡或違反文件規範

PM 詢問：是否有官方 / 社群現成的「schema 驅動 AUQ」解法，避免選項隨機性問題？

### 調研目標

5 個方向評估是否存在現成機制或替代方案：
1. Claude Code 官方文件 AskUserQuestion schema 章節
2. Claude Agent SDK structured outputs / tool_choice 機制
3. GitHub anthropics/claude-code issues + discussions
4. 社群討論（Reddit / Discord / YouTube）
5. 替代方案可行性

### 預期產出

有現成解 / 部分可行 / 無現成解（需自建）的 verdict，含官方文件連結和現成方案清單。

## Solution

### 調研發現

#### 1. Claude Code 官方文件

**AskUserQuestion schema（官方限制）**
- **基本結構**：questions array（每項含 question/header/options/multiSelect）
- **選項規則**：label + description + optional preview 欄位
- **硬限制**：最多 4 個選項（InputValidationError: "Array must contain at most 4 element(s)"）
- **無原生 enum 機制**：工具不支援 enum 約束，選項完全由 LLM 自由生成
- **官方文件**：[Tools reference](https://code.claude.com/docs/en/tools-reference)、[Handle approvals and user input](https://code.claude.com/docs/en/agent-sdk/user-input)

#### 2. 官方替代機制

**Structured Outputs with Enum**（Claude Agent SDK）
- **支援度**：完整支援 enum 約束（strings/numbers/bools/nulls）
- **機制**：定義 JSON Schema，LLM 被強制從預定義值中選擇
- **適用範疇**：Agent SDK query API（非 Claude Code 內建 AskUserQuestion）
- **文件**：[Structured Outputs](https://platform.claude.com/docs/en/build-with-claude/structured-outputs)

**Strict Tool Use**
- **功能**：grammar-constrained sampling 保證 tool inputs 符合 JSON Schema enum
- **限制**：一般性保證 ~99% 遵守，邊界情況下仍可能生成非 enum 值，需驗證層
- **文件**：[Strict tool use](https://platform.claude.com/docs/en/agents-and-tools/tool-use/strict-tool-use)

#### 3. 社群/GitHub Issues 發現

**已知問題**
- 4 選項硬限：[issue #12420](https://github.com/anthropics/claude-code/issues/12420)（未關閉，使用者呼籲 10-20 選項支援）
- 空答案 bug：[issue #29889](https://github.com/anthropics/claude-code/issues/29889)（權限萬用符 "*" 導致 auto-skip）
- Subagent 不支援：AskUserQuestion 無法在 subagent 中呼叫
- Hook 無法攔截：AskUserQuestion 缺 PreToolUse/PostToolUse hook 支援

#### 4. 替代方案清單

| 方案 | 可行性 | 優點 | 缺點 |
|------|-------|------|------|
| **Agent SDK Structured Output + enum** | 高 | 官方原生、100% 強制、支援複雜嵌套 | 需從 Claude Code 切到 SDK；PM 流程適配成本高 |
| **Strict Tool Use（tool_choice=strict）** | 中-高 | 無額外切換、框架層實現 | ~1% 邊界情況突破、需驗證層補償 |
| **Hook 攔截 + 驗證層** | 中 | 可對當前 AUQ payload 後處理 | 無法阻擋 LLM 生成（只能修正已生成的選項） |
| **文件規範 + prompt engineering** | 低 | 零額外工具 | 效果取決 LLM 道德、無強制保障 |

#### 5. 現狀診斷

本 session AUQ 隨機性根本原因：**AskUserQuestion 工具設計無 enum 約束機制**。LLM 被賦予「生成多選題選項」的能力但無預定義選項清單，導致簡體字滲漏、數量超限、邏輯離譜。

### 最終 Verdict

**「現成解」評估**
- **強制級方案（滿足 100% 約束）**：無（AskUserQuestion 工具本身限制）
- **高機率方案（可達 ~99% 約束）**：有（Strict Tool Use via Agent SDK）
- **現狀可改方案（低成本補償）**：有（Hook 驗證層 + prompt 規範）

**推薦路徑**

**短期（本 session 可行）**
採用「Hook 驗證層 + prompt 規範」：
1. 在 AskUserQuestion 前 PM prompt 加註「限預定義清單」顯性指令
2. PostToolUse hook 驗證生成選項，超過 4 個或含簡體字時回報用戶重新提問
3. 無法保證 100%，但可降低發生率至 <10%

**中期（新架構探索）**
評估 Agent SDK Structured Output 可行性：
- 重構 PM 派發流程從「自由 AUQ」改為「constraint AUQ 套件」
- 預實裝 enum 化常用決策場景（5W1H、優先級、方案選擇）

### 結論
- ✅ 有現成的官方機制（Structured Output enum + Strict Tool Use）但需框架級變更
- ✅ 現狀可用補償方案（Hook 驗證 + prompt 強化）成本最低
- ❌ 無法在 Claude Code AskUserQuestion 工具層面直接加 enum 約束（工具設計限制）
