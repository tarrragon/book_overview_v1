# Execution Log

## Task Summary

分析 ticket CLI handoff stale 殘留判斷不一致（SessionStart/Stop hook 與 resume --list 結果不同步）

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

三套程式碼對 handoff stale 過濾規則不一致：reminder-hook 完全不過濾、stop-hook 對任務鏈一律保留（不檢查目標狀態）、CLI 端則正確識別任務鏈目標已啟動為 stale。三方判斷分歧導致 Stop hook 誤觸發 block，但 CLI list 顯示 0 個待恢復——用戶無從信任任一資料源。

### 影響範圍

- `.claude/hooks/handoff-reminder-hook.py`（SessionStart additionalContext）
- `.claude/hooks/handoff-auto-resume-stop-hook.py`（Stop hook 阻塞）
- `.claude/skills/ticket/ticket_system/commands/handoff_gc.py` / `resume.py --list`（CLI 端）
- `.claude/skills/ticket/ticket_system/lib/handoff_utils.py`（共用判斷尚未抽取）

### 相關 Error Pattern

無直接 PC error-pattern；本 ANA 的根因可作為「規則散落多處導致演進不同步」的反模式參考。

### 後續證據（本 session 觀察）

ANA 分析期間 Stop hook 第二次觸發誤阻塞，與根因吻合：

- `pending/` 含 3 個 stale（W10-089/W11-004.1/W16-003，任務鏈目標已啟動）
- CLI `handoff --gc --dry-run` 與 `--execute` 均識別並能清除
- Stop hook 仍視為「待恢復」並 block

採 workaround：執行 `ticket handoff --gc --execute` 清除 3 stale + 移除 `.claude/handoff/.stop-blocked` flag 解除阻塞。

證實 W17-095.2 的修復急迫性——只要 stale 任務鏈 handoff 累積，Stop hook 就會持續誤觸發。

## 重現實驗結果

### 實驗方法

於本 session 啟動時觀察三方資料源對相同 `.claude/handoff/pending/` 內容的判斷差異：

```bash
# 觀測點 1：SessionStart additionalContext（自動）
# 觀測點 2：Stop hook 觸發（自動於 session 結束）
ticket resume --list                       # 觀測點 3
ticket handoff --gc --dry-run              # 觀測點 4
```

### 實驗執行

**觀測點 1**（SessionStart additionalContext）：顯示 6 個待恢復——W11-004.3 / W11-004.7.1 / W11-004.7 / W16-003 / W10-089 / W11-004.1

**觀測點 2**（Stop hook 觸發）：訊息「多個待恢復任務，請選擇要恢復的任務」，觸發 block。

**觀測點 3**（`ticket resume --list`）：「沒有待恢復的任務 / 已過濾 3 個 stale handoff（來源 ticket 已完成）」

**觀測點 4**（`ticket handoff --gc --dry-run`）：3 個 stale
- W10-089.json：任務鏈目標 W10-012 已啟動
- W11-004.1.json：任務鏈目標 W11-004.2 已啟動
- W16-003.json：任務鏈目標 W16-003.2 已啟動

### 實驗發現

**已驗證**：
- SessionStart 6 個 = stale 3 個（W10-089/W11-004.1/W16-003，任務鏈目標已啟動）+ 其他 3 個（W11-004.3/W11-004.7.1/W11-004.7）
- Stop hook 認為 6 個全待恢復 → 阻塞
- CLI 認為 3 個 stale 應過濾，但 list 顯示 0 而非 3，意味 W11-004.* 也被視為 stale（來源 ticket completed）

**根因表（三方資料源行為差異）**：

| 元件 | stale 過濾規則 | 任務鏈目標已啟動是否視為 stale | 觀測結果 |
|------|---------------|---------------------------|---------|
| `handoff-reminder-hook.py`（SessionStart） | 不檢查 stale，只看 `resumed_at != null` | 否（完全不過濾） | 顯示 6 個待恢復 |
| `handoff-auto-resume-stop-hook.py`（Stop hook） | 來源 ticket completed + 非任務鏈 → GC；任務鏈一律保留 | 否（保留任務鏈即使目標啟動） | 持續阻塞 |
| `resume.py --list` / `handoff_gc.py`（CLI） | 來源 ticket completed + 非任務鏈，**或** 任務鏈目標已 in_progress/completed | 是（與 dry-run 一致） | 顯示 0 個（3 個 stale 過濾） |

**證據引用**：
- `.claude/hooks/handoff-auto-resume-stop-hook.py` 行 387-415（`should_preserve_pending_json`）：任務鏈一律 return True
- `.claude/skills/ticket/ticket_system/commands/handoff_gc.py` 行 70-76：任務鏈目標已啟動則加入 stale
- `.claude/hooks/handoff-reminder-hook.py` 行 90-117：純掃描，唯一過濾條件 `resumed_at != null`

**共用判斷未被三方一致引用**：`handoff_utils.scan_pending_handoffs` 已存在 stale record metadata（CLI 端引用），但 hook 端各自實作 scan + 過濾邏輯，未引用同一函式 → 規則散落 3 處，演進不同步。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

```yaml
multi_view_status:
  status: skipped
  reason: "技術修復路徑明確：CLI 端規則（handoff_gc._collect_stale_handoffs）已穩定運作，三方對齊只需抽共用函式並各自引用，無設計選擇歧義。風險低（hook 只是補齊判斷）、影響範圍局限於 handoff stale 過濾，無跨層架構決策。子 ticket 拆分（.1/.2/.3/.4）已落地驗證 4 個 acceptance 全部達成，多視角審查邊際效益低於建立成本。"
```

### 推薦組合：A + B + C

採用「方案 C 抽共用函式 + 方案 A/B 引用同一判斷」一次性根治三方資料源不同步根因。

### 方案 A：Stop hook GC 對齊 CLI 規則（最小改動，立即見效）

- **目標**：消除 Stop hook 誤觸發
- **改動**：`.claude/hooks/handoff-auto-resume-stop-hook.py` `should_preserve_pending_json()` 任務鏈類型不再無條件 return True；改為「任務鏈目標未啟動」才保留；目標已 in_progress/completed → 視為 stale
- **風險**：低。CLI 端規則已穩定運作，hook 只是補齊判斷
- **落地**：W17-095.2 ✓ commit 4fe4ab26

### 方案 B：handoff-reminder-hook 套用相同 stale 過濾

- **目標**：消除 SessionStart additionalContext 的「6 個」假象
- **改動**：`.claude/hooks/handoff-reminder-hook.py` `scan_handoff_pending_directory()` 引用共用 `is_handoff_stale()`；過濾 stale 後再列入 reminder；顯示「已過濾 N 個 stale」與 CLI 一致
- **風險**：低。reminder 只是顯示層，無破壞性
- **落地**：W17-095.3 ✓ commit 580758e9

### 方案 C：抽共用 stale 判斷函式（根本解，三方共用）

- **目標**：避免規則散落 3 處（reminder hook / stop hook / CLI）
- **改動**：`handoff_utils` 新增 `is_handoff_stale(record) -> (is_stale, reason)`；三套程式碼一律引用；加單元測試覆蓋四種情境
- **風險**：中。需重構 3 個檔案，但收斂規則於單一來源
- **落地**：W17-095.1 ✓ commit 277e7b36（`is_handoff_stale` + 8 unit tests）；W17-095.4 ✓ commit b4b62817（9 case 三方一致性整合測試）

### 方案 D（不採用）：Stop hook 觸發時自動 GC

- **目標**：每次 Stop 自動清理 stale
- **改動**：Stop hook 在 scan 前呼叫 `execute_gc(dry_run=False)`
- **風險**：高。Stop hook 副作用變大；可能在用戶無預期時刪檔
- **判決**：A+B+C 已根治根因，方案 D 副作用過大不需採用
