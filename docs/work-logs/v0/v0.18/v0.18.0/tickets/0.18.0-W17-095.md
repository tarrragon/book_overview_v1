## 後續證據（本 session 觀察）

ANA 分析期間 Stop hook 第二次觸發誤阻塞，與根因吻合：
- pending/ 含 3 個 stale（W10-089/W11-004.1/W16-003，任務鏈目標已啟動）
- CLI `handoff --gc --dry-run` 與 `--execute` 均識別並能清除
- Stop hook 仍視為「待恢復」並 block

採 workaround：執行 `ticket handoff --gc --execute` 清除 3 stale + 移除 `.claude/handoff/.stop-blocked` flag 解除阻塞。

證實 W17-095.2 的修復急迫性——只要 stale 任務鏈 handoff 累積，Stop hook 就會持續誤觸發。

## 根因分析

### 三方資料源行為差異（根因）

| 元件 | stale 過濾規則 | 任務鏈目標已啟動是否視為 stale | 觀測結果 |
|------|---------------|---------------------------|---------|
| `handoff-reminder-hook.py`（SessionStart additionalContext） | **不檢查 stale**，只看 `resumed_at != null` | 否（完全不過濾） | 顯示 6 個待恢復 |
| `handoff-auto-resume-stop-hook.py`（Stop hook 阻塞） | 來源 ticket completed + 非任務鏈 → GC；任務鏈一律保留 | **否**（保留任務鏈即使目標啟動） | 持續阻塞 |
| `resume.py --list` / `handoff_gc.py`（CLI） | 來源 ticket completed + 非任務鏈，**或** 任務鏈目標已 in_progress/completed | **是**（與 dry-run 一致） | 顯示 0 個（3 個 stale 過濾） |

### 殘留 3 筆 handoff 的命運

3 筆殘留：W10-089 / W11-004.1 / W16-003，均為 `to-sibling`/`to-child`，且任務鏈目標皆已啟動。

- CLI（resume --list / handoff gc dry-run）：判定為 stale，過濾顯示但**不自動移除**（需 `--execute`）
- Stop hook GC（`should_preserve_pending_json`）：保留所有任務鏈類型，**不檢查目標狀態** → 每次 Stop 都觸發誤阻塞
- SessionStart hook：完全不過濾 → 把 stale + recent 全列為「6 個待恢復」

### 證據引用

- `.claude/hooks/handoff-auto-resume-stop-hook.py` 行 387-415（`should_preserve_pending_json`）：任務鏈一律 return True
- `.claude/skills/ticket/ticket_system/commands/handoff_gc.py` 行 70-76：任務鏈目標已啟動則加入 stale
- `.claude/hooks/handoff-reminder-hook.py` 行 90-117：純掃描，唯一過濾條件 `resumed_at != null`

### 共用判斷未被三方一致引用

`handoff_utils.scan_pending_handoffs` 已存在 stale record metadata（CLI 端引用），但 hook 端各自實作 scan + 過濾邏輯，未引用 `handoff_utils` 的同一函式 → 規則散落 3 處，演進不同步。

## 修復方向（建議）

### 方案 A：Stop hook GC 對齊 CLI 規則（最小改動，立即見效）

**目標**：消除 Stop hook 誤觸發。

**改動**：`.claude/hooks/handoff-auto-resume-stop-hook.py` `should_preserve_pending_json()`
- 任務鏈類型不再無條件 return True
- 改為：「任務鏈目標未啟動」才保留；目標已 in_progress/completed → 視為 stale（與 `handoff_gc.py` 行 70-76 對齊）
- 引用 `handoff_utils.is_ticket_in_progress_or_completed` 與 `extract_direction_target_id`

**風險**：低。CLI 端規則已穩定運作，hook 只是補齊判斷。

### 方案 B：handoff-reminder-hook 套用相同 stale 過濾

**目標**：消除 SessionStart additionalContext 的「6 個」假象。

**改動**：`.claude/hooks/handoff-reminder-hook.py` `scan_handoff_pending_directory()`
- 引用 `handoff_utils.scan_pending_handoffs()` 與 `_collect_stale_handoffs` 同一判斷
- 過濾 stale 後再列入 reminder
- 顯示時也提示「已過濾 N 個 stale」與 CLI 一致

**風險**：低。reminder 只是顯示層，無破壞性。

### 方案 C：抽共用 stale 判斷函式（根本解，三方共用）

**目標**：避免規則散落 3 處（reminder hook / stop hook / CLI）。

**改動**：
- `handoff_utils` 新增（或暴露）`is_handoff_stale(record) -> (is_stale, reason)`
- 三套程式碼（reminder-hook / stop-hook / handoff_gc / resume）一律引用此函式
- 加單元測試覆蓋四種情境（任務鏈目標啟動、來源 completed 非任務鏈、from_status==completed、未完成）

**風險**：中。需重構 3 個檔案，但收斂規則於單一來源。

### 方案 D：Stop hook 觸發時自動 GC（副作用較大，不建議獨立採用）

**目標**：每次 Stop 自動清理 stale。

**改動**：Stop hook 在 scan 前呼叫 `execute_gc(dry_run=False)`。

**風險**：高。Stop hook 副作用變大；可能在用戶無預期時刪檔。建議僅作為方案 A/B/C 的補充。

### 推薦組合

**A + B + C**：先以方案 C 抽共用函式，再讓 A/B 引用同一判斷。一次性根治三方資料源不同步根因。

### 建議子任務拆分

| Ticket | Type | 範圍 |
|--------|------|------|
| W17-095.1 | IMP | 抽 `handoff_utils.is_handoff_stale()` 共用函式 + 單元測試 |
| W17-095.2 | IMP | Stop hook (`should_preserve_pending_json`) 引用共用函式 |
| W17-095.3 | IMP | reminder hook (`scan_handoff_pending_directory`) 引用共用函式 + 顯示「已過濾 N 個」 |
| W17-095.4 | IMP | 補測試：三套程式碼對 4 種情境（任務鏈目標啟動 / 非任務鏈 completed / from_status=completed / 未完成）行為一致 |
