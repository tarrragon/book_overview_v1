## Problem Analysis

### 問題根因

`update_source_spawned_tickets` 與 `update_parent_children` 採用 **read-modify-write** 三步驟，無任何 lock 保護：

```
1. load_ticket(version, ticket_id)  → 從磁碟讀整個 ticket
2. append child_id to list (記憶體操作)
3. save_ticket(ticket, path)         → 整檔覆寫
```

並發呼叫時暴露三種失敗模式：

| 失敗模式 | 機制 | 損害程度 |
|---------|------|---------|
| Lost update | A/B 都 load 到 spawned=[X]，各自 append 後 save，後寫者完全覆蓋前寫者 | 中（資料遺失） |
| Empty read | A 在 `open("w")` 截斷檔案瞬間，B 用 `open("r")` 讀到 0 byte 檔案 | 中（B 寫入時可能誤建空陣列） |
| File corruption | 兩個 process 的 partial write 在 OS-level page cache 交錯 | 高（YAML 解析失敗，ticket 完全壞掉） |

### 影響範圍

| 檔案 | 受影響函式 | 寫入路徑 |
|------|----------|---------|
| `.claude/skills/ticket/ticket_system/lib/parser.py` | `save_ticket()` (L280) | 所有 ticket 寫入入口（不限 spawned/children） |
| `.claude/skills/ticket/ticket_system/lib/ticket_builder.py` | `update_parent_children()` (L703) | parent.children append |
| `.claude/skills/ticket/ticket_system/lib/ticket_builder.py` | `update_source_spawned_tickets()` (L768) | source.spawned_tickets append |

所有透過 `ticket track create --parent <id>` 與 `ticket track create --source <id>` 的並行呼叫都受影響。同理，任何透過 `save_ticket` 的修改（status 更新、acceptance 勾選等）若與其他 process 並行也會 race。

### 相關 Error Pattern

- **PC-078**（parallel-session-ticket-misjudge）：歷史上已發生並行 session 同時操作 ticket 的混亂歸因
- **PC-100**（spawned-ticket-context-inheritance）：spawned 流程的多階段寫入若 race 會導致 Context Bundle 不完整
- **PC-073**（ANA spawned not children）：CLI 修補時引入 `update_source_spawned_tickets`，但未同步加 lock 保護
- **ARCH-020**：validator 邏輯漂移問題的並發版——同樣是「多處平行寫入未統一收斂」家族
## 重現實驗結果

### 實驗方法

直接模擬 `update_source_spawned_tickets` 內部 load → append → save 三步驟，無 lock：

- N=20 並行 process（macOS Python 3.14, fork 模式）
- 每個 process 對同一 source 檔案 append 一個獨特 child-XXX
- 預期結果：spawned_tickets 應有 20 個 unique entries
- 變因：注入 `sleep_ms` 於 load 與 save 之間，放大 race window
- 每組 sleep_ms 跑 10 trials

實驗在 `tempfile.TemporaryDirectory` 內進行，未污染 repo。腳本直接用 `yaml.safe_load/dump` 模擬 `parser.load_ticket/save_ticket` 行為（邏輯等價，bypass 內部 cache 簡化測試）。

### 實驗執行

執行指令（heredoc 直餵 python3，fork 模式繞 spawn 限制）：

```python
import multiprocessing as mp
mp.set_start_method("fork", force=True)
# ... race_worker: load → optional sleep → append → save
with mp.Pool(20) as pool:
    pool.map(race_worker, [(path, f"child-{i:03d}", sleep_ms) for i in range(20)])
```

收集指標：
- mean count（最終 spawned 長度，期望 20）
- lost / total（lost_update 數 / N=200）
- empty_reads（worker 讀到空檔案次數）
- worker_yaml_err（worker 讀檔時 YAML 解析失敗次數）
- final_corrupted（trial 結束後最終檔案無法 parse 的次數）

### 實驗發現

**量化結果（200 個並發 update / 組）**：

| sleep_ms | mean count | lost / total | lost rate | empty_reads | worker_yaml_err | final_corrupted |
|----------|------------|--------------|-----------|-------------|-----------------|-----------------|
| 0 | 6.7 / 20 | 133 / 200 | 66.5% | 58 | 10 | 0 / 10 |
| 1 | 3.3 / 20 | 167 / 200 | 83.5% | 46 | 6 | 0 / 10 |
| 5 | 2.3 / 20 | 177 / 200 | 88.5% | 42 | 2 | 1 / 10 |
| 10 | 1.6 / 20 | 184 / 200 | 92.0% | 22 | 0 | 1 / 10 |

**已驗證的事實**：

1. **Race 在最現實情境（sleep=0）即觸發 66.5% lost rate**——20 並行 process 中平均僅 6.7 個 update 倖存，相當於每 3 次並發呼叫就有 2 次資料遺失。
2. **Empty read 是真實風險**——sleep=0 時 200 次 worker 讀取中 58 次（29%）讀到空檔案。根因：`open(path, "w")` 在 Python 進入 `yaml.safe_dump` 前已 truncate 檔案，並發 reader 撞到 0 byte 視窗。
3. **檔案物理損毀可重現**——sleep=5/10 時各 1/10 次最終檔案無法 YAML 解析。實際樣本（`/tmp/w14005_corruption_samples/`）：

```yaml
spawned_tickets:
- child-019
:
- child-000
- child-004
- child-014
- child-018
status: in_progress
```

`:` 後接 sequence 而無 mapping key——典型 partial write 交錯：Process B 寫入完整新內容後，Process A 在自己的 file offset 繼續寫剩餘部分。

4. **隨 sleep_ms 增加，lost rate 上升但 worker_yaml_err 下降**——延遲加大讓 read 與 write 視窗較分散，worker 較少撞到截斷瞬間；但 save 之間的時間差變大，後寫者覆蓋更徹底。

**仍未驗證的假設**（不影響方案選擇）：

- 真實 ticket CLI 並發頻率：本實驗無法量化「實際使用中多久撞一次 race」。理論上單一開發者並發機率較低，但多 worktree / 多 session / hook 自動派發場景會放大。
- macOS APFS vs Linux ext4 行為差異：本實驗在 macOS APFS 上，partial write 交錯模式可能異於 Linux。但 lost update 是 logical race，與檔案系統無關，必然存在。

**結論**：

| 結論 | 證據強度 |
|------|---------|
| Race condition 100% 確認存在 | 直接證據（200/200 trials 均觀察到 lost update） |
| 三種失敗模式同時可能發生 | 直接證據（lost + empty_read + corruption 全部重現） |
| File lock（fcntl.flock 或 atomic rename）必要 | 推論強：三模式皆由「load → save 間另一 process 介入」造成，lock 一次解決 |
| 不需 sqlite 重型方案 | 推論強：問題是檔案層級 read-modify-write，sqlite 引入 schema 遷移與 lock contention 過度設計 |

### WRAP 決策紀錄

**W — Widen Options**：三方案——(A) fcntl.flock advisory lock、(B) atomic rename + fsync、(C) filelock 第三方函式庫。排除 sqlite（實驗已證明過度設計）。

**R — Reality-Test**：針對六維度實測評估（見下方評估表）。fcntl 不支援 Windows；atomic rename 需大幅改寫 save_ticket 流程；filelock 跨平台但引入外部依賴。

**A — Attain Distance**：Consider the Opposite——「不加 lock 行不行？」不行，實驗 66.5% lost rate 已證明。「只保護 update_parent/spawned 夠嗎？」不夠，save_ticket 是唯一寫入入口，所有 caller 都有 race 風險。

**P — Prepare to Be Wrong**：選 fcntl.flock 但保留 filelock fallback 介面，若未來需 Windows 支援可替換底層實作。

---

### 1. 三方案評估表

| 維度 | (A) fcntl.flock | (B) atomic rename + fsync | (C) filelock library |
|------|----------------|--------------------------|---------------------|
| **crash safety** | 進程死亡自動釋放（OS 回收 fd） | 最強：rename 是 POSIX atomic op，不存在 partial write | 同 (A)（底層用 flock/LockFile） |
| **cross-platform** | macOS/Linux 原生；**Windows 不支援** | macOS/Linux 原生；Windows 需 `os.replace` 但語意相同 | macOS/Linux/Windows 全支援 |
| **實作成本** | 低：save_ticket 前後加 `with flock(path)` | 中高：需改寫為 write-to-temp → fsync → rename 流程 | 低：`pip install filelock` + `with FileLock(path + ".lock")` |
| **效能** | 極低開銷（kernel syscall） | 額外 1 次 fsync + 1 次 rename（~1ms） | 同 (A) + 極小 Python 包裝開銷 |
| **與 _ticket_cache 互動** | 無影響：lock 在 save_ticket 內部，cache invalidation 在 lock 釋放前完成 | 需調整：rename 後 cache key 的 path 不變，但需確保 rename 後再 invalidate | 同 (A) |
| **與 hook 系統相容性** | hook 透過 CLI subprocess 呼叫 ticket CLI，各 subprocess 獨立取 lock，正確 | 同左 | 同左（lockfile 可跨 process） |

**結論**：選 **(A) fcntl.flock** 作為主方案。理由：

1. 本專案目標平台為 macOS/Linux（Chrome Extension 開發環境），Windows 非當前需求
2. 實作成本最低——僅需在 `save_ticket()` 加一層 context manager
3. crash safety 由 OS 保證，無需手動 cleanup
4. 保留升級路徑：未來若需 Windows 支援，可替換為 filelock library（API 相容）

### 2. Lock Granularity 選擇

| 粒度 | 描述 | contention 風險 | deadlock 風險 | 適用性 |
|------|------|----------------|--------------|--------|
| **per-ticket-file lock** | 對每個 ticket .md 檔建立 `{path}.lock` | 極低：不同 ticket 零等待 | 極低：單一 lock 無順序問題 | 最佳 |
| directory-level lock | 整個 tickets/ 目錄一把 lock | 高：任何 ticket 寫入都阻塞其他 | 低 | 過度限制 |
| global lockfile | 全 CLI 共用一把 lock | 極高：所有操作序列化 | 低 | 不可接受 |

**選擇 per-ticket-file lock**。

**Deadlock 防護**：`update_parent_children` 和 `update_source_spawned_tickets` 內部會先 save 自身 ticket 再 save parent/source ticket，涉及兩把 lock。為避免 A→B / B→A deadlock：

- **方案**：在 `save_ticket` 層級加 lock（不在 caller 層級），每次 `save_ticket` 呼叫獨立取鎖、寫入、釋鎖。兩次 `save_ticket` 呼叫之間不持有任何 lock。
- **代價**：兩次 save 之間存在極短的不一致窗口（parent 已更新但 child 還沒），但這是已知且可接受的——最壞情況是重跑命令即可補齊。
- **Why this works**：deadlock 需要「持有 A 的 lock 同時等待 B 的 lock」，本方案中每個 lock 的持有時間 = 單次 `save_ticket`（~5ms），釋放後才取下一把。

### 3. 遷移路徑設計

#### 3.1 Inject 點

選擇 **`parser.save_ticket()` 層級**（非 caller 層級）。理由：

- `save_ticket` 是所有寫入的唯一入口（grep 確認 30+ 呼叫點，跨 8 個模組）
- 在此層級加 lock，所有 caller 自動受保護，零修改成本
- 若在 caller 層級（如 `update_parent_children`）加 lock，會遺漏其他 caller

#### 3.3 Backward Compatibility

**一次性切換**（非漸進式）。理由：

- 變更範圍 = 僅 `parser.py` 的 `save_ticket` 函式（~10 行新增）

#### 3.4 實作骨架（概念碼，非最終實作）

```python
import fcntl
from contextlib import contextmanager

@contextmanager
def _file_lock(ticket_path: Path):
    """Per-ticket-file advisory lock via fcntl.flock."""
    lock_path = ticket_path.with_suffix(ticket_path.suffix + ".lock")
    lock_path.parent.mkdir(parents=True, exist_ok=True)
    fd = open(lock_path, "w")
    try:
        fcntl.flock(fd, fcntl.LOCK_EX)
        yield
    finally:
        fcntl.flock(fd, fcntl.LOCK_UN)
        fd.close()

def save_ticket(ticket, ticket_path):
    with _file_lock(ticket_path):
        # ... 既有邏輯不變 ...
```
