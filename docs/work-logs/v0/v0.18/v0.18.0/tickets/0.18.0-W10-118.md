### 問題根因

4 個 hook（phase4-decision-enforcement / ticket-quality-gate / acceptance-gate self_check_visibility_checker / ticket_validator._is_placeholder）共通設計缺陷：**字面 regex 缺語意分流，豁免機制不完備**。各 hook 獨立設計時未考慮 ticket type / 上下文 / 章節 / 表格等情境，導致：

- A：規則引用文字（「禁止 Phase 5 再決定」）誤判為延後話術
- B：對 ANA 類型仍套用 Flutter Bloc 風格章節要求（Layer 1-5 / test/*.dart）
- C：自檢結果 H3 加任何補充說明（如版本標註）即抓不到，warning 永久亮燈
- D：表格 cell 中的 `N/A`（不適用標示）被誤判整個 Solution 為佔位符

4 者皆在 `ticket track complete` 流程觸發，**互相疊加迫使 PM 同時規避 4 個字面遊戲**，產生 acceptance 用詞改寫 / Solution 表格用詞限縮 / 章節命名拘謹等 workaround 設計，與「品質防護」設計初衷相悖（quality-baseline 規則 5 失敗案例學習原則明示「規則應提煉教訓而非製造儀式」）。

### 案例詳述

#### 案例 A：PC-093 hook 字面抓觸發詞，無「規則引用」豁免

**Hook**：`.claude/hooks/phase4-decision-enforcement-hook.py`
**觸發點**：偵測「Phase 4 評估」「Phase 5 再決定」等延後話術
**問題**：
- acceptance 文字「禁止 Phase 5 再決定」是引用 decision-trigger-binding 規則 1.5 名稱
- hook 字面匹配抓到 5 個位置（line 50/169/193/212），全是規則引用而非實際延後話術
- 合法 exempt category（tdd-transition / baseline-gated / ticket-tracked / user-override）無「rule-quote」類別

**修復方向候選**：
1. 新增 exempt category: `rule-reference`（引用規則名稱，附 reason 引用規則路徑）
2. 改進 hook 語意判斷（前後文 negation 詞如「禁止」「不應」「反模式」附近的觸發詞應豁免）
3. 規則名稱字面改寫（避免 hook 抓不到的同義詞替代）

#### 案例 B：ticket-quality-gate-hook 對 ANA 類型不適用

**Hook**：`.claude/hooks/ticket-quality-gate-hook.py`
**觸發點**：PostToolUse:Write 偵測 ticket md，要求 c2 (acceptance_criteria 章節 + test_plan + references) 和 c3 (Layer 標示)
**問題**：
- 要求「### 驗收條件」章節（與 frontmatter acceptance 重複）
- 要求「測試檔案路徑（test/.../*.dart）」（Flutter 專用，本專案為 Chrome Extension JS；對 ANA/DOC 類型無關）
- 要求「[Layer X]」標示（流程設計類 ANA 無明確 Layer）
- hook 是 PostToolUse 但 blocking error，會阻擋後續 hook 與工具

**修復方向候選**：
1. hook 識別 type=ANA/DOC 時放寬 c2/c3 規則
2. 移除 Flutter 專用要求（test/*.dart），改為通用「測試規劃」段落或專案語言感知
3. Layer 標示改為可選（Framework-Meta / Cross-Cutting 等抽象 Layer 接受）

#### 案例 C：自檢結果 H3 含括號補充 regex 抓不到

**Hook**：`.claude/hooks/acceptance-gate-hook.py`（Layer 1 自檢觀測 step）
**觸發點**：偵測 Solution 內是否有 `### 自檢結果` 章節
**問題**：
- 我寫的 `### 自檢結果（Layer 1 — W17-064）` hook 抓不到
- 改成精確「### 自檢結果」才被識別
- regex 設計過嚴（要求標題完全等於「### 自檢結果」，不容子標題說明）

**修復方向候選**：
1. regex 改為前綴匹配 `^### 自檢結果`（允許後綴補充說明）
2. 文件規範明示「自檢結果章節標題必須完全等於 ### 自檢結果」

#### 案例 D：表格 cell 中 N/A 被 _is_placeholder 誤判（PC-138 重現）

**位置**：`.claude/skills/ticket/ticket_system/lib/ticket_validator.py` `_is_placeholder` 函式
**觸發點**：W17-094 已加 \b 字邊界，但 N/A 在表格 cell 仍中 `\bN/A\b`
**問題**：
- Solution 中 M4 行「不可行 | N/A | — | —」N/A 是表格欄位值（不適用標示）
- _is_placeholder 抓到 N/A 後判定整個 Solution 章節為「未填寫」
- 阻擋 complete 直到改 N/A 為「不適用」

**修復方向候選**（PC-138 建議已存）：
1. _is_placeholder 排除表格情境（regex 偵測 `| N/A |` 包夾豎線時豁免）
2. 預處理時剝除 markdown 表格 cell 內容後再判斷 placeholder

1. **字面 regex 缺語意分流**：不依 type / 上下文 / 章節區分（Flutter vs JS、規則引用 vs 實際話術、表格 vs 散文、子標題補充 vs 純標題）

### 修復策略決策

評估 3 方案（α 獨立 / β 大重構 / γ 拆分平衡）：

| 方案 | 描述 | 優劣 |
|------|------|------|
| α | 4 個 hook 各自獨立修復 | 範圍小、可平行；缺整體 UX 視角 |
| β | 整合為「ticket complete UX 重構」單一大 ticket | 整體一致；認知負擔高、易超 16 小時上限 |
| γ | 拆 4 個 IMP（每 hook 1 個）+ 1 個 DOC（更新 ticket-body-schema 與 acceptance 文字規範） | 平衡平行性與整合性，符合 cognitive-load.md 3b 派發前閾值 |

**採用方案 γ**（已決策狀態）。理由：

1. 4 個 hook 的修復本質互相獨立（regex 改進 / type-aware 觸發 / 前綴匹配 / 表格情境豁免），合併為單一 ticket 違反 cognitive-load.md「功能職責數 > 2 須拆分」閾值
2. 修復後需更新規則文件（ticket-body-schema 對 ANA 章節要求、acceptance 文字規範禁忌詞）—— DOC ticket 統一收斂
3. 4 個 IMP 可依優先級排序（A 高—rule-quote 影響規則引用普遍性 / B 中—Flutter 假設明顯不適用本專案 / C 中—warning 不阻擋但永久亮燈影響觀測 / D 中—表格用詞限縮）
