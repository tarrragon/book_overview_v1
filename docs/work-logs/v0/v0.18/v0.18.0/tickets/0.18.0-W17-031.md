### Simplified WRAP（claim checkpoint）

**W (Widen) — 候選方案**:
- A. 4 children 並行（.1/.2/.4/.5）：最快但 4 thyme 同時 commit main 違反 PC-092（commit race）
- B. 分波並行（第一波 .1/.2/.5 並行 → .3 等 .1）：仍有 3 thyme 同時 commit main 風險
- C. 純序列（.5 → .2 → .1 → .3，.4 PM 評估）：一次一個 thyme，主 repo cwd 安全
- D. 先派 .5 一個試水（推薦）：確認流程順暢後依結果決定後續

**A (Attain distance) — 機會成本**:
- 4 children 並行雖快但 PM 注意力分散；其他 90+ pending tickets 同等 P1 待處理
- 本 group 中優先（IMP/P1），序列保品質可接受
- 真實受益：序列派發 thyme 不爭資源，PM 可同時關注其他子題分析

**P (Prepare to be wrong) — 最可能失敗**:
- 風險 1（最高）：.1/.3/.4 都動 track_runqueue.py，並行派發必撞檔案（PC-092）
- 風險 2：thyme commit 時 stage 無關檔案（PC-092 副效應），prompt 必須明示精確 git add path
- 風險 3：派發位置誤判 — children 全動 .claude/，必須主 repo cwd（不在 worktree，ARCH-015 hardcoded 保護）
- 風險 4：commit 後忘記 ticket 父子 acceptance 同步（PM 須在每 child complete 後勾本 group AC）

**結論**：採方案 C（純序列），第一個派發 W17-031.5（盲區 E，動 hook.py 最獨立）。
