#### 1.3 find_violations 新分支插入位置

**現行結構**（hook 第 82-95 行）：

```
for each char in text:
  code = ord(char)
  [分支 A] if char in SIMPLIFIED_CHARS:
      append (path, char, code, "簡體字")
      continue
  [分支 B] for (start, end) in EMOJI_RANGES:
      if start <= code <= end:
          append (path, char, code, "emoji")
          break
```

**擴充後結構**（插入新分支 C）：

```
for each char in text:
  code = ord(char)
  [分支 A] if char in SIMPLIFIED_CHARS:        # 不動
      append (path, char, code, "簡體字")
      continue
  [分支 B] for (start, end) in EMOJI_RANGES:    # 不動
      if start <= code <= end:
          append (path, char, code, "emoji")
          break
      after the for/else fallthrough:            # python for-else 或 flag 變數
  [分支 C] if char in JAPANESE_ONLY:             # 新增
      append (path, char, code, "日文漢字")
      continue
```

**分支順序理由**：

| 順序決策 | 理由 |
|---------|------|
| A（簡體）在前 | 既有行為保持不變；Phase 1 Spec §API「SIMPLIFIED_CHARS 判定順序優先」明文要求；E3 案例驗證 |
| B（emoji）維持中間 | 既有行為保持不變；emoji 在 CJK block 外，與 JAPANESE_ONLY 無重疊衝突 |
| C（日漢）在末端 | 新增功能應排最後；若未來清單有爭議字可更早 fail-fast 而不影響既有邏輯 |

**虛擬碼（pseudocode，非可執行）**：

```
function find_violations(text, field_path):
  violations = []
  for each char in text:
    code = codepoint_of(char)

    # 分支 A：簡體（既有）
    if char in SIMPLIFIED_CHARS:
      violations.append(tuple(field_path, char, code, "簡體字"))
      continue

    # 分支 B：emoji（既有）
    matched_emoji = false
    for each (start, end) in EMOJI_RANGES:
      if start <= code <= end:
        violations.append(tuple(field_path, char, code, "emoji"))
        matched_emoji = true
        break
    if matched_emoji:
      continue

    # 分支 C：日文漢字（新增）
    if char in JAPANESE_ONLY:
      violations.append(tuple(field_path, char, code, "日文漢字"))
      continue

  return violations
```

> 實作注意：既有 code 使用 `break` 跳出 emoji 迴圈但未 `continue` 外層；新增分支 C 須確保 emoji 命中後不會連鎖 fallthrough 進入分支 C。建議用 for-else 或 flag 變數，Phase 3b 實作者依語言慣用法擇一。

### 2. 資料結構選擇

| 結構 | 選擇 | 理由 |
|------|------|------|
| JAPANESE_ONLY | `frozenset[str]` 17 字 | O(1) lookup；與既有 SIMPLIFIED_CHARS 對稱；不可變避免誤改 |

### 5. 行為鏈式推演（Phase 2 v1.5.0 新增要求）

#### 5.1 核心行為鏈

```
PM 生成 AUQ payload
  ↓ [前置] JAPANESE_ONLY frozenset 已載入
  ↓ [前置] find_violations 可呼叫
PM 呼叫 AskUserQuestion tool
  ↓ [中間] PreToolUse hook 攔截
hook 解析 stdin JSON → scan_payload → find_violations
  ↓ [分支點] 日漢字 / 簡體 / emoji / 共用字 / 純繁體
依類別決定 exit code
  ↓
[正常結果] exit 0 or exit 2 with BLOCK_MESSAGE_TEMPLATE
```

#### 5.2 四維度行為分支覆蓋

| 維度 | 覆蓋案例 | 範例 |
|------|---------|------|
| 正常流程 | A 類 + E 類 | A2（純日漢命中）、E1（混合命中） |
| 異常流程（畸形輸入） | F5, F6 + 既有 `test_malformed_items_skipped` | 空字串、純 ASCII、非 dict |
| 邊界條件 | B 類 + C 類 + D3 | 共用字邊界、繁中異體邊界 |
| 中斷操作 | 不適用（純函式無可中斷動作） | — |

#### 5.3 前置驗證斷言（每個 When 前的 Given 都有對應斷言）

- 類別 A：`Given JAPANESE_ONLY 已載入` → A1 直接以成員性斷言驗證前置
- 類別 B/C：`Given 排除字不在清單` → B5、C1 明確成員性驗證
- 類別 D：`Given 既有 SIMPLIFIED_CHARS / EMOJI_RANGES 不變` → D2、D4 直接驗證既有邏輯路徑
- 類別 E：`Given 多類別共存` → E3 以 positive control 驗證類別分支存在
- 類別 F：`Given field_path 契約未變` → F1-F3 直接驗證欄位傳遞

#### 5.4 停止判斷

- 四個維度的場景皆有至少一個案例（中斷除外，純函式無此維度）
- 無法再提出新的「如果...那麼...」組合（已涵蓋 17 可入 + 5 禁入 + 既有回歸 + 欄位定位 + 訊息模板）
- 所有 Then 皆為可直接觀察的函式返回值或 frozenset 成員性
