## 實作摘要（W14-034）

### Helper API（hook_utils）

新增 `hook_utils.get_effort_level(payload, default="medium") -> str`：

- 優先序：`payload['effort']['level']` > `$CLAUDE_EFFORT` > `default`
- 正規化為 `low` / `medium` / `high`，未知值降為 `default`
- `default` 本身若非合法值則回退 `medium`
- 位置：`.claude/hooks/hook_utils/hook_io.py`，並從 `hook_utils/__init__.py` re-export
- 設計目標：向後相容，W14-036 / W14-037 / W14-038 可直接呼叫

### 4 個 Hook 改動

| Hook | 行為 | low effort |
|------|------|------------|
| `acceptance-gate-hook.py` | strict gate（complete 命令前） | 短路 emit allow JSON |
| `ticket-quality-gate-hook.py` | strict gate（ticket Write/Edit） | 短路 emit allow JSON |
| `ticket-frontmatter-validator-hook.py` | strict validator | 短路 `emit_allow_output(None)` |
| `phase4-decision-enforcement-hook.py` | PC-093 偵測 | **blocked 永遠阻擋**；low 僅抑制 warn/info audit |

### PC-093 防護驗證

phase4 hook 的修改採「上半段不變、下半段條件化」設計：

1. `blocked` 判斷在 effort 檢查之前，確保任何 effort level 命中 PC-093 模式皆 `return 2`（exit code 2 = 阻擋）
2. 僅 `warn/info/markers` audit 輸出在 `effort == "low"` 時抑制（純 token 節省，不改變強制決策）
3. 額外 `logger.info` 記錄「effort=X 仍強制阻擋（PC-093 規則 2）」留痕

### 各 hook logger.info 留痕

| 條件 | 訊息範本 |
|------|---------|
| low 短路 | `effort=low，<hook> 短路放行` |
| medium/high 完整路徑 | `effort=<level>，執行完整 <檢查名> 驗證` |
| phase4 blocked | `blocked=N，effort=<level> 仍強制阻擋（PC-093 規則 2）` |
| phase4 low 抑制 audit | `effort=low，抑制 warn/info audit 輸出（warned=N info=N markers=N）` |

### 不在本 ticket 範圍

- W14-036 / W14-037 / W14-038 將於後續 ticket 套用 `get_effort_level` 至其餘候選 hook
