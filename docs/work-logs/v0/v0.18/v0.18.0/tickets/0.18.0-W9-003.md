### 根因分類（4 類，滿足驗收條件 2）

**類 A：認知摩擦差（Friction Imbalance）**

| 動作 | 步驟數 | 心智負擔 |
|------|-------|---------|
| 寫 memory | 1（Write 單一檔案 + 更新 MEMORY.md 索引） | 低 |
| 升級為框架規則 | 5+（判斷跨專案性 → 找 rules/methodologies 位置 → 寫內容 → 更新索引 → 在 memory 標記「已升級」） | 高 |

PM 在 session 高壓下會選擇「先寫 memory 保留資訊」，但「之後再升級」的第二步永遠沒到。

**類 B：邊界判斷缺失（Scope Misjudgment）**

- Memory 和 Ticket **都是專案層級儲存**（`~/.claude/projects/<project>/memory/`），但 PM 的心智模型誤將 memory 視為「跨 session 持久層」而非「專案層級儲存」
- 寫 memory 當下未評估「此原則是否跨專案適用」
- 預設行為是「先寫 memory」→ 等用戶指正才升級（本 session 之前 PM 不清楚 memory 是專案層級）

**類 C：工具提示偏向（Tool Guidance Bias）**

| 提示來源 | 偏向 |
|---------|------|
| CLAUDE.md auto-memory 章節 | 詳述「如何寫」，未述「何時升級」 |
| continuous-learning skill | 聚焦「捕獲」，未強制「升級路徑」 |
| Memory tool description | 鼓勵多寫，未提示「跨專案原則應寫到框架」 |

工具設計潛在假設是「寫 memory 是終點」，而非「寫 memory 是評估起點」。

**類 D：用戶介入作為唯一校正機制（Reliance on User Correction）**

已升級的 6 個案例（#1, #7, #8, #9, #10, #11）中，至少 4 個是「用戶指正後 PM 才升級」：
- #1（框架不引用 ticket）：用戶指正 memory 也是專案層級
- #7（窮盡五問）：用戶先指出「先窮盡間接方式」PM 才主動探索
- #11（版本狀態同步）：用戶發現 `ticket --wave` 查詢錯誤才觸發

PM 自身**無主動 memory audit 流程**，依賴用戶巡檢，失敗率高（5/13 ≈ 38% 未升級）。

### S1：新增 quality-baseline 規則 7

**位置**：`.claude/rules/core/quality-baseline.md`（自動載入，PM 強制閱讀）

**新增內容**：

```markdown
### 規則 7：Memory 寫入必須評估跨專案升級

**寫入 feedback 類 memory 時，必須同時評估是否升級為框架規則**

> **來源**：0.18.0-W9-003 — PM 有 5/13（38%）的 feedback memory 僅存 memory 未升級，包含跨專案適用的原則（如「框架/產物分離」「Ticket 引導優先於 Hook」）。Memory 本身是專案層級儲存，不會 sync 到其他專案，跨專案原則若僅存 memory 會在其他專案消失。

**強制檢查問題**：寫入 feedback memory 時，必須回答：

| 檢查問題 | 是 | 否 |
|---------|----|----|
| 此原則對其他專案也適用嗎？ | **同時升級框架** | 標記為 `project_` 前綴 |
| 此原則是通用品質/流程原則嗎？ | 升級到 `rules/core/` | 保留 memory |
| 此原則是 PM 行為規範嗎？ | 升級到 `rules/core/pm-role.md` 或 `pm-rules/` | 保留 memory |
| 此原則是錯誤學習嗎？ | 升級到 `error-patterns/` | 保留 memory |

**升級後處理**：
- 原 memory 檔案開頭加註「本原則已升級為框架規則」
- 列出升級目的地路徑
- 保留 memory 作為本專案的 context 提醒索引

**禁止行為**：
- 以「之後再升級」為由僅寫 memory
- 寫入 feedback memory 時未評估跨專案性
```

2. **寫 memory ≠ 完成知識管理**：寫入 memory 只是**起點**，還需評估跨專案性並決定升級位置。工具目前的提示偏向「如何寫」而非「寫完之後」。

3. **原則層與執行層的落差是共通結構**（與 W9-002 呼應）：PM 識別原則正確，但落地到所有相關檢查清單/框架層的步驟被略過。需要「自動化同步機制」（Hook 提示或 skill 強制步驟）。

4. **認知摩擦差決定行為選擇**：寫 memory 1 步 vs 升級框架 5+ 步，高壓下 PM 永遠選低摩擦路徑。降低升級摩擦（提供決策樹、自動化提示）比單純訂規則更有效。

5. **規則 6 落地驗證**：此案屬「框架修改優先於專案進度」，一次改善（規則 7 + skill 升級 + hook + PC-061）避免所有後續專案重犯「memory 停滯」問題。
