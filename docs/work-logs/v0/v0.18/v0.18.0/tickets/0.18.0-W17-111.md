## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

兩層共振造成 PM 寫 ANA Solution multi_view_status 時容易誤用 nested YAML 結構，hook regex 跨行抓到下一行字串並判定值非法：

1. **Hook regex 跨行匹配**：`.claude/hooks/acceptance_checkers/multi_view_checker.py:89` `_parse_field` 用 `^\s*multi_view_status\s*:\s*(.+?)\s*$`。Python re 的 `\s` 含 `\n`，第二個 `\s*` 跨行吞掉換行+縮排，`(.+?)` 從 nested 第二行 `  status: skipped` 起匹配，`value = "status: skipped"` → 不在 allowed_values（reviewed/skipped/n_a） → 警告觸發。

2. **Schema 文件未明示禁 nested**：`.claude/config/ana-solution-schema.yaml` examples 全部展示 flat 格式（行 33-45），但未明示「禁用 nested 結構」。schema 描述「reviewed 需附 reviewers / conclusion 子欄位」容易讓 PM 聯想為 nested object。

### 影響範圍

- `.claude/hooks/acceptance_checkers/multi_view_checker.py`（hook 解析邏輯與訊息）
- `.claude/config/ana-solution-schema.yaml`（schema 範例文件）
- 未來所有 ANA ticket 收尾的 PM（PC-117 過渡期靠用戶自律）

### 相關 Error Pattern

- PC-117（本 session 落地）：用戶側防護 — PM 寫 multi_view_status 必用 flat YAML
- PC-110（同類）：「validator 與用戶寫法分歧」反模式

## 重現實驗結果

### 實驗方法

於 W17-095 收尾過程，PM 寫 nested YAML 後執行 `ticket track complete 0.18.0-W17-095`，觀察 acceptance-gate-hook 訊息：

```yaml
# Solution 區段內容（觸發誤判）
multi_view_status:
  status: skipped
  reason: "..."
```

### 實驗執行

執行 `ticket track complete 0.18.0-W17-095` 觸發 PreToolUse hook：

```
[WARNING] Acceptance Gate: ANA Ticket multi_view_status 值非法

Ticket: 0.18.0-W17-095
目前值: status: skipped
合法值: reviewed, skipped, n_a
```

事後改 flat YAML：

```yaml
multi_view_status: skipped
reason: "..."
```

無警告通過。

### 實驗發現

**已驗證的事實**：

| 事實 | 證據 |
|------|------|
| Python re `\s` 含 `\n` | Python 標準行為，`re.MULTILINE` 不影響 `\s` |
| `\s*` 跨行吞 newline + 縮排 | regex `^\s*multi_view_status\s*:\s*(.+?)\s*$` 在 nested YAML 下抓到第二行內容 |
| Schema examples 為 flat | `.claude/config/ana-solution-schema.yaml` 行 33-45 全部 flat |
| Schema 未明示禁 nested | 同檔案內無 forbid 註解 |
| Hook 訊息對用戶不夠清楚 | 顯示「目前值: status: skipped」未提示「nested 結構誤用」 |

**仍未驗證的假設**：

| 假設 | 評估方法 |
|------|---------|
| 改 schema 範例加註解可降低誤用率 | 需追蹤未來 ANA ticket 收尾誤用率（W17-082 類觀察期）|
| 改 hook 邏輯支援 nested 自動扁平化是否引入新邊界 | 需 case 矩陣覆蓋 |

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

```yaml
multi_view_status: skipped
reason: 技術修復路徑明確（hook regex 行為 + schema 文件缺漏均為事實），改善方向是 cost-benefit trade-off 而非設計選擇分歧；W17-095 ANA 收尾踩坑案例已直接驗證根因；無跨層架構決策；多視角審查邊際效益低於建立成本。
```

### 推薦組合：方案 B + A（低成本雙通道）

採用「方案 B 改 schema 範例加禁 nested 註解 + 方案 A 改 hook 訊息明示 nested 結構誤用」雙通道防護。方案 C（hook 邏輯支援 nested）暫不採用——成本中等且可能引入新邊界，B+A 已能堵住 95%+ 誤用案例。

### 方案 A：改 hook 訊息明示 nested 結構誤用

- **目標**：當 hook 偵測到 value 含 `:` 字元時，提示「可能寫成 nested 結構」並給 flat 範例
- **改動**：`.claude/hooks/acceptance_checkers/multi_view_checker.py` `_format_invalid_value_warning` 加分支：若 `value.contains(":")` 則訊息加附「偵測到值含冒號，可能誤用 nested YAML 結構。schema 要求 flat 格式：`multi_view_status: skipped`」
- **風險**：低。只改訊息文字，不影響判斷邏輯
- **預估成本**：30 分鐘 + 補測試

### 方案 B：改 schema 範例加禁 nested 註解（最低成本）

- **目標**：在 schema examples 處明示「禁用 nested 結構」
- **改動**：`.claude/config/ana-solution-schema.yaml` 行 33-45 加 forbidden_format 段落，明示 nested YAML 會被 hook 跨行誤判
- **風險**：極低。純文件改動
- **預估成本**：10 分鐘

### 方案 C（暫不採用）：改 hook 邏輯支援 nested 自動扁平化

- **目標**：hook 自動辨識 nested 結構並扁平化解析
- **改動**：`_parse_field` 偵測下一行是縮排則嘗試 nested parse；改用 yaml.safe_load(section) 解析整段
- **風險**：中。yaml.safe_load 對 markdown 雜訊敏感；需 case 矩陣覆蓋（含 nested + flat + 空值 + 註解 + 多重 yaml block）
- **判決**：B+A 已堵住主要誤用案例，C 邊際效益低且引入新邊界

| Ticket | Type | 範圍 | 預估成本 |
