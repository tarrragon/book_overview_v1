### 問題根因

phase4-decision-enforcement-hook 的 `format_block_message` 拒絕訊息僅提及「加 <!-- PC-093-exempt: cat:reason --> 標記」一句，未列白名單完整 category 清單也無適用情境說明。當 agent 在 Phase 4 撞到 BLOCK 時，只看到「需加標記」但不知：

1. 合法 category 有哪些（白名單）
2. 各 category 適用什麼情境
3. inline 標記是首選路徑（而非改寫文字繞過）

導致 agent 選擇「最簡字串繞過路徑」（改寫文字、改章節名）— W10-089.1 已實證此反模式，屬 PC-093 同精神（找捷徑而非正確路徑）。

### 修復摘要

改寫 `format_block_message` 拒絕訊息：

1. **標題後立即加「優先嘗試 inline 標記」提示段**（2 行），明示這是最低成本解法，並警告「改寫文字 = PC-093 同精神反模式」
2. **新增「合法豁免 category 白名單」段**，完整列出 **5 個** category（acceptance 寫 4 但實際 EXEMPT_CATEGORIES frozenset 為 5 個：`tdd-transition` / `baseline-gated` / `ticket-tracked` / `user-override` / `rule-quote`，已自行 grep 確認與 decision-trigger-binding.md 一致）
3. **每個 category 附「— 適用情境」一行說明**（如「rule-quote — 引用 .claude/rules/ 或 .claude/pm-rules/ 規則名稱（reason 須含規則路徑）」）
4. **訊息末段指向權威規則**：`.claude/rules/core/decision-trigger-binding.md「Hook 引用豁免機制」章節`
5. **更新文案**：原「二選一」改「三選一」（執行/移除/豁免本就是 3 條，前版表述錯誤一併修正）

### 設計決策

- **白名單以原始碼 `EXEMPT_CATEGORIES` 為單一真實來源**：訊息字串硬編，但測試用 `for category in (...)` 對 5 個全列項目驗證存在性，未來新增 category 時測試會紅燈提醒同步訊息（防 ARCH-020 漂移）
- **inline 提示放標題正下方而非結尾**：agent 讀錯誤訊息常掃首段；放開頭命中率最高
- **保留 grep 訊號（`AUQ 選項骨架`、`PC-093 Phase 4 強制決斷`）**：兼顧人類可讀性與 hook log 過濾
