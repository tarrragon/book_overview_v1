### 問題根因

`extract_and_write_context_bundle` (context_bundle_extractor.py L851-880) 含 logical read-modify-write 序列：

1. L852 `load_ticket(version, ticket_id)` 讀 target ticket frontmatter + body
2. L863-877 計算 extract 結果、merge auto-extracted block、`_replace_section_body`
3. L880 `save_ticket(target, ticket_path)` 整檔覆寫

並行情境下兩個 process 對同一 target 同時呼叫此函式，可能：
- Process A load → Process B load（皆讀到舊 body）
- Process A save → Process B save（B 覆寫 A 寫的內容，A 的更新遺失或產生重複 marker / 累積區塊）

實測（race regression test 無 lock 時）：N=20 並行下，最終 body 出現 2 個 `<!-- auto-extracted: v1 -->` marker，與 W14-005 lost-update / 重複累積模式同源。

### 修復摘要

以 `with file_lock(ticket_path): ...` 包圍 `extract_and_write_context_bundle` 完整的 load → modify → save 序列。

### 設計決策

1. **沿用 W14-042.1.2 共用 helper**：直接 `from .file_lock import file_lock`，不重複實作；遵循 file_lock docstring 規定的「禁止巢狀」契約（save_ticket 內部無 lock，僅外層包圍一次）。

2. **ticket_path 計算提前**：file_lock 需要 target_path 決定 lock file 位置（`{path}.md.lock`），故 `get_ticket_path(version, ticket_id)` 從原本的 L879 提前至 try 之前。功能等價（純路徑計算，無副作用）。

3. **整個函式 body 移入 lock 區塊**：包含 load + merge + save 三步驟。早 return 路徑（`target is None`、`not rendered`、`no_change_idempotent`）仍位於 lock 內，正常 release（with 區塊離開時 finally 觸發 LOCK_UN）。

4. **Exception 處理保留原契約**：`ContextBundleExtractionError` 仍從 load 失敗時拋出，with 區塊正常釋放鎖。

### 不在本 ticket 範圍

- W14-042.1.4 caller 審計識別的其他 7 處（lifecycle.py 等）：各自獨立 ticket（W14-044 等）。
- save_ticket 內部加 lock：W14-042.1.2 已明確決議不在 save 層加鎖（巢狀風險 + single-write atomicity 已足）。

### 變更檔案
