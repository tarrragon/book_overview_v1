# Execution Log

## Task Summary

評估 CC 2.1.97 /agents 命令 Running 分頁能否取代或補強現有 PM 判斷背景代理人機制（dispatch-active.json + agent-commit-verification-hook）；對照最近 commits（W5-021 permissionMode 批次修復 + PC-059 retry5）中處理過的案例，確認 /agents 分頁能否顯示 run_in_background、worktree 隔離、失敗/超時代理人

---

## Problem Analysis

### 問題根因

PM 判斷背景代理人狀態時有多個盲點，近期連續觸發 retry：

| Pattern | 盲點描述 | Retry 次數 |
|---------|---------|-----------|
| PC-045 | 代理人未即時完成 → PM 自己寫產品程式碼 | 2 次 |
| PC-050 模式 A | PM 在 main 看 git status → 忽略 feature 分支有 commit | 多次 |
| PC-050 模式 B | 多代理人派發時只看一個就行動 | 1 次 |
| PC-050 模式 C | 共用分支失去隔離 | 2 次 |
| PC-050 模式 D | 讀 transcript 中間狀態推論失敗 | 1 次 |
| PC-059 | permissionMode 受 subagent cwd 限制 | 5 次 retry |

**根因共通性**：PM 看不見背景代理人的「即時執行狀態」，只能靠檔案系統證據（dispatch-active.json、git status、transcript）間接推論，且各來源會不一致或有延遲。

### 影響範圍

- `.claude/hooks/active-dispatch-tracker-hook.py`（PostToolUse 清理派發記錄）
- `.claude/hooks/agent-commit-verification-hook.py`（檢查 worktree/feature 分支未合併 commit）
- `.claude/hooks/lib/dispatch_tracker.py`（dispatch-active.json 讀寫）
- `.claude/pm-rules/pm-role.md`（失敗判斷前置步驟、完成確認 SOP）
- `.claude/dispatch-active.json`（活躍派發狀態）

### 相關 Error Pattern

- PC-045 — PM 代理人失敗時自行撰寫產品程式碼
- PC-050 — 過早代理人完成/失敗判斷（含變體 X、Y、模式 A-D）
- PC-059 — Agent tools vs runtime permission 邊界

## Solution

### Part 1：現有機制能力清單（程式碼審查結論）

| 資訊類型 | 現有來源 | 更新時機 | 可靠度 |
|---------|---------|---------|-------|
| 活躍派發數量 | `dispatch-active.json` | PostToolUse 清理（Hook） | 高（但有 stale 風險 → 超時清理） |
| 派發者、代理人類型、描述 | `dispatch-active.json` | 派發時寫入 | 高 |
| 代理人實際執行中？ | 無直接來源 | — | **盲點** |
| 代理人回合耗盡？ | 無直接來源（需等 task-notification） | — | **盲點** |
| 未 commit 變更 | `agent-commit-verification-hook`（完成時 git status） | PostToolUse | 中（只在完成時觸發） |
| worktree/feature 分支 commit | 同上（hook 內 `git worktree list` + `git branch`） | PostToolUse | 中（只在完成時觸發） |
| Orphan 分支偵測 | `active-dispatch-tracker`（`detect_orphan_branches`） | PostToolUse | 中 |

**現有機制的根本限制**：所有資訊都是**事後觸發**（PostToolUse Hook）或**靜態快照**（dispatch-active.json），PM 無法在代理人執行中途得知「它還活著嗎？」「它卡在哪個階段？」

### Part 2：PM 可用的觀察工具（實證結果，非推測）

**結論先行**：`/agents` 是 REPL UI（PM 無法呼叫），但 PM 有其他工具可達到相同效果。

#### 2.1 實測方法

派發 2 個 Explore 背景代理人（掃描 .claude/ 深度報告），在執行中以 parallel tool calls 呼叫 `TaskOutput` / `TaskList` / `Bash(cat dispatch-active.json)`，記錄實際返回值。

#### 2.2 工具能力實證

| 工具 | 呼叫方式 | 實證結果 | 適用場景 |
|------|---------|---------|---------|
| **TaskOutput** (`block=false, timeout=3000`) | 用 Agent tool 返回的 `agentId` 當 `task_id` | **✓ 可用**：返回 `<status>running</status>` + `<task_type>local_agent</task_type>` | 確認特定代理人是否仍在執行 |
| **TaskList** | 無參數 | ✗ 不適用：「No tasks found」；這是 TodoList 系統，非 background agent 清單 | 無（誤導性） |
| **dispatch-active.json** | `cat` 或 Bash | ✓ 可用：Hook 維護的派發記錄 | 計數 / 失敗判斷前置步驟 |
| **completion notification** | 自動到達（system-reminder） | ✓ 可用：下一輪 PM 回合的 system-reminder 中 | 完成事件驅動 |

**關鍵矯正**：TaskOutput 的 DEPRECATED 說明（「For local_agent tasks: use the Agent tool result directly」）**部分誤導**。實測證明它對 `local_agent` 狀態查詢有效，且其 `<status>` 標籤是 PM 唯一能主動取得「代理人是否仍在執行」的資料來源。

#### 2.3 TaskOutput 的使用風險

| 風險 | 說明 | 緩解 |
|------|------|------|
| Context 污染 | 返回值包含截斷的 JSONL transcript（數十 KB） | 只讀 `<status>`、`<retrieval_status>`、`<task_type>` 標籤，忽略 `<output>` 內容 |
| 誤用 transcript | PC-050 模式 D 禁止讀 transcript 推論代理人狀態 | 嚴禁從 output body 推論；只信任 `<status>` |

#### 2.4 意外發現：回合時序

代理人完成得比預期快（2 個測試派發都在 PM 下個回合之前完成並送達 completion notification）。這代表：
- PM 派發背景代理人後，**下一次 PM 回合開始時 80%+ 機率代理人已完成**
- 「即時觀察中途狀態」的需求其實比想像少
- 真正需要觀察的場景是：代理人執行時間長（例如需要讀 50+ 檔案的大型 Explore）

### Part 3：能力覆蓋對照表（實證後修訂）

| 場景 | dispatch-active.json + Hook | TaskOutput（PM 可呼叫） | 覆蓋結論 |
|------|----------------------------|------------------------|---------|
| **1. `run_in_background: true` 代理人是否存活** | 間接（派發記錄存在不等於仍在執行） | ✓ `<status>running</status>` 直接判定 | **TaskOutput 補強** |
| **2. Worktree 隔離代理人** | 完成後 Hook 檢查 worktree commit | 同 1（有 agentId 即可查） | 兩者互補 |
| **3. 代理人失敗/超時** | 超時清理 Hook + task-notification | `<status>` 會顯示 completed/error | 兩者皆可 |
| **4. 代理人回合耗盡**（PC-050 模式 D） | 無法偵測 | ✓ `<status>running</status>` 可確認仍執行中 → 不誤判 stop | **TaskOutput 根本解法** |
| **5. 未 commit 變更檢查** | Hook 完成時 git status | 不提供 | Hook 不可取代 |
| **6. 並行派發計數** | `dispatch-active.json` 長度（Source of Truth） | 需逐個 agentId 查詢 | `dispatch-active.json` 更適合 |

### Part 4：結論與建議行動（實證後修訂）

**結論**：`/agents` 是 REPL UI，PM 無法呼叫；但 PM 擁有更強的 **TaskOutput** 工具，可補強 PC-050 模式 D 的盲點。

| 現有機制 | 建議 | 理由 |
|---------|------|------|
| `dispatch-active.json` | **保留（Source of Truth）** | Checkpoint 1.85 計數 + Hook 邏輯依賴 |
| `agent-commit-verification-hook` | **保留** | 唯一的 git commit 檢查來源 |
| `active-dispatch-tracker` | **保留** | Orphan 分支偵測獨特能力 |
| `pm-role.md` 失敗判斷前置步驟 | **升級** | 加入 Step 0.5：以 TaskOutput 查 `<status>` 再判失敗 |
| `pm-role.md` 完成確認 SOP | **升級** | 允許 TaskOutput 作為 dispatch-active.json 補充驗證（非取代） |
| PC-050 模式 D 防護規則 | **修訂** | 從「完全禁止讀 transcript」改為「允許讀 `<status>` 標籤，禁止讀 output body」 |

**建議行動**（生成後續 Ticket）：

1. **W7-001（P1，DOC）**：修改 `.claude/rules/core/pm-role.md` 「失敗判斷前置步驟」，新增 Step 0.5：
   ```
   對每個可能失敗的 agentId 呼叫 TaskOutput(task_id, block=false, timeout=3000)
   只讀 <status> 標籤，若為 running 則不可判失敗
   ```
2. **W7-002（P1，DOC）**：修訂 PC-050 模式 D 的防護規則
   - 舊版：「禁止主動讀取 agent transcript output 檔推論 agent 是否失敗」
   - 新版：「允許呼叫 TaskOutput 讀 `<status>` 標籤；禁止讀 `<output>` body 推論」
3. **W7-003（P2，DOC）**：於 `.claude/references/` 新增 `pm-agent-observability.md`，含：
   - TaskOutput 安全使用範本（只讀 status）
   - dispatch-active.json + TaskOutput + completion notification 三者分工表
   - Context 污染風險警告（為何禁止讀 output body）

**兩套機制互補**（非替代）：
- `dispatch-active.json`（持久化）：回答「我派發了幾個？Hook 認為哪些完成了？」
- **TaskOutput**（即時查詢）：回答「這個特定代理人現在還在執行嗎？」
- `agent-commit-verification-hook`（完成時）：回答「代理人留下了什麼 git 證據？」
- completion notification（事件驅動）：回答「代理人何時完成？」
