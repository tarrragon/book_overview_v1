### 問題根因

W10-098 為了讓既有 draft PROP（PROP-001 / PROP-003）通過 hook 章節檢查，採取「draft → 暫設 light」grandfather 妥協（方案 B+D）。WRAP Consider the Opposite 階段提出選項 E：應修改 hook 讓 `status=draft` 自動豁免章節檢查，使 `evaluation_level` 維持實際強度（heavy/standard），與 `status` 解耦。

**結構性問題（重現實驗發現 1+2 證實）**：

`light` 語意在現行落地中承載兩義：
- 設計語意：拼字修正 / 格式調整 / 重命名等低摩擦變更
- 實際語意：draft 探索期暫存（grandfather 妥協）

兩義混淆造成：

| 觀察維度 | 後果 |
|---------|------|
| 讀取 PROP 真實強度 | 必須同時看 `evaluation_level` + `status` 才能判斷（無法單欄位推論） |
| 未來新增拼字修正 PROP | 與 draft 暫設混在 light 池內，無法區分原始用途 |
| promote draft → discussing | 兩步驟（改 level + 補章節），易遺漏 level 更新 |
| 對齊 ADR/RFC 慣例 | draft 自然語意是「探索中、章節可不完整」，現行機制與業界慣例脫節 |

### 影響範圍

| 層級 | 檔案 | 影響類型 |
|------|------|---------|
| Hook 強制層 | `.claude/hooks/proposal-evaluation-gate-hook.py` | 若選方案 C/D，新增 status=draft 豁免分支（~10 行） |

## 重現實驗結果

### 實驗方法

盤點 `docs/proposals/PROP-*.md` 所有 PROP 的 `frontmatter.evaluation_level` 與 `frontmatter.status` 配對，比對 W10-098 落地後 `light` 機制的真實承載分布。指令：

```bash
for f in docs/proposals/PROP-*.md; do
  awk '/^---$/{c++; if(c==2)exit} /^id:/{id=$0} /^status:/{s=$0} /^evaluation_level:/{l=$0}
       END{ ... print id "  status="s "  level="l }' "$f"
done
```

### 實驗發現

**已驗證的事實**：

1. **light 機制當前承載 0% 設計目的**：規則文件規則 1 將 `light` 定義為「拼字修正 / 格式調整 / 重命名 / spec 盤點批次建立」用途。11/11 個既有 PROP 中**沒有任何一個**屬於此類。
2. **light 機制當前承載 100% grandfather 妥協**：唯二的 `light` 用例（PROP-001 / PROP-003）皆是 `status=draft` 暫設，本質強度為 heavy（PROP-001 target_version=v2.0+ 跨版本）或 standard。
3. **W10-098 採方案 B+D 已落地**：`draft → 暫設 light` 規避章節檢查的 grandfather 妥協方式已寫入 12 PROP frontmatter，且兩個 draft 案例的 ticket md 註記「promote 時重評」。
4. **hook 行為機制**：`proposal-evaluation-gate-hook.py:160-162` 對 `level == "light"` 直接 `return False, ""`（跳過章節檢查），與 `status` 欄位完全無關。
5. **規則 1 與規則 2 的耦合點**：規則 1 強制 frontmatter 必標 `evaluation_level`，規則 2 強制 standard/heavy 必含章節；兩條規則對 draft 探索期 PROP 無豁免分支。

**仍未驗證的假設**：

- 假設 A：「未來真有拼字修正類 light PROP 出現」的頻率——目前 0 案例，無法判斷選項 D（保留 light 純語意）是否仍有必要。
- 假設 B：「status=draft 永久滯留作章節豁免規避」的濫用風險強度——需 stale draft 追蹤資料佐證。
- 假設 C：選項 E 修改後對既有 11 PROP 的回滾成本（PROP-001/003 重評 level 是否會誘發 hook 阻擋）。

### 初步推薦（PM 立場，待多視角審查覆核）

**方案 D（C + light 收斂純語意）**，理由：

1. **重現實驗證實 light 機制當前 0% 設計目的**（11/11 案例）。保留現狀（A）等於把 grandfather 妥協結構性化。
2. **C 與 D 同樣解開 status × level 耦合**，但 D 額外修補 light 語意債務，邊際成本低（規則文件 ~20 行改動）。
3. **E 過於激進**：理論上純粹但放棄了未來「真有拼字修正類 PROP」的合法 light 用例；且現行 hook level 列舉值改動牽涉測試覆寫。
4. **B 是中間妥協**：細分 light 但仍把 draft 概念塞 level 欄，語意污染未根治。

### 多視角審查結論（Phase 2）

**linux 視角**（dispatched 2026-05-12，產出 ~600 字）：

- 偏好排序：**E > D > C > A > B**
- 核心論證：light 池 11/11 證實 0% 設計用例，是 dead code（YAGNI 鐵則）；status × level 本該正交（status 描述生命週期，level 描述工程強度），D 仍讓 level 兼任 status 提示
- 對 D 立場：**弱支持，傾向覆核改 E**

- 一句總結：用 `evaluation_level=light` 承載 `status=draft` 的豁免邏輯，是把「事實欄位」與「判斷欄位」混用，會在每次新增 PROP 時繼續繁殖兩義性

**basil 視角**（dispatched 2026-05-12，產出 ~900 字）：

  - 條件 2：規則文件規則 1 必須新增 status=draft 豁免條款（避免 PC-118 規則-行為脫節）
- hook 改動位置建議：**`check_prop_content` 內 line 162 之後**（保留規則 1 frontmatter 強制力，複用既有 light 分支對稱性）
