## Problem Analysis

### 問題根因

`commands/track_acceptance.py:548` 的 section 匹配脆弱：
```python
section_pattern = rf"## {re.escape(section)}\n(.*?)(?=\n##|\Z)"
```
要求標題前後完全一致（單一空白、結尾 `\n`）。實務踩坑：
- `## Solution ` 末尾空白 → SECTION_NOT_FOUND
- `##  Solution`（雙空白）→ SECTION_NOT_FOUND
- 中文標題或標題後加副註 → SECTION_NOT_FOUND
- 用戶不知該 ticket 實際標題為何

### 影響範圍

### 設計決策（W17-004 落差 11 兩方案合併）

- A 正規化匹配：pattern 改為 `rf"^##\s+{re.escape(section)}\s*$"`（多行模式 + 容許前後空白），與 acceptance_auditor.py:364 同步
- B 錯誤訊息改善：SECTION_NOT_FOUND 時 grep 該 ticket md 所有 `^## ` 標題列出，引導用戶用正確名稱

### 修改摘要

- `commands/track_acceptance.py:548` section_pattern 改用 MULTILINE + `^##\s+{section}\s*$` 容許前後空白（與 acceptance_auditor.py:364 同步）
- 移除 `\Z` 後行為改用 `\n## ` 分隔次區段
- SECTION_NOT_FOUND 錯誤訊息：grep `^##\s+.+$` 列出該 ticket 所有現存 ## 標題引導用戶

### 設計決策

- 採 A+B 合併方案（規格 Problem Analysis）：A 正規化容尾空白 / B 錯誤訊息列既有標題
- 不抽 helper（避 scope creep；散落多處屬獨立重構 ticket）
- 邊界錨點 `^...$` 與 `\s*$` 確保 `Solutions` / `Solution alt` 不誤匹配
