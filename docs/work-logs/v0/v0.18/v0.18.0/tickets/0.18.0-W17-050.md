## 根因分析（5W1H + 因果鏈）

### 觸發事件

W17-048 等多張 ticket 執行期間，代理人（包括主規劃者）產出持續違反 compositional-writing 與 document-writing-style 規範：簡體字混入 AskUserQuestion payload（PC-072）、資訊優先序顛倒（PC-066 案例鏈）、禁令無正向錨點（PC-080）、Unicode escape 錯字形（PC-085）。

### 直接原因（What）

規則存在但缺少常駐執行者。document-writing-style.md v1.2.0 已明示「最高優先原則：二次審查強制執行」，但：

1. 代理人無法主動載入 SKILL（平台系統限制）
2. 主線程 PM 在高 context 壓力下降低自審機率（PC-066）
3. 既有代理人職責分散，無單一角色常駐執行文字品質審查

### 根本原因（Why）

文字品質規範與執行層脫節。類比 linux 之於架構品質的關係——規則需要專責代理人擔任「常駐委員」，才能在多視角審查中必然在場。

### 因果鏈

規則宣告（document-writing-style v1.2.0 + compositional-writing SKILL）
    → 代理人無法載入 SKILL（平台限制）
    → 規則僅靠 prompt 傳達 + 代理人自律
    → 高壓情境下規則被跳過（PC-066 機率分布）
    → 產出違反（PC-072 / 074 / 080 / 085 重現）
    → 下游修正成本倍增（codex 實驗：第一次生成 vs 第二次修正的成本差）

### 一、一致性/衝突清單

#### 1.1 basil- 命名前綴與既有群組定位

**判定：有衝突（warning）**

既有 basil- 前綴代理人為 basil-event-architect（事件驅動架構）與 basil-hook-architect（Hook 系統實作），兩者皆為「架構建造者」（architect）角色，屬於設計/實作類代理人。新提案 basil-writing-critic 為「審查者」（critic）角色，職責性質與既有 basil 群組不同。

**條款引用**：agent-definition-standard.md 未明示前綴必須對應群組，但 document-writing-style.md 三明示原則要求命名決策應交代 Why。Solution 2.1 僅解釋 basil 的語言學聯想，未論述與既有 basil-* 群組的關係。

## 二、新代理人職責邊界

### 2.1 代理人命名提案

**提案名稱**：basil-writing-critic（暫訂）

| 屬性 | 值 | 理由 |
|------|-----|------|
| 名稱骨幹 | basil（羅勒） | 沿用專案香料命名慣例；basil 在英文中有「語言藝術守護者」的次要聯想 |
| 角色後綴 | writing-critic | 明示職責是「審查書面文字」而非撰寫；critic 對應 linux 的 reviewer 定位 |
| 備選名稱 | basil-prose-auditor / basil-text-critic | 若 basil 已被其他代理人占用，由 PM 在 IMP 階段最終裁決 |

**Why**：命名需一眼可辨職責（writing = 文字、critic = 審查），並與既有常駐委員 linux 對稱（linux 審架構，basil 審文字）。

**Consequence**：命名若混淆（例如叫 basil-writer），會誤導 PM 派發去做撰寫任務，破壞審查者職責純度。

**Action**：IMP 階段確認名稱前先以 ls .claude/agents/ 確認無命名衝突。

### 2.2 核心職責（3 職責 + L3 兜底設計）

> **設計版本**：v2（W17-067 修改，依 W17-066 多視角審查 PM 彙整 R-1 落地）
> **設計依據**：linux 審查 L-C1 Critical + PM 仲裁共識 C-1：職責三/四（禁用字、亂碼）為可規則化偵測，應主歸 Hook 層（L1 防線）；basil 聚焦三大語意判斷職責 + L3 兜底 fallback。

#### 職責一：三明示結構驗證

審查書面文字是否對每段論述明示「為何 / 後果 / 下一步」三層資訊，並偵測隱含表達 6 句型。

**Why**：規則文件若缺三明示，讀者需自行推論隱藏涵義，規則無法獨立執行；這正是 document-writing-style.md v1.2.0 的核心契約。隱含表達 6 句型（saffron warning 1 / W17-066）為三明示違規的具體模式——句型本身即「條件模糊、責任外推、理想未落地」，是三明示缺失的症狀。

**Consequence**：遺漏三明示會讓規則堆積為「看似涵蓋但不可執行」的條目；PM 與代理人在壓力情境下跳過該規則，規則設計意圖失效。隱含表達若漏偵測，讀者仍需猜測「適用條件」與「下一步」，效果等同缺 Action。

**Action**：掃描每段論述，標註缺 Why / Consequence / Action 中任一項的段落，同時掃描以下 6 句型並回報修正方向。

**隱含表達 6 句型偵測表**（來源：document-writing-style.md「反模式：人性化/隱含表達」章節）：

| 句型 | 違規原因 | 重寫方向 |
|------|---------|---------|
| 「希望讀者理解…」 | 把責任推給讀者，不交代依據 | 明示：「此原則依據 X，違反會導致 Y」 |
| 「按理應…」「自然而然…」 | 假設共識，實際共識可能不存在 | 明示條件：「當 X 條件成立時，應做 Y」 |
| 「通常來說」「一般情況下」 | 條件模糊，讀者無法判斷是否適用 | 明示邊界：「除了 X/Y 情境外，應做 Z」 |
| 「規則寫得好卻沒人用」 | 描述症狀但不指出成因和修正動作 | 明示因果+動作：「因為 X 機制缺失，導致 Y 發生；修正方向：Z」 |
| 「假設讀者會注意到」 | 把規則的有效性寄託於讀者自律 | 明示強制點：「Hook A 在時機 B 強制檢查；人工 fallback 在 C」 |
| 「理想情況下」 | 未指明理想與現實差距的處理方式 | 明示落差處理：「理想 X；現實 Y；先做 Z 達到 W」 |

#### 職責二：資訊優先序檢查

審查技術論述的句子/短段落層級資訊順序，確保「核心原則 → 示例 → 邊界提醒」的順序正確。

**Why**：顛倒順序會讓讀者先記脫離脈絡的示例，再倒推抽象概念；認知路徑反向造成主次不分，是 AI 產出技術文章的系統性模式。

**Consequence**：讀者必須讀完整句才能理解前文示例代表的意義，認知負擔上升；不熟悉主題的讀者容易誤以為示例本身是結論。

**Action**：逐句檢查「若讀者只讀第一句，是否知道這個概念是什麼」；若答案為否，標註需將原則移到句首。

#### 職責三：正面陳述審查

審查文件是否對每條「禁止 X」的反模式提供對應的「正確做法 Y」正向錨點。

**Why**：document-writing-style.md 二次審查清單明示「負向對比有正向錨點」是核心審查項；只列禁令不教修正方向，讀者知道不能做什麼但不知道往哪走。

**Consequence**：單向禁令會讓代理人陷入 WRAP 限制性解法（禁止 X / 規避 X）的認知陷阱（PC-066 / PC-080），遮蔽探索性解法（找正確方式做 X）。

**Action**：掃描文件每個「禁止」「不可」「禁用」條目，確認其後或同段中有對應的「應改為」「正確做法是」「改用」錨點；缺失則標註需補。

### 2.2.1 Hook 層化設計（L1/L2 防線）與 L3 兜底

**設計原則**：可規則化偵測（禁用字、字元集污染）應主歸 L1 Hook 層，用機器取代 LLM 語意消耗；basil 作為 L3 語意審查層，兼掃 L1/L2 漏網之魚。

| 防線 | 執行者 | 職責 |
|------|--------|------|
| L1 — 同步阻擋 | charset hook / language-guard hook | emoji、禁用詞、簡體字、Unicode escape 等可規則化字元 |
| L2 — 非阻擋掃描 | basil 審查既存檔案時的 Grep 掃描 | Hook 規則尚未覆蓋的模式（新增情境） |
| L3 — 語意兜底 | basil 語意判斷 | 三明示缺失、資訊優先序顛倒、正面陳述缺錨點、隱含表達句型（語意層，Hook 無法覆蓋） |

**Why**：W17-066 linux L-C1 Critical 指出「用 opus 跑 grep 等同浪費算力」；Hook 層在 commit 時同步阻擋的成本遠低於 LLM 事後審查。

**Consequence**：若禁用字偵測仍由 basil 主責而非 Hook，每次派發 basil 需額外 17% token 用於可規則化掃描（ginger 估算），且 Hook 同步阻擋的即時防護缺失。

**Action**：IMP 執行時，禁用字與字元集偵測職責轉移至 W17-068 Hook 擴充；basil 僅在審查既存檔案時保留 L2 掃描（Grep 指令），不在核心職責中主責 L1 偵測。

### 2.3 與 linux 的職責邊界分工

| 維度 | linux（架構常駐委員） | basil-writing-critic（文字常駐委員） |
|------|---------------------|-----------------------------------|
| 審查對象 | 程式碼、架構、設計決策 | 書面文字（規則 / 註解 / ticket body / 分析報告 / prompt / commit message） |
| 品質維度 | Good Taste、特例消除、向後相容、複雜度 | 三明示、資訊優先序、禁用字、字元集、正面陳述 |

## 三、與 parallel-evaluation skill 的整合機制

### 3.1 整合層級：常駐委員 vs 情境視角

basil 的整合模式與 linux 對稱——常駐委員機制（所有情境都可加入），但觸發頻率依情境不同。

**Why**：文字品質不是每個審查場景都需要——程式碼變更審查（情境 A）關注重點是程式碼本身，不是 ticket 文字；但規劃/結論/系統設計類場景（情境 D / F / G）產出物本質是文字，必須加入。

**Consequence**：若將 basil 設為情境 A-G 全部強制加入，會浪費 token（程式碼審查中 basil 無事可做）；若只設為單一情境選用，會失去常駐委員的覆蓋價值（結論審查場景容易遺漏派發）。

**Action**：採「預設加入 + 情境差異化頻率」策略，具體如下表。

### 3.2 情境加入策略

| 情境 | 加入頻率 | 理由 |
|------|---------|------|
| A: 程式碼變更審查 | 選用 | 標的為程式碼，basil 僅檢註解品質；除非變更含大量註解/docstring 修改 |
| B: 重構決策評估 | 選用 | 標的為結構；僅在評估報告本身也需要審查時加入 |
| C: 架構方案評估 | 強制 | 架構決策文件為書面產物，需審明示性與資訊優先序 |
| D: 功能設計評估 | 強制 | 功能規格為書面產物 |
| E: 系統冗餘偵測 | 選用 | 標的為程式碼/模組 |
| F: 結論/報告審查 | 強制 | 結論報告本身即為書面文字，明示性是核心審查維度 |
| G: 系統設計評估（規則/Skill/方法論變更） | 強制 | 標的即規則與方法論文件 |

**Why**：情境 C / D / F / G 的產出本體是文字，basil 必須在場；情境 A / B / E 標的是程式碼，basil 為選用以節省 token。

**Consequence**：強制情境不加入會讓 W17-048 類事件（規則/方法論違反明示性）重現；選用情境強制加入會造成情境 A 每次派 4-5 個 agent 的 token 浪費。

**Action**：在 parallel-evaluation SKILL.md 的「常駐委員」章節新增 basil 一行，在情境表補齊「basil 加入頻率」欄；同步更新 lens-configurations.md 新增 basil 視角描述，lens-prompts.md 新增 basil prompt 模板。

### 3.3 與 Worth-It Filter 的對應

basil 的審查發現依嚴重度對應 Worth-It Filter，走同一套規則：

| basil 嚴重度 | 對應 Worth-It Filter | 判定標準 |
|--------------|---------------------|---------|
| 嚴重（Critical） | 高幅度 | 禁用字命中、簡體字、Unicode escape 錯字形、三明示全缺 |
| 中等（Warning） | 中幅度 | 資訊優先序錯、正面陳述缺錨點、單段論述缺 Action |
| 輕微（Info） | 低幅度 | 可選的邊界說明、風格建議 |

**Why**：沿用既有 Worth-It Filter 的判定框架，讓 PM 不需額外學習 basil 專屬嚴重度標準。

**Consequence**：若另創嚴重度體系，PM 需維護兩套判定邏輯，增加派發摩擦。

**Action**：IMP 階段將本表寫入 basil agent definition 的「輸出格式」章節，並在 parallel-evaluation lens-configurations.md 的 basil 視角補充對應表。

### 3.4 語言代理人加入規則的邊界確認

parallel-evaluation SKILL.md 已定義「語言代理人加入規則」（Flutter / Python / Go 涉及時加入對應語言專家）。basil 與此規則正交：

| 維度 | 語言代理人（parsley / thyme / fennel） | basil-writing-critic |
|------|------------------------------------|--------------------|
| 審查對象 | 程式碼與框架慣例 | 書面文字明示性 |
| 加入觸發 | 標的涉及該語言/框架 | 標的涉及書面文字產出 |
| 並存情境 | 例如 G + Python（Hook 規則 + 文字品質）同時派 thyme + basil | 不重疊 |

**Why**：語言代理人與 basil 審查不同維度，可並存而不互斥。

**Consequence**：若將 basil 視為取代語言代理人，會失去框架專業審查能力；反之亦然。

**Action**：在 basil agent definition 的「與其他代理人邊界」明示「不替代語言代理人的框架慣例審查」。

## 四、代理人定義檔骨架

### 4.1 檔案路徑與 frontmatter

**目標路徑**：.claude/agents/basil-writing-critic.md

frontmatter 骨架：

- name: basil-writing-critic
- description: 文字品質常駐審查委員（compositional-writing + document-writing-style 執行者）。審查書面文字的三明示結構、資訊優先序、禁用字、字元集污染、正面陳述。parallel-evaluation 情境 C / D / F / G 強制加入，與 linux 並列常駐。產出審查報告，不修改文件。Use when: 規則/方法論變更後、分析報告產出後、Ticket 規劃完成後、Phase 1 功能規格產出後。
- tools: Read, Grep, Glob, Bash
- color: green
- model: opus
- effort: low

主文開頭沿用 linux / bay 慣例：`@.claude/agents/AGENT_PRELOAD.md` + 一句話定位。

**Why**：frontmatter 的 description 是 PM 派發時的一句話決策依據，必須明示「審什麼 / 何時派 / 產出什麼」；觸發詞（Use when 列表）讓 ToolSearch 能命中。

**Consequence**：description 若模糊（例如「文字審查工具」），PM 在多視角派發時容易遺漏，違反常駐委員意圖。

**Action**：IMP 階段撰寫 description 時，參照 linux 的 description 句式（角色定位 + 觸發情境），確保可 grep 命中。

### 4.2 區塊 1：允許產出

| 產出類別 | 範圍 |
|---------|------|
| 文字審查報告（Markdown） | 違規位置清單、修正方向、嚴重度評分（Critical / Warning / Info）、對應 Worth-It Filter 幅度 |
| 明示性改寫建議 | 針對缺 Why / Consequence / Action 的段落提供重寫骨架（不代寫完整文字） |
| 禁用字替換清單 | 命中位置 + 正確替代詞（引用 language-constraints.md 規則 2） |
| 字元集污染報告 | 簡體字 / 繁日共用字 / emoji / Unicode escape 錯字形的行號與修正建議 |
| 唯讀分析操作 | Read / Grep / Glob / Bash（唯讀掃描指令） |

**Why**：允許產出必須與 tools 欄位對應（Read / Grep / Glob / Bash），避免執行時拒絕工具。

**Consequence**：若允許產出宣稱「Edit 修正」但 tools 沒有 Edit，代理人會在執行時失敗並回退，浪費 token。

**Action**：IMP 階段嚴格對齊 tools 與允許產出，修正由 PM 或 PM 派發的 mint-format-specialist 處理。

### 4.3 區塊 2：禁止行為

絕對禁止清單：

1. 禁止修改任何文件 — 唯一產出是審查報告；修正工作交由 PM 或其他代理人
2. 禁止審查架構與程式碼結構 — 此為 linux 職責；basil 不評估 Good Taste、複雜度、特例消除
3. 禁止替代語言代理人的框架慣例審查 — 例如不評估 Dart / Python / Go 的語言慣例
4. 禁止撰寫原創內容 — 僅提供「缺 Why / 缺 Action」的改寫骨架，不代寫完整段落
5. 禁止跳過二次審查 — 自己的產出（審查報告本身）也必須套用三明示原則，提交前自我掃描
6. 禁止使用簡體字、禁用詞、emoji — 違反即為自身違規，需重新輸出繁體中文版本
7. 禁止替代 PM 做派發決策 — 僅報告發現與嚴重度，不決定後續由誰修正

違規時必須停止並升級到 rosemary-project-manager。

**Why**：禁止行為明示職責邊界，避免 basil 越界替代 linux 或語言代理人。

**Consequence**：邊界模糊會導致 parallel-evaluation 多委員結論重疊稀釋，PM 的 Worth-It Filter 難以分類。

**Action**：IMP 階段在撰寫後請 linux + saffron 多視角審查本區塊，確認無與既有代理人職責衝突。

### 4.4 區塊 3：適用情境

| 維度 | 說明 |
|------|------|
| TDD Phase 標註 | Phase 1（功能規格產出後）、Phase 4（重構決策報告產出後）、N/A（獨立任務如規則變更審查、ANA 結論審查） |

### 4.6 輸出格式章節骨架
