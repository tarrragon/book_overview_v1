### 修復摘要

於 migrate.py 加入 collision detection 雙層工具強制（W14-047 ANA 方案 1 的 L2 工具層）：

1. **新增 helper `_check_target_collision(target_id, version)`**：回傳 `{path, title, status}` 或 None，集中 target 路徑解析與存在性檢查邏輯。

2. **`_migrate_single_ticket` 新增 `force_overwrite: bool = False` 參數**：
   - dry-run 階段：若 target 已存在，輸出 `WARN_MIGRATE_TARGET_EXISTS`（含路徑、既有標題、既有狀態、提示 --force-overwrite 旗標）。
   - 實際執行：預設拒絕並 return 1（`ERROR_MIGRATE_TARGET_EXISTS`）；`force_overwrite=True` 時記錄 `INFO_FORCE_OVERWRITE` audit log 後繼續。
   - 例外：`source_id == target_id` 視為 in-place rename，不算 collision。
   - 在 `load_and_validate_ticket` 前加入實體檔案存在檢查（規避 process-scoped cache 在冪等 re-run 場景的誤命中）。

3. **`_batch_migrate` 加入 pre-scan**：dry_run=False 且 force_overwrite=False 時，先掃描所有 (source, target) pair，蒐集 collision 列表；任一 collision 即輸出 `ERROR_BATCH_COLLISION` 並 return 1，不執行任何 migration（fail-fast）。Pre-scan 跳過 source 不存在的條目（讓 idempotent re-run 保持原本 skip 語意）。

4. **argparser 新增 `--force-overwrite` 旗標**（default=False）。

5. **訊息常數**：`command_tracking_messages.py:MigrateMessages` 新增 5 個常數
   - `WARN_MIGRATE_TARGET_EXISTS` / `ERROR_MIGRATE_TARGET_EXISTS` / `ERROR_BATCH_COLLISION` / `INFO_FORCE_OVERWRITE` / `ARG_FORCE_OVERWRITE`

### 設計決策

- **fail-fast 優於 partial-success**：批量遷移撞一個 ID 即全部停，避免「部分覆寫部分跳過」的混亂中間狀態。
- **--force-overwrite 寫入 stdout audit**：採用簡單 print，未引入新的 audit log 檔案；保持工具職責單一，必要時可由 W14-047 規則層引導用戶補建獨立 audit ticket。
- **source-exists check 提前到 load 前**：解決 `_ticket_cache` 在冪等 re-run 場景命中導致誤判 source 仍存在的問題（test_w11_reorganization_idempotency 驗證）。
