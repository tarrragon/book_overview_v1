### 問題根因

sync-claude-push.py 的 `parse_commit_type` 僅辨識 `<type>(scope): desc` 形式的 conventional commit，無法辨識 git 預設的 `Revert "..."` 格式；且 `categorize_commits` 沒有「淨效應」邏輯，導致同批推送同時含 X 與 revert(X) 時，遠端 commit log 與 CHANGELOG 仍列原 X，閱讀者看不到 X 已被撤回。

### 影響範圍

- `.claude/scripts/sync-claude-push.py`：`parse_commit_type` / `categorize_commits` / `generate_commit_summary` 的 display_order / `VERSION_BUMP_WEIGHTS`
- 觸發案例：W14-031 revert 推送後遠端 auto summary 仍列原 chore migration

### 相關 Error Pattern

無既有 PC 直接對應；屬 sync 腳本工具改善。

### 實作摘要

1. `parse_commit_type` 新增 git 預設 `Revert "<subject>"` 識別，歸類為 `revert` type。
2. 新增 `parse_revert_info(subject)`：支援三種格式（`revert(scope):`、`revert:`、`Revert "..."`），回傳 `(original_subject, original_ref)`；ref 優先萃取 Ticket ID（含 Wave 編號），其次裸 7+ hex hash。
3. 新增 `_normalize_subject_for_match`：剝 conventional prefix + 外圍引號 + lower，用於 revert ↔ 原 commit 配對。
4. 重寫 `categorize_commits` 加入淨效應邏輯：
   - 先掃所有 revert 建立「被撤回原 subject」集合
   - 同批的非 revert commit 若 normalized subject 命中該集合 → 跳過（被抵銷）
   - revert 行附註原 commit ref：`<original_subject> (原 commit: <ref>)`
5. `display_order` 將 `revert` 排首位；`VERSION_BUMP_WEIGHTS["revert"] = "patch"`。
6. 新增 `--dry-run` flag 與 `run_dry_run()`：只 shallow-clone 取得 last-sync timestamp（失敗則 fallback 全歷史）、collect+categorize+generate 後直接輸出，不修改 VERSION/CHANGELOG、不 push。

### 設計決策

- 配對採 normalized subject 比對（而非 hash / ticket ID）：覆蓋率最高，含 ticket ID 缺漏的 revert 也能配對。
- ref 萃取保留原 ticket 編號（不經 `strip_project_specific_info`）：revert 行的價值在「能追溯」，移掉 ticket ID 等於失去資訊。trade-off 是 revert 行可能含專案特定字串，但屬可接受例外（revert 本身即低頻、且摘要受眾理解該欄位作用）。
- `--dry-run` 使用 `--depth=1` 縮短 clone 成本；clone 失敗時 fallback 至全歷史，避免本地網路問題阻斷驗證。

### 不在本 ticket 範圍

- 不修改 sync-claude-pull.py（只影響 push 端摘要）。
- 不改變既有非 revert commit 的分類行為（測試覆蓋確認）。
