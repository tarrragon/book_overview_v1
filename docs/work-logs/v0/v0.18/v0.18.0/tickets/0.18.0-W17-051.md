## 問題敘述

代理人系統呈現三個交織的成本與品質問題：

1. **Token 浪費**：30 個代理人中 17 個（57%）掛 opus 或更高 model（含 bay 的 claude-opus-4-6[1m]），但多數代理人任務屬機械執行、格式處理或樣板套用，不需深度推理
2. **估時驅動決策**：PM 與代理人出現「預估太久所以跳過/簡化」的模式，讓估時取代「是否應該做」的本質判斷，違反敏捷 SRP 精神
3. **追求一次完美**：代理人嘗試在單次派發內完成理解、設計、實作、驗證全流程，出錯時整包重做，未利用「第二次檢測成本低於第一次生成」的 codex 實驗結論

## 問題根因（因果鏈）

**根因 A：model 選擇的認知偏見**
- 設計代理人時預設「拉到最高 model 就不會錯」
- 違反 SRP：任務越單純越不需高階推理，高 model 只增 Token 浪費
- 對比：decision point（pepper、sage、linux）適合 opus；execution point（thyme、parsley、cinnamon）適合 sonnet

**根因 B：估時禁令缺席**
- ai-communication-rules.md 已有規則 5（反諂媚），但未擴展到估時場景
- 缺明文禁令 → 估時成為最容易量化的決策依據 → 結構性遮蔽正確但慢的解法
- 本質：用戶/PM 用估時壓代理人 = 用資訊不對稱逼走捷徑，屬規則 5 變體

**根因 C：多輪驗收未落地**
- document-writing-style v1.2.0 已驗證「二次審查強制執行」，但只落地在文件撰寫
- 代理人工作流無對應機制：無自檢輪、無專責委員輪、無收斂輪
- W17-050 常駐委員是 Layer 2 的核心，但缺 Layer 1 自檢介面

#### 規則 6 條款文字（可直接合併）

### 規則 6：禁止以估時為決策依據

**Why**: 估時是可量化的誘餌，容易取代「是否應該做」的本質判斷。在 AI 輔助開發中，估時準確度低（token budget 波動大），用估時當決策依據等於用雜訊當訊號。此外，估時框架會讓代理人傾向「討好式選擇快的路徑」，遮蔽正確但慢的解法。

**Consequence**: 以估時驅動決策會導致（1）跳過多視角審查（「太耗時」）；（2）選簡化版而非正確版；（3）把「做不到」偽裝成「時間不夠」；（4）代理人輸出品質隨估時壓力下降。

**Action**: 禁止在決策陳述中出現以下框架：

| 禁止句型 | 替代句型 |
|---------|---------|
| 「預估花太久所以跳過 X」 | 「X 的價值是 Y；若超出 Wave 容量則建 Ticket 延後」 |
| 「這樣會花太多 token」 | 「此方案需要 Z 資源；若超出預算則拆 Wave」 |
| 「先做快的，慢的之後補」 | 「先做正確的；若有容量約束則依優先級排序」 |
| 「時間緊迫不跑多視角」 | 依摩擦力方法論判斷階段，前期階段無豁免 |

**邊界說明**：本規則禁止的是用估時當決策依據，不是禁止估時本身。估時仍可作為（1）資源規劃輸入；（2）Wave 容量檢查；（3）向用戶報告進度。關鍵是估時不進入「做 vs 不做」「做 A vs 做 B」的決策邏輯。

**與規則 5 的銜接**：規則 5 §5.6 機制 4 禁止「最佳實踐/業界標準」等諂媚框架；本規則禁止「估時過長」作為諂媚用戶的捷徑。兩者同為 receiver-end 前提查驗的延伸。

## 多視角審查：ginger 成本量化視角

### 審查結論

### 一、Token 淨效益宣稱驗證

**Solution 宣稱**：降級省 200k/Wave、驗收增 60k/Wave、淨省 140k/Wave。

**ginger 驗證**：

| 驗證項 | Solution 宣稱 | 實測結果 | 誤差 |
|-------|-------------|---------|------|
| 30 agent 總數 | 30 | 30（含 AGENT_PRELOAD 元文件） | 通過 |
| 高 model agent 數 | 17 | 17（opus 15 + claude-opus-4-6[1m] 2） | 通過 |
| 高 model 比例 | 57% | 58.6%（17/29 有效 agent） | 可接受 |
| sonnet/haiku 比例 | 17%（「僅 5 個」） | 38%（11/29：sonnet 8 + haiku 3） | 嚴重錯誤 |
| 降級後 opus 比例 | 33%（10/30） | 33%（10/30） | 通過 |

**Why 誤差 1 嚴重**：linux 在 Problem Analysis 稱「僅 5 個為 sonnet/haiku」，實際為 11 個。此錯誤不影響降級清單正確性（分類清單獨立核對），但會誤導後續決策者對「有多少空間可降級」的判斷（真實空間比宣稱的小，因為已有 11 個在低 model）。

**Token 節省數學推導**：

假設 Wave 派發量：15 ticket/Wave（W17 實測規模），每 ticket 平均呼叫 1 個主 agent + 0.3 個副審查 agent = 約 20 次 agent 呼叫/Wave。

以 Claude 官方 current-gen pricing（USD/1M tokens）推算 per-call 成本：
- opus 4.6：input $15、output $75（1M context 變體約 1.5 倍加成）
- sonnet 4.6：input $3、output $15
- haiku 4.6：input $1、output $5（估計值，官方未明示當代 haiku）

降級 7 個 opus→sonnet（thyme-python、parsley、fennel、thyme-extension、cinnamon、lavender、star-anise），每次 agent 呼叫約 input 15k + output 8k：
- opus 成本：15k × $15 + 8k × $75 = $0.225 + $0.6 = $0.825/call
- sonnet 成本：15k × $3 + 8k × $15 = $0.045 + $0.12 = $0.165/call
- 每次降級節省：$0.66/call

若這 7 個被降級 agent 在 Wave 中平均被呼叫 3 次/Wave = 21 次呼叫：
- 降級節省：21 × $0.66 = $13.86/Wave
- token 節省（output 換算）：21 × 8k × 0.8 = 134k tokens/Wave

再加 1 個 sonnet→haiku（thyme-documentation-integrator）：節省約 10k/Wave。

**實際降級節省**：約 140k tokens/Wave（Solution 宣稱 200k 偏高 30%，但量級正確）。

**Layer 1+2 驗收增量**：
- Layer 1（所有 ticket）：15 × (2k+1k) = 45k/Wave
- Layer 2（60% 觸發）：9 × (3k+2k) = 45k/Wave
- 合計 90k/Wave（Solution 宣稱 60k 偏低 50%）

**修正後淨效益**：140k 節省 − 90k 增量 = 50k tokens/Wave（Solution 宣稱 140k 偏高 180%）。

**為何 Solution 估算偏差大**：
1. 未考慮 Layer 2 委員本身需跨多個 ticket 做交叉比對（input token 膨脹）
2. 未計入 Layer 1 自檢的 output token（修正清單本身消耗 output）
3. 降級節省高估——實際 opus→sonnet 在短 prompt 場景差距縮小，長 prompt 才顯著

## 決策評估

**核心決策**: 值得做（三項改革同時落地，但需分階段導入以控制破壞半徑）

**理由**:
- 問題真實存在：30 個代理人中 17 個掛 opus/bay 等高 model，實際任務多為機械執行或文字處理，Token 浪費與推理延遲並存
- 估時禁令本質上是規則 5（權力不對等）在專案估時場景的具體化，缺這條規則讓「時間壓力」成為跳過品質的免死金牌
- 多輪驗收哲學與 document-writing-style v1.2.0「二次審查強制執行」原則同源，是正確延伸
- 破壞風險可控：model 屬 frontmatter 獨立欄位，改動不影響 prompt 語意，可隨時回滾

## Problem Analysis（根因）

### 因果鏈

**症狀 1：Token 浪費**
- 30 個代理人 → 29 個有 model 宣告 → 僅 5 個為 sonnet/haiku（17%），其餘 24 個為 opus 或 bay-quality-auditor 的 claude-opus-4-6[1m]（83%）
- 多數代理人任務為機械執行（格式化、樣板套用、固定流程實作），不需深度推理
- 根因：agent 設計時預設「拉到最高 model 就不會錯」的直覺式偏見，違反 SRP 精神（任務越單純越不需高階推理）

**症狀 2：估時驅動決策**
- 觀察：PM 與代理人出現「預估要花太久所以跳過/簡化」的決策模式
- 根因：缺乏明文禁令 → 估時自然成為最容易量化的判斷依據 → 遮蔽「是否應該做」的本質判斷
- 結構問題：這是規則 5 §5.6 機制 4「反諂媚設計」的變體。估時本身不是壞事，但把估時當決策依據會讓代理人選擇討好（快的路徑）而非正確

**症狀 3：追求一次完美**
- 觀察：代理人嘗試在單次 prompt 內完成「理解 + 設計 + 實作 + 驗證」全部環節
- 根因：model 高 + Token 多 + 無明確分輪 → 代理人傾向 all-in-one → 出錯時整包重做
- 對比：document-writing-style v1.2.0 已驗證「二次審查比強迫首次完美成本更低」，但該原則只落地在文件撰寫，未擴展到代理人工作流

## 關鍵發現

### 資料結構
- **現狀**: agent frontmatter 的 model 欄位是扁平字串（opus/sonnet/haiku/claude-opus-4-6[1m]），無任務屬性 tag
- **問題**: 無法機械判斷某 agent 是否該用高 model；未來若要動態分配 model，需補 task_type 欄位
- **建議**: 本次 ANA 不動資料結構（破壞太大），由 spawned IMP ticket 處理；降級先依人工分類清單執行

### 複雜度分析
- **本質複雜度**: 代理人成本優化是三維問題（model × prompt 長度 × 輪次）
- **實作複雜度**: 本 ANA 方案只動 model 維度（最易驗證），prompt 長度另由 W17-049 處理，輪次另由 W17-050 處理
- **評估**: 複雜度匹配，三張 ANA 正交拆分是 good taste 設計

### 特例情況
- claude-opus-4-6[1m]：兩個 agent 用此變體（saffron、bay），因需 1M context 處理大範圍審查，不降級
- rosemary-project-manager：用 inherit，特殊值表示跟隨主 session model，不動
- 三個 DEPRECATED agent（john-carmack、memory-network-builder）：不動，避免破壞向後相容
- language-agent-template：範本檔，不動

### 風險點
- **向後相容性**: model 降級不影響 prompt 語意，風險低；但 sonnet 對長 prompt 的指令遵守度可能低於 opus，需監測
- **依賴影響**: 無直接依賴破壞；間接影響是降級的 agent 若實際任務比分類估計複雜，可能產出品質下降
- **緩解措施**: 降級後一個 Wave 內監測失敗率，若某 agent 失敗率上升 >20% 則回滾至 opus

## 成長機會

### 從本 ANA 學到的原則
1. **Model 選擇是 SRP 的延伸**：一個 agent 只做一件事 → 一件事需要的推理深度決定 model → 拉滿 model 是認知偏見而非最佳策略
2. **估時禁令是權力不對等的變體**：用戶/PM 用估時壓代理人 = 用資訊不對稱逼代理人走捷徑；規則 5 的精神應擴展到專案估時場景
3. **多輪驗收是摩擦力方法論的時間維度應用**：原本摩擦力只談單一動作的強度（象限 A/B/C），多輪驗收引入「時間重複」維度——同樣摩擦力分多輪施加比一次施加更有效

### 可沉澱為 memory 的教訓
- memory 候選：AI 輔助開發中，model 高不等於品質好；SRP 精神要求 model 配合任務而非任務配合 model
- memory 候選：估時作決策依據是權力不對等反模式，與「業界最佳實踐」同構
