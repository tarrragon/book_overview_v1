## 分析結論

### 1. set-acceptance vs check-acceptance 語意對照表

| 特性 | check-acceptance | set-acceptance |
|------|-----------------|-----------------|
| **語意** | 切換驗收條件狀態（模糊） | 顯式設置狀態（明確） |
| **單一操作** | `index` 或 `--all [--uncheck]` | `--check/--uncheck/--all-check/--all-uncheck` |
| **多個 index** | 不支援單次多 index（需拆呼叫） | 支援單次多 index：`--check 1 2 3` |
| **語義清晰度** | 高（toggle 動作隱含） | 最高（目標狀態顯式） |
| **UI 介面風格** | Unix toggle 模式（如 chmod u+x） | 聲明式 API 風格（如 Kubernetes --image）|
| **使用場景推薦** | 已廢棄，保留為向後相容 | W14-030 新推薦標準 |

**根本差異**：
- `check-acceptance <id> 1 --uncheck` → "對 index 1 進行 toggle"（動作驅動）
- `set-acceptance <id> --uncheck 1` → "將 index 1 設置為取消勾選"（狀態驅動）

---

### 2. 適用場景清單（≥3 例）

#### 場景 1：新建 Ticket 初始化驗收條件（推薦 set-acceptance）
```bash
# 場景：Ticket 剛建立，4 個驗收條件都未勾選（空白狀態）
# 目標：快速將所有條件改為待驗收（勾選全部）

# 推薦（set-acceptance）
ticket track set-acceptance 0.18.0-W17-008.8 --all-check

# 不推薦（check-acceptance）
ticket track check-acceptance 0.18.0-W17-008.8 --all
# 原因：--all 語意不清，使用者需從上下文推論「勾選」vs「取消勾選」
```

**決策依據**：
- 新建後的 Ticket 通常驗收條件都是空白，需要全量初始化
- set-acceptance --all-check 語意明確，不會被誤讀為取消勾選全部
- check-acceptance --all 缺乏方向性，容易誤操作

---

#### 場景 2：修改既有 Ticket 的部分驗收條件（set-acceptance 優勢明顯）
```bash
# 場景：正在執行 Ticket，3 個條件已完成，需勾選；1 個條件先跳過，保持未勾選
# 目標：單次命令勾選 index 1, 2, 3（多個）

# 推薦（set-acceptance）
ticket track set-acceptance 0.18.0-W17-008.8 --check 1 2 3

# 不推薦（check-acceptance）— 需要 3 次呼叫
ticket track check-acceptance 0.18.0-W17-008.8 1
ticket track check-acceptance 0.18.0-W17-008.8 2
ticket track check-acceptance 0.18.0-W17-008.8 3
# 原因：N 倍 token 消耗；多次 I/O；風格與 `rm file1 file2 file3` 違和
```

**決策依據**：
- Unix 哲學：如果操作可並行，應支援批量操作（如 `cp src1 src2 dest/`）
- set-acceptance 支援多 index，單次完成，對應 "Atomic update" 原則
- check-acceptance 每次只能單一 index，是設計限制，不應引導使用者採用

---

#### 場景 3：Ticket 實行中發現部分條件無法達成，需取消勾選（set-acceptance 最清晰）
```bash
# 場景：中途發現驗收條件 2 的需求被 blockedBy，暫時取消勾選
# 目標：解除某個已勾選的驗收條件

# 推薦（set-acceptance）
ticket track set-acceptance 0.18.0-W17-008.8 --uncheck 2

# 可接受（check-acceptance）
ticket track check-acceptance 0.18.0-W17-008.8 2 --uncheck
# 原因：set-acceptance --uncheck 更符合「取消」的語意直覺
# check-acceptance 需額外 --uncheck flag 才能表達反向操作
```

**決策依據**：
- set-acceptance --uncheck 不需要額外 flag，目標狀態寫在 --uncheck 後
- check-acceptance 的 toggle 設計在此場景下增加認知負擔

---

#### 場景 4：驗證 Ticket 的所有驗收條件都已檢查並完成（推薦 validate）
```bash
# 場景：接手他人的 Ticket，檢查驗收條件是否完整填寫
# 目標：驗證 4 關鍵欄位（status/completed_at/acceptance/who）合規

# 使用 validate
ticket track validate 0.18.0-W17-008.8

# 輸出例：
# [INFO] 0.18.0-W17-008.8 frontmatter 合規性檢查
# ✓ status: completed
# ✓ completed_at: 2026-04-20
# ✗ acceptance: 驗收條件 3 未勾選
# ✓ who: rosemary
# 建議：ticket track set-acceptance 0.18.0-W17-008.8 --check 3
```

**決策依據**：
- validate 是檢查型命令，不修改狀態，與 check-acceptance/set-acceptance 職責分離
- 提供修復建議，對應 "helpful error messages" 原則（類似 Linux 的 `make` error 提示）

## set-acceptance vs check-acceptance 決策指引

### 何時用 set-acceptance（推薦標準，W14-030）

使用 **`set-acceptance`** 適用於以下所有場景：

1. **單一勾選**（新建或中途補勾）
   ```bash
   ticket track set-acceptance <id> --check 1
   ```

2. **多個勾選（單次）** ⭐ set-acceptance 專優
   ```bash
   ticket track set-acceptance <id> --check 1 2 3
   # 不應分多次：ticket track check-acceptance <id> 1 2 3 （違反 Unix 批量慣例）
   ```

3. **取消勾選**
   ```bash
   ticket track set-acceptance <id> --uncheck 2
   ticket track set-acceptance <id> --uncheck 2 4 6    # 支援多個
   ```

4. **全部勾選**
   ```bash
   ticket track set-acceptance <id> --all-check
   ```

5. **全部取消勾選**
   ```bash
   ticket track set-acceptance <id> --all-uncheck
   ```

### 何時用 check-acceptance（向後相容，新代碼禁用）

- **僅**在遺留 Ticket 上使用（涉及舊文件或 hook 依賴）
- 新 Ticket 和所有新手指引應指向 `set-acceptance`

| **多 index 支援** | ❌ 需拆呼叫 | ✓ --check 1 2 3 |
| **flag 風格** | toggle: --all [--uncheck] | 聲明式: --all-check/--all-uncheck |
| **推薦用於新代碼** | ❌ | ✓ W14-030 定標 |

- `validate <id>`：檢查 frontmatter 4 欄位合規，不修改狀態（W14-030）

### 補充：根因分析

**為什麼新手易誤用？**

1. **語言模糊性**：「check」既可解釋為「檢查」（動詞），也可解釋為「勾選」（UI 動作）
   - check-acceptance 的 --all --uncheck 組合在英文語法上歧義（是「檢查全部」還是「取消勾選全部」？）
   - set-acceptance 的 --all-check 消除歧義（明確 = 設置為已勾選狀態）

2. **设计不对称**：
   - check-acceptance 不支援多 index，違反 Unix 慣例（`rm a b c` 可批量）
   - 導致使用者遇到「多條件修改」時必須拆多次呼叫，增加 token 成本

3. **文件滯後**：
   - W14-030 引入 set-acceptance 後，track-command.md 第 76-80 行改為推薦，但未說明「何時用哪個」
   - 新手無決策樹，猜測時容易選錯

**修復效果**：
- 補完此決策樹後，新手的認知負擔從「理解兩個命令」降至「查表選命令」
- 對應 Linux man page 的職責：提供決策樹，而不只是語法
