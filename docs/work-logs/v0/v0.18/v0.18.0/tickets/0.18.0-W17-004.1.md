## Linux POSIX 介面設計成功模式清單

本清單作為 W17-004.2（CLI 審計）與 W17-004.3（深度分析）之量尺基準。

### 1. Man Page 結構規範
**參考**: man(7) - Linux man-pages project standard structure (NAME, SYNOPSIS, DESCRIPTION, OPTIONS, RETURN VALUE, ERRORS, EXAMPLES, SEE ALSO)
**POSIX**: IEEE Std 1003.1-2017, 2.3 "Documentation Practices"
**為何 good taste**: 統一的文件結構讓開發者快速定位關鍵資訊，降低學習曲線，確保新手與專家都能高效查閱——是 Unix 哲學「易學難精」的實踐。

### 2. errno 全域一致性
**參考**: errno(3), perror(3), strerror(3); Linux man-pages errno(3) 詳述所有標準 errno 值的語義 (ENOENT, EACCES, EAGAIN, etc.)
**POSIX**: IEEE Std 1003.1-2017, 2.3 "Error Handling" & <errno.h> definition; POSIX.1-2008 Chapter 2.3 列舉所有標準 errno 值與其含義
**為何 good taste**: 全域 errno 常數讓所有系統呼叫與庫函式共用同一套錯誤語言，避免重複定義，降低 API 學習成本——是可組合性的基礎。

### 3. POSIX 命名慣例
**參考**: man(3) - libc function naming (getuid, setpgid, fork, execve 等遵循動詞-目標物模式); POSIX File System 命名 (/dev/null, /proc/self/fd, etc.)
**POSIX**: IEEE Std 1003.1-2017, 2.2 "Terminology" & Function Naming Guidelines — 動詞+名詞+修飾符的前綴模式 (get-, set-, is-, has-, mk-, rm-)
**為何 good taste**: 可預測的命名讓開發者憑直覺猜測函式行為（如 getpid vs setpid），減少查文件的頻率，提高 API 自說明性。

### 4. 無歧義原則
**參考**: open(2) - 清楚區分 O_RDONLY, O_WRONLY, O_RDWR flags，無容易混淆之互斥項; chmod(2) - mode bits 清晰定義 (S_IRUSR, S_IWUSR, etc.)
**POSIX**: IEEE Std 1003.1-2017, 2.3 "Interface Specification" — 每個操作應有單一明確的語義，避免上下文相依性與多重解釋
**為何 good taste**: 消除歧義讓非同步環境中的行為可預測，減少邊界條件 bug，是多執行緒/多程序系統可靠性的基礎。

### 5. Syscall 行為契約
**參考**: write(2), read(2), select(2) - 清楚說明部分完成的情況 (short read/write)、信號中斷 (EINTR)、阻塞與非阻塞模式的確切行為
**POSIX**: IEEE Std 1003.1-2017, 2.4.3 "Signal Concepts" & 2.8.2 "File Descriptor I/O" — 定義系統呼叫在中斷、超時、部分完成時的行為
**為何 good taste**: 明確的行為契約讓開發者能可靠地撰寫健壯的系統程式，避免在 race condition 或信號中斷時的未定義行為。

### 6. /proc 透明機制
**參考**: proc(5) - Linux /proc 虛擬檔案系統（/proc/self, /proc/[pid]/fd, /proc/[pid]/maps, /proc/meminfo 等）提供非破壞性檢視內核狀態
**POSIX**: IEEE Std 1003.1-2017 未正式包含 /proc（Linux 特定），但遵循 POSIX File I/O 原則——一切皆檔案，讀取如同讀普通檔案
**為何 good taste**: 不需特殊系統呼叫即可檢視程序狀態，避免專有工具或內核接口，符合 Unix 哲學「資訊應以文本形式公開」。

### 7. 錯誤回傳一致性
**參考**: return value conventions (man(2), man(3)) — 成功返回 0 或正值，失敗返回 -1（syscall）或 NULL（庫函式）且設定 errno
**POSIX**: IEEE Std 1003.1-2017, 2.3 "Return Values" — 標準化的回傳模式讓呼叫端統一的錯誤檢查方式 (if (write(...) == -1) { perror(...) })
**為何 good taste**: 一致的失敗訊號讓錯誤處理邏輯可複用、可自動化，使語言綁定與包裝層易於實現正確的異常轉換。

### 8. 介面少量但正交
**參考**: dup(2), dup2(2) vs dup3(2) - 基礎函式 (dup) + 增強版本 (dup2) 設計原則; open(2) 與 openat(2) 的正交擴展; fork(2) 與 vfork(2) 的最小分化
**POSIX**: IEEE Std 1003.1-2017, 2.1 "General Concepts" — 最小化函式數量，通過旗標與參數組合達成功能擴展，避免 N×M 的爆炸
**為何 good taste**: 正交的小介面易於學習、文件化、測試與維護，也易於在新場景中組合使用——是 Unix 哲學「Do one thing and do it well」的最高體現。
