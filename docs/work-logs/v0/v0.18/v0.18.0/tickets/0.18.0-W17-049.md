### 問題現象

TDD Phase 3b 派發任務過大事件的三個實證訊號：

1. 代理人認知負擔過載（回合耗盡、context 超標、無法一次完成原計畫工作）
2. PM 派發 prompt 撞 agent-prompt-length-guard Hook 30 行硬上限（W17-046/W17-047/W17-048 實證）
3. 複雜任務的 Context Bundle 膨脹到單一子任務 > 20K tokens（phase3b-prompt-template 預算上限）

**W17-048 是典型案例**：原本應拆分為「auto-load 引用（認知層）+ Hook 軟提示（工具層）+ 模板骨架補強（資產層）」三張子 ticket 的任務，最初被視為單一派發項，撞 Hook 上限後才被迫拆分。該案例暴露「3b 派發前缺乏強制的拆分檢查點」。

### 問題根因（5W1H 因果鏈）

根因可定位在「3b 派發的決策路徑缺少一道強制門」。

| 層級 | 根因 | 支撐證據 |
|------|------|---------|
| **規範層** | task-splitting.md 列出拆分觸發條件但無「3b 派發前強制執行」錨點 | tdd-flow.md 3b 拆分評估區段雖強制，但只要求評估「是否拆分」，未要求在 prompt 產生前比對 context 預算 |
| **工具層** | agent-prompt-length-guard 只檢 prompt 本體行數（30 行硬上限 + 10 行軟提示），不檢 ticket body / Context Bundle 體量 | Hook 主旨是偵測 prompt 膨脹，而非偵測任務過大；兩者不同維度 |
| **認知層** | PM 在 Phase 3a→3b 轉換時，認知焦點在代理人能不能做完，而非 SRP 代理人能不能做好 | task-splitting.md v5.0.0 已宣告拆分不是為了做完，而是為了做好，但 3b 派發前沒有強制檢查清單讓此原則落地 |
| **資產層** | cognitive-load.md 有拆分閾值，但閾值欄位散佈在三份文件（cognitive-load / task-splitting / thresholds），PM 派發時需跨檔比對 | PM 派發摩擦力高，傾向略過比對直接派發 |

**根因摘要**：3b 派發決策路徑上缺一個整合性檢查門（check-gate），該門應：(a) 位於 PM 寫 prompt 之前；(b) 對比 ticket 的 context 體量與拆分閾值；(c) 與既有 agent-prompt-length-guard Hook 分工（前者管任務大小、後者管 prompt 大小）。

- 摩擦力方法論：設計與評估時增加摩擦力，執行時降低摩擦力，送出前做檢查

### 多視角審查：ginger 視角

審查角度：認知負擔指數（指標 = 變數+分支+巢狀+依賴）+ PM 派發前實際執行成本 + 工作流摩擦力。

#### 衝突點 1：8 項閾值 vs linux 提議 3 項

**結論：同意 linux 砍至 3 項，建議保留組合為「功能職責數 / 修改檔案數 / Context Bundle tokens」。**

PM 派發前執行成本量化對比（每次 3b 派發都要跑一次）：

| 指標 | 量測動作 | 認知負擔指數 | 是否真能降低回合耗盡 |

8 項全跑累計約 25 認知負擔點，每次派發 PM 多扛 25 點 → 必然略過。3 項約 5 點，可實際執行。**真正能降低代理人回合耗盡的只有 #2/#3/#6**。

#### 衝突點 2：兩階段 Hook 擴充 vs CLI 方案

**結論：同意 linux 反論採 CLI 為主，但 Hook 階段 2 不應完全刪除—建議「CLI 主動 + Hook 兜底軟提示」雙層。**

維度對比：

| 維度 | 兩階段 Hook | CLI 工具 | 雙層方案 |
|------|------------|---------|---------|
| 派發延遲 | +50-100ms（每次派發強制執行） | 0（PM 主動） | +20ms（Hook 只讀 metadata 不算 token） |
| 可觀測性 | 高（每次都跑） | 低（漏跑無紀錄） | 高 |
| PM 摩擦力 | 中（強制但被動） | 低（主動但易忘） | 低 |
| 失敗模式 | 軟提示疲勞 → 忽略 | 直接跳過 | 兩道訊號互補 |

linux 主張「真正檢查點在 PM 寫 prompt 之前」正確；但 CLI 為唯一入口的失敗模式是 PM 趕時間直接跳過。建議：CLI 為主動檢查（PM 工作流），Hook 階段 2 退化為**僅檢查 #3 where.files 數**（metadata 純讀取，零 token 成本，零派發延遲），作為 CLI 漏跑的兜底。

#### 衝突點 3：4 張 IMP vs 砍至 2 張

**結論：採 linux 結構但補充——W17-052/053 合併為 IMP-A，W17-055 升必做，W17-054 不刪除改縮範圍。**

| 既建 Ticket | 建議處置 | 理由 |
|------------|---------|------|
| W17-052（task-splitting 檢查清單） | 保留，內容改為 3 項閾值 | 核心規範 |
| W17-053（tdd-flow 錨點） | 合併入 W17-052 | 修 1-2 行不值獨立 ticket（linux 正確） |
| W17-054（Hook 擴充） | 保留但縮範圍：僅 where.files 計數軟提示 | 兜底機制；零成本實作 |
| W17-055（CLI Skill）| 升為必做 | CLI 是主檢查點 |

撤銷 W17-053 並併入 W17-052；W17-054 縮減 scope 不撤銷。

#### Final 建議：第三方案

採 linux 骨幹（3 閾值 + CLI 主檢查 + IMP 精簡）+ ginger 補強（Hook 退化為零成本 metadata 兜底）：

1. **3 項閾值**：功能職責數 / 修改檔案數 / Context Bundle tokens（linux 原案）
2. **檢查機制**：CLI 主動（PM 派發前）+ Hook 階段 2 僅檢 where.files 數（兜底軟提示）
3. **Ticket 結構**：W17-052（合併 053，3 項閾值 + tdd-flow 錨點）/ W17-054（縮範圍 metadata 兜底）/ W17-055（CLI 必做）；W17-053 撤銷

理由：linux 砍到骨幹方向正確（PM 派發前認知負擔必須極低），但「單一檢查點」失敗模式是漏跑無紀錄。Hook 退化方案保留可觀測性零成本，符合摩擦力方法論「執行階段降摩擦 + 送出前做檢查」。

### Q1：8 項拆分閾值是否過度設計？

**結論：嚴重過度設計。8 項砍到 3 項即可。**

逐項評估：

| # | 指標 | 建議 | 理由 |
|---|------|------|------|
| 1 | 測試群組數 > 3 | 刪除 | 測試群組是實作細節，不是任務大小訊號；群組可多可少跟 SRP 無必然關係 |
| 2 | 功能職責數 > 1 | 保留 | **SRP 核心指標，這是唯一真正必要的判斷** |
| 3 | 修改檔案數 > 5 | 保留 | 客觀、可量測、與認知負擔強相關；這是「夠用的 proxy」 |
| 4 | 跨架構層級數 > 2 | 刪除 | 高度重複於 #3；若 5 檔跨 3 層，#3 已觸發；若 2 檔跨 3 層，案例極少 |
| 5 | 認知負擔指數 > 10 | 刪除 | 需要手算（變數數+分支數+巢狀深度+依賴數），PM 派發前根本不會算；「可判別」是幻覺 |
| 6 | Context Bundle tokens > 20K | 保留 | 唯一有硬性 budget 上限的指標（agent 真的會爆），必須留 |
| 7 | Subagent tool call > 15 | 刪除 | 與 #3 修改檔案數高度相關；Phase 3a 策略中數動作是主觀判讀 |
| 8 | 讀取總行數 > 1000 | 刪除 | 讀取行數是「agent 讀多少」而非「任務多大」；context budget（#6）已涵蓋本質 |

**保留三項**：功能職責數（SRP 本質）、修改檔案數（客觀 proxy）、Context Bundle tokens（硬性 budget）。

Linus 原則對照："If you need more than 3 levels of indentation, you're screwed"——同理，如果需要 8 項閾值才能判斷任務大不大，表示問題本身沒想清楚。好的設計應該一眼看懂，不需 8 項交叉比對。

### Q2：兩階段 Hook 擴充是否違反 SRP？

**結論：策略 A（擴充現有 Hook）違反 SRP，但策略 B 的理由站不住。應採取第三條路——維持現 Hook + CLI 工具。**

問題分析：
- 現 Hook 名為 agent-prompt-length-guard，語意邊界是「prompt 行數」
- 擴充為兩階段後名不副實，實際做兩件事（prompt 大小 + 任務大小）
- 「避免 Hook 數量膨脹」是假理由——80+ hooks 不是問題，每個單一職責才是品質

**但**新建獨立 Hook 也是過度設計。理由：
- 任務大小檢查**不應該是 Hook**（攔截點）
- 派發時才檢查已經太晚（Context Bundle 早已寫好）
- 真正的檢查點在「PM 寫 prompt 之前」不是「PM 送出派發時」

**Good Taste 方案**：
- 現 Hook 維持原職責不動（規則六：失敗案例學習——現 Hook 已驗證穩定，不亂改）

- PM 派發時 Hook 不再檢查任務大小（避免重複防護造成 PM 邊際疲勞）

差異對比：原設計 Hook 軟提示 + CLI 輔助（雙層防護），改為**單一檢查點**（CLI 作為唯一入口），符合「一件事做好」。

## 產出項 2：Hook / CLI 檢查點落點

### 設計目標

與既有 `agent-prompt-length-guard-hook.py` 分工銜接，前者管「任務大小」、後者管「prompt 大小」。兩者都在 PreToolUse 階段執行，但量測不同維度，避免耦合。

### Hook 架構設計

**策略 A（推薦）：擴充現有 Hook 為兩階段檢查**

於 `agent-prompt-length-guard-hook.py` 新增第二階段檢查：

| 階段 | 量測對象 | 閾值 | 退出碼 | 訊息 |
|------|---------|------|--------|------|
| 階段 1（既有） | prompt 行數 | > 30 行 | 2（阻擋） | 既有硬上限訊息 |
| 階段 1（既有） | prompt 行數 | 10-30 行且缺模板關鍵字 | 0（軟提示） | 既有軟提示訊息 |
| 階段 2（新增） | ticket body Context Bundle tokens | > 20K | 0（軟提示，不阻擋） | 「任務可能過大，建議拆分為多個子 ticket（對照 task-splitting.md 8 項閾值）」 |
| 階段 2（新增） | ticket where.files 數 | > 5 個 | 0（軟提示） | 「修改檔案 N 個超過 5 個建議閾值，請確認 SRP 或拆分」 |

**為何階段 2 只軟提示不阻擋**：
- 不拆分並非絕對錯誤（可能是合理的 group coordinator ticket）
