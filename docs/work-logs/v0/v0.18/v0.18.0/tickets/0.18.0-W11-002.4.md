## Problem Analysis

## 函式對照表

兩個 predicate 同為 AND 語義（所有 blocker 已解除），差異點：

| 條件 | _check_unblocked_tickets (line 947-950) | _can_cascade_unblock (line 1246-1256) |
|------|-------------------------------------------|------------------------------------------|
| 已解除 status | completed only | completed 或 closed |
| blockedBy 為空 | all([]) → True | not blocked_by → True (early return) |
| blocker 找不到 | {}.get("status") = None ≠ "completed" → False | 顯式 return False（保守保留 blocked） |
| 邏輯類型 | AND | AND |

## 結論

語義非完全等價，差異點為「closed 是否視為已解除」：
- cascade unblock（lifecycle skip 規則）已將 closed 視為 terminal，須含 closed
- 建議列表場景保留原行為，僅認 completed

兩處對「找不到 blocker」皆採保守 False，可合併。

## 設計決策

抽出 _is_fully_unblocked(ticket, ticket_map, *, include_closed_as_resolved: bool)，差異透過 keyword-only 參數顯性揭露：

- _check_unblocked_tickets 使用 include_closed_as_resolved=False（保留原 conservative 行為）
- _can_cascade_unblock 使用 include_closed_as_resolved=True（與既有 cascade 規則一致）

行為不變（純重構）。
