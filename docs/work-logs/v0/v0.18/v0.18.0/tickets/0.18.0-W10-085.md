## 結論：where.files 標記規則

### 規則 1：列「實質修改會發生的位置」（選項 D 優先）

| 修改類型 | where.files 列法 |
|---------|-----------------|
| 骨架索引變更（版本號、索引表、導航） | 只列骨架 |
| 實質內容擴充（新規則、修改現有規則細節） | 列 references/（不強制列骨架，除非需同步版本號） |
| 同步變更（索引+實質都改） | 兩者都列 |

**判別問題**：PM 撰寫 where.files 時自問：「此 ticket 的修改本質是改變規則『入口索引』還是『規則本身的內容』？」
- 改索引 → 骨架
- 改內容 → references
- 同步 → 兩者

### 規則 2：PM 撰寫 where.files 前的拆分偵測

PM 列 where.files 時，若看到路徑包含 `rules/core/` 或 `pm-rules/`，必須檢查是否存在對應 `references/X.md` 或 `references/X-details.md`：

```bash
# 偵測 pairing 的快速命令（建議 CLI 整合）
for path in $(echo "$where_files" | grep -E "(rules/core|pm-rules)/"); do
  basename=$(basename "$path" .md)
  find .claude/references -name "${basename}*.md"
done
```

### 規則 3：Agent 延伸 where.files 外檔的行為規範

agent 若發現實際修改必須延伸到 where.files 未列的檔案（例如骨架-references 拆分配對的另一邊）：

| 情境 | 允許行為 |
|------|---------|
| 延伸符合 ticket 意圖（規則實質內容落在 references/） | **允許延伸**，必須 append-log 記錄：「延伸至 X.md 因 [理由]」 |
| 延伸超出 ticket 意圖（新增無關模組） | **禁止延伸**，應停止並回報 PM |

**預設禁止默默擴展未記錄**。

### 規則 4：acceptance-auditor 擴充檢查項

acceptance-auditor 檢查 where.files 完整性時，新增：
- 若 where.files 含 `rules/core/X.md` 或 `pm-rules/X.md`，掃描 `references/X.md` 或 `references/X-details.md`
- 若存在對應 references/ 但未列於 where.files：WARN「骨架拆分對未列出，確認修改是否同步 references/」
