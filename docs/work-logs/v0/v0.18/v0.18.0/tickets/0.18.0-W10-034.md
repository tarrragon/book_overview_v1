### 上層銜接

本分析經用戶 2026-04-14 觀察，發現真正根因更大——**前期決策鏈（PROP→Phase 1）摩擦力倒置**，與摩擦力管理方法論「決策點增加摩擦、執行點降低摩擦」原則相反。W10-034 的 A+B 方案將作為 **W10-035 Meta ANA**（開發流程摩擦力配置倒置結構性分析）的「Phase 1 層實例」納入。W10-034 保留原 scope 結論完整，不擴張；結構性問題由上層 W10-035 處理。
## 分析結論

### 根因修正

原假設「/spec 未觸發導致缺口未發現」不完整。經 Reality Test 驗證，真根因為雙重問題：

1. **執行落差**：lavender agent definition 明文要求使用 /spec init 產生獨立 feature-spec.md（line 69/212/301），但 W5-002 Phase 1 實際將規格寫入 Ticket body 繞過工具，無機制強制驗證。
2. **偵測不足**：即使 lavender 100% 執行 /spec validate，現有 3 維度對本次 3 類缺口覆蓋率僅 30-50%，對「實作決策定位」類缺口無對應維度。

### 推薦方案：A + B 組合（MVP）

- **A（執行層）**：新增 feature-spec-existence-guard-hook，PM 派發 sage 前強制驗證 feature-spec.md 存在；agent acceptance 加自檢。
- **B（偵測層）**：/spec validate 擴充第 4 維度「實作決策定位」，並將本次 3 缺口納入範例。

A+B 獨立落地互不依賴，組合解決執行層 + 偵測層雙重問題，預估可將 3 類缺口偵測率從 30-50% 提升至 80-90%。
