## Problem Analysis

### 問題根因

subagent（如 thyme-documentation-integrator）具有 Edit 工具權限，可直接修改 ticket YAML frontmatter 的任意欄位。當 subagent 寫入非標準值時（如 `status: complete` 而非 `completed`、`completed_at: T24:00:00` 無效 ISO 8601、acceptance 多條擠在單行），既有機制無法阻擋：

1. **CLI 不是唯一入口**：`ticket track complete/set-status/check-acceptance` 等命令有驗證邏輯，但 subagent 可直接 Edit 檔案繞過
2. **Hook 不偵測 ticket 檔案修改**：既有 PreToolUse hooks 不檢查 Edit 目標是否為 ticket .md 及修改欄位是否合規
3. **Skill 規則純文字**：派發 prompt 要求「用 CLI」但無強制機制，依賴 subagent 紀律

核心問題：**ticket 系統的 Source of Truth 原則（CLI 為唯一合法路徑）缺乏執行層強制**，subagent 可繞過後 CLI 才發現瑕疵（如 `ticket track list` 顯示 [?]、check-acceptance --all 只勾第一項）。

### 影響範圍

| 層級 | 受影響內容 |
|------|----------|
| Ticket 系統 | ticket CLI 查詢結果不正確、complete 時 hook 警告「未找到驗收記錄」 |
| Hook 系統 | acceptance-gate-hook / complete 清單警告無法區分「subagent 瑕疵」vs「PM 真的遺漏」 |
| 代理人生態 | thyme/parsley 等實作型 subagent 若複製 W14-019.2 模式，瑕疵會擴散 |
| PM 工作流 | PM 需手動 Edit 修復 frontmatter（本 session 至少 2 次：W14-023 AC split / 歷史 019.2 status 修正）|

## WRAP 四階段分析

### W（Widen Options）

| 方案 | 描述 | 成本 | 覆蓋 | 誤報 | 選用 |
|------|------|------|------|------|------|
| A. PreToolUse Hook 硬阻擋 | Edit ticket frontmatter 特定欄位時直接阻擋 | 高 | 全欄位 | 中 | 候選（可選，非強制）|
| B. Skill/派發 prompt 警告 | 派發 prompt 明列各欄位 YAML 格式要求 | 低 | 依 prompt 涵蓋 | 低 | 是（本 session 已部分驗證） |
| C. CLI 強化 | 新增 `ticket track set-acceptance --check INDEX` 等明確子命令 | 中 | 提供正確路徑 | 低 | 是（根本解法） |
| **D**. PostToolUse 驗證 Hook | 事後偵測 ticket frontmatter 異常，寫 stderr 提醒 PM 修復 | 中 | 全欄位 | 低 | **是（主路徑）** |
| E. 組合 D+B+C | 事後驗證 + prompt 警告 + CLI 工具 | 中-高 | 最完整 | 低 | 候選 |
| F. 暫不處理 | 靠 PM 事後手工修復 | 零 | 零 | 零 | 否 |

### R（Reality Test）

| 假設 | 驗證狀態 | 證據 |
|------|---------|------|
| subagent 手寫 frontmatter 是常見模式 | **已驗證** | 本 session 3/5 案例觀察到（含歷史 W14-019.2）|
| prompt 警告可 100% 解決 | **已反駁** | 覆蓋率依 prompt 涵蓋欄位，未提的欄位仍有瑕疵風險 |
| PreToolUse 阻擋誤報高 | **部分驗證** | PM 本 session 做 Edit AC split 是「合理修復」，會被 A 方案誤擋 |

### A（Attain Distance）

**方案 D（PostToolUse 驗證）機會成本**：
- 直接成本：開發 ticket-frontmatter-validator-hook（~0.5 天）
- 延遲成本：現有瑕疵需 PM 繼續手工修復 1-2 個 session 週期
- **收益**：不擋合理修復、涵蓋全欄位、與既有 Hook 生態一致

**方案 A（PreToolUse 硬阻擋）機會成本**：
- 直接成本：開發 + 維護白名單（哪些修改合法）
- 風險：誤擋 PM 合理修復，PM 需 `--force` 繞過 → 養成繞過習慣反而更糟

**方案 C（CLI 強化）機會成本**：
- 直接成本：新增 3-5 個 CLI 子命令
- 風險：subagent 訓練慣性仍用 Edit，新命令未被使用

**結論**：方案 D 為主（防護網）、B 為輔（教育性）、C 為長期補強（正確路徑）。A 降級為可選。

### P（Prepare to be Wrong）

12 小時後方案 D 最可能失敗的原因：

1. **PostToolUse 警告被忽略**：PM 過忙未看到 stderr → 防護：警告需同時寫入日誌 + 下次 complete 時 gate 提醒
2. **驗證規則涵蓋不全**：新 frontmatter 欄位未加入檢查 → 防護：規則以白名單模式（明列檢查哪些欄位），未列者不檢查，擴充門檻低
3. **subagent 學習繞過模式**：偵測到警告後改用其他繞過方式 → 防護：方案 B（prompt 警告）並行，建立正確行為範式

## 結論：採組合方案 E（D+B+C）分階段

### Phase A（立即，spawned IMP）

**IMP-1：PostToolUse 驗證 Hook**（主路徑）
- 新建 `ticket-frontmatter-validator-hook.py`
- 偵測目標：Edit/Write 對 `docs/work-logs/**/tickets/*.md` 的 YAML frontmatter
- 檢查欄位：
  - `status` ∈ {pending, in_progress, completed, blocked, closed}
  - `completed_at` 符合 ISO 8601（`YYYY-MM-DDTHH:MM:SS`）
  - `acceptance` 為 list 型別且每項為獨立字串（不含 `,[ ]` 或 `,[x]` 這類合併模式）
  - `who` 為 object 含 `current` 欄位（非 plain string，PC 已知模式）
- 違規時：stderr 提醒 + 寫入 hook-logs + 不阻擋（事後修復）

**IMP-2：派發 prompt 模板強化**（輔助）
- 更新 thyme/parsley/fennel 代理人 frontmatter 或 .claude/pm-rules/context-bundle-spec.md
- 明列 frontmatter 欄位 YAML 格式要求
- 本 session 已驗證 acceptance 欄位 100% 改善率

### Phase B（後續版本，視 Phase A 效果）

**IMP-3：CLI 子命令強化**（長期補強）
- 新增 `ticket track set-acceptance --index N --check/uncheck`
- 新增 `ticket track validate <ticket-id>`（驗證 frontmatter 合規）
- 提供 subagent 正確路徑

### 否決方向

| 否決項目 | 理由 |
|---------|------|
| 方案 A 立即實施 | 誤報率高、會擋 PM 合理修復、可能養成 --force 繞過習慣 |
| 方案 F 暫不處理 | 本 session 實測 3+ 案例顯示瑕疵真實存在且影響工作流 |
| 只做 B 不做 D | prompt 警告無法涵蓋所有欄位，系統防護仍必要 |
