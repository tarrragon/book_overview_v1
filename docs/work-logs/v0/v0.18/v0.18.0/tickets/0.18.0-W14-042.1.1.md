## 問題

`_force_fork_mode` fixture 使用 `try/except RuntimeError: pass` 吞掉 `mp.set_start_method('fork', force=True)` 失敗訊號。在 macOS Python 3.13+ 預設 spawn 模式下，若 multiprocessing 已被 init 過、且當前模式不是 fork，except 會靜默通過，後續測試於 spawn 模式執行。

## 後果

spawn 模式下 child process 不繼承主 process 的 monkeypatch（`ticket_builder.get_ticket_path` / `load_ticket`），worker 會 import 真實模組、操作真實 ticket dir 或直接 import 失敗，使 race regression 測試的 lost-update 偵測失效，產生 false GREEN（看似通過但實際上未驗證 race scenario）。

## 根因

「靜默 except」遮蔽了 fork mode 不可用這個關鍵前提條件。

### 修復摘要

在 `_force_fork_mode` fixture 末尾追加 `assert mp.get_start_method() == 'fork'`，附明確失敗訊息說明 spawn/forkserver 會造成 false GREEN 的具體機制（monkeypatch 不傳播至 child process）。

### 行為變化

- set_start_method 拋 RuntimeError 但當前已是 fork：assert 通過（正常情況，set 過一次的 idempotent 情境）
- set_start_method 拋 RuntimeError 且當前非 fork：assert 失敗，明確指出無法切到 fork mode
- set_start_method 成功：assert 通過

### 不在本 ticket 範圍

- race regression 的 GREEN 實作（lock 注入）由父 ticket / 後續子任務處理
- 本任務僅關閉 false GREEN 漏洞，不改變 race detection 邏輯本身
