## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

三個獨立根因造成用戶觀察到的噪音：

**根因 1：Stop event 語意誤解**
- Hook docstring 寫「session 結束時掃描」，但 Claude Code `Stop` event 在**每個 assistant turn 結束**時觸發，非 session 結束
- Claude Code 無 `SessionEnd` event（PM Context 中「SessionEnd」為推測，實際不存在）
- 結果：hook 每 turn 都執行，而非只在 session 結束時

**根因 2：缺乏 in_progress ticket 豁免**
- 任務進行中（有 in_progress ticket 存在）也觸發 handoff 同步提醒
- 邏輯倒置：任務未完成時提醒 handoff 無意義，handoff 是 session 結束時的動作
- 同類 hook `handoff-auto-resume-stop-hook.py` 已有 stop flag 防重複觸發機制（300 秒內抑制），本 hook 無此防護

**根因 3：Handoff section 擷取範圍過寬**
- `_extract_handoff_section()` 找到最早的 handoff 關鍵字（index 42082），往後掃到下一個 H1/H2 標題才停
- worklog 有多個歷史 handoff 段落（多 session 累積），段落間以 H3（`###`）分隔而非 H2
- 結果：擷取範圍 = 42082~91343（49K chars），包含所有歷史 handoff 段落
- 從中解析出 283 個 ticket ID，經 terminal 狀態過濾後剩 ~12 個未完成 ticket
- 這 12 個 ticket 每 turn 重複列出，因為 worklog 內容不變

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `.claude/hooks/stop-worklog-handoff-sync-check-hook.py` | 主要修復對象 |
| `.claude/pm-rules/session-switching-sop.md` | 引用本 hook 行為，可能需同步更新 |

### WRAP 分析

#### W — Widen（擴增選項）

| 方案 | 描述 | 優點 | 缺點 |
|------|------|------|------|
| A. Stop flag 防重複觸發 | 仿 handoff-auto-resume-stop-hook 的 stop flag 機制，首次觸發後寫 flag，TTL 內不再觸發 | 最小改動，已有同類參考 | 治標不治本，首次仍可能在任務進行中觸發 |
| B. 只擷取最新 handoff 段落 | 修改 _extract_handoff_section() 從最後一個 handoff 關鍵字往後擷取，而非從最早的 | 精準對應當前 session 的 handoff 意圖 | 需確認「最新」定義（最後一個關鍵字 or 最後一個 H3 handoff 段落） |
| C. in_progress ticket 豁免 | 偵測是否有 in_progress ticket，有則跳過整個 hook | 邏輯正確（任務進行中不該提醒 handoff） | 需讀取 ticket frontmatter，增加 I/O |
| D. 組合方案（B + C + A） | B 修正擷取範圍 + C 加 in_progress 豁免 + A 加 stop flag 防重複 | 三個根因都處理 | 改動較大 |
| E. 停用 hook | 暫時從 settings.json 移除本 hook | 立即消除噪音 | 失去 handoff 同步檢查功能 |

#### R — Reality-test（現實檢驗）

- 方案 A 只解決頻率問題，不解決擷取範圍過寬（根因 3）和邏輯倒置（根因 2）
- 方案 B 單獨實施可解決 12→正確數量，但任務進行中仍會觸發
- 方案 C 單獨實施可避免任務中觸發，但結束時仍列 283 IDs 中的 12 個
- 方案 D 是唯一覆蓋三個根因的方案，改動量約 30-40 行，風險可控
- 方案 E 是最安全的短期止血，但犧牲功能

**最可能失敗原因**：方案 D 的 in_progress 偵測（C）依賴 ticket frontmatter 讀取，若 ticket 目錄結構異常會 false negative（fail-open 設計可緩解）

#### A — Attain distance（機會成本）

- 選 D 的機會成本：約 1 個 IMP ticket（P1），預估 20-30 分鐘
- 不選 D 的機會成本：每 session 每 turn 重複噪音持續訓練 PM 忽略提醒，hook 信任度下降

#### P — Prepare to be wrong（預備失敗）

- 若方案 D 實作後仍有 edge case → 降級為方案 E（停用 hook），建新 ANA 重新評估
- 驗證方式：修復後連續 5 turn 觀察，確認只在最後一 turn（無 in_progress ticket 時）觸發且只列當前 session 的 handoff ticket

### 決策結論

**選擇方案 D（B + C + A 組合）**，三個修復對應三個根因：

| 根因 | 修復 | 對應方案 |
|------|------|---------|
| 根因 1：每 turn 觸發 | 加 stop flag 防重複（TTL 2 小時，與 handoff-auto-resume 一致） | A |
| 根因 2：in_progress 也觸發 | 偵測 in_progress ticket 存在時跳過 | C |
| 根因 3：擷取範圍過寬 | 改用最後一個 handoff 關鍵字位置往後擷取 | B |
