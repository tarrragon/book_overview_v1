### Spawned Tickets

決議「保留欄位、view 不渲染、v2 規格無需補丁」，不需後續修復或升級 Ticket。

### 原始用途確認

pending_checks 欄位由 017.8 設計，用途為：
1. SAFE_CALL 在資料源 I/O 失敗時 append `auto_detectable=False` 的 PendingCheck
2. 作為 `ready_for_clear` 推導的輸入條件：`all(not c.auto_detectable for c in pending)` — 若存在 auto_detectable=True 的 check，ready_for_clear 被阻擋
3. 017.13 AC3 決議「保留 + 文件化」，理由：(a) ready_for_clear 依賴此 list，(b) 既有測試驗證 reason 文字為對外契約

### 決議：v2 規格無需新增渲染輸出

**結論**：pending_checks 是正確的「內部推導輸入」，其效果已透過 `ready_for_clear` 間接呈現於 view 層。不需要在 checkpoint_view.py 額外直接渲染 pending_checks 內容。

**理由**：
1. pending_checks 只在 I/O 降級時產生（SAFE_CALL fallback 路徑），此時 ready_for_clear=False 已傳達「不可 clear」語意
2. data_sources dict 已記錄具體 I/O 錯誤訊息（checkpoint-status 命令渲染此 dict），使用者可從中得知降級原因
3. 直接渲染 pending_checks 會與 data_sources 資訊重複，增加認知負擔

**建議**：
- 不修改 v2 規格（不新增 pending_checks 渲染輸出）
- 不棄用 lib 層 pending_checks 欄位（017.13 AC3 已決議保留，ready_for_clear 依賴它）
- v1 §5.2 的「pending_checks → 補充需人工確認訊號」描述為「介面充足性確認」，確認欄位存在且 lib 暴露即可，不等於 view 層必須直接渲染
- 本 ticket 結論：欄位保留，view 層不渲染，v2 規格正確無需補丁
