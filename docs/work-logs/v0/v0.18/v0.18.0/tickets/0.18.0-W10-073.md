## Problem Analysis

### 問題根因

用戶觀察到 `/ticket` 啟動後，基礎知識載入佔用 Opus 1M context 約 50% 容量。

根因在於 Claude Code 自動載入範圍：`CLAUDE.md` + `.claude/rules/**/*.md`，當前共 **11 個檔案 / 2166 行 / ~85KB**，其中大量內容屬於「條件性才需要」的規則（例如 `quality-common.md` 378 行僅在代理人寫程式碼時才需要，PM 派發時完全用不到）。

| quality-common.md | 378 | 15076 | 代理人寫程式碼 | 高（可外移） |
| pm-role.md | 346 | 17820 | 主線程派發 | 中（骨架保留） |
| CLAUDE.md | 250 | 7922 | 啟動 | 低（必須載入） |

### 相關 Error Pattern

- 無直接相關 error-pattern，但設計目標符合 `.claude/rules/README.md` 所述的「開放封閉原則」（開放內容自動載入，封閉細節按需讀取）
- 本 ticket 將驗證現有「骨架+references」設計模式的執行徹底度

### 判斷原則（核心依據）

**拆分的判斷維度是「認知負擔」與「按需載入」，不是「行數」。**

這是卡片盒筆記法（Zettelkasten）的核心精神，也是本專案文件設計的核心精神：

| 正確的判斷問題 | 錯誤的判斷問題 |
|--------------|--------------|
| 這段知識在什麼情境才需要理解？ | 這檔案多少行？ |
| 每段知識是否原子化（單一職責）？ | 骨架是否「夠短」？ |
| 按需載入時能否獨立理解？ | 行數超過某閾值就要拆？ |
| 保持載入會增加閱讀者在無關情境的認知負擔？ | 檔案規模是否超標？ |

**推論**：

- 一個 500 行的檔案若內容**全部原子化且在同一情境共同需要**，不應因行數拆分
- 一個 100 行的檔案若**混合多個情境才用到的知識**，即使短也應拆分
- 拆分後的每一個碎片必須能獨立被理解（原子性），否則只是把完整概念切碎而已
- 保留判斷標準不是「這段很重要」，而是「所有使用場景都需要立即載入」

### 初步分析（用戶對話中已完成）

**拆分候選分類**（依「按需載入情境匹配度」評估，非行數）：

1. **高信心可外移**（啟動情境完全不匹配此知識的使用情境）
   - `quality-common.md` — 內容是「代理人寫程式碼時的品質規則」，PM 派發時完全不需要。情境錯配明確，可外移到 `references/` 供代理人按需讀取。
   - `observability-rules.md` — 內容是「寫產品程式碼時的 catch/日誌/全域錯誤」，啟動時不需要。情境錯配明確。
   - `cognitive-load.md` — 內容是「任務拆分時的量化閾值」，僅 PM 拆任務時需要。可整併至 `pm-rules/task-splitting.md`（PM 按需讀取）。

2. **中信心需檢視原子化狀態**（檔案可能混合多個情境的知識）
   - `pm-role.md` — 「角色辨識 + 禁止產品程式碼」是每次主線程回應都需要的核心約束；「失敗判斷前置步驟 SOP」「TaskOutput 安全範本」是特定情境才觸發的程序。兩者情境不同，應評估拆分為「常態規則」+「情境程序」。
   - `quality-baseline.md` — 7 條規則是否全部在啟動時都需要？或某幾條規則有明確的觸發情境（例如規則 4 可觀測性僅在寫程式時才需要）？需逐條評估。
   - `document-format-rules.md` — 「禁用 Emoji」「狀態標記」是所有文件輸出都適用（啟動情境匹配）；「跨檔案引用格式」「規格文件引用穩定性」是寫特定文件時才需要。情境混合，應評估。

3. **低信心建議保留**（所有使用場景都需要立即載入的知識）
   - `CLAUDE.md` — 專案身份、指令、架構是任何操作前都必須知道
   - `language-constraints.md` — 語言約束是每次輸出都適用
   - `tool-discovery.md` — 五問流程是「PM 準備說做不到」時的立即觸發防護，不能等讀取後才用
   - `bash-tool-usage-rules.md` — cd 污染/git 串接是每次 Bash 呼叫的立即約束，不能遲載
   - `rules/README.md` — 目錄索引本身就是導航，必須預載才能找到其他規則

```bash

wc -c CLAUDE.md .claude/rules/**/*.md
```

- 11 個檔案納入自動載入

### 實驗發現

**已驗證事實**：
1. Claude Code 自動載入範圍確實為 `CLAUDE.md + .claude/rules/**/*.md`（`rules/README.md` 明確記載）
2. 當前自動載入 2166 行，對於 Opus 1M context 約佔用固定成本
3. 部分檔案內容與啟動時機不匹配（例如 quality-common 是代理人才需要的程式碼品質規則）

**待驗證假設**（交由 11 個子 ticket 的 WRAP 分析驗證）：
1. 每個檔案的「啟動必要性」評估是否正確
2. 骨架+references 模式是否適用於所有候選檔案
3. 拆分後的引用耦合（有多少其他檔案引用將被移出的內容）
