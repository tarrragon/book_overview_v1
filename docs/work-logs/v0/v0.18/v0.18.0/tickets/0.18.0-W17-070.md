## 重現實驗結果

### 實驗方法

直接呼叫 `_is_placeholder` 和 `_is_section_empty`（hook 端）函式，傳入 W17-056 事件的實際章節內容結構，驗證兩處 false negative。

實驗環境：`(cd .claude/skills/ticket && uv run python3 -c "...")`

實驗輸入：
1. `_is_placeholder` 傳入 `<!-- Schema note -->\n\n<!-- To be filled -->\n\n---`
2. `_is_placeholder` 傳入純 `---`
3. `validate_execution_log` 傳入模擬 W17-056 pre-fix body（Schema 章節只含 note+separator，實質內容在自定義 H2）
4. Hook 端 `_is_section_empty` 傳入同結構

### 實驗執行

**Test 1**：`_is_placeholder('<!-- Schema[IMP/Solution]: 選填 -->\n\n<!-- To be filled by executing agent -->\n\n---')`
- 結果：`False`（應為 `True`）
- 根因：HTML 註解剝除後剩 `---`，`_is_placeholder` 未識別 `---` 為非實質內容

**Test 2**：`_is_placeholder('---')`
- 結果：`False`（應為 `True`）
- 根因確認：`---` 本身不在任何 placeholder 模式中

**Test 3**：`validate_execution_log('W17-056', body_with_custom_h2)`
- 結果：`(True, [])`（表示「全部已填寫」，應為 `(False, ['Problem Analysis', 'Solution', 'Test Results'])`）
- 根因：雙層共振——(A) section 擷取被 `## 實作摘要` 切斷，schema section 僅含 note+separator；(B) `_is_placeholder` 對 `---` 返回 False

**Test 4**：Hook 端 `_is_section_empty(content, 'Solution')` 傳入含 `---` 的 schema section
- 結果：`False`（表示「非空」，應為 `True`）
- 根因確認：與 validator 端同構——HTML 剝除後 `---` 殘留被視為實質內容

### 實驗發現

**已驗證事實**：

1. `_is_placeholder` false negative 已確認：`---` 不在剝除清單中，任何含 `---` 的空殼章節都會被放行
2. `validate_execution_log` section 擷取切斷已確認：自定義 H2 會截斷 schema section 範圍
3. Hook 端 `_is_section_empty` 存在同構 bug：與 validator 端 `_is_placeholder` 獨立實作但同樣未剝 `---`
4. 雙層共振已確認：根因 A（`---` 未剝）+ 根因 B（自定義 H2 切斷）任一單獨成立即可觸發 false negative；兩者同時存在時 100% 觸發
5. W17-032 修復確認為單向：只覆蓋 false positive 分支，未測 false negative

**仍未驗證的假設**：

- Agent 自定義 H2 行為是否跨 agent 通用（僅觀察到 thyme-documentation-integrator 一例，樣本不足判斷通用性）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 雙根因因果鏈分析（三層）

#### 層 1：Agent 行為層（源頭）

Agent（thyme-documentation-integrator）寫 `## 實作摘要` / `## 驗證指令與結果` 作為自定義 H2 章節，而非在 `## Solution` / `## Test Results` 下用 H3 子章節。

失效原因：`agent-definition-standard.md` v1.1.0「執行責任：Ticket body 填寫」條款規定「使用 `ticket track append-log <id> --section "<章節名>"`」，但未明示「禁止自定義 H2 章節」「必須用 Schema 定義的章節名」。Agent 缺乏禁令指引，自然以自認合理的章節名撰寫。

#### 層 2：Validator 邏輯層（偵測失敗）

兩個獨立 bug 共振：

(a) `_is_placeholder` 剝除 HTML 註解後未剝除 markdown 分隔符 `---`。剩餘 `---` 被判為非 placeholder。

(b) `validate_execution_log` 的 section 擷取邏輯使用「到下一個任意 `##` 或 `###`」作為邊界（line 439-443），自定義 H2 截斷 schema section 範圍。即使 `_is_placeholder` 修好，只要 agent 寫自定義 H2，schema section 仍只含 note+separator 而非實作內容。

#### 層 3：Hook 體系層（最後防線失敗）

`acceptance_checkers/execution_log_checker.py` 的 `_is_section_empty` 是 `_is_placeholder` 的獨立重寫版本，但包含同構 bug（未剝 `---`）。且同樣使用 `(?=^## )` regex 作為 section 邊界，被自定義 H2 切斷。

acceptance-gate-hook 呼叫 `check_execution_log_filled` 時，Solution 和 Test Results 都被判為「已填寫」，未觸發阻擋。

**偵測缺口完整清單**：

| 缺口 | 層級 | 說明 |
|------|------|------|
| `---` 未列入剝除清單 | Validator + Hook | 兩處獨立實作都遺漏 |
| Section 邊界被任意 H2 切斷 | Validator + Hook | 應限定為 Schema 定義的章節名 |
| 無自定義 H2 偵測 | Hook | 無任何 hook 檢查 ticket body 是否含非 Schema 的 H2 章節 |
| `_is_placeholder` 與 `_is_section_empty` 未共用 | 架構 | 兩處獨立維護，bug 修復無法同步 |

### 三類方案 ROI 比較

| 方案 | 修復範圍 | 實作成本 | 效益 | ROI |
|------|---------|---------|------|-----|
| A. Validator bug fix | `_is_placeholder` 加 `---` 剝除 + `validate_execution_log` section 邊界改用 Schema 章節名 | 低（約 10 行） | 堵住 false negative；但 Hook 端仍有同構 bug | 中 |
| B. Hook 端同步修復 | `_is_section_empty` 加 `---` 剝除 + regex 邊界修正 | 低（約 5 行） | 堵住 Hook 端 false negative | 中 |
| C. Agent 規則強化 | `agent-definition-standard.md` 新增禁止自定義 H2 條款 | 低（文字修改） | 從源頭減少觸發；但無強制力（agent 可能不讀規則） | 低-中 |
| D. Pre-complete hook 偵測自定義 H2 | 新增 hook 檢查 ticket body 含非 Schema H2 時警告 | 中（新 hook 或擴展既有 hook） | 強制性偵測源頭問題 | 高 |
| A+B+C（建議組合） | 三層同時修復 | 低-中 | 堵住兩根因所有路徑 | 高 |

### Spawned IMP 規格建議

#### IMP-1：症狀修復（Validator + Hook `---` 剝除 + section 邊界修正）

**範圍**：

1. `ticket_validator.py` `_is_placeholder`：HTML 註解剝除後增加 markdown 分隔符剝除（`re.sub(r'^---+$', '', ..., flags=re.MULTILINE)`）
2. `ticket_validator.py` `validate_execution_log`：section 擷取邊界從「任意 `##`/`###`」改為「Schema 定義的章節名（Task Summary / Problem Analysis / Solution / Test Results / Completion Info / NeedsContext / Exit Status / 重現實驗結果）+ 文件結尾」
3. `acceptance_checkers/execution_log_checker.py` `_is_section_empty`：同步修正 (1)(2) 的邏輯；或考慮統一呼叫 `_is_placeholder` 消除重複實作
4. 補單元測試：覆蓋 `---` 殘留 false negative、自定義 H2 切斷 false negative 兩場景

**驗收條件**：
- `_is_placeholder('---')` 返回 `True`
- `_is_placeholder('<!-- note -->

<!-- placeholder -->

---')` 返回 `True`
- `validate_execution_log` 對含自定義 H2 的 body 正確偵測 schema section 為空
- Hook 端 `_is_section_empty` 對同場景返回 `True`
- 既有測試全部通過（無 regression）

#### IMP-2：根因防護（Agent 規則 + 偵測 hook）

**範圍**：

1. `agent-definition-standard.md`「執行責任：Ticket body 填寫」條款新增：「禁止在 ticket body 中建立 Schema 定義以外的 H2 章節（`## 自定義標題`）。實作內容必須寫在 Schema 章節下，可用 H3 子章節（`### 子標題`）組織」
2. 擴展既有 hook（acceptance-gate 或 ticket-quality-gate）：complete 時掃描 body 中的 H2 章節名，與 Schema 定義比對；發現非 Schema H2 時輸出 warning（不阻擋，避免誤殺既有 ticket）
3. PC-110 防護章節補完：填入本 ANA 結論的修復方案與 spawned IMP ticket ID

**驗收條件**：
- `agent-definition-standard.md` 含禁止自定義 H2 條款
- Hook 掃描到 `## 實作摘要` 等非 Schema H2 時輸出 warning
- PC-110 防護章節已更新
