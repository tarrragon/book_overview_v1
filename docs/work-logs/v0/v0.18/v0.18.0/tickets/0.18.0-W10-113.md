## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 觀察事實：本次 /ticket session 的 tool call 盤點

從用戶輸入 /ticket 到 AskUserQuestion 顯示選項，主線程實際執行 7 個 tool call：

| # | Tool Call | 耗時主因 | 是否必要 |
|---|----------|---------|---------|
| 1 | runqueue --context=resume --top 3 | SKILL.md 規範 step 1 | 必要但與 SessionStart hook 重複 |
| 2 | list --status pending,in_progress --format=list | 試錯（list 不是有效值） | 不必要：SKILL.md 文檔不完整 |
| 3 | list --status pending,in_progress --format=table | 重試成功，但 dump 67 筆 | 不必要：list 無預設 limit |
| 4 | list --status in_progress | 為了單獨抽出 in_progress | 不必要：步驟 3 已包含 |
| 5 | runqueue --top 5 | 為了組 AskUserQuestion 選項 | 不必要：與 step 1 重複 |
| 6 | ToolSearch select:AskUserQuestion | AskUserQuestion 是 deferred tool | 不必要：核心 PM 工具不該 deferred |
| 7 | AskUserQuestion 本體 | 顯示選項 | 必要 |

**結論**：真正必要的只有 step 1 + step 7。其他 5 次都是設計問題造成的浪費。

### 設計根因清單

#### 根因 A：scheduler 邏輯重複執行三次

- SessionStart hook 已跑過 runqueue（產出「無 resume 候選」+ spawned 推進清單）
- /ticket 入口重跑（runqueue --context=resume）
- 為組 AskUserQuestion 選項又跑第三次（runqueue --top 5）

資訊都是同一份，但無快取機制讓後續呼叫複用 hook 結果。

#### 根因 B：dashboard 視角缺失

/ticket 入口需要的是「in_progress + top N ready + stale 警示」聚合視圖，但目前需要拼三個 CLI：list --status in_progress + runqueue --top N + stale-list。PM 接手時實際只問 1 個問題（下一個做哪個），系統卻要求拼 3 個查詢。

#### 根因 C：list 命令無預設限制

list --status pending,in_progress 直接 dump 67 筆，PM 必須掃過全部才能定位有意義項目。預設應該 --top 10 by priority 或只顯示 stale + ready 前段，全量需明確 --all。

#### 根因 D：SKILL.md 與 CLI 行為不一致

SKILL.md 範例寫 list --status pending,in_progress 無 --format，--format=list 是合理猜測但不存在（CLI 只接受 table/ids/yaml）。SKILL hook 已警告觸發 ADJ ticket 建議。

#### 根因 E：AskUserQuestion 是 deferred tool

PM 規則明訂「列選項必用 AskUserQuestion」，但它每次 session 都要 ToolSearch 載入 schema。核心 PM 工具被歸類為 deferred = 流程強制依賴前置動作。

#### 根因 F：SessionStart hook 與 /ticket 流程斷裂

SessionStart hook 已產出「無 resume 候選」「Spawned 推進清單 21 筆」「下 session 建議項目」段落。但 /ticket 入口不讀 hook output，又自己跑一遍。Session 內資訊未被持久化共享。

### 待評估的改善方向

| # | 改善點 | 預期效果 | 待評估項 |
|---|--------|---------|--------|
| M1 | 新增 ticket track dashboard 一次回傳 in_progress + top N ready + stale | 7 tool call -> 2 個 | 設計複雜度、是否與既有命令重疊 |
| M2 | SessionStart hook 快取 runqueue 結果到 .claude/cache/，/ticket 讀檔 | 消除 scheduler 重複執行 | 快取一致性、失效策略 |
| M3 | list 預設加 --top 10 by priority，全量需 --all | dump 67 -> 10 筆 | 是否破壞既有 caller 假設 |
| M4 | AskUserQuestion 從 deferred 改為 default load（CC platform 設定層） | 消除 ToolSearch 前置 | 是否可在 settings.json 控制、跨 session 影響 |
| M5 | SKILL.md 補 --format 可選值與 list 預設行為說明 | 消除試錯往返 | 文檔更新範圍、是否含其他類似缺口 |

### WRAP 三問（claim 後 ANA 強制要求）

**W (Widen)** — 候選方案：

1. 5 個改善方向全部 spawn ticket（完整重構）
2. 只取最痛點（M1 dashboard）局部優化
3. 無 trigger 延後（被駁回，違反 decision-trigger-binding 規則）
4. 放棄重構，接受現狀

**A (Attain distance)** — 機會成本：投入此 ANA + spawn IMP 會擠壓 W10-096 監測任務與 W10-102/103 規則 8 違規修復。但 PM 接手是高頻動作（每 session），技術債利息高。

**P (Prepare to be wrong)** — 失敗預想：M2 hook 快取若失效策略沒設計好（過時 runqueue 結果誤導 PM），會反向引入新 bug。防護：M2 ticket 必含 TTL 或事件驅動失效設計。

**R (Reality Test)** — 已完成，見下方重現實驗結果章節。M4 假設被駁回，需改設計方向。
