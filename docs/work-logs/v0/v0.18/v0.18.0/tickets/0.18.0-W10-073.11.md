### 判斷原則（父 ticket 共通指引）

**拆分依據是「認知負擔」與「按需載入」，不是行數。**

這是卡片盒筆記法（Zettelkasten）的核心精神：

- 正確問題：這段知識在什麼情境才需要理解？每段知識是否原子化？按需載入時能否獨立理解？
- 錯誤問題：檔案多少行？骨架是否夠短？行數超過某閾值就要拆？

**推論**：
- 500 行的檔案若內容全部原子化且在同一情境共同需要 → 不應拆分
- 100 行的檔案若混合多個情境才用到的知識 → 即使短也應拆
- 拆分後的每一碎片必須能獨立被理解（原子性），否則只是把完整概念切碎
- 保留判斷標準不是「這段很重要」，而是「所有使用場景都需要立即載入」

**WRAP 分析時的具體提問**：

1. **W (擴增選項)**：列出所有可能的原子化切分方式（不是以行數為切點，而是以「情境匹配度」為切點）
2. **R (現實測試)**：每個切分方案下，各個碎片是否能獨立被理解？保留在 auto-load 的碎片是否「所有場景都需立即載入」？
3. **A (保持距離)**：從 sync 到其他專案的角度看，這些知識是否真的每個專案啟動都需要？
4. **P (準備出錯)**：若拆分後發現某個碎片其實常被需要，如何快速恢復到 auto-load？

## WRAP 分析：rules/README.md (50 行) 拆分可行性

### R (Reality Test) — 現實測試

**讀者行為模擬**：

| 讀者 | 真的會讀本檔嗎？ | 需要哪一段？ |
|------|---------------|------------|
| PM（session 啟動） | 否，會直接看 CLAUDE.md 的 `@.claude/rules/README.md` 引用 | L1-12（導航） |
| 代理人（派發時） | 否，只讀 agents/<name>.md 和被點名的 rules | 幾乎不讀此檔 |
| 框架維護者（改規則時） | **是**，需查「這屬於 rules 還是 skill」 | L26-46（職責分離） |
| 新 Skill 建立者 | **是**，需查框架資產 vs 專案產物原則 | L26-46 |

**關鍵發現**：
- L1-12（平台機制 + 目錄表）確實是**每個角色啟動時的導航價值**，但目前 50 行全量載入等於「每個 session 都吞了 30 行根本不會看的職責分離規則」
- L14-46（兩個職責分離章節）是**情境觸發型知識**（設計決策時才需），不符合「auto-load = 所有場景必需」原則
- 依父 ticket 核心原則「拆分依據是認知負擔與按需載入，不是行數」：本檔的 L14-46 **正是**混合情境才用到的知識的典型樣本，即使短也應拆

**投資報酬率**：
- 保留 50 行成本：每 session × 每角色 × ~200 tokens = 小但永久
- 精簡成本：一次 Edit + 建立 references/framework-asset-separation.md + 更新 CLAUDE.md `@` 引用

**引用耦合檢查**（7 處引用）：
- `CLAUDE.md:@.claude/rules/README.md` → 精簡後仍有意義（導航表保留）
- `.claude/README.md:354` → 「完整規則索引」指向 rules/README.md，精簡後仍指目錄表
- `.claude/templates/CLAUDE-template.md:41` → 同上
- `ARCH-012` / `PC-061` 引用「專案設定與代理人知識分離」「框架/產物分離」具體章節 → **此處需同步更新引用到新 references/ 檔案**
- `skills/ticket/CYCLE_DETECTION_GUIDE.md:353` → 泛指規則，精簡後仍有效

### A (Attain Distance) — 保持距離

**從其他專案 sync 視角**：
- 本檔所有內容都是**跨專案通用**（框架機制、目錄分工、職責分離原則），無專案特定識別符
- sync 到新專案時：L1-12 有即時導航價值；L14-46 只在首次設計規則 / Skill 時有用
- 結論：sync 無死連結風險，但 L14-46 的「每 session 都載入」對新專案同樣是冗餘投資

**從規則系統演進視角**：
- rules/core/ 目前 9 個檔案，未來可能增減（如 W10-073 正在評估其他檔案拆分）
- L5-12 的目錄表若內容變動，需同步維護 → 低風險（改動頻率低）
- L14-46 的兩個職責分離章節若規則改變，本檔和 ARCH-012/PC-061 需同步改 → 精簡後反而**減少同步點**（集中在 references/ 單點）
