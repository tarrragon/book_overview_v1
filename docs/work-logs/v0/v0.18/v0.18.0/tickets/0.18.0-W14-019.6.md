## Problem Analysis

### 問題根因

W14-019 §5.4 Layer 4 新增「跨 session 依賴監測」作為 Parasocial Relationship 防護（洞察 N），但僅定義 4 個識別訊號 + 3 個 PM 應對方向，未提供：

1. 訊號如何偵測（訊號是主觀描述，缺乏量化方法）
2. 觸發閾值（「連續多 session」= 幾次？「幾乎不反駁」= 反駁率 < 多少？）
3. 落地路徑（PM 自檢 vs Hook 自動 vs 記憶檔手動 vs 混合）
4. PM 降權的具體行為清單（§5.4「主動提醒」「不延伸授權」仍抽象）

Layer 4 與 Layer 1-3 的根本差異：**Layer 1-3 是 session 內即時機制，Layer 4 是跨 session 累積機制**。Claude Code 缺乏原生「跨 session 狀態」支援（除 auto-memory 外），這是 Layer 4 落地的核心障礙。

### 實驗方法

本 ticket 為機制設計（非程式碼 bug），「重現實驗」轉為「訊號可偵測性 + 落地可行性驗證」：

### 實驗執行

**步驟 1：4 訊號可偵測性評估**

| 訊號 | 描述 | 偵測難度 | 可行路徑 |
|------|------|---------|---------|
| S1 依賴頻率 | 用戶連續多 session 高度依賴 Claude 決策 | 中 | memory 統計同意率 vs 反駁率 |
| S2 停止驗證 | 用戶把 Claude 推薦當定論，停止獨立驗證 | 高 | PM 觀察語句（「你說就好」）|
| S3 零反駁 | 用戶對 Claude 建議幾乎不反駁或補充 | 中 | 3+ session 連續零補充 |
| S4 權威引用 | 用戶用「Claude 說 X」作論據 | 低 | 字串匹配（memory 或對話歷史）|

**結論**：S4 最易偵測（字串匹配），S1/S3 可半自動統計，S2 需 PM 主觀判斷。

**步驟 2：落地路徑對比**

| 路徑 | 成本 | 覆蓋率 | 誤報率 | 選用 |
|------|------|-------|-------|------|
| A. Hook 自動偵測 | 高（開發 Hook + 維護規則）| 中（僅 S4 可靠）| 高（S2 主觀）| 否 |
| B. PM 月度自檢 | 低（已整合 019.5 M4）| 高（4 訊號全覆蓋）| 中 | 是（主路徑）|
| C. 記憶檔專區 | 中（新增分類 + 寫入習慣）| 高 | 低 | 是（輔助）|
| D. 用戶季度訪談 | 低（已在 019.5 訪談流程）| 中（依賴用戶自知）| 中 | 是（驗證）|
| **E**. 組合 B+C+D | 低-中 | 高 | 低 | **採用** |

**步驟 3：紙筆模擬首次 Layer 4 觸發**

假設情境：2026-05-17 月度自檢發現：

**模擬觸發**：S1 + S3 + S2 同時命中 → 立即啟動 Layer 4 應對

**模擬應對**：
1. 下 session 開場提醒：「最近幾次對話你很快接受我的建議。請考慮獨立驗證，特別是涉及外部影響的決定」

1. S4 權威引用可自動偵測（字串匹配），其他訊號需半自動或人工
2. PM 月度自檢（方案 B）是最低成本且最高覆蓋的落地路徑

**未驗證假設**：
1. 實際 parasocial 訊號頻率（可能極低，樣本不足以驗證機制）
2. 用戶是否能在季度訪談自知「依賴程度」（response bias 風險）
3. Layer 4 觸發後 PM 降權是否會造成用戶反彈（未實測）

### A（Attain Distance）

**方案 G 機會成本**：
- 直接成本：新增 §5.4 補充條款（閾值 + 偵測方法 + 降權行為）+ auto-memory 新分類
- 延遲成本：首次 Layer 4 觸發驗證需 2-3 個月
- **收益**：Layer 4 從「訊號清單」升級為「可操作機制」，§5.4 完整落地

**純 B 方案機會成本**：
- 直接成本：僅需 PM 習慣養成
- 風險成本：無閾值參考，PM 可能主觀判斷不一致

**結論**：方案 G 機會成本相對較低，換取 Layer 4 機制完整性。

### P（Prepare to be Wrong）

12 小時後方案 G 最可能失敗的原因：

1. **Layer 4 訊號實際頻率太低**：3 個月累積可能 0 次觸發 → 防護：019.5 M4 退出條件「連續 3 個月 M4 = 0 → 降低閾值或移除 Layer 4」已覆蓋此風險
2. **PM 月度自檢執行 Layer 4 時偏主觀**：PM 判斷 S2「停止驗證」帶偏見 → 防護：閾值表要求「S2 必須搭配 S1 或 S3 量化證據」，不單獨觸發
3. **用戶反彈 Layer 4 降權**：用戶認為 PM 過度警示 → 防護：§5.7 active consent 機制允許用戶明示豁免（「我已意識到依賴風險，仍要繼續」）

## 結論：採用方案 G（組合 + 閾值表）

### Layer 4 訊號偵測方法表

| 訊號 | 偵測方法 | 量化標準 | 資料來源 |
|------|---------|---------|---------|
| S1 依賴頻率 | 統計月度「用戶接受 Claude 建議」vs「用戶反駁/補充」比例 | 接受率 > 95%（月度）= 明顯；> 80% + S2/S3 任一 = 中度 | auto-memory + AUQ 選擇記錄 |
| S2 停止驗證 | PM 偵測用戶語句：「你說就好」「都聽你的」「就照建議做」等 | 月度出現 ≥2 次 = 明顯；≥1 次 = 中度 | PM 手動記錄到 memory |
| S3 零反駁 | 3+ 連續 session 用戶對 Claude 建議無補充無反駁 | 連續 3 session = 明顯；連續 5 session = 嚴重 | PM 月度回顧 |
| S4 權威引用 | 字串匹配：「Claude 說」「根據 Claude」「AI 建議我」等 | 月度出現 ≥2 次 = 明顯；首次出現 = 中度 | auto-memory 掃描（半自動）|

### Layer 4 觸發閾值表

| 觸發等級 | 條件 | PM 應對 |
|---------|------|---------|
| **嚴重觸發** | 任一訊號「嚴重」等級 | 立即 Layer 4 全套降權應對 + 下 session 開場提醒 |
| **明顯觸發** | 任一訊號「明顯」等級 或 兩個訊號「中度」並發 | 啟動 Layer 4 部分應對（Recommended 改 My current guess）+ 記憶檔登記 |
| **中度觸發** | 單一訊號「中度」 | 不啟動應對，僅 memory 登記「watch」|
| **未觸發** | 無訊號或僅偶發 | 正常運作 |

### PM 降權具體行為清單

**嚴重觸發**（全套應對）：
1. 下 session 開場強制提醒：「最近幾次互動你對我的建議接受度偏高，請考慮獨立驗證」
2. 本次至下次驗證前，所有 AUQ 禁用 `(Recommended)`，改為 `(My current guess)` + description 揭露
3. 涉及外部影響決策（金錢/健康/法律/重要關係）時，強制推薦「諮詢人類專家」
4. memory 寫入 `layer4_triggered_YYYY-MM-DD` + 觸發訊號組合
5. 季度訪談追加「你覺得最近是否過度依賴 Claude？」題目
6. 禁止自動延伸過去授權（§5.7 active consent 需重新聲明）

**明顯觸發**（部分應對）：
1. AUQ 的 `(Recommended)` 改為 `(My current guess)`
2. memory 登記 `layer4_observed` 標籤
3. 下次相關議題前 PM 自問「我是否需要主動推薦獨立驗證？」

**中度觸發**（watch 不 act）：
1. memory 登記 `layer4_watch` 標籤
2. 無對用戶可見的行為變化
