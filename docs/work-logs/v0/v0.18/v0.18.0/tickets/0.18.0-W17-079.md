## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

PC-111 v1.2.0 R5「素材跨層誤推」揭示：讀者易將「實作層/工具層/協作層」素材（如 git index race / subagent 局部視角 / Team shared state）提升至「認知層/架構層」論述（如 working memory 混淆）。

Q4 素材抽象層級檢查（防護 A）已加入 PC-111，但僅為讀者自檢；引導讀者的三個 SKILL 及 error-pattern 撰寫骨架**自身沒有層級顯性化設計**，無法在「讀者第一次閱讀時」就引導正確的層級理解。三 SKILL 的 description、Use when、Why、Action 等引導文字若以跨層語言描述，讀者的 Q1-Q3 自檢會找到素材（引用檢查通過），但 Q4 層級檢查需要讀者主動識別 SKILL 本身是否引導跨層——此為結構性盲點。

### 影響範圍

- `.claude/skills/wrap-decision/SKILL.md` — 警戒術語清單未標層級歸屬
- `.claude/skills/compositional-writing/SKILL.md` — 五大原則無「層級顯性化」獨立條款（僅原則 3 散文單句隱晦提到）
- `.claude/skills/agent-team/SKILL.md` — description 與快速決策表使用協作層術語，易被讀者上推至認知層
- `.claude/error-patterns/README.md` — 文件格式模板無「抽象層級」欄位
- `.claude/skills/skill-design-guide/SKILL.md`（待檢查）— SKILL 設計規範可能缺「層級標記」欄位

### 相關 Error Pattern

- **PC-111 v1.2.0** — R5 素材跨層誤推（本 ANA 直接觸發）
- **PC-066** — 決策品質 autopilot（同宗：跳過自檢）
- **PC-080** — WRAP A 階段框架檢查未做
- **PC-088** — LLM tool selection 步驟數估算偏誤
- **pm-judgment-interference-map v1.1.0 因子 1.4** — 素材錯置抽象層級

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### WRAP 四階段草案（主線程 Phase 1 起點；Phase 3 後依代理人回饋 refine）

#### W — Widen Options（候選改進方向）

| # | 候選方向 | 類別 | 成本估計 |
|---|---------|------|---------|
| A | 在 SKILL frontmatter 加 `abstraction_layers` 欄位 | 結構性 | 每 SKILL 1-3 行 |
| B | 在 SKILL 五大原則/觸發表加「層級顯性化」獨立條款 | 規則性 | 每 SKILL 5-20 行 |
| C | 在 error-pattern README 補「抽象層級」必填欄位 | 模板性 | README 20 行 + 既有 PC 追溯補欄 |
| D | 不改 SKILL 只改 error-pattern 模板（最小干預） | 最小化 | README 20 行 |
| E | 把三 SKILL 合併入一個 cross-layer-awareness methodology | 重構性 | 新建 methodology 200 行 |
| F | 不修改任何 SKILL/PC，加 lint hook 偵測跨層誤推 | Hook 性 | 新建 Hook 150 行 |

**假設層級多元性檢查**：六方向對應不同根因假設——A/B 假設問題在 SKILL 撰寫引導；C/D 假設問題在 PC 模板；E 假設問題在三 SKILL 概念重疊；F 假設問題在檢測而非文件。多元性足夠。

**方案性質三類涵蓋**（PC-095）：新增工具（F）/ 改造既有（A/B/C/D）/ 零工具或純文件（E 方法論無新 Hook）—— 三類齊備。

#### R — Reality Test（實境檢驗，已執行 + 待執行）

已執行（重現實驗章節）：
- 三 SKILL 關鍵字次數證實「層級缺失」假設

待執行（需代理人佐證）：
- 實際閱讀情境下三 SKILL 是否誘發讀者跨層？（需 linux / thyme 判斷）
- PC-111 R5 再發生基本率估計（需 bay 審計評估）
- 加層級欄位 vs 加 Hook 的 ROI（需 bay 評估）

**基本率問題**：類似「文件骨架缺防護設計導致用戶跨層誤推」的案例在 Claude Code 生態中基本率？（Zoom Out）

#### A — Attain Distance（機會成本）

**Tool selection check**（PC-088 物化檢查）：
- Q1 物化：方向 F 把「引導讀者」物化為「Hook 偵測」—— 是否忽略「問題在撰寫不在讀者」的本質？
- Q2 步驟數：A+B+C 三方向並行執行需 3 個 IMP；D 單獨做只需 1 個；E 重構需 1 個大型 IMP
- Q3 目的地：產出是「SKILL/PC 撰寫引導正確層級」，首選修改 SKILL/PC 本身（方向 A-D），次選新 Hook（F）
- Q4 白名單：首選低摩擦—— D（最小修改）優先；其次 B 增量添加條款

**10/10/10 規則**：
- 10 分鐘後：方向 D 最快完成，但僅覆蓋 PC 撰寫未來新增
- 10 個月後：方向 B 覆蓋 SKILL 讀者，長期影響更大
- 10 年後：方向 E methodology 成為知識組織基礎，但可能與 compositional-writing SKILL 重疊

**核心優先事項對齊**：W17-079 AC 5 明示「提出層級顯性化設計原則」—— 支持方向 A/B/C 組合；AC 8「產出 spawned IMP 落實至少一項」—— 傾向先小後大。

#### P — Prepare to be Wrong（行前預想）

**3 個最可能失敗原因**：

| # | 失敗原因 | 可能性 | 防護 |
|---|---------|--------|------|
| 1 | 加層級標記但讀者不看（讀者依賴既有習慣） | 40% | 在 SKILL 第一屏加顯性提示，非藏在 frontmatter |
| 2 | 層級分類本身難以界定（實作/工具/協作邊界模糊） | 30% | 提供具體對照表 + 邊界範例（引用 pm-judgment-interference-map 因子 1.4） |
| 3 | SKILL 維護者覺得麻煩，新 SKILL 忽略此欄位 | 20% | 加 lint hook 檢查 SKILL frontmatter 完整性 |

**回退計畫**：若 A-F 全部失敗，PC-111 v1.2.0 Q4 仍是最後防線。

#### Consider the Opposite（驗收條件 7 必檢）

**反論**：「SKILL / PC 現狀足夠，問題在讀者而非設計」

**支持反論的論點**：
- PC-111 v1.2.0 Q4 已讓讀者做素材層級檢查
- 讀者自檢是最終防線，不應把所有責任推給文件骨架
- 加層級標記增加維護成本，可能與 compositional-writing SKILL 的「atomic」「最小認知負擔」原則衝突

**反駁反論的論點**：
- Q4 是讀者主動檢查，前提是讀者已識別「這是跨層語境」。但 SKILL 自身若以跨層語言引導，讀者根本不會意識到需要 Q4 檢查（結構性盲點）
- W17-078 事件即為實例：PC-092 / feedback / agent-team SKILL 三素材被跨層提升前，Q1-Q3 均通過（引用檢查合格），Q4 未觸發，因為 SKILL 引導本身就是跨層的
- 加層級標記的成本與 PC-111 R5 的重現事件成本相比是長期勝出

**結論**：反論**部分成立**（Q4 確實降低再發生率），但「SKILL 本身引導跨層」的結構性問題 Q4 無法修復。因此仍需 SKILL 層級顯性化，但可**聚焦最小干預**（方向 B + C 組合優先）。
