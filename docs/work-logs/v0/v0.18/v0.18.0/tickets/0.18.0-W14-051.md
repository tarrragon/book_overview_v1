### 實驗方法

使用 Python 直接呼叫 `get_tickets_dir()` 測試不同版本號的路徑產出，模擬 migrate 寫入端行為。

### 實驗執行

**指令**：
```python
from ticket_system.lib.paths import get_tickets_dir
for v in ['0.18.0', '1.0.0', '2.0.0', '0.19.0']:
    print(get_tickets_dir(v))
```

**輸出**：
| 版本 | 產出路徑 | 是否正確 |
|------|---------|---------|
| 0.18.0 | `docs/work-logs/v0/v0.18/v0.18.0/tickets` | 正確（三層已存在） |
| 1.0.0 | `docs/work-logs/v1/v1.0/v1.0.0/tickets` | 正確（但僅因手動 mv 建立了 v1/ 目錄） |
| 2.0.0 | `docs/work-logs/v2.0.0/tickets` | **錯誤**（應為 `v2/v2.0/v2.0.0/tickets`） |
| 0.19.0 | `docs/work-logs/v0/v0.19/v0.19.0/tickets` | 正確（v0/ 目錄已存在） |

### 實驗發現

**已驗證事實**：
- `get_tickets_dir()` 在 `paths.py:99` 檢查 `hierarchical.exists() or hierarchical.parent.exists()`，若三層目錄結構不存在則 fallback
- `paths.py:103` 檢查 `major_dir.exists()`，若 `v{major}/` 不存在也 fallback
- 對全新主版本（如 2.0.0），三個條件皆為 False，落入 line 107 flat 路徑 `v2.0.0/tickets/`
- migrate 命令在 `migrate.py:279` 使用 `target_path.parent.mkdir(parents=True, exist_ok=True)` 建立目錄，但建立的是 flat 結構
- 1.0.0 目前正確僅因 commit 042bebb9 手動 mv 建立了 `v1/v1.0/v1.0.0/` 目錄

**結論**：根因在 `get_tickets_dir()` 的存在性檢查邏輯——它將「目錄尚未建立」誤判為「應使用 flat 結構」。

### 根因定位

**檔案**：`.claude/skills/ticket/ticket_system/lib/paths.py`，`get_tickets_dir()` 函式（行 69-108）。

**分支邏輯**（行 99-107）：
```python
# 行 99：僅在目錄已存在時使用三層結構
if hierarchical.exists() or (hierarchical.parent.exists()):
    return hierarchical
# 行 103：major 目錄存在也使用三層
if major_dir.exists():
    return hierarchical
# 行 107：fallback 到 flat 結構
flat = root / WORK_LOGS_DIR / versioned / TICKETS_DIR
return flat
```

**根因**：`get_tickets_dir()` 的設計假設為「三層結構目錄已預先存在」，這對 v0.x 系列成立（開發過程中手動建立），但對全新主版本（v1+, v2+）首次遷移時三個條件全部為 False，導致 fallback 到不一致的 flat 結構。

### 修復方案

**策略**：移除存在性檢查，當 `len(parts) >= 2` 時**一律**回傳三層階層路徑。flat 結構僅作為無法解析 major.minor 時的最終 fallback。

**具體修改**（paths.py `get_tickets_dir()`）：
```python
parts = bare_version.split(".")
if len(parts) >= 2:
    major = parts[0]
    minor = f"{parts[0]}.{parts[1]}"
    hierarchical = root / WORK_LOGS_DIR / f"v{major}" / f"v{minor}" / versioned / TICKETS_DIR
    return hierarchical  # 一律回傳三層，不做存在性檢查

# flat fallback 僅用於無法解析的版本號
flat = root / WORK_LOGS_DIR / versioned / TICKETS_DIR
return flat
```

**影響評估**：
- 所有正常版本號（X.Y.Z）皆有 `len(parts) >= 2`，一律走三層
- 既有 v0.x 三層目錄已存在，行為不變
- 新主版本遷移時 `mkdir(parents=True)` 會自動建立完整三層
- flat fallback 僅在極端情況（版本號格式異常如 "1"）觸發

**邊界測試案例**：
| 版本 | 預期路徑 |
|------|---------|
| 1.0.0 | `v1/v1.0/v1.0.0/tickets/` |
| 2.0.0 | `v2/v2.0/v2.0.0/tickets/` |
| 10.3.1 | `v10/v10.3/v10.3.1/tickets/` |
| 0.19.0 | `v0/v0.19/v0.19.0/tickets/`（不變） |

**是否需建 main.md 骨架**：否。migrate 已有 `target_path.parent.mkdir(parents=True, exist_ok=True)`，修復 path resolution 後目錄會自動建立。worklog 的 main.md 屬版本規劃流程，非 migrate 職責。
