## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

**直接原因**：agent 定義和 AGENT_PRELOAD.md 均未將 `ticket track check-acceptance` + `ticket track complete` 列為代理人收尾責任。

**因果鏈**（5-Why）：

1. **Why**: subagent commit 後只回報不 complete → 因為 agent 不知道自己有 complete 責任
2. **Why**: agent 不知道 → 因為 AGENT_PRELOAD.md 第 2 節「Ticket 操作規範」只覆蓋 query / append-log，無 complete 步驟
3. **Why**: AGENT_PRELOAD 漏了 → 因為歷史設計將 complete 視為 PM 專屬操作（pm-role.md 列「Ticket 生命週期操作」含 complete）
4. **Why**: PM 專屬 → 因為 acceptance-gate-hook 是 PM 端防護，設計時未考慮 agent 自律路徑
5. **Why**: 未考慮 agent 自律 → 因為早期 agent 為純執行者，無生命週期管理意識

**三維度缺口分析**：

| 維度 | 現狀 | 缺口 |
|------|------|------|
| Agent 定義（agent-definition-standard.md） | W17-016.5 新增「執行責任：Ticket body 填寫」，覆蓋 append-log 但止於 complete 前 | **無「收尾責任：check-acceptance + complete」章節** |
| AGENT_PRELOAD.md | §2 Ticket 操作只有 query / append-log；§6 執行檢查清單無 complete 項 | **無 complete 步驟** |
| Dispatch prompt 模板 | PM 派發時偶爾在 prompt 尾部加 complete 指示，但非標準化 | **無強制模板要求** |
| CLI Hook | acceptance-gate-hook 在 PM 端 complete 前觸發驗證 | Hook 設計為 PM 端防護，不影響 agent 端自律缺口 |

### 影響範圍

- 所有實作類 agent（thyme-python-developer、parsley-flutter-developer、fennel-go-developer、thyme-documentation-integrator、cinnamon-refactor-owl）
- 每次 PM 需額外執行 check-acceptance --all + complete，約 2-3 tool call 浪費
- W17-020、W17-016.3 為近期具體案例

### 相關 Error Pattern

- 無直接對應的既有 PC error-pattern（本 ticket 將產出新 PC）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 重現實驗結果

### 實驗方法與發現

靜態分析 AGENT_PRELOAD.md、agent-definition-standard.md、5 個實作類 agent 定義檔。

**已驗證事實**：
1. AGENT_PRELOAD.md 2 節「Ticket 操作規範」僅覆蓋 query / append-log，無 complete 步驟
2. agent-definition-standard.md「執行責任」止於 body 填寫，未延伸至 complete
3. 部分 agent 尾部有「優先使用 CLI 命令（ticket track complete 等）」通用提示，非結構化流程
4. acceptance-gate-hook 在 agent 自行 complete 時同樣觸發，無安全風險

---

## 根因結論

**根因**：歷史設計將 complete 視為 PM 專屬操作。W17-016.5 新增 body 填寫責任時止步於「complete 前」，未順勢將 complete 本身納入 agent 責任鏈。

**核心判斷：agent 自律 vs hook 代補 vs PM 驗收**

推薦方案：**agent 自律為主，hook 為安全網，PM 驗收降級為抽查**。

理由：
- agent 自律成本最低（每次省 PM 2-3 tool call）
- acceptance-gate-hook 已在 complete 時自動觸發（無論由 agent 或 PM 執行），安全網已存在
- PM 不需要在每個 ticket 手動 complete，只需在 handoff 時確認狀態

---

## 改善方案

### 方案 A：症狀修復（AGENT_PRELOAD.md 補 complete 流程）

**修改位置**：AGENT_PRELOAD.md

1. 2.3「更新時機」表格新增「任務完成」列：

| 時機 | 更新什麼 | 範例 |
|------|---------|------|
| 任務完成 | check-acceptance + complete | ticket track check-acceptance --all ID && ticket track complete ID |

2. 執行檢查清單新增：
- [ ] 任務完成後執行 ticket track check-acceptance --all + complete

### 方案 B：根因防護（agent-definition-standard.md 新增收尾責任）

**修改位置**：agent-definition-standard.md「執行責任」章節

在「Ticket body 填寫」之後新增「收尾責任：Ticket 完成」章節：

| 時機 | 操作 |
|------|------|
| body 填寫完畢 + commit 完成 | ticket track check-acceptance --all <id> |
| acceptance 全數通過 | ticket track complete <id> |
| acceptance 有未通過 | 在 NeedsContext 記錄缺口，不 complete |

**不適用範圍**（與 body 填寫同）：純分析型 agent 由 PM 派發時明確指示。

### 方案 C：PC error-pattern 建立

建立 PC-105（暫編）：subagent-no-complete-after-commit

| 欄位 | 內容 |
|------|------|
| 症狀 | subagent commit 後只回報 PM，未執行 complete |
| 根因 | AGENT_PRELOAD / agent-definition-standard 無 complete 收尾流程 |
| 防護 | 方案 A + B 落地後，agent 自律為主；acceptance-gate-hook 為安全網 |
| 偵測 | PM handoff 時若發現 ticket 仍為 in_progress，即為此 pattern |

### 建議執行順序

1. IMP ticket：實施方案 A + B（修改 AGENT_PRELOAD.md + agent-definition-standard.md）
2. IMP ticket：建立 PC error-pattern
3. 無需新增 hook（acceptance-gate-hook 已覆蓋安全網）
