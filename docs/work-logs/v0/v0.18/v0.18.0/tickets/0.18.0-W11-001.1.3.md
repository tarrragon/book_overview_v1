## Solution

實作內容（對應 Phase 3a strategy Phase E/F/G）：

**1. `commands/claim_verification.py` 新增 `prompt_user_decision`**

- 處理 y/n 二選一互動：接受 y/yes/空字串 → 'y'；n/no → 'n'
- `auto_yes=True` 短路直接回 'y'，不受 tty 影響（F5/F7）
- 非 tty + 無 `--yes` → 回 'n' + stderr 警告（F6）
- 最多 3 次無效輸入後 fail-closed 預設 'n'（F4）
- EOFError 降級回 'n'（支援非互動邊界）

**2. `commands/lifecycle.py` 新增 `TicketLifecycle.claim_with_verification`**

- 決策樹：
  - `skip_verify + auto_yes` 同時為真 → stderr warning 提示 --yes 被忽略
  - `skip_verify=True` → 印「已跳過」後走 `self.claim`
  - 非 tty 且無 flag → fail-closed return 1
  - `collect_ac_verifications` ValueError → 降級直接 claim
  - pairs 空（S1） → 直接 claim
  - `run_all_verifications` KeyboardInterrupt → return 130
  - `none_verifiable`（S2） → 摘要 + 直接 claim
  - `all_passed`（S4） → stderr 拒絕訊息 + return 1（不 prompt）
  - `has_failures`（S3） → render + prompt → y 則 claim / n 則 return 1
- 新增 `sys` import 與 project_root / claim_verification 的跨模組 import
- `execute_claim` 改為委派至 `claim_with_verification`，讀取 `args.skip_verify` / `args.yes`

**3. `commands/track.py` 新增 CLI flags**

- `--skip-verify` (dest=skip_verify)
- `--yes` / `-y` (dest=yes)

**4. 測試檔案 `tests/test_claim_with_verification.py`**

- Group F：7 案例（F1-F7，互動決策 + auto_yes 短路 + 非 tty 處理）
- Group G：3 案例（G1-G3，parse_ac + match_template 配對）
- Group H：12 案例（H1-H12，S1-S7 全情境 + flag 組合 H10-H12）
- Group K：2 案例（K2 YAML 損毀降級、K3 非 tty 拒絕；K1 已由 Group E test_e6 覆蓋）
- 使用 `unittest.mock.patch` 隔離 `claim`、`collect_ac_verifications`、`run_all_verifications`、`resolve_project_cwd`，避免檔案系統依賴
