### 歷史證據盤點

git log 顯示 W5-019 執行期間（2026-04-12）有多次 permissionMode 試錯：

1. `8ad3cf29` fix(agent): thyme-python-developer 加入 permissionMode: acceptEdits
2. `f2f6ab07` feat(framework): settings.local.json permissions.allow 加入 Edit/Write/Grep
3. `66a14aa5` fix(agent): thyme permissionMode 改為 bypassPermissions
4. `4021c41a` docs(PC-059): retry4 修訂

**PC-059 retry4 commit message 已揭示 retry3 失敗教訓**：
- `acceptEdits` 僅自動接受 **cwd 或 additionalDirectories** 內的編輯
- subagent 繼承主 session cwd = 主倉庫；worktree 在主倉庫外部
- 故 acceptEdits 於 worktree 場景無效
- 官方安全保證：`bypassPermissions` 下 `.claude/commands`、`.claude/agents`、`.claude/skills` 允許

### 根因三假設的證據強度

| 假設 | 結論 | 證據 |
|-----|------|------|
| A. W5-019 靠繼承主 session 權限 | **部分成立** | PC-059 文件明確說 permissions.allow 對 subagent 無效；但 retry4 commit 說 bypassPermissions 為 worktree 場景標準值——暗示 W5-019 retry4 應該成功 |
| B. bypassPermissions 值被拒 | **部分成立** | 它不是完全無效，但在 cwd 外的 `.claude/agents/` 路徑可能被視為「其他目錄仍提示」 |
| C. 機制有未文件化限制 | **強成立** | bypassPermissions 的「例外允許」是否基於路徑字面 match 或 cwd 相對，文件未明確；thyme 失敗即為此限制具象化 |

### 本次失敗的精確根因

thyme 失敗的真正原因是 **cwd 不對齊**：

- thyme 繼承主 session cwd = 主 repo（`/Users/tarragon/Projects/book_overview_v1`）
- PM prompt 指定的 Edit 目標是 worktree 絕對路徑（`/Users/tarragon/Projects/book_overview_v1-0.18.0-W5-021/.claude/agents/*.md`）
- 對 subagent 視角，worktree 是「外部絕對路徑」，不是「cwd 下的 .claude/agents/」
- bypassPermissions 的「`.claude/agents` 允許」判斷可能是基於 cwd 相對路徑識別
- 結果：Edit 被拒，符合 PC-059 retry3 描述（「worktree 在 cwd 外部」）

PC-059 retry4 聲稱 bypassPermissions 解決 worktree 場景，**但這是假命題**——它只解決 cwd 內 .claude/agents/ 的編輯。worktree 場景 **仍需 cwd 對齊**（但 IMP-056 禁止 subagent cd）。

### Claude Code 合法 permissionMode 值

來自 PC-059 文件（官方 https://code.claude.com/docs/en/sub-agents#permission-modes）：

| 值 | 行為 |
|----|------|
| `default` | 標準檢查含提示（背景自動拒） |
| `acceptEdits` | 自動接受 cwd/additionalDirectories 內的 Edit |
| `auto` | 分類器評估 |
| `dontAsk` | 自動拒（allow 清單仍有效） |
| `bypassPermissions` | 跳過大部分提示（部分目錄仍提示） |
| `plan` | 唯讀探索 |

**所有值在 subagent cwd 外的 worktree 編輯場景都不可靠**。

### 三根因假設的證據評估

**假設 A（W5-019 當時能成功是繼承主 session 權限）**：證據強度**中**。
- 17:28 的 W5-001 Phase 3b 在 permissionMode 與 settings allow 都不存在時即成功，支持「另有其他授權來源」。

- Claude Code 官方 changelog 2.0.43 明文 'Added permissionMode field for custom agents'，證明此欄位存在且生效。

**假設 C（subagent permissionMode 存在未文件化限制）**：證據強度**中**。
- 同一欄位值（bypassPermissions）在近距離兩次派發中產生不同結果，顯示存在非 frontmatter 因素。
- 可能變因：主 session 的 accept-edits 狀態、worktree vs 主 repo、tool set 交集、dispatch-active.json 內容、session resume 狀態。

### 最可能根因（綜合推斷）

**主 session 的 accept-edits 狀態是 subagent Edit 能否成功的實際決定因子**，subagent frontmatter 的 permissionMode 在目前 Claude Code 版本中可能**僅能放寬但不能覆蓋**主 session 的限制。W5-019 成功時主 session 處於 accept-edits，W5-021 失敗時主 session 未開啟或已退出該模式。這解釋：

此推斷與 memory/feedback_agent_tools_runtime_permission.md（PC-059）'permissions.allow 無效於 subagent' 的觀察一致，進一步延伸為：**subagent 的實際寫入授權來自主 session 的互動模式，而非 frontmatter 或 settings.allow**。
