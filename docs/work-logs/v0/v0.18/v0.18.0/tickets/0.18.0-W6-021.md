## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

**直接原因**：`readmoo-adapter.extractAllBooks()` 缺少 lazy load / scroll-to-end 流程，僅呼叫 `waitForBookElements()` 等待**首批**元素出現（length > 0 即 resolve），未驗證 DOM count 是否達目標總數（944）。

**設計缺口三層**（從淺到深）：

1. `waitForBookElements()` resolve 條件：`getBookElements().length > 0` 即返回（adapter line 282-292）；無「等到 count 達 target N」邏輯。
2. `extractAllBooks()` 直接消費 `bookElements`，無迴圈 / 滾動 / 重試（adapter line 580-617）。
3. `content-modular.js` 進入點未串接任何 SPA 路由監聽（`onURLChange` 在 `event-utils.js:767` 定義但 `content-modular.js` 未調用），路由切換返回 library 時 content script 只看到當下 DOM snapshot。

**對照 docs/bookstores/readmoo.md §虛擬 scroll / lazy load**：文件明確指出「完整提取需 scroll 觸發 lazy load（透過底部「更多...」按鈕或 scroll-to-end）。提取流程應等待 `.library-item` 數量穩定於目標總數」。MCP E2E checklist Step 3（line 248-265）甚至提供現成 evaluate_script 範本（最多 30 輪 scroll + 點「更多」按鈕）。**規格已明定，但 src/content/ 從未實作。**

### 影響範圍

- **直接影響**：`src/content/adapters/readmoo-adapter.js`（缺 scroll-to-end loop）、`src/content/content-modular.js`（缺路由監聽接線）。
- **功能影響**：
  - 場景 A（直接 reload library）：依書庫大小不同，首批 96 後使用者需手動 scroll 才看到更多；提取結果可能只有 96/944（約 10%）。
  - 場景 B（SPA 路由切換返回 library）：MutationObserver 在 96 命中後 resolve，提取同樣只有 96。
  - 場景 C（小書庫 < 96 本）：無感（首批即全部）。
- **波及檔案數**：核心 2 檔（adapter + content-modular），相關 utils 已備（event-utils.onURLChange、MCP checklist 演算法）。

### 相關 Error Pattern

未發現直接相關的 PC error-pattern。`feedback_extraction_fault_tolerance` memory 提到「提取流程容錯設計，多策略 fallback」但未涵蓋此 lazy load 場景。

### 實驗發現

**已驗證的事實**：

1. `extractAllBooks()` 無 scroll-to-end loop，無「等到 count 達 target N」邏輯。
2. `content-modular.js` 未串接 SPA 路由監聽。
3. `waitForBookElements()` 設計為「等到至少一個元素出現」而非「等到 lazy load 完成」。
4. W6-012.9.4 實機觀察「count=96 而非 944」為**真實提取缺口**，非 ANA 設計偏差或 MCP 工具誤差。
5. `docs/bookstores/readmoo.md` §虛擬 scroll / lazy load 已書面規格化 scroll-to-end 流程（包含現成演算法範本），但 src/ 從未實作。

**判定（對應 ticket 列 4 個選項）**：

| 選項 | 判定 |

| (c) 用戶實際操作不會走此路徑（低風險） | 不成立（場景 A「直接 reload」即觸發） |
| (d) 補修復（spawn IMP） | **採此選項** |

**未驗證假設**（不影響本 ANA 結論，留作 IMP 階段驗證）：

- scroll-to-end 完成判定條件的最佳閾值（穩定 3 輪無變化？對照 total-count text？兩者取交集？）。
## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 分析結論

**S4 路由切換命中 96 並非「路由切換特有」問題，而是提取流程**普遍**缺少 lazy load 處理**。場景 A（直接 reload）會同樣命中 96/944，只是 W6-012.9.4 ANA 沒把 reload 場景的提取結果與書庫總數對照，所以漏看。

**根因**：`docs/bookstores/readmoo.md` §虛擬 scroll / lazy load 規格已書面定義，但 src/content/ 從未實作對應流程（adapter 與 content-modular 雙缺）。MCP E2E checklist Step 3 的 scroll-to-end 演算法只用於「手動驗證 e2e 流程」，未被 production extraction code 採納。

**決策：補修復（spawn IMP）**。
