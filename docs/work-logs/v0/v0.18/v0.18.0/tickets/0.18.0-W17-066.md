#### 二、Good Taste 改進空間

##### 改進 1（核心）：把 Hook + Agent 重新分層，消除「特殊情況」

當前設計把五職責塞給單一 agent 是「八個特殊情況變一個通用流程」**的反向操作**——把可規則化的兩項（職責三、四）跟需語意判斷的三項（職責一、二、五）綁在一起，因為「都是文字品質」就放同一籃。這是分類錯誤。

正確分層：

| 層 | 載體 | 職責 | 觸發 |
|---|------|------|------|
| L1 同步阻擋 | Hook（既有 charset hook 擴充） | 禁用字、簡體字、emoji、Unicode escape 錯字形 | 寫檔時、commit 時 |
| L2 非同步審查 | basil-writing-critic | 三明示、資訊優先序、正面陳述 | parallel-evaluation 情境 D/F/G 加入 |
| L3 兜底 fallback | basil 也掃 L1 規則 | L1 漏網的歷史檔案（rules/methodologies 既存） | 審查既有文件時 |

**Why**：規則化的東西用 grep 不用 LLM 是 Linus 80 年代以來的常識；把 grep 級別工作丟給 opus 既浪費 token 又會引入 LLM 幻覺風險（agent 可能漏掃）。
**Consequence**：現設計把 L1/L2 混在 LLM 的後果是：(a) opus token 浪費在 grep 工作；(b) 同步阻擋變非同步審查，違規檔案可能先 commit 才被發現；(c) Hook 失能訊號被 agent 表面通過遮蔽。
**Action**：IMP 階段先建 L1 Hook ticket（擴充 charset hook 到 emoji + Unicode escape codepoint 比對），basil 改聚焦三明示、資訊優先序、正面陳述（從五職責減為三職責）。

##### 改進 2：扁平化「常駐委員」概念，避免擴張

§3.1「basil 為第二位常駐委員，與 linux 並列」是危險先例。常駐委員越多，每次 parallel-evaluation 派發成本越高，而「常駐」的字面意義就是無 opt-out。如果未來再來「commit message 常駐委員」「測試品質常駐委員」「i18n 常駐委員」，PM 每次 review 派 8 個 agent。

**Linus 觀察**：linux agent 之所以能無條件常駐，是因為 Good Taste / 架構複雜度是**所有產出**的元維度。basil 的「文字品質」也接近元維度但**不完全是**——情境 A/B/E 標的是程式碼，basil 在場無事可做（§3.2 自己也承認）。

**Action**：把「常駐委員」概念拆成 (1) `universal_lens`（linux 唯一），(2) `default_lens_per_scenario`（basil 列入 C/D/F/G 預設但允許 opt-out）。架構詞彙的清晰會防止後人繼續加「第三位常駐」。

##### 改進 3：Solution §7.2 spawned_tickets 表的 ID 命名違反既有規範

§7.2 列「W17-050.1 / W17-050.2 / W17-050.3 / W17-050.4」屬於 sub-id 命名。既有 ticket 系統使用 `{version}-W{wave}-{seq}` 序號（W17-051、W17-052 等），不用 `.subseq`。雖屬 §7.2 範疇（PM 在 complete 時建立），但作為 ANA 建議若被照抄會造成命名污染。**註**：本項屬輕度建議（W17-050 已實際使用 W17-056~059 + W17-066，未照抄 §7.2 命名），但 IMP 階段應更新 §7.2 使用真實 ID 避免後人考古困惑。

#### 1.1 document-writing-style v1.2.0 覆蓋（warning）

basil 五大職責涵蓋 document-writing-style 的核心要求，但「反模式：人性化/隱含表達」章節（六個反模式句型：「希望讀者理解」「按理應」「通常來說」「規則寫得好卻沒人用」「假設讀者會注意到」「理想情況下」）未被任何職責明確涵蓋。

- 職責一（三明示）覆蓋「三明示」與「二次審查清單」
- 職責二（資訊優先序）覆蓋「核心原則二」
- 職責五（正面陳述）覆蓋「負向對比有正向錨點」
- **缺口**：隱含表達偵測（條件模糊、責任外推）與三明示互補但不等同——三明示要求每段有 Why/Consequence/Action，隱含表達偵測的是「按理應」「通常來說」等模糊條件詞

**修正建議**：IMP 階段將隱含表達反模式偵測併入職責一（三明示）作為子項，或獨立為職責六。成本低（清單已存在於 document-writing-style.md），效益高（隱含表達是 PC-066 壓力場景下最易出現的違規類型）。

#### 1.2 compositional-writing 三大支柱五大原則覆蓋（info）

basil 五大職責覆蓋三大支柱中的 Explicit Intent（意圖顯性，職責一二五），部分覆蓋 Atomization（職責一間接涵蓋），但未覆蓋：

- **原子化（Atomization）**：「一段只承載一個概念」的獨立審查未明示
- **可查詢性（Searchability）**：「關鍵字前置、grep 友善格式」未涵蓋
- **欄位設計（Field Design）**：「what/why/acceptance 不混淆」未涵蓋

此三項在實務中較難由審查者執行（需知讀者搜尋意圖、需知欄位設計背景），建議 IMP 階段評估是否列為選用審查子項而非強制。不構成 critical 缺陷，因 basil 的核心價值在於執行「已有規則但無執行者」的缺口，五大職責已涵蓋最高頻違規面。

#### 1.3 parallel-evaluation 整合（info，無衝突）

W17-050 提案的情境差異化策略（C/D/F/G 強制，A/B/E 選用）與既有 SKILL.md 的「常駐委員」機制一致。目前 SKILL.md 寫「所有情境自動加入 linux」，basil 採部分情境強制是合理的差異化——linux 審架構（所有情境都涉及），basil 審文字（僅書面產出情境涉及）。

需注意：SKILL.md 需修改「常駐委員」段落以容納「分情境強制」的第二種模式，目前段落只描述「全情境自動加入」。此修改已列入 W17-050 落地清單項目 2，無遺漏。

#### 1.4 language-constraints 禁用字偵測重疊（info，互補無衝突）

#### 1.5 ai-communication-rules 規則 5 關係（info，無衝突）

規則 5（權力不對等下的對話品質）聚焦 PM 與用戶的互動，basil 審查的是代理人產出的書面文字，兩者維度正交。basil 不需要審查「權力不對等」維度，因其標的不涉及 PM-用戶對話。
