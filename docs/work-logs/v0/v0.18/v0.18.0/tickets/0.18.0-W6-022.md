### 問題根因

`_apply_context_resume` 對 handoff JSON 僅解析 `direction` 字串，未讀取 W17-164 寫入端落地的頂層 `target_ticket_id` 欄位。`direction=context-refresh` 不屬於 `is_task_chain_direction` 認定的任務鏈方向，落入 else 分支將 source ticket_id（W6-012, status=completed）加入候選集；但 `tickets` 來自 `list_tickets(version)` 經 `_is_listable` 過濾僅含 `pending + blockedBy=[]` 的 ticket，導致 candidate 與 tickets 交集為空。

### 影響範圍

- 檔案：`.claude/skills/ticket/ticket_system/commands/track_runqueue.py`（單一函式 `_apply_context_resume`）
- 功能：PM 裸 `/ticket` 路由依賴 `runqueue --context=resume`，當 handoff direction=context-refresh + target_ticket_id 存在時會誤判「無 resume 候選」，與 `resume --list` 結果不一致 → PM 漏接 cross-ticket handoff

### 相關 Error Pattern

無直接對應的 error-pattern；本案屬於「寫入端先落地、讀取端未跟進」典型 follow-up 漏鉤（W17-164 完成 writer，W6-022 補 reader）。
## 觀察事實

本 session 啟動時 SessionStart hook 顯示「待恢復任務：1. 0.18.0-W6-012」，但 PM 跑裸 `/ticket` 路由執行 `ticket track runqueue --context=resume --top 3` 卻回報「無 resume 候選；當前無 handoff pending ticket」。`ticket resume --list` 同時段可正確找到該 handoff。

## Handoff JSON 內容（.claude/handoff/pending/0.18.0-W6-012.json）

- `ticket_id`: "0.18.0-W6-012"（source，status=completed）
- `direction`: "context-refresh"
- `target_ticket_id`: "0.18.0-W13-001"（W17-164 落地的絕對指向欄位）
- `from_status`: "completed"

## 根因定位

檔案：`.claude/skills/ticket/ticket_system/commands/track_runqueue.py` L262-284 `_apply_context_resume`

```python
for source_id, info in handoff_info.items():
    direction = ((info or {}).get("direction") or "") if isinstance(info, dict) else ""
    if is_task_chain_direction(direction):
        target = extract_direction_target_id(direction)
        candidate_ids.add(target if target else source_id)
    else:
        candidate_ids.add(source_id)   # ← bug：context-refresh 落這支
return [t for t in tickets if t.get("id") in candidate_ids]
```

`direction="context-refresh"` 不屬於 `is_task_chain_direction` 認定的任務鏈方向（僅 to-sibling/to-parent/to-child），落入 else 加入 `source_id`=W6-012。但 `tickets` 來自 `list_tickets(version)` 經 `_is_listable` 過濾僅含 pending+blockedBy=[]，W6-012 status=completed 被濾掉 → candidate_ids 與 tickets 交集為空。

## 為何 resume --list 沒受影響

`resume --list` 直接列舉 handoff JSON 檔案內容，不經 `list_tickets` 的 status 過濾，故能呈現。

## 修復方向（與 acceptance 對齊）

W17-164 已在 handoff JSON 寫入 `target_ticket_id` 欄位作為絕對指向。`_apply_context_resume` 應優先採用該欄位：

```python
target_explicit = (info or {}).get("target_ticket_id")
if target_explicit:
    candidate_ids.add(target_explicit)
    continue
# fallback 既有 is_task_chain_direction 邏輯
```

### 修復摘要

`_apply_context_resume`（`.claude/skills/ticket/ticket_system/commands/track_runqueue.py`）在解析 direction 之前，先檢查 handoff JSON 頂層的 `target_ticket_id`（W17-164 落地的絕對指向欄位）：

- 若 `target_ticket_id` 為非空字串，直接加入 candidate_ids 並 `continue`。
- 否則 fallback 既有 `is_task_chain_direction` / `extract_direction_target_id` 邏輯，保持向後相容。

修復後 `runqueue --context=resume` 與 `resume --list` 對 `direction=context-refresh + target_ticket_id` 形式 handoff 的結果集一致。

### 設計決策

| 議題 | 決策 |
|------|------|
| 為何優先 target_ticket_id？ | W17-164 已將其定義為「絕對指向」欄位語意，優先級高於 direction 字串解析（後者僅作為向後相容 fallback） |
| target_ticket_id 空字串/None 處理 | 視為缺失，走原 direction 邏輯，避免將空候選加入結果集 |
| 是否同步修改 resume --list？ | 否；resume --list 直接讀 handoff JSON，本來就會優先使用 target_ticket_id，無需變更 |

### 不在本 ticket 範圍

- handoff JSON 寫入端（W17-164 已完成）
- resume CLI 行為（未受影響）
