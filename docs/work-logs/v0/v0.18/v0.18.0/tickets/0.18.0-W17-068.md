## W17-068 實作摘要（4 處修改）

### 修改 1：.claude/hooks/language-guard-hook.py — 新增 emoji + 隱含表達 6 句型偵測

新增三項：
- `EMOJI_RANGES` 常數（與 askuserquestion-charset-guard-hook.py 對齊 6 個範圍）
- `find_emoji_chars(text)` — 掃描 emoji codepoint，去重後回 [(char, cp), ...]
- `find_implicit_expressions(text)` — 掃描 7 個隱含表達 pattern（document-writing-style v1.2.0 反模式表 6 條，「按理應/自然而然」「通常來說/一般情況下」拆為獨立 pattern）

`IMPLICIT_EXPRESSION_PATTERNS` 7 個句型（每個含修正方向 hint）：
- 希望讀者理解 / 按理應 / 自然而然 / 通常來說 / 一般情況下 / 假設讀者會注意到 / 理想情況下

`main()` 在偵測到前一輪 assistant 回應違規時，三類分別寫 stderr + logger.warning（規則 4）。
**警告非阻擋**：exit 0（符合 PCB §自我驗證重點：語意需 basil 兜底）。

### 修改 2：.claude/hooks/askuserquestion-charset-guard-hook.py — 新增 PC-085 字形混淆 self-check

新增三項：
- `CONFUSABLE_PAIRS` 常數：26 組繁/簡/日新字體相鄰 codepoint 對照表，每組 4 元組 (trad_cp, simp_cp_or_None, jp_cp_or_None, gloss)
- `validate_confusable_pairs_consistency()` — 啟動 self-check 函式，回傳 dict 含 critical / info 雙鍵
- `main()` 啟動時呼叫 self-check：critical 級寫 stderr + logger.error（清單設計筆誤），info 級僅 logger.info（漏網提示，清單漸進擴充屬預期）

四檢查規則：
- 規則 1（info）：簡體 cp 應在 SIMPLIFIED_CHARS（漏網）
- 規則 2（critical）：繁體 cp 不應在 SIMPLIFIED_CHARS（PC-085 首發案例「遺/遗」筆誤）
- 規則 3（critical）：日文新字體 cp 不應在 SIMPLIFIED_CHARS（PC-074 教訓）
- 規則 4（critical）：繁日共用字（trad == jp）不應在 JAPANESE_ONLY（PC-084 禁入）

### 修改 3：隱含表達 6 句型偵測（合併入修改 1）

實作位置：language-guard-hook.py，PCB 明示「警告非阻擋」設計。
語料來源：.claude/rules/core/document-writing-style.md L115-L120。
偵測機制：每次 UserPromptSubmit 掃描前一輪 assistant 回應，命中即列舉 pattern + 修正方向。

### 修改 4：新增 hook 測試（45 個新測試）

**.claude/hooks/tests/test_language_guard.py（新建，35 個測試）**：
- 類別 A：既有韓文/日文假名偵測回歸（5 tests）
- 類別 B：W17-068 emoji 偵測（9 tests，含 6 個 codepoint 範圍各別驗證 + 去重 + 邊界）
- 類別 C：W17-068 隱含表達 6 句型偵測（13 tests，每句型 1 test + multi-pattern + clean negative + hint 結構）
- 類別 D：子行程整合（8 tests，驗證 exit 0 + stderr 雙通道 + 警告非阻擋契約）

**.claude/hooks/tests/test_askuserquestion_charset_guard.py（擴充類別 H，10 個測試）**：
- H1-H5：CONFUSABLE_PAIRS schema + PC-085 首發案例「遺/遗」+ PC-084「鑑」+ PC-074「讀/读/読」三胞胎驗證
- H6-H7：dict schema 契約 + 既有清單通過 critical 級檢查
- H8-H10：3 種注入污染情境偵測（繁體誤入 / 漏網 info / 繁日共用字誤入 JAPANESE_ONLY）
- H11：雙通道隔離契約（info 不污染 critical）

### 設計決策記錄

1. **CONFUSABLE_PAIRS 設計為 4-tuple schema（含 gloss 字形說明）**：遵循 PC-085 §防護措施 "同行註解必須含正體中文字元且字形可肉眼辨識" 行為準則。

2. **self-check 分 critical / info 雙級（非單級 warning list）**：
   - 原設計單級會在每次 hook 執行吐 12 行漏網警告，污染 stderr 雙通道契約（test_violating_payload_stderr_minimal_or_empty 失敗）
   - PC-074 教訓：清單漸進擴充屬預期狀態（只收實際遇過字），漏網不應升為 stderr 警告
   - critical 級僅在「設計筆誤」時觸發（規則 2/3/4）

3. **隱含表達 7 pattern（非 6）**：document-writing-style 表格列 6 列，但「按理應/自然而然」「通常來說/一般情況下」分別同列含 2 模式，故拆成獨立偵測項保持精確（hint 共用同列指引）。

4. **效能驗證**：純邏輯 benchmark
   - language-guard 三類偵測：0.21 ms/次
   - PC-085 self-check：0.01 ms/次
   - charset scan_payload：0.03 ms/次

   subprocess 測試的 100-300ms 為 Python interpreter cold start 系統開銷（與 hook 邏輯無關），純邏輯遠低於 100ms 預算。

### 規範遵守

- **規則 4（觀測性）**：所有偵測雙通道輸出（stderr 警告 + logger 持久化）
- **agent-definition-standard**：未修改 frontmatter，僅擴充三區塊內容
- **Python 3.9 相容**：使用 Optional/List/Tuple，未使用 PEP 604
- **hook_utils 統一**：所有 logger 透過 setup_hook_logging，stdin 透過 read_json_from_stdin
- **PC-085 同行註解規範**：CONFUSABLE_PAIRS 每筆含 gloss 字形說明，避免後續維護筆誤
