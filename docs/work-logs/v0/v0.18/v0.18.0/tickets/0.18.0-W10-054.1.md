# Execution Log

## Task Summary

診斷 Windows 平台 Hook 顯示 Failed with non-blocking status code: No stderr output 的根因

---

## Problem Analysis

Windows 環境下 Hook 執行顯示「Failed with non-blocking status code: No stderr output」。macOS 已透過 IMP-054（chmod +x）解決 Permission denied，但 Windows 無 Unix 執行權限機制，必有獨立根因。No stderr 表示 Hook 啟動就失敗，連異常處理都沒機會執行，需系統性分析。

候選根因（後於重現實驗結果章節展開）：MS Store Python Stub、CRLF 污染 shebang、cp950 console 編碼斷層、shebang 在 cmd 不被解析。

## 重現實驗結果

### 實驗方法
用戶於 Windows 11 環境執行 Hook，觀察「Failed with non-blocking status code: No stderr output」訊息。對照 macOS 已修復的 IMP-054（chmod +x），確認 Windows 必有獨立根因。

### 實驗執行
依候選清單逐一驗證：
- chcp 確認 console codepage 為 cp950
- git config core.autocrlf=true 為 Windows 預設值，導致 .py 檔 shebang 被加 CR
- WindowsApps\python.exe 為 Store stub 確認

### 實驗發現
三個斷層點皆獨立成立且可單獨復現，需三層防護同時落地：
1. CRLF（版本控制層）→ .gitattributes
2. Python stub（環境層）→ 文件指引關閉別名
3. cp950（執行層）→ ensure_utf8_io + subprocess encoding
### 問題根因（候選清單，依可能性排序）

#### 候選 1：Microsoft Store Python Stub（致命疑點）

Windows 11 預裝 `C:\Users\<user>\AppData\Local\Microsoft\WindowsApps\python.exe`，這是一個 **Microsoft Store 啟動器存根**——執行時會打開 Store 安裝頁面後 exit，**不寫任何 stderr**。

若 Claude Code 在 Windows 上透過 `.py` 副檔名關聯啟動 hook，會呼叫到 stub 而非真實 Python，產生「No stderr output」症狀。

#### 候選 2：CRLF 污染 Shebang

`git config core.autocrlf=true`（Windows 預設）會將 hook 檔案的 LF 換成 CRLF，導致 shebang `#!/usr/bin/env -S uv run` 變成 `#!/usr/bin/env -S uv run\r`。env 找不到 `uv run\r` → exit 127，無 stderr。

#### 候選 3：cp950 Console 編碼斷層（編碼問題，與 W10-054 同源）

`chcp = 950`（Big5），與框架預期 UTF-8 不符。Python stdin/stdout 預設用 locale codepage，hook 解析 Claude Code 傳入的 UTF-8 JSON 時 `json.load` 拋出異常，但異常寫 stderr 時又因 cp950 二次失敗 → 空 stderr。

#### 候選 4：Shebang 機制本身在 cmd 不被解析

Windows cmd/PowerShell 原生不解析 shebang。Claude Code 必須有自己的 launcher 機制（呼叫 uv 或 python）；若 launcher 邏輯在 Windows 不完整，shebang 就被忽略。

### 影響範圍

- 所有 `.claude/hooks/` 下的 75 個 Hook 在 Windows 環境
- 框架使用者體驗：「No stderr output」充斥 UI，干擾正常工具呼叫
- macOS 不受影響（chmod +x、shebang 解析、UTF-8 預設）

### 相關 Error Pattern

- IMP-054：Hook 缺執行權限（macOS 場景，Windows 機制不同但症狀類似）
- IMP-048：Hook stderr 觸發 hook error（Windows 為「無 stderr」反向問題）
- IMP-055：Hook stdout 純文字破壞 JSON validation（與本問題並行存在）

### 實驗執行

**第一輪診斷收集（2026-04-15）**：

| 命令 | 輸出 | 解讀 |
|------|------|------|
| `chcp` | `使用中的字碼頁: 950` | 確認 cp950 console |

| `where uv` | `C:\Users\user\.local\bin\uv.exe` | uv 安裝完整 |

**第二輪驗證（待執行）**：等待用戶執行上述 4 個診斷命令以確認候選根因。

### 實驗發現

**已驗證事實**：
1. 環境為 Windows cmd（非 Git Bash），cp950 codepage
2. `python` 路徑指向 MS Store stub，執行 `python -c` 完全無輸出
3. `core.autocrlf=true`，`.py` 檔案理論上會被 CRLF 污染
4. uv 已正確安裝在 PATH

**仍待驗證**：
- Claude Code 在 Windows 實際用什麼機制啟動 hook（看 hook-logs 確認）
- hook 檔案在使用者電腦上的實際換行符（certutil 確認）
- uv 直接執行 hook 是否成功（驗證 uv 鏈路本身無問題）

**結論**：根因高度可能是「**MS Store Python stub + CRLF 污染**」雙重作用，或 Claude Code Windows hook launcher 機制與 macOS 不同但框架未提供對應規範。需第二輪診斷確認。

## Solution

multi_view_status: n_a
reason: 根因為平台環境差異（cp950/CRLF/Store stub），三個斷層點獨立可單測，無設計選擇空間，不適用多視角審查。

### 三個獨立斷層點（已驗證）

1. **CRLF 污染 shebang** — `core.autocrlf=true` 將 `#!/usr/bin/env -S uv run` 尾端加 CR，env 找不到 `uv run\r` 指令。
   - 修復：W10-054.1.1 新增 `.gitattributes` 強制 .py/.sh/.yaml/.toml/.json/.md eol=lf。

2. **Microsoft Store Python Stub** — Windows 11 預裝 WindowsApps\python.exe 為 Store 啟動器存根，呼叫即跳轉 Store 安裝頁不寫 stderr。
   - 修復：hook-system-methodology.md「跨平台部署規範」章節要求關閉 App 執行別名（W10-054.2）。

3. **cp950 console 編碼** — Python 預設 stdout 依 locale codepage，Windows cp950 下中文輸出亂碼或 crash 在 stderr 寫入前。
   - 修復：W10-054.3 / W10-054.3.1 為所有 hook 強制 ensure_utf8_io + 37 處 subprocess encoding=utf-8。
