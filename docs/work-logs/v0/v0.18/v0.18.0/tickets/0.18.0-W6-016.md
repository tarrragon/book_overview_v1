### 問題根因

**Warning 類型**：esbuild `commonjs-variable-in-esm`
**根因**：兩個檔案在 ES module（含頂層 `export` 宣告）中同時使用 CommonJS 的 `module.exports`，造成雙模式 export 模式（dual CJS/ESM）。esbuild 將檔案識別為 ESM（因有頂層 `export`），但 `module.exports = ...` 在 ESM context 中 `module` 是全域變數而非 CJS module 物件，可能不按預期運作。

**真實 Warning 位置（與 ticket title 行號不完全一致）**：

| 檔案 | 真實 warning 行 | ticket title 標註 | 程式碼 |
|------|---------------|-----------------|--------|
| src/core/messages/MessageDictionary.js | 372-373 | 372/379 | `if (typeof module !== 'undefined' && module.exports) { module.exports = { MessageDictionary, GlobalMessages } }` |
| src/core/logging/Logger.js | 476-477 | 414/424 | `if (typeof module !== 'undefined' && module.exports) { module.exports = { Logger, LOG_LEVELS } }` |

> ticket title 的 414/424 不是 Logger.js 的 warning 行；414 是 `name: this.name`、424 是註解列。真實 module.exports 位於 476-477。建立 ticket 時行號標錯（W6-012.1 source 估算）。

### 影響範圍

**Source 檔案**（2 個）：
- src/core/messages/MessageDictionary.js:372-380（雙模式 export pattern）
- src/core/logging/Logger.js:476-487（同上 pattern）

**入口載入鏈**：
- background.js（ESM bundle）：直接 `import { Logger }` / `import { GlobalMessages }`（行 18-19）→ ESM path，`module.exports` 不會執行（runtime `module` undefined，guard `typeof module !== 'undefined'` 為 false）。
- popup.js（IIFE bundle）：使用 `require('src/core/logging/Logger')`（行 35, 58）→ 透過 esbuild bare-specifier alias 解析；popup 為 IIFE 格式但 source 檔仍以 ESM 形式編譯，`require` 由 esbuild 在打包時內聯。
- content.js：未直接 import 這兩檔（但 transitive 依賴可能存在，未深掃）。

**Warning 次數**：8 次（4 個 source × 2 入口 bundle 重複；建置產物 build/development/src/.../ 下也出現一次，bundle 在多入口時重複 inline 該檔案各觸發一次）。

### 相關 Error Pattern

- `.claude/references/chrome-extension-quickref.md`「CJS/ESM 雙模式」測試環境差異欄：「模組需同時支援 module.exports 和 export」——歷史設計為 Jest（jsdom + CJS）和 Chrome（ESM）共用 source 的妥協。
- 無對應 PC error-pattern；屬「歷史 CJS/ESM 雙模式遷移殘留」非反模式。

### 決策

**結論：不立即修復；spawn 後續 IMP ticket 處理為宜（P3 / 低優先）。**

#### 三選項評估（WRAP - Widen）

| 選項 | 內容 | 機會成本 | 風險 |
|------|------|---------|------|
| A. 立即修復（單行可解） | 移除 `if (module) { module.exports = ... }` 分支，僅保留 ESM `export { ... }` | 可能破壞 Jest 測試（CJS-mode require）| 中—需同步驗證所有測試通過 |
| B. spawn 後續 IMP ticket | 建立獨立 ticket 完整評估 Jest/Chrome 雙環境後再清理 | 延後處理時間 | 低—純技術債，warning 不影響 v0.18.0 release |
| C. 接受 warning 不處理 | 視為 known issue，加 `--log-override:commonjs-variable-in-esm=silent` 抑制 | 失去未來提醒 | 低—但隱藏問題 |

**選 B 理由**：
1. 修復**非單行可解**：兩檔的 dual-mode pattern 是歷史 Jest CJS + Chrome ESM 共存設計（見 `.claude/references/chrome-extension-quickref.md`「CJS/ESM 雙模式」）。直接移除 `module.exports` 可能破壞既有 Jest 單元測試。
2. 真實修復範圍：需評估 Jest config（`testEnvironment` / `moduleFileExtensions` / Babel transform）是否能讓 Jest 直接消費 ESM `export`，這是跨多檔的架構決策。
3. v0.18.0 為「測試重寫與品質恢復」階段（CLAUDE.md §8 里程碑），warning 不阻擋 release。
4. 機會成本：修復需 1-2h 包含跑全測試套件，超出 W6 Wave 容量；建議 v0.19.0 或 v0.20.0 處理。
