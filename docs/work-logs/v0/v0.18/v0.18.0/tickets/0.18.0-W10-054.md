## Task Summary

診斷 Hook 與 Skill CLI 在 Windows 平台執行時中文亂碼根因，補齊跨平台編碼規範並修復受影響的 Hook

---

## Problem Analysis

### 問題根因

Windows console 預設 codepage 為 OEM (cp950/cp936)，與框架預期 UTF-8 不符；Hook subprocess 呼叫多未指定 encoding，Python stdout 預設依 locale。並行斷層：CRLF 污染 shebang、Microsoft Store Python Stub。

### 影響範圍

- `.claude/hooks/` 75 個 Hook 在 Windows 環境
- `.claude/skills/` CLI 在 Windows console
- hook-system-methodology.md 缺跨平台部署規範
- hook-architect-technical-reference.md 缺 UV 單檔模式 UTF-8 範本

### 相關 Error Pattern

- IMP-062 windows-hook-encoding-startup-failure（本 ticket 落地）
- IMP-067 windows-git-mode-loss-on-new-files
- IMP-054 hook permission denied（macOS 同源症狀）

---

## 重現實驗結果

### 實驗方法

於 Windows 11 cmd 環境執行 `chcp` / `where python` / `git config core.autocrlf` / 直接 uv run hook，比對 macOS 預期。

### 實驗執行

| 命令 | Windows 觀察 | macOS 對照 |
|------|------------|-----------|
| chcp | 950 (Big5) | UTF-8 |
| where python | WindowsApps stub | /usr/local/bin/python |
| git config core.autocrlf | true | input |
| 中文輸出 | 亂碼 / 空 stderr | 正常 |

### 實驗發現

三個獨立斷層點皆獨立可單獨復現：
1. cp950 console（執行層）→ ensure_utf8_io + subprocess encoding
2. CRLF 污染 shebang（版本控制層）→ .gitattributes 強制 LF
3. MS Store Python stub（環境層）→ 文件指引關閉 App 執行別名
