### 品質不足 PROP 共通模式（強制機制設計依據）

#### 模式 1：Spec 盤點類 PROP 系統性低品質（PROP-001/002/003）

3 個「source: spec」的 PROP 全部低品質，共通特徵：
- 長度 < 60 行（遠低於平均）
- 僅 In/Out Scope + AC，無替代方案、無失敗防護、無機會成本分析
- 全部 draft 狀態停滯（2026-03-30 迄今無進展）
- 缺少觸發時機與優先級論證

**根因**：批次盤點時「為盤點而建提案」，未做個別評估投入。

**強制機制對策**：spec 盤點類 PROP 應走**輕量模板**（只需 AC 和單一版本），或要求「盤點發現 > 實際提案」兩階段分離。

#### 模式 2：跨版本影響 PROP 未觸發加強評估（PROP-007/008）

跨版本提案本應是「重量級」（跨 3-4 個大版本影響 UC 與 model），但：
- PROP-007 本端摘要 52 行，依賴 APP 端完整版，本專案端無獨立評估
- PROP-008 雖有 322 行但缺少「若 APP 端改動導致對齊成本暴增」的失敗防護
- 兩者 ticket_refs 都為空，confirmed 後無下游追蹤

**強制機制對策**：跨版本/跨專案 PROP 必須有本端的 Reality Test（不只引用 APP 端）。

#### 模式 3：WRAP 執行深度不一（PROP-009 單方向 vs PROP-010 完整 6 候選）

兩個 framework 類 PROP 質量懸殊：
- PROP-009（102 行）：只有單一方向的 3 面向分析，無「不這樣做可以怎樣做」的替代方案
- PROP-010（258 行）：完整 6 候選、每候選獨立評估表、MVP 組合、風險應變、與相關提案關係對照

**強制機制對策**：framework 類 PROP 必須明示「候選方案數 >= 3」，防止單方向論證。

#### 模式 4：狀態流轉與 ticket_refs 不一致（PROP-002/007/008/009）

4 個 confirmed/approved PROP 的 ticket_refs 實況：
- PROP-002：confirmed 但 W1-001 僅為分析 ticket，實作未啟動
- PROP-007：confirmed 但 ticket_refs 為空
- PROP-008：confirmed 但 ticket_refs 為空（Phase 1 應該已派生 ticket 但未派生）
- PROP-009：approved 但 ticket_refs 為空

**強制機制對策**：confirmed/approved 狀態必須綁定至少 1 個實作 ticket，否則回退為 discussing 或要求補建 ticket。

#### 模式 5：Reality Test 章節完全缺席（10/10 PROP 都無此章節）

即便高品質 PROP（PROP-004/010）也沒有明確的「Reality Test」章節，雖然 PROP-004 事實上做了「待確認→確認」驗證、PROP-010 有觸發案例，但這是敘事中自然帶出，不是強制結構。

**強制機制對策**：模板層級新增「Reality Test / 觸發案例實證」章節作為必填，讓假設與現況對照顯性化。

### 關鍵發現摘要

1. **評估倒置驗證成立**：40% PROP 為低品質，20% 高品質；且「重量級提案」（跨版本/跨專案）反而評估更薄弱
2. **spec 盤點批次生成是系統性問題**：PROP-001/002/003 同一日產出且全部停滯，需區分「盤點清單」與「正式提案」
3. **WRAP 未被強制**：10 個 PROP 中僅 PROP-010 做完整 WRAP（6 候選）；高品質與 WRAP 使用高度相關
4. **狀態與實作脫鉤**：4/7 個 confirmed+ PROP 無實作 ticket，狀態機制形同虛設
5. **Reality Test 無結構位置**：既有模板無此章節，導致即便做了也無法被稽核
## WRAP 四階段分析（PM 前台執行）

### W（Widen）—— 強制機制候選方案

| 候選 | 機制 | 優點 | 缺點 |
|------|------|------|------|
| 1. 純規則層 | pm-rules/proposal-evaluation-gate.md 強制 PM checklist | 低成本、靈活 | PM 自覺仍可能略過 |
| 2. 純 Hook 層 | PreToolUse 攔截 docs/proposals/ Write 未通過評估者 | 強制力高 | 可能誤觸小提案（拼字修正） |
| 3. 規則+Hook 雙重 | 規則定義流程 + Hook 自動化強制 | 防護層次完整 | 實作成本最高 |
| 4. 命令層封裝 | /propose 命令封裝完整評估流程 | UX 整合度最高 | 只攔截命令不攔截直接 Write |
| 5. 分級強制 | 輕量提案只要 Reality Test；重量提案完整 WRAP + 多視角 | 避免過度摩擦 | 分級判定規則需精準定義 |

**偏好組合**：候選 5（分級）+ 候選 3（規則+Hook）= 依提案規模分級強制 WRAP，輕量提案豁免完整審查，重量提案 Hook 強制阻擋

### A（Attain distance）—— 機會成本

推進 M-1 擠壓：
- M-3（Phase 3b Hook 密度審計）— 10+ Hook 盤點規模較大
- M-4（saffron checklist 強化）— 可在 M-1 完成後快速推
- M-5（Hook 設計指引）— 依賴 M-1 和 M-3 結論

推進優先判斷：M-1 是 Meta ANA 方向 1 P1，不能延後（錯誤提案假設會蔓延所有 Phase）

### P（Prepare to be wrong）—— 失敗防護

| 失敗情境 | 防護 |
|---------|------|
| 分級判定規則過嚴：小提案（拼字修正）被卡 | 分級表明確列「豁免條件」（重命名/拼字/格式） |
| 分級判定規則過鬆：重大提案通過輕量審查 | 分級判定多維度（檔案數、跨版本、UC 變動、相依性） |
| Hook 誤觸非 PROP 的 docs/ 變更 | Hook 明確限定 docs/proposals/ 路徑 + 非 README 檔案 |
| proposal-evaluation-gate.md 寫完不被引用 | 在 decision-tree、pm-role、tdd-flow 三處規則引用 |
| 現有 10 個 PROP 不符新規則造成焦慮 | 規則自公布日起對新提案強制，既有提案不追溯 |

### R（Reality Test）—— 盤點 10 個 PROP 評估品質

派發 saffron-system-analyst 執行 Reality Test 盤點。盤點維度：

1. **審查強度**：PROP 是否經過 WRAP/多視角/Reality Test？單人決策還是審查通過？
2. **評估完整性**：提案內有「替代方案」「失敗防護」「機會成本」章節嗎？
3. **檔案規模**：PROP 文件行數（估算評估深度）
4. **跨版本影響**：target_version 跨度
5. **UC 影響**：是否變更既有 UC 或新增 UC
6. **狀態流轉**：draft → discussing → confirmed → implemented 的時間線是否合理

預期產出：10 PROP 品質分級表（分為高/中/低三級），以及品質不足 PROP 的共通模式（作為強制機制設計依據）。

## PM 設計決策（基於 saffron 盤點結論）

### 最終設計：三層強制機制組合

1. **規則層**：`.claude/pm-rules/proposal-evaluation-gate.md`（本 Session 已產出 v1.0.0）

### 分級設計（規則 1）

基於 saffron 共通模式 1/2/3 設計三級：
- **輕量（light）**：spec 盤點批次 / 拼字修正 / 格式調整 / 重命名
- **標準（standard）**：單版本 / 1-2 UC 變動 / 非架構改動
- **重量（heavy）**：跨版本 / 跨專案 / framework 類 / 3+ UC 結構變動

### 各級評估要求（規則 2）

- 輕量：AC + 單版本即可
- 標準：WRAP 三問（W/A/P）+ Reality Test + 至少 2 候選方案
- 重量：完整 WRAP 四階段 + 本端獨立 Reality Test + 至少 3 候選方案 + 多視角審查

### 五大共通模式對應對策（saffron → 規則）

| 模式 2 跨版本未加強評估 | 規則 2 heavy 級別要求本端 Reality Test |
