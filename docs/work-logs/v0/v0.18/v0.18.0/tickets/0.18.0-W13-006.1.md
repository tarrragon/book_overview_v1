## Problem Analysis

## 問題描述

W13-003 落地的 askuserquestion-charset-guard-hook 於攔截時（exit 2），stderr 輸出
BLOCK_MESSAGE_TEMPLATE (L80-97) 包含 4 個區塊：

1. 錯誤摘要 + 違規清單（面向用戶可接受）
2. 「為什麼阻止」段落（面向 AI 的背景教育）
3. 「修復方式」1-4 點（面向 AI 的操作指引）
4. 「詳見 PC-072 路徑」（面向 AI 的延伸閱讀）

Claude Code PreToolUse Hook stderr 被 CLI 直接渲染到用戶 terminal，
完整的 AI 修復指引變成用戶 terminal 雜訊，違反 UX 乾淨原則。

## 根因

Hook 只使用單一 stderr 通道混合輸出 human-facing 與 AI-facing 訊息。
Claude Code Hook 實際有兩條通道可分離：

- stdout JSON: hookSpecificOutput.permissionDecisionReason → 傳給 Claude（不顯示用戶）
- stderr + exit 2: 同時顯示在用戶 terminal + 供 Claude 讀取

未利用此雙通道設計導致人機訊息混雜。

## 官方規格驗證（Reality-test 補完）

WebFetch https://code.claude.com/docs/en/hooks 確認：

### 關鍵發現（修正原先假設）

1. **exit 0 + JSON 與 exit 2 互斥**：JSON output only processed on exit 0；exit 2 忽略任何 JSON
2. **permissionDecisionReason 語意**（官方原文）：
   - allow/ask: shown to user but not Claude
   - deny: **shown to Claude**
   - defer: ignored
3. **exit 2 的 stderr 實際行為**（官方原文）：
   > stderr text is fed back to Claude as an error message

   即 exit 2 的 stderr 主要 audience 是 Claude，非用戶。原 ANA 假設「stderr 顯示給用戶」需修正為「Claude Code UI 選擇性顯示 hook error 給用戶（可觀測性設計），內容其實是給 Claude 的」
4. exit 0 時 stdout 寫入 debug log 但不進 transcript（PreToolUse 類）
5. 字元集：未明確記載 UTF-8 限制；additionalContext/systemMessage/plain stdout 上限 10,000 字元（reason 未明示，但共享此規模建議）

### 修正後的方案 C（最終形式）

**Hook 輸出策略**：
| 通道 | 用途 | 內容 |
|------|------|------|
| stdout JSON | 主要阻擋 + AI 指引 | `{hookSpecificOutput: {hookEventName: "PreToolUse", permissionDecision: "deny", permissionDecisionReason: "<完整 BLOCK_MESSAGE_TEMPLATE>"}}` |
| stderr | （留空或極簡違規清單） | Claude Code exit 0 時 stderr 寫入 debug log，不進 transcript |
| file log (logger) | 持久化觀察 | logger.warning 保留（既有） |
| exit code | 控制 | 0（讓 JSON 生效） |

**使用效果**：
- Claude 透過 permissionDecisionReason 取得完整修復指引（exit 2 的 stderr 機制等效，但走 JSON 通道更乾淨）
- 用戶 terminal 看到的是 Claude 基於 deny reason 產生的回應（自然整合到對話流）
- terminal 雜訊消失（原本 BLOCK_MESSAGE_TEMPLATE 長段不再直接印出）
- file log 仍保留供 debug

### 驗收條件更新建議（for W13-006.2 IMP）

1. 攔截時 exit code = 0（非 2）
2. stdout 為合規 JSON（含 hookEventName/permissionDecision/permissionDecisionReason 三欄）
3. permissionDecisionReason 包含完整違規清單 + 修復指引（BLOCK_MESSAGE_TEMPLATE 內容）
4. stderr 無輸出（或僅一行 summary，不含完整指引）
5. logger.warning 保留寫入 file log
6. 單元測試：驗證攔截 payload 後 stdout parse 為合規 JSON、exit 0、stderr 空/極簡
7. 實機測試：觸發 AUQ 違規後確認 Claude 正確接收 reason 並停止（Claude 端行為觀察）
## WRAP 分析

### W - Widen Options

| 方案 | 機制 | 優點 | 缺點 |
|------|------|------|------|
| A. stderr 極簡 + log file 補完整 | stderr 一行摘要；完整指引寫 log | stderr 乾淨 | AI 預設拿不到指引；需多讀 log |
| B. stderr structured JSON | stderr 輸出 JSON 分 human/ai 欄位；UI 只顯示 human | 單通道分離 | CLI 不解析 stderr JSON，行不通 |
| C. stdout JSON (permissionDecisionReason) + stderr 簡短摘要 | AI 指引走 permissionDecisionReason；stderr 只給用戶看違規位置 | 符合官方雙通道設計；規則 4 可觀測性保留；log 仍存檔 | 需實測 JSON output 阻擋行為等同 exit 2 |
| D. Verbose env var | HOOK_VERBOSE=1 才輸出完整 | 最小改動 | AI 預設失去教育；非預期 env 依賴 |
| E. stderr 分層空行 | 第一行摘要 + 空行後完整 | 無程式改動 | UI 仍渲染全部，無法分離 |
| F. stderr 只留 PC-072 連結 | 違規清單 + 一行連結；AI 自己 Read PC-072 | 最精簡 | AI 多一步讀檔；修復反應延遲 |

### R - Reality Test

- 方案 B 不可行：Claude Code CLI 將 stderr 視為純文字渲染，不解析 JSON。
- 方案 E 不可行：無法達成分離目標。
- 方案 A/F 需 AI 額外讀檔，增加一次 tool call。
- 方案 C 利用 Claude Code 既有 hookSpecificOutput.permissionDecisionReason 機制
  （IMP-055 證實 stdout JSON 結構化輸出可用），讓 AI 在看到 deny 時直接拿到
  修復指引，stderr 只顯示用戶友善摘要。
- 方案 D 的 env var 依賴違反「預設可觀測」原則。

### A - Attain Distance

一年後維護者讀 terminal 時看到：
「錯誤：AUQ payload 含 2 處污染 [questions[0].options[0].label: '与' (U+4E0E) 簡體字]」
這是理想訊號密度——足以定位問題，不污染畫面。AI 端的長段指引透過官方 reason 通道
傳遞，不干擾人類閱讀。
