### 關鍵發現

Hook 系統當前呈現「被動防禦反射弧」：錯誤發生 → 加 Hook → 觸發頻率上升 → 誤報累積 → 降級延後。前期階段因錯誤延遲顯現不會觸發此反射弧，導致前期長期缺 Hook。

**Hook 設計指引的核心主張**：從「發生問題才補 Hook」轉為「依階段特性主動設計 Hook」。前期階段應在設計時就規劃 Hook（即使當下無明顯錯誤）；後期階段應設定降級觀察期（連續 N 次無錯自動降級）。

## PM 產出：hook-stage-balance-methodology.md v1.0.0

### 方法論核心架構

**4 個核心原則**：
1. Hook 設計依階段特性主動規劃（不等錯誤發生）
2. 前期階段 Hook 先行（Proposal/Phase 0/Phase 1 新增必帶 Hook 設計）
3. 後期階段 Hook 設降級觀察期（連續 5 次無錯 → 2 Wave → 3 Wave）
4. Hook 本身受摩擦力方法論約束（禁 Hook 疊加 Hook）

**Hook 階段分佈理想密度表**：Phase 0-4 + Proposal 各階段的理想 Hook 密度 + 現況對比

**Hook 設計決策樹**：新增 Hook 前依階段判斷主動設計 vs 被動防禦分支

**Hook 生命週期管理**：Active → Observing → Deprecating → Archived 四階段 + hook-lifecycle.yaml 追蹤格式

**5 個反模式**：Hook 疊加 Hook / 前期不設計等錯誤 / 後期無降級條件 / Hook 密度與摩擦力曲線相反 / Hook 存在但無統計
