### 問題陳述

L5 Agent SDK Structured Outputs 重構被 W10-135 確認為唯一 100% 強制約束 AUQ 選項的路徑。本 ticket 評估此重構對既有 hook 系統（96 unique scripts / 121 registrations）、ticket-system、pm-rules 的破壞性，並識別漸進路徑。

### 盤點方法

1. 解析 `.claude/settings.json` 取得所有 hook 註冊（按 event type + matcher 分類）
2. 識別與 AUQ 直接互動的 hooks
3. 評估 L5 重構對各 hook 類別的影響面
4. 對比漸進路徑 vs 直接重構的累積成本

### 實驗發現

**既有 hook 系統規模**：96 unique scripts / 121 registrations / 6 event types

| Event Type | Registrations | 與 AUQ/Agent 派發相關 |
|-----------|--------------|---------------------|
| PreToolUse | ~40 | AUQ charset guard (1)、Agent dispatch (6)、Task dispatch (4) |
| PostToolUse | ~38 | Agent dispatch tracker/logger/count (3) |
| SessionStart | ~22 | 無直接 AUQ 關聯 |
| UserPromptSubmit | ~10 | auq-option-pattern-detector (1) |
| SubagentStop | 2 | agent-commit-verification (1) |
| Stop | ~5 | 無直接 AUQ 關聯 |

**直接 AUQ 互動 hooks（3 個）**：
1. `askuserquestion-charset-guard-hook.py`（402 行）— PreToolUse:AskUserQuestion，攔截簡體字
2. `askuserquestion-reminder-hook.py`（127 行）— PreToolUse:Task，提醒 PM 用 AUQ
3. `auq-option-pattern-detector-hook.py`（318 行）— UserPromptSubmit，偵測選項模式

**間接 Agent 派發 hooks（10 個）**：
- agent-dispatch-validation、agent-prompt-length-guard、dispatch-record、file-ownership-guard、worktree-base-distance-check、worktree-commit-before-dispatch（PreToolUse:Agent）
- active-dispatch-tracker、agent-dispatch-logger、dispatch-count-guard（PostToolUse:Agent）
- agent-commit-verification、subagent-stop-dispatch-cleanup（SubagentStop）

### L5 Agent SDK 重構影響面評估

#### 核心認知：L5 解決的問題範圍極窄

L5 Agent SDK Structured Outputs 重構的目標是**僅解決 AUQ 選項的 enum 約束**（簡體字、數量超限、邏輯離譜）。這是 96 個 hooks 中 3 個 AUQ hooks 試圖解決的問題。

#### 影響面分層

| 影響層 | 影響範圍 | 嚴重度 |
|-------|---------|--------|
| **直接廢棄**（3 hooks） | charset-guard、reminder、option-pattern-detector 被 enum 取代 | 正面（減少 847 行 hook 程式碼） |
| **Agent 派發重構**（10 hooks） | 若從 Claude Code Agent tool 改為 SDK Agent，所有 PreToolUse:Agent / PostToolUse:Agent / SubagentStop hooks 的 matcher 和 input schema 須重寫 | critical |
| **其餘 hooks（83 scripts）** | PreToolUse:Bash/Edit/Write、SessionStart、Stop 等不涉及 AUQ 或 Agent 派發 | 無影響 |
| **ticket-system** | ticket CLI 本身不依賴 AUQ，但 PM 流程中 ticket 與 AUQ 互動（如 claim 確認）需適配 | 低 |
| **pm-rules** | askuserquestion-rules.md、behavior-loop-details.md、decision-tree.md 中 AUQ 相關段落須重寫 | 中 |

#### 關鍵破壞性：Agent 派發 hooks 是連鎖爆炸點

L5 不僅改 AUQ。Agent SDK Structured Outputs 若用於 PM 派發代理人（取代 Claude Code 內建 Agent tool），10 個 Agent 派發 hooks 全部需要重寫：

- **PreToolUse:Agent** matcher 會失效（SDK Agent 可能不經 PreToolUse 事件）
- **PostToolUse:Agent** 同理
- **SubagentStop** event 是否保留取決於 SDK Agent 的生命週期管理

這 10 個 hooks 是 PM 流程的核心防護（派發驗證、prompt 長度限制、檔案所有權、worktree 檢查、dispatch 計數）。一旦失效，等於撤除派發品質閘門。

### 漸進路徑分析

#### 路徑 A：先做 L1+L2.5+L3.5 短期組合

| 步驟 | 內容 | 成本 | AUQ 問題解決度 |
|------|------|------|--------------|
| L1 強化 | 現有 3 個 AUQ hooks 改進 | 低 | <10% 發生率 |
| L2.5 | Few-shot scenario library | 中 | 進一步降低 |
| L3.5 | LLM judge 評分 | 中 | 高（但非 100%） |
| 累積 | — | 中 | ~95% |

**優勢**：不觸碰 Agent 派發 hooks，零破壞性
**劣勢**：永遠達不到 100%；3 層疊加的維護成本隨時間增長

#### 路徑 B：直接 L5 Agent SDK 重構

| 步驟 | 內容 | 成本 | AUQ 問題解決度 |
|------|------|------|--------------|
| SDK 整合 | Agent SDK Structured Outputs enum 化 AUQ | 高 | 100% |
| Hook 重寫 | 10 個 Agent 派發 hooks 適配新 event model | 高 | — |
| pm-rules 更新 | AUQ 相關規則重寫 | 中 | — |
| 累積 | — | 極高 | 100% |

**優勢**：一次到位，100% 解決
**劣勢**：破壞面大（10 hooks + pm-rules）、依賴 Claude Code 對 Agent SDK 整合的成熟度（W10-135 確認成本高）

#### 路徑 C（漸進路徑）：L1+L3.5 先行，觀測後決定 L5

| 階段 | 內容 | 觸發條件 |
|------|------|---------|
| 階段 1 | L1 hooks 強化 + L3.5 LLM judge | 立即 |
| 階段 2 | 觀測 30 天 AUQ 違規率 | 階段 1 完成 |
| 階段 3 | 若違規率 > 5% 啟動 L5 | 觀測數據驅動 |

### 「我可能錯在哪裡」自問

1. **低估 Agent SDK 成熟度**：若 Claude Code 未來版本原生整合 Agent SDK（如 tool_choice:strict 直接套用於 AskUserQuestion），L5 成本可能大幅降低。我的分析基於「當前需自行整合」假設，但 Claude Code 演進速度快，此假設可能在數週內過時。

2. **高估 hook 重寫成本**：若 Agent SDK Agent 仍觸發 PreToolUse/PostToolUse 事件（只是 input schema 不同），10 個 hooks 可能只需微調而非重寫。我尚未驗證 Agent SDK 的 event model 是否與 Claude Code hook system 相容。

3. **錯誤框定問題**：AUQ 選項隨機性可能不值得 L5 級別的投資。3 個 AUQ hooks + prompt 規範已將問題降至可容忍水準。本分析可能被「追求 100%」的完美主義驅動，而非價值驅動。

4. **忽略 Strict Tool Use 中間解**：W10-135 提到 `tool_choice=strict` 可達 ~99%，且不需要 Agent SDK 全面重構。這個中間層的實作成本和 hook 系統影響我未深入評估。

### 系統衝突檢查

#### 跨模組依賴衝突
- **有衝突**：L5 重構影響 hook system（10 個 Agent hooks）+ pm-rules（AUQ 相關 3+ 文件）+ ticket-system（PM 流程適配）
- 嚴重度：critical（Agent 派發 hooks 是品質閘門）

#### 既有 UC 變動影響
- **有變動**：PM 派發流程（UC 級別）會因 Agent SDK 替換而改變
- 既有測試案例需更新：是（hook 測試）
- 嚴重度：warning

#### 跨版本相容性
- **有影響**：Agent SDK 重構為框架級變更，不可漸進回退
- Migration 路徑：待設計
- 嚴重度：warning

### 衝突檢查結論
- 存在 critical 衝突（Agent 派發 hooks 連鎖影響），不建議直接啟動 L5 重構
- 建議先走路徑 C（漸進觀測），用數據決定是否需要 L5

### 結論與建議

**核心判斷**：L5 Agent SDK 重構對 AUQ 問題的解決是 100% 有效的，但其破壞面（10 個 Agent 派發 hooks + pm-rules）遠超 AUQ 問題本身的嚴重度。

**建議路徑**：路徑 C（漸進觀測）

| 短期（本版本） | 中期（觀測後） |
|--------------|--------------|
| L1 hooks 強化（已有 3 hooks 改進） | 數據驅動決定 L5 |
| L3.5 LLM judge（若 W10-136.3 支持） | Agent SDK 成熟度追蹤 |

**對 PM 的整合決策建議**：
- 若 4 視角共識偏向「L5 重構」，應先建 POC ticket 驗證 Agent SDK event model 與既有 hook system 的相容性（自問 #2），再決定全面重構
- 若偏向「hook 強化組合」，本視角無衝突，可直接進行
