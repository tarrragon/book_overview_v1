## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 逐 Hook Rubric 評估

---

#### 1. ticket-file-access-guard-hook.py

**R1 強制動作**：三種 deny 場景：(a) Read ticket 檔案、(b) Edit ticket frontmatter 欄位、(c) Write ticket 檔案。均回 exit 2 + deny message 引導使用 `/ticket` CLI。

**R2 觸發條件維度**：多維度。檢查順序為：工具類型（Read/Edit/Write）→ 路徑匹配（兩組 regex）→ 內部呼叫豁免（env var）→ Handoff 恢復模式豁免 → Edit 時區分 frontmatter vs body 內容。共 4+ 維度。

**R3 豁免機制**：完備。(a) `TICKET_TRACKER_INTERNAL=1` 環境變數豁免內部呼叫；(b) Handoff 恢復模式豁免 Read；(c) Edit body 區段（`## Problem Analysis` 等）允許通過；(d) 非 ticket 路徑一律放行。

**R4 已知衝突規則對照**：ARCH-015 不衝突——ARCH-015 約束 subagent 對 `.claude/` 的 Edit/Write，而本 hook 約束的是 `docs/work-logs/*/tickets/` 路徑，兩者路徑域不重疊。ticket 檔案不在 `.claude/` 下（`.claude/tickets/` 由 ticket-path-guard-hook 另行攔截）。無實際觸發劇本。

**R5 ARCH-018 風險分級**：**P3**（無觸發路徑）。多維度條件 + 完備豁免機制，設計已充分情境感知。

---

#### 2. mcp-write-tool-on-text-file-guard-hook.py

**R1 強制動作**：當 serena MCP 寫入工具（5 種）操作非程式碼副檔名（`.md/.txt/.yaml/.yml/.json/.toml`）時，回 exit 2 + deny，引導改用 Edit/Write。

**R2 觸發條件維度**：雙維度。工具名稱（是否屬 `SERENA_WRITE_TOOLS` 集合）× 副檔名分類（是否屬 `NON_CODE_EXTENSIONS`）。兩條件 AND 成立才 deny。

**R3 豁免機制**：隱式豁免。(a) 非 serena 寫入工具直接放行；(b) 程式碼副檔名放行；(c) 無副檔名或無路徑放行。無顯式 exempt list，但副檔名白名單機制等效。

**R4 已知衝突規則對照**：ARCH-015 不衝突——本 hook 僅約束 serena MCP 工具對非程式碼檔的使用，不涉及路徑域（`.claude/` vs 其他）。serena 工具本身可能因 MCP server 設定有額外限制，但那是 MCP 層面，與本 hook 無交集衝突。無實際觸發劇本。

**R5 ARCH-018 風險分級**：**P3**（無觸發路徑）。雙維度條件清晰，副檔名分類邏輯穩健。

---

#### 3. mcp-run-tests-validator.py

**R1 強制動作**：當 `mcp__dart__run_tests` 的 `roots` 參數缺少 `paths` 或 `paths` 為空時，回 exit 2 + deny，防止全量測試卡住。

**R2 觸發條件維度**：雙維度。工具名稱（僅匹配 `mcp__dart__run_tests`）× roots 參數內容驗證（每個 root 物件必須含非空 `paths`）。

**R3 豁免機制**：隱式豁免。(a) 非目標工具直接放行；(b) roots 含有效 paths 即放行。無需顯式 exempt list，因為條件本身就是精確的參數驗證。

**R4 已知衝突規則對照**：**本專案為 Chrome Extension（JavaScript），非 Flutter/Dart 專案**。`mcp__dart__run_tests` 工具在本專案 runtime 中不存在，本 hook 永遠走 `tool_name != "mcp__dart__run_tests"` 的早期放行路徑。與任何規則無衝突。觸發劇本：無——目標工具不存在於本專案。

**R5 ARCH-018 風險分級**：**P3**（無觸發路徑）。Hook 設計合理，但在本專案為死碼（dead code）——目標工具不存在。此為跨專案共用 hook 設定的副作用，非 ARCH-018 風險。

**附註（P2 觀察）**：死碼 hook 增加每次 PreToolUse 的 hook 執行開銷（啟動 Python + 讀 stdin + 早期 return）。建議 PM 評估是否移除或標記為非啟用。

---

#### 4. askuserquestion-charset-guard-hook.py

**R1 強制動作**：當 AskUserQuestion payload 含簡體字、日文漢字或 emoji 時，回 deny（透過 stdout JSON），阻擋污染內容送達用戶。注意：exit code 為 0（非 2），依賴 stdout JSON 的 `permissionDecision: deny` 生效。

**R2 觸發條件維度**：多維度。工具名稱（僅匹配 `AskUserQuestion`）× payload 內容掃描（三類字元：簡體字集合 / 日文漢字集合 / emoji unicode 範圍）。掃描粒度到每個字元。

**R3 豁免機制**：隱式豁免。(a) 非 AskUserQuestion 工具直接放行（return 0）；(b) payload 無違規字元即放行；(c) stdin 解析失敗時放行。另有 PC-085 self-check 機制驗證清單設計一致性（啟動時自檢，不影響 deny 決策）。

**R4 已知衝突規則對照**：ARCH-015 不衝突——本 hook 攔截的是 AskUserQuestion 工具的 payload 內容，與路徑域完全無關。唯一理論風險：若某合法繁體中文字被誤收入 `SIMPLIFIED_CHARS` 或 `JAPANESE_ONLY`，會產生 false positive deny。但 PC-085 self-check 機制已防護此風險（啟動時交叉驗證 CONFUSABLE_PAIRS）。無實際觸發劇本。

**R5 ARCH-018 風險分級**：**P3**（無觸發路徑）。多維度內容掃描 + self-check 防護，設計成熟。

---

#### 5. ticket-path-guard-hook.py

**R1 強制動作**：當 Write/Edit 操作的目標路徑匹配 `.claude/tickets/*.md` 時，回 exit 2 + deny，引導使用 `/ticket create` 和正確路徑 `docs/work-logs/*/tickets/`。

**R2 觸發條件維度**：雙維度。工具名稱（Write/Edit）× 路徑匹配（正則 `^\.claude/tickets/.*\.md$`）。

**R3 豁免機制**：隱式豁免。(a) 非 Write/Edit 工具放行；(b) 非 `.claude/tickets/` 路徑放行。無內部呼叫豁免（與 ticket-file-access-guard 不同），因為 ticket CLI 不會在 `.claude/tickets/` 下建檔。

**R4 已知衝突規則對照**：ARCH-015 理論交集存在但不衝突——`.claude/tickets/` 屬 `.claude/` 子目錄，ARCH-015 禁 subagent 在 worktree 內 Edit `.claude/`。但本 hook 的阻擋方向與 ARCH-015 一致（都是禁止寫入 `.claude/` 下的檔案），兩者是「同向強制」而非「反向衝突」。無實際觸發劇本。

**R5 ARCH-018 風險分級**：**P3**（無觸發路徑）。雙維度條件，且與 ARCH-015 同向而非衝突。

---

### 整合表格

| Hook | R1 強制動作 | R2 維度 | R3 豁免 | R4 衝突 | R5 風險 |
|------|-----------|---------|--------|---------|---------|
| ticket-file-access-guard | Read/Edit frontmatter/Write ticket → deny | 多維度（4+） | 完備（env var / handoff / body 區段） | 無（路徑域不重疊 ARCH-015） | **P3** |
| mcp-write-tool-on-text-file-guard | serena 寫入工具 × 非程式碼副檔名 → deny | 雙維度 | 隱式（副檔名白名單） | 無 | **P3** |
| mcp-run-tests-validator | dart run_tests 缺 paths → deny | 雙維度 | 隱式（非目標工具放行） | 無（目標工具不存在於本專案） | **P3** |
| askuserquestion-charset-guard | AUQ payload 含簡體/日漢/emoji → deny | 多維度 | 隱式 + PC-085 self-check | 無 | **P3** |
| ticket-path-guard | Write/Edit `.claude/tickets/` → deny | 雙維度 | 隱式（非目標路徑放行） | 與 ARCH-015 同向（非衝突） | **P3** |

### 結論

Batch B 的 5 個 hook **全部為 P3（無觸發路徑）**，無 P0/P1/P2 風險。原因：

1. 這 5 個 hook 均為 path/charset/access 守衛類，約束域明確且窄（特定路徑模式或特定工具 + 內容），不涉及派發行為或 worktree 決策，因此與 ARCH-015/ARCH-018 的典型衝突場景（派發類 hook 強制 worktree）無交集。
2. 所有 hook 均具備至少雙維度條件檢查，無「單維度一律強制」的 ARCH-018 反模式。
3. ticket-file-access-guard 的豁免機制最為完備（4 種豁免路徑），可作為其他 hook 的設計參考。

### 附帶觀察（非 ARCH-018 風險）

- **mcp-run-tests-validator.py 為死碼**：本專案為 Chrome Extension（JavaScript），`mcp__dart__run_tests` 工具不存在於 runtime。每次 PreToolUse 觸發時此 hook 白白啟動一次 Python 程序。建議 PM 評估是否停用或移除。
