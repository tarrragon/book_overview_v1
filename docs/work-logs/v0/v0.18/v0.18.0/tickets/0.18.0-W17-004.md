## 本 Session 直接觀察到的 8 個落差

### 落差 1：section 枚舉 vs 文件約定自相衝突（最嚴重）
- 文件要求：`.claude/pm-rules/context-bundle-spec.md` 指示用 `append-log --section "Context Bundle"` 寫入
- CLI 實測：append-log 的 section 枚舉僅允許 `Problem Analysis / Solution / Test Results / Execution Log`
- 結果：系統自相衝突的 live 案例——遵守文件指示會被 CLI 拒
- Linux 對比：POSIX 規範與 syscall 實作不會對立，`write(2)` 的 flags 在 man 和 kernel 必須一致

### 落差 2：show 不存在（命名反 Unix 直覺）
- 實測：`ticket track show` → invalid choice
- 應為：`ticket track full` 或 `summary`
- Linux 對比：`git show`、`docker show`、`kubectl get/describe` 是通用慣例；show 語意穩定

### 落差 3：add-spawned 不接多 ID
- 實測：`add-spawned W17-001 W17-002 W17-003` → unrecognized arguments
- 必須：拆兩次呼叫
- Linux 對比：`rm a b c`、`ln -s a b c dest/` 都接多目標，unix-y CLI 慣例

### 落差 4：append-log 的 content 是 positional 而非 --content
- 實測：`--content "..."` 被拒
- 實際：`ticket_id content` positional
- 問題：CLI 各子命令的「內容」欄位風格不一（create 用 `--what`、append-log 用 positional）

### 落差 5：錯誤訊息雙軌混雜
- CLI argparse error（英文 stderr）
- 後接 PostToolUse Hook 「SKILL 引導品質回饋」（中文 markdown）
- 兩者有時自相矛盾：argparse 說「invalid choice」，Hook 說「查 SKILL.md」
- Linux 對比：POSIX 錯誤 = errno + strerror()，單一語意通道

### 落差 6：create --source-ticket 有隱藏副作用
- 觀察：create 時指定 --source-ticket 會**自動**建立 spawned_tickets 連結
- 我後來 add-spawned 才從「已存在」訊息推論出此副作用
- 問題：SKILL.md 未顯式說明此自動行為，PM 會重複操作
- Linux 對比：`open(O_CREAT|O_EXCL)` 的副作用在 man 明寫；無隱藏動作

### 落差 7：parent vs source-ticket 語意歧義
- 兩者互斥但外觀相似（都建立父子關係）
- 實際差異需讀 PC-073 error-pattern 才懂：
  - parent → children 阻擋父 complete
  - source → spawned 不阻擋源 complete
- Linux 對比：`fork()` vs `clone()` 都有 man 清楚分辨 behaviors、flags、side effects

### 落差 8：ticket 目錄實際位置與文件示範不一致
- 文件示範：`docs/work-logs/v0.18.0/tickets/`
- 實際：`docs/work-logs/v0/v0.18/v0.18.0/tickets/`（三層版本目錄）
- PM 用第一個路徑 ls 會 No such file or directory

## 分析任務規劃（給執行代理人）

### 階段 1：Linux POSIX 成功模式盤點

## 回應用戶原問題

> 是不是我對於 ticket 系統的設計有疏失？

**部分是，部分不是**：

| 層級 | 結論 |
|------|------|
| 架構方向 | **正確**——Ticket = PCB，/proc = docs/work-logs/，agent definition = ELF，設計直覺與 Linux 一致 |
| 介面細節 | **有瑕疵**——17 項具體落差，其中 P0 兩項極低成本可補 |
| 引導文件 | **有瑕疵**——文件要求 vs CLI 實作存在自相矛盾（section 枚舉、路徑示範、隱藏副作用） |
| 設計複雜度 | **偶然 > 本質**——40 子命令中 ~6 便利 + ~2 可刪減，但 ~27 本質必需；真痛在命名與分組不一致，非總量 |

**核心教訓**：你設計 Ticket 系統時 Linux 類比是對的，但 CLI 實作階段缺少 POSIX 那種「介面即契約」的紀律——許多子命令獨立演化，未對齊同一套慣例（nargs、option 風格、錯誤訊息、文件同步）。修復方向是**用 P0/P1 清理介面基線**，不是重新設計架構。
