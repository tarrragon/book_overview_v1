評估 AUQ 強制給 2-4 選項、(Recommended) 標記、強制 Hook 等設計對規則 5 機制的衝擊，決定是否調整 askuserquestion-rules.md

## Problem Analysis

### 問題根因

W14-019 §5.9 衝擊表列出 4 項現行 AUQ 設計與規則 5 的衝突，全部標為「嚴重」或「中等」衝擊：

1. **強制給 2-4 個選項**：規則 5 指標 1（提供封閉選項並暗示最佳解）指向此設計為強勢敘事源
2. **(Recommended) 標記 + 放第一位**：規則 5 指標 4（權威框架）+ 位置偏誤，對應洞察 B GDPR Confirmshaming 研究
3. **強制觸發 Hook**：規則 5 判定為「整體掌控」（Lifton 條件 1 Milieu Control）
4. **用戶選 Other 後繼續推進**：主體性訊號被忽略，洞察 AA Lifton 延伸應用

但推翻這些設計會破壞三條核心防護：
- PC-014：合理化跳過（以「非正式任務」為由豁免 AUQ）
- PC-064：無意識疏失（PM 列純文字選項未意識到應用 AUQ）
- PC-066：Recommended 設計不當導致決策品質防護失效

核心矛盾：「§5.9 修正方向」與「PC-014/PC-064/PC-066 既有防護」在部分子項上方向相反，不可同時滿足。

### 影響範圍

### 實驗發現

**已驗證事實**：

1. §5.9 衝擊判定依據全為**通論研究**（Lifton / GDPR / Anthropic / NICHD），**無 Claude Code 在地實證**
2. askuserquestion-rules.md 三條防護（PC-014/PC-064/PC-066）均有**實際 incident 記錄**（在地實證）
3. §5.9 四項修正中，**兩項直接衝突 PC-064**（強制選項 + 釐清題純文字），**兩項可共存**（Recommended 標記 + Other 重新框架）
4. PC-066 規則 6 的「Recommended 為 WRAP/多視角」設計與 §5.9「(Recommended) 改 (My current guess)」方向一致，不衝突

**未驗證假設**：

1. §5.9「Claude 用 AUQ 時的權威誘導」在 Claude Code 實際情境下的強度（無量化數據）
2. 用戶對 AUQ 選項的依賴率 vs 使用 Other 的比例（尚未收集）
3. 「釐清題用純文字」在 CLI + Hook 環境下的可行性（Hook 可能誤判為開發命令——這正是規則 3 存在的原因）

**結論**：在地實證強度不對等——PC 防護有 incident 支撐，§5.9 修正是假設性方向。推翻 PC-064 的任何方案都必須先有在地實證反駁。

#### A（Attain Distance）機會成本

執行 Option A（完全採用 §5.9）的機會成本：
- **直接成本**：需同步修改 14 個 Hook、17 個場景、3 條 PC 防護規則
- **風險成本**：推翻 PC-064 後 PM 可能再次列純文字選項（PC-064 根因是無意識疏失，非規則不清楚）
- **機會成本**：擠壓 v0.18.0 其他 ticket 的資源（W14 Wave 尚有其他待派發任務）

執行 Option G（分期落地）的機會成本：
- **直接成本**：僅需修改 askuserquestion-rules.md 規則 6（小幅補丁）
- **延遲成本**：§5.9 衝擊在 3 個月內未被完整處理
- **收益**：保留 PC 防護 + 取得在地實證再做重大調整

#### P（Prepare to be Wrong）失敗預想

12 小時後 Option G 最可能失敗的原因：

1. **「(My current guess)」仍被用戶視為 Recommended**：標籤改了但位置偏誤未解 → 防護：同步在 description 明示「Claude 無充分資訊優勢」
2. **規則 5 Power Index 自檢 PM 忘記做**：新增規則後 PM 未養成習慣 → 防護：在 askuserquestion-rules.md 規則 6 之後新增規則 6.5 並設計 Hook 提醒
3. **019.5 監測指標設計不出可量化數據**：3 個月後仍無法判斷是否進入 Phase C → 防護：019.5 必須先定義可量化指標（已在 ticket AC）

### 結論：採用方案 G（分期落地）

**決策依據**：

1. 在地實證不對等——PC 防護有 incident，§5.9 修正為假設
2. PC-064 核心禁令不可在無在地反駁下推翻
3. §5.9 四項修正中 2 項與 PC-064 衝突、2 項可共存；可共存部分（Recommended 標記、Other 重新框架）優先落地
4. 衝突部分（釐清題純文字）等 019.5 監測數據後再決策

**Phase A（立即與 019.2 同步落地）**：

- 修改 askuserquestion-rules.md 規則 6：ANA/重大決策路由/Session 關鍵分歧情境下，`(Recommended)` 改為 `(Recommended by WRAP/多視角)` 或 `(My current guess)`
  - 前者用於有 WRAP/多視角支撐的選項
  - 後者用於無充分證據的傾向（必須在 description 明示「Claude 無充分資訊優勢」）
  - 仍放第一位（PC-066 要求）
- 保留所有 PC-064 禁令不動（禁止純文字列選項）

**Phase B（019.4 決策題/釐清題分流完成後）**：

- askuserquestion-rules.md 新增規則 6.5「AUQ 前置自檢」：
  - ANA / 重大決策路由 / Session 關鍵分歧 三類情境下，PM 用 AUQ 前必須自檢規則 5 Power Index（§5.5 八項指標）
  - 勾選 ≥3 項時，AUQ question 文字必須揭露 Claude 當前傾向（讓用戶知道 Claude 有強勢敘事傾向）
  - 不勾選或 <3 項時，維持現行 AUQ 流程
- 依 019.4 分類結果，決定釐清題是否需要特殊 AUQ 包裝（如 question 明示「自由描述，notes 欄位記錄」）

**Phase C（規則 5 落地 3 個月 + 019.5 監測數據充足後）**：

- 依監測指標決定：
  - 連續 2 次 Other 觸發「重新框架」的 Hook 實作可行性
  - 釐清題純文字機制的實證可行性（是否真的會誤判為開發命令）
  - 強制 Hook 的 Milieu Control 實證強度

**否決方向**：

| 否決項目 | 理由 |
|---------|------|
| 立即推翻 PC-064「禁止純文字列選項」 | 違反核心防護，§5.9 證據不足 |
| 連續 2 次 Other 立即實作 Hook | 閾值「2 次」無實證依據，Hook 實作複雜度高 |
| 立即把所有 (Recommended) 改掉 | PC-066 既有規則 6 已有部分防護，通過改 ANA/重大決策路由情境就足夠 |
