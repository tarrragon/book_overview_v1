## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 重現實驗摘要

**已驗證事實**：
1. 現有 body-check 不含 Layer 1 自檢驗證
2. W17-061 落地採純自律制（「建議插入」）
3. W17-051 IMP-D 意圖「防繞過」但未明示強制程度
4. 自檢結果可省略（無發現時），硬擋「必須有 ### 自檢結果」不合理
5. acceptance-gate-hook exit code：0=允許、2=阻擋、1=錯誤

---

### 維度一決策：Hook 行為強度 — 選項 B（warning + 豁免）

**結論**：warning + 可豁免（exit 0 + stderr 提醒），非硬擋。

**理由**：

1. **quality-baseline 規則 4 精神**：Hook 失敗必須可見（不可靜默），但規則 4 要求的是「可見性」非「阻擋性」。Layer 1 自檢是品質提升措施，非 data integrity 防護（與 body schema 必填章節不同層級）。

2. **自檢結果可省略的設計矛盾**：agent-self-check-template.md line 171 明確允許「無發現可省略」。若硬擋「必須有 ### 自檢結果」，則與模板設計衝突；若硬擋「必須執行過自檢」，則無法區分「執行了但無發現」與「根本沒執行」——兩者的 ticket body 外觀完全相同。

3. **skip-gate 精神**：skip-gate 的核心是「阻擋必須有明確的、可機械判定的違規條件」。自檢是否執行無法從 body 文字機械判定（除非強制所有自檢都寫結果，但這與模板設計矛盾）。

4. **quality-baseline 規則 5 精神**：所有發現必須追蹤。warning 輸出已滿足「發現 + 可見」要求。

**與 W17-061 衝突解法**：不修改 W17-061 落地的「建議非強制」語意。Hook 的 warning 是對 PM 的提醒（「此 ticket 的 Solution 中未見 ### 自檢結果，建議確認代理人是否已執行 Layer 1」），而非對代理人的阻擋。這與 W17-061「建議插入」定位一致——PM 派發時建議插入 + Hook complete 時 warning 提醒 = 雙層建議制。

---

### 維度二決策：檢查觸發範圍 — 全 IMP/ANA/DOC，無排除

**結論**：對所有 IMP/ANA/DOC ticket 的 complete 觸發 warning 檢查，不排除「純機械」或「同 session 已執行」。

**理由**：

1. **「純機械」無法機械判定**：ticket frontmatter 無 `task_nature: mechanical` 欄位，ticket title/what/how 的文字分析（NLP）會引入 false positive/negative。人為在 frontmatter 加標記等同要求 PM 預判任務性質，摩擦力過高且容易遺漏。

2. **「同 session 已執行」無持久化機制**：Hook 是 stateless 的（每次 complete 獨立執行），無法知道「同 session 是否已跑過自檢」。建立持久化（如 `.claude/state/self-check-history.json`）成本過高且引入新的 state management 複雜度。

3. **warning 成本極低**：既然行為是 warning（非硬擋），對純機械 ticket 多顯示一行 warning 的成本幾乎為零。PM 或代理人看到 warning 後自行判斷是否需要補執行即可。

4. **簡化實作**：無排除邏輯 = 無判定邏輯 = 無 false positive = 實作最簡。

---

### 維度三決策：豁免機制 — 不需要

**結論**：因行為為 warning（非阻擋），不需要豁免機制。

**理由**：

1. **豁免機制解決的是「被擋住無法繼續」**：硬擋需要豁免出口（`--force` / `--skip-self-check`）。warning 不阻擋 complete 流程，代理人和 PM 可自行忽略 warning。

2. **審計性由 warning 本身提供**：warning 輸出到 stderr，PM 在 conversation 中可見（quality-baseline 規則 4 滿足）。不需要額外的審計欄位。

3. **減少 CLI flag 膨脹**：現有 CLI 已有 `--skip-verify` 等 flag，再加 `--skip-self-check` 會增加認知負擔。

---

### W17-051 IMP-D 與 W17-061 語意衝突解法

**衝突描述**：W17-051 line 613「確保已填寫」暗示硬擋 vs W17-061 落地「建議非強制」。

**解法**：將 W17-051 IMP-D 的意圖重新詮釋為「使自檢流程可觀測」而非「強制自檢流程」。具體做法：

1. Hook 在 complete 時檢查 ticket body 是否含 `### 自檢結果`
2. 若不含，輸出 warning：「[Layer 1] Solution 中未見 ### 自檢結果 子章節。若代理人已執行自檢但無發現，此為正常；若未執行，建議 PM 確認。」
3. 不阻擋 complete

這滿足 W17-051 IMP-D「防繞過」的精神（透過 warning 使繞過行為可見），同時不違反 W17-061「建議非強制」的設計。
