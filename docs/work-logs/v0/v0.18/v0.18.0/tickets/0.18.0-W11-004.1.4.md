## Solution

### 實作摘要

於 `.claude/hooks/agent-dispatch-validation-hook.py` 新增分層判決：

1. **`HIGH_CONFIDENCE_KEYWORDS`**（frozenset）：`{"git commit", "git 寫入", "ticket CLI", "分支操作"}`
   - 選擇理由：這四類皆為直接可執行的命令動詞，語意極難被合法引用「描述」（已被規則 A/B/C/D 白名單過濾後），剩餘匹配基本為真越界。
   - 低信心保留 warn：`實作`、`修改檔案`、`設計功能規格`、`執行測試`、`修改規格`、`執行重構`、`系統審查` 屬於 natural language 描述動詞，FP 難以窮盡處理，避免誤阻塞。

2. **Bypass 雙通道**（`_is_bypass_requested`）：
   - env var：`AGENT_DISPATCH_BYPASS=1`
   - prompt marker：`[BYPASS-DISPATCH-VALIDATION]`
   - 降級後仍印 stderr 警告並寫 events.jsonl 供審計。

3. **分區函式**（`_partition_by_confidence`）：依 `HIGH_CONFIDENCE_KEYWORDS` 分離 conflicts 為 high/low 兩組。

4. **`_emit_keyword_conflict_warning_if_any` 重構**：
   - 回傳值從 `None` 改為 `"block" | "warn" | "pass"`
   - 高信心 + 未 bypass → 印 `_KEYWORD_CONFLICT_BLOCK_TEMPLATE` + 回傳 `"block"`
   - 其他情況（僅低信心 / 高信心被 bypass）→ 維持 warn template，bypass 路徑附加 `[BYPASS]` 註記
   - 所有真違規（含 block path）仍寫 events.jsonl 供誤報率統計

5. **`main()` 調整**：接收 verdict，`"block"` 時 `return 2`。

### 判決訊息設計（對應 AC「block 訊息含 agent 職責與建議」）

Block template 包含：
- Agent 名稱與高信心衝突摘要
- Agent 職責摘要（指引改派方向：實作代理人 / PM 前台）
- 兩種 bypass 方式說明（env var / prompt marker）
- 規則引用（agent-definition-standard.md / W5-001 error-pattern）

### 誤報率評估

依 Phase A 建立的 annotate/stats 基礎：
- 現有 events.jsonl 47 筆事件中，27 筆含高信心關鍵字（會被新邏輯 block）
- 高信心集合經設計限制為「直接命令動詞」，已通過 W11-004.1.3 四層白名單過濾（引號/否定/路徑/meta-task）後才進入判決
- Bypass 通道提供 PM 應急覆寫，實務上誤擋亦可即時解除
- AC4「誤報率 <= 10%」：實測需累積 annotate 標註後由 `dispatch_stats.py` 產出，本 ticket 完成架構落地
