### 修復摘要

新增 `ticket track dispatch-readiness <ticket_id>` 子命令，作為派發前
認知負擔閾值與綜合就緒度檢查入口，獨立於既有 `dispatch-check`（W10-017.2
活躍派發狀態）與 `dispatch-validate`（W17-003 CB 合理性檢查）。

### 設計決策

**三閾值對應**（來源 `.claude/references/cognitive-load-execution-details.md`）：

| 閾值 | 軟上限 | 強制拆分 | 取得方式 |
|------|-------|---------|---------|
| 1. 功能職責數 | > 2 | > 4 | `acceptance` 條目近似（CLI 無法精確自動推導） |
| 2. 修改檔案數 | > 5 | > 10 | `where.files` 計數 |
| 3. Context Bundle tokens | > 3000 | > 5000 | section 字元數 / 4 |

**Exit code 語意**（在 docstring 警示，避免跨命令解讀）：

- 0 = 全通過
- 1 = 軟性警告（任一項超軟上限但未達強制拆分）
- 2 = 硬性失敗（任一項超強制拆分閾值 / ticket 不存在 / IO/YAML 錯誤）

**仿 dispatch-validate 結構**：純函式（`check_responsibility_count` /
`check_file_count` / `check_context_bundle_tokens`）+ CLI 入口
`execute_dispatch_readiness` + `register_dispatch_readiness`，便於單元測試。
