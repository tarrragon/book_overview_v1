### AC #4 處理說明（測試斷言改用常數引用）

依 ticket WRAP 階段 P 段防護策略「本次只搬不改文案」，測試以 substring assertion 對外部使用者訊息做契約驗證；常數化後文案完全相同，substring 斷言仍正確且穩定。

若未來文案需更動，再依 substring assertion 調整為 `LifecycleMessages.CASCADE_UNBLOCKED_HEADER` 引用即可（已具備條件）。本 ticket 範圍內視為「已具備可改寫條件、保留現狀符合 WRAP 防護策略」。
### 修復摘要

將 commands/lifecycle.py 中 cascade 相關硬編碼訊息字串遷至 lib/command_lifecycle_messages.py 的 LifecycleMessages 類別，符合 quality-common.md §1.3.1（使用者訊息禁止硬編）。

### 新增常數（LifecycleMessages）

- CASCADE_UNBLOCKED_HEADER：[Cascade] 以下子 Ticket 已自動解鎖（blocked → pending）：
- CASCADE_UNBLOCKED_ITEM：item 模板（id + title）
- CHILDREN_PENDING_WARNING_HEADER：[Warning] 父 Ticket 完成時尚有未完成的子 Ticket：
- CHILDREN_PENDING_WARNING_ITEM：item 模板（id + status + title）
- CHILDREN_PENDING_WARNING_HINT：提示文字
- CASCADE_SAVE_FAILED：cascade 解鎖 {ticket_id} 儲存失敗：{error}

### 改動點

1. _print_cascade_unblocked：print 改用 LifecycleMessages.CASCADE_UNBLOCKED_HEADER + format_msg(CASCADE_UNBLOCKED_ITEM, ...)
2. _print_children_warnings：header / item / hint 三段全部改用常數
3. _classify_child 同流程的 cascade save f-string warning 改用 format_msg(CASCADE_SAVE_FAILED, ticket_id=..., error=...)

### 設計決策

- 本次只搬不改文案：所有 substring 與既有測試 assertion 保持一致。
- 常數放入既有 LifecycleMessages 類別而非新建類別，與 SPAWNED_* 系列同類別歸屬。
- 命名遵循既有慣例（*_HEADER / *_ITEM / *_HINT）。
