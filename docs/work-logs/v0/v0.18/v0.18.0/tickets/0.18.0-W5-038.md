分析 PM 提供選項時未使用 AskUserQuestion 工具的行為疏失根因

---

## Problem Analysis

### W（Widen）候選改善方案

基於已驗證真根因（Hook 層缺失 + 行為慣性 + 規則落地深度不足），候選方案：

| # | 方案 | 作用層 | 優點 | 缺點 |
|---|------|--------|------|------|
| 1 | Hook 偵測 PM 選項 pattern（Stop 或 UserPromptSubmit） | Infrastructure | 最接近根因，事後糾正 | Hook 延遲、false positive 風險 |
| 2 | pm-role.md「行為循環」增補「列選項必用 AUQ」 | 規則 | 自動載入，每次 session 可見 | 規則已有（規則 1/3 未解 5 次疏失），可能仍被稀釋 |
| 3 | CLAUDE.md 頂部顯眼提醒 | 規則（強制載入） | 最顯眼 | 被其他 context 稀釋 |
| 4 | 組合方案（1+2+3） | 多層 | 多重防護、提升落地率 | 工作量 3 倍 |
| 5 | 預載入 AskUserQuestion schema（非 deferred） | Runtime | 消除 ToolSearch 摩擦 | 非主因、非本專案可控 |

**推薦**：方案 4 組合（規則先行 → Hook 補強 → CLAUDE.md 最終防線）
**理由**：單一層防護已證實不足（規則 1/3 存在但未生效）；多層才能覆蓋「規則稀釋 + 行為慣性 + context 擁擠」的複合失效。

---

### A（Attain distance）機會成本

- **執行成本**：方案 4 拆為 4 個 Ticket，估 1-2 個 Wave 完成
- **擠壓目標**：W5-002 Phase 2 起步、W5-037 Hook 修復排序
- **不做的成本**：
  - 未來每次 session PM 列選項時可能再犯
  - 每次疏失 = 用戶糾正成本 + context 浪費（本 session 約 10 輪對話修正）
  - PC-014 教訓已顯示此類疏失會重複 3+ 次
  - 規則 7（Memory 升級檢查）亦要求跨 session 原則必須上升為框架層
- **判斷**：一次性改善成本 << 長期累積疏失成本。**值得立即排入 W5**。

---

### P（Prepare to be wrong）最可能失敗原因

**12h 後最可能失敗情境**：Hook 偵測誤報率高 → PM 疲憊忽略提醒 → Hook 實質效果歸零（回到起點）

**其他失敗情境**：
- 規則升級後 PM 仍因 context 稀釋遺忘（規則已明確但未生效的前例）
- CLAUDE.md 提醒被 auto-compact 優先壓縮（顯眼程度不足）
- Hook 偵測 pattern 過窄（漏報）或過寬（誤報）

**防護措施**：
1. Hook MVP 只偵測**高信心度 pattern**（連續 3 個以上 A/B/C 標記 + 冒號結尾）
2. Hook 設為 **warning 非 block**（避免阻礙正常流程）
3. false positive 豁免清單（如引用既有文件的選項、歷史回顧性列表）
4. **先做規則層（方案 2、3）再做 Hook（方案 1）**——規則是基線，Hook 是補強，順序不可倒
5. 升級為 PC error-pattern（PC-064 類）確保跨 session 學習

**已驗證的真根因（Infrastructure 層 + 行為層雙重失效）**

1. **Hook 層失效（最主要）**：askuserquestion-reminder-hook.py 只處理 Task 派發的多 Ticket 提醒，**沒有 Hook 在 PM 輸出文字含「列選項」pattern 時攔截或提醒**。5 次疏失全部發生在 commit 前的 session 中途決策點，沒有 Hook 會觸發。

2. **行為慣性（次要）**：Claude base model 訓練資料中列 markdown 選項給用戶選是常見 CLI 對話模式，無 Hook 預警時，主動意識容易被稀釋（尤其 session context 複雜、多輪決策疊加）。

3. **規則落地深度不足（連帶）**：askuserquestion-rules.md 在 pm-rules/ 下，非自動載入；pm-role.md 的「行為循環」章節未將「列選項」明確列為觸發條件。PM 在 session 中後期 context 擁擠時，主動 Read 的記憶可能稀釋。

**被反證的假設**

- ToolSearch 摩擦論：僅首次成立，不解釋後續 4 次
- 規則不明確論：規則文件遠比初始假設明確
