分析 PM 派發 subagent prompt 範本化執行機制的系統缺口

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題現象

本 session 兩次撞 `agent-prompt-length-guard-hook.py` 30 行限制（派發 W17-046.1 與 W17-047 實作時），每次都是「寫 prompt → 送出 → Hook 擋 → 補 Context Bundle 到 ticket → 重寫精簡 prompt → 送出」的繞路流程。

### 問題根因

規範與模板**都已存在**（`context-bundle-spec.md`、`agent-dispatch-template.md`、`phase3b-prompt-template.md`、PC-040），但 PM 派發工作流中模板未被主動套用，導致 Hook 事後擋而非事前引導。根因位於三個層級：

- **認知層**：`pm-rules/` 按需讀取，`rules/core/pm-role.md` 未強制引用派發模板 → PM auto-load context 中看不到模板存在
- **工具層**：Hook 只事後擋 30 行，不事前引導 → PM 撞牆才補救
- **資產層**：模板檔案存在但結構偏「完整規範」而非「快速填空骨架」，PM 需讀 154 行才能理解用法

### 影響範圍

- 檔案：`.claude/rules/core/pm-role.md`、`.claude/hooks/agent-prompt-length-guard-hook.py`、`.claude/references/agent-dispatch-template.md`
- 行為：每次派發 subagent 都有撞 Hook 的風險，平均每次派發重寫 1 次 prompt，浪費 ~500 tokens/次 + 打斷 PM 流程連續性
- 規模：本專案 80+ hooks、40+ agents，派發頻率高，累積成本可觀

### 相關 Error Pattern

- **PC-040**（context-in-prompt-not-ticket）：本問題的症狀記錄，但 PC-040 僅記錄行為模式，未設計事前防範機制
- **PC-088**（多步驟 perplexity 盲）：W17-023 處理的子議題（路徑預算自檢），與本 ticket 議題互補但不重疊

### 流程斷點

PM 派發的認知流程（實證來自本 session）：

```
1. 想派發 → 2. 從零寫 prompt → 3. 送出 → 4. Hook 擋 → 5. 補 Context Bundle → 6. 重寫 prompt → 7. 送出
```

理想流程：

```
1. 想派發 → 2. 填空模板骨架 → 3. Context Bundle 已在 ticket → 4. 送出 → 5. 一次通過
```

斷點在步驟 2：PM 從零寫 prompt 而非填空模板，因 auto-load context 中**看不到模板**。

### 工作流斷點分析（acceptance 2）

PM 派發的認知流程（實證來自本 session）：

```
1. 想派發 → 2. 從零寫 prompt → 3. 送出 → 4. Hook 擋 → 5. 補 Context Bundle → 6. 重寫 prompt → 7. 送出
```

理想流程：

```
1. 想派發 → 2. 填空模板骨架 → 3. Context Bundle 已在 ticket → 4. 送出 → 5. 一次通過
```

**斷點位置**：步驟 2。PM 從零寫 prompt 而非填空模板，因為：

- **認知層斷點**：`pm-rules/` 按需讀取，`rules/core/pm-role.md` 無引用派發模板 → PM auto-load context 中**看不到**模板存在
- **工具層斷點**：Hook 只事後擋長度，不事前引導 → PM 撞牆才補救
- **資產層斷點**：模板檔案存在但結構偏「完整規範」而非「快速填空骨架」，PM 需讀 154 行才能理解用法

### Prompt 產生過程調查（acceptance 3）

PM 腦中預設模型：

```
prompt = 任務描述 + 背景 + 步驟 + commit policy + 範圍鎖定 + 完成要求
```

期望模型：

```
prompt = Ticket ID + 「讀取 ticket」指引 + 簡短動作 + 完成方式
ticket body = 任務描述 + 背景 + 步驟 + commit policy + 範圍鎖定 + 完成要求
```

**落差**：PM 把「確保 agent 了解完整情境」當成 prompt 責任，但這責任應由 ticket 承擔。
