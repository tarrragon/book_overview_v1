**規模量化**：1 次 smoke test → 48 個 false positive missing ID + 1 個 orphan（W17-083 自身 handoff，已通過 GC 清理）。

### 根因識別

**真正根因**（與 ticket why 假設不同）：

hook line 275 掃描範圍錯誤：

```python
worklog_ids = _extract_ticket_ids(content, active_version=version) if has_keywords else []
```

`content` 是**整個 worklog 全文**（含整個版本所有歷史 ticket 提及）。Hook **沒呼叫 `extract_handoff_section(content)`** 限定到「下個 Session 接手 Context」段落。

**driver 函式對照**：

| 函式 | 範圍 | 是否被 hook 呼叫 |
|------|------|----------------|
| `extract_recent_content(path, since_mtime)` | mtime 過濾後回完整 content（命名誤導：實際不限定段落） | 透過 line 258 mtime 檢查模擬，但邏輯不等價 |
| `extract_handoff_section(content)` | 從首個 HANDOFF_KEYWORD 到下一個 H1/H2 | **未被呼叫** |
| `extract_ticket_ids(text)` | 對輸入字串抓 ID | 被呼叫，但輸入錯誤（整個 content 而非 section） |

**對 Phase 2 sage 測試規格的影響**：

19 RED 測試（S3 11 個）全綠但運行 false positive，原因為測試 input 是「精心構造的小段 worklog 內容」（已含 handoff 段標記 + 少量 ticket ID），未模擬真實 worklog（含整版本歷史的 v0.18.0-main.md，> 700 行 ticket 進度條目）。測試覆蓋了「方案邏輯正確」但未覆蓋「對真實規模 worklog 不漏抓 / 不誤抓」。

**ticket why 三假設的判定**：

| 假設 | 結論 |
|------|------|
| H1 extract_recent_content 未呼叫 | **部分成立**（命名誤導；實際根因是 extract_handoff_section 未呼叫） |
| H2 mtime 過濾門檻失準 | **次要因素**（smoke test session_start=0 時 line 258 過濾自動失效，但即使 session_start > 0 通過 mtime 後仍掃整個 content，仍會 false positive） |
| H3 session_start_timestamp 缺失 fallback | **不成立**（即使有時間戳，掃描範圍錯仍 false positive） |

**推薦：方案 A**

實作示意：

```python
# .claude/hooks/stop-worklog-handoff-sync-check-hook.py line 275 前
from worklog_parser import extract_handoff_section  # SOT-mirror 同步：對應 lib

# detect_sync_drift 內：
if has_keywords:
    section = _extract_handoff_section(content)  # 新增 hook 內 helper（SOT-mirror）
    worklog_ids = _extract_ticket_ids(section, active_version=version)
else:
    worklog_ids = []
```
