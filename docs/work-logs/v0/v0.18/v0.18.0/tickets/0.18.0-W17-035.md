### 補充脈絡（W17-029 Linux 再評估）

W17-029 已完成 batch dispatch + template 可行性評估，初步結論為「暫不做 batch dispatch CLI，先做 template + dispatch-plan」。使用者進一步追問：若不做 batch dispatch CLI，原需求如何解決？從 Linux 系統設計重新評估後，結論更精確：

- Linux 不用 `batch_execve()` 解決大量任務啟動，而是保留小 primitive：`fork` / `exec` / `wait` / `signal` / scheduler。
- 批量工作由 shell / `make` / `xargs -P` / systemd 這類 orchestration layer 描述與編排。
- 對本專案來說，對應解法不是 batch dispatch CLI，而是 **Makefile-like dispatch-plan template + prompt snippets + auto Context Bundle + runqueue**。

本 ticket 的範圍因此應包含 dispatch-plan template 評估，並將 batch dispatch CLI 明確列為「非首輪候選 / 僅 tripwire 觸發後升級」。

### 與 W17 未完成任務的關係

| Ticket | 關係 |
|--------|------|
| W17-029 | 本 ticket 的輸入：提供 D 方向可行性與 Linux 再評估結論 |

| W17-027 / W17-031 | runqueue / scheduler 消化 dispatch-plan 的後續入口；本 ticket 不改 scheduler |
| W17-030 | tripwire 監測 T3：若 PM 仍拼單避免派發，再升級 batch CLI 評估 |
| W17-033 | 保持 agent/ticket process 邊界後，agent 自主 complete 是 lifecycle 完整性補強 |
