## Problem Analysis

W17-035 分析 PM 派發 prompt 反覆撞 PC-040/PC-065 的 meta 根因：缺少可重用短 prompt skeleton、dispatch-plan 結構與 snippet，而不是需要 batch dispatch CLI。

### 補充脈絡（W17-029 Linux 再評估）

W17-029 已完成 batch dispatch + template 可行性評估，初步結論為「暫不做 batch dispatch CLI，先做 template + dispatch-plan」。使用者進一步追問：若不做 batch dispatch CLI，原需求如何解決？從 Linux 系統設計重新評估後，結論更精確：

- Linux 不用 `batch_execve()` 解決大量任務啟動，而是保留小 primitive：`fork` / `exec` / `wait` / `signal` / scheduler。
- 批量工作由 shell / `make` / `xargs -P` / systemd 這類 orchestration layer 描述與編排。
- 對本專案來說，對應解法不是 batch dispatch CLI，而是 **Makefile-like dispatch-plan template + prompt snippets + auto Context Bundle + runqueue**。

本 ticket 的範圍因此應包含 dispatch-plan template 評估，並將 batch dispatch CLI 明確列為「非首輪候選 / 僅 tripwire 觸發後升級」。

### 若後續建 IMP 的範圍（本 ANA 需評估）

首輪 IMP/DOC 若成立，範圍應限制為：

- 更新 `.claude/references/agent-dispatch-template.md`：新增 dispatch-plan table 與欄位說明。

- 不新增 `batch dispatch` CLI，不新增自動批量 Agent 派發執行器。

### 問題根因

根因不是「hook 不夠嚴」或「需要 batch dispatch CLI」，而是 PM 派發介面缺少可重用的短 prompt skeleton 與 dispatch-plan 結構。

- PC-040 的直接原因：PM 為了讓 agent 一次取得足夠 context，把分析、檔案、策略寫進 prompt，而不是先持久化到 Ticket Context Bundle。
- PC-065 的直接原因：PM 草擬 prompt 時先寫任務敘述，忘記把機器可解析的 `Ticket: {id}` 放在第一行；並行派發時同一個錯誤模板會放大成 N 次 hook block。
- 共同的 meta 根因：現有文件同時要求「職責邊界聲明」「禁止範圍」「精準 staging」「短 prompt」，但沒有把這些要求收斂成一個 <=30 行、第一行固定 `Ticket: {id}` 的 snippet。
- W17-029 Linux 類比後的修正：應將多任務編排放在 dispatch-plan（類 Makefile / systemd unit graph），而不是把多個 agent 任務合成一個 batch execution CLI。

### 實驗發現

## Solution

結論採文件/template/snippet 落地：建立並完成 W17-044，補 dispatch-plan template 與 PM prompt snippets；batch dispatch CLI 排除於首輪。

### 候選方案比較

| 方案 | 解決 PC-040 | 解決 PC-065 | 成本 | 風險 | 結論 |
|------|-------------|-------------|------|------|------|
| 只靠現有 hook | 只阻擋 | 只阻擋 | 低 | PM 仍反覆重試 | 不足 |
| memory 強化 | 中 | 中 | 低 | 不可驗證、跨 session 仍可能遺漏 | 可輔助，不作主方案 |
| skill / reference snippet | 高 | 高 | 中 | 文件需維護 | 首輪主方案 |
| dispatch-plan template | 高 | 中 | 中 | PM 需先填表 | 首輪主方案 |
| agent builder / prompt generator | 中 | 高 | 中高 | 可能過早工具化 | 延後 |
| batch dispatch CLI | 低 | 中 | 高 | PC-042/PC-092/PC-104 風險上升 | 排除首輪 |

### 建議落地

建立一張後續 DOC：`0.18.0-W17-044`。

範圍限定為文件/template/snippet：

- 在 `.claude/references/agent-dispatch-template.md` 新增短 prompt skeleton，第一行固定 `Ticket: {id}`，範例不超過 30 行。
- 新增 dispatch-plan table：`ticket` / `agent` / `files` / `deps` / `context source` / `commit policy` / `run mode`。
- 在 `.claude/pm-rules/parallel-dispatch.md` 定義：group ticket、spawned ticket、2+ agent 派發前先寫 dispatch-plan。
- 在 `.claude/pm-rules/dispatch-gate.md` 或 `.claude/skills/ticket/SKILL.md` 增加派發前檢查點。
- 明確禁止首輪新增 batch dispatch CLI；若 W17-030 T3 偵測 PM 仍拼單，才另開工具化評估。

### 不拆子任務的理由

W17-035 是 ANA ticket，工作重點是根因分類與方案選型；落地是一個一致的文件/template 更新面，不需要以 child 拆分。使用 spawned ticket 比 child 更符合語意：`W17-044` 是分析結論衍生的後續任務，不是 W17-035 的分析子步驟。
