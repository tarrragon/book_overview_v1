### 問題根因

**根因（單一）**：兩個 hook 在判斷「目標檔案是否允許編輯」時，使用「本專案 cwd context」而非「目標檔案所屬 repo context」推導決策依據，造成跨專案路徑的維度錯位。

**具體機制**：

1. `main-thread-edit-restriction-hook.py` + `lib/path_permission.py`
   - `normalize_path()` 用 `CLAUDE_PROJECT_DIR` 或 `Path.cwd()` 將絕對路徑轉相對路徑
   - 跨專案場景下，目標檔案絕對路徑（如 `/Users/.../blog/content/skills/...`）與 `CLAUDE_PROJECT_DIR`（`/Users/.../book_overview_v1`）無前綴交集 → 維持絕對路徑形式
   - `ALLOWED_PATTERNS` 全為相對路徑 regex（`^\.claude/...`、`^docs/...`）→ 絕對路徑必然不匹配
   - `BLOCKED_PATTERNS` 同樣不匹配 → 落入「預設拒絕」分支
   - 結果：任何外部 repo 路徑都被預設拒絕，無視該路徑在外部 repo 內可能完全合法

2. `branch-verify-hook.py`
   - `_resolve_cwd_for_branch_detection()` 已正確從 file_path 父目錄推導 cwd（這部分邏輯是對的）
   - `get_current_branch(cwd=file_dir)` 取得**目標 repo 的 branch**（正確）
   - 若目標 repo 在保護分支（如 blog repo 的 main），進入 `is_exempt_path_on_protected_branch(file_path, cwd=file_dir)`
   - 該函式內 `get_project_root(cwd=cwd)` 取得**目標 repo 的 root**（正確）
   - 但豁免清單 `exempt_prefixes` / `exempt_exact` 是**本專案的設計約定**（`.claude/`、`docs/`、`CLAUDE.md` 等）
   - 若目標 repo 用不同的目錄結構（如 blog 用 `content/`），目標檔案路徑相對於目標 repo root 後不匹配豁免清單 → deny
   - 實際阻擋訊息中的「豁免路徑」清單是本專案的清單，但用來判斷外部專案，使用者讀到後也無法理解為何外部 blog 的 `content/skills/` 不在豁免清單

### 影響範圍

- 兩個 hook：`.claude/hooks/main-thread-edit-restriction-hook.py`、`.claude/hooks/branch-verify-hook.py`
- 一個共用模組：`.claude/hooks/lib/path_permission.py`（normalize + ALLOWED/BLOCKED 模式）
- 一個分支設計約定：每個專案的「豁免路徑」可能不同，目前硬編碼於 `branch-verify-hook.py` 內

**受影響場景**：

- 主線程在本專案 session 中嘗試編輯外部 repo 檔案（如跨專案修補 blog/skills 等）
- subagent 不受 `main-thread-edit-restriction` 影響（已有 `is_subagent_environment` early return），但仍受 `branch-verify-hook` 阻擋
- 編輯**非 git 管理**的外部檔案（如 `~/Downloads/notes.md`）：path_permission 預設拒絕 + branch-verify 透過 `is_exempt_path_on_protected_branch` 第 128 行 `not file_path.startswith(project_root)` 豁免（這分支邏輯**已對外部檔案放行**，但 main-thread hook 仍會擋）

### 重現實驗（靜態追蹤）

**方法**：讀取 hook 原始碼追蹤決策路徑，對照觸發案例的 file_path（外部 repo 絕對路徑）模擬執行流程。本 ticket 為靜態程式碼分析型 ANA，不需動態重現。

**模擬輸入**：tool_name=Edit, file_path=`/Users/mac-eric/project/blog/content/skills/compositional-writing/skill.md`, 主線程 cwd=book_overview_v1。

**main-thread-edit-restriction-hook.py 路徑追蹤**：

1. is_subagent_environment → false
2. get_current_branch(cwd=file_dir) → blog 的 branch（如 main）
3. is_allowed_branch('main') → false
4. check_file_permission(file_path):
   - normalize_path：CLAUDE_PROJECT_DIR=book_overview_v1，與 blog 路徑無前綴關係 → 維持絕對路徑
   - is_allowed_path：ALLOWED_PATTERNS 全為相對路徑 regex → 不匹配
   - is_blocked_path：同樣不匹配
   - 路徑非 .claude/ 開頭（仍是絕對路徑） → 落入預設拒絕
5. 結論：deny `EDIT_BLOCKED_DEFAULT_DENY`

**branch-verify-hook.py 路徑追蹤**：

1. _resolve_cwd_for_branch_detection → blog 父目錄
2. get_current_branch(cwd=file_dir) → 'main'（blog）
3. is_protected_branch('main') → true
4. is_exempt_path_on_protected_branch:
   - get_project_root(cwd=blog_dir) → blog repo root（正確）
   - normalize 為 `content/skills/...`
   - exempt_prefixes（.claude/, docs/, scripts/experiments/）不匹配
   - 返回 false
5. is_project_file=true（file_path.startswith(blog_root)）
6. 結論：deny，訊息顯示**本專案的豁免清單**（誤導，blog 沒有 .claude/）

**已驗證的事實**：

1. main-thread hook 對外部 repo 必然 deny：ALLOWED/BLOCKED 為相對路徑 regex，外部絕對路徑不匹配任何模式 → 預設拒絕
2. branch-verify hook 的 branch 判斷已正確切到目標 repo，但豁免清單硬編碼本專案約定 → 外部 repo 保護分支必擋
3. branch-verify 第 128 行 `not file_path.startswith(project_root)` 豁免分支只對 auto-memory 等「不在任何 git repo」的路徑生效；當 cwd 已切到目標 repo 時，此分支不觸發
4. 兩個 hook 阻擋邏輯維度不同：main-thread 是白名單路徑模式，branch-verify 是保護分支+豁免清單；根因都是 context 錯位，但修法需分別處理

### 跨專案路徑識別與正確判斷策略

**核心設計原則**：所有路徑判斷必須以「目標檔案所屬 repo」為決策上下文，不是「主線程 cwd context」。

**Step 1：目標 repo 偵測**

新增共用函式 `lib.git_utils.find_target_repo(file_path: str) -> Optional[str]`：

- 從 file_path 父目錄逐層向上找 `.git` 目錄（檔案或子目錄皆可，後者支援 worktree）
- 找到回傳 repo root；找不到回傳 None（外部非 git 檔案）
- 與既有 `_resolve_cwd_for_branch_detection` 的差別：明確語意為「找 repo root」而非「找一個能跑 git 的 cwd」

**Step 2：判斷檔案歸屬**

| 情境 | target_repo | 判斷依據 |
|------|------------|---------|
| 目標檔案在本專案內 | == get_project_root() | 走現行邏輯（白名單 + 本專案豁免清單） |
| 目標檔案在另一個 git repo | != get_project_root() | 走「跨專案邏輯」 |
| 目標檔案不在任何 repo | None | 直接放行（既有第 128 行邏輯延伸） |

**Step 3：跨專案邏輯（main-thread-edit-restriction）**

| 設計選項 | 行為 | 評估 |
|---------|------|------|
| A. 完全放行（不檢查白名單） | 信任外部 repo 自有規則 | 最寬鬆；風險：誤編輯非預期檔案 |
| B. 讀取目標 repo 的 .claude/（若存在）並套用其 ALLOWED_PATTERNS | 尊重目標專案規則 | 最完整但實作複雜（需設計跨 repo 規則載入） |
| C. 跨專案場景下僅做 branch-verify，path_permission 整體跳過 | 將「白名單」職責限於本專案內部設計約定 | 推薦：path_permission 是本專案內部規範，本就不該套用到外部 |

**推薦選項 C**：path_permission.py 的設計目的是「保護本專案不被主線程亂改 src/ 等程式碼」，這是本專案內部的設計約定。對外部 repo 套用本專案的白名單在語意上不合理。修法：在 main-thread hook 進入 path_permission 前，先判斷 target_repo，若非本 repo 則 skip path_permission 檢查（仍走 branch-verify）。

**Step 4：跨專案邏輯（branch-verify）**

| 設計選項 | 行為 | 評估 |
|---------|------|------|
| A. 跨專案時不做 branch-verify | 完全信任用戶 | 太寬鬆；用戶可能不知道目標 repo 在保護分支 |
| B. 跨專案時保留 branch-verify，但豁免清單改為通用清單（如 README.md / CHANGELOG.md / .gitignore） | 退回最小通用集 | 推薦：通用文件可在任何 repo 的 main 編輯 |
| C. 跨專案時 branch-verify 行為 = 自動建 feature branch（見後段評估） | 主動處理 | 進階方案，留 IMP follow-up |

**推薦：B 為基線，C 為進階選項**。基線版本：跨專案保護分支時，僅放行 README.md、CHANGELOG.md、.gitignore、.gitattributes 等通用文件；其他檔案 deny 並提示「目標 repo 在保護分支，請切換或啟用自動建分支」。

### 自動建 feature branch 方案可行性評估

**核心設想**：偵測到目標 repo 在保護分支時，自動建立 `feat/cross-project-edit-{timestamp}` 分支再放行。

**可行性分析**：

| 維度 | 評估 | 說明 |
|------|------|------|
| 技術可行性 | 中 | git checkout -b 可程式化執行；但 PreToolUse hook 在 Edit 之前同步執行，建分支屬副作用 |
| 安全性 | 低-中 | 副作用 hook 違反 PreToolUse 設計慣例（應只決策 allow/deny） |
| Race condition | 高風險 | 多個跨專案 Edit 並行時可能建出重複分支或衝突 |
| 未提交變更處理 | 高風險 | 目標 repo 可能有未提交變更，自動 checkout 會中斷用戶工作流 |
| 用戶授權邊界 | 不明 | 主線程 session 中對「另一個 repo」做 git 寫操作，超出本 session 的隱含授權範圍 |
| 復原成本 | 高 | hook 失敗後留下半成品分支，需用戶手動清理 |

**結論：不建議在 hook 內自動建分支**。

**替代方案：deny 訊息中提供準備好的指令**

當偵測到目標 repo 在保護分支時，deny 訊息應包含：

1. 切換到目標 repo 並建分支的完整 shell 指令（含 cd）
2. 警告若有未提交變更
3. 推薦先確認用戶意圖再執行

範例 deny 訊息：

```
跨專案編輯被阻止：目標 repo 在保護分支

目標檔案: /Users/.../blog/content/skills/...
目標 repo: /Users/.../blog
目標 repo 當前分支: main（保護分支）

建議操作：
  cd /Users/.../blog
  git status        # 確認無未提交變更
  git checkout -b feat/cross-project-{purpose}
  # 然後重試編輯
```

**進階：若用戶明確要求自動化**

可建立**獨立 slash command** `/cross-repo-prepare <target_path> <branch_name>` 處理：

- 檢查目標 repo 未提交變更
- 切換或建立分支
- 回報結果讓用戶確認

這是顯式 opt-in 路徑，避免 hook 隱式副作用。本 ticket 的 IMP follow-up 不包含此項，留作 P2 後續觀察。
