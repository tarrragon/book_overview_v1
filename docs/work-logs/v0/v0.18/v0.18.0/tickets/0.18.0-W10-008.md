```markdown
## WRAPCheck 欄位（PC-066 防護）

> **核心理念**：每回合主動自問「我是否處於 WRAP 觸發條件之一？」，防止 context 沉重時系統性遺忘輔助決策系統。

**填寫格式**：

(三反引號) WRAPCheck: [觸發狀態] - [說明] (三反引號)

**觸發狀態**：

| 狀態 | 使用情境 | 範例 |
|------|---------|------|
| 未觸發 | 本回合不符合任何 WRAP 觸發條件 | `WRAPCheck: 未觸發 - 機械性執行（讀檔案/勾選驗收）` |
| 已觸發-已執行 | 觸發條件且已執行 WRAP（含本 ticket 早先回合） | `WRAPCheck: 已觸發-已執行 - 本 ticket claim 時已完成簡化 WRAP 三問` |
| 已觸發-即將執行 | 本回合偵測觸發，下個動作將執行 WRAP | `WRAPCheck: 已觸發-即將執行 - 連續失敗 2 次，下一動作 /wrap-decision` |
| 觸發但豁免 | 觸發條件成立但有合理豁免理由（必須說明） | `WRAPCheck: 觸發但豁免 - 純文件格式修正，無設計選項可擴增` |

**WRAP 觸發條件速查**（與 wrap-decision SKILL 同步）：

- 連續失敗 2+ 次（同一問題或同一 ticket）
- PM 即將輸出「做不到 / 無法 / 禁止 / 不可能 / CLI 不支援」
- ANA Ticket claim 或分析過程中
- 重大決策（升級規則 / 新建 Skill / 改架構 / 廢棄方案）
- Debug / 錯誤根因調查
- 提案可行性評估
- 偏離核心 Wave / 救火排擠 > 30 分鐘
- 高 context 訊號（tool call > 50、session 已長）

**WRAPCheck 與 decision-tree.md 的關係**：

| 系統 | 角色 |
|------|------|
| `decision-tree.md` Context 重度檢查層 | 派發前的規則層強制路由（系統視角） |
| `WRAPCheck` 欄位 | 每回合的自我提示（PM 自我視角） |
| `decision-quality-guard-hook` | 自動偵測層（待落地） |

三層協同：規則強制 + 自我提示 + Hook 自動偵測。
```
