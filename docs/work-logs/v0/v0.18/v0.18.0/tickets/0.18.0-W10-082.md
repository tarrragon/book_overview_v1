#### 2.1 主流程 main

```

IF event_name == 'PostToolUse' AND matches MAIN_GATE_CMD:

#### 2.2 Phrase 掃描 F2

```
hits = []
FOR line_no, line IN enumerate(lines, 1):
    stripped = strip_exempt_markers_for_scan(line)   # 避 marker 內含 phrase 誤判
    FOR rule IN table:
        FOR match IN rule.pattern.finditer(stripped):
            hits.append(Hit(line_no, rule.id, rule.level, match.group()))
RETURN hits
```

同行可多規則命中（INT-9；I2 與 M3 可同命中），不去重，partition 依豁免狀態處理。

#### 2.3 豁免距離匹配 F6

```
# Phase 1 §3.2: marker 在 phrase 同行或前 1 行內生效
FOR marker IN markers (若 marker.valid):
    IF marker.line_no == hit.line_no:       RETURN True
    IF marker.line_no == hit.line_no - 1:   RETURN True
RETURN False
```

DIST-3 前 2 行不生效；DIST-4 後行不生效；DIST-5 每個 hit 獨立比對。

```
# F3
MATCH r'<!--\s*PC-093-exempt\s*:\s*([^:]+?)\s*:\s*(.+?)\s*-->' on text
IF no match: RETURN None                             # EX-N5/EX-N8
RETURN ExemptMarker(g1.strip(), g2.strip())

# F4

| ID | Regex（re.IGNORECASE + re.MULTILINE） | Rationale | 案例 |
|----|--------------------------------------|-----------|------|
| M1 | `Phase\s*[0-9]+\s*(?:再\|在)?(?:決定\|決斷\|判斷\|評估)` | 遞迴推給未來 phase，PC-093 核心反模式 | (a) W10-017.8 TD1「Phase 4 視 E1 結果決定 in-memory LRU」；(b) TD5「Phase 4 決定無 .git 時 raise」 |
| M2 | `(?:之後\|以後\|日後\|未來)\s*(?:再\|才)?(?:決定\|決斷\|說\|考慮\|處理)` | 無時間錨點＝永久延後 | (a)「use_cache 之後再決定」；(b)「CheckpointStateError 以後再處理」 |
| M3 | `保留\s*(?:以防萬一\|以備不時之需\|彈性\|擴展性\|擴充性)` | 將「不用」偽裝為「保留」（根因 D） | (a)「保留 use_cache 以防萬一」；(b)「保留 extension error 擴展彈性」 |

#### 1.2 WARN（警告）

| ID | Regex | Rationale | 案例 |
|----|-------|-----------|------|
| W1 | `視\s*.{1,30}?\s*(?:結果\|情況\|狀況\|需求)\s*(?:再\|而)?\s*(?:決定\|判斷\|評估)` | 帶條件的延後，條件明確可豁免 | (a)「視 baseline 結果再決定」；(b)「視實測情況決定」 |
| W2 | `(?:未來\|以後)\s*(?:可能\|或許\|也許)\s*(?:需要\|會用\|要用)` | 根因 B「預留樂觀」話術 | (a)「未來可能需要 cache」；(b)「以後或許會用到」 |
| W3 | `先(?:保留\|留著\|不動)\s*(?:再說\|吧)?` | 根因 C「決策疲勞」口語 | (a)「先保留再說」；(b)「先不動吧」 |

#### 1.3 INFO（提醒）

| ID | Regex | Rationale | 案例 |
|----|-------|-----------|------|
| I1 | `(?:TBD\|TODO\|FIXME)\s*[:：]?\s*(?:Phase\s*[0-9]+\|之後\|未來)` | 標記式延後，若有後續 ticket 可豁免 | (a)「TODO: Phase 4 決定」；(b)「FIXME: 之後處理」 |
| I2 | `(?:擴展\|擴充)\s*(?:彈性\|空間\|介面)` | PC-093 偽裝詞前綴 | (a)「保留擴展彈性」；(b)「提供擴充介面」 |

**合計**：8 條（M×3 / W×3 / I×2）

### 2. 觸發時機決策

#### 候選比較

| 方案 | Event | 優點 | 缺點 |
|------|-------|------|------|
| A | PostToolUse(Bash: ticket track phase X phase4) | 精準鎖定進入 Phase 4 | 只查當下，事後修改漏失 |
| B | PostToolUse(Bash: ticket track append-log) | 每次寫入即時掃 | 高頻觸發，Phase 1/2/3 過度偵測 |
| C | UserPromptSubmit | 覆蓋所有互動 | stdin 不含 ticket 上下文，須額外 read |
| D | PreToolUse(Bash: ticket track complete) | 結案閘門 | 時機晚，重修成本高 |

#### 決策：A 主 + D 輔（雙閘門，獨立 Hook）

- **主觸發 A**：PostToolUse on `ticket track phase <id> phase4`；命中 MUST-block → Exit 2；WARN/INFO → stdout
- **輔助觸發 D**：PreToolUse on `ticket track complete <id>`；二次掃描防 Phase 4 後又加入延後話術
- **排除 B**：append-log 在 Phase 1-3 寫「Phase 4 再決定」是合法過渡態
- **排除 C**：須主動 read ticket md 成本高且不知當下對應哪個 ticket

### 3. 豁免機制

#### 3.1 合法延後案例（必須豁免）

| 類型 | 案例 | 合法性 |
|------|------|--------|
| TDD 階段過渡 | Phase 1 寫「Phase 2 補 RED 測試」 | TDD 流程本身 |
| 明確條件門檻 | 「baseline > 80ms 時加 cache」 | 可驗證門檻（PC-093 §防護 2） |
| 跨 ticket 追蹤 | 「延後至 W11-xxx，已建 follow-up」 | 有後續 ticket |
| 用戶明確拒絕 | PM 判斷特殊情境 | 規則 5 用戶 autonomy |

#### 3.2 豁免語法

在命中 phrase **同行或前 1 行內** 加入標記：

```
<!-- PC-093-exempt: <category>:<reason> -->
Phase 4 再決定是否保留 use_cache。

或同行後綴：
Phase 4 再決定。<!-- PC-093-exempt: baseline-gated:baseline>80ms 才啟用 -->
```

**欄位規範**：
- `category`: `tdd-transition` | `baseline-gated` | `ticket-tracked` | `user-override`
- `reason`: ≥ 10 字元；`baseline-gated` 必含數值；`ticket-tracked` 必含 ticket id regex

#### 3.3 豁免審計（Phase 4 進入時必輸出）

```
[PC-093 豁免清單] 當前豁免：
  line 42: baseline-gated — baseline>80ms 才啟用
  line 87: ticket-tracked — 延後至 W11-005

Phase 4 結束前必須清償（改執行或移除），剩餘豁免於 complete 時 WARN。
```

### 5. 與既有 Hook 互動

#### 職責比對

| 既有 Hook | 職責 | 重疊? |
|-----------|------|-------|
| phase-completion-gate-hook | Phase 完成 worklog 上報檢查 | 無（上報 vs 話術） |
| acceptance-gate-hook | complete 前 8 項 checker | 部分（輔助觸發 D 同時作用） |
| phase-contract-validator-hook | Phase 契約驗證 | 毗鄰不重疊（契約 vs 話術） |
| ticket-quality-gate-hook | ticket 品質通用閘門 | 部分重疊 |

#### 決策：獨立 Hook（不整合）

Rationale:
1. **單一職責**：PC-093 延後話術偵測與上報/驗收職責正交
2. **Event 時機不同**：主觸發 Bash 命令 vs phase-completion-gate 監控 Write/Edit
3. **可觀察性**：獨立 `phase4-decision-enforcement` named logger 便於除錯
4. **失敗隔離**：regex 誤報時可單獨 disable 不傷 acceptance-gate
5. **協作點**：核心掃描邏輯抽函式 `scan_ticket_for_pc093(path) -> dict`，供 acceptance_checkers/ 新增子模組（如 `pc093_checker.py`）複用；Phase 3b 實作時統一
