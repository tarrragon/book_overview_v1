## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### saffron 系統一致性分析結論

### AC1: 既有設施盤點

見「重現實驗結果」。8 個設施全數盤點，無一完整涵蓋「事實宣稱驗證」。

### AC2: 重複造輪子檢測

**結論：D 不重複造輪子，但需精確定位切入點避免語意混淆。**

| 候選方案 | 與既有系統重複度 | 判定 |
|---------|---------------|------|
| A（pm-rules 規則文件） | 低——既有規則無一聚焦「事實宣稱驗證」 | 不重複 |
| B（Hook 強制） | 中——與 dispatch-gate + agent-dispatch-validation 功能域重疊，但偵測目標不同 | 邊界模糊，易膨脹 |
| C（獨立 Skill） | 高——與 wrap-decision 觸發條件重疊（都是「派發前品質 checkpoint」） | **重複造輪子** |
| D（整合 wrap-decision） | 低——延伸既有 R 階段或新增子步驟 | 不重複 |
| E（三段式 ticket） | 高——與 ANA 完整 WRAP 強制重疊 | **過度設計** |

**C 為重複造輪子**：wrap-decision 已是「派發前認知偏誤防護」的統一入口，另建 Skill 會造成 PM 決策時需記住「何時用 wrap-decision、何時用 dispatch-pre-validation」的認知負擔。

### AC3: D 切入點識別

**建議切入點：WRAP R 階段（Reality Test）新增「事實宣稱驗證」子步驟。**

理由：
1. R 階段既有職責是「現實檢驗」——驗證假設是否符合現實，語意最接近
2. 現有 R 階段聚焦「基本率」（統計性驗證），缺「確定性驗證」（grep/查文件驗證事實）
3. 新增為 R 階段子步驟（如 R.2 事實宣稱驗證）而非第五階段，避免改變 W-R-A-P 四階段結構

**具體切入**：
- 在 `.claude/skills/wrap-decision/SKILL.md` 的 R 階段段落新增「R.2 事實宣稱驗證」子節
- 觸發條件：「代理人結論含事實宣稱（如『hook 為強制』『無引用』『已刪除』）時，PM 必須 grep/查文件驗證」
- 不改變 WRAP 整體結構，不新增工具/hook/skill

**潛在風險**：wrap-decision SKILL.md 聲明「獨立於任何專案框架」（第 12 行）。「事實宣稱驗證」本身是通用概念（不限於本專案），但範例（grep hook）帶專案色彩。建議：規則寫通用（「驗證事實宣稱」），範例放專案落地層（pm-rules/）。

### AC4: A 位置建議

**建議位置：`.claude/pm-rules/dispatch-gate.md` 新增「事實宣稱驗證」段落。**

理由：
1. dispatch-gate.md 已是「所有派發前強制」的統一入口（decision-tree 第 151 行引用）
2. 新增段落不需建新檔案，維護成本最低
3. 與 Context Bundle、並行化檢查並列，語意一致（都是「派發前品質關卡」）

替代位置分析：

| 位置 | 優點 | 缺點 |
|------|------|------|
| dispatch-gate.md（建議） | 統一入口，已被 decision-tree 引用 | 段落數增加 |
| 獨立 pm-rules/assumption-verification.md | 獨立性高 | 新增檔案，PM 需記住讀 |
| decision-tree.md Context 重度檢查層 | 已有 PC-066 入口 | 該層設計為偵測訊號而非操作指引 |

### AC5: 整合衝突分析

**結論：D+A 組合無 critical 衝突。**

| 衝突維度 | 檢查結果 | 嚴重度 |
|---------|---------|--------|
| wrap-decision SKILL 通用性 | R 階段新增通用概念（事實驗證）不違反通用性；專案範例放 pm-rules/ | info |
| dispatch-gate 段落數 | 新增 1 段（約 15 行），不影響可讀性 | info |
| claim 三問與 R 階段的觸發時機差異 | claim 三問是 ticket 認領時；R 階段是 WRAP 執行時。兩者不衝突——claim 三問觸發 WRAP，WRAP 內 R 階段包含事實驗證 | 無衝突 |
| 既有 hook 系統 | 不新增 hook，不與既有 hook 衝突 | 無衝突 |
| W10-138 L3 schema | 本方案不涉及 schema 變更，不與 W10-138 衝突 | 無衝突 |

### AC6: 自問段——我可能錯在哪裡

1. **低估 PM 不讀規則風險**：D+A 都是自律機制（PM 須主動讀 dispatch-gate + 執行 WRAP R 階段）。若 PM 自動駕駛（PC-066），兩者都會被跳過。但 claim 三問已有 CLI 自動提示機制（W10-028），若 R 階段驗證寫入 WRAP SKILL 且 CLI 提示引導至 WRAP，鏈條完整度尚可。
2. **高估 W10-137 代表性**：W10-137 可能是罕見案例（代理人宣稱錯誤事實 + PM 差點直接採信）。若此模式罕見，固化機制的維護成本可能高於收益。但 PC-066 案例庫顯示「自動駕駛」是系統性問題，W10-137 只是新變體。
3. **R 階段語意擴展過度**：將「事實宣稱驗證」塞入 R（Reality Test）可能讓 R 階段職責膨脹。但 R 原義就是「現實檢驗」，事實驗證是其自然延伸。
4. **wrap-decision SKILL 跨專案 sync 問題**：若 R 階段新增內容含本專案 grep 範例，sync 到其他專案會不適用。建議：SKILL 只寫通用原則，範例放 pm-rules/。

### 總結建議

**支持 D+A 組合**，具體實施：

1. **D**：wrap-decision SKILL.md R 階段新增「R.2 事實宣稱驗證」通用子步驟（約 10 行，不含專案範例）
2. **A**：dispatch-gate.md 新增「事實宣稱驗證」段落（約 15 行，含 W10-137 案例 + grep 操作指引）
3. **不需要 B/C/E**：不新增 hook、不新增 skill、不改 ticket 流程
