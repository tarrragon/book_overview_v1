### WRAP 結論：方案 D + 提示行（複合方案）

#### W. Widen Options

| 選項 | 做法 | 風險 |
|------|------|------|
| A | IMP complete 反向 hook：自動掃 source ANA 並提醒 | hook 邏輯複雜 |
| B | 自動 close source ANA | 高（body 可能未補完，永久空白） |
| C | SessionStart 顯示 stuck ANA 列表 | 噪音風險，PM 麻木 |
| **D** | **新增 `ticket track stuck-anas` CLI（顯式查詢，無噪音）** | **低** |
| E | 純文件規則：「IMP complete 時必查 source ANA」 | 依賴自律 |
| F | 不做 | W17-008.10 案例已否決 |

**Consider Opposite**：「自動 close 是好設計嗎？」否——ANA 結論需要 body 完成度檢查，自動關閉會固化 placeholder（W17-008.10 證據）。**B 排除**。

#### R. Reality-Test

- **基本率**：GitHub/Jira/Linear 對 parent issue 在 sub-issues 全 close 時皆**只通知不自動 close**，理由與 B 排除理由相同
- **Steelman F**：W17-008.10 是 PM 注意力問題，hook 提醒可能淪為 noise → 支持 D（按需查詢）優於 C（always 顯示）
- **Ooch**：D 可在 30 分鐘內 prototype（lifecycle.py 加掃描函式 + register CLI 命令）

#### A. Distance

- **機會成本**：D 占用 hook 系統資源最少；A 高耦合風險
- **10/10/10**：D 短期低感、長期穩定；C 短期有感、長期變噪音
- **優先順序**：P2 不應擠壓核心開發

#### P. Prepare

| 方案 | Premortem 主因 | 回退 |
|------|--------------|------|
| D | PM 不主動跑命令 → 補：IMP complete 後加單行提示「→ source ANA <id> 可考慮 complete」 | 低 |

### 推薦：D + IMP complete 後單行提示

**設計**：
1. 新增 `ticket track stuck-anas` CLI 命令（顯式查詢，輸出表格）
2. 在 lifecycle.py IMP complete handler 末尾，若 source_ticket 非空，查詢該 ANA 的所有 spawned，若全 completed，輸出單行提示：
   `→ Source ANA 0.18.0-W17-008.10 spawned 全 completed，可考慮執行 ticket track complete`
3. 提示為 stdout 而非 hook，不阻擋流程

**ROI**：~50 LOC（CLI 命令 + 提示行），測試 ~30 LOC。可視 008.12 盤點結果決定是否優先。

`single_view_only`：機制設計屬 linux 視角強項（complexity vs benefit），結論明確（業界基本率支持），P2 + ROI 中等，無需多視角 panel。
