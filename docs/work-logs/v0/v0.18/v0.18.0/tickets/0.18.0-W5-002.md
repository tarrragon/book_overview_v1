### Phase 1 功能規格（lavender）

本規格建立於 PM 已寫入之 Context Bundle 與 WRAP 三問之上，不重述已決策事項，聚焦「claim 如何整合 ac_parser/validation_templates 的端到端行為契約」。

#### 1. 功能範圍與非範圍

**範圍（In-scope）**：
- `ticket track claim <ticket_id>` 在狀態更新**之前**執行 AC 自動驗證
- 新增 CLI flag `--skip-verify`（逃生閥）
- 對每個 AC 呼叫 `match_template`；若 `is_verifiable=True` 則以 `subprocess.Popen` 執行 `command`，依 `parse_strategy` 判定結果
- 互動式 y/n 二選一決策；y 繼續既有 claim 流程、n 取消
- 全部可驗證 AC 已達成時：**拒絕 claim 並提示改用 complete**（不進 prompt、不轉 complete）
- 輸出限制 15 行內的摘要
- 中斷處理（Ctrl-C / timeout）確保子程序終止、回傳非零

**非範圍（Out-of-scope）**：
- 新增模板（W5-001 已硬編碼 5 個，變更由 W5-001 owner 處理）
- 修改 `ac_parser` 或 `validation_templates` 的 API
- 驗證結果快取、持久化（每次 claim 重新執行）
- 並行執行多個驗證 command（MVP 循序執行，簡化 Ctrl-C 處理）
- 從 claim 內部呼叫 `complete`（拔除 c 選項後無此跨類別呼叫）
- 針對非 npm 環境的替代驗證（由 `unverifiable_reason` 標記「環境不符」，不重試）
- Windows 相容（MVP POSIX only，process group 終結依賴 `os.killpg`）

#### 2. User Story / 使用情境

| # | 情境 | AC 狀態 | 預期行為 |
|---|------|---------|---------|
| S1 | 無 AC Ticket | `acceptance` 為空或 None | 跳過驗證，直接進入既有 claim 流程，無額外輸出 |
| S2 | AC 全部 unverifiable | 所有 AC 的 `match_template` 回 `None` 或 `is_verifiable=False` | 顯示「無可驗證 AC（N 項）」，直接 claim，無 y/n/c 提示 |
| S3 | 部分可驗證、部分或全部未達成 | 至少 1 個 AC 執行後判定未達成 | 顯示結果摘要 → 提示 y/n，預設 y（正常開工） |
| S4 | 全部可驗證 AC 已達成（外溢達成） | 所有可驗證 AC 皆判定已達成 | **拒絕 claim**：stderr 提示「AC 已達成，請改用 `ticket track complete` 或檢討 Ticket 拆分」，return 1，不進 prompt |
| S5 | `--skip-verify` 指定 | 任何 AC 狀態 | 不執行驗證、不提示，直接走既有 claim 流程；輸出 1 行提示「已跳過 AC 驗證」 |
| S6 | 驗證執行中 Ctrl-C | 驗證執行到一半 | 中止當前 subprocess（含其子程序群）、回傳 130、不更新 Ticket 狀態 |
| S7 | 驗證 timeout | 某 command 超過 `timeout_sec` | 該 AC 標記「timeout」（視為未達成），繼續下一個 AC，不中斷整體流程 |

#### 3. CLI 介面定義

```
ticket track claim <ticket_id> [--skip-verify] [--yes]
```

| Flag | 型別 | 預設 | 行為 |
|------|------|------|------|
| `--skip-verify` | bool flag | False | 完全跳過 AC 驗證與 y/n/c 提示，直接 claim |
| `--yes` / `-y` | bool flag | False | 非互動模式：驗證仍執行，但自動選 `y`（繼續 claim）；適用 CI/script |

**互動式輸入規格**：
- Prompt 文字：`繼續 claim？[y] 繼續 / [n] 取消 (預設: y): `
- 僅在 S3 場景（有可驗證 AC 且未全部達成）進 prompt；S4 直接 return 1 不 prompt
- 接受輸入：`y`/`Y`/`yes`、`n`/`N`/`no`、空字串（採預設 y）
- 無效輸入：重新 prompt，最多 3 次後預設 `n`（取消）
- 非 tty 環境（`sys.stdin.isatty()=False`）且無 `--yes`：視為 `n`（安全預設），stderr 警告「非互動環境，已取消；請加 --yes 或 --skip-verify」

```python
@dataclass(frozen=True)
class VerificationResult:
    ac: AC
    status: Literal['passed', 'failed', 'unverifiable', 'no_template', 'timeout', 'env_error']

@dataclass(frozen=True)
class VerificationSummary:
    total: int
    passed: int
    failed: int
    unverifiable: int           # 含 no_template + manual + env_error + timeout

```

#### 5. 資料流

```
claim(ticket_id, skip_verify, auto_yes)
    │
    ├─ skip_verify? ─yes─► 輸出「已跳過」→ 既有 claim 流程 → return 0
    │
    └─ collect_ac_verifications(ticket_id)
          │  uses: parse_ac → list[AC]
          │  uses: match_template → ValidationCommand | None
          ▼
       list[(AC, ValidationCommand | None)]
          │
          ▼
       run_all_verifications(pairs)
          │  for each pair:
          │    - vc is None → VerificationResult(status='no_template')
          │    - vc.is_verifiable=False → VerificationResult(status='unverifiable', message=unverifiable_reason)
          │    - is_verifiable=True → execute_verification(ac, vc)
          │        │
          │        ▼
          │     subprocess.run(vc.command, shell=True, timeout=vc.timeout_sec, capture_output=True)
          │        │
          │        ▼
          │     apply_parse_strategy(vc.parse_strategy, stdout, stderr, exit_code)
          │        │
          │        ├─ tail_lines: 取 stdout 最後 5 行，regex 找 /(\d+) failed/ 或 /(\d+) passing/；exit_code=0 且無 failed → passed
          │        ├─ exit_code: exit_code=0 → passed，否則 failed
          │        ├─ coverage_number: regex 擷取覆蓋率百分比；>= 門檻 → passed（門檻暫訂 80%，見缺口 §D）
          │        └─ manual: 不應在此被呼叫（is_verifiable=False）
          ▼
       list[VerificationResult]
          │
          ├─ summarize_results → VerificationSummary
          │
          ▼
       status == 'no_ac' 或 'none_verifiable' ─► 不 prompt，直接 claim（顯示 1 行摘要）
       status == 'all_passed' ─► stderr 提示「AC 已達成，請改用 complete 或檢討拆分」→ return 1（不 prompt、不 claim）
       status == 'has_failures' ─► render_results → print → prompt_user_decision
                │
                ├─ 'y' ─► 既有 claim 流程（status 更新、輸出 checklist）
                └─ 'n' ─► return 1（未更新 Ticket）
```

#### 6. 輸出格式規格（15 行內硬上限）

**S3 場景（有可驗證 AC 且存在未達成）**：

```
[AC 驗證] Ticket 0.18.0-W5-002 (6 項)
  [PASS] #1 npm test 通過 (npm_test_pass, 12.3s)
  [FAIL] #2 覆蓋率 >= 80% (coverage_threshold, 實測 72%)
  [SKIP] #3 lint 通過 (lint_pass, exit=2)
  [N/A]  #4 flaky 測試已修復 (flaky_fixed: 需人工審查)
  [----] #5 文件已更新 (無模板匹配)
  [PASS] #6 輸出 15 行內 (manual 檢查已達成)

結果：2 passed / 1 failed / 3 unverifiable
繼續 claim？[y] 繼續 / [n] 取消 (預設: y):
```

**行數核算（上限 15）**：標題 1 + AC 清單最多 N 行（N <= 10，超出則摘要為「... (N-10 more)」）+ 空行 1 + 結果摘要 1 + prompt 1 = 最多 14 行（有餘裕）。

**S2 場景（全部 unverifiable）**：2 行
```
[AC 驗證] Ticket 0.18.0-W5-002：6 項 AC 皆無法自動驗證（5 no_template + 1 manual）
已略過驗證，直接進入 claim 流程...
```

**S4 場景（全部可驗證 AC 已達成）**：2 行，stderr 輸出，return 1
```
[AC 驗證] Ticket 0.18.0-W5-002：N 項可驗證 AC 皆已達成
建議改用 `ticket track complete <ticket_id>`，或檢討 Ticket 是否需拆分
```

**S5 場景（--skip-verify）**：1 行
```
[AC 驗證] 已跳過（--skip-verify）
```

**狀態標籤對應**：PASS=passed, FAIL=failed, SKIP=timeout/env_error, N/A=unverifiable(manual), ----=no_template

#### 7. 錯誤處理邊界

| 觸發條件 | 偵測方式 | 對應行為 | Exit Code |
|---------|---------|---------|-----------|
| AC command timeout | `subprocess.TimeoutExpired` | 該 AC 標 `status='timeout'`、`message='超過 {N}s 未完成'`、終止 process group | 繼續下一 AC |
| subprocess 執行失敗（`FileNotFoundError`，如無 npm） | `OSError` on `subprocess.run` | `status='env_error'`、`message='環境缺 npm 或 command 不可用'` | 繼續下一 AC |
| Ctrl-C 中斷 | `KeyboardInterrupt` | 終止當前 subprocess（`os.killpg`）、print `中斷：未更新 Ticket` 到 stderr | 130 |
| `parse_ac` 拋 `FileNotFoundError` | try/except | 既有錯誤處理（`load_and_validate_ticket` 已處理此情境） | 1 |
| `parse_ac` 拋 `ValueError`（YAML 損毀） | try/except | print `[Warning] AC 解析失敗：{err}`、降級為「跳過驗證，直接 claim」 | 走既有 claim |
| 非 tty 且無 `--yes` 與 `--skip-verify` | `sys.stdin.isatty()` | stderr 警告後 `return 1` | 1 |
| `--yes` 且驗證全 failed | summary.status 分支 | 自動選 `y`、顯示結果後繼續 claim（使用者已明示接受） | 走既有 claim |
| 全部可驗證 AC 已達成（S4） | summary.status == 'all_passed' | stderr 輸出建議訊息、不進 prompt | 1 |

**子程序終結可靠性要求**：
- 使用 `subprocess.Popen(..., start_new_session=True)` 或 `preexec_fn=os.setsid`（POSIX）建立 process group
- `KeyboardInterrupt` / `TimeoutExpired` 時呼叫 `os.killpg(pgid, SIGTERM)`，1 秒後再 `SIGKILL`
- 驗證執行前 print `執行中: {template_name}... (timeout {N}s，Ctrl-C 中斷)`（stderr，不計入 15 行輸出上限）

#### 補遺 1：c 選項二次確認的實作位置 [作廢 — 2026-04-14 PM 仲裁拔除 c 選項]

> **本補遺整章作廢**：PM 採納 linux Good Taste P0 否決權，§B.2「MVP 加二次確認」已推翻，c 選項完全移除。`confirm_transfer_to_complete` 不再存在。保留原文僅供歷史追溯。Phase 2 不得基於本補遺設計測試。

**決策**：採 (b) SRP 拆分為獨立函式 `confirm_transfer_to_complete`。

| 項目 | 值 |
|------|---|
| 函式位置 | `commands/claim_verification.py` |
| 函式簽名 | `confirm_transfer_to_complete(auto_yes: bool) -> bool` |
| 呼叫位置 | `prompt_user_decision` 內部：當使用者選 `c` 後，再呼叫此函式二次確認；回傳 `True` 才回傳 `'c'`，否則重新進 prompt 主迴圈 |
| Prompt 文字 | `確定轉為 complete 並記錄 AC 外溢達成? [y/N]: ` |
| 預設值 | N（空字串、無效輸入皆視為 N） |
| 無效輸入處理 | 3 次後預設 N（取消轉 complete，回主 prompt 重新選擇 y/n/c） |
| `--yes` 行為 | `auto_yes=True` 時直接回 `True`（已在 §3 標示 --yes 為 non-interactive 短路） |
| 非 tty 行為 | `sys.stdin.isatty()=False` 且 `auto_yes=False` → 回 `False`（安全預設，不轉 complete） |

**拆分理由**：`prompt_user_decision` 已負責 y/n/c 三選一主邏輯（含 S3/S4 預設切換、非 tty 警告），再塞入二次確認會讓認知負擔指數超過 10；獨立函式也讓 Phase 2 測試 mock 目標單純（mock `confirm_transfer_to_complete` 即可隔離二次確認路徑）。

#### 補遺 2：subprocess 的 cwd 與 env

|------|---|
| `cwd` | 專案 root，透過 `subprocess.run(['git', 'rev-parse', '--show-toplevel'], capture_output=True, text=True, check=True).stdout.strip()` 取得；失敗（非 git 專案 / git 不可用）時 fallback 為 `os.getcwd()`（ticket CLI 執行目錄） |
| `env` | **Inherit parent**（不傳 `env=` 參數，使用 `os.environ`），包含 `PATH`、`FLUTTER_ROOT`、`NODE_PATH` 等 |
| 取得時機 | `execute_verification` 執行前一次性取得 `project_root`，由 `run_all_verifications` 傳入（避免每個 AC 重複呼叫 git） |
| 實作位置 | 新增 helper `resolve_project_cwd() -> str` 於 `lib/project_root.py`（採納 saffron 建議提升至 lib/，跨命令共用） |

**完整 subprocess 呼叫**：

```python
subprocess.Popen(
    vc.command,
    shell=True,
    cwd=project_cwd,           # 補遺 2：專案 root
    env=None,                  # 補遺 2：inherit os.environ
    stdout=subprocess.PIPE,
    stderr=subprocess.PIPE,
    start_new_session=True,    # §7 process group 終結要求
)
```

**理由**：
- `cwd=專案 root` 確保 `npm test` / `flutter test` 等命令在正確目錄執行（使用者可能在 `docs/` / `.claude/skills/ticket/` 等子目錄執行 `ticket track claim`）
- `env` 完全 inherit 避免 FLUTTER_ROOT / NVM 等環境變數遺失（whitelist 維護成本高且易遺漏）
- fallback `os.getcwd()` 涵蓋非 git 專案情境（未來若 ticket CLI 被其他專案重用）

**對 Phase 2 測試影響**：

#### 補遺 3：--yes 與 --skip-verify 衝突時的使用者提示

**決策**：加 stderr 訊息告知 `--yes` 已被忽略。

| 項目 | 值 |
|------|---|
| 觸發條件 | `skip_verify=True` 且 `auto_yes=True` 同時為真 |
| 訊息通道 | stderr（不污染 stdout 的結果輸出、不計入 15 行上限） |
| 訊息文字 | `[Warning] --yes 已被忽略（同時指定 --skip-verify 時不執行驗證，--yes 無作用）` |
| 觸發位置 | `claim_with_verification` 入口處，於「已跳過」1 行提示**之前**輸出 |
| 退出路徑 | 訊息輸出後繼續走 skip_verify 路徑（既有 claim 流程），exit code 0 |

**理由**：
- 使用者顯式同時傳兩個 flag 通常是 CI script 維護時的殘留（複製貼上 `--yes` 習慣），靜默忽略會讓 CI 行為與預期不符時難以排查
- stderr 而非 stdout 避免污染「已跳過」摘要輸出，也符合 Unix 慣例（warning 走 stderr）
- 不改變既有優先級決策（PM §B.3：`--skip-verify` 優先），僅加可見性
