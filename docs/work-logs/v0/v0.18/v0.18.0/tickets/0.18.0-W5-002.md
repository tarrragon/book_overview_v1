## Problem Analysis

### 簡化 WRAP 三問（claim Checkpoint）

**W（Widen）—— 有其他做法嗎？**
1. [預設] claim 前自動執行所有可驗證 AC，結果 + y/n/c 三選一
2. [候選] `ticket track claim --verify` 顯式觸發，預設跳過驗證（低侵入、但易被遺忘）
3. [候選] claim 後僅警告顯示 AC 狀態，不中斷（最弱防護，回到 PC-055 問題）

選 (1) 理由：PC-055 的核心問題是「代理人做已完成工作」，必須在 claim 路徑上強制 prompt，非可選。但加 `--skip-verify` 逃生閥以處理不適用場景（見 P）。

**A（Attain distance）—— 機會成本是什麼？**
- 時間成本：每次 claim 多 0-300 秒（視 AC 類型）
- 擠壓目標：TDD Phase 1 起步時間
- 判斷：W5 整體目的是降低錯誤派發率（每次誤派發 = 代理人回合耗盡 + PM 重新規劃），單次誤派發成本 >> claim 驗證成本。值得。

**P（Prepare to be wrong）—— 最可能失敗的原因是什麼？**
- 最可能：npm test 類 command 在大專案耗時過長，使用者被迫等待 / Ctrl-C 中斷後子程序洩漏
- 12h 後失敗情境：使用者為了加速放棄使用 claim 命令，回到手動 claim → 功能形同不存在
- 防護措施：
  1. 支援 `--skip-verify` flag（逃生閥）
  2. 統一 timeout（使用 `validation_templates` 的 timeout_sec）
  3. 子程序終結必須可靠（`subprocess.run(..., timeout=...)` + 終止訊號傳遞）
  4. 每個驗證執行前顯示「執行中...」提示，給予中斷機會
### Context Bundle（PM 前台分析）

**W5-001 產物（可直接使用的 API）**：
- `ticket_system.lib.ac_parser.parse_ac(ticket_id) -> list[AC]`
  - 回傳 `AC(index, text, checked, raw)` frozen dataclass
  - 已處理 checkbox 前綴剝除（`[x]` / `[ ]` / 無前綴視為未勾選）
- `ticket_system.lib.validation_templates.match_template(ac_text) -> ValidationCommand | None`
  - 5 個模板：npm_test_pass / coverage_threshold / lint_pass / flaky_fixed / skipped_evaluated
  - 大小寫無關子字串匹配，第一註冊者勝出
  - 回傳 `ValidationCommand` 含 `command` / `timeout_sec` / `parse_strategy` / `is_verifiable` / `unverifiable_reason`

**整合位置（TicketLifecycle.claim）**：
- 檔案：`.claude/skills/ticket/ticket_system/commands/lifecycle.py:307-375`
- 插入點：在 `validate_claimable_status` 之後、`_print_phase_prerequisite_warning` 之前（或之後，視 phase 警告順序）
- 實際執行 claim 的三行：line 362-364（status/assigned/started_at 設定）

**必要的新模組/函式**：
1. AC 驗證執行器：對 `is_verifiable=True` 的 AC 執行 command，解析 parse_strategy
2. 結果呈現器：將每個 AC 的 (已達成 / 未達成 / 無法驗證 / 無模板匹配) 列印（限 15 行）
3. y/n/c 互動器：讀 stdin，y=繼續、n=abort return 1、c=轉 complete（呼叫既有 complete 流程）

**AC 解析與驗證的邊界**：
- AC 無 `acceptance` 欄位 → 跳過驗證，直接 claim
- AC 清單全部 `unverifiable`（無模板匹配或 `command=None`）→ 顯示「無可驗證 AC」，直接 claim
- AC 全部 `checked=True` → 自動觸發「可能已外溢達成」提示，預設 c 選項

**風險與已知陷阱**：
- `npm test` 執行耗時（可達 300 秒），使用者可能無法等待 → 必須支援 `--skip-verify` flag 或 timeout 中斷
- Ctrl-C 必須能正常中斷、回傳非零（不 leak 子程序）
- 部分環境無 npm（僅 Python 工具鏈）→ command 執行失敗應標記「環境不符」非「未達成」
