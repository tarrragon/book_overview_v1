## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

<!-- PC-093-exempt: ticket-tracked:引用 W14-042.1 Phase 4 評估歷史結論 -->
W14-042 修復 ticket_builder.py 兩個 update_* 函式的 logical race。bay-quality-auditor 與 linux 在 Phase 4 評估指出其他 caller（context_bundle_extractor / lifecycle 等）有相同 load→modify→save pattern 未受保護。本 ticket 系統審計所有 caller 確認潛在 race 點。

### 影響範圍

`.claude/skills/ticket/ticket_system/` 全部含 `save_ticket` 的 commands 與 lib 模組。

### 相關 Error Pattern

- W14-005 重現實驗（lost_rate 55.6-71.9%）證實 read-modify-write 序列即使 single f.write atomic 仍會 race
- PC-073 follow-up 為 spawned 非 children
## 重現實驗結果

### 實驗方法

`grep -rn "save_ticket" .claude/skills/ticket/ticket_system/ --include="*.py"` 列出所有 save_ticket 呼叫位置；對每處檢查同函式是否有對應 load_ticket（read-modify-write pattern）。

### 實驗執行

掃描 lib/ 與 commands/ 共 18 個 Python 檔案。識別含 load→modify→save 序列的函式並評估並發機率：

| Caller 位置 | Pattern | 並發機率 | 觸發時機 |
|---|---|---|---|
| context_bundle_extractor.py:852→880 | load → 修改 _body → save | **中** | claim 觸發自動 context bundle 注入 |
| commands/lifecycle.py:324→477/734/813/895 | load → 修改 status/started_at → save | 中 | claim/complete/release CLI 命令 |
| commands/track_acceptance.py:484→506, 528→617 | load → 修改 acceptance → save | 低-中 | check-acceptance / set-acceptance |
| commands/fields.py: 5 處 set-X | load (上游) → 修改 5W1H → save | 低 | set-who/what/when/where/why/how/etc |
| commands/track_relations.py:77→196/301/358 | load → 修改 blockedBy/relatedTo → save | 低 | set-blocked-by / set-related-to |
| commands/migrate.py:140→195/345 | load → 修改 → save | 極低 | 手動 migration 命令 |
| commands/version_shift.py:225/396 | load (從 path) → 修改 → save | 極低 | 版本遷移命令 |
| commands/track_batch.py:346 | load 上游 → 修改 → save | 低 | batch 命令 |
| commands/track_set_acceptance.py:156 | load → 修改 → save | 低 | set-acceptance toggle |
| **ticket_builder.py update_***（已保護） | load → modify → save | 高 | — |

### 實驗發現

1. **context_bundle_extractor.extract_and_apply_context_bundle** 為次高風險點：claim 自動觸發，若 PM 與 worktree agent 同 ticket 操作可能並發
2. **lifecycle.py claim/complete/release** 為中風險：PM CLI + hook + agent worktree 可能對同 ticket 並發狀態變更
3. **fields/track_acceptance/track_relations** 為低風險：通常單一 PM session 序列執行，但 PC-040 dispatch 後 agent 並行 commit 偶有並發
4. **migrate/version_shift/batch** 為極低風險：手動命令，並發機率近零

所有未保護的 caller 在概念上具相同 race 模式，但實際並發機率由使用者操作模式決定。優先級依機率排序。
