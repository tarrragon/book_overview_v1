### 共同特徵提煉（5 條）

**特徵 1：全部發生在代理人完成回報後的 PM 回覆中**

**Why**：代理人完成通知觸發 PM 的「結果彙報 + 下一步規劃」行為模式。此模式中列選項是自然反應（「完成了，接下來有 A/B/C 可做」），但正是這個自然反應最容易繞過 AUQ 工具呼叫。

**Consequence**：代理人完成回報是 PM 最高頻的決策點，若此場景系統性繞過 AUQ，則 AUQ 規則的實際覆蓋率遠低於名義覆蓋率。

**Action**：L4 行為層 checklist 應將「代理人完成回報後」標記為 AUQ 必檢觸發點。

**特徵 2：工作記憶越沉重，違規頻率越高**

**Why**：5 次違規的工作記憶狀態從中度（#1）遞增至極高度（#5），與 PC-066 自律失效曲線吻合——context 使用率越高，主動觸發規則的機率越低。

**Consequence**：session 後段（context > 60%）幾乎必然違規，因為 PM 的注意力已被代理人狀態追蹤、ticket 管理等任務佔滿，AUQ 規則在注意力競爭中落敗。

**Action**：L2 hook 設計必須覆蓋 session 後段場景，不能依賴 PM 自律；hook 是唯一在高 context 壓力下仍可靠的防護層。

**特徵 3：低 stakes 感知降低 AUQ 使用動機**

**Why**：5 次違規場景中，PM 均將列選項視為「快速確認下一步方向」而非「重大決策」。PC-064 陷阱模式第一條「這只是快速確認，用純文字比較方便」精確命中。低 stakes 感知讓 PM 認為不值得為此多做一步工具呼叫。

**Consequence**：AUQ 規則的觸發條件（規則 1：2+ 候選項）不區分 stakes 高低，但 PM 的行為決策隱性地以 stakes 作為豁免依據——這是規則與行為的落差來源。

**Action**：L3 規則升級應明確「AUQ 觸發不因 stakes 高低豁免」；L4 checklist 應含「即使覺得不重要，仍需 AUQ」。

**特徵 4：Recommended 標記或隱性推薦幾乎全程存在**

**Why**：5 次中至少 4 次含顯式 (Recommended) 標記或隱性推薦語氣。PM 在列選項時傾向預設推薦一個方向，這降低了使用 AUQ 的動機（「反正我已經告訴用戶哪個比較好了」）。

**Consequence**：帶 Recommended 標記的純文字選項比不帶標記的更危險——用戶更可能直接接受推薦而不深思，與 ai-communication-rules 規則 5（權力不對等）的主體性保護原則衝突。

**Action**：L2 hook 應將「含 Recommended / 推薦 / 建議 字樣 + 列選項 + 無 AUQ」視為高嚴重度違規訊號；L3 規則應將 Recommended 標記列為 AUQ 強制觸發的額外訊號。

**特徵 5：表格類型多元但 hook 偵測盲區一致**

**Why**：5 次違規涵蓋三種呈現形式（Markdown 列表、Markdown 表格、隱性建議），但現有 hook（askuserquestion-reminder-hook）僅偵測 Task 派發場景，對 PM 回覆文字的選項 pattern 完全無覆蓋。

**Consequence**：無論 PM 用何種表格形式列選項，hook 都無法攔截。偵測盲區不因表格類型而異——盲區在 hook 的觸發條件（僅 Task 派發），非 pattern 解析能力。

**Action**：L2 hook 設計的核心挑戰是將觸發條件從「Task 派發」擴展到「PM 回覆文字含選項 pattern」，且須覆蓋多種呈現形式（列表、表格、隱性建議）。

### 與 PC-064 三層複合失效根因對照

| PC-064 根因層 | 本 session 5 違規的映射 | 補強發現 |

| 規則落地深度不足（連帶） | CLAUDE §1.1 已寫但 session 後段幾乎無效 | 補強：規則效力隨 context 使用率遞減，session 後段需 hook 兜底 |

### 結論

狀態 (a) 已決策：5 違規時刻表與 5 條共同特徵已完成結構化提煉。核心發現為——(1) 觸發場景高度一致（代理人完成回報後）；(2) 工作記憶壓力與違規頻率正相關；(3) 低 stakes 感知是行為層主要豁免藉口；(4) Recommended 標記加劇權力不對等風險；(5) hook 偵測盲區在觸發條件層而非 pattern 解析層。以上 5 條特徵為 W17-174.3（L3 規則升級）和 W17-174.4（L4 行為層 checklist）提供具體訊號來源。
