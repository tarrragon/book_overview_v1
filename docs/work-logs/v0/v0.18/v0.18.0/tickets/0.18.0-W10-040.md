## Problem Analysis

### 問題根因

ARCH-017 v1.1.0 引入「串行兄弟」合法例外（4 條件），但方法論為文字約束，PM 在 context 沉重時可能濫用「串行兄弟」為藉口繞過 ARCH-017 主原則。需 hook 自動化檢查 4 條件，避免人為判斷漂移（PC-066 模式）。

### 影響範圍

- `.claude/hooks/`（新增 sibling-blockedby-validator-hook.py）
- `.claude/skills/ticket/ticket_system/commands/lifecycle.py`（claim/complete 觸發點）
- `.claude/error-patterns/architecture/ARCH-017-sibling-hidden-dependency.md`（規範依據）

### 相關 Error Pattern

- ARCH-017 v1.1.0：兄弟 Ticket 隱藏依賴（本 ticket 自動化目標）
- PC-066：context 沉重時自律失效（本 ticket 防護動機）

### 簡化 WRAP 三問

#### W (Widen) — 候選方案

| 方案 | 說明 | ROI |
|------|------|-----|
| (a) 純警告 hook | PostToolUse claim/complete 偵測違規後 stderr 警告 | 低成本，弱強制力 |
| (b) 強制阻擋 hook | PreToolUse claim/complete 阻擋違規通過 | 高強制，誤報成本高 |
| (c) 純文件約束 | 不寫 hook，靠 SKILL + 方法論宣導 | 最低成本，最弱強制力 |
| (d) 混合方案（推薦） | 預設 warn，違規累計或重複時 block；提供 --acknowledge flag 由 PM 確認豁免 | 平衡強制力與誤報成本 |

選 (d)：4 條件中 2 個（規格→實作時序、不可深度化）是語義判斷，純 block 誤報率高；warn + ack 流程能保留 PM 判斷空間。

#### A (Attain) — 機會成本

執行 W10-040 會擠壓：
- 同 wave 高優先 ticket（70+ pending）
- Hook 維護成本：ARCH-017 規則演化時需更新 hook 邏輯
- 4 條件語義判斷的測試覆蓋成本（每條件 ≥ 2 案例）

替代不做的成本：PM 自律 + 文件約束已部分阻擋（atomic-ticket-methodology + ARCH-017 v1.1.0 已存在），但 PM 在 context 沉重時自律失效（PC-066 模式）。

#### P (Prepare) — 失敗預想

12 小時後失敗最可能原因：
1. 「規格→實作時序」靠 type 推測（ANA→IMP）誤報純 IMP 兄弟序列
2. 「不可深度化」無法自動判斷，需 PM 確認 → 加 --acknowledge flag
3. Hook 解析 ticket frontmatter 時遇舊版欄位漂移失敗

防護：預設 warn 不 block；提供 --acknowledge flag；Hook 內加 try/except + fallback warn-only 模式。

## Solution

### TDD Phase 1：功能規格

> 採 WRAP 結論方案 (d) 混合：預設 warn + 累計違規或 --acknowledge flag 控制 block。

### 1. Hook 觸發點

| 觸發點 | 行為 | 理由 |
|--------|------|------|
| `ticket track claim` PreToolUse 或 PostToolUse | 偵測 + warn / block | claim 是 PM 確認任務開始的 checkpoint |
| `ticket track complete` PreToolUse | 偵測 + warn / block | complete 是任務鏈結束的最後攔截點 |

### 2. 偵測邏輯

對目標 ticket 與其兄弟（同 parent_id）執行：

```
For each sibling_id in target.blockedBy:
    if sibling.parent_id == target.parent_id:  # 同層兄弟
        檢查 4 條件 → 任一不滿足 → 標為違規
```

### 3. 4 條件檢查規則

| 條件 | 檢查方法 | 違規訊號 |
|------|---------|---------|
| 單向（A→B，B 不依賴 A 以外的兄弟） | 兄弟集合內依賴計數 | B.blockedBy 含 ≥ 2 個兄弟 |
| 無環 | DAG 檢查 | 存在循環引用 |
| 規格→實作時序 | 依 type 啟發式判斷（ANA→IMP / DOC→IMP 視為合法） | 純 IMP→IMP 兄弟序列（誤報率高，warn-only） |
| 不可深度化 | 無法自動判斷 | 一律 warn 並要求 --acknowledge |

### 4. 行為分級

| 違規數 | 行為 |
|--------|------|
| 條件 1-2 不滿足 | block（結構錯誤可自動判定） |
| 條件 3-4 不滿足且無 --acknowledge | warn + 提示加 --acknowledge |
| 含 --acknowledge | 通過並記錄至 ticket Problem Analysis |

### 5. 邊界 case

- 缺 parent_id 或 children → skip（無兄弟結構）
- blockedBy 引用不存在 ticket → 既有 hook 已處理，不重複
- frontmatter 解析失敗 → fallback warn-only

### 6. CLI 整合

新增 `--acknowledge "<reason>"` flag 至 `ticket track claim/complete`，違規通過時將 reason 寫入 ticket Problem Analysis 章節（追蹤責任）。

### 7. 接續 Phase 2 入口

Phase 2（sage-test-architect）需設計：
- 4 條件每條 ≥ 2 違反案例 + 1 合法案例
- block vs warn 行為分離測試
- --acknowledge flag 整合測試
- frontmatter fallback 測試
