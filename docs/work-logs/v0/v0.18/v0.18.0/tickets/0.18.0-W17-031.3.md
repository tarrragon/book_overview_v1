### 設計決策

判定規則對應 ticket md「Readiness 判定規則」表：

| Tag | 判定條件 |
|-----|---------|
| READY | exit_status=success 或 Context Bundle 段落非空 |
| NEEDS-CTX | exit_status=needs_context |
| BLOCKED | exit_status=blocked |
| FAILED | exit_status=failed |
| NO-CB | 無 exit_status 且無 Context Bundle 段落 |

partial_success / 未知 status 值 → fallthrough 至 Context Bundle 判定（fail-open，避免 schema 演進破壞）。

### 實作摘要

1. `track_runqueue.py` 新增 `_has_context_bundle(ticket)`：透過 `find_section(body, "Context Bundle")` 定位，以正則去除 HTML 註解後判斷 strip() 是否非空。
2. 新增 `_compute_readiness(ticket, handoff_info)`：依規則表計算 readiness tag；exit_status 對應映射 + Context Bundle fallback。
3. `_render_list` 在 priority 後插入 `[<readiness>]` 標籤；既有 priority 排序與 blockedBy 過濾邏輯不變。
4. `render_runqueue` 在 list 視圖時無條件載入 handoff_info（不再僅 resume 模式），供 readiness 計算使用。
5. 既有 W17-031.1 exit_status tag 行為（resume 模式 `[needs_context]` 等 suffix）完全保留，未動。

### 不在本 ticket 範圍

- dispatch-check / handoff 子命令的 readiness 整合（W17-003 範圍）
- batch dispatch CLI（W17-029 已決策排除）
