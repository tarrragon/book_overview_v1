## Problem Analysis
並行 session commit bleed 分析：W17-010 commit b5b6ec7b 因寬範圍 git add 混入 W17-020/W17-016.3 實作。詳見下方根因與實驗。

### 重現實驗結果

#### 實驗方法

透過 git log / git show 重建事件時序，驗證哪些檔案被哪個 commit 收錄。

#### 實驗執行

1. `git show b5b6ec7b --stat`：確認 15 files changed，含 track_runqueue.py（W17-020 範圍）和 ticket_validator.py（W17-016.3 範圍）
2. `git show b5b6ec7b -- track_runqueue.py`：diff 明確包含 W17-020 的 context 參數新增和 resume 訊息分支
3. `git show b5b6ec7b -- test_track_runqueue.py`：新建 106 行測試檔，docstring 明寫 (W17-020)
4. `git show b5b6ec7b -- ticket_validator.py`：diff 含 validate_execution_log_by_type 完整函式（W17-016.3 範圍）
5. `git show 9d351fe7 --stat`：W17-020 commit 僅改 1 file（ticket md），程式碼 diff 為零
6. `git show 182e9404 --stat`：W17-016.3 commit 僅改 2 files（1 行 mock patch + ticket md）
7. 時序：b5b6ec7b 21:51 -> 9d351fe7 21:56（5 分鐘後）-> 182e9404 22:14（23 分鐘後）

#### 實驗發現

**已驗證事實**：
- b5b6ec7b 確實收錄了 W17-020 的完整實作（track_runqueue.py + test_track_runqueue.py）
- b5b6ec7b 確實收錄了 W17-016.3 的核心實作（ticket_validator.py 的 validate_execution_log_by_type + 中文 placeholder 偵測）
- W17-020 / W17-016.3 的後續 commit 實質為空或極小（修 mock + 更新 ticket md）
- commit message 與 diff 範圍完全不對齊

**仍未驗證假設**：
- 無法確認是 PM 手動 git add . 還是 agent 自動使用寬範圍 add（需 session transcript，不可取得）
- 無法確認三個 ticket 是同一 session 並行派發還是不同 terminal（時序接近暗示同 PM session）
### 問題根因

**直接原因**：W17-010 session 的 commit（b5b6ec7b，21:51:35）使用了範圍過寬的 `git add`，將同一 working tree 上 W17-020（track_runqueue.py context 參數 + 6 tests）和 W17-016.3（validate_execution_log_by_type 函式 + placeholder 偵測強化）的未 commit 檔案一併 staged 並 commit。

**因果鏈**：
1. 三個 ticket（W17-010 / W17-020 / W17-016.3）在同一 session 或並行 session 中同時修改主 repo working tree
2. W17-010 agent commit 時 staging 範圍涵蓋了 ticket 範圍外的已修改檔案（track_runqueue.py / ticket_validator.py / test_track_runqueue.py）
3. W17-020 agent 後續 commit（9d351fe7，21:56）發現程式碼 diff 已空，僅能 commit ticket md
4. W17-016.3 agent commit（182e9404，22:14）同理，主實作已在 b5b6ec7b，僅補 mock patch

**根本原因**：並行 ticket 共用同一 working tree + 無 commit scope guard（agent commit 前未驗證 staged 檔案是否屬於自身 ticket where.files 範圍）。此為 PC-092（並行代理人 git index 競爭）的變體——不是 index.lock 競爭，而是 staging 範圍溢出。

## Solution
改善方案三方向：流程規範（禁 git add .）+ commit scope guard hook + worktree 隔離。建議 PC-108 error pattern。

## 分析報告

### 問題敘述

2026-04-20 21:51，W17-010 commit（b5b6ec7b）收錄了 15 files，其中至少 4 個檔案屬於 W17-020 和 W17-016.3 的實作範圍。後續兩個 ticket 的 commit 實質為空（僅 ticket md 或 1 行 mock 修補）。commit message 標示 W17-010 但 diff 涵蓋三個 ticket 的程式碼。

### 根因分析（5W1H）

| 項目 | 內容 |
|------|------|
| What | b5b6ec7b 的 staging 範圍溢出，收錄了非 W17-010 範圍的檔案 |
| Who | W17-010 的執行 agent（或 PM commit 動作） |
| When | 2026-04-20 21:51:35，三個 ticket 並行開發窗口內 |
| Where | 主 repo working tree（非 worktree 隔離） |
| Why | 並行 ticket 共用 working tree + commit 前無 scope 驗證 |
| How | agent/PM 使用 git add . 或 git add -A，將 working tree 全部已修改檔案 staged |

**因果鏈**：共用 working tree -> 多 ticket 修改同時存在 -> 寬範圍 git add -> staging 溢出 -> commit 邊界錯誤 -> 後續 agent 撲空

### 改善方案

#### 方向一：症狀修復（commit scope guard hook）

在 agent commit 前新增 pre-commit hook 或 dispatch hook 邏輯：
- 從 ticket where.files 取得允許的檔案路徑 pattern
- 比對 git diff --cached --name-only 與 ticket scope
- 若 staged 檔案包含非 ticket 範圍的檔案，警告或阻擋

**優點**：直接防止溢出。**限制**：where.files 可能不完整；新建檔案無法預先列入。需設計 escape hatch。

#### 方向二：根因防護（worktree 隔離強化）

並行 ticket 強制使用獨立 worktree：
- PM parallel-dispatch 時自動建立 worktree
- 每個 agent 在獨立 worktree 工作，物理隔離 staging 區
- commit 後 merge 回 main

**優點**：從根本消除共用 working tree 問題。**限制**：.claude/ 修改受 CC runtime 限制（ARCH-015）；worktree 管理成本；已有 PC-092 討論此方向。

#### 方向三：流程規範（agent commit 最佳實踐）

強化 agent definition 和 PM 派發 prompt：
- Agent commit 時禁止 git add . / git add -A，改用 git add <specific-files>
- PM 派發 prompt 明列允許修改的檔案
- PM commit 前執行 git diff --cached --name-only 自檢

**優點**：零成本、立即可用。**限制**：依賴紀律，無強制機制。

### 建議

1. 短期（低成本）：方向三。在 agent-definition-standard.md 禁止行為段落新增「禁止 git add . / git add -A，必須 git add <specific-files>」。PM 派發 prompt 含 where.files 清單。
2. 中期：方向一。建 commit-scope-guard hook，從 dispatch-active.json 讀取當前 ticket 的 where.files，比對 staged files。
3. 長期：方向二配合 PC-092 的 worktree 隔離策略一併推進。

### PC Error Pattern 草稿

建議新增 **PC-108: 並行 ticket commit staging 溢出（commit bleed）**，歸類為 PC-092 的子模式：

- **症狀**：agent commit 的 diff 包含非自身 ticket 範圍檔案；後續 ticket 的 commit diff 為空或僅含 ticket md
- **根因**：共用 working tree + git add . / git add -A
- **防護**：(1) agent 禁用寬範圍 add (2) commit-scope-guard hook (3) worktree 隔離
- **嚴重度**：中（程式碼不遺失但 git 歷史歸因錯誤 + agent 工作浪費）
- **姊妹模式**：PC-092（index.lock 競爭）、PC-076（跨 session 遺留）

### 後續 Ticket 建議

本 ANA 不建 IMP（依 ticket 指示「不建 IMP，建議範圍在 Solution 描述即可」）。若 PM 決定落地，建議：
- IMP-A：agent-definition-standard.md 新增禁止 git add . 條款（方向三，P1）
- IMP-B：commit-scope-guard hook 實作（方向一，P2）
