### AC2：真陽 vs 誤報分類表（5+ sample cases）

**結論前提**：日誌不記錄 prompt 內容。以下分類基於日誌元資料（時間戳、路徑、session 特徵）與 hook 偵測機制的結構性分析。

**結構性風險分析**（替代無法直接分類的限制）：

S2 的關鍵字清單含「禁止」「無法」，這兩個詞在本專案的使用語境：

| 關鍵字 | 規則/SKILL 引用頻率 | 用戶 prompt 引用機率 |
|--------|-------------------|-------------------|
| 「禁止」 | 極高（rules/ 中 > 100 處） | 高——用戶引述規則、描述 ticket 約束時常用 |
| 「無法」 | 高（error-pattern、ticket body） | 中——用戶描述問題時使用 |
| 「做不到」 | 中（ai-communication-rules 等） | 低——用戶較少自陳做不到 |
| 「不支援」 | 中 | 低 |
| 「限制性解法」 | 低（僅 WRAP 相關文件） | 極低 |

基於用語頻率分析，「禁止」「無法」兩個關鍵字的引用語境觸發機率遠高於實質決策語境。但實際觸發僅 9 次 / 8 天，說明 cooldown 機制和 `min_prompt_length` 有效壓制了絕對觸發量。

### AC3：誤報率計算結果

**無法精確計算**。原因：日誌不記錄 prompt 內容，9 筆真實觸發全部歸類為「無法判定」。

**結構性估計**：基於關鍵字語境分析，「禁止」「無法」的引用語境使用頻率遠高於決策語境。保守估計誤報率 > 50%（引用語境觸發為主）。但由於絕對觸發量極低（每日約 1.1 次），即使 100% 誤報，對用戶體驗的實際干擾也極低。

### AC4：推進建議

**建議：關閉 W10-057（YAGNI）**

理由：

1. **絕對觸發量極低**：9 次 / 8 天 = 每日 1.1 次，cooldown 機制有效壓制重複觸發
2. **用戶體驗干擾極低**：S2 為 advisory 模式（exit 0 + stderr 提醒），不阻擋操作。每日約 1 次提醒對工作流影響可忽略
3. **語義區分的實作成本 vs 收益不對稱**：實作引用語境排除需要 prompt 上下文分析（如判斷關鍵字是否在引號/程式碼區塊內），複雜度高，但解決的痛點（每日約 1 次提醒）極小
4. **現有緩解機制已足夠**：`min_prompt_length: 20` + `warn_cooldown_seconds: 300` 已有效控制觸發頻率

**附加建議**（不開 IMP，作為觀察紀錄）：若未來誤報體感上升，優先考慮提高 `min_prompt_length` 至 50-100（低成本調整），而非實作語義區分。
