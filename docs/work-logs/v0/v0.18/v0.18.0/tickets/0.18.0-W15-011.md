## 實驗設計：PC-088 v2 雙軌驗證

### 實驗假設

**主假設 H1（步驟數估算偏誤）**：
在多步驟 tool 選擇情境中，未被提示計算總步驟數時，LLM 選擇的路徑平均 tool call 數 > 被提示時，且差距 > 1 步。

**Null H0**：兩組路徑步驟數分布無統計顯著差異。

**替代假設**（實驗需能反證）：
- H2：偏好中途回饋 → 預測：即使提示步驟數，仍傾向多步
- H3：訓練頻率偏誤 → 預測：提示步驟數無效，需直接禁用常見 pattern
- H4：認知負擔規避 → 預測：在任務複雜度高時 bias 更強

---

### E3：Retrospective Audit（優先執行，低成本）

**目的**：建立 base rate，量化「當前環境下 PM 多步路徑使用頻率」

**資料來源**：
- `.claude/dispatch-logs/` 最近 50 個 session 的 bash 命令記錄
- 或 grep `~/.claude/projects/-Users-tarragon-Projects-book-overview-v1/` 的 transcript

**編碼規則**（canonical task → tool path 分類）：

| Canonical Task | 短路徑 | 多步路徑 |
|---------------|--------|---------|
| 傳長文字 > 1KB 到 CLI | heredoc `cat <<'EOF'` | Write /tmp + cat / Read |
| 修改檔案局部 | Edit / MultiEdit | Read + Write 整檔 |
| 查字串 | Grep | Bash grep / find |
| 查檔案 | Glob | Bash find / ls |

**量測**：
- `long_path_rate = 多步路徑次數 / (短路徑 + 多步路徑)`
- 按 task 分組計算

**判讀**：
- `long_path_rate > 0.3` → 支持存在偏誤（不區分機制）
- `long_path_rate < 0.1` → PC-088 v2 缺實證，降級

---

### E2：A/B Prompt 因果實驗（選擇性執行）

**觸發條件**：E3 顯示 `long_path_rate > 0.3` 才投入

**設計**：
- **Task set**：20 個 canonical tasks（4 類別 × 5 實例）
- **Control 組**：標準 task prompt，無任何路徑提示
- **Treatment 組**：相同 task + 附加指令「執行前先估算從現在到完成共需幾步 tool call，若 > 2 步請搜尋是否有 1 步工具」
- **執行環境**：新 session、不載入 `.claude/` 完整規則（避免 PC-088 本身污染）
- **重複**：每組 n=20（同 task 跑一次，跨 20 task）

**量測**：
- `avg_steps_control` vs `avg_steps_treatment`
- `single_tool_usage_rate`（使用專用工具 Grep/Glob/Edit/heredoc 的比率）

**統計檢驗**：
- Paired Wilcoxon signed-rank test（同 task 配對）
- 顯著水準 α = 0.05

**判讀**：
| 結果 | 支持假設 | 動作 |
|------|---------|------|
| treatment 顯著少於 control 且差距 > 1 步 | H1 成立 | PC-088 v2 維持 |
| 無顯著差異 | H1 不成立 | 改寫 PC-088：結論改為「現象存在但機制未定」 |
| treatment 顯著多於 control（反向） | 提示反效果 | 重新設計防護（Layer 1 提示無效）|
