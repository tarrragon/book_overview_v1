#### 二、命名與可讀性

| 項目 | 評估 | 說明 |
|------|------|------|
| ErrorEnvelope 欄位（component / action / errno / hint） | **Good** | 四個欄位皆名詞自說明：component（來源模組）、action（操作動詞）、errno（錯誤分類，借 POSIX errno 概念易聯想）、hint（人讀建議）。**唯一可商榷**：`action` 在 OO 語境易誤解為「將執行的動作」，這裡實際為「失敗的動作」。可考慮 `failed_action` 或 `operation`，但會讓 callsite 更冗長。**維持現狀，文件補一行說明即可**。 |
| `errno` 借用 POSIX 概念 | **Good taste** | Python 生態 `errno` 模組為標準命名（OSError.errno 等），複用此心智模型降低學習成本。比 `error_code` / `code` / `kind` 更精準。 |
| ERROR_ENVELOPE_VERSION_MARKER 命名 | **Good** | 全大寫 + 描述用途 + 版本後綴，前後雙底線值「`__error_envelope_v1__`」做為 sentinel 也清晰。**唯一風險**：v2 升級時值要改但常數名要不要改？建議改 v2 時值改為 `__error_envelope_v2__`，常數名不變（命名描述用途，值描述版本）——目前正是此設計，無需動。 |
| ArgparseFormatErrorParser 命名 | **Acceptable，名稱冗長但精準** | 17 字符 class 名揭示「argparse + format_error + parser」三概念。可考慮 `EnvelopedArgumentParser`（13 字符）或 `BusinessAwareArgumentParser`，但前者犧牲「format_error 整合」訊號，後者意涵漂移。**維持現狀**。 |
| `_classify_argparse_error` 內部使用 `_ARGPARSE_BUSINESS_ERROR_PATTERNS[0]` / `[1]` index 取 pattern | **Bad smell（P2 修）** | 用 index 取 pattern 後再對應 errno，缺一層命名。建議改為 dict 或 namedtuple/dataclass：`PATTERNS = {"INVALID_CHOICE": re.compile("invalid choice"), "INVALID_VALUE": re.compile(r"invalid \S+ value")}`，則 classify 可改為 `for errno, pat in PATTERNS.items(): if pat.search(message): return errno`。讀者不需追蹤「[0] 是哪個 errno」，認知負擔下降。**5 行碼修，零行為差異**。 |
| INVALID_VALUE 與 INVALID_CHOICE 兩支只測一支（盲區 3 缺口） | **命名層面 OK，覆蓋層面缺** | 命名一致（都 INVALID_*），但 _ARGPARSE_BUSINESS_ERROR_PATTERNS 兩支只有一支被整合測試覆蓋。命名與測試的不對稱會誤導讀者以為「兩支都受測」，建議補測（與 linux 視角 P3 一致）。 |
| hint 內中文長句包含路徑/格式提示 | **Acceptable** | hint 多處為中文長句（如「找不到 source ticket: {id}（請確認 ID 正確且檔案存在）」），對人讀友善但對 grep / 自動化處理不友善。**設計權衡可接受**：errno 給程式判別，hint 給人讀，兩者職責分開即可。 |

#### 三、Python 慣例

| 項目 | 評估 | 說明 |
|------|------|------|
| `@dataclass(frozen=True)` 用於 ErrorEnvelope | **Good，地道** | frozen=True 讓 envelope hashable + 不可變，符合 value object 慣例。`Optional[hint]` + 預設 None 也符合 PEP 8。**可進一步**：加 `slots=True`（Python 3.10+）可省記憶體 + 防 attribute typo——若專案 Python ≥ 3.10 建議補上，P3。 |
| `Union[str, ErrorEnvelope]` + isinstance 分發 | **Acceptable，未到地道** | thyme task 提示考慮 `functools.singledispatch`。**評估後不建議改**：(1) singledispatch 對只有兩個分支的場景過度設計；(2) 兩種輸入路徑迥異（str.format 走 kwargs，envelope 走 attribute access），singledispatch 各 register 還是會寫兩個函式體，行數不減反增；(3) 現行 isinstance 寫法非常 Pythonic（Python 官方文件 typing 章節 Union 範例就是 isinstance）。**維持現狀**，linux 視角已稱「good taste」。 |

| `# pragma: no cover (super().error 會 sys.exit)` | **Good** | 明確標註且附理由，符合 coverage best practice。 |
| 全域常數 `_ARGPARSE_BUSINESS_ERROR_PATTERNS` 用 `_` 前綴標 module-private | **Good** | 符合 PEP 8。 |
| `from typing import Union` | **可現代化（P3）** | Python 3.10+ 可改 `str \| ErrorEnvelope`。但這需確認專案 Python 最低版本（uv tool 環境）。若已 ≥ 3.10 可全檔遷移（messages.py 內也有其他 Union/Optional），但屬獨立 P3 follow-up，不在本 group 範圍。 |
| `_render_envelope` module-private function | **Good** | 將渲染邏輯抽出 format_error 是良好分離（測試友善 + 邏輯內聚），命名加 `_` 也正確。 |
| `sys.stderr.write(... + "\n")` 而非 `print(..., file=sys.stderr)` | **Acceptable，風格一致即可** | 兩種寫法皆地道。本檔混用兩種（messages.py:441 用 stderr.write；create.py 多處用 print）。**建議統一**（與 emit_error helper 收斂同步處理），但屬樣式問題 P3。 |

#### 二、未覆蓋盲區

#### 盲區 1：marker 字串硬編碼於 hook，未跨檔案驗證一致性

**現況**：
- `messages.py:ERROR_ENVELOPE_VERSION_MARKER = "__error_envelope_v1__"`
- `skill-cli-error-feedback-hook.py:ENVELOPE_VERSION_MARKER = "__error_envelope_v1__"`（hardcoded literal）
- `test_marker_value_is_v1` 只 assert messages.py 端值；hook 端未被 import 比對

**風險**：升級至 v2 時，若 messages.py 改了但 hook 漏改，messages 端測試會跟著 import 自動通過（assert 成 `v2 == v2`），hook 端無人偵測。**Linus 評語**：testing the wrong invariant。

**建議**：
1. （短期）整合測試補一條 `test_hook_marker_in_sync`：直接讀 hook .py 檔內容用 regex 抽出 `ENVELOPE_VERSION_MARKER = "(.+)"` 與 messages 端比對。
2. （長期 / 已建追蹤）若有共用 constants 模組，把 marker 提到 `.claude/shared/error_envelope_constants.py` 或類似位置，hook 與 ticket skill 同 import。**但**注意 hook 是獨立 uv script，跨目錄 import 成本高，可能 P3 而非 P2。

#### 盲區 2：envelope marker 在「使用者輸出字串」中誤命中？

**分析**：marker 為 `__error_envelope_v1__`（前後雙底線 + 版本後綴），屬「unlikely-by-accident」設計。
- ticket 的 5W1H/Solution/Problem Analysis 內容由用戶填寫，理論上可包含任意字串
- 但 marker 必須完整出現於 stderr/stdout 才命中；普通文檔內容不會湊巧形成此特殊 token

**風險評估**：
- 真實風險：**極低**。連測試樣本中也是「在 envelope 渲染時主動輸出」才有此 token。
- 假想風險：用戶若在 ticket Solution 章節寫「這個 bug 可由 `__error_envelope_v1__` 觸發」並透過 `ticket track query` 顯示——此時 hook 偵測到 marker 跳過引導補充，但其實命令成功（exit_code=0）。**已有 exit_code=0 short-circuit 在 main() 第 290 行，先攔截**，所以即使 marker 在輸出中也不會觸發 hook 邏輯。

**結論**：盲區實際被 exit_code=0 short-circuit 預先擋住，**設計穩健**。但這條保護沒被測試明確覆蓋。

**建議**：補測 `test_envelope_marker_in_successful_output_does_not_trigger`：模擬 exit_code=0 且 stdout 含 marker，驗證 hook 仍 short-circuit。低優先級（P3）。

#### 盲區 3：整合測試 5 場景覆蓋度

| 已覆蓋 | 缺口 |
|--------|------|
| legacy 路徑：invalid-section / section-not-found | legacy 路徑：FILE_NOT_FOUND、ACCEPTANCE_CRITERIA_INDEX_OUT_OF_RANGE 等其他 ErrorMessages 常數 |
| envelope 路徑：CHECKLIST_VALIDATION_FAILED、INVALID_CHOICE | envelope 路徑：create.py 的 SOURCE_PARENT_MUTUALLY_EXCLUSIVE、BLOCKED_BY_NOT_FOUND、DECISION_TREE_MISSING_ALL/PARTIAL（commit message 列出但整合測試未場景化） |
| argparse 業務 vs 語法分流（場景 2 + 5） | INVALID_VALUE（`type=int` 轉型失敗）— 只有 INVALID_CHOICE 被覆蓋；regex 分類的另一支路無端到端驗證 |
| hook 偵測 envelope 跳過 | hook 與 CLI 真實串接的端到端測試（hook 是 PostToolUse 由 Claude Code runtime 觸發，整合測試僅驗證 CLI 端輸出格式，hook 端只有 unit test） |

**建議**：補一個 `test_scenario_6_invalid_value_type`（type=int 收 float / 字串）驗證 INVALID_VALUE 分類路徑，避免 ArgparseFormatErrorParser 兩支分類只測一支。建追蹤 ticket，P3。

#### 盲區 4：hook chain 順序

`skill-cli-error-feedback-hook` 的 main flow 順序：
1. 是否 Bash + ticket/skill CLI
2. exit_code == 0 跳過
3. is_excluded_error 跳過（業務邏輯詞彙模糊匹配）
4. **is_envelope_output 跳過（W17-008.5.5 新增）**
5. detect_skill_error_type → feedback

**潛在問題**：第 3 步的 `EXCLUDED_ERROR_PATTERNS` 含「ticket not found / no pending ticket」等詞彙，用 `re.search` + `IGNORECASE` 對 `stderr + stdout` 全文搜。若 envelope 的 hint 欄位剛好含「ticket not found」（例如 errno=TICKET_NOT_FOUND 的 hint 描述），第 3 步會先攔截，**第 4 步 envelope check 永遠跑不到**。

實際影響：兩條跳過路徑都 short-circuit 不補充引導，**結果一致**（hook 不輸出 feedback），所以**此盲區無功能性 bug**，僅是「邏輯歸因」不純。Linus 評語：works by accident, not by design。

**建議**：把 envelope check 提到 excluded check 之前（envelope 一定是 CLI 主動輸出的結構化錯誤，比模糊詞彙匹配優先級高）。改動只動順序不動行為，零風險。建議 P2 提案級小修。
