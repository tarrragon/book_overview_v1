### 分析報告（saffron-system-analyst return 2026-04-21）

四軸分析完成，識別 4 個規則缺口，建議 5 張衍生 Ticket。

#### 軸 A：decision-tree 第八層擴充

**現況**：Checkpoint 2（completion-checkpoint-rules.md L49-108）判定邏輯僅看 ticket 本身 status 和 Wave 內 pending 數量，**完全不查詢 spawned_tickets 欄位的子 ticket 狀態**。ANA complete 後 spawned IMP 為 pending，若 spawned 不在同 Wave 或被歸為 pending 但不被 decision-tree 主動推進，則無人接手。

**缺口風險**：
- W17-022 completed 後衍生 W17-032/033/034 全 pending，若 PM session 切換，下次 Re-center Protocol（step 2 runqueue）可能排序低於其他 P1 ticket
- W12-002 歷史重現：4 張 spawned IMP 靠人工記憶推進，延遲多輪 session
- Checkpoint 2 情境 C 可能錯判「Wave 完成」（pending spawned 未計入）

**建議動作**：Checkpoint 2 情境 B/C 判定前新增「spawned 狀態掃描」步驟——ANA 類 ticket complete 時自動列出 spawned_tickets 中 pending/in_progress 項目，觸發「推進提醒」或加入 runqueue 優先佇列。

#### 軸 B：Priority 繼承機制

**現況**：ANA W17-022 為 P1，衍生 W17-032 為 P1，但 W17-033/034 為 P2。Priority 降級判定依據不明——ticket-lifecycle.md 無「spawned ticket priority 繼承」規則；`ticket create` CLI 無預設 priority 繼承邏輯；PM 手動指定時無強制參考來源 ANA priority。

**缺口風險**：
- P1 ANA 衍生 P2 IMP 時，runqueue 排序將 P2 推遲至 P1 全完成後
- 語意上「P1 分析結論的落地執行」卻被排在其他 P1 後面
- 無規則約束時 PM 可能系統性將 spawned 降級

**建議動作**：spawned IMP 預設繼承 source ANA priority；若需降級必須在 ticket body 明示理由（新增 `priority_override_reason` 欄位）。位置：ticket-lifecycle.md「ANA 衍生 Ticket」章節新增 priority 繼承條款。

#### 軸 C：Wave 完成判定

**現況**：Checkpoint 2 情境 C 條件為「ticket completed + 同 Wave 無 pending」。若 ANA complete 後 PM 進入 Checkpoint 2 情境 B，只會做「任務切換 Handoff」——不會主動識別哪些 pending 是「剛 spawned 的高優先推進目標」vs「原本排隊的低優先 pending」。

**缺口風險**：
- Session-start hook 摘要「Wave N 有 M 個 pending」不區分「原生 pending」vs「ANA 衍生 spawned pending」
- Wave 完成判定邏輯正確但推進路徑無差異化：spawned 與一般 pending 被同等對待
- 若 spawned 跨 Wave（被手動分到 W18），當前 Wave 判定完成，spawned 被延到下版本

**建議動作**：
1. Runqueue 排序演算法新增「spawned_from_completed_ana」加權：同 priority 下 spawned IMP 排在非 spawned 前面
2. Wave 完成判定增加「spawned 清點」：情境 C 前置條件追加「本 Wave 無 pending 且本 Wave 已 completed ANA 的 spawned_tickets 皆非 pending」

#### 軸 D：Handoff 觸發條件補強

**現況**：Handoff 觸發於 Checkpoint 0 和 Checkpoint 2 情境 A/B。Handoff 內容模板（session-switching-sop.md）記錄「下次優先處理」項目，**但無強制欄位要求列出 spawned pending**。

**缺口風險**：
- 跨 session 遺忘：ANA complete 在 session A，spawned 推進需在 session B，但 handoff 未顯式記錄
- Re-center Protocol step 2（runqueue）只按 priority 排，不按「spawned 急迫性」排
- 若 handoff 發生在 ANA complete 的同一個 Checkpoint 循環，spawned 資訊最容易丟失

**建議動作**：Handoff 模板新增「Spawned 推進清單」強制欄位：ANA complete 後進入 handoff 時自動列出 spawned_tickets 中 pending 項目 + source ANA ID + priority。位置：session-switching-sop.md handoff 模板章節。

### 規則檔案設計雛形：spawned-ticket-execution-guidance.md

- **觸發時機**：ANA complete 後（Checkpoint 2 進入前）、Session-start 時、Handoff 產出時
- **推進規則**：Priority 預設繼承 + Runqueue 加權 spawned_from_completed_ana + Wave 完成判定前置條件
- **責任歸屬**：
  | 階段 | 責任人 |
  |------|--------|
  | spawned 建立 + priority 指定 | PM（ANA complete 時） |
  | spawned 推進排程 | PM（Checkpoint 2 + runqueue） |
  | spawned 執行 | 代理人（正常派發流程） |
  | spawned 遺漏偵測 | Hook（session-start + handoff） |
- **Handoff 強制欄位**：`## Spawned 推進清單`（ANA complete 後觸發）
