## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W17-008.5.2/.5.3/.5.4 完成後，`format_error` 升級為雙路徑（legacy str + ErrorEnvelope）。ErrorEnvelope 渲染時帶 `__error_envelope_v1__` 版本標記（`.claude/skills/ticket/ticket_system/lib/messages.py:18` ERROR_ENVELOPE_VERSION_MARKER）。

當前 `skill-cli-error-feedback-hook` 對所有 ticket/skill CLI 錯誤都會：

1. 命中 SKILL_ERROR_PATTERNS（如 unrecognized arguments）→ 補中文 markdown 引導
2. 命中 EXCLUDED_ERROR_PATTERNS（如 ticket not found）→ 跳過

但對「ErrorEnvelope 已輸出完整結構化錯誤」的情況沒有處理，造成業務錯誤已給 envelope 訊息又被 hook 重複補一份引導，UI 噪音。

### 影響範圍

- **改動檔案**：`.claude/hooks/skill-cli-error-feedback-hook.py`
  - 新增 envelope 偵測函式（建議 `is_envelope_output(stderr, stdout) -> bool`）
  - 在 `is_excluded_error` 之後、`detect_skill_error_type` 之前增加 envelope 偵測 → 命中跳過
- **新增測試**：`.claude/hooks/tests/test_skill_cli_error_feedback_hook.py`（若已存在則擴充）
  - envelope 命中跳過案例
  - envelope 未命中保留原行為案例（SKILL_ERROR_PATTERNS 仍補引導）
- **文件補充**：
  - `.claude/hooks/skill-cli-error-feedback-hook.py` docstring 新增「envelope 偵測模式」說明
  - `.claude/hooks/README.md` 或 `.claude/error-patterns/` 對應條目（依專案慣例）

### 相關 Error Pattern

無新增 error pattern。屬 W17-008.5 group 結構化錯誤通道整合的最後一塊。

### Hook 改動位置（精準）

`skill-cli-error-feedback-hook.py:283-294` 流程：

```python
# 排除業務邏輯錯誤（Ticket 不存在、無法認領等）
if is_excluded_error(stderr, stdout):
    ...
    return EXIT_SUCCESS

# 偵測 SKILL 引導缺陷錯誤  ← 在這之前插入 envelope check
error_type = detect_skill_error_type(stderr, stdout)
```

新增邏輯：

```python
# 跳過 ErrorEnvelope 已輸出完整結構化訊息的情況（W17-008.5.5）
if is_envelope_output(stderr, stdout):
    logger.debug("ErrorEnvelope 已輸出完整訊息，跳過補充: %s", command[:80])
    print(json.dumps(DEFAULT_OUTPUT, ensure_ascii=False))
    return EXIT_SUCCESS
```

`is_envelope_output` 實作：

```python
ENVELOPE_VERSION_MARKER = "__error_envelope_v1__"

def is_envelope_output(stderr: str, stdout: str) -> bool:
    """偵測輸出是否含 ErrorEnvelope 版本標記。

    與 messages.py:ERROR_ENVELOPE_VERSION_MARKER 同步。命中表示 CLI
    已輸出完整結構化錯誤，hook 不需重複補充引導。
    """
    return ENVELOPE_VERSION_MARKER in stderr or ENVELOPE_VERSION_MARKER in stdout
```

### 為何不直接加入 EXCLUDED_ERROR_PATTERNS

EXCLUDED_ERROR_PATTERNS 是 regex 清單，依設計做訊息文字模式比對。ErrorEnvelope 標記是固定字串、版本化欄位（將來可能升級為 v2），語意層次不同——envelope 偵測屬「輸出格式辨識」非「業務錯誤辨識」。獨立函式更清晰，將來 v2 升級時也只需改 marker 常數一處。

### 改動摘要

1. `.claude/hooks/skill-cli-error-feedback-hook.py`
   - 模組 docstring 新增「Envelope 偵測模式 (W17-008.5.5)」段落，說明 marker 與 messages.py 同步來源
   - 新增 module-level 常數 `ENVELOPE_VERSION_MARKER = "__error_envelope_v1__"`，註明與 `ticket_system/lib/messages.py:ERROR_ENVELOPE_VERSION_MARKER` 同步、v2 升級兩處同改
   - 新增純函式 `is_envelope_output(stderr, stdout) -> bool`：偵測 stderr/stdout 任一含 marker
   - 主流程在 `is_excluded_error` 之後、`detect_skill_error_type` 之前插入 envelope check：命中時 `logger.debug` + 輸出 DEFAULT_OUTPUT 並 return（既有原行為保留於未命中分支）

2. `.claude/hooks/tests/test_skill_cli_error_feedback_hook.py`（新建）
   - Marker 同步驗證 1 案例
   - is_envelope_output 純函式 4 案例（stderr 命中 / stdout 命中 / 雙空 / 無 marker）
   - 主流程整合 5 案例對齊 ticket Context Bundle：envelope-stderr / envelope-stdout / SKILL_ERROR_PATTERN 命中 / EXCLUDED_ERROR_PATTERN 命中 / 無 pattern 命中

### 設計決策

- 為何不直接加入 EXCLUDED_ERROR_PATTERNS：envelope 標記是「輸出格式辨識」非「業務錯誤辨識」，獨立函式 + 常數比 regex 清單更清晰，v2 升級時只需改一處常數
- envelope check 順序晚於 is_excluded_error：保留既有業務錯誤短路徑優先，envelope 為輔助過濾層
