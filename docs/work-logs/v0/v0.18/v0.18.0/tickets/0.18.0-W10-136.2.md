## Linus 視角：Good Taste 評估報告

multi_view_status: linux-only（本 ticket 為 4 視角之一，整合在 W10-136 父 ticket）

### Core Judgment

**Worth doing**：L1 + L3（hook wrapper 強制 schema）+ L2 輕量 scenario library。
**Not worth doing**：L2.5 few-shot 自動學習、L3.5 LLM judge、L5 Agent SDK 重構。

### 1. 5 層次架構複雜度評估

| 層次 | Taste Score | 理由 |
|------|------------|------|
| L1（hook 事後過濾） | Acceptable | 現狀基線，但效果弱 |
| L2（scenario library） | **Good taste** | 把「PM 寫什麼選項」從每次決策變成查表，特例 → 常規 |
| L2.5（few-shot 自學） | Garbage | 引入「違規→label→注入」三段管線，把資料結構問題（沒有規範選項池）用機器學習補；典型的「在錯地方加聰明」 |
| L3（hook wrapper 強制 schema） | **Good taste** | 既有 hook 模式延伸 1 個檔案，schema 是純資料結構，沒有條件分支 |
| L3.5（LLM judge） | Garbage | 用一個隨機系統（LLM judge）判定另一個隨機系統（AUQ payload）的合規性。Linus：「You're solving randomness with more randomness.」 |
| L4 | N/A | 官方確認不可行 |
| L5（Agent SDK 重構） | Garbage（在當前情境）| 為了 < 10% 的 AUQ 違規率重寫整個 dispatch loop。10 行 if 改寫成 4 行的 good taste 反例——這是 4 行 if 改寫成 100 行框架。 |

### 2. L5 是否屬「Sledgehammer 砸蒼蠅」

**Yes，明確屬於。**

論證三點：
1. **問題規模 vs 解法規模**：AUQ 違規率 < 10%（W10-135 數據），L5 是框架級重寫（影響 PM dispatch、所有 AUQ 觸發點、放棄原生 UI）。比例失衡 > 10:1。
2. **「Never break userspace」**：L5 放棄 AUQ tool 等於放棄 Claude Code 原生 UI。用戶看到的對話框會從「Claude Code 內建選單」變成「Claude 文字輸出選項」。這是破壞用戶體驗的「正確但不可接受」變更。
3. **不可逆性**：L1+L2+L3 任何時候可加可減，L5 一旦切換等於離開 Claude Code 框架的 happy path，未來 SDK 變更將承擔遷移成本。

**反問自己**（強制自問段，見下方）：L5 是否是「真正解決問題」？答：L5 解決的是「100% 確定性」需求，但專案實際需求是「< 10% 違規率」，這兩個目標差一個數量級。

### 3. 最簡可行組合 vs 最強組合對比

| 組合 | 新增複雜度 | 預期違規率 | 不可逆性 | ROI |
|------|----------|----------|---------|-----|
| **最簡：L1+L2+L3** | 認知負擔 +5 | 推估 < 5% | 低（可加可減） | **最高** |
| L1+L2.5+L3.5（用戶建議的中度） | 認知負擔 +13 | 推估 < 3% | 中（pipeline 一旦建立難移除） | 中 |
| 最強：L5 | 認知負擔 +12 + 框架遷移 | < 1% | 高（離開 Claude Code happy path） | **低** |

**Linus 建議**：先做 L1+L2+L3。觀察 6-8 週違規率。若仍 > 5%，再評估加 L3.5；若 < 5%，停手。

「Premature optimization is the root of all evil」適用：在沒有 L1+L2+L3 的 baseline 數據前，討論 L3.5/L5 是空談。

### 4. 既有 PM Hook 機制延伸覆蓋 L3.5 的可行性

**部分可行，但不建議**。

可行面：
- `.claude/hooks/askuserquestion-charset-guard-hook.py` 等已實作 PreToolUse 攔截，加 LLM judge call 技術上可行（subprocess claude -p ...）。
- Hook 已有 stderr/log 雙通道（規則 4），judge 結果可整合既有 observability。

不建議面：
- LLM judge 在 hook 內 call 等於在 PreToolUse 同步等待 LLM 回應，PM 派發延遲 +2-5 秒/次。
- judge 本身會錯（LLM 隨機性），需 retry/fallback policy → 又一層條件分支地獄。
- 既有 hook 是純規則比對（regex / charset），加 LLM 是「在快機制裡塞慢機制」典型反模式。

**結論**：L3.5 即使技術可行，也屬於 PC-066「在錯地方加聰明」家族。

### 5. 強制自問：「我可能錯在哪裡」

prompt 警告我前兩輪都選「不重構，hook 強化」，本輪可能繼續同方向被框定。誠實檢視：

**自問 1**：我是否因為前兩輪選 hook 強化，本輪就用同框架評估 L5？
答：部分是。但 L5 的「砸蒼蠅」判定不是來自 hook 偏好，是來自規模比例（10:1）+ 不可逆性 + 破壞 UI 三項硬指標。即使我從零開始，這三項仍會把 L5 打回票。

**自問 2**：L5 是否有「我看不見的長期價值」？
可能。例如未來 AUQ 場景擴展到 100+ scenario，L2 scenario library 會撐不住，L5 的 schema 化反而清晰。但**這不是當前問題**——當前是 < 20 個 scenario，L2 完全夠用。「為未來想像的問題重寫」正是 YAGNI 反模式（PC-093 家族）。

**自問 3**：是否我在保守維護現狀，怕新東西？
檢驗點：我同意 L3 + L2（兩個新增），不同意 L2.5/L3.5/L5（三個新增）。如果是純保守，我會反對所有新增。所以判別標準是「資料結構乾淨度」而非「新舊」。

**自問 4**：bay devil's advocate 視角（W10-136.3）若反論 L2 scenario library 也是「集中化反模式」，我能否接受？
能。若 bay 拿出 scenario library 在實際運作中變成「另一個被遺忘的目錄」的證據，我會降級至「只做 L3」。L3 是最低限度必須做的（hook 已是專案內建模式延伸）。

### Linus-style Solution

**第一步永遠是先簡化資料結構**：
- L3 hook wrapper 定義 schema（option_id、label、value）→ 把「PM 自由發揮文字」變成「填空」。
- L2 scenario library 是 schema 的實例化儲存（YAML/JSON），不是新機制。

**消除特例**：
- 現狀的特例：每次 AUQ 都是「PM 重新發明選項格式」。
- L3+L2 後：所有 AUQ 走同一條 schema 校驗路徑。10 行 if 變 4 行表查。

**最笨但最清晰的實作**：
- Hook 校驗：option 必填欄位 → 缺則拒。
- Scenario library：1 個目錄 + N 個 YAML，PM 派發前可選引用。
- **不需要** LLM judge、不需要 few-shot pool、不需要 SDK 重構。

**零破壞性**：
- 對既有 AUQ 呼叫向下相容（schema 寬鬆模式 + 嚴格模式 flag）。
- 不改 Claude Code runtime UI。

### 整合決策準則對父 ticket 的輸入

依 W10-136 整合決策準則：
- linux 視角投票：**支持「L1+L2+L3 短期實施」**。
- 反對 L5 重構（明確 sledgehammer）。
- 反對 L3.5（隨機系統解隨機系統）。
- 對 L2.5 持保留（待 bay 視角看是否有自學需求的硬證據）。

dissenting view（PC-101）：若其他三視角全部支持 L5，我的反對保留意見聚焦在「框架不可逆性」與「< 10% 違規率不應觸發框架級重構」，請父 ticket 仲裁時保留我的論證進入 ANA 結論。

### Tripwire 設計（觀測退出條件）

實施 L1+L2+L3 後，6-8 週內若以下任一觸發 → 重啟 L3.5/L5 評估：
- AUQ 違規率仍 > 5%（量化失敗訊號）
- scenario 數量 > 50（L2 集中化反模式風險）
- 用戶反映 AUQ UI 體驗下降（L3 schema 太嚴）
- 出現 hook 校驗無法覆蓋的新型違規（如語意層而非格式層）
