## Solution

新增 `.claude/skills/ticket/tests/test_ac_drift_regression.py`，透過 `execute_claim`
entry point（argparse namespace）模擬 W3-001 原始觸發案例：

- Fixture：`_w3_001_ticket_fixture()` 建立 `created` 為 9 天前的 pending Ticket
  （含三條 AC：整合測試通過 / flaky 修復 / 無不穩定測試）
- Mock：`collect_ac_verifications` + `run_all_verifications` 回傳三項 passed
  VerificationResult（模擬 AC 已由上游外溢達成），並 mock `load_ticket` 讓
  `execute_claim` 前段 stale 檢查拿到 fixture

四個測試驗證 W11-001.1 + W11-001.2 協力防護：

1. `test_regression_stale_info_warning_emitted` — 驗證 stale INFO 警告輸出
   （`[INFO]` 標記 + Ticket ID + `>= 7 天` 閾值）
2. `test_regression_ac_achieved_message_emitted` — 驗證 stderr 輸出「已達成」
   + 建議 `ticket track complete`
3. `test_regression_claim_rejected_exit_code_one` — 驗證 claim 被拒絕（rc == 1）
4. `test_regression_all_three_signals_co_occur` — 綜合驗證三訊號同時出現

關於 AC「y/n/c 三選一」的實作偏離：
目前 `claim_with_verification` S4（all_passed）分支直接 return 1，不進入 prompt。
本測試依實作現況驗證「拒絕 claim」，比原 AC 更嚴格（完全不給 claim 機會）；
若未來擴充三選一互動，測試需同步補齊。此差異已在測試 module docstring 明示。
