## Problem Analysis

## Context Bundle

### 觀察事實
1. Remote `tarrragon/claude.git` 的 git commit log：v1.17.0 (85f0da8, Mac) → **v1.36.2 (84adf5a, Windows, amanda@pintech.com.tw)**，中間無任何 commit
2. 但 v1.36.2 的 CHANGELOG 有完整線性歷史：1.17.0 → 1.18.0 → 1.19.0 → ... → 1.36.0 → 1.36.1 → 1.36.2（19 個新 entry）
3. 單次 commit 把 VERSION 從 1.17.0 直接改為 1.36.2
4. 822 個檔案變更，+81641 行 / -6122 行

### 邏輯矛盾
- `sync-claude-push.py::bump_version()` 最多 +1 minor 或 +1 major，絕對產生不出 1.17.0 → 1.36.2
- CHANGELOG 記錄的 19 個中間版本（1.18..1.36）在 git log 完全不存在
- 若 push 失敗 `sys.exit(1)`（line 608），腳本不會累積狀態
- 每次 push.py 都 `rmtree(temp_dir)`（finally block），`.claude/` 本地 CHANGELOG 不被修改

### 最可能的形成路徑（推論）
amanda 的 Windows 機器上的 `.claude/` 目錄是**從其他來源複製來的**（例如她個人的另一個已累積到 1.36.x 的專案 clone），非從 book_overview_v1 或 remote 直接來。執行 sync-push 時：
- `copy_filtered(claude_dir, temp_dir)` 把 local `.claude/CHANGELOG.md`（已累積到 1.36.2）覆蓋到 temp_dir
- `saved_changelog` 已先讀入 remote 版（僅到 1.17.0）
- `update_changelog(temp_dir, new_version, commit_message, saved_changelog)` line 553 依 saved_changelog prepend new_entry
- **但最終 CHANGELOG 結果顯示完整 1.17 → 1.36.2 歷史**，意味著 update_changelog 實際沒生效或被 copy_filtered 覆蓋的 local CHANGELOG 留下了
- VERSION 的 1.36.2 寫入路徑不明——需要驗證 push.py 在 Windows 是否存在路徑順序或 write 時機差異

### 核心脆弱點（不需重建完整機制也可識別）
1. **缺 sanity check**：new_version vs remote_version 差距無上限驗證（正常應 <= 1 minor），jump 過大時未 fail-fast
2. **copy_filtered / write_text 順序依賴**：line 533 copy 後 line 550 write，任何亂序都會把 local CHANGELOG/VERSION 汙染洩漏到 push
3. **bump_suggestion 固定 patch 當 user_message 存在**：line 501-504 不分析 local commit 類型，導致 user message 時版號變化失準
4. **無 encoding 防護**：read_text 未 strip BOM；Windows 檔案若含 BOM，extract_version_string 會回傳 `\ufeff1.17.0`，bump_version fallback 到 `1.0.1`
5. **push 失敗無診斷持久化**：失敗僅 print_color + sys.exit，下次執行無從比對前次狀態

### 資料充足度評估
- 實證資料充足指向「local `.claude/` 已污染」為必要條件（W = widen 有多個替代假設但此為最合理）
- 不充足：無法取得 amanda 的 Windows 機器 sync-push stdout 日誌或本地 `.claude/` 快照，無法 100% 確認 1.36.2 的 write 路徑
- 行動決策：治本應聚焦「脆弱點」而非「特定機制」——即使根因細節未全解，防護 sanity check 可阻止類似事件再發

### 修正策略
1. 加入 new_version sanity check：若 `new_major > remote_major + 1` 或 `new_minor > remote_minor + 1` 或 `diff > 1 minor/patch` 即 fail-fast
2. 加入 encoding BOM strip：讀 VERSION 時 `.lstrip('\ufeff')`
3. copy_filtered 前明確移除 temp/VERSION、temp/CHANGELOG.md，強制重建
4. 驗證 local `.claude/VERSION` vs remote VERSION：若 local > remote 且非 patch 級差異，require user 確認
5. push 失敗時 dump 完整 diagnostics（remote_version、new_version、bump_suggestion、commit_message、commit_count）到 ~/.claude-push-failure.log

### 版號決策
- v1.36.2 無法撤回（已 push 到 remote，force push 會影響所有 downstream sync-pull）
- 接受現狀：下次 push 從 1.36.2 bump → 1.36.3 或 1.37.0（視本次變更類型）
- CHANGELOG 中間的虛假版本保留（清理需複雜 rewrite，投入成本 > 收益）

## 實作摘要

### 修改項目

#### 1. `extract_version_string()` 加入 BOM strip
line 352 新增 `.lstrip("\ufeff")`。防止 Windows VERSION 檔案含 UTF-8 BOM 時 bump_version 正規式 parse 失敗（舊版會 fallback 到 1.0.1）。

#### 2. 新增 `validate_version_bump(remote_version, new_version)` 函式
Sanity check：新版號相對 remote 必須恰好滿足三種合法 bump 型態之一：
- major bump: X+1.0.0
- minor bump: X.Y+1.0
- patch bump: X.Y.Z+1

任何其他跳躍（含反向、非連續、跨 minor 多版）立即 fail-fast，輸出紅色錯誤並 sys.exit(1)。

#### 3. 在 main 的 line 552 呼叫 validate
bump_version 算出 new_version 後立即驗證，確保寫入 VERSION 檔前已通過 sanity check。
