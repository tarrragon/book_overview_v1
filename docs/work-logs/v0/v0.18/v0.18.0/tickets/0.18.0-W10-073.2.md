## WRAP 分析：pm-role.md（346 行）拆分可行性

### 核心問題

`pm-role.md` 位於 `.claude/rules/core/`，每個 session 自動載入。但檔案第 10 行明寫：
> 如果你正在執行 Ticket 開發任務（已認領的 IMP/ANA/DOC 等），**忽略本規則**，繼續你的工作。

這代表：**代理人 session 明確不需要這份規則**，但自動載入機制不分角色，代理人 session 仍照載 346 行。這是 WRAP 分析的關鍵切入點。

### R — Reality Test（現實測試）

**關鍵問題 1：代理人 session 是否需要 pm-role.md 的哪些章節？**

逐章節檢視：

| 章節 | 代理人需要？ | 理由 |
|------|-----------|------|
| 角色辨識（1-14） | **需要（辨識用）** | 代理人需讀此才知道「我是代理人，忽略本規則」，但**讀一次就夠，不需每次 load** |
| 核心原則（16-39） | 不需要 | 「主線程禁止寫程式碼」對代理人無意義 |
| 行為循環（41-107） | 不需要 | 「派發後行為」、「AUQ 觸發」都是 PM 專屬 |
| 工作階段切換 SOP（109-160） | 不需要 | /clear、worklog 記錄都是 PM 行為 |
| 代理人失敗 SOP（162-292） | 不需要 | PM 如何判斷代理人失敗，對代理人自己無意義 |
| PM 流程規則（294-319） | 不需要 | 場景路由指向 pm-rules/，代理人不走此路由 |
| Re-center Protocol（321-330） | 不需要 | PM 迷失時的定位機制 |

**結論**：代理人 session 只需要「角色辨識」一段（14 行）就能識別身份。其餘 332 行對代理人完全無用。自動載入對代理人是**96% 浪費**。

**關鍵問題 2：PM 是否每個 session 一開始都需要讀全部 346 行？**

| 章節 | PM 每 session 啟動需立即載入？ | 理由 |
|------|-------------------------|------|
| 角色辨識 + 核心原則（1-39） | **是** | 核心禁令（不寫程式碼、不自己解決）是行為底線，必須常駐 |
| 行為循環（41-107） | **是** | 派發位置判斷、AUQ 觸發是每個對話都可能觸發的即時決策 |
| 工作階段切換 SOP（109-160） | **否** | 只在「切換焦點 / /clear」前才需要，PM 可在觸發情境下 Read |
| 代理人失敗 SOP（162-292） | **否** | 只在「懷疑代理人失敗」時才需要，情境明確 |
| PM 流程規則路由表（294-319） | **是** | 每次決策時都可能需要查路由，但短（26 行） |
| Re-center Protocol（321-330） | **是** | 迷失時的救生索，但短（10 行） |

**結論**：PM 立即需要的是 1-107 + 294-330 = 約 145 行（行為循環 + 路由 + Re-center）。代理人失敗 SOP（131 行）和工作階段切換 SOP（52 行）都是「情境觸發」的，可以移出 auto-load。

**原子性檢驗（Zettelkasten）**：
- 代理人失敗 SOP 是**自成一體的程序知識**（Step -1~4、失敗類型表、滑坡場景），符合原子化，能獨立理解，適合抽出
- 工作階段切換 SOP 同樣自成一體（/clear 前檢查、快照命令），適合抽出
- 行為循環 + AUQ 強制是「每個對話都可能觸發」的即時決策規則，不適合抽出（除非路由指引夠強）
- 角色辨識 + 核心禁令是「最高頻參照」的底線，必須 auto-load

### P — Prepare to be Wrong（準備出錯）

**失敗場景 1：拆分後 PM 在關鍵情境忘記 Read 子檔**
- 情境：代理人派發後，PM 沒感覺到「失敗懷疑」訊號，直接跳過 Step 0.5 推論失敗（PC-070 的再發）
- 緩解：在 auto-load 保留的部分中，寫一張**觸發情境 → 必讀子檔**路由表（類似選項 D 的「核心禁令 + 場景路由表」設計）
- 緩解：pm-rules/ 子檔的文件名必須「自說明」（如 `agent-failure-sop.md`、`session-switching-sop.md`），讓 PM 看路由表就知道該 Read 哪個
- 回退路徑：若 1 個月內觀察到 PM 忘記 Read 子檔的案例 ≥ 2 次，將該子檔恢復 auto-load

**失敗場景 2：引用更新不完整，產生 broken link**
- 情境：45+ 處引用 pm-role.md 的檔案中，有指向「#代理人失敗 SOP」「#失敗判斷前置步驟」等子章節的錨點引用。拆分後錨點失效
- 緩解：使用 `/broken-link-check` skill 在拆分後全掃；必要時保留 pm-role.md 作為「重導向索引檔」（內容只剩章節清單 + 指向子檔連結）
- 回退路徑：broken-link 掃描結果作為拆分是否完成的驗收指標

**失敗場景 3：拆分碎片化導致「查規則比執行還累」**
- 情境：PM 遇「代理人失敗」時要 Read 2-3 個子檔才能拼湊完整 SOP（例如失敗判斷前置步驟 + 失敗類型表 + 重試守則 各自獨立檔）
- 緩解：**不要過度切**。選項 C 把 131 行的代理人失敗 SOP 整段搬為一檔，保持原子完整性，不進一步切割
- 回退路徑：若 PM 反映「找規則累」，合併相關子檔

**失敗場景 4：選項 B（整檔遷移）導致新手 PM 不知「不寫程式碼」禁令**
- 情境：新 session 啟動，PM 沒 Read pm-role-core.md，直接遇到「代理人失敗」情境就自己動手（PC-045 再發）
- 緩解：**選項 B 有這個風險 → 不選 B**。核心禁令必須 auto-load
- 回退路徑：不適用，直接排除選項 B
