## Task Summary

分析 ticket CLI 父子任務的 complete 順序邏輯缺陷：子任務被父阻塞無法 complete，父可越過子直接 complete，違反「父任務是子任務彙總」的設計意圖

---

## Problem Analysis

### 問題根因

**核心發現：CLI 目前完全沒有父子互鎖邏輯**。add-child、complete、release 三個命令互相獨立，不讀寫對方的狀態。

1. **add-child 只寫資訊性欄位**（track_relations.py:242-312）
   - 僅更新 `parent.children`、`child.parent_id`、`child.chain`
   - 不觸碰 `status`，不寫 `blockedBy`
   - 不會因建立父子關係而自動 block 子或 pending 父

2. **complete 不檢查 children**（lifecycle.py:374-489）
   - 僅檢查 status、acceptance、execution_log、blockedBy(透過 claim checklist)
   - 完全不讀取 `children` 欄位
   - 父可在子未完成時獨立 complete，違反「父是子彙總」的常規設計

3. **status=blocked 的唯一來源是 release**（lifecycle.py:491-519）
   - 全 codebase 只有一處 `ticket['status'] = STATUS_BLOCKED`
   - 無任何自動 block 邏輯（不因 add-child 也不因父 pending）

4. **_check_unblocked_tickets 只處理 blockedBy，不處理 parent_id**（lifecycle.py:618-640）
   - 父 complete 時掃描 `blockedBy` 含該 id 的 ticket 給出解鎖建議
   - 但 add-child 不寫 blockedBy，所以此機制對父子關係無效

### 本 session 實際情境還原

- W5-013 add-child W5-014（但 W5-013.children 最終為 []，疑似另一 bug 或未保存）
- W5-014 被手動 release 變 blocked（blockedBy=[]，無自動解鎖依據）
- W5-013 complete 成功（CLI 不檢查 children）
- W5-014 仍 blocked 且無人能自動喚醒

### 影響範圍

- .claude/skills/ticket/ticket_system/commands/lifecycle.py（complete/release/claim）

- .claude/skills/ticket/ticket_system/lib/chain_analyzer.py（分析層，未影響生命週期）

## WRAP 分析

### W (Widen Options) — 擴增選項（5 個方案）

**方案 A：父完成前強制檢查子全部完成（硬約束，父被子阻塞）**
- complete() 新增：若 children 非空且任一子 status != completed → 拒絕 complete
- 語義：父是子的彙總，子未完成父不能宣告完成
- 類似 Jira Epic/Story 模型

**方案 B：父完成自動 cascade 解除子阻塞（軟約束，父 complete 解救子）**
- complete() 新增：對所有 children（且 status=blocked）自動轉為 pending
- 語義：父任務結束，子任務自動獲得可執行狀態
- 類似 GitHub Issues close 觸發子 issue 狀態更新

**方案 C：雙向約束（父子互鎖）**
- A + B 組合：父不能提前完成 + 子隨父 complete 解鎖
- 最嚴謹但實作最複雜

**方案 D：add-child 自動建立 blockedBy 關聯（復用現有機制）**
- add-child 時自動把 parent_id 加到 child.blockedBy（若 child.status != completed）
- 復用 _check_unblocked_tickets 的既有自動解鎖邏輯
- 父子關係語義上翻譯為依賴關係，不新增生命週期路徑

**方案 E：放棄父子生命週期耦合，僅做提示（最小變更）**
- complete 時若 children 未全完成 → 警告但不阻止
- release/claim 時顯示父子關係提示
- 父子關係維持純資訊性，依賴 blockedBy 處理真實阻塞

### R (Reality Test) — 檢驗機會

每個方案的可驗證假設：
- 方案 A/C：常規設計習慣是否適用本專案？（子可能是 spawn-out 獨立工作，非強彙總）
- 方案 B：父 complete 時子還在 blocked 是否代表子有獨立阻塞原因？（可能誤解凍結）
- 方案 D：blockedBy 語義能否承載父子關係？（是，都表示「依賴另一 Ticket」）
- 方案 E：用戶是否願意承擔手動同步成本？（本 session 的情境就是不願意）

**實證資料**：本 session 的 W5-013/014 是 **ANA 分析 + 延伸 ANA**，不是強彙總關係。W5-014 是 W5-013 衍生出的獨立分析主題，父先完成、子後獨立完成是合理的。這削弱方案 A/C，支持方案 B/D/E。

### A (Attain Distance) — 機會成本

- 方案 A/C 會阻斷「先完成綜合分析、再各自深入」的工作模式（本 session 典型情境）→ 機會成本高
- 方案 B 單獨使用：無法防止父獨立 complete 的問題（但這未必是問題，見上）
- 方案 D：增加 add-child 副作用，可能與使用者預期衝突（add-child 原本不改狀態）
- 方案 E：最低成本，但需靠使用者紀律

### P (Prepare to be Wrong) — 準備出錯

引入失敗信號：
- 方案 B/D：觀察 6 個月內是否有「父 complete 但子應保持 blocked」的案例
- 方案 A/C：若出現「使用者繞過父子而改用 relatedTo」就是失敗信號

## 建議方案：B + E 組合（漸進式）

**立即（B）**：complete 時自動解鎖 blocked children（僅針對 blockedBy 含 parent_id 或空 blockedBy 的 blocked children，轉為 pending）+ 顯示提示。

**同步（E）**：complete 時若有未完成 children，**警告但不阻止**，列出 children 狀態讓使用者決策。

**理由**：
1. 保留本 session 揭示的工作模式（父先完成綜合、子獨立延伸）— 不採用方案 A/C 的硬約束
2. 修復核心痛點（子孤兒 blocked）— 方案 B 的自動解鎖
3. 最小侵入性（不改變 add-child 語義，不新增 blockedBy 副作用）— 排除方案 D
4. 保留升級路徑（未來若真需硬約束可疊加方案 A）

**不採用方案 D 的關鍵理由**：add-child 應是純結構操作，不應隱式寫 blockedBy（違反單一職責）。

## 現存不一致 Ticket 遷移指引（W5-014）

**情境**：W5-014 status=blocked, blockedBy=[], parent_id=0.18.0-W5-013（已 completed）

**遷移步驟**：
1. 手動執行 `ticket track claim 0.18.0-W5-014` 直接認領（claim 應接受從 blocked → in_progress，需驗證）
   - 若 claim 拒絕 blocked → 需先手動編輯 frontmatter 把 status 改為 pending
2. 正常執行 W5-014 分析工作
3. 完成後 `ticket track complete 0.18.0-W5-014`

**預防**：IMP Ticket 實作後，類似情境會由父 complete 自動觸發子解鎖，不需手動干預。
