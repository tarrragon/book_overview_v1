### 變更摘要

在 `.claude/settings.local.json` 新增 `worktree` 區塊：

```json
"worktree": {
  "bgIsolation": "none"
}
```

此設定來自 Claude Code v2.1.143 release notes：
> Added `worktree.bgIsolation: "none"` setting to let background sessions edit the working copy directly without `EnterWorktree`, for repos where worktrees are impractical

### 決策依據

#### 為何啟用（適用情境匹配）

本專案為 Chrome Extension（Manifest V3），符合 release notes 描述「worktrees are impractical」全部三項特徵：

| 特徵 | 本專案狀況 |
|------|-----------|
| build artifact 不在 git 索引內但體積大 | `build/development/`（gitignored）+ `node_modules/`，每個 worktree 需重跑 `npm install --legacy-peer-deps` + `npm run build:dev`（esbuild bundle 三入口點） |
| 重型編譯產物 | esbuild IIFE bundle 對 background/content/popup 三入口點完整 bundle，worktree 切換後需完整重 build |
| 工具鏈索引敏感 | jest jsdom 環境、Chrome Extension manifest 路徑、Babel transform cache 切換 worktree 後皆需重建 |

#### 與既有 memory 條目關聯

`feedback_worktree_agent_issues.md` 子項 3「worktree 不自動合併依賴分支」於 2026-04-13 標記為「保留於 memory 作為專案 context 索引」。`bgIsolation:none` 啟用後背景 session 不建 worktree，依賴合併需求自然消解。此議題正式評估由 ANA W17-216 追蹤（已雙向 related_to）。

### WRAP 三問

#### W（Widen）— 候選方案

| 方案 | 評估 | 選擇 |
|------|------|------|
| A. 啟用 `bgIsolation:"none"`（本方案） | 零成本迴避 worktree 重 build；風險為背景 agent 與前景共用工作副本 | 採用 |
| B. 保留預設 worktree 隔離 | 完整隔離但每個 background session 需重跑 `npm install` + `npm run build:dev`（約數十秒至分鐘級） | 拒絕（成本與本專案 build 重型不匹配） |
| C. framework 層改善 worktree create 自動合併 blockedBy | 屬子項 3 完整解；但需 Claude Code 上游或專案層 worktree skill 改造 | 轉 ANA W17-216 評估是否仍需追蹤 |

選擇非默認值（預設為自動建 worktree），符合 Consider the Opposite。

#### A（Attain distance）— 機會成本

啟用後背景 agent 直接編輯工作副本，可能影響前景未提交變更。目前前景工作流尚未頻繁使用 `claude --bg` / `claude agents`，擠壓成本接近 0；專案實際進度（v0.17→v0.18 過渡期）以前景為主，無平行多 agent 需求。

#### P（Prepare to be wrong）— 失敗預想

最可能失敗情境：未來引入頻繁前景 + 多 `claude agents` 背景並行修改同一檔案，引發未提交變更衝突（對應 memory `feedback_parallel_session_ticket_misjudge.md` PC-078 並行 session 識別模式）。

防護措施：啟用後若實際遇衝突，立即建 IMP ticket 評估改回 worktree 隔離或設計 sub-worktree 排程策略；ANA W17-216 評估時納入此監測項。
