### Phase 1 設計範圍指引（PM 派發 context）

**設計目標**：對 `parser.save_ticket()` 加入 fcntl.flock per-ticket-file advisory lock，三模式 race 全消除（lost update / empty read / file corruption）。實作骨架已在 W14-005 Solution §3.4。

### 1. API 設計

| 元素 | 規範 |
|------|------|
| `_file_lock(path: Path)` | 內部 helper context manager，私有不對外 |
| `save_ticket(ticket, ticket_path)` | signature 不變，caller 透明 |
| Lock file 命名 | `{ticket_path}.lock` 後綴 |

### 2. 錯誤處理

- `fcntl.flock` 失敗（資源不足 / 被另一 process 長期持有）：blocking 等待還是 timeout？建議 blocking（advisory lock 一般情境短暫，timeout 增加複雜度）
- `permission denied` / `disk full`：rethrow，由 caller 處理
- Crash safety：OS fd 回收自動釋放，無需主動清理

### 3. 邊界條件

- Lock file cleanup：保留不刪（advisory lock，crash 後自動釋放，下次 reuse 不衝突），加入 .gitignore
- 與 `_ticket_cache` 互動順序：取 lock → 修改 → cache invalidate → 釋 lock（lock 內完成 cache invalidate 確保下次 load 取最新）
- **不在範圍**：Windows 支援（fcntl POSIX-only，後續 ticket 用 filelock library fallback）

### 4. Acceptance 細化方向

將 IMP 4 項 acceptance 拆為可驗證的 Phase 2 RED 測試項目：
- per-ticket-file lock 隔離（不同 ticket 並發無等待）
- 同一 ticket 序列化（同 ticket 並發 → 0 lost update）
- Crash recovery（process kill 後 lock 自動釋放）
- Cache invalidation timing 正確

### 5. 參考實作骨架（W14-005 Solution §3.4）

```python
import fcntl
from contextlib import contextmanager

@contextmanager
def _file_lock(ticket_path: Path):
    lock_path = ticket_path.with_suffix(ticket_path.suffix + ".lock")
    lock_path.parent.mkdir(parents=True, exist_ok=True)
    fd = open(lock_path, "w")
    try:
        fcntl.flock(fd, fcntl.LOCK_EX)
        yield
    finally:
        fcntl.flock(fd, fcntl.LOCK_UN)
        fd.close()

def save_ticket(ticket, ticket_path):
    with _file_lock(ticket_path):
        # ... 既有邏輯不變 ...
```

### Phase 1 Feature Spec（Full 模式，6 區段）

#### 1. Purpose

為 `parser.save_ticket()` 提供 per-ticket-file 並發寫入安全保護，消除 W14-005 重現實驗確認的 race condition（lost update / empty read / file corruption 三模式，66.5% lost rate）。對 caller 完全透明（signature 不變），影響範圍為 ticket CLI 所有寫入路徑（update_source_spawned_tickets / update_parent_children / claim / complete / append-log）。

#### 2. API Signatures

| 元素 | Signature | 可見性 |
|------|-----------|--------|
| `_file_lock` | `@contextmanager def _file_lock(ticket_path: Path) -> Iterator[None]` | 私有（模組內） |
| `save_ticket` | `def save_ticket(ticket, ticket_path: Path) -> None` | 公開，signature 不變 |

**Lock file 命名**：`{ticket_path}{suffix}.lock`（例：`0.18.0-W14-042.md.lock`）

**鎖型別**：`fcntl.LOCK_EX` blocking exclusive advisory lock，per-ticket-file。

#### 3. GWT Scenarios

**場景 1：同一 ticket 並發寫入序列化**
- Given: 兩個 process A、B 同時呼叫 `save_ticket(t, P)`，P 為同一 path
- When: A 先取得 lock，B 等待
- Then: A 完成寫入並釋放後 B 取得 lock；最終檔案內容為 B 寫入版本（last-writer-wins），無 lost update / empty read / corruption

**場景 2：不同 ticket 並發無阻塞**
- Given: process A 寫 ticket P1，process B 寫 ticket P2（P1 ≠ P2）
- When: 兩者同時呼叫 save_ticket
- Then: 兩 lock 互不干擾，並發完成（per-file lock 隔離）

**場景 3：Crash recovery**
- Given: process A 持有 P.lock，未釋放即被 kill -9
- When: process B 後續呼叫 save_ticket(t, P)
- Then: OS 自動回收 fd 釋放 flock，B 立即取得 lock 並完成寫入；lock file 殘留但不影響後續使用

**場景 4：Cache invalidation 時序**
- Given: `_ticket_cache` 已 cache ticket P 內容
- When: save_ticket(t, P) 執行
- Then: cache invalidate 動作在 lock 範圍內完成（取 lock → 寫入 → invalidate → 釋 lock），下次 load 必取到最新內容

**場景 5：權限或磁碟錯誤**
- Given: lock file 目錄無寫入權限，或磁碟已滿
- When: `open(lock_path, "w")` 拋 `PermissionError` / `OSError`
- Then: 例外直接 propagate 給 caller，不吞錯

#### 4. Error Handling

| 情境 | 處理策略 |
|------|---------|
| `fcntl.flock` blocking 等待 | 預設 blocking，advisory lock 一般情境短暫；不設 timeout（避免複雜度） |
| `PermissionError` (lock file 建立失敗) | rethrow 給 caller |
| `OSError` (disk full 等) | rethrow 給 caller |
| Process crash 持鎖中 | OS fd 回收自動釋放 flock，無需主動清理 |
| Lock file 殘留 | 保留（不刪），下次 reuse 不衝突；加入 .gitignore 避免誤 commit |

#### 5. Dependencies

| 項目 | 說明 |
|------|------|
| `fcntl` (stdlib) | POSIX-only，macOS / Linux 支援；Windows 不在本 ticket 範圍 |
| `contextlib.contextmanager` | stdlib |
| 既有 `parser.save_ticket` 邏輯 | 包覆於 `with _file_lock(...)` block 內，邏輯不變 |
| `_ticket_cache` | invalidate 動作須在 lock 範圍內 |
| 測試環境 | pytest + multiprocessing fork mode（macOS Python 3.13 預設 spawn，需顯式 set_start_method('fork')） |

#### 6. Acceptance（細化）

對應 IMP 4 項 acceptance，細化為 Phase 2 RED 可驗證測試項目：

- [ ] **A1**：`parser.save_ticket()` 加入 `_file_lock` 包覆，per-ticket-file lock；signature 不變
- [ ] **A2-1**：Race regression test — N=20 並行 fork 同一 ticket 寫入，0 lost update（最終內容為其中一個 writer 的完整 payload，無 corruption / empty）
- [ ] **A2-2**：不同 ticket 並發測試 — 兩個 process 並發寫不同 ticket，總時間 ≈ 單次寫入時間（驗證 per-file 隔離）
- [ ] **A2-3**：Crash recovery 測試 — 子 process 持鎖中被 SIGKILL，主 process 後續寫入立即成功（< 1s）
- [ ] **A2-4**：Cache invalidation 測試 — save 後 immediate load 必取得新內容
- [ ] **A3**：`npm run test:hooks` 既有 hook test suite 全綠（含 ticket_system 全部既有測試）
- [ ] **A4**：`.gitignore` 加 `*.lock`（位於 `.claude/skills/ticket/ticket_system/` 或 root level，視既有 .gitignore 結構決定）
