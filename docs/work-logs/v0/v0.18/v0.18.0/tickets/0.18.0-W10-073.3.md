- 500 行的檔案若內容全部原子化且在同一情境共同需要 → 不應拆分

## A — Attain Distance（保持距離）

**6 個月後視角 + sync 到其他專案的視角**：

1. **平台約定面**：CLAUDE.md 是 Anthropic 平台約定用法，其他專案也會有自己的 CLAUDE.md。本檔案**不會 sync** 到其他專案（被 `.claude/scripts/sync-claude-push.py:L495,582` 明確排除），所以拆分決策是**專案內決策**，不影響框架 sync 品質。
2. **其他專案視角**：每個專案的 CLAUDE.md 都會有「專案身份 + 技術選型 + 開發指令」，但「專案特定規範（§6）」和「Chrome Extension 規範（§7.3）」是本專案獨特的，其他專案會換成 Flutter/Python 規範。這提示：§6 和 §7.3 本來就是「可被外部檔 100% 取代的塊」，拆出去是合理的解耦。
3. **6 個月後視角**：
   - 錯誤處理體系會擴充（新增錯誤類別）→ 外移能讓 CLAUDE.md 不需每次同步擴充

## P — Prepare to be Wrong（準備出錯）

**失敗場景與回退路徑**：

### 失敗場景 1：代理人沒讀到外移的規範 → 寫錯誤處理時用 `throw new Error`
- **機率**：中（需要依賴代理人主動讀 `@import` 或遇到需要時讀）
- **防護**：`@` 語法是 Claude Code 平台級別的 inline include，根據平台行為會在載入 CLAUDE.md 時一併載入被 `@` 引用的檔案（不是 lazy reference）。所以 §2 `@.claude/rules/core/quality-baseline.md` 現在已經這樣用。
- **回退**：若實測發現 `@import` 也會被預載（等於沒節省），退回選項 A（保持現狀），零成本回退。

### 失敗場景 2：`@` 引用的檔案被 lazy load，代理人沒主動讀
- **機率**：低（既有的 §2/§3 `@import` 實測是 inline include）
- **防護**：外移後仍在章節標題放 1 行摘要指引（例如「錯誤處理規範詳見 @docs/project-specific-conventions.md——寫產品程式碼前必讀」）
- **回退**：若代理人沒讀，加入 agent prompt 模板強制讀外移檔

### 失敗場景 3：章節 6 的錯誤處理規範太重要，外移後執行代理人漏看
- **機率**：中
- **防護**：已透過 `src/core/errors/` 目錄 README + 代理人指引雙重保護；CLAUDE.md 仍保留「強制規範」骨架（3 行），只把錯誤類別表外移
- **回退**：加回到 CLAUDE.md 成本極低（複製貼上）

### 失敗場景 4：Anthropic 未來升級 CLAUDE.md 處理規則（例如改用 YAML 前言、或停止支援 `@import`）
- **機率**：低
- **防護**：`@` 語法是 CC 現行功能，如未來變更，整個框架都需調整，非本 ticket 風險
- **回退**：若 `@` 停支援，將 CLAUDE.md 整體保留 inline（把外移的內容複製回來）

### 失敗場景 5：外移檔案 broken-link
- **機率**：低（已有 `/broken-link-check` 工具）
- **防護**：新增檔後跑一次 broken-link-check
- **回退**：修正路徑
