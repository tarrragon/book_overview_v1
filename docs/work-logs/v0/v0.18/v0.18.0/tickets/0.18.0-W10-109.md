### Phase 1 介面規格設計（PM 前台代做，W10-099 ANA 已具備 Phase 1 等級細節，補強規範化產出）

#### 1. 豁免優先序表（規則層敘述，物理 OR 不依賴順序）

| 優先序 | 豁免條件 | hook 觸發點 | 設計理由 |
|-------|---------|-----------|---------|
| P1 | Edit/MultiEdit 改動 < 30 字元（MICRO_EDIT_THRESHOLD） | `is_micro_edit()` 早 return 0（line 318-320） | 拼字/格式類微調不該被章節檢查阻擋（既有） |
| P2 | `status: draft`（任何 level） | `check_prop_content` line 162 之後新增分支早 return False | draft 為探索期，章節通常未完整；frontmatter `evaluation_level` 仍強制（規則 1 不豁免）以避免標記混亂 |
| P3 | `evaluation_level: light`（任何 status） | line 160-162 既有早 return False | light 為純拼字/格式/重命名類，無須完整評估章節（既有） |
| P4 | level=standard 或 heavy 且 status≠draft | 章節檢查（line 168-176） | 真正評估強度的 PROP 必須符合規則 2 章節要求（既有） |

**設計原則**：P2 與 P3 在物理層為 OR 關係（兩個獨立早 return），規則層敘述採 P2 先於 P3 順序對齊「狀態優先於分級」的語意（draft 探索期是更基礎的概念）。

**互斥保證**：P1 在 `is_micro_edit` 階段就早 return（main 函式 line 318），完全不進入 `check_prop_content`；P2/P3 在 `check_prop_content` 內各為獨立分支；P4 是 fall-through 預設路徑。

#### 2. hook `check_prop_content` status=draft 分支契約

**Given**:
- `content` 為 PROP-*.md 完整內容（已通過 `is_target_file` = "prop"）
- `fm = parse_frontmatter(content)` 解析成功（`fm != None`）
- `level` 已驗證為 VALID_LEVELS 之一（`light` / `standard` / `heavy`）

**When**:
- `fm.get("status")` 為非空字串
- `str(fm.get("status")).lower().strip() == "draft"`
- 觸發位置：line 162 既有 light 早 return 之後、line 164 章節檢查之前

**Then**:
- `logger.info("PROP status=draft，跳過章節檢查（豁免優先序 P2）")`
- `return False, ""`

**Edge cases**:
| 情境 | 行為 |
|------|------|
| 缺 `status` 欄位 | 不豁免（fall-through 到 P3/P4） |
| `status: ""` 空字串 | 不豁免 |
| `status: DRAFT` 大寫 | 經 `.lower().strip()` 豁免（保持寬容） |
| `status: draft` + `level: light` | 物理上 P3 先早 return（line 162），P2 不觸發；行為等價 |
| `status: confirmed` | 不豁免（規則 4 仍強制 ticket_refs） |

#### 3. 規則 1 修訂大綱（light 純語意收斂）

**現況**：規則 1「分級判定表」light 列定義為「spec 盤點批次建立 / 拼字修正 / 格式調整 / 重命名」。

**修訂方向**：
- 在「分級判定表」light 列**移除「spec 盤點批次建立」**（此用例對齊 draft 探索期語意，應改用 P2 豁免）
- 在「豁免判定」表後新增明示條款：
  > **light 不可作為 draft 暫設規避手段**。Draft 探索期 PROP 即使章節未完整，應保持 `status: draft` 並標明實際強度的 `evaluation_level`（standard/heavy），由 hook 自動豁免（P2 優先序）。

**Why（document-writing-style 三明示）**：
- **Why**：W10-099 重現實驗證實 light 100% 承載 grandfather 妥協、0% 設計用例。混用導致 PROP 真實強度與 evaluation_level 脫節。
