## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### Devil's Advocate 分析：為何不該重構 Rules 系統

### 一、Rules 30K 全載入的真正價值（3 條）

**V1：零遺漏防護保證**。全載入意味 PM 每個 session 都擁有完整規則視野。任何 lazy-load / RAG / hook-driven 方案都引入「漏載」風險。當前 115KB / 15 檔的 rules 是 146 個 PC error-pattern 的經驗結晶——每條規則背後都有至少一次實際違規事件。漏載一條 = 重蹈一次覆轍。

**V2：交叉引用完整性**。rules 之間有 45+ 處 PC-xxx 交叉引用。quality-baseline 規則 5 引用 decision-trigger-binding；ai-communication-rules 規則 5 引用 power-asymmetry-rules；cognitive-load 引用 task-splitting。全載入讓 PM 在任何決策點都能即時跳轉關聯規則。分拆載入後，PM 引用 A 規則時 B 規則不在 context，交叉引用斷裂。

**V3：新人代理人一致性基線**。每個 subagent 透過 AGENT_PRELOAD 繼承 rules 子集，但 PM 需要完整集來驗收代理人產出是否合規。PM 若只載入部分 rules，驗收時可能放過違規——這比「多花 token 載入」的代價嚴重得多。

### 二、重構失敗代價

**C1：規則碎裂**。將 15 檔拆成「核心 / 情境 / 細節」三層後，每次新增規則都需決定「放哪層」。分類標準本身就是主觀判斷，團隊成員（PM 的不同 session）會產生分類不一致，久而久之三層邊界模糊，退化為「三個目錄的 15 個檔案」——比原本更混亂。

**C2：新違規模式湧現**。重構過渡期（至少 3-5 個 session），PM 同時面對舊規則和新規則兩套系統。過渡期內的每個決策都有「用了舊版 / 漏了新版」的風險，預期產生 3-5 個新 PC error-pattern。修復這些 pattern 的成本可能超過 30K token 節省的收益。

**C3：Hook 系統連鎖斷裂**。當前 hooks 引用 rules 路徑（如 phase4-decision-enforcement-hook 引用 decision-trigger-binding.md）。重構改路徑 = 所有 hook 需同步修改。Hook 是靜默守護者，斷裂後不會報錯——只會靜默失效（quality-baseline 規則 4 的反面）。

### 三、X/Y/Z/K 各自的致命缺陷

**X（80/20 刪除低價值 rules）致命缺陷**：「低頻違規」不等於「低價值」。decision-trigger-binding（11.8KB）可能一季只觸發 2 次，但每次觸發都防止一個 PC-093 級延後決策累積。刪除「低頻」規則等於拆除低頻但高衝擊的安全網。

**Y（動態壓縮 summary 版）致命缺陷**：Summary 版必然丟失三明示（Why/Consequence/Action）中的 Consequence 和 Action 細節。PM 讀到 summary「禁止無 trigger 延後」但不知具體替代寫法（規則 3 替換對照表），等於知道禁令但不知怎麼合規——違規率反而上升。

**Z（Cognitive checkpoint 驅動載入）致命缺陷**：PM 必須先「知道自己需要什麼規則」才能觸發載入。但規則的價值恰恰在於 PM 不知道自己需要時仍能被提醒。這是 unknown unknowns 問題——你不知道你不知道什麼，所以無法主動請求載入。

**K（Rules 改 YAML schema + hook 強制）致命缺陷**：將 prose 規則機械化為 YAML 會丟失規則的「教育功能」。當前 rules 的三明示結構（Why/Consequence/Action）不只是「告訴 PM 不能做什麼」，而是「教 PM 為什麼不能做」。YAML schema 只剩 constraint，PM 不理解 Why 就會嘗試繞過 hook——產生對抗而非合作。

### 四、質疑「Context budget 是真痛點」

**反論 1：30K / 200K = 15%，這是合理的基礎設施成本**。作業系統核心占記憶體 5-15% 從未被視為「問題」。Rules 是 PM 的作業系統——15% 是廉價的。

**反論 2：用戶的真正痛點可能不是 token 而是注意力**。30K rules 載入後 PM 不需逐字閱讀，只在決策點被 hook 觸發時才關注。真正消耗注意力的是工作 context（ticket 來回、工具輸出），不是靜態 rules。如果痛點是「PM 決策品質下降」，解法應是改善決策流程（如 WRAP 觸發頻率），而非縮減規則集。

**反論 3：「較小 model 規則占比 60%+」是假設情境**。當前 PM 使用 200K context model。為尚未發生的 20K model 情境重構現有系統，違反 YAGNI 原則。

### 五、反論證據

- 115KB / 15 檔已是 L2 lazy-load 後的結果（references/ 已外移）。再瘦身的邊際收益遞減。
- 4/4 第一輪視角共識「L3 hook-driven」方向，但無人量化 trigger map 維護成本。trigger map 是另一個需要全載入的東西——把問題從「rules 全載入」搬到「trigger map 全載入」。
- 本專案 rules 累積速度約 2 條/月。照此速率 5 年後 ~180KB，仍在 200K 的 20% 以內。

### 六、「保留現狀 + 微調」最佳路徑

存在。具體路徑：

1. **刪重複段落**（不刪規則）：cognitive-load.md 的「跨進程同步修復豁免」和「監測校準框架」各 ~3KB，這些是特定案例的固化，可移至 references/ 按需讀取，不影響規則完整性。預估節省 6-8KB。
2. **壓縮版本歷史**：每個 rules 檔案尾部的 Version 歷史占 1-2KB/檔，15 檔合計 15-20KB。移至 CHANGELOG 或刪除（git log 已追蹤），預估節省 15-20KB。
3. **合併小檔**：quality-common.md（1.2KB）已只剩索引，可併入 quality-baseline.md。tool-discovery.md（2.9KB）和 tool-selection.md（4.2KB）可合併。預估減少 2-3 個檔案。

以上三步預估從 115KB 降至 85-90KB（約 22K tokens），節省 ~8K tokens/session，不引入任何結構性風險。

### 我可能錯在哪裡

Rules 系統可能已進入「沉沒成本陷阱」——我為保留現狀辯護的動機，可能部分來自「已投入大量精力建立這套系統」的認知偏誤，而非純粹的技術判斷。如果從零開始設計，未必會選全載入。
