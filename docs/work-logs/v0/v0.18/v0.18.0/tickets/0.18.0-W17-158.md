### 實測錯誤訊息（2026-05-07 session 結束時）

```
Stop hook error: Hook JSON output validation failed — (root): Invalid input
```

### Schema 對照

Claude Code 對 hookSpecificOutput.additionalContext 欄位的 schema 規定：

| Event | additionalContext 允許？ |
|-------|------------------------|
| PreToolUse | 否（用 permissionDecisionReason / updatedInput） |
| UserPromptSubmit | 是（required） |
| PostToolUse | 是（optional） |
| PostToolBatch | 是（optional） |
| **Stop** | **否（schema 完全沒列）** |

### 違規位置

`.claude/hooks/stop-worklog-handoff-sync-check-hook.py` line 309-314：

```python
output = {
    "hookSpecificOutput": {
        "hookEventName": "Stop",
        "additionalContext": warning,  # <- 違反 Stop event schema
    }
}
print(json.dumps(output, ensure_ascii=False))
```

### 影響範圍

5/5 stop hooks 全部 fail（hook 機制應為「任一輸出違反 schema 則整批 fail」），導致：
1. 此 hook 想傳達的「worklog 寫了 handoff 但 CLI 缺 handoff」同步檢查訊息丟失
2. 其他 4 個 stop hooks 也跟著 fail（連鎖效應）
3. session 結束時用戶看到一大段 schema 錯誤訊息（PC-127 風險）

### 修復方向候選

| 方案 | 欄位 | 評估 |
|------|------|------|
| A | `systemMessage` (string) | 顯示給用戶，類似 Stop hook 訊息——首選 |
| B | `stopReason` (string) | 顯示停止原因，語意較窄 |
| C | 直接 stderr 輸出（不用 JSON） | hook docstring line 21 提到的合併機制可能因此失效 |

建議首選方案 A（systemMessage），保留多 hook 合併能力。需 WebFetch 確認 Claude Code 文件。

### 衍生洞察（資料層）

修復後 hook 將正確輸出，session 結束時會看到「48 個 ticket 在 worklog 提到但無 CLI handoff」的清單——這是另一個資料層問題（W17-156 範圍應評估是否覆蓋；hook 偵測精度本身屬 W17-155 範圍）。本 ticket 限定 schema 違反修復，資料清理另開 ticket（如需）。

**Stop event 不在清單內** ← 確認本 ticket 根因

**Stop event 合法 hookSpecificOutput / 頂層欄位**：

| 欄位 | 位置 | 用途 |
|------|------|------|
| `systemMessage` | top-level（非 hookSpecificOutput 子欄位）| 顯示 warning 訊息給用戶（**首選**）|

**修復方案確認**：採方案 A 的修正版——`systemMessage` 是 **top-level 通用欄位，不是 hookSpecificOutput 子欄位**。
