### 問題根因

撰寫「禁止絕對主義 / 推崇機會成本語氣」的規範性文字時，作者自身使用絕對主義語氣（「必須」「禁止」「強制」），形成內容主張與形式表現的元層矛盾。W17-060 / W17-122 連續兩次案例顯示此模式可重現。

### 影響範圍

- 規範性文字（rules / methodologies / skills 主文 / pm-rules / error-patterns 防護段）
- 受影響行為：規則被讀者複製其形式（絕對語氣）而非其內容（機會成本框架）
- 相關 framework 路徑：`.claude/rules/`、`.claude/pm-rules/`、`.claude/methodologies/`、`.claude/error-patterns/`、`.claude/skills/*/SKILL.md`

### WRAP 四階段

**W (Widen Options)**：列三個處置方向

| 選項 | 說明 |
|------|------|
| A. 升級新 PC（PC-130 候選編號） | 獨立 error-pattern，含兩案例庫與防護機制 |
| B. 擴充既有 PC-066 加 dogfooding 維度 | 在 PC-066 加新章節「規範性文字 dogfooding」 |
| C. 不升級，依賴 W17-122 三層防護機制 | 已建 SKILL trigger（W17-125）+ Layer 2 流程（W17-124）+ commit msg 標記 |

**R (Reality Test)**：見 Problem Analysis「重現實驗結果」段（兩案例 P0 各 1，可重現）

**A (Attain Distance)**：機會成本

| 選項 | 落地成本 | 機會成本 |
|------|---------|---------|
| A 新 PC | 寫 PC（~3-5K token）+ 至少 2 案例引用 | 增 1 個 PC 維護負擔；後人讀 PC 須先理解 PC-066 邊界 |
| B 擴充 PC-066 | ~2K token | 模糊 PC-066 焦點（決策語氣 vs 書面文字 dogfooding） |
| C 不升級 | 0 | 系統性事後補做（W17-124 / 128 / 129 共 3 ticket 才結案）；事前不可發現 |

**P (Prepare to be Wrong)**：行前預想失敗

- A 風險：新 PC 與 PC-066 邊界仍不清晰，後人混用 → 防護：PC 開頭明示「決策語氣參考 PC-066；書面文字 dogfooding 參考本 PC」交叉引用
- B 風險：PC-066 變雜訊，dogfooding 章節被淹沒 → 防護：略
- C 風險：第三案例必然出現（兩次規律下無防護），事後補做累積 → 已實證（W17-124/128/129）

### 升級決策

**選 A：升級為新 PC（建議編號 PC-130，與 W17-122 ANA 預留編號一致）**

**理由**：
1. 兩案例證據充足且各 P0 1 條暗示系統性閾值（非偶發）
2. 與 PC-066 範疇可分離（決策框架 vs 書面文字產出）
3. 事後補做成本（3 ticket）顯著高於事前一次定義 PC
4. W17-122 三層防護（hook + claim + Layer 2）需要 PC 作為依據條款，否則防護的觸發條件無錨點

### PC-130 草案

#### 標題
PC-130: 規範性文字 dogfooding 違規（內容禁止絕對主義，形式採用絕對主義）

#### 症狀
撰寫「禁止 X」「推崇 Y」類規範性文字時，作者自身使用 X 語氣或未體現 Y 語氣，造成內容主張與形式表現的元層矛盾。

#### 根因（機制層）
1. **規範性文字的權威需求**：寫規則時 LLM 預設選擇「必須 / 禁止 / 強制」絕對命令詞，認為這樣才有約束力
2. **內容焦點遮蔽形式自審**：作者注意力集中於「禁止 X」的精準定義，忽略「自身正在使用 X」
3. **與 PC-066 共振**：context 沉重時規範性文字撰寫尤其易違規（PC-066 限制性語言預設反應在書面文字場域的延伸）

#### 案例庫

| 案例 | Ticket | 違規 | 修復成本 |
|------|--------|------|---------|
| 1 | W17-060（ai-communication-rules.md 規則 6） | P0 1 + P1 6 絕對主義 | W17-124 v1.2.1 + W17-128 兩輪修正 |
| 2 | W17-122 ANA Solution（三層防護章節） | P0 1 + P1 6 dogfooding 違規 | W17-124 / W17-128 / W17-129 三 ticket |

#### 防護（已分散落地，本 PC 統合錨點）

| 層 | 機制 | 落地 ticket |
|----|------|-----------|
| 事前 | claim 時若 ticket where.files 含 framework 路徑 + type=IMP，提示 Read compositional-writing SKILL | W17-125 |
| 事中 | PM 自做 framework 規則編輯 6 步驟流程（含 Layer 2 委員審查） | W17-124 |
| 事後 | commit msg 標 Layer 2 by [agent-name] 或 N/A by [理由] | W17-126（待落地） |
| 偵測 | framework-rule-edit-skill-trigger-hook 偵測 Edit framework 路徑時若無 SKILL Read 即提醒 | W17-127 |

#### 與 PC-066 邊界

| 維度 | PC-066 | PC-130 |
|------|--------|--------|
| 場景 | 決策框架選擇（限制性 vs 探索性） | 書面規範性文字撰寫 |
| 觸發條件 | context 沉重時 PM 主動決策 | Edit framework 路徑 + 撰寫規範性段落 |
| 防護機制 | Hook 偵測「無法 / 禁止」關鍵字 → 觸發 WRAP | claim 時 SKILL trigger + Layer 2 委員 |
