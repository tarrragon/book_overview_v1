### 實驗執行（思想實驗 — ANA 分析型）

基於 Hook 事件 schema 和既有 signal 處理邏輯推演（非實機測試，ANA 分析型 ticket 職責範圍）。

**觸發案例（True Positive）**：

| # | 事件 | tool_name | tool_input 特徵 | 預期行為 |
|---|------|-----------|----------------|---------|
| TP-1 | Write /tmp/solution.md | Write | file_path 含 /tmp/ 且 .md | 觸發 advisory |
| TP-2 | Bash: cat /tmp/x.md | Bash | command 含 cat /tmp/ | 觸發 advisory |
| TP-3 | Write 後接 Bash cat /tmp | Write+Bash | 連續 /tmp 寫讀 | 首次 Write 即觸發 |

**不觸發案例（True Negative）**：

| # | 事件 | tool_name | tool_input 特徵 | 預期行為 |
|---|------|-----------|----------------|---------|
| TN-1 | Write docs/work-logs/x.md | Write | 不含 /tmp/ | 不觸發 |
| TN-2 | Bash: npm test | Bash | 不含 /tmp 相關 | 不觸發 |
| TN-3 | Bash: echo wrote to /tmp | Bash | 僅 echo 含 /tmp | 不觸發（min_command_length） |

**誤報風險案例（False Positive 候選）**：

| # | 場景 | 預期 | 防護 |
|---|------|------|------|
| FP-1 | Write /tmp/test-output.log | 不應觸發 | .md 副檔名過濾 |
| FP-2 | Bash: python3 /tmp/script.py | 不應觸發 | command_pattern 限定 cat/read |
| FP-3 | Write /tmp/debug.md | 可容忍觸發 | advisory + cooldown 300s |

### 實驗發現

1. **Write /tmp/*.md 是最高信噪比偵測點**：PC-087 核心 pattern 是三步繞路，攔截第一步即可
2. **Bash cat /tmp 是輔助偵測點**：若 Write 已觸發則 cooldown 壓制
3. **FP-3 是可容忍的 false positive**：advisory 不阻擋，cooldown 防疲勞轟炸；與 charset-guard 阻擋式 FP 傷害本質不同
4. **不需跨步序列偵測**：單步偵測已足夠，無需複雜狀態機

### 核心設計一句話

偵測 PostToolUse 中 Write/Bash 對 /tmp/*.md 的操作，advisory 提醒 heredoc/Edit 替代路徑，復用 RestrictiveKeywordsStrategy 的 command_pattern 匹配機制。

---

### 1. 可偵測的多步 pattern 訊號

**推薦偵測點**：兩個獨立 tool_matcher，合併為單一 signal（同 cooldown）。

| 偵測點 | event_source | tool_matcher | 匹配邏輯 | 信噪比 |
|-------|-------------|-------------|---------|--------|
| A. Write /tmp/*.md | PostToolUse | Write | tool_input.file_path 匹配 /tmp/.*\.md$ | 高（PC-087 核心 pattern） |
| B. Bash cat/read /tmp | PostToolUse | Bash | command_pattern 匹配 cat\|head\|tail\|less 後接 /tmp/ | 中（輔助，cooldown 可壓制） |

**不推薦**的偵測方式：
- 跨步序列偵測（Write 後接 cat）：需狀態機，複雜度高，單步已足夠
- tool call 次數閾值（> N）：太泛，與具體 pattern 無關
- UserPromptSubmit 關鍵字：PM prompt 不會明說「我要寫 /tmp」

**技術可行性**：現有 Hook 架構的 tool_matcher 支援 Write/Bash；但 Write 的 file_path 匹配需新增 `file_path_pattern` 欄位（現有 command_pattern 只適用於 Bash）。替代方案：在 Hook Python 中為 Write tool 特殊處理 tool_input.file_path，由 YAML 的 command_pattern 統一配置（將 Write 的 file_path 視為等效 command）。

**推薦實作路徑**：新增 `TmpFileDetourStrategy` class，因為：
- Write tool 的匹配邏輯（file_path）與 Bash tool 的匹配邏輯（command）不同
- 現有 RestrictiveKeywordsStrategy 基於 UserPromptSubmit，不適用 PostToolUse
- 現有 ConsecutiveFailuresStrategy 基於計數累積，語意不同
- 新 Strategy 可同時處理 Write（file_path 匹配）和 Bash（command 匹配）兩個偵測點

### 2. False Positive 風險評估

| FP 場景 | 發生頻率 | 傷害程度 | 防護措施 | 容忍度 |
|---------|---------|---------|---------|--------|
| Write /tmp/debug.md（合理 debug） | 低 | 極低（advisory） | cooldown 300s | 可容忍 |
| Write /tmp/test-output.log | 中 | 無（.md 過濾排除） | file_path_pattern 限 .md | 已防護 |
| Bash: python3 /tmp/script.py | 低 | 無（command_pattern 排除） | 限定 cat/head/tail/less | 已防護 |
| Bash: echo "wrote /tmp/x.md" | 極低 | 極低 | min_command_length=15 | 可容忍 |

**與 charset-guard FP 的關鍵差異**：
- charset-guard 是**阻擋式**（exit 1），FP 直接中斷工作流 → 傷害大
- 本 signal 是**advisory**（exit 0 + stderr），FP 只多一行提醒 → 傷害極低
- 因此 FP 容忍閾值遠高於 charset-guard，不需 PC-074/084/085 等級的精密防護

### 3. 與既有 signals 職責切分

| Signal | 偵測目標 | 事件源 | 觸發時機 | 本 signal 差異 |
|--------|---------|--------|---------|--------------|
| S1 consecutive_failures | 代理人連續失敗 | PostToolUse (Task) | 失敗累積 >= 2 | 本 signal 偵測一次性路徑選擇 |
| S2 restrictive_keywords | PM 說「做不到」 | UserPromptSubmit | 限制性語句 | 本 signal 偵測工具使用行為 |
| S3 ana_claim | claim ANA ticket | PostToolUse (Bash) | ticket claim 時 | 本 signal 不限 ticket type |
| S4 reflection_depth_challenge | 反思深度質疑 | UserPromptSubmit | 用戶挑戰深度 | 本 signal 由工具行為觸發 |
| **S5 tmp_file_detour（新）** | **/tmp 繞路寫入** | **PostToolUse (Write/Bash)** | **Write /tmp/*.md 或 cat /tmp** | **路徑選擇偵測，非失敗/語句** |

**category 歸類**：`reflection_trigger`（建議是）。理由：
- 不是標準 WRAP 觸發（不是「做不到」或「連續失敗」）
- 是認知偏誤的行為觀察（PC-088 的 observable pattern）
- 引導方向是「自檢路徑選擇」而非「執行 WRAP 四階段」
- 訊息前綴應為 `[Reflection Trigger]` 而非 `[WRAP 絆腳索]`

### 5. 新 signal YAML 片段草案

```yaml
  - id: tmp_file_detour
    category: reflection_trigger
    enabled: true
    event_sources: ["PostToolUse"]
    tool_matcher: "Write|Bash"    # 需 Hook 支援多 tool_matcher（或拆為兩個 signal）
    # Write tool: 匹配 tool_input.file_path
    file_path_pattern: "/tmp/.*\\.md$"
    # Bash tool: 匹配 tool_input.command
    command_pattern: "\\b(cat|head|tail|less)\\s+/tmp/"
    min_command_length: 15
    reset_conditions:
      - manual_reflection_invocation
      - session_end
    message_template: |
      [Reflection Trigger] 偵測到 /tmp 中介檔操作（{matched_file}）。
      路徑預算自檢：這個內容能直接用 heredoc 傳入嗎？
        heredoc 直傳          — 1 步完成（bash-tool-usage-rules 規則五）
        Edit 直接修改 ticket  — 精確替換，無中介
        當前路徑              — Write + cat + pipe = 3 步
      參考：PC-087、PC-088 v2（單步敏感、總步驟盲）
```

**實作注意**：現有 tool_matcher 是單值字串，不支援 "Write|Bash"。有兩個選項：
- **選項 A（推薦）**：拆為兩個 signal（`tmp_file_detour_write` + `tmp_file_detour_bash`），共用 cooldown key
- **選項 B**：擴展 Hook 支援 tool_matcher 為 list 或 regex

### 6. FP 防護策略

| 防護層 | 機制 | 設定值 | 理由 |
|-------|------|--------|------|
| 副檔名過濾 | file_path_pattern 限 .md | /tmp/.*\.md$ | 排除 .log/.py/.json 等合理暫存 |
| 命令過濾 | command_pattern 限 cat/head/tail/less | 見 YAML | 排除 python3/node 等執行類命令 |
| 最小長度 | min_command_length: 15 | 15 | 排除短 echo 類誤觸 |
| cooldown | warn_cooldown_seconds（全域 300s） | 300 | 同一 session 5 分鐘內不重複 |
| advisory mode | hook_mode: advisory | exit 0 | FP 不阻擋工作流 |
| reset | manual_reflection_invocation / session_end | — | 手動觸發反思後重置 |

### 7. 與 D/E 方向的協同關係

| 方向 | 本 signal (G) 的角色 | 關係 |
|------|---------------------|------|
| D（W17-023 認知層 prompt） | G 的觸發頻率是 D 效果的**前後對照指標**：D 上線前後比較 G 觸發率 | 互補（G 驗證 D） |
| E（W17-024 結構層 CLI） | G 的高頻 pattern 是 E 的**bundle 候選來源**：哪些 /tmp 繞路最常見 → 優先做 CLI 整合 | 互補（G 驅動 E 的 ROI） |
| D+E+G 整體 | 三者無互斥：D 降低偏誤認知、E 消除繞路必要、G 觀察殘餘 pattern | 三層互補 |

**資料驅動回饋迴路**：G advisory 的 state.json 累積觸發紀錄 → 定期統計 → 觸發率高的 pattern 排入 E 的 CLI bundle 優先級 → E 上線後 G 觸發率應下降 → 若不降則回查 D 的 prompt 效果。
