## Solution

### WRAP 分析

#### W — Widen（擴增選項）

除 Context Bundle 列的 3 個選項外，補充第 4 選項：

| 選項 | 描述 |
|------|------|
| A | 全數移到 `docs/`，CLAUDE.md 用 `@docs/...` @import 或直接內嵌 |
| B | 保留 `.claude/references/`，在規則 8 引入「專案特定 references 子分類」例外 |
| C | 新增 `.claude/project/` 目錄區隔，sync-push 排除此目錄 |
| D | **混合策略**：強違反（`project-specific-conventions.md`）移 docs/；弱違反（`chrome-extension-quickref.md`）泛化（移除 Readmoo 字樣）後保留 `.claude/references/` 作為通用 Chrome Extension 技術速查 |

#### R — Reality Test（現實模擬）

模擬 sync 到假想 Flutter 專案的影響：

| 選項 | Sync 到 Flutter 專案的表現 | 嚴重度 |
|------|--------------------------|--------|
| A | 2 檔在 docs/，不 sync → Flutter 專案看不到（符合預期） | 無污染 |
| B | 2 檔 sync 過去 → 雖不自動載入，但 references/ 多出「Readmoo 錯誤處理」「Chrome Extension 限制」檔，代理人按需讀取時可能誤入 | 中等污染 |
| C | 2 檔在 `.claude/project/`，sync-push 排除 → Flutter 專案看不到；但新增一層目錄語義（`.claude` 是框架，但 `.claude/project` 是專案），`.claude/rules/README.md` 框架/產物分離表要加一層，sync-push EXCLUDE_PATTERNS 要改 | 無污染但架構成本高 |
| D | `project-specific-conventions.md` 不擴散（強違反被移除）；`chrome-extension-quickref.md` 泛化後變通用技術參考，對 Flutter 專案是冗餘但無誤導 | 低污染 |

#### A — Attain Distance（拉開距離）

**6 個月後觀察**：

- 若選 B：規則 8 例外會持續被援引擴張（新外移章節也會援引此例外），最終瓦解「框架/產物分離」
- 若選 C：隨專案發展會累積 5-10 個 `.claude/project/*.md`，需獨立索引，變成第二個文件體系
- 若選 A：自然擴展為 `docs/` 文件生態，與 `docs/chrome-extension-dev-guide.md`、`docs/struct.md` 一致
- 若選 D：規則 8 保持乾淨，強違反走 docs/，弱違反透過泛化變通用知識，可持續擴展

**類比案例**：Rails gem 是跨專案框架（類比 `.claude/`），`config/application.rb` 是專案配置（類比 `CLAUDE.md`），專案特定常數不放 Rails gem。本案邏輯同理。

#### P — Prepare to be Wrong（失敗準備）

| 風險 | 可能性 | 退路 |
|------|-------|------|
| `@docs/xxx.md` 在 Claude Code 中不支援跨目錄 @import | 低（官方語法支援相對路徑） | 改用絕對路徑 `@/Users/.../docs/...`；或內嵌回 CLAUDE.md（內容僅 ~50 行，不會過大） |
| 移到 docs/ 後代理人找不到 | 極低（`.claude/references/` 本來就是按需讀取） | CLAUDE.md 加 `> 本專案特定規範位於 docs/...` 指引 |
| Chrome Extension 相關人員找不到 quickref | 低（`docs/chrome-extension-dev-guide.md` 已存在） | `docs/README.md` 索引加註 |

### 量化比較

| 維度 | A（移 docs/） | B（規則例外） | C（新目錄） | D（混合） |
|------|---------------|---------------|-------------|-----------|
| 跨專案 sync 死連結風險 | 無 | 高 | 無 | 低 |
| CLAUDE.md 路徑變更處數 | 2 | 0 | 2 | 0-1 |
| 規則一致性 | 高 | 低（削弱規則 8） | 中（增一層例外） | 高 |
| 實施檔案異動數 | 3 | 2（但留債） | 5+ | 3 |
| 遷移成本工時 | ~30 min | ~15 min | ~60 min | ~45 min |
| 後續擴展成本 | 低 | 高（例外擴張） | 中 | 低 |
| 風險等級 | 低 | 中高（架構瓦解） | 中 | 低 |

### 最終建議：選項 D（混合策略）

**核心邏輯**：規則 8 違反不是二元的，應按違反程度分流處理。

**具體行動**：

1. **`project-specific-conventions.md`（強違反）→ 移到 `docs/`**
   - 新路徑：`docs/project-conventions.md`（或整併入 `docs/struct.md` §錯誤處理體系段落）
   - CLAUDE.md §6 改為 `@docs/project-conventions.md`（若語法支援），或直接內嵌回 §6（~50 行，不會讓 CLAUDE.md 過大）
   - 刪除 `.claude/references/project-specific-conventions.md`

2. **`chrome-extension-quickref.md`（弱違反）→ 泛化保留**
   - 移除「Readmoo 頁面結構」字樣，改為「平台頁面結構會變更，選擇器需多層 fallback」
   - 保留在 `.claude/references/` 作為跨 Chrome Extension 專案技術速查
   - CLAUDE.md §7.3 引用路徑不變

3. **規則層補強**：
   - `reference-stability-rules.md` 或 `document-format-rules.md` 新增「CLAUDE.md 章節外移決策樹」：章節含 `src/` 路徑/專案名稱 → 移 docs/；純技術平台知識 → 可保留 `.claude/references/`
   - DOC-010 更新或新建 DOC-011 記錄本次教訓

**為何不選 A**：`chrome-extension-quickref.md` 的 Chrome Extension 限制是跨 Chrome Extension 專案通用知識。未來若有另一 Chrome Extension 專案可直接複用，不應全移 docs/ 把通用知識「關在本專案內」。

**為何不選 B**：規則 8 核心價值是「框架/產物分離」。開例外會被援引擴張，最終瓦解分離。此為 ARCH-007 已教訓過的模式。

**為何不選 C**：增加 `.claude/project/` 一層讓新進代理人學習曲線變陡（要記「.claude 是框架，但 .claude/project 是專案」）；sync-push 要新增排除規則，工具複雜度擴散。簡單的 A 已能解決強違反，不需新結構。

|------|-------------------|---------------|

| C | 需改 EXCLUDE_PATTERNS | 消除（但工具要改） |
| D | 無 | 大幅降低（強違反消除，弱違反泛化後為通用知識） |

### CLAUDE.md @import 路徑穩定性

| 選項 | CLAUDE.md 變更 |
|------|---------------|
| A | line 141、172 改為 `@docs/...`（2 處，需驗證跨目錄 @import 支援） |
| B | 無變更 |
| C | line 141、172 改為 `@.claude/project/...`（2 處） |
| D | line 141 改為 `@docs/...` 或內嵌（0-1 處）；line 172 不變 |

選項 D 穩定性最佳（僅 1 處或 0 處變更）。
