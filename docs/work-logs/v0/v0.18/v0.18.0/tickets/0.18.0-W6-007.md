## 重現實驗結果

### 實驗方法

驗證 chrome-devtools-mcp 是否可作為本專案 Chrome Extension E2E 測試新路線。步驟：

1. `npx -y chrome-devtools-mcp --help` 取得官方 CLI 參數清單與工具特性
2. 對照本專案既有 E2E 架構（`tests/e2e/` + Puppeteer 22.15.0）
3. 評估 chrome-devtools-mcp 的 extension 載入機制與 CI 可行性

### 實驗執行

**chrome-devtools-mcp CLI 參數要點**：

- `--headless` / `--isolated` 支援，等同 Puppeteer 對應能力
- `--chromeArg=--load-extension=<path>` 可載入 unpacked extension（透過 chromeArg 透傳）
- `--categoryExtensions` 啟用 extension 專屬工具集（getExtensions/getExtensionLogs 等）
- `--browserUrl` / `--wsEndpoint` / `--autoConnect` 三種連接既有 Chrome 的方式
- `--slim` 提供 3 件最小工具（navigation/script/screenshot）

**關鍵限制（官方 help 明示）**：

> "categoryExtensions: ... currently only supported with a pipe connection. autoConnect, browserUrl, and wsEndpoint are not supported with this feature until 149 will be released."

即「載入 unpacked extension + 操作 extension」必須由 chrome-devtools-mcp **自啟動** Chrome（pipe 模式），無法接到已執行的 Chrome instance。Chrome 149 釋出後此限制可能解除。

**架構性差異**：

| 維度 | Puppeteer (現有) | chrome-devtools-mcp |
|------|------------------|---------------------|
| 介面 | Node.js API | MCP protocol (stdio/pipe) |
| 驅動者 | Jest test 腳本（程式化斷言） | MCP client（LLM/Claude Code） |
| CI 可行性 | 標準 Jest + headless | 需 MCP client 互動，不適合無人值守 |
| 斷言寫法 | `expect(...).toBe(...)` 程式碼 | LLM 解讀 snapshot/console，非結構化 pass/fail |
| Extension 工具 | 自行用 puppeteer API | 內建 categoryExtensions（功能更直接） |
| 適用情境 | 自動化迴歸測試 | 互動式探索、LLM-assisted debug |

**本專案既有 E2E 配置**：

- Jest + Puppeteer 22.15.0
- 五大分類：workflows / integration / performance / deployment / validation
- npm scripts：`test:e2e` / `test:e2e:workflow` / `test:e2e:integration` 等

### 實驗發現

**已驗證的事實**：

1. chrome-devtools-mcp 在本機環境可由 npx 啟動（`/opt/homebrew/bin/npx`），免額外安裝
2. 載入 unpacked extension 在參數層可行（`--chromeArg=--load-extension`）
3. extension 工具集存在，但僅限 pipe 連接模式（不能 attach 既有 Chrome）
4. CLI 無 `--script` / `--test-file` 類自動執行接口，所有操作須由 MCP client 即時下指令

**未驗證的假設（本 POC 範圍外）**：

- Chrome 149 釋出後 `--browserUrl + categoryExtensions` 是否真能 attach
- chrome-devtools-mcp 透過 MCP 協定的 snapshot 結構能否被 LLM 穩定解讀為 pass/fail

**結論：不可作為 Puppeteer E2E 替代品**。理由：

- chrome-devtools-mcp 設計為 MCP server，需 MCP client（如 Claude Code、Cline 等 LLM agent）即時下指令，無法產出機械可判定的 pass/fail
- CI/CD 無法掛 LLM 作為 E2E 執行者（成本、不穩定、不可重複）
- 既有 Jest + Puppeteer 已涵蓋五大類測試，遷移無收益

**結論：可作為開發期輔助工具，補位 Puppeteer 不擅長的情境**：

- 開發期手動驗證 popup 互動（Claude Code 直接驅動 Chrome 看 extension 行為）
- LLM-assisted debug：把 console/network/snapshot 餵給 LLM 找 bug
- 視覺確認 mock Readmoo page 上 extension 行為

### POC 結論

**判定：不採用 chrome-devtools-mcp 作為 Puppeteer E2E 替代品**

### 採用建議

| 角色 | 工具 | 理由 |
|------|------|------|
| 自動化 E2E（CI/CD） | Puppeteer 22.15 + Jest（現狀） | 程式化斷言、CI 可行、五大類已涵蓋 |
| 開發期手動驗證 | chrome-devtools-mcp（新增） | LLM 驅動、可看 popup、debug snapshot |

### 不採用為 E2E 替代的核心理由

1. **不是測試 runner**：chrome-devtools-mcp 是 MCP server，需 LLM/MCP client 互動驅動，無法 headless 跑出機械 pass/fail
2. **CI 不可行**：CI 環境無 LLM agent；要在 CI 跑等於每次測試呼叫 LLM API，成本與不穩定性不可接受
3. **無遷移收益**：現有 Puppeteer 已穩定，五大分類齊備，遷移工作量 vs 收益失衡

### 可選後續 Ticket（不強制）

**SUGGESTED 2**：建立追蹤 Ticket — Chrome 149 釋出後重評 `--browserUrl + categoryExtensions` attach 模式（目前官方明示不支援，未來可能改變）。
