## 深度反思紀錄（主線程親自回顧）

### 候選動機 6 假設

| # | 假設 | Reality Test |
|---|------|------------|
| H1 | Shell 習慣（傳統 sysadmin 模式）| 部分匹配——訓練資料常見 |
| H2 | 把長文字物化為檔案的心智模型 | 高度匹配——Markdown 表格 + 多段視為檔案 |
| H3 | 規避長 heredoc 在 Bash tool 的不確定性 | 部分匹配——未主動驗證容量 |
| H4 | 模組化錯覺（Write 持久化 + Bash 管道）| 匹配——無意識分離產生與注入 |
| H5 | 避免 Edit 既有 ticket 的 old_string 摩擦 | 匹配——150 行檔案下意識避開 |
| H6 | 降低失敗重試成本 | 弱匹配——事後合理化 |

### 真實根因（3 層深度）

表層（H1，PC-087 原版）：shell 習慣——解釋不了為何不用 heredoc

第一層深因（H2 + H4）：我把長文字 payload 物化為檔案。內容超過 20 行含表格/多段 markdown 時，思考切換到「檔案模式」，把它當文件系統物件而非命令參數。

第二層深因（H3 + H5）：認知負擔規避。heredoc 看起來不乾淨、Edit 需要 old_string 精確匹配，Write /tmp 是最低單步複雜度的表面路徑。

真實根因（最深）：當內容規模觸發「檔案感」時，我選擇增加步驟換取每步心智簡潔。違反 Occam's Razor 但個別步驟感覺更舒服。

### H1 vs 真因的差別

- H1 是被動歸因（訓練資料如此）
- 真因是主動認知策略（減少單步複雜度）
- 差別決定防護方向：H1 → 加規則；真因 → 加 check heuristic「是否把參數當檔案了？」

### PC-087 原版的 meta 問題

寫根因時我也在規避認知負擔：
- 「shell 習慣」= 1 分鐘可產出的表層答案
- 真因分析需要自我觀察 5 層，慢且痛
- 這本身是一個 error-pattern：「寫 error-pattern 時自己走認知負擔規避」

### 教訓與行動

1. 技術規則：寫 ticket 長內容前自問「payload 真需要獨立檔案嗎？」若答案只是「方便」= 規避訊號
2. 防護 Hook：main-thread-edit-restriction-hook 訊息可補「是否想用 heredoc 或 Edit ticket？」
3. 方法論：根因分析品質 checklist——若根因 1 句話就完，80% 是表層。真因應能解釋「為何不是 B/C/D」而不只「為何是 A」
4. PC-087 重寫：補「真實根因 3 層深度」章節，保留原 H1 為表層、加入 H2-H5 為深層
