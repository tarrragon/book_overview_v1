### WRAP 四階段分析

**Widen**：除了「mint 形似字辨識失誤」的直覺歸因，還有哪些可能根因？

| 候選根因 | 可能性 | 證據 |
|---------|--------|------|
| A. LLM token-level 形似字混淆（污/汙/汲字形相近） | 高 | 9 處中 3 處錯，且錯的方向一致（汲），非隨機 |
| B. mint prompt 缺字元驗證指令 | 高 | mint agent definition 無任何 normalize 白名單或字元驗證機制 |
| C. haiku model 中文字辨識能力不足 | 中 | mint 使用 haiku model + effort: low，降低了精細字元辨識能力 |
| D. 既有 PC-074/131/132 防護不覆蓋 subagent 產出 | 高 | PC-074 聚焦 hook false positive；PC-131 聚焦外部工具；PC-132 聚焦 hook log 回收。均未覆蓋「subagent 文字 normalize 誤改」 |

**Reality-test**：commit diff 實證（見重現實驗結果章節）確認 3/9 處錯改為「汲」，mint 自我報告也顯示認知為「汲」非 typo。

**Attain distance**：此問題不只是 mint 個案——任何 subagent 執行字元 normalize 任務時，都可能因 LLM 形似字混淆產生靜默誤改。目前無 post-commit 自動驗證機制。

**Prepare to be wrong**：方案需同時考慮 false positive（正確字被標記）和 false negative（錯字漏檢）的平衡。

### 問題根因

直接原因：mint-format-specialist（haiku model, effort: low）執行「污→汙」normalize 任務時，將 3 處「污染」誤改為「汲染」。「汲」(U+6C72, 汲取) 與「汙」(U+6C59, 汙染) 字形相近，LLM tokenizer 層面混淆。

系統性原因：
1. mint agent definition 無字元 normalize 白名單或驗證機制
2. 無 post-commit 字元驗證 hook 偵測形似字誤改
3. PC-074/131/132 字元家族防護不覆蓋此場景（見下方分析）

- 檔案：`.claude/skills/compositional-writing/references/principles/agent-team-context-isolation.md`
- 影響行：L17, L29, L74（3 處「汲染」，已由 commit 0aa3658a 修復）

### 實驗發現

**已驗證事實**：
1. mint 在同一 commit 中 9 處 normalize，6 處正確（污→汙）、3 處錯誤（污→汲），錯誤率 33%

6. 目前 `.claude/` 下已無殘留「汲」字

**仍未驗證的假設**：
- haiku model 是否比 sonnet/opus 更容易產生此類形似字混淆（需跨 model 對照實驗，超出本 ticket 範圍）
## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 候選防護方案評估

| 維度 | 方案 A: post-commit 形似字驗證 hook | 方案 B: mint agent prompt 加白名單 | 方案 C: dynamic validation Skill |
|------|-----------------------------------|----------------------------------|--------------------------------|
| 機制 | PreCommit hook 掃描 diff 中是否出現已知形似字混淆對（如 汲↔汙） | mint prompt 加入「normalize 目標字白名單 + 完成後自驗」指令 | 獨立 Skill 供任何 agent 呼叫驗證 normalize 結果 |
| 覆蓋範圍 | 所有 agent 的所有 commit | 僅 mint agent | 需 agent 主動呼叫 |
| false positive 風險 | 低（只檢查已知混淆對，非全字集掃描） | 無（prompt 層面，不阻擋） | 低 |
| 實作成本 | 中（新 hook + 混淆對清單維護） | 低（prompt 修改） | 高（新 Skill 定義 + 整合） |
| 防護強度 | 強（commit 層面硬阻擋） | 弱（依賴 LLM 自律，haiku 自律能力有限） | 中（依賴 agent 主動呼叫） |
| 可擴展性 | 高（混淆對清單可持續擴充） | 低（每個 agent 需個別加） | 中 |

### 推薦方案

**方案 A + B 組合**（雙層防護）：

1. **方案 B 先行**（低成本、快速落地）：在 mint agent definition 中加入 normalize 任務的「目標字白名單」指令段落，要求 normalize 完成後 grep 驗證目標字是否正確。
2. **方案 A 跟進**（系統性防護）：建立 PreCommit hook，維護已知形似字混淆對清單，掃描 commit diff 是否引入混淆字。初始清單：`{汲: 汙, 汚: 汙}`。

**不採方案 C 的理由**：Dynamic Skill 需要每個 agent 主動呼叫，與 haiku model 的 effort: low 設定衝突——低 effort agent 越不可能主動呼叫額外驗證工具。

### PC-150 error-pattern 設計

新建 `PC-150-subagent-homoglyph-normalize-misreplacement.md`（subagent 形似字 normalize 誤替換）：

- 症狀：subagent 執行字元 normalize 任務時，將目標字替換為形似字而非正確字
- 根因：LLM tokenizer 層面形似字混淆 + 缺 post-commit 驗證
- 案例：W11-028 mint 將 3/9 處「污」錯改為「汲」而非「汙」
- 防護：方案 A（hook）+ 方案 B（prompt 白名單）
- 姊妹模式：PC-074 / PC-085 / PC-131
