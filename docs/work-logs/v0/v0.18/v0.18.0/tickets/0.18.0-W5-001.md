**2.1 validation_templates — 模組級常數 + 匹配**

```
模組初始化（import 時執行一次）：
    定義 _TEMPLATES 為有序 tuple，順序固定為：
        (npm_test_pass, coverage_threshold, lint_pass, flaky_fixed, skipped_evaluated)
    每個 Template 欄位對照 §4 表格填入（patterns 全部小寫）

function match_template(ac_text):
    if ac_text is empty or only whitespace:
        return None
    normalized = ac_text.lower()
    for template in _TEMPLATES:      // 有序遍歷，第一個勝出
        for pattern in template.patterns:
            if pattern in normalized:   // 子字串匹配
                return ValidationCommand(
                    template_name = template.name,
                    command       = template.command,
                    timeout_sec   = template.timeout_sec,
                    parse_strategy= template.parse_strategy,
                    is_verifiable = (template.command is not None),
                    unverifiable_reason = template.unverifiable_reason
                )
    return None

function list_templates():
    return list(_TEMPLATES)    // 淺拷貝成 list；Template 本身 frozen 不可改
```

**2.2 ac_parser — 載入 + 剝除 + 組裝**

```
function parse_ac(ticket_id):
    // 步驟 1：解析版本
    components = id_parser.extract_id_components(ticket_id)
    if components is None:
        raise ValueError("無效的 ticket_id 格式: " + ticket_id)
    version = components["version"]

    // 步驟 2：載入 Ticket（既有 API 回傳 None 而非 raise）
    ticket = parser.load_ticket(version, ticket_id)
    if ticket is None:
        raise FileNotFoundError("找不到 Ticket: " + ticket_id)

    // 步驟 3：取出 acceptance 欄位
    acceptance = ticket.get("acceptance")
    if acceptance is None:
        return []
    if acceptance is empty list:
        return []

    // 步驟 4：型別防護
    if not isinstance(acceptance, list):
        raise ValueError("acceptance 欄位型別錯誤，預期 list，實際: " + type(acceptance).__name__)

    // 步驟 5：逐項解析
    result = []
    for index, raw_item in enumerate(acceptance):
        raw_str = str(raw_item)          // 保留原樣 → raw 欄位
        stripped = raw_str.lstrip()      // 允許前方空白

        checked = False
        text = stripped

        // 子步驟 5a：檢查 checkbox 前綴（YAML 引號已由 YAML parser 剝除，此處只剩 "[ ] xxx" 樣式）
        if stripped starts_with "[x]" or "[X]":
            checked = True
            text = stripped[3:].lstrip()
        elif stripped starts_with "[ ]":
            checked = False
            text = stripped[3:].lstrip()
        else:
            // 未帶 checkbox：保守視為未勾選，text = 原文（見技術債 4.2）
            checked = False
            text = stripped

        result.append(AC(index=index, text=text, checked=checked, raw=raw_str))

    return result
```

### 2. 資料結構定義

```python
# ac_parser.py
@dataclass(frozen=True)
class AC:
    index: int              # 於 acceptance list 的 0-based 位置；用於 PM 對應顯示「[AC 1]」
    text: str               # 已剝除 "[ ]" / "[x]" 前綴的純文字；模板匹配輸入
    checked: bool           # frontmatter 中該項是否已勾選；Phase 1 MVP 不寫回，但需紀錄
    raw: str                # 原始字串（含 checkbox）；除錯與回寫保留用

# validation_templates.py
@dataclass(frozen=True)
class Template:
    name: str               # 模板識別名（如 "npm_test_pass"）；錯誤訊息與日誌引用
    patterns: tuple[str]    # 關鍵字片段（小寫比對）；命中任一即匹配
    command: str | None     # Shell 指令；None 代表 unverifiable
    timeout_sec: int        # 預設 30；超過即視為阻塞（PROP-010 風險應變）
    parse_strategy: str     # "tail_lines" | "exit_code" | "coverage_number" | "manual"
    unverifiable_reason: str | None  # 若為手動則寫明原因（如「需要主觀判斷」）

@dataclass(frozen=True)
class ValidationCommand:
    template_name: str           # 取自 Template.name，利於日誌追溯
    command: str | None          # Shell 指令；None = unverifiable
    timeout_sec: int
    parse_strategy: str
    is_verifiable: bool          # command is not None 時 True；方便呼叫端分流
    unverifiable_reason: str | None
```

**欄位理由**（每個欄位為何需要）：
- `AC.index`：PM 輸出「[AC 1]」需要；frontmatter 沒有明確 id。
- `AC.text`：模板匹配唯一輸入；剝除 checkbox 避免 "[ ]" 干擾比對。
- `AC.checked`：W5-002 可能以此決定「已勾選的 AC 是否仍需驗證」（保留選項）。
- `AC.raw`：未來回寫 frontmatter 時保留原樣式，避免格式漂移。
- `Template.patterns` 用 tuple：immutable 利於測試與共享；小寫比對避免大小寫陷阱。
- `Template.command=None`：統一表達 unverifiable，不用另開子類。
- `Template.timeout_sec`：PROP-010 風險「驗證耗時 > 30 秒視為阻塞」的落地欄位。
- `parse_strategy`：W5-002 執行後解析輸出的策略指示（本 Ticket 不實作執行器，只宣告）。
- `ValidationCommand.is_verifiable`：derive 欄位，讓呼叫端用布林分流而非 None 檢查。
- `unverifiable_reason`：PM 顯示「跳過原因」，符合 PROP-010「< 15 行但需清楚」。

### 5. 關鍵設計決策

**決策 1：模板匹配策略 → 關鍵字子字串（大小寫無關）**

- 評估：正則靈活但難維護、關鍵字簡單但可能誤配、混合增加複雜度。
- 選擇關鍵字：PROP-010 已明示「失敗率 > 20% 則改明示對應，不做 NLP」。5 個模板規模下，關鍵字足夠且可預測。
- 風險：中文/英文混用關鍵字需兩組 pattern（已反映於表格）。
- 升級路徑：若 W5-005 回歸測試顯示誤配率過高，Template 增加 `regex_patterns` 欄位即可向後相容。

**決策 2：unverifiable AC 標記 → Template.command=None + is_verifiable 布林**

- 評估：三選項（Optional[ValidationCommand]、特殊型別 UnverifiableCommand、統一 ValidationCommand with flag）。
- 選擇統一型別 + 布林：呼叫端只需一個分支（`if vc.is_verifiable`），且可同時拿到 `unverifiable_reason` 做 PM 顯示。
- Optional return 的缺點：呼叫端無法區分「無匹配」與「匹配但不可驗證」，導致 PM 輸出失去「為何跳過」的資訊。
- 最終：`match_template` 回傳 `ValidationCommand | None`；None=無匹配，ValidationCommand(is_verifiable=False)=匹配但需人工。

**決策 3：規則庫承載方式 → 純 Python 模組常數，不用 YAML**

- 評估：YAML 熱載入 vs Python dict vs dataclass tuple。
- 選擇 Python 常數：規則庫變更頻率低（< 月/次），變更必然伴隨測試更新，YAML 熱載入增加 I/O、解析失敗路徑、測試雙軌（程式碼 + 配置）。
- 規則庫定義為模組級 `_TEMPLATES: tuple[Template, ...]`，透過 `list_templates()` 暴露只讀副本。
- 未來若需跨語言共享，再提煉為 YAML（YAGNI）。

### 7. Phase 2 測試策略綱要（供 sage-test-architect）

**ac_parser 測試場景**：
1. **正常 Ticket**：輸入 W5-001 自身（5 項 acceptance 全未勾選），預期 5 個 AC 物件，index 0-4，checked 全 False。不可遺漏：這是主路徑。
2. **已部分勾選**：mock frontmatter 含 `- '[x] 已完成'` 與 `- '[ ] 未完成'`，預期 checked 欄位正確。不可遺漏：W5-002 會依賴 checked。
3. **無 acceptance 欄位**：mock 無此欄位的 Ticket，預期 []（不 raise）。不可遺漏：非 TDD 類型 Ticket 可能無 AC。
4. **空 acceptance 清單**：`acceptance: []`，預期 []。不可遺漏：型別邊界。
5. **型別錯誤**：acceptance 是 str 或 dict，預期 raise ValueError。不可遺漏：防止 Ticket 格式腐化靜默通過。
6. **找不到 Ticket**：不存在的 ID，預期 FileNotFoundError。不可遺漏：錯誤傳遞正確。
7. **raw 欄位保真**：輸入含特殊字元的 AC，raw 與原始 YAML 值一致。不可遺漏：未來回寫基礎。

**validation_templates 測試場景**：
1. **每個模板正向匹配**：5 個模板各一個真實 AC 文字（取自既有 Ticket），預期正確回傳對應 ValidationCommand。不可遺漏：規則庫核心契約。
2. **每個模板負向拒絕**：不相關文字（如「更新文件」），預期不匹配。不可遺漏：避免誤配引發錯誤驗證。
3. **大小寫無關**：「NPM TEST 全部通過」能匹配 npm_test_pass。不可遺漏：PM 可能大小寫混用。
4. **多模板衝突優先序**：構造同時命中兩模板的文字，預期第一註冊者勝出且行為穩定。不可遺漏：決定性保證。
5. **unverifiable 模板回傳結構**：flaky_fixed 匹配時 is_verifiable=False、unverifiable_reason 非空。不可遺漏：PM 顯示跳過原因的基礎。
6. **無匹配回傳 None**：「部署至 production」等完全不相關文字回傳 None。不可遺漏：區分「無匹配」與「不可驗證」。
7. **list_templates 為只讀**：嘗試修改回傳值不影響後續查詢。不可遺漏：防止測試互汙染。

### 設計規格完成聲明

**關鍵取捨**：
1. **採用關鍵字子字串匹配而非正則**：犧牲靈活度換取可維護性與 PROP-010 一致性（反 NLP）；若後續誤配率高，Template 可向後相容加 regex_patterns 欄位。
2. **規則庫硬編碼於 Python 而非 YAML**：犧牲熱載入換取減少 I/O 與測試複雜度；規則庫變更頻率低，YAGNI 原則適用。
3. **ValidationCommand 統一型別 + is_verifiable 布林，而非 Optional/子類**：呼叫端單一分支，unverifiable_reason 同步傳遞，減少 W5-002 條件分支。
4. **timeout 欄位宣告但本 Ticket 不強制執行**：邊界清晰留給 W5-002；本 Ticket 純設計不碰 subprocess。

**風險**：
- **關鍵字誤配**（如「測試通過」誤配非 npm_test_pass 場景）：Phase 2 測試場景 4（多模板衝突）與 W5-005 回歸必須覆蓋；若誤配率 > 20% 觸發 PROP-010 應變。
- **patterns 中英文雙寫成本**：5 個模板尚可，擴張到 20+ 會成維護負擔；若發生，提煉為 YAML（決策 3 的升級路徑）。
- **frontmatter checkbox 解析脆弱**：YAML list item 可能為 `- '[ ] text'` 或 `- "[x] text"` 或無引號；parse_ac 須以字串前綴匹配而非正則解析，以涵蓋多種引號樣式。

**與 Context Bundle／PROP-010 的矛盾**：無。PROP-010 MVP 規格與 Ticket Context Bundle 一致，本設計未發現自相矛盾之處。

### 給 Phase 3a pepper 的實作提示

1. **YAML 引號變體（§設計風險第 3 點）**：frontmatter `acceptance` list item 可能為 `- '[ ] text'`、`- "[x] text"`、或 `- [ ] text`（無引號）。parse_ac 對 checkbox 剝除應以字串前綴匹配（如 `text.lstrip().startswith(("[ ]", "[x]", "[X]"))`）而非正則，以涵蓋所有引號樣式。測試 `test_raw_preserves_original_yaml_value_with_special_chars` 已覆蓋 '%' 與 '>=' 等特殊字元。

2. **Template 有序 tuple 初始化時機（§決策 3）**：`_TEMPLATES` 為模組級 `tuple[Template, ...]`，必須在 import 時即完成常數化（凍結），並於 `list_templates()` 回傳 `list(_TEMPLATES)` 副本。測試 `test_mutating_returned_list_does_not_affect_subsequent_calls` 會驗證外部 clear() 不汙染內部。注意：若使用 dataclass(frozen=True)，則 Template 物件本身亦不可變。

3. **load_ticket 回傳 None 轉 FileNotFoundError（§1 vs 既有 parser）**：既有 `parser.load_ticket()` 找不到 Ticket 時回傳 None，非 raise。parse_ac 需自行判斷 `if ticket is None: raise FileNotFoundError(f"找不到 Ticket: {ticket_id}")`。測試 `test_nonexistent_ticket_id_raises_file_not_found` 會驗證此錯誤傳遞。

4. **版本解析**：parse_ac 僅收 ticket_id（如 "0.31.0-W5-001"），需自行從 ID 解析版本（如 "0.31.0"），或透過既有 `id_parser` 協助。測試 fixture 使用 `monkeypatch.setenv("CLAUDE_PROJECT_DIR", ...)` 隔離檔案系統。

5. **衝突優先序決定性（§4 表格順序）**：`_TEMPLATES` 排列順序必須為 (npm_test_pass, coverage_threshold, lint_pass, flaky_fixed, skipped_evaluated)。測試 `test_first_registered_wins_when_multiple_match` 會構造「npm test 全部通過且覆蓋率 >= 80%」驗證 npm_test_pass 勝出。

6. **大小寫無關匹配**：`match_template` 應對 `ac_text.lower()` 做 substring 比對；Template.patterns 本身已規定小寫（§2）。測試含「NPM TEST」與「ESLint」等混合大小寫。
