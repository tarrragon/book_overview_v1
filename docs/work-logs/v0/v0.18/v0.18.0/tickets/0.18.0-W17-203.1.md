### 問題根因

PM 在派發前須手動掃描子任務 `where.files` 判斷可否平行派發。當 children 數 >= 3 且檔案重疊度不直觀時（例如同 `.claude/skills/ticket/` 下不同檔），人工判斷易遺漏「同目錄弱衝突」與 PC-137 並行上限。`askuserquestion-rules.md` 規則 7 規定的「可平行組 / 全衝突 / 部分可平行」三輸出語意缺乏 CLI 強制層支撐。

### 影響範圍

- 派發決策摩擦上升（PM 每次需逐 ticket grep `where.files` 後手動配對）
- 並行 race condition 風險（同檔 Edit / `.claude/` 三檔以上同步觸發 PC-137）
- AUQ 規則 7 應用不一致（PM 易跳過「全衝突需顯式說明」步驟）

### 相關 Error Pattern

- PC-137：>= 3 並行 `.claude/` Edit 觸發 CC runtime deny
- askuserquestion-rules.md 規則 7：選項含 ≥ 2 pending 子任務時必含平行派發選項

### 派發 Context（PM 補料 2026-05-17）

### 目標
新增 `ticket track parallel-check <id>` 子命令，自動分析目標 ticket 的 children/兄弟 ticket 的 `where.files`，分組為「可平行派發」/「衝突任務」/「單獨派發」，輔助 PM 套用 askuserquestion-rules.md 規則 7。

### 規則 7 上游脈絡（必讀）
- 規則檔：`.claude/pm-rules/askuserquestion-rules.md`（L221 起，規則 7 主文）
- 觸發門檻：選項含 ≥ 2 pending 子任務/兄弟
- 三輸出語意：
  - 可平行組（檔案互斥）
  - 全衝突 → 顯式列「無可平行派發」並說明
  - 部分可平行 → 列具體互斥子集 + 衝突項另列
- PC-137：≥ 3 並行 .claude/ Edit 強制 warning，建議拆批

### 演算法骨架
1. 讀目標 ticket → 取 children 清單（或 parent_id 推兄弟）
2. 過濾 status=pending（terminal 狀態不參與派發決策）
3. 對每對 (a, b) 比對 where.files 路徑前綴：
   - 同檔（完整路徑相等）→ 強衝突
   - 同目錄前綴（共同父目錄深度 ≥ 某閾值，建議用「最深共同祖先」）→ 視為衝突（保守策略，避免 race）
   - 完全不相交 → 可平行
4. 連通分量：用 union-find 把衝突的 ticket 連在一起，每個連通分量內僅能單一派發
5. 跨連通分量的「單身節點」（無衝突關係）可組成「可平行集合」
6. PC-137 檢查：若可平行集合中 ≥ 3 個 ticket 的 where.files 觸及 .claude/ 前綴 → warning + 建議拆批至 ≤ 2

### 衝突判定建議
- 同檔：路徑完全相等 → 強衝突
- 同目錄：取 Path.parents 集合交集，最深共同祖先深度 ≥ N（建議 N=3，即 .claude/skills/ticket/ 級）視為弱衝突
- 跨頂層目錄（如 docs/ vs src/）→ 無衝突
- 路徑前綴比較使用 Path API，禁 string startswith（避免 foo/bar vs foo/barbaz 誤判）

### 輸出格式範例（text）
=== parallel-check 0.18.0-W17-203 ===

[可平行派發] 2 ticket(s)（檔案互斥）
  - 0.18.0-W17-203.a  files=[.claude/hooks/x.py]
  - 0.18.0-W17-203.b  files=[docs/y.md]

[衝突任務] 1 group(s)
  Group #1（共同檔案 .claude/skills/ticket/SKILL.md）:
    - 0.18.0-W17-203.c
    - 0.18.0-W17-203.d

[單獨派發] 0 ticket(s)
[PC-137 警告] 無（.claude/ Edit 數 < 3）
