### 雙向映射 Schema 設計

#### 1. SKILL.md 觸發條件章節 ↔ YAML signals[].id 對應

設計原則：SKILL.md 觸發條件表列出人類可讀情境（14 列），YAML signals 為機器可讀子集合（4 個已實作訊號）。映射為 N:1 關係——多個 SKILL 情境可映射到同一 YAML signal，且 SKILL 情境可無對應 YAML signal（代表尚未自動化）。

| SKILL.md 觸發情境 | YAML signal id | 備註 |
|-------------------|----------------|------|
| 連續失敗 | consecutive_failures | 1:1 |
| 被困住 | restrictive_keywords | 語意重疊 |
| 偏離核心 | （無） | 未自動化 |
| 重大決策 | （無） | 未自動化 |
| 救火排擠 | （無） | 未自動化 |
| 分析任務 | ana_claim | 1:1 |
| 根因深度反思 | reflection_depth_challenge | 語意相近 |
| 反思深度質疑 | reflection_depth_challenge | 1:1 |
| 提案評估 | （無） | 未自動化 |
| 個人化建議 | （無） | 未自動化 |
| CLI/規則 autopilot（3.1） | （無） | 未自動化 |
| 既有結論錨定（3.2） | （無） | 未自動化 |
| 規則失敗草率改規則（3.3） | （無） | 未自動化 |
| 多步驟成功率盲點（3.4） | （無） | 未自動化 |

**一致性約束**：每個 YAML signal id 必須至少對應 1 個 SKILL.md 觸發情境（YAML 不可有 orphan signal）。SKILL.md 可有未自動化的情境（人工觸發）。

#### 2. SKILL description Triggers 段 ↔ YAML keywords 隸屬關係

SKILL description `Triggers:` 段為語意類別標籤（如 `stuck`, `限制性解法`）。YAML keywords 為該類別下的具體偵測詞。隸屬關係定義：

| SKILL description Trigger 類別 | 對應 YAML signal | YAML keywords 欄位路徑 |
|-------------------------------|------------------|----------------------|
| stuck, blocked, loop, no progress | consecutive_failures | `.failure_detection.keywords` + threshold 邏輯 |
| 限制性解法 | restrictive_keywords | `.keywords[]` |
| 分析 | ana_claim | `.command_pattern`（regex） |
| debug | consecutive_failures | 同 stuck 類別 |
| 個人化建議, 具體推薦 | （無） | 未自動化 |

**一致性約束**：YAML keywords[] 中的每個詞必須能歸屬到至少一個 SKILL description Trigger 類別（禁止 orphan keyword）。新增 keyword 時需標注所屬 Trigger 類別。

#### 3. Version 比對規則

**決策**：採用 SKILL.md 主文 footer `**Version**: X.Y.Z` 標記作為 SKILL version 來源（非 frontmatter）。

理由：
- SKILL.md frontmatter 目前只有 `name` 和 `description`，加 `version` 需變更 Claude Code runtime 解析假設
- 主文 footer 已存在且持續維護（目前 v2.3.0）
- YAML 已有 `version: "1.2.0"` 欄位

**比對規則**：
- YAML version 與 SKILL version 獨立演進（不要求相同數值）
- 一致性檢查僅驗證「非回退」：新版 >= 舊版（semver compare）
- Hook 讀取方式：regex `\*\*Version\*\*:\s*(\d+\.\d+\.\d+)` 從 SKILL.md 尾部 20 行提取

### W10-055 IMP 三項 Acceptance 重寫

原 AC 全部作廢，替換為以下 3 項（可自動驗證）：

#### AC1：YAML signals orphan 檢查

> Hook 驗證：對 `wrap-triggers.yaml` 每個 `signals[].id`，在 `.claude/skills/wrap-decision/SKILL.md` 觸發條件表（`## 觸發條件` 章節的 Markdown 表格）中至少存在 1 行 match。Match 方式：YAML signal id 的 snake_case 轉為空格分隔詞後，觸發條件表「情境」欄含該詞或在映射檔中有顯式對應。

驗證指令：`grep -c "signal_id_word" SKILL.md` 或讀取映射檔 JSON/YAML。

#### AC2：YAML keywords orphan 檢查

> Hook 驗證：對 `wrap-triggers.yaml` 每個 signal 下的 `keywords[]` 和 `failure_detection.keywords[]`，在 SKILL.md description frontmatter `Triggers:` 段的類別詞中，存在語意歸屬（由映射檔定義）。

驗證指令：讀取映射檔，確認每個 keyword 有 `belongs_to_trigger_category` 欄位。

#### AC3：Version 非回退檢查

> Hook 驗證：`wrap-triggers.yaml` 的 `version` 欄位與 SKILL.md footer `**Version**:` 欄位，各自 >= git HEAD~1 中同檔案同欄位的值（semver compare）。任一回退 → 警告。

驗證指令：`git show HEAD~1:.claude/config/wrap-triggers.yaml | grep version` vs 當前值。

### Hook 規格

#### 觸發時機

#### 檢查項目

| 項目 | 邏輯 | 結果 |
|------|------|------|
| Signal orphan | 每個 YAML signal id 在映射檔中有對應 SKILL 情境 | 警告 |

| Version 非回退 | semver(current) >= semver(previous) for both files | 警告 |
| 映射檔存在 | triggers-alignment 檔案存在且可解析 | 阻擋（exit 2） |

#### 警告 vs 阻擋策略

- **警告（exit 0 + stderr）**：Signal orphan、Keyword orphan、Version 回退。理由：advisory mode 與既有 tripwire hook 一致，不阻擋開發流程。
- **阻擋（exit 2）**：映射檔不存在或無法解析。理由：映射檔是檢查的前提，缺少時所有其他檢查無意義。

#### Observability log 要求（依 observability-rules.md 規則 1-3）

- 每次觸發必須 stderr 輸出前綴 `[WRAP Consistency]`
- 異常（YAML parse error / 映射檔 missing）必須同時 stderr + logger.error
- 正常通過可 logger.info（含檢查耗時）
- 禁止空 except 靜默吞掉
