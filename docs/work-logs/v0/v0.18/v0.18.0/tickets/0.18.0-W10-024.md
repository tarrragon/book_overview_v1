## Problem Analysis

### Context Bundle 擴充（2026-04-15 PC-070 實證後）

**根因定位**：PostToolUse(Agent) 觸發時機

`active-dispatch-tracker-hook.py` L60-L93 邏輯：
1. L61: PostToolUse(Agent) 觸發 → `clear_dispatch(project_root, agent_description)`
2. L84: 查詢剩餘活躍派發 `get_active_dispatches`
3. L85-93: 無剩餘 → 廣播 `[OK] 所有代理人已完成`

**關鍵 bug**: `run_in_background=true` 模式下，Agent tool call 在代理人**啟動時**立即返回（帶 agentId），PostToolUse(Agent) 此時就觸發，但代理人仍在背景 running。Hook 在此時點清除 dispatch 記錄並廣播「已完成」是**過早**的判定。

**實證證據（2 次觸發）**:

| 案例 | 代理人 | 派發時間 | Hook 廣播 | 實際狀態 |
|------|--------|---------|---------|---------|
| 1 | W10-056.2（thyme） | 15:16 派發 | 派發後立即廣播「完成」 | running（真正完成於 15:23） |
| 2 | W10-056.3（thyme） | 15:32 派發 | 派發後 <1 分鐘廣播「完成」 | running（TaskOutput status 確認） |

兩次都是 background agent，Hook 在代理人真正完成前就廣播「所有代理人已完成」，誘發 PM 進入錯誤驗收 SOP。

**修正方向建議**（供執行者參考）:

1. **時機調整**：PostToolUse(Agent) + `run_in_background=true` 時不清除 dispatch，改以 task-notification 為完成訊號
2. **或**：清除邏輯保留，但廣播判斷改用「task completion event」而非「PostToolUse」作為訊號源
3. **或**：若 Hook 無法感知 background vs foreground，改以「Agent 的 tool_response 是否含完成狀態」判定
4. **訊息細緻化**（原 W10-024 scope）:
   - 有活躍派發 → [WAIT]
   - 無活躍派發且未派發過 → 不輸出
   - 無活躍派發且剛完成驗收 → [OK]

## Solution

採用方案：**B**（時機修正 + 三態訊息細緻化）。

### 根因修復

`active-dispatch-tracker-hook.py` 在 PostToolUse(Agent) 時無條件執行 `clear_dispatch` 並依「剩餘活躍派發」廣播完成訊息。對 `run_in_background=true` 的派發，PostToolUse 在代理人**啟動時**（agentId 返回即觸發）就執行，代理人仍在背景 running，Hook 此時清記錄並廣播「所有代理人已完成」是過早判定，誘發 PC-070。

修復做法：
1. 新增 `is_background = tool_input.get("run_in_background", False)` 判斷。
2. 背景代理人路徑：**不呼叫 `clear_dispatch`**（保留記錄讓 task-notification 事件處理）、**不輸出 [OK]/[WAIT] 訊息**，但仍執行 `cleanup_expired` / `detect_orphan_branches` housekeeping。
3. 前景代理人路徑保留原清理語意，但廣播改為三態：
   - 有剩餘活躍派發 → `[WAIT]`
   - 無剩餘 且 `cleared=True`（本次確實清掉一筆）→ `[OK]`
   - 無剩餘 且 `cleared=False`（未派發過 / 已被清過）→ **不輸出 dispatch-state 訊息**
4. 若 `messages` 為空（純啟動、純未派發），直接 `return 0` 不輸出 additionalContext，避免 PM 每次 Bash 後看到雜訊。
