# Execution Log

## Task Summary

新增 /clear 前強制 main 未提交變更檢查規則

---

## Problem Analysis

### 問題根因

session 結束前 PM 在 main 分支可能累積未提交變更（決策紀錄、規則調整、跨 ticket 中介改動）。若隨 `/clear` 丟失 context，後人僅見 disk 上檔案不知決策理由——記憶在 context、檔案在 disk，兩者斷鏈。既有 `session-switching-sop.md` 列出「未提交變更應 commit」但無強制動作、無 main vs worktree 差別對待，導致 PM 在 context 沉重時為求快而跳過 commit。

### 影響範圍

- `.claude/pm-rules/session-switching-sop.md`（/clear 前確認章節 + 禁止行為）
- `.claude/pm-rules/askuserquestion-rules.md`（場景 #11a/#11b 前置條件）
- 透過 `pm-role.md` 情境路由觸達

### 相關 Error Pattern

無直接 PC，但與「context 沉重時跳過 commit」之決策疲勞傾向（PC-066 系列）同源。

---

## Solution

### 變更摘要

#### 1. session-switching-sop.md（v1.1.0 → v1.2.0）

新增「main vs worktree 差別對待（強制）」子章節：

- 三明示（Why/Consequence/Action）說明為何強度分流
- 對照表：主倉庫 main 有未提交 → 強制 commit；worktree 有未提交 → 提示不強制
- 提供判別 CLI（`git rev-parse --show-toplevel` / `--abbrev-ref HEAD` / `git status --short`）
- 「禁止行為」表新增條款：main 有未提交時直接呈現 #11 視為違規

#### 2. askuserquestion-rules.md（v3.11.0 → v3.12.0）

- 場景表 #11a/#11b 觸發條件加註「前置：main 必須 clean」
- 「選項前提檢查」之上新增「#11 /clear 前置條件（強制）」章節：對照表列出主倉庫 main / worktree / 全 clean 三狀態的處理動作
- 引用 session-switching-sop.md 為 SSOT

### 設計決策

| 決策 | 理由 |
|------|------|
| main 強制 / worktree 提示 | worktree 由代理人持有可能有並行 terminal；強制 commit 會搶用其他 terminal 編輯中檔案 |
| 透過 #11 前置條件落地 | #11 是 commit-handoff-hook 的 PM AUQ 觸發點，是 /clear 進入的最近上游節點 |
| 不強制 Hook 阻擋本次落地 | Hook 實作（PreToolUse 偵測 /clear 關鍵字）由 W10-017.7 條件式追蹤，本 ticket 先固化規則層 |

### 不在本 ticket 範圍

- Hook decision-checkpoint-pre-clear-hook 實作（W10-017.7 已追蹤）
- 自動化測試案例（規則層落地，本身為自律規範）
