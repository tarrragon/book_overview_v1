## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W6-012.2.1 修復後新提取的書籍主 ID 為 `reader-{privacyBookId}`（如 `reader-18337744`），但用戶於 W6-012.2.1 之前已提取的 `chrome.storage.local.readmoo_books` 仍以 `cover-{slug}` 為 id（如 `cover-pirited`）。若無遷移機制，下次提取會把同一本書視為「新書」與舊記錄並存，破壞書庫去重與用戶自訂 tag/狀態。

### 影響範圍

| 檔案 | 角色 |
|------|------|
| `src/background/events/event-coordinator.js:562` | 主要 storage 寫入點（`chrome.storage.local.set({ readmoo_books: storageData })`） |
| `src/background/messaging/popup-message-handler.js:385-388` | 讀取 readmoo_books 並回傳給 popup |
| `src/background/constants/module-constants.js:521` | `READMOO_BOOKS: 'readmoo_books'` 常數 |
| `src/background/lifecycle/install-handler.js:194-195` | onUpdated 呼叫 `migrationService.migrate(previousVersion, currentVersion)`，但 dependencies.migrationService 未實作 wire |
| `src/data-management/migration/v1-to-v2.js` | 既有 stand-alone schema migration（v1 → v2 schema 3.0.0），無 caller 串接 |
| `src/background/domains/data-management/services/cross-device-sync-service.js` | 跨裝置同步可能含舊 ID |
| `src/background/domains/extraction/services/data-processing-service.js:196` | dataProcessors 註冊 readmoo_books 處理器 |
| `src/export/` (CSV/JSON 匯出) | 匯出檔含舊 cover-XXX ID，外部資料無法自動遷移 |

### 既有 migration 框架現狀

- `MigrationService` 是 install-handler 的 dependency injection 介面，**但無實作 class**（grep 全 src 無 `class MigrationService` 或 `new MigrationService`）
- `v1-to-v2.js` 是孤兒模組（無 import 引用），原本要在 install-handler.onUpdated 觸發但未 wire
- BookSchemaV2.SCHEMA_VERSION = '3.0.0'

### 相關 Error Pattern

無直接對應 PC。屬「schema 演進 + 不可逆 ID 變更」場景，需參考 BookSchemaV2 既有 v1→v2 設計模式（先備份 → 轉換 → 寫入 → 刪除備份 → 失敗回滾）。

### 實驗發現

**已驗證的事實**：

1. `readmoo_books` 單一 storage key 集中，遷移點明確
2. 既有 migration 框架（v1-to-v2.js + MigrationService 介面）為孤兒——v1-to-v2.js 含完整 backup/restore 邏輯但無 caller
3. 受影響欄位：`bookData.id`（會從 `cover-XXX` 變為 `reader-{privacyBookId}`）
4. 比對 key：`bookData.identifiers.privacyBookId` 早已寫入（W6-012.2 Phase 2 確認）；對含此欄位的舊記錄，遷移為 deterministic 1:1 轉換
5. 邊界 case：
   - 5a) `identifiers.privacyBookId` 為空（privacy element 缺失 fallback case）→ 無法 deterministic 遷移
   - 5b) coverId 為 `openbook` 等預設圖 → 多本舊書共用同一 cover-XXX ID（已 corrupt）
   - 5c) 用戶自訂 tag / 閱讀狀態（v2 schema 引入）綁定在舊 ID 上 → 遷移時需保留並 rebind 到新 ID

**仍未驗證的假設**：

- H1: 是否存在 import 路徑（手動匯入 JSON 含 cover-XXX）需同時處理？需檢視 `src/data-management/import/` 或 popup import handler 路徑
- H2: cross-device-sync 是否將 ID 作為跨裝置主 key？若是則一裝置遷移完成後同步會把舊 ID 再 push 回來

H1/H2 屬實作期需驗證的細節，不影響本 ANA 結論的方向（仍是「主動遷移 + 比對 privacyBookId」）。

### 合併策略（候選方案）

| 方案 | 觸發時機 | 優點 | 缺點 |
|------|---------|------|------|
| A 主動遷移（首次升級） | install-handler onUpdated 觸發 schema 升級 | 一次到位、用戶無感、與既有 v1-to-v2 框架對齊 | 大型書庫遷移時間長；需 backup/rollback 機制 |
| B 延遲遷移（提取時合併） | 下次用戶觸發提取，新舊比對合併 | 無啟動時開銷；錯誤可重試 | 遷移時間取決於用戶操作頻率；中間期新舊並存產生暫態錯誤 |
| C 混合（A 主動 + B 後備） | onUpdated 主動 + 提取時補捕 | 覆蓋率最高 | 邏輯複雜度高，需確保不重複遷移 |

**推薦方案：A 主動遷移**

理由：

- W6-012.2.1 已將 SCHEMA_VERSION 從 cover-XXX 含意演進到 reader-XXX，屬 schema 級變更，與既有 BookSchemaV2 v1→v2 同性質
- v1-to-v2.js 框架可重用（先備份 → 轉換 → 寫入 → 刪除備份 → 失敗回滾）
- 主動遷移可一次性解決，避免 B 的「中間期暫態錯誤」
- 配合 W6-012.5 E2E 驗證自然涵蓋遷移路徑

### 合併規則設計

| 案例 | 條件 | 處理 |
|------|------|------|
| 1 正常遷移 | 舊 id=`cover-XXX` + identifiers.privacyBookId 存在 | 新 id=`reader-{privacyBookId}`；保留所有自訂欄位（readingStatus、tagIds、progress） |
| 2 privacyBookId 為空 | 舊 id=`cover-XXX` + identifiers.privacyBookId='' | 保留舊 id；遷移日誌標記 `requires_manual_review`；UI 顯示「待重新提取」徽章 |
| 3 cover-openbook 共用 ID | 多本舊書 id 都是 `cover-openbook`（已 corrupt） | 用 title + extractedAt 作為 secondary key 去重；保留所有記錄但加 `legacy_duplicate` 標記 |
| 4 同 privacyBookId 多筆 | 遷移後同 reader-XXX ID 有舊新兩條（極少見，可能是手動匯入舊備份覆蓋） | 合併規則：時間較新的記錄為主；自訂欄位（tag/狀態）採並集 |
| 5 跨裝置 sync 衝突（H2 待驗） | 遷移完一裝置後，sync 服務 push 來另一裝置的 cover-XXX | sync 比對時也加入 privacyBookId 後備 key（屬 IMP 子任務） |

### 遷移實作流程（仿 v1-to-v2.js）

1. install-handler.onUpdated 偵測版本升級到含本修復的版本
2. 讀 `readmoo_books`，掃 `books[].id` 是否含 `cover-` 前綴
3. 若有：備份原資料至 `migration_backup_cover_to_reader`
4. 對每本書依「合併規則」表處理；失敗的書記入遷移日誌
5. 寫回 `readmoo_books`；schema_version 升至下一版本（如 `3.1.0`）
6. 驗證寫入成功後刪除備份
7. UI 通知用戶遷移完成 + 受影響書籍數 + 待重新提取數
