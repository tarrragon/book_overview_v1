### 觀察到的反模式

#### 反模式 1：WRAP W (Widen) 階段不夠廣泛 → R 階段過早收斂

**Why**：第一輪 WRAP（W17-098）只列出單一假設「runtime hardcoded deny」，沒系統性列出可能變因。

**案例（W17-098）**：
- W 階段只比對 PM vs subagent 結果差異，得出「runtime 層問題」結論
- R 階段直接宣稱「成本/收益不對稱不執行」——但實際成本只是「派發 1 個 thyme 跑簡單 Edit」極低
- A/P 階段流於形式

**後果**：結論寫入 PC-115 + 更新 ARCH-015 memory，被 W17-099 一個簡單實驗就推翻。

**核心瑕疵**：W 階段把「現有觀察的可能解釋」當成「所有可能假設」。應該列「實驗能告訴我什麼」而非「現有資料能推什麼」。

---

#### 反模式 2：用「偶發 / 不可重現 / transient」當分析失敗的合理化標籤

**Why**：W17-099 推翻 W17-098 後直接寫「偶發 deny，條件不明」當收場。

**案例**：
- W17-098 deny / W17-099 success 兩個觀察值，沒進一步控制變因就標籤化「偶發」
- 用戶質疑「偶發 deny 這種結論是不合理的」
- 觸發第二輪 + 第三輪 WRAP，提出 5 個變因 + 設計 E1/E2/Test G 對照實驗
- 結論：5 個變因 4 個被否證，剩下 1 個（PM prefix 接觸史）與所有實驗對應

**核心瑕疵**：「偶發」是一種反詞——把「我沒找到原因」描述成「沒有原因」。系統行為極少真正偶發；多數情況是隱藏變因未被識別。AI 推理特別容易掉入這個陷阱因為它看似科學中立。

---

#### 反模式 3：推論粒度錯誤（檔案層級 vs prefix 層級）

**Why**：Test G 設計時用「PM 是否已 Edit 過該檔」（檔案層級）作為變因，否證後跳到「不是 PM 接觸史」結論。

**案例**：
- Test G W17-102 編輯 `.claude/methodologies/atomic-ticket-methodology.md`（PM 該 session 從未碰過）
- 結果 success → 結論「PM 接觸該檔並非決定因子」
- **遺漏**：PM 雖未碰過該檔，但已 Edit 過 `.claude/skills/ticket/...` 等多個 `.claude/` 內檔案
- 真正的變因應該是「PM 是否已 Edit 過 prefix（`.claude/`）內任一檔」（前綴層級）

**核心瑕疵**：實驗設計時假設變因有特定粒度（檔案），但決定因子可能在不同粒度（前綴 / 模組 / session）。否證後沒檢查「是否變因粒度搞錯」就跳結論。

---

#### 反模式 4：推翻後仍 anchor 在原框架

**Why**：W17-098 第一輪寫「runtime hardcoded deny」；W17-099 推翻後仍寫「runtime 偶發性 deny」——只是把 hardcoded 換成偶發，沒跳出「runtime 層問題」框架。

**案例**：
- W17-099 success 應觸發「重新審視所有變因」，但實際只是把絕對結論軟化為「偶發」
- 真正的破解需要用戶質疑「偶發不合理」才觸發

**核心瑕疵**：第一個假設一旦寫入文件，AI 推理會傾向修補它而非完全推翻。需要外部觸發（用戶質疑）才能跳脫。

### 元學習

本 session 的反模式不是個案，是 AI 推理的系統性傾向：
- AI 傾向尋找已有資料能解釋的最簡假設（Occam's razor 過度套用）
- 當假設被推翻，傾向「修補」原框架而非完全重啟思考
- 「偶發」是 AI 在沒有結論時的心理避震器

人為觸發（用戶質疑「不合理」）是這次破解的關鍵。長期解法是把這個質疑內化為 skill / 絆腳索，不依賴外部觸發。
