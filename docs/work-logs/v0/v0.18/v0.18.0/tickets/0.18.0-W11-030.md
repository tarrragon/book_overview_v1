## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`.claude/settings.local.json` 第 199-209 行對 `PreToolUse:Agent` 重複註冊 `agent-dispatch-validation-hook.py`，且註冊命令使用 `python3 $CLAUDE_PROJECT_DIR/...` 形式強制走系統 python 解譯器，繞過 hook 檔案 shebang `#!/usr/bin/env -S uv run --quiet --script` 與 pep723 deps（`pyyaml`）。系統 python 未安裝 pyyaml → `framework_paths.py:36` top-level `import yaml` 失敗 → ModuleNotFoundError 顯示為 PreToolUse:Agent hook traceback；同時 settings.json 註冊路徑走 shebang 成功，導致 hook log 顯示 INFO 放行紀錄（詳見下方 append-log）。

### 影響範圍

- `.claude/hooks/agent-dispatch-validation-hook.py`（受影響的 hook 進入點）
- `.claude/hooks/lib/framework_paths.py`（top-level import yaml 為 trigger point；本身非根因）
- `.claude/settings.local.json` 第 199-209 行（重複註冊根因位置）
- 用戶體驗：每次派發 Agent 都見 hook error traceback（雖 non-blocking），違反 quality-baseline 規則 4「Hook 失敗必須可見」的反向精神（不應產生誤導性失敗訊號）

### 相關 Error Pattern

- **PC-124**（subagent pytest 通過 ≠ hook subprocess 環境通過）：本案非此模式，pep723 deps 已正確宣告
- **PC-135**（lib refactor caller sync）：本案非此模式，pyyaml 已在 caller pep723 宣告
- **新模式**：Hook 雙重註冊 shebang vs python3 直呼不一致 → 由 0.18.0-W11-030.2 (DOC) 建立 PC- 文件

### 實驗方法

1. 直接 python3 執行 hook（模擬 settings.local.json 觸發路徑）

### 實驗執行

**步驟 1（python3 直呼）**：

```
echo '{"tool_name":"Agent","tool_input":{"subagent_type":"thyme","prompt":"x"},...}' \
  | python3 .claude/hooks/agent-dispatch-validation-hook.py
```

輸出：
```
Traceback (most recent call last):
  File ".claude/hooks/agent-dispatch-validation-hook.py", line 45, in <module>
    from lib.framework_paths import get_categories as _get_framework_categories
  File ".claude/hooks/lib/framework_paths.py", line 36, in <module>
    import yaml
ModuleNotFoundError: No module named 'yaml'
exit=1
```

**步驟 2（uv run --script）**：

```
echo '{...same...}' | uv run --quiet --script .claude/hooks/agent-dispatch-validation-hook.py
```

輸出：exit=0（hook 正常放行，hook log 寫入 INFO）

**步驟 3（兩處註冊形式對照）**：

| 檔案 | 行 | 註冊命令 | 觸發方式 |
|------|---|---------|---------|
| `.claude/settings.json` | 218 | `$CLAUDE_PROJECT_DIR/.claude/hooks/agent-dispatch-validation-hook.py` | 走 shebang `#!/usr/bin/env -S uv run --quiet --script` → uv pep723 deps `pyyaml` 自動安裝 → 成功 |
| `.claude/settings.local.json` | 205 | `python3 $CLAUDE_PROJECT_DIR/.claude/hooks/agent-dispatch-validation-hook.py` | 強制 python3 直呼，**繞過 shebang 與 pep723 deps** → 系統 python 無 pyyaml → 失敗 |

**步驟 4（framework_paths top-level import）**：

`.claude/hooks/lib/framework_paths.py:36` — `import yaml` 為模組級 top-level import。導致 `from lib.framework_paths import ...` 必觸發 yaml 載入，無延遲機會。

**步驟 5（caller 統計）**：

```
grep -rln "from lib.framework_paths" .claude/hooks/
```

三個 caller：
- `agent-dispatch-validation-hook.py`（settings.json + settings.local.json 雙註冊）
- `layer-boundary-validator-hook.py`（僅 settings.json 註冊）
- `commit-msg-layer2-marker-check-hook.py`（僅 settings.json 註冊）

僅 `agent-dispatch-validation-hook` 在 settings.local.json 出現重複註冊（python3 形式），其他 caller 無此問題。

### 實驗發現

**已驗證事實**：

1. **Q1 解答**：hook log INFO 與 stderr traceback 同時出現的原因是 hook 被註冊兩次：settings.json（uv 走 shebang，成功寫 log）+ settings.local.json（python3 直呼，失敗吐 stderr）。Claude Code 對同一 event matcher 的多 hook 命令逐一執行，互不關聯。
2. **Q2 解答**：pep723 deps 已正確宣告 pyyaml，但 settings.local.json 註冊命令以 `python3` 開頭強制走系統解譯器，繞過 shebang，pep723 失效。**非 PC-124（uv ephemeral cache miss）也非 PC-135（caller deps 漏宣告）**，而是「caller 觸發方式繞過 pep723 機制」的全新模式。
3. **Q3 解答**：framework_paths.py top-level `import yaml` 本身合理（uv 路徑下正常），但若想對 non-uv caller 容錯，可考慮 try/except + 延遲 import；但**真正根因是 settings.local.json 重複註冊**，延遲 import 只是把錯誤位置往後移（_load_config 仍會炸）。

**未驗證假設**（已排除）：

- ~~uv ephemeral env cache miss~~：步驟 2 多次跑 uv 均成功，無 cache miss 現象。
- ~~lib 共用模組漏宣告 transitive deps~~：pyyaml 已在 caller pep723 deps 宣告，問題非 transitive。

**結論**：屬全新模式「**Hook 雙重註冊：shebang 路徑 vs python3 直呼路徑不一致**」。應建 PC error-pattern 並修復 settings.local.json 重複註冊。

### WRAP 決策

**W (Widen options)** — 候選修正方向：

| 方案 | 描述 | 優劣 |
|------|------|------|
| A. 移除 settings.local.json 重複註冊 | 直接刪除第 199-209 行 PreToolUse Agent block | 根因修復；風險：若該行是用戶有意覆寫 settings.json 行為（如測試）會丟失意圖 |

**R (Reality-test)** — 證據對照：

- 步驟 5 證據：settings.local.json 僅 agent-dispatch-validation 出現 python3 重複註冊，其他兩個 framework_paths 消費者無此問題 → 重複註冊是歷史遺留（commit e5071559 2026-03-30 「同步 .claude 配置變更」）

- 用戶未表達需保留此覆寫意圖，且註冊形式（python3 直呼）違反專案 hook 標準（shebang + pep723）

**A (Attain distance)** — 機會成本：

- 方案 A 成本：1 行刪除；後續若 settings.json 有同事修改不會被 settings.local.json 干擾
- 方案 C 單獨成本：增加 framework_paths.py 防禦邏輯，但根因未解，未來其他 non-uv caller 仍會踩雷
- 方案 D 邊際成本：A 已根治，C 屬可選防護；CC 環境下無第二個 non-uv caller 預期出現

**P (Prepare to be wrong)** — 反向假設：

- 若 settings.local.json 該 block 是某機制（如 CC plugin / 用戶 local override）所需，刪除會破壞。但 git blame 顯示為 3/30 配置同步，無功能性註解 → 風險低
- 若未來需支援 non-uv 環境（如純 python3 系統 hook），方案 C 可作後備；現階段不必過度設計

**決策：採方案 A（移除重複註冊）**，C 留作 follow-up 防禦選項但本 ANA 不強制。
