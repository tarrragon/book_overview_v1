### WRAP 三問（PM 派發前評估）

**W (Widen) — 候選方案**:
1. **短路退出 + sort by mtime desc**（ticket strategy 主選）：直接掃 ticket 檔案、依 mtime 排序，遇第一個 in_progress 即 return。實作成本低，避免全掃。
2. **改用 `ticket track list --status in_progress` CLI**（ticket strategy 替代）：呼叫已有 CLI 取得 in_progress 清單。實作最簡，但多一次子進程開銷（fork+解析）。
3. **快取結果**：在 hook 內以 mtime 為 key 緩存上次結果。複雜度高、記憶體常駐風險、不適用於 short-lived hook。

**選 1**：CLI 開銷可能 > 直接掃檔（hook 對延遲敏感），且短路退出可在 hot path 達到 << 100ms 目標。

**A (Attain distance) — 機會成本**:
擠壓 W14-020（ANA 規則 6 補強）與 W17-008（group 清理 11 項）。但 W11-019 是 hot path 性能修復（每次 UserPromptSubmit 多 1052ms），影響開發體驗，優先序合理。

**Why**: P1 性能修復影響每次互動延遲；ANA/group 為品質提升任務，可後續補做。

**P (Prepare to be wrong) — 最可能失敗**:
F1 巢狀降到 ≤3 時拆分 try-except 後，錯誤捕捉邊界改變，導致 SA guard 對某類異常的 fallback 行為與重構前不同（違反 acceptance 第三條「SA guard type/phase 過濾行為不變」）。

**防護**: agent 必須跑現有 hook 相關測試（若有）；無測試則需手動驗證 SA guard 對 IMP type 與 Phase 4 的過濾行為（依 W11-004.2 ticket 原意）；commit 前比對 grep "in_progress\|SA guard\|except" 變更前後語意一致性。

### 修改摘要

`get_active_in_progress_ticket` 改寫為短路退出 + mtime desc 排序：

1. 取 repo root → 列出 `docs/work-logs/**/tickets/*.md`，依 file mtime 由新到舊排序
2. 逐一解析 frontmatter，遇第一個 `status=in_progress` 即 return（不再全掃 + 排序）
3. 單檔解析失敗回 None（透過抽出的 `_parse_ticket_frontmatter` helper），不影響其他 ticket 掃描

### 巢狀深度（F1）

抽出 `_parse_ticket_frontmatter` helper 後，`get_active_in_progress_ticket` 巢狀深度從 5（try → for → try → if → append）降至 2（try → for），helper 自身深度 1。

### 排序鍵變更說明

由 `started_at` (frontmatter 字串) 改為 `file mtime`。對 SA guard 用途（識別「當前進行中的 ticket」）mtime 是更好 proxy——claim/append-log/complete 都會更新檔案，反映實際活躍程度。一般情境同時間僅 1 個 in_progress ticket，兩種排序鍵結果一致。Ticket how.strategy 明示採用此方案。

### 行為保留驗證

- `_should_silence_sa_review`: 6 case (None / DOC / ANA / IMP+Phase 4 / IMP+phase3b / IMP+None) 全通過
- `detect_skip_intent`: 5 case (4 skip 模式 + 1 普通對話) 全通過
- E2E: hook stdin/stdout JSON 結構正確
