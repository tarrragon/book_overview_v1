## Task Summary

評估 ticket track snapshot/handoff-ready/checkpoint-status 三命令是否合併為 snapshot --check flag (linux L1)

---

## Problem Analysis

### WRAP ANA (PM 前台, 2026-04-19)

#### R (Reality Test) — 重現實驗

**接受的根因假設**：三命令分裂導致 active_handoff 在 snapshot vs handoff-ready 中被迫承擔兩種解釋（linux L1 + lavender §2 共同記錄）。

**最小重現**（讀規格不需執行程式碼）：
- snapshot 情境（lavender §2）：active_handoff is None → ready；is not None → 「先處理 pending handoff」（**阻擋 Go**）
- handoff-ready 情境（lavender §2）：active_handoff is not None → 「視為通過項」（**Go 訊號**）

**已驗證事實**：
- lavender 規格 §2 確實記錄此語意分裂
- 017.8 lib/checkpoint_state.py PRIORITIES 用同一 state，CLI 層詮釋分歧

**未驗證假設**：
- 合併為 --check flag 是否真能消除語意分歧？（仍可能在 flag 不同值間出現相同分歧）
- PM 實際使用三個動詞 vs 一個 flag，認知負擔差距是否顯著？

#### W (Widen) — 候選方案

| # | 方案 | 描述 | 影響範圍 |
|---|------|------|----------|
| A | 維持三命令 | lavender 規格現狀 + §2 雙語意明示文件化 | 0（不改） |
| B | snapshot --check=handoff\|clear\|none | linux L1 提案，三命令合併為一 | 大（CLI 重設計 + invalidate 017.8 metrics caller 維度） |
| C | snapshot 主入口 + 兩 alias | snapshot 為主，handoff-ready/checkpoint-status 為 subset | 中（雙重維護更糟） |
| D | lib 拆 active_handoff_blocking / _present | 雙欄位消除語意分歧 | 大（修改 017.8 已固定介面，違反邊界） |
| E | 維持三命令 + 規格化雙語意 | 在 lib 層建立 view function `handoff_status_for(caller)` 統一語意路由 | 小（lib 加 view function，CLI 純調用） |

#### A (Attain distance) — 機會成本

| 方案 | 短期成本 | 中期維護 | 風險 |
|------|---------|---------|------|
| A | 0 | 雙語意持續，新進者困惑 | 低 |
| B | 大 (重設計 + metrics) | PM 認知負擔 1/3 | 中（flag 命名爭議） |
| C | 中 | alias 雙重維護 | 中 |
| D | 大 | 違反「不改 lib」邊界 | 高 |
| E | 小 (新增 view function) | 雙語意但有單一 source of truth | 低 |

執行此 ANA 擠壓對象：017.1 Phase 1 重規格（lavender 等待中）、011.1 文件原則、其他 W10 待辦。

#### P (Prepare to be wrong) — 12h 失敗預判

| 方案 | 最可能失敗 | 防護 |
|------|----------|------|
| A | 雙語意持續成為 PM 累積認知負擔 | 後續 6 個月 PM 使用回顧時主動評估 |
| B | --check=handoff 不直覺，PM 要求回滾 | 先做 PM 試用 mockup 再決定 |
| E | view function 名稱選不好，CLI 層仍各自詮釋 | 命名同 lib `format_phase_label` 風格 |

#### 決策

**選 E**：lib 新增 `handoff_status_for(state, caller: str) -> tuple[bool, str]` view function（caller in {"snapshot", "handoff-ready"}），返回 (is_blocking_or_signal, message)。CLI 三命令調用此 view，**不自行詮釋 active_handoff**。

**理由**：
1. **最低風險**：不違反「不改 lib」邊界（新增 view function 屬擴增，非修改既有）
2. **保留 W10-017 WRAP 結論方向**：三命令架構不動，避免回溯 017.8 metrics 設計
3. **解決真實問題**：active_handoff 語意路由集中在 lib，消除 CLI 層詮釋分歧
4. **不阻擋 017.1 Phase 2**：lavender 重規格時加入「lib 新增 view function」一條，sage 設計測試覆蓋三 caller 路徑

**B 方案延後條件**：若 PM 使用 6 個月後仍覺得三命令認知負擔過高，啟動 B 方案重構（屆時 017.8 metrics caller 維度合併亦較成熟）。
