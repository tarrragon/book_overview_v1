### 問題根因

`commands/lifecycle.py` 的 `TicketLifecycle.claim` / `complete` / `release` / `close` 四個方法皆為典型 `load → modify → save` 三步序列，期間不持任何 file-level lock。W14-005 重現實驗已證實同類型 pattern（ticket_builder.update_*）在並發下 100% 觸發 lost-update（lost rate 55.6%~71.9%）。本檔受 PM CLI、Hook（如 acceptance-gate-hook）、subagent worktree 多源同時觸發，存在實際 race 風險。

### 影響範圍

- `commands/lifecycle.py`：
  - `claim` L417-477：`load_and_validate_ticket` → 修改 status/assigned/started_at → `save_ticket`
  - `complete` L594-734：`load_and_validate_ticket` → 多項驗證（含互動 prompt）→ 修改 status/completed_at/_body → `save_ticket`
  - `release` L780-813：`load_and_validate_ticket` → 修改 status/assigned/started_at=None → `save_ticket`
  - `close` L819-895：`load_and_validate_ticket` → 修改 status/closed_at/closed_by/close_reason → `save_ticket`
- 不在本 ticket 範圍：
  - `_cascade_unblock_children` L1707（complete 後對 children 的 save L1784）——操作目標為**不同 ticket file**（children），race domain 與 parent 解耦，屬獨立 race 區；W14-042.1.4 audit 已歸類為「cascade-level 並發」，由後續 ticket（或併入 W14-045 批量保護範圍）處理。本 ticket 僅保護 parent ticket 自身的 4 處狀態變更。

### 相關 Error Pattern

- W14-005 race-condition baseline / W14-042 ticket_builder 修復前例
- IMP-003（修 bug 時 scope 變更 → 必須檢查所有 caller）
- `lib/file_lock.py` 巢狀禁令：同一 `with file_lock(p)` 內不可再次 `file_lock(p)`

### 設計決策

1. **lock 目標路徑取得**：採用 `Path(get_ticket_path(self.version, ticket_id))` 作為 lock target，與 `_append_unique_to_list_field` 一致；不依賴 ticket 載入後的 `_path`（lock 必須在 load 前取得）。
2. **lock 範圍**：包圍從 `load_and_validate_ticket` 到 `save_ticket` 之整段；validation 失敗（early return）時 `with` 自動釋鎖，無洩漏。
3. **complete 內互動 prompt 的鎖持有**：`_handle_ana_spawned_confirmation` 在互動環境會 `input()` 阻塞；持鎖期間阻塞屬可接受設計（complete 本身互斥，且 prompt 是人為操作不應並發）。非互動環境（CI / hook）走 yes_spawned 或直接 return 2，鎖持有時間 < 1 秒。
4. **巢狀檢查**：4 個方法彼此不互相呼叫；`save_ticket` 自身不加 lock（file_lock.py 設計）；`_post_complete_cascade` 在 parent save 後執行且操作不同檔——無巢狀風險。
5. **save_ticket 路徑**：lock 後仍以 `resolve_ticket_path(ticket, ...)`（讀 `_path`）作為實際 save 目標，與既有 `_append_unique_to_list_field` 一致。lock target 與 save target 可能略有差異（_path vs get_ticket_path），但 file_lock 用獨立 `.lock` sidecar，差異容忍。

### 修復摘要

在 `commands/lifecycle.py` 的 `TicketLifecycle.{claim, complete, release, close}` 4 個方法中，以 `with file_lock(lock_target):` 包圍從 `load_and_validate_ticket` 到 `save_ticket` 的完整序列。

### 變更內容

1. **import**：新增 `from ticket_system.lib.file_lock import file_lock`
2. **claim** (約 L417-487)：lock 包圍 load → validate → modify status/assigned/started_at → save
3. **complete** (約 L594-746)：lock 包圍 load → 多階段 validation（含 schema check / spawned confirmation / pending children block）→ modify status/completed_at/_body → save；cascade unblock 與 worklog/handoff 保持在 lock 外（操作不同 ticket，無巢狀）
4. **release** (約 L792-829)：lock 包圍 load → validate (only in_progress) → modify status=blocked → save
5. **close** (約 L834-912)：lock 包圍 load → validate (status != closed) → modify status/closed_at/closed_by/close_reason → save

### 關鍵設計

- **Lock target 統一用 `Path(get_ticket_path(self.version, ticket_id))`**：與 `lib/ticket_builder._append_unique_to_list_field` 一致。Lock target 在 load 前取得，避免依賴 ticket dict 的 `_path`。
- **早期 return 自動釋鎖**：validation 失敗的 early return 在 `with` 區塊內，context manager 保證 fcntl.LOCK_UN。
- **互動 prompt 持鎖**：`complete` 內 `_handle_ana_spawned_confirmation` 的互動 `input()` 在 lock 內，屬可接受設計（同 ticket 的 complete 操作本應互斥；非互動 CI / hook 走 yes_spawned / return-2 不阻塞）。
- **巢狀檢查**：
  - 4 個方法彼此無互呼叫；
  - `save_ticket` 自身不加 lock（file_lock.py 設計）；
  - `_post_complete_cascade → _cascade_unblock_children` 對 children 的 save 在 parent lock 釋放後執行，且操作不同檔——無巢狀；
  - W14-042.1.4 audit 已將 cascade 列為獨立 race 區（不在本 ticket 範圍，由 W14-045 或 follow-up 處理）。

### 不在本 ticket 範圍

- `_cascade_unblock_children` L1784 對 children 的 save：操作不同 ticket file，與 parent race domain 解耦。
- `claim_with_verification`：本方法在 verification 通過後呼叫 `self.claim`，claim 內已加鎖；verification 本身僅讀取不寫入。
- `fields.py / track_acceptance.py / track_relations.py / track_set_acceptance.py`：W14-045 範圍。
