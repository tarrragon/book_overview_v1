## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### Bay 獨立技術品質審計：RAG 提案技術風險評估

#### 一、關鍵技術風險清單

| # | 風險 | Severity | 說明 |
|---|------|----------|------|
| R1 | 中英混雜 embedding 品質 | HIGH | 本專案文件約 70% 繁體中文 + 30% 英文技術術語混合。通用 embedding model（OpenAI / sentence-transformers）對 CJK 混雜文本的語意相似度準確率顯著低於純英文場景。實測常見：「skip-gate」與「測試跳過防護」語意等價但 cosine similarity < 0.5。 |
| R2 | Stale index / 索引一致性 | HIGH | 265+ error-patterns + 63 rules 持續增長。每次 commit 可能新增/修改/刪除檔案，embedding index 必須同步更新。任何 stale entry 導致 retrieve 錯誤文件或漏 retrieve 新文件。需建立 re-index pipeline（git hook 或 CI），引入額外基礎設施維護負擔。 |
| R3 | Rules RAG 漏 retrieve 違規風險 | CRITICAL | Rules RAG 失敗成本不對稱：漏 retrieve 一條 rule = PM 違規。以 95% 命中率要求計算，63 條 rules 中每次查詢允許漏 3 條。但品質基線規則（如規則 1 測試通過率 100%）漏 retrieve 一次即可能導致測試紅燈進 main。Embedding 語意匹配無法保證確定性覆蓋——這是 RAG 與 FSM/rule-based 系統的根本差異。 |
| R4 | Embedding model 漂移 | MEDIUM | 本地 model（如 all-MiniLM-L6-v2）版本升級後 embedding 空間改變，歷史 index 失效需全量 re-embed。API model（OpenAI）版本退役後同理。5 年週期內至少經歷 2-3 次 model 世代交替。 |
| R5 | 跨 rule 引用斷裂 | MEDIUM | 現有規則系統大量使用顯式路徑引用（`@.claude/rules/core/quality-baseline.md`）。RAG 按語意 retrieve 無法保證引用鏈完整性——retrieve 到規則 A 但漏掉其依賴的規則 B，PM 執行時缺少前置條件。 |

### 二、雙提案 Verdict

**提案 A（Ticket/PC RAG）：有條件支持，建議先實驗**

理由：失敗成本低（漏 retrieve = 少個歷史參考），265+ error-patterns 確實超過人工 grep 有效管理範圍。但需先驗證 R1（中英混雜 embedding 品質）是否達 60% 以上命中率。建議以 30 筆 PC pattern 做 retrieval 精度實驗後再決策。

**提案 B（Rules/PM-Rules RAG）：反對**

理由：R3 風險不可接受。Rules 是確定性約束（必須 100% 覆蓋），RAG 是機率性檢索（無法保證覆蓋）。將確定性需求交給機率性系統是架構層級的類型錯誤。既有三層機制（L1 全載入核心 rules + L2 lazy-load references + L3 hook-driven 情境載入）已覆蓋此需求，且 107 個 hook 已部分實現情境觸發載入。強化 L3（擴展 hook trigger map）的確定性收益遠高於 L4 RAG 的不確定性收益。

### 三、5 年維護曲線對比

| 年份 | RAG 系統維護成本 | 強化 grep + lazy-load |
|------|----------------|---------------------|
| Y0 | 高（建置 embedding pipeline、vector DB、re-index hook） | 低（擴展現有 hook trigger） |
| Y1 | 中（維護 index 一致性、調整 retrieval 閾值） | 低（新增 rule 時更新 trigger map） |
| Y2-3 | 高峰（embedding model 世代交替、全量 re-embed） | 穩定低 |
| Y4-5 | 中（持續適應 model 變化、vector DB 升級） | 穩定低 |

RAG 的維護曲線呈鋸齒形（model 交替期飆升），grep + hook 呈平坦線。

### 四、安全性 / 隱私評估

- **本地 embedding model**：無隱私風險，但需 ~500MB 儲存 + 初次載入延遲
- **API embedding model**：規則文件含專案內部流程，送外部 API 有資訊洩漏風險。且 API 依賴引入網路故障單點
- **Verdict**：若實施提案 A，僅限本地 model

### 五、既有資產覆蓋率評估

| 需求場景 | 既有機制 | 覆蓋率 |
|---------|---------|--------|
| PM 決策路由（claim/complete/dispatch） | decision-tree.md FSM + hook 強制 | 95% |
| Error pattern 查詢 | `/error-pattern query` CLI + grep | 70%（超過 200 筆時人工選擇困難） |
| Rules 情境載入 | auto-load core + lazy-load ref + hook trigger | 85% |
| 歷史 ticket 參考 | `ticket track query` + source_ticket 鏈 | 60%（跨版本搜尋較弱） |
| Context Bundle 組裝 | claim 自動抽取 + PM 手動補充 | 80% |

**綜合覆蓋率**：既有機制覆蓋 ~78% 需求。缺口主要在跨版本 ticket 搜尋（提案 A 甜蜜點）和大量 error-pattern 語意檢索。Rules 載入覆蓋率 85% 可透過擴展 hook trigger 提升至 95%+，無需 RAG。
