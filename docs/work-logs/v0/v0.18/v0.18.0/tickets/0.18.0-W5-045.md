## Problem Analysis

### 問題根因

W5-001 Phase 2 派發 sage 越界事件根因之一：PM 派發 prompt 中含「實作」「建立 .py」等字串時，agent-dispatch-validation Hook 僅檢查派發位置（.claude/ vs worktree），不檢查 prompt 內容是否違反 agent 的「禁止行為」宣告。

W5-043 標準化結構後，agent 檔皆含「## 禁止行為」區塊，提供穩定解析錨點。W5-045 在此基礎上增加關鍵字衝突掃描防線。

### 影響範圍

- `.claude/hooks/agent-dispatch-validation-hook.py` — 新增 parser + matcher 函式
- `.claude/hooks/tests/test_agent_dispatch_validation_hook.py` — 新增至少 3 個單元測試
- 不影響既有 worktree/staging 驗證邏輯（新增分支，非重構）

### 相關 Error Pattern

- W5-001 派發 sage 越界（根因 A — agent 職責邊界模糊；W5-043 已封堵結構面）
- PC-086 Hook exec bit（新增 Hook 不需，僅擴充既有檔）

### 獨立驗證（PC-007）

## Solution

### Phase 1 Context Bundle（PM 前台規劃，2026-04-18）

#### 禁止行為 section 格式觀察（樣本掃描 4 agents）

| 格式類型 | 範例 agent | 結構 |
|---------|-----------|------|
| A. 階層式 | sage, saffron, lavender | `## 禁止行為` → `### 絕對禁止` → 編號清單 + `**禁止XX**` |
| B. 扁平式 | acceptance-auditor | `## 禁止行為` → 編號清單 + `**禁止XX**` |

共通錨點：每條禁止項皆以 `**禁止XX**` 粗體開頭（標籤化，易 regex 抽取）。

#### Parser 設計（建議方案）

**輸入**：agent .md 檔路徑
**輸出**：List[str]，每條為從 `**禁止` 到該行結尾的純文字 pattern

**策略**：
1. 讀檔，定位 `## 禁止行為` 開頭、下一個 `## ` 結尾（或 EOF）
2. 在該區段內 regex 匹配 `\*\*禁止([^*]+)\*\*` 的 capture group
3. 結果為該 agent 的 prohibited_actions list

**範例輸出（sage-test-architect）**：
- "實作程式碼"
- "設計功能規格"
- "直接執行測試修復"

#### Matcher 設計（建議方案）

**策略：關鍵字映射表（W5-045 初版）**

從 prohibited action 字串萃取關鍵字，對 prompt 做子字串/regex 匹配：

| prohibited action 關鍵字 | prompt 衝突觸發 pattern |
|------------------------|----------------------|
| "實作" / "實作程式碼" | `實作`、`編寫.*程式碼`、`寫入.*\.py`、`Write.*\.py` |
| "修改檔案" | `Edit.*\.py`、`修改.*\.py`、`修正.*\.py` |
| "git commit" | `git commit`、`提交.*commit` |
| "設計功能規格" | `設計.*規格`、`撰寫.*規格` |
| "直接執行測試修復" | `修復.*測試`、`fix.*test` |

**實作建議**：定義 `FORBIDDEN_KEYWORD_MAP: Dict[str, List[re.Pattern]]`，作為第一版關鍵字清單；未來可擴充。

**偵測邏輯**：
1. 對每個 agent 的 prohibited_actions，查 FORBIDDEN_KEYWORD_MAP 取 regex patterns
2. 對 prompt 逐 pattern 匹配
3. 有匹配 → 收集衝突項

#### Warn vs Block 策略（建議）

| 階段 | exitcode | 行為 | 適用條件 |
|------|---------|------|---------|
| 初版 (W5-045) | 0 | stderr warn | 任何衝突都先 warn，避免阻斷合法工作 |
| 後續 (可選) | 2 | block | 若日後發現特定高信心 pattern 誤報率 < 5%，可升級 block |

**理由**：避免第一版過於激進阻斷 PM 工作；warn 先累積 pattern 資料，再決定升級。

#### 實作步驟建議

1. 在 hook.py 新增 `_extract_prohibited_actions(agent_md_path: Path) -> List[str]`
2. 定義模組級 `FORBIDDEN_KEYWORD_MAP` 常數
3. 新增 `_detect_keyword_conflicts(prompt: str, prohibited_actions: List[str]) -> List[str]`
4. 在主流程中（通過位置驗證後）呼叫，衝突時 stderr warn + exitcode 0
5. 測試：至少 3 case
   - AC-T1: sage agent + 含「實作」prompt → warn
   - AC-T2: linux agent + 含「git commit」prompt → warn
   - AC-T3: 正常 prompt（無衝突） → pass

#### 已知限制（Prepare to be wrong）

1. **中英文混用**：prohibited action 是中文（"實作"），但 prompt 可能用英文「implement」→ 映射表需雙語
2. **語意近似**：「修改」vs「修正」vs「更新」→ 初版用寬泛 regex 接受誤報
3. **正當覆蓋**：用戶明確授權 override（例如 "已確認例外，請執行 X"）→ 初版不處理，warn-only 不阻斷

### 簡化 WRAP 三問

- **Widen**：候選方案比較
  1. 關鍵字映射表（建議）— 可控、可擴充
  2. LLM semantic match — 昂貴、PreToolUse 不可同步呼叫
  3. 全 prompt regex scan — 誤報率高，無 agent 脈絡
- **Attain distance**：機會成本
  - 此任務排擠 v0.18.0 發布檢查（若延後至 v0.19.0 則發布可加速）；用戶選擇繼續本 session 表示優先度高
- **Prepare to be wrong**：最可能失敗
  - FORBIDDEN_KEYWORD_MAP 覆蓋不足 → 初版 3 組衝突 case 驗證，後續依實際派發 log 擴充

### Phase 3a: 策略

Parser：以 `^## 禁止行為$` 至下一個 `^## ` 或 EOF 定位區塊，regex 抽取 `\*\*禁止([^\*\n]+?)\*\*` capture group。
Matcher：FORBIDDEN_KEYWORD_MAP 將 prohibited action 關鍵字（如「實作」「git commit」）映射為 prompt 偵測 regex 清單；prohibited action 若「包含」某 key 則套用該組 patterns。
整合：`_emit_keyword_conflict_warning_if_any` 在 main() 中於 IMPLEMENTATION_AGENTS gate 之前執行（W5-001 sage 越界案例中 sage 非實作代理人，必須放在 gate 前才會觸發）。

### Phase 3b: 實作

- hook.py 新增 ~140 行（parser + map + matcher + warning helper）
- FORBIDDEN_KEYWORD_MAP 覆蓋 6 組關鍵字（實作、修改檔案、git commit、設計功能規格、直接執行測試修復、執行測試）含中英文 pattern
- Warn-only 策略（exitcode 0 + stderr）：符合 Solution Context Bundle「初版累積資料後再評估升級 block」
