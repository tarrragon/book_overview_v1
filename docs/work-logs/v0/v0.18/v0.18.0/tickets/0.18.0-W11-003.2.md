# Execution Log

## Task Summary

補強 ticket CLI complete 父 ticket 時檢查子 ticket 狀態防呆

---

## Problem Analysis

### 問題根因

ticket CLI 的 `track complete` 命令未檢查子 ticket 狀態，導致 PM 可能在父 ticket 有未完成子任務時錯誤標記父 ticket 為 completed，造成狀態樹不一致和工作流斷裂。

### 影響範圍

`.claude/skills/ticket/scripts/` 下 `track complete` 相關實作。間接影響 PM 的 AskUserQuestion 推薦邏輯（PM 在無 CLI 防呆時可能推薦錯誤的 complete 選項）。

### 相關 Error Pattern

需新建 PC 類 error-pattern：「PM 在父子 ticket 鏈進行中推薦父 ticket 標 completed」。

---

### 再次發生案例：W10-049 分析任務（2026-04-15）

**事件描述**：PM 在 W10-049（ANA 分析 Ticket）建立完 4 個子任務（W10-049.1~.4）並 commit 後，於 AskUserQuestion 中將「W10-049 標為 completed (Recommended)」列為推薦選項。

**錯誤性質**：
- 分析任務的「分析工作完成」不等於「父 ticket 可關閉」——延伸出子任務後，父任務必須等子任務全部完成才能 complete
- PM 把「驗收 4 項已達標」誤判為「可 completed」，但驗收條件達標只表示分析工作完成，子任務狀態才決定父 ticket 可否關閉
- 用戶介入糾正：「保持 pending 等子任務執行」

**根本原因（重申 W10-022 必要性）**：
1. **CLI 缺乏防呆**：track complete 命令未檢查 children 欄位的子 ticket 狀態
2. **PM 認知缺陷**：分析任務和後續實作被視為斷點而非連貫工作流
3. **AskUserQuestion 推薦邏輯**：PM 生成選項時未考慮父子鏈狀態，將 completed 列為推薦

**驗證 W10-022 設計的正確性**：本案例證明 acceptance 中「實作子 ticket 狀態檢查邏輯」「有未完成子 ticket 時 complete 需 --force 旗標」是必須的——若 CLI 有此防護，PM 即使誤推薦 completed，用戶或 PM 也會在 CLI 回應中得到警告。

**擴充建議（給執行 agent）**：除了 CLI 層防呆，可考慮在 `.claude/skills/manager` 或 PM 規則中新增條款：「父 ticket 有未完成子任務時，AskUserQuestion 不可將 completed/handoff 等切斷工作流的選項列為 Recommended」。但此擴充屬 PM 行為層，非本 ticket 核心；可在執行 W10-022 時評估是否需 spawn 新 ticket。

### 修改摘要

在 `ticket track complete` 流程新增 **Step 3.7：pending children blocking**，置於 Step 3.6（ANA spawned confirmation）之後、Step 4（執行完成操作）之前，並提供 `--force` 旁路。

### 變更檔案

| 檔案 | 變更 |
|------|------|
| `.claude/skills/ticket/ticket_system/commands/lifecycle.py` | `TicketLifecycle.complete()` 新增 `force` 參數；Step 3.7 呼叫 `_handle_pending_children_block`；新增 helper `_collect_pending_children` + `_handle_pending_children_block`；`execute_complete` 從 `args.force` 取值傳入 |
| `.claude/skills/ticket/ticket_system/commands/track.py` | `p_complete` 新增 `--force` flag |
| `.claude/skills/ticket/tests/test_track_lifecycle.py` | 新增 `TestCompletePendingChildrenBlocking`（3 場景）+ 更新 W11-002.5 contract 測試容納 Step 3.7 額外的 `list_tickets` 呼叫 |
| `.claude/skills/ticket/SKILL.md` | 常用範例加 `--force` 用法 + complete 子命令說明補上 W11-003.2 行為 |

### 邏輯設計

- `_collect_pending_children(children_ids, version)`：透過 `list_tickets` 取得 ticket_map，回傳非 terminal（status 不在 `TERMINAL_STATUSES={completed, closed}`）的 `(id, status)` 清單。
- `_handle_pending_children_block(ticket, version, force) -> Optional[int]`：
  - children 空 / 全 terminal → `None`（通過）
  - 含非 terminal + `force=False` → stderr 列出 + 引導 `--force`，回傳 1
  - 含非 terminal + `force=True` → stderr 警告但繼續，回傳 `None`

### W5-019 cascade 保護

Step 3.7 通過（含 `--force` 路徑）才會進入 Step 4，後續 `_post_complete_cascade`（line 762）照舊執行。`--force` 旁路場景下，未完成 children 若為 `blocked` 且 blockedBy 全解除，仍會被 cascade 解鎖為 `pending`。

### W11-002.5 contract 測試調整

原測試斷言 `parent_save_idx < first_list_idx`。Step 3.7 在 parent save 前呼叫 `list_tickets` 做 pending children 檢查，導致該斷言過嚴。修正為斷言：parent save 後仍有至少一次 `list_tickets`（即 cascade 用的那次）。W11-002.5 真正契約（cascade 載入 ticket_map 時 parent 已 completed in disk）仍被保留。
