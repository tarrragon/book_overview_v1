#### 五、Hook Trigger 機制設計（概念層，非實作）

**核心原則**：在 AUQ hook 觸發前插入 deterministic 過濾層。

```
PM 準備觸發 AUQ
    |
    v
[Deterministic Filter] 查詢系統狀態
    |
    +-- 場景在 D1-D10 清單
    |   +-- 系統狀態可推斷 → 自動執行推斷結果，skip AUQ
    |   +-- 系統狀態不可推斷（異常/缺資料）→ fallback 到 AUQ
    |
    +-- 場景不在 D1-D10 清單 → 正常觸發 AUQ
```

**具體機制**：

| 層次 | 機制 | 說明 |
|------|------|------|
| 規則層 | decision-tree.md 新增「deterministic bypass」路由 | 列出 D1-D10 場景及其自動執行邏輯 |
| Hook 層 | 既有 AUQ hooks 增加前置 guard | 對 D1-D10 場景，hook 先查系統狀態，可推斷時輸出「[AUTO] 已自動執行：{動作}」而非提醒 AUQ |
| Fallback 層 | 任何 deterministic 推斷失敗時降級為 AUQ | 確保不會因自動化而跳過真正需要用戶判斷的場景 |

**預估效果**：

- 觸發頻率降低：10/18 場景（D1-D10）bypass → **-55% 觸發頻率**（與 W10-136 預估 -30~40% 吻合，偏樂觀因含邊界場景）
- 保守估計（僅高信心度場景 D1-D9，排除 D10）：9/18 → **-50% 觸發頻率**

#### 六、強制自問段：我可能錯在哪裡

1. **過度自信於 scheduler 排序**：D2（優先級）和 D6（Handoff 方向）假設 `runqueue` 排序完全對齊用戶意圖。但用戶可能有系統外資訊（如「今天精力差想做簡單的」），scheduler 無法捕捉。**緩解**：加入用戶偏好覆寫機制（如 `--override-next`），但預設走 scheduler。

2. **認知負擔閾值不是二元的**：D3（任務拆分）假設 > 10 強制拆分、≤ 10 不拆。實際上 7-10 區間可能需要用戶判斷拆分粒度。**緩解**：僅 > 10 和 ≤ 5 自動化，6-10 仍 AUQ。

3. **#16 錯誤學習的 commit type 過濾可能漏掉重要學習**：docs: commit 也可能含架構教訓。**緩解**：除 commit type 外也檢查 diff 是否含 error-pattern 目錄變更。

4. **並行/序列判斷（D4/D5）假設檔案重疊是唯一衝突維度**：邏輯依賴（非檔案級）可能導致並行執行出錯。**緩解**：除 `where.files` 外也檢查 `blockedBy` + `relatedTo`。

5. **-55% 頻率降低可能高估**：部分 D 場景在實際 session 中出現頻率低（如 D8 parallel-evaluation、D9 錯誤追蹤），加權頻率計算可能只有 -30~35%。**緩解**：W10-136 tripwire 設計已覆蓋——落地後監測實際 bypass 率。

6. **Deterministic 過濾可能削弱用戶對系統行為的感知**：自動化太多讓用戶不知道 PM 在做什麼。**緩解**：自動執行時仍輸出簡短通知（「[AUTO] 已依 runqueue 排序路由至 W10-XXX」），保持透明。
