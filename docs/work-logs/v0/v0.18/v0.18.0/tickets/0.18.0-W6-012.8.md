### 問題根因

E2E 驗證觀察「Readmoo Library header 顯示『擁有 944 本書』，但 storage / overview 只有 928 本」。差異 = 16 = 封存 15 + 借出 1。父 ticket W6-012.5 將「filter 切換可看 944」列為 AC（已勾選 PASS），但實際上：

1. **Overview UI 不存在 status filter 元件**（`src/overview/overview.html` 確認）：只有 `#searchBox`（標題搜尋）+ `#sortSelect`（排序），無封存/借出/狀態下拉選單或 toggle。
2. **Overview controller 已有 status filter 邏輯但未綁定 UI**：`overview-page-controller.js:101,496-509,535-536` 已實作 `this.statusFilter` 欄位、`setStatusFilter()` 方法、`applyCurrentFilter()` 管線（狀態 → Tag → 文字搜尋 → 排序），但無 DOM 事件接線。
3. **944 vs 928 差異源於提取層，不是 UI 過濾**：依 `docs/bookstores/readmoo.md:184,257` 與 W6-012.5 提取結果（`finalCount=928`），Readmoo Library 頁的 `a[href*="/api/reader/"]` 在瀑布流載完後僅 928 筆——封存/借出書籍不渲染為 `.library-item`，提取器（`src/content/adapters/readmoo-adapter.js:60`）以 `.library-item` 為主選擇器，故 storage 永遠拿不到 944。

### 影響範圍

| 層級 | 現況 | 缺口 |
|------|------|------|
| 提取層（content/extractors） | 只抓 visible `.library-item`（928） | 無法存取封存/借出書籍的 DOM（Readmoo 未渲染） |
| Storage 層 | 完整保存提取結果（928） | 無封存/借出資料可存 |
| Overview Controller 層 | 已有 `statusFilter` 邏輯 + `applyCurrentFilter` 管線 | 無 DOM 元件觸發 setStatusFilter() |
| Overview UI 層（HTML） | 只有搜尋框 + 排序 | 缺 status filter 下拉/toggle 元件 |
| Schema 層 | `readingStatus` 6 種列舉（unread/reading/finished/queued/abandoned/reference） | 無 `archived`/`borrowed` 狀態定義 |

**關鍵衝突**：W6-012.5 AC「filter 切換可看 944」與系統現況不一致：

- 系統不可能透過「UI filter 切換」看到 944，因為 944 那 16 本根本沒進 storage
- 要看到 944 需要提取層改造（探索 Readmoo 封存頁 API / 切換 tab 提取），不是 overview UI 工作

### 相關 Error Pattern

- 父 ticket W6-012.5 acceptance「filter 切換可看 944」屬於誤認需求邊界（將提取層問題誤標為 UI filter 功能）—— PC-055 模式（AC 與實況漂移）
- W6-012.5 AC 勾選 PASS 但實機未驗證「filter 切換動作」（因為元件不存在）—— 屬於可勾選但無動作的 AC

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 現況 vs 需求差距總表

| 項目 | 現況 | UC-06 v1.0 規格 | W6-012.5 隱含期望 | 差距 |
|------|------|---------------|------------------|------|
| 6 種 readingStatus 篩選 UI | 缺（controller 邏輯已有） | 必要 | 隱含覆蓋 | 缺 UI 元件 + DOM 接線 |
| 封存/借出篩選 UI | 缺 | **未要求** | 「切換可看 944」 | 規格未定義，且資料層也不支援 |
| Tag 篩選 UI | 缺（controller `setTagFilter` 已有） | 必要 | 未測 | 缺 UI 元件 |
| 944 本完整提取 | 928（去除封存 15 + 借出 1） | 未明定 | 「切換可看 944」 | 提取層需改造才能存到 944 |

### 944 vs 928 差異定位

差異在**提取層**，不是 storage 層也不是 UI 層：

```
Readmoo Library header: 「擁有 944 本書，其中封存 15 本，借出 1 本」
  └── DOM `.library-item` 渲染：928 本（封存/借出書籍 Readmoo 預設不渲染）
        └── content/adapters/readmoo-adapter.js 提取：928 本
              └── chrome.storage.local.readmoo_books：928 本
                    └── overview.currentBooks：928 本
                          └── (statusFilter 邏輯存在但無 UI 接線)
```

要從 overview 「切換」看到 944 在現有架構下不可能。可行路徑：

| 路徑 | 改造範圍 | 複雜度 |
|------|---------|-------|
| A. 提取器探索 Readmoo 封存/借出 tab/API | content/adapters + extractors，可能需 SSO 後 API 呼叫 | 高（含實機探索 + Schema 擴充） |
| B. 僅顯示 Library header 統計（不提取封存/借出明細） | overview 顯示「總書庫 944（其中封存 15、借出 1，本工具僅追蹤 928）」 | 低（UI 標示，不需資料） |
| C. 不做（接受 928 為提取上限） | 修正 W6-012.5 AC 錯誤期望 | 零 |

### 建議方案：路徑 C（不做）+ Schema/UI follow-up 分開排程

**理由**：

1. UC-06 v1.0 規格未要求「封存/借出」篩選，路徑 A 屬於 v2.0+ 多書城支援階段的需求探索（讀者借閱模型在不同書城差異大）。
2. v0.18.0 / v0.19.0 焦點是「測試重寫 + 打包內測」，提取層改造會延後內測時程。
3. v0.20.0 規劃「中文圖書分類法 + Tag 管理」，封存/借出語意應在該階段重新評估為 tag 或 status。
