## Problem Analysis

## Context Bundle（補齊）

### 背景（源自 W17-001 WRAP + W17-002 依賴）
- W17-001 WRAP 將「A 強制檢查」從主方案降為「C 的安全網」
- 角色：C 自動化完成後，dispatch-check 不再要 PM 手填，而是檢查**自動抽取結果**是否合理
- Linux 類比：execve() 參數驗證——kernel 在載入 ELF 前檢查 binary 格式合理性，不檢查使用者意圖

### 具體實作範圍（經 W17-004 審計確認）
- `.claude/skills/ticket/cli/` 中 dispatch-check 相關邏輯
- W17-002 產出的 context_bundle_extractor.py 輸出格式

### 合理性檢查規則（具體化）
1. **欄位非空**：source 抽取後 Context Bundle 不可全空（全空表示抽取失敗）
2. **內容長度**：每欄位 ≥ 50 字元（避免空殼文字）
3. **檔案存在**：where.files 列出的檔案在檔案系統存在
4. **語意合理**（進階）：acceptance 至少含 3 項（4V 原則）
5. **可選 LLM 審查**：若時間預算夠，可用 LLM 審查 Context Bundle 是否「真的能讓 agent 上手」

### Tripwire T2 量化指標
- Hook 擋人 ≤ 10/week（超過表示合理性規則過嚴）
- 若 C 自動抽取錯誤率 > 20%，啟動 Safety Net 模式（強制 PM review）

### 與其他 ticket 關係
- source: W17-001
- blockedBy: W17-002（必須有抽取結果才能檢查）
- relatedTo: W17-029（dispatch-plan / Linux scheduler 類比邊界）
- 角色轉變：原「A 強制檢查」→「C 的第二道防線」
- 邊界補充：dispatch-check 檢查 readiness、Context Bundle 與 dispatch-plan 產出是否足以派發；不負責產生 dispatch-plan、不實作 batch dispatch CLI，也不取代 hook / scheduler 的執行控制。

### 重要：本 ticket 是 ADJ 類型不是 IMP
- 調整現有 dispatch-check CLI，不新建命令
- Phase 2 開始（跳過 Phase 1，功能已定義）
### 問題根因

C 方案（Context Bundle 自動填料）需要第二道防線檢查抽取結果是否合理（欄位非空、長度足夠、檔案存在、acceptance 項數達標），避免「填料成功但內容為空殼」的失敗模式。

### 影響範圍

- 新建 `.claude/skills/ticket/ticket_system/commands/track_dispatch_validate.py`

**邊界**：不觸碰既有 `ticket track dispatch-check`（W10-017.2 落地的活躍派發狀態查詢 CLI）。新建獨立子命令 `dispatch-validate` 對應本 ticket 的 Context Bundle 合理性檢查職責。詳見 W17-209 ANA Solution 方案 A。
