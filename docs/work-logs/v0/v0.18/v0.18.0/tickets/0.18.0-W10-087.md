**根因**: hook #17 AUQ 邏輯「檢查 ticket 執行期間是否有新增 error-pattern」只看時間窗口，不驗證 PC 是否實際屬於該 ticket 的 scope

## 採用策略：(b) source_ticket 前瞻 + (a) ticket md 引用 fallback

### 設計決策

**策略評估**（Problem Analysis 列三候選）：
- (a) 時間窗 + commit author/scope 比對 — 單獨使用過依賴 git log，失敗情境多
- (b) PC frontmatter `source_ticket` 欄位 — 精確但對 legacy PC 不適用
- (c) frontmatter 豁免欄位 `exclude_from_ticket_scope_check` — 豁免概念與歸屬語意不符，粒度粗

**採用混合 (b)+(a)**：新 PC 強制 `source_ticket` 前瞻（原子化 Zettelkasten）；
legacy PC 無 frontmatter 時回退至「當前 ticket md 是否引用該 PC ID 或 basename」。
