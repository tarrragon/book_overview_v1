### Hook 現況

`.claude/hooks/agent-dispatch-validation-hook.py` 的 `main()` 主邏輯依優先順序處理路徑分類：

1. `has_external_claude=True` → 阻擋（runtime 必拒）
2. `has_main_repo_claude=True` 且 `has_other=False` → 放行（ARCH-015 豁免）
3. `isolation == "worktree"` → 放行
4. 其餘（has_other 無 worktree、空 prompt 無 worktree）→ 阻擋（強制 worktree）

問題：multi-view review 場景派發 thyme/parsley/fennel 等實作代理人擔任「審查」角色時，prompt 雖含 `src/` 等路徑（屬被審查目標的引用），代理人僅讀不寫，但落入 (4) 被強制 worktree 阻擋（源案例：W10-017.1 v2 派發 thyme 審查被擋）。

### 偵測位置

於 main() 步驟 (1) 與 (2) 之間插入審查模式偵測（步驟 1.5）：
- 維持外部 .claude/ 阻擋（runtime 必拒邊界不可豁免）
- 命中關鍵字 → 直接放行，跳過 (2)/(3)/(4) worktree 判斷

### 關鍵詞清單（寫死於 hook，依 ticket 邊界要求）

`REVIEW_MODE_KEYWORDS = ("審查", "review", "掃描", "scan", "評估", "evaluate")`

大小寫不敏感比對。任一命中即視為審查模式。

### 修復摘要

1. 新增 `REVIEW_MODE_KEYWORDS` 常數（6 個關鍵字）與 `_is_review_mode_prompt(prompt)` helper（大小寫不敏感比對）。
2. `main()` 在外部 .claude/ 阻擋之後、worktree 強制邏輯之前，插入步驟 (1.5)：實作代理人 + prompt 命中審查關鍵字 → 直接放行。
3. 未動其他邏輯：外部 .claude/ 阻擋、ARCH-015 豁免、worktree 通過、強制 worktree 等既有路徑保持不變。

### 設計決策

- **關鍵字寫死於 hook**：依 ticket 邊界要求，不外掛配置檔。
- **大小寫不敏感**：用 `prompt.lower()` + `kw.lower()` 雙向小寫，涵蓋 `Review/REVIEW/Scan` 等變體。
- **保留外部 .claude/ 阻擋**：runtime 必拒邊界不能豁免（即使審查也無法寫入外部 .claude/）。
