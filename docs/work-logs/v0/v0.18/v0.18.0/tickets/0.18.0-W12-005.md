## Task Summary

修改 acceptance-gate-hook 的 spawned_tickets 檢查語義：不只檢查存在，還要檢查狀態（類似 children 檢查），防止防護性 ANA 在 spawned IMP 仍 pending 時被 complete

## Problem Analysis

### Context Bundle（PC-040：PM 派發前分析輸入）

#### 問題根因

PC-075（W12-004 發現）：防護性 ANA（如 W12-002）將衍生實作任務放入 `spawned_tickets` 後，其 AC 中「spawned IMP 仍 pending」的情況下仍能通過 `acceptance-gate-hook`。`check_ana_has_spawned_tickets()` 只驗存在性，不檢狀態。結果：ANA 的防護責任脫鉤於 spawned IMP 的實際落地狀態。

Phase 1 已落地（W12-004）：
- `.claude/hooks/acceptance_checkers/ana_spawned_checker.py::check_spawned_tickets_status()` 計算非 terminal spawned 並回傳警告訊息
- `AcceptanceCheckResult.spawned_non_terminal_warning` 專用欄位（不抑制 scene #9/#1 gate）
- Checklist 項目 5 顯示「含非 terminal 項目」警告

殘留缺口：WARNING 層級被忽略時，無強制 checkpoint 讓 PM 顯性決策。

#### WRAP 評估結論（完整模式，2026-04-17）

**錨點**：影響決策品質 + 持續回報 > 一次性投入，值得投入。

**資料充足度**：部分充足（W12-004 現況、PC-075 根因、PC-073 教訓明確；Phase 1 實地效果、現存 ANA+spawned 分布、方案 E 走向為未知）。可進行方案評估，最終 Phase 3（完全替代 Phase 1）決策需等 Phase 1 觀察數據（對應 W13-001）。

**Consider the Opposite 重大發現**：
原假設「spawned 應 BLOCK」的相反假設「spawned 設計目的就是獨立追蹤不阻擋（PC-073 原意）」有強論述支持。→ 問題重新定義為「如何在尊重 spawned 獨立性的前提下，防止防護性 ANA 的責任漂移」。原方案 A/D 嚴格 BLOCK 違反 PC-073，應排除。

**擴增選項**：A（嚴格 BLOCK + 逃生閘）、C（CLI 互動確認）、D（純 BLOCK 無逃生閘）、I（雙欄位語義分離）、J（ANA 子類型）、K（Phase 1 警告強化為 blocking confirmation）、L（延後至 W13-001）。

**採方案 K**（原因見 decision_tree_path.rationale）：
1. 不違反 PC-073 spawned 獨立性原則
2. 顯性化 PM 決策（y/N prompt 解決 PC-075 根因：「看不見」spawned 未完成）
3. 工時較低（1-1.5h），不排擠 W11-003 CLI bug
4. 保留 W13-001 未來評估方案 E 的空間
5. 與 children_checker 哲學差異對應明確（children=強依賴 BLOCK；spawned=獨立追蹤 + 顯性確認）

**行前預想最可能失敗**：
- 非互動環境（CI/CD / 非 TTY stdin）blocking confirmation 卡死 → 對應防護：`--yes-spawned` flag 必需
- Hook 層重複檢查導致雙重 gate → 對應防護：acceptance-gate-hook spawned 檢查明確保持 WARNING 語義（不升級），CLI 層獨立控制 block
