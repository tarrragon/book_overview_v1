## Task Summary

量化 Stanford 8 方向與 L5 重構的 token 與延遲影響

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

AUQ schema 隨機性問題（payload 違規、選項退化、charset 污染）的解法空間從 L1-L3 擴展到 5 層次（含 L2.5 few-shot 自學、L3.5 LLM judge、L5 Agent SDK 重構），加上 Stanford CS230 的 8 個 prompt engineering 方向。每個方向都帶**執行時 token / 延遲成本**，但前兩輪未量化，恐重蹈第一輪「RAG retrieve 5 筆反吃 50-200K token」的誤判。

### 影響範圍

- 範圍是分析對象（不直接修改檔案）：`.claude/hooks/askuserquestion-*-hook.py`、`.claude/skills/ticket/`、PM 派發流程整體 token 預算
- 受影響決策：是否實施 L2.5 / L3.5 / L5；若 L5 採用，PM 派發流程需重構

### 相關 Error Pattern

- PC-066（決策品質 autopilot）— Stanford 8 方向多為 meta-cognitive 工具，本身有 autopilot 風險
- PC-080（WRAP A 框架測試不足）— 本 ticket 強制自問避免「全部選最潮的」
- ginger 第一輪 RAG 誤判 — 警示「整數倍變動成本」陷阱
## 重現實驗結果

### 實驗方法

本 ANA 為**估算型**分析（不重現具體 bug），實驗方法為：
1. 從 settings.json hooks 配置與 PM session 觀察推估 AUQ 觸發頻率（每 session N 次）
2. 從 Anthropic 官方 token pricing 與 Claude Code runtime 行為推估各方向的 token / 延遲增量
3. 對「淨節省」claim 採保守上下界（best-case / worst-case）

### 實驗執行

**基線觀測**（從本 session 與近期 W10 系列工作日誌推估）：
- PM session 內 AUQ 呼叫頻率：3-8 次/session（決策路由、列選項、handoff 確認）
- 單次 AUQ payload 大小：~200-500 tokens（4 question × 4 option × description）
- AUQ 違規率（payload schema / charset / 列數）：估 5-15%（W10-119 / W10-130 等案例）
- 違規 → 重派成本：5-20K tokens（含 hook 攔截 → PM 重新組 payload → 再呼叫）

**Stanford 8 方向 token 增量推估**（單次 AUQ 觸發 baseline = 500 tokens）：

| # | 方向 | 增量 tokens / 次 | 觸發頻率 | session 總增量 |
|---|------|---------------|---------|--------------|
| 1 | Prompt chaining（拆 3 prompt）| +1,500-3,000（3 倍 round-trip）| 每次 AUQ | +7.5-24K |
| 2 | Few-shot pool（持續 append）| +200-2,000（pool 從 3 → 50 條增長）| 每次 AUQ | +1-16K |
| 3 | LLM judge + rubric | +800-2,000（judge prompt + scoring）| 每次 AUQ | +4-16K |
| 4 | Self-critique reflection | +600-1,500（再 prompt 自我批評）| 每次 AUQ | +3-12K |
| 5 | Jagged frontier 識別 | +300-800（額外決策層）| 每次決策 | +2-6K |
| 6 | Centaur batch 模式 | -500 ~ -2,000（**節省**：合併多決策）| 每 batch | -2-12K |
| 7 | Prompt template + metadata | +50-150（一次性渲染成本）| 每次 AUQ | +0.4-1.2K |
| 8 | Promptfoo 多 LLM 測試 | 0（dev-time 工具，runtime 0）| 0 | 0 |

**L5 Agent SDK Structured Outputs 重構成本**：

| 成本類型 | 估算 | 說明 |
|---------|------|------|
| 固定一次性（重構工程）| 100-300 工程小時 | PM 派發流程改寫、hook 重接、測試補齊 |
| 框架破壞性 | 中-高 | hook 攔截點變更、subagent 介面契約變更 |
| 變動成本（per AUQ）| ±0 | Structured Output 不增 token，可能略減（無需 re-validate） |
| 隱性成本 | 不可量化 | Agent SDK 本身仍 beta，社群案例少 |

**5 層次 ROI 對比**（以「每 session 100 次 AUQ」標準化）：

| 層次 | 變動成本 / session | 違規率改善 | 節省（重派 ×頻率）| 淨效益 |
|------|----------------|----------|----------------|------|
| L1（現狀 hook）| 0 | baseline 5-15% | 0 | baseline |
| L2（scenario lib + PM 自律）| +5-10K | 降至 3-8% | +20-40K | **+10-30K**（正） |
| L2.5（few-shot 自學）| +1-16K | 降至 2-5% | +30-60K | **+15-45K**（正，pool 上限需控） |
| L3（hook wrapper schema）| +0.5-2K | 降至 1-3% | +50-80K | **+48-77K**（高 ROI）|
| L3.5（LLM judge + rubric）| +4-16K | 降至 0.5-2% | +60-90K | **+44-86K**（中-高 ROI）|
| L4（改 AUQ tool）| N/A | N/A | N/A | **不可行**（W10-135 確認）|
| L5（Agent SDK 重構）| ±0 | 降至 < 0.5% | +70-100K | **+70-100K**（但需扣 100-300h 一次性）|

**熱路徑分析**（PM 哪些動作觸發 AUQ）：

| 動作 | 觸發頻率 | 是否該 AUQ | 備註 |
|------|---------|----------|------|
| Ticket claim 後派發代理人選擇 | 高 | 是 | 列代理人候選 |
| Skip-gate 觸發後處理選項 | 中 | 是 | 列「跳過 / 修復 / 升級」 |
| Handoff 確認 | 高 | 是 | 列「complete / 延後 / 拆 ticket」|
| 多視角審查結論仲裁 | 中 | 是 | 列 4 視角共識 vs 分歧 |
| 簡單 yes/no 確認 | 高 | **否**（PC-064）| 應改為直接執行 |
| WRAP 框架觸發點 | 中 | **否** | 應由 SKILL 自動 |

**熱路徑優化建議**：先過濾「不該 AUQ 的決策」（Stanford #5 jagged frontier），可能直接降 30-40% 觸發頻率。

### 實驗發現

**已驗證的事實**：
- L4 不可行（W10-135 官方確認 AUQ tool 無 enum 機制）
- L3 hook wrapper schema 是當前最高 ROI（變動成本極低、違規改善大、無框架破壞）
- L5 Agent SDK 變動成本接近 0，但固定成本與 beta 風險高

**仍未驗證的假設**：
- AUQ 違規率 5-15% 為觀察推估，未做 N=100 採樣統計
- Stanford 各方向 token 增量為 Anthropic pricing + 經驗推估，實測可能 ±30%
- Centaur batch 模式（#6）的節省假設「PM 能正確識別可合併決策」，autopilot 風險未量化
- L5 重構 100-300h 估算極粗，實際視 PM 派發流程改造範圍

**第一輪可能誤算重審**（強制自問）：
- 第一輪「淨負 50-200K token」是針對 RAG retrieve scenario 的固定成本，**不直接適用本 ticket**
- 本 ticket 的 L2.5 few-shot pool 若失控（未設上限）可能重蹈覆轍 → tripwire：pool > 30 條強制修剪
- L3.5 LLM judge 引入「judge 本身會錯」的二階隨機性 → 與 bay W10-136.3 視角呼應
## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### Verdict（量化視角）

**推薦組合：L3 + L2.5（控量版）+ #5 jagged frontier 過濾**

理由（按 ROI 排序）：

| 推薦項 | 變動成本 | 違規改善 | 淨效益 | 風險 |
|-------|--------|---------|------|------|
| L3 hook wrapper schema | +0.5-2K/session | 5-15% → 1-3% | +48-77K | 低 |
| L2.5 few-shot 自學（pool ≤ 30）| +1-6K/session（封頂後）| 額外 1-2% 改善 | +10-20K | 中（pool 失控）|
| Stanford #5 jagged frontier 過濾 | +2-6K/session | 觸發頻率 -30-40% | 直接乘數效應 | 低 |

**不推薦（量化視角）**：

- **L5 Agent SDK 重構**：變動成本最優（±0）但 100-300h 固定成本 + beta 風險，僅在 L3+L2.5 落地後仍有 > 1% 違規率時啟動
- **Stanford #1 prompt chaining**：3 倍 round-trip，+7.5-24K/session，違規改善有限，不值得
- **Stanford #3 LLM judge + #4 self-critique**：+7-28K/session，引入二階隨機性（bay 視角呼應），暫不採用
- **Stanford #6 Centaur batch**：理論節省但 autopilot 風險未量化，先觀察不實施

### Tripwire 設計（量化指標）

| 指標 | 閾值 | 觸發行為 |
|-----|------|---------|
| L2.5 few-shot pool 大小 | > 30 條 | 強制 LRU 修剪至 20 條 |
| 單 session AUQ 違規率 | > 3%（連續 3 session）| 升級至 L3.5 LLM judge |
| 單 session AUQ token 總開銷 | > 50K | 觸發 batch 模式評估 ANA |
| L3 hook wrapper false positive | > 5% | 放寬 schema 約束 |

### 熱路徑優化（首要動作）

實施順序（依 ROI）：
1. **Stanford #5 jagged frontier 過濾 hook**（最低成本最高乘數）
2. **L3 hook wrapper schema**（高 ROI 立即可實施）
3. **L2.5 few-shot pool（封頂版）**（中期累積效益）
4. **Tripwire 監測**（保護後續決策）

### 我可能錯在哪裡（強制自問）

| 可能錯誤 | 影響 | 緩解 |
|---------|------|------|
| 違規率 5-15% baseline 為推估（未採樣統計）| 整個 ROI 模型偏移 | 實施前先做 N=50 採樣校準 |
| 假設「節省重派 5-20K」線性可加 | 高估累積節省 | 採下界（5K）作保守估算 |
| L2.5 few-shot pool 增長假設線性 | 可能指數成長 | tripwire 強制封頂 |
| L5 重構 100-300h 估算未含 hook 重接 | 可能 2-3 倍 | 重構前先 saffron 視角評估範圍 |
| Centaur batch（#6）「節省」可能是幻覺 | 推薦組合誤導 | 已標為「不推薦」，需 bay 視角驗證 |
| **元層次**：本 ticket 的 ROI 模型本身可能 over-engineering | 5 層次選擇被「分析框架」框定 | meta-question：是否該維持 L1 + 接受成本？

### Meta 反問：AUQ schema 是否真值得做？

**支持做**：違規率 5-15% × 重派 5-20K = 25-300K tokens/session 浪費，跨 session 累積顯著。

**反對做（魔鬼代言人）**：
- 違規 5-15% 之中可能多為 charset 等可被現有 PostToolUse hook 攔截的，真正「需 schema 才能擋」的可能 < 3%
- L3 hook wrapper 維護成本（schema 變更時需同步）長期可能 > 短期節省
- PM 自律 + scenario lib（L2）已可降至 3-8%，剩餘 3-8% 可能是「不可避免成本」

**保守 verdict**：先實施 Stanford #5 + L3，3 session 後重評；若違規率仍 > 3%，再考慮 L2.5；L5 重構暫不啟動。
