### 判斷原則（父 ticket 共通指引）

**拆分依據是「認知負擔」與「按需載入」，不是行數。**

這是卡片盒筆記法（Zettelkasten）的核心精神：

- 正確問題：這段知識在什麼情境才需要理解？每段知識是否原子化？按需載入時能否獨立理解？
- 錯誤問題：檔案多少行？骨架是否夠短？行數超過某閾值就要拆？

**推論**：
- 500 行的檔案若內容全部原子化且在同一情境共同需要 → 不應拆分
- 100 行的檔案若混合多個情境才用到的知識 → 即使短也應拆
- 拆分後的每一碎片必須能獨立被理解（原子性），否則只是把完整概念切碎
- 保留判斷標準不是「這段很重要」，而是「所有使用場景都需要立即載入」

**WRAP 分析時的具體提問**：

1. **W (擴增選項)**：列出所有可能的原子化切分方式（不是以行數為切點，而是以「情境匹配度」為切點）
2. **R (現實測試)**：每個切分方案下，各個碎片是否能獨立被理解？保留在 auto-load 的碎片是否「所有場景都需立即載入」？
3. **A (保持距離)**：從 sync 到其他專案的角度看，這些知識是否真的每個專案啟動都需要？
4. **P (準備出錯)**：若拆分後發現某個碎片其實常被需要，如何快速恢復到 auto-load？

## Solution

# WRAP 分析：bash-tool-usage-rules.md (207 行) 拆分可行性

## W (Widen) — 擴增選項

列出 5 個候選方案（非僅以行數為切點，以「情境匹配度 × 違規成本」為切點）：

**選項 A：保持現狀（不拆分）**
- 維持 207 行自動載入
- context 節省: 0 行
- 風險: 低

**選項 B：骨架保留 + 詳細案例遷移到 references（部分拆分）**
- 自動載入保留三條規則的「強制規範表 + 檢查清單 + 安全做法速查」約 80 行
- 詳細根因圖解 / 典型混淆案例 / chpwd 深度說明遷移到 `.claude/references/bash-tool-usage-details.md` 約 127 行
- context 節省: 約 127 行（約 61%）
- 風險: 中（代理人第一次違規時需主動讀 references）

**選項 C：完全遷移到 references（只在 rules/core/ 保留索引卡片）**
- 自動載入保留 20 行指向 references 的「三規則一句話摘要 + 違規時讀哪個 reference」
- 完整規則遷移到 `.claude/references/bash-tool-usage-rules.md` 全部 207 行
- context 節省: 約 187 行（約 90%）
- 風險: 高（規則一是高頻違規，延遲到「出錯才讀」已太遲）

**選項 D：拆成三個獨立 rules 檔（各規則一檔，全部自動載入）**
- `bash-cd-pollution-rules.md`（約 60 行）
- `bash-output-handling-rules.md`（約 55 行）
- `bash-git-chaining-rules.md`（約 50 行）
- 三檔全部自動載入
- context 節省: 0 行（甚至略增，因需各加 header/footer）
- 風險: 低，但不解決 context 問題

**選項 E：Hook 完全取代（新建三個 PreToolUse:Bash Hook）**
- 新建 `bash-cd-pollution-hook.py`、`bash-taskoutput-reminder-hook.py`、`bash-git-chain-guard-hook.py`
- 規則檔降為 30 行指南
- context 節省: 約 177 行（約 85%）
- 風險: 非常高（Hook 開發/維護成本 + pattern match 漏判 + 教育價值流失）

## R (Reality Test) — 現實測試

**每個碎片是否能獨立被理解**：
- 選項 B 的骨架（80 行）能否獨立提醒正確行為？→ 可以，表格 + 檢查清單足以讓代理人「寫出正確命令」
- 選項 C 的索引卡片（20 行）能否獨立觸發查表？→ 不行，高頻違規（PC-046 50+）若規則剛載入就摘要化，代理人容易略過
- 選項 E 的 Hook 能否取代教育？→ 不行，Hook 只能在違規發生時輸出 stderr 警告，無法教「為什麼子 shell 正確」和「為什麼不能串接 git commit && git merge」

**保留在 auto-load 的碎片是否「所有場景都需立即載入」**：
- 三條規則適用範圍：代理人只要呼叫 Bash 工具就可能踩到任一條。PM 前台幾乎每個 session 都會執行 git/uv/ticket 指令，屬高共用知識
- 骨架（選項 B）保留的「表格 + 檢查清單」對應「每次下 Bash 前在心裡 checklist」的使用情境，符合「每次都需要」
- 詳細案例（選項 B 遷移部分）對應「被 Hook 擋下 / 遇到錯誤訊息後 debug」的使用情境，屬條件性需要
- **Hook 防護覆蓋度僅 1/3** 強化了規則本體必須保留 auto-load 的必要性（Hook 不足以兜底）

**從實際違規數據看**：PC-046（cd 多餘）50+ 次違規證實，即使 rule 已 auto-load，規則一仍高頻踩雷。若降為 references 按需讀取，違規率很可能上升

## A (Attain Distance) — 保持距離

**從跨專案 sync 角度**：
- `.claude/` 透過 sync 機制同步到其他專案。其他專案的代理人也會使用 Bash 工具
- 三條規則的技術根因（CC Bash 持久 shell / CC TaskOutput API / git index.lock）是 Claude Code runtime 固有特性，與專案無關 → 所有 sync 到的專案都需要此規則
- 此規則屬於「Claude Code runtime 使用規範」，與 language-constraints / document-format-rules 同屬 framework-level 必要知識

**從其他規則檔對比**：
- `quality-common.md`（378 行）在 W10-073.3 討論中被識別為「僅代理人寫程式碼時需要」，可考慮拆分。**bash-tool-usage-rules 不同**，PM 和代理人都會用 Bash
- 最接近的參照是 `language-constraints.md`（繁中/禁用詞）：全 session 適用、教育價值 > 純 Hook 攔截 → 兩者都該保留 auto-load，但細節可搬 references

**反 sunk cost 檢視**：
- 目前 207 行是歷次疊加的結果（v1.0 → v1.4.0 陸續新增 chpwd、git 串接）。若純看「新加的內容是否需要 auto-load」，chpwd Shell 警告 15 行其實更像條件性知識（非所有 shell 環境都有 chpwd hook），可考慮搬 references

## P (Prepare to be Wrong) — 準備出錯

**選項 B 可能的失敗場景與回退**：

| 失敗場景 | 症狀 | 回退路徑 |
|---------|------|---------|
| 代理人不主動讀 references | 違規率未下降反上升 | 在規則骨架檢查清單新增「不懂原因？→ 讀 references/bash-tool-usage-details.md」顯式指引，並在 bash-edit-guard-hook.py 擋到疑似違規時輸出 reference 路徑 |
| 骨架表格不足以教育新場景 | 出現表格未覆蓋的違規（如新 shell hook 類型） | references 擴充後，若該場景高頻則把對應行拉回骨架 |
| 其他專案 sync 後找不到 references | 引用路徑斷裂 | 規則 8 要求：references 路徑用 `.claude/references/` 絕對路徑，sync 時會一併帶過去，不會斷裂 |
| 違規率短期反彈 | 代理人剛切換不適應 | 觀察 2-3 週違規趨勢。若 PC-046 類違規 > 基準值 30% 則回退到選項 A |

**關鍵絆腳索**：若 90 天內 PC-046 occurrences 增長率 > 現況基準 1.5 倍，表示拆分失敗，自動回退選項 A（git revert 即可）

## 最終建議：選項 B — 骨架保留 + 詳細案例遷移 references

**量化節省**：
- 自動載入從 207 行降至 ~80 行，節省 127 行（約 61%）
- context 節省不如選項 C/E，但保留了教育價值和 Hook 覆蓋缺口的兜底

**為何非 A**：三條規則的詳細案例（根因圖解、典型混淆案例、chpwd 深度說明）屬「理解為什麼」的條件性知識，不是每次 Bash 呼叫都需要立即載入。保留在 auto-load 浪費 127 行 context

**為何非 C**：規則一是 50+ 次違規的高頻問題，Hook 覆蓋率為 0，規則本身是唯一防線。降為索引卡片會讓「正確命令格式」這類高共用知識延遲載入，違規成本太高

**為何非 D**：不解決 context 問題，且三規則主題相關（都是 Bash 工具使用），拆三檔增加檔案管理成本

**為何非 E**：三規則 Hook 覆蓋僅 1/3，開發新 Hook 成本高且無法教育；即使 Hook 完成，規則本身仍需保留教育文件。且 Hook 遺漏率高於規則遺漏率（pattern match vs 讀 checklist）
