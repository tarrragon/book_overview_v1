## ANA 結論 (PM 前台分析, 2026-04-19)

### 評估標的
`CheckpointState` 兩個欄位：
- `data_sources: Dict[str, str]` — 記錄 5 個資料源（git_status / dispatch_active / handoff / ticket_query / git_worktrees）狀態，值為 "ok" 或錯誤訊息字串
- `pending_checks: List[PendingCheck]` — 待處理檢查項清單，影響 `ready_for_clear` / `next_action`

### 語意差異

| 欄位 | 語意層 | 用途 | 粒度 |
|------|-------|------|------|
| `data_sources` | 結果記錄（資料源層） | metrics log + 觀測診斷 | 細粒度（每個 _read_* 一筆） |
| `pending_checks` | 流程影響（業務語意層） | `ready_for_clear` 判斷 + `next_action` 推導 | 業務抽象（可包含非資料源的待處理項） |

### 重疊程度

| 重疊項 | 程度 |
|--------|------|
| 資訊重複 | 部分重複（data_source 失敗會反映在 pending_checks，但 pending_checks 也含其他類型如 dispatch_active 中的待處理 agent） |
| 語意重複 | 不重複（觀測 vs 流程判斷） |
| 修改頻率 | data_sources 僅資料源 _read_ 失敗時 mutate；pending_checks 由 _derive_checkpoint 推導 |

### 4 視角共識（Phase 4 cinnamon R8）

> 「data_sources dict 與 pending_checks list 部分語意重疊（同樣記錄 data_source_X 失敗），非純重複但值得 Phase 4 後續斟酌是否合併」

### 合併方案評估

| 方案 | 收益 | 成本 |
|------|------|------|
| A 完全合併（單一 list） | 減少 dataclass 1 欄位 | 失去細粒度（觀測層需再過濾）；觀測 log 結構變動 |
| B 保留兩者 + 共用 helper（如 `_record_data_source_status` 同時寫入兩者） | 同步保證 + 結構不變 | 引入 helper 但兩處 mutate 點仍在 |
| C 保留現狀 + 註解強化邊界文件化 | 零成本 | 不解決潛在 drift 風險 |

### ANA verdict

**選方案 C：保留現狀 + 註解強化邊界文件化**

理由：
1. **語意分離合理**：兩欄位代表兩個語意層（觀測 vs 流程），合併會失去明確邊界
2. **修改點分離**：`data_sources` 在 DATA_SOURCES table for loop 內 mutate（每個資料源），`pending_checks` 在 `_derive_checkpoint` 由業務邏輯產生，邏輯路徑不同
3. **drift 風險低**：兩者由不同來源產生，當資料源失敗時 `_derive_checkpoint` 自然會推導出對應 pending check，無需強制同步
4. **YAGNI 原則**：合併只在資訊重複嚴重影響可讀性時才有收益，目前 5 個資料源狀態 + 業務 pending 兩者並存不重，閱讀時可清楚區分

### 後續建議（不開新 Ticket，僅文件化）

在 `CheckpointState` dataclass docstring 內加註：
- `data_sources`: 觀測層（資料源狀態快照），mutate by DATA_SOURCES for loop
- `pending_checks`: 業務層（待處理項抽象），推導 by `_derive_checkpoint`
- 兩者非冗餘，分屬不同語意層

此註解可在後續任何 IMP 時順帶加入（不必獨立 Ticket）。
