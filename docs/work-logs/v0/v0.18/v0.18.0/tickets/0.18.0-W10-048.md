## 診斷結論：兩個 Hook 因 stdin 欄位命名 bug 全失效；一個 Hook 健康無問題

### 失效模式診斷表

| Hook | 註冊事件 | 觸發次數（log 檔數）| 命中目標次數 | 失效根因 | 設計目的剩餘有效性 |
|------|---------|------|------|---------|---------|
| dispatch-count-guard-hook | PostToolUse(Agent) + UserPromptSubmit | 160 | 0（全部 log 為 "未知的 Hook 事件: "） | **stdin 欄位 bug**：讀取 `hookEventName` (camelCase)，官方規範為 `hook_event_name` (snake_case)。永遠進入 else 分支 → no-op | 有效（派發計數警示對 PM 仍有價值） |
| file-ownership-guard-hook | PreToolUse(Agent) | 68 | 0（全部 log 為 "非目標觸發"） | **stdin 欄位 bug**：`is_target_trigger()` 讀 `hookEventName` + `toolName` (camelCase)，官方規範為 `hook_event_name` + `tool_name`。永遠返回 False → 跳過所有 Agent 派發 | 有效（檔案 ownership 防護是 ARCH-015 防線之一） |
| process-skip-guard-hook | UserPromptSubmit | 85 | **健康**：log 多為 "Hook execution time" 屬正常無 skip-intent；2026-05-05 兩條 INFO log 證實在 active dispatch 與 JSON 控制字元情境下正確路由 | **無 bug**。"全空執行" 的描述為誤判：UserPromptSubmit 大多數用戶輸入本就不含 skip pattern，hook 正常 fall-through | 有效且運作中 |

### 根因分析

**Source of camelCase bug**：
- `dispatch-count-guard-hook.py` 與 `file-ownership-guard-hook.py` 來自 commit `2d64dc5f`（外部 .claude v1.1.53 chore sync），引入時即帶有 camelCase 欄位讀取
- 對照本專案其他健康 hooks（`active-dispatch-tracker-hook`、`agent-dispatch-logger-hook`、`process-skip-guard-hook`）：要嘛只在「輸出」端用 camelCase（`hookSpecificOutput.hookEventName`，符合 Claude Code stdout schema），要嘛根本不從 stdin 讀 event name，僅靠 settings.json 註冊事件路由
- 官方 spec（`.claude/hook-specs/claude-code-hooks-official-standards.md`）與 IMP-055 明確定義 stdin 欄位為 snake_case：`hook_event_name`、`tool_name`

**為何長期未被偵測**：
- 兩 hook 失效模式為「靜默 no-op」，不阻擋任何流程，PM 與代理人不會看到錯誤
- log 內容為 DEBUG 級別，需主動審計（W10-035.3 ginger 密度審計才暴露）
