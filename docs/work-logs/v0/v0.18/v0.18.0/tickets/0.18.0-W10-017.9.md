### F 方案：SessionStart Hook 自動注入（重評）

**父 Ticket 原述**：2 小時寫 Hook + 摘要文件、每 session 固定 token 開銷、優點被動注入

**實況校正**：
- SessionStart 機制已成熟（16 hooks / 1 matcher）— F 實質是「第 17 個 hook」
- 邊際成本：80-150 行 Python（對照既有 session-experience-persistence-reminder-hook.py 複雜度）
- token 開銷可量化：14 項 checkpoint 摘要若 300-500 token，每 session 全載入
- 已有類似先例：tech-debt-reminder、branch-status-reminder 證明「開場注入摘要」模式可行

**預判評分**：
| 維度 | 評估 |
|------|------|
| 實作成本 | 極低（既有 matcher + 既有模板 + 既有 hook_utils） |
| Token trade-off | 每 session +300 token 固定成本 vs 每次 handoff 省 1200 token |
| ROI | >20x（1 session/day × 10 handoff/session 假設） |
| 破壞性 | 無（新增而非修改既有 hook） |
| 可逆性 | 高（刪 hook 即回退） |

**條件式 go 判準**（4 週後）：
- T1（使用率不升）與 F 無關（F 是被動注入，不需 PM 主動呼叫）
- T2（completion-checkpoint-rules Read 次數下降）— **F 直接衝擊此指標**，下降 < 50% 則 F 失敗
- T5（handoff→/clear tool call 下降）— F 應貢獻 3-5 次 tool call 節省
