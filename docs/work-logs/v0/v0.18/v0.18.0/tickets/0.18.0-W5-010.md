### 規則提煉（初稿，待 DOC 階段精煉）

**規則名**：失敗案例學習原則

**核心主張**：
- 失敗案例承載的資訊量遠大於成功案例（成功往往多路徑可達，失敗則直指特定瓶頸）

- 改善計畫與修復架構是錯誤的正向產物，應主動建立

**禁止行為**：
- 發現派發錯誤/設計錯誤時自動回退既成工作（除非工作本身有害）
- 將疏失歸咎於個案而不提煉為規則
- 以「避免犯錯」為由過度保守決策

**鼓勵行為**：
- 疏失發生時同步建立：(a) 事件紀錄 ANA Ticket、(b) 改善 IMP Ticket、(c) memory feedback 條目
- 主動將個案教訓升級為框架層規則或 Hook
- 將「錯誤」視為系統邊界的探測訊號而非個人責任

### 規則 7（quality-baseline.md）：失敗案例學習原則

**核心承諾**：發現疏失時的第一反應必須是「萃取經驗」而非「回退已完成工作」。失敗案例比成功案例承載更多系統邊界資訊。

| 要求 | 說明 | 可協商 |
|------|------|--------|
| 保留可用既成工作 | 若既成工作品質可用，即使流程有瑕疵也保留 | 否 |
| 三件事同步建立 | ANA Ticket 記根因 + IMP Ticket 落地修復 + memory feedback 跨 session 可見 | 否 |
| 主動升級框架 | 個案教訓必須提煉為框架層規則或 Hook（rule 6 框架優先） | 否 |
| 禁止過度保守 | 不以「避免犯錯」為由拒絕探索或延後決策 | 否 |

**唯一豁免**：既成工作本身有害（資料損壞、安全漏洞、破壞性操作）才允許回退。

**禁止行為**：

| 行為 | 原因 |
|------|------|
| 發現疏失就主動撤銷既成工作（除非有害） | 失去學習機會、浪費已完成工作 |
| 將個案歸咎於「這次不小心」而不建 Ticket/rule | 疏失會重複發生，系統性問題需系統性防護 |
| 以「避免再犯」為由轉為極度保守 | 過度保守削弱探索能力，框架應靠規則和 Hook 防護，不靠個人記憶 |
| 修復完成後不建 error-pattern | 違反規則 5「所有發現必須追蹤」 |

**鼓勵行為**：

| 行為 | 理由 |
|------|------|
| 疏失發生同步建 ANA + IMP + memory | 確保教訓不遺失，三通道互補 |
| 把 prompt/Ticket/worktree 的「成功案例」固化為模板 | 正向案例的固化價值 = 負向案例的預防價值 |
| 在新規則中明確記錄觸發案例 | 規則文本自帶「為什麼」，後人閱讀不需重新考古 |
