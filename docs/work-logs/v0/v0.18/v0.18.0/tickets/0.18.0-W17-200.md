### 問題根因

`ticket track list` 命令底部僅顯示 stale ticket 彙總計數（`format_stale_list_summary`），PM 評估時無法定位個別 stale ticket，需手動讀取多個 ticket md 的 frontmatter `created` 欄位才能確認哪些 ticket 已過閾值。

無直接相關 PC；設計上對齊 W17-008.15 stuck-anas 模式（version-agnostic 命令註冊）+ W17-031.4 stale ticket 機制（lib/staleness.py 復用）。
## 目標

新增 `ticket track stale-list` 子命令，列舉 stale ticket 明細（ID + 標題 + days_since_updated + 等級），補 PM 評估 stale ticket 時需手讀 frontmatter 的缺口。

## 既有基礎（直接複用）

| 閾值常數 | 同上 | `STALE_INFO_DAYS=7` / `STALE_WARNING_DAYS=14` / `STALE_CRITICAL_DAYS=30` |
| `format_stale_list_summary()` | 同上 | 既有 `list` 命令底部的彙總顯示來源 |
| `format_stale_warning()` | 同上 | 單一 ticket stale 警告格式（query 命令使用） |

### 修復摘要

新增 `ticket track stale-list` 子命令，列舉 pending 且建立日期超過閾值的 ticket 明細。預設 `--threshold warning` 顯示 warning + critical 級；info / all 顯示三級；critical 僅 critical。

### 實作要點

| 元件 | 路徑 | 說明 |
|------|------|------|
| 新命令模組 | `.claude/skills/ticket/ticket_system/commands/track_stale_list.py` | `execute_stale_list` + `register_stale_list`；參考 stuck-anas 模式 |
| 註冊點 | `commands/track.py` | import (line 140-143)、_create_version_agnostic_handlers dict、register() 末段 |
| 復用 | `lib/staleness.py` | `calculate_stale_level` + `_parse_created`，不重定義閾值 |
| 測試 | `tests/test_track_stale_list.py` | 13 個測試覆蓋 threshold / status filter / 排序 / wave / format / 空集合 |
| 文件 | `SKILL.md` + `references/track-command.md` | 新增 stale-list 章節（用法 / Flag 表 / 範例 / 設計約束） |

### 設計決策

- **欄位選擇 `created` 而非 `updated`**：對齊既有 `staleness.py`（calculate_stale_level 以 created 計算年齡），避免雙軌邏輯；ticket prompt 表述「days_since_updated」實質為「days since created」。
- **`--threshold all` 同 `info`**：依 prompt 指示三級全收，保留 `all` 作為更直覺的別名。
- **僅列 `status=pending`**：in_progress stale 由 `is_stale_in_progress` + runqueue 已涵蓋（W17-031.4），避免職責重疊。
- **`--format ids` 一行一 ID**：適合與 xargs / 其他 ticket 命令串接。

- 不改既有 `list` 命令彙總顯示
