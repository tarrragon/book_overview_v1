### 判斷原則（父 ticket 共通指引）

**拆分依據是「認知負擔」與「按需載入」，不是行數。**

這是卡片盒筆記法（Zettelkasten）的核心精神：

- 正確問題：這段知識在什麼情境才需要理解？每段知識是否原子化？按需載入時能否獨立理解？
- 錯誤問題：檔案多少行？骨架是否夠短？行數超過某閾值就要拆？

**推論**：
- 500 行的檔案若內容全部原子化且在同一情境共同需要 → 不應拆分
- 100 行的檔案若混合多個情境才用到的知識 → 即使短也應拆
- 拆分後的每一碎片必須能獨立被理解（原子性），否則只是把完整概念切碎
- 保留判斷標準不是「這段很重要」，而是「所有使用場景都需要立即載入」

**WRAP 分析時的具體提問**：

1. **W (擴增選項)**：列出所有可能的原子化切分方式（不是以行數為切點，而是以「情境匹配度」為切點）
2. **R (現實測試)**：每個切分方案下，各個碎片是否能獨立被理解？保留在 auto-load 的碎片是否「所有場景都需立即載入」？

#### 最終建議：**方案 2（完全外移）**

**建議理由**：

1. **情境匹配度完全錯配**：可觀測性規則是「寫 catch 區塊/日誌/全域錯誤處理」的具體指引，與 PM 每次啟動的「派發、驗收、回覆」無任何關聯。rules/ 自動載入違反卡片盒筆記法核心原則（情境錯配）。

3. **既有防護已足夠**：
   - `quality-baseline.md` 規則 4 已有完整的 except 區塊檢查要求 + stderr/日誌雙通道要求
   - `quality-baseline.md` 規則 4 行 107 已引用 `references/observability-rules.md` 作為語言細節入口
   - `references/quality-dart.md` 和 `references/quality-go.md` 各有獨立的可觀測性章節
   - 代理人透過 quality-baseline（自動載入）即可找到 references 入口

4. **rules/ 獨有內容（規則 2、3）的保留策略**：
   - 規則 2「統一日誌工具（禁止 `debugPrint`/`print`/`console.log`）」：可納入 `references/observability-rules.md` 新增「工具選擇」章節

   - Context 節省：89 行 / 2996 bytes
   - 占當前自動載入總量（2166 行 / 84992 bytes）約 **4.1% 行數 / 3.5% bytes**
