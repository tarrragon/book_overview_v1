## Task Summary

L2 hook 設計 偵測 PM 回覆 N 選項 Markdown 表格 + 後續無 AUQ 反模式規格 + 技術可行性

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

PC-064 防護表「Hook 偵測 PM 回覆文字含選項 pattern」標「建議實施」，W5-042 已實作 `auq-option-pattern-detector-hook.py` 並部署至 `settings.json`（UserPromptSubmit）。但現行 hook 偵測範圍有一處缺口：

**現行覆蓋**：
- 連續 3+ 行首選項標記（`A./B./C.` 或 `1./2./3.`）+ 問句結尾 → 命中
- 二元問句（`要繼續嗎？` 等）→ 命中

**缺口**：PM 以 Markdown 表格（`| 選項 | 說明 |` 格式）呈現 N 個候選方案，後續未呼叫 AskUserQuestion。W17-174 父 ticket 記錄的 5 次違規中，多數以表格形式列選項而非 `A./B./C.` 列表，現行 hook 完全未覆蓋此 pattern。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `.claude/hooks/auq-option-pattern-detector-hook.py` | 需擴充偵測規則（新增 Markdown 表格路徑） |
| `.claude/hooks/tests/test_auq_option_pattern_detector.py` | 需新增表格路徑測試 |
| `.claude/plans/hooks/auq-option-pattern-detector-spec.md` | 需補充 §3.x 表格偵測規則 |

### 相關 Error Pattern

- PC-064：PM 列純文字選項未用 AskUserQuestion（本案直接關聯）
- PC-134：ANA self-reference irony（分析 X 違規的 ticket 自身犯 X）
### 重現實驗結果

### 實驗方法

閱讀現行 `auq-option-pattern-detector-hook.py` 源碼，構造 Markdown 表格選項場景對照現行 regex。

### 實驗執行

現行 `OPTION_MARKER_RE`：`^[ ]{0,2}(?:[A-Ea-e]\.|[1-5]\.)[ \t]+\S`

測試輸入（模擬 PM 表格列選項）：

```
| 方案 | 說明 | 建議 |
|------|------|------|
| A. 繼續 | 直接推進 | Recommended |
| B. 暫停 | 先做分析 | |
| C. 回退 | 重新設計 | |

要選哪個方案？
```

- `OPTION_MARKER_RE` 不命中（行首是 `|` 不是 `A.`）
- `has_option_markers()` 返回 False → 整條選項路徑不命中
- 二元問句路徑也不命中（「要選哪個」不在 `BINARY_QUESTION_KEYWORDS`）

### 實驗發現

1. 現行 hook 對 Markdown 表格形式的選項完全無偵測能力
2. 現行豁免 E3 把 `|...|` 行視為假陽性豁免，與「偵測表格選項」需求衝突
3. 缺口明確，需新增第三條偵測路徑

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### Hook 偵測規格設計

### 1. 新增偵測路徑：Markdown 表格選項（§3.4-bis）

在現行 §3.2（選項標記）、§3.3（二元問句）之外，新增第三條偵測路徑 §3.4-bis（Markdown Table Option Path）。

**必須同時滿足兩條件才算命中**：

| 條件 | 規則 | 理由 |
|------|------|------|
| (A) Markdown 表格存在 | 訊息含 >= 3 行 Markdown 表格資料列（不含標題列與分隔列） | 3 行 = 至少 3 個候選項，對齊現行 §3.2 的「3+ 標記」閾值 |
| (B) 選項語境 | 訊息結尾 400 字內出現 `OPTION_QUESTION_KEYWORDS` 任一，或最後 50 字含問號 | 複用現行 `has_question_ending()` 函式，無需新邏輯 |

**Markdown 表格資料列判定 regex**：

```python
TABLE_DATA_ROW_RE = re.compile(
    r"^\|[^|]+\|",   # 行首 | 後至少一個非 | 字元，再接 |
    re.MULTILINE,
)
TABLE_SEPARATOR_RE = re.compile(
    r"^\|[\s:~-]+\|",  # 分隔列（只含 -、:、空白、~）
    re.MULTILINE,
)
```

計算方式：`table_data_rows = len(TABLE_DATA_ROW_RE.findall(text)) - len(TABLE_SEPARATOR_RE.findall(text)) - header_count`

其中 header_count = min(1, separator_count)（有分隔列則扣 1 列作標題）。

**閾值**：`table_data_rows >= 3`

### 2. 豁免規則調整

#### E3 衝突處理

現行 E3（「pattern 全落在 Markdown table cell 則豁免」）與新路徑衝突。調整方案：

- E3 豁免**僅適用於 §3.2 選項標記路徑**（A./B./C. 落在表格 cell 內時豁免）
- E3 **不適用於 §3.4-bis 表格路徑**（表格路徑的目標就是偵測表格，自豁免無意義）

#### 新增 E6：純資料表格豁免

| 豁免 | 判定規則 | 範例 |
|------|---------|------|
| E6 | 表格欄位名不含「選項/方案/策略/Option/推薦/Recommended」任一關鍵字，且結尾無問句 | 純數據報表（測試結果、效能指標） |

**Why**：PM 經常用表格呈現分析結果（測試覆蓋率、效能指標），這些不是選項，若無豁免會產生大量誤報。

**Consequence**：缺少 E6 會讓 hook 在每次 PM 用表格呈現資料時誤觸發，訓練 PM 忽略提醒（ADR-5 原則：寧可漏報不可誤報）。

**Action**：E6 判定邏輯檢查表格標題列（第一行 `|...|`），不含選項語境關鍵字即豁免。

### 3. 技術可行性評估

#### 3.1 Hook 階段

維持現行 UserPromptSubmit，無需變更。新路徑複用現行 transcript 讀取機制，在 `detect_and_build_output()` 內新增分支。

#### 3.2 實作複雜度

| 維度 | 評估 |
|------|------|
| 新增程式碼量 | ~40-60 行（`has_table_options()` 函式 + E6 豁免 + 整合到 `detect_and_build_output`） |
| 現行函式改動 | `detect_and_build_output()` 新增第三條 `hit_table_path` 分支；E3 加條件限定 |
| 測試新增 | ~6 個測試案例（3 TP + 3 FP） |
| 風險 | 低。新路徑獨立於現行兩條路徑，不影響既有偵測行為 |

#### 3.3 Payload vs Content 檢測

| 方式 | 優點 | 缺點 | 結論 |
|------|------|------|------|
| Content（transcript 文字） | 可偵測純文字回覆中的表格；覆蓋完整 | 需解析 transcript JSONL | **採用**（對齊現行架構） |
| Payload（工具參數） | 精準度高 | PM 列選項不伴隨工具呼叫，payload 不存在 | 不適用 |

維持現行 Content 檢測架構，無需改變。

### 4. 誤報率評估

| 場景 | 觸發？ | 處理 |
|------|--------|------|
| PM 用表格列 3+ 選項 + 問句結尾 | 真陽性 | 命中提醒 |
| PM 用表格呈現測試結果 | E6 豁免（欄位名無選項關鍵字） | 靜默 |
| PM 用表格呈現效能指標 | E6 豁免 | 靜默 |
| PM 用表格列規則對照（含 A./B. 但無問句） | 條件 B 不滿足 | 靜默 |
| PM 用表格列 Ticket 清單 | E6 豁免（欄位名無選項關鍵字） | 靜默 |
| 程式碼 block 內的 Markdown 表格 | 現行 code block 排除 | 靜默 |
| PM 引用規則文件中的表格 | E1（引用既有文件豁免） | 靜默 |
| PM 用表格呈現歷史決策 | E2（歷史回顧豁免） | 靜默 |

**預估誤報率**：基於上述 8 場景分析，E6 + E1 + E2 + code block 排除可覆蓋主要假陽性場景。預估誤報率 < 10%，符合 ADR-5「寧可漏報不可誤報」原則。

**邊界風險場景**：

| 場景 | 風險 | 緩解 |
|------|------|------|
| 表格欄位名含「方案」但實為純分析表 | 低誤報風險 | 條件 B（問句結尾）作二道防線 |
| 表格只有 2 行資料列 | 漏報 | 可接受：2 選項可用二元問句路徑覆蓋 |

### 5. 阻擋 vs 警告策略

**結論：維持 Warning-only（exit 0 + additionalContext）**

**Why**：對齊現行 hook 設計原則（ADR-4）和 PC-064 摩擦力 C 象限定位——增加一步提醒的摩擦足以達到防護效果，阻擋會對正常表格使用場景產生過度摩擦。

**Consequence**：若改為阻擋（exit 2），PM 每次用表格呈現任何資訊都需面對 permission prompt，規則疲勞會導致 PM 繞過或忽略。

**Action**：IMP 實作維持 exit 0，提醒訊息複用現行 `AUQOptionPatternMessages.REMINDER`（已覆蓋「選項列表」和「二元確認」情境描述）。
