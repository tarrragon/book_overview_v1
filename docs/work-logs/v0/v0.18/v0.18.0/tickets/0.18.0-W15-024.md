## 解決方案

### close 條件規則（4 條）

**C1. close 合法理由（枚舉，frontmatter 必填 close_reason）**：

| close_reason | 語意 | 範例 |
|-------------|------|------|
| goal_achieved | 目標已達成 | 正常完成 |
| requirement_vanished | 需求消失（環境變更使 ticket 無意義） | 上游 API 移除 |
| superseded_by | 被上游 ticket 取代（需附 ticket ID） | follow-up 合併 |
| not_executable_knowledge_captured | 無法執行且知識已轉移 error-pattern（需附檔案路徑） | 實驗條件長期不可達 |
| duplicate | 與既有 ticket 重複 | 需附 ticket ID |
| cancelled_by_user | 用戶明示取消 | 需附理由 |

**C2. 禁止的假 close**：

- 「等 data / 等觀察」→ 應 migrate 至下版本或 rescope
- 「暫緩 / 之後再說」→ 應 migrate 或拆 ANA 改設計
- 「算了不重要」→ 應 cancel 並記入決策日誌

**C3. 條件型 ticket 建立時預定義三後果**：

條件觸發型 ticket（如 W15-015「E3 > 0.3 才投入 E2」）建立時必須預定義：

- 觸發 → 執行動作
- 未觸發 → close with `not_executable_knowledge_captured` + 指明 error-pattern 位置
- 未觸發且仍有價值 → rescope 或 migrate 條件

**C4. CLI 驗證**：

`ticket track close <id> --reason <enum>` 必填；retrospective close 允許 `unknown` 但標記 audit。

### 區分 close / migrate / cancel / error-pattern-only 邊界

| 情境 | 動作 | 理由 |
|------|------|------|
| 目標達成 | complete | 正常 |
| 目標達成但被整合 | close superseded_by | 責任轉移到上游 |
| 需求消失 | close requirement_vanished | 外部變更 |
| 條件不足且知識完成 | close not_executable_knowledge_captured | 不拖延 |
| 條件不足但仍有價值 | rescope 或 migrate | 保留路徑 |
| 重複任務 | close duplicate | 歸一 |
| 用戶取消 | close cancelled_by_user | 尊重決策 |
| 僅需記知識不需 ticket | 不建 ticket，直接寫 error-pattern | 節省追蹤成本 |
