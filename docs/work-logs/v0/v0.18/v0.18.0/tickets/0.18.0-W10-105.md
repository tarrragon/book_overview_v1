## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

phase4-decision-enforcement-hook M1 模式對 P4 評估字串敏感，agent 處理含此字串的合法評估章節時撞 hook。雖然 hook 已支援 inline 標記豁免（PC-093-exempt cat:reason 格式），但 category 白名單（baseline-gated/tdd-transition/ticket-tracked/user-override）未明確列在拒絕訊息中，agent 用錯 category 後選擇放棄 inline 走字串繞過路徑。

### 影響範圍

- `.claude/hooks/phase4-decision-enforcement-hook.py`（拒絕訊息設計）
- 所有實作代理人（thyme/parsley/fennel）：撞 hook 後的行為一致性
- 元級影響：agent 字串繞過會在 ticket md 留下 audit trail 缺失，違反 quality-baseline 規則 6

### 相關 Error Pattern

- PC-093（YAGNI 累積反模式）：本 ANA 是 PC-093 的元案例
- PC-094（TD 清單即時校準）：W10-083 觸發本 ANA
- PC-IMP-071（append-log placeholder 重複）：本 ticket md 結構受此 known issue 影響

### Hook 白名單實證（自 commit 6caa521b）

phase4-decision-enforcement-hook 對 inline 標記的 `category` 欄位有硬編碼白名單：

- `baseline-gated`
- `tdd-transition`
- `ticket-tracked`
- `user-override`

非白名單值（如 `pattern-reference`、`note-only`）會被擋。

### 根因初步判斷

agent 失敗的真實根因是「category 白名單不明確」而非「inline 機制不存在」。hook 訊息可能：

1. 未主動列出 4 個合法 category（agent 需自行翻 hook 程式碼）
2. 拒絕訊息不夠精確（agent 看到「不認可」就放棄 inline 改走字串繞過）

### ROI 評估候選方向

| 方案 | 描述 | 預估成本 |
|------|------|---------|
| A. 修 hook 訊息 | 拒絕 inline 標記時列出白名單 4 值 + 各 category 適用情境 | 低（單檔 hook 改訊息）|
| B. 規範 agent 流程 | 在 thyme/parsley/fennel agent definition 補「遇 PC-093 hook 阻擋優先試 inline 標記，白名單見 hook 第 X 行」 | 中（多 agent definition 改動）|
| C. 提取白名單到常數 + 文件 | hook 常數 export 至 SKILL.md / methodology | 中（hook 重構 + 文件同步）|

### 結論預判

方案 A 投資回報率最高（單檔改、收益直接）。方案 B 與 C 為補強。

### 重現實驗摘要

**實驗 1（W10-089.1 thyme #2）**：未嘗試 inline 標記直接走字串繞過（改章節名「Phase 4 評估」→「重構評估結論」+ frontmatter when/why 改「多視角審查」）。Agent 報告「inline 未被認可」描述不準確（git log 5541ff08~8182b1ca 無 PC-093-exempt 字串證明未嘗試）。<!-- PC-093-exempt: ticket-tracked:本行為 W10-105 ANA 元描述 PC-093 模式，已建 W10-107/W10-108 追蹤後續 IMP -->

**實驗 2（W10-083 thyme #1）**：嘗試 inline 標記用 category `pattern-reference`（標準格式）被擋（非白名單），改 `tdd-transition` 通過（commit 6caa521b）。

**白名單實證**：`baseline-gated`、`tdd-transition`、`ticket-tracked`、`user-override`。非白名單值會被擋。

### ROI 方案比較

| 方案 | 內容 | 成本 | 收益 | 風險 |
|------|------|------|------|------|
| A. 修 hook 訊息 | 拒絕 inline 時主動列白名單 4 值 + 各 category 適用情境 + 「優先嘗試 inline」開頭提示 | 低（單檔 hook 改訊息） | 直接解決根因；agent 不需翻 hook 程式碼 | 訊息變長可能擠壓其他訊息空間 |
| B. 規範 agent 流程 | thyme/parsley/fennel agent definition 補「遇 PC-093 阻擋優先試 inline 標記」 | 中（多 agent definition + 持續同步維護） | 上游教育性高 | 違反 DRY，且每新增實作 agent 都需同步；agent definition 已重 |
| C. 提取白名單到常數 + 文件 | hook 常數 export 至 SKILL.md / methodology | 中-高（hook 重構 + 文件同步） | 多處可重用 | 過度抽象；當前 4 值穩定，無需常數化 |
| D. 結案不動作 | 接受 agent 偶選字串繞過 | 零成本 | — | PC-093 同精神反模式累積（規則 6 失敗案例學習應防護） |

### 結論：採方案 A，建立 spawned IMP

**理由**：
1. 方案 A 投資回報率最高（ROI = 收益/成本）：單檔改 hook 訊息，直接解決 agent 認知缺口
2. 方案 B 違反 DRY 且維護負擔重
3. 方案 C 屬過度抽象（白名單 4 值近期穩定）
4. 方案 D 接受 PC-093 同精神反模式累積，與 quality-baseline 規則 6 相悖

**Action**：建立 spawned IMP ticket：擴增 phase4-decision-enforcement-hook 拒絕 inline 標記時的訊息可達性（列白名單 4 值 + 各 category 適用情境 + 「優先嘗試 inline」開頭提示）。
