### 派發前置調查（W10-083 / W10-089.1 對比證據）

兩個並行 thyme agent 處理同性質「Phase 4 評估章節含『Phase 4』字串觸發 PC-093 hook」場景時，採取了不同路徑，可作為 W10-105 ANA 的關鍵對照組：

| Agent | Ticket | 行為 | 結果 |
|-------|--------|------|------|
| thyme #2 | W10-089.1 | 改章節名「Phase 4 評估」→「重構評估結論」+ frontmatter when/why 改「多視角審查」 | 字串繞過成功；agent 報告「inline 標記未被認可」（描述不準確）|
| thyme #1 | W10-083 | 嘗試 inline 標記 `<!-- PC-093-exempt: pattern-reference:... -->`；被擋 → 改 `tdd-transition` 通過（commit `6caa521b`）| inline 標記實際可用，但 category 必須在白名單 |

### Hook 白名單實證（自 commit 6caa521b）

phase4-decision-enforcement-hook 對 inline 標記的 `category` 欄位有硬編碼白名單：

- `baseline-gated`
- `tdd-transition`
- `ticket-tracked`
- `user-override`

非白名單值（如 `pattern-reference`、`note-only`）會被擋。

### 根因初步判斷

agent 失敗的真實根因是「category 白名單不明確」而非「inline 機制不存在」。hook 訊息可能：

1. 未主動列出 4 個合法 category（agent 需自行翻 hook 程式碼）
2. 拒絕訊息不夠精確（agent 看到「不認可」就放棄 inline 改走字串繞過）

### ROI 評估候選方向

| 方案 | 描述 | 預估成本 |
|------|------|---------|
| A. 修 hook 訊息 | 拒絕 inline 標記時列出白名單 4 值 + 各 category 適用情境 | 低（單檔 hook 改訊息）|
| B. 規範 agent 流程 | 在 thyme/parsley/fennel agent definition 補「遇 PC-093 hook 阻擋優先試 inline 標記，白名單見 hook 第 X 行」 | 中（多 agent definition 改動）|
| C. 提取白名單到常數 + 文件 | hook 常數 export 至 SKILL.md / methodology | 中（hook 重構 + 文件同步）|

### 結論預判

方案 A 投資回報率最高（單檔改、收益直接）。方案 B 與 C 為補強。
