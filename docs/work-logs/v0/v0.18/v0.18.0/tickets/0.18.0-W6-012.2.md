## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

**本問題不是「缺乏真實 book ID 來源」，而是「程式碼層 ID 策略順序錯誤導致 privacy ID 即使取到也被 cover-XXX 覆寫」。**

關鍵程式碼位置：

- `src/content/adapters/readmoo-adapter.js:368-381` `extractBookIdFromPrivacy()` — 已實作從 `<div class="privacy" id="privacy-{digits}">` 抽取真實 book ID。
- `src/content/adapters/readmoo-adapter.js:461-561` `parseBookElement()` — 把 privacy ID 放進 `effectiveReaderId`，並用它構建 `effectiveUrl`，繞過 SPA 佔位 URL。
- `src/content/adapters/readmoo-adapter.js:515` `generateStableBookIdWithInfo(effectiveReaderId, title, cover)` — 把 effectiveReaderId（含 privacy ID）傳入 ID 生成器。
- `src/content/adapters/readmoo-adapter.js:871-894` `applyIdGenerationStrategiesWithInfo()` — **策略順序為 cover → title → reader-link，cover 永遠優先**，即使 readerId 是真實 privacy ID 也會被覆寫成 `cover-{coverId}`。

行為觀察：

| 環節 | 預期 | 實機現狀 |
|------|------|---------|
| DOM 抓 privacy element | 取到 `privacy-{digits}` | 假設 H1：DOM 中 privacy element 不存在；假設 H2：格式不符 `^privacy-(\d+)$` |
| effectiveReaderId | 真實 book ID | 若 H1/H2 成立則為空字串 |
| effectiveUrl | `/api/reader/{privacyBookId}` | 若 H1/H2 成立則維持 SPA dummy URL `/api/reader/210017268000101` |
| bookData.id | 真實 book ID | `cover-{coverId}` — **此為 cover 策略勝出，與 privacy 是否取到無關**（程式碼層 bug，已 100% 確認） |

### 影響範圍

- `src/content/adapters/readmoo-adapter.js:871-894` ID 策略順序需調整（cover 策略應降到 fallback）。
- `src/extractors/readmoo-data-validator.js:133-222` `readmoo-cover-url` 驗證規則可能仍合理（cover URL 安全檢查），但 ID 主來源不應為 cover。
- `docs/bookstores/readmoo.md` 觀察點章節：缺 ID 來源優先級表 + privacy element 偵測指引。
- 下游影響：書庫匯出 CSV 的 `id` 欄位（`cover-XXX` 對 Readmoo 帳號內未必穩定，封面圖路徑改版會破壞 ID 穩定性）、跨書城跳轉、後續 sync 機制。

### 相關 Error Pattern

待 query：`/error-pattern query` 主題「ID 策略優先級錯誤」「fallback 策略凌駕主來源」（暫未明確對應）。

### 實驗發現

**已驗證的事實**：

1. 程式碼層已有 privacy element ID 抽取機制（`extractBookIdFromPrivacy()`）。
2. `parseBookElement()` 正確將 privacy ID 放入 `effectiveReaderId` 並重建 `effectiveUrl`。
3. `generateStableBookIdWithInfo()` 策略順序為 **cover → title → reader-link**，cover 永遠優先（這是 W6-012.2 標題描述「id 為 cover-XXX」的直接原因）。
4. `bookData.id = idInfo.id`，由策略順序決定主 ID 欄位。

**仍未驗證的假設**：

- **H1**：Readmoo library 頁 DOM 中可能沒有 `[id^="privacy-"]` 元素（解釋 URL 仍為 dummy）。
- **H2**：privacy element 存在但 ID 格式不符 `^privacy-(\d+)$`。
- **H3**：Readmoo library 載入時 XHR/Fetch API 是否直接返回包含 book ID 的 JSON response（若是，可改用 network interception 取 ID，更穩定）。
- **H4**：SPA 內部 state（Vue/React store / `window.__INITIAL_STATE__`）是否暴露真實 book ID。

**結論**：可立即進行的修復為「調整 ID 策略順序：將 reader-link（含 privacy ID）提升為最高優先，cover 降至 fallback」；此修復覆蓋 80% 場景。H1-H4 實機驗證可進一步確認是否需要 network/state 層補強策略。

| A | 純策略順序調整：reader-link > title > cover (fallback) | privacy element 存在於 DOM（H1 否） |
| B | 增加 network interception，攔截 library 載入 XHR 取 book ID | privacy element 不存在但 API 有（H1 是 + H3 是） |

### 真實 book ID 來源確認

**結論**：Readmoo SPA 在 library 頁的書籍容器 `.library-item` 內已暴露真實 book ID 於 `<div class="privacy" id="privacy-{8位數字}">` 元素。`extractBookIdFromPrivacy()` 已實作完整提取邏輯，無需引入新的 ID 來源（如 network 攔截 / SPA state）。

### 真正需要的修正：ID 策略優先級重排

當前 `src/content/adapters/readmoo-adapter.js:871-894` `applyIdGenerationStrategiesWithInfo()` 策略順序：

```
cover → title → reader-link (with 'unstable-' prefix)
```

**修正後策略順序**：

```
reader-link (privacy ID) → cover → title → fallback
```

理由：

- privacy ID 是 Readmoo 內部穩定 book ID（8 位數字，與 Readmoo 帳號內 reader URL 一一對應），改版風險最低
- cover 路徑（如 `pirited_210x315.jpg`）依 CDN 命名規則生成，封面更新或 CDN 路徑變更會破壞 ID 穩定性（已知反例：W6-010 觀察過部分書籍 cover 是 `openbook.png` 預設圖，coverId 會塌縮為 `openbook`）
- title 易受版本後綴影響（「【二版】」「（增訂版）」）

### 具體實作改動點

1. **修改 `tryReaderStrategy` 返回值**：移除 `unstable-` 前綴，改回 `reader-{id}`（與 fallback 策略名稱對齊；privacy ID 是穩定來源，不應標記為 unstable）。
2. **重排 `applyIdGenerationStrategiesWithInfo` 順序**：reader-link 提升為第一優先，cover 降到第二，title 第三，fallback 維持。
3. **新增單元測試**：覆蓋以下場景
   - privacyBookId 存在 → id 為 `reader-{privacyBookId}`，strategy = `reader-link`
   - privacyBookId 不存在 + cover 存在 → id 為 `cover-{coverId}`，strategy = `cover`
   - privacy 不存在 + cover 是 `openbook.png` → 不採 cover（coverId 為 `openbook` 不穩定）→ fall through 到 title
   - 全部失敗 → fallback hash
4. **更新 `tryCoverStrategy` 增加 known-unstable 過濾**：當 coverId ∈ `['openbook', 'undefined', 'placeholder']` 等通用 fallback 圖片時應視為 null，避免 96 本書共用同一 cover ID。

### 影響評估

| 項目 | 影響 |
|------|------|
| 既有 cover-XXX 已儲存的書籍 | 修復後 ID 會變更為 reader-XXX；需提供資料遷移策略（建議：使用 `identifiers.privacyBookId` 比對既有 cover-XXX 記錄並合併） |
| CSV 匯出 / 跨書城跳轉 | 修復後變穩定，可作為 cross-store unique key |
| 測試 | 需新增 4 個 ID 策略測試案例 |
