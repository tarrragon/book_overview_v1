### Phase 1 設計產出（lavender-interface-designer）

#### 問題根因

**直接原因**（2026-04-24 事件）：
前 session commit `6d0a8fc2` 在 `docs/work-logs/v0/v0.18/v0.18.0/v0.18.0-main.md` 新增「下個 Session 接手 Context」段落並明示「下 session 優先建議：W17-079」，但未執行 `ticket handoff W17-079` CLI，導致 `.claude/handoff/pending/` 目錄為空。下個 session 啟動時 `session-start-scheduler-hint-hook.py` 呼叫 `ticket track runqueue --context=resume --top 3` 無 resume 候選，接手者看不到前 session 明確指定的 ticket，被迫手動輸入 W17-079。

**系統性根因**：雙軌交接機制（worklog 人類敘述軌 + .claude/handoff/pending/ 機器讀取軌）缺乏同步偵測。紀律層（session-switching-sop.md v1.1.0）已補但無自動化防護。PM 在 session 收尾時 context 已重且步驟多（commit、完成勾選、memory、worklog、CLI），僅靠自律易漏 CLI 步驟。

#### 影響範圍

| 面向 | 影響 |
|------|------|
| 功能 | session-start-scheduler-hint-hook runqueue resume 候選為空，PM 無法自動繼承前 session 決策 |
| 使用者體驗 | 接手者需手動重讀 worklog 找 ticket ID，浪費 token 和注意力 |
| 檔案範圍 | .claude/hooks/（新增或擴充 hook）、.claude/skills/ticket/ticket_system/commands/handoff.py（可能新增 --from-worklog 子命令）、.claude/pm-rules/session-switching-sop.md（引用本機制） |

| 自動化程度 | 高（被動偵測 + 提醒） | 高（commit 時強制） | 低（需 PM 記得下指令） | 高（Stop 提醒 → PM 用 CLI 快速落地） |
| 誤報風險 | 中（Stop 觸發頻繁，須條件收斂） | 中（commit 不一定含 worklog handoff 變更） | 無（PM 顯性觸發） | 中（繼承 A 的誤報） |
| 漏報風險 | 低（session 結束必觸發） | 高（PM 可能 commit 前就 /clear） | 高（全靠 PM 自律） | 低 |
| 實作成本 | 中（新 hook + worklog 解析） | 低（擴充既有 commit-handoff-hook.py） | 低（handoff.py 新增子命令 + worklog 解析） | 中（A 的成本 + C 的成本，但 C 可作為 A 的執行工具） |
| 對既有流程干擾 | 低（additionalContext 提醒，不阻塞） | 中（commit 流程已有多個 hook） | 無 | 低 |
| 可測試性 | 高（解析輸入固定，單元測試易） | 中（需 mock commit stdout） | 高（純 CLI 邏輯，單元測試易） | 高 |
| 反諷點 | Stop hook 本身已多（session-context-guard、handoff-reminder），再加一個增加載入成本 | commit 時才提醒已晚，PM 可能已切 session | 需要 PM 記得這個命令存在（又回到自律） | 互補：Stop 提醒是「警示」，CLI 是「一鍵落地」 |
| ROI 分數（1-5） | 3 | 2 | 3 | 5 |

**ROI 說明**：
- A：被動偵測但需新增 hook，誤報需調條件，單獨使用效益中等
- B：commit 時機太晚、漏報高，單獨方案 ROI 低
- C：低成本但依賴自律，與紀律規則重疊無自動化防護，單獨方案 ROI 中
- D：Stop hook 負責偵測（人可能忘記下 CLI），CLI 子命令負責「被提醒後一鍵落地」，兩者互補最小化漏報與誤報

**推薦方案：D（組合 A + C）**

### 2. 偵測關鍵字清單（worklog 解析用）

以下關鍵字擇一出現於 worklog 段落標題或段落首 3 行即視為「交接意圖」：

| 關鍵字類別 | 字串清單 |
|-----------|---------|
| 標題式（段落標題） | `下個 Session 接手 Context`、`下一 Session 接手`、`下 Session 接手`、`接手指引`、`Handoff Context`、`Session Handoff` |
| 建議式（句中） | `下 session 優先建議`、`下個 session 優先建議`、`下一 session 優先建議`、`下 session 優先順序建議`、`下次 session 建議`、`建議下 session` |
| 續行式 | `本 session 結束時.*仍 in_progress`、`未完成清單`、`Spawned 推進清單`（ANA complete 後強制欄位） |

**Ticket ID 提取 regex**（W5、W6、W10、W17 等 Wave ID 格式）：
```
\b(\d+\.\d+\.\d+)[-_](W\d+[-_][\d\w]+(?:\.\d+)?)\b
```

或更寬鬆（容許省略版本前綴）：
```
\b(W\d+[-_][\d\w]+(?:\.\d+)?)\b
```
若只抓到 Wave 短 ID，用當前版本補全（從 `todolist.yaml` 讀 active version）。

### 3. 偵測流程虛擬碼

```
# Stop Hook 主邏輯（方案 A）

def on_session_stop(event):
    # 步驟 1: 找 active version 的 worklog
    active_version = read_active_version_from_todolist()
    worklog_path = find_worklog_path(active_version)
    if not worklog_path.exists():
        return  # 無 worklog 無需檢查

    # 步驟 2: 判斷 worklog 是否在本 session 內被修改
    session_start_time = event.get("session_start_timestamp")
    if worklog_path.mtime < session_start_time:
        return  # 本 session 未動 worklog

    # 步驟 3: 擷取 worklog 最新段落（末 N 行或本 session diff）
    recent_content = extract_recent_content(worklog_path, session_start_time)

    # 步驟 4: 關鍵字偵測
    if not any(kw in recent_content for kw in HANDOFF_KEYWORDS):
        return  # 無交接意圖

    # 步驟 5: 提取 ticket ID 清單
    worklog_ticket_ids = extract_ticket_ids(recent_content)  # regex match

    # 步驟 6: 比對 .claude/handoff/pending/ 現況
    pending_dir = Path(".claude/handoff/pending")
    pending_ticket_ids = {p.stem for p in pending_dir.glob("*.json")}

    missing = set(worklog_ticket_ids) - pending_ticket_ids
    orphan = pending_ticket_ids - set(worklog_ticket_ids)  # CLI 有但 worklog 無

    # 步驟 7: 輸出（四情境）
    if not missing and not orphan:
        return  # 雙軌一致，無需輸出
    output_warning(missing, orphan, suggested_commands)
```

**CLI 子命令（方案 C）邏輯**：

```
ticket handoff --from-worklog [--worklog-path PATH] [--dry-run]

def from_worklog_handoff(args):
    worklog_path = args.worklog_path or auto_detect_worklog()
    content = worklog_path.read_text()
    ticket_ids = extract_ticket_ids_from_handoff_section(content)

    for tid in ticket_ids:
        if (pending_dir / f"{tid}.json").exists():
            print(f"[SKIP] {tid}: 已存在 pending handoff")
            continue
        if args.dry_run:
            print(f"[DRY-RUN] 將執行：ticket handoff {tid}")
        else:
            # 呼叫既有 _execute_handoff 邏輯（或 subprocess ticket handoff <tid>）
            result = execute_handoff(tid)
            print(f"[{'OK' if result == 0 else 'FAIL'}] {tid}")
```

### 5. 測試場景（四情境矩陣）

| 情境 | worklog 關鍵字 | `.claude/handoff/pending/` | 預期行為 |
|------|---------------|---------------------------|---------|
| 1. 雙軌一致 | 含「下 session 優先建議：W17-079」 | 有 `0.18.0-W17-079.json` | 不輸出（正常） |
| 2. worklog 有 / pending 無 | 含「下 session 優先建議：W17-079」 | 空目錄 | 輸出警告 + 建議 CLI |
| 3. pending 有 / worklog 無 | 無關鍵字 | 有 `0.17.0-W5-001.json` | 輸出「孤立」提醒（低優先級） |
| 4. 雙軌皆無 | 無關鍵字 | 空目錄 | 不輸出（正常結束） |

額外邊界情境：

| 邊界 | 預期行為 |
|------|---------|
| worklog 檔案不存在 | 靜默退出（不阻塞 session stop） |
| worklog 關鍵字出現在程式碼區塊內 | 仍視為觸發（簡化邏輯；極少發生） |
| 提取到 Wave 短 ID（如 `W17-079` 無版本前綴） | 用當前 active version 補全 → `0.18.0-W17-079` |
| ticket ID 在 worklog 提到但已 completed | 不視為 missing（查 `load_ticket` 狀態） |
| CLI 執行失敗（hook 無法執行子命令） | 僅輸出建議，不自動執行 |

### 6. 與既有 hook 的整合

**Session-start 側**（`session-start-scheduler-hint-hook.py`）：
- 無需修改；本機制是「前 session 結束時偵測」，非「新 session 啟動時偵測」
- 若補落 CLI handoff 後，下個 session 啟動時 runqueue resume 自然會有候選

**Stop hook 側**：
- 既有 `handoff-auto-resume-stop-hook.py`：處理不同場景（自動 resume），可並存
- 既有 `session-experience-persistence-reminder-hook.py`：類似 Stop hook，不衝突
- **新增 hook 命名建議**：`stop-worklog-handoff-sync-check-hook.py`

**CLI 側**（`handoff.py`）：
- 新增 `--from-worklog` 子命令
- 引用既有 `_execute_handoff` 重用邏輯
- 新增 `ticket_system/lib/worklog_parser.py` 作為 worklog 解析共用模組（供 Stop hook 和 CLI 共用，避免 DRY 違反）

### 8. 風險與邊界聲明

| 風險 | 緩解 |
|------|------|
| Stop hook 載入時間延長 | worklog 解析限制在最新 200 行內，regex 預編譯 |
| 誤報（worklog 舊段落誤觸發） | 用 mtime + session_start_time 過濾，只看本 session 變更 |
| CLI subprocess 執行失敗於 hook 內 | 僅輸出建議，不自動執行（方案 D 精髓：hook 提醒 + CLI 一鍵落地） |
| 與 handoff-auto-resume-stop-hook 重複觸發 | 功能正交：前者處理「自動 resume」，本 hook 處理「雙軌同步偵測」 |
| Session 無 worklog 變更但有 CLI handoff | 屬情境 3「孤立」，輸出低優先級提醒，不阻塞 |

### 9. 推薦方案總結

**方案 D：組合 A（Stop Hook 偵測）+ C（CLI --from-worklog 子命令）**

**理由**：
1. Stop hook 自動偵測解決「PM 忘記執行 CLI」的核心漏報問題
2. CLI 子命令提供「被提醒後一鍵落地」的最低摩擦執行路徑
3. 兩者共用 `worklog_parser.py` 避免 DRY 違反
4. 與既有紀律規則（session-switching-sop.md v1.1.0）形成「紀律 + 自動化」雙層防護
5. 總實作成本約 3.5-4 小時，ROI 最高
