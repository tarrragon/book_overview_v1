### 3. 偵測流程虛擬碼

```
# Stop Hook 主邏輯（方案 A）

def on_session_stop(event):
    # 步驟 1: 找 active version 的 worklog
    active_version = read_active_version_from_todolist()
    worklog_path = find_worklog_path(active_version)
    if not worklog_path.exists():
        return  # 無 worklog 無需檢查

    # 步驟 2: 判斷 worklog 是否在本 session 內被修改
    session_start_time = event.get("session_start_timestamp")
    if worklog_path.mtime < session_start_time:
        return  # 本 session 未動 worklog

    # 步驟 3: 擷取 worklog 最新段落（末 N 行或本 session diff）
    recent_content = extract_recent_content(worklog_path, session_start_time)

    # 步驟 4: 關鍵字偵測
    if not any(kw in recent_content for kw in HANDOFF_KEYWORDS):
        return  # 無交接意圖

    # 步驟 5: 提取 ticket ID 清單
    worklog_ticket_ids = extract_ticket_ids(recent_content)  # regex match

    # 步驟 6: 比對 .claude/handoff/pending/ 現況
    pending_dir = Path(".claude/handoff/pending")
    pending_ticket_ids = {p.stem for p in pending_dir.glob("*.json")}

    missing = set(worklog_ticket_ids) - pending_ticket_ids
    orphan = pending_ticket_ids - set(worklog_ticket_ids)  # CLI 有但 worklog 無

    # 步驟 7: 輸出（四情境）
    if not missing and not orphan:
        return  # 雙軌一致，無需輸出
    output_warning(missing, orphan, suggested_commands)
```

**CLI 子命令（方案 C）邏輯**：

```
ticket handoff --from-worklog [--worklog-path PATH] [--dry-run]

def from_worklog_handoff(args):
    worklog_path = args.worklog_path or auto_detect_worklog()
    content = worklog_path.read_text()
    ticket_ids = extract_ticket_ids_from_handoff_section(content)

    for tid in ticket_ids:
        if (pending_dir / f"{tid}.json").exists():
            print(f"[SKIP] {tid}: 已存在 pending handoff")
            continue
        if args.dry_run:
            print(f"[DRY-RUN] 將執行：ticket handoff {tid}")
        else:
            # 呼叫既有 _execute_handoff 邏輯（或 subprocess ticket handoff <tid>）
            result = execute_handoff(tid)
            print(f"[{'OK' if result == 0 else 'FAIL'}] {tid}")
```

### 5. 測試場景（四情境矩陣）

| 情境 | worklog 關鍵字 | `.claude/handoff/pending/` | 預期行為 |
|------|---------------|---------------------------|---------|
| 1. 雙軌一致 | 含「下 session 優先建議：W17-079」 | 有 `0.18.0-W17-079.json` | 不輸出（正常） |
| 2. worklog 有 / pending 無 | 含「下 session 優先建議：W17-079」 | 空目錄 | 輸出警告 + 建議 CLI |

| 4. 雙軌皆無 | 無關鍵字 | 空目錄 | 不輸出（正常結束） |

額外邊界情境：

| 邊界 | 預期行為 |
|------|---------|
| worklog 檔案不存在 | 靜默退出（不阻塞 session stop） |
| worklog 關鍵字出現在程式碼區塊內 | 仍視為觸發（簡化邏輯；極少發生） |

| ticket ID 在 worklog 提到但已 completed | 不視為 missing（查 `load_ticket` 狀態） |
| CLI 執行失敗（hook 無法執行子命令） | 僅輸出建議，不自動執行 |

### 6. 與既有 hook 的整合

**Session-start 側**（`session-start-scheduler-hint-hook.py`）：
- 無需修改；本機制是「前 session 結束時偵測」，非「新 session 啟動時偵測」
- 若補落 CLI handoff 後，下個 session 啟動時 runqueue resume 自然會有候選

**Stop hook 側**：
- 既有 `handoff-auto-resume-stop-hook.py`：處理不同場景（自動 resume），可並存
- 既有 `session-experience-persistence-reminder-hook.py`：類似 Stop hook，不衝突
- **新增 hook 命名建議**：`stop-worklog-handoff-sync-check-hook.py`

**CLI 側**（`handoff.py`）：
- 新增 `--from-worklog` 子命令
- 引用既有 `_execute_handoff` 重用邏輯
- 新增 `ticket_system/lib/worklog_parser.py` 作為 worklog 解析共用模組（供 Stop hook 和 CLI 共用，避免 DRY 違反）

### 8. 風險與邊界聲明

| 風險 | 緩解 |
|------|------|
| Stop hook 載入時間延長 | worklog 解析限制在最新 200 行內，regex 預編譯 |

### 9. 推薦方案總結

**方案 D：組合 A（Stop Hook 偵測）+ C（CLI --from-worklog 子命令）**

**理由**：
1. Stop hook 自動偵測解決「PM 忘記執行 CLI」的核心漏報問題
2. CLI 子命令提供「被提醒後一鍵落地」的最低摩擦執行路徑
3. 兩者共用 `worklog_parser.py` 避免 DRY 違反
4. 與既有紀律規則（session-switching-sop.md v1.1.0）形成「紀律 + 自動化」雙層防護
5. 總實作成本約 3.5-4 小時，ROI 最高
