### 實驗方法

工具：chrome-devtools-mcp（已 install_extension 完成，Extension ID `jjojfonkallmalpccagibhbhbbepjkom` v0.18.0 Enabled）。
觀察點：`list_console_messages` 過濾 `WAIT_FOR_BOOK_ELEMENTS_TIMEOUT` / `WAIT_FOR_BOOK_ELEMENTS_FOUND` / `BOOK_CONTAINERS_FOUND`；
`evaluate_script` 查 `document.querySelectorAll('.library-item').length` 與 `window.location.href`。

四情境 budget 上限各 10 tool calls。實際使用：S1=5、S2=0（無資源，N/A）、S3=0（permission denied，N/A）、S4=6、Baseline reload=3。

### 實驗執行

**情境 1：未登入狀態（isolatedContext 無 Readmoo cookie）**
- 步驟：`new_page url=https://read.readmoo.com/library isolatedContext=scenario1_nologin`
- 結果（無 hash 路徑）：S3 XML `AccessDenied` 回應，host 仍為 `read.readmoo.com`，但 body 為錯誤頁，無 `.library-item`。
- 接著 `navigate_page url=https://read.readmoo.com/#/library`：**redirect 至 `next.readmoo.com/zh-TW/auth/signin`**（OAuth login flow）。
- 關鍵發現：host 變為 `next.readmoo.com`，**content script (matches `read.readmoo.com`) 完全未注入**，`hasContentScript: false`，`waitForBookElements` **從未執行**。
- 判定：S1 **不觸發** WAIT_FOR_BOOK_ELEMENTS_TIMEOUT（content script 不在該 host）。

**情境 2：空書庫（無測試資源）**
- 步驟：無 0-book 測試帳號可用；亦無 cookie 控制權限可模擬「登入但空書庫」狀態。
- 替代評估：依 readmoo-adapter.js:241 邏輯，若 `getBookElements()` 全程回傳 `[]`，3000ms timeout 後會走 `finalElements.length === 0` 分支 → 觸發 **warn 級別 WAIT_FOR_BOOK_ELEMENTS_TIMEOUT**。
- 判定：S2 **N/A（無法實機驗證），但根據程式碼推論為唯一已知會觸發 finalCount:0 warn 的情境**。

**情境 3：節流網路（Slow 3G）**
- 步驟：嘗試 `mcp__chrome-devtools__emulate networkConditions=Slow 3G`。
- 結果：**Permission denied**（tool 未授權使用），無法注入 throttle。
- 判定：S3 **N/A（mcp 權限受限）**。

**情境 4：SPA 路由切換（dashboard → library）**
- 步驟：在已 hydrate 的 library 頁（928 items）執行 `window.location.hash = '#/shelves'`（實際 redirect 到 `#/dashboard`）；hash 切回 `#/library`。
- t=500ms：96 items；t=2000ms：96；t=5000ms：96（穩定）。
- Console 觀察：
  - `GET_BOOK_ELEMENTS_CALLED` 來自 `waitForBookElements` (content-modular.js:2361)，`directLibraryItemCount: 96`。
  - `WAIT_FOR_BOOK_ELEMENTS_FOUND` source=mutation, count=96（**未走 timeout 路徑**）。
  - 0 筆 warn 級別 WAIT_FOR_BOOK_ELEMENTS_TIMEOUT。
- 判定：S4 **不觸發** finalCount:0 timeout；但揭露另一問題：SPA 路由切換後僅渲染 96/944 items（lazy load 未捲動觸發），屬不同議題（incomplete extraction），不在 W6-012.9.4 範圍。

**Baseline（reload `#/library`，logged in，944 books）**
- t=0：兩次 poll directLibraryItemCount: 0（SPA hydration window）→ FALLBACK_SELECTOR_ATTEMPT、LAST_RESORT_STRATEGY 觸發。
- t≈90ms：第三次 poll directLibraryItemCount: 96，MutationObserver 命中。
- `WAIT_FOR_BOOK_ELEMENTS_FOUND source=mutation count=96`。
- **無 WAIT_FOR_BOOK_ELEMENTS_TIMEOUT log**。
- 結論：正常 reload 情境下 SPA hydration 在 ~90ms 完成，遠低於 3000ms timeout，MutationObserver 機制有效。

### 實驗發現

**已驗證事實**：
1. v0.18.0 當前 `waitForBookElements` 預設 3000ms timeout + MutationObserver 在「登入 + 有書 + 正常網路」條件下**從未觸發** finalCount:0 warn。
2. 未登入會導致 host 變為 `next.readmoo.com`，**content script 完全不執行**，與 timeout 無關。
3. SPA 路由切換（dashboard→library）渲染速度 < 500ms 完成首屏 96 items，亦不觸發 timeout。
4. SPA 首屏只渲染 96/944 items（疑似 lazy load / 虛擬列表），是**另一個 incomplete extraction 議題**，與 finalCount:0 timeout 無因果關係。

**仍未驗證假設**：
1. 空書庫（0 books）情境—只能依程式碼推論會觸發 warn finalCount:0，無實機資料。
2. Slow 3G 節流網路—需 `emulate` 權限，當前 mcp session 受限。

### 方案 D 評估

**方案 D 原始假設**：finalCount:0 timeout 在多種情境發生 → 需調整 wait 策略（拉長 timeout、改用 long-polling、或重試）。

**實機數據結論**：
1. **登入 + 有書 + 正常網路**：MutationObserver 在 ~90ms 內命中，3000ms timeout 完全綽綽有餘。
2. **登入 + SPA 路由切換**：< 500ms hydration，亦不觸發 timeout。
3. **未登入**：content script 不執行，與 timeout 機制無關（屬另一個 UX 議題：popup 在登入頁應顯示提示而非試圖提取）。
4. **唯一可能觸發 finalCount:0 warn 的情境**：登入但 0 book 帳號（程式碼推論，未實機）。

**判定：方案 D 不值得在當前範圍實施。**

理由：
- finalCount:0 timeout warn 在 v0.18.0 build 的常見使用情境下實際不發生。
- 唯一推論會觸發的情境（空書庫）屬「正常結果」而非「失敗」，當前程式碼路徑回傳空 `[]` 並非 bug，warn 級別 log 可降級或加 context。
- 若用戶報告 finalCount:0 warn，需先收集 user agent / 是否 0 book 帳號 / 是否 Slow 3G，再決定是否拉長 timeout。盲目調整策略缺乏實證支撐。
