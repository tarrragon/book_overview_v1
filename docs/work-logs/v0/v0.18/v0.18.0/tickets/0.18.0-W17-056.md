## Problem Analysis

### 問題根因

代理人無法載入 SKILL（平台限制），`compositional-writing` 與 `document-writing-style` 規範無落地執行者；多次出現違反禁用字、亂碼、資訊優先序的代理人產出（PC-072 charset contamination、PC-080 WRAP A 選項不完備、PC-074 繁簡共用字檢查等）。規則本身正確但缺少多輪驗收機制，生成式犯錯難以靠規則單一通道避免（codex 實驗驗證）。因此需建立類似 linux 的常駐文字品質審查委員 agent，以多輪驗收取代一次完美的期待。

- `## 適用情境`：強制觸發（情境 C/D/F/G）vs 選用觸發分開列出

每項職責含：執行步驟（4-5 步）、Why/Consequence/Action 三明示。

1. 職責一：三明示結構驗證（含四項檢查清單）

4. 職責四：亂碼與字元集污染偵測
5. 職責五：正面陳述審查

#### 輸出格式章節（§4.6）

完整模板含：審查標的 / Critical 表 / Warning 表 / Info 表 / 全文風險總結 / 修正優先序 / basil 自我審查。

#### 與其他代理人邊界（§4.7）
