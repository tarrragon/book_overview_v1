### 問題根因

W10-114 重構評估發現 track_dashboard.py 跨模組 import track_runqueue 的私有函式（`_foo` 命名），破壞「`_` = 模組私有」Python 慣例。原評估標註為 TD-1 但屬「需動 4+ 檔案」範圍，因此另開本 ticket 作 follow-up 決策化。

### 影響範圍

跨模組 private function import 完整 inventory（grep 證據）：

| Consumer | Imported private functions | 文件化現狀 |
|----------|---------------------------|-----------|
| `commands/track_dashboard.py:34` | `_compute_readiness, _filter_by_wave, _get_pending_handoff_info, _is_unblocked_pending, _priority_rank` | 已在 module docstring lines 16-21 明示「複用既有」 |
| `commands/resume.py:52` | `_priority_rank` 單一函式 | 無註解，僅 import 行 |

**確認單純 cross-module 而非系統性**：

**Rule of Three 對照**：當前 2 consumers，未達抽出共用模組的 rule-of-three 觸發點。

### 實驗發現

| Rule of three（3 consumer 觸發抽出）未滿足 | 當前 2 consumer，差 1 |

#### 仍未驗證假設

| 假設 | 驗證狀態 |

#### 結論

跨模組 `_` 命名違規屬「同 package 內部 sibling-module 存取」，Python 慣例允許套件內部默契放寬（如 stdlib 廣泛存在類似模式）。當前 2 consumer 處於 rule-of-three 邊界內，**抽出共用模組屬 premature abstraction**。

### 結論：採方案 B（維持現狀 + 補文件 + 設 trigger）

#### 推薦理由

1. **Rule of three 未滿足**：2 consumer 抽出屬 premature abstraction（Refactoring 第二版第 1 章原則）
2. **介面未穩定**：track_dashboard.py W10-114 剛上線，函式介面演化中；抽出後易分叉
3. **同 package internal sibling-module access 屬 Python 默契可接受**：stdlib 廣泛存在類似模式
4. **track_dashboard.py 已自我文件化**：只需補 resume.py 註解 + track_runqueue.py trigger 標記
