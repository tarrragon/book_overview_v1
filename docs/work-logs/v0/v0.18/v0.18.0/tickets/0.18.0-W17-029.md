## Solution

## 完整 WRAP 分析（先行）

> 執行 `wrap-decision` skill：本 ticket 是 INV / Analysis 類型，且會影響 PM→agent 派發流程與後續規則設計；不能只用 claim 簡化三問，需跑完整 WRAP（錨點確認、Step 0、W/R/A/P、tripwire）。

### Anchor Check（錨點確認）

**客戶 / 使用者**
- 主要客戶：PM 與被派發的 subagent。PM 需要低摩擦、低重複的派發流程；subagent 需要完整但不爆量的任務上下文。
- 次要客戶：ticket 系統維護者。派發規則、template 與 batch 機制若落地，會變成長期維護面。

**核心目標**
- 不是「一定要做 batch dispatch」，而是降低 PM 派發成本，同時維持 subagent 首次成功率與可回收性。
- D 方向是 W17-001 的控制組 / 反向選項：若 A/B/C 已降低摩擦，D 可不做；若 T3 顯示 PM 開始拼單避開派發成本，D 需重新升級。

**流程價值判斷**
- 影響開發效率：是。PM 派發 prompt 重複、上下文補料與多次重派會直接吃掉迭代時間。
- 不做的代價：缺少 D 方向基線，W17-001 的 T1/T2/T3 tripwire 無法判定「派發成本是否高到需要減少派發次數本身」。
- 進入 Step 0：成立。

### Step 0 — Data Sufficiency Gate（資料充足度）

**目前資料不足以直接決定「做 batch」或「不做 batch」**。已知資料只足以確認 W17-001 漏追 D 方向；尚不足以判斷 D 的 ROI。

需要補齊的資料：
1. **派發頻率基線**：W10-W17 近 3 月每 wave / 每類任務 / 每代理人的派發次數。
2. **重複 prompt 證據**：派發 prompt 中哪些片段重複，重複率與差異點是什麼。
3. **batch 安全性證據**：哪些任務同代理人、同類型、檔案 scope 無重疊、可共享 context；哪些會因合併踩 PC-042 / PC-104。
4. **A/B/C 已落地效果**：W17-002（auto Context Bundle）、W17-010（exit status / NeedsContext）與 W17-003（dispatch-check）對 D 的替代程度。

Step 0 結論：
- 先做資料蒐集與可行性評估，不進入實作。
- 本 ticket 的 acceptance 應以「輸出基線 + 場景矩陣 + template 候選 + A/B/C/D 關係 + 建議」作為完成標準。

### W — Widen Options（擴增選項）

先寫出隱性假設：**「PM 派發成本高，是因為單次派發 prompt 組裝成本與重複上下文成本過高」**。此假設未驗證，因此選項必須跨不同根因，而不是全部指向新增 batch 工具。

候選方向：
1. **D0 不做 batch，只保留控制組觀察**
   若 A/B/C 已讓派發成功率與補料成本達標，D 只留下 tripwire 與統計基線。
2. **D1 只做 template，不做 batch dispatch**
   把高重複 prompt 片段抽成模板，PM 仍逐一派發；降低組裝成本，但保留任務隔離。
3. **D2 batch dispatch 作為人工流程規則，不做工具**
   定義何時可同一輪派多個 agent / 同代理人多任務，但不新增 CLI；適合先試水溫。
4. **D3 擴展 ticket tooling，提供 batch dispatch 或 dispatch plan 生成**
   從 `.claude/skills/ticket/` 既有 batch-create 經驗延伸，但需避免把 ticket 批量建立誤認為 agent 批量執行。
5. **D4 與 C 合併：auto Context Bundle + template**
   由 W17-002 產生 Context Bundle，template 只負責穩定框架；不追求減少派發次數。
6. **D5 強化 A/B/C，不採 D**
   若根因其實是上下文缺料、exit status 不清或 dispatch-check 未到位，應投資 A/B/C，而非 batch。

反向問題：
- 若「減少派發次數」本身是錯誤目標，真正問題可能是派發單位切得不穩、Context Bundle 缺欄位、或 agent exit status 不足。
- 若 batch 讓單次任務變大，可能提升 PM 表面效率，但降低 subagent 成功率與錯誤定位能力。

### R — Reality Test（實境檢驗）

需要以 repo 內資料逐項核對，不能靠直覺判斷 batch/template 候選。

**基線統計測試**
- 搜尋 W10-W17 ticket / docs 中的 dispatch 記錄、Agent 派發語句、subagent 任務描述、parallel dispatch 規則引用。
- 產出表格：wave、ticket、agent 類型、任務型態、檔案 scope、是否可共享 context、是否可能 batch-safe。

**template 候選測試**
- 對 `.claude/agents/`、`.claude/pm-rules/parallel-dispatch.md`、W10-W17 票據中的派發片段做重複片段比對。
- 高價值 template 必須同時符合：
  - 重複率高；
  - 差異點集中在 ticket id / files / acceptance / context bundle；
  - 不會把任務特有判斷藏進通用模板。

**batch-safe 測試**
- batch 只可考慮同代理人、同任務型態、可共享背景、檔案 scope 無重疊或有明確 ownership 的任務。
- 排除條件：
  - 任務需要不同 agent 專長；
  - 任務之間有順序依賴；
  - 任一子任務可能超過 PC-042 tool-call 預算；
  - 任務邊界不清，會觸發 PC-104 execution boundary misjudgment；
  - 合併後失敗時無法清楚回收或重派。

**最小試水溫（Ooch）**
- 不先做工具。先抽 10-20 筆近期 dispatch，人工分類「可 batch / 可 template / 不適合」。
- 若人工分類無法穩定一致，代表 batch 規則還不成熟，不應進入實作。

**Steelman**
- 反對 D 的最強理由：batch 會犧牲任務隔離、增加 context 體積、讓失敗回收更困難；若 A/B/C 已解決上下文問題，D 可能是過度工程。
- 支持 D 的最強理由：若 PM 大量重複相同派發骨架，template 可低成本降低摩擦；若同質小任務很多，batch / dispatch plan 可降低切換成本。

### A — Attain Distance（拉開距離）

**機會成本**
- 做 batch 工具的成本高於 template / 分析規則：需要 CLI 或 PM workflow 支援、錯誤回收規則、agent ownership schema、測試與文件維護。
- 若現在直接做 D3，可能排擠 W17-003 dispatch-check、W17-010 NeedsContext 協議、W17-002 auto Context Bundle 的收斂驗證。

**短期可逆決策**
- 先做統計與 template 候選清單是低風險、可逆。
- 先落地工具是高風險、較不可逆，需等基線證據支持。

**距離後判斷**
- 本 ticket 不應以「是否能做 batch」收尾，而應以「在什麼條件下值得做、先做哪一層」收尾。
- 預設排序：D0/D1/D2 優先於 D3；除非資料顯示 batch-safe 任務比例很高且 PM 拼單現象已出現。

### P — Prepare to Be Wrong（準備犯錯）

**Premortem**
- 失敗模式 1：派發記錄不完整，導致頻率基線低估。
- 失敗模式 2：把 ticket batch-create 誤當 agent batch dispatch，混淆工具能力與流程能力。
- 失敗模式 3：template 抽象過度，變成另一個 PM 必須維護的儀式。
- 失敗模式 4：batch 規則忽略 PC-042 / PC-104，讓 subagent 任務變大但成功率下降。
- 失敗模式 5：A/B/C 的改善效果尚未穩定，就過早把 D 判定為需要或不需要。

**Tripwires**
- 若 W10-W17 樣本中 **batch-safe 任務 < 20%**：不做 batch dispatch，最多做 template / dispatch checklist。
- 若高重複 prompt 骨架 **> 50%**，但檔案 scope / acceptance 差異大：做 template，不做 batch。
- 若出現 W17-001 T3「PM 開始拼單合併以避免派發」且一週多次：升級 D2/D3 評估。
- 若 A/B/C 完成後 3 個月內重派發或補料成本仍未下降：重新打開 D 方向，檢查 template + auto PCB 混合方案。
- 若 batch 試點造成 agent needs_context / partial_success 增加：停止 batch，回退到單任務派發 + template。

### 後續執行計畫（對應 acceptance）

1. 產出 W10-W17 派發頻率基線：用 `rg` / `git log` 盤點 dispatch 記錄與 ticket 內派發語句。
2. 建立 batch feasibility matrix：依 agent 類型、任務型態、scope ownership、依賴關係、PC-042 / PC-104 風險分類。
3. 建立 template candidate list：列出重複 prompt 片段、可參數化欄位、風險與建議落點。
4. 建立 A/B/C/D 關係表：標出 D 與 W17-002、W17-003、W17-010 的替代 / 協同 / 前置關係。
5. 給出落地建議：不做 / 做 template / 條件式 batch / 做工具，並附 tripwire 條件。

### AC2 — Batch Dispatch 可行場景

**適合 batch / dispatch-plan 的任務形態**

| 場景 | 可行性 | 例子 | 條件 |
|------|--------|------|------|
| 同質 child tickets | 高 | W10-073 11 張 WRAP 拆分分析、W10-074 14 張 writing skill 子任務、W17-016 5 張 body schema 子任務 | 同 parent、同 type、互不依賴、where.files 無交集 |
| 文件 / reference 批次改寫 | 高 | W10-074.3~.7、W12-002.4.x、W17-008.4/.6 | thyme-documentation-integrator；可共享背景；每張 ticket 改不同檔 |
| 機械性程式清理 | 中 | W10-054.3.1 subprocess encoding、W14-009 local re-import、W17-013 dead flag | thyme-python-developer；需精準檔案 ownership；不共享同一核心檔 |
| CLI 子命令群 | 中 | W10-017.x、W17-008.x、W17-011.x、W17-027.x | 若都改 `track.py` / parser 註冊則不適合真並行；可 batch 建 dispatch-plan，但執行需序列或拆 ownership |
| Read-only 多視角分析 | 高 | parallel-evaluation、WRAP analysis、linux/saffron/bay/pepper 類審查 | 不修改檔案；可同時派多個分析視角 |
| 驗證 / audit 類 | 高 | acceptance-auditor、project-compliance、test report | 明確 SOP、只讀或產出報告，不互相依賴 |

**不適合 batch 的場景**
- 高設計耦合：如 W17-001/W17-009/W17-028 類架構分析，batch 會讓決策上下文互相污染。
- 多 ticket 共享同一核心檔：例如多個 CLI 子任務同時改 `.claude/skills/ticket/ticket_system/commands/track.py`。
- `.claude/` 主 repo 共享 index 場景：不能靠 worktree 隔離時，batch commit 風險高，需精準 staging 或 PM 統一 commit。
- 需要前後序依賴的落地鏈：例如先有 Context Bundle extractor，再 wire-in，再文件化。
- 任一子任務可能接近 PC-042 tool-call 上限，或任務邊界可能觸發 PC-104。

**適合的代理人類型**
- **thyme-documentation-integrator**：最適合 template / docs / reference 批次，風險低。
- **thyme-python-developer**：適合機械性、檔案 ownership 清楚的 batch；不適合多子任務同改 parser / lifecycle 核心。
- **basil-hook-architect**：適合 hook 子模組批次，但需要明確 hook 檔案邊界。
- **acceptance-auditor / project-compliance-agent / saffron-system-analyst / linux / bay-quality-auditor**：適合 read-only batch analysis / verification。
- **rosemary-project-manager**：適合產生 dispatch-plan 與整合，不適合作為「把多個實作壓成一個巨大 prompt」的替代。

### AC3 — Template 化候選項

## Linux 角度再評估（2026-04-22 補充）

### Linux 對應模型

Linux 不以「batch exec syscall」解決大量任務啟動問題。它保持 `fork/exec/wait/signal` 等 primitive 小而穩定，將批量組合交給 shell / `make` / `xargs -P` / systemd / scheduler 這些上層 orchestration。這個模型的重點不是把多個 process 合成一個大 process，而是：

- process 邊界清楚：PID、exit status、signal、wait、資源限制都可獨立處理。
- PCB 完整：task metadata、cwd、fd、env、credentials、scheduling info 在啟動前就準備好。
- argv 精簡：`execve()` 只放必要命令與參數，不把全部上下文塞進 command line。
- scheduler / runqueue 負責「下一個 runnable task」，不負責合併 task。
- orchestration layer 負責 dependency graph、並行度與重試策略。

### 對本專案的修正解讀

| Linux 機制 | 本專案對應 | 目前狀態 |
|------------|------------|----------|
| PCB / task_struct | Ticket frontmatter + Context Bundle | W17-002 已落地 auto extraction |
| argv/envp | Agent prompt | 仍需 snippet / template 降低重複組裝 |
| signal | NeedsContext | W17-010 已落地 schema |
| wait / exit status | Exit Status / PM 收斂 | W17-010 已落地；W17-033 待落地 agent 自主 complete |
| runqueue / scheduler | `ticket track runqueue` / handoff | W17-011、W17-027、W17-031 持續整合 |
| shell / make / systemd | dispatch-plan template | 尚未落地；應由 W17-035 評估 |

因此，Linux 類比支持更精確的結論：**不要做 `batch dispatch CLI`，要做 Makefile-like dispatch-plan + prompt snippets**。

### 為何 dispatch-plan 比 batch dispatch CLI 更符合 Linux

`batch dispatch CLI` 類似新增一個 `batch_execve()`：一次把多個任務塞給 runtime。這會弱化 ticket / agent 的獨立生命週期，增加 PC-042、PC-092、PC-104 風險。

`dispatch-plan` 則類似 Makefile / systemd unit graph：描述多個 job 的依賴、ownership、context source、run mode 與 commit policy，但仍保留每個 job 的獨立 ticket / agent / exit status。
