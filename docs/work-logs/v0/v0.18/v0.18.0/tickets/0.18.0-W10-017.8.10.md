## ANA 結論 (PM 前台分析, 2026-04-19)

### 評估標的
`_query_in_progress_tickets()` 透過 `subprocess.run(["ticket", "track", "list", ...])` 呼叫 ticket CLI，依賴 CLI JSON 輸出格式。CLI schema 變更會靜默破壞此函式（bay C3 視角發現）。

### 4 視角共識（Phase 4 bay C3）

> 「目前 CLI 和此模組由同一 ticket_system 維護，格式變更風險可控。長期可考慮 library-level API 替代 CLI 呼叫」

### 替代方案評估

| # | 方案 | 收益 | 成本 | 風險 |
|---|------|------|------|------|
| 1 | 保持現狀（subprocess 呼叫 CLI） | 零成本，進程隔離 | CLI schema 變更靜默破壞 | 中（同套件可控） |
| 2 | 改 library API（直接 import ticket_system.cli 函式） | 移除 subprocess overhead + 編譯期型別檢查 | 需要 CLI 提供 library-level API；失去進程隔離 | 中（CLI 改重構時連鎖） |
| 3 | 加 schema 版本檢查（CLI 加版本欄位 → 函式驗證） | 顯式契約 + 早期失敗 | 需要 ticket CLI 配合修改 | 低（增量改善） |
| 4 | wrapper 隔離（封裝 `_ticket_query_client.list_in_progress_tickets()`） | 明確 CLI dependency boundary | 新增模組 | 低 |

### ANA verdict

**選方案 1（保持現狀）+ 後續長期防護 spawned Ticket（方案 3 或 4）**

理由：
1. **drift 風險可控**：ticket CLI 與 checkpoint_state.py 在同一 `ticket_system` 套件內維護，CLI schema 變更時開發者會自然注意此 caller
2. **JSON 解析已有防護**：函式內部以 try/except 處理 JSON parse 錯誤（DATA_SOURCES table 的 SAFE_CALL fail-open），不會造成系統崩潰
3. **CLI 改變頻率低**：ticket-system v1 已穩定，schema 變更為低頻事件
4. **方案 2 失去進程隔離**：直接 import 會繼承 CLI module-level 副作用（如 logger 初始化），不利測試
5. **方案 3/4 收益實質但需跨模組協調**：適合長期 spawned Ticket（不在當前 017.8 範圍）

### Spawned Ticket 建議

建議後續建立框架級 Ticket（ticket-system 改善，超出 017.8 範圍）：
- **建議題目**：`ticket CLI list 子命令加 JSON schema 版本欄位 + checkpoint_state.py 加版本驗證`
- **理由**：明確契約、早期失敗、方案 3 的具體落地
- **優先級**：低（drift 風險可控時不急迫）
- **觸發條件**：當 ticket CLI 實際發生過 schema 變更導致 checkpoint_state.py 出錯一次後

當前 ANA 結論不立即建 spawned Ticket（依「條件觸發才建」原則），但記錄此建議供未來參考。

### 短期建議（不開新 Ticket）

- 依賴 ticket CLI `list --status` JSON 輸出 schema
- 失敗時 fail-open（記錄 errors，不阻塞主流程）
- 長期防護見 ANA 017.8.10 spawned Ticket 建議
