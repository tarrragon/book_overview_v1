## Task Summary

評估 handoff_status_for(state, caller: str) 簽名是否升級為 dataclass/strategy 避免 caller-string dispatch 退化

---

## Problem Analysis

### 問題根因

`handoff_status_for(state, caller: str)` 使用字串 dispatch 模式：caller 參數為 `str`，透過 `_HANDOFF_STATUS_ROUTES: Dict[str, _HandoffStatusBuilder]` dict 路由到對應 builder 函式。

linux L2 視角指出：表面上消除了 if-elif，但本質仍是 caller-string dispatch——新增第 4 caller 時必須同步修改 `CheckpointCaller` Literal + `_HANDOFF_STATUS_ROUTES` dict + 實作對應 builder 函式，三處散落無編譯期保證。

### 影響範圍

- `checkpoint_view.py`：`handoff_status_for` + `_HANDOFF_STATUS_ROUTES` + 3 個 builder 函式
- `checkpoint_state.py`：`CheckpointCaller` Literal 定義
- 呼叫端：`_build_ready_check_items`、`render_ready_check`、CLI commands 層

目前僅 3 caller（snapshot / handoff-ready / checkpoint-status），擴展頻率極低。

### 對照 PC-088 結果

PC-088 核心是「LLM 對單步敏感但對路徑總步驟盲」的估算偏誤。本案的 PC-088 對偶是：

- linux L2 將 dict-routing 歸類為「偽裝」if-elif，暗示需升級為 dataclass/strategy
- 但升級本身是「每步感覺精緻，總步驟更多」的路徑：引入 Protocol/ABC + 3 個策略類別 + 註冊機制 + 型別檢查配置
- 實際收益（新增 caller 時的安全性）需對照實際擴展頻率

結論：dict-routing 已經是正確的短路徑。linux L2 指出的「偽裝」問題在 W10-017.13 AC1 已用 dict dispatch 解決（消除 if-elif），且 `_VALID_CALLERS` 從 `get_args(CheckpointCaller)` 派生確保單一 source of truth。PC-088 對偶模式在此案例中不成立——現行 dict-routing 才是真正的短路徑。

### 相關 Error Pattern

對照 PC-088（LLM tool selection 偏誤），本案結論為對偶不成立：dict-routing 即正確短路徑。

### 三方案評估表

| 方案 | 優點 | 缺點 | 適用時機 |
|------|------|------|---------|
| A. Dataclass tag（CallerConfig dataclass 攜帶路由邏輯） | 型別安全、IDE 自動補全、新增 caller 編譯期保證 | 過度工程（3 caller + 極低擴展頻率）；需重構所有呼叫端從 str 改 CallerConfig | caller > 5 且各 caller 攜帶多個配置欄位 |
| B. Strategy pattern（Protocol/ABC + 3 策略類別） | OCP 完美遵循、可獨立測試每個策略 | 引入 3 個類別 + 1 個 Protocol + 1 個 registry = 5 新檔案/類別；認知負擔指數從 3 升至 8+ | caller 行為差異巨大、需要獨立生命週期管理 |
| C. Dict-routing（現行方案） | 極簡、單檔自包含、新增 caller 僅加 1 函式 + 1 dict entry；`_VALID_CALLERS` 從 Literal get_args 派生確保同步 | 無編譯期完整性保證（漏加 dict entry 只有 runtime KeyError） | caller 少（< 5）、行為簡單（單一 handoff_id 判定）、擴展頻率低 |

### 決策分析

**現況評估**：
1. 目前 3 caller，半年內無新增 caller 需求
2. 每個 caller 的邏輯極簡（1 個 if-else，3 行函式）
3. `_VALID_CALLERS` 已從 `get_args(CheckpointCaller)` 派生，dict key 不同步時 runtime 立即 KeyError（非靜默）
4. 認知負擔指數：現行 = 3（dict + 3 builder），策略模式 = 8+（Protocol + 3 類別 + registry + 呼叫端改造）

**PC-088 對偶判定**：linux L2 建議升級的動機是「消除字串 dispatch」，但 dict-routing 本身已是字串 dispatch 的最輕量解法。升級到 dataclass/strategy 是「感覺更精緻但總步驟更多」的典型 PC-088 偏誤路徑。

### 最終決議

**保留現狀（方案 C dict-routing）**，不升級為 v2.x。

理由：
1. 擴展頻率不支持升級投資（3 caller，半年無新增）
2. 現行防護已足夠（Literal 派生 + runtime ValueError）
3. 升級成本 > 收益（認知負擔翻倍，5+ 新實體，呼叫端全改）
4. TD-A 錨點已標記，未來第 4 caller 出現時可快速定位升級點

### 規格補丁

不需要。決議為保留現狀，無需向 lavender Phase 1 v2 規格提補丁。TD-A 錨點（checkpoint_view.py L67-68）已充分標記未來升級入口。
