### 問題根因

對應父 ANA W17-176 的三個獨立根因：

**根因 1：Stop event 語意誤解**
- Hook docstring 寫「session 結束時掃描」，但 Claude Code Stop event 在每個 assistant turn 結束時觸發
- 本 hook 無防重複觸發機制（相較 handoff-auto-resume-stop-hook 有 stop flag + 7200s TTL）

**根因 2：缺乏 in_progress ticket 豁免**
- 任務進行中也觸發 handoff 提醒，邏輯倒置（handoff 是 session 結束的動作）

**根因 3：`_extract_handoff_section()` 範圍過寬**
- 用 `find` 取最早 handoff 關鍵字位置，往後掃到下一個 H1/H2 才停
- worklog 多 session 累積以 H3 (`###`) 分隔 handoff 段落，無法被 H2 切斷
- 結果擷取 49K chars / 283 IDs，過濾後重複列出 ~12 ticket 每 turn

### 影響範圍

### 修復摘要（方案 D：A + B + C 三層組合）

**根因 1 修復（A：Stop flag 防重複觸發）**

新增 `_get_stop_flag_path` / `_is_blocked_this_session` / `_mark_blocked_this_session` 三函式：

| 設計點 | 值 |
|-------|----|
| flag 路徑 | `.claude/handoff/.stop-sync-check-blocked` |
| TTL | 7200s（與 handoff-auto-resume 一致） |
| 標記時機 | 僅在實際輸出 warning 後標記（非每次觸發都標記） |
| 過期處理 | 自動刪除 flag |
| Fail-open | 損壞 flag 刪除並回傳 False |

**設計取捨**：標記 flag 移至「實際輸出 warning 後」而非「函式入口」，避免「無 drift 也消耗 flag」造成下次有 drift 時被誤判跳過。

**根因 2 修復（C：in_progress ticket 偵測豁免）**

新增 `_has_in_progress_ticket` 函式，使用 hook_utils 的 `scan_ticket_files_by_version` + `parse_ticket_frontmatter`：

| 設計點 | 值 |
|-------|----|
| IN_PROGRESS_STATUSES | `{"in_progress"}` |
| Fail-open 策略 | scan 異常 → False（不阻塞 hook 後續邏輯） |
| 邏輯位置 | `detect_sync_drift` 開頭，version 偵測後立即檢查 |

**根因 3 修復（B：`_extract_handoff_section` 改取最後一個關鍵字）**

關鍵改動（hook 與 SOT 雙寫同步，ARCH-020）：

```python
# Before: find + earliest_idx
earliest_idx = -1
for kw in HANDOFF_KEYWORDS:
    idx = content.find(kw)
    if idx >= 0 and (earliest_idx < 0 or idx < earliest_idx):
        earliest_idx = idx

# After: rfind + latest_idx
latest_idx = -1
for kw in HANDOFF_KEYWORDS:
    idx = content.rfind(kw)
    if idx > latest_idx:
        latest_idx = idx
```

### 設計決策

**為何不改用 SessionEnd event？**
- 父 ANA 重現實驗確認 Claude Code 無 SessionEnd event（PM 推測為錯誤）
- 改用 Stop + 防重複是唯一可行路徑

**為何 fail-open（in_progress 偵測異常 → 繼續執行）？**
- 假陽性（無 in_progress 卻誤判有 → 跳過）會掩蓋真實 drift（嚴重）
- 假陰性（有 in_progress 卻誤判無 → 仍觸發）只是恢復原行為（已知噪音，可接受）
- 偏向「寧可誤報不可漏報」反向，因為 stop flag 本身會抑制後續觸發

### 不在本 ticket 範圍

- 重新設計 handoff section 的 H 級分隔（H3 → H2）：屬 worklog 寫入規範議題
- 全 hook 系統的 in_progress 豁免統一抽取：本 ticket 聚焦單一 hook
