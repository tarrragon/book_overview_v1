## Problem Analysis
<!-- Schema[DOC/Problem Analysis]: 選填（若 DOC 起因於缺陷或盤點結論可填）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W10-115 落地 track_query priority 排序後，與既有 track_runqueue 兩處 priority 處理邏輯並存：

| 函式 | 模組 | 輸出 | 用途 | Consumer 數 |
|------|------|------|------|----------|
| `_normalize_priority(value)` | track_query.py:606 | str ("P0".."P3"/"P9") | sort key tuple `(p_norm, created, tid)` | 1（同模組） |
| `_priority_rank(ticket)` | track_runqueue.py:80 | int (0..3/99) | sort key tuple `(rank, id)` | 3（含 resume.py + track_dashboard.py 跨模組） |

兩者共享「P0-P3 排序 / 未知排最後」的 domain 語意，但實作獨立。

### 影響範圍

`.claude/skills/ticket/ticket_system/commands/track_query.py`、`track_runqueue.py`。間接：resume.py / track_dashboard.py（透過 `_priority_rank`）。

### 相關 Error Pattern

無直接對應 PC，屬「premature abstraction vs domain knowledge SSOT」設計取捨，與 W10-119 同家族（rule of three 邊界判斷）。

### Reality Test：實際 grep 結果

```
track_query.py:606: def _normalize_priority(value: Any) -> str
track_runqueue.py:58: _PRIORITY_ORDER = {"P0": 0, "P1": 1, "P2": 2, "P3": 3}
track_runqueue.py:80: def _priority_rank(ticket: Dict) -> int
```

未知值處理：normalize → "P9"；rank → 99（皆排最後，語意一致）。
## Solution
<!-- Schema[DOC/Solution]: 免填（DOC 類型以變更摘要取代）（.claude/pm-rules/ticket-body-schema.md） -->

### 候選方案

| 方案 | 描述 | 介面 | 成本 |
|------|------|------|------|
| A | 抽出 `lib/ticket_priority.py` 提供 `normalize_priority(value) -> str` 作為單一 SSOT；callers 各自轉 int / 直接用 str | str + 對應轉換層 | 中：新檔 + 更新 2 caller + 既有測試驗證 |
| B | 抽出 `lib/ticket_priority.py` 提供雙函式 `normalize_priority(value) -> str` 與 `priority_rank(value) -> int` | str + int 並列 | 中高：新檔 + 雙函式 + 更新 2 caller + 測試 |
| C | 維持兩處實作 + 補 cross-reference 註解（兩函式 docstring 註明另一個變體位置與差異） | 兩 module-private | 極低（純文件） |

### 評估

| 維度 | A | B | C |
|------|---|---|---|
| Rule of three | 部分：normalize 1 + rank 3 cross-module = 4 usage，但分別針對 str/int 形式各 1 / 3 | 同 A | 不適用（不抽出） |
| Domain SSOT（priority schema 變更時改動點數） | 1 處 | 1 處 | 2 處 |
| API 穩定性承諾 | 公開 lib 介面需 backward compat | 同 A，加倍 | 私有，無承諾 |
| 介面對齊 | 須 caller 額外轉換（normalize → int 仍需查表） | 介面歧異仍存在 | 不解決 |
| 與 W10-119 trigger 相容 | 與「第 3 cross-module consumer 抽出 lib/runqueue_helpers」衝突（_priority_rank 已 2 cross-module，再加 lib/ticket_priority 變成雙抽出） | 同 A | 不衝突 |

### Reality Test：W10-119 trigger 衝突點

W10-119 結論：當 `_priority_rank` 第 3 個 cross-module consumer 出現時，連同其他 4 個 runqueue 內部 helper 抽出至 `lib/runqueue_helpers.py`。若本 ticket 改抽 `lib/ticket_priority.py`，會造成：

- `_priority_rank` 被抽到 `lib/ticket_priority.py`（本 ticket 方案）
- 其他 runqueue helper 仍待第 3 consumer 抽到 `lib/runqueue_helpers.py`（W10-119 結論）

兩個 lib 位置語意重疊（都是 ticket-related helpers），且 `_priority_rank` 與 `_compute_readiness` 等 runqueue helpers 有共生關係（同為 runqueue 排序語意），抽到不同位置會分散 cohesion。

### 結論：採方案 C（維持現狀 + 補 cross-reference）

#### 推薦理由

1. **Rule of three 邊界判斷**：normalize 1 個 consumer 不滿足抽出條件；rank 3 cross-module 滿足，但 W10-119 已將其納入「連同 runqueue helpers 抽出 lib/runqueue_helpers」trigger
2. **避免與 W10-119 衝突**：本 ticket 抽 lib/ticket_priority 會分散 cohesion，違背 W10-119 統一抽出策略
3. **Domain SSOT 影響有限**：priority schema 變更（如新增 P4）為低頻事件；2 處同步改動成本可接受
4. **介面差異本就需要**：track_query 偏好 str（穩定型別、可讀 sort）；track_runqueue 偏好 int（高效 sort）；抽出 lib 仍須 caller 轉換

#### 落地清單

| 動作 | 檔案 | 內容 |
|------|------|------|
| 1 | `commands/track_query.py:606` `_normalize_priority` docstring | 加 cross-reference 註解指向 `track_runqueue._priority_rank`，並說明兩者差異（str vs int） |
| 2 | `commands/track_runqueue.py:80` `_priority_rank` docstring | 加 cross-reference 註解指向 `track_query._normalize_priority`，並說明兩者差異 |
| 3 | 若 W10-119 trigger 觸發抽 `lib/runqueue_helpers.py`，順便將 `_normalize_priority` 一起遷入 lib | 屬 W10-119 trigger 後續工作，本 ticket 不獨立追蹤 |
