### 問題根因

`test_migrate_reverse_refs.py` 的 `_patch_get_project_root` fixture 在 line 63-69 含 `try: import ticket_system.lib.paths ... except ImportError: pass` 跨版本容錯區塊。fixture docstring（line 55）已說明「並非所有版本都暴露 paths.get_project_root」的設計動機，但 except 區塊本身無行內註解，違反 quality-baseline 規則 4 異常可觀測性「空 catch 必須加日誌或在註解中說明靜默原因」。

### 影響範圍

- `.claude/skills/ticket/ticket_system/tests/test_migrate_reverse_refs.py` line 63-69 fixture 區塊
- 不涉及其他檔案、不涉及程式碼邏輯變更（純註解 / contextlib.suppress 替換）

### 相關 Error Pattern

quality-baseline 規則 4 異常可觀測性（catch 區塊必須有日誌或靜默原因說明）。

### 實作策略選項

| 方案 | 內容 | 評估 |
|------|------|------|
| A（選定） | 在 except 區塊上方加單行 `# ImportError 為預期 fallback：...` 註解 | 最小變更、保留現有結構、符合 quality-baseline 規則 4 |
| B | 改用 `contextlib.suppress(ImportError)` 包覆 try block | Pythonic 但需重寫整段、可能影響 monkeypatch 流程閱讀性 |
| C | 加 logging warning | fixture 是測試輔助，logging 噪音不必要 |

選方案 A。

### WRAP 三問

- **W（Widen）**：見上表三方案，選 A（最小侵入）。
- **A（Attain distance）**：機會成本極低（單行註解修改），不影響其他 ticket。
- **P（Prepare to be wrong）**：最可能失敗 = 註解措辭與 fixture docstring 重複造成冗餘。防護 = 註解聚焦「為何 ImportError 是預期」而非重複「跨版本容錯」描述。

在 fixture `_patch_get_project_root` 的 `try/except ImportError` 區塊上方插入單行註解，說明：

- ImportError 為預期 fallback 情境
- 觸發原因：舊版 ticket_system 未抽出 `lib/paths.py`
- 行為：import 失敗時跳過該模組 monkeypatch，不影響 `commands.migrate` 與 `lib.ticket_loader` 兩個模組的 patch

只動註解一行，未改邏輯與 import 結構。
