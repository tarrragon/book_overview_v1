## Phase 1 Feature Spec (lavender-interface-designer)

> **Phase**: 1 (Feature Design) | **Scope**: CLI 介面 + 輸出規格 + 既有命令複用策略
> **Not in scope**: 實作程式碼、測試案例（Phase 2 sage 接手）

### 1. CLI 介面規格

#### 1.1 命令簽名

```
ticket track dashboard [--top N] [--wave N] [--no-stale] [--stale-threshold MINS] [--format {text|json}]
```

#### 1.2 Flag 規格

| Flag | 型別 | 預設值 | 行為 |
|------|------|-------|------|
| `--top` | int | `5` | top N ready 區塊的列數上限；複用 runqueue `--top` 語意 |
| `--wave` | int | None（取最新 active wave） | wave 範圍過濾；複用 runqueue `--wave` 語意 |
| `--no-stale` | flag | False（預設顯示 stale 章節） | 隱藏 stale warning 區塊 |
| `--stale-threshold` | int (分鐘) | `60` | stale 判定門檻；複用 `stale-list` 既有預設值 |
| `--format` | enum | `text` | `text`（PM 預設）或 `json`（機器消費，給 hook / 自動化） |

#### 1.3 輸出格式（text，三章節結構）

```
=== Dashboard (wave={W}, version={V}) ===

[In Progress] {count} ticket(s)
  - {id}  {title}  (started_at: {ts}, agent: {who.current})
  - ...

[Ready Top {N}]  priority 排序，可直接 claim
  [1] [P0] [ready] {id}  {title}
  [2] [P1] [ready] {id}  {title}
  ...

[Stale Warning] {count} ticket(s) over {threshold}min   (用 --no-stale 隱藏)
  - {id}  stale={mins}m  status={status}  agent={who.current}
  - ...

Hint: ticket track claim <id>     # 從上方 [N] 編號直接取 id
```

#### 1.4 `[N]` 編號規則

| 規則 | 說明 |
|------|------|
| 編號範圍 | 僅 **Ready Top N** 章節使用 `[1]..[N]`，從 1 起算 |
| 編號穩定性 | 同次命令輸出內穩定；不跨命令保持（每次重新從 1 編） |
| In Progress / Stale 不編號 | 該兩章節項目用 `-` bullet，不混淆 claim 語意 |
| 編號與 id 對應 | PM 必須以 `id` 而非 `[N]` 傳給 `claim`（編號僅是視覺輔助，避免 PM 拼錯複雜 id） |
| 空章節處理 | 三章節各自於 count=0 時印「(none)」一行，保留章節標題（讓輸出結構穩定） |

### 3. 與既有命令的關係（複用策略）

| 既有命令 | 複用方式 | 不複用的部分 |
|---------|---------|-------------|
| `track_runqueue.py` | 直接 import `_priority_rank`、`_is_unblocked_pending`、`_filter_by_wave`、`_compute_readiness`、`_apply_context_resume` 等內部函式；複用 ready 篩選 + 排序邏輯 | 不複用 `_render_list` / `_render_dag` / `_render_critical_path`（dashboard 自有輸出格式） |
| `track.py`（list） | 不直接呼叫子進程；複用其 ticket loading 入口（共用 ticket index loader） | 不複用 `--status` filter 流程；dashboard 自行對 in_progress / pending 分流 |
| `track_stale_list.py` | 直接 import stale 判定函式（`_is_stale` / `_compute_stale_minutes`）+ threshold 預設值 | 不複用 stdout 渲染；dashboard 自有 stale 區塊格式 |

**關鍵原則**：dashboard 是 **聚合 view**，內部以 Python function call 複用，**禁止以 subprocess 呼叫既有 CLI**。理由：(1) 避免重複載入 Python interpreter；(2) 避免重複解析 ticket YAML；(3) 單一 ticket index 載入即可完成三章節計算。

### 4. 效能目標

| 指標 | 目標 | 量測方式 |
|------|------|---------|
| Subprocess 數 | **0**（除 Python 自身外不 fork） | 程式碼審查 + Phase 2 測試 mock subprocess 監測 |
| Ticket index 載入次數 | **1 次**（in_progress / ready / stale 共用同一份） | 注入 ticket loader spy 計次 |
| 預期執行時間（wave=10, ~20 ticket） | **< 200ms** wall clock | Phase 2 可選效能斷言（非 acceptance 強制） |
| 輸出行數上限 | in_progress + N + stale ≤ 30 行（預設 N=5） | 視覺檢視 |

### 6. 邊界條件 & 異常處理

| 情境 | 行為 |
|------|------|
| `--top 0` | 章節保留但內容顯示 `(none)`；不拋錯 |
| `--top` 大於可用 ready 數 | 顯示所有可用 ready，不補空列 |
| `--wave` 指定不存在的 wave | 三章節皆 `(none)`，wave header 仍顯示用戶輸入值 |
| `--no-stale` | 完全不渲染 `[Stale Warning]` 區塊（連 header 都不印） |
| `--format json` + `--no-stale` | json `stale` 欄位為 `null`（區分 `[]` 空清單 vs 「未計算」） |
| 無 active version | 退出碼 1 + stderr 提示「No active version detected」 |
| Ticket index 載入失敗 | 退出碼 1 + stderr 印 root cause；不部分渲染 |
