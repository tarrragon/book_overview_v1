### Rules RAG 的差異化定位

| 維度 | Ticket / PC RAG（原提案） | Rules RAG（補充提案） |

| 架構適配 | 純 RAG 可行 | 可能需要 hybrid（hook-driven + RAG） |

### Rules 載入的層次化方案

| 層次 | 機制 | 優缺點 |
|------|------|-------|
| L1: 全載入（現狀） | CLAUDE.md @ 引用全展開 | 一致性 100%，token 開銷最高 |
| L2: 部分 lazy-load（已採用） | references/ 按需 Read | 部分節省，仍有入口檔案開銷 |
| L3: hook-driven 情境載入 | 偵測 ticket type / 動作類型 → 載入對應 rule subset | 確定性高，需維護 trigger map |
| L4: RAG 按需載入 | embedding semantic match | 彈性高，命中率風險 |
| L5: 混合分層（推薦研究方向） | 核心 rule 全載入（L1）+ 情境 rule hook trigger（L3）+ 細節 rule RAG（L4） | 需要設計 rule 分類 |

| bay | **有條件支持**（30 筆精度實驗）| 反對（R3 CRITICAL 違規風險）| 強化 L3 即可達 95%+ |
| ginger | 不實施（淨負 50-200K token）| 不實施改 L3（節省 22K/session）| L3 hook-driven Rules 情境載入（P0）|
| saffron | reject（Context Bundle 已覆蓋）| reject 改 L3（確定性 > 機率性）| 沿 L3 擴充 hook trigger map |

### 共識度

| 結論 | 共識 |
|------|------|
| 提案 B（Rules RAG）reject + 改 L3 hook-driven | 4/4 = 100% |
| 提案 A（Ticket/PC RAG）不做完整 RAG | 3/4 = 75% |

| 在既有設施上擴充，不新建獨立模組 | 4/4 = 100% |

### Dissenting View（PC-101 強制記錄）

**bay 對提案 A 持「有條件支持」立場**，理由：
- 265+ error-patterns 超過人工 grep 有效管理範圍
- 失敗成本低（漏 retrieve = 少個參考，與 Rules RAG 違規風險不同）
- 既有資產覆蓋率僅 ~78%，缺口在跨版本 ticket 搜尋（60%）+ 大量 error-pattern 語意檢索（70%）
- 建議：先以 30 筆做 retrieval 精度實驗驗證

**3 視角反論**（linux / ginger / saffron）：
- ripgrep 對 1289 ticket / 265 PC 秒級回應（linux）
- retrieve 5 筆注入 PCB 反吃 50-200K token，淨負（ginger）
- Context Bundle + error-pattern skill 已覆蓋（saffron）

**主線程仲裁**：
- bay 觸到的 unknown unknowns 痛點（PM 不知 pattern 存在時無法 grep）真實存在
- 但 ginger 的 token 競爭量化證明全量 RAG 經濟性反向
- **採折衷**：拒絕完整 RAG，但保留「30 筆精度實驗」作為驗證 ticket。若實驗證實 grep 命中率 < 60%，重啟 RAG 評估；若 ≥ 60%，採 bay 反論方向（強化 grep + L3）

### 最終決策

**提案 B：實施 L3 hook-driven Rules 情境載入**（P0，4/4 共識）
- 依 ticket type / TDD phase / 動作類型載入對應 rule subset
- 預期節省 22-30K/session 固定 token 成本
- 命中率目標 95%+（hook-driven 確定性）
- 在既有 hook 系統 + CLAUDE.md @ lazy-load 機制上擴充

**提案 A：不實施完整 RAG，分兩階段驗證**
- 階段 1（先做）：30 筆 retrieval 精度實驗（驗證 bay 主張）
  - 採樣 30 筆「PM 派發 / 規則查詢」場景
  - 量測現行 grep + Context Bundle 命中率
  - 若命中率 ≥ 60% → 提案 A 永久不實施，強化 grep CLI flag（linux 推薦 A 方向）
  - 若命中率 < 60% → 進入階段 2 評估輕量方案
- 階段 2（條件觸發）：輕量 RAG 評估（SQLite-vec / TF-IDF / hybrid）
  - 不做完整 vector RAG
  - 限定 PC pattern 範圍（ticket 用結構化 grep）
  - 注入機制改為 summary 而非全文（避免 token 競爭）

### Tripwire 設計（融合 linux + bay）

**重啟完整 RAG 評估的條件**（ALL 必滿足）：
1. L3 hook-driven 落地 ≥ 12 個月
2. 30 筆精度實驗結果命中率 < 60%
3. ripgrep + summary cache 失敗率 > 30%（採樣實證）
4. 量化痛點 ≥ 10 案例 / 季

**回滾條件**（任一觸發即回滾）：
- L3 trigger map 維護成本 > 預期 2 倍
- L3 命中率 < 90%
- 引入新 PM 違規模式
