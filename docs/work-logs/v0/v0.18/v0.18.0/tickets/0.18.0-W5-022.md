## Problem Analysis

### 驗證結果：無生產缺陷，但有測試契約弱點

### 根因評估

**生產環境：無缺陷**（case a）

lifecycle.complete() 的執行順序已確認安全：
- L451-452：`ticket["status"] = STATUS_COMPLETED`（in-memory）
- L455：`save_ticket(ticket, ticket_path)` — **先落盤**（含 cache invalidation，parser.py L360）
- L482：`list_tickets(version)` — 從檔案重讀
- L490：`_cascade_unblock_children(ticket, version)` — 內部再次呼叫 `list_tickets`

由於 Python 檔案 I/O 為同步操作，`save_ticket` 返回後 parent 的 `status=completed` 已持久化並已失效快取。cascade 內的 `list_tickets` 會從磁碟讀到新狀態，`_can_cascade_unblock` 檢查 parent（作為 child 的 blocker）時正確取得 `completed`。

**測試環境：存在契約漏洞**（case d 成立）

TC-A1 fixture 結構（test_track_lifecycle.py L378-396）：
- `env.save_ticket = MagicMock(return_value=None)` — 不寫檔、不 mutate
- `all_tickets = [parent] + children_list + extras` — **共用 parent dict 引用**
- `env.list_tickets.return_value = all_tickets` — 返回相同列表

執行流程：
1. complete() L451 直接 mutate `ticket["status"] = completed`
2. save_ticket（Mock）no-op
3. cascade 內 list_tickets() 回傳 `all_tickets`，其中 `parent` dict 因步驟 1 已為 completed（in-memory 共用引用）
4. `_can_cascade_unblock` 見 blocker=completed → 解鎖 child

**測試通過依賴「共用引用的 in-memory mutation」，而非「save→load 的持久化往返」。** 若有人將 `complete()` 重構為「cascade 先於 save」，生產環境會讀到磁碟上舊狀態導致 cascade 失敗，但測試仍會因共用引用而綠燈 — 這是典型的矇混場景。

### 驗證結論

**不修改 lifecycle.py**：當前實作的 save→list→cascade 順序正確，生產環境行為符合預期。

**建議建立 follow-up IMP Ticket（W5-028）** 補強測試契約：

1. **補充整合測試（真實檔案系統）**：使用 `tmp_path` 建立真實 ticket 檔案，不 mock save_ticket，驗證 parent 狀態確實先落盤再觸發 cascade。此測試可偵測「save 先後順序被破壞」的回歸。

2. **補充順序斷言**：在 TC-A1 加入 mock_calls 序列檢查，確認 `save_ticket(parent, ...)` 在 `list_tickets(...)`（cascade 內）之前被呼叫。這可在純 Mock 環境下防止順序重構回歸。

3. **在 lifecycle.py 的 `_cascade_unblock_children` 入口處加 docstring 註解**：明確記錄「契約：呼叫前 parent 必須已 save 落盤，否則 ticket_map 會讀到舊 status」，讓未來維護者理解順序敏感性。

### 為何選擇補測試而非修程式

- 當前程式碼正確，修改只會引入回歸風險
- 根本問題是「測試矇混」而非「實作錯誤」
- 測試補強成本低、效益高（防止順序被誤改）
