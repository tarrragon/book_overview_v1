## 重現實驗結果（補充）

### 實驗方法
以 grep/wc 掃描全部 30 個 agent 定義檔。

### 實驗發現
1. AGENT_PRELOAD.md 覆蓋全 30 agent，623 words，6 段強制規則
2. 實作類 agent 有 TDD 執行流程段落可局部插入
3. pepper(Phase3a) 不需此機制
4. 新增段落約 150-250 token
5. LLM 是否遵循自檢仍需 A/B 測試驗證
### 問題根因

PC-088 v2（2026-04-18 reframe 版）結論：LLM 對 tool call 路徑的**步驟數估算**存在系統性偏誤。機制為「單步 perplexity 敏感、總步驟盲」—— 每個 next-token 的 perplexity 是直接感受，但「10 個 token 組成的序列」只能靠推理總計，結果 LLM 誤把「步驟多但每步簡單」的路徑估為短路徑（例：Write /tmp → cat → pipe 三步 vs heredoc 一步）。

本 ticket 處理此偏誤的**認知層補強方向（方向 D）**：評估在實作類代理人派發 prompt 前置「先列總步驟再執行」自檢段落，強制 LLM 在執行前產生明確的路徑預算，避免執行後才發現總成本超估。

## 推薦結論：條件式推薦

推薦在 AGENT_PRELOAD.md 新增路徑預算自檢段落，但限定 Phase 3b 實作類代理人適用，且須搭配 W17-024（結構層）和 W17-025（觀察層）協同部署。

## 1. 三位置候選對比表

| 維度 | A: PRELOAD 段落 5 之後 | B: 各 agent TDD Step 1 之前 | C: PM 派發 prompt 動態注入 |
|------|----------------------|---------------------------|--------------------------|
| 覆蓋範圍 | 全 30 agent（含不需要的） | 僅 4 實作類 agent | 僅被派發時觸發 |
| 維護成本 | 改 1 檔 | 改 4 檔 | 改 pm-rules 範本 |
| 精準度 | 低（分析類 agent 不需要） | 高（僅實作類） | 最高（可按 task 條件注入） |
| 與既有段落衝突 | 低（緊鄰 Phase 3b 查詢限制，主題連貫） | 低（TDD Step 0 概念，不覆蓋 Step 1-5） | 無（獨立於 agent 定義） |
| token 成本 | 每次載入 +150-250 token，含不需要的 agent | 僅需要時 +150-250 token | 僅派發時 +100-150 token |
| 實作複雜度 | 低 | 中 | 高（需改 PM 派發邏輯） |

**推薦位置：A（PRELOAD 段落 5 之後）**

理由：(1) 維護成本最低（1 檔）；(2) 已有 Phase 3b 條件前綴機制（段落 5 已標註 Phase 3b 強制），可複用同一條件守衛；(3) 對非實作類 agent 的 token 浪費約 200 token，相對其總 prompt 體量（數千 token）可忽略。

## 2. token 成本估算

| 項目 | 估算值 |
|------|--------|
| 自檢段落文字量 | 約 80-120 字（中文），含表格 |
| 對應 token 數 | 約 150-250 token（中文字 token 比約 1.5-2x） |
| PRELOAD 現有 token | 約 800-1000 token |
| 增幅比例 | +15%-25% |
| 每次派發攤提 | 所有 agent 派發皆載入，但僅 Phase 3b 實作類執行 |
| 每日典型派發次數 | 10-30 次 |
| 每日額外 token 總量 | 1500-7500 token（可忽略） |

## 3. 與既有 prompt 段落的衝突風險

| 既有段落 | 衝突風險 | 說明 |
|---------|---------|------|
| 段落 5: 查詢範圍限制 | 無衝突，互補 | 段落 5 限制「讀取」；新段落限制「執行路徑選擇」 |
| TDD 執行流程 Step 1-5 | 無衝突 | 新機制在 Step 1 之前觸發，屬 Step 0 概念 |
| 先寫 RED 再寫 GREEN | 無衝突 | 路徑預算針對單一 tool call 路徑，不影響 TDD 順序 |
| 先盤點既有工具再新建（PC-068） | 互補 | 盤點是「選什麼」；路徑預算是「怎麼選」 |

## 5. 具體 prompt 片段草案

以下段落建議插入 AGENT_PRELOAD.md 段落 5 之後（新段落 5.5 或重編號為段落 6，既有段落 6 順延為 7）：

```markdown
### 5.5 工具路徑預算自檢（Phase 3b 強制）

> **來源**：PC-088 v2 — LLM 對 tool call 路徑的步驟數估算存在系統性偏誤（單步 perplexity 敏感、總步驟盲）。

執行 Bash / Write / Edit 等工具前，若操作涉及多步組合（例如寫入中介檔再讀取再傳入），先在內部列出步驟數再決定路徑：

| 自檢問題 | 判斷 |
|---------|------|
| 這個操作需要幾個 tool call？ | 列出具體步驟數 |
| 有沒有單步替代方案？ | heredoc / Edit / 直接參數 |
| 多步方案的每步是否都必要？ | 刪除可省略步驟 |

**規則**：若多步方案 >= 3 步且存在 <= 1 步的替代方案，優先選擇短路徑。

**常見短路徑對照**：

| 長路徑（避免） | 短路徑（優先） |
|--------------|--------------|
| Write /tmp -> cat -> pipe | heredoc 直傳 |
| echo > file -> Read -> 傳入 | 直接用參數或 heredoc |
| 多次 Edit 同一檔案 | 一次 Edit 含所有修改 |
```

## 6. 推薦結論（正式）

**條件式推薦實作**，條件如下：

| 條件 | 說明 |
|------|------|
| 位置 | AGENT_PRELOAD.md 段落 5 之後，標註 Phase 3b 強制 |
| 觸發範圍 | Phase 3b 實作類代理人（thyme-py/thyme-ext/parsley/fennel） |
| 前置依賴 | 無（可獨立部署） |
| 協同部署 | 建議與 W17-025（觀察層）同步，以量測有效性 |
| 預期 token 成本 | +150-250 token/次派發（可忽略） |
| 風險 | 低（無破壞性，worst case 僅 LLM 忽略自檢段落） |
| 回退方案 | 刪除段落即可，無其他依賴 |

**不推薦**以下做法：
- 在每個 agent 各自維護（維護成本 4x，內容同步困難）
- PM 派發 prompt 動態注入（實作複雜度高，且 PM prompt 範本目前無結構化注入機制）
- 對所有 Phase 都啟用（分析類/設計類 agent 不需要路徑預算）
