## Solution

### 變更摘要

1. **`_validate_create_checklist` 擴充**（create.py）：在原 4 項（where.files / acceptance / decision_tree_path / when）外新增 4 項 — `who`（拒「pending」「待定義」與空字串）、`what`（拒空）、`why`（DOC 類型豁免，其他類型拒「待定義」）、`how_strategy`（拒「待定義」）。

2. **新增 `_enforce_create_checklist(missing, force)`**（create.py）：
   - `missing == []` → 直接 return
   - `missing != [] and not force` → 印錯誤訊息（含缺失欄位清單與 --force 提示）後 `sys.exit(1)`
   - `missing != [] and force` → 印 WARNING 但放行（保留快速建立逃生閥）

3. **`_persist_and_report` 改寫**（create.py）：原 WARNING 列印區塊替換為 `_enforce_create_checklist(missing_fields, force=getattr(args, "force", False))`。

4. **新增 `--force` argparse 旗標**（create.py register）：`action="store_true"`，help 文字明示為 PROP-009 阻擋的逃生閥，標註不建議用於正式 Ticket。

5. **既有 fixture 補強**（test_create_source_ticket.py `_make_args`）：`who` 從 `None` 改為 `"thyme-python-developer"`；新增 `force=False` / `quiet=False` / `verbose=False` / `json_output=False` 預設，避免新驗證誤擋既有整合測試。

### 設計決策

- 為何把阻擋邏輯抽出 `_enforce_create_checklist`：原 `_validate_create_checklist` 是純函式（回傳 list），保持其純度便於單元測試；阻擋（含 sys.exit）是另一責任，分離後雙重職責清楚，且阻擋與 --force 邏輯可獨立測試（`test_create_validation_blocking.py`）。
- 為何 `who == "pending"` 視為缺失：`_parse_cli_args_to_config` 的 fallback 預設為 `"pending"`，若不視為缺失則「未指定 --who 也通過」。AC 要求 5W1H 全欄位必填，故必須拒。
- 為何 DOC 類型豁免 why：與既有 `_parse_cli_args_to_config` 中 PC-018 的 why 必填驗證對齊（DOC 類型不要求 why）。
- 為何不在 `_parse_cli_args_to_config` 直接 sys.exit：保留「先收集所有缺失欄位再一次回報」的 UX，避免使用者修一個 commit 一次循環。

- W11-003 父 ticket 的其他子任務（W11-003.1~.4）不在本範圍。
- `where.layer` 欄位的驗證升級不在本 ticket（AC 未列）。
