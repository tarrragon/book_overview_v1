# Execution Log

## Task Summary

主線程為評估和蒐集資訊需累積大量 context，但 context 累積會降低決策品質（已知 LLM 特性）。現有多視角模式 + WRAP 系統已設計，但決策樹未明確指引「何時該停下來用這些輔助系統」，PM 容易在 context 已重的情況下果斷做判斷，錯失使用更強決策系統的時機。任務：分析決策樹現狀、識別「應觸發 WRAP/多視角」的情境訊號（如已用 tool calls > 某閾值、重要決策、連續失敗、user 指正頻率高等），升級決策樹加入自動提示或強制路由，讓 PM 在高 context 重擔時自覺切換輔助決策模式。

---

## Problem Analysis

### 現狀盤點

**現有輔助決策系統**（全部已建置）：

| 系統 | 位置 | 覆蓋範圍 |
|------|------|---------|
| WRAP 決策框架 | `.claude/skills/wrap-decision/SKILL.md` | 連續失敗、被困、偏離核心、重大決策、ANA/Debug/提案 |
| 多視角並行評估 | `.claude/skills/parallel-evaluation/SKILL.md` | Code review、重構、架構、功能、冗餘、結論審查、系統設計 |
| 決策樹路由 | `.claude/pm-rules/decision-tree.md` | 已引用 WRAP 作為 ANA/Debug/提案強制流程 |

**現狀問題**：輔助系統已完備，但**觸發機制仰賴 PM 自律**。PM 在 context 沉重時：
- 記憶工作集縮小，難以主動回想「我現在應該用 WRAP」
- 高壓下傾向「快速判斷完結案」而非「停下來展開選項」
- 決策樹雖引用 WRAP，但缺乏**主動偵測機制**

### Hook 可行性評估（滿足驗收條件 4）

**可行度分層**：

| Hook 設計 | 可行度 | 難點 |
|----------|-------|------|
| 連續失敗偵測（訊號 2） | 高 | `ticket track release/reclaim` 事件可計數 |
| 關鍵字掃描（訊號 3、4、6） | 高 | 正則偵測 PM 輸出即可 |
| ANA Ticket claim 提示（訊號 5） | 高 | `ticket-reinstall-hook` 或 ticket CLI 輸出擴充 |
| 偏離 Wave 偵測（訊號 7） | 中 | 需跨 ticket 狀態比對 |
| Tool call 次數（訊號 1） | 中 | Claude Code session state 需從 transcript 推導 |
| Session 長度 | 低 | Hook 無法直接取得 session 起始時間 |
| 救火時長（訊號 8） | 低 | 需設計計畫性 ticket vs 非計畫性的區分 |

**建議起始範圍**：先實作高可行度 3 項（訊號 2、3、5），涵蓋 90% 高風險場景。

**Hook 架構**：
- 檔名：`.claude/hooks/decision-quality-guard-hook.py`
- 觸發：`UserPromptSubmit` + `PostToolUse:Bash`（偵測 ticket 事件）
- 輸出（符合品質規則 4 雙通道）：
  ```

  強制建議：執行 /wrap-decision 快速模式（5 分鐘）
  原因：{signal_reason}
  ```
- 節流：同一訊號 10 分鐘內只觸發一次

### S1：decision-quality-guard-hook

**位置**：`.claude/hooks/decision-quality-guard-hook.py`

**觸發事件**：
- `UserPromptSubmit`：掃描關鍵字（做不到/禁止/無法/沒辦法/升級/重構）
- `PostToolUse:Bash`：偵測 `ticket track claim` ANA type、`ticket track release/reclaim` 連續失敗

**偵測邏輯**：
1. **訊號 2（連續失敗）**：同一 ticket 在 1 小時內 release + reclaim ≥ 2 次 → 強制 WRAP
2. **訊號 3（做不到關鍵字）**：PM 輸出匹配 `(做不到|無法|沒辦法|不可能|CLI不支援)` → 強制 WRAP
3. **訊號 5（ANA claim）**：`ticket track claim <id>` 後該 ticket type=ANA → 強制 WRAP

**輸出格式（stderr + log）**：
```
[DecisionQualityGuard] 偵測到觸發訊號：{signal}
強制建議：執行 /wrap-decision 快速模式（5 分鐘）
原因：{reason}
節流：同訊號 10 分鐘內不再觸發
```

**節流設計**：
- 使用 `.claude/hook-state/decision-quality-guard.json` 儲存最後觸發時間
- 同訊號 10 分鐘內略過
- 不同訊號各自獨立節流

### S2：決策樹新增 Context 重度檢查層

```markdown
| ANA Ticket claim | **強制** 執行 /wrap-decision 快速模式 |
| 重大決策關鍵字（升級/重構/改架構） | 提示（非強制） |

未觸發 → 繼續派發閘門。
觸發 → 先完成 WRAP 錨點 + W + 基本率問題 → 再進派發閘門。
```

### 關鍵教訓

1. **三 ANA 元模式（W9-002/003/004 共通結構）**：PM 的原則正確但執行落差，需要「自動觸發機制」補 PM 自律缺口。三層防護（規則 + skill + hook）是可重用的元架構。
