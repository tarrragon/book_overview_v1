### 問題根因

W11-004.7.1 Problem Analysis 使用 `<!-- PC-093-exempt: history:... -->` 標記豁免 PC-093 hook 檢查，但 `history` 不在現有 EXEMPT_CATEGORIES 白名單，被標 INVALID（informational 等級未阻擋 complete）。屬白名單覆蓋不足。

### 現有白名單

`.claude/hooks/phase4-decision-enforcement-hook.py:57-63`：
- tdd-transition
- baseline-gated（reason 需含數字）
- ticket-tracked（reason 需含 W{wave}-{seq}）
- user-override
- rule-quote（reason 需含 .claude/rules/ 或 .claude/pm-rules/ 路徑）

### history 實際用例分析

W11-004.7.1:66 範例：
> `<!-- PC-093-exempt: history:本 ticket 即執行 parent W11-004.7 多視角審查發現的 polish 工作，非延後項；以下描述為動機脈絡 -->`

語意：「描述歷史脈絡 / 動機背景，非延後決策」。常見於 IMP ticket 引用 parent ANA 的審查發現作 Problem Analysis 開頭。

### 與既有 category 重疊評估

| 既有 category | 是否可覆蓋 history 用例 | 差異 |
|--------------|-------------------|------|
| ticket-tracked | 部分可（reason 含 parent ticket ID 即合法） | history 語意更精確；ticket-tracked 暗示「等該 ticket 完成後處理」，history 是「引用已完成歷史」 |
| rule-quote | 不可 | 引用對象是 ticket 而非規則 |
| 其他 | 不可 | 語意不符 |

### 設計選項

**選項 A：納入 history 為合法 category**
- 語意：reason 描述歷史脈絡/動機，不是延後決策
- 驗證：reason 需含 ticket ID（W{wave}-{seq}）或明確時間錨點，避免變成自由文字逃生閥
- 優點：語意清晰，撰寫者無需在 ticket-tracked 與「歷史引用」間判斷
- 缺點：白名單膨脹，與 ticket-tracked 部分重疊

**選項 B：拒絕 history，hook 訊息引導改用 ticket-tracked**
- 改進 hook ERR_MESSAGE_MAP，category-whitelist 訊息加「常見錯用：history → 改用 ticket-tracked + reason 含 parent ticket ID」
- 優點：白名單保持精簡，DRY
- 缺點：撰寫者需理解語意差異（延後 vs 歷史脈絡）

### 決策

採選項 A：納入 `history` 為合法 exempt category，附「reason 須含 ticket ID 作歷史錨點」約束。

**決策依據**：
- 語意精確性：`history` 描述「引用已完成歷史 / 動機脈絡」（如 W11-004.7.1:66 引用 parent W11-004.7 多視角審查），與 `ticket-tracked` 的「等待該 ticket 完成後處理」語意不同
- 防自由文字逃生閥：強制 reason 含 `W\d+-\d+` 格式 ticket ID，與 `ticket-tracked` 同款錨點驗證

**為何不選 B（拒絕 + hook 訊息引導）**：B 雖保持白名單精簡，但會讓撰寫者在「延後 vs 歷史脈絡」語意判斷上耗費認知成本；A 將語意差異固化為機制，更符合三明示原則（規則命名即傳達語意）
