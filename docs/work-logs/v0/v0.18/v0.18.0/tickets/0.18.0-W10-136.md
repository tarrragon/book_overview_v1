## 第三輪分析輸入（兩個來源整合）

### 來源 A：Stanford CS230 Prompt Engineering 8 方向

| # | Stanford 概念 | 對 AUQ 應用 |
|---|--------------|------------|
| 1 | Prompt chaining | AUQ 拆三 prompt：決策識別 → 選項池選 → 渲染 description |
| 2 | Few-shot examples 持續更新 | 違規 → label → append 到 examples pool；下次插示例 |
| 3 | LLM judge + rubric | Hook 用 LLM judge 評分 AUQ payload（替代脆弱 regex）|
| 4 | Self-critique / reflection | AUQ 生成後強制自我批評再修正 |
| 5 | Jagged frontier | 識別「不該 AUQ」的決策（hook 可推斷的）|
| 6 | Centaur vs Cyborg | AUQ 改 batch 模式：累積多決策一次問 |
| 7 | Prompt template + metadata | scenario_id + metadata 渲染（強化 L4 實作模型）|
| 8 | Promptfoo 多 LLM 測試 | dev-time 工具：scenario 庫 batch 測試 |

### 來源 B：W10-135 官方/社群調研

**AUQ tool 根本限制**（官方確認）：
- 硬上限 4 個選項
- **無 enum 約束機制** ← L4「改造 AUQ tool」**確認不可行**
- 官方文件：[Tools reference](https://code.claude.com/docs/en/tools-reference)

**官方替代機制**：

| 機制 | 強制度 | 實作成本 |
|------|------|---------|
| Claude Agent SDK Structured Outputs with enum | 100% | 高（框架級重構）|
| Strict Tool Use via tool_choice strict | ~99% | 中（驗證層補償）|
| Hook 驗證層 + prompt 強化 | < 10% 發生率 | 低 |

**社群已知問題**：
- GitHub issue #12420：4 選項限制太嚴
- GitHub issue #29889：空答案 bug
- Hook 支援、Subagent 支援均缺失

**官方推薦路徑**：
- 短期：Hook 驗證層 + PM prompt 強化
- 中期：Agent SDK Structured Output 重構

## 第三輪 4 視角整合結論

### Verdict 矩陣

| 層次 | linux | bay | ginger | saffron | 共識 |
|------|-------|-----|--------|---------|------|
| L1 hook 過濾 | 採 | 採 | 採（基線）| 採 | 4/4 採 |
| L2 scenario library | 採 | 隱含（L7）| 隱含（#5）| 隱含（路徑 C）| 4/4 採 |
| L2.5 few-shot 自學 | 拒（骯髒包裝）| 拒（survivorship + Goodhart）| 採（with cap ≤ 30）| 未明確 | **3:1 拒** |
| **L3 hook wrapper schema** | **採（最 good taste）** | **採** | **採（+48-77K 最高 ROI）** | 路徑 C 未明列 | **3-4/4 採** |
| **L3.5 LLM judge** | 拒（用隨機解隨機）| 拒（Cohen kappa 0.4-0.7 < 0.9）| 拒（二階隨機性）| **採（路徑 C 先行）** | **3:1 拒** |
| L5 Agent SDK 重構 | 拒（不可逆 + Never break userspace）| 拒 | 拒（100-300h beta）| 漸進（POC 先 + Claude Code 演進 hedge）| 3:1 拒 + saffron 條件 |
| **新 #5/L7 deterministic 過濾** | 隱含 | **採（L7）** | **採（#5 觸發頻率 -30-40%）** | 未提 | 2/2 顯性採 |

### 量化整合（ginger 主軸）

| 推薦項 | 變動成本/session | 違規改善 | 淨效益 |
|-------|---------------|--------|------|
| L3 hook wrapper | +0.5-2K | 5-15% → 1-3% | **+48-77K** |
| L7/#5 deterministic 過濾 | +2-6K | 觸發頻率 -30-40% | 乘數效應 |
| L1 強化 | 持平 | < 10% 發生率 | 基線 |

### bay devil's advocate 量化反論金句

- 60% 覆蓋率 × 100% 可靠性 > 100% 覆蓋率 × 70% 可靠性（期望品質 0.6 > 0.49）
- LLM judge Cohen kappa 0.4-0.7 < 工程閾值 0.9
- Few-shot pool「N 代後只教 LLM 如何不被 hook 攔截」（Goodhart's law 應用）
- Self-critique 5 種已知偏見（sycophancy / position / verbosity / anchoring / confidence）

### saffron 新關鍵發現

L5 真正破壞點不在 AUQ 而在 **Agent 派發 hooks（10 個品質閘門）**：
- 派發驗證 hooks
- PreToolUse:Agent prompt 長度檢查
- 檔案所有權驗證
- worktree 檢查
- dispatch 計數
- SubagentStop event handlers

L5 重構需重寫 10 個 hooks 的 matcher 與 input schema → PM 系統失去核心安全網。

### Dissenting View 處理（PC-101 強制）

**saffron 採 L3.5 LLM judge（3:1 反對）**：

- saffron 論點：L1 強化 + L3.5 LLM judge 先行，觀測違規率數據後決定後續
- 3 反對視角證據：
  - linux：用隨機系統解隨機系統（PC-066 同構）
  - bay：Cohen kappa < 0.9 工程閾值
  - ginger：二階隨機性，淨效益負

**主線程仲裁**：採 saffron「漸進觀測」精神但不採其 L3.5 工具選擇。改用 deterministic L7/#5 過濾（bay/ginger 共識）達成同等效果，且不引入二階隨機性。

### 主線程第三輪仲裁

**最終實施組合**（融合 4 視角共識點）：

| Phase | 動作 | 證據基礎 | 負責層次 |
|-------|------|---------|---------|
| **A (P0)** | L3 hook wrapper schema 設計與實作 | 3-4/4 共識，ROI +48-77K | runtime 強制 |
| **A (P0)** | L7/#5 deterministic scenario 過濾（識別「不該 AUQ」決策）| bay/ginger 顯性 + linux/saffron 隱含 | 從根本剔除 |
| **B (P1)** | L1 既有 hook 強化（charset/format 規則擴充）| 4/4 共識 | 安全網 |
| **B (P1)** | Promptfoo-style dev-time 測試（dev tool 層）| Stanford #8 + bay 認可 | dev-time 品質 |

### Tripwire 設計（融合 4 視角）

**重啟更激進方案的條件**（任一觸發即評估）：

1. L3 落地 6-8 週後違規率仍 > 5%（linux/ginger）
2. scenario 庫 > 50 個 scenario（linux）
3. token 總開銷 > 50K（ginger）
4. **Claude Code 官方原生支援 tool_choice:strict for AUQ**（saffron hedge）→ 重評 L5
