## Task Summary

審查 所有 75+ hooks 的單維度強制邏輯與 ARCH-018 風險

## Problem Analysis

### 問題根因

ARCH-018（hook 單維度強制 vs nested rule 衝突）由 W10-041/W10-042 觸發案例（branch-verify-hook 強制 worktree vs ARCH-015 禁 worktree）確認為 hook 設計通用反模式。需系統性審查專案內所有強制阻擋類 hook 是否潛藏類似結構性風險。

### 影響範圍

`.claude/hooks/*.py` 中含「強制阻擋」邏輯（permissionDecision: deny / exitCode: 2）的 10 個業務 hook（Reality Test 已校準範圍，詳見上方）。

### 相關 Error Pattern

- **ARCH-018**: hook-blanket-requirement-vs-nested-rule-conflict（本 ticket 主要審查焦點）
- **ARCH-015**: subagent-claude-write-protection（W10-042 的衝突對手規則）
- **PC-077**: hook-vs-architecture-conflict（升級到方法論的 follow-up：W13-014）

### WRAP 分析

**W (Widen)** — 候選方案：
- (a) PM 前台逐一審查 10 hook（精細但慢）
- (b) bulk-evaluate 並行派發 2 批給 bay-quality-auditor（**已選**）
- (c) 聚焦 5 個 worktree/dispatch 高風險 hook（範圍縮小）
- (d) close 不做（W10-042 已修，可能 over-engineering）

選擇 (b) 理由：10 hook 雖小於原預期，但仍非 PM 前台一氣呵成的合理量；bay-quality-auditor 並行可在 30-60 分鐘內取得結構化報告，且兩批報告可互為對照組。

**A (Attain distance)** — 機會成本：
- 排擠 W10-045/046/047/048 共 4 個 P2 hook 相關 ticket（總工作量約 1 wave）
- 兩個 bay-quality-auditor 派發的 token 成本（每個 audit 報告約 5-10K tokens）
- 收益估算：若識別 1+ 個潛藏 ARCH-018 風險 hook，可避免未來撞牆事件；若全部清白，提供結構化背書，後續類似 hook 設計有參考標準

**P (Prepare to be wrong)** — 失敗預測：
- 最可能失敗：bay-quality-auditor 對 ARCH-018 風險判別準則不一致，產出兩份不可橫向比對的報告
- 防護措施：在子 ticket PCB 提供標準化評估 rubric（檢查項：強制動作 / 觸發條件 / 路徑分類豁免機制 / 已知衝突規則對照），確保兩份報告對齊
- 次要失敗：可能找出大量「理論風險但實務無觸發路徑」的偽陽性
- 防護：rubric 強制要求每個風險點附「實際觸發劇本」，無觸發劇本者降為 P3 觀察項

### 實驗發現

- 這 10 個 hook 才是 ARCH-018 風險（單維度強制 vs nested rule conflict）的潛在發生點

**仍未驗證的假設**：

**範圍重新校準**：

## Solution

### Dispatch Plan

並行派發 2 個 bay-quality-auditor 子任務，每批 5 hook，使用統一評估 rubric：

| 子 ticket | Batch | 範圍 | 派發代理人 |
|---------|-------|------|----------|
| 0.18.0-W10-043.1 | A: dispatch/lifecycle | branch-verify, task-dispatch-readiness-check, agent-ticket-validation, acceptance-gate, main-thread-edit-restriction | bay-quality-auditor |

### 評估 Rubric（兩批共用，寫入子 ticket PCB）

每個 hook 必須回答以下 5 點：

1. **強制動作**：hook 在何種條件下回 deny / exit 2？
2. **觸發條件維度**：條件是單維度（單一 flag/path）還是多維度（多條件 AND/OR）？
3. **路徑分類 / 豁免機制**：是否已有 exempt list / classifier？覆蓋哪些情境？
4. **已知衝突規則對照**：ARCH-015 / 其他 nested rules 是否會與此 hook 強制邏輯撞擊？列出實際觸發劇本（無劇本則降 P3）
5. **ARCH-018 風險分級**：P0（已撞牆）/ P1（高機率撞牆）/ P2（理論風險）/ P3（無觸發路徑）

- 對 P2/P3 記錄為背書（無需立即行動）
- 完成 4 條 acceptance 並 complete W10-043

### 整合結論（W10-043.1 + W10-043.2）

**全 10 個強制阻擋類 hook ARCH-018 風險分布**：

| 風險級 | 數量 | Hook 清單 |
|-------|------|----------|
| P0（已撞牆） | 0 | — |
| P1（高機率撞牆） | 1 | task-dispatch-readiness-check |
| P2（理論風險） | 2 | agent-ticket-validation, main-thread-edit-restriction |
| P3（無觸發路徑） | 7 | branch-verify, acceptance-gate, ticket-file-access-guard, mcp-write-tool-on-text-file-guard, mcp-run-tests-validator, askuserquestion-charset-guard, ticket-path-guard |

**核心發現**：

1. **W10-042 修復有效**：原始 ARCH-018 觸發案例 branch-verify-hook 已具備完善豁免，無殘留風險
2. **單一 P1 風險**：task-dispatch-readiness-check 的 explicit phase pattern 掃描範圍過寬（前 500 字元），易誤命中 Context Bundle 引用的上游 Phase 標記
3. **P2 風險低但值得改善**：兩個 hook 在邊界擴展時可能需要調整
4. **附帶發現（死碼）**：mcp-run-tests-validator.py 在 Chrome Extension 專案中為死碼（dart MCP 不存在）

**Follow-up Tickets（依規則 5 全數追蹤）**：

| Ticket | 類型 | 處理對象 | 優先級 |
|--------|------|---------|-------|
| 0.18.0-W10-043.3 | IMP | task-dispatch-readiness-check phase pattern 修正 | P1 |
| 0.18.0-W10-043.4 | IMP | agent-ticket-validation checklist + main-thread-edit-restriction deny 訊息（合併） | P2 |
| 0.18.0-W10-043.5 | ANA | mcp-run-tests-validator 死碼移除三選項評估 | P2 |

**結構化背書**：

10 個強制阻擋類 hook 中 7 個（70%）為 P3，所有 hook 均具備至少雙維度條件檢查，不存在「單維度全量強制」反模式（ARCH-018 典型）。本次審查可作為「Hook 設計成熟度」基線，後續新增強制阻擋 hook 應通過此 5 點 rubric。

**規則 6 驗證**：本 ticket Reality Test 將範圍從 75+ 校準為 10 個（範圍縮減 7-8 倍），驗證了「先量化再決策」的價值——若直接接受 ticket 標題的「75+」估算，會傾向選 (a) PM 前台逐一審查（耗時）或 (d) close 不做（規避）。bulk-evaluate (b) 在實際範圍 10 hook 下仍具效率（30-60 分鐘並行 vs PM 前台順序耗時相當），且兩批可互為對照組驗證 rubric 一致性。
