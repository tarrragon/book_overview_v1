### Devil's Advocate 分析：為何機制化「派發前假設驗證」可能不必要或有害

---

#### 1. 機制化反向風險清單

| 風險 | 機制 | 嚴重度 |
|------|------|--------|
| R1：規則疲勞（rule fatigue） | 59 檔 + 175+ checklist 項已超載；新增項目的邊際注意力接近零 | 高 |
| R2：checklist 形式化（checkbox theater） | PM 勾選 checklist 不等於真正驗證；W10-137 的救援價值來自 PM 的判斷力而非 checklist 存在 | 高 |
| R3：過度防護的機會成本 | 每次派發多一步驗證 = 每次派發多消耗 token + 時間；99% 的派發不需要假設驗證 | 中 |
| R4：遞迴防護悖論 | PC-066 說「自律失效需 Hook 化」→ 本提案說「假設驗證需機制化」→ 下次會說「機制化遵循需 Hook 化」→ 無窮遞迴 | 高 |
| R5：虛假安全感 | checklist 存在讓 PM 以為「有防護」而降低警覺，實際 checklist 覆蓋不了所有假設驗證場景 | 中 |

---

#### 2. W10-137 普遍性質疑

**核心論點：W10-137 是孤立案例，不是普遍現象。**

證據：

| 維度 | 分析 |
|------|------|
| 樣本量 | N=1。僅此一次被觀察到「派發前 grep 驗證救下錯誤決策」 |
| 觸發條件特殊性 | 需同時滿足：(a) 多視角分析中某視角做出強烈建議（刪規則），(b) 該建議基於未驗證假設（hook 是強制的），(c) PM 恰好有足夠 context 意識到需要驗證。三者交集機率低 |
| 歷史對照 | 專案歷史中大量派發決策未經「假設驗證」但未出問題。W10-134/W10-136 兩輪 devil's advocate 本身就是假設驗證的替代品，且已被部分採納 |
| 倖存者偏誤 | 我們只看到「驗證救下的案例」，看不到「未驗證但也沒出問題的案例」（後者遠多於前者） |
| 對照組缺失 | 無法量化「不驗證時出問題」的基準頻率。沒有基準就無法證明機制化的 ROI |

**結論**：以 N=1 的救援案例推動機制化，屬於可得性偏誤（availability bias）——因為案例鮮明而高估其頻率。

---

#### 3. 不固化的真實成本

| 成本項 | 評估 |
|--------|------|
| 下次再犯機率 | 低。W10-137 本身已成為 memory/error-pattern，PM 在類似場景會自然回想。經驗學習比 checklist 更有效 |
| 再犯後果 | 中。最壞情況是刪除一條有價值的規則，但可 git revert。不是不可逆傷害 |
| 修復成本 | 低。git revert + 重新評估 = 30 分鐘。遠低於建立和維護機制的成本 |
| 累積風險 | 低。此類「基於未驗證假設的破壞性決策」極少見（N=1 in 140+ tickets） |

**成本結論**：不固化的真實成本 = 低機率 x 低後果 x 低修復成本 = 極低。

---

#### 4. 不機制化的最佳路徑

**存在，且已部分到位：**

| 既有防護 | 覆蓋度 | 說明 |
|---------|--------|------|
| 多視角審查（parallel-evaluation） | 高 | W10-134 到 W10-140 的整個流程本身就是多視角審查。devil's advocate 視角已經在做「質疑假設」的工作 |
| PC-066 memory | 中 | PM 已有「自律失效」的經驗記憶，類似場景會自動警覺 |
| git revert 安全網 | 高 | 即使錯誤決策被執行，規則刪除可完全還原 |
| quality-baseline 規則 6（失敗案例學習） | 高 | W10-137 本身就是規則 6 的實踐——從失敗案例提煉教訓，無需額外機制 |
| ANA 多視角評估流程 | 高 | 高風險決策已走多視角，不需要在「每次派發」層級加防護 |

**最佳路徑**：不新增任何機制。將 W10-137 記錄為 PC error-pattern（如 PC-1XX：多視角結論中未驗證假設的破壞性建議），讓 PM 在類似場景自然觸發警覺。這比 checklist 更符合人類認知——故事比清單好記。

---

#### 5. 反論證據：為何 PM 自律 + 既有 dissenting view 已足夠

**論點 A：W10-137 的救援行為本身證明自律有效。**

如果 PM 自律無效，W10-137 就不會被攔下。正是 PM 主動 grep 驗證才避免了錯誤。這證明現有機制（bay devil's advocate 提醒 + PM 判斷力）已足夠觸發驗證行為。用 W10-137 來論證「需要機制化」是自相矛盾的——它同時證明了「不需要機制化也能攔下」。

**論點 B：checklist 不能替代判斷力。**

W10-137 的救援關鍵不是「有 checklist 提醒 PM 去驗證」，而是 PM 的技術判斷力讓他意識到「hook 是否強制」是個需要驗證的假設。checklist 寫「驗證假設」不會讓沒有判斷力的 PM 知道哪些是假設、怎麼驗證。

**論點 C：175+ checklist 項的存在本身就是警告。**

每新增一條 checklist 都是在稀釋所有既有項目的注意力權重。27 檔含 checklist 意味著 PM 的認知負擔已超載。此時正確的方向是精簡而非新增。

---

#### 6. 對 5 候選方案的反向評估

| 方案 | 反論 |
|------|------|
| A. 規則文件 | PM 已有 44 個 pm-rules 檔要讀。第 45 個的邊際注意力 = 0。PC-066 本身就證明了自律規則在壓力下失效 |
| B. Hook 強制 | 「高風險動作」的定義無法窮舉。Hook 要麼太鬆（漏抓）要麼太緊（每次派發都觸發 = 被忽略）。且 PC-066 的解法已是 Hook，再加一層 Hook 是遞迴 |
| C. 獨立 Skill | 與 D 重複，且增加工具數量 |
| D. 整合 wrap-decision | 相對最不有害，但 WRAP 已有 W/R/A/P 四階段，加第五問會稀釋既有四問的注意力 |
| E. 三段式 | 父 ticket 已拒絕，不再評估 |

**最不有害的選項**：如果必須機制化，D 是唯一可接受的——但我的立場是「不需要機制化」。

---

### 總結 verdict

**建議：不機制化。**

理由：
1. N=1 案例不足以支撐機制化（可得性偏誤）
2. 現有防護（多視角 + memory + git revert）已覆蓋此風險
3. 175+ checklist 項已超載，新增邊際價值為零
4. W10-137 的成功恰恰證明自律有效，無需外部強制
5. checklist 不能替代判斷力，會產生虛假安全感

**如果 PM 仲裁後仍決定機制化**：方案 D（整合 WRAP 第四問）是唯一可接受路徑，但應作為 WRAP 的可選擴充而非強制步驟。

dissenting_view: 本分析承認一個弱點——「N=1 但後果嚴重」的場景在安全工程中常被視為正當的機制化理由。但本案的後果可 git revert，不屬於不可逆損害，安全工程類比不成立。
