## continueOnBlock 適用範圍評估

### 評估範圍

`.claude/settings.json` PostToolUse 共 5 個 matcher 組（Bash / mcp__dart__run_tests / Edit / Write / Agent + 一個無 matcher 全域組），共 30 個 hook 條目（去重後 29 個獨立 hook）。

### 分類方法

對每個 hook 檢視原始碼判斷 block 行為：

- **無 block 型**：原始碼無 `exit(2)` / `decision: "block"` / `permissionDecision: "deny"` 任何輸出形式 → exit 0 純觀測/提醒。**continueOnBlock 對其無意義**（本來就不擋）。
- **必擋型**：block 是「補正後繼續就會破壞下一步前提」的硬擋。
- **可放寬型**：block 屬「提醒 / 訊息回饋」性質，繼續推進無破壞。

### 必擋型詳評：phase4-decision-enforcement-hook

| 項目 | 內容 |
|------|------|
| Block 觸發 | 偵測「Phase X 再決定」「延後評估」「Phase 4 評估」等 PC-093 反模式 |
| 為何不可放寬 | quality-baseline 規則 2（Phase 4 不可跳過）+ PC-093（無 trigger 延後）+ PC-066（決策品質自律失效）三重防護。block 後若繼續推進 → 違規話術固化進 ticket / commit，下一步動作（complete / handoff）已建立在錯誤前提上，事後無法低成本回退 |
| Fallback 機制 | hook 訊息已包含 exempt marker 寫法（PC-146 / EXEMPT_PROXIMITY_LINES=1）；合法引用可加 `<!-- PC-093-exempt: ... -->` 通過 |

### 可放寬型詳評：l10n-sync-verification-hook

| 項目 | 內容 |
|------|------|
| Block 觸發 | l10n 鍵值在 src/ 與 lang files 間不同步 |
| 為何可放寬 | l10n 不同步是「需要補正」而非「決策被固化」；訊息透過 continueOnBlock 回饋給 Claude，可在同回合補上鍵值或 spawn ticket |
| 副作用評估 | (1) 訊息回饋是否誤觸決策迴圈？l10n 缺鍵是事實性訊息（非決策話術），不會被 Claude 詮釋為延後決策 → 不誤觸 PC-066/PC-093 防護。(2) PC-066/PC-093 防護是否仍生效？此 hook 與 PC-066/PC-093 防護無關（後者由 phase4-hook + WRAP tripwire 負責）→ 仍生效。 |
| Fallback 機制 | 若 Claude 收到 reason 後忽略未補正，spawn ticket 機制（規則 5）仍會在後續審查發現 l10n 漂移；本專案無 l10n 流程（Chrome Extension v1 純繁中），實際觸發機率低 |
