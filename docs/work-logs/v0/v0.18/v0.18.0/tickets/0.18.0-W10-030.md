### WRAP 完整分析

#### W (Widen options) — 選項擴增

不要只想「自動指數計算 + 強制觸發」。框架測試：能否一句話概括？「在 claim hook 加指數計算」屬單一方向，須擴增。

| 候選 | 說明 | ROI 評估 |
|------|------|---------|
| (a) 純自動分級 | claim hook 計算指數，>10 簡化 / >15 完整（W10-027 原設想） | 高成本（指數計算需 PCB/AST 解析），低增益（已有更早閾值） |
| (b) 提示式分級 | claim 印出指數值，深度由用戶決定 | 中成本，弱效力（用戶可忽略） |
| (c) 弱訊號加成 | 保留類型分級，加 where.files 數量、acceptance 數量等弱訊號 | 低成本，與 ticket create 警告重疊 |
| (d) 混合分級 | 自動建議 + 用戶 override | 高成本（雙層 UI） |
| (e) **不做（推薦）** | 類型分級 + 3b 派發前閾值 + create 警告已三層覆蓋；W10-030 設想已被超車 | 最高 ROI |
| (f) 後驗校準 | 先記錄當前分級命中率，看是否真有問題 | 中成本，但需先有實證 |
| (g) 移到拆分階段 | 將指數計算用於 ticket 拆分時自動建議拆分數，而非 claim 強制 WRAP | 中成本，與認知負擔本意對齊 |

#### R (Reality Test) — 真根因驗證

見上方 Problem Analysis Reality Test 章節。結論：H1/H2 已被 W17-052 + ticket create 警告超車。

#### A (Attain distance) — 機會成本

執行 W10-030（純自動分級）會擠壓：
- 同 wave 高優先 ticket（W10-040 / W10-046 等）
- 認知負擔指數計算的維護成本（hook 跑時間、規則更新追蹤）
- 與既有三層攔截（cognitive-load.md / 3b 派發閾值 / ticket create 警告）的協調成本

更高 ROI 的事：選項 (e) 不做 + 將 W10-030 範圍轉為「驗證既有三層攔截是否足夠」的 follow-up 監測 ticket。

#### P (Prepare to be wrong) — 失敗預想

12 小時後若選 (e) 不做失敗，最可能原因：
1. 三層攔截實際命中率低 → 需要選項 (f) 後驗校準
2. PM 在派發階段忽略 3b 派發前閾值 → 屬流程遵循問題，加 hook 強制比加自動分級更直接
3. 用戶持續期待 W10-027 中期方案落地 → 需明示已超車並結案

防護措施：建立 follow-up 監測 ticket（選項 (f)），規定 N 個 ticket 後觀察「分級失準率」是否 > X%，> X 才重啟設計。

### 結論：選項 (e) 不做 + (f) 監測 follow-up
