### 設計交付物 3：hook wrapper 攔截邏輯

**Hook 名稱**：`auq-schema-enforcement-hook.py`（IMP 階段建立，本 ticket 不寫）

**Hook 類型**：PreToolUse（matcher: AskUserQuestion）

**與既有 askuserquestion-charset-guard-hook.py 的關係**：並存。Charset guard 攔截字元集；schema enforcement 攔截結構。兩者皆 PreToolUse + AskUserQuestion matcher，獨立判定（任一阻擋即整體阻擋）。

**攔截流程（pseudo-code，純設計）**：

```
input: AUQ tool payload (JSON)
  - questions[].question: str
  - questions[].header: str
  - questions[].multiSelect: bool
  - questions[].options[].label: str
  - questions[].options[].description: str

step 1: scenario_id 解析
  方法 A：payload 含 metadata.scenario_id（理想；需 PM 主動標註）
  方法 B：fallback——根據 question 文字 fingerprint 匹配 _registry.yaml
    - 取 question 前 30 字元 → 對 _registry.yaml 各 scenario.question.template 做相似度比對
    - 命中閾值 >= 0.85 視為匹配
    - 無匹配 → category=unknown，進入「unknown scenario 處理」分支

step 2: 載入對應 scenario YAML
  - 路徑：.claude/config/auq-scenarios/{category}/{scenario_id}.yaml
  - 讀取失敗 → 警告但放行（保守降級，避免 hook bug 阻塞 PM）

step 3: 驗證選項池（依 category）
  case static:
    - 取 payload.options[*].label 與 scenario.options[*].label 做集合比對
    - 多/少/錯選項 → exit 2 + stderr 列出 diff
  case semi-dynamic:
    - 取 payload.options[*].label 對 scenario.options[*].label_template 做變數替換比對
    - 變數槽位允許任意值，骨架必須一致
    - 缺少必要選項或多出非預期選項 → exit 2
  case dynamic:
    - 不驗證 label 內容（runtime 生成）
    - 僅驗證 constraints（max_options / recommended_position）

step 4: 驗證 constraints（所有 category 共用）
  - len(options) <= constraints.max_options
  - options[constraints.recommended_position - 1] 標記為 recommended
  - 若 scenario 在 recommended_must_be.when_in 範圍內，
    options[recommended_position-1].label 必須包含 then_label_contains_one_of 任一字串

step 5: 驗證 PC-066 反模式
  - options[0] 不得為「直接執行」「跳過評估」「快速處理」「先做了再說」等省力路徑
  - 命中 → exit 2 + 提示改用 WRAP/多視角

step 6: 驗證 PC-064 假選項訊號
  - 任一 option.description 含「顯然不適用」「雖然但」「不推薦但」 → exit 2

unknown scenario 處理：
  - 不阻擋（exit 0）
  - 寫入 stderr 警告：「scenario 未註冊，建議在 _registry.yaml 補齊 scenario_id」
  - 累積觀測：count + log 至 .claude/logs/auq-unknown-scenarios.log，週期審查補齊 registry
```

**退出碼**：

| 退出碼 | 行為 | 觸發條件 |
|-------|------|---------|
| 0 | 放行 | 全部驗證通過 / unknown scenario / scenario YAML 載入失敗（保守） |
| 2 | 阻擋 + stderr | 選項池不符 / constraints 違反 / PC-066 / PC-064 命中 |

**stderr 訊息格式**（IMP 階段參考）：

```

  - Recommended 不在第 1 位（實際在第 3 位）

修正方式：
```

| **Phase 2：static-strict** | 8 個靜態池場景 | 對靜態池 strict 驗證（exit 2）；其他場景仍 observe-only | 4 個半動態場景 YAML 完成 + 觀測 ≥ 30 違規攔截案例 |

**Phase 1 風險緩衝**：

- Hook 預設 mode=`observe-only`（環境變數 `AUQ_SCHEMA_MODE=observe`）
- IMP 階段先實作 hook 但 mode=observe，shadow 跑 1-2 weeks 確認無誤殺再切 strict
- 每 scenario YAML 加入 `enforcement_mode` 欄位（observe / strict / disabled），允許單一 scenario 獨立切換

**回滾路徑**：

| 觸發 | 行動 |
|------|------|
| Hook 大量誤殺（> 10% 合法 AUQ 被擋） | 全域切回 `AUQ_SCHEMA_MODE=observe` |

### 設計交付物 5：「我可能錯在哪裡」自問段

**質疑 1：scenario_id fingerprint 匹配可能誤判**

- 風險：question 文字相似但語意不同（如 #11a 與 #11b 都含「Ticket 已完成」前綴）

**質疑 2：靜態池假設可能過於樂觀**

- 風險：實際 PM 對「靜態場景」也常加入動態變數（如 #16 加上 commit hash 引用）
