### PM 前置定位

**Ticket where.files 修正**：原列 `src/utils/AppLogger.js` 不存在，實際 Logger 位於 `src/core/logging/Logger.js`（main，470+ 行）+ `src/core/logging/Logger.simple.js`。

### 根因定位

**問題位置**：`src/core/logging/Logger.js:181-228` `_consoleOutput` 方法
```js
_consoleOutput (level, logEntry) {
  const prefix = `[${level}]`
  // ...
  case 'INFO':
    console.info(prefix, logEntry)  // ← 第二參數 logEntry 為 object
}
```

**症狀**：W6-012.1 實機 MCP 驗證觀察 console 大量 `[INFO] [object Object]`（library page 16+ 次、overview page 多次）。

**根因**：`console.info` 對 object 第二參數依賴 DevTools UI 展開，但：
- chrome-devtools-mcp `list_console_messages` 取得 console message text 形式
- object 第二參數降為 `[object Object]` toString fallback
- 日誌寫到 storage/檔案/CI 也會丟失 object 結構

### Logger logEntry 結構

```js
{
  timestamp,
  level,
  name,
  messageKey,
  message,
  data: { ... }  // 業務 payload，最關鍵
}
```

### 修復方向（依 ticket strategy）

1. `JSON.stringify(logEntry, replacer, 2)` 展開為可讀字串，circular reference replacer 防護
2. 改為 `console.info(\`[\${level}] \${jsonString}\`)`——單字串輸出
3. 保留 stack trace（Error 物件用 `error.stack`）
4. 單元測試覆蓋 object / circular / Error 三 payload

### MCP 驗證計畫

修復後重複 W6-012.1 MCP 流程（library 提取 + overview 開啟），觀察 console 不再有 `[INFO] [object Object]`，改為含 messageKey、data 欄位的可讀字串。

### 設計決策

**1. 為何採單字串輸出取代 `console.info(prefix, logEntry)` 雙參數**

| 方案 | 優點 | 缺點 |
|------|------|------|
| 原 `console.info(prefix, logEntry)` | DevTools UI 可展開 | chrome-devtools-mcp / storage / CI 取 text 時降為 `[object Object]` |
| **採 `console.info(\`prefix serialized\`)`** | 任何 text 取得通道都可讀 | DevTools UI 看到字串（縮排 2 仍可讀） |
| 用 util.inspect | 輸出更漂亮 | Node 限定，瀏覽器無；違反 Chrome Extension 環境要求 |

選單字串輸出，犧牲 DevTools UI 物件展開體驗，換取 MCP / CI / storage 通道一致可讀。

**2. 循環引用防護用 `WeakSet` 而非 try/catch retry**

- `JSON.stringify` 原生對循環引用拋 `TypeError`
- WeakSet 在 replacer 中追蹤已序列化物件，遇重複時回傳 `'[Circular]'`
- 不依賴 try/catch retry：性能更佳、行為可預測、輸出含 '[Circular]' 可辨識位置

**3. Error 物件展開 `{ name, message, stack }`**

- 標準 `JSON.stringify(new Error())` 回傳 `'{}'`（Error 屬性多為 non-enumerable）
- replacer 偵測 `value instanceof Error` 主動展開三欄
- 保留 stack 給 debug，name 給錯誤分類

**4. 序列化失敗的最後防線**

- 雙層 try/catch：第一層 JSON.stringify 失敗時降級為含 messageKey 的最小 JSON
- 第二層失敗時回傳固定字串 `'[Logger Serialize Failed]'`
- 確保 Logger 在任何 payload 下都不拋例外影響業務流程
