### 問題根因

Handoff 機制偏離設計初衷的根因有三層：

**Root Cause 1：direction 概念模型缺統一分類**
- `constants.py` 定義兩組：`TASK_CHAIN_DIRECTION_TYPES = ("to-sibling", "to-parent", "to-child")` 和 `NON_CHAIN_DIRECTION_TYPES = ("context-refresh", "next-wave")`
- `handoff.py:1228` 的 `_VALID_AUTO_DIRECTIONS` 額外接受 `to-source`，但 `to-source` 不在任一常數組中
- `handoff_utils.py:26` 的 `_KNOWN_DIRECTION_VALUES` 從 constants 衍生，也不含 `to-source`
- 結果：`to-source` 是 auto handoff 合法方向，但對 GC、stale 判定、is_valid_direction() 都是「未知類型」

**Root Cause 2：GC 非任務鏈分支邏輯反轉**
- `handoff_gc.py:78-81`：非任務鏈分支判定 `from_status != "completed"` 時才標 stale
- 語意：ticket 已 completed（line 64 確認）但 handoff 建立時就已是 completed → 不算 stale
- 但 `handoff_utils.py:233-237` 的 `check_stale` 邏輯不同：`is_ticket_completed(from_ticket)` 直接標 stale
- **GC 與 utils 判定邏輯不一致**（GC 獨立重寫而非呼叫 utils，ARCH-020 同構問題）

**Root Cause 3：handoff 缺絕對 target 指向**
- 設計初衷：handoff = 讓下 session 快速找到「該做的 ticket」（target）
- 實際實作：handoff JSON 記錄 `ticket_id`（= from_ticket，來源）+ `direction`（相對方向）
- `to-source` 語意矛盾：from_ticket 自己已 completed，direction 指回 source，但 sources_declared=0 → 無目標可指向
- SessionStart / Stop hook 讀 pending/ 時看到此 JSON，解讀為「有待恢復任務」，實際無任務可恢復

### AC1: L1 立即修復清單與優先序

L1 目標：消除孤兒 handoff JSON 的產生與累積。

| 修復項 | 優先序 | 說明 |
|--------|--------|------|
| L1-A: GC 改用 handoff_utils.check_stale | P1 | handoff_gc.py:54-81 的 _collect_stale_handoffs 改為呼叫 handoff_utils.check_stale(record_dict)，消除 ARCH-020 重複邏輯。GC 目前的非任務鏈判定（from_status != completed）與 utils 不一致是 W10-047.4 漏判的直接原因 |
| L1-B: constants.py 決策：移除 to-source | P1 | to-source 不加入任何常數組，改為從 _VALID_AUTO_DIRECTIONS 移除。理由：(1) chain_analyzer 不產生此方向 (2) 無自動化流程使用 (3) 語意矛盾（自己 completed 指回 source 但 source 可能不存在）。若有「回溯 source」需求，應由 source_ticket 欄位 + runqueue 路由處理，不需要 handoff direction |
| L1-C: handoff --auto 建立前 terminal check | P2 | 在 _execute_auto_handoff 加入防護：若 ticket status 為 terminal（completed/closed）且 direction 為非任務鏈，拒絕建立 handoff JSON（completed ticket 的非任務鏈 handoff 無消費者）|

**Spawn 規劃**：一張 IMP ticket 涵蓋 L1-A + L1-B + L1-C（跨進程同步修復，符合認知負擔豁免條款：2 進程 / ≤5 檔 / 同根因）。

### AC2: L2 中期改善設計

| 改善項 | 說明 | Schema 變更 | 向後相容 |
|--------|------|------------|---------|
| L2-A: handoff --next <target-ticket-id> | 新增 CLI 子命令/flag，直接指定「下 session 該做的 ticket」，繞過 direction 間接指向。JSON 新增 `target_ticket_id` 欄位 | 新增選填欄位 target_ticket_id | 相容：舊 JSON 無此欄位時 fallback 至 direction 邏輯 |
| L2-B: SessionStart 提示語改寫 | 從「待恢復任務」改為「下 session 建議項目」。語意差異：「恢復」暗示中斷，「建議」表達指引 | 無 schema 變更 | 純文字改動，完全相容 |
| L2-C: Stop hook 跳過 terminal JSON | handoff-auto-resume-stop-hook.py 的 pending 掃描加入過濾：ticket status 為 terminal 時不計入「待恢復」 | 無 schema 變更 | 相容（GC 最終會清理，此為即時過濾） |

**風險評估**：L2-A 涉及 handoff JSON schema 擴展，需確保所有讀取端（GC / SessionStart / Stop / resume）都處理 target_ticket_id 欄位的有無。建議 L2-A 獨立 IMP、L2-B+C 可合併一張 IMP。

### AC3: L3 長期 schema 拆分可行性評估

**三選結論：視 L2 結果再評（狀態 b，需建 follow-up ticket）**

分析：
- chain handoff（任務鏈內部跳轉：to-sibling/to-parent/to-child）語意清晰，目標由 direction 後綴 `:TARGET_ID` 指定
- session handoff（跨 session 交接）目前由 context-refresh / next-wave 承擔，語意為「提供 context 給下 session」
- 兩者混用同一 JSON schema 不是當前問題的根因（根因是 direction 常數不一致 + GC 重複邏輯）
- L2-A 的 target_ticket_id 欄位若落地成功，session handoff 的絕對指向需求已被滿足，拆分收益降低

**結論**：不建議現階段拆分。建 follow-up ANA ticket，trigger 為「L2-A 落地 + 運行 2 個 Wave 後重評」。

### AC4: direction 概念模型釐清

**概念對照表：source 視角 vs target 視角**

| direction | 座標系 | from_ticket 語意 | 指向目標 | 問題 |
|-----------|--------|-----------------|---------|------|
| to-child | source 視角 | parent | child ticket（direction 後綴） | 明確 |
| to-sibling | source 視角 | sibling A | sibling B（direction 後綴） | 明確 |
| to-parent | source 視角 | child | parent ticket（direction 後綴） | 明確 |
| context-refresh | 無方向性 | 任意 ticket | 無目標（自身 context 刷新） | 明確 |
| next-wave | 無方向性 | 任意 ticket | 無目標（Wave 跨越提示） | 明確 |
| to-source | **矛盾** | completed ticket | source_ticket 欄位指向的 ticket | **from_ticket 已 completed 時 source 已無可操作性；sources_declared=0 時無目標** |

**決策建議**：移除 to-source。理由：
1. 任務鏈方向（to-child/sibling/parent）已覆蓋所有有向跳轉
2. 非任務鏈方向（context-refresh/next-wave）覆蓋無向場景
3. 「回到 source」的需求由 source_ticket 欄位 + runqueue --context=resume 路由處理，不需 handoff direction 承擔
4. to-source 在 constants.py 的兩組常數皆不存在，實質是未分類的「幽靈方向」
