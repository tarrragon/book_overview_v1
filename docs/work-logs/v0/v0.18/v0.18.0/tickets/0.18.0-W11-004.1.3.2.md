## Problem Analysis

### 問題根因

Phase 4 bay-quality-auditor TD-2 高風險發現：

`_is_meta_task_prompt(prompt)` 命中時，`_detect_keyword_conflicts` 在 L563 位置直接 `return []`，整體跳過所有衝突偵測。

### 安全漏洞場景

PM 派發「修改 .claude/rules/...」的 meta-task 時，若 prompt 同時包含真實越界行為，該越界會靜默通過：

- 範例 1：`修改 .claude/rules/core/quality-baseline.md 規則 X，同時 git commit 新增章節`
- 範例 2：`編輯規則文件加入新約束，請實作 src/foo.py 的 helper`
- 範例 3：`更新 FORBIDDEN_KEYWORD_MAP 加入 git push pattern 後執行 git push origin main`

上述 prompt 前 100 字元命中 meta-task pattern（修改 .claude/rules/ / 編輯規則文件 / 更新 FORBIDDEN_KEYWORD_MAP），使整個 prompt 免於檢測，但後段的真實違規行為未被攔截。

### 影響範圍

- `.claude/hooks/agent-dispatch-validation-hook.py::_detect_keyword_conflicts`

## 實作摘要（TD-2 per-match 降級）

### 變更點 1：_detect_keyword_conflicts（hook.py L548-）
- 移除 meta-task 整體豁免（原 L563-565 的 return []）
- 仍呼叫 _is_meta_task_prompt 取得 (is_meta, meta_reason)
- 新增 _META_TASK_VERBS = {"修改檔案","編輯檔案","更新檔案","修改","編輯","更新"}
- 每筆 conflict 若 is_meta 且 keyword in _META_TASK_VERBS → 附加 whitelist_reason
- 其他 keyword（實作 / git commit / git 寫入 / ticket CLI 等動作性越界）即使 prompt 為 meta-task 仍作真違規

### 變更點 2：_emit_keyword_conflict_warning_if_any（hook.py L693-）
- 分離 meta_filtered（有 whitelist_reason）vs real_conflicts
- meta_filtered 僅 logger.debug，不寫 stderr、不寫 events.jsonl
- real_conflicts 維持原行為（stderr 警告 + logger.info + events.jsonl，僅含 real）

### 設計理由
- meta-task 合法情境（修改規則 / agent md）命中「修改/編輯/更新」是語境必然 → 降級
- 但 git commit / push / 實作 / ticket CLI 等動作性動詞與規則文件編輯無關 → 真違規
- per-match 粒度避免 TD-2 安全漏洞：前 100 字元命中 meta-pattern 即免除所有檢測
### Context Bundle（Phase 3b 派發用）

**實作檔案**：`.claude/hooks/agent-dispatch-validation-hook.py`（主 repo，豁免 worktree）

**當前行為**（整體豁免，錯誤）：
```python
# _detect_keyword_conflicts 中
is_meta, _ = _is_meta_task_prompt(prompt)
if is_meta:
    return []  # ← 跳過全部偵測，漏報真違規
```

**修復策略**：Meta-task 命中改為 **per-match 降級**，不跳過偵測：
- 保留 `_is_meta_task_prompt` 函式
- 從 `_detect_keyword_conflicts` 開頭的「整體 return []」移除
- 在規則 D 命中時，conflict 仍生成，但改寫入 logger.debug（不進 events.jsonl）
- 需要：`_detect_keyword_conflicts` 回傳能分辨 meta-match 的方式，或由 caller `_emit_keyword_conflict_warning_if_any` 在 meta-task 時靜默不 print stderr、不寫 events.jsonl，但仍 logger.debug

**推薦實作骨架**：
1. 在 conflict dict 加 `whitelist_reason` 欄位（or similar），記錄 meta-task 的 reason
2. `_detect_keyword_conflicts` 返回全部 conflicts（含 meta-match 的）
3. `_emit_keyword_conflict_warning_if_any` 過濾：meta-task 命中的 → logger.debug；其他 → stderr + events.jsonl

**測試要求**（對應 AC）：
- 測試 1：`修改 .claude/rules/X.md 同時 git commit` → `git commit` 仍應觸發（寫 events.jsonl）
