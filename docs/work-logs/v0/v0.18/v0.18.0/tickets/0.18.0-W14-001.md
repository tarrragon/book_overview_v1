## Problem Analysis

_verify_
### 問題根因

**直接原因**：`_shift_single_ticket`（version_shift.py:187）接受 `ticket_path: Path` 參數（測試 temp 目錄），但內部呼叫 `load_ticket(from_version, ticket_id)`，後者透過 `get_project_root()` 解析真實 git toplevel，不尊重傳入的 ticket_path。因此測試場景下 load_ticket 回傳 None，`_shift_single_ticket` 回 (False, None)，update_count 停在 0。

**為何生產環境看不到**：生產環境 `tickets_dir = project_root/docs/work-logs/v{ver}/tickets`，load_ticket 解析出的路徑恰好一致，能正常工作。測試用 tempfile 目錄時路徑解析失配暴露問題。

**設計異味**：project_root 參數從 `_shift_ticket_files` 往下傳遞時中斷（未傳給 load_ticket），違反 dependency injection。

### 影響範圍

- **目標檔案**: .claude/skills/ticket/ticket_system/commands/version_shift.py（`_shift_single_ticket` L168-217；`_process_ticket_for_cross_refs` L329-379 也有同樣問題）
- **測試檔案**: .claude/skills/ticket/tests/test_version_shift.py（22 tests 中 test_basic_version_shift 等多個會觸發）
- **生產影響**: 目前無回歸（因為生產環境 project_root 恰好與 load_ticket 隱式解析一致），但資料完整性依賴巧合

### 相關 Error Pattern

無既有 error-pattern。屬 PC-068 Phase 3a 既有 utility 調用時未驗證參數邊界的變體（load_ticket 的路徑解析是隱性依賴）。

### 修復策略（Phase 3a）

**選項**：直接從 ticket_path 讀檔並用 parse_frontmatter 解析，避開 load_ticket 的隱式路徑解析

**理由**：
- 最小變動：只改一個函式，不動 load_ticket 公共介面
- 測試隔離性：ticket_path 已是正確目標，無需再透過版本解析
- 保持一致：save_ticket 已接受 Path 參數（L213），讀寫兩端對稱

**實作要點**：
1. `_shift_single_ticket` 改為直接讀 ticket_path 檔案
2. 用 parse_frontmatter 分離 frontmatter/body
3. 保留原有 body（_body 附加到 dict 以便 save_ticket 恢復）
4. `_process_ticket_for_cross_refs` 同樣處理

### Phase 4 重構評估
