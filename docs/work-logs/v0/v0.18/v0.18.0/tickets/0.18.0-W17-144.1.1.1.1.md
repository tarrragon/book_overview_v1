## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W17-144 收斂於「污染源在 AI token 預測機率分佈」（普遍 token 偶發），但用戶觀察否定此推論：**正常對話無簡體字，但 AUQ payload 60-70% 命中率**。token 偶發應該所有輸出都偶爾出現（< 1%），AUQ 命中率遠超此值，指向 AUQ-specific 機制。

### 影響範圍

`AskUserQuestion` tool 的 payload 生成路徑——從模型決定呼叫 AUQ 到 hook 攔截前的所有處理層。

### 相關 Error Pattern

- PC-072（AUQ payload 字元集污染）— 主要追蹤對象
- PC-085（CJK codepoint 鄰近性）— 物理基礎
- W17-144 推論修正——「token 偶發」假設不成立

### 候選機制假設

| AUQ 命中遠高於對話正文 | 普遍 token 偶發 | H3 |

**留下的最可能假設**：**H3 — 模型在生成 AUQ JSON 時切換到「結構化資料 mode」，繁體 prime 弱化**。

### H3 的潛在物理機制

模型生成「自由對話文字」時：
- 注意力集中於語義流暢度
- 繁體 prime 強烈（用戶 / 系統 prompt 使用繁體）
- token 預測穩定挑繁體形

模型生成「AUQ JSON 結構」時：
- 注意力分散到「JSON 結構正確性」+「question/options/labels 簡潔表達」
- 繁體 prime 被「結構化資料常見訓練樣本」（多為簡體中文網站 / 中國大陸 SaaS）稀釋
- 短 label / description 中 token 預測偶爾挑簡體形

這解釋了為何：
- AUQ payload 命中字集中於「实/独/发/决」等高頻動詞 / 形容詞（短 label 的選詞）
- 對話正文長段落幾乎不命中（語義流暢的繁體 prime 強）

### 結論：H3「結構化資料 mode 繁體 prime 弱化」為主要假設

| 結論項 | 內容 |
|--------|------|
| W17-144 推論修正 | 從「普遍 token 偶發」修正為「AUQ-specific 結構化資料 mode 弱化」 |
| 真根因（高可信度）| H3 — AUQ JSON 結構化生成時繁體 prime 被簡體訓練樣本稀釋 |
| 完全排除假設 | H4 / H1 / H2 / H5 / H6（中介層轉換不存在）|
| 仍未驗證假設 | H3 的具體物理機制（attention head 切換 / training data 分布）—— 屬模型內部黑盒，外部無法直接驗證 |

### 防護策略不變

無論真根因為 H3 還是其他模型內部機制，**防護策略不變**：
1. `askuserquestion-charset-guard-hook.py` 在 PreToolUse 階段攔截（已運作）
2. detector.py self-test 5 層 + Method 6 hook log 反推擴充 ANCHORS（W17-144.1.1.1 已落地）
3. PM 寫 AUQ 時注意力分配——本 session 觀察到「對 PC-072 主題敏感時更易自證」（meta-層污染）

H3 不需要修復程式碼層（無法修復模型內部 prime），但需要：
- 維持並強化 hook 層攔截（運作中）
- PM 自我意識：寫 AUQ 短 label 時刻意 double-check 繁體形
- 持續收割 hook log 擴充 ANCHORS / hook SIMPLIFIED_CHARS
