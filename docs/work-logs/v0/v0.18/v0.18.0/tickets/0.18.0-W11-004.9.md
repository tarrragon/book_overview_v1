## Solution

### 根因
`scan_hook_errors` 舊實作以 `HOOK_ERROR_KEYWORDS = ("ERROR", "FAIL", "Exception", "Traceback", "TypeError", "NameError")` 做純字串 `in` 比對，對 log 檔全文逐一匹配關鍵字。INFO/DEBUG 等級訊息若回放使用者命令（例如 `grep -rn 'ERROR' src/`、命令含 `FAIL`/`Exception` 字樣）即被誤判為 Hook 錯誤，造成 PM 視野循環污染。

### 修改檔案
1. `.claude/hooks/agent-commit-verification-hook.py`
   - 新增 `import re`
   - 移除 `HOOK_ERROR_KEYWORDS` tuple，改為兩個 regex：
     - `HOOK_ERROR_LOG_LEVEL_RE = re.compile(r"\] (ERROR|CRITICAL|FATAL) -")` — 精確匹配 `[timestamp] LEVEL - message` 格式中的 ERROR/CRITICAL/FATAL log 行
     - `HOOK_ERROR_TRACEBACK_RE = re.compile(r"^Traceback \(most recent call last\):", re.MULTILINE)` — 行首 Python Traceback 標記
   - 新增 `_has_hook_error(content: str) -> bool` 函式封裝判斷邏輯（附 docstring 說明設計意圖）
   - `scan_hook_errors` 迴圈內 `any(kw in content ...)` 改呼叫 `_has_hook_error(content)`

2. `.claude/hooks/tests/test_agent_commit_verification_hook_scan_errors.py`（新增）
   - 13 個單元測試涵蓋 AC 三個場景 + WARNING/DEBUG 降噪 + 邊界案例

### 設計決策
- **為何不直接修 regex 常數而新增 `_has_hook_error` 函式**：封裝雙判斷（log level + Traceback）便於日後擴充（例如加 stack trace 偵測），也讓 scan 迴圈保持可讀。
- **為何 WARNING 不視為錯誤**：W11-004.9 目標是降誤報，WARNING 在現行 hook 多用於非致命狀況（git timeout、dispatch lookup miss），納入會延續誤報問題。若日後需要，另行評估。
- **Traceback regex 用 `^` + MULTILINE**：避免 INFO log 中提到「參考 Traceback 文件」等誤觸（已於 `test_traceback_in_middle_of_line_not_detected` 驗證）。
