## Problem Analysis

### 問題根因

PC-097 揭露同一筆 I/O 結果（檔案讀取、git status、HTTP 請求、DB 查詢）在單次請求生命週期內被兩個或多個位置各自獨立呼叫，導致：

1. **效能浪費**：相同 I/O 執行 N 次（大 repo 的 git status 數百 ms / 次）
2. **資料漂移**：兩次呼叫間實際狀態若有變動，呼叫方拿到不一致快照（race window）
3. **時間欄位不一致**：第一次取得 computed_at_T1 寫入 state，第二次取得時實際是 T2 但仍標 T1 → 時間戳謊報
4. **責任分裂**：兩處呼叫各自擁有「我的 git status」心智模型，難以追蹤該以哪份為準

### 影響範圍

**已修復（無需處理）**：

- `commands/track_snapshot.py:233-234`（PC-097 案例 1）：已改為從 `state.uncommitted_files` 取值，註解明示「不再呼叫 git status」
- `lib/checkpoint_state.py:381` 的 `_read_git_status` 已為單一採集點

**待稽核範圍**：

| 路徑 | 稽核目標 |
|------|---------|
| `.claude/skills/ticket/ticket_system/commands/track_agent_status.py` | 與 lib 層 checkpoint_state 是否有重複 git/file I/O |
