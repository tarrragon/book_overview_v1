### MigrationService 設計

- `constructor({ storage, logger?, steps? })`：storage 必填；缺則拋錯（fail-fast）。
- 預設 step 清單由 `createDefaultSteps()` 產生，目前僅含 `v1-to-v2`（包裝 `src/data-management/migration/v1-to-v2.js#migrateV1ToV2`）。
- `applies(prev, curr)` 採寬鬆策略：v1-to-v2 永遠 applies=true，實際是否跳過交由 step 內部依 `storage.schema_version` 判定（避免重複實作版本判定邏輯）。
- `migrate(prev, curr)` 回傳結構化 report：`{ success, executed[], skipped[], failed? }`。
- 三段失敗保護：
  1. `applies` 拋例外 → `failed.phase='applies'`，後續 step 中止。
  2. `run` 拋例外 → `failed.phase='run'`，後續 step 中止，error 寫入 logger.error。
  3. `run` 回傳 `{ error }`（軟失敗，如 v1-to-v2 配額不足）→ 同樣中止後續。
- `registerStep(step)` 公開介面供 W6-012.2.2.2 後續追加 cover-to-reader step；介面驗證（id/applies/run）+ 重複 id 偵測。

### install-handler DI wire

- 既有：`this.migrationService = dependencies.migrationService || null`（onUpdated 若無注入則靜默跳過 migration）。
- 變更後：`dependencies.migrationService || this._createDefaultMigrationService()`。
- `_createDefaultMigrationService()` 在 `chrome.storage.local` 可用時自動建立預設實例；不可用（如測試環境未 mock chrome）時回傳 null 並寫 warn，不阻斷 background 啟動。
- onUpdated 既有呼叫 `this.migrationService.migrate(prev, curr)` 不需改動，自然走預設 step 清單。
