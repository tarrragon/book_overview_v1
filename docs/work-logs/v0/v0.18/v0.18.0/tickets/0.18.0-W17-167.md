## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->


### 問題根因

ANA complete 流程存在「規劃-落地」斷裂：ANA Solution 章節寫了 spawn 規劃（IMP/DOC/ANA 表格），但 complete 前無機制驗證規劃是否已轉為實際 ticket。

**Root Cause 1：acceptance 勾選與 spawn 落地是兩件事**
- ANA acceptance 常含「產出 spawned IMP/DOC 清單」條目
- 代理人在 Solution 寫了表格即勾選 `[x]`，但「寫表格」不等於「建 ticket」
- acceptance-gate-hook 檢查 `[x]` 勾選狀態，不檢查 frontmatter spawned_tickets 與 Solution 表格的一致性

**Root Cause 2：spawned_tickets 被重定位為「弱 metadata」後失去強制力**
- W17-120.2 / PC-091 將 ANA 落地統一用 children（`--parent <ANA-ID>`），spawned_tickets 不再阻擋 complete
- 但 ANA 由 saffron（分析代理人）執行，saffron 無權建 ticket（ticket create 是 PM 職責）
- 結果：saffron complete 時 Solution 有規劃但 frontmatter 必然為空，PM 若未察覺則永久遺忘

**Root Cause 3：check_ana_has_spawned_tickets 只檢「有無」不檢「一致性」**
- 現行檢查：spawned_tickets 和 children 皆空時輸出 warning（不阻擋）
- 缺失檢查：Solution 內 spawn 規劃表格列了 N 項，但 spawned_tickets + children 只有 M 項（M < N）
- 語意解析難度：Solution 表格格式不標準化，正則比對可行但誤報率需評估

### 三層防護設計

#### L2 Hook 設計：ANA complete 前 spawn 一致性檢查

**規格**：在 `acceptance-gate-hook.py` Step 2.5 之後新增 Step 2.5.2「ANA Solution spawn 一致性檢查」。

**檢查邏輯**：
1. 解析 Solution 章節，以正則掃描 spawn 規劃表格行（匹配 `| IMP |`、`| DOC |`、`| ANA |` + `P[0-3]` 的行）
2. 計算規劃數量 N（表格行數）
3. 讀取 frontmatter spawned_tickets 數量 S 和 children 數量 C
4. 若 N > 0 且 S + C == 0：**阻擋 complete**，輸出訊息「Solution 規劃了 N 項 spawn，但 spawned_tickets 和 children 皆為空。請先建立 ticket 或在 Solution 標註『無需建 ticket：[理由]』」
5. 若 N > 0 且 S + C > 0 但 S + C < N：**warning**（不阻擋），提示「規劃 N 項但實際只有 S+C 項，請確認是否有遺漏」
6. 豁免條件：Solution 內含「無需建 ticket」或「不 spawn」等顯性否定標記時跳過檢查

**Why**：acceptance 勾選「產出 spawned 清單」只檢查文字產出，不檢查 ticket 實際建立。此檢查在 complete 時攔截「寫了規劃但沒建 ticket」的斷裂。

**Consequence**：不加此檢查，ANA Solution 的 spawn 規劃會靜默遺忘（W17-162 案例已證明）。

**Action**：建 IMP ticket 實作此檢查。

**誤報率評估**：
- 正則匹配 `| (IMP|DOC|ANA) |` + `P[0-3]` 的表格行，誤報風險低（非表格的散文不含此格式）
- 少數 ANA 的 Solution 使用非表格格式列 spawn 規劃（如散文「建議建 IMP 處理」），此類不會被偵測，為 false negative（可接受，降低誤報優先）
- 豁免標記（「無需建 ticket」）防止合法不 spawn 的 ANA 被誤擋

**實作可行性**：高。正則解析 Solution 章節 + frontmatter 計數比對，複雜度與現有 checker 相當。建議新增 `acceptance_checkers/ana_spawn_consistency_checker.py`。

#### L3 規則升級

**規則 5 延伸（quality-baseline.md）**：

現行規則 5：「所有發現必須追蹤」。

修訂草案：在規則 5 適用場景段落新增：

> **ANA Solution 內 spawn 規劃**：ANA Solution 章節若含 IMP/DOC/ANA spawn 規劃表格，規劃項目必須在 complete 前轉為實際 ticket（spawned_tickets 或 children），或在 Solution 中顯性標註「無需建 ticket：[理由]」。acceptance 勾選「產出 spawned 清單」不等於「已建 ticket」。

**ticket-lifecycle.md 修訂草案**：

在 ANA complete 條件段落新增：

> ANA complete 前置條件（追加）：
> - Solution 章節的 spawn 規劃表格項目已全部轉為 ticket（spawned_tickets 或 children 欄位記錄），或已在 Solution 顯性標註豁免理由
> - acceptance-gate-hook Step 2.5.2 檢查通過（或使用 `--force` 豁免並記錄理由）

**ticket-body-schema.md 修訂草案**：

在 ANA Solution 章節規格新增子節：

> ### Spawn 落地確認（ANA 專屬，選填但被 hook 偵測）
> 若 Solution 含 spawn 規劃表格，必須在 complete 前確認：
> - [ ] 所有規劃項目已建 ticket（spawned_tickets 或 children 記錄）
> - [ ] 或標註「無需建 ticket：[理由]」

#### L4 行為層：ANA 驗收 checklist 三明示

PM 派發 ANA 完成驗收時，checklist 必含以下問題：

| 問題 | Why | Consequence | Action |
|------|-----|-------------|--------|
| Solution spawn 規劃表格的每一項，是否都已建對應 ticket？ | Solution 寫了規劃不等於建了 ticket，acceptance 勾選只檢文字產出 | 未建 ticket 的規劃會靜默遺忘，違反規則 5（所有發現必須追蹤）且觸發 PC-093 模式 | 比對 Solution 表格 vs frontmatter spawned_tickets/children，缺漏立即建 ticket |
| frontmatter spawned_tickets 或 children 數量是否 >= Solution 規劃數量？ | 數量不一致代表有遺漏 | 部分規劃遺忘，後續 Wave 無法排入 | 逐項核對，補建遺漏 ticket |
| 若合法不 spawn，Solution 是否有顯性否定標記？ | 否定標記讓 hook 豁免且讓後人理解決策理由 | 無標記時 hook 阻擋 complete，或後人誤以為是漏建 | 在 Solution 寫「無需建 ticket：[具體理由]」 |
