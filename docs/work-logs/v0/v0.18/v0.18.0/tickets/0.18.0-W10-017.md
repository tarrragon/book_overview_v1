### 已知可自動化候選（初步）

1. **git commit 狀態檢查**：腳本化（pre-/clear hook 已部分實現）
2. **dispatch-active.json 清點**：PC-050 已定義，但需要每次手動 cat；可做成 `ticket check-dispatch` 命令
3. **worklog 更新提醒**：Hook 可自動檢測 ticket complete 後 worklog 是否同步
4. **下一個 Ticket 選擇**：`ticket track snapshot` 已存在，但 PM 仍需思考路由；可考慮 `ticket suggest-next`
5. **AskUserQuestion #11 觸發**：Wave 收尾 hook 已實現，但前置條件（commit 狀態）檢查分散
6. **Checkpoint C1/C1.5/C2 定位**：可由腳本從 git log + dispatch-active 推導

### 已知必須 PM 判斷（不可自動化）

- 代理人失敗根因分析（需 context 理解）
- Ticket 優先級決策
- Skip-gate 事件分類
- 跨 Ticket 依賴判斷

### CLI 指令優化方向（新增）

現有 `/ticket` 雖然底層是腳本，但輸出仍是「原始資料」需 PM 解讀：

| 現況 | 問題 | 優化方向 |
|------|------|---------|
| `ticket track list` | PM 要從 pending 清單中挑下一個 | `ticket suggest-next`：直接回傳建議 Ticket + 理由（依賴、優先級、當前 Checkpoint） |
| `ticket track snapshot` | PM 要從 git status / dispatch-active 推導 Checkpoint | `ticket checkpoint-status`：直接回傳「目前在 C1.5，下一步應 merge」 |
| `cat dispatch-active.json` | PM 要 parse JSON 判斷是否可驗收 | `ticket dispatch-check`：直接回傳「可驗收 / 仍有 2 個代理人運行中」 |
| `git status` + `git log` 推測 | PM 要組合多指令判斷狀態 | `ticket handoff-ready <id>`：直接回傳「可 /clear / 需先 commit main / 需先驗收代理人」 |

**核心設計原則**：CLI 指令應輸出「決策建議 + 理由」，而非「原始狀態資料」。PM 只做最終確認，不做中間推論。
