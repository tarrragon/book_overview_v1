### 分類 × 應否擴充

| 類型 | 建議 | 理由 |
|------|------|------|
| 純 alias（list→ids）| 擴充 + warning | 摩擦為零，Postel 原則 |
| 同義輸入（逗號=空格）| 擴充 | 零風險，符合 Unix 多 CLI 慣例 |
| 結構化 flag vs positional | 雙軌支援 | backward compat + 結構化（git/docker 範例）|
| section 值匹配 | 需 ANA | 是設計邊界還是 UX 缺口 |
| 子命令一致性（--yes）| 需 ANA | 設計遺產還是演進 |

### 應保留嚴格的場景（防護性 vs 摩擦性區分）

| 場景 | 嚴格理由 |
|------|---------|
| Type-aware body schema 必填章節 | 產品品質防護（W17-016.x） |
| Acceptance gate（spawned 未完不可 complete）| W12-004/W13-001 刻意防護 |
| Hook 格式要求（`Ticket: {id}`）| PC-040 對抗 prompt 污染 |
| acceptance 4V 原則 | 規格品質 |

**判別準則**：嚴格是「防護使用者免於實質錯誤」還是「懲罰使用者猜錯語法」？前者保留、後者擴充。

## W 階段擴增（審查後補 4 選項 F/G/H/I + 3 順序 J/K/L）

### 原 W 階段的結構性偏見

原 A/B/C/D/E 全假設「新增工具」。Plan 視角揭露結構性偏見：整份分析預設「pm-rules 不動 + 新增 CLI/Hook/Skill」，排除「改造既有載入機制」與「零新工具」路線。

### F 方案：SessionStart Hook 自動注入

- 新 session 開場 Hook 自動把 completion-checkpoint-rules 14 項摘要 cat 到 context
- 繞過 CLI/Skill 查詢，零 PM 主動 tool call
- 優點：被動注入 > 主動查詢；維護成本極低
- 缺點：增加每 session 固定 token 開銷（不論是否真的要 handoff）
- 成本：約 2 小時（寫 Hook + 摘要文件）

### G 方案：Skill Preload 精簡摘要

- 把 pm-rules 46 檔壓成一份 200-token 核心摘要，session 開始自動載入
- 利用既有 Skill 載入機制
- 優點：從源頭減少讀取需求，比 snapshot 增強更徹底
- 缺點：需先撰寫高品質摘要；摘要過簡可能漏細節
- 成本：約 8-12 小時（摘要 + Skill 整合）

### H 方案：純文件重構（不新增工具）

- 把 46 檔 pm-rules 合併精簡到 5 檔核心
- 刪掉「機械性檢查」改成「工具就這樣做，不用讀」（隱含的 guidance → 顯性的 action）
- 優點：**零實作成本**；PM 根本不需要新工具；可與 A/E 共存
- 缺點：重構範圍大；破壞性變更；需跨 Ticket 協調
- 成本：約 20-30 小時（拆分 + 合併 + 跨引用修正）

### I 方案：Memory 層自動狀態追蹤

- 用 serena memory（或 write_memory）在 checkpoint 轉換時自動記錄當前狀態
- PM 新 session 開場 read_memory 而非從 git+dispatch 推導
- 優點：持久化跨 session；符合 continuous-learning Skill 設計
- 缺點：需 Hook 觸發 memory 寫入；memory 膨脹風險
- 成本：約 12 小時（Hook + memory schema）

### 新增順序替代

- **J 序：觀測先行**（017.core 的 log 機制先做 4 週 baseline 量測 → 有數據再決 Phase 1 內容）

### 候選方案清單（5 個，非二元）

#### 方案 A：全 CLI 化（查詢+建議型子命令）

新增 4 個 ticket CLI 子命令：

| 命令 | 取代 | 輸出形式 |
|------|------|---------|
| `ticket track dispatch-check` | PC-050 cat+python3 片段 | 「仍有 N 個代理人：... / 無活躍派發，可繼續」 |

| `ticket track suggest-next` | PM 從 list 挑 | 「建議 W10-018（同 Wave 最高 P1，無阻塞）」 |

**缺點**：仍需 PM 主動呼叫；無強制性
**維護成本**：中（4 個新命令，需覆蓋現有 CLI 變更）

#### 方案 B：Hook 強制攔截

| Hook | 攔截時機 | 行為 |
|------|---------|------|
| PreToolUse `/clear` 攔截 | 用戶 invoke `/clear` 前 | 執行 5 項檢查，不通過則 block + 列出缺項 |
| UserPromptSubmit `handoff` 偵測 | 用戶輸入含 handoff 關鍵字 | 自動 cat 當前 checkpoint 狀態 |
| PostToolUse commit 後 | git commit 成功 | 自動跑 dispatch-check + 提示下一步 |

**優點**：強制性高；PM 無法忘記；Hook 生態已成熟
**缺點**：誤判風險（Hook 誤 block 合法操作）；可觀測性差（Hook 邏輯藏在 .py）
**維護成本**：高（Hook 錯誤偵測難）

#### 方案 C：SKILL Wizard（/handoff-to-clear）

新增 Skill：

```
/handoff-to-clear <ticket-id>
  → Step 1: dispatch-check
  → Step 2: git status + commit 提示
  → Step 3: worktree 檢查
  → Step 4: worklog 檢查
  → Step 5: handoff --status
  → Step 6: 執行 handoff
  → Step 7: AskUserQuestion: 確認 /clear?
```

**優點**：linear 流程易懂；PM 一個命令走完
**缺點**：靈活性低（必須完整走完）；與現有 Hook 重疊
**維護成本**：中（新 Skill + 需與現有 Hook 協調）

#### 方案 D：CLI + Hook 混合（查詢用 CLI，強制用 Hook）

| 層次 | 工具 | 責任 |
|------|------|------|
| 查詢層 | CLI 子命令（方案 A） | PM 主動查詢 checkpoint、handoff-ready、suggest-next |
| 強制層 | PreToolUse Hook | /clear 前強制執行 handoff-ready，不通過則 block |
| 提示層 | SessionStart reminder | 新 session 開場顯示待恢復 + 狀態摘要 |

**優點**：分層設計，各司其職；保留 PM 靈活性同時有強制防護
**缺點**：實作工作量大（CLI + Hook 都要動）
**維護成本**：高（跨 3 層需協調變更）

#### 方案 E：snapshot 增強（最小變更）

改 `ticket track snapshot` 輸出結構，新增「建議下一步」欄：

```
=== Project Snapshot ===
...（原有內容）

--- 當前建議 ---
  目前位置：C1.5（未提交 1 檔，無活躍代理人）
  建議動作：
  1. git add + git commit（未提交變更）
  2. 檢查 worktree（目前 main 無需合併）
  3. ticket handoff W10-017 --to-sibling W10-018

--- /clear Ready Check ---
```

**優點**：單點改動；不新增命令；PM 熟悉的入口
**缺點**：snapshot 輸出變長；推論邏輯集中在一處（可讀性挑戰）
**維護成本**：低（單一函式擴充）
