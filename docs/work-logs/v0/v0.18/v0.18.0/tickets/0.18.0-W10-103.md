### 問題根因

W10-080 grep 全掃描識別 7 個 `.claude/` 檔案含產品名稱（Readmoo）或書城列舉（博客來/Kindle/Kobo）。規則 8（reference-stability-rules.md）要求 `.claude/` 框架文件禁止引用專案層級識別符，因 `.claude/` 透過 sync-push/sync-pull 擴散到其他專案，產品名稱會變成跨專案誤導。

但「產品名稱出現」不等於「實質違反」——若產品名稱僅作「情境舉例」（如 dry-run 測試的具體任務範例、業務情境驅動的程式碼註解正例），泛化反而失去脈絡。故需逐檔評估違反強度與處理方式。

## 重現實驗結果

### 實驗方法

對 7 個候選檔案執行 grep 找出產品名稱出現位置，再用 sed 擷取每處 ±10 行上下文，依「產品名稱在語意中扮演的角色」判定 A/B/C 分類：

- **A 類**：產品名稱作為「框架描述」（如「本專案是 X」「目標網站 X/Y/Z」），跨專案 sync 後會錯誤宣告或誤導
- **B 類**：產品名稱作為「具體舉例」（如測試範例的系統名稱、業務情境驅動的程式碼註解、業務意圖正例），泛化會失去脈絡
- **C 類**：產品名稱為中性提及，可選擇性處理

依「角色」判定，7 檔可乾淨分類為：

- **A 類（2 檔）**：thyme-extension-engineer.md（聲明「本專案」）、oregano-data-miner.md（書城列表）——兩者均為「框架描述」性質，跨專案 sync 會誤導

- **C 類（0 檔）**：無中性案例
