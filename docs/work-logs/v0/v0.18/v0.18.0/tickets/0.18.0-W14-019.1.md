## Problem Analysis

### 問題根因

WRAP 框架（基於 Heath 兄弟《Decisive》）現有的 wrap-decision skill 主要描述 4 階段原則（Widen / Reality-Test / Attain Distance / Prepare to Be Wrong），但缺少以下實踐方法：

1. **多輪迭代搜尋方法論**：不是一次性搜尋，而是迭代精準化
2. **反向驗證（Consider the Opposite）實踐範本**：不只警告偏誤，提供具體執行格式
3. **悖論識別機制**：規則本身可能違反它要保護的價值
4. **自我暴露偏好實踐**：不假裝中立，透明化推理鏈

本 session 規則 5 設計過程實證了這些方法可顯著提升決策品質——四輪查詢從 8 個發散主題擴展到 28 個洞察（A-BB），第四輪反向搜尋更揭露了規則 5 設計的核心悖論（規則 5 本身是 paternalism）。這些方法價值跨情境可重用，必須提煉為 skill 擴充。

### 影響範圍

- 主檔：`.claude/skills/wrap-decision/SKILL.md`
- 衍生：`.claude/skills/wrap-decision/references/case-studies.md`（補入本 session 實證）
- 衍生：`.claude/skills/wrap-decision/references/project-integration/`（如有對應檔案）
- 跨專案：因 wrap-decision 屬框架資產，sync-push 後其他專案可同步受益

### 相關 Error Pattern

- **PC-080**（A 階段方案表面多元實際全指向同類型解法）：本 session 第一輪 7 選項犯了 PC-080，需在 wrap skill 加入「框架測試」前置檢查
- **PC-063**（ANA premature solution convergence）：本 session 多輪查詢避免了過早收斂，可作為正面案例
- **PC-072**（規則擺設）：規則 5 草擬時警示，wrap 擴充也須避免變成擺設章節

## 重現實驗結果

### 實驗方法

本 session 規則 5 設計過程作為實驗。觀察四輪查詢方法的演化、發現量、用戶介入點。

### 實驗執行

四輪查詢執行統計：

| 輪次 | 關鍵字策略 | 搜尋數 | 新洞察數 | 累積洞察 |
|-----|----------|-------|---------|---------|
| 1 | 發散型（談判 / 訊問 / 教育 / 醫病 / UX 廣泛主題） | 8 | A-F (6 個) | 6 |
| 2 | 具體化（華人案例 / MI / Anthropic CAI / 兒童 / Stanford / Lifton） | 8 | G-L (6 個) | 12 |
| 3 | 精準化（LLM sycophancy / parasocial / DarkBench / MITI / NICHD / 盧正細節 / Voss tactical empathy） | 8 | M-S (7 個) | 19 |
| 4 | 反向驗證（每個結論找反例 / 批評 / 反駁） | 8 | U-BB (8 個) | **27** |

關鍵介入點：

- **用戶補充警方/談判/師長四例**（第一輪後）→ 暴露 PM narrow framing → 觸發第二輪轉向「權力結構」
- **用戶覆蓋 v0.20.0 推薦選 v0.18.0**（建 ticket 時）→ 暴露 pm-quality-baseline 規則 6 漏洞 → 衍生 W14-020
- **用戶指出「這不是對錯題」也是強勢框架**（第三輪後）→ 規則 5 機制 4 元層次升級 → 從「禁止暗示」改為「主動暴露」
- **用戶要求做反向搜尋**（第四輪）→ 揭露規則 5 是 paternalism 悖論

### 實驗發現

**已驗證事實**：

- 多輪迭代搜尋確實累積洞察（每輪平均新增 6-8 個）
- 反向驗證揭露單向搜尋無法發現的悖論（第四輪「規則 5 是 paternalism」是最尖銳發現）
- 暴露偏好讓用戶能介入糾正方向（v0.18.0 vs v0.20.0 案例）
- 用戶補充（特別是案例 + 反向觀察）是設計者盲點的最強發現器
- WebSearch 平行 fire 8 個比序列 fire 效率高（單次 session 即可完成 32 次搜尋）

**仍未驗證假設**：

- 第五輪是否還能有顯著新發現（邊際效益遞減點未知）
- 「多輪迭代」對非規則設計類議題（如純技術 debug）是否同樣有效
- 反向驗證的「最佳輪次」是否一定在第四輪
- 沒有用戶補充的場景（純 PM 自我運作）這套方法是否仍有效

## Solution

### 1. WRAP skill 擴充章節設計（4 個新章節）

#### 章節 A：多輪迭代查詢方法論（補入 W 階段）

**四輪結構**：

| 輪次 | 名稱 | 關鍵字策略 | 目的 |
|-----|------|----------|------|
| 1 | 發散型 (Divergent) | 廣泛關鍵字鋪面，5-8 個不同領域 | 建立問題地圖，找出可能相關的所有方向 |
| 2 | 具體化 (Concrete) | 基於第一輪發現深入特定主題 | 將抽象主題轉為具體案例 |
| 3 | 精準化 (Precise) | 從前兩輪提煉精準關鍵字深挖 | 找到最高質量的學術來源與案例細節 |
| 4 | 反向驗證 (Inverse) | 每個結論找反例 / 批評 / 反駁 | 防止確認偏誤，揭露悖論 |

**進入下一輪訊號**（至少 3 條）：

1. 該輪新洞察 < 上一輪 50% → 邊際效益遞減
2. 已涵蓋正反雙面 → 可進入收斂
3. 用戶補充重大反向觀察 → 立即進入反向驗證輪
4. 發現自我參照悖論 → 進入元層次反思輪
5. 出現結論互相衝突 → 進入優先級裁決輪

**邊界條件**：

- 不一定要 4 輪——簡單議題可能 2 輪夠
- 反向驗證輪幾乎不可省略（PC-080 防護）
- 每輪 8 個搜尋是基於 WebSearch 平行 fire 上限的經驗值

#### 章節 B：反向驗證實踐範本（補入 R 階段）

反向搜尋標準格式：

```
| 我們的結論 | 反向搜尋目標 |
|----------|-----------|
| [結論 1] | [批評/反例/反駁/限制] |
| [結論 2] | [批評/反例/反駁/限制] |
```

**至少 8 種反向方向類型**：

1. 方法論的學術批評（如 MI 缺乏理論基礎）
2. 關鍵案例的反方論點（如冤案的警方立場）
3. 結論的失效情境（如 MI 對強制 client 失效）
4. 數字 / 統計的方法論限制（如 DarkBench 30-61% 的限制）
5. 普適性的文化 / 情境限制（如 Voss 不適用長期關係）
6. 隱性 paternalism 警示（如規則本身違反它要保護的價值）
7. trade-off 揭露（如 sycophancy 的 engagement-harm trade-off）
8. 自我參照悖論識別（如反 narrow framing 本身就是 narrow framing）

#### 章節 C：悖論識別檢查清單（補入 A 階段）

設計規則 / 流程 / 系統時，必檢查（至少 4 條）：

- [ ] 此規則是否本身違反它要保護的價值？（如：保護 autonomy 的規則限制了某方 autonomy）
- [ ] 此規則是否通過 4 條件 benevolent paternalism 測試？
    1. 實質可預防的傷害
    2. 介入大概率防止傷害
    3. 利益大於風險
    4. 最小限制介入
- [ ] 此規則的「正確示範」是否本身就是它要禁止的模式？
- [ ] 此規則的設計者立場是否被透明化？
- [ ] 此規則是否預留「用戶可覆蓋」機制？

#### 章節 D：自我暴露偏好實踐（補入 P 階段）

每次提供建議時：

| 實踐 | 禁止 | 正確 |
|-----|------|------|
| 暴露偏好 | 假裝中立提問 | 「我傾向 X，理由 Y」 |
| 暴露推理鏈 | 只給結論 | 列出推理步驟讓用戶可追溯 |
| 暴露盲點 | 假裝完整考慮 | 「我可能漏掉的角度有 Z」 |
| 標記偏誤 | 標 (Recommended) | 改為「我目前的猜測」或不標 |
