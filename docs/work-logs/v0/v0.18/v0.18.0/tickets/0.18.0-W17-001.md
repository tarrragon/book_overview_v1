## Problem Analysis

## 用戶原始觀察

用戶剛上 Linux 核心設計課程（PID/process/thread），發現 ticket 系統與 subagent 的設計接近 process/PCB 模型。期望：PM 派發 ticket 給 subagent 就像 Linux fork/exec——整包 PCB 丟過去即執行，主線程不需花額外 prompt 來回。實際現況：仍花大量 token 反覆補 prompt。

## 系統現狀（Explore Agent 調查）

### Ticket self-containment
- Frontmatter：who/what/when/where/why/how/acceptance/blocked_by/status/completed_at
- Context Bundle 區段：PM 透過 `append-log --section "Context Bundle"` 寫入（Inline 摘要、API 簽名、相關檔案、驗收條件、約束）——**注意：CLI 實測 section 枚舉僅允許 Problem Analysis/Solution/Test Results/Execution Log，Context Bundle 是約定俗成非枚舉，需後續澄清**
- 執行日誌：Problem Analysis/Solution/Test Results
- 問題：Context Bundle 是 PM 額外責任，常被省略

### Prompt 限制
- `.claude/hooks/agent-prompt-length-guard-hook.py` 硬性 30 行限制
- 超過即 PreToolUse 阻擋（本 session 首次派發 34 行被拒，縮到 23 行通過）
- 目標模板：Ticket ID + 任務 + Ticket 路徑 + Read Ticket

### Agent 輸入預期
- Agent 被要求 Read Ticket 取得 Context Bundle
- subagent 約 20 tool call 預算，Context Bundle 不完整導致耗盡於 Read/Grep

### 已知摩擦案例
- PC-040：Context 塞 prompt 未存 Ticket，失敗後 prompt 消失需重組
- PC-047：PM prompt 含「參考 X」「先 grep」導致代理人耗盡預算
- PC-057：PM prompt 超出 agent 職責範圍，代理人無防線照做

## Linux ↔ Claude 對照表

| Linux | Claude Code | 設計意圖 |
|-------|-------------|---------|
| PID | Ticket ID（0.18.0-W10-017.2） | 跨 session persistent 唯一識別 |
| PCB / task_struct | Ticket frontmatter + Context Bundle | 子行程啟動即備好的環境包 |
| /proc/<pid>/ | docs/work-logs/v{ver}/tickets/{id}.md | 可觀察、可 IPC 的持久化檔案 |
| ELF binary | Agent definition (.claude/agents/*.md) | 載入哪套職責 / 允許哪些 syscall |
| fork() | Agent(subagent_type=..., prompt=...) | PM 派發新 subagent |
| exec() argv | Agent prompt（≤ 30 行） | 只傳入口參數，不傳整個程式碼 |
| ARG_MAX | Hook 30 行限制（PC-040） | 超過代表 PCB 沒填好、不是 argv 塞太多 |
| waitpid() / exit code | ticket track complete + status | 結構化回收，非自由文字 |
| signal(7) | append-log 區段 | 父子 IPC，不經 stdin/stdout 多輪對話 |
| zombie process | in_progress 但無人推進的 ticket | 需 release 回收 |

## 三處設計失配

### 失配 1：PCB 半填就 fork → child 永遠在 page fault
- Linux：fork() copy-on-write mm_struct/files_struct，child 啟動就有完整 address space
- 專案：PM 常省略 Context Bundle，把 context 塞 prompt（argv）
- 結果：子 agent 啟動時 PCB 空，被迫 Read/Grep 「swap in」→ 20 tool call 預算很快用光
- 根因：PC-040

### 失配 2：argv 當程式碼用 → exec() 載入錯誤 image
- Linux：execve() 用 ELF entry point 執行，argv 只是啟動參數；binary 規定 capability
- 專案：Agent definition 是 ELF，規定允許產出/禁止行為/適用情境（agent-definition-standard.md 三區塊）
- PM 常在 prompt 塞「順便做 X」超出 agent 職責，相當於傳 argv 要 /bin/ls 格式化磁碟
- 代理人無防線照做 = privilege escalation
- 根因：PC-057

### 失配 3：沒有 waitpid()，parent 靠 polling 猜 exit code
- Linux：wait() 回傳 exit status + resource usage，parent 一次拿完
- 專案：Subagent 結束回自由文字；真實狀態在 ticket append-log/status 欄位需 Read 才知
- 沒有「執行中 raise 需要補 context」的 signal 通道 → 只能失敗後 PM 事後補料重 fork

## 修復方向（ROI 排序）

| # | 修復 | Linux 類比 | 落地動作 |
|---|------|-----------|---------|
| 1 | Context Bundle schema 強制化 | PCB 結構化 | CLI dispatch-check（W10-017.2 已有）在派發前校驗欄位，缺則擋 |
| 2 | Agent capability 顯性聲明 | ELF capability bits | 強化 agent-definition-standard 三區塊，派發前對照（目前靠自律） |
| 3 | Exit code 標準化 | waitpid() | Subagent 結束寫結構化區段（Completion/NeedsMoreContext/Blocked），PM CLI 查而非讀自由文字 |
| 4 | Signal 通道 | SIGUSR1 | 執行中可透過 append-log 的 NeedsContext 區段 raise 需求，PM 非同步回填 |

## 一句話結論

Ticket 系統方向正確（= Linux PCB + /proc 模型），但實際派發時 PM 把 fork() 當成 system()——把 context 塞進 argv 而非預先寫進 PCB。解法不是多寫 prompt，是派發前 CLI 強制驗證 PCB 完整性。

# WRAP 決策分析（對「ROI #1 Context Bundle schema 強制化」原提議）

## W — Widen your options（擴增選項）

原提議只聚焦「強制檢查（A）」，執行擴增後得到五選項：

| # | 選項 | 核心機制 | Linux 類比 |
|---|------|---------|-----------|
| A | 強制檢查 | CLI dispatch-check 派發前擋缺欄位 | 擋 execve() 前的參數驗證 |
| B | Agent 自救 | agent 啟動發現 PCB 不足主動 raise NeedsContext 並 abort claim | SIGSEGV 時進 handler |
| C | 自動化填料 | skill 從 source/相關 ticket 抽取自動填 Context Bundle | Kernel loader 自動載入 ELF dependencies |
| D | 接受摩擦 | 承認 PM 寫 PCB 不可約化；改投資減少派發次數本身 | 不修 exec 成本，改減 fork 頻率 |
| E | 混合 A+B+C | 三層防線：擋派發 + 擋執行 + 自動化 | 完整 process lifecycle |

**Vanishing options test**：若 A 和 D 都禁用，只能選 B 或 C，結果會選 C——因為 Linux exec loader 本身就是自動化填料，沒要人手填 PCB。**這提示 C 是方向正確的 delta**。

**Find someone who solved this before**：Linux 的解法從來不是「強制 PM 填 PCB schema」，而是「kernel 自動從 ELF 抽取 mm_struct」。這點原提議完全忽略。

## R — Reality-test your assumptions（現實測試假設）

**Consider the opposite**：假設「強制 Context Bundle schema」是錯的，為何可能錯？
- 風險 1（ritual compliance）：PM 填「最小通過內容」應付 CLI，品質未提升
- 風險 2（checker 盲點）：CLI 能檢欄位齊全，無法檢內容品質
- 風險 3（繞路）：PM 放棄小任務派發，改 PM 前台處理，分工惡化

**Zoom out（base rate）**：Hook 30 行限制已存在，用戶仍感摩擦——說明單靠「擋人」沒解決問題。base rate 不支持「多加 checker」假設。

**Zoom in（本 session 案例）**：
- 第一次派 Explore 34 行被擋 → 縮到 23 行通過
- 但 agent 仍讀了 10+ 檔案自動補齊，任務成功
- 關鍵：agent 有能力補，只是不知道該補什麼——指向「context discoverability」問題而非「PCB 必填」問題

**Ooching**：先選 3 個 agent 類型小範圍試自動化填料（C），成本遠低於全面鋪開 A。

## A — Attain distance（取得距離再決定）

**10/10/10**：
- 10 分鐘後：A 看似有效（PM 填表就通過）
- 10 個月後：C 的自動化 stitch 會穩定取代人工；A 的 checker 顯得多餘
- 10 年後：Agent 平台原生支援 workflow state 協議（Claude Code、LangGraph、AutoGen 已在往這方向），C 思路成為標準

**核心價值觀檢核**：用戶原話「不想花額外 prompt 來回」，指向「降低 PM 認知/token 成本」。A 讓 PM 花更多時間填表——與核心目標**有衝突**。C 讓系統替 PM 抽取資訊——與核心目標**對齊**。

**朋友視角**（想像資深 DevOps）：「CI precheck 是好的，但要求每次都填 metadata 會被繞路。真正有效是自動化抓取」——支持 C。

## P — Prepare to be wrong（準備犯錯）

### Premortem（假設 6 個月後失敗）

| 失敗模式 | 訊號 | 緩解 |
|---------|------|------|
| F1 ritual compliance | PCB 欄位齊全但內容空洞，agent 失敗率不降 | 加內容品質檢查或轉 C |
| F2 繞路崩潰 | PM 轉全前台處理，subagent 派發次數下降 | 放寬 schema，加 C 自動化 |
| F3 系統內耗 | Hook 維護成本 > 派發改善收益 | 停 A，投資 C |
| F4 診斷錯誤 | 真痛在批次派發不便，而非 PCB | 改做 batch dispatch + template |

### Preparade（假設成功）
- PCB 自動產生 > 80%、手動補 < 20%
- agent 首次成功率 > 90%
- 重派發次數下降 50%

### Tripwires（預警線）
- T1：實作後 3 個月派發失敗率未降 30% → 停 A 轉 C
- T2：Hook 擋人 > 10/week 且 PM 縮 prompt 也過不了 → 放寬或改 B
- T3：PM 開始拼單合併以避免派發 → 派發成本過高，檢查 overhead

## 修訂後結論

**原 ROI 排序錯誤**。Linux 真實類比指向「自動化填料（C）」而非「強制檢查（A）」。修訂如下：

| 新順序 | 修復方向 | Linux 類比 | 可落地子 Ticket |
|--------|---------|-----------|----------------|
| 1 | **C — 自動化 Context Bundle 抽取** | Kernel ELF loader | IMP 設計 ticket create/claim 時自動從 related/blocked_by/source ticket 抽取 Context Bundle 欄位 |
| 2 | **A — CLI dispatch-check 作為 C 的安全網** | execve 參數驗證 | ADJ 擴充 dispatch-check 檢查自動填料結果合理性（而非要 PM 手填） |
| 3 | **B — Agent 自救能力** | SIGSEGV handler | ANA 設計 agent NeedsContext raise 協議（長期，先試 C） |
| 4 | **D — 減少派發次數本身** | 減 fork 頻率 | INV batch dispatch + template 可行性（控制組，觀察 tripwire） |
