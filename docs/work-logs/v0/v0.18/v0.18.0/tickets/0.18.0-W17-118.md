**Phase 1**：在 scan_pending_handoff_tasks 解析每筆 record 後、進入 ticket completed 分支前，先呼叫 is_handoff_stale(data)：
- stale=True 且非「最近 < 300 秒」→ 視同 GC 條件，刪除檔案（與既有 GC 一致）

**Phase 2**：加入「最近 N 秒內建立的 handoff 視為下 session 接手點」豁免：
- 從 data.get(\"timestamp\") 解析建立時間
- 若 now - timestamp < RECENT_HANDOFF_WINDOW_SECONDS（建議 300 秒）→ 計入 recent_tasks 而非 pending_tasks
- 對應 v2.4.0 既有「最近任務允許退出」語意，不阻塞

**Phase 3**：unit test 5+ 案例：
1. 純 stale（任務鏈目標已啟動，來源 in_progress）→ GC
2. 純 active（單個 active handoff，timestamp > 300 秒）→ blocking（單任務行為）
3. stale + active 混合 → stale 被過濾，active 被計入 → blocking active

5. 多 active（皆 > 300 秒，皆非剛建）→ blocking 多任務清單（既有行為保留）

### 修復摘要

`.claude/hooks/handoff-auto-resume-stop-hook.py` 的 `scan_pending_handoff_tasks` 在原邏輯中只在 `is_ticket_completed` 分支內過濾 stale，導致 ticket 仍 `in_progress` 但任務鏈目標已啟動的 stale handoff 流入 pending_tasks → blocking exit。本次修復導入 Phase 1+2：

- **Phase 1（stale 計數前過濾）**：在解析每筆 record 後，先呼叫 `is_handoff_stale(data)`；stale 且非剛建 → 直接 `unlink()` 並 `continue`，不進入 completed/active 分支。對齊既有 GC 一致性，與 `ticket resume --list` 的篩選邏輯收斂。
- **Phase 2（剛建豁免窗口）**：新增常數 `RECENT_HANDOFF_WINDOW_SECONDS = 300` 與 helper `is_handoff_recently_created(record)`，解析 `record["timestamp"]`（CLI 寫入的 `datetime.now().isoformat()` 格式）；若 `now - timestamp < 300s` → 進 `recent_tasks` 不計入 pending（不阻塞，視為下 session 接手點）。剛建判定優先於 stale 過濾，確保剛建即使在任務鏈下也不被 GC。

### 設計決策

1. **剛建優先於 stale**：剛建 handoff 即使對應目標已啟動（stale）仍應保留並標為 recent，因其用途為下 session 接手點，當前 session 不該決定其去留。順序為「剛建 → recent」→「stale → GC」→「completed/active 分類」。
2. **timestamp 解析防護**：`is_handoff_recently_created` 以 `try/except (ValueError, TypeError)` 包裹 `fromisoformat`，timestamp 缺失或格式錯誤時保守回傳 `False`，走原路徑（PCB 防護 (b)）。
3. **常數命名清晰區分**：新增 `RECENT_HANDOFF_WINDOW_SECONDS=300`（handoff 建立後接手點豁免窗口），與既有 `STOP_FLAG_EXPIRY_SECONDS=7200`（session stop flag 防重複觸發）名稱不重疊。
4. **不調整 should_preserve_pending_json 對外契約**：Phase 1 在其上游攔截 stale，到達 completed 分支的 record 必為非 stale，should_preserve_pending_json 行為與測試一致。

### 不在本 ticket 範圍

- 不修改 `handoff_utils.is_handoff_stale` 既有四情境判定。
- 不調整 `resume.py` / `handoff_gc.py`。
- 不重構 `should_preserve_pending_json` 對外契約。
