### 修復方案

採方案 B（lib 層 API 接受預採集參數注入）於 hook 層收斂：

1. **新增 `_load_frontmatter_cached(cache, ticket_id, project_root, logger)`** 作為 hook 內單一採集點函式（命名遵循規則 3：`_load_*` 表達 I/O）。cache 命中直接回傳；未命中才走 `find_ticket_file` + `parse_ticket_frontmatter` 並寫入 cache。

2. **`is_ticket_recently_started` 新增 `frontmatter_cache: Optional[Dict[str, Optional[dict]]] = None` 參數**：未傳維持每次解析（向後相容單一查詢場景）；傳入則透過 `_load_frontmatter_cached` 走快取。

3. **`scan_pending_handoff_tasks` 建立 request-scope `frontmatter_cache: Dict[str, Optional[dict]] = {}`** 並傳入 `is_ticket_recently_started`，確保同一 scan 內同一 ticket frontmatter 只解析一次。

### before / after I/O 呼叫次數對照

| 場景 | before | after |
|------|--------|-------|
| 單次 `is_ticket_recently_started` 對同一 ticket 連續呼叫兩次 | 2 次 `parse_ticket_frontmatter` | 1 次（cache hit） |
| `scan_pending_handoff_tasks` 處理單筆 handoff record（含 `is_ticket_recently_started` 呼叫） | 同一 ticket 在 hook 層 `is_ticket_recently_started` 路徑讀 1 次 frontmatter | 同一 ticket 在 hook 層讀 1 次後寫入 cache（後續同 scan 命中為 0 次） |

### 不在本 ticket 範圍

`is_handoff_stale`（lib SSOT）與 `is_ticket_completed`（delegate 至 lib `is_ticket_terminal`）目前仍各自走 lib 內 `_load_ticket_status` 讀 frontmatter。完全收斂需在 lib 層擴充「接受預採集 frontmatter 參數」介面，屬於 lib API breaking change，超出本 ticket scope（W15-029.2 限定為 hook / command 層稽核 + 修復）。已於規範文件「未完全收斂的部分」段落顯性記錄此邊界。
