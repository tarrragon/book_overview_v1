## 量化分析（ginger 視角：機制觸發頻率 + 認知負擔 + 維護成本）

### 1. 機制觸發頻率（ANA → IMP 轉換實測）

| 指標 | 數量 | 來源 |
|------|------|------|
| 90 天內 ANA tickets | 304 | `find docs/work-logs ... -mtime -90 \| grep "type: ANA"` |
| 90 天內 IMP tickets | 712 | 同上 grep IMP |
| ANA 含 spawned_tickets 非空 | 106 | grep `spawned_tickets:$` |
| **ANA → IMP 觸發率** | **~35%（106/304）** | 實際 spawn 比率 |
| **每月觸發次數估算** | **~35 次/月** | 106 / 3 個月 |
| **每 wave 觸發次數估算** | **~8-10 次/wave** | W10 wave 含 289 tickets，~80 ANA |

**結論**：ANA → IMP 是高頻路徑（每月 ~35 次）。任何在此路徑加負擔的機制都會被頻繁觸發。

### 2. 認知負擔量化

#### 方案 D（wrap-decision 第四問）成本

| 項目 | 估算 | 計算 |
|------|------|------|
| 第四問 prompt 文字 | ~150 tokens | claim hook 既有 W/R/A/P 四問 ~600 tokens，+25% |
| PM 思考 + 回答 | ~300-500 tokens | 寫入 Problem Analysis 一段 |
| 觸發點 | 每次 claim ANA | 304 ANA / 90 天 = ~100 次/月 |
| **每月 token 增量** | **~45,000-65,000 tokens** | 100 × (150+450) |

#### 方案 A（pm-rules 規則段）成本

| 項目 | 估算 |
|------|------|
| 規則檔字數 | ~80-120 行（含對照表 + W10-137 案例 + 三明示）|
| auto-load 每 session | +600-900 tokens 進 PM context（rules/core/ 自動載入；若放 pm-rules/ 則按需讀取，0 baseline） |
| **若放 pm-rules/（按需讀取）** | **0 baseline + 觸發時 ~900 tokens** |
| **若放 rules/core/（auto-load）** | **每 session +900 tokens** |

#### D+A 組合每月總成本

| 配置 | 每月 token | 備註 |
|------|----------|------|
| D（claim 第四問）only | ~50,000 | 觸發頻繁但短 |
| A（rules/core/ auto-load）only | ~30,000-50,000 session 數 × 900 | 30 session/月則 ~27,000 |
| A（pm-rules/ 按需）only | ~5,000-10,000 | 35 次/月 × 900 |
| **D + A（pm-rules/ 按需）** | **~55,000-60,000** | 主線程預判方案 |
| **D + A（rules/core/ auto-load）** | **~75,000-100,000** | 過度方案 |

### 3. 維護成本量化

| 維護項目 | 頻率估算 | 成本 |
|---------|---------|------|
| Checklist 案例新增（如 W10-137 類事件累積）| ~1-2 次/月（PC-143/145 速率參考）| 低（10 分鐘/次）|
| 規則疲勞檢測（每季 review）| 1 次/季 | 中 |
| 與 wrap-decision 同步維護 | 每次 wrap-decision 改版 | 中（耦合風險）|
| Hook 方案（B）若選 | 持續調整誤判閾值 | **高**（PC-138/142/144 已證 hook 字面偵測常誤判）|

### 4. 與本 session 三輪累積成本對比

| 項目 | 數量 | token 估算 |
|------|------|----------|
| 本 session ANA 數 | 13 | — |
| 本 session 子 ticket | 8 | — |
| 累積 insertions | ~4,500+ 行 | ~150,000-200,000 tokens（產出）|
| **本次 W10-140 評估成本（4 視角 ANA）** | **4 個並行 subagent** | **~15,000-25,000 tokens（評估開銷）** |

**對比**：W10-140 評估成本 ~20,000 tokens；D+A 組合每月運作成本 ~55,000-60,000 tokens；**1 次評估 ≈ 0.4 月運作成本**。

### 5. ROI 結論

#### 救援價值估算

| 來源 | 量化 |
|------|------|
| 本 session W10-137 案例：避免「直接刪 ticket-skill-sync-check.md」錯誤決策 | 1 次救援，避免 ~3,000-5,000 token 重做成本 + PC-066 自律失效風險 |
| PC 家族頻率（PC-143/145，相似「假設未驗證」模式）| ~2 次/月 顯性記錄 |
| **隱性救援估算（未被記錄但被攔截）** | **~3-5 次/月**（ginger 推估，需 W10-140 後續驗證）|

#### ROI 公式

```
ROI = (救援價值 × 救援次數) / 機制成本
    = (~5,000 tokens × 5 次/月) / ~55,000 tokens/月
    = 25,000 / 55,000
    = 0.45（投資 1 token，產出 0.45 token）
```

**初看 ROI < 1，但須加入隱性收益**：
- 避免 PC error pattern 累積（每 PC 平均 ~10,000 tokens 維護成本）
- 避免代理人時間浪費（每次重派 ~20,000-50,000 tokens）

**修正 ROI**：

```
修正 ROI = (5,000 + 隱性 30,000) × 5 次 / 55,000
        = 175,000 / 55,000
        = 3.18（投資 1 token，產出 3.18 token）
```

### 6. 強制自問段（檢驗前三輪「先做最簡再評估」結論的有效性）

| 自問 | 答案 |
|------|------|
| Q1: D+A token 成本真的低嗎？ | **否**。每月 ~55,000-60,000 tokens 不算低，相當於 ~3 個中型 ANA 的成本 |
| Q2: 「先做最簡」是否仍適用？ | **是，但須調整**：先做 A（按需讀取版）觀察 30 天，確認觸發 ≥ 5 次/月再加 D |
| Q3: D 第四問是否會被 PM 跳過？ | **高機率（PC-066 自動駕駛）**。claim hook 既有 W/R/A/P 已被忽略案例存在；加第四問不會自動提升遵循率 |
| Q4: A 規則段是否會被 PM 讀？ | **按需讀取版會被忽略**（依賴 PM 主動）；auto-load 版成本過高 |
| Q5: 是否真的需要機制化？ | **保守結論：先 A（按需）+ tripwire**。觀察 30 天，若 PC-W10-137 類事件 ≥ 3 次/月才升級 D |

### 7. ginger 最終建議

| 階段 | 動作 | 條件 |
|------|------|------|
| **Phase 0（立即）** | 僅做 A（pm-rules/ 按需讀取，含 W10-137 案例 + 三明示）| 成本最低，~5,000-10,000 tokens/月 |
| **Phase 1（30 天觀察）** | 統計「派發前假設驗證」相關 PC 新增頻率 | tripwire：≥ 3 次/月 |
| **Phase 2（觸發 tripwire 後）** | 加 D（wrap-decision 第四問）| 成本升至 ~55,000 tokens/月 |
| **拒絕** | B（hook 強制）、C（獨立 skill）、E（三段式 ticket 類型）| 過度設計 |

### 8. dissenting view（PC-101 護欄）

主線程預判 D+A 組合可能高估「機制化即解」的效力。本量化分析顯示：
- D 單獨 ROI 偏低（PM 跳過機率高）
- A auto-load 成本過高
- **真正最小可行 = 僅 A 按需 + tripwire**

若 4 視角共識「立即 D+A」，ginger 視角的 dissenting view 需保留為附註，並在 Phase 1 觀察期主動量測。
