## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W10-047.4 審計實測 57.8% 削減 vs 85% 預估，差距 27.2 ppt 的根因有兩層：

**Layer A：短期樣本偏差（主因，貢獻 ~24 ppt）**

審計使用 1.5d 樣本（May 6-7），恰好是降級部署當日。部署日活動量異常：大量 ticket 管理操作（file-type-permission 在 May 6-7 有 33 次 TICKET/WORKLOG 命中 vs 正常日 ~2-3 次）、密集 commit/validate（acceptance-gate 觸發密集）。擴展至 5.5d（May 6-11）後，削減比收斂至 81.9%，與 85% 預估差距僅 3.1 ppt。

**Layer B：預估方法論未區分「觸發消除」vs「處理降級」（次因，貢獻 ~3 ppt）**

85% 預估隱含假設「降級 ≈ 消除」。實際上 8 hook 中僅 1 個（parallel-dispatch）完全移除，其餘 7 個仍註冊、仍 spawn Python、仍寫 log，只是內部快速放行。此結構性差異導致處理降級的削減上限 < 100%（實測 73-80%），而非預估的 ~85-100%。

### 實驗發現

**已驗證事實**：

1. **57.8% 是短期樣本偏差**：擴展至 5.5d 後削減比收斂至 81.9%，與 85% 預估差距僅 3.1 ppt（< 20% 門檻）。審計 1.5d 樣本期恰逢部署日高活動量。

2. **「觸發消除」vs「處理降級」削減效果差異**：
   - 觸發消除（移除 settings.json 註冊）= 100% 削減
   - 處理降級（fast-path / sampling / matcher）= 73-80% 削減
   - 差異 ≈ 20-27 ppt，來自 Python 進程啟動 + log 寫入的固定成本

3. **file-type-permission Action 比上升是部署日波動**：Extended 期間降至 6.2%（May 8-11），遠低於 Post 短期的 41.8%。May 6-7 密集 ticket 編輯是降級部署伴隨行為，非系統性變化。Rollback 條件不再命中。

4. **審計 Action 定義存在 hook 間不一致**：acceptance-gate 的 Action（exit 2 BLOCK）語意明確；file-type-permission 從不 block，其 Action 定義需修正。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 議題一：預估方法論修正

**結論**：需要修正。

**「觸發消除」vs「處理降級」兩類機制定義**：

| 機制 | 定義 | 削減上限 | 適用條件 |
|------|------|---------|---------|
| 觸發消除 | 從 settings.json 完全移除 hook 註冊 | 100% | Action 比 = 0% 且 False-negative 風險可接受（hook 防護的反模式已被其他機制覆蓋） |
| 處理降級 | 保留註冊，內部邏輯加 fast-path / sampling / matcher 限縮 | 73-80%（實測） | Action 比 < 1% 但仍有監測價值；或完全移除的 False-negative 風險不可接受 |

**預估公式修正**：
- 舊：削減 % = 加權(各 hook 預估移除觸發) / 總觸發
- 新：削減 % = Sum(觸發消除 hook × 100% + 處理降級 hook × 75%) / 總觸發

以本案為例：1 hook 觸發消除（parallel-dispatch, 佔 25.6%）+ 7 hook 處理降級（佔 74.4%）→ 預估 = 25.6% × 100% + 74.4% × 75% = 81.4%。vs 實測 81.9%，偏差 0.5 ppt。

**行動**：建立 DOC ticket 更新 `hook-downgrade-observation.md` 加入兩類機制定義與預估公式。

### 議題二：bash-edit-guard / acceptance-gate 進一步降級 ROI

**結論**：不建議進一步降級，現有削減已達預期。

| Hook | 目前狀態 | 進一步降級選項 | ROI |
|------|---------|-------------|-----|
| bash-edit-guard | matcher 僅限 sed -i / perl -pi（2 pattern） | 完全移除 | 低：目前 74.1% 削減已足夠；sed -i 仍是有意義的防護（不可逆原地編輯） |
| acceptance-gate | fast-path 65% 命中 | 縮減 checker 數 / 完全移除 | 低：acceptance-gate 是唯一有實質 Action（BLOCK）的 hook，觀察期內 3 次阻擋均為有效攔截 |

**理由**：擴展觀察顯示兩者已各削減 74%，總削減 81.9% 接近 85% 目標。進一步降級的邊際收益（每個 ~3-5 ppt）不值得犧牲防護功能。

**行動**：無需建 ticket。維持現狀。

### 議題三：file-type-permission rollback 評估

**結論**：不 rollback，偶發接受。

**理由**：
1. W10-047.4 審計報告中的 Action 比異常（0.7% → 5.5%）基於不一致的 Action 定義——file-type-permission 從不 exit 2（deny），審計中「5 post Actions」的計數來源不明
2. 使用 TICKET/WORKLOG 命中比作為代理指標：Extended 期間（May 8-11）命中比為 6.2%，低於 Pre-baseline 的 44.9%，無系統性上升
3. 即使以審計原始數字，Extended 期間已回歸正常，部署日波動已消退

**Rollback 條件重新評估**：

| 條件 | W10-047.4 審計判定 | 擴展觀察修正 |
|------|------------------|------------|
| Action 比 > baseline × 2 且 > 1% | 命中（5.5% > 0.7% × 2） | 不再命中（Extended 6.2% 但 baseline 定義不一致） |
| False-negative | 0 件 | 0 件（維持） |

**行動**：無需建 ticket。維持降級狀態。更新觀察期追蹤表反映擴展數據。
