### 觸發脈絡

本 session 用戶反映 cc-statusline 沒顯示 main 未提交變更。診斷過程中執行 `git worktree list --porcelain`，意外發現大量 locked agent worktree 殘留。

### 觀察事實

**數量統計**（建 ticket 當下，2026-05-03）：

| 指標 | 數值 |
|------|------|
| 總 worktree 數 | 26 |
| 主 worktree（main） | 1 |
| 人工建的 feat worktree | 1（feat/0.18.0-W5-015） |
| `.claude/worktrees/agent-*` | 24 |
| agent-* 全部 locked | 是（24/24） |

**Lock 形式**：每個 agent worktree 在 `git worktree list --porcelain` 輸出中均含

```
worktree /Users/tarragon/Projects/book_overview_v1/.claude/worktrees/agent-XXXXXXXX
HEAD <sha>
branch refs/heads/worktree-agent-XXXXXXXX
locked claude agent agent-XXXXXXXX (pid NNNNN)
```

**衝擊**：每個 agent worktree 在 statusline 顯示為 `worktree-agent-XXXXXXXX ~1`，淹沒 main 的 dirty 顯示；佔用 disk inode 與檔案空間。

### 初步推論（待 ANA 驗證或推翻）

| 推論 | 待驗證點 |
|------|---------|
| Claude Code agent isolation:worktree 機制建 worktree 後加 lock 防止 pruning | 真有此機制？lock 規格？ |
| Agent 結束時 cc runtime 應釋放 lock 並清理 worktree | runtime 是否有此邏輯？或設計上要外部清理？ |
| 專案缺 session-end / agent-stop hook 做 GC | 實際 hook 設定確認 |
| Lock 的 PID 多數已死（agent 早結束） | `ps -p <pid>` 逐一驗證 |

quality-baseline 規則 5「所有發現必須追蹤」+ rules/core/decision-trigger-binding.md「無 trigger 延後禁用」雙重作用：發現流程性問題不可口頭記錄即離場，需建 ANA 走完根因分析後決定 spawn 何種 IMP。

- agent 派發決策層的 worktree 使用策略檢討（在 isolation 機制本身的設計分析後再評估）
### 待 ANA 認領者擴充驗證（重現實驗待補項）

| 待驗證項 | 建議指令 |
|---------|---------|
| 各 agent worktree 的 PID 是否還活著 | `git worktree list --porcelain \| grep "^locked" \| grep -oE "pid [0-9]+" \| awk '{print $2}' \| while read p; do ps -p $p > /dev/null && echo "$p alive" \|\| echo "$p dead"; done` |
| 各 worktree 建立時間（年齡） | `for d in .claude/worktrees/agent-*; do echo "$(stat -f '%Sm' $d) $d"; done \| sort` |
| 各 worktree dirty 內容（為何全 dirty=1） | `for d in .claude/worktrees/agent-*; do echo "=== $d ==="; git -C "$d" status --porcelain; done` |
| .claude/hooks/ 是否有 worktree GC 相關 hook | `ls .claude/hooks/ \| grep -iE "worktree\|cleanup\|gc\|stop"` |
| .claude/skills/worktree/ 邏輯與已實作子命令 | `ls .claude/skills/worktree/ && cat .claude/skills/worktree/SKILL.md` |
