## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

cc Agent tool 設 `isolation: "worktree"` 時，cc runtime 自動建 `.claude/worktrees/agent-XXX` worktree 並加 git lock（含 cc CLI process PID）。Agent 結束 / cc CLI 崩潰時 runtime **不釋放 lock 也不刪 worktree**；專案無對應 GC 機制（缺 SessionStart/SessionEnd 清理 hook），導致 lock 隨時間累積為殭屍。詳見「重現實驗結果」第 4 / 5 節既有 hooks 盤點與 settings.json 缺口分析。

### 影響範圍

- `.claude/worktrees/agent-*`：4 週累積 24 個殭屍 worktree（PID 全 dead）
- cc-statusline 顯示：殭屍 worktree 在第三行被列為 `worktree-agent-XXX ~1`，淹沒 main 分支 dirty 訊號（觸發本 ANA 的初始症狀）
- disk 佔用：每個 worktree ~MB 級檔案 + git metadata
- 派發決策層：未來 isolation:worktree 派發若延續現行流程會持續加速累積（4 週 24 個 → 月增約 30+）

### 相關 Error Pattern

未直接命中既有 PC error-pattern。本問題屬「cc runtime 機制（建立完整、清理空白）+ 專案配套缺口」之流程設計類缺陷，與 `PC-009 Handoff first` / `PC-115 ARCH-015` 等 worktree 相關 PC 無因果關係。本 ANA 結論落地後若殘留模式重現，可考慮升級為新 PC 描述「外部工具自動建立資源 vs 專案 GC 責任邊界」。

### WRAP 四階段執行記錄

**W（Widen Options）— 5 個改善方案候選**

| 方案 | 描述 | 優點 | 缺點 |
|------|------|------|------|

| 階段 | 理想流程 | 實際流程 | 缺哪一道防線 |
