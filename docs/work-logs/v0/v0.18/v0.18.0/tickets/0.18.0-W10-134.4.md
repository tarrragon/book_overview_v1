### 提案 A（Ticket/PC RAG）Verdict: 不建議實施

**理由**：

1. **重複造輪子風險點 1**：Context Bundle 自動抽取已覆蓋「歷史 ticket 脈絡注入」核心需求。RAG 的增量價值僅在 semantic similarity（跨 ticket 關聯發現），但 ticket frontmatter 的 source_ticket/blockedBy/spawned_tickets 已提供確定性連結，比 embedding 相似度更可靠。

2. **重複造輪子風險點 2**：error-pattern skill 的 grep-based 查詢已覆蓋「PC pattern 檢索」需求。PC pattern 的命名規範（PC-NNN 編號 + 標題關鍵字）使 keyword search 命中率極高，semantic search 的邊際收益低。

3. **整合衝突**：RAG retrieval 結果注入 PM context 會與 Context Bundle 的注入點競爭。兩套系統同時注入會造成 token 預算衝突（父 ticket 估算工作 context 已占 50-70%），且 PM 需額外判斷 RAG 結果 vs Context Bundle 結果的優先序。

### 提案 B（Rules/PM-Rules RAG）Verdict: 不建議實施，改擴充既有 L3 方案

**理由**：

1. **既有 hook-driven 情境載入已是更精確方案**：107 個 hook 已覆蓋主要決策 trigger 點，pm-role.md 情境路由表已是確定性 FSM。RAG 的 95%+ 命中率要求（父 ticket 分析）在規則載入場景極難保證，漏 retrieve 直接導致 PM 違規。

2. **lazy-load 已部分解決 token 競爭**：rules/core 自動載入（~30K）+ references 按需 Read 的分層已運作。進一步優化應沿 L3 方向（hook-driven 情境載入）而非跳到 L4（RAG），因 L3 的確定性遠高於 L4 的機率性。

3. **整合衝突**：Rules RAG 的 trigger 點（每個 session 啟動）與 SessionStart hook 重疊。新建 RAG 模組會與既有 hook 系統形成平行的載入通道，增加維護成本且可能產生載入順序衝突。

### 整合策略建議

**建議方向：在既有設施上擴充，不新建獨立模組。**

| 需求 | 既有設施擴充方向 | 優先級 |
|------|----------------|-------|
| 跨 ticket 關聯發現 | 擴充 Context Bundle：claim 時額外掃描同 wave/同 module 的 completed ticket | 低（當前 source_ticket 連結已足夠） |
| PC pattern 語意搜尋 | 擴充 error-pattern skill：加入 tag-based 分類索引（比 embedding 維護成本低） | 低（grep 已夠用） |
| Rules 情境載入優化 | 沿 L3 方向：擴充 hook trigger map，依 ticket type/TDD phase 載入對應 rule subset | 中（可降低 ~30K 固定成本） |
| 歷史決策參考 | 維持現狀：PM 需要時手動 grep/rg 查詢 + error-pattern skill | 無需新建 |
