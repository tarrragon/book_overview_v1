### 問題根因

三個不同來源的 ticket 共用 `dispatch-check` 命名空間，但功能完全不同：

| 來源 | 實際功能 | 命名 |
|------|---------|------|
| W10-017.2（已實作） | 檢查 `dispatch-active.json` 活躍派發狀態（exit 0/1/2） | `ticket track dispatch-check` |
| W17-003（pending ADJ） | 檢查 Context Bundle 自動填料的合理性（欄位非空、長度、檔案存在） | 「擴充 dispatch-check」 |
| W17-053（pending IMP） | 實作派發前三項閾值 CLI（功能職責數/修改檔案數/CB tokens） | `ticket track dispatch-check` |

根因：W10-017.2 先佔了 `dispatch-check` 名稱做「活躍派發狀態查詢」；W17-003 和 W17-053 各自規劃了不同的「派發前檢查」功能卻沿用同一名稱，三者語意完全不同。

### 影響範圍

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 三方功能差異與重疊面

| 維度 | W10-017.2（既有） | W17-003（pending ADJ） | W17-053（pending IMP） |
|------|-------------------|----------------------|----------------------|
| 語意 | 活躍派發狀態查詢 | CB 自動填料品質驗證 | 認知負擔閾值檢查 |
| 輸入 | `dispatch-active.json` | Context Bundle 抽取結果 | ticket `where.files` + CB |
| 輸出 | exit 0/1/2 | pass/warn/fail | pass/warn/fail |
| 時機 | 派發後/收到通知時 | 派發前（CB 抽取後） | 派發前（拆分決策） |
| 重疊 | 無 | 無 | 無 |

結論：三方功能完全正交，無重疊。

### CLI 命名重劃方案

### 方案 A：既有保留，新功能用獨立子命令

| 子命令 | 功能 | 對應 ticket |
|--------|------|------------|
| `dispatch-check`（保留不動） | 活躍派發狀態查詢 | W10-017.2（已完成） |
| `dispatch-validate` | CB 填料合理性驗證 | W17-003 |
| `dispatch-readiness` | 認知負擔閾值 + 綜合派發就緒度 | W17-053 |

優點：不碰既有實作和測試，零破壞風險。
缺點：三個 `dispatch-*` 子命令，使用者需記住差異。

### 方案 B：統一為 dispatch-check 子模式（--mode flag）

| 呼叫方式 | 功能 |
|---------|------|
| `dispatch-check --active`（預設） | 活躍派發狀態查詢 |
| `dispatch-check --validate` | CB 填料合理性驗證 |
| `dispatch-check --readiness` | 認知負擔閾值 |

優點：單一入口，概念統一。
缺點：需改既有實作加 flag parsing；exit code 語意在不同 mode 下不同，增加混淆。

### 方案 C：語意分群重命名

| 子命令 | 功能 | 對應 ticket |
|--------|------|------------|
| `dispatch-status`（重命名既有） | 活躍派發狀態查詢 | W10-017.2 需 IMP 重命名 |
| `dispatch-validate` | CB 填料合理性驗證 | W17-003 |
| `dispatch-readiness` | 認知負擔閾值 | W17-053 |

優點：每個名稱自說明，語意最清晰。
缺點：需改既有命令名（但引用處極少，rg 確認 SKILL.md/pm-rules 無引用）。

### 推薦：方案 A

理由：零破壞風險，既有 W10-017.2 不需任何修改；W17-003/053 各自新建子命令，互不干擾。方案 C 語意更佳但需額外 IMP 改名+測試遷移，收益不成正比。

### W17-003 / W17-053 範圍重劃建議

**建議：保持獨立，不合併。**

理由：
1. 功能正交：W17-003 檢查 CB 品質（「填料對不對」），W17-053 檢查認知負擔（「ticket 太大嗎」）
2. 依賴鏈不同：W17-003 blockedBy W17-002（CB 抽取器），W17-053 blockedBy W17-052（閾值定義）
3. 合併會違反 SRP，增加單一 ticket 認知負擔

**範圍調整**：
- W17-003：scope 改為「新建 `dispatch-validate` 子命令」（非「擴充 dispatch-check」），title 需更新
- W17-053：scope 改為「新建 `dispatch-readiness` 子命令」（非沿用 dispatch-check），title + where.files 路徑需修正
- 兩者均需在 Problem Analysis 註明不觸碰既有 `dispatch-check`
