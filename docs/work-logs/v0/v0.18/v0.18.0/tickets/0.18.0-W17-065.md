# Phase 1 規格設計（lavender，2026-05-04）

本 ANA 的目標是「設計實驗方法」而非執行實驗。Phase 1 產出規格供下游 W17-062/063 依規格建立 6×N IMP + N ANA 子 ticket。本規格刻意提多方案而非預設唯一解，留多視角審查空間（PM 將派 linux + ginger 審查）。


### 多視角審查：linux 視角

審查對象：Phase 1 lavender 規格四項產出 + 5 個留白決策點。Linus 視角不評估「值不值得做」（用戶已接受 136 張 ticket 成本），只評估「方法本身是否最簡潔有效」。

#### 整體 Taste Score

**Acceptable（偏向 over-engineering）**

規格涵蓋面完整、可執行性高、留白決策合理。但實驗本身規格化過度，把一個簡單問題（「降級會不會壞掉」）包裝成跨 19 代理人 × 6 重複 × 3 視角的工業級驗證。這不是壞品味，但確實有 second-system effect 的影子——對「降級這件事」的恐懼導致每個維度都選了「保險」方案，而非「最少能回答問題」方案。

**核心質疑**：Bad programmers worry about the code. Good programmers worry about data structures. 這個實驗的「資料結構」是什麼？是 19 個代理人 × 6 份產出 × 3 視角的對照矩陣（19 × 6 × 3 = 342 個觀察點）。但**真正回答決策的數據結構**只需要：「這個代理人降級後會不會壞？」——一個 bool。用 342 個觀察點得出 19 個 bool，資訊密度極低。

#### 5 個留白決策點結論

##### Q1：視角組合 → **β 雙視角（linux + ginger）**

| 方案 | Linus 判定 | 理由 |
|------|----------|------|
| α 三視角（含 basil） | **Garbage** | 規格自己說 basil 屬「不重要差異」軸不入回滾判準。**既然不入判準，叫它來幹嘛？** 這就是典型的 special case——「為了完整性加一個視角，但又補充說它不算數」。Good code has no special cases。 |
| β 雙視角（linux + ginger） | **Good taste** | 兩視角分別覆蓋「邏輯/判斷」與「執行能力/成本」，正交且都入判準。最少視角回答最多問題。 |
| γ 四視角 | **Garbage** | acceptance-auditor 的客觀指標已經是 ginger 視角的子集（acceptance_met 數字 ginger 一定會看）。加它純粹疊床架屋。 |

**Action**：採 β。文字品質違規由二次審查層（既有 basil 流程）兜底，**不需要**在實驗中當第三視角。省下 19 × basil 派發 ≈ 19 次 token。

##### Q2：樣本數 → **n=3，但採絕對門檻而非統計推論**

n=3 確實無法做 t-test，但規格已正確選擇「絕對門檻」（≥1 件邏輯錯誤即觸發回滾）。這是 pragmatic 的選擇——既然門檻是「出現一次就回滾」，n 加到 5 反而**增加偽陽性風險**（樣本越多越可能撞到偶發失誤）。

**Action**：堅持 n=3。若擔心偶發失誤，**改用「重跑驗證」機制**：n=3 中出現 1 次回滾觸發條件時，再跑 2 次該代理人原 model 作為對照——若原 model 也偶發同類錯誤，視為模型無關，**通過**。這比 n=5 更精準且總成本更低（19 × 2 額外執行 vs 19 × 2 全升級）。

##### Q3：任務長度 → **B 中（10-12 tool calls）**

lavender 預判正確。Linus 補強理由：

- **A 過短**：5-8 tool calls 內模型差異被 prompt 主導，看不出推理能力。
- **C 過長**：15-18 觸碰 PC-042 ~20 tool call 限制，「執行能力下降」會與「平台限制」混淆。**這是把實驗本身的 bug 當成被測對象的 bug**，taste 極差。
- **B 中**：避開兩端，且 prompt + 工具呼叫的比例正常。

**Action**：B 中。**但判斷複雜度仍取 C（3+ 規則交互）**——任務長度與判斷複雜度是兩個正交維度，lavender 已正確區分，不要混為一談。

##### Q4：派發策略 → **序列 6 IMP，每完成 1 IMP commit + /clear session**

lavender 自評已經給出正確答案。Linus 補強：

| 策略 | Linus 判定 |
|------|----------|
| 全並行 6 | **Garbage**：同代理人 6 並行必撞 PC-092 commit race + 同 session context 污染。**這不是「快」，是把垃圾資料生產得更快**。 |
| 並行 3+3 | **Acceptable**：分 batch 後 race 風險降低，但仍同 session，原 model #1 的產出會污染 #2 #3 的推理。 |
| 序列 6 + /clear | **Good taste**：唯一能保證 6 份產出真正獨立的方法。時間成本 6× 但這是**正確 vs 錯誤**的差異，不是快慢的差異。 |

**Action**：序列 + /clear。用戶既已接受 136 張 ticket 的成本，再接受 6× 執行時間是合理的——**否則前面的成本全部白花**（污染樣本得出的結論不能用）。

##### Q5：實驗總規模（136 張）→ **可砍至 ~50 張，等價結論**

這是 Linus 視角最關鍵的觀察。**規格設計把「19 代理人全部驗證」當成不可協商的前提，但這個前提本身就值得質疑**。

**重新審視資料結構**：19 個代理人不是同質的。用 W17-051 既有分類就能砍出三類：

| 類別 | 代理人數 | 是否全跑實驗？ | 理由 |
|------|---------|--------------|------|
| **Haiku 候選**（mint/thyme-doc/rosemary） | 3 | **全跑** | 跨 model 級距大（Sonnet→Haiku），風險最高，3 個全跑 = 18 IMP |
| **Sonnet 候選 - 單純執行類**（13 個） | 13 | **抽樣 3 個代表** | 同類型同職能（如多個實作類代理人），跨 model 級距小（Opus→Sonnet），抽 thyme-python（實作）+ acceptance-auditor（審查）+ star-anise（重構）3 個代表 = 18 IMP |
| **邊界類**（cinnamon/lavender/thyme-ext） | 3 | **全跑** | ginger 已標記高風險，必須個別驗證 = 18 IMP |

**砍後規模**：

| 項目 | 原規格 | 砍後 | 節省 |
|------|-------|------|-----|
| IMP 總數 | 114 | 54（9 代理人 × 6） | -60 |
| 分析 ANA | 19 | 9 | -10 |
| 容器 ANA | 2 | 2 | 0 |
| 本 ANA | 1 | 1 | 0 |
| **合計** | **136** | **66** | **-70（-51%）** |

**Action 前提**：抽樣的 3 個 Sonnet 代表通過後，**將結論外推至同類群其他代理人**（記錄為「同類群推論」而非「實證」）。若抽樣中任一回滾，**該整類群全部不降級**（保守外推）。這是用「類別不變性」假設換實驗成本——若假設不成立（同類代理人對 model 敏感度差異大），代價是全類群保守不降，**沒有錯誤決策風險**，只是少省一些 token。

**這比「全跑 19 個」更符合 pragmatism**——花 51% 成本得到 100% 決策資訊（保守側）。

#### 整體評價：Over-engineering 還是 Pragmatic？

**Verdict：Over-engineering，但有救**。

| 觀察 | 評估 |
|------|------|
| 三維度選擇矩陣（5 維 × 3 選項 = 15 格） | 過度規格化。實際上「任務長度 B + 判斷複雜度 C + 工具 B」三個決策即可，其他維度（輸出型態/上下文規模）由代理人職能自然決定，不需獨立決策。 |
| 6 類代理人對齊任務型態表 | 合理且必要。 |
| 三檔判定（通過/警告/回滾） | **「警告」這檔是 garbage**。「設 2 Wave 監測期 + 失敗率 > 20% 回滾」聽起來嚴謹，但 2 Wave 內失敗率統計的母體在哪？這是把不確定的決策包裝成看似量化的流程。**砍掉「警告」檔，改為「通過 / 回滾」二元**——邊界情況保守判回滾，後續若有強證據再開單獨 ticket 重評。 |
| 命名規範 4 層後綴（W17-062-A1-1） | 過深。建議扁平化：`W17-062-mint-1`（直接用代理人名替 A{n}），可讀性高 100 倍。 |
| Phase 4 結論「無需重構」 | 正確。 |

#### Second-system effect 風險

**有，且訊號明確**：

1. **「3 視角中 ≥ 2 視角」這種多重保險語法** — 第一系統（W17-051 ginger 修正）就是用「多視角」修補「直接降級」的不安，第二系統（本 ANA）又用「3 視角複數判定」修補「多視角」的不安。**每多一層保險，原始決策就更難被推翻**——最後沒人敢說「降級」，因為要推翻 3 視角共識比推翻 1 視角共識難 3 倍。
2. **136 張 ticket 規模** — 把「降級會不會壞」這個 yes/no 問題包裝成跨 19 代理人 × 6 樣本 × 3 視角的工業驗證，**規模本身已經改變問題性質**——實驗執行中必有意外（PC-042 / PC-092 / context drift），這些意外會被誤讀為「降級的問題」，反而污染結論。
3. **量化門檻表的精確性幻覺** — 「acceptance_met 降幅 ≥ 33%」聽起來精確，但 n=3 下這只是「降 1 個 AC」。把粗粒度判斷包裝成百分比是 second-system 的典型症狀。

#### Linus 推薦的最簡方案（總結）

| 維度 | Linus 推薦 | vs lavender 預判 |
|------|----------|-----------------|
| 視角 | β 雙視角（linux + ginger） | 砍 basil |
| 樣本數 | n=3 + 觸發後重跑 2 次原 model 對照 | 用「重跑」取代「擴 5」 |
| 任務長度 | B 中 + 判斷複雜度 C | 同 lavender |
| 派發策略 | 序列 + /clear | 同 lavender 自評 |
| 規模 | 9 代理人（3 Haiku + 3 Sonnet 代表 + 3 邊界） | 砍至 51% |
| 判定檔次 | 通過/回滾 二元 | 砍「警告」 |
| 命名 | W17-062-{agent_name}-{1..6}-R | 扁平化 |

**最終規模**：54 IMP + 9 分析 ANA + 2 容器 ANA + 1 本 ANA = **66 張 ticket**（原 136 的 49%）。

**核心訊息**：If you need more than 3 levels of indentation, you're screwed. 這個實驗如果需要 4 層 ticket 後綴 + 5 維度選擇矩陣 + 3 視角複數判定，就是 screwed。砍到能用最少維度回答決策的程度，才是 good taste。

### 多視角審查：ginger 視角

審查焦點：認知負擔、派發效能、token 預算。本視角不評估「實驗該不該做」（用戶已決定），只評估「給定要做，方法論的資源效率與認知設計是否合理」。與 linux 結論大方向一致（規模偏大、警告檔贅餘、命名過深），但在 Q4 派發策略與 Q5 規模上有具體微調。

#### 5 個留白決策點獨立判斷

##### Q1：視角組合 → 同意 linux β 雙視角（linux + ginger）

認知負擔角度補強：每多 1 視角，分析 ANA 需在報告中對齊 1 套維度欄位 → PM 派發前讀規格的概念追蹤數 +N。三視角報告骨架已有 6 欄對照矩陣 + 重要/不重要差異分類 + 視角結論，認知負擔指數已達 ~9（接近「需重構」閾值 11）。加 basil 變 4 視角後跨視角對齊矩陣概念數爆破到 ~13，PM 讀單份分析 ANA 需切換脈絡 4 次。

token 預算角度：basil 視角產出純粹是「文字品質清單」，這類產出在現有 basil-writing-critic 流程中本就是 commit 後審查層。實驗中加 basil = 19 × basil 派發（每次 ~3K tokens 讀規格 + 6 份產出 + 2K 寫報告 ≈ 5K）= ~95K tokens，純損失（不入回滾判準）。

採 β。

##### Q2：樣本數 → 同意 linux「n=3 + 觸發後重跑 2 次原 model 對照」，但補充早停條件

linux 的「重跑機制」是好設計，但缺一個成本上限。補充：

| 重跑觸發條件 | 動作 | 成本上限 |
|-------------|------|---------|
| n=3 降級組出現 1 次回滾觸發條件 | 重跑該代理人原 model 2 次作對照 | +2 IMP / 代理人 |
| 重跑 2 次中原 model 也出現同類錯誤 | 視為模型無關，通過 | 終止，不再重跑 |
| 重跑 2 次中原 model 皆無錯誤 | 確認回滾 | 終止 |
| 重跑後仍無法判定（1 次有錯 1 次無錯） | 直接判回滾（保守側） | 終止，禁無限延伸 |

Why：n=3 + 條件式重跑的 worst case 是 19 × (6+2) = 152 IMP，需早停條件防止偶發失誤觸發級聯重跑。

##### Q3：任務長度 → 同意 lavender + linux「B 中（10-12 tool calls）+ 判斷複雜度 C」

無補充。但提醒派發效能：B 中（10-12 tool calls）已接近 PC-042 ~20 tool call 限制的 60%，實驗中若代理人在「讀 frozen context + 思考 + 寫產出」之間產生意外迭代（例如 acceptance gate 抽查需重讀），可能突破限制。建議在範本骨架的 Forbidden 段補一條：「禁止超過 14 tool calls，超過視為執行能力下降」——把工具預算明示為 acceptance 一部分，而非靠執行時自律。

##### Q4：派發策略 → 微調 linux 結論：序列 6 IMP + /clear，但 token 預算實際可能未省

這是與 linux 結論最不同的點。Linus 主張「序列 + /clear」是唯一保證樣本獨立的方法，認知品質角度成立。但需量化 token 預算：

| 策略 | 樣本獨立性 | token 成本（單代理人 6 IMP） |
|------|----------|----------------------------|
| 全並行 6（同 session） | 低（context 污染） | ~6 × 8K = 48K（共享 context） |
| 並行 3+3（跨 session） | 中（batch 內仍共享） | ~6 × 12K = 72K（每 batch 一次 preload） |
| 序列 6 + 每次 /clear | 高（完全獨立） | ~6 × 18K = **108K**（每次重 preload 系統規則 + AGENT_PRELOAD + ticket md） |

序列 + /clear 比並行貴 ~2.25 倍 token，因為每 IMP 都要重新載入：
- AGENT_PRELOAD.md 與 agent definition (~3K)
- 自動載入 rules/core/*.md (~5K)
- ticket md + Context Bundle (~3K)
- 任務輸入（frozen context 凍結檔案）(~3-5K)

= 每次 ~14-16K preload，6 次 = ~90K 純 preload + ~18K 實際工作 = ~108K。

19 代理人 × 108K = 2.05M tokens 純樣本獨立性成本。

**ginger 微調建議**：採折衷方案 **C+「序列 + /clear，但每代理人 6 IMP 共用 1 個 frozen context 檔案載入」**：

1. 每代理人實驗開始前，PM 把 frozen context（輸入檔案 snapshot + Forbidden 規則）寫入該代理人實驗組 ANA 的 Context Bundle。
2. 6 個 IMP 的 ticket md 各自從同一份 Context Bundle 引用（透過 source_ticket / parent_id 機制），而非每個 IMP 獨立複製 context。
3. /clear 後新 session 的代理人讀 ticket md → 看到 parent ANA 的 Context Bundle 引用 → 一次性載入。

這樣保留了 /clear 帶來的樣本獨立性，但避免了「每次重複貼大段 frozen context 到 prompt」的 token 浪費。預估能把 108K 降到 ~85K（節省 ~20%）。

19 代理人 × 85K ≈ 1.6M tokens，比純序列 + /clear 省 ~450K，比並行 3+3 仍貴 ~70%（合理代價換樣本獨立）。

##### Q5：規模 → 同意 linux 抽樣外推大方向，但具體抽樣選擇微調

linux 的「9 代理人抽樣」邏輯（3 Haiku 全跑 + 3 Sonnet 代表 + 3 邊界全跑）方向正確，但 Sonnet 抽樣的「3 個代表」需要更明確的代表性論證。否則「結論外推至同類群」會被質疑外推依據不足，反而需要事後追加實驗，倒省不到。

**抽樣調整建議**（仍維持 9 代理人總數）：

| 類群 | 候選代理人（W17-051 sonnet 候選 13 個） | 抽樣選擇 | 代表性論證 |
|------|--------------------------------------|---------|----------|
| 實作執行類 | thyme-python、parsley、fennel、bay | 抽 1：thyme-python | 這 4 個共享「RED→GREEN」核心職能；thyme-python 為本框架最高頻派發代理人，行為失誤影響範圍最大 |
| 分析審查類 | acceptance-auditor、project-compliance、saffron-system-analyst | 抽 1：acceptance-auditor | 這 3 個共享「規則套用 + 結論輸出」職能；acceptance-auditor 在 Phase 3a/4 高頻被派發，且行為直接影響 ticket complete |
| 重構/設計類 | star-anise、sassafras、ginger、cinnamon-refactor-owl | 抽 1：star-anise | 這 4 個共享「多約束權衡 + 設計輸出」職能；star-anise 在 Wave 收尾期高頻被派發 |
| 其他 sonnet 候選 | sage、basil-writing-critic | 不獨立抽樣 | sage 屬測試設計屬性已被「實作執行類」覆蓋；basil 為二次審查層，本身就是降級判準工具，不入候選 |

抽樣依據：取「同類群中最高頻派發者」+「失誤代價最大者」雙重標準。若該代表通過降級，外推至同類群的論證強度高（同類群其他代理人派發頻率更低 → 失誤代價同等或更小）。

ginger 與 linux 規模一致（66 張 ticket），但抽樣選擇有具體論證依據。

#### 額外評估：規格本身的認知負擔

##### 評分標準「警告」檔的認知負擔

linux 主張砍掉「警告」檔，ginger 強烈同意，補強認知負擔角度論證：

| 指標 | 三檔（通過/警告/回滾） | 二元（通過/回滾） |
|------|---------------------|-----------------|
| PM 讀分析 ANA 後的決策狀態數 | 3 | 2 |
| 警告檔的後續追蹤負擔 | 「2 Wave 監測期 + 失敗率 > 20% 回滾」需追蹤 ≥ 19 個代理人 × 2 Wave 統計 → **新增持久追蹤狀態** | 0 |
| 認知負擔指數 | 警告檔引入「2 Wave 失敗率閾值」「監測期狀態」「失敗率計算母體」3 個新概念 → 指數 +3 | 0 |
| 違反 decision-trigger-binding | **是**（「2 Wave 後重評」是無 trigger 延後決策，違反 rules/core/decision-trigger-binding.md 規則 1） | 否 |

**砍「警告」檔不是規格簡化，是規則合規修正**。decision-trigger-binding 規則 1 明示：所有決策只能是「已決策」或「綁 ticket trigger 延後」，不存在「2 Wave 後再評」這種無 ticket trigger 的延後狀態。改二元（通過/回滾）後，邊界情況保守判回滾，後續若有強證據再開單獨 ticket 重評，符合規則。

##### 命名後綴 4 層 vs 扁平的識別成本

linux 主張扁平化命名，ginger 從 PM 派發識別成本角度補強：

| 命名格式 | 範例 | PM 派發時識別成本 | grep 友善度 |
|---------|------|-----------------|-----------|
| 4 層後綴 | `0.18.0-W17-062-A1-1` | 需記憶 A1=mint、-1=原 model 第 1 次（兩層映射） | 低（grep `mint` 找不到 ticket） |
| 扁平（linux 建議） | `0.18.0-W17-062-mint-1` | 直接讀懂代理人名 + 樣本序號 | 高（grep `mint` 直接命中） |

PM 在實驗執行期需頻繁 `ticket track list` 查狀態，4 層後綴需 PM 記住 A1-A19 對應哪個代理人——19 個對應關係超過 Miller's Law 7±2 工作記憶上限。扁平命名直接消除這個記憶負擔。

進一步：建議 `-1` ~ `-3` 為原 model，`-4` ~ `-6` 為降級 model，但加語義後綴更清楚：`-orig-1` ~ `-orig-3`、`-down-1` ~ `-down-3`。代價是 ticket id 長度 +6 字元，但讀懂成本降到零。最終格式：`0.18.0-W17-062-mint-orig-1`、`0.18.0-W17-062-mint-down-1`、`0.18.0-W17-062-mint-R`。

1. Q2 樣本數：重跑機制加早停條件（worst case 上限 152 IMP，避免級聯）
2. Q3 任務長度：把「≤ 14 tool calls」明示為 acceptance 一條，不靠執行自律
3. Q4 派發策略：frozen context 寫入實驗組 ANA 的 Context Bundle，6 IMP 引用而非各自複製，token 省 ~20%

### 產出 1：實驗任務範本設計原則

#### 設計目標

範本須同時滿足：(a) 可複現（6 次執行輸入完全一致）；(b) 能暴露代理人真實能力差異（不能只測機械套用）；(c) 規模適中（單次 < 15 tool calls，避免 PC-042 回合耗盡污染對照）。

#### 三維度選擇矩陣

| 維度 | 選項 A（保守） | 選項 B（中庸） | 選項 C（暴露） | lavender 預判 |
|------|--------------|--------------|--------------|--------------|
| 任務長度 | 5-8 tool calls | 10-12 tool calls | 15-18 tool calls | B（避開 A 過短遮蔽差異 / C 觸碰回合限制） |
| 工具呼叫類型 | 純 Read | Read + Grep + Edit | Read + Grep + Edit + Bash | B（涵蓋主要工具，不引入 Bash 變異） |
| 判斷複雜度 | 單條規則套用 | 2-3 規則交互 | 3+ 規則含邊界 | C（模型差異主要在交互推理，B 不夠暴露） |
| 輸出型態 | 純編輯 | 編輯 + 摘要回報 | 編輯 + 設計決策說明 | 依代理人職能對齊（見下表） |
| 上下文規模 | < 5 KB | 5-15 KB | > 15 KB | B（避免 input bloat 干擾） |

**判別準則**：以 B 為基線，**判斷複雜度提到 C** 暴露推理差異。任務長度若採 C 風險過高（會與「執行能力下降」混淆，因 PC-042 限制本就會讓長任務頻繁失敗）。

### 產出 2：多視角分析規範

#### 視角組合（候選方案）

| 方案 | 視角組合 | 優點 | 缺點 |

| α 三視角 | linux（架構/邏輯） + ginger（成本量化） + basil-writing-critic（文字品質） | 涵蓋邏輯/成本/風格三軸 | 三視角中 basil 屬「不重要差異」軸，比重需明示 |

### 產出 3：評分標準

#### 三檔判定

| 判定 | 條件 | 動作 |
|------|------|------|
| **通過** | 3 視角中 ≥ 2 視角無重要差異，且重要差異視角僅報「文字品質」類差異 | 確認可降級，更新 frontmatter |
| **警告** | 1-2 視角報告 1 項重要差異，但非「執行能力下降」類 | 設 2 Wave 監測期，期間記錄失敗率，超 20% 回滾 |
| **回滾** | 任一視角報告以下任一項：(a) 邏輯/判斷錯誤 ≥ 1 件；(b) 執行能力下降（acceptance_met < 原 model 平均）；(c) 回合耗盡頻率 > 原 model 2 倍 | 立即回滾 frontmatter，本代理人標記 `model_downgrade_blocked`，記錄案例 |

#### 量化門檻（重要差異判別）

| 差異類型 | 量化門檻 | 來源依據 |
|---------|---------|---------|
| 邏輯錯誤 | 結論與規格矛盾 ≥ 1 件 / 3 次降級組 | acceptance 客觀比對 |
| 判斷錯誤 | 多約束情境中遺漏約束 ≥ 1 項 / 3 次 | 視角分析認定 |
| 執行能力下降 | acceptance_met 降幅 ≥ 33%（即 3 次降級組比 3 次原 model 組少完成 ≥ 1 個 AC） | ticket Exit Status 欄位 |
| 回合耗盡頻率 | 降級組 ≥ 2/3 次出現 partial_success/failed，且原 model 組 ≤ 1/3 | ticket Exit Status 欄位 |
| 文字品質違規 | 不論次數 | 不入判準（記錄即可） |

#### 統計門檻說明

3 次重複是最小可信樣本。lavender 自評：3 次無法做嚴格統計顯著性檢定（n=3 變異大），採「絕對門檻」（如 ≥ 1 件邏輯錯誤即觸發）而非「比較均值」更穩健。若多視角審查認為 3 次過少，可在本規格擴增為 5 次（成本 +66%），由 PM 決策。
