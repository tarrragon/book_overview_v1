### 額外證據（2026-05-05 本 session 即時觀察）

W17-145 ticket 派發過程中，PM 寫 AskUserQuestion description 時 hook 即時攔截一個簡體字 `实` (U+5B9E) 出現於「实際」（應為「實際」）。這提供了 PC-072 污染源的具體實證：

| 觀察 | 涵義 |
|------|------|
| 污染發生於 AI 即時生成（PM 自己打字） | 污染源不在 framework 文件 |
| Hook 偵測到並阻擋 | 防護層運作正常 |
| 字 `实` 與 `實` codepoint 相鄰 + 語意相同 | 模型 token 預測時偶爾選錯近鄰字（CJK 字元 codepoint 鄰近性 / PC-085）|

**對 W17-144 的啟示**：用 zhtw-mcp 掃描 `.claude/` 中 AskUserQuestion 引用文件可能得到「無污染」結果，因為污染源是 AI 生成過程而非靜態文件。若掃描結果為 0 命中，應將結論導向「污染源在 token 預測機率分佈，不在文件」，並記錄此即時案例作為強佐證。

## 重現實驗結果

### 實驗方法

兩階段實驗：
1. **第一輪**：自製 Python 腳本掃 74 個 markdown/text 候選檔案找簡體字 codepoint
2. **第二輪**：直接用 `mcp__zhtw-mcp__zhtw` 工具對含已知污染字元的測試文字做能力驗證

### 實驗執行

#### 第一輪：自製 Python 腳本（失敗，犯 PC-074 同類錯誤）

腳本路徑：`/tmp/w17144_scan.py`，SIMPLIFIED_SET 包含繁簡共用字（件/本/保/言/系/明 等），導致 6911 個誤報，74 個檔案全部「中招」，明顯 false positive 爆炸。

**自我教訓**：自製簡體字偵測必須只列繁/簡明確不同的字元對；繁簡共用字（PC-074 範圍）必須排除。第一輪失敗本身即是 PC-074 的活案例自證。

#### 第二輪：直接用 zhtw-mcp 實測能力

**測試 1**：傳入含已知污染字的混合文字（独立、违反、补充、隶屬、遗留、⚡扯、补 spawned_tickets、隸/遺/補/獨 對照組）

```json
{
  "detected_script": "traditional",
  "issues": [{"found": "主線程", "suggestions": ["主執行緒"], "rule_type": "cross_strait", "severity": "warning"}],
  "summary": {"errors": 0, "warnings": 1, "info": 0}
}
```

完全沒偵測到「独/违/补/隶/遗/⚡」，僅報 1 個跨海峽用語警告。

**測試 2**：傳入 53 個明顯簡體詞（独立/违反/补充/...红色/绿色）

```json
{
  "detected_script": "simplified",
  "s2t_applied": true,
  "issues": [{"found": "視頻", "suggestions": ["影片","影音","視訊"], "severity": "info"}],
  "summary": {"errors": 0, "warnings": 0, "info": 1}
}
```

工具自動 S2T 轉繁，**53 個簡體字本身完全沒被標記為 issue**，只報 1 個 info（「視頻」轉繁後跨海峽用語建議）。

**測試 3**：傳入含 7 個 emoji 的文字（⚡✅❌🔴📝🔧✔️⚠️🚨💡）

```json
{
  "issues": [],
  "summary": {"errors": 0, "warnings": 0, "info": 0}
}
```

emoji 完全不在工具規則範圍。

### 實驗發現

#### 已驗證事實

1. **zhtw-mcp 不偵測簡體字**：工具設計目標是「自動 S2T 轉換 + 跨海峽用語 + 翻譯腔 + AI 寫作 + 教育部標準字」。簡體字輸入會被視為「需要轉換」（s2t_applied=true）而非「需要報錯」。
2. **zhtw-mcp 不偵測 emoji**：emoji 完全不在工具的 1736 條規則中。
3. **zhtw-mcp 對 PC-072 場景不適用**：W17-144 原 strategy 假設「用 zhtw-mcp 系統性掃描 .claude/ 找污染源」在工具能力層即不成立。

#### 額外證據（W17-145 即時觀察，已寫入 Problem Analysis 額外證據段）

本 session 派發 W17-145 時，PM 寫 AskUserQuestion description 即時打入簡體字 `实` (U+5B9E)，被 `askuserquestion-charset-guard-hook.py` 攔截。

| 觀察 | 涵義 |
|------|------|
| 污染發生於 AI 即時生成（PM 自己打字） | 污染源不在 framework 文件 |
| Hook 偵測到並阻擋 | 防護層運作正常 |
| 字 `实` 與 `實` codepoint 相鄰 + 語意相同 | 模型 token 預測時偶爾選錯近鄰字（PC-085）|

#### 推論結論

PC-072 污染源 = **AI 生成過程的 token 機率分佈相鄰污染**，不在靜態文件。靜態文件即使全乾淨，污染仍會在每次 AUQ payload 生成時隨機出現。現有 `askuserquestion-charset-guard-hook.py` 是正確且唯一可行的防線層級（生成時攔截），不需修改文件源。

#### 仍未驗證的假設

- 是否有特定 token 上下文（如 JSON escape 邊界、長 description）會放大污染機率：本 ticket 範圍外，可能值得後續 ANA 分析 hook log 統計。

### 解釋為何 zhtw-mcp 不適用

zhtw-mcp 工具職責：
- 跨海峽用語正規化（軟件→軟體、視頻→影片）
- 教育部標準字校正
- 翻譯腔（歐化句法）偵測
- AI 寫作模式偵測

**不在工具職責**：偵測「混入文件中的簡體字 codepoint」、偵測 emoji。

工具設計者邏輯：簡體字輸入 = 用戶不熟悉繁體 → 自動 S2T 轉換為輸出，不視為錯誤。這對「翻譯/正規化」場景合理；但對 PC-072 場景（找污染）不合理。

### PC-072 防護的正確架構

| 層級 | 工具 | 職責 |
|------|------|------|
| 生成攔截（運作中） | `.claude/hooks/askuserquestion-charset-guard-hook.py` | PreToolUse 偵測 AUQ payload 的簡體字/emoji，阻擋送出 |
| 文件正規化（工具能力） | `mcp__zhtw-mcp__zhtw` | 跨海峽用語 / 翻譯腔 / 教育部標準字（正常文件審稿用） |
| Hook 字元集污染偵測 | 自製 Python（避免 PC-074 共用字陷阱） | 補充 zhtw-mcp 不覆蓋的「文件中混入簡體字」場景 |

### 不開 IMP follow-up 的理由

| 候選 IMP | 是否開 | 理由 |
|---------|-------|------|
| 修文件中的污染 | 不開 | 推論污染源不在文件，無修復目標 |
| 強化 zhtw-mcp 能力 | 不開 | 工具是外部 MCP，不應改造為偵測器 |
| 補強 askuserquestion-charset-guard | 不開 | 現有 hook 已運作（W17-145 即時案例證明），無新缺陷 |
| 自製文件全掃工具（補 PC-072 文件側監測） | **建議追蹤但本 ticket 不開** | 屬「污染源不在文件但仍想全掃」的延伸主題，與 W17-144 acceptance 不直接綁定；若未來 PC-072 再現且懷疑文件污染，可獨立開 ANA |
