### 新增章節：ID 演進歷程

在 `docs/bookstores/readmoo.md` 現有「Book ID 來源優先級（必讀）」章節之後，新增「ID 演進歷程」章節，涵蓋四段內容：

#### 1. 歷史 ID 策略 cover-XXX

- 早期實作：`bookData.id = cover-{coverPathSegment}`（取自封面 CDN 路徑）
- 來源檔：`src/content/adapters/readmoo-adapter.js` 的 `applyIdGenerationStrategiesWithInfo` 舊版本（cover 策略優先）

#### 2. 發現問題（W6-012.2 ANA）

- 用戶實機測試發現 link 為 dummy URL `/api/reader/210017268000101`，下游功能受阻
- ANA chrome-devtools-mcp 證實：
  - DOM 已有 96 個 `[id^="privacy-"]` element（與 .library-item 1:1 對應），格式 `privacy-{8位數字}`
  - extension 已實作 privacy element 提取邏輯，但 `bookData.id` 主欄位被 cover 策略覆寫
- 真根因：ID 策略順序錯誤（cover → title → reader-link），cover 永遠優先

#### 3. 修復方案 reader-{privacyBookId}（W6-012.2.1）

- 策略順序改為 reader-link > cover > title > fallback
- cover 增加 known-unstable 過濾（openbook / undefined / placeholder / default）
- `bookData.id` 從 `cover-XXX` 變為 `reader-{8位數字}`，與 Readmoo 內部 book ID 一一對應

#### 4. 遷移流程（W6-012.2.2 / .2.1 / .2.2）

- MigrationService class（W6-012.2.2.1）統一處理 schema 演進
- cover-to-reader migration step（W6-012.2.2.2）依 5 案例合併規則處理：
  1. 正常遷移（identifiers.privacyBookId 存在）
  2. privacyBookId 為空 → 保留舊 ID + manual_review 標記
  3. cover-openbook 共用 ID → secondary key 去重
  4. 同 privacyBookId 多筆 → 取新並集 tag
  5. cross-device sync 衝突（範圍外 follow-up）
- 備份隔離 key `migration_backup_v3_1`，schema_version → 3.1.0
- 觸發時機：install-handler.onUpdated 偵測版本升級

#### 5. 用戶可見影響

- 升級後首次啟動：背景自動遷移，UI 顯示完成通知 + 受影響書籍數
- privacyBookId 為空的書籍：UI 標記「待重新提取」徽章
- cover-openbook corrupt 書籍：列入 legacy_duplicate 清單，建議用戶人工確認
- CSV 匯出檔的 `id` 欄位變更：升級前匯出的檔案 id=cover-XXX，升級後 id=reader-XXX；外部資料無法自動遷移，但 `identifiers.privacyBookId` 可作為 cross-version key
