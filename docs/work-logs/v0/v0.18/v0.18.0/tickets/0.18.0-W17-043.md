## 觸發案例紀錄

### 案例 1（建立動機）
2026-04-21 W17-041 PCB append-log 操作時觸發場景 #17 AUQ「錯誤學習經驗確認」，PC-075 異動由 W17-042 DOC ticket completed 落地，本次操作不修改 error-pattern，屬於誤觸發。

### 案例 2
2026-04-21 W17-036 ANA complete 操作時再次觸發場景 #17 AUQ「錯誤學習經驗確認」，PC-075 異動同樣由 W17-042 已落地，W17-036 本身是 PC-075 下游傳播的觸發者但不新增 error-pattern。本次 complete 前觸發屬場景 #17 的原始設計時機（complete 前），但判定邏輯仍誤判：它只看「session 內有無異動過的 error-pattern」，不看「異動是否屬本 ticket 職責」。

### 誤觸發模式總結

| 觸發點 | 實際操作 | 誤觸發原因 |
|-------|---------|-----------|
| append-log Bash 命令 | W17-041 PCB 寫入 | Hook 在 ticket complete 以外的 Bash 也觸發 |
| complete Bash 命令 | W17-036 complete | Hook 判定 session 內 PC-075 有異動即觸發，不問異動歸屬 |

## 分析方向

1. **Hook 觸發時機校正**：場景 #17 應限於「ticket complete 成功後」且「該 ticket 本身新增/修改了 error-pattern」
2. **異動歸屬檢查**：需比對「session 內異動的 error-pattern」與「當前 ticket 的變更範圍」，避免跨 ticket 誤觸發
3. **降噪替代**：若無精確歸屬判定能力，改為 daily session summary 級別的提醒，而非每 Bash 操作都觸發

## 建議衍生 IMP（待 ANA 分析後決定）

- 修正 scenario-17 hook 觸發條件（限 complete 成功 post-event + 本 ticket 職責內 error-pattern 異動）
