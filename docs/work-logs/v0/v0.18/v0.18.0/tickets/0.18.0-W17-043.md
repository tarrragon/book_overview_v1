## Problem Analysis

### 觸發案例紀錄

#### 案例 1（建立動機）

2026-04-21 W17-041 PCB append-log 操作時觸發場景 #17 AUQ「錯誤學習經驗確認」。PC-075 異動由 W17-042 DOC ticket completed 落地，本次 append-log 操作不修改 error-pattern，屬於誤觸發。

#### 案例 2

2026-04-21 W17-036 ANA complete 操作時再次觸發場景 #17 AUQ。PC-075 異動同樣由 W17-042 已落地，W17-036 本身是 PC-075 下游傳播的觸發者但不新增 error-pattern。本次 complete 前觸發屬場景 #17 的原始設計時機（complete 前），但判定邏輯仍誤判：它只看 session 內有無異動過的 error-pattern，不看異動是否屬本 ticket 職責。

#### 誤觸發模式總結

| 觸發點 | 實際操作 | 誤觸發原因 |
|-------|---------|-----------|
| append-log Bash 命令 | W17-041 PCB 寫入 | Hook 在 ticket complete 以外的 Bash 也觸發 |
| complete Bash 命令 | W17-036 complete | Hook 判定 session 內 PC-075 有異動即觸發，不問異動歸屬 |

### 分析方向

1. Hook 觸發時機校正：場景 #17 應限於 ticket complete 成功後，且該 ticket 本身新增/修改了 error-pattern
2. 異動歸屬檢查：需比對 session 內異動的 error-pattern 與當前 ticket 的變更範圍，避免跨 ticket 誤觸發
3. 降噪替代：若無精確歸屬判定能力，改為 daily session summary 級別的提醒，而非每 Bash 操作都觸發

### 案例 1（append-log 誤觸發）已被 W10-047.1 修復

修復方式：acceptance-gate-hook.py 第 572-583 行增加降級 fast-path（commit `05f328b7`，2026-05-06 01:01）。在重操作前以最低成本判斷：tool_name 非 Bash 或 command 非 ticket track complete/batch-complete 直接放行。

```python
if input_data.get("tool_name") != "Bash" or not is_complete_command(_fp_command):
    logger.debug("Fast-path skip: 非 ticket track complete 命令")
    _output_allow_json()
    return EXIT_SUCCESS
```

`is_complete_command()` 僅匹配 `ticket track complete` 或 `ticket track batch-complete`，故 append-log/claim/show 等 Bash 命令均不觸發場景 #17。

時序對照：案例 1 發生 2026-04-21 → 當時無 fast-path；W10-047.1 fast-path 上線 2026-05-06 01:01。

### 案例 2（complete 時跨 ticket 誤判）已被 W10-087 修復

修復方式：`acceptance_checkers/error_pattern_attribution.py`（commit `bfd856dd`，2026-04-19 15:55）增加 `filter_error_patterns_by_ticket_scope()`，採短路求值兩階判定：

1. PC 檔案 frontmatter `source_ticket` 等於當前 ticket_id 即歸屬
2. 無 frontmatter 時，檢查 ticket md 是否引用該 PC（basename 或 PC ID）即歸屬

時序對照：案例 2 發生 2026-04-21；W10-087 attribution filter 上線 2026-04-19 15:55，案例 2 發生時 fix 已 deployed。
