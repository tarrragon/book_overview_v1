## Problem Analysis

### 問題根因

`validate_execution_log_by_type` 在 `ticket_validator.py:566-572` 使用 `body.find("## {section}")` / `body.find("### {section}")` 做 section header 定位，屬 **substring** 匹配而非 line-anchored 匹配。

當 ticket body 含 backtick 包住的章節名（如 Problem Analysis 內寫「該章節指 `## Test Results`」），`body.find("## Test Results")` 會命中 backtick 內部的 substring，而非真正的行首 header。後續 `content_start = body.find("\n", section_start) + 1` 會跳到 backtick 結束後那一行，而 `next_section_idx` 則命中真正的 `^## Test Results` 行首，導致 `section_content` 被擷取為空字串，誤判為 placeholder。

重現指令（已實測）：
```
body = "## Problem Analysis\n\n這裡指 `## Test Results` 的內容\n\n## Test Results\n\npytest 全綠。\n"
body.find("## Test Results")  # 返回 31（backtick 內）而非行首
# 後續 content_start=53, next_section_idx=54，extract 為空字串 → false positive
```

### 影響範圍

- `.claude/skills/ticket/ticket_system/lib/ticket_validator.py::validate_execution_log_by_type`
- 呼叫鏈：ticket complete 流程驗證 body schema 時對含自我引用章節名 (backtick 包住) 的 body 誤擋（false positive）

W17-071 已修復 `validate_execution_log`（通用三章節版本）的同類 bug，改用 `re.search(rf"^{level} {re.escape(section)}\b", ..., re.MULTILINE)`；但 `validate_execution_log_by_type`（type-aware 版本）在 W17-016.3 新增時仍沿用 `body.find`，屬同家族漏網之魚。

### 相關 Error Pattern

與 W17-071 根因 B（section 邊界偵測不精確）同家族；本 ticket 補 type-aware 路徑的漏網之魚。無既存 error-pattern 需新增；若累積第三例同類 bug 應考慮建立「string-based section header detection」pattern。

### 修復策略

將 `validate_execution_log_by_type` 的 section header 定位從 `body.find(pattern)` 的 substring 匹配改用 `re.search(rf"^{level} {re.escape(section)}\b", body, re.MULTILINE)` 的 line-anchored regex（W17-071 已在 `validate_execution_log` 套用相同 pattern，本次補 type-aware 版本漏網之魚）。

### 修改檔案

- `.claude/skills/ticket/ticket_system/lib/ticket_validator.py::validate_execution_log_by_type`
  - 替換 `header_patterns = [f"### {section}", f"## {section}"]` + `body.find()` 迴圈
  - 改為 `for level in ("###", "##")` + `re.search(rf"^{level} {re.escape(section)}\b", body, re.MULTILINE)`
  - docstring 標註 W17-074，補說明 line-anchored 避免 backtick 誤判
- `.claude/skills/ticket/ticket_system/tests/test_ticket_validator.py::TestValidateExecutionLogByTypeBacktickBoundary`
  - 新增 6 個 regression cases（TC-074-01 ~ TC-074-06）
  - 覆蓋：backtick 包住的章節名引用、多個 backtick 引用、ANA 雙方向引用、backtick 引用但真章節缺失、backtick 引用 + 真章節空殼、段落中行中 `## ` 非行首字串

### Dissenting View

無並行代理人產出互斥結論；本 ticket 為 W17-071 ANA 衍生的 IMP，根因分析已於 W17-071 確定屬同家族修復，無重大替代方案爭議。

### 可替代方案（已納入考量）

- 方案 A（已採用）：line-anchored regex（`^## Section\b` multiline）— 與 W17-071 修復方式一致，維持整個 validator 的偵測策略對稱性。
- 方案 B：保留 `body.find` 但額外檢查前後字元是否為 backtick — 特例 patch，覆蓋面窄且未解決「行中 `## `」類情境；拒絕。
- 方案 C：抽取 `_find_section_header(body, section)` 共用函式同時給 `validate_execution_log` / `validate_execution_log_by_type` 呼叫 — DRY 方向正確但超出本 ticket 範圍，留待後續同家族問題第三次出現時升級。
