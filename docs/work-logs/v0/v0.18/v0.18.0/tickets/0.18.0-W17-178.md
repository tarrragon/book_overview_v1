### 問題根因

**根因分類：(c) 規則語意未涵蓋 W17-176 案例**

`ana_spawn_consistency_checker.py` 的 `_SPAWN_ROW_PATTERN` 正則只匹配「row-per-spawn」表格格式：

```
| 1 | IMP | P1 | 修復 A | a.py | thyme |
```

正則要求同一行內同時出現 `| (IMP|DOC|ANA) |` 和 `P[0-3]`。

W17-176 的 Solution 使用「key-value」格式：

```
### Spawned IMP 規劃

| 欄位 | 值 |
|------|------|
| action | 修復 stop-worklog-handoff-sync-check-hook.py 三個根因 |
| priority | P1 |
| who | thyme-extension-engineer |
```

此格式中：
- `IMP` 出現在 H3 標題（`### Spawned IMP 規劃`），不在表格行內
- `P1` 出現在 `| priority | P1 |` 行，但該行無 `IMP/DOC/ANA` cell
- 正則匹配結果 N=0，checker 在 L128-130 直接 return（「無 spawn 規劃表格行」），從未進入比對邏輯

**直接原因**：W17-168 實作時的 RED 測試僅覆蓋「row-per-spawn」格式（W17-162 / W17-167 案例），未考慮代理人可能使用「key-value」格式撰寫 spawn 規劃。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `.claude/hooks/acceptance_checkers/ana_spawn_consistency_checker.py` | 正則 `_SPAWN_ROW_PATTERN` 涵蓋格式不足 |

### 相關 Error Pattern

- W17-168 實作時的格式假設過窄（無對應 PC，本案為首次發現）
- 與 PC-063（重現實驗必填）相關：本 ANA 需重現實驗驗證
### 重現實驗

**實驗方法**：以 W17-176 的 Solution spawn 表格為輸入，測試 `_SPAWN_ROW_PATTERN` 匹配行為。

**實驗執行**：

1. 對 W17-176 key-value 格式表格執行正則：`matches = _SPAWN_ROW_PATTERN.findall(section)` → 結果 `[]`，N=0
2. N=0 觸發 L128-130 early return `(False, None)`，checker 判定「無 spawn 規劃」直接通過
3. Step 2.5.2 確實被呼叫（type=ANA 條件滿足），但因 N=0 放行

**實驗發現（已驗證事實）**：

1. 正則 `_SPAWN_ROW_PATTERN` 要求同一行同時含 `| (IMP|DOC|ANA) |` 和 `P[0-3]`
2. W17-176 的 key-value 格式將 type 放在 H3 標題、priority 放在獨立行，正則無法匹配
3. 根因確認為 **(c) 規則語意未涵蓋**：正則只識別 row-per-spawn 格式

### WRAP 分析

#### W — Widen（擴增選項）

| 方案 | 描述 | 優點 | 缺點 |
|------|------|------|------|
| A. 擴充正則涵蓋 key-value 格式 | 新增正則匹配 `### Spawned (IMP|DOC|ANA)` H3 標題 | 直接修復漏判 | 格式持續演化則持續補正則，治標 |
| B. 改用 heading keyword 偵測 | 掃描 H3 標題含 `Spawn` + `IMP/DOC/ANA`，不依賴表格行格式 | 格式無關，涵蓋面最廣 | 可能誤判（如討論段落提及 Spawned IMP） |
| C. 雙策略組合（正則 + heading） | 保留現有 row-per-spawn 正則，新增 heading-based 偵測作第二層 | 向後相容 + 涵蓋新格式 | 兩套邏輯維護成本 |
| D. 標準化 spawn 表格格式 | 在 ticket-body-schema 強制統一 row-per-spawn 格式 | 從源頭解決 | 需所有代理人遵循，短期無法強制 |

#### R — Reality-test（現實檢驗）

- 方案 A 最小改動但仍依賴特定格式，未來新格式會再次漏判
- 方案 B 涵蓋面最廣但有誤判風險；可用 `spawned_tickets+children==0` 作為必要條件降低誤報（H3 含 spawn keyword + 實際無 ticket = 高可信度阻擋）
- 方案 C 向後相容但兩套邏輯長期維護成本高
- 方案 D 是根本解法但需要 Schema 層配合，短期不可行作為獨立修復

**最可能失敗原因**：方案 B 在非 spawn 語境（如「不 spawn」討論段落）誤報 → 豁免標記機制已存在可緩解

#### A — Attain distance（機會成本）

- 選 C 的成本：1 IMP（P2），約 20 分鐘，涵蓋面足夠
- 不修復的成本：後續 ANA 使用 key-value 格式時 spawn 規劃繼續漏建
- 方案 D（標準化）可作為獨立 DOC ticket 後續處理，不阻擋本次修復

#### P — Prepare to be wrong（預備失敗）

- 若方案 C 實作後仍有漏判格式 → 建新 ANA 再次擴充，或加速推動方案 D
- 驗證方式：以 W17-176 的 Solution 為測試樣本，確認修復後 N>0

### 決策結論

**選擇方案 C（雙策略組合）**：保留現有 row-per-spawn 正則（策略 1），新增 heading-based spawn 偵測（策略 2）。

修復邏輯：
1. 策略 1（現有）：`_SPAWN_ROW_PATTERN` 掃描表格行 → N1
2. 策略 2（新增）：掃描 H3 標題含 `Spawn` 關鍵字 + `IMP|DOC|ANA` → N2（每個匹配的 H3 計為 1 項 spawn 規劃）
3. N = max(N1, N2)，取較大值
4. 後續邏輯不變（N>0 且 S+C==0 → 阻擋）
