### 問題根因

5 個 hook 各自硬編碼扁平 ticket 路徑 docs/work-logs/v{version}/tickets/，未使用 hook_utils/hook_ticket.py 中已存在的共用 helper（find_ticket_file / find_ticket_files / scan_ticket_files_by_version）。這些 helper 已支援扁平與三層階層結構，但 5 個 hook 在 helper 建立前各自實作了路徑組裝邏輯，未隨 W17-176.2.1 同步更新。

### 影響範圍

5/5 全部為 (a) 需修。每個 hook 都對硬編碼路徑做實際 file IO（glob / exists / read），不是純字串引用：

- commit-handoff-hook.py:173 — scan_wave_tickets() glob + 讀 frontmatter 判斷 wave 完成
- tech-debt-reminder.py:72 — find_tickets_directory() 回傳 tickets_dir 供後續讀取
- post-git-commit-hook.py:177 — _scan_wave_tickets() glob + 讀 frontmatter 判斷 wave 完成
- prompt-submit-hook.py:197 — resolve_ticket_path() 構建路徑做 .exists() 存在性驗證
- file-ownership-guard-hook.py:453 — get_active_tickets() glob + 讀 frontmatter 掃描同 Wave ticket

### 相關 Error Pattern

- ARCH-020（跨進程重複驗證邏輯）：5 個 hook 各自重寫路徑組裝，與 hook_utils 共用 helper 形成 ARCH-020 模式
## 重現實驗結果

### 實驗方法

在專案根目錄執行 Python 腳本，比較扁平路徑與三層階層路徑的 ticket 可見性差異。

### 實驗執行

指令：
```
python3 -c "from pathlib import Path; flat=Path('docs/work-logs/v0.18.0/tickets'); nested=Path('docs/work-logs/v0/v0.18/v0.18.0/tickets'); print(f'Flat exists: {flat.exists()}'); print(f'Nested exists: {nested.exists()}'); print(f'Nested count: {len(list(nested.glob(chr(34)+'*.md'+chr(34))))}');"
```

輸出：
```
Flat exists: False
Nested exists: True
Nested count: 817
```

### 實驗發現

已驗證事實：
1. 扁平路徑 `docs/work-logs/v0.18.0/tickets/` 不存在（`v0.18.0/` 目錄存在但無 `tickets/` 子目錄，僅有 `reports/`）
2. 三層階層路徑 `docs/work-logs/v0/v0.18/v0.18.0/tickets/` 存在且含 817 個 ticket 檔案
3. 所有使用硬編碼扁平路徑的 hook 在 v0.18.0 版本中回傳空結果，功能靜默失效
4. `hook_utils/hook_ticket.py` 已有 `scan_ticket_files_by_version()` 和 `find_ticket_file()` 支援雙結構，可直接替換

### 分類結論

5/5 全部為 (a) 需修。無 (b) 或 (c) 類。

### 統一同步策略：是否抽 helper

不需要新建 helper。hook_utils/hook_ticket.py 已有完整的共用函式：

| 用途 | 共用 helper | 支援扁平+三層 |
|------|-----------|-------------|
| 掃描特定版本所有 ticket | scan_ticket_files_by_version(project_root, version, logger) | 是 |
| 查找單一 ticket 檔案 | find_ticket_file(ticket_id, project_root, logger) | 是 |
| 掃描所有版本 ticket | find_ticket_files(project_root, version, logger) | 是 |

修復策略：每個 hook 刪除自己的硬編碼路徑組裝邏輯，改為呼叫對應的 hook_utils helper。5 個 hook 已全部 import hook_utils，無需新增依賴。

### IMP spawn 規劃表（待 PM 建立）

建議合併為 1 個 group IMP（符合跨進程同步修復豁免條款：5 hook 同根因同修法、預估修改檔案數 = 5、邏輯同步性高）：

| ticket | priority | who | acceptance |
|--------|----------|-----|-----------|
| (待 PM 建立) 修復 5 hook 硬編碼扁平 ticket 路徑改用 hook_utils helper | P1 | thyme-extension-engineer | (1) 5 hook 不再含硬編碼扁平路徑組裝 (2) 改用 scan_ticket_files_by_version 或 find_ticket_file (3) 既有 hook 測試全綠 (4) 重現實驗驗證：hook 在三層階層結構下能正確找到 ticket |

備選方案：若 PM 認為 5 hook 過多，可拆為 2 個 IMP（wave-scan 類 3 hook + 單 ticket 查找類 2 hook）。

### Spawn 落地確認

- IMP spawn 規劃 1 項：待 PM 建立（本 ANA 為 read-only 分析者，無 ticket create 權限）
