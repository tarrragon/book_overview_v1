### (A) 兩天前的搜尋方法盤點

PC-050 模式 D 新增前後，PM 實際嘗試過的路徑：

| 搜尋路徑 | 使用工具 | 結論 |
|---------|---------|------|
| 找專案 Hook 機制 | `grep/find .claude/hooks/` | 找到 `dispatch-active.json` + `agent-commit-verification-hook`（只在代理人完成後觸發） |
| 找專案內建 API | `grep "dispatch_tracker\|active-dispatch"` | 找到檔案系統持久化機制，但無法查詢 runtime 狀態 |
| 讀 transcript 檔案 | `Read /tmp/claude-501/tasks/{agentId}.output` | 讀到流式 JSONL，誤判中間狀態為「失敗」→ 觸發 PC-050 模式 D |
| 系統級工具發現 | **未執行** | 沒有想到用 ToolSearch 查 CC runtime 能力 |

**選擇的解法**：規則化「禁止讀 transcript」——用**封閉防護**取代**尋找正確工具**。

### (B) 為何沒想到 ToolSearch？根因分析

此為核心盲點。整理出五個成因：

**成因 1：ToolSearch 被框架為「AskUserQuestion 專用前置步驟」**

`.claude/pm-rules/askuserquestion-rules.md` 第 29、47、49、146 行皆描述為「使用 AskUserQuestion 前必須執行」。PM 的心智模型把 ToolSearch 當成**單一用途鑰匙**（只為 AskUserQuestion 服務），而非**通用能力發現機制**。

**成因 2：`search-tools-guide` skill 缺少「CC Meta-Tools」章節**

該 skill 覆蓋 Grep/Serena/WebSearch/Dart MCP/Glob，都是**程式碼 / 網頁搜尋**。PM 遇到「觀察代理人」需求時，腦中對應的 skill 是「找工具 = 找程式碼」，不會聯想到「找 CC runtime 的能力」。

**成因 3：每次 session 的 deferred tools 清單被當成背景資訊**

CC runtime 在每個 session 的 system-reminder 中列出所有 deferred tools（含 TaskOutput/TaskList/TaskStop/SendMessage/WebFetch 等）。但這段訊息**沒有互動性**（不是提示 PM 主動去用），PM 當成系統雜訊跳過。

**成因 4：問題框架錯誤（限制性解法 vs 探索性解法）**

PM 當時面對「讀 transcript 誤判」時，自然的反應是**「禁止讀 transcript」（限制性解法）**。這個框架下，PM 的搜尋目標是「如何強制不讀」，而不是「是否有工具可正確判斷狀態」。

如果問題框架是「如何正確取得代理人的 runtime 狀態」，就會觸發「找查詢工具」的思路，ToolSearch 才可能進入視野。

**成因 5：feedback memory 寫下原則但未落地檢查清單**

同日建立的 `feedback_exhaust_indirect_before_impossible.md` 列出四問（Hook / 檔案 / 流程 / 既有 API），但**沒有第五問「CC runtime deferred tools」**。原則正確但覆蓋不全，同一 session 內新教訓未即時擴充檢查清單。

### S1：升級 search-tools-guide skill

在 SKILL.md 新增章節：

```
## Claude Code Meta-Tools（平台能力發現）

### ToolSearch — Deferred Tools 發現機制

CC runtime 將部分工具以 deferred 模式提供（每 session 的 system-reminder 會列出）。
使用前必須 `ToolSearch("select:<tool_name>")` 載入 schema，否則呼叫失敗。

| 需求 | 對應 deferred tool |
|------|------------------|
| 查詢背景代理人 runtime 狀態 | TaskOutput（只讀 <status>，禁讀 body） |
| 派發背景代理人 | TaskCreate + Agent |
| 停止失控代理人 | TaskStop |
| 向代理人發送新指令 | SendMessage |
| 讓用戶做決策 | AskUserQuestion |
| 抓取網頁 / 文件 | WebFetch / WebSearch |
| 排程自動執行 | CronCreate / CronList / CronDelete |
```

並在選擇決策樹加入分支：

```
搜尋需求
    |
    +-- 找 CC runtime 能力（觀察代理人、排程、網頁抓取） → ToolSearch
```

### S2：抽象 ToolSearch 規則

建立 `.claude/rules/core/tool-discovery.md` 或併入 search-tools-guide，內容：

> **規則**：遇到「我想做 X 但不知道怎麼做」時，在宣告「做不到」或使用「封閉性限制」解法之前，必須執行 `ToolSearch` 搜尋是否有 deferred tool 對應此需求。

然後 askuserquestion-rules.md 的「ToolSearch 前置載入」改為引用此通用規則 + 具體到 AskUserQuestion 的用例。

### S3：pm-agent-observability.md 補註

在 TaskOutput 章節開頭加入：

> TaskOutput 是 CC deferred tool，新 session 首次使用前需執行 `ToolSearch("select:TaskOutput")` 載入 schema。

### S4：feedback memory 新增第五問

`feedback_exhaust_indirect_before_impossible.md` 的四問擴充為五問：

```
(1) Hook 能推送嗎？
(2) 檔案系統能追蹤嗎？
(3) 流程能繞過嗎？
(4) 既有模組有 API 但沒接線嗎？
(5) CC runtime 有 deferred tool 嗎？（執行 ToolSearch 搜尋）
```

### S5：新增 error pattern

建立 `.claude/error-patterns/process-compliance/PC-060-meta-tool-discovery-blindness.md`：
- 症狀：PM 在「找工具」需求中忽略 CC runtime deferred tools，導致採用限制性解法
- 根因：search-tools-guide 只覆蓋程式碼搜尋；ToolSearch 被框架為 AskUserQuestion 專用
- 防護：每 session 主動掃描 deferred tools 清單；tool-discovery 檢查清單第五問

### 關鍵教訓

1. **心智模型錯誤比工具缺失更危險**：ToolSearch 早已存在於 `askuserquestion-rules.md`，但被「單一用途」框架限制，讓 PM 不會在「觀察代理人」情境聯想到它。
2. **限制性解法 vs 探索性解法**：面對誤判問題時，PM 傾向「禁止行為」而非「找正確工具」。後者需主動擴大搜尋範圍。
3. **原則建立當下就要擴充檢查清單**：`feedback_exhaust_indirect_before_impossible.md` 的四問在建立的同一 session 中就有反例（ToolSearch），但未即時擴充。原則性 feedback 必須同時更新所有相關檢查清單。
4. **規則 6 落地驗證**：此案屬「框架修改優先於專案進度」，一次改善（抽象 ToolSearch、補充 search-tools-guide）惠及所有後續派發與工具發現場景。
