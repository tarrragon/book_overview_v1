### (B) 錯誤訊息雙軌衝突

#### 已驗證的 4 個關鍵案例

**案例 1：append-log section 衝突 [最嚴重]**
- 文件要求：context-bundle-spec.md 明確 
- CLI 實測：VALID_SECTIONS = ["Problem Analysis", "Solution", "Test Results", "Execution Log"]
- 結果：Context Bundle 未列舉，append-log 拒絕，訊息「無效的 section」
- 原因：架構邊界不清（執行日誌 vs 派發準備）

**案例 2：add-spawned 不接多 ID**
- 文件未說明限制，CLI 實測無 nargs=「*」或「+」
- 違反 Unix 慣例（rm a b c、ln -s a b c dest/）
- 必須逐一呼叫，高摩擦

**案例 3：create --source-ticket 缺席 [PC-073 根因]**
- 文件：ANA 衍生 IMP 應用 spawned_tickets
- CLI：無 --source-ticket 參數，PM 被迫用 --parent 繞路
- 結果：children 誤設、complete 被 hook 擋住

**案例 4：錯誤訊息格式雙軌**
- argparse：英文 POSIX 風格（unrecognized arguments）
- PostToolUse Hook：中文 markdown（查 SKILL.md）
- 問題：時序未定義，用戶不知應讀哪一個

#### 統一方案建議

**方案 1：統一驗證層（推薦）**
- argparse 只負責語法，業務驗證由 execute_* 統一執行
- 所有錯誤用 format_error() 中文訊息 + 修復建議

**方案 2：SKILL 文件同步機制**
- VALID_SECTIONS 單一來源（constants 或 track_acceptance），禁止手寫範例
- 防止 PC-007 (command-guidance-implementation-mismatch) 重演

**優先度**：
- **P0**：append-log section 加 "Context Bundle"（5 行代碼）
- **P1**：add-spawned nargs="+"（2 行代碼）
- **P2**：create --source-ticket 參數（中等代碼 + 文件）

### (C) 語意歧義熱點

#### 熱點 1：parent vs source-ticket [PC-073 根本病因]

**現狀衝突**：
- parent_id + children[]：父子，children 未完成父無法 complete（硬性阻塞）
- source_ticket + spawned_tickets[]：衍生，無阻塞，獨立排程
- 但 --source-ticket 參數不存在，PM 誤用 --parent 導致 PC-073 問題

**驗證案例（W12-002）**：PM 用  建 4 個 IMP，期望獨立執行，實際被 hook 擋住

**重命名建議**：
- parent_id → blocking_parent（強調阻塞語意）
- children → atomic_children（強調必須同時交付）
- source_ticket → spawned_from（單向明確）
- spawned_tickets → derived_tickets

#### 熱點 2：children vs spawned_tickets

同上，根本修復需補 --source-ticket 參數

#### 熱點 3：section 枚舉邊界 [落差 1]

Context Bundle 邏輯上應為一級 section，但被歸為「Execution Log 的子標題」

**方案 A（推薦）**：VALID_SECTIONS 改為 6 項，加入 "Context Bundle"
**方案 B**：重構 section 為 lifecycle phases（成本高）

#### 熱點 4：add-* 命名不一致

add-acceptance（追加驗收條件）vs add-spawned（新增關聯），模式相反

**折衷方案**：支援新舊命名 alias，漸進廢棄舊名
