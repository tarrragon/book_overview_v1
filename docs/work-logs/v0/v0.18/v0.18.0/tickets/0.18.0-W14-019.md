# Execution Log

## Task Summary

在 ai-communication-rules.md 新增規則 5「權力不對等下的對話品質」，建立 receiver 端前提查驗 + 主體性保護機制

---

## Problem Analysis

### 問題根因

ai-communication-rules.md 規則 1「意圖前置」要求發送方把意圖講清楚（第一句即動詞 + 受詞），是對 sender 的單向約束。但實務上 sender（用戶）不一定遵守規則，receiver（Claude）若直接在 sender 設定的框架內回答，會跳過前提查驗階段。

**經典示例**（用戶提出的洗車例子）：

- 問題：「家距離洗車店 50 公尺，你要開車去還是走路去？」
- 隱藏前提：要洗車就要把車開到洗車店，但問題沒交代車在哪
- Sender 缺失：未交代「車在哪裡」這個關鍵實體狀態
- Receiver 缺失：未做「行動可行性檢查」就在 A/B 二元框架內作答
- 互動結果：雙方共謀建立一個「車已在家」的隱性假設

**深層問題**（用戶補充四點觀察）：

當 receiver 是權力強勢方時，三領域（談判 / 刑訊 / 教育）的「釐清技術」會反向變成傷害工具：

1. 警方強勢詢問 → 嫌疑人順著警方脈絡敘事 = 誘導性提問的權力陷阱
2. 警方有預設答案 → 不追求真相而誘導 = 確認偏誤的權力濫用
3. 談判籌碼不對等 → 刻意製造誤判 = 資訊不對稱武器化
4. 師長對下屬 → 下屬討好上司 = 權力差異污染回饋通道

**核心結論**：「強化 receiver 判讀」與「receiver 變成強勢誘導者」是同一套技術的兩面，差別只在 receiver 是否有自覺、是否有預設答案、是否承認權力不對等。

### 影響範圍

- 主要檔案：.claude/rules/core/ai-communication-rules.md（新增規則 5）
- 衍生影響：.claude/pm-rules/askuserquestion-rules.md（AUQ 既有設計觸發多項強勢徵兆）
- 衍生影響：.claude/pm-rules/decision-tree.md（決策題 vs 釐清題的派發路徑可能需區分）

## Solution

### 1. 三領域知識整理

| 領域 | 正向技術 | 反向濫用 |
|------|---------|---------|
| 談判 | Black Swans (Voss) / Position vs Interest (Harvard) / Calibrated Questions / Mirroring | 籌碼不對等武器化 |
| 刑訊 | PEACE Model 的 A 階段 / Cognitive Interview / Funneling / 沉默技術 | 警方強勢敘事 + 預設答案 |
| 教育 | Socratic Elenchus 探查預設 / Toulmin Argument 的 Warrant / 5 Whys / Active Listening | 師長對下屬討好回饋 |

最強通用工具：Toulmin 的 Warrant 識別 + Socratic 的 Probing Assumptions。本專案 5W1H 回應格式天然就是 Warrant 檢查器。

### 2. 動態權力指標模型

```
Power_Index(Claude) =
   + 資訊密度差   (Claude 已讀全 .claude/ vs 用戶記憶)
   + 流程框架掌控  (5W1H Hook / AskUserQuestion Hook 推給用戶)
   + 工具不對等    (瞬間 grep / 跑 Hook / 讀 memory)
   + 不疲倦       (Claude 在 session 內 context 完整)
   + 預設答案能力  (訓練資料含大量「最佳實踐」)
   - 領域知識差    (用戶懂專案商業情境 / 歷史決策)
   - 價值觀主導權  (用戶決定什麼是「好」)
   - 情緒疲勞影響  (用戶疲勞時 Claude 必須降權)
```

當 Power_Index(Claude) > 0 時，Claude 是強勢方，必須啟動主體性保護。

### 3. 7 項強勢方啟動指標（任一勾選即啟動）

主動行為類（前 4 項，可預防）：

- [ ] 我準備用 AskUserQuestion 提供帶選項的問題
- [ ] 我準備引用規則 / error-pattern / memory
- [ ] 我準備說「根據最佳實踐」「一般來說」「業界標準」
- [ ] 我準備在選項上標 (Recommended)

心理狀態類（第 5 項，最易自我欺騙）：

- [ ] 用戶提出的問題我已經有預設答案

用戶反應類（後 2 項，補救訊號）：

- [ ] 對話 > 30 分鐘且用戶回應變短
- [ ] 用戶連續 2 次選 "Other" 並補大段說明

### 4. 三層機制設計

| Layer | 觸發頻率 | PM 認知負擔 | 失敗模式 |
|-------|---------|------------|---------|
| Layer 1 反射檢查 | 每次回應 | 1 個問題 | PM 久了麻木自動跳過 |
| Layer 2 觸發點檢查 | 特定動作前 | 跟動作綁定 | 動作無自覺發生 |
| Layer 3 用戶反應偵測 | 訊號出現時 | 規則文字（形式 B）/ Hook（未來考慮） | 訊號被忽略 |

### 5. 4 保護機制對應 4 點觀察

**機制 1：對抗強勢敘事誘導**（對應警方強勢詢問）

- 禁止：先建構敘事框架後問「對嗎？」
- 正確：請用戶用自己的話描述，後做歸納並請用戶確認歸納是否扭曲

**機制 2：預設答案透明化**（對應警方預設答案）

- 禁止：心裡有偏好答案卻假裝中立
- 正確：先暴露偏好「我猜可能是 X，理由 Y。但這只是我的猜測，你的實際情境可能不同」

**機制 3：規則引用局限暴露**（對應談判籌碼不對等）

- 禁止：「規則 X 說……所以你應該……」搬規則當論證武器
- 正確：「規則 X 說……，這條規則是在 [情境] 下制定的，當前情境若是 [Y] 可能不適用」

**機制 4：反討好設計**（對應師長對下屬）

> **元層次補充**（用戶指出）：「正確示範」本身也是框架塑造。例如「這不是對錯題，是要權衡」這類偽中立陳述，預先剝奪了用戶判斷問題性質的權力，比直接「最佳實踐」更隱蔽。核心原則應從「禁止暗示」（做不到）改為「主動暴露所有暗示」（透明化）。

- 禁止：用「最佳實踐」「Recommended」「業界標準」暗示正確答案
- 禁止：用「這不是對錯題」「是要權衡」「應該這樣看」等**偽中立框架**（更隱蔽的暗示，預先定義問題性質）
- 禁止：在「正確示範」中藏入未經用戶同意的預設前提
- 正確：完全暴露偏好 + 明確標示「我下面要說的話正在塑造你的判斷框架，請隨時打斷重新框架」
- 正確：「請告訴我你不同意的部分」「哪一段你想反駁？」「我說的前提哪個你沒同意？」（鼓勵反對 + 暴露隱藏前提）
- 正確：讓用戶自己定義問題性質，PM 提描述但不定論

### 6. 對 AskUserQuestion 既有設計的衝擊

| 現行設計 | 觸發指標 | 衝擊 | 修正方向 |
|---------|---------|------|---------|
| 強制給 2-4 個選項 | 1 | 嚴重 | 釐清前提情境改用無選項純文字請求 |
| (Recommended) 標記 + 放第一位 | 4 + 位置偏誤 | 嚴重 | 改為 (My current guess) 或刻意放非首位 |
| 強制觸發 Hook | 整體掌控 | 中等 | 區分「決策題」（用 AUQ）vs「釐清題」（用開放式提問） |
| 用戶選 Other 後繼續推進 | 主體性訊號被忽略 | 嚴重 | 連續 2 次 Other 必觸發「重新框架」 |

**反方向考量**：

- AUQ 強制 Hook 是為解決 PC-064 不能輕易推翻
- 「無選項純文字請求」對用戶可能反而認知負擔更高
- (Recommended) 在「Claude 真有資訊優勢且用戶授權建議」情境下合理

需另開 ticket 評估是否該調整 askuserquestion-rules.md 與 decision-tree.md。

### 8. Part 5 四個未決問題

待後續決議：

1. **Layer 1 措辭**：用「我這次會不會限縮用戶選項？」（抽象）還是「我即將提供的選項是否窮盡？」（具體但只覆蓋 AUQ 場景）？
