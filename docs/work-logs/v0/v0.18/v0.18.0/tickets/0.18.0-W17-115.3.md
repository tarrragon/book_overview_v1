### 關鍵設計要求（從 W17-115 ANA Solution 摘要）

**觸發機制**：PostToolUse Bash matcher，git commit 命令成功完成。

**偵測流程**：
1. `git show --name-only HEAD` 取本次 commit 的 file list
2. 路徑過濾：檢查是否含 `.claude/skills/ticket/ticket_system/**` 改動
3. commit msg 詞元過濾：含 `feat` / `refactor` 詞元觸發；`fix` / `test` / `docs` / `chore` 不觸發
4. 滿足條件時輸出 INFO 級別提示

**提示內容**：
- 列出本次 commit 的 ticket skill src 改動檔案
- 待掃描清單：`.claude/skills/ticket/SKILL.md`、`.claude/pm-rules/decision-tree.md`、`.claude/pm-rules/*.md`
- 提示語：「若本次改動屬行為變更，請執行 grep 確認決策層引用無過時」

**meta 防護**：
- 路徑白名單豁免：本 hook 自身路徑（`.claude/hooks/ticket-skill-sync-check-hook.py`）— 避免 PC-099 自我引用誤判
- ticket 級豁免：commit 範圍含 ticket skill src **且**同 commit 已含 `.claude/pm-rules/` 或 `.claude/skills/ticket/SKILL.md` 改動 → 視為已同步，不提示
- 訊息級別：INFO，**不阻擋** commit

**PC-098 防護**：hook 程式碼內容不引用 W17-XXX 等 ticket ID 作動機/銜接（規則檔 v1.1.0 已示範通用陳述）

### 實作摘要

依 W17-115 ANA Solution「C 路徑」落地 commit-level sync-check hook。

#### 新增檔案

- `.claude/hooks/ticket-skill-sync-check-hook.py`（~250 行）
  - PostToolUse Bash matcher
  - 偵測流程：tool_name 驗證 → git commit 命令過濾 → commit 成功標記 → commit type 過濾（feat/refactor 觸發）→ git show --name-only HEAD 取 file list → meta 自我引用豁免 → ticket skill src 路徑過濾 → 同步豁免（pm-rules/ 或 SKILL.md）→ 輸出 INFO 提示
  - 訊息級別 INFO，不阻擋 commit
- `.claude/hooks/tests/test_ticket_skill_sync_check_hook.py`（23 test cases）
  - 純函式單元測試 14 條（command 判別 / commit type 提取 / 路徑過濾）
  - 主流程整合測試 8 條（含 ANA Solution 指定的 5 個情境 + 補充 3 個邊界）

#### 修改檔案

- `.claude/settings.json`：在 PostToolUse Bash matcher hooks 陣列末尾追加 ticket-skill-sync-check-hook 註冊（timeout 5000ms）

#### 設計決策

- **commit-level vs Write-level**：採 commit-level（PostToolUse Bash + git show HEAD），commit 是行為變更明確邊界，避免 Write 階段中間態誤報
- **meta 防護兩層**：路徑白名單豁免（hook 自身改動）+ ticket 級豁免（同 commit 含 pm-rules/ 或 SKILL.md）
- **PC-098 防護**：hook 程式碼內僅以通用陳述描述設計理由，不嵌入 W17-XXX ticket ID（PC-099 自我引用 + PC-098 跨層引用雙合規）
- **訊息形式**：純 INFO additionalContext，不阻擋 commit；提供建議掃描清單與 grep 指令，PM 自行判斷後續動作

#### 不在本 ticket 範圍

- 規則 / PC 文件撰寫（W17-115.1 / W17-115.4 各自處理）
- worktree 隔離（ARCH-015 限制，主 repo 執行）
