## 重現實驗結果

### 實驗方法

本 ticket 為分類/判別框架設計（非程式碼 bug），「重現實驗」轉為「場景掃描 + 分類可行性檢驗」：

1. 掃描 askuserquestion-rules.md 17 個強制使用場景
2. 對每個場景試分類為「決策題 / 釐清題 / 混合題」
3. 評估分類判別標準的可操作性（能否從情境快速判斷，不依賴 PM 主觀）

### 實驗執行

**步驟 1：17 場景題型試分類**

掃描 .claude/pm-rules/askuserquestion-rules.md 場景列表，對每個場景試分類：

| # | 場景 | 答案空間 | 試分類 | 判別依據 |
|---|------|---------|--------|---------|
| 1 | 驗收方式確認 | 有限選項（各驗收方式） | 決策題 | 選項窮盡、PM 執行路徑分歧 |
| 2 | Complete 後下一步 | 有限選項（next ticket / release / pause） | 決策題 | 選項窮盡 |
| 3 | Wave/任務收尾確認 | 有限選項（情境 C1/C2） | 決策題 | 選項窮盡 |
| 4 | 方案選擇 | 有限選項（多方案並列） | 決策題 | 核心決策題 |

| 7 | 派發方式選擇 | 有限選項（subagent/Teams/序列） | 決策題 | 選項窮盡 |

| 13 | 後續任務路由 | 有限選項 | 決策題 | 選項窮盡 |
| 14 | parallel-evaluation 觸發確認 | 二元 | 決策題 | PC-053 防護 |
| 15 | Bulk 變更前備份 | 二元 | 決策題 | 安全防護 |
| 16 | 錯誤學習經驗確認 | 二元 | 決策題 | 錯誤學習雙通道 |
| 17 | 錯誤經驗改進追蹤 | 有限選項 | 決策題 | 選項窮盡 |

**結論**：17 個強制場景中，**16 個為決策題**（答案空間有限、選項可窮盡），**1 個為混合題**（#6 任務拆分）。**沒有純釐清題場景**。

**步驟 2：釐清題實際情境盤點**

既有 17 場景未涵蓋「純釐清題」。嘗試列舉 PM 實際可能遇到的釐清題：

| 情境 | 答案空間 | 為何不適合 AUQ 選項 |
|------|---------|-------------------|
| 「新 feature 的具體需求是什麼？」 | 開放文字 | 無法枚舉 |
| 「上次 session 卡在哪？」 | 開放描述 | 用戶心智模型未知 |
| 「你對這個規則有什麼想補充？」 | 開放建議 | 答案未定形 |
| 「這個名稱你覺得貼切嗎？不貼切你會怎麼改？」 | 開放 + 建議 | 釐清+設計混合 |
| 「你認為 Claude 太強勢嗎？」（規則 5 §5.11 監測） | 主觀感受 | 強迫選項會誘導答案 |

**關鍵發現**：這些情境目前**不在 AUQ 強制使用場景**內——因為它們多半發生在「用戶主動找 PM 討論」而非「PM 找用戶決策」。

**步驟 3：判別標準可操作性檢驗**

候選判別標準：

| 標準 | 可操作性 | 風險 |
|------|---------|------|
| **選項窮盡性**（能列 ≤5 個覆蓋 90%+ 答案？）| 高 | 明確二分，不易誤用 |
| **答案維度**（選擇 vs 描述）| 中 | PM 可能主觀判定「這是描述」以跳過 AUQ |
| **因果後果**（答案是否分歧 PM 執行路徑？）| 高 | 客觀可判（路徑分歧就是決策題）|
| **用戶認知負擔**（選 vs 描述哪個省力？）| 低 | 太主觀，易被濫用 |

**已驗證事實**：
1. 現行 17 個強制場景幾乎全是決策題（16/17），§5.9「釐清題 illusion of control」問題**不出現在既有強制場景**
2. 釐清題實際發生在「用戶→PM」方向的對話（用戶主動詢問/討論），而非「PM→用戶」的決策點
3. 判別標準以「選項窮盡性 + 因果後果」最可操作（選項能列且分歧路徑 → 決策題）

**未驗證假設**：
1. §5.9 描述的「AUQ illusion of control」是否真的在既有 17 場景發生（需實證）
2. 若將「釐清題」納入強制規範，是否會增加 PC-014/PC-064 濫用風險（需實測）
3. Hook 系統能否自動識別釐清題（技術可行性未評估）

**結論**：**§5.9 描述的釐清題問題與既有 17 場景的重疊度極低**。PM 用 AUQ 的 17 個強制場景幾乎全為決策題，釐清題問題實際發生在「未納入強制場景的用戶主動對話」。這改變了 019.4 的分析方向。
### 問題根因

W14-019 §5.9 衝擊表揭示「AUQ 強制 2-4 選項」對**釐清題**（clarification questions）構成 illusion of control——當答案空間本質上是開放文字描述時，強制選項會:

1. 隱性框架用戶答案（用戶傾向從選項中挑，而非描述真實需求）
2. Recommended 位置偏誤進一步放大（用戶在 context 沉重時選首位 Recommended）
3. Other 選項雖存在但使用率低（實證：PC-066 發現 Recommended 設計不當時用戶傾向順從）

但同時存在對立防護：
- **PC-064**：PM 不得列純文字選項（以防無意識疏失繞過 AUQ）
- **PC-014**：合理化跳過 AUQ

核心矛盾：**若允許 PM 自主判別「釐清題」而跳過 AUQ 選項，PC-064 防護可能被濫用（以「這是釐清題」為由省略 AUQ 工具）**

## 結論：採用方案 G（分層組合）

### Phase A（立即，與 W14-023 同期或接續）

**修改 1：askuserquestion-rules.md 新增「題型判別輔助」章節**

位置：規則 6 之後、場景列表之前（約 line 148）

內容結構：
```
## 題型判別輔助（非強制，教育性）

PM 用 AUQ 前的自檢：這是哪種題型？

| 題型 | 判別標準 | 處理方式 |
|------|---------|---------|
| 決策題 | 答案空間可列 ≤5 個選項覆蓋 90%+、不同答案導致 PM 執行路徑分歧 | 強制 AUQ + 規則 1-6 |
| 釐清題 | 答案為開放文字描述、無法窮舉選項 | 開放式提問（PC-064 邊界條款適用） |
| 混合題 | 決策 + 補充描述組合 | AUQ 決策層 + description 或 notes 欄位補充 |
```

**修改 2：askuserquestion-rules.md 新增「PC-064 適用邊界」章節**

位置：規則 3（禁止列純文字選項）之下，作為子章節

內容：
```
### PC-064 適用邊界（釐清題不適用）

PC-064 核心禁令（禁止列純文字選項）適用於「PM→用戶決策提問」場景：
- 有明確選項空間可窮舉
- 不同答案導致 PM 執行路徑分歧

**不適用**：釐清題（答案為開放文字描述），如：
- 「新 feature 的具體需求是什麼？」
- 「上次 session 卡在哪？」
- 「你對這個規則有什麼想補充？」

釐清題可用開放式提問（不列選項），但 PM 必須主動自檢：
1. 這真的沒有可窮舉的選項空間嗎？
2. 不同答案會導致 PM 執行路徑分歧嗎？

兩者皆否才可判定為釐清題。濫用此邊界（PM 把決策題偽裝成釐清題以跳過 AUQ）仍屬 PC-014 違規。
```

### Phase B（019.6 或新 ticket：混合題機制）

針對 #6 任務拆分（現行唯一混合題場景），設計「AUQ 決策層 + description/notes 開放補充」機制：
- 決策層用 AUQ 列選項（拆 / 不拆）
- 補充層讓用戶在 notes 欄位或後續對話補充「怎麼拆」
- 避免單 AUQ 強迫用戶在「不夠資訊」下做決策

### Phase C（019.5 監測指標 3 個月後）

依實證決定：
- 釐清題強制場景是否需要新增（方案 B）
- PC-064 邊界條款是否被濫用
- Hook 是否需要自動識別題型

### 否決方向

| 否決項目 | 理由 |
