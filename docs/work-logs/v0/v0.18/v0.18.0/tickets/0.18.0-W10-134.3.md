## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### ginger 效能架構視角 verdict

### 1. 量化基線（實測 corpus）

| 項目 | 數量 | 總 bytes | 平均 tokens（÷4 chars） |
|------|------|---------|----------------------|
| ticket md | 1289 | 10.0 MB | ~2.5M tokens 全集 |
| PC pattern md | 265 | 1.12 MB | ~280K tokens |
| rules/ 自動載入 | - | 117 KB | ~29K tokens |
| pm-rules/ 47 檔（按需） | 47 | 353 KB | ~88K tokens |
| 單 ticket 中位數 | - | ~7.7 KB | ~1.9K tokens（123 行） |

PM 預算 200K，自動載入 rules/ + CLAUDE.md ≈ 30K（15%）。

### 2. 雙提案 token / 延遲對比

| 維度 | 提案 A: Ticket/PC RAG | 提案 B: Rules RAG |
|------|---------------------|-------------------|
| 成本性質 | **變動成本**（每次 retrieve 增量） | **固定成本**（每 session 啟動節省） |
| 單次 retrieve 注入 | 5 筆 × 1.9K = **9.5K tokens / 次**（4.75% 預算） | 命中規則 ~3-5K tokens |
| 觸發頻次 | claim/dispatch 高頻（每 ticket 1-3 次） | session 啟動 1 次 |
| 累積 / session | 高頻 PM 100+ 動作 → 數十次 retrieve = 50K-200K **新增**負擔 | 30K → 8K = **節省 22K**（11% 預算） |
| 命中率敏感度 | 漏 retrieve = 少參考（容忍） | 漏 retrieve = PM 違規（致命） |
| 延遲 | embedding query ~50-200ms + ranking | session 啟動一次性 200-500ms |
| 對 PM 主流程衝擊 | **負衝擊**（額外 IO + token 膨脹） | **正衝擊**（減 context 壓力） |

**關鍵不對稱**：A 是「花 token 換可能的決策品質」，B 是「省 token 換確定性風險」。A 的成本可量化（高），B 的收益可量化（高）但風險難量化（規則漏載 = 隱性違規）。

### 3. Verdict

| 提案 | Verdict | 理由 |
|------|---------|------|
| A: Ticket/PC RAG | **不實施完整 RAG，改輕量 hook-driven keyword** | 增量 50-200K tokens / session，實際 PM 高頻動作會把 200K 預算吃掉 30%+；命中漏失成本低，不值得引入 vector infra 維護曲線 |
| B: Rules RAG | **不實施 pure RAG，改 L3 hook-driven 情境載入** | 節省 22K 是真收益但 95% 命中率對 embedding 是高難度；hook 條件式載入（如「派發代理人 → 載入 agent-dispatch-template」）可達 100% 命中且零延遲 |

### 4. 架構簡化方向（≥ 2）

**方向 1: Metadata + TF-IDF 替代 vector**
- ticket frontmatter 已有 type/wave/priority/spawned/blockedBy 結構化欄位
- PC 已有 ID 命名規則（PC-XXX-keyword）
- 用 SQLite FTS5 + frontmatter index 即可達 80% retrieval 需求，零 embedding 成本，查詢 < 10ms
- 增量成本：建索引 ~5 分鐘 / 全量；維護 = git hook 觸發 reindex

**方向 2: Hook-driven 情境載入（L3）取代 Rules RAG**
- 既有 hook 系統已實現部分（charset-guard / phase4-guard 等）
- 擴展為「動作 → 規則 subset」映射表（YAML 維護）
- 範例：`ticket claim` → 載入 ticket-lifecycle.md；`Agent dispatch` → 載入 agent-dispatch-template.md
- 100% 命中率、零 token 浪費、無 embedding infra
- 唯一成本：trigger map 的人工維護（< 50 條 entries）

**方向 3（補充）: Lazy summary cache**
- 對 ticket / PC 預生成 100-token summary 存 frontmatter
- retrieve 時只注入 summary（5 × 100 = 500 tokens vs 9.5K，**節省 19x**）
- 全文按需 Read（PM 已有 Read 工具）
- 兼容方向 1 的 FTS5 結構

### 5. 熱路徑分析

| PM 動作 | 頻率（每 ticket） | 是否該觸發 retrieval |
|---------|---------------|------------------|
| ticket create | 1 | ❌ 規格未定，retrieval 噪音大 |
| ticket claim | 1-2 | ⚠️ PCB 自動抽取已覆蓋大部分需求 |
| Agent dispatch prompt 撰寫 | 1-3 | ✅ 真正甜蜜點（歷史相似 prompt 參考） |
| complete / handoff | 1 | ❌ 結論已成形 |
| 遇 bug / 錯誤 | 變動 | ✅ PC pattern 查詢（但 grep "PC-" + keyword 已夠） |

**熱路徑只有 2 個**（dispatch + bug）。Dispatch 用方向 3（summary cache）即可；bug 場景用既有 grep 已覆蓋（PC ID 命名本來就為 grep 設計）。

### 6. 優先序建議

| 優先 | 動作 | 理由 |
|------|------|------|
| **P0** | 方向 2: Hook-driven Rules 情境載入（L3） | 固定成本優化收益最大（22K / session × N sessions），零延遲零風險；既有 hook 基礎可復用 |
| **P1** | 方向 3: Ticket/PC summary cache | 19x token 節省，兼容 P0；可與 frontmatter migration 合併 |
| **P2** | 方向 1: SQLite FTS5 metadata index | 取代任何 RAG infra 念頭；建索引成本低 |
| **不做** | Vector RAG（提案 A 完整版 / 提案 B pure RAG） | A 加重 PM token 負擔（與初衷相反）；B 命中率不足 95% |

**核心判斷**：固定成本優化（B 改 L3）優先於變動成本優化（A 改 summary cache），因為前者每次 session 都收益且風險可控；後者只在熱路徑收益且引入維護曲線。

### 7. Tripwire 設計

| 訊號 | 閾值 | 行動 |
|------|------|------|
| L3 hook 載入規則漏失導致 PM 違規 | 30 天內 ≥ 3 次 | 該規則改回 auto-load |
| Summary cache miss rate | > 40% | 暫停推 P1 |
| FTS5 查詢延遲 | p95 > 50ms | 檢查索引 / 降回 grep |

### 8. dissenting view 預留

若 linux / saffron 視角主張「既有 grep 已足夠連 P0 都不需做」，本視角同意降級——L3 hook trigger 本質就是「結構化的 grep 替代」，方向上一致；只在「systematic 統一管理」價值上有分歧。
