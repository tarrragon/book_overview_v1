### 問題根因

phase4-decision-enforcement-hook 的 `collect_exempt_markers` / `scan_lines_for_phrases` 對 ticket body 逐行掃描，未識別 `<!-- Schema[...]: ... -->` placeholder template 區塊邊界。當 ticket body 內含 schema placeholder 區塊（如 ticket-body-schema.md 提供的範本），其中作為 marker 格式說明的範例字串 `<!-- PC-093-exempt: cat:reason -->` 會被當作實際 exempt marker 解析，且 `cat` 不在白名單 → 觸發 `[INVALID: category-whitelist]` warning 噪音。

### 影響範圍

- `.claude/hooks/phase4-decision-enforcement-hook.py`：`collect_exempt_markers` + `scan_lines_for_phrases`
- 任何 IMP/ANA/DOC ticket body 含 Schema placeholder 區塊（範例文字）且 marker 範例為 cat:reason / <category>:<reason>
- 噪音層級：WARN（不阻擋 complete），但污染 hook 輸出可達性，與 W10-108 改善方向背道

### 修復摘要

引入 `compute_schema_placeholder_lines(lines)` helper 計算 Schema placeholder template 區塊的行號集合：

- **起點**：`SCHEMA_PLACEHOLDER_START = r"<!--\s*Schema\[[^\]]+\]\s*:"` 匹配 `<!-- Schema[IMP/Problem Analysis]: ... -->` 等標記
- **終點**：下一個 H2（`^\s*##\s`）或 `---` 水平分隔符（`^\s*---\s*$`），不含邊界行本身

兩處掃描函式（`scan_lines_for_phrases` 與 `collect_exempt_markers`）於迴圈開頭跳過 placeholder 行：

```python
if idx in placeholder_lines:
    continue
```

達成：placeholder 區塊內的範例字串既不被當作延後話術（phrase）也不被當作實際 exempt marker（避免 `[INVALID: category-whitelist]` 噪音）。

### 設計決策

| 設計點 | 決策 | 理由 |
|-------|------|------|
| 區塊邊界 | 下個 `## ` H2 或 `---` 分隔符 | ticket body schema 採此兩種分節符；optional fields 後通常緊跟 `---` |
| 邊界行歸屬 | 不屬 placeholder（不跳過） | 邊界後即為使用者實際內容 |
| 跳過粒度 | 整段（phrase + marker 雙跳） | placeholder 區塊內任何延後話術都屬範例（如 schema note 內示意「Phase 5 再決定」） |
| 與既有 EXEMPT_MARKER_STRIP 關係 | 互補 | EXEMPT_MARKER_STRIP 處理「同行 marker 內含 phrase」；本機制處理「整個 placeholder 區塊」 |
