### 判斷原則（父 ticket 共通指引）

**拆分依據是「認知負擔」與「按需載入」，不是行數。**

這是卡片盒筆記法（Zettelkasten）的核心精神：

- 正確問題：這段知識在什麼情境才需要理解？每段知識是否原子化？按需載入時能否獨立理解？
- 錯誤問題：檔案多少行？骨架是否夠短？行數超過某閾值就要拆？

**推論**：
- 500 行的檔案若內容全部原子化且在同一情境共同需要 → 不應拆分
- 100 行的檔案若混合多個情境才用到的知識 → 即使短也應拆
- 拆分後的每一碎片必須能獨立被理解（原子性），否則只是把完整概念切碎
- 保留判斷標準不是「這段很重要」，而是「所有使用場景都需要立即載入」

**WRAP 分析時的具體提問**：

1. **W (擴增選項)**：列出所有可能的原子化切分方式（不是以行數為切點，而是以「情境匹配度」為切點）
2. **R (現實測試)**：每個切分方案下，各個碎片是否能獨立被理解？保留在 auto-load 的碎片是否「所有場景都需立即載入」？
3. **A (保持距離)**：從 sync 到其他專案的角度看，這些知識是否真的每個專案啟動都需要？
4. **P (準備出錯)**：若拆分後發現某個碎片其實常被需要，如何快速恢復到 auto-load？

## WRAP 分析 — language-constraints.md (99 行) 拆分可行性

### 檔案結構盤點（99 行）

| 章節 | 行數區間 | 內容類型 | 載入需求 |
|------|---------|---------|---------|
| 開頭 + 強制規則導引 | L1-L8 | 規則宣告 | 所有場景必須 |
| 規則 1 繁體中文 | L9-L18 | 硬性約束（10 行） | 所有輸出必須 |
| 規則 2 禁用詞彙表 | L20-L35 | 對照表（10 條詞彙） | 所有輸出必須 |
| 規則 3 禁止 Emoji | L37-L55 | 7 情境對照 + 理由說明 | 所有輸出必須 |
| 規則 4 技術術語保留原文 | L57-L67 | 5 類對照表 | 所有輸出必須 |

**原子性判斷**：L9-L67（四條規則，58 行）是**緊密相關、同時需要的單一概念單元**——每次輸出文字都必須同時通過這四關，無法按情境拆分。L71-L88（檢查機制章節）是**元資訊**，不影響即時決策，且內容與實際 Hook 能力**不一致**（見下節）。

### 關鍵事實驗證（Reality Test 前置）

檢視 `.claude/hooks/` 實際 Hook 能力：

| 宣稱的 Hook 防護（規則原文） | 實際 Hook 行為 | 落差 |
|------------------------|-------------|------|
| 「PostToolUse (Write) 檢查新建檔案是否包含禁用詞彙」 | **無此 Hook**。僅 `worklog-format-check.py` 檢表格內特定 emoji（CLI crash 防護） | 規則內的檢查機制章節是**錯誤陳述** |
| 「UserPromptSubmit 提醒使用繁體中文回應」 | `language-guard-hook.py` 僅偵測韓文/日文假名（Unicode AC00-D7AF / 3040-30FF），輸出警告到 stderr | 只覆蓋「輸出切成韓日文」這個極端邊界，**不檢簡體字、不檢禁用詞彙、不檢 emoji** |

**推論**：Hook 防護極不完整。規則內文字是主要守門員；若移到 references 按需讀取，代理人可能不會主動 Read 該檔，導致規則 1-4 實質失效。

### R (Reality Test) — 現實測試

**逐碎片「所有場景都需立即載入？」檢驗**：

| 碎片 | 所有場景需立即載入？ | 理由 |
|------|-----------------|------|
| 規則 1（繁中）| 是 | 每一次輸出第一個字就要決定語言 |
| 規則 2（禁用詞彙）| 是 | 每一個中文詞都要過濾（智能/文檔/數據等高頻詞） |
| 規則 3（禁 emoji）| 是 | 所有 markdown 和對話輸出都要避免 emoji |
| 規則 4（技術術語保留）| 是 | 判斷哪些英文名詞不能中譯時要用 |
| 檢查機制章節（L71-L88）| **否** | 僅在「想理解違規後會發生什麼」時才相關，非即時決策依據；且內容與實際 Hook 不符（誤導） |
| 參考資源（L92-L99）| 否 | 指向 terminology-dictionary.md，該字典本身就是按需查詢 |

**原子性判斷**：規則 1-4 是一個不可切分的單元（任何中文輸出都是「同時過四關」）。檢查機制章節是**不同概念層**（元資訊/制度設計），可切分。

**錯誤陳述風險**：保留「檢查機制」段落會讓代理人誤以為有 PostToolUse 禁用詞彙 Hook，做出錯誤期待（例如「反正 Hook 會擋，我輸出簡體字也沒差」）。

### A (Attain Distance) — 保持距離

**跨專案 sync 視角**：

- `.claude/` 會 sync 到其他專案（sync-push/sync-pull skill）。其他專案可能完全不使用本專案的 Hook 系統，或 Hook 版本不同步。
- 若規則倚賴「Hook 會擋」的前提，sync 到 Hook 闕如的專案時規則會實質失效。
- 規則 1-4 的文字內容是**跨專案唯一可靠的守門員**，必須保留 auto-load。
- 檢查機制章節描述特定 Hook 實作，跨專案 sync 會產生誤導（如 `language-constraints.md` 規則 8「框架文件禁引用專案層級識別符」精神的近似違反——雖非 ticket ID，但將本專案特定 Hook 作為「檢查機制」陳述，本身就是一種耦合）。
- **從此視角看，反而應該刪除檢查機制章節（而非搬 references），讓規則保持 Hook 無關的純原則**。

**Hook 能力強化趨勢評估**：Hook 目前只偵測韓日文（最低門檻）。擴展到簡體字/禁用詞彙需要維護詞彙表、處理偽陽性（技術術語含簡字變體）、對話 transcript 解析等，工程成本高且副作用大。未來 Hook 不太可能完全取代規則文字。

### P (Prepare to be Wrong) — 準備出錯

**失敗情境 1**：刪除「檢查機制」章節後，有人誤以為規則沒有自動防護，不敢違規通報 → 回退：在 CLAUDE.md 或 pm-role.md 簡述「Hook 部分覆蓋（韓日文偵測），規則文字為主要守門員，違規請通報」。

**失敗情境 2**：拆分後某代理人不載入 references 導致違規飆升 → 回退：將該 references 章節合併回主檔（git revert）。由於元資訊章節本身不影響即時決策，這個風險極低。

**失敗情境 3**：Hook 實際能力後來擴增（W10-XXX 升級 language-guard-hook 支援簡體字/禁用詞偵測），但規則已刪除檢查機制章節 → 回退：在 references 建立新的 `language-hook-coverage.md` 記錄當前 Hook 能力（由 Hook 維護者同步更新），不再放在規則檔。
