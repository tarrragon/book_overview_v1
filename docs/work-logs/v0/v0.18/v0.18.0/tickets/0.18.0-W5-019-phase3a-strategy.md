## 1. 實作策略設計

### 1.1 總體方針

採「整合測試驅動 + 後置副作用注入」策略：

- 在 `complete()` 既有流程尾端（所有驗證通過、父 ticket save 完成後）注入 cascade 副作用區塊
- Cascade 邏輯抽為模組級 private helper `_cascade_unblock_children`，職責單一
- 輸出格式化與狀態變更解耦：helper 只回傳「結果資料」，由呼叫端決定列印順序
- save 失敗採 try-except 侷限在 save 呼叫，記錄 warning 後繼續下一個 child（non-fail-fast）

### 1.2 RED → GREEN → REFACTOR 節奏

**階段 1（RED+GREEN 主幹）**：
1. 先寫 fixture（`make_parent_child_tickets` + `complete_env`）— 交付無此則所有測試無法啟動
2. 寫 TC-A1（最簡單的正向 cascade）→ 紅燈
3. 在 `complete()` 第 487 行（`_auto_handoff_if_needed` 之前）inline 寫一個最小可通過版本（不抽 helper）→ TC-A1 綠燈
4. 寫 TC-A2（警告但不阻止）→ 紅燈
5. 擴充 inline 邏輯加入 warning 列表收集與格式化輸出 → TC-A2 綠燈
6. 寫 TC-A3（其他依賴保留）→ 紅燈
7. 擴充 blockedBy AND 語義判斷 → TC-A3 綠燈

**階段 2（RED+GREEN 邊界）**：
8. 批次寫 TC-B1~B6（不涉及 save 失敗）→ 視 fail 狀況補 guard
9. 寫 TC-B7（save 失敗）→ 加 try-except 圍 save_ticket
10. 寫 TC-B8 parametrize（AND 語義完整覆蓋）→ 驗證既有邏輯
11. 寫 TC-C1/C2 輸出格式 → 視需要微調字串

**階段 3（REFACTOR）**：
12. 所有 13 測試綠燈後，將 inline 邏輯抽為 `_cascade_unblock_children(parent, version) -> (unblocked, pending)`
13. 再抽兩個 print helper：`_print_cascade_unblocked(unblocked)` 與 `_print_children_warnings(pending)`
14. 重跑測試套件確認全綠

### 2.2 _cascade_unblock_children helper

```
function _cascade_unblock_children(parent_ticket, version):
    """
    對 parent.children 中 blocked 且可解鎖者執行 status=pending 寫入，
    並收集所有未完成 children 的狀態供警告輸出。

    回傳: (unblocked_list, pending_list)
      unblocked_list: [{id, title}, ...]  — 已解鎖（blocked → pending）
      pending_list:   [{id, title, status}, ...]  — 未完成且未解鎖
                      （含 pending / in_progress / blocked-with-other-deps）
    """
    unblocked = []
    pending   = []

    children_ids = parent_ticket.get("children", [])
    if 空: return ([], [])

    all_tickets = list_tickets(version)
    ticket_map  = {t.id: t for t in all_tickets}

    for child_id in children_ids:
        child = ticket_map.get(child_id)
        if child is None:
            continue                              # §6.5 找不到就跳過

        child_status = child.get("status", PENDING)

        if child_status in (COMPLETED, CLOSED):
            continue                              # §6.4 / §6.6 不處理已完成

        if child_status == BLOCKED:
            if _can_cascade_unblock(child, ticket_map):
                # --- 嘗試解鎖 ---
                child["status"] = PENDING
                try:
                    save_ticket(child, resolve_ticket_path(child, version, child.id))
                    unblocked.append({"id": child.id, "title": child.title})
                except Exception as err:
                    print format_warning(f"cascade 解鎖 {child.id} 儲存失敗：{err}")
                    # §6.7 non-fail-fast，此 child 不計入 unblocked，也不列入 pending
                    continue
            else:
                # blocked 但還有其他未完成依賴 → 列入警告（保留 blocked）
                pending.append({"id": child.id, "title": child.title, "status": BLOCKED})
        else:
            # pending / in_progress → 警告但不改動
            pending.append({"id": child.id, "title": child.title, "status": child_status})

    return (unblocked, pending)


function _can_cascade_unblock(child, ticket_map):
    """依 §6.8 AND 語義：child.blockedBy 中所有 ticket 皆 completed/closed 才可解鎖"""
    blocked_by = child.get("blockedBy", [])
    if blocked_by 空:
        return True                               # §6.8 B8d 異常狀態仍解鎖
    for blocker_id in blocked_by:
        blocker = ticket_map.get(blocker_id)
        if blocker is None:
            return False                          # 找不到 blocker 時保守保留 blocked
        if blocker.get("status") not in (COMPLETED, CLOSED):
            return False
    return True
```

### 2.3 輸出 helper

```
function _print_cascade_unblocked(unblocked):
    print ""
    print "[Cascade] 以下子 Ticket 已自動解鎖（blocked → pending）："
    for item in unblocked:
        print f"   - {item.id}: {item.title}"

function _print_children_warnings(pending):
    print ""
    print "[Warning] 父 Ticket 完成時尚有未完成的子 Ticket："
    for item in pending:
        print f"   - {item.id} [{item.status}]: {item.title}"
    print "   （提示：父 complete 不阻止，但建議確認子任務是否需要獨立推進或 close）"
```

## 3. 資料結構選擇

| 名稱 | 型態 | 選擇理由 |
|------|------|---------|
| `unblocked_list` | `List[Dict[str, str]]` with keys `id`, `title` | 不需要完整 ticket dict，只需輸出欄位；降低呼叫端認知負擔 |
| `pending_list` | `List[Dict[str, str]]` with keys `id`, `title`, `status` | 多一個 status 欄位供格式化使用 |
| `ticket_map` | `Dict[str, dict]` | `_analyze_next_steps` 慣例一致，供 blockedBy 反查 |
| helper 回傳 | `Tuple[unblocked, pending]` | 明確區分兩類產出，呼叫端分別處理輸出 |

**不使用 dataclass / namedtuple 理由**：專案其他 helper（`_check_pending_children`、`_check_unblocked_tickets`）皆採 dict 風格，維持一致性。

## 4. 資料流程圖

```
complete(ticket_id)
    │
    ├─[既有驗證] status / acceptance / execution_log
    │
    ├─[既有狀態寫入] parent.status=completed → save_ticket(parent)
    │
    ├─[既有輸出] INFO_COMPLETED + worklog + 驗收提示 + next-steps
    │
    ├─[新增 cascade 區塊]────────────────────┐
    │                                       │
    │   parent.children 非空？              │
    │       │                               │
    │       └─→ _cascade_unblock_children(parent, version)
    │               │
    │               ├─ list_tickets(version) → ticket_map
    │               │
    │               └─ for each child_id in parent.children:
    │                       │
    │                       ├─ child 不存在 → skip
    │                       ├─ status ∈ {completed, closed} → skip
    │                       ├─ status == blocked 且 _can_cascade_unblock:
    │                       │     → child.status = pending
    │                       │     → try save_ticket
    │                       │         success → unblocked.append
    │                       │         fail    → print warning, skip
    │                       ├─ status == blocked 但不可解鎖 → pending.append(blocked)
    │                       └─ status ∈ {pending, in_progress} → pending.append
    │
    ├─ unblocked 非空 → _print_cascade_unblocked(unblocked)
    ├─ pending   非空 → _print_children_warnings(pending)
    │
    └─[既有] _auto_handoff_if_needed(...) → return 0
```

## 5. 控制流程圖（cascade 決策樹）

```
每個 child_id：
    │
    ticket_map.get(child_id)
    │
    ├─ None ────────────────────────────────→ skip（§6.5）
    │
    └─ child：
         │
         child.status
         │
         ├─ completed/closed ────────────────→ skip（§6.4/§6.6）
         │
         ├─ pending/in_progress ─────────────→ pending_list.append
         │
         └─ blocked：
              │
              _can_cascade_unblock(child)
              │
              ├─ False ─────────────────────→ pending_list.append(status=blocked)
              │
              └─ True：
                   │
                   child.status = pending
                   save_ticket(child)
                   │
                   ├─ success ──────────────→ unblocked_list.append
                   │
                   └─ exception ────────────→ print warning（含 child_id, err）, skip
```

## 7. 權宜方案與技術債務

### 7.1 最小可行實作

本次實作只覆蓋 **complete → 直接 children 一層 cascade**。以下刻意不處理：

- 孫子不遞迴（§6.1 by design）
- parent_id 反查（§6.2 單一真實來源）
- close 命令的 cascade（另開 Ticket）

### 7.2 已知限制

| 限制 | 說明 | 影響 |
|------|------|------|
| save 失敗的 warning 訊息格式未在規格嚴格定義 | 測試採寬鬆比對（含 child_id 即可） | 若未來需機器解析需補格式常數 |
| children 清單大（>50）未截斷 | 規格 §9 有提未實作 | 極端案例輸出過長，Phase 4 可加 |
| `_can_cascade_unblock` 遇到找不到 blocker 時保守保留 blocked | 若資料不一致可能永遠 blocked | 由人工 release 或修資料 |
| cascade 載入全版本 tickets 兩次（complete 自身 + helper） | `_analyze_next_steps` 已載入 all_tickets，helper 再載入一次 | 效能浪費，Phase 4 可傳 ticket_map 注入 |

### 7.3 技術債務標記（Phase 4 建議）

| 項目 | 建議行動 |
|------|---------|
| all_tickets 重複載入 | 將 `_analyze_next_steps` 載入的 all_tickets 傳遞給 `_cascade_unblock_children`，避免重覆 I/O |
| warning 訊息常數化 | 將 `[Cascade]` / `[Warning]` 區塊字串提取至 `messages.py` 統一管理 |
| closed 常數化 | 若 `STATUS_CLOSED` 未集中定義，本實作順手補上 |
| save 失敗記錄格式 | Phase 4 討論是否統一為 `format_warning(...)` 風格 |

## 8. 語言特定實作注意事項（Python / Phase 3b）

### 8.1 Python 慣例

- **Type hint 必須**：`def _cascade_unblock_children(parent_ticket: Dict[str, Any], version: str) -> Tuple[List[Dict[str, Any]], List[Dict[str, Any]]]:`
- **Docstring 必須**：參考 `_check_unblocked_tickets` 的 Google style 風格
- **import 增補**：`Tuple` 若尚未 import，加入 `from typing import ...`
- **log module**：本 helper 不另引日誌；失敗訊息直接 `print(format_warning(...))`，與既有 complete 風格一致

### 8.2 Mock 可測性要求（測試依賴）

Phase 2 測試 mock 以下符號於 `ticket_system.commands.lifecycle` 模組空間：
- `load_ticket`, `save_ticket`, `list_tickets`
- `validate_completable_status`, `validate_acceptance_criteria`
- `append_worklog_progress`, `_auto_handoff_if_needed`

**實作要求**：`_cascade_unblock_children` 必須透過模組級 `list_tickets` 與 `save_ticket`（已 imported），**禁止**用完整路徑 `ticket_system.repositories.save_ticket(...)` 或重新 import，否則 patch 失效。

### 8.3 錯誤處理範圍

```python
# 正確：只包 save 呼叫
child["status"] = STATUS_PENDING
try:
    save_ticket(child, resolve_ticket_path(child, self.version, child_id))
    unblocked.append({"id": child_id, "title": child.get("title", "")})
except Exception as err:
    print(format_warning(f"cascade 解鎖 {child_id} 儲存失敗：{err}"))
    continue

# 錯誤：包住整個分支，難以定位失敗原因
try:
    child["status"] = STATUS_PENDING
    save_ticket(...)
    unblocked.append(...)
except Exception:
    ...
```

### 8.4 父 ticket dict 可變性

`execute_complete` 呼叫 helper 時，parent_ticket 已經 save 完成。helper 內部載入的 child dict 若透過 `list_tickets` 拿到，**與** `ticket_map` 是不同物件（或同物件，視實作）。**重要**：修改 child.status 後呼叫 save_ticket 即可；不須回頭更新 ticket_map。
