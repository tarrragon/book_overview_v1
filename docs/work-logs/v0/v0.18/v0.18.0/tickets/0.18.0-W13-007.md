### 用戶關鍵觀察

1. 異常**只在 AskUserQuestion**，正常對話與文件輸出無此問題
2. 涵蓋三類：簡體字、日文漢字、emoji
3. 不只是 token 偶發，為穩定重現的模式
4. 生成式 AI 應有能力在生成權重加入輔助防止偏差，需分析為何 AUQ 專屬

### WRAP 四問（下 session 待執行）

**W (Widen)** — 候選假設（至少 4 個）：
- H1：JSON escape 層（\uXXXX）讓模型 token 選擇過程相異於對話 Markdown
- H2：AUQ field length constraint（label ≤12 字）觸發「相近字替代」策略
- H3：AUQ markdown rendering context 讓模型以為需要「視覺區分」用漢字變體
- H4：AUQ payload 的 JSON structure 讓 attention distribution 偏離繁體優先權重
- H5：AUQ 訓練資料偏差（若存在）導致 payload 生成時多源語言混入

**R (Reality Test)** — 重現實驗設計：
- 實驗 A：在 AUQ 產 10 個 payload 記錄漢字污染次數
- 實驗 B：在相同語意對話層產 10 個回覆對比污染次數
- 實驗 C：AUQ JSON 手動寫入繁體 U+XXXX 是否被模型改寫
- 實驗 D：讓模型解釋 AUQ payload 生成時的 token 選擇依據

**A (Attain distance)** — 機會成本分析：
- 清單擴充（W13-003 模式）vs 通用 CJK 繁/簡/日分辨器 vs 生成層提示工程調整
- 不同方案的工程成本與防護覆蓋率

**P (Prepare to be wrong)** — 最可能失敗路徑：
- 清單式防護無法窮盡（本 Ticket 已知）
- 通用 CJK 偵測可能誤判（如部首、合法雙字形）
- 生成層調整需模型 prompt-engineering 深入研究
