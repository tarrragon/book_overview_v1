## Problem Analysis

### 問題根因

W17-139 整合的 hook 用 `claude mcp list` 探測 zhtw-mcp 註冊狀態。`claude mcp list` 內部對所有 user 註冊的 MCP server（Gmail / Google Drive / Linear / Greptile / Calendar）做健康檢查，平均耗時 8.9 秒。每個 session 啟動，hook 透過 spawn claude CLI 隱性對所有遠端 MCP server 發 HTTP 請求。核心衝突：探測機制範圍（list 全部 server + 全部 health check）遠大於探測目的（單一 server 是否註冊），對應 ARCH-022 根因 1。

### 影響範圍

- `.claude/hooks/zhtw-mcp-availability-check-hook.py` 函式 `_check_mcp_registered`
- 跨專案 framework sync 後每個專案 session 啟動都付此成本
- 用戶不知情的隱性網路流量

### 相關 Error Pattern

ARCH-022（hook-cli-probe-with-cross-boundary-side-effects，本 ticket 同時建立）。核心原則：framework hook 不應對 user-scope 設定產生隱性副作用。

### 修復摘要

| 項目 | 變更前 | 變更後 |
|------|-------|-------|
| 註冊探測 | `subprocess.run(["claude", "mcp", "list"])` | 直讀三層 scope JSON 檔 |
| Binary 版本探測 | `subprocess.run(["zhtw-mcp", "--version"])` | 移除（zhtw-mcp 0.1.0 不支援，等上游 release） |
| `import subprocess` | 是 | 移除 |
| 平均探測耗時 | 8.9 秒 | 0.48 毫秒（提升 ~18000 倍（純 Python 邏輯量測，端到端 hook 啟動含 uv overhead 另計）） |

### 三層 Scope 探測順序

1. user scope: `~/.claude.json` 的 `mcpServers`
2. project scope: `<project_dir>/.mcp.json` 的 `mcpServers`
3. local scope: `<project_dir>/.claude/settings.local.json` 的 `mcpServers`

任一 scope 含 `"zhtw-mcp"` 鍵即視為已註冊。三狀態返回值：

| 情境 | _scope_has_zhtw_mcp | _check_mcp_registered |
|------|---------------------|----------------------|
| 任一 scope 命中 | True | True |
| 全部檔案不存在或無 zhtw-mcp 鍵 | False | False |
| 任一檔案存在但 JSON 解析失敗 | None | None（probe inconclusive） |

### 設計決策

- **Project 路徑來源**：優先讀 `CLAUDE_PROJECT_DIR` 環境變數，無時 fallback 至 `Path(__file__).parent.parent.parent`（hook 位置推算）。
- **JSON 解析失敗的處理**：返回 None 而非 False，避免將「設定檔損毀」與「未註冊」混淆；hook 訊息標示 "scope probe inconclusive"。
- **不再 fallback claude CLI**：依 ARCH-022，CLI fallback 反而引回原本的副作用。讀檔失敗即承認不確定（返回 None），hook 訊息標 "scope probe inconclusive"，由用戶看到後自行檢查設定檔；避免靜默回退到有副作用的探測。

### 不在本 ticket 範圍

- Binary `--version` 探測機制：依用戶決策，等上游 zhtw-mcp 發布 release 後再用標準路徑（追蹤 ticket: W17-140）。
