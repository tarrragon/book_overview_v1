### 問題根因

Phase 4 多視角評估發現 `_resolve_path_classification` 三項可改善點（linux + bay）：

1. **L2/L3 分支內 `_merge_ticket_scope` 重複呼叫**（linux #2 Garbage）：L2 與 L3 互斥分支各自呼叫 `_merge_ticket_scope(ticket_ids)`，雖實際只執行一次，但讀者需追蹤兩處呼叫站才能確認語意，且若 L2 後續改為 fall-through 將立即出現 N+1 I/O。
2. **L1/L2/L3 互斥隱晦**（linux #1/#6）：elif 結構表達互斥但缺乏顯式註解，讀者需腦補狀態機。
3. **L3 假設邊界缺警示**（linux #6）：「純 .claude/ ticket → 將 has_other 降為 False」隱含「ticket where.files 為 source of truth」假設，若 where.files 漏填會誤豁免，需註解警示。
4. **`_merge_ticket_scope` 內 `except Exception` 缺規則 4 雙通道**（bay #1）：靜默降級 project_root_str 為空字串，未說明靜默原因。
