# Execution Log

## Task Summary

評估 CC 2.0.72 的 Claude in Chrome Beta 工具能否用於本專案 Readmoo Chrome Extension 的 E2E 測試：能否載入未發布 unpacked extension、能否操作 extension popup/content script、能否驗證書庫資料提取流程、與現有 Puppeteer E2E 測試比較優劣

---

## Problem Analysis

Schema 補正：本 ANA 的問題分析已對照三類工具完成。現有正式 E2E 需求是可重跑、可斷言、能載入 build/development unpacked extension 並驗證 popup/content script/service worker。Claude in Chrome 偏人工輔助驗證；Chrome DevTools MCP 是官方 Chrome DevTools 團隊提供的 MCP/CLI 候選，具備成為新 E2E 方法的可能性，但需要獨立 POC 驗證 extension 載入、popup 操作、console/network 讀取與 pass/fail 輸出。
### 問題根因

本專案是 Manifest V3 Chrome Extension，正式 E2E 需求包含：

- 載入本機 `build/development` unpacked extension。
- 操作 `chrome-extension://<extension-id>/...` popup / overview 頁。
- 在 Readmoo 或模擬 Readmoo 頁面注入 content script 並驗證書籍提取流程。
- 在 CI 或本機測試中可重跑、可斷言、可產出穩定失敗訊號。

Claude in Chrome 的官方定位是 Claude Code 與 Chrome extension 連接後，讓 Claude 在可見 Chrome 視窗中測試 web apps、讀 console / DOM、點擊、填表、抽取網頁資料。它強調互動式 build-test-verify、除錯與人工協作，不是 headless 測試 runner，也沒有官方文件承諾能載入另一個本機 unpacked extension 作為受測目標。

補充調查後確認：Chrome 官方另有 **Chrome DevTools MCP**。這不是 Claude in Chrome，而是 Chrome DevTools 團隊提供的 MCP / CLI 工具，能讓支援 MCP 的 agent 控制與檢查 live Chrome browser。它更接近「新的 E2E 測試方法」候選，因為它走 DevTools 能力與 Puppeteer 自動化，且提供 headless、isolated profile、連接既有 Chrome、傳入 Chrome args 等配置。

### 影響範圍

- `package.json`：現有 E2E 指令以 Jest + Puppeteer 執行。
- `jest.e2e.config.js`：E2E 使用 Node environment。
- `tests/e2e/setup/extension-setup.js`：使用 Puppeteer `--disable-extensions-except` / `--load-extension` 載入 `build/development`。
- `tests/helpers/e2e-test-suite.js`：大量流程使用 mock Readmoo page、mock extension state 與可程式化斷言。
- `tests/e2e/**`：現有正式 E2E / validation / deployment / performance 測試。

### 相關 Error Pattern

未新增 error-pattern。這次是工具可行性評估，沒有發現新的可重用錯誤模式；既有教訓適用於「不能把 beta 工具能力當成正式測試保證」與「任務描述中的工具名稱需用官方文件重新驗證」。本次修正了初版分析只比較 Claude in Chrome、漏掉 Chrome DevTools MCP 的問題。

---

## Solution

Schema 補正：結論更新為「Claude in Chrome 不替代正式 E2E；Chrome DevTools MCP 進入新 E2E 方法 POC」。後續 POC 應測 npx chrome-devtools-mcp@latest 搭配 --headless、--isolated、--chromeArg=--load-extension=<build/development>，以及 --browserUrl 連接手動啟動的 debug Chrome。成功標準是可非人工重跑最小 UC-01 smoke flow，並回傳可機械判定的 pass/fail。
後續 POC ticket 已建立：0.18.0-W6-007（實測 Chrome DevTools MCP 作為 Chrome Extension E2E 新路線）。此 ticket 追蹤 chrome-devtools-mcp 載入 unpacked extension、操作 popup/mock Readmoo page、讀 console/network/snapshot、產出 pass/fail 的實測結果。
### 評估結論

**技術可行性：部分可，但需分成兩個工具判斷。**

Claude in Chrome 可用於人工輔助 QA、debug 與探索性驗證；不建議取代現有 Puppeteer E2E，也不建議納入 CI 的正式驗收鏈。

Chrome DevTools MCP 是不同工具，具備成為新 E2E 測試方法的可能性；但本 ticket 尚未實機驗證 `chrome-devtools-mcp` 能否穩定載入本專案 unpacked extension、操作 popup/content script、產出可重跑 assertion。因此建議建立獨立 POC ticket，不在本 ticket 直接替換現有 Puppeteer E2E。

### Claude in Chrome 官方能力與限制

官方文件確認：

- Claude Code 可用 `claude --chrome` 或 `/chrome` 連接 Chrome。
- Chrome integration 會開啟可見 Chrome 視窗，能讀 DOM / console、點擊、填表、抽取資料、錄製 GIF。
- 會共享使用者已登入的 browser state；遇到 login / CAPTCHA 會暫停要求人工處理。
- 目前支援 Google Chrome / Microsoft Edge，不支援 Brave、Arc 等其他 Chromium browser，也不支援 WSL。
- 官方目前要求 Claude in Chrome extension 1.0.36+、Claude Code 2.0.73+ 與 direct Anthropic paid plan。

任務描述中的 CC 2.0.72 是功能加入版本，但以目前官方文件為準，實際使用門檻是 Claude Code 2.0.73+。

### Chrome DevTools MCP 官方能力

官方 Chrome for Developers 文件確認：

- Chrome DevTools MCP 是 public preview，讓 AI coding assistant 直接用 Chrome DevTools debug web pages。
- 可用於 real-time verification、console / network error diagnosis、user behavior simulation、layout debugging、performance tracing。
- 支援 MCP server 與 CLI；官方安裝範例包含 Claude Code 與 Codex。
- `chrome-devtools-mcp` 可控制與檢查 live Chrome browser，並使用 Puppeteer 自動化 action。
- 主要配置包含 `--headless`、`--isolated`、`--browserUrl` / `--wsEndpoint`、`--autoConnect`、`--executablePath`、`--chromeArg`。
- 可連接既有 Chrome session，也可啟動隔離 profile；這比 Claude in Chrome 更接近可腳本化測試基礎。

### 本專案現況

現有 E2E 已具備 Puppeteer-based extension runner：

- `npm run test:e2e` 使用 `jest --config jest.e2e.config.js tests/e2e`。
- `tests/e2e/setup/extension-setup.js` 以 headed Chrome 載入 `build/development`：
  - `--disable-extensions-except=<extensionPath>`
  - `--load-extension=<extensionPath>`
- popup 以 `chrome-extension://${extensionId}/src/popup/popup.html` 開啟。
- Readmoo flow 目前多數用 `tests/e2e/fixtures/readmoo-mock-page.html` 或 `E2ETestSuite` mock state 驗證。

### 與現有 E2E 對照

| 評估面 | Claude in Chrome Beta | Chrome DevTools MCP | 既有 Puppeteer E2E | 判斷 |
|---|---|---|---|---|
| 載入 unpacked extension | 官方未承諾 | 可能可透過 `--chromeArg=--load-extension=...` 或連接已啟動的 debug Chrome 實測 | 已用 `--load-extension` / `--disable-extensions-except` 明確載入 | MCP 需 POC |
| 操作 extension popup | 可能可點擊可見 UI，但 extension internal page / toolbar popup 未承諾 | 可控制 Chrome tab；是否能穩定開 `chrome-extension://<id>/...` 需實測 | 已能以 `chrome-extension://<id>/...` 開頁並斷言 | MCP 需 POC |
| content script / service worker 驗證 | 可讀 DOM / console，偏人工 debug | DevTools 能力更強，可讀 console/network/snapshot；service worker lifecycle 仍需實測 | 已用 target / worker / mock state 管理 | Puppeteer 保留，MCP 評估 |
| Readmoo 登入狀態 | 可共享使用者 browser login | 可連接既有 Chrome session 或 isolated profile；真站登入流程可人工/自動混合 | CI 不依賴真帳號，主要用 mock page / fixture | MCP 可補足真站 smoke |
| 可重跑與 CI | Beta、可見視窗、人工介入，CI 不穩 | 有 headless / isolated / CLI 潛力，但 agent-driven assertion 穩定性需設計 | 可由 npm script 重跑，失敗訊號可機械判定 | MCP 可能成新路線 |
| 除錯體驗 | 可讀 console、DOM、network request | 官方 DevTools 能力，含 performance trace / screenshot / snapshot | 需在測試腳本內補 logging / trace | MCP 優勢明顯 |
| 維護成本 | 少寫腳本，但需人監督 | 可能用 prompt / CLI / MCP tools 寫較少測試膠水，但需標準化 assertion | 需維護 fixtures 與 script，但結果可自動化 | 先 POC 再決策 |

### 採用建議

不採用 Claude in Chrome 作為正式 E2E 替代方案。

採用 Chrome DevTools MCP 作為「候選新 E2E 方法」進入 POC，但現階段不替換 Puppeteer。

可在以下場景「選用」：

- 本地開發後的人工 smoke test。
- Readmoo 真站登入後，快速檢查 DOM 變更、console error、extension UI 是否仍可操作。
- 需要錄製 demo GIF 或把瀏覽器觀察結果轉成 bug report。
- Puppeteer E2E 失敗但錯誤訊息不足時，用 Chrome integration 或 Chrome DevTools MCP 做現場 debug。

不建議用於：

- CI gate。
- release blocking test。
- 需要固定 assertion、穩定重跑、無人工介入的驗收。
- 未經 POC 直接取代 `tests/e2e/setup/extension-setup.js` 的 extension 載入流程。

### 後續 Ticket

建議建立後續 POC ticket：

- 類型：ANA 或 IMP/POC
- 目標：實測 Chrome DevTools MCP 是否能載入 `build/development` unpacked extension、開啟 popup、操作 mock Readmoo page、讀 console/network、產出可機械判定的 pass/fail。
- 必測路線：
  - `npx chrome-devtools-mcp@latest --headless --isolated --chromeArg=--load-extension=<build/development>`
  - 連接手動啟動的 Chrome：`--browserUrl=http://127.0.0.1:9222`
  - 如 Chrome 版本支援，測 `--autoConnect`
- 成功標準：能用非人工步驟重跑一個最小 UC-01 smoke flow，且失敗時回傳可供 CI 或 ticket 記錄使用的明確錯誤。
