### 問題根因

**根因分類：(a) 原始投訴錯誤歸因 + W17-176.2.1 已修真因**

原始投訴「stop hook 每 turn fire 提醒 W17-174」實為**錯誤歸因**。每 turn fire 的是 `handoff-prompt-reminder-hook.py`（UserPromptSubmit 事件），非 stop hook。

Stop hook (`handoff-auto-resume-stop-hook.py`) 行為正確：
1. 使用 `is_handoff_stale()` 正確判定 W17-174 JSON 為 stale（`from_status=completed` + `is_ticket_completed()` via `find_ticket_file`）
2. GC 邏輯正常刪除 stale JSON（L553 `file_path.unlink()`）
3. 有 `has_been_triggered_this_session` 防重複，每 session 最多觸發一次
4. Stop 事件本身每 session 只在退出時觸發，不可能「每 turn fire」

真正每 turn fire 的是 prompt reminder hook，其 `is_ticket_completed` 路徑解析 bug（不支援三層階層）導致 W17-174 completed 狀態無法被偵測，**已由 W17-176.2.1（commit fdc3ee3e）修復**。

### 影響範圍

- `.claude/hooks/handoff-prompt-reminder-hook.py` — 已修復
- `.claude/handoff/pending/0.18.0-W17-174.json` — 殘留 stale JSON，下次 stop hook 執行（本 session flag 過期後）會自動 GC

### 相關 Error Pattern

- ARCH-020（跨進程同步遺漏）：W17-165 修 stop hook 用 `find_ticket_file`，但未同步 prompt reminder hook
- W17-176.2 正確識別此模式，W17-176.2.1 已修復

### 驗證實驗

1. `is_handoff_stale(W17-174 JSON)` → `(True, '來源 ticket 0.18.0-W17-174 已 completed')`
2. stop hook `is_ticket_completed` 使用 `find_ticket_file`（L379-380），支援三層階層
3. stop hook 有 `has_been_triggered_this_session` guard（L685），每 session 只觸發一次
4. prompt reminder hook commit `fdc3ee3e` 已改用 `find_ticket_file`（L107, L139）
