### 結論：legacy str template 不淘汰（三視角共識）

W17-008.5.6.2 三視角審查（linux + bay + thyme）對 format_error 雙路徑的共識：

| 視角 | 結論 |
|------|------|
| linux | 「雙路徑勝出 vs envelope-only 立即淘汰」——立即淘汰需要一次改 30+ callsite，且 INVALID_SECTION 等含「有效值清單」的訊息不適合套 envelope 四元組（hint 會塞整列表，扭曲 schema 用途）|
| bay | 「legacy str 雙路徑長期維護」屬 P3 風險而非技術債——「雙路徑非技術債而是職責分離（legacy 適合附帶清單的複合訊息）」|
| thyme | functools.singledispatch 兩分支場景過度設計，現行 isinstance 已是 Pythonic |

故結論：legacy str 路徑保留作為「複合訊息」場景的設計選擇，不規劃淘汰時程。

### legacy 適用場景盤點

當前仍走 legacy str 路徑的 callsite 性質分類：

| 場景類型 | 範例 | 適合保留 legacy 原因 |
|---------|------|--------------------|
| 含「有效值清單」的訊息 | INVALID_SECTION（列出有效 section）、INVALID_SECTION（列出 valid sections）| envelope 四元組 hint 欄位塞整列表會扭曲 schema 用途 |
| 純複合訊息 | SECTION_NOT_FOUND（顯示現有 ## 標題列舉）、TICKET_NOT_FOUND（無分類需求）| 業務分類粒度低，errno + hint 結構過重 |
| 系統錯誤回顯 | argparse 純語法錯誤 POSIX 路徑（W17-008.5.4 設計界線）| 沿用 POSIX 慣例已是黃金標準 |

當前走 envelope 路徑的場景（適用 envelope 條件）：

- 業務驗證（CHECKLIST_VALIDATION_FAILED / WHY_REQUIRED 等含分類意義 errno）
- argparse 業務錯誤（INVALID_CHOICE / INVALID_VALUE，配 ArgparseFormatErrorParser）
- 跨 hook ↔ CLI 邊界（envelope marker 給 hook 偵測）

### 使用指引（已由 W17-008.5.7 PC-105 文件補強落實）

W17-008.5.7（commit `44899080`）已在 `.claude/skills/ticket/SKILL.md` + `references/track-command.md` 補上：

- format_error 雙路徑說明
- ErrorEnvelope 結構與適用時機
- ArgparseFormatErrorParser 業務 vs 語法錯誤分流
- hook envelope 偵測跳過行為

新 callsite 撰寫時可從這些文件 grep 命中使用範例，依場景選擇 legacy str 或 envelope。

### 為何不需獨立 trigger ticket

原 acceptance 預設「legacy 為技術債需淘汰」，三視角共識否定此前提。後續處理路徑：

1. **PC-105 文件指引**：已由 W17-008.5.7 落實（commit `44899080`）
2. **新場景判斷**：每個新增錯誤 callsite 應依「複合訊息 vs 結構化分類」判斷選用 legacy 或 envelope（指引在 references/track-command.md）
3. **未來擴充風險處置**（修正 PC-120 違規）：bay 視角識別「callsite 增長後選擇隨意」屬 P3 風險。**永久決策：不建監測 ticket、不規劃定期走查**。理由：(a) 當前 callsite 數量穩定；(b) PC-105 文件已落實使用指引（W17-008.5.7），新 callsite 撰寫者可直接查文件；(c) 若日後出現「選擇隨意」實證案例，重新建獨立 ANA 評估該案例驅動的需求。本決策為終局，不綁 trigger ticket。

故本 DOC ticket 結論為「規劃結果：不淘汰 + 落實使用指引（已透過 W17-008.5.7）」，不需 spawn 後續 IMP ticket。
