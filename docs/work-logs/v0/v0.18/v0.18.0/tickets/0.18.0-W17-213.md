### 變更摘要

依 W17-053 linux 審查 3 項建議改善：

1. **建議 1 DRY helper**：新建 `ticket_system/lib/dispatch_common.py`，抽出 `load_and_unpack(args, version)` 共用前置：取 `ticket_id` → `load_ticket` → 判 None / `_yaml_error` → 解包 `_body` / `where.files` / `acceptance`。回傳 `DispatchLoadResult` dataclass 含 `error_exit_code`（None = 成功，整數 = 呼叫端直接 return）。
   - 評估後決定提前落地（雖只兩個呼叫端）：(a) 兩處 boilerplate ~18 行完全重複；(b) 未來新增第三個 dispatch-* 命令時可直接使用；(c) 抽取成本低（純函式 + dataclass）。
   - `dispatch_validate.py` / `dispatch_readiness.py` 各自的 `execute_*` 改用 `load_and_unpack`，移除重複的 `sys.stderr.write` 與欄位解包。
2. **建議 2 常數重命名**：`_RESPONSIBILITY_SOFT_MAX` → `_RESPONSIBILITY_PASS_MAX`，貼合 `> N 即離開 pass 區` 的閘門語意；comment 補一句說明來源（rules/cognitive-load.md 3b 派發前閾值）。其餘 `_FILES_SOFT_MAX` / `_CB_TOKENS_SOFT_MAX` 保留（本 ticket 範圍僅 acceptance 那項精確化）。
3. **建議 3 近似性 PM 覆核提示**：
   - `references/track-command.md` 在「閾值來源」段落後追加「近似性警告（W17-213）」三段說明：(a) 驗證類條目導致高估、(b) 多職責合併導致低估、(c) WARN/FAIL 時 PM 應手動覆核。
   - `SKILL.md` L96 dispatch-readiness 表格行追加「閾值 1 以 acceptance 條目近似，含驗證類條目時可能高估，PM 於 WARN/FAIL 應手動覆核——詳見 references/track-command.md」。

### 設計決策

- helper 拆獨立 module 而非放 `lib/parser.py`：保持單一職責，未來新增 dispatch-* 命令時 import 路徑清晰。
- 測試 patch 點從 `commands.track_dispatch_*.load_ticket` 改為 `lib.dispatch_common.load_ticket`：對應 helper 內實際 import。
- 補 `tests/test_dispatch_common.py` 6 項測試覆蓋成功 / 失敗（缺 ticket_id / None / yaml_error）/ 預設值 / where 非 dict 等邊界。

### Layer 2 豁免理由

純程式碼品質重構（DRY 抽取 + 常數重命名 + docs 補近似性說明），無新增 framework 規定，亦無對外規範語句變動。SKILL.md 與 track-command.md 補的近似性說明屬既有功能說明補完，非新規則引入。
