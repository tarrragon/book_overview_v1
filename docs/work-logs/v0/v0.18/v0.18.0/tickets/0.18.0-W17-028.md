## 5 盲區修補設計

### 盲區 A：runqueue 讀 Exit Status

**現況**：`_apply_context_resume` (line 106-115) 掃描 `handoff/pending/*.json` 只取 `ticket_id`，不讀 JSON 內其他欄位。`_get_pending_handoff_ticket_ids` (line 79-103) 只 `data.get("ticket_id")` 後丟棄整個 dict。

**修補設計**：

1. 改 `_get_pending_handoff_ticket_ids` 為 `_get_pending_handoff_info`，回傳 `Dict[str, Dict]`（key=ticket_id, value=完整 handoff JSON）
2. `_apply_context_resume` 從回傳 dict 取 `exit_status` 欄位（W17-010 schema: `success|needs_context|blocked|partial_success|failed`）
3. `_render_list` 在每筆 ticket 後標註 exit_status tag，例如 `[needs_context]` / `[blocked]`
4. 新增排序權重：`needs_context` 和 `blocked` 排在正常 ticket 之後（降優先），讓 PM 先看可立即接手的

**API 變更**：
- `_get_pending_handoff_ticket_ids() -> Set[str]` → `_get_pending_handoff_info() -> Dict[str, Dict]`
- `_render_list` 接收額外 `handoff_info: Dict[str, Dict]` 參數
- 無 CLI 介面變更（透明增強）

**回歸範圍**：`track_runqueue.py` 單元測試需新增 exit_status 標註驗證；既有 resume context 測試需適配回傳型別變更。

### 盲區 B：handoff --auto 整合 Context Bundle extractor

**現況**：`_handle_auto_handoff` (line 1090-1176) 手動組裝 `handoff_data` dict，只填 ticket_id/direction/timestamp/from_status/title/what/chain/auto_generated 共 8 欄位。W17-002 的 Context Bundle extractor 未被呼叫。

**修補設計**：

1. 在 `handoff_data` 寫入後、print 成功訊息前，呼叫 Context Bundle extractor CLI（或直接 import extractor 模組）
2. 將 extractor 輸出存入 handoff JSON 的新欄位 `context_bundle`（嵌入式，不另存檔）
3. 失敗時 warning 降級（handoff 本體仍成功），不阻塞 --auto 流程

**API 變更**：
- handoff JSON schema 新增 `context_bundle` 欄位（optional）
- `_handle_auto_handoff` 新增 extractor 呼叫邏輯

**依賴**：W17-002 extractor 已 completed（W17-002.1 + W17-002.2），API 穩定。

**回歸範圍**：handoff --auto 測試需驗證含/不含 context_bundle 兩路徑。

### 盲區 C：runqueue 前置 readiness 標註

**現況**：`_render_list` 輸出欄位為 `priority | ticket-id | title`，無 PCB 完整性資訊。PM 依 runqueue 建議派發後才發現缺 context。

**修補設計**：

1. runqueue 輸出每筆 ticket 時，檢查 ticket YAML 的 Context Bundle 章節是否非空做輕量 readiness 判定

3. 不阻塞排序——readiness 只是標註，不影響排序邏輯

**與盲區 A 的交互**：盲區 A 提供 exit_status 標註，盲區 C 額外加 PCB 完整性。兩者可合併為一個 `[tag]` 欄位。
