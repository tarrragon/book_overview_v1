## Problem Analysis

### 觸發案例：W5-001 Phase 2 派發越界（2026-04-12 16:52）

**事件經過**：
1. PM 於 16:50 派發 sage-test-architect 執行 W5-001 Phase 2 測試設計。
2. Prompt 內容要求 sage 建立 `.claude/skills/ticket/tests/test_ac_parser.py` 和 `test_validation_templates.py`，並執行 pytest 確認 RED 狀態。
3. sage 於 16:52 完成（status=completed, tool_uses=12, duration=187s），實際產出：
   - 2 個 .py 測試檔案（共 30 個 test function）
   - Ticket Test Results section 寫入對應表與 pepper 提示
4. 用戶指出派發粒度設計疑慮；PM 檢討後發現 sage 職責定義為「**禁止實作程式碼和超出職責範圍的工作**」，PM prompt 逾越了代理人定位。

**三類派發問題**：

### 問題 A：代理人職責邊界模糊或被 PM prompt 稀釋
- sage-test-architect 職責描述明示「禁止實作程式碼」，但當 PM prompt 要求寫 .py 時，sage 仍執行。
- 代理人缺少「拒絕違反自身職責的任務」內建行為。
- PM 派發前未對照代理人職責做最終自檢。

### 問題 B：粒度評估缺失
- 14 個測試場景 × 2 個檔案的工作量，接近 subagent ~20 tool call 限制（feedback_agent_task_granularity）。
- 本次 sage 實際用 12 tool calls 險過關，但若測試更複雜或 lavender Solution 細節更多，易觸發回合耗盡。
- PM 派發前缺 "工作量 vs tool call 預算" 的量化檢查。

### 問題 C：PM 派發前缺 forcing checklist
- 現有 pm-rules 提到代理人分工，但無派發前的「最終檢查清單」（如：代理人職責 vs prompt 要求一致？工作量拆分到 <15 個 tool calls？Forcing function 已加入？）
- 類似問題在過往版本出現多次（PC-018、PC-040、PC-042），但皆以個案 feedback 記錄，未系統化為派發前 Hook 或規則。

**影響範圍**：
- 所有 TDD Phase 派發（Phase 1/2/3a/3b/4）
- 所有需要背景代理人並行的場景（大範疇 review、多檔案修改）
- PM 決策樹的「代理人選擇 → prompt 撰寫」步驟

**參考來源**：
- sage-test-architect agent 定義（見 /agents 清單）
- feedback_agent_task_granularity（~20 tool calls workaround）
- feedback_pm_is_ai_needs_complete_rules（PM 需完善規則不能靠經驗）
- quality-baseline-rule-6（框架修改優先）

**用戶裁示**：不回退 sage 產出（工作品質可用），但建 Ticket 追蹤改善，避免後續 W5-002~005 與其他版本重犯。

### 從 W5-001 Phase 1-3b 實戰學到的改善方案

本 session 除派發越界外，同時產出多個**正向案例**驗證 prompt 設計的有效性：pepper Phase 3a 產出品質高、thyme-A/B 並行無衝突、PC-057 防護在後續派發中實際生效（thyme 在職責邊界聲明後未越界）。將實戰經驗轉為改善方案：

---

#### 方案 1：Agent definition description 格式標準化

**問題對應**：根因 A（代理人職責邊界模糊）

**內容**：規範每個 .claude/agents/*.md 的 description 欄位必須含結構化區塊：

```
## 允許產出
- （明列此 agent 可產生的產出類型）

## 禁止行為
- （明列此 agent 不可做的事，包含哪種檔案、哪種操作）

## 適用情境
- （TDD Phase 標註、獨立任務類型）
```

**落地建議**：新建 .claude/rules/agent-definition-standard.md 規範 + 批量審視既有 agent 補充缺失區塊。

**預估 IMP Ticket**：0.18.0-W5-011（規則新增 + 批量 patch）

---

#### 方案 2：PM prompt 模板「職責邊界聲明」骨架固化

**問題對應**：根因 C（PM 派發前缺自檢清單）

**內容**：本 session 派發 pepper / thyme-A / thyme-B 的 prompt 都使用了統一結構：

```
Ticket: {id}

## 職責邊界聲明

{agent-name} 的 agent 定義為「{description 引文}」。

**允許的產出**：{列出}
**禁止的產出**：{列出}

本 prompt 符合職責邊界，請繼續執行。
```

這個開場讓 agent 在執行前先「自我檢核」prompt 是否越界。實戰證明 pepper/thyme 無一越界。

**落地建議**：固化為 .claude/references/agent-dispatch-template.md，pm-rules 引用為派發強制骨架。

**預估 IMP Ticket**：0.18.0-W5-012（模板固化 + pm-rules 引用）

---

#### 方案 3：agent-dispatch-validation Hook 增加關鍵字衝突掃描

**問題對應**：根因 A + C（沒有自動防線）

**內容**：現有 agent-dispatch-validation 只檢查 Ticket ID 格式。增強：

- 讀取目標 agent 的 description 中「禁止行為」區塊（依方案 1 標準化後）
- 掃描 prompt 是否含衝突關鍵字（如 agent 禁止「實作程式碼」而 prompt 要求「建立 .py」→ 警告或 block）
- 預設 warn，嚴重衝突 block（exitcode=2）

**落地建議**：修改 .claude/hooks/agent-dispatch-validation/（Python），新增 conflict_scanner 模組。

**預估 IMP Ticket**：0.18.0-W5-013（Hook 擴充 + 單元測試）

**依賴**：方案 1（需先標準化 agent description 結構）

---

#### 方案 4：TDD Phase 代理人職責清單（pm-rules/tdd-flow.md 擴充）

**問題對應**：根因 C（PM 派發決策時缺參考）

**內容**：tdd-flow.md 加表格：

| Phase | 代理人 | 允許產出 | 禁止產出 |

| 0 | saffron | 系統審查報告 | 程式碼、測試 |
| 1 | lavender | 功能設計文件 | 程式碼、測試檔 |

| 3a | pepper | 實作策略/虛擬碼 | 真實 .py 程式碼 |
| 3b | {lang}-developer | 真實程式碼 + 測試 .py | 修改既有測試契約 |

**落地建議**：pm-rules/tdd-flow.md 新增本表格。

**預估 IMP Ticket**：0.18.0-W5-014（DOC 文件更新）
