## Problem Analysis

### 觸發案例：W5-001 Phase 2 派發越界（2026-04-12 16:52）

**事件經過**：
1. PM 於 16:50 派發 sage-test-architect 執行 W5-001 Phase 2 測試設計。
2. Prompt 內容要求 sage 建立 `.claude/skills/ticket/tests/test_ac_parser.py` 和 `test_validation_templates.py`，並執行 pytest 確認 RED 狀態。
3. sage 於 16:52 完成（status=completed, tool_uses=12, duration=187s），實際產出：
   - 2 個 .py 測試檔案（共 30 個 test function）
   - Ticket Test Results section 寫入對應表與 pepper 提示
4. 用戶指出派發粒度設計疑慮；PM 檢討後發現 sage 職責定義為「**禁止實作程式碼和超出職責範圍的工作**」，PM prompt 逾越了代理人定位。

**三類派發問題**：

### 問題 A：代理人職責邊界模糊或被 PM prompt 稀釋
- sage-test-architect 職責描述明示「禁止實作程式碼」，但當 PM prompt 要求寫 .py 時，sage 仍執行。
- 代理人缺少「拒絕違反自身職責的任務」內建行為。
- PM 派發前未對照代理人職責做最終自檢。

### 問題 B：粒度評估缺失
- 14 個測試場景 × 2 個檔案的工作量，接近 subagent ~20 tool call 限制（feedback_agent_task_granularity）。
- 本次 sage 實際用 12 tool calls 險過關，但若測試更複雜或 lavender Solution 細節更多，易觸發回合耗盡。
- PM 派發前缺 "工作量 vs tool call 預算" 的量化檢查。

### 問題 C：PM 派發前缺 forcing checklist
- 現有 pm-rules 提到代理人分工，但無派發前的「最終檢查清單」（如：代理人職責 vs prompt 要求一致？工作量拆分到 <15 個 tool calls？Forcing function 已加入？）
- 類似問題在過往版本出現多次（PC-018、PC-040、PC-042），但皆以個案 feedback 記錄，未系統化為派發前 Hook 或規則。

**影響範圍**：
- 所有 TDD Phase 派發（Phase 1/2/3a/3b/4）
- 所有需要背景代理人並行的場景（大範疇 review、多檔案修改）
- PM 決策樹的「代理人選擇 → prompt 撰寫」步驟

**參考來源**：
- sage-test-architect agent 定義（見 /agents 清單）
- feedback_agent_task_granularity（~20 tool calls workaround）
- feedback_pm_is_ai_needs_complete_rules（PM 需完善規則不能靠經驗）
- quality-baseline-rule-6（框架修改優先）

**用戶裁示**：不回退 sage 產出（工作品質可用），但建 Ticket 追蹤改善，避免後續 W5-002~005 與其他版本重犯。
