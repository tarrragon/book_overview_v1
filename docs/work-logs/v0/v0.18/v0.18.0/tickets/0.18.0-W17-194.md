### 問題根因

既有 `git-index-lock-cleanup-hook.py` 的 lock 殘留警告訊息僅提示「可能有其他 git 操作進行中」，未列舉外部 GUI app（Fork / GitKraken / SourceTree / VS Code Git）週期性 fork `git status`/`git diff` 子進程的可能來源；PM 排查時被既有規則錨定為「自家 hook/Bash 串接」，遺漏 GUI app 情境，白等 lock 自清。

### 影響範圍

- 唯一顯示 index.lock 警告訊息的 hook：`.claude/hooks/git-index-lock-cleanup-hook.py`（line 86-90，`else` 分支處理 lock 較新仍存在情境）
- 其他三個 hook 僅有 comment 或 filter pattern，不顯示警告，不在本 ticket 修改範圍：
  - `agent-dispatch-logger-hook.py:61`（filter 黑名單）
  - `post-commit-fetch-hook.py:42`（comment）
  - `agent-dispatch-validation-hook.py:141`（comment）

### 相關 Error Pattern

- `.claude/error-patterns/process-compliance/PC-139-git-index-lock-source-misattribution-gui-app-fork.md`（W17-193 落地的 PC，本 IMP 為其防護鏈第二段「強化既有 hook 提示讓 PM 不誤判 lock 來源」）

else:

### 修改摘要

修改 `.claude/hooks/git-index-lock-cleanup-hook.py` 的 `else` 分支（lock 未過期情境）警告訊息，從原本單行「可能有其他 git 操作進行中」擴充為三層來源排查清單：

1. 自家 hook/Bash 串接 git 寫入（指向 `bash-tool-usage-rules.md` 規則三）
2. 外部 GUI app fork 子進程（Fork / GitKraken / SourceTree / VS Code Git）— cross-link PC-139，含 `ps aux | grep -iE "(Fork|GitKraken|SourceTree)\.app|VS.Code.*git"` 偵測指令
3. 確認無自家操作後可執行 `rm .git/index.lock` 安全捷徑

並在 `else` 分支上方加入 PC-139 業務情境註解，說明 GUI app 短暫持有 lock 為何屬合法情境而非自家串接。

### 測試擴增

新增 `.claude/hooks/tests/test_git_index_lock_cleanup_hook.py`（5 test cases）：
- `test_fresh_lock_message_contains_gui_app_names`：訊息含 4 個 GUI app 名稱
- `test_fresh_lock_message_contains_ps_aux_detection`：regex 比對 `ps aux | grep` + GUI app pattern
- `test_fresh_lock_message_contains_pc139_reference`：訊息含 PC-139 識別碼與完整 error-pattern 檔名
- `test_no_lock_returns_none`：無 lock 回傳 None
- `test_stale_lock_auto_removed`：過期 lock 自動清理路徑回歸測試

### 不在本 ticket 範圍
