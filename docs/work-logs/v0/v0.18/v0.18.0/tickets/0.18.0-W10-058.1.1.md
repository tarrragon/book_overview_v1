### 問題根因

`wrap-decision-tripwire-hook.py` 的 S1/S2/S3 訊號偵測機制存在三個結構性缺陷，導致 14 天命中率僅 0.23%（4/1741），且可辨識的 2 個 hit 全為偽陽性：

1. **缺測試環境豁免**：hook 在 pytest 子進程環境執行時仍正常偵測，導致測試 fixture 字串被當作 PM 真實語句觸發 signal（hit 2 案例）。
2. **缺語境判別（純 keyword 比對先天不足）**：「不可能 / 做不到 / 無法」等關鍵字在「PM 主張」（自動駕駛訊號）vs「技術陳述事實」之間語意完全不同，純 keyword 比對無法區分（hit 4 案例）。
3. **observability 不全且事後加入**：W10-101 於 2026-05-06 02:01 才補上 observability logging（matched_keyword + prompt_excerpt）；在此之前的 hit（hit 1 / hit 3）永久無法回溯判定真偽陽性，造成歷史樣本不可分析。

### 影響範圍

- `.claude/hooks/wrap-decision-tripwire-hook.py`：S1-S3 偵測邏輯（純 keyword 比對 + 無 context filter）
- `.claude/config/wrap-triggers.yaml`：訊號設定（缺豁免條件 schema）
- W10-058 父任務範圍判定：本 ANA 結論決定 W10-058 是否擴 S4-S8
- 既有 hook 信任度：偽陽性訊號累積會降低 PM 對 hook 警告的注意力

### AC2 三個結構性缺陷（總結）

| 缺陷 | 證據 | 嚴重度 |
|------|------|--------|
| **缺測試環境豁免** | hit 2 觸發詞來自 pytest fixture，純粹是測試輸入污染 | 高（高頻誤報來源） |
| **缺語境判別** | hit 4「不可能」在技術陳述句中被當作 PM 主張誤判 | 中（純 keyword 比對先天限制） |
| **observability 不全且事後加入** | hit 1/3 永久不可判定，事後補強無法回溯舊樣本 | 中（影響長期評估能力） |

### AC3 改善方案

| 方案 | 描述 | 預估成本 | 預估效益 |
|------|------|---------|---------|
| **F1 pytest 環境豁免** | hook 偵測 `PYTEST_CURRENT_TEST` env var 或 cwd 含 `pytest-of-` 路徑即 skip detection | 低（~ 5 行 code） | 消除測試污染類偽陽性（hit 2 類） |
| **F2 強制 observability** | 移除 W10-101 之前的「無 observability 路徑」；signal hit 必同步寫 matched_keyword + prompt_excerpt（已部分落地） | 已完成（W10-101） | 確保未來 hit 100% 可判定 |
| **F3 context-aware filter (黑名單)** | 觸發詞前後 ±20 字含「Stop 事件 / hook / 訊號 / 偵測 / 不會 / 機制」等技術詞時 suppress | 中（需詞表 + 邊界調整） | 消除語境誤判（hit 4 類） |
| **F4 context-aware filter (語法模式)** | 偵測「X 在 Y 時不可能 Z」「不可能『...』」等技術陳述語法 pattern | 高（需 NLP 或 regex 樣本累積） | 較通用但 ROI 待量化 |

**最小可行改善（MVP）**：F1 + F2 — 立即消除高頻誤報來源 + 確保未來樣本可分析。F3/F4 待樣本累積後再決定。

### AC4 IMP 切分建議

**建議切分為 2 個獨立 IMP**：

1. **IMP-X.1 pytest 環境豁免**（首選，最高優先）
   - Where: `.claude/hooks/wrap-decision-tripwire-hook.py` main() 早期 return
   - 改動：偵測 `os.environ.get("PYTEST_CURRENT_TEST")` 或 `Path.cwd()` 含 `pytest-of-` 即 `return 0`
   - AC：新增 pytest 環境 unit test 驗證跳過行為；既有 4 hit 測試集（hit 2）改為斷言「pytest path 下不觸發」
   - 預估工時：30 分鐘

2. **IMP-X.2 context-aware filter (黑名單版)**（次優先）
   - Where: 同檔 + `wrap-triggers.yaml` 新增 `context_blacklist` schema
   - 改動：在 keyword match 後檢查前後 ±N 字含 blacklist 詞即 suppress
   - AC：hit 4 案例測試斷言不觸發；3+ 黑名單詞範例
   - 預估工時：1-2 小時（含詞表設計）

**不建議獨立 IMP** 的事項：
- F2 observability：W10-101 已完成
- F4 語法模式：樣本量不足，需 IMP-X.2 落地後累積樣本再評估
