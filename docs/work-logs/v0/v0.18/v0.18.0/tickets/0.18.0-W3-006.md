### 現場重現\n\nPM 執行 /ticket handoff W2-001 --to-sibling W3-002 後，ticket resume --list 正確顯示 1 個待恢復任務（5 個 stale 自動過濾）。但告知用戶 /clear 時，Stop hook 回應「多個待恢復任務，請選擇要恢復的任務」並阻擋退出。\n\n### 根因定位\n\nHook 的 _get_ticket_file_path (line 97-120) 只處理舊扁平路徑：\n  docs/work-logs/v{version}/tickets/{ticket_id}.md\n\n實際本專案已遷移到三層階層結構：\n  docs/work-logs/v{major}/v{major.minor}/v{version}/tickets/{ticket_id}.md\n\n導致 is_ticket_completed 和 is_ticket_recently_started 因檔案路徑不存在而全部返回 False。\n\n### 連鎖效應\n\n.claude/handoff/pending/ 現況：\n- 5 個 direction=auto 的 stale handoff (W2-015/W2-016/W3-007.1/W3-010.1/W4-001.1) - ticket 找不到\n- 1 個 direction=completed 的 handoff (W1-002) - direction 值非法\n- 1 個 direction=to-sibling:W3-002 的新 handoff (W2-001) - 剛建立\n\nstop hook 邏輯：\n- W2-001: ticket completed (路徑找不到誤判為 false，但實際能進入 chain_directions 分支) → 加入 pending_tasks\n- W1-002: ticket completed 誤判為 false + direction=completed 不在 non_blocking_directions → 加入 pending_tasks\n- 結果: len(pending_tasks) = 2 → 觸發多任務分支\n\n### 可用的正確函式\n\nhook_utils/hook_ticket.py 的 find_ticket_by_id (line 680+) 已正確處理雙路徑結構，但 handoff-auto-resume-stop-hook.py 沒用它，而是自己實作了錯誤版本。
