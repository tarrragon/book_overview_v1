### 4. 過度防護的訊號

ticket-skill-sync-check（136 行，**自動載入**，0 PC 引用）是最明顯案例。設計動機是「ticket skill 改了 SKILL.md 沒同步更新」，但：
- 0 PC pattern 證實此類事件曾累積成模式
- 同類問題本應由 commit hook 處理（規則已宣告 hook 為 C 路徑強制層）
- 「自律層 + hook 兜底」的雙層設計，自律層若無 PC 證實必要，純粹是 token 浪費

**估計可釋放 token**（保守）：
- Tier A 全刪：rules/core 移除 136 行（auto-load 直接省）+ pm-rules 移除 387 行（按需省）
- Tier B 瘦身：cognitive-load 砍至 ~150 行 = auto-load 省 224 行
- Tier C 合併去重：估省 200-400 行

### 5. 「我可能錯在哪裡」（強制自問）

1. **PC 引用次數 ≠ 違規預防價值**：高違規 rule 多被引用是因為已發生過，沒違規的 rule 可能正是「規則寫對了所以沒事」。零引用未必等於零價值。
   - 反駁：但 ticket-skill-sync-check 的設計動機已宣告由 hook 兜底，自律層存在僅為「閱讀提醒」，這在 30K auto-load 預算下成本過高。
2. **新規則需要時間累積 PC**：剛建立的規則（如 ticket-skill-sync-check 2026-05-03 落地）尚未有時間累積。
   - 反駁：134 行的「自律層 + hook 兜底」設計從第一天就應由 hook 兜底，自律層只需 30 行條目即可。
3. **「合併」可能違反單一職責**：把 communication + writing + format 合併會降低可定位性。
   - 反駁：三檔交叉引用已嚴重，讀者本來就要跳轉 3 次；合併後章節結構維持單一職責。
4. **本審計 grep 的是檔名字串**：未涵蓋「概念引用但未提檔名」的情況，可能低估 cognitive-load 等概念性規則的實際影響。
   - 補強：cognitive-load 374 行中「監測校準框架」「跨進程同步豁免」屬 ANA 結論一次性固化，這部分客觀屬 methodology 而非 rule，與 PC 引用無關。
5. **ticket-skill-sync-check 雖 0 引用但屬「事前防護」**：規則 6 失敗案例學習是事後固化，但有些規則設計是事前防患。
   - 反駁：事前防患若無 hook 強制，就只是宣告。本檔已自承「hook 為 C 路徑強制層」，自律層的存在是 belt-and-suspenders 的雙重投入。

### 6. 重構優先級清單（≤ 800 字總控）

**P0（auto-load 立即受益，預估省 ~360 行 / ~5K tokens）**

1. 刪除 rules/core/ticket-skill-sync-check.md（136 行）：hook 已是強制層，自律層零 PC 證實必要。改寫為 30 行附錄塞入 quality-baseline.md。
2. cognitive-load.md 砍半（374 → ~150 行）：移除「監測校準框架」「跨進程同步豁免」「失效偏差率」等 ANA 結論章節到 .claude/methodologies/cognitive-load-design-methodology.md（已存在）。

**P1（按需 rules 整理，無 token 直接受益但減維護負擔）**

3. 刪除 4 個零引用 pm-rules（skill-index, methodology-index, hook-governance, skill-creation-flow，共 387 行）：以 `ls .claude/skills/ + .claude/methodologies/` 取代索引；hook-governance 移到 .claude/hooks/README.md。
4. 合併 4 個 routing 檔（question/command/incident/tdd-completion）為單一 pm-rules/routing-tables.md（4 表共一檔）。

**P2（合併重構，需 PR-level 重新組織）**

5. ai-communication-rules + document-writing-style + document-format-rules 合併為單一檔：先以 references/ 拆出範例段，主檔保留 ≤ 200 行核心規則。
6. parallel-first + parallel-dispatch + two-stage-dispatch 合併為 dispatch-strategy.md。

**核心發現**：rules 系統存在嚴重 80/20 失衡。pm-rules 中 9% 檔案承載 45% 違規追蹤；15/44 = 34% pm-rules 為零引用。這不是「需要 RAG 檢索」的問題，是**rules 內容本身需要瘦身**。RAG 是用更貴方法繞過該瘦身的本質工作。

**對父 ticket 第一輪結論的衝擊**：第一輪「採 L3 hook-driven Rules 情境載入」結論成立，但本審計顯示更便宜的 P0 動作（刪除 136 行 + 砍半 374 行）即可釋放 ~5K auto-load tokens，**不需 hook trigger map 維護成本**。L3 應建立在已瘦身的 rules 基礎上，而非作為避免瘦身的替代方案。
