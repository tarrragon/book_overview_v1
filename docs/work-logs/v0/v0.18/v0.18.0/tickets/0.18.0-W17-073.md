## Task Summary

分析 跨進程同步修復類 IMP 的拆分準則（以 W17-071 IMP-1 為實驗案例）

---

## Problem Analysis

### 觸發事件

W17-071 IMP-1（validator + hook body-check 同步修復）thyme 執行時長 **30.3 分鐘**，超出 PM 預估 20 分鐘上限 50%。用戶於執行期間觀察「比預期久」，觸發 WRAP 反思：當初規劃 IMP-1 是否責任負擔過重？為何沒有拆分？

### WRAP 反思結論（Consider the Opposite）

真正該問的問題不是「本 IMP 要不要拆」，而是「**跨進程同步修復類 IMP 的拆分準則如何界定**」。單案例爭論無法產出可複用規範；需從模式層面回答。

### 問題根因

拆分權衡（「bug 一致性」vs「代理人認知負擔」）在 PM 決策路徑上**缺顯性記錄**：

- PM 規劃 IMP-1 時隱性選擇「同 bug 的多處修復同 commit 保一致性」→ 未觸發拆分
- 此權衡理由未寫入 ticket、methodology 或 pm-rule，無法被後人複用或挑戰
- 未來類似 spawned IMP（跨進程同步修復）重複面對同議題時，每次都要重新權衡，無基線數據

### 與 W17-049 關聯性（互補維度）

本 ANA 與 W17-049「泛 3b 派發拆分紀律強化方案」互補而非重疊：

| 維度 | W17-049 | W17-073（本 ANA）|
|------|---------|------------------|
| 切入點 | 所有 3b 派發任務 | 跨進程同步修復類 IMP |
| 維度類型 | **廣度** — 通用 3b 拆分紀律 | **深度** — 特定類型精化準則 |
| 觸發案例 | W17-048（派發範本化，prompt 撞 Hook 30 行上限） | W17-071（實作階段 4 改動點，執行 30 分鐘） |
| 過載訊號 | 派發前（prompt 體量、Hook 撞擊） | 執行中（時長超預期） |
| 根因層級 | 規範 / 工具 / 認知 / 資產四層 | 拆分權衡缺顯性記錄（方法論層） |
| 介入點 | Hook 擴展 + 3b 派發前檢查門 | atomic-ticket-methodology + task-splitting 條款 |
| 共同 where | `task-splitting.md` + `cognitive-load.md` | `task-splitting.md` + `cognitive-load.md` |

**產出組合模式**：W17-049 建立「3b 派發前通用拆分閾值」，W17-073 建立「跨進程修復類拆分矩陣」。W17-073 的矩陣應作為 W17-049 通用閾值下的**專章特例**，兩者收斂後形成完整拆分決策樹。

若兩 ANA 產出時機接近，需評估是否合併為單一 methodology 章節；若時機分散，後完成者應引用先完成者的結論並標注特例關係。
