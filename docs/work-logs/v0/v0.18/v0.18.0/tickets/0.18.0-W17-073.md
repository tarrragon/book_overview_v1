## Problem Analysis

### 觸發事件

### WRAP 反思結論（Consider the Opposite）

真正該問的問題不是「本 IMP 要不要拆」，而是「**跨進程同步修復類 IMP 的拆分準則如何界定**」。單案例爭論無法產出可複用規範；需從模式層面回答。

### 問題根因

拆分權衡（「bug 一致性」vs「代理人認知負擔」）在 PM 決策路徑上**缺顯性記錄**：

- PM 規劃 IMP-1 時隱性選擇「同 bug 的多處修復同 commit 保一致性」→ 未觸發拆分
- 此權衡理由未寫入 ticket、methodology 或 pm-rule，無法被後人複用或挑戰
- 未來類似 spawned IMP（跨進程同步修復）重複面對同議題時，每次都要重新權衡，無基線數據

- W17-071 實證 28 分 37 秒一次完成全綠，證明豁免條款在實務上可行且 ROI 高

### 跨進程同步修復豁免條款（提議納入 cognitive-load.md）

擬於 `.claude/rules/core/cognitive-load.md` §3b 派發前閾值（PM 派發檢查門）§三項核心閾值 之後新增子章節：

```markdown
### 跨進程同步修復豁免條款

**適用情境**：當 IMP 滿足以下「跨進程同步修復」全部特徵時，可豁免「功能職責數 > 2 須拆分」閾值：

| 特徵 | 判定 |
|------|------|
| 跨進程數 | 2-3 個進程（如 CLI lib + Hook，或 validator + checker + linter） |
| 邏輯同步性 | 各進程實作同一根因修復，同步修法防止漂移（ARCH-020 教訓） |
| 修改檔案數 | ≤ 5（含測試檔，超過走通用閾值） |
| 預估執行時長 | ≤ 30 分鐘（可由 PM 依過往案例估算，超過走通用閾值） |
| 根因數 | ≤ 2（多根因仍應拆 ANA → 多 IMP） |

**Why**：跨進程同步修復的拆分會破壞「同 bug 同 commit」一致性，導致兩 IMP 修法漂移風險（A 修了但 B 漏修，或 A 用方案 X、B 用方案 Y）；且 PM overhead（4 次 claim/complete）與測試重複設定成本高於合併執行。

**Consequence**：誤套用通用拆分閾值會：
1. 增加 ARCH-020 模式的修法漂移風險（首例已見：W17-032 只修 validator 漏修 hook，3 天後 W17-056 重爆 → W17-071 同步修復才終結）
2. 拆 4 ticket 後 PM overhead = 4× claim + 4× complete + 4× commit + 4× 驗收
3. 測試難以共用 fixture（每 ticket 重建測試環境）

**Action**：豁免時必須於 IMP Problem Analysis 顯性記錄：
- 跨進程清單（哪些進程同步修復）
- 邏輯同步性說明（同根因 / 同修法）
- ARCH-020 或 PC-110 引用（理論依據）
- 預估執行時長（與基線案例對照）

**首筆基線案例**：W17-071（28 分 37 秒 / 3 檔 / 12 測試 / 全綠 / 2 進程 / 2 根因）
```

### 拆分方案 ROI 對照（acceptance #2）

以 W17-071 為對照樣本評估四種拆法：

| 拆法 | PM overhead | 一致性風險 | 執行時長 | 測試成本 | 適合場景 |
|------|------------|----------|---------|---------|---------|
| **單 IMP（W17-071 實際）** | 1 claim + 1 complete + 1 commit | 低（同 commit 同步） | 28 分 37 秒 | 一次設定 12 測試 | 跨進程同步修復豁免條款命中時 |
| 雙拆（按進程） | 2× claim + 2× complete + 2× commit | 中（兩進程修法可能漂移） | 估 35-45 分（含 PM 仲裁） | 兩次測試設定 | 跨進程邏輯獨立時 |
| 三拆（按根因 × 進程） | 3-4× claim + 3-4× commit | 高（4 修法獨立執行） | 估 50-70 分 | 4 次測試設定 | 不適合此案例 |
| 父子 group | 1 父 + N 子 ticket（5+ 操作） | 中（需父子協調） | 估 40-60 分 | 子 ticket 各自設定 | 多根因 ANA 拆 IMP 時用 |

**結論**：W17-071 樣本證明「跨進程同步修復豁免條款」在 ROI 上明顯優於其他拆法。

### Consider the Opposite 檢查（acceptance #6）

**對立面**：「拆分增加 PM overhead 且破壞 bug 一致性 → 應永遠合併執行」是否成立？

**反論驗證**：
1. **N > 3 進程時**：合併執行的代理人認知負擔超出 7±2 工作記憶上限，仍應拆分
2. **跨 domain 修復時**：例如 validator + 文件 + Hook 各為不同 domain，雖跨進程但 domain 不同，合併會混淆責任
3. **多根因時**：根因 > 2 的修復應先拆 ANA → 多 IMP，避免單 IMP 過載
4. **PM 估時超 30 分鐘時**：超過經驗豁免上限，應拆分降低風險

**結論**：對立面在「合併永遠優於拆分」是無效的；豁免條款設「跨進程 ≤ 3 / 檔案 ≤ 5 / 時長 ≤ 30 分 / 根因 ≤ 2」邊界正是回應這四個反論案例。
