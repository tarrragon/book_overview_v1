## 問題現象

派發 `claude-code-guide` 查詢 Claude Code v114-v119 changelog 時，Hook 擋下並回報：

```
派發任務必須引用有效的 Ticket ID（格式：Ticket: {id} 或 #Ticket-{id} 或 [Ticket {id}]）
```

## 問題本質

**現行設計**（agent-ticket-validation-hook.py L61-65）：

```python
TICKET_EXEMPT_AGENT_TYPES = [
    "Explore",  # 探索代理人：用於蒐集資訊以建立 Ticket
]
```

僅 `Explore` 豁免，其他情報蒐集類 agent 全被擋。

## 受影響的情報蒐集類 agent

| Agent | 用途 | 工具集 | 現狀 |
|-------|------|-------|------|
| `Explore` | codebase 探索 | 唯讀 | 已豁免 |
| `claude-code-guide` | Claude Code/SDK/API 文件查詢 | Bash/Read/WebFetch/WebSearch | **被擋** |
| `general-purpose` | 複雜問題多步驟研究 | 全工具 | **被擋** |
| `Plan` | 架構規劃、實作計畫 | 唯讀（no Edit/Write） | **被擋** |
| `feature-dev:code-explorer` | 既有功能深度分析 | 唯讀 | **被擋** |

## 根因

**規則 3（所有發現必須追蹤）與規則 5（工具發現精神）的執行邊界未區分**：

- 規則 3：執行期發現問題 → 建 ticket
- 規則 5（search-tools-guide）：宣告「做不到」前先窮盡工具 → 允許決策前情報蒐集

現行 Hook 把「派發 agent」一律視為「執行任務」，未區分：
- 情報蒐集（決策前，建 ticket 的前置動作）
- 執行任務（決策後，需 ticket 追蹤）

## 觸發案例

Session `5W1H-20260424-093453` 中：
1. 用戶問 Claude Code v114-v119 變更 → 需派發 `claude-code-guide` 查官方 changelog
2. Hook 擋下 → PM 改用 WebFetch 直接查 → 完成情報蒐集
3. WebFetch 成功，但流程顯示 Hook 阻擋了正當查詢路徑

說明 Hook 管制邊界過嚴，違反 search-tools-guide 的「先窮盡工具」精神。

## WRAP 決策結論

### 採用方案 A：擴充白名單（立即修復）

**判別準則**：

| 可豁免 | 不可豁免 |
|-------|---------|
| agent 工具僅含唯讀（Read/Grep/Glob/WebFetch/WebSearch/Bash 查詢類） | agent 可 Edit/Write/git commit |
| 用途為情報蒐集、文件查詢、分析規劃 | 用途為實作/重構/測試執行 |

**新增白名單**（`TICKET_EXEMPT_AGENT_TYPES`）：

```python
TICKET_EXEMPT_AGENT_TYPES = [
    "Explore",                      # codebase 探索（已存在）
    "claude-code-guide",            # Claude Code / SDK / API 文件查詢
    "general-purpose",              # 複雜問題多步驟研究
    "Plan",                         # 架構規劃、實作計畫（唯讀）
    "feature-dev:code-explorer",    # 既有功能深度分析
]
```

**Hook 註解補充**：明示判別準則（唯讀 vs 可寫），並標註「若白名單 > 10 或誤用率升高，升級為方案 B」。

### WRAP 四階段結論摘要

| 階段 | 關鍵發現 |
|------|---------|
| W | 6 方案（A 白名單 / B 自動分類 / C prompt 標籤 / D frontmatter 類別 / E prompt 內容判斷 / F proposal slot） |
| R | 基本率：~40% 豁免機制用白名單；今日案例 claude-code-guide 僅含唯讀工具，風險自限 |
| A | 核心迭代（v0.18 測試修復）> 工具改善 → 排除 B-F 完整實作（>2 小時） |
| P | 最大失敗原因「白名單漏列」可低成本補救；最壞情況「膨脹至 20+」升級為 B |

### 被放棄方案（鋼鐵人測試記錄）

| 方案 | 優點 | 放棄理由 |
|------|------|---------|
| B 自動分類 | 不需維護、新 agent 自動生效、符合開放封閉原則 | 2 小時成本擠壓核心迭代；白名單短期內不會膨脹到需要自動化 |
| C prompt 標籤 | 顯式宣告、可控 | 需改 PM 習慣、訓練成本高 |
| D frontmatter 類別 | 標準化 | 需全量改 agent 定義，過度投資 |
| E prompt 內容判斷 | 不依賴 agent type | 自然語言判斷易誤判 |
| F proposal slot | 保留追蹤完整性 | 過度設計 |

### 未來升級路徑（絆腳索）

- 觸發條件：白名單 agent 數 > 10 或誤用率升高
- 升級方案：方案 B（讀 agent definition 的 tools 欄位，僅含唯讀工具時自動豁免）
- 記錄位置：Hook 檔頭註解 + 本 ticket Solution
