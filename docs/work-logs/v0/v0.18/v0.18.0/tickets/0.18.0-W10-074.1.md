### 4. 四個關鍵決策的候選分析

#### 決策 1：Marketplace 可攜性標準

| 選項 | 含義 | 驗收方式 | 成本 | 推薦 |
|------|------|---------|------|------|
| A. 技術剝除 | Skill 內無框架路徑/專案 ticket ID/Hook 依賴 | 自動化 grep 掃描（`scripts/portability-check.sh`） | 低（可持續） | - |
| B. 使用者零上下文可用 | 新使用者讀完 SKILL.md 即能理解並套用 | 新使用者 dry-run 測試 | 高（一次性） | - |
| **C. 兩者並行**（推薦） | Phase 1 用 A（日常檢查），Phase 2 用 B（發佈前） | 階段化 | 中 | **是** |

**推薦 C**，理由：A 可自動化但只檢查形式；B 是最終品質門但成本高。兩者並行可兼顧日常守門與發佈品質。

#### 決策 2：Metric 框架（M1-M5 候選）

**認知負擔類**：
- **M1**：「找到答案」路徑長度 — 讀者從 SKILL.md 開始，需要開啟幾個檔案才能解決問題？目標 ≤ 2
- **M2**：reference 獨立理解率 — 隨機挑選一個 reference，讀者不看其他 reference 能否獨立理解？目標 100%

**Token 類**：
- **M3**：SKILL.md 主檔 token 上限 ≤ 5000 tokens（遵循 Anthropic 建議 < 5k tokens）
- **M4**：單一 reference 觸發載入 token ≤ 3000 tokens（避免單檔過大）
- **M5**：grep 定位某問題的 token 成本 — 用 grep 搜尋關鍵字，能否在 < 500 tokens 內定位答案？

**這 5 個 metric 是評估工具，不是硬性 KPI**。目的是讓「達成/未達成」可被重複驗證。

#### 決策 3：既有 methodologies/ 檔案處理

受影響檔案清單：

| 檔案 | 現有引用情況 | 候選處理 |
|------|------------|---------|
| `comment-writing-methodology.md` | 被 quality-common.md 1.6 節引用 | A 取代 / **B 保留 pointer** / C 引用翻轉 |
| `worklog-writing-methodology.md` | 被 doc-flow skill 引用 | A / **B**（推薦） / C |
| `methodology-writing-methodology.md` | 主要被 methodology-writing skill 自身引用 | **A 取代**（推薦）/ B / C |
| `document-format-rules.md`（rules/core） | 自動載入，被多處引用 | **只取其「寫作規範」部分融入 Skill，其餘保留**（混合方案） |
| `quality-common.md` 1.6 節 | 自動載入，被多處引用 | **保留為框架規則，Skill 作為跨專案方法論**（分層共存） |

**推薦方案**（混合）：
- 單純方法論檔（comment-writing、worklog-writing、methodology-writing）→ B 保留 pointer（不破壞現有引用）
- `.claude/rules/core/` 下的規則 → 保留（框架約束），Skill 作為「跨專案可攜的方法論內容來源」
- 未來框架內其他檔案引用時，優先指向 Skill

這符合「框架資產 vs 專案產物分離」精神，也保留向後相容。

#### 決策 4：W10-011 融入映射

W10-011 原計畫擴充的 4 項內容，映射到新 Skill 的 reference：
