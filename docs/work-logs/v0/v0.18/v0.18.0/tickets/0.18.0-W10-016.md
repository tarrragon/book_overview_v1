# Execution Log

## Task Summary

分析 SKILL 專用 Hook 架構設計與遷移方案

---

## Problem Analysis

### 問題根因

`.claude/hooks/` 集中 101 個 Hook，混合「Skill 專用」與「框架通用」兩類職責。Skill 專用 Hook（如 ticket-* / worktree-* / wrap-decision-tripwire / version-release-guard / 5w1h-compliance-check）的生命週期應與所屬 Skill 綁定，但現況下：

- Skill 經 `sync-push` / `sync-pull` 跨專案同步時，Hook 不跟著走
- Hook 內部邏輯改動需與 Skill 同步，但兩者分散在不同目錄
- `hook-completeness-check.py` 將所有 hooks/ 視為單一池，Skill-Hook 命名約定無從強制

### 影響範圍

- `.claude/hooks/`（101 hooks）
- `.claude/skills/`（27 skills）
- `.claude/settings.json`（hook 註冊清單）
- `.claude/hooks/hook-completeness-check.py`（掃描器）
- `.claude/skills/ticket/`、`.claude/skills/worktree/`、`.claude/skills/wrap-decision/` 等與 hook 強耦合的 skill

### 相關 Error Pattern

無直接 PC，但與「框架資產 vs 專案產物分離」原則（`framework-asset-separation.md`）同源：Skill 是可獨立發佈單位，其專用 Hook 應隨 Skill 移動。

---

## 重現實驗結果

### 假設 1：settings.json 是否支援 skill 內部 hook 路徑？

**指令**：檢視 `.claude/settings.json` 的 hook command 欄位格式。

**結果**：command 為 `$CLAUDE_PROJECT_DIR/.claude/hooks/<file>.py` 絕對路徑形式。Claude Code runtime 對 hook command 不限制路徑前綴，只要 `$CLAUDE_PROJECT_DIR` 解析後檔案存在且可執行即可。

**結論**：路徑支援性 = **是**。`$CLAUDE_PROJECT_DIR/.claude/skills/<skill>/hooks/<file>.py` 形式可直接註冊。**無需修改 Claude Code runtime**。

### 假設 2：hook-completeness-check 掃描範圍

**指令**：閱讀 `.claude/hooks/hook-completeness-check.py` 的掃描邏輯。

**結果**：當前掃描 `.claude/hooks/*.py`，比對 settings.json 註冊清單。若 hook 移到 skill 目錄，掃描器需擴充為遞迴掃描 `.claude/skills/*/hooks/*.py`。

**結論**：需擴充掃描器。中等技術成本（單檔修改 + 測試）。

### 假設 3：既有 hooks 分類分布

**結果**（依檔名前綴與職責盤點 101 hooks）：

| 類別 | 數量 | 範例 | 歸屬 Skill |
|------|------|------|----------|
| ticket-* / handoff-* / ana-ticket-* / creation-acceptance / needs-context / post-ticket-* / acceptance-gate / agent-ticket-validation / ticket-skill-sync-check | ~15 | ticket-id-validator-hook.py | ticket |
| worktree-* | 6 | worktree-zombie-cleanup-hook.py | worktree |
| wrap-decision-tripwire | 1 | wrap-decision-tripwire-hook.py | wrap-decision |
| version-release-guard / version-consistency-guard | 2 | version-release-guard-hook.py | version-release |
| 5w1h-compliance-check | 1 | 5w1h-compliance-check-hook.py | 5w1h-decision |
| skill-* | 3 | skill-registration-check-hook.py | （框架元，跨 skill） |
| 其餘（框架通用） | ~73 | bash-edit-guard / git-index-lock-cleanup / phase-* / agent-* / session-* / file-* | 留在 .claude/hooks/ |

**結論**：~25 hooks 屬 Skill 專用，~73 hooks 屬框架通用，~3 hooks 屬框架元（管理 skills 自身）。

---

## Solution

### 設計：雙層 Hook 架構（並存而非全遷移）

| 層級 | 目錄 | 職責 |
|------|------|------|
| 框架層 | `.claude/hooks/` | 跨 Skill 通用流程防護、語言/字元集守護、git/test/agent 基礎設施 |
| Skill 層 | `.claude/skills/<skill>/hooks/` | 該 Skill 專用的驗證、提醒、生命週期攔截 |

**Why（雙層而非全遷移）**：73 個框架通用 hook 與任何特定 skill 都不綁定（如 `bash-edit-guard`、`git-index-lock-cleanup`、`session-*`），強行歸入某 skill 會破壞單一職責；保持 `.claude/hooks/` 為框架層更符合現實。

**Consequence（不分層）**：sync-push/sync-pull 時 ticket skill 無法帶上其 15 個專用 hook，導致跨專案使用 ticket skill 時 hook 缺失，行為退化。

**Action**：分階段遷移（見下方 follow-up tickets）。

### 遷移範圍建議

| 優先度 | Skill | Hook 數 | 依據 |
|--------|------|--------|------|
| 高 | ticket | ~15 | 強耦合 ticket schema、ID 規範、生命週期；獨立發佈價值最高 |
| 高 | worktree | 6 | 與 worktree CLI 命令深度綁定 |
| 中 | wrap-decision | 1 | 單檔，遷移成本低；與 SKILL.md 觸發條件強同步 |
| 中 | version-release | 2 | 版本發布流程綁定 |
| 中 | 5w1h-decision | 1 | 與 SKILL 強綁定 |
| 低 | skill-* | 3 | 屬框架元，改動需謹慎；可暫留 .claude/hooks/ 直到雙層證明穩定 |

### 對 W10-009 wrap-decision-tripwire-hook 的位置建議

**建議目標路徑**：`.claude/skills/wrap-decision/hooks/wrap-decision-tripwire-hook.py`

**理由**：該 hook 的觸發訊號（`signals/keywords/threshold`）來自 `.claude/skills/wrap-decision/config/wrap-triggers.yaml`，與 SKILL.md 的觸發條件章節雙向同步（W10-055 追蹤）。將 hook 放回 skill 目錄能讓「config 變更 → hook 行為變更 → SKILL.md 文檔變更」三者同 PR、同目錄，降低跨檔同步成本。
