## Problem Analysis

### 問題根因

W12-002 父任務識別 emoji 為主污染源並啟用 AUQ charset guard hook，但未驗證 (1) 簡體字 / (2) 日文漢字是否尚有獨立污染通道。W13-007 推測「emoji 共現觸發簡體 token 偏差」為主機制（假設充分性待驗）。

### 掃描方法

沿用 .claude/hooks/askuserquestion-charset-guard-hook.py 的 SIMPLIFIED_CHARS 清單（PC-074 已驗證不含繁簡共用字）；新建 JAPANESE_ONLY 候選清單（65 字元）涵蓋常見日文新字體專屬字。以 Python 腳本分層 count-only 掃描（避免 PM token 污染），脈絡分類區分引用 vs 污染。

### 影響範圍分層

| 層級 | 狀態 | 說明 |
|------|------|------|
| 自動載入 (CLAUDE.md + rules/) | zero hit | W12-002 已徹底消毒 |
| pm-rules/ / references/ / methodologies/ / skills/ / agents/ | zero hit | 完全乾淨 |
| error-patterns/ PC-072/074/075 | 35 simp hits | 全為污染案例引用（必要討論） |
| hooks/ 偵測相關檔案 | 102 simp + 0 jp hits | SIMPLIFIED_CHARS 定義與測試 fixture（技術必要） |
| hooks/tests/test_project_root_symmetry.py | 5 jp hits U+4E21「両」 | 真污染：docstring 寫「両個」應為「兩個」 |

### 清單設計缺陷發現（PC-074 類比）

首版 JAPANESE_ONLY 候選清單誤列「鑑」U+9451 為日文專屬；實為繁日共用字（繁中「鑑別」、「借鑑」、「鑑於」）。造成 PC-050/PC-070/pm-agent-observability.md 共 11 處 false positive。已於 AC5 驗證階段移除。類比 PC-074 首發經驗：清單擴充前必須逐字驗證繁體是否也用。

- PC-074 charset hook 繁簡共用字 false positive（本 ANA 重現於繁日共用字情境）

### 重現實驗結果

#### 實驗方法

- 腳本：/tmp/w12-002-5/scan_hanzi.py, scan_v2.py, locate_hits.py, masked_context.py, final_scan.py
- 指令：python3 /tmp/w12-002-5/*.py（PM 前台直接執行，不派發代理人）
- 輸出策略：僅回傳 file path + hit count + codepoint，禁止回傳漢字 line 內容到 PM context

#### 實驗執行

1. 掃描範圍：CLAUDE.md + .claude/rules + pm-rules + references + methodologies + error-patterns + skills + agents + hooks
2. 分層統計：每層 files_with_hits / simplified_count / japanese_count
3. 脈絡定位：可疑 hit 逐檔逐行標記類別（quote / list / rule-table / code-block / other）
4. Masked context：以 [H:U+XXXX] 替換所有 CJK 字元後輸出上下文人工判讀
5. 修正 false positive：移除「鑑」U+9451 後重跑

#### 實驗發現

**已驗證事實**：

| 事實 | 證據 |
|------|------|
| 自動載入層（auto-load + 常用 on-demand）零漢字污染 | final_scan 輸出 files=0 |
| error-patterns 35 simp hits 全為污染案例必要引用 | masked_context 人工判讀 |
| hooks 102 simp hits 全為 SIMPLIFIED_CHARS 定義或測試 fixture | locate_hits 分類為 list/quote |
| test_project_root_symmetry.py 5 jp hits U+4E21「両」為真污染 | 應為繁中「兩」(U+5169)，docstring 誤用日文新字體 |
| 「鑑」U+9451 為繁日共用字，不可列入 JAPANESE_ONLY | PC-050/070/observability 11 處正常繁體使用被誤報 |

**W13-007 emoji 共現假設評估**：
- **簡體字維度**：假設充分。自動載入層無簡體污染，AUQ hook 已攔截 PM payload 層，歷史 PC-072 記錄的污染多為 PM context 累積效應（context-refresh 方向正確）。
- **日文漢字維度**：假設不充分。test_project_root_symmetry.py 的「両」與 emoji 無共現關係，為 PM 撰寫測試碼時的 Write/Edit 路徑獨立污染源。

### 解決方案

**結論分三類**：

#### A. 現況評估

emoji 共現假設在簡體字維度充分（加上 AUQ hook 已止血），日文漢字維度不充分但僅發現 1 個真污染檔案，範圍極小。

#### B. 需建立的後續 IMP

1. **IMP-A**：修正 test_project_root_symmetry.py 5 處「両個」→「兩個」（U+4E21 → U+5169）
2. **IMP-B**（可選）：擴充 AUQ charset guard hook 新增日文新字體偵測 + 逐字驗證 JAPANESE_ONLY 清單不含繁日共用字

#### C. 需建立的追加 Error Pattern

- 新建 PC（或更新 PC-074）：**繁日共用字類比繁簡共用字**，日文漢字清單設計需同樣做「繁體是否也用此字」的逐字驗證。觸發案例：本 ANA 首版清單誤列「鑑」U+9451。
