## 誤報類型 2：驗收檢查誤報（W5-013 範疇）

### A. 根因分析

**觸發位置**：process-skip-guard-hook.py:89-100 SKIP_ACCEPTANCE 模式
- ("不需要","驗收") / ("跳過","驗收") / ("省略","驗收")
- ("直接","完成")   <-- 最易誤報
- ("不做","審查")

**為何誤報**：
1. Hook 是 UserPromptSubmit，僅在用戶送出 prompt 瞬間觸發，**與代理人是否在背景執行無關**
2. 觸發邏輯（detect_skip_intent, line 128-150）只做字串 in 比對，**完全不檢查 .claude/dispatch-active.json**
3. 用戶只要 prompt 中同時含「直接」「完成」兩字（如「你直接讓代理人去完成 X」）就會誤報
4. 設計意圖：防止 PM 說「跳過驗收直接 complete」；實際行為：任何含這兩字的 prompt 都觸發，無上下文感知

**落差**：
- 設計假設：關鍵字 pair = 強烈省略意圖
- 實際：「直接/完成」「不做/審查」是高頻日常用語，在派發、規劃、Phase 4 重構等場景大量出現，語義與「跳過驗收」無關

### B. 修正方案

**方案 1（保守過濾 + 上下文感知，推薦）**：
在 detect_skip_intent 前加 3 個 guard：
1. Active dispatch guard：若 .claude/dispatch-active.json 非空，跳過 SKIP_ACCEPTANCE 與 SKIP_AGENT_DISPATCH 的偵測（代理人還在跑，談「完成」語義不同）
2. ticket track complete 語境 guard：僅當 prompt 同時提及 ticket ID 或 "complete" 命令時才啟用 SKIP_ACCEPTANCE
3. 降低關鍵字 pair 假陽性：移除 ("直接","完成") 和 ("不做","審查")，保留更明確的 ("跳過","驗收")、("省略","驗收")

優點：保留核心防護、假陽性大減；實作 ≤ 30 行
缺點：仍是關鍵字層面，極端案例漏檢
誤殺風險：低

**方案 2（重構為語義意圖分類器）**：
用結構化規則（Ticket 狀態 + 當前 phase + 命令語境）判斷是否為省略意圖，關鍵字 pair 改為「弱訊號」而非「強判定」。
優點：根本解決；缺點：實作複雜，需重新定義觸發模型

**推薦**：方案 1（先止血），後續若誤報仍多再評估方案 2。

## 誤報類型 3：Cache 過時誤報（W5-013 範疇）

### A. 根因分析

**觸發位置**：acceptance-gate-hook.py:283-286（checklist 生成）+ acceptance_checker.py:23-59（has_acceptance_record）

**為何誤報**：
1. has_acceptance_record 只搜尋 **ticket content body** 關鍵字：「驗收結果: 通過」、「Acceptance Audit Report」、「驗收通過」、「驗收者：」、「Auditor:」、「PM 直接驗收」、「acceptance-auditor」
2. 但 ticket track check-acceptance --all 將狀態寫入 **frontmatter 的 acceptance: list 欄位**（track_acceptance.py:433 ticket["acceptance"] = acceptance_list，每個項目為 "[x] ..." 字串）
3. **兩個資料源完全脫鉤**：CLI 寫 frontmatter，Hook 讀 body 關鍵字，互不交集
4. 結果：check-acceptance --all 全勾選後，Hook 仍顯示「[WARNING] 1. acceptance 未全勾選」

**並非 cache 問題**：沒有實際 cache 層。是 **資料來源不匹配**（misaligned source of truth），外觀上像是過時狀態。

**落差**：
- 設計意圖：兩種驗收途徑（CLI 勾選 / 人工寫 Audit Report）任一完成即算驗收
- 實際：Hook 只認 Audit Report 類關鍵字，CLI 勾選完全不計

### B. 修正方案

**方案 1（補一個 frontmatter 來源檢查，推薦）**：
修改 has_acceptance_record（或新增 has_acceptance_checklist_done）：
- 解析 frontmatter 的 acceptance: list
- 若所有項目皆以 "[x]" 或 "- [x]" 開頭 → 視為驗收通過
- 與既有關鍵字檢查 OR 合併：return keyword_match or checklist_all_checked

優點：最小改動、對齊 CLI 與 Hook 的雙來源；實作 ≤ 20 行
缺點：需在 acceptance-gate checklist 項目 1 的訊息同步更新

**方案 2（統一驗收狀態為單一 source of truth）**：
廢止 frontmatter acceptance: list，改為寫入 Acceptance Audit Report 段落（即使 CLI 勾選也落成 body 文字）。
優點：單一真相；缺點：破壞既有 CLI 語義，影響 check-acceptance 多處、需 migration。

**推薦**：方案 1。方案 2 延後至大型 ticket 系統重構時再評估。
