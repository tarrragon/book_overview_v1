## Problem Analysis


### WRAP 三問

**W（Widen）候選方案**：
1. **選定**：`_find_handoff_file` 擴充 fallback 讀 JSON 的 `ticket_id` 欄位（兼容所有命名）
2. 候選 A：批次重命名 legacy `v*-handoff.json` 檔為 `{id}.json`（有 side effect，且未確認寫入端已停用舊格式）
3. 候選 B：改 handoff.py 寫入端統一使用新 v 前綴命名（需同步改 _find_handoff_file，波及面大）

**A（Attain distance）機會成本**：
- 本 ticket 估時 0.5-1 小時，擠壓 W15-007 bash 規則五實作
- 但 resume 是 PM 日常接手流程基石，bug 未修每次 session 切換都要手動繞路，長期成本更高

**P（Prepare to be wrong）失敗原因預想**：
- 最可能失敗：fallback 掃描效能在大量 handoff 時變慢（目前 archive 數十檔尚可接受）。防護：pending 目錄通常 < 10 檔，掃描成本可忽略；若未來擴大可加 cache
### Context Bundle

**症狀**：
- `ticket resume --list` 可列出 0.18.0-W15-007
- 但 `ticket resume 0.18.0-W15-007` 報 `[Error] Ticket 0.18.0-W15-007 無待恢復的交接檔案`

**根因**：
`.claude/skills/ticket/ticket_system/commands/resume.py:_find_handoff_file` (L76-118) 搜尋策略：
1. L94: `dir_path / f"{ticket_id}.json"` - 搜尋 `0.18.0-W15-007.json`（未命中）
2. L99: `dir_path / f"{ticket_id}.md"` - 搜尋 `0.18.0-W15-007.md`（未命中）
3. L105-116 fallback: 掃描 direction 欄位反向匹配（direction="to-sibling" 不含 target id，未命中）

但實際檔案為 `v0.18.0-W15-007-handoff.json`（legacy 命名格式，來自舊版 handoff 寫入邏輯；`.claude/handoff/archive/` 亦存在 `v0.17.2-W2-impl.json` 等 v 前綴歷史檔案佐證）。

**對比**：
- `list_pending_handoffs` 經由 `scan_pending_handoffs()` glob 所有 `*.json` 並讀取 `ticket_id` 欄位，故 --list 成功
- `_find_handoff_file` 僅靠檔名模式比對，故單檔查詢失敗

### Solution

**策略**：在 `_find_handoff_file` 的 fallback 區塊（L103-117）擴充，除了現有的 `direction` 反查外，新增「讀取 JSON 的 `ticket_id` 欄位比對」作為第三層 fallback，避免依賴檔名約定。

**實作步驟**：
1. 修改 `_find_handoff_file`：在 direction 反查失敗後，繼續掃描 JSON 的 `ticket_id` 欄位
2. `mark_handoff_as_resumed` / `archive_handoff_file` 已透過 `_find_handoff_file` 取得 Path，受惠於同一修正
3. 新增 regression test：tests/ 建立 fixture `v{id}-handoff.json` 並驗證 resume 可載入

**不修改的部分**：
- 不改 handoff.py 的寫入檔名（保持 `{ticket_id}.json`）
- 不批次重命名既有 v-prefixed legacy 檔案（讓 fallback 吸收即可，避免 side effect）
