## TDD Phase 1 功能設計（2026-04-18）

### 1. 設計目標

linux Good Taste：三個 CLI（snapshot / handoff-ready / checkpoint-status）共用同一決策推導。抽出 checkpoint_state() 核心函式，讓所有投影 CLI 都是其薄包裝；017.6 觀測 log 為橫切關注點併入。

### 2. CheckpointState dataclass 規格

位置：.claude/skills/ticket/ticket_system/lib/checkpoint_state.py

欄位定義（對應 completion-checkpoint-rules.md 所列 Checkpoint 0/0.5/1/1.5/1.8/1.85/1.9/2/3）：

| 欄位 | 型別 | 說明 | 資料來源 |
|------|------|------|---------|
| current_phase | str | Checkpoint ID（"0"/"0.5"/"1"/"1.5"/"1.8"/"1.85"/"1.9"/"2"/"3"） | 推導 |
| phase_label | str | 人類可讀標籤（如 "C1.5 未提交變更待 commit"） | 推導 |
| next_action | str | 下一步建議動作 | 推導 |
| ready_for_clear | bool | 是否通過 /clear 5 項 Ready Check | 推導 |
| pending_checks | List[PendingCheck] | 未通過檢查項（check_id / reason / blocker / auto_detectable） | 推導 |
| active_agents | int | dispatch-active.json 中未完成代理人數 | dispatch-active.json |
| uncommitted_files | int | git status 未提交檔案數 | git status --porcelain |
| unmerged_worktrees | List[str] | 未合併 worktree 路徑清單 | git worktree list --porcelain |
| active_handoff | Optional[str] | handoff pending 檔的 ticket_id | .claude/handoffs/pending/*.json |
| in_progress_tickets | List[str] | 當前版本 in_progress ticket IDs | ticket track query |
| data_sources | Dict[str, str] | 各資料來源時間戳（除錯用） | 各來源 mtime |
| computed_at | str | ISO 時間戳 | datetime.utcnow().isoformat() |

輔助 dataclass：PendingCheck (frozen) 含 check_id / reason / blocker / auto_detectable 四欄位。auto_detectable=False 代表無法自動驗證（如 worklog 一致性）。

### 3. checkpoint_state() 函式簽名

位置：ticket_system/lib/checkpoint_state.py

簽名：
  def checkpoint_state(
      ticket_id: Optional[str] = None,
      *,
      use_cache: bool = False,
      log_metrics: bool = True,
  ) -> CheckpointState

語意：
- ticket_id=None 使用當前 in_progress 推導
- use_cache: 60 秒內快取（Phase 1 僅留接口）
- log_metrics: 是否寫入 pm-automation-metrics.jsonl
- Raises: CheckpointStateError（關鍵來源失敗且無 fallback）

資料來源優先順序（fail-open）：

| 順序 | 資料來源 | 失敗 fallback |
|------|---------|-------------|
| 1 | git status --porcelain | uncommitted_files=-1 + 記 log |
| 2 | .claude/state/dispatch-active.json | 缺檔時 active_agents=0 |
| 3 | .claude/handoffs/pending/*.json | 缺目錄時 active_handoff=None |
| 4 | ticket CLI query（當前版本 in_progress） | 錯誤時空清單 |
| 5 | git worktree list --porcelain | 錯誤時空清單 |

單一來源失敗不阻斷整體；data_sources 欄位記錄錯誤，pending_checks 以 auto_detectable=False 標記。

### 4. Checkpoint 推導邏輯（核心決策）

「從高到低第一個命中」的優先級：

| 優先級 | 判定條件 | current_phase | next_action |
|-------|---------|---------------|------------|
| 1 | active_agents > 0 | "1.85" | 等待代理人完成或 ticket track agent-status |
| 2 | unmerged_worktrees != [] | "1.9" | 合併 worktree 並清理 |
| 3 | uncommitted_files > 0 | "1" | git add + git commit |
| 4 | active_handoff is not None | "2" | ready for /clear（有待恢復 handoff） |
| 5 | in_progress_tickets 非空且 ticket_id 指定 | "0.5" | ticket track append-log 記錄階段 |
| 6 | 其他 | "3" | ready for /clear 或選下個 Ticket |

ready_for_clear = (current_phase in {"2","3"}) AND pending_checks 全為 auto_detectable=False

### 6. 觀測 log 橫切契約（017.6 併入）

位置：.claude/logs/pm-automation-metrics.jsonl（append-only JSONL）

Schema：

  {
    "ts": "2026-04-18T12:34:56.789Z",
    "event": "checkpoint_state",
    "caller": "snapshot|handoff-ready|checkpoint-status|dispatch-check",
    "ticket_id": "0.18.0-W10-017.8",
    "current_phase": "1",
    "ready_for_clear": false,
    "active_agents": 0,
    "uncommitted_files": 2,
    "duration_ms": 47.3,
    "data_source_errors": []
  }

寫入時機：checkpoint_state() 函式末尾（回傳前）統一寫入；呼叫端用 caller 欄位標示自己。log_metrics=False 時不寫（單元測試隔離）。

寫入策略：失敗不阻斷主流程（info level log + stderr warning，遵守 quality-baseline 規則 4）；檔案 > 10MB 自動 rotate 為 .1.jsonl（保留一份歷史）。

4 週 baseline 目的（對應 017.9 評估依據）：
- 呼叫頻率（驗證 PM 自動化是否被實際使用）
- current_phase 分布（哪些 Checkpoint 最常卡住）
- duration_ms（驗證 <100ms 假設）
- data_source_errors（驗證 fail-open 是否掩蓋真實問題）
