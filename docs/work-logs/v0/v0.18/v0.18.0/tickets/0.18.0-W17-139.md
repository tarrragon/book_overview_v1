### 問題描述

用戶要求將 sysprog21/zhtw-mcp（Rust 寫成的繁體中文檢查 MCP server）整合到 .claude/ 框架，作為文字品質審查的機械化補強層。zhtw-mcp 提供 1100+ 詞彙規則（簡轉繁、跨境用語）、教育部國字標準字（裏→裡、着→著）、全形標點強制（，。：）、AI 寫作偵測等能力，與既有 basil-writing-critic agent 的邏輯結構審查（三明示、資訊優先序）形成互補。

### 跨專案部署挑戰

framework `.claude/` 透過 sync-claude-{push,pull}.py 雙向同步到獨立 repo 後 sync 到其他專案。直接寫死 zhtw-mcp 配置會造成兩個問題：

1. 無 zhtw-mcp 需求的專案（例如英文專案）被硬性阻擋
2. 一個專案的 opt-out 決策被 sync 帶到其他專案，污染各自決策

### 設計約束

| 約束 | 來源 |
|------|------|
| Hook 失敗必須可見（stderr 或日誌） | quality-baseline 規則 4 |
| 跨專案 framework 不應引用專案層識別符 | document-format-rules 規則 8 |
| 軟性檢查不阻擋 session 啟動 | 跨專案部署現實考量 |
| Per-project opt-out 決策不參與 sync | sync-exclusion-guide 類型 B |

### 用戶選定方案

兩階段 AskUserQuestion 後鎖定 Layer 1+2 + settings.local.json：
- Layer 1：MCP 註冊寫 settings.local.json（不 sync）
- Layer 2：SessionStart 可用性檢查 hook
- 跳過 Layer 3（basil-writing-critic agent 升級，作為後續可選）

### 變更檔案

| 檔案 | 動作 | 說明 |
|------|------|------|

### Hook 三狀態設計

| 狀態 | 偵測方式 | 輸出通道 | 訊息 |
|------|---------|---------|------|
| 未安裝 | `shutil.which("zhtw-mcp")` 為 None | stderr | 平台對應 curl 安裝命令 + 註冊命令 + opt-out |
| 已安裝未註冊 | binary OK，`claude mcp list` 不含 zhtw-mcp | stderr | binary 路徑 + 註冊命令 + opt-out |
| 全 OK | binary OK，registered=True | stdout（單行） | `[zhtw-mcp Check] OK: <版本> (registered)` |
| Skip flag | `.claude/.zhtw-mcp-skip` 存在 | 完全靜默 | 僅寫日誌不輸出 |

### 跨專案部署設計

| 變更 | sync 行為 | 理由 |
|------|----------|------|
| Hook 檔案 | sync（framework 共享） | 所有專案共用同一檢查邏輯 |
| settings.json hook 註冊 | sync | hook 在所有專案啟用 |
| settings.local.json mcpServers | 不 sync | 每專案獨立 opt-in MCP |
| `.zhtw-mcp-skip` flag | 不 sync（已加入排除清單） | per-project opt-out 決策不污染他專案 |
| sync 腳本本身的排除規則 | sync | 規則更新跟隨 framework |

### 關鍵設計決策

**Soft check vs hard block**：選擇 stderr + exit 0 折衷。Hard block 會在英文專案 session 啟動就噴錯（過度耦合），靜默通過違反 quality-baseline 規則 4（Hook 失敗必須可見）。Soft check 兩者兼顧。

**Skip flag 為何需要 sync 排除**：`.zhtw-mcp-skip` 是「per-project opt-out 決策資料」。若隨 sync 流動，A 專案決策跳過會污染 B 專案的判斷。歸類為 sync-exclusion-guide 類型 B（Local-only settings），與 `settings.local.json` 同類處理。
