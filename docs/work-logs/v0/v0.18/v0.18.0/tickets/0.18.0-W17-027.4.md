## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`handoff --auto` 目前只負責產生 `.claude/handoff/pending/<ticket>.json`，stdout 只告知檔案已建立。這完成了「留下接手資料」但沒有完成「提示下一個可接手 ticket」的 scheduler 銜接，PM 仍需額外記得執行 `ticket track runqueue --context=resume --top 3`。

這個缺口造成 `handoff --auto` 與 W17 scheduler 入口各自存在但未形成單一操作閉環：auto handoff 生成 pending handoff，runqueue 才知道 resume context 的候選；兩者若不在同一 stdout 中串接，/clear 或 session handoff 前的 PM 決策摩擦仍會保留。

### 影響範圍

- `.claude/skills/ticket/ticket_system/commands/handoff.py`：`_execute_auto_handoff()` 成功路徑需追加 runqueue 建議段落。
- `.claude/skills/ticket/ticket_system/commands/track_runqueue.py`：應復用既有 `runqueue --context=resume --top 3` 排序與輸出邏輯；若需更乾淨整合，可抽出 render helper。
- `.claude/skills/ticket/tests/test_handoff_auto.py`：更新既有 auto handoff 成功路徑輸出預期，新增有候選/無候選測試。
- `.claude/handoff/pending/`：auto handoff 建檔後立即成為 runqueue resume context 的輸入。

### 相關 Error Pattern

- PC-075 spawned/衍生任務狀態未推進的下游風險：handoff 若不列出下一步，completed ANA/IMP 後的後續推進容易跨 session 遺忘。
- PC-050 類 PM 過早或錯誤判斷代理人/流程狀態：需讓 CLI 輸出可觀測，而不是依賴 PM 記憶下一個查詢命令。

### WRAP 簡化三問（Claim 版）

#### W（Widen）-- 有其他做法嗎？

1. **方案 A：在 `handoff --auto` 成功後直接復用 `track_runqueue.execute_runqueue()` 輸出 `--context=resume --top 3`**  
   優點：最貼近 acceptance；復用 runqueue 排序與 context=resume 語意；不重做 scheduler 邏輯。缺點：`execute_runqueue()` 目前直接 print，若要測試輸出需用 `capsys` 或抽 helper。

2. **方案 B：在 `handoff.py` 新增 helper，直接呼叫 `track_runqueue` 的 render/過濾函式組裝提示文字**  
   優點：stdout 可控，容易精準輸出「目前無待恢復 ticket」。缺點：會碰觸 `track_runqueue` 內部私有函式，耦合比方案 A 高；若 runqueue 排序日後改動，helper 可能漏同步。

3. **方案 C：只輸出下一步指令提示，不實際呼叫 runqueue**  
   優點：改動最小。缺點：不滿足「提示下一個可接手 ticket」與「無結果訊息」驗收，只把摩擦延後給 PM。

4. **方案 D：用 subprocess 呼叫 `ticket track runqueue --context=resume --top 3`**  
   優點：最接近使用者 CLI 行為。缺點：依賴全域安裝版 `ticket`，可能與 repo 內程式碼不同步；測試慢且容易受 PATH 影響。前一輪 W17-013 已遇到全域 `ticket` 與 repo 內 parser 不一致，故不採用。

**選擇**：方案 A 為主。若實作時發現 `execute_runqueue()` 的直接 print 讓無結果訊息難以精準測試，允許在 `track_runqueue.py` 小幅抽出 `render_runqueue(args, version) -> str`，讓 `execute_runqueue()` 負責 print、`handoff.py` 負責嵌入同一段文字。

#### A（Attain distance）-- 機會成本是什麼？

- 預估成本：小型 IMP，約 1 個 helper + 2-3 個測試案例。
- 擠壓目標：W17 scheduler 後續整合工作（W17-027 group 其他 children）與 W17-033/035 文書任務。
- 不做代價：`handoff --auto` 仍只留下 JSON，不提示下一步；PM 在 /clear 或 session 切換前需額外想起 runqueue 查詢，W17-027 入口接線不完整。
- 結論：一次性小改動可降低每次 handoff 後的接手決策摩擦，回報高於成本。

#### P（Prepare to be wrong）-- 最可能失敗的原因是什麼？

- 最可能失敗原因：為了滿足「目前無待恢復 ticket」而在 `handoff.py` 重寫 runqueue 過濾/排序邏輯，導致與 `track_runqueue.py` 的 context=resume 行為漂移。
- 防護措施：實作時優先呼叫/抽用 `track_runqueue` 單一來源；測試需同時覆蓋「有 pending handoff 候選」與「無 pending handoff 候選」；不使用 subprocess 呼叫全域 `ticket`。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 實作細節草案

#### 1. `handoff.py` 新增 auto runqueue hint helper

建議新增 helper：

```python
def _print_auto_runqueue_hint(version: str) -> int:
    """在 handoff --auto 成功後輸出 resume context 的下一步候選。"""
```

行為：
- 只在 `_execute_auto_handoff()` 成功寫入 JSON 後呼叫。
- 輸出段落標題，例如：

```text

下一步候選（runqueue --context=resume --top 3）
------------------------------------------------------------
```

- 使用 `argparse.Namespace(format="list", top=3, context="resume", wave=None)` 呼叫 runqueue。
- 預設不讓 runqueue failure 影響 handoff JSON 建立結果；若 runqueue 內部錯誤，顯示 warning，但 `_execute_auto_handoff()` 仍回傳 0，因為核心 handoff 檔已建立。

#### 2. runqueue 復用方式

優先方案：
- 從 `ticket_system.commands.track_runqueue` import `execute_runqueue`。
- 在 helper 中呼叫 `execute_runqueue(ns, version)`，用 `capsys` 在測試層驗證 stdout。

若需要精準處理無結果訊息，允許小幅重構 `track_runqueue.py`：
- 新增 `render_runqueue(args, version) -> str`，內部包含目前 `execute_runqueue()` 的 list/filter/render 流程。
- `execute_runqueue()` 改為 `print(render_runqueue(args, version))`。
- `handoff.py` 呼叫 `render_runqueue()`，若結果含 context=resume 空集合，額外輸出 `目前無待恢復 ticket`，或由 `render_runqueue()` 支援該字串。

#### 3. 無結果訊息

Acceptance 要求明確顯示：

```text
目前無待恢復 ticket
```

觸發條件：
- `runqueue --context=resume --top 3` 經 pending handoff 交集與 `status=pending blockedBy=[]` 過濾後無 runnable ticket。
- 注意：剛建立的 handoff JSON 可能指向 `from_ticket_id`，但該 ticket 若不是 `pending` 或被 blocked，runqueue list 仍會為空；此時仍應顯示上方訊息，而不是只依 pending handoff 檔案存在判斷。

#### 4. 測試策略

更新 `.claude/skills/ticket/tests/test_handoff_auto.py`：

1. 更新既有成功測試：`_execute_auto_handoff()` stdout 除了 `HANDOFF_FILE_CREATED`，也包含 runqueue 建議段落標題。
2. 新增有候選測試：
   - 建立 auto handoff 來源 ticket。
   - 建立一張 `status=pending, blockedBy=[]` 且對應 pending handoff JSON 的 ticket（或讓 auto handoff 來源 ticket 本身為 pending，依實作選擇）。
   - 驗證 stdout 含 `runqueue --context=resume --top 3` 與候選 ticket id。
3. 新增無候選測試：
   - 建立 auto handoff 檔，但 ticket 狀態為 `in_progress` 或 `blockedBy` 非空。
   - 驗證 stdout 含 `目前無待恢復 ticket`。
4. 回歸測試：
   - `python3 -m pytest .claude/skills/ticket/tests/test_handoff_auto.py -q`
   - 若抽 `track_runqueue.render_runqueue()`，同步跑 `python3 -m pytest .claude/skills/ticket/tests/test_track_runqueue.py .claude/skills/ticket/ticket_system/tests/test_track_runqueue.py -q`

#### 5. 非目標

- 不新增新的 scheduler 排序規則。
- 不用 subprocess 呼叫全域 `ticket`。
- 不改 handoff JSON schema。
- 不改 manual handoff 流程，只影響 `--auto` 成功路徑。

### 實作完成摘要

落地方案採 WRAP 選定的「復用 runqueue 單一來源」：

1. `track_runqueue.py`
   - 新增 `render_runqueue(args, version) -> str`，承接原本 `execute_runqueue()` 內的 list/filter/render 流程。
   - `execute_runqueue()` 改為呼叫 `render_runqueue()` 後 print，保留原 CLI 行為。
   - unsupported format 改由 `render_runqueue()` raise `ValueError`，`execute_runqueue()` 捕捉後輸出 `[ERROR] ...` 並回傳 1。

2. `handoff.py`
   - 新增 `_print_auto_runqueue_hint(version)`。
   - `_execute_auto_handoff()` 成功建立 handoff JSON 後輸出：
     - `下一步候選（runqueue --context=resume --top 3）`
     - 同一份 `track_runqueue.render_runqueue()` 產出的 resume context 清單
   - 若 runqueue 輸出沒有候選項，額外顯示 `目前無待恢復 ticket`。
   - 不使用 subprocess，不依賴全域安裝版 `ticket`。

3. `test_handoff_auto.py`
   - 新增有候選測試：`status=pending` 的 auto handoff 來源 ticket 會出現在 stdout。
   - 新增無候選測試：auto handoff 來源 ticket 不是 runnable 時顯示 `目前無待恢復 ticket`。
   - 既有 JSON schema 與 manual handoff 測試維持通過。
