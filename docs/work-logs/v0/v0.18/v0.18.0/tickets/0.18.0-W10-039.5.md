### WRAP 完整分析

**W（Widen）**：
- 方案 A：判違規，將 W10-036.2 重組進 W10-036.1 內部（如 .1.5）
- 方案 B：判串行例外，補充 atomic-ticket「串行兄弟」模式說明（**採用**）
- 方案 C：判 Wave 分離，將 .2 移到下 Wave
- 方案 D：調整 ARCH-017 定義，移除「兄弟平行無依賴」絕對化表述（配套 B 一起做）

**R（Reality Test）**：見 Problem Analysis「重現實驗結果」章節——已驗證 ARCH-017 自身允許此模式。

**A（Attain Distance）**：
- 10 年後回看，這個分析結論的保存價值在於「方法論原則有例外但需明文」，而非具體的 W10-036 案例
- 機會成本：若選 A/C 會引入重組工作（~2h），且消除不了本質時序依賴
- 10 年後看 B 最清晰：正面承認「串行兄弟」為合法模式，避免後人重踩同樣模糊

**P（Prepare to be wrong）**：
- 最可能失敗：「串行兄弟」被濫用為所有兄弟依賴的藉口
- 防護：在方法論中明文列出串行兄弟的 4 條合法條件（單向、無環、規格→實作、不可深度化）
- 絆腳索：若發現新 Ticket 引用「串行兄弟」為理由但結構仍是 hub 發散，應視為違規

### 結論：選方案 B + D

**三選一分析結果**：
- (a) 違規：**否**。依賴不可透過深度化消除（Hook 非方法論子），亦無隱藏 hub。
- (b) 串行兄弟合法例外：**是**。屬「規格先於實作」單向時序，ARCH-017 自身範例已允許。
- (c) Wave 分離：**否**。強行分離會阻塞父任務當前 Wave 驗證，無效益。

### 串行兄弟合法模式的四條件

1. **單向依賴**：A → B，B 不依賴 A 以外的兄弟
2. **無循環**：兄弟依賴形成 DAG 不成環
3. **規格→實作時序**：A 為規格/介面/方法論，B 為實作/驗證/Hook
4. **不可深度化**：B 若成為 A 的子，會違反內容邊界（如 Hook 不是方法論的子章節）

四條件**全滿足**才算合法串行兄弟，否則回歸 ARCH-017 重組範式。
