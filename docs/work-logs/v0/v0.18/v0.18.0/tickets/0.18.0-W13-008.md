### 問題根因

Hook 系統在「設計即攔截」模式下，少數高頻 hook 觸發次數顯著高於平均，反映設計缺陷（規則寬鬆 / 觸發條件粗糙 / 邏輯漏洞）而非個案失誤。目前缺乏：

## Solution

### 推薦方案：E + F 組合（hybrid）— 修訂後

**核心原則**：監測機制應寄生於既有基礎設施，閾值採相對基線而非絕對魔數，輸出僅作 stderr 提醒而非自動建 ticket。**Why**：監測系統自身不應成為新的高頻 hook 或 ticket 噪音源（observability paradox）；絕對閾值在 hook 設計差異巨大的情境下必然產生分類碎片化（6 分類 = 設計問題的訊號）。**Consequence**：若採絕對閾值 + 自動建 ticket，會在 false positive 高的 hook 上暴增無人 claim 的 ticket，反向印證 W13-008 動機。**Action**：採方案 E（擴充既有 `hook-health-monitor.py`）+ 方案 F（CLI 子命令給 PM 手動觸發 / dry-run）。

### 推薦結論

1. **複用既有基礎設施（DRY）**：`.claude/hooks/hook-health-monitor.py` 已是 SessionStart hook，已動態解析 settings.json，已有 stderr 報告機制。**Consequence**：不複用即違反 saffron critical 發現；新建獨立 hook 將重複造輪。**Action**：擴充 `hook-health-monitor.py` 加入觸發頻率掃描（scope 從 SessionStart hooks 擴大為全部 hooks，屬合理擴充）。
2. **單點接入降低風險**：監測邏輯收斂於 1 個既有 hook + 1 個 CLI 子命令，避免方案 C 的 96 hook 全改面。**Consequence**：分散接入會讓監測本身成為新的攔截源。**Action**：核心邏輯 lib 化於 `.claude/hooks/lib/hook_health.py`，供 hook-health-monitor 與 ticket CLI 共用。
3. **相對基線取代絕對閾值**：以「自身 7 天均值的 N 倍（N=2）」作為主要判定，hook-type 僅作 2 類粗分（高頻可接受 vs 應低頻）。**Consequence**：絕對閾值（6 分類）在 hook 異質性高時無法穩定校準，且新增 hook 必須維護表格。**Action**：見下方「量化標準」章節。
4. **純 stderr 提醒，不自動建 ticket**：偵測命中時於 SessionStart 輸出 stderr 摘要，由 PM 評估後手動建 ANA。**Consequence**：自動建 ticket 路徑複雜度逼近方案 C 風險面（CLI subprocess + 去重 + 模板填料），且 false positive 直接污染 ticket 系統。**Action**：自動建 ticket 留待 quarterly review 證實價值後再評估（建追蹤 ticket 或於本 ticket 留 `<!-- PC-093-exempt: ticket-tracked:0.18.0-W13-008 quarterly review 後評估 -->` 標記）。
5. **Session 邊界改用 marker file**：由 hook-health-monitor 在 SessionStart 寫 `.claude/state/last-session-start.marker`，後續掃描讀取精確邊界，取代 timestamp gap heuristic。**Consequence**：timestamp gap heuristic 在 PM 長 session 或多 session 並行時系統性失準。**Action**：見下方「Session 邊界」章節。

### 否決方案

| 方案 | 否決理由 | 修正方向 |
|------|---------|---------|
| A. 純被動 log aggregator | 等同現狀（273M 日誌無人讀的延續） | → 採 E（寄生 SessionStart 主動呈現） |

| D. ML pattern detector | YAGNI；6 個 PC 印證的問題用閾值即可偵測 | → 採 E + F（相對基線閾值） |

### 量化標準（acceptance 2）— 修訂後

**原則先行**：頻率判定主依賴 hook **自身歷史基線**，hook-type 僅作 2 類粗分降低 false positive。**Why**：絕對閾值表格（6 分類）會隨新增 hook 持續膨脹，且每個分類的「合理範圍」缺乏客觀依據（憑設計者直覺）。**Consequence**：絕對閾值維護成本與 hook 數正相關，最終回到「無人維護的 273M 日誌」同構問題。**Action**：採相對基線 + 2 類權重。

### Hook 分類（2 類）

| 類別 | 描述 | 範例 | 基線倍率閾值 |
|------|------|------|------------|
| 高頻可接受 | PreToolUse 決策 / 認知品質類 | phase4-decision-enforcement / wrap-decision-tripwire | N=3（容忍尖峰） |
| 應低頻 | 其他全部（PostToolUse / SessionStart / charset / validator 等） | acceptance-gate / charset-guard / version-release-guard | N=2 |

### 觸發判定（相對基線）

對每個 hook：

1. 計算「過去 7 天每日觸發次數的均值」`baseline = sum(last_7_days) / 7`
2. 計算「最近 1 天觸發次數」`recent = today_count`
3. 觸發條件：`recent > baseline * N`（N 由分類決定）
4. 額外升級訊號（任一即標 critical）：
   - 連續 3 天 recent 持續超閾值
   - 該 hook 對應 ≥ 2 個 PC error-pattern 已記錄

**Bootstrap**：首次運行（無 7 天歷史）時退化為「絕對下界」`recent > 100 / day`（單一保守魔數，僅 bootstrap 用），記錄歷史滿 7 天後自動切換。

### Session 邊界

由 `hook-health-monitor.py` 在 SessionStart 寫入 `.claude/state/last-session-start.marker`（內容：ISO timestamp）。後續掃描比對檔案 mtime > marker timestamp 視為「本 session 觸發」。**Why**：timestamp gap > 1 小時的 heuristic 在長 session 或多 session 並行時失準。**Consequence**：誤判 session 邊界會讓單 session 統計不可信，污染相對基線。**Action**：marker file 由 hook-health-monitor 寫入，純讀取不需修改其他 hook。

### W — Widen Options（擴增選項）

PM 預設四方案（A/B/C/D）+ 自主擴增兩個方向：
