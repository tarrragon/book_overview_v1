### 2. 雙提案 Verdict

#### 提案 A（Ticket / PC RAG）：**不實施**

理由（good taste 三問）：

1. **資料結構已對**：ticket frontmatter（type / source_ticket / blockedBy / files）+ PC 五目錄分類 = structured index。RAG 是「對結構化資料再外掛 embedding」，特例（無向量也能查的場景）反而成為常規。
2. **特例膨脹**：embedding model 選型、vector DB 維護、re-embed 觸發、命中率監測、stale index 防護，全是新增複雜度；而當前痛點不是「找不到」而是 PC-066 / PC-105 類「找到但不讀 / 不依規則執行」——RAG 解錯問題。
3. **sledgehammer**：1289 ticket × 平均 5KB ≈ 6MB 純文字；ripgrep 全掃 < 200ms。引入 Python embedding stack（sentence-transformers / chromadb / 向量重建 pipeline）對應 100x 複雜度增量、0x 實際收益增量。

> 引述 Linus：「Bad programmers worry about the code. Good programmers worry about data structures.」既有 frontmatter + 目錄分類 = 對的資料結構；RAG 是在對的結構上再貼一層糟的結構。

#### 提案 B（Rules / PM-Rules RAG）：**不實施，改 L3 hook-driven 強化**

理由：

1. **失敗成本高 + 命中率要求 95%+**：規則漏 retrieve = PM 違規。RAG 命中率本質機率性；對「不可漏」場景使用機率工具是設計倒置。
2. **L1+L2+L3 混合已部分到位**：core/* 全載入（不可漏的底線）+ references/ lazy-load（按需細節）+ hook 情境提示（dispatch-strategy / claim-precheck 已在做）。剩下的工作不是「換 RAG」，是「補完 L3 trigger map」——這是修現有結構，不是換結構。
3. **20K 小 model 場景非主要訴求**：此專案 PM 跑在 1M context 模型；為假想小 model 引入 embedding stack 是 YAGNI（PC-093 模式）。

### 3. 推薦輕量替代方案（依優先序）

| 方案 | 投入 | 收益 | 採用建議 |
|------|------|------|---------|
| **A. 強化 ticket CLI 查詢**：新增 `ticket track search --type ANA --keyword "RAG" --recent 30d` 組合 flag | 低（既有 search 擴 flag） | PM 派發前 1 行即可拉相似 ticket | 推薦立即建 ticket |
| **B. PC 檢索 helper**：`/error-pattern query <keyword>` 改為支援多 keyword + 目錄過濾，輸出 top-5 path + 一行摘要 | 低（既有 skill 增強） | PC 命中率提升、無新依賴 | 推薦建 ticket |
| **C. Hook-driven rule preload trigger map**：claim 時依 ticket type / who 欄位 stdout 提示 `Read .claude/pm-rules/<sop>.md` | 中（trigger map 維護） | L3 補完，不引入 embedding | 推薦研究 |
