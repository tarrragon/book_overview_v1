## Problem Analysis

### 問題根因

Phase 4 多視角發現（linux P1 / bay TD-4/8 / 實作自述局限）：三個白名單規則的邊界參數為魔術常數，存在誤報/漏報風險。

**問題 1（規則 C）**：`_is_in_path_context` 用「向前/向後視窗內無空白」反向判斷路徑上下文。正例如 `.claude/xxx禁止` 的 before="xxx"（無空白）會被誤判為路徑內。應改為正向 token 驗證（`[\w/.\-]+$`）。

**問題 2（規則 B）**：`_has_negation_prefix` 視窗 10 字元過窄。中文常見句型「請不要在這個 ticket 裡面實作」否定詞到關鍵字距離超 10 字，漏判為真違規。

**問題 3（規則 D）**：`_is_meta_task_prompt` 視窗 100 字元為任意魔術常數。Ticket prompt 第一行常為 `Ticket: 0.18.0-...`（30 字），meta-task 描述可能在 200-500 字位置。

### 影響範圍

- `.claude/hooks/agent-dispatch-validation-hook.py::_has_negation_prefix`

- 相關常數：`_PATH_PREFIXES`、`_NEGATION_WORDS_ZH`、`_META_TASK_PATTERNS`

### 規則 C：路徑判斷改正向 token（AC1）

新增 `_PATH_TOKEN_CHARS`（ASCII 路徑字元集合，刻意排除 CJK，因 Python `\w` 預設涵蓋 CJK 會讓關鍵字誤判為路徑字元）與 `_is_path_token()` helper。
`_is_in_path_context` 改寫：
  1. Ticket ID 檢查為最優先
  2. 匹配字串本身必須為 `_is_path_token`（CJK 關鍵字直接拒絕）
  3. 前綴/副檔名 between 片段改用 `_is_path_token` 取代「無空白」反向判斷

效果：.claude/xxx禁止 的 禁止、.claude/foo-bar.md 禁止 的 禁止 不再誤判為路徑內容；原 ASCII 路徑片段識別不受影響。

### 規則 B：否定前綴視窗 10→20 + 逗號句子邊界（AC2）

新增常數 _NEGATION_WINDOW_SIZE = 20（取代 hardcoded 10）。
_SENTENCE_BOUNDARY_CHARS 新增 六 個標點（全形/半形逗號、驚嘆、問號）。
_has_negation_prefix 改用常數。

效果：中長距否定（14+ 字）現可識別；跨逗號仍不算；原跨句號測試保留。

### 規則 D：meta-task 視窗第一段 + 500 字 fallback（AC3）

_is_meta_task_prompt 改為：
  - prompt.split("\n\n", 1)[0] 取第一段
  - 若第一段 >= 100 字 → 用第一段
  - 否則 fallback prompt[:500]（涵蓋 Ticket: ID\n\n修改 .claude/rules/... 格式）

取代原 prompt[:100] 魔術常數。
效果：Ticket 標題段後第二段 meta-task 描述可正確命中。

    # 路徑延續
