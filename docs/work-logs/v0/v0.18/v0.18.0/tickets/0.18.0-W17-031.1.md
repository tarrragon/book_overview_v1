### 修復摘要

擴充 `track_runqueue.py` 讓 `--context=resume` 模式讀 handoff JSON 的 `exit_status.status` 並在 list 視圖以 tag 形式顯示，取代原 `blockedBy=[]` runnable 標記。

### 設計決策

1. **新增 `_get_pending_handoff_info()`**（取代純 ID set）：回傳 `Dict[str, Dict]`，保留完整 handoff 資料供 tag 解析。原 `_get_pending_handoff_ticket_ids()` 改為 thin wrapper（向後相容 `_apply_context_resume` 與測試）。
2. **新增 `_get_exit_status_tag(handoff_info)`**：純函式解析 `exit_status.status` 欄位，採 fail-open 設計：
   - handoff 為 None / 非 dict → None（不標籤）
   - `exit_status` 缺失 / 非 dict → None（fail-open，相容舊 handoff JSON）
   - status 為 `success` 或未知值 → None（不標籤）
   - status 為四類 (`needs_context` / `blocked` / `failed` / `partial_success`) → 回傳對應字串
3. **`Literal` 型別 + `_TAGGED_EXIT_STATUSES` 集合常數**：消除魔法字串、單一 source of truth，schema 演進時只改一處。
4. **`_render_list` 新增 `handoff_info` 參數**：僅 resume 模式注入 handoff dict；非 resume 模式 `handoff_info={}`，行為完全不變。tag 存在時 suffix 為 `[<status>]` 取代 `blockedBy=[]`，符合 AC#3「不顯示 runnable 標註」。
5. **`render_runqueue` 條件式載入**：`context == "resume"` 才呼叫 `_get_pending_handoff_info()`，避免非 resume 模式 IO 開銷。

### 認知負擔評估

- 新增函式：2 個（`_get_pending_handoff_info`、`_get_exit_status_tag`），各 < 20 行
- 函式分支深度：`_get_exit_status_tag` 4 個 guard clause（扁平結構，深度 1）
- 命名：tag 字串與 `ExitStatusTag` Literal 對齊 W17-010 schema 命名
