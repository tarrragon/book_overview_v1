### 問題根因

**直接原因**：ticket skill 源碼（`.claude/skills/ticket/ticket_system/`）發生行為變更（子命令名稱更換、flag 必填性改變、complete 條件調整等）後，相關決策層文件（`.claude/pm-rules/decision-tree.md`、`pm-rules/*.md`、`.claude/skills/ticket/SKILL.md`）缺乏結構化的同步機制，仰賴 PM 自律補償，補償往往滯後。

**深層原因**：memory feedback「ticket skill 行為變更需同步檢查決策樹」已沉澱原則，但僅作為一條 memory 條目缺乏執行框架。無觸發條件定義（何時掃描）、無掃描清單（掃描哪些文件）、無豁免定義（哪些改動不需掃描），導致規則無法自動套用。

### 規則設計決策

**放置位置**：建新檔 `.claude/rules/core/ticket-skill-sync-check.md`（自動載入，PM 與所有代理人皆生效），不併入 `decision-trigger-binding.md`（避免與決策狀態合法性規則混合，職責清晰分離）。

**規則結構**：

| 規則 | 說明 |
|------|------|
| 規則 1 | 行為變更必須觸發同步掃描（含三明示 Why/Consequence/Action + 四步掃描指令） |
| 規則 2 | 重大行為變更範例表（5 個歷史案例）作為判別參考 |
| 規則 3 | 純修復型改動豁免（bug fix / 格式調整不觸發全量掃描） |

**三明示結構覆蓋**：

- **Why**：ticket skill 是 PM 操作 ticket 的唯一介面；命令語意改變但文件不更新會導致靜默失效
- **Consequence**：跳過掃描讓決策層累積過時引用；補償成本隨時間遞增
- **Action**：四步掃描指令（git diff → grep SKILL.md → grep decision-tree → grep -rln pm-rules）

**掃描目標清單覆蓋 acceptance AC-2**：

| 目標 | AC-2 對應 |
|------|---------|
| `.claude/skills/ticket/SKILL.md` | SKILL.md |
| `.claude/pm-rules/decision-tree.md` | decision-tree.md |
| `.claude/pm-rules/*.md` | pm-rules/*.md |

- PC-118 撰寫（W17-115.1）：獨立 DOC Ticket，互不阻塞
- commit-level hook（W17-115.3）：獨立 IMP Ticket，本規則為人工自律層，hook 為機制層兜底
