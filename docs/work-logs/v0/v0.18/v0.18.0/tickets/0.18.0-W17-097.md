# Execution Log

## Task Summary

分析 ticket CLI status 過濾系統性缺口（TERMINAL_STATUSES 採用率 1/7）與重構策略

---

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W14-004 引入 `TERMINAL_STATUSES = frozenset({STATUS_COMPLETED, STATUS_CLOSED})` 作為終止狀態的單一真實來源，但僅覆蓋 `lifecycle.py` 的 spawned/children 終結檢查（line 1328）。其餘所有 status 過濾場景仍 hard-code `!= STATUS_COMPLETED`，未考慮 `closed` 狀態。

直接後果：`closed` 狀態的 ticket 在 board、relations、query、stuck_anas 等視圖中被當作「未完成」顯示，造成假象（W17-011.2 P0 假象事件）。

### 影響範圍

共 12 個檔案、26+ 個過濾點涉及 status 過濾邏輯。原始 ticket 列 5 個缺口點，實際掃描發現遠多於此。chain_analyzer.py（8 點）和 handoff.py（3 點）是最大遺漏。

### 相關 Error Pattern

無直接對應的 PC error-pattern。本缺口屬於 W14-004 設計引入但未完整推廣的常數採用率不足系統性問題。

---

## 重現實驗結果

### 實驗方法

使用 `rg -n '!= STATUS_COMPLETED|!= "completed"|== "completed"|== STATUS_COMPLETED' ticket_system/ -g '*.py'` 掃描全 codebase status 過濾點，逐一判斷語意分類。

### 實驗執行

1. grep 掃描得 26+ 個 status 過濾點跨 12 個檔案
2. 逐一讀取每個檔案的上下文（前後 5-10 行），判斷該過濾點的業務語意
3. 分類為 A（terminal）/ B（success-only）/ C（分組展示）/ D（from_status 語意不同）四類
4. 交叉比對 TERMINAL_STATUSES 的實際 import/使用位置，確認僅 lifecycle.py:1328 使用

### 實驗發現

已驗證事實：
- TERMINAL_STATUSES 定義於 constants.py:129，僅被 lifecycle.py:1328 使用（採用率 1/7 檔案）
- 19 個過濾點應改用 TERMINAL_STATUSES（A 類）
- 7 個過濾點語意正確無需改（B 類，success-only）
- 1 個分組展示缺口（C 類，track_relations.py）
- 2 個 from_status 語意不同無需改（D 類）
- lifecycle.py:1092 和 checkpoint_state.py:427 使用 hard-code 字串 `"completed"` 而非常數

### Step 2: 常數設計建議

**推薦方案：維持現有 TERMINAL_STATUSES，不新增 ACTIVE_STATUSES**

理由：
1. 現有 `TERMINAL_STATUSES = {completed, closed}` 語意清晰，19 個過濾點全部可用 `in/not in TERMINAL_STATUSES` 表達
2. `ACTIVE_STATUSES = {pending, in_progress, blocked}` 是 TERMINAL 的補集，新增只增加維護負擔無額外語意價值
3. `superseded` 在語意上屬 terminal（任務已結案），但目前未加入 TERMINAL_STATUSES。建議本次不擴充（superseded 票量極少且有獨立語意），但在 track_relations.py 分組中加入

常數層唯一變更：在 constants.py 的 `__all__` 確認 TERMINAL_STATUSES 已 export（已存在，無需變更）。

### Step 3: Lint/grep 防護方案

**推薦方案：grep-based 單元測試**

| 層級 | 機制 | 說明 |
|------|------|------|
| 單元測試 | `test_no_bare_status_completed_filter.py` | grep 掃描確認無新增 bare `!= STATUS_COMPLETED` 過濾 |
| 允許清單 | success-only 語意的檔案:行 加入 allowlist | 避免誤報（B 類場景） |

不建議用 AST 分析（過度工程）或 CI（本專案無 CI pipeline）。
