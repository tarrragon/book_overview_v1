### 2. dispatch_tracker.py 變更虛擬碼

#### 2.1 record_dispatch 修改

```
FUNCTION record_dispatch(project_root, agent_description, tool_use_id, ticket_id="", files=[], branch_name="", agent_id=null):
  // tool_use_id 為必填（Phase 2 T1.1.2 驗證缺失拋錯）
  ACQUIRE state_lock(project_root)
    state = read_state(project_root)
    entry = {
      "agent_description": agent_description,
      "tool_use_id": tool_use_id,        // 新增：必填
      "agent_id": agent_id,              // 新增：可選，預設 null
      "ticket_id": ticket_id,
      "files": files,
      "branch_name": branch_name,
      "dispatched_at": now_utc_iso()
    }
    APPEND entry TO state.dispatches
    write_state(project_root, state)
  RELEASE lock
```

#### 2.2 update_dispatch_agent_id 新增

```
FUNCTION update_dispatch_agent_id(project_root, tool_use_id, agent_id) -> bool:
  ACQUIRE state_lock(project_root)
    state = read_state(project_root)
    matched = false
    FOR EACH entry IN state.dispatches:
      IF entry.tool_use_id == tool_use_id:
        IF entry.agent_id is not null AND entry.agent_id != agent_id:
          LOG warning "覆寫 agent_id"
        entry.agent_id = agent_id
        matched = true
        BREAK  // tool_use_id 應唯一
    IF matched:
      write_state(project_root, state)
    RETURN matched
  RELEASE lock
```

#### 2.3 clear_dispatch_by_id 新增

```
FUNCTION clear_dispatch_by_id(project_root, agent_id) -> bool:
  ACQUIRE state_lock(project_root)
    state = read_state(project_root)
    original_count = LENGTH(state.dispatches)
    // 過濾掉所有匹配 agent_id 的記錄（防禦重複）
    state.dispatches = [d FOR d IN state.dispatches IF d.agent_id != agent_id]
    removed = original_count - LENGTH(state.dispatches)
    IF removed > 1:
      LOG warning "agent_id 重複匹配 {removed} 筆"
    IF removed > 0:
      write_state(project_root, state)
      RETURN true
    RETURN false
  RELEASE lock
```

#### 2.4 clear_dispatch_by_description_fallback 新增

```
FUNCTION clear_dispatch_by_description_fallback(project_root, description) -> bool:
  // 清理 description 匹配且 dispatched_at 最早的一筆
  ACQUIRE state_lock(project_root)
    state = read_state(project_root)
    candidates = [d FOR d IN state.dispatches IF d.agent_description == description]
    IF candidates is empty:
      RETURN false
    // 按 dispatched_at 升序排列，取最早
    oldest = SORT(candidates, key=dispatched_at)[0]
    REMOVE oldest FROM state.dispatches
    write_state(project_root, state)
    RETURN true
  RELEASE lock
```

### 5. 技術決策

**決策 1：SubagentStop 新建獨立 hook 檔案**

理由：active-dispatch-tracker-hook.py 已有 PostToolUse 邏輯（補 agent_id + housekeeping）。SubagentStop 職責不同（清理 + 廣播），混在同一檔案需用 event 判斷分支，增加認知負擔。新建 `subagent-stop-dispatch-cleanup-hook.py` 職責單一、測試獨立。

**決策 2：SubagentStop fallback 不依賴 description（因 input 無此欄位）**

SubagentStop input schema 僅有 agent_id、agent_type、agent_transcript_path、last_assistant_message，無 description。Fallback 策略改為：清理 dispatches 中 agent_id=null 且 dispatched_at 最早的一筆（FIFO 語義）。這比 description 匹配更簡單且不依賴外部欄位。

Phase 2 T2.4.3 測試驗證此路徑。clear_dispatch_by_description_fallback 仍保留作為 lib API（未來可能有其他 caller），但 SubagentStop handler 不使用。

**決策 3：PostToolUse update_agent_id 找不到 entry 記 debug（非 warning）**

Q2 實證：前台模式 SubagentStop 先到 0.615s，PostToolUse 遲到時 entry 已清。這是正常現象（T3.3.1 覆蓋），不應產生 warning 噪音。

### 行為鏈式推演（重點場景）

#### 場景：Background 代理人完成清理（T3.1.1）

- Given: project_root 存在且 dispatch-active.json 為空

#### 場景：Race — SubagentStop 先到（T3.3.1）
