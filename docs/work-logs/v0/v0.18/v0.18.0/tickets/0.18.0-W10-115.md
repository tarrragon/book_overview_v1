#### 2. 核心函式虛擬碼

##### 2.1 `_normalize_priority(value) -> str`

```
INPUT: value (任意型別，來自 ticket.get("priority"))
OUTPUT: "P0" | "P1" | "P2" | "P3" | "P9"

IF value 不是 string THEN return "P9"
strip value 兩端空白後 uppercase
IF result IN {"P0", "P1", "P2", "P3"} THEN return result
return "P9"
```

##### 2.2 `_sort_tickets_by_priority(tickets) -> sorted list`

```
INPUT: list[dict]（每個 dict 含 priority / created / id 鍵，皆可能缺失）
OUTPUT: list[dict] 依三階層排序後的新 list

DEFINE sort_key(t):
  p_norm = _normalize_priority(t.get("priority"))
  created = t.get("created") or ""   # 缺值排最前？依 spec「同 priority 內舊者在前」，空字串字典序最小，自然排最前，符合 D2
  tid = t.get("id") or t.get("ticket_id") or ""
  return (p_norm, created, tid)
  # tuple 比較自動實現三階層：(P0,P1,P2,P3,P9) 字典序遞增即「P0 在前」

return sorted(tickets, key=sort_key)
```

關鍵設計：
- P0/P1/P2/P3/P9 字典序遞增等於 priority 高到低，無需 mapping table
- 純函式無副作用，回傳新 list，方便 D 群組單元測試
- 缺欄位用 `.get()` + 預設值統一處理，無 KeyError 風險

##### 2.3 `_resolve_top_conflict(args) -> (effective_top, warning_msg or None)`

```
INPUT: argparse.Namespace（含 args.top, args.all）
OUTPUT: (int_or_None, optional_warning)

IF args.all == True:
  IF args.top is explicitly set (非預設):
    return (None, "[WARN] --all overrides --top")
  return (None, None)
ELSE:
  return (args.top, None)   # args.top 預設 10
```

「explicitly set」判別方式：argparse 預設值改用 sentinel（如 `DEFAULT_TOP = 10`），檢查 `args.top != DEFAULT_TOP` 或檢查 `args.top is not None` 並把 default 設為 None；推薦後者，由 helper 補回 10 預設。

##### 2.4 `_apply_top_limit(tickets, top) -> limited list`

```
INPUT: tickets (sorted list), top (int or None)
OUTPUT: list

IF top is None: return tickets    # --all 路徑
IF top == 0: return []
return tickets[:top]
```

##### 2.5 list 主邏輯整合（在 `_execute_list_single_version` / `_all_versions` / `_cross_version` 三處）

```
... 既有篩選 (status, wave) ...

# --- Phase 3a 新增段 ---
```

#### 3. argparse 結構

於 `_register_query_commands()` list parser（行 430-439）追加：

```
DEFINE _parse_top_arg(value: str) -> int:
  try: n = int(value)
  except ValueError: argparse.ArgumentTypeError("--top must be integer")
  IF n < 0: raise argparse.ArgumentTypeError("--top must be >= 0")
  return n

p_list.add_argument("--top", type=_parse_top_arg, default=None,
                    help="限制輸出最多 N 筆（預設 10），依 priority 排序後取前 N")
p_list.add_argument("--all", action="store_true",
                    help="取得全量輸出（覆蓋 --top）")
```

註解設計：`default=None` 允許 `_resolve_top_conflict` 區分「用戶未指定」vs「用戶顯式指定 10」；最終預設 10 由 `_resolve_top_conflict` 末段補：
```
IF args.all: return (None, ...)
return (args.top if args.top is not None else 10, None)
```

衝突 warning 寫入 `sys.stderr.write(msg + "\n")`，不影響 stdout 內容（TC C3 用 capsys.readouterr() 分別檢查 out/err）。

`--top must be >= 0` 由 argparse.ArgumentTypeError 自動轉換為 exit 2（argparse 既有行為），不需手動 sys.exit。

## Phase 1 — Feature Design Specification

### 1. CLI 介面變更

#### 1.1 新增 flag

| Flag | 型別 | 預設值 | 說明 |
|------|------|--------|------|
| `--top N` | int | `10` | 限制輸出最多 N 筆，依 priority 排序後取前 N 筆 |
| `--all` | bool | `False` | 取得全量輸出，等同關閉 `--top` 限制 |

#### 1.2 排序規則（預設 priority 排序）

| 階層 | 排序鍵 | 規則 |
|------|--------|------|
| 1（主鍵） | priority | P0 > P1 > P2 > P3（高優先級在前） |
| 2（次鍵） | created | 較舊者在前（FIFO，priority tie-break） |
| 3（末鍵） | id | 字典序（最終 stable tie-break） |

priority 缺失（空字串 / None / 非 P0-P3 值）的 ticket 視為 P9（排在最末）。

### 2. 既有 flag 互動

| 既有 flag | 與 --top / --all 互動 |
|-----------|----------------------|
| `--status` | 先依 status 篩選，再排序後套 `--top` 取前 N |
| `--wave` | 同上，先 wave 篩選再排序限制 |
| `--format` (table/json/list) | 不影響取數邏輯，只影響輸出格式 |
| `--limit`（若既有） | 若同時存在，`--top` 優先；建議 deprecation note |

執行順序：篩選（status/wave）→ 排序（priority/created/id）→ 限制（top/all）→ 格式化輸出。

### 3. 衝突與邊界處理

| 情境 | 行為 |
|------|------|
| `--all` 與 `--top N` 同時指定 | `--all` 優先（取全量），stderr 輸出 warning：`[WARN] --all overrides --top` |
| `--top 0` | 輸出空集合 + header（不報錯） |
| `--top N`（N < 0） | 報錯 exit 2：`--top must be >= 0` |
| 篩選後結果 < N | 輸出實際筆數，不補齊不報錯 |
| 無 ticket 符合篩選 | 輸出 header + 空 body（既有行為延續） |
| priority 欄位缺失 | 視為 P9 排末位（不報錯，向後相容歷史 ticket） |

### 4. 輸入/輸出範例

#### 4.1 預設行為（無 flag）

輸入：`ticket track list --status pending,in_progress`
輸出：最多 10 筆，依 P0 → P3 排序，table 格式（既有預設）。

#### 4.2 指定 top

輸入：`ticket track list --status pending --top 5`
輸出：最多 5 筆 pending ticket，priority 排序。

#### 4.3 全量輸出（向後相容）

輸入：`ticket track list --status pending,in_progress --all`
輸出：所有符合的 ticket（等同 W10-113 觀察到的 67 筆 dump 行為）。

#### 4.4 同 priority 內排序

輸入：`ticket track list --top 20`，假設有 5 筆 P1（created 不同）
輸出：5 筆 P1 依 created 從舊到新排列，再接 P2 筆數。

#### 4.5 priority 缺失

輸入：包含 1 筆無 priority 欄位的舊 ticket
輸出：該 ticket 排在所有 P0-P3 之後（視為 P9）。

### 5. 與 dashboard 命令的功能邊界

| 維度 | `ticket track list` | `ticket track dashboard` |
|------|---------------------|--------------------------|
| 定位 | 單一篩選 + 排序視圖 | 聚合視圖（多區塊） |
| 篩選 | 依 `--status` / `--wave` 任意組合 | 固定 in_progress + ready + stale |
| 排序 | 預設 priority，可未來擴充 | priority + readiness 內建 |
| 預設筆數 | top 10（本 ticket 新增） | top N ready（W10-114 已落地） |
| 用途 | PM 精確查詢特定 status / wave | PM 快速接手總覽 |

兩命令並存，list 不取代 dashboard。
