### Context Bundle（PC-040：PM 派發前分析輸入）

#### 問題根因（來自 PC-073）

PM 執行 ANA 類型 Ticket 完成後，為調查結論建立後續防護 IMP 時：
1. 只能用 `ticket create --parent {ANA-ID}`（CLI 唯一關聯參數）
2. CLI 自動設定為 ANA 的 `children`（父子關係）而非 `spawned_tickets`（衍生關係）
3. PM 執行 `ticket track complete {ANA-ID}` 時被 acceptance-gate-hook 擋下
4. 必須手動 Edit 5+ 個檔案清除 parent_id/children、填 source_ticket 才能 complete

#### 概念模型區分

- 父子關係（children/parent_id）：必須一起交付；子未完成父無法 complete
- 衍生關係（spawned_tickets/source_ticket）：獨立交付時機；衍生項不阻擋 source complete

#### 既有現況

`ticket create --help` 目前關聯參數僅有三個：
- `--parent`（父子）

#### 1. 執行流程虛擬碼（execute 擴充）

```
函式 execute(args):
    # Step 0：版本解析（既有）
    version = resolve_version(args.version)
    若 version 無效 → 回傳 1

    若版本未在 todolist.yaml 註冊 → 輸出錯誤並回傳 1

    # Step 1：Ticket ID 與 wave 解析（既有）
    resolved = _resolve_ticket_id_and_wave(args, version)
    若 resolved 為 None → 回傳 1
    (version, ticket_id, wave) = resolved

    ticket_type = args.type 或 "IMP"
    tdd_result = suggest_tdd_sequence(ticket_type)

    # Step 1.5（新增）：source-ticket 前置驗證
    # 設計原則：所有錯誤路徑在持久化前結束；fail-fast 順序一致

        source_version = extract_version_from_ticket_id(args.source_ticket)

        若 source_ticket 為 None:
            輸出 CreateMessages.SOURCE_TICKET_NOT_FOUND（帶引導「ticket track list」）
            回傳 1

        # 子步驟 4：狀態警告（allow + warning；不阻擋）
        若 source_ticket.status == "completed":
            輸出 CreateMessages.SOURCE_TICKET_COMPLETED_WARN
            # 非 ANA type 無額外警告（決策 §8：消除特例）

    # Step 2：CLI 參數轉 TicketConfig（擴充新欄位）
    config = _parse_cli_args_to_config(args, version, ticket_id, wave, tdd_result)
    # _parse_cli_args_to_config 內部新增 source_ticket=args.source_ticket
    若 config 為 None → 回傳 1

    # Step 3：驗證 + 持久化 + 回報（擴充關係層）
    回傳 _persist_and_report(args, config, version, ticket_id, tdd_result)

函式 update_source_spawned_tickets(source_ticket_id, new_ticket_id):

    source = load_ticket(resolved_version, source_ticket_id)
```

#### 2. Helper 抽取決策（linux #1 N=2 DRY 邊界判斷）

**結論：不抽 `_append_to_ticket_list_field` helper，本 Ticket 僅複寫現有模式**

**論據對照**：

| 維度 | 抽 helper 論據 | 不抽 helper 論據 |
|------|--------------|--------------|
| DRY | 兩函式結構同形（extract_version → load → normalize → if not in: append + save），差異僅欄位名 | N=2 是 Rule of Three 的警戒線非觸發線；第三次出現時才抽 |
| 測試負擔 | 抽 helper 要多寫一層單元測試 + 兩個 wrapper 測試 | 只需為新函式寫 6 個測試（A1-A6），可直接覆蓋並行業務場景 |
| 回報語義 | helper 返回 True/False 需由 caller 翻譯為「parent 失敗」vs「source 失敗」訊息 | 各函式直接印 caller 所需訊息（但實際上目前訊息 print 在 caller 層，不影響） |
| 既有風險 | 抽 helper 會動到 `update_parent_children`（已穩定、被多測試覆蓋）→ 擴大本 Ticket 變更面 | `update_parent_children` 零變動，本 Ticket 風險限縮在新函式 |
| 未來擴展 | 若 W10-040 / migrate / fields 增第三 list field，helper 免再複寫 | 第三次出現時再抽，屆時有三個具體 call site 做為 helper 設計依據，避免 N=2 推測出的抽象失準 |

**最終決策**：**不抽**（對齊 linux 自己的原評語「不是強制要求，是 good taste 建議」與 Rule of Three 原則）。
**延後追蹤**：若本版本內出現第三個 list-field 追加場景（例如 W10-040 實作），於當次 PR 執行 refactor。記在本 Ticket Phase 4 評估備忘即可，不另建 Ticket。

**_normalize 處理**：`update_source_spawned_tickets` 內可**直接重用 `_normalize_children`**（它的語義是「將任意輸入正規化為 list[str]」，無 children 特化邏輯）。不需改名。若 thyme 覺得命名過於 children 導向，可作為 Phase 4 的純命名 refactor（改為 `_normalize_list_field`）不納入本 Phase。

#### 3. Argparse 互斥策略選擇

**結論：採選項 B（執行層顯式檢查）**

**論據對照**：

| 面向 | A：argparse mutually_exclusive_group | B：執行層 if-check |
|------|-------------------------------------|-------------------|
| 訊息可控 | argparse 預設訊息為英文「not allowed with argument」；需自訂須經 `formatter_class` 或 override error() | 直接用 CreateMessages.SOURCE_PARENT_MUTUALLY_EXCLUSIVE 多行中文訊息含 PC-073 引用 |
| 檢查順序 | argparse 在 parse_args() 階段立即 fail，無法插入其他檢查之前/之後 | 可自由排序：互斥 → 格式 → 存在 → 狀態（測試 B4 要求） |
| Exit code | argparse 預設 exit 2（系統錯誤），與 CLI 其他 error path exit 1 不一致 | 顯式 return 1，與既有 VERSION_NOT_DETECTED / PARENT_UPDATE_FAILED 等一致 |
| 測試覆蓋 | 需 capture SystemExit + 解析 stderr；pytest 需 `pytest.raises(SystemExit)` pattern | 直接 assert return code 與 stdout 訊息（重用既有 test 模式） |
| 程式碼量 | `parser.add_mutually_exclusive_group()` 1 行定義 | 3-4 行 if-check（可讀性高） |
| 可發現性 | `--help` 自動標註互斥 | 需手動在 help 文字加「與 --parent 互斥」說明（已在 §2.1 規格明列） |

**決策依據**：三視角共識「fail-fast 檢查順序」與「失敗訊息可讀」兩項共同指向選項 B。選項 A 的 argparse 原生支援看似優雅，但訊息控制代價高於省下的 3 行 if-check。

**實作提示（給 thyme）**：
- 互斥檢查放 `execute()` 本體，在 Step 1.5 的最前段（載入任何 ticket 之前）
- 訊息常數 SOURCE_PARENT_MUTUALLY_EXCLUSIVE 使用多行 format string（見 §2.5 規格）
- help 文字顯式標註「與 --parent 互斥」（測試 C2 驗證）

#### 1. Purpose（目的）

**問題背景**：`ticket create` CLI 目前僅提供 `--parent` 建立父子關係（children）。當 PM 基於 ANA Ticket 的調查結論建立後續防護 IMP 時，實際需要的是**衍生關係**（spawned_tickets / source_ticket）——衍生項有獨立交付時機，不應阻擋 source 的 complete。但 CLI 無對應參數，PM 只能先用 `--parent` 建立，再手動 Edit 5+ 個 ticket frontmatter 檔案清除 parent_id/children、填入 source_ticket（見 PC-073）。

**目標用戶**：PM 執行 ANA Ticket 完成流程、需建立衍生 IMP/ADJ 的 session。

**核心價值**：從源頭修補 CLI 能力缺口，讓 ANA 衍生 IMP 的 spawned 關係一步到位；避免事後手動 Edit、避免 acceptance-gate-hook 誤擋 ANA complete；與 W12-005（CLI complete 層 blocking confirmation）形成「前因 × 後果」完整防護鏈。

```python
def update_source_spawned_tickets(
    source_ticket_id: str,
    new_ticket_id: str,
) -> bool:
    """更新 source Ticket 的 spawned_tickets 欄位（append + 去重）。

    Args:
        source_ticket_id: source Ticket ID
        new_ticket_id: 新建立的衍生 Ticket ID

    Returns:
        bool: 成功更新返回 True，失敗返回 False

    Implementation:
        1. 從 source_ticket_id 提取版本號
        2. 載入 source Ticket
        3. 若 source 不存在，返回 False
        4. 取得 spawned_tickets 欄位，確保為 list 型別（防禦性：字串→list）
        5. 若 new_ticket_id 不在清單中，append
        6. 儲存 source Ticket
        7. 返回 True

    Side Effects:
        - 修改 source Ticket 檔案
        - 更新 source["spawned_tickets"] 清單
    """
```

#### 3. GWT Scenarios（行為場景）

##### 場景 1：正常 spawned 建立（happy path）

- **Given**: source Ticket `0.18.0-W12-002`（type=ANA, status=completed）存在，其 `spawned_tickets: []`
- **When**: PM 執行 `ticket create --action 實作 --target 'CLI 增強' --wave 12 --type IMP --source-ticket 0.18.0-W12-002 --why "PC-073 防護"`
- **Then**:
  - 新 Ticket 建立成功，frontmatter 含 `source_ticket: 0.18.0-W12-002`、`parent_id: null`、`children: []`
  - source Ticket 的 `spawned_tickets` 更新為 `[<新 ID>]`
  - stdout 含 `[INFO] 已雙向關聯：source=0.18.0-W12-002 新增 spawned=<新 ID>`
  - exit code = 0

##### 場景 2：source 不存在（error path）

- **Given**: source Ticket `0.18.0-W99-999` 不存在
- **When**: PM 執行 `ticket create ... --source-ticket 0.18.0-W99-999 ...`
- **Then**:
  - 不建立新 Ticket
  - stderr/stdout 含 `[ERROR] --source-ticket 指定的 0.18.0-W99-999 不存在，請以 ticket track list 確認 ID`
  - exit code = 1

##### 場景 3：--source-ticket 與 --parent 同用（error path）

- **Given**: source Ticket `0.18.0-W12-002` 與 parent Ticket `0.18.0-W12-001` 皆存在
- **When**: PM 同時提供 `--parent 0.18.0-W12-001 --source-ticket 0.18.0-W12-002`
- **Then**:
  - 不建立新 Ticket
  - stderr/stdout 含互斥說明訊息（引 PC-073）
  - exit code = 1
  - **互斥檢查先於 source 存在性檢查**（fail-fast）

##### 場景 4：source ticket ID 格式無效（error path）

- **Given**: 提供 `--source-ticket invalid-format`
- **When**: CLI 執行 ID 驗證
- **Then**:
  - 沿用既有 `validate_ticket_id` 邏輯，輸出 `[Error] 無效的 Ticket ID 格式: invalid-format`
  - 不建立新 Ticket
  - exit code = 1

##### 場景 5：source 已 completed（warning path，允許建立）

- **Given**: source Ticket `0.18.0-W12-002` 存在且 status=completed
- **When**: PM 執行 `ticket create ... --source-ticket 0.18.0-W12-002 ...`
- **Then**:
  - 新 Ticket 建立成功（不阻擋）
  - source Ticket 的 `spawned_tickets` 更新
  - stdout 含 `[WARNING] source 0.18.0-W12-002 狀態為 completed，仍可建立衍生 Ticket...`
  - stdout 含 `[INFO] 已雙向關聯...`
  - exit code = 0
  - 理由：spawned 關係的語義正是「source 可先 complete，衍生項獨立排程」

##### 場景 6：source 為非 ANA type（允許建立，不特別警告）

- **Given**: source Ticket `0.18.0-W12-005`（type=IMP, status=completed）
- **When**: PM 執行 `ticket create ... --source-ticket 0.18.0-W12-005 ...`
- **Then**:
  - 新 Ticket 建立成功
  - source 的 spawned_tickets 更新
  - 無 type 相關警告
  - exit code = 0
  - 理由：spawned 關係本身不限 type；spawned 常見於 ANA 衍生 IMP，但 IMP 衍生後續 IMP 也合法（如驗證中發現的後續工作）

##### 場景 7：與 --blocked-by / --related-to 共存（allowed）

- **Given**: source `0.18.0-W12-002` 存在；blocked-by `0.18.0-W12-005` 存在；related-to `0.18.0-W12-003` 存在
- **When**: PM 執行 `ticket create ... --source-ticket 0.18.0-W12-002 --blocked-by 0.18.0-W12-005 --related-to 0.18.0-W12-003 ...`
- **Then**:
  - 新 Ticket 建立成功，四欄位正確寫入：`source_ticket`、`blockedBy`、`relatedTo`；`parent_id: null`
  - source 的 spawned_tickets 更新
  - exit code = 0

##### 場景 8：update_source_spawned_tickets 失敗的容錯（edge case）

- **Given**: source Ticket 存在且 ID 格式有效，但在新 Ticket 寫入後、更新 source 時發生 IO 錯誤（例如檔案權限問題）
- **When**: create 流程進行到關係層
- **Then**:
  - 新 Ticket 已持久化（不回滾，符合既有 `update_parent_children` 行為）
  - stdout 含 `[WARNING] 新 Ticket 已建立，但更新 source {source_id}.spawned_tickets 失敗，請手動檢查`
  - exit code = 0（與 parent 更新失敗行為一致）
  - 後續報告章節仍正常輸出

##### 場景 9：source 為自身 ID（error path）

- **Given**: 尚未建立的新 Ticket ID 將為 `0.18.0-W12-010`
- **When**: PM 誤打 `--source-ticket 0.18.0-W12-010`
- **Then**:
  - 存在性檢查自然 fail（因為自己還未建立）
  - 輸出 `SOURCE_TICKET_NOT_FOUND`
  - exit code = 1
  - 理由：無需額外「自我引用」檢查，存在性檢查已涵蓋

#### 4. Error Handling（錯誤處理）

| 錯誤情境 | 檢查順序 | 層級 | 回傳值 | 訊息 |
|---------|---------|------|--------|------|
| `--source-ticket` ID 格式無效 | 1 | CLI parse 後、解析前 | exit 1 | `INVALID_TICKET_ID_FORMAT` |
| `--source-ticket` 與 `--parent` 同用 | 2 | 持久化前 | exit 1 | `SOURCE_PARENT_MUTUALLY_EXCLUSIVE` |
| source Ticket 不存在 | 3 | 持久化前 | exit 1 | `SOURCE_TICKET_NOT_FOUND` |
| source 已 completed | 4 | 持久化前（warning only） | 建立成功 | `SOURCE_TICKET_COMPLETED_WARN` |
| 新 Ticket 寫入失敗 | 5 | 持久化層 | 沿用既有行為 | 沿用既有訊息 |
| update_source_spawned_tickets 失敗 | 6 | 關係層（容錯） | exit 0 + warning | `SOURCE_UPDATE_FAILED` |

**錯誤處理原則**：
1. **fail-fast**：互斥/存在性檢查先於任何檔案寫入
2. **warning 不阻擋**：source 狀態相關為 warning，寫入與否交給 PM 判斷
3. **關係層失敗不回滾新 ticket**：與 `update_parent_children` 的既有行為對齊，避免引入新的事務語義
4. **訊息引用 PC-073**：互斥訊息必須指引用戶理解父子 vs 衍生的語義差異
