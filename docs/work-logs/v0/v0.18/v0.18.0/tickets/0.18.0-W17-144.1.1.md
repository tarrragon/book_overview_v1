## Task Summary

評估 charset-pollution-detector.py 的 OpenCC 依賴是否需補雙權威源（教育部常用國字標準字體表 + Unicode Han Database）以提升健壯性

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W17-144.1 落地的 charset-pollution-detector.py 把 OpenCC 當「簡繁判定權威」，但用戶 WRAP 質疑後實證 OpenCC 對台灣標準字 30% 誤判率：

| 字 | OpenCC s2t 判斷 | 實際 |
|----|---------------|------|
| 台 (U+53F0) | → 臺（誤判為簡體） | 台灣行政院公文標準字 |
| 群 (U+7FA4) | → 羣（誤判為簡體） | 教育部標準字 |
| 干 (U+5E72) | → 幹（語意誤映射） | 教育部標準字（干涉/干擾） |
| 才 (U+624D) | 不轉 | 教育部標準字 ✓ |
| 了 (U+4E86) | 不轉 | 教育部標準字 ✓ |

OpenCC STCharacters 表來自中國大陸 GB 標準，對「台灣現代繁體標準字」判斷有偏差。

### 關鍵發現（本 session 即時實證）：zhtw-mcp 是更精準的權威源

實測：傳入「台灣的群眾才是真正了解只有布料才能干涉這件事」（含 OpenCC 誤判的「台/群/才/了/只/布/干」）給 zhtw-mcp：

```json
{
  "detected_script": "traditional",
  "s2t_applied": false,
  "issues": [{"found": "台灣", "suggestions": ["臺灣"], "rule_type": "variant",
              "context": "MoE 標準用字 (地名用「臺」)"}]
}
```

| 觀察 | 涵義 |
|------|------|
| detected_script: "traditional" | zhtw-mcp 正確識別「群/才/了/只/布/干」是繁體 |
| 「台灣」單獨報 variant warning | zhtw-mcp 有 context-aware 判斷（地名 context 用「臺」） |
| 不誤把「群/才/干」標為簡體 | 比 OpenCC 更接近台灣 MoE 標準 |

**結論**：zhtw-mcp 內部規則庫（ruleset_hash b6c461b9... / 1736 rules）對台灣標準字判斷比 OpenCC 更準確。用戶提問「zhtw-mcp 能否作為權威來源」答覆：**對「台灣繁體標準字驗證」場景可（甚至優於 OpenCC）**；對「找混入簡體字」場景仍不適用（auto S2T，W17-144 已驗證）。

### 影響範圍

`.claude/scripts/charset-pollution-detector.py` 的 `_build_anchors()` 函式目前用 OpenCC 動態驗證，未來擴充 ANCHORS 時可能踩 OpenCC 偏差雷區。

### 相關 Error Pattern

- PC-072 — 主要追蹤對象（AUQ payload 簡體污染）
- PC-074 — 字元集判定的共用字陷阱（W17-144.1 二度自證）
- W17-144 — zhtw-mcp 對「找污染」場景不適用的初探結論

### 額外證據（本 session）

WRAP 分析時用戶質疑「OpenCC 並不是完全正確的權威資料」，並實時提問「新增的 zhtw mcp 能不能作為權威來源」，我即時測試後確認 zhtw-mcp **可作為台灣繁體標準字權威來源**。這改變本 ANA 的策略空間：原本是「OpenCC + 教育部標準字 + Unihan 三源」，現可改為「OpenCC + zhtw-mcp 雙源」（zhtw-mcp 已涵蓋教育部標準）。

### WRAP 完整四階段結論

#### W（Widen Options）— 候選方案

| # | 方案 | 整合成本 | 對 detector 影響 |
|---|------|--------|-----------------|
| A1 | zhtw-mcp 只用於 self-test 階段 | 中（需 MCP runtime）| detector runtime 仍純 OpenCC |
| A2 | zhtw-mcp 取代 OpenCC 為主要過濾器 | 高（需 MCP runtime + 位置定位 fallback）| detector 增加 MCP 依賴 |
| A3 | 完全靜態 hardcoded ANCHORS + 反向白名單 | 低（純 Python）| 維護成本回升 |
| A4 | **維持現狀（OpenCC + 種子手選）+ self-test 補強** | 極低（純註解 + 反向驗證）| 最小變更 |

#### R（Reality Test）— 兩輪實測

**測試 1**：傳 90+ 簡體字（独/违/没/务/...）給 zhtw-mcp
- 結果：`detected_script: "simplified"`, 0 issues
- 確認 zhtw-mcp 識別簡體文字（自動 S2T，不報錯）

**測試 2**：傳對應 90 繁體字 + OpenCC 誤判字（獨/違/...台/群/才/干/了/只/布）
- 結果：`detected_script: "traditional"`, 0 issues
- 確認 zhtw-mcp 對「台/群/才/干/了/只/布」**正確識別為繁體**（OpenCC 0% 偏差）

| 字 | OpenCC s2t | zhtw-mcp detected | 真相 |
|----|-----------|------------------|------|
| 台 | → 臺（誤判簡體）| traditional ✓ | 台灣行政院標準字 |
| 群 | → 羣（誤判簡體）| traditional ✓ | 教育部標準字 |
| 干 | → 幹（誤判簡體）| traditional ✓ | 教育部標準字 |
| 才 | 不轉 ✓ | traditional ✓ | 教育部標準字 |

#### A（Attain Distance）— 機會成本

| 方案 | context 增量 | 維護成本 | 故障風險 |
|------|------------|---------|---------|
| A1 整合 zhtw-mcp self-test | 中（MCP 呼叫 + fallback）| 中 | MCP 不可用時 self-test 跳過 |
| A2 zhtw-mcp 取代 OpenCC | 高 | 高 | runtime MCP 依賴 |
| A3 hardcoded + 反向白名單 | 低 | 高（手動維護兩表）| PC-074 共用字陷阱重現 |
| **A4 維持 + self-test 補強** | **極低** | **極低** | **零新增風險** |

#### P（Prepare to be Wrong）— 失敗預想

最可能失敗：**未來有人從 OpenCC STCharacters 表全量抽取「常用簡體字」加入 _ANCHORS_SEED**，把「台/群/干」等台灣標準字誤入種子，導致 detector 誤報合法繁體文件。

**防護**：在 `_ANCHORS_SEED` 註解明示「禁從 OpenCC STCharacters 表全量抽取，因 OpenCC 對台灣標準字 30% 偏差（已實證 台/群/干）」+ self-test 加「台灣標準字反向白名單」測試案例。

### 結論：採行 A4（維持現狀）+ self-test 補強

| 選項 | 是否採行 | 理由 |
|------|---------|------|
| 採行雙權威源（OpenCC + zhtw-mcp） | **否** | zhtw-mcp 整合需 MCP runtime，detector 失去 standalone 性；現 _ANCHORS_SEED 手選種子已過濾 OpenCC 誤判字，雙權威源 ROI 不足 |
| 維持現狀（單權威 OpenCC） | **是（補強）** | 加 self-test「台灣標準字反向白名單」+ 註解警告擴充 ANCHORS 的禁用方法 |
| 改用其他工具 | **否** | 已實證 OpenCC + 種子手選方案足夠；未來若有更精確開源工具可重評 |
