#### Hook 2: task-dispatch-readiness-check.py

| Rubric | 評估結果 |
|--------|---------|
| 1. 強制動作 | 在 strict 模式下，若代理人分派與任務類型不匹配，回 deny（阻擋 Task 工具）；warning 模式只記錄不阻擋 |
| 2. 觸發條件維度 | 多維度：工具類型（僅 Task）AND Handoff recovery 豁免 AND 代理人名稱直接匹配（agent_to_task_map）OR 關鍵字權重評估 |
| 3. 路徑分類/豁免機制 | (a) Handoff recovery 模式豁免；(b) 未知代理人直接放行；(c) agent_to_task_map 直接匹配優先跳過關鍵字判斷；(d) strict/warning 雙模式可配置 |
| 4. 已知衝突規則對照 | **P1 風險**：關鍵字權重評估（detect_task_type）基於 prompt 前 500 字元的正則匹配，可能誤判任務類型導致錯誤 deny。例：prompt 同時含 Phase 3b 和 Phase 4 關鍵字時，優先匹配第一個命中的 Phase 標記，若代理人為 cinnamon-refactor-owl 但 prompt 開頭含 `[Phase 3b]` 標記（如引用上游 ticket 的 Phase 標記），會誤判為 Phase 3b 任務而 deny |
| 5. ARCH-018 風險分級 | **P1**（高機率撞牆）— 關鍵字權重誤判在複雜 prompt 中有合理觸發路徑 |

**觸發劇本**：PM 派發 cinnamon-refactor-owl 執行 Phase 4 重構，prompt 的 Context Bundle 引用了上游 Phase 3b ticket 內容（含 `[Phase 3b]` 字樣）。Hook 掃描 prompt 前 500 字元命中 `[Phase 3b]` explicit phase pattern，判定任務為 Phase 3b，與 cinnamon 的 Phase 4 預期不匹配，觸發 deny。

**建議修正方向**：explicit phase pattern 應僅匹配 prompt 第一行（意圖宣告行），而非前 500 字元全文掃描；或改為只在 `prompt_head` 中未命中時才 fallback 到關鍵字權重。

#### Hook 3: agent-ticket-validation-hook.py

| Rubric | 評估結果 |
|--------|---------|
| 1. 強制動作 | 若 Task 派發的 prompt 未包含有效 Ticket ID 或 Ticket 不存在，回 deny（exit 2） |
| 2. 觸發條件維度 | 多維度：工具類型（僅 Task）AND Handoff recovery 豁免 AND 代理人類型豁免（TICKET_EXEMPT_AGENT_TYPES）AND Ticket 存在性/決策樹驗證 |
| 3. 路徑分類/豁免機制 | (a) Handoff recovery 模式豁免；(b) 5 種唯讀代理人類型豁免（Explore/claude-code-guide/general-purpose/Plan/feature-dev:code-explorer）；(c) 豁免白名單有明確判別準則（唯讀 vs 可寫）和升級路徑（> 10 項時改動態分類） |
| 4. 已知衝突規則對照 | 無已知衝突。豁免白名單覆蓋所有已知的唯讀探索型代理人。理論上新增唯讀代理人類型時若忘記加入白名單會被誤擋，但有明確升級路徑記錄 |
| 5. ARCH-018 風險分級 | **P2**（理論風險）— 新增唯讀代理人未加入白名單時會被誤擋，但觸發路徑需要「新增代理人 + 忘記更新白名單」雙重失誤 |

**觸發劇本**：開發者新增一個唯讀分析型代理人（如 bay-quality-auditor 類型的自定義 subagent_type），但未將其加入 TICKET_EXEMPT_AGENT_TYPES。該代理人被派發時因 prompt 不含 Ticket ID（情報蒐集類任務通常無 Ticket）而被 deny。

**建議修正方向**：維持現狀（白名單機制已有明確升級路徑規劃），但建議在新增代理人的 checklist 中加入「確認是否需要加入 TICKET_EXEMPT_AGENT_TYPES」提醒。

| Rubric | 評估結果 |
|--------|---------|
| 1. 強制動作 | 子任務未完成時回 deny（exit 2）阻擋 `ticket track complete`。其餘檢查項（acceptance/5W1H/error-pattern/execution log/multi_view/custom H2）為 warning 不阻擋 |
| 2. 觸發條件維度 | 多維度：工具類型（僅 Bash）AND 命令匹配（ticket track complete/batch-complete）AND subagent 環境偵測（跳過 AskUserQuestion 提醒）AND 8 個獨立 checker 模組 |
| 3. 路徑分類/豁免機制 | (a) subagent 環境直接跳過（避免 subagent 無法回答 AskUserQuestion）；(b) 唯一阻擋條件為子任務未完成（硬性前置條件），其餘皆為 warning/提醒；(c) ANA/DOC 類型有差異化處理路徑 |
| 4. 已知衝突規則對照 | 無已知衝突。唯一阻擋邏輯（children 未完成）是正確的前置條件，不存在「正確操作被誤擋」的 ARCH-018 模式。W17-120.2 已退場 ANA spawned 阻擋（改為 children 統一路線），消除了先前的雙重阻擋風險 |

**結論**：此 hook 是 Batch A 中設計最成熟的，8 個 checker 模組職責清晰，阻擋與警告層級分離明確，subagent 環境偵測到位。無 ARCH-018 風險。

#### Hook 5: main-thread-edit-restriction-hook.py

| Rubric | 評估結果 |
|--------|---------|
| 1. 強制動作 | 主線程在保護分支上編輯非白名單路徑時回 deny（exit 2）。預設拒絕安全策略 |
| 2. 觸發條件維度 | 多維度：工具類型（僅 Edit/Write）AND subagent 環境偵測（跳過）AND 開發分支偵測（跳過）AND 路徑白名單/黑名單雙層判斷（path_permission.py） |
| 3. 路徑分類/豁免機制 | (a) subagent 環境直接跳過（此 hook 僅限制主線程）；(b) feat/fix 開發分支直接跳過；(c) 白名單覆蓋：`.claude/**`、`docs/**`、`CLAUDE.md`、`CHANGELOG.md`、`.gitignore`、`.gitattributes`、Claude Code memory/plan 系統路徑；(d) 黑名單：`lib/`、`test/`、`*.dart`、`.claude/lib/*.py`、`backend/`、`*.go` |
| 4. 已知衝突規則對照 | **P2 風險**：白名單中 `.claude/hooks/.*` 允許主線程編輯 hook .py 檔案，但 pm-role.md 禁止主線程寫產品程式碼。hook .py 不是產品程式碼（屬框架資產），所以邏輯上不衝突。但若未來 `.claude/lib/*.py` 邊界調整（目前黑名單擋 `.claude/lib/*.py`），可能出現灰色地帶。此外，`src/` 目錄既不在白名單也不在黑名單，但被預設拒絕策略擋下——這是正確行為，只是依賴隱式邏輯 |
| 5. ARCH-018 風險分級 | **P2**（理論風險）— `.claude/lib/*.py` 的白名單/黑名單邊界在未來擴展 `.claude/` 子目錄時可能需要調整 |

**觸發劇本**：未來若在 `.claude/lib/` 下新增非 .py 的可執行檔（如 .sh script），白名單中無對應模式但也不在黑名單中，會被 `.claude/` 非白名單路徑攔截規則擋下。雖然行為正確（保守拒絕），但錯誤訊息可能誤導使用者以為是 bug。

**建議修正方向**：維持現狀（預設拒絕策略是安全的），但建議在 deny 訊息中區分「在黑名單中」和「不在白名單中」兩種情況，幫助使用者理解拒絕原因。
