### 重構摘要

執行 Phase 4 多視角共識三項：

| 重構項 | 變更內容 | 行數效果 |
|--------|---------|---------|
| (a) `_write_ticket` YAML 序列化 | 38 行手寫分支（dict / list-of-string / list-of-dict）改為 `yaml.safe_dump(frontmatter, sort_keys=False, allow_unicode=True)`，frontmatter 改用 dict literal + `**extra_fields` 合併 | 38 行 → 17 行（含 docstring） |
| (b) AC2 五個單欄位測試 parametrize | 五個 method（`test_blockedby_updated` / `test_relatedto_updated` / `test_parent_id_updated` / `test_source_ticket_updated` / `test_spawned_tickets_updated`）合併為單一 `test_single_field_updated`，採 `(field, build_initial, build_expected)` 三元 parametrize，`ids` 用欄位名讓失敗訊息可讀；list 形式驗證部分替換，scalar 形式驗證單值替換 | 五個 13 行 method（共 ~65 行）→ 單一 ~20 行 method |
| (c) AC4 主驗收測試拆 helper + 子 method | `test_w11_reorganization_full_consistency` 由 73 行單體拆為 `_setup_w11_scenario`（建情境 + 跑 batch migrate）+ 四個 `_verify_*`（child rename / parent children / external refs / no stale old ids），主測試 body 縮為 6 行（setup + assert rc + 4 個 verify call）；class 常數 `P_ID / C1..C3_NEW` 集中 id mapping，移除測試體內重複定義 | 主測試 73 行 → 7 行（其餘下放 helper，認知指數每個 helper 均 ≤ 10） |

### 設計決策

- yaml.safe_dump 使用 `sort_keys=False` 保留欄位插入順序，避免破壞既有測試假設（fm["children"] 順序比對）
- parametrize ids 採欄位名（非 index），便於 CI 失敗訊息定位
- AC4 helper 用 instance method + class 常數，符合 pytest test class 風格；不引入 fixture 是因為 4 個 verify 都讀同一 tickets_dir，無需共享可變狀態

### 不在本 ticket 範圍

- AC4 idempotency 測試（`test_w11_reorganization_idempotency`）未拆，因認知指數已可接受（無重複 helper 機會）
- TestAC3 三個批量測試也存在類似結構重複，但跨遷移引用情境（兩步 chain / dict children）變化較大，未強制 parametrize 以保持可讀性
