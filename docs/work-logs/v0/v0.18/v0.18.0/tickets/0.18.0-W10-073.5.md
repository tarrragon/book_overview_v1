### 判斷原則（父 ticket 共通指引）

**拆分依據是「認知負擔」與「按需載入」，不是行數。**

這是卡片盒筆記法（Zettelkasten）的核心精神：

- 正確問題：這段知識在什麼情境才需要理解？每段知識是否原子化？按需載入時能否獨立理解？
- 錯誤問題：檔案多少行？骨架是否夠短？行數超過某閾值就要拆？

**推論**：
- 500 行的檔案若內容全部原子化且在同一情境共同需要 → 不應拆分
- 100 行的檔案若混合多個情境才用到的知識 → 即使短也應拆
- 拆分後的每一碎片必須能獨立被理解（原子性），否則只是把完整概念切碎
- 保留判斷標準不是「這段很重要」，而是「所有使用場景都需要立即載入」

**WRAP 分析時的具體提問**：

1. **W (擴增選項)**：列出所有可能的原子化切分方式（不是以行數為切點，而是以「情境匹配度」為切點）
2. **R (現實測試)**：每個切分方案下，各個碎片是否能獨立被理解？保留在 auto-load 的碎片是否「所有場景都需立即載入」？
3. **A (保持距離)**：從 sync 到其他專案的角度看，這些知識是否真的每個專案啟動都需要？
4. **P (準備出錯)**：若拆分後發現某個碎片其實常被需要，如何快速恢復到 auto-load？

### 最終建議：選項 C（規則 7+8 合移至 references/reference-stability-rules.md）

**建議理由**：
1. **原子性充足**：規則 7、規則 8 雖涵蓋不同對象（規格 vs 框架），但「引用穩定性」是共同主題，合為一檔兩個章節邏輯自洽，閱讀者不會迷失
2. **按需情境明確**：兩規則都有明確觸發場景（編輯 docs/spec/ 或 .claude/），且觸發時代理人很容易判斷要讀 references
3. **量化效益明顯**：節省 ~92 行（227 → ~135 行，縮減 40%）
4. **回退成本低**：僅 ~10 處引用需更新，broken-link-check 可驗證完整性
5. **符合卡片盒筆記法精神**：規則 1-6 是「每次輸出都需要」的基本功，規則 7-8 是「特定情境才會啟動」的進階規範

**不建議選項 D（積極拆分）的原因**：
- 規則 4（Markdown）和 YAML frontmatter 雖非每次用到，但段落極短（各 ~10-15 行），節省行數有限但提升按需讀取摩擦
- 規則 5（檔案命名）建立 ticket 時就需要，且 ticket CLI 提示常觸發

**不建議選項 B（僅移規則 8）的原因**：
- 規則 7 使用率更低（<10%），比規則 8 更適合按需
- 只移一個規則，references/ 檔案結構不平衡
