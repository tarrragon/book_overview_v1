### 階段 2：Linux scheduler 對照表細化

| Linux 層級 | Linux 機制 | Ticket 對應 | 目前實現 | 缺口 |
|-----------|-----------|-----------|---------|------|
| **Core: Runqueue** | struct rq; per-CPU FIFO | runqueue 模擬：blockedBy（入隊條件）+ priority（nice） | blockedBy 存在；priority 欄位存在 | 無 CLI 直接查詢「可立即入隊」 |

| **Priority: nice** | -20~19；影響時間片；相同 runqueue 優先執行 | priority P0-P3；嚴格偏好無 soft preemption | priority 欄位有；無 nice 概念 | 無 CLI respecting priority order |

| **Debug: /proc/sched_debug** | runqueue 內容；待執行隊列；task 狀態 | ticket track schedule DAG 視覺；runqueue（可立即做的排序清單） | **缺** | **C：無 DAG + runqueue 聯動展示** |

**結論**：blockedBy + priority 邏輯基礎存在，但缺 3 個 CLI 命令對應 Linux 的核心調度概念。

### 階段 3：真缺口清單（≥3 項 + W17 live 證據）

#### 缺口 A：「可立即做」過濾（schedule() equivalent）
**定義**：返回當前可執行任務（TASK_RUNNING 狀態 ⟹ blockedBy 全 completed）  
**Linux 對應**：schedule() 從 runqueue 選一個；runqueue = ready 進程  
**W17 live 案例**：  
```
W17 pending 17 個：
- W17-005/006：P0，blockedBy=[]  ✓ 可立即做（無阻擋）
- W17-002：P1，blockedBy=[W17-005] → 不可做（W17-005 still pending）
- W17-003：P2，blockedBy=[W17-002] → 不可做（傳遞阻擋）
```

#### 拓撲排序算法選擇

使用 **Kahn 算法**（基於 in-degree）而非 DFS：

```python
def topological_sort_kahn(tickets):
    """
    Kahn 算法：適合交互展示（層級分組）
    
    輸入：tickets list，含 blockedBy 欄位
    輸出：List[List[ticket]]，按層級分組
    """
    in_degree = {t['id']: len(t.get('blockedBy', [])) for t in tickets}
    queue = [t['id'] for t in tickets if in_degree[t['id']] == 0]
    
    levels = []
    while queue:
        levels.append(queue[:])  # 同層級的票
        next_queue = []
        for tid in queue:
            for dependent in find_dependents(tid):
                in_degree[dependent] -= 1
                if in_degree[dependent] == 0:
                    next_queue.append(dependent)
        queue = next_queue
    
    return levels  # [(layer 0), (layer 1), ...]
```

**優點**：
- 層級分組天然對應 schedule 輸出
- 每層內可並行執行
- 易於檢測環（queue 無法遍歷全部節點）
- 複雜度 O(V+E)，可視化友好
