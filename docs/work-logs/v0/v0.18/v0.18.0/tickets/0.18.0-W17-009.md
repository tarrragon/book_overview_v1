## Problem Analysis

## Context Bundle

### 背景（W17-001 Linux 類比的第 5 維度）

W17-001 類比表覆蓋：PID / PCB / /proc / ELF / fork / exec / argv / ARG_MAX / waitpid / signal / zombie 共 10+ 項，但**未涵蓋 scheduler 層**（runqueue / schedule()/ TASK_RUNNING / nice）。

用戶在本 session 追問：「建立完整任務鏈，是否有串連理解執行順序，或 /clear 後從某 ticket 推斷接手的標準？」—— 正是 scheduler 層。

**與 W17-007 差異**：
- W17-007: 單 process exit code 協議（waitpid/signal）——「一個 agent 做完了怎麼報告」
- W17-009: 跨 process scheduling（runqueue/schedule）——「多個 ticket 該選哪個做」

### 現有工具盤點（待 Explore 細審）

| 工具 | 粗略能力 | 不確定處 |
|------|---------|---------|
| `ticket track list [--wave] [--status]` | 列 ticket | 能否按 priority 排序、能否過濾「可立即做」 |
| `ticket track tree <id>` | 樹狀 parent/children | 是否顯示 blocked 狀態 |
| `ticket track chain <id>` | 依賴鏈 | 是否含拓撲順序 |
| `ticket track deps <id>` | 依賴列表 | 與 chain 差異 |
| `ticket track handoff-ready` | handoff 就緒 | 實際輸出內容需 Read 確認 |
| `ticket handoff / resume` | session 切換 | handoff 檔是 PM 手寫還是 CLI 自動生 |
| `ticket track agent-status` | agent 狀態 | 側重單 agent 非全局排程 |
| `ticket track board` | 看板 | 格式與過濾能力 |

### Linux scheduler 對照（設計量尺）

| Linux | Ticket 應有 | 狀態 |
|-------|------------|------|
| `schedule()` kernel | `ticket track next` 選下一個可做 | **缺** |
| runqueue | 隱性（blockedBy + priority 組合） | 無 CLI 直接呈現 |
| `TASK_RUNNING` filter | `--ready` 過濾 blockedBy 全 completed | **缺** |
| `nice` 微調 | priority (P0-P3) | 有欄位無 CLI 生效 |
| `top` 排序 | `list --sort priority,blocked-depth` | **缺** |
| `ps -ef` tree | `tree` | 部分有（未標 blocked 狀態） |
| `/proc/sched_debug` | DAG 視覺化 | **缺** |

### W17 Live 證據（scheduler 缺口痛點）

17 個 W17 pending ticket：
- W17-002 blockedBy W17-005
- W17-003 blockedBy W17-002
- W17-008.1-11 無相互 blockedBy（group children，可並行）
- W17-005/006/007/008/008.1-11 各自 priority P0-P2 混合

若 /clear 後接手要回「先做哪個」：
1. 需 `ticket track list --wave 17 --status pending`

5. 最終結論：W17-005 / W17-006 先做（P0 介面基線）

這 5 步完全可自動化，但現無 CLI。**應 1 命令得答**。
