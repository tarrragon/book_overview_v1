## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 改動 1: agent-authoring-guide checklist 補強

**檔案**: `.claude/references/agent-authoring-guide.md` 「檢查清單（新增／修改代理人時）」區段

**新增條目**：

> 唯讀分析型代理人（不需 Ticket ID 即可派發，如 Explore/code-explorer/Plan 類型）？→ 確認是否需加入 `.claude/hooks/agent-ticket-validation-hook.py` 的 `TICKET_EXEMPT_AGENT_TYPES` 白名單，否則該代理人會因 prompt 不含 Ticket ID 被 deny（W10-043.1 audit P2 風險）

**理由**：W10-043.1 Batch A audit 識別 agent-ticket-validation-hook P2 風險為「新增唯讀代理人 + 忘記更新白名單雙重失誤」。在 checklist 加入提醒可預防此情境，無需修改 hook 程式碼（白名單機制本身無瑕疵）。

### 改動 2: main-thread-edit-restriction-hook 不需修改

**檔案**: `.claude/hooks/lib/path_permission.py` `check_file_permission()` 函式

**現況分析**：deny 訊息已透過三個獨立常數區分：

| 拒絕情境 | GateMessages 常數 |
|---------|-------------------|
| 黑名單命中（程式碼檔案） | `EDIT_BLOCKED_PROGRAM_FILES` |
| `.claude/` 非白名單路徑 | `EDIT_BLOCKED_CLAUDE_INVALID_PATH` |
| 預設拒絕（不在白名單也不在黑名單） | `EDIT_BLOCKED_DEFAULT_DENY` |

W10-043.1 Batch A audit 的建議「在 deny 訊息中區分黑名單命中和不在白名單兩種情況」**已經實作**。本 ticket Reality Test 確認：現有 hook 行為符合 audit 建議，無需修改程式碼。

### 整合結論

W10-043.4 兩個 P2 改善項中：
- 改動 1（agent checklist）：實作 1 行 markdown checklist 補強
- 改動 2（deny 訊息區分）：已存在，無需動作（屬 audit 建議與既有實作對齊驗證）

兩者皆無需 hook 程式碼變動，故不涉及 TDD 完整流程，亦無新增測試需求（既有測試覆蓋三種 deny 情境）。
