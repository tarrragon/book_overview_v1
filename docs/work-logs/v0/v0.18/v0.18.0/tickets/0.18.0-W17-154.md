## Solution

multi_view_status: skipped
multi_view_skip_reason: 設計類 ANA 範圍小（單一 PC-115 文件改寫 + 表格新增），三個子問題的選項在 Test Results 已用證據逐一收斂，無架構分歧。Trigger 機制非高風險決策（base rate < 5%，預期觸發頻率低），紀律遵循率將由 W17-055 30 天觀測追蹤。

### Step 1：Deny 事件記錄方式 — 手動 PM 寫入（唯一可行）

**Why**：Hook 自動偵測 deny 在當前 CC 架構下不可行（重現實驗結果驗證 1）。`.claude/hook-logs/` 對 runtime deny 無紀錄，PreToolUse 全 allow。

**How**：PM 觀察到 subagent 對 `.claude/` Edit deny 時，立即在 PC-115「Deny 事件累積」表追加一列。

### Step 2：計數位置 — PC-115 新增獨立「Deny 事件累積」表

**Why**：

- 獨立 yaml（如 `.claude/state/pc115-deny-events.yaml`）脫離 PC-115 上下文閱讀
- 混入 Base Rate 表破壞語意（success 觀察 vs deny 紀錄）
- 新增獨立表讓 PC-115 讀者直接看到 trigger 進度

**Where**：PC-115「Base Rate 觀察」段落之後新增一節「Deny 事件累積（trigger ≥3 重啟調查）」

**欄位**：`# | 日期 | Session 起始 | 案例 ticket | 檔案路徑 | cwd | CC 版本 | 處理動作`

### Step 3：≥3 門檻 trigger 動作 — 半自動（PM 寫入第 3 行時建 ticket）

**Why**：

- 全自動 hook 監測表行數 ROI 低（base rate 強支持 transient，頻率低）
- 純手動易遺漏（PM 在繁忙 session 可能忽略門檻）
- 半自動讓「寫第 3 行」動作本身觸發決策點（forcing function）

**Action 規範**：當 PM 在「Deny 事件累積」表寫入第 3 行時，必須同時：

1. 建立 ANA ticket（標題：「重啟調查 PC-115 / ARCH-015 subagent .claude/ Edit deny 真因（trigger ≥3 達標）」）
2. 在「處理動作」欄填入該 ticket ID（例：「W17-177 重啟調查」）
3. trigger 觸發 + 對應 ANA ticket 進入 in_progress 後，計數歸零重新累積

**安全網**（不在本 ticket scope）：未來若紀律不足，可加 hook 變體偵測 PC-115 表行數變化，count >= 3 時 stderr 提示。納入 W17-055（30 天觀測）追蹤。

### Step 4：PC-115 line 112 改寫

PC-115 「收斂說明」段落改寫為含「Trigger 機制」表格，明示記錄方式 / 門檻動作 / 計數歸零三步驟。已落地。
