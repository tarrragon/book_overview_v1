# Phase 2 測試案例設計：父子 cascade 與 children 警告

## 1. 測試策略總覽

### 1.1 測試分層決策

| 層級 | 應用 | 決策理由 |
|------|------|---------|
| 整合測試（主要） | `lifecycle.py::Lifecycle.complete()` 端到端行為 | complete 涉及多個協作函式（list_tickets、save_ticket、_analyze_next_steps），必須驗證整體流程 |
| 單元測試（helper） | `_cascade_unblock_children(parent, version)` | Phase 3b 拆分後的 helper 需獨立覆蓋 blockedBy AND 判斷、cascade 條件分支 |

> 本 Ticket 的核心語義是「complete 後的副作用」，屬於組合行為驗證，優先採整合測試路徑（以 mock 取代 I/O），helper 單測作為邊界條件補充。

### 1.2 Sociable Unit Tests 原則

| 依賴 | 策略 | 理由 |
|------|------|------|
| `load_ticket` / `list_tickets` | Mock | 外部 I/O 依賴 |
| `save_ticket` | Mock（含 side_effect 模擬失敗） | 外部 I/O；需驗證呼叫次數與狀態 |
| `validate_completable_status` / `validate_acceptance_criteria` | Mock 為成功 | 非本功能驗證範圍（已由 TestComplete 既有測試覆蓋） |
| `append_worklog_progress` | Mock 為 no-op | I/O；非本功能驗證範圍 |
| `_analyze_next_steps` / `_auto_handoff_if_needed` | 允許真實呼叫或 mock | 後者建議 mock 以隔離；前者讀取 all_tickets 可真實執行 |
| Ticket dict 結構 | 真實 dict | 內層資料，不 mock |

### 1.3 測試檔案組織

| 項目 | 決策 |
|------|------|
| 檔案位置 | `.claude/skills/ticket/tests/test_track_lifecycle.py` |
| 測試類別 | 新增 `TestCompleteCascadeChildren` 類別（與既有 `TestComplete` 並列） |
| Fixture 位置 | 本檔案內定義（見 §2），若後續多檔案需要可提升至 `conftest.py` |
| 命名慣例 | 沿用既有：`test_complete_<情境描述>` |

## 2. Fixture 設計

### 2.1 核心 Fixture：`make_parent_child_tickets`

```python
@pytest.fixture
def make_parent_child_tickets():
    """
    建立父子 Ticket dict 的 helper factory。

    使用方式：
        parent, children = make_parent_child_tickets(
            parent_id="P0",
            children_spec=[
                ("C1", "blocked", ["P0"]),
                ("C2", "pending", []),
            ],
            parent_extra={"acceptance_criteria": [{"text": "ac1", "completed": True}]},
        )

    Returns:
        (parent_dict, [child_dict, ...])
    """
```

職責：
- 建構標準 parent ticket dict：id, status=in_progress, title, children=[...], acceptance_criteria(全勾)
- 建構 child ticket dict 清單：id, status, blockedBy, parent_id=parent_id, title
- 呼叫端可傳 `parent_extra` / `child_extra` 覆寫欄位

### 2.2 Mock Bundle：`complete_env`

```python
@pytest.fixture
def complete_env(monkeypatch):
    """
    封裝 complete() 執行所需的 mock bundle。

    Returns:
        SimpleNamespace(
            load_ticket=MagicMock,

            list_tickets=MagicMock,
            validate_completable_status=MagicMock (預設回傳成功),
            validate_acceptance_criteria=MagicMock (預設回傳成功),

```

職責：
- 統一 patch `ticket_system.commands.lifecycle` 模組內的上述函式
- `set_tickets(parent, children)` helper 讓測試內一行完成資料源設定
- `save_ticket` 以 MagicMock 記錄所有呼叫，供驗證 cascade 後的 save 行為

### 2.3 save 失敗模擬

```python
def _save_fails_for(ticket_ids: set):
    """產生 save_ticket 的 side_effect：指定 id 的 save 拋 IOError，其餘成功。"""
    def _side_effect(ticket_dict, *args, **kwargs):
        if ticket_dict.get("id") in ticket_ids:
            raise IOError(f"mock save failure for {ticket_dict['id']}")
        return None
    return _side_effect
```

## 3. 測試案例清單

測試類別：`TestCompleteCascadeChildren`（共 11 個測試案例）

### 3.1 三大使用場景（對應 Phase 1 §4）

#### TC-A1：正常 cascade — 唯一 blocked 子自動解鎖

**對應**：Phase 1 場景 1、AC1

```
Given:
  - parent P0 (status=in_progress, children=[C1], acceptance 全勾)
  - child C1 (status=blocked, blockedBy=[P0], parent_id=P0)
  - list_tickets 回傳 [P0, C1]

When:
  execute_complete(args=Mock(ticket_id="P0"), version="0.18.0")

Then:
  - 回傳 0
  - save_ticket 被呼叫至少 2 次（P0 complete + C1 unblock）
  - 可在 save_ticket.call_args_list 中找到 id=C1 且 status="pending" 的 dict
  - capsys.readouterr().out 含字串 "[Cascade]"
  - 輸出含 "C1"
  - 輸出不含 "[Warning] 父 Ticket 完成時尚有未完成的子 Ticket"
```

#### TC-A2：警告但不阻止 — 有 pending / in_progress children

**對應**：Phase 1 場景 2、AC1

```
Given:
  - parent P0 (children=[C1, C2], acceptance 全勾)
  - C1 (status=pending, blockedBy=[], parent_id=P0)
  - C2 (status=in_progress, blockedBy=[], parent_id=P0)

When:
  execute_complete(args, "0.18.0")

Then:
  - 回傳 0
  - C1 / C2 在 save_ticket 呼叫中「status 未被修改」
    （驗證方式：檢查 save_ticket.call_args_list 中無以 C1/C2 為 id 的新狀態寫入，或確認 dict 狀態不變）
  - stdout 含 "[Warning] 父 Ticket 完成時尚有未完成的子 Ticket"
  - stdout 含 "C1 [pending]" 和 "C2 [in_progress]"
  - stdout 不含 "[Cascade]"
```

#### TC-A3：其他依賴保留 blocked — blockedBy 尚有未完成項

**對應**：Phase 1 場景 3、AC1

```
Given:
  - parent P0 (children=[C1])
  - C1 (status=blocked, blockedBy=[P0, X], parent_id=P0)
  - X (status=in_progress, 不在 P0.children 中)
  - list_tickets 回傳 [P0, C1, X]

When:
  execute_complete(args, "0.18.0")

Then:
  - 回傳 0
  - C1 status 保持 blocked（無 save_ticket 呼叫將 C1 改為 pending）
  - stdout 含 "[Warning]"
  - stdout 含 "C1 [blocked]"
  - stdout 不含 "[Cascade]"
```

### 3.2 八項邊界情境（對應 Phase 1 §6）

#### TC-B1：孫子不遞迴（§6.1）

```
Given:
  - P0 (children=[C1])
  - C1 (status=blocked, blockedBy=[P0], children=[GC1])
  - GC1 (status=blocked, blockedBy=[C1], parent_id=C1)

When: complete(P0)

Then:
  - C1 被解鎖為 pending（save 被呼叫）
  - GC1 狀態保持 blocked（save_ticket 未對 GC1 寫入狀態變更）
  - stdout 僅列 C1 於 [Cascade]，不列 GC1
```

#### TC-B2：多層 parent_id 資料不一致（§6.2）

```
Given:
  - P0 (children=[])  # 空 children 但其他 ticket 聲稱 parent_id=P0
  - C_orphan (status=blocked, blockedBy=[P0], parent_id=P0)

When: complete(P0)

Then:
  - C_orphan 不被 cascade（權威來源是 P0.children）
  - stdout 不含 [Cascade] 或 [Warning] 區塊（P0.children 為空）
```

#### TC-B3：children 為空集（§6.3）

```
Given:
  - P0 (children=[] 或無 children 欄位)

When: complete(P0)

Then:
  - 回傳 0
  - stdout 不含 "[Cascade]" 也不含 "[Warning] 父 Ticket"
  - save_ticket 僅對 P0 自身呼叫
```

#### TC-B4：全 completed 的 children（§6.4）

```
Given:
  - P0 (children=[C1, C2])
  - C1 (status=completed)
  - C2 (status=closed)

When: complete(P0)

Then:
  - 回傳 0
  - stdout 不含 "[Cascade]" 也不含 "[Warning] 父 Ticket"
  - save_ticket 不對 C1/C2 呼叫
```

#### TC-B5：找不到 child_id（§6.5）

```
Given:
  - P0 (children=[C1, C_ghost])
  - C1 (status=blocked, blockedBy=[P0])
  - ticket_map 中無 C_ghost

When: complete(P0)

Then:
  - 回傳 0（不拋例外）
  - C1 正常 cascade
  - stdout 不提及 C_ghost
```

#### TC-B6：closed 視同 completed（§6.6）

```
Given:
  - P0 (children=[C1])
  - C1 (status=closed)

When: complete(P0)

Then:
  - stdout 不含 [Warning]（closed 不列警告）
  - stdout 不含 [Cascade]（closed 不 cascade）
```

#### TC-B7：cascade 儲存失敗（§6.7）

```
Given:
  - P0 (children=[C1, C2])
  - C1 (status=blocked, blockedBy=[P0])
  - C2 (status=blocked, blockedBy=[P0])
  - save_ticket 在被呼叫且 ticket_dict.id==C1 時拋 IOError
    使用 fixture _save_fails_for({"C1"})

When: complete(P0)

Then:
  - 回傳 0（父 complete 仍成功）
  - stdout 含 "[Warning]" 含 C1 的 save 失敗訊息
  - C2 仍被 cascade 成功（save_ticket 對 C2 的呼叫有發生且 status=pending）
  - 不 fail-fast：C1 失敗不中斷 C2 處理
```

#### TC-B8：blockedBy 多項 AND 語義（§6.8）

子案例 B8a：`blockedBy=[P0]` 單項 → 可解鎖
子案例 B8b：`blockedBy=[P0, X]` 且 X.status=completed → 可解鎖
子案例 B8c：`blockedBy=[P0, X]` 且 X.status=pending → 保留 blocked
子案例 B8d：`blockedBy=[]` 但 status=blocked（異常狀態）→ 可解鎖（依 §6.8 規格）

```
實作建議：以 @pytest.mark.parametrize 提供四組資料，共用一個測試函式。

parametrize 資料：
  ("single_parent",      ["P0"],      {}, True),           # B8a
  ("other_completed",    ["P0", "X"], {"X": "completed"}, True),   # B8b
  ("other_pending",      ["P0", "X"], {"X": "pending"}, False),    # B8c
  ("empty_blockedby",    [],          {}, True),           # B8d

Then: 若 expected_unblock 為 True，stdout 含 [Cascade] 且 C1 被 save 為 pending；
      若 False，C1 保持 blocked 且出現在 [Warning] 列表。
```

## 5. 行為鏈式推演

### 5.1 TC-A1 鏈式展開（示範）

```
行為鏈：
  前置 → complete 觸發 → acceptance 驗證 → 狀態寫入 → cascade 副作用 → 輸出

前置驗證斷言（測試內必須先確認）：
  - parent dict 具備 id / children / acceptance_criteria 欄位 ✓（由 fixture 保證）
  - child dict status=blocked、blockedBy=[parent_id] ✓（由 fixture 保證）
  - mocks 正確 patch（load_ticket / list_tickets 回傳預期資料）✓

When：execute_complete(args, version)

中間狀態（可選驗證）：
  - parent 的 save 先於 child 的 save（順序驗證：Phase 1 §2.1 規定 cascade 是 complete「成功後」的副作用）
  - cascade 訊息印於「驗收提示」之後、「next-steps」同一區間（§7.1）

最終結果：
  - return 0
  - child 狀態 pending
  - stdout pattern 符合 §3.3
```

### 5.2 四維度分支覆蓋檢查

| 維度 | 涵蓋測試 |
|------|---------|
| 正常流程 | TC-A1, TC-B8a/b/d（可解鎖路徑）|
| 異常流程 | TC-B5（找不到 child）, TC-B7（save 失敗）|
| 邊界條件 | TC-B1（遞迴邊界）, TC-B2（資料不一致）, TC-B3（空集）, TC-B4（全完成）, TC-B6（closed）|
| 中斷操作 | N/A（cascade 是同步後置副作用，無中斷語義） |

> 「中斷操作」維度不適用：cascade 在 complete 返回前同步執行，無異步取消場景。此為有意識的推演結論，非遺漏。

## 6. 與 Phase 1 規格的差異與風險

### 6.1 差異

無。測試案例嚴格對應 Phase 1 §4（三場景）與 §6（八邊界）+ §3.3/§3.4（輸出格式）。

### 6.2 識別的風險

| 風險 | 影響 | 緩解 |
|------|------|------|
| cascade 順序驗證困難 | 測試可能過度耦合實作 | 只驗證「最終 save 結果」與「stdout 包含關鍵字串」，不強制 call_args_list 的絕對順序 |
| capsys 捕獲可能遺漏中文字符 | 輸出字串比對失敗 | 使用 `in` 子字串匹配而非精確 `==`；關鍵 ASCII 標籤（`[Cascade]` / `[Warning]`）作為錨點 |
| _analyze_next_steps 真實執行的副作用 | 非本功能驗證範圍的輸出干擾 | 允許其輸出存在；測試僅 assert `[Cascade]` / `[Warning]` 子字串，不 assert 完整輸出 |
| TC-B7 save 失敗的 warning 格式未在 §3 明確規範 | 測試 assert 可能與實作對不上 | 測試只 assert 「含 C1 id」+「含 failure/error/save 等關鍵字」的寬鬆匹配，具體字串待 Phase 3b 定義 |
| TC-B2 資料不一致為設計斷言 | 若 Phase 3b 實作改掃描 parent_id 會與測試衝突 | 測試作為「單一真實來源」設計決策的守門員；若未來需改 Schema，需同步更新本測試與 Phase 1 §6.2 |
