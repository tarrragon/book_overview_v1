盤點「跳過 SA 前置審查」提醒的觸發邏輯，確認哪些場景應排除，規劃場景過濾方案。

## Problem Analysis

### 觸發位置

`.claude/hooks/process-skip-guard-hook.py:66-76` 的 `SKIP_SA_REVIEW` 模式，原始關鍵字對：

```python
"SKIP_SA_REVIEW": {
    "pairs": [
        ("不需要", "sa"),
        ("跳過", "sa"),
        ("不做", "架構審查"),
        ("不需要", "審查"),
        ("跳過", "系統分析"),
    ],
}
```

### 根因

觸發函式 `detect_skip_intent` 僅檢查兩個關鍵字是否同時出現於 prompt，**完全不讀取 ticket 狀態**：

- 當前 ticket 的 type（IMP/DOC/ANA/REF）
- 當前 ticket 的 current_phase
- ticket status、wave 等上下文

### 誤報場景

<!-- PC-093-exempt: ticket-tracked: 描述既往 W5-001 重構評估場景，非延後決策；實作已由 W11-004.2 落地 -->
W5-001 既有模組重構評估，PM prompt 自然提及「重構不需要 SA 審查」即命中關鍵字對，無上下文過濾。

### 落差總結

| 層面 | 設計意圖 | 實際行為 |
|------|---------|---------|
| 觸發條件 | 新功能或架構變更前 | 任何含關鍵字對的 prompt |
| 排除情境 | DOC/ANA/REF、既有模組重構 | 無排除 |
| 上下文感知 | 讀 ticket frontmatter | 0 |

### 相關 Error Pattern

無直接對應 PC，但屬「Hook 缺上下文導致誤報」類別。

## Solution

### Multi-View Status

```yaml
multi_view_status: n_a
reason: 純 hook 邏輯改善（單檔加 ticket 上下文 guard），無架構/設計爭議；方案明確且 W11-004.2 已落地驗證有效
```

### 推薦方案（方案 3 — hybrid）

關鍵字精確化 + ticket type/phase guard 雙層架構。

**方案 1（ticket 上下文 guard）**

於 `main()` 加入 ticket 狀態查詢，對 SKIP_SA_REVIEW 套用 type/phase guard：

```python
if skip_type == "SKIP_SA_REVIEW":
    ctx = get_current_ticket_context()
    if ctx.get("type") in ("DOC", "ANA", "REF"):
        continue
    if ctx.get("current_phase", "").startswith("Phase-4"):
        continue
```

優點：精準過濾。缺點：每 prompt 50-100ms 開銷（cold path 化解）。

**方案 2（關鍵字精確化）**

移除 `("不需要","審查")` 等過泛組合，保留具體 SA 語境字串。優點：零開銷。缺點：仍可能誤報。

**方案 3（hybrid，推薦）**

兩者疊加。關鍵字止住明顯假陽性，guard 處理系統性誤報。
