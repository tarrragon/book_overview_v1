# Devil's Advocate 分析：為何「用 LLM 解決 LLM 隨機性」是本質矛盾

## 1. Stanford 8 方向各自引入的新隨機性源頭

| # | Stanford 方向 | 引入的新隨機性 | 嚴重度 |
|---|-------------|-------------|-------|
| 1 | Prompt chaining（拆三 prompt） | 三次 LLM 呼叫 = 隨機性三倍化。每次呼叫獨立採樣，錯誤傳播鏈：決策識別錯 → 選項池錯 → 渲染錯。中間態無法 deterministic 驗證 | 高 |
| 2 | Few-shot examples 持續更新 | 訓練資料品質依賴 LLM 標注（誰判定「這次是違規」？）；pool 漂移無 ground truth 校準；survivorship bias——只收集被 hook 攔截的案例，漏網案例永遠進不了 pool | 高 |
| 3 | LLM judge + rubric | **judge 本身是 LLM**，受同樣隨機性影響。Rubric 評分的 inter-rater reliability 未知。judge 與被評者可能共享相同偏見（同模型或同系列模型） | 致命 |
| 4 | Self-critique / reflection | 已知缺陷：LLM 傾向「肯定自己的輸出」（sycophancy bias）；即使要求批評，批評深度隨機；修正後可能引入新錯誤 | 高 |
| 5 | Jagged frontier | 邊界判定本身需要 LLM 推斷「這個決策是否適合 AUQ」——又是一次隨機呼叫 | 中 |
| 6 | Centaur vs Cyborg（batch 模式） | 批次累積引入時序依賴；批次大小影響 context window 壓力，間接影響輸出品質 | 低 |
| 7 | Prompt template + metadata | scenario_id 映射若由 LLM 選擇 → 選擇本身隨機；若由 deterministic 匹配 → 實質退化為 L3 hook | 低（若 deterministic）|
| 8 | Promptfoo 多 LLM 測試 | dev-time 工具，不影響 runtime 隨機性。唯一安全方向 | 無 |

**結論**：8 方向中僅 #8（Promptfoo）和 #7（若 deterministic 匹配）不引入新隨機性。其餘 6 個方向每個都在既有隨機性之上疊加新的隨機層。

## 2. LLM Judge 可信度評估

### 核心矛盾：Quis custodiet ipsos custodes?（誰來監管監管者？）

| 問題 | 分析 |
|------|------|
| Judge 的 ground truth 來源 | 無。AUQ payload 品質沒有客觀標準答案，rubric 本身由人寫但由 LLM 解讀——解讀是隨機的 |
| Judge 與被評者的獨立性 | **不獨立**。若使用同一模型（Claude），judge 和被評者共享相同的訓練偏見、相同的 sycophancy 傾向、相同的語言偏好 |
| Judge 一致性 | 學術研究顯示 LLM judge 的 Cohen's kappa 通常在 0.4-0.7（moderate agreement），遠低於工程系統需要的 >0.9 |
| Judge 成本 | 每次 AUQ 呼叫額外增加一次 LLM 呼叫，token 成本翻倍，延遲翻倍 |
| Judge 錯誤的後果 | False positive（拒絕正確 AUQ）→ 用戶體驗受損；False negative（放行錯誤 AUQ）→ 系統目的失敗。兩者都比「不用 judge」的後果更差 |

**致命缺陷**：LLM judge 的錯誤率本身不可觀測。Hook regex 誤判時至少可以 grep log 找到。LLM judge 誤判時，你不知道它誤判了——因為判斷「judge 是否正確」需要另一個 judge，形成無限遞迴。

## 3. Self-critique 偏見分析

### 已知 LLM 自我批評偏見

| 偏見類型 | 對 AUQ 的影響 |
|---------|-------------|
| Sycophancy bias（討好偏見） | LLM 傾向認為自己的輸出「大致正確，僅需微調」，不會徹底否定自己的 AUQ 設計 |
| Position bias（位置偏見） | Self-critique 傾向修改末尾內容而非開頭，但 AUQ 的關鍵決策（是否該問、問什麼）在開頭 |
| Verbosity bias（冗長偏見） | 批評後的修正版通常更長更複雜，但 AUQ 需要簡潔 |
| Anchoring bias（錨定偏見） | 第一版輸出成為錨點，critique 圍繞錨點微調而非重新思考 |
| Confidence calibration | LLM 的自信度與實際正確率相關性弱；「高信心修正」可能是錯誤的 |

**核心問題**：Self-critique 在 AUQ 情境的效果不可預測。可能改善也可能惡化，且無法事前判斷——這正是「隨機性」的定義。

## 4. Few-shot Pool 自我學習的 Garbage In Garbage Out 風險

### 學習迴路的退化路徑

```
初始 pool（人工標注，品質高）
    ↓
LLM 使用 pool 生成 AUQ → 部分被 hook 攔截
    ↓
攔截案例被標為「違規」→ 加入 pool 作為負面範例
    ↓
問題 1：未被攔截的違規案例永遠不進 pool（survivorship bias）
問題 2：hook 攔截的可能是 false positive → 污染 pool
問題 3：pool 逐漸反映 hook 的偏見而非真正的品質標準
    ↓
N 代之後：pool 只教 LLM 「如何不被 hook 攔截」而非「如何產出好 AUQ」
```

### 量化風險

| 風險 | 機率 | 後果 |
|------|------|------|
| Pool 被 false positive 污染（hook 誤攔正確 AUQ 加入負面範例） | 中（hook regex 已知有 false positive） | pool 品質逐代下降，LLM 學會迴避正確但被誤攔的模式 |
| Pool 漏網偏見（只收集被攔截案例，漏網的「壞 AUQ」不進 pool） | 高（漏網案例無觀測機制） | pool 只代表「hook 能攔截的違規類型」，對新型違規無教學效果 |
| Pool 規模膨脹（累積案例越多，few-shot 佔用 context 越大） | 確定 | context window 壓力增加，主任務推理品質下降 |
| Pool 過時（早期案例的規則已修改但範例仍在） | 高（無自動清理機制） | LLM 學到過時規則，與現行規則矛盾 |

## 5. 反論證據：為何 Deterministic Hook 仍是更好選擇

### Deterministic vs Probabilistic 系統特性對比

| 特性 | Deterministic Hook（L1+L3） | LLM-based（L2.5+L3.5） |
|------|---------------------------|----------------------|
| 可預測性 | 100%：相同輸入永遠相同輸出 | 0-100%：相同輸入可能不同輸出 |
| 可除錯性 | 高：grep log 即可定位 | 低：需重現 LLM 推理過程（不可能） |
| 可審計性 | 高：規則清單即審計報告 | 低：需理解 LLM 為何做此判斷 |
| 錯誤可修復性 | 高：改規則，改後保證修復 | 低：改 prompt/rubric，不保證修復 |
| 成本 | 零（本地 regex/AST） | 每次呼叫 token 成本 + 延遲 |
| 覆蓋率 | 有限（只攔已知模式） | 理論上更廣（但不可靠） |
| 誤判後果 | 可觀測、可修復 | 不可觀測、不確定是否修復 |

### 「覆蓋率」論述的反駁

支持 LLM-based 方案的主要論據是「deterministic hook 覆蓋率有限，無法攔截未預見的違規模式」。反駁：

1. **覆蓋率 vs 可靠性的 trade-off**：100% 覆蓋率但 70% 可靠性（LLM judge）vs 60% 覆蓋率但 100% 可靠性（deterministic hook）——後者的期望攔截品質更高（0.6 * 1.0 = 0.6 > 0.7 * 1.0 * 0.7 = 0.49，考慮 judge 自身可能放行錯誤）
2. **增量擴展路徑存在**：deterministic hook 的覆蓋率可透過「觀察 → 新增規則」持續擴展，每次擴展都是 100% 可靠的
3. **LLM 覆蓋率不可量化**：你無法測量 LLM judge 攔截了多少「未預見模式」——因為「未預見」本身不可列舉

## 6. 純確定性最佳路徑：L1 + L3 強化 + Promptfoo 驗證

### 推薦架構

| 層次 | 機制 | 隨機性 | 實作成本 |
|------|------|--------|---------|
| L1 | Hook 事後過濾（charset/emoji/簡體字） | 零 | 已實作 |
| L3 | Hook wrapper 強制 schema（選項數/必填欄位/格式） | 零 | 中（第二輪已規劃） |
| L7（新提議） | scenario_id deterministic 匹配 + 靜態模板庫 | 零 | 低 |
| Promptfoo | dev-time 測試（Stanford #8） | 零（測試時間） | 低 |

### L7 詳述：Deterministic Scenario Matching

Stanford #7（prompt template + metadata）的**純確定性版本**：

- scenario_id 由 Hook 根據 context 特徵 deterministic 匹配（if-else / pattern matching）
- 每個 scenario 對應靜態 prompt 模板（人工撰寫、人工審核）
- LLM 只負責填入模板變數，不負責決策結構

這保留了 Stanford #7 的結構化優勢，但完全移除 LLM 在「決策結構」層面的隨機性。

### 路徑總結

**保留 L1 + L3 + deterministic L7 + Promptfoo**，放棄 L2.5（few-shot 自學）和 L3.5（LLM judge）。

理由：在 AUQ schema 品質問題的當前規模（PC-072 記載的違規率 < 10%），deterministic 方案的覆蓋率足夠，且每次擴展都是 100% 可靠的增量。引入 LLM-based 層在當前階段是 overengineering，且帶來不可觀測的新風險。
