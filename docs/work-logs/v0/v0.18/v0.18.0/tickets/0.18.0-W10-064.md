### 調研結論

**已驗證事實**（雙重來源：`.claude/hook-specs/claude-code-hooks-official-standards.md` + Context7 `/ericbuess/claude-code-docs`）

CC runtime 官方支援 9 種 Hook events：
- SessionStart / SessionEnd
- UserPromptSubmit
- PreToolUse / PostToolUse（兩者有 matcher）
- Stop（main thread 完成）
- **SubagentStop**（subagent 完成，關鍵）
- PreCompact / Notification

**SubagentStop 確認存在且可區分代理人**：

input schema（Context7 驗證）：
```json
{
  "hook_event_name": "SubagentStop",
  "stop_hook_active": false,
  "agent_id": "def456",
  "agent_type": "Explore",
  "agent_transcript_path": "~/.claude/projects/.../subagents/agent-def456.jsonl",
  "last_assistant_message": "Analysis complete. Found 3 potential issues..."
}
```

**五大問題回答**

1. 完整 Hook events 清單：上述 9 種（非代理人聲稱 18+）
2. 是否有 TaskCompleted/SubagentStop 類 event：
   - SubagentStop ✅ 存在
   - SubagentStart、TaskCompleted ✗ 不存在（代理人幻覺）
3. Stop vs SubagentStop：
   - Stop = main thread 停止（非 agent）
   - `run_in_background: true` 代理人完成觸發 **SubagentStop**（非 Stop）
4. SubagentStop input context：含 `agent_id`、`agent_type`、`agent_transcript_path`、`last_assistant_message` — **足以讓 active-dispatch-tracker 精準清理對應 task**
5. 不需其他機制（SubagentStop 已足夠）

### 架構升級可行性

**目前保守策略（W10-060/061）**：
- background 派發時 Hook 跳過驗證邏輯（因 PostToolUse(Agent) 在啟動時觸發非完成時）
- 依賴 PM 主動查 dispatch-active.json + TaskOutput 確認狀態

**SubagentStop-driven 升級方案**：
- active-dispatch-tracker-hook 改掛 SubagentStop event：依 `agent_id` 精準清理對應記錄（目前 dispatch-active.json 以 agent_description 匹配不精準）
- agent-commit-verification-hook 改掛 SubagentStop event：真正完成時才驗證 commit（目前在啟動時誤驗證）
- 從根因層消除 PC-050 模式 E（Hook 訊號誤判）

**現場佐證**：當前 session `.claude/dispatch-active.json` 殘留 7 筆舊記錄（agent_description 空 ticket_id）未被 PostToolUse(Agent) 清理成功 — 正是 SubagentStop 升級要解決的精準清理問題。
