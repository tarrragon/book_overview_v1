## Problem Analysis

### Phase 0 前置分析（PM 前台，2026-04-16）

#### 現有架構分析

**agent-commit-verification-hook.py**（636 行），掛 PostToolUse(Agent)，功能 5 大塊：

1. **未 commit 檔案偵測**（get_uncommitted_files）— 排除 .claude/ 和 docs/ 路徑
2. **Worktree 未合併偵測**（get_unmerged_worktrees）— 檢查非 main 的 worktree 分支
3. **Feature 分支未合併偵測**（get_unmerged_feature_branches）— 排除 worktree 已涵蓋的
4. **Hook error 摘要**（scan_hook_errors）— 掃描最近 5 分鐘 hook-logs
5. **PM 立即動作摘要**（build_pm_action_summary）— 綜合前四項動態生成

**核心問題**（第 526-537 行）：Background 派發時 PostToolUse 在啟動時觸發，代理人還沒工作，所有檢查都會誤報。現用 is_background_dispatch 完全跳過。

#### W10-066 共享基礎

W10-066 建立了 SubagentStop 基礎設施（dispatch_tracker.py 新 API + SubagentStop event binding）。W10-067 可複用：
- SubagentStop event 已有 binding（W10-066 新增）
- agent_id 可從 SubagentStop input 取得
- 觸發時機天然正確（代理人真完成時），不需 background 跳過邏輯

#### Scope 分析

Ticket 有 5 個 AC，分兩大塊：

**A. Hook 升級**（AC1-2, AC5）：
- 改掛 SubagentStop event（或同時保留 PostToolUse 做 CWD 提醒）
- 移除 background 跳過邏輯
- 測試覆蓋兩路徑

**B. 文件更新**（AC3-4）：
- pm-agent-observability.md 新增 SubagentStop 為第五觀察工具
- PC-050 模式 E 防護章節附註長期解法

#### 技術考量

1. **事件選擇**：commit verification 所有 5 大功能都需要在代理人「真完成」後觸發，改掛 SubagentStop 是正確選擇
2. **SubagentStop input schema 差異**：SubagentStop 無 tool_input（不含 description/isolation/run_in_background）。需調整：
   - agent_description 可從 last_assistant_message 或 dispatch-active.json 交叉查詢（W10-066 的 lib API 可用）
   - isolation（worktree 判斷）需從 dispatch-active.json entry 的 branch_name 欄位推斷
3. **CWD 還原提醒**：可能仍需在 PostToolUse 觸發（因為 CWD 污染發生在代理人執行期間，SubagentStop 時 CWD 可能已被污染）
4. **Hook 體量**：636 行認知負擔高（> 300 行閾值），Phase 4 應考慮拆分

#### 依賴

- W10-066 Phase 3b（basil 執行中）必須先完成：SubagentStop event binding 和 dispatch_tracker.py 新 API
- 文件更新部分（AC3-4）可與 Hook 升級並行
