## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

**直接原因**：ticket skill src（`.claude/skills/ticket/ticket_system/`）行為改動後，相關決策層文件（`.claude/pm-rules/decision-tree.md`、其他 `pm-rules/*.md`、`SKILL.md`）沒有同步更新機制；仰賴 PM 事後察覺再補償，W17-114（決策樹閉環流程）與 W17-113（禁用無 trigger 延後決策）即此類事後補償的明確案例。

**深層原因**：行為層（ticket skill 程式邏輯）與決策層（PM 規則／決策樹）耦合鬆散——程式可直接修改，但決策文件需要人類事後同步思考。當前無自動化或強制性機制連結兩者，memory feedback「ticket skill 行為變更需同步檢查決策樹」雖已沉澱原則，但缺結構化落地。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 三路徑 ROI 評估表

| 路徑 | 形式 | 成本 | 效益 | 局限 | 對齊規則 |
|------|------|------|------|------|---------|
| **A. PC error-pattern** | 撰寫 PC-118 描述「ticket skill 行為變更未同步決策層」反模式 | 低（~30 分鐘，~80-120 行 markdown） | 經驗沉澱、可被引用、自動進入 PC 索引 | 純被動描述、無強制機制；PM 須先閱讀過 PC 才會回顧 | document-writing-style 三明示原則 + PC 索引機制 |
| **B. rules 規則** | 補強 `rules/core/` 或 `pm-rules/`：ticket skill 行為變更時必須掃描相關決策層引用並同步 | 中（~1-2 小時，~50-80 行 + parallel-evaluation 多視角審查） | 規則自動載入、PM/agent 形成檢查清單；可在 commit 訊息或 review 階段提醒 | 仰賴 PM 自律；context 沉重時失效（PC-066）；無自動偵測 | rules/core 自動載入 + decision-trigger-binding 規則 1 |
| **C. commit-level hook** | PostToolUse Bash matcher（git commit）：偵測 commit 範圍含 ticket skill src 改動時，提示掃描 decision-tree.md / pm-rules/ | 高（~200 行 Python + 測試 + meta 防護） | 自動化強制邊界提醒；commit 是行為變更明確邊界（比 Write-level 更精準）；不會被忽略 | 設計風險：誤報（修 bug 不需同步）、meta hook 自我引用（PC-099 教訓）；維護成本高 | hook system 設計慣例 + bash-tool-usage-rules |

### WRAP 擴增的額外路徑（檢驗排除）

| 路徑 | 排除原因 |
|------|---------|
| **D. PR template checklist** | 本專案多為直接 commit，PR template 對 PM 自律觸達率不高 |
| **E. 自動文件生成** | decision-tree.md 含 PM 心智模型與判斷邏輯，無法從 ticket skill src 反向自動產生 |
| **F. SessionStart 周期 audit** | 與 C 重疊但時機鬆散，誤報率高且對當前 commit 回饋延遲大 |

### 決策結論（狀態 a：已決策結論）

**採 A + B + C 三層組合**

理由：

1. 單獨 A：純描述無防護力，事後補償會繼續發生
2. 單獨 B：rules 自律在 context 沉重時失效（PC-066），需 hook 兜底
3. 單獨 C：缺經驗沉澱（PC-118）和規則指引（rule），PM 收到 hook 提示時無對照標準
4. 組合 A+B：缺強制性，hook 兜底缺位
5. 組合 B+C：缺 PC 描述，hook 觸發時 PM 無「為何要做」的依據
6. **組合 A+B+C**：三層覆蓋（描述層 → 規則層 → 強制層），各自彌補單獨路徑局限；總成本仍可控（~5-7 小時）

### W17-115.3 hook 偵測邏輯設計（acceptance 第 3 條要求）

**觸發路徑**：PostToolUse hook，Bash matcher 匹配 `git commit` 命令成功完成。

**偵測流程**：

1. 取得 commit 的 file list（`git show --name-only HEAD`）
2. 檢查是否含 `.claude/skills/ticket/ticket_system/**` 路徑（ticket skill src 改動）
3. 若是，進一步檢查 commit msg 是否含 `feat` / `refactor` 詞元（行為變更性質）；排除純 fix / test / docs commit
4. 滿足條件時輸出提示

**提示內容**：

- 列出本次 commit 的 ticket skill src 改動檔案
- 引用待掃描清單：
  - `.claude/skills/ticket/SKILL.md`（對外契約）
  - `.claude/pm-rules/decision-tree.md`（決策路由）
  - `.claude/pm-rules/*.md`（情境 SOP）
- 提示：「若本次改動屬行為變更，請執行 `grep -rln 'ticket track' .claude/pm-rules/` 確認上述文件無過時引用」

**meta 防護**（PC-099 / PC-110 教訓應對）：

| 防護機制 | 實作方式 |
|---------|---------|
| 路徑白名單豁免 | 本 hook 自身路徑（`.claude/hooks/ticket-skill-sync-check-hook.py`） |
| ticket 級豁免 | commit 範圍含 ticket skill src 但同 commit 已含 decision-tree.md / pm-rules/ 改動 → 視為已同步，不提示 |
| 訊息級別 | INFO（不阻擋 commit），僅作提醒 |
| 命名衝突防護 | 與 doc-sync-check-hook 區分：本 hook 專注 ticket skill ↔ 決策層同步，不取代 doc-sync-check |

**reference**：`.claude/hooks/doc-sync-check-hook.py`（262 行同步檢查 hook）作為設計骨架。
