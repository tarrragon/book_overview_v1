### 對比理想摩擦力分佈

依摩擦力管理方法論「決策點增加摩擦、執行點降低摩擦」原則：

| 階段 | 現況 | 理想 | 倒置程度 | 理由 |
|------|-----|------|--------|------|
| Proposal 評估 | 低 | **高** | **嚴重倒置** | 決策點：頻率低、不可逆（錯誤假設蔓延所有 Phase）、跨版本影響 |
| Phase 1 規格 | 低 | **高** | **嚴重倒置** | 決策點：規格錯誤放大到測試/實作/重構，修復成本隨階段指數上升 |
| Phase 0 SA | 中 | 中高 | 略倒置 | 決策點（系統一致性） |
| Phase 2 測試 | 中 | 中 | OK | 半決策半執行 |
| Phase 3a 策略 | 中 | 中 | OK | 半決策半執行 |
| Phase 3b 實作 | 高 | 低 | **輕度倒置** | 執行點：高頻、可逆、局部影響（10+ Hook 可能過度防衛） |
| Phase 4 重構 | 中高 | 中 | 略過度 | 已由多視角提供 |

### 關鍵發現

1. **Proposal 評估嚴重倒置**：11 個已接受的 PROP 中，無一個經過強制 WRAP 或多視角審查。錯誤提案假設會蔓延所有後續 Phase，是最根本的摩擦力缺口。

2. **Phase 1 規格嚴重倒置**：W10-034 已證實 /spec 執行落差 + validate 維度不足。Phase 1 的錯誤成本遠高於 Phase 3b 的錯誤成本（後者可透過 Red-Green 迭代修正，前者會形成全版本結構性偏差）。

3. **Phase 3b 輕度倒置**：10+ Hook 形成高摩擦執行環境，可能與摩擦力方法論 v2.0「連續 5 次無錯降級」原則不符。需審計哪些 Hook 已滿足降級條件但仍保持高摩擦。

4. **摩擦力方法論 v2.0 本身的缺口**：現有方法論聚焦「單一動作的象限判斷」（象限 A/B/C），**缺少「開發流程各階段的摩擦力曲線」章節**。方法論層面需補強。

5. **倒置的結構性原因**：Hook 系統是被動防禦（發現問題才補 Hook），前期決策環節因為「錯誤較晚才顯現」沒有對應的 Hook 設計壓力。後期實作環節因為錯誤立即可見，Hook 自然累積。

## Meta 分析結論

### 根因

開發流程中前期決策環節（Proposal、Phase 1）摩擦力**嚴重倒置**，與摩擦力管理方法論「決策點增加摩擦、執行點降低摩擦」原則相反。結構性原因為 Hook 系統被動防禦（錯誤立即可見處累積 Hook），前期決策環節錯誤較晚顯現因此缺乏 Hook 設計壓力。

### 再平衡策略

**P1（本 Wave 必做）**：
- 方向 1：Proposal 評估強制 WRAP + 多視角（子 M-1）
- 方向 2：Phase 1 /spec 執行 + 偵測（由 W10-034 A+B 承接，本 Meta 不重複）
- 方向 5：friction-management-methodology v3.0 加「流程階段摩擦力曲線」章節（子 M-2）

**P2（本版本或下版本）**：
- 方向 3：Phase 0 SA 衝突檢查強化（子 M-4）
- 方向 4：Phase 3b Hook 密度審計與降級（子 M-3）

**P3（方法論整合後再推）**：
- 方向 6：Hook 設計指引前期階段 Hook 先行（子 M-5）

### 與 W10-034 的關係

W10-034 是本 Meta ANA 的 **Phase 1 層單點實例**。本 Meta 提供整體視角與策略架構，W10-034 提供 Phase 1 層的具體實作方案（A+B）。兩者互補，不重複。

### 子 Ticket 建立策略

本 session 建議優先建立 M-1、M-2（P1），其餘可隨用戶節奏排入 Wave。M-1 本身又是 ANA，實作前需再做一輪 Reality Test（盤點 PROP 評估品質）。

### 對摩擦力方法論的反饋

現有 v2.0 聚焦「單一動作象限判斷」，本 Meta ANA 過程暴露方法論缺少「流程階段摩擦力曲線」視角。v3.0 擴充後，本 W10-035 的 Reality Test 盤點可作為方法論範例。
