## Problem Analysis

### 背景資訊（WRAP 分析必讀）

**問題發現經過**：
2026-04-11 派發代理人實作 W3-003 後，PM 看到 git status 只有 Ticket 文件變更就判定「代理人失敗，只更新日誌沒有實作程式碼」，準備重新派發。但用戶從 terminal 上看到大量 `JSON validation failed` 錯誤，指出 PM 的判斷完全錯誤。

**三個層次的問題**：

1. **PM 判斷偏誤**：
   - 跳過了 pm-role.md 定義的「失敗判斷前置步驟」（檢查分支、worktree、dispatch-active.json）
   - 看到表面現象就歸因，沒有調查根因
   - 違反規則：「只有 dispatch-active.json 為空且所有分支都沒有代理人的 commit 後，才能判定代理人失敗」

2. **Hook error 可見性缺口**：
   - terminal 上的 Hook error 訊息只有**用戶**看得到
   - **代理人（subagent）看不到** — Hook error 是 CLI UI 層面的提示，不注入代理人 context
   - **主線程（PM）也看不到** — 背景代理人的 terminal 輸出不回傳給主線程
   - 結果：PM 和代理人都對 Hook 系統問題渾然不知

3. **錯誤歸因模式**：
   - PM 的預設假設是「代理人能力不足」而非「環境有問題」
   - 這與 PC-005（CLI failure assumption attribution）是同類偏誤
   - 需要在判斷流程中加入「環境檢查」步驟

**WRAP 分析方向**：
- W：有哪些方式讓 Hook error 對 PM/代理人可見？（日誌檔、summary 注入、dispatch 前環境檢查）
- R：Claude Code CLI 是否提供 Hook error 的程式化存取？（API、日誌檔、env var）
- A：修改 PM 判斷規則 vs 修改 Hook 錯誤傳播機制，投入產出比？
- P：如果選擇修改規則但不改機制，PM 仍然看不到錯誤，規則會被遺忘
