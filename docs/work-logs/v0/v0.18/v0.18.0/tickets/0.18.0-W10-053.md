## Task Summary

診斷 subagent Edit .claude/ 路徑被權限層拒絕 + Bash sed/awk 被 bash-edit-guard 攔截的雙向阻塞根因，提案放寬方案

## Problem Analysis

### 過時性聲明

本 ticket 建立於 15 天前，原假設「subagent 在主 repo 主分支也無法 Edit .claude/」。該假設已在 W17-041 被實測推翻，並升級至 `.claude/methodologies/dispatch-rules.md`（ARCH-015）。

### 問題根因

原始假設錯誤。CC runtime 對 `.claude/` 的 hardcoded 保護是 **per-worktree** 而非 per-target-path：subagent 從主 repo cwd 派發時，即便 target 在 `.claude/` 也能 Write/Edit；只有外部 worktree 才被擋。W10-049.4 thyme agent 當時的「雙拒絕」回報，根因為派發 cwd 為 worktree（非主 repo），與 ARCH-015 限制相符，非框架缺陷。

### 影響範圍

- 已落地：`.claude/rules/core/pm-role.md`「派發位置」段、`.claude/methodologies/dispatch-rules.md`（ARCH-015）
- 不需修改：`.claude/hooks/`、`.claude/settings.json`（原假設要放寬的目標）

### 相關 Error Pattern

無新增；本 ticket 結論本身可作為「過時 ticket 經實證後關閉」的範例。

## 重現實驗結果

### 實驗方法

引用 W17-041 已執行的實證（v2.1.114+）：在主 repo cwd 派發 subagent 嘗試 Edit/Write `.claude/` 下檔案，並對照 worktree cwd 的相同操作。

### 實驗執行

W17-041 紀錄：

| 派發 cwd | target | 操作 | 結果 |
|---------|--------|------|------|
| 主 session 工作樹（主 repo） | `.claude/**` | Write/Edit | 通過 |
| feature/* worktree | `.claude/**` | Write/Edit | 拒絕（CC runtime hardcoded） |
| 任意 cwd | `docs/**`、`src/**` | Write/Edit | 通過（一般規則） |

### 實驗發現

**已驗證事實**：分界線是「派發 cwd 是否在主 session repo 樹內」，不是「target path 是否在 .claude/」。

**原假設已推翻**：「subagent 在主 repo 主分支也無法 Edit .claude/」為錯誤；該情境通過。

**第二題（Bash sed/awk 攔截）**：經查 `.claude/rules/core/bash-tool-usage-rules.md`，sed/awk 是 framework best-practice 引導（避免該等工具修檔），非 hardcoded block。遇到時改用 Edit 即可，不構成阻塞，不需獨立 ticket。

#### 2. Bash sed/awk 攔截（原假設另一半）
