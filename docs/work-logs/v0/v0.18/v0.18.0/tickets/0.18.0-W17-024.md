## Top 3 建議（IMP 範圍描述）

### 建議 1：`ticket track complete --check-all`（ROI 30.0）

**範圍**：在 `complete` 子指令增加 `--check-all` flag，執行 complete 前自動呼叫 `check-acceptance --all`。

**動機**：幾乎每次 complete 前都要先 check-acceptance --all，是最高頻的固定二步序列。

**實作要點**：在 complete handler 內，若 `--check-all` 傳入，先呼叫 check_acceptance(all=True) 再執行 complete 邏輯。預估 0.5h。

### 建議 2：`ticket create --claim`（ROI 24.0）

**範圍**：在 `create` 子指令增加 `--claim` flag，建立後自動 claim 給 `--who` 指定的代理人（或 PM 自己）。

**動機**：PM 自建 Ticket 後立即 claim 是最常見序列，尤其 ANA/DOC 類型 PM 自己做的場景。

**實作要點**：create 成功後呼叫 claim handler。預估 0.5h。

### 建議 3：`ticket create --inherit-from <source_ticket_id>`（ROI 4.2，但解決 PC-100 系統性問題）

**範圍**：在 `create` 子指令增加 `--inherit-from` flag，自動從 source ticket 的 Problem Analysis 提取 Context Bundle 並 append-log 到新 ticket。

**動機**：PC-100 指出 ANA 衍生 IMP 時空殼 PCB 是系統性問題。Bundle 後 LLM 無需記住「create 完要 append-log 繼承 context」。

**實作要點**：讀取 source ticket md → 提取 Problem Analysis 區塊 → append-log 到新 ticket。需處理 section 解析。預估 2-3h。與 `--source-ticket` 可共用但語意不同（source-ticket 管 spawned 關係，inherit-from 管內容繼承）。

### 未入選但值得觀察

- **P9 batch-set**（ROI 7.2）：開發成本低但使用場景較散，可在 P2/P3 實作後視需求追加。
- **P4 append-log --replace**（ROI 5.3）：有實際需求但頻率不夠高，延後。
- **P7 error-pattern CLI**（ROI 2.1）：開發成本高且頻率低，不建議近期投入。
