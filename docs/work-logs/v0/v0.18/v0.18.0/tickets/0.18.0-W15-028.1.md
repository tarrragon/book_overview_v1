### 變更摘要

新增 `.claude/references/cli-exit-code-rules.md`（v1.0.0），作為 PC-096 防護的 SSOT 規範文件。

### 章節結構

1. **適用範圍** — 列出本規範適用的 CLI 命令類別與豁免項目（純語法錯誤類保留 sys.exit(1)）
2. **強制規則**
   - 規則 1：三值 exit code 定義（0=GO/SUCCESS、1=INTERNAL_ERROR、2=BUSINESS_REJECT/NO-GO），含 Why/Consequence/Action 三明示
   - 規則 2：路徑分離（except 與業務判定使用不同 return 值），含 Why/Consequence/Action
3. **Unix 慣例對照** — grep / diff / test 三值對照表，並說明本規範與 grep 慣例的對應差異（本規範 2=business reject，grep 2=error）及差異原因
4. **Phase 1 設計檢查清單** — 8 項檢查供 CLI 命令規格 review 使用
5. **套用範例**
   - 正例：track_handoff_ready.py:62-97（四條路徑語意清晰）
   - 反例：track_audit.py:127-145（業務拒絕與 except 共用 return 1）
6. **與既有錯誤訊息機制的銜接** — 說明 ArgparseFormatErrorParser / ErrorEnvelope / format_error 與本規範的協同關係（exit code = 機器訊號；ErrorEnvelope = 人類可讀說明）
7. **與其他規則的邊界** — 對照 PC-096 / quality-baseline 規則 4 / PC-005
8. **檢查清單 + 相關文件**

### 設計決策

- 採 2=business reject 而非 2=error：跟隨專案先行案例的既成事實，避免再造混亂。差異原因在「Unix 慣例對照」章節明示。
- 正反例皆引實際檔案行號：方便後續 IMP（W15-028 修復違規點）對照施工。
- 強制 exit code 與 ErrorEnvelope 同 commit 釋出：避免分階段釋出造成人類除錯資訊缺失。

### 三明示覆蓋

每個強制規則（規則 1、規則 2、Phase 1 檢查清單、format_error 銜接）皆含 Why / Consequence / Action 三段，符合 .claude/rules/core/document-writing-style.md 三明示原則。
