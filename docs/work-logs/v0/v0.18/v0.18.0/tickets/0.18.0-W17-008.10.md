## Solution

### 1. 角色對照表

| 參數 | 類型 | 實際角色 | 目前使用狀態 |
|------|------|---------|------------|
| `version_str` | positional 必填 | 指定版本號查詢目標（如 `0.31.0`） | 有效使用（track_query.py:540） |
| `--version` → `version_param` | optional flag | 預期備選版本指定方式 | 未使用（dead code） |

### 2. 使用場景分析

**使用場景 1 - 推薦方式（目前使用）**
```bash
/ticket track version 0.31.0
# 等價於：/ticket track version v0.31.0
# 執行流程：args.version_str = "0.31.0" → 正常執行
```

**使用場景 2 - 嘗試使用 --version flag（功能未實現）**
```bash
/ticket track version --version 0.31.0
# 預期行為：指定版本進度摘要
# 實際行為：args.version_param 被設定但從未被讀取
# 結果：TypeError（args.version_str 缺失）
```

### 3. 核心問題

**混淆點**：
- `--version` 在 Unix CLI 慣例中通常指程式版本（如 `git --version`），但此處表示"目標版本"
- `ARG_VERSION_STR` 描述「版本號（如 0.31.0 或 v0.31.0）」vs `ARG_VERSION_PARAM` 描述「版本號（自動偵測）」—— 兩者文件說明互相矛盾
- `version_param` 定義但未使用 = dead code，造成維護負擔

**根因**：設計中可能原意提供多種版本指定方式（positional 或 flag），但實現只完成 positional，導致不一致

### 4. 重命名建議與向後相容策略

#### 方案 A：移除 --version flag（推薦）
**根據**：
- `version_param` 完全未使用
- 單一職責：positional argument 清晰且必須
- 簡化 CLI 表面積

**實施步驟**：
```python
# 移除第 397 行：
# p_version.add_argument("--version", dest="version_param", help=...)

# 改進幫助文本（第 396 行）：
p_version.add_argument("version_str", help="版本號（如 0.31.0 或 v0.31.0；v 前綴可選）")
```

**向後相容**：
- 如未曾有使用者呼叫 `track version --version 0.31.0`，移除無影響
- 若需保留相容性，可在 execute_version() 中檢查 args.version_param 並優先使用

#### 方案 B：保留並實現 --version flag（代價大）
**若選此方案**：
1. 在 track_query.py:540 改為：
   ```python
   target_version_str = args.version_param or args.version_str
   ```
2. 更新幫助文本為更清晰的名稱（見方案 C）
3. 維護成本：每個版本命令變體都需測試

#### 方案 C：重命名以澄清語意（最佳實踐）
若選方案 B，建議改為：
```python
# 更清晰的旗標名稱
p_version.add_argument("--target-version", dest="target_version", 
                      help="目標版本號（可選；預設取 positional 參數）")
```
這樣：
- 避免與全域 `--version` 衝突
- 名稱與角色一致
- 文件與實作相符
