### CE-3 逐欄位去留決策

#### 欄位 1: `signals[S1].failure_detection` (primary + fallback)

**原始設計意圖**（Phase 3a L~1331 YAML 註解）:
- `primary: agent_commit_verification` — 讀取 agent-commit-verification-hook 的判定結果
- `fallback: task_output_error_marker` — 從 tool_response 抓取 error/exception/failed/timeout 關鍵字

**實作現況**: 完全未讀取；Python `_is_failure` L401-L414 直接硬編碼檢查 `status == failed/error` 再 substring match `error/exception/failed/timeout`。

**與 linux-1+C2 的關係**: 同一問題的兩個視角 — linux 看到「硬編碼違反 W10-052」，CE-3 看到「YAML 設計意圖未實現」。兩者應合併處理。

**決策**: **部分實作（簡化）** 合併於 .3

實作方向（給 .3）:
1. YAML 結構簡化為：
   ```yaml
   failure_detection:
     keywords: ["error", "exception", "failed", "timeout"]
     structured_statuses: ["failed", "error"]
   ```
2. Python `_is_failure` 改讀 `sd.raw.get('failure_detection', {}).get('keywords', [])` 和 `structured_statuses`
3. **移除** primary/fallback 二分命名（YAGNI — 跨 hook agent_commit_verification primary 策略尚未需要，需要時再加）
4. 滿足 W10-052 source-of-truth 約束（移除 Python 硬編碼常數）

---

#### 欄位 2: `signals[S1].ticket_context_key`

**原始設計意圖**（Phase 3a YAML 註解）: 跨 ticket 切換重置用（key 名稱可配置）

**實作現況**: Python `apply_ticket_switch_reset` L535-L543 硬編碼使用 `state['current_ticket']`，未讀 YAML flag。

**決策**: **移除** 於 .3

理由:
- state schema 在 `_initial_state` L220-L226 已內化定型為 `current_ticket` 固定 key
- 跨 hook 不共享 state，無 key 重命名需求
- 過度設計（speculative generality），違反 YAGNI
- 移除後 Python 邏輯不變（已正確使用 current_ticket）

---

#### 欄位 3: `output.include_signal_id`

**原始設計意圖**（推測）: 控制訊息是否顯示 signal_id（追蹤哪個 signal 觸發）

**實作現況**:
- `render_message` L558-L563 的 safe dict 無條件填入 `signal_id: sd.id`
- 所有 3 個 message_template 都未使用 `{signal_id}` placeholder
- flag 讀取 0 次

**決策**: **移除** 於 .3

理由:
- 當前訊息已用 `{matched_keyword}` / `{count}` / `{ticket_id}` 等 placeholder，清楚看出是哪個 signal
- 額外顯示 signal_id 收益小（冗餘訊息）
- 移除同時應從 `render_message` safe dict 移除 `'signal_id': sd.id` 這一行（清理無用程式碼）

---

**原始設計意圖**（推測）: debug 用 — 提醒訊息附上 state 快照

**實作現況**: 完全未實作；YAML 預設 `false`；Python 無對應邏輯。

**決策**: **移除** 於 .3

理由:
- 功能完全未實作
- debug 情境已有 logger.info 日誌足夠
