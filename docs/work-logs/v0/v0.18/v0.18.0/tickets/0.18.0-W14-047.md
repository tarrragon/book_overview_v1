## Problem Analysis

### 問題根因

雙層失誤共同造成：

- **L1（PM 規則缺口）**：`.claude/skills/ticket/references/migrate-command.md` 無前置檢查條款。`handoff-command.md:120` 有「前置檢查（強制）」條款（檢查殘留 pending handoff），但 migrate 規則缺對等的「檢查目標版本既有 ticket」要求。PM 配置批量遷移目標 ID 範圍時未強制 `ls` 目標目錄。
- **L2（工具缺陷）**：`.claude/skills/ticket/ticket_system/commands/migrate.py:_migrate_single_ticket` 兩處缺 collision check：
  - **Bug A** `migrate.py:295-299` dry_run 區塊僅 print 標題狀態，未呼叫 `target_path.exists()`
  - **Bug B** `migrate.py:343-345` 實際執行直接 `save_ticket(ticket, target_path)`，預設覆寫既有檔案不警示

### 影響範圍

- 工具：`.claude/skills/ticket/ticket_system/commands/migrate.py` line 260-360
- 規則：`.claude/skills/ticket/references/migrate-command.md`（缺前置檢查章節）
- 學習：`.claude/error-patterns/process-compliance/`（缺對應 PC 文件）
- 本次事件案例：v0.19.0 W1-001~W1-003（v1.0 路線圖 IMP 父級 ticket）被覆寫為 ANA 監測 ticket；靠 git status 在 commit 前發現才得以 rollback（commit d1780ccb 為 W3-001~W3-011 重做後的成功遷移）

- **memory IMP-061**：migrate 工具 parent_id typo 且不同步更新依賴欄位——與本次不同類型（IMP-061 是欄位內容 bug，本次是 target collision 預設覆寫）

## Solution

### W (Widen) — 候選方案

| # | 方案 | 優 | 缺 |
|---|------|---|---|
| 1 | 雙層防護（推薦） | L1 規則 + L2 工具強制；PM 與工具任一層都有保護；符合「不依賴單點自律」原則 | 實作成本較高（規則 + 工具兩處） |
| 2 | 純工具層 | 工具強制，PM 無自律負擔 | 規則層無更新；PM 仍可能誤用 `--force-overwrite` 旗標 |
| 3 | 純規則層 | 實作成本低 | 依賴 PM 自律；工具層無強制保護；不符 PC-066 反 autopilot 原則 |
| 4 | 共用 handoff 的 `_validate_target_ticket_exists` | DRY 統一架構 | handoff 要求「目標必須存在」，migrate 要求「目標必須不存在」，語義相反難共用 |

### A (Attain distance) — 機會成本

- 本 ANA 屬 P2 防護性質，無高優先 ticket 受阻
- 並行 wave 任務（W14-022 charset guard 重構）無依賴關係
- 若不處理：未來版本遷移時撞 ID 風險仍存在（已用 W10-119 / W17-188 等版本作多次遷移，下次必撞）
- 機會成本低，預期收益高（防止 v1.0 路線圖被破壞重演）

### P (Prepare to be wrong) — 失敗預想

| 失敗模式 | 防護 |
|---------|------|
| 規則層更新後 PM 仍不自律 | 同時做工具層強制（方案 1 而非方案 3） |
| 工具層強制後出現繞過旗標濫用 | `--force-overwrite` 預設拒絕；旗標使用須記錄至 audit log |
| 規則 + 工具修了但 PC 未升級，未來新類別失誤無防護 | 落地 PC error-pattern 文件，提供未來同類失誤識別模板 |

### 採行方案：方案 1（雙層防護）

| 層級 | 修復內容 | 落地位置 |
