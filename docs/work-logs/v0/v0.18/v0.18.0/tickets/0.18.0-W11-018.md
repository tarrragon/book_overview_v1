#### 1. 演算法虛擬碼：compute_fenced_block_lines(lines) -> Set[int]

語言無關，CommonMark 0.31 fenced code block 子集（FENCE-1~7）。單次線性掃描 O(n)。

```
INPUT: lines = List[str]（1-based 行號對應 idx）
OUTPUT: fenced_lines : Set[int]

CONST INDENT_MAX = 3            # FENCE-6：indent ≤ 3 才視為 fence
FENCE_START_REGEX = ^(?P<indent> {0,3})(?P<fence>BACKTICK{3,}|~{3,})(?P<info>.*)$
  # FENCE-3：info string 不影響邊界
  # backtick fence 的 info 不可含 backtick（CommonMark）；tilde fence 可含任何字元

STATE:
  fenced_lines ← empty set
  in_block     ← False
  open_char    ← None         # '`' or '~'
  open_len     ← 0            # 起始 fence 字元數

FOR idx, raw IN enumerate(lines, start=1):
    IF NOT in_block:
        m ← FENCE_START_REGEX.match(raw)
        IF m matched:
            # FENCE-6 / EDGE-8：Tab 視為 4 空格 → indent_visual ≥ 4 不啟用
            indent_visual ← visual_width(m.indent)   # Tab=4
            IF indent_visual > INDENT_MAX:
                CONTINUE
            # backtick fence 的 info string 不可含 backtick
            IF m.fence starts with '`' AND '`' in m.info:
                CONTINUE
            in_block  ← True
            open_char ← m.fence[0]
            open_len  ← len(m.fence)
            fenced_lines.add(idx)       # FENCE-4：起始 fence 行屬區塊
    ELSE:
        # 已在區塊中：先加入後再判斷是否 close（確保 close 行也屬區塊）
        fenced_lines.add(idx)
        IF is_closing_fence(raw, open_char, open_len):
            in_block  ← False
            open_char ← None
            open_len  ← 0

# FENCE-5：未閉合 fence 至檔尾 — 因迴圈中 in_block 期間每行已 add，自然涵蓋
RETURN fenced_lines


FUNCTION is_closing_fence(raw, open_char, open_len):
    # close fence：同字元、長度 ≥ open_len、indent ≤ 3、fence 後僅尾部空白
    m ← match ^(?P<indent> {0,3})(?P<fence>[BACKTICK~]+)\s*$ AGAINST raw
    IF NOT m matched:               RETURN False
    IF visual_width(m.indent) > 3:  RETURN False
    IF m.fence[0] ≠ open_char:      RETURN False  # EDGE-6：混用不閉合
    IF len(m.fence) < open_len:     RETURN False  # EDGE-4：短 fence 不閉合長 fence
    RETURN True
```

關鍵不變量：
- in_block 為 True 期間，當前行先加入 fenced_lines 再檢查 close（保 FENCE-4）。
- 起始 fence 字元決定整個區塊閉合字元（EDGE-6）。
- Tab visual width = 4（EDGE-8 / CommonMark）。

## Phase 1 功能規格：Fenced Code Block 範例語境豁免

### 1. Purpose（需求分析）

**核心問題**：phase4-decision-enforcement-hook.py 現有 4 種豁免機制（REF_LINE / SCHEMA_PLACEHOLDER / hook_self_reference / EXEMPT_PROXIMITY）皆不覆蓋 Markdown fenced code block 範圍。當規則文件 / ticket md / error-pattern 在 ```...``` 區塊內示範 PC-093 反模式（如「Phase 5 再決定」「之後再說」）或展示 exempt marker 範例時，validator 會：

- 將範例中的延後話術誤判為實際 BLOCK / WARN / INFO 命中
- 將範例中的 `<!-- PC-093-exempt: cat:reason -->` 誤收為實際豁免宣告，進入豁免審計清單造成噪音

**使用者場景**：
- PM / agent 撰寫規則文件（`.claude/rules/`、`.claude/methodologies/`）需在 code block 內展示反模式
- error-pattern 文件需引用實際違規語句作為案例
- ticket md 的 Solution / Problem Analysis 章節以 fenced block 引用 commit message / 規格片段
- spec 文件展示 marker 使用範例供讀者複製

**現有 4 豁免機制覆蓋對照**：

| 機制 | 覆蓋範圍 | 不覆蓋 fenced block 的原因 |
|------|---------|--------------------------|
| REF_LINE_PATTERN | 行首 `- [ref]` 或 `[ref]` | 限 Context Bundle 自動抽取行 |
| SCHEMA_PLACEHOLDER | `<!-- Schema[type/section]: ... -->` 至下個 H2/--- | 限 ticket body schema 範例 |
| hook_self_reference | frontmatter 宣告整檔豁免 | 限 hook 自我設計 ticket |
| EXEMPT_PROXIMITY | marker 同行或前 1 行 | 限實際宣告豁免，非範例展示 |

結論：fenced code block 是普遍 Markdown 結構，現有四機制皆為「特定文件結構」豁免，缺通用「範例展示語境」豁免。

### 2. 功能規格

#### 2.1 偵測規則：Fenced Code Block 邊界

採 CommonMark 0.31 fenced code block 子集規則（覆蓋 99%+ 實務用法）：

| 規則 ID | 規則 | 說明 |
|--------|------|------|
| FENCE-1 | 起始 fence | 行首（允許 0-3 空格 indent）出現 3+ 連續 backtick 或 3+ 連續 tilde |
| FENCE-2 | 結束 fence | 同類字元（backtick / tilde 不可混用）≥ 起始 fence 長度，行內不得有其他內容（允許尾部空白） |
| FENCE-3 | Language hint | 起始 fence 後同行的字串視為 info string，不影響邊界判定 |
| FENCE-4 | 範圍 | fence 起始行與結束行**自身**亦屬區塊範圍（fence 本身的字元不會被誤判為 phrase / marker） |
| FENCE-5 | 未閉合 | 起始 fence 後文件結束前未見配對結束 fence，視為「fence 至檔尾」皆屬區塊（容錯處理） |
| FENCE-6 | Indented fence | indent ≤ 3 空格的 fence 視為有效 fenced block；indent ≥ 4 空格屬 indented code block，**不在本次處理範圍**（範圍限縮見 §4） |
| FENCE-7 | Nested fence | 不支援巢狀（CommonMark 規範亦不支援）；內層相同字元 fence 若長度 < 外層起始長度則視為內容，否則閉合外層 |

#### 2.2 豁免效果

落入 fenced code block 範圍的行（含起始 / 結束 fence 自身行）：

| 處理對象 | 行為 |
|---------|------|
| Phrase 掃描（M1-M3 / W1-W3 / I1-I2） | 整行跳過，不產生 Hit |
| Exempt marker 蒐集 | 整行跳過，不蒐集為 ExemptRef（避免範例 marker 進入豁免清單噪音） |
| Hit 豁免距離匹配 | N/A（因 phrase 已不命中） |

#### 2.3 與既有 4 豁免機制的關係（優先級與覆蓋順序）

採「先範圍排除，再規則匹配」分層設計，避免機制間互相干擾：

| 順序 | 階段 | 機制 | 早期退出 |
|------|------|------|---------|
| 1 | 檔級 | hook_self_reference（frontmatter 宣告） | 整檔 return 0 |
| 2 | 行級範圍 | **Fenced code block**（本 ticket 新增） | 行級 skip |
| 3 | 行級範圍 | SCHEMA_PLACEHOLDER 區塊 | 行級 skip |
| 4 | 行級範圍 | REF_LINE_PATTERN | 行級 skip |
| 5 | 行級內容 | EXEMPT_MARKER（marker 解析） | 命中後 EXEMPT_PROXIMITY 匹配 |
| 6 | 行級內容 | Phrase regex 掃描 | 命中產生 Hit |

**關鍵設計決策**：fenced block 排在 SCHEMA_PLACEHOLDER 之前，因 fenced block 是更通用的 Markdown 結構；SCHEMA_PLACEHOLDER 是 ticket-body-schema 專屬 HTML comment template，兩者區塊邊界互不重疊（schema placeholder 用 `<!-- ... -->`，fenced block 用 backtick / tilde），不會互相干擾。

#### 2.4 API / 介面定義

新增函式（位於 `phase4-decision-enforcement-hook.py`）：

```python
FENCED_BLOCK_START = re.compile(r"^(?P<indent> {0,3})(?P<fence>`{3,}|~{3,})(?P<info>[^`]*)$")
# 注意：tilde fence 的 info string 可含 backtick；backtick fence 不可

def compute_fenced_block_lines(lines: List[str]) -> set:
    """計算 fenced code block 內的 1-based 行號集合（含 fence 自身行）。

    規則：
    - 起始 fence: indent ≤ 3 空格 + 3+ 連續 ` 或 ~
    - 結束 fence: 同字元 + ≥ 起始長度 + 行內僅尾部空白
    - 未閉合 fence: 視為至檔尾
    - 不支援 indented fence（indent ≥ 4）
    - 不支援 nested fence（CommonMark 規範一致）

    回傳：所有屬於 fenced block 的行號集合（含起始與結束 fence 行）。
    """
```

整合點：
- `scan_lines_for_phrases`：在 placeholder_lines 檢查後新增 fenced_lines 檢查
- `collect_exempt_markers`：同樣新增 fenced_lines 檢查
- 兩函式共用同一個 `compute_fenced_block_lines(lines)` 呼叫結果（建議在 main 中計算一次後傳入，或維持各函式內部計算保持 API 一致性 — Phase 3 實作決定）
