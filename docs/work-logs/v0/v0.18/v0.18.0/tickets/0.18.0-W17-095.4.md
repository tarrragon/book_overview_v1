### 實作摘要

新建 `.claude/skills/ticket/tests/test_handoff_stale_consistency.py`，固化「handoff_utils / stop-hook / reminder-hook 三方對同一 handoff record 的 stale 判斷必一致」的整合測試。未改動任何 production 程式碼。

### 設計決策

1. **共用 stub 層**：以 `_ScenarioStub` 替換 handoff_utils 內 4 個底層探測函式（`is_ticket_in_progress_or_completed` / `is_ticket_completed` / `extract_version_from_ticket_id` / `load_and_validate_ticket`），讓三方入口共享同一份 ticket 狀態認知，避開實際 ticket fs 依賴。

2. **三方入口共用同一 `is_handoff_stale` 物件**：stop-hook / reminder-hook 在 import 時將 `is_handoff_stale` 綁定為 module 層級名稱。測試前手動將 `hook.is_handoff_stale` 重新指派至 reload 後的 utils_module 函式物件，確保 patch utils 內部探測時三方都受影響。

3. **reminder-hook 透過 tmp_path 寫實體 JSON 檔**：scan 函式從目錄載入，故用 tmp_path fixture 隔離。stop-hook 直接餵 dict（其 should_preserve 是 record-level 函式）。

4. **9 case 矩陣 + 3 parametrized 一致性檢查**：除 9 個獨立 case 驗證個別入口行為，再加 3 個 parametrized test 直接斷言「三方對同一 record 結論必一致」（is_stale ↔ not preserve ↔ stale_count），讓未來新入口加入時可直接擴展矩陣。

### 邊界遵守

- 僅新建 test 檔，未改 handoff_utils.py / stop-hook / reminder-hook / handoff_gc.py
- git add 精確路徑（test 檔 + ticket md），未用 `git add .` / `git add -A`

### 不在本 ticket 範圍

- 新增第 4 個入口時的擴展（測試骨架已預備，但實際擴展屬未來工作）
