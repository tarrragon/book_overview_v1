設計機制解決 Ticket SPEC 與實際狀態漂移問題

---

## Problem Analysis

### 觸發場景（2026-04-12 Session）

PM 在 handoff 恢復流程中執行 `ticket resume 0.18.0-W2-001`（已完成）→ 被引導到 `0.18.0-W3-002`（驗證 100% 測試通過率）→ 發現 W3-002 被 W3-001 阻塞 → 準備 claim W3-001 前跑 `npm test` 先確認現況，結果發現：

| 測試類型 | 通過數 | 失敗數 | Skipped |
|---------|-------|--------|---------|
| 單元測試 | 4085 | 0 | 29 tests / 1 suite |
| 整合測試 | 634 | 0 | 12 tests / 1 suite |
| **合計** | **4719** | **0** | **41 tests / 2 suites** |

**W3-001 AC「整合測試全部通過、flaky test 修復」實際上已被上游 Ticket（W1-002、W2-001、W3-006）修復外溢達成。**

### 問題根因

1. **Ticket AC 有時效性**：建立當下（2026-04-03）有效，9 天後可能已被其他 Ticket 外溢達成
2. **修復外溢效應**：單元測試修復可能連帶影響整合測試，反之亦然
3. **PM 依賴警覺性**：本次靠 PM 主動跑測試才發現，差點派發代理人做「已完成工作」
4. **Ticket 系統缺乏 AC 實況對照能力**：`ticket track query` 只顯示 frontmatter 的 `[ ]` 標記，不會實際執行驗證

### 影響範圍

- `pm-rules/decision-tree.md` — PM claim 前的驗證流程
- `pm-rules/ticket-lifecycle.md` — Ticket 生命週期階段規則
- `rules/core/quality-baseline.md` — 品質檢查清單
- `.claude/skills/ticket/` — CLI 命令（特別是 claim 流程）
- `.claude/hooks/` — 可能需要新 hook 自動偵測漂移

### 相關 Error Pattern

待查詢 `error-pattern query drift` 確認既有記錄。初步判斷這是新的模式（PC-055 候選）。
