## Problem Analysis

### PM 前台 Phase 1 分析（2026-04-29）

#### 問題根因

- `process-skip-guard-hook.py:66-76` 的 SKIP_SA_REVIEW 模式採純 keyword pair 偵測，無上下文感知；任何 IMP/DOC/ANA 任務只要文字命中 ("跳過","sa") / ("不做","審查") 就觸發提醒
- SKIP_ACCEPTANCE 含 ("不做","審查") pair，與 SKIP_SA_REVIEW 重疊但無區分；同一句話可能命中多模式
- "直接/完成"（SKIP_ACCEPTANCE）、"跳過/sa"（SKIP_SA_REVIEW）這類組合在重構/修復/維護任務中是合法措辭（例：「直接修復後完成」），高假陽性
- detect_skip_intent (lines 128-150) 僅讀 user_input_lower，未讀取 ticket frontmatter

#### 影響範圍

- .claude/hooks/process-skip-guard-hook.py（233 行，主檔）
- .claude/hooks/lib/ask_user_question_reminders.py（532 行，提醒文案邏輯）
- 相依工具：.claude/hooks/lib/frontmatter_parser.py:159 `read_ticket` 已具備 ticket 解析能力
- Active ticket 來源候選：(a) `ticket track list --status in_progress` (需 subprocess) (b) docs/work-logs/v*/tickets 直接掃描 in_progress

#### AC 漂移檢查（PC-055）

Ticket 建立 17 天，逐項對照 acceptance：
- AC1（派發前查詢 ticket type 和 current_phase）：仍適用，是 type/phase guard 核心需求
- AC2（IMP + Phase 4 排除 SA 提醒）：仍適用
- AC3（DOC/ANA type 全面排除）：仍適用
- AC4（移除/改寫高假陽性關鍵字 pair）：仍適用，今日驗證 SKIP_ACCEPTANCE.("直接","完成") 與 SKIP_SA_REVIEW.("不做","審查") 仍是高 FP 組合
- AC5（測試覆蓋三種排除場景）：仍適用
- AC6（既有測試持續 GREEN）：仍適用

#### 建議實作策略

1. 在 detect_skip_intent 觸發 SKIP_SA_REVIEW 後，讀取當前 in_progress ticket（用 frontmatter_parser.read_ticket）
2. 排除規則：
   - `type ∈ {DOC, ANA}` → 全面靜音
   - `type == IMP AND current_phase == "Phase 4"` → 靜音 SKIP_SA_REVIEW
3. 關鍵字精確化：
   - 移除 SKIP_ACCEPTANCE.("直接","完成")（與「直接修復完成」重疊度過高）
   - 移除 SKIP_SA_REVIEW.("不做","審查")（與一般審查討論重疊）
   - 保留 ("跳過","sa")、("不需要","sa") 等顯式 SA 詞
4. 性能：active ticket 只在偵測到 skip intent 後才讀取（cold path）
5. 失敗回退：若無 in_progress ticket 或讀取失敗，維持原行為（不靜音）

#### 相關 Error Pattern

- PC-007（接手獨立驗證範圍）— 已執行，hook 邏輯與 AC 對應確認
- PC-055（pending >7 天 AC 漂移）— 已逐項審查
