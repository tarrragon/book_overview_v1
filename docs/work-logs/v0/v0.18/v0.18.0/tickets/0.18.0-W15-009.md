### R — Reality Test（落地可行性）

| 設計點 | 可行性 | 風險 |

| T7 S3 訊息擴充 | 高（只改 yaml message_template） | 訊息變長用戶忽略，需分段提示 |
| T3 SKILL 章節 | 高（純文件更新） | 無強制力，PM 可能遺漏 — 由 tdd-flow.md Checkpoint 補強 |

### A — Attain distance（機會成本）

- 若建獨立 Hook：~200 LOC + 測試 + 註冊；後續 triggers 擴充要雙處維護
- 方案 4 成本：~20 行 yaml + hook ~30 行 category 判斷 = 小增量
- 對下個 Ticket（W15-010 終止條件）的 context 排擠：低（本 Ticket 設計細節已由 W15-012 框定）

### P — Prepare to be wrong

| 失敗模式 | 防護 |

| T6 關鍵字「再想想」誤觸（用戶只是閒聊） | 加 min_prompt_length ≥ 30 + 需在質疑上下文（可改進但先上 substring） |

## 方向調整（2026-04-18 用戶 reframe）

原方向：設計「自動觸發反思」機制對抗 LLM 短路徑 bias。

用戶 reframe 後發現：
- LLM 短路徑偏好本身**符合** friction-management-methodology（摩擦力是中性工具，執行點/象限 A 應低摩擦）
- 真正的問題是「**必要時加摩擦**」（決策點/象限 C）而非「對抗短路徑偏好」
- PC-088 原框架「對抗 bias」與本專案摩擦力方法論衝突，已重寫為「步驟數估算偏誤 + 決策點摩擦」框架

### 調整後目標

不是「在所有 tool call 前加 meta-check」，而是「**決策點強制摩擦 + 執行點保持短路徑預設**」。

4 個觸發點重新定位：
- T6（用戶質疑）：決策品質檢查，對應象限 C
- T7（ANA claim）：決策點（分析任務）
- T3（Phase 4 重構）：決策點（品質強化）
- T5（連續錯誤）：原定義為「偵測」，調整為「累計 2 次同錯才觸發反思」（與 friction 升級訊號一致）

### AC 調整

原 AC「4 觸發點優先級排序」→ 調整為「4 觸發點對應摩擦力象限分類 + 升級/降級訊號」

### 1. wrap-triggers.yaml 擴充（新增 reflection_triggers 類別）

```yaml
# 新增訊號定義（並列於既有 signals）
signals:
  # ... 既有 S1/S2/S3 維持 ...

  - id: reflection_depth_challenge
    category: reflection_trigger  # 新增欄位，區別於 wrap_tripwire
    enabled: true
    event_sources: ["UserPromptSubmit"]
    keywords:
      - "太表層"
      - "不夠深"
      - "再想想"
      - "這解釋不了"
      - "為何不是"
      - "更深一層"
      - "還有其他可能嗎"
      - "introspection"
    match_mode: "substring"
    case_sensitive: false
    min_prompt_length: 30
    reset_conditions:
      - manual_reflection_invocation
      - session_end
    message_template: |
      [Reflection Trigger] 偵測到反思深度質疑（關鍵字：{matched_keyword}）。
      建議採用 three-phase-reflection 方法論：
        Phase 1: 列 5+ 候選假設 + Reality Test + 2 層深因
        Phase 2: WRAP 檢驗 Phase 1 結論
        Phase 3: 與既有方法論對照檢查（friction/cognitive-load/ai-communication）
      參考：.claude/methodologies/three-phase-reflection-methodology.md
```
