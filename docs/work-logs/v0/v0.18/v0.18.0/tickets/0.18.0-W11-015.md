## Problem Analysis

### 問題根因

snapshot 測試耗時 ~9s/test 的雙層成因（2026-05-18 PM 二度驗證確認仍成立）：

1. **`_scan_all_versions` 仍被無條件呼叫**：
   - `track_snapshot.py:65` `_render_full_snapshot` 路徑無條件呼叫 `_scan_all_versions()`
   - `track_snapshot.py:123` `_render_degraded_snapshot` fail-open 路徑同樣呼叫
   - 兩條路徑都會迭代 `docs/work-logs/` 全目錄（量測：169 個版本目錄 / 189 個總目錄）

2. **既有 mock 顆粒度錯位**：
   - `test_track_snapshot.py:63-72` `_patch_checkpoint_state` helper 用 `monkeypatch.setattr(mod, "checkpoint_state", fake)` 替換 state 推導
   - 但 `_render_full_snapshot` 與 `_render_degraded_snapshot` 在拿到 mocked state 後**仍走 `_scan_all_versions()` 真實 I/O**
   - 結論：mock 上游不等於下游被遮蔽（常見「mock 顆粒度錯位」陷阱）

3. **真正耗時熱點推測非 `_scan_all_versions` 本身**：
   - `_scan_all_versions` 純做 `iterdir() + regex match`，不開檔，169 目錄 I/O 應 < 1s
   - 真正熱點推測為後續 `_print_version_progress` → `list_tickets(version)`，**為每個版本載入所有 ticket md 檔案**
   - W11-015 原始 acceptance「mock `_scan_all_versions` fixture」設計範圍可能不足以達 < 100ms AC

## Solution

### 修復摘要

於 `.claude/skills/ticket/ticket_system/tests/conftest.py` 新增 autouse fixture `_mock_track_snapshot_filesystem_scan`：

- mock `track_snapshot._scan_all_versions` → 回傳單一假版本清單 `["0.0.0-fixture"]`
- mock `track_snapshot.list_tickets` → 回傳空清單
- `raising=False`：允許 track_snapshot 模組不存在時靜默跳過（不影響其他測試集）
- 包覆 `try/except ImportError`：對非 snapshot 測試零侵入

### 設計決策

| 決策 | 選擇 | 理由 |
|------|------|------|
| Mock 位置 | `track_snapshot` 模組命名空間（非 `ticket_loader.list_tickets` 原址） | track_snapshot.py:37 用 `from ticket_loader import list_tickets`，已建立模組級綁定；mock 必須對齊綁定點（IMP-064 同家族陷阱） |
| autouse=True | 是 | 17/17 snapshot 測試都需要，個別加 fixture 反覆 reference 增加噪音 |
| Mock 範圍 | 兩個函式 | Phase 1 量測證實 `_render_full_snapshot` + `_render_degraded_snapshot` 兩條路徑都呼叫 `_scan_all_versions`，且下游 `_print_*` 三函式 × 每版本 `list_tickets` 為主要熱點 |
| 假資料內容 | 單一假版本 + 空 ticket 清單 | 既有測試斷言聚焦 header / 區塊存在性（"--- 版本進度 ---"），不檢查具體內容 |
| `_patch_checkpoint_state` helper | 保留 | 與 fixture 組合（fixture 處理 I/O，helper 處理 state 推導） |

### 不在本 ticket 範圍

- 不改 `track_snapshot.py` 產品碼（範圍限制）
- 不延伸 mock 至 `_get_git_branch`（S8 測試需原始 subprocess spy 行為，不可遮蔽）
- IMP-064 同家族「Mock 顆粒度錯位」error-pattern 升級：本次修復作為驗證樣本，是否沉澱為獨立 PC 由 PM 決定
