## 用戶補充：Handoff 系統原始設計哲學（2026-04-14）

W10-036 不僅修正「父子責任傳遞」，應同時記載 handoff 系統的原始設計意圖：

### 核心設計哲學

**Ticket 結構應產生「任務鏈」（Task Chain），支援在鏈上上下 / 水平移動而不損失 context。**

- **上下移動**：父↔子（ANA 父 → ANA/IMP/DOC 子 → 子之子）
- **水平移動**：兄弟（同 parent 下的子 ticket 互相交接）
- **Context 保留**：handoff 機制確保角色切換時 context bundle 完整傳遞

### 本 session 現成範例

本次 session 自然產生了典型任務鏈結構，可作為設計哲學範例：

```
W5-002 (IMP, Phase 1 規格) ─ Phase 1 PM 驗收暴露缺口
    ↓ 衍生上層
W10-034 (ANA, Phase 1 規格完整性) ─ 單點分析
    ↓ 衍生更上層
W10-035 (ANA, Meta 摩擦力倒置) ─ 結構性分析
    ├─ W10-035.1 ANA Proposal 評估強制機制
    ├─ W10-035.2 DOC friction v3.0 流程階段曲線
    ├─ W10-035.3 ANA Phase 3b Hook 密度審計
    ├─ W10-035.4 IMP SA 系統衝突檢查
    └─ W10-035.5 ANA Hook 設計指引前期先行
    ↓ 衍生理念修正（橫向）
W10-036 (DOC, 父子責任理念修正) ← 本 ticket
```

### 對本 Ticket scope 的影響

W10-036 應同時處理以下關聯面向，不只處理單一責任傳遞：

1. **任務鏈原則**：文件化「Ticket 結構 = 任務鏈」作為設計基礎
2. **上下水平移動的規範**：
   - 上移：子 ticket 發現更大問題時建上層 parent 或 related-to 橫向 ANA
   - 下移：父 ticket 產生 spawned children
   - 水平：兄弟間 handoff（handoff to-sibling）
3. **Context 保留機制**：handoff 檔案內容、append-log 鏈結、chain 參照如何保證 context 不損失
4. **父子責任傳遞**（原本範圍）：父 complete 需子全部 completed/closed
