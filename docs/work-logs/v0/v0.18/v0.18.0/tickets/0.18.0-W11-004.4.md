## Task Summary

檢討 session 間殘留狀態導致 hook 誤報（dispatch-active 殘留 + 模式變更遺留）

## Problem Analysis

### 問題根因

Session 間殘留狀態有兩類來源：

**來源 A：dispatch-active.json 記錄殘留**

- **寫入時機**：`dispatch-record-hook.py`（PreToolUse Agent）寫入新記錄
- **正常清理**：`subagent-stop-dispatch-cleanup-hook.py`（SubagentStop handler）清理已完成的記錄
- **超時清理**：`active-dispatch-tracker-hook.py`（PostToolUse Agent）呼叫 `cleanup_expired(max_age_hours=4)`
- **根因**：session 異常結束（crash/force-quit/`/clear`）時 SubagentStop 未觸發，記錄殘留。超時 GC 的 4 小時 TTL 遠大於一般 session 間隔，短時間（如 35 分鐘）內重啟會看到 stale 記錄。且 GC 只在 PostToolUse(Agent) 時觸發——若新 session 未派發代理人，GC 完全不會執行。
- **session-start hook 缺 GC**：`session-start-sync-exclusion-check-hook.py` 已將 `dispatch-active.json` 列入 KNOWN_EXCLUDED（識別為 runtime state），但未執行任何清理動作。

**來源 B：working tree 未提交變更遺留**

- hook_output_validator.py 的 chmod +x 變更來自前 session 代理人執行，session 結束時未 commit
- 此類遺留非 dispatch-active 問題，屬 session-switching-sop 執行不完整（/clear 前未 commit）

### 影響範圍

- `.claude/dispatch-active.json`：殘留記錄導致多個 PostToolUse Hook 反覆提醒「N 個代理人執行中」
- `.claude/hooks/lib/hook_output_validator.py`：chmod 變更觸發「有未提交變更」警告
- 影響所有依賴 dispatch-active 狀態的 Hook（wrap-decision-tripwire、agent-commit-verification 等）

### 相關 Error Pattern

- PC-076（跨 session 未提交遺留）已涵蓋來源 B
- 來源 A（dispatch-active GC 不足）尚無 error-pattern

### 重現實驗結果

**已驗證事實**：
1. 當前 dispatch-active.json 含 3 筆記錄（W11-004.3/4.4/4.5），dispatched_at 均為 2026-04-29T07:33，距今約 8 小時——已超過 4h TTL，下次 PostToolUse(Agent) 會被清理
2. `cleanup_expired` 預設 max_age_hours=4，只在 `active-dispatch-tracker-hook`（PostToolUse Agent）中被呼叫
3. `session-start-sync-exclusion-check-hook` 識別 dispatch-active.json 為 runtime state 但不清理
4. hook_output_validator.py 目前無未提交變更（已被某個 session 解決）

**仍未驗證假設**：
- /clear 是否觸發 SubagentStop event（推測不會，需實驗確認）

## Solution


---

### 建議方案

#### 方案 A（推薦）：session-start hook 加入 dispatch-active GC

在 `session-start-sync-exclusion-check-hook.py` 啟動時呼叫 `cleanup_expired(max_age_hours=1)`。

**優點**：
- 最自然的 GC 時機——新 session 開始即清理前 session 殘留
- 不依賴 PostToolUse(Agent) 觸發，即使新 session 不派發代理人也能清理
- TTL 降至 1 小時（session 間隔通常 > 1 小時），幾乎涵蓋所有 stale 場景
- 改動最小（session-start hook 已 import pathlib，只需加 dispatch_tracker import + 一行呼叫）

**缺點**：
- session-start hook 原則上應「不阻塞」，但 cleanup_expired 是 O(n) 讀寫 JSON，n 通常 < 10，耗時可忽略

#### 方案 B：降低 active-dispatch-tracker 的 TTL

將 `cleanup_expired(max_age_hours=4)` 降為 `max_age_hours=1`。

**優點**：改動一行
**缺點**：仍依賴 PostToolUse(Agent) 觸發，若新 session 不派發代理人則 GC 不會執行

#### 方案 C：dispatch-record-hook 寫入時先 GC

在 PreToolUse(Agent) 寫入新記錄前先呼叫 cleanup_expired。

**優點**：寫入前清理，確保讀取時不含 stale
**缺點**：PreToolUse 路徑增加延遲；仍需代理人派發觸發

### 推薦組合

**方案 A + B**：session-start 加 GC（TTL=1h）+ active-dispatch-tracker TTL 從 4h 降為 1h。雙觸發點確保覆蓋。

### 來源 B 處理（chmod 遺留）

已有 PC-076 覆蓋。session-switching-sop 已規定 /clear 前 commit。本 ANA 不重複建 ticket。
