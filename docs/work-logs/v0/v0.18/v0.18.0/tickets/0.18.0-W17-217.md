## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`phase-completion-gate-hook.py` 已具備 worklog 主檔排除邏輯（W10-072.1 落地），但 regex 過於嚴格，漏匹配 `-main.md` 命名變體：

```python
# .claude/hooks/phase-completion-gate-hook.py:128
WORKLOG_MAIN_FILE_REGEX = re.compile(r"^v\d+\.\d+\.\d+\.md$")
```

此 regex 只匹配 `v0.18.0.md` 格式，**不匹配**本專案實際使用的多種主檔命名：

| 實際命名格式 | 範例 | regex 命中？ |
|------------|------|-----------|
| `v{minor}-main.md` | `v0.18-main.md` | 否 |
| `v{patch}-main.md` | `v0.18.0-main.md` | 否（本 session 三次誤判） |
| `v{patch}-work-log.md` | `v0.10.4-refactor-work-log.md` | 否 |
| `v{patch}.md` | （理論上） | 是（但專案無此檔案） |

實際結果：所有 `*-main.md` 與 `*-work-log.md` 編輯都會通過排除檢查，進入 `is_phase_completion_report()` 內容判定，觸發空 schema 警告。

### 影響範圍

| 範圍 | 影響 |
|------|------|
| 本 session 受影響檔案 | `docs/work-logs/v0/v0.18/v0.18.0/v0.18.0-main.md` 編輯 4 次皆觸發誤判 |
| 全專案 worklog 主檔 | 12+ 個 `*-main.md` + `*-work-log.md` 命名檔案皆受影響 |
| PM 注意力成本 | 每次 worklog 進度條目更新都產生 stderr 警告 + 訊息疲勞累積 |
| 真正警告品質下降 | autopilot 化忽略警告（PC-066 模式）→ 真正 Phase 報告問題可能漏掉 |

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

multi_view_status: skipped
reason: 本 ANA 根因明確（regex 漏匹配 `-main.md` / `-work-log.md` 命名變體），修復方向 = 擴充 regex；不涉及架構或設計決策，故不需多視角審查；IMP 落地時跑現有 hook 測試覆蓋驗證 false negative 風險即可

### 結論

`phase-completion-gate-hook.py` 在 W10-072.1 已建立 worklog 主檔排除機制，但 `WORKLOG_MAIN_FILE_REGEX` 過於嚴格，漏匹配本專案實際使用的多種主檔命名變體（`-main.md` / `-work-log.md`）。屬於「排除清單實作與現實脫節」的 hook 設計問題，與 PC-142（phase4-hook 字面誤判規則引用）為同家族。

### 修復方案候選

| 方案 | 描述 | 評估 |
|------|------|------|
| A. 擴充 regex（推薦） | `^v\d+(\.\d+)*(-(main\|work-log)(.*))?\.md$` 涵蓋 main/work-log suffix 變體 | 最小改動、最快落地、不影響既有 Phase 報告識別 |
| B. 多 regex 清單 | 分別維護 minor 主檔 / patch 主檔 / work-log 主檔等多個 regex | 維護成本高，新增命名格式需修多處 |
| C. 反轉白名單 | 只當 basename 命中明確 Phase 報告命名（如 `phase-N-completion.md`）才檢查 | 改動較大，需全專案搜尋現有 Phase 報告命名取得共識 |
| D. 純內容判定 | 忽略檔名，純看內容是否有 `## Phase X 評估` 等明確 Phase 標題 | 已有 `_has_phase_section_heading()` 邏輯可重用，但需重整 `is_worklog_write_operation()` 早期排除流程 |

**推薦方案 A**：最小改動原則，落地成本最低；既有測試覆蓋只需追加 4-6 個案例（`v0.18-main.md` / `v0.18.0-main.md` / `v0.18.0-work-log.md` / `v0.18.0.md`）。

### WRAP 四階段分析

#### W（Widen）

候選方案見上表 A/B/C/D。Consider the Opposite：相反方向 = 「完全移除主檔排除邏輯，回到原 W10-072.1 前狀態」— 拒絕（會誤判更多）。

#### R（Reality-test）

- 量化證據：本 session 三次連續觸發 + grep 確認 12+ 個專案實際主檔命名與 regex 不匹配
- regex 設計意圖（程式碼註解行 123）原本宣告「v{major}.{minor}.{patch}.md 命名模式」，但本專案實際命名加 `-main` suffix；設計與實作斷層
- 風險評估：方案 A 的 false negative 風險可由 hook 既有測試套件 + 新增 4-6 個 fixture 案例完整覆蓋

#### A（Attain distance）

採方案 A 機會成本接近 0：
- regex 修改 + 測試 ≈ 1 個 IMP ticket（< 1 小時）
- 不採方案 A 的累積成本：每天每位 PM 處理 worklog ≈ 3-5 次 stderr 警告 → 訊息疲勞 → PC-066 autopilot 風險擴散

#### P（Prepare to be wrong）

最可能失敗：方案 A regex 過於寬鬆誤排除合法 Phase 報告（例如某新命名格式 `v0.18.0-phase-completion-report.md` 也命中 `-` suffix）。

防護措施：
1. IMP 落地時 regex 用具體 suffix 白名單（`main` / `work-log` / 未來明確列舉），不用泛化 `(-.*)` 通配
2. spawned IMP 必須附測試案例：3 正例（觸發排除）+ 3 反例（不應排除）
3. 修改後在本 session 4 個編輯歷史回放驗證
