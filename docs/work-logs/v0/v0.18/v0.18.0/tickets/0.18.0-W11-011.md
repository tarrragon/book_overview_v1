## Problem Analysis

## WRAP 分析

### 事實盤點

**觸發場景**：W10-074 任務鏈完成後 commit，commit-handoff-hook 顯示「場景 11 路由提示」：
```
有其他 Wave pending → #3a（Wave 收尾）
無任何 pending    → #3b（/version-release check）
```

**PM 實際情境**：Wave 10 有 82 pending + Wave 11 有 25 pending（W10-074 完成只是局部），總計 83+ pending。

**PM 錯誤行為**：生成 AUQ 3 個選項，選項 3 為「版本發布檢查」，description 含「仍有 83 pending，卻先確認顯然不時，但可行」——前後矛盾的 disclaimer 等於自承假選項，違反規則 5.6 機制 4（illusion of control）。

### R (Reality Test)

**規則是否清楚**：askuserquestion-rules.md:218 場景 3 寫「情境 C1：有其他 Wave → #3a；情境 C2：全完成 → 技術債整理 + /version-release check」——規則本身清楚區分 C1 vs C2，**是 PM 錯用，非規則模糊**。

**Hook 提示是否清楚**：post-commit hook 提示同樣明確「有 pending → #3a；皆無 → #3b」。**Hook 也無誤**。

**根因定位**：
1. **PM 未驗證前提就把兩種路徑都列為選項**——場景 3 的 C1 和 C2 是互斥分支，但 PM 把 C2 的動作（/version-release check）當作「次選」硬塞進 C1 情境的 AUQ 選項。
2. **缺乏「選項前提驗證」自檢**：AUQ 規則書未明文禁止「附 disclaimer 合理化不適用選項」。
3. **規則 5.6 機制 4 反諂媚已涵蓋原則層面**，但缺「AUQ 送出前檢查每個選項的前提條件」的操作層 checkpoint。

### W (Widen options)

| 方案 | 內容 | 成本 | 效果 |
|------|------|------|------|
| A | 修 hook 提示更嚴格（列具體條件） | 低 | 有限——PM 本就有判斷能力，hook 不變也能正確路由 |
| B | 新增 AUQ 規則「禁止假選項原則」，明定「選項前提不成立即刪除，禁用 disclaimer 合理化」 | 中 | 直擊執行層缺口 |
| C | B + 在 PM 規則 pm-rules/pm-role.md 或 rules/core/ai-communication-rules.md 加自檢 checkpoint | 中 | 規則層強化 |
| D | 接受、不改規則（視為個案） | 0 | 違反規則 6 失敗學習原則 |

### A (Attain distance)

**機會成本**：B 需修 1-2 檔 ~30 行；相比「假選項污染用戶決策品質」的長期代價（信任成本、PM 行為漂移），值得投入。

### P (Prepare to be wrong)

**最可能失敗**：修完規則後，PM 仍可能以其他形式產假選項（如「不推薦」「僅供參考」等變體 disclaimer）。

**防護**：規則 5.6 機制 4 已涵蓋 illusion of control，新增規則明文禁止「前提不成立選項」+「disclaimer 合理化」，形成**具體可檢查條款**而非抽象原則。

### 結論與建議

**選方案 B**（最小有效修正）：在 `.claude/pm-rules/askuserquestion-rules.md` 新增「禁止假選項原則」段，明定：

> **選項前提檢查（強制）**：產出 AUQ 選項前，必須對每個選項自問「此選項的觸發前提是否成立？」。若前提不成立（如 Wave 未全完成卻提 /version-release check），**必須刪除該選項**。禁止用 disclaimer（如「顯然不時但可行」「不推薦但列出」）合理化不適用選項——此類 disclaimer 即是「假選項」訊號，違反規則 5.6 機制 4。

**可在同 Ticket 收尾**（改動小，不需衍生 IMP）。修改後本 Ticket 可直接 complete，不建 follow-up。
