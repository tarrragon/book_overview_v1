### A: Attain distance（10/10/10 框架）

| 時間距離 | 影響評估 |
|---------|---------|
| 10 分鐘後 | 加 hook 後消除 markdown 誤選風險，程式碼仍可用 serena |
| 10 個月後（v0.20+ Tag-based Book Model + i18n） | JS 程式碼大幅變動，serena 跨檔 rename / find_references 價值上升 |
| 10 年後（MCP 生態演進） | LSP 標準化完成，serena 可能被原生 CC LSP 取代；屆時直接 disable hook 即可（單次操作，無遺留成本） |

### P: Prepare to be wrong（每選項可逆性與決策錯誤成本）

| 選項 | 決策錯誤成本 | 可逆性 |
|------|------------|-------|
| A | PC-112 在 agent 層規則衰減時復現；每次新 agent 派發踩雷 | 高（隨時可加 hook） |
| B | hook 邏輯誤擋程式語言副檔名（如 .vue / .svelte 含 markdown 區塊） | 高（disable hook 即可） |
| C | 「沒 serena 就不能跨檔 rename」；重裝成本約 1 小時（重啟 MCP） | 中（需重新配置） |
| E | 6 個月後 review 時可能忘記取消，變成隱性技術債 | 高（review milestone 設 ticket 強制） |

### 結論：推薦 B（即 W17-090 hook + rule）

**Why**：
- serena 對程式碼有實證價值（JS 符號樹解析有效）
- 文件操作不該用 serena（PC-112 確認）
- Hook 是 hard enforcement，agent 層 soft 規則會在 context 衰減時失效
- 可逆性高，未來想移除只需 disable hook + remove from settings

**移除（C）的條件**：
- 設 6 個月後 review（建 W18 或 W19 觀察 ticket）
- 觀察期間真實 mcp__serena__ 寫入工具呼叫次數
- 若 < 5 次/月 → 移除（無實際 ROI）
- 若 ≥ 5 次/月 → 維持限縮配置

**W17-090 不變更，可解除 blockedBy**：原計畫的 hook + rule 仍是正確方向，本 ANA 確認此方向無需調整。
