### 發現（三結論）

**結論 1：PM 層 serena read-only 使用率 0%，非偶有遺漏而是完全未啟用**

**結論 2：13 大檔樣本中 11 個是 markdown（規則/SKILL/worklog/ticket md），對 serena 本質價值有限**
serena 設計用於 LSP-based 符號級操作，對純 markdown 的 `get_symbols_overview` 只能切到 H2/H3 標題層級，與 Read+offset/limit 的 ROI 差距遠不到「5-10× token」等級。W17-091 引用之 migrate.py 529 行（Python 程式碼）屬少數真正 5-10× 受益情境。**這是「習慣未養成」之上的更深層原因：對 markdown-heavy 工作流，serena read-only ROI 本就不到一般想像的數量級**。

**結論 3：觸發語境多為「用戶簡短回應後 PM 反射式 Read」，無顯性探索階段可插 hook 提醒**
13 樣本中用戶語境多為 `'是'` / `'D'` / 空（連續 toolUse）。AskUserQuestion 後續或 ticket 切換 re-center 的反射動作沒有「先理解結構」的顯性決策點。PreToolUse hook 提醒會在每次大檔 Read 觸發 → 高頻噪音；精準提醒需更上層 session 任務類型判斷，成本高於收益。
