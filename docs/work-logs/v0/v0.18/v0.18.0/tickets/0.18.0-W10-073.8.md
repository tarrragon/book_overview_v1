### 判斷原則（父 ticket 共通指引）

**拆分依據是「認知負擔」與「按需載入」，不是行數。**

這是卡片盒筆記法（Zettelkasten）的核心精神：

- 正確問題：這段知識在什麼情境才需要理解？每段知識是否原子化？按需載入時能否獨立理解？
- 錯誤問題：檔案多少行？骨架是否夠短？行數超過某閾值就要拆？

**推論**：
- 500 行的檔案若內容全部原子化且在同一情境共同需要 → 不應拆分
- 100 行的檔案若混合多個情境才用到的知識 → 即使短也應拆
- 拆分後的每一碎片必須能獨立被理解（原子性），否則只是把完整概念切碎
- 保留判斷標準不是「這段很重要」，而是「所有使用場景都需要立即載入」

**WRAP 分析時的具體提問**：

### 2. R — Reality Test（現實測試）

**原子性與獨立理解性檢驗**：

- A/B（規則 1-4 + 核心理念）：**原子性 OK**——四條規則彼此正交且自洽，獨立讀也能懂，每條都綁定一個明確觸發情境。
- C（使用速查）：**非原子**——是 SKILL 中同主題內容的精簡版；SKILL 的工作流程示意圖與用途對照表使其更完整。
- D（反模式表）：**半原子**——SKILL 有同名反模式章節（第 151-159 行），條目略異但重疊度高。
- E（相關文件）：**非原子**——純導覽表，跳轉時再載入即可。

**保留在 auto-load 的價值判斷**（「所有場景都需立即載入？」）：

| 內容 | 每 session 載入的必要性 | 理由 |
|------|----------------------|------|
| 規則 1（五問） | 高 | PC-060 核心防護：觸發「宣告做不到前檢查」的心智閘門，失效即 regression |
| 規則 2（限制性 vs 探索性框架） | 高 | 獨立認知偏誤防護（framing bias），若藏在 SKILL，PM 不會在設計禁止規則時主動 Read |
| 規則 3（ToolSearch 通用性） | 中 | 防止「AskUserQuestion 專用前置」的錯誤框架重犯；每次用 ToolSearch 都受影響 |
| 規則 4（system-reminder 主動檢視） | 中 | 每 session 啟動時的行為提示；auto-load 才能觸發「首次掃一次」 |
| 使用速查 / 反模式 / 相關文件 | 低 | 執行時才需要；SKILL 已有更完整版本 |

**結論**：A+B 是「觸發層」（防護心智模型），必須 auto-load；C+D+E 是「執行層 + 導覽層」，按需查閱即可。

### 3. A — Attain Distance（保持距離）

**從其他專案 sync 視角檢視**：

- `.claude/` 會透過 sync-push 同步到所有專案。每個專案 session 啟動都承受 ~408 tokens（其中 C+D+E 約 ~180 tokens 是與 SKILL 重複的概念）。
- 假設 10 個專案 × 平均 20 sessions/週 × 52 週 = ~10,400 session-loads/年。若能節省 ~180 tokens/session，一年約節省 ~1.87M tokens 的重複載入。
- 更重要的機會成本：**概念重複讓規則位置不清**——C/D/E 在 rule 和 skill 都有，修改時可能漏改一邊導致不一致（現狀確實存在，如反模式條目兩邊略異）。

**從新人／新代理人視角**：

- 新 session 看到 auto-load 的 tool-discovery.md，若只看到 A+B（觸發規則）更聚焦；C+D+E 是「進一步執行細節」，應在 SKILL 裡一次看完整版，避免 rule/skill 間跳躍。
- 單一事實來源（SSOT）：使用速查、反模式、相關文件應以 SKILL 為 SSOT，rule 僅負責「何時觸發 SKILL」。
