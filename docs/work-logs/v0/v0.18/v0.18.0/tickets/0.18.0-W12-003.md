### 設計修復方案（至少 2 方案並列 + 權衡）

#### 方案 A：完整級差設計（scope 欄位 + hook 分支邏輯）

**改動**：
1. Ticket schema 新增 ANA `scope` 欄位：`protective`（防護性）/ `investigative`（研究性）
2. ana_spawned_checker.py 讀 scope：
   - `protective` → 檢查 spawned 全 terminal，否則 block
   - `investigative` → 維持現行（警告但不 block）
3. ticket create 建 ANA 時強制選 scope（AskUserQuestion）

**權衡**：
- 優：語義最清楚，防護性 ANA 自動守門
- 劣：新增 schema 欄位有遷移成本（既有 ANA 需回填）
- 劣：PM 建 ANA 時判斷「是否防護性」增加認知負擔

#### 方案 B：純紀律層（不改 hook，僅規則）

**改動**：
1. ticket-lifecycle 規則新增「防護性 ANA 清單」：列明哪些 ANA 需等 spawned IMP 完成才 complete
2. PM 手動遵守
3. acceptance-gate-hook 訊息補充「提醒 PM 檢查 spawned 狀態」

**權衡**：
- 優：零程式碼改動，立即可執行
- 劣：依賴紀律，PC-073/PC-072 已證紀律層不可靠
- 劣：清單維護難（誰決定一個 ANA 是否防護性）

#### 方案 C：旗標機制（Ticket-level 選擇性啟用）

**改動**：
1. Ticket frontmatter 新增 `spawned_completion_required: bool`（ANA only）
2. ANA 建立時 PM 判斷是否需要「spawned 全 terminal 才能 complete」
3. ana_spawned_checker.py 讀旗標，啟用時檢查狀態
4. ticket create 提示 ANA 類型「是否為防護性？」附預設值

**權衡**：
- 優：語義單純，不需要分類學
- 優：最小改動（新增 bool 欄位比新增 enum scope 簡單）
- 劣：仍依賴建 Ticket 時的正確判斷
- 優：旗標未填時維持現狀（不破壞歷史 Ticket）

#### 方案 D：最小改動 — 警告層（Hook 警告但不 block）

**改動**：
1. ana_spawned_checker.py 擴充：spawned 任一非 terminal 時輸出警告（不 block）
2. 警告訊息列出非 terminal 的 spawned ID 和狀態
3. PM 自行判斷是否繼續 complete

**權衡**：
- 優：零破壞性，行為可逆
- 優：立即觀察到問題（警告可見）
- 劣：仍依賴 PM 意志力，不能強制落地
- 優：實作成本最低，可做過渡方案
