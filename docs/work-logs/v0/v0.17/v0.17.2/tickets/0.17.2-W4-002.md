## CQRS 讀取模型設計

### 設計原則

不引入新的儲存層（如 SQLite），而是在現有 Ticket YAML frontmatter 基礎上新增 CLI 查詢子命令。CQRS 的「讀寫分離」體現在：
- **寫入端**：ticket create/set-*/append-log（已有，不變）
- **讀取端**：新增跨維度查詢命令（掃描 YAML frontmatter 聚合）

### 新增查詢命令建議

| 命令 | 用途 | 實作方式 | 優先級 |
|------|------|---------|--------|
| ticket track search --ref UC-01 | 查詢某 UC/Spec/Prop 相關 Tickets | 掃描 where.files + why 欄位 | P1 |
| ticket track search --file path | 反向查詢：哪些 Ticket 修改了某檔案 | 掃描 where.files 欄位 | P2 |
| ticket track list --version all --status pending | 跨版本待辦 | list 支援 --version all | P1 |
| ticket track decisions --version 0.17.2 | 已完成 Ticket 的技術決策摘要 | 掃描 execution log 提取關鍵結論 | P2 |

### 不建議的方案

| 方案 | 理由 |
|------|------|
| SQLite 讀取層 | 過度工程，Ticket 數量 < 500，YAML 掃描 < 1s |
| 額外索引檔案 | 需維護一致性，增加寫入端複雜度 |
| ElasticSearch 式全文搜尋 | grep/ripgrep 已足夠 |

### 規格文件標準化建議

| 建議 | 說明 | 優先級 |
|------|------|--------|
| 補齊 spec frontmatter | export-interchange-format-v2.md 等 3 個檔案 | P1 |
| doc nav 引用完善 | proposals-tracking.yaml 的 spec_refs 需補齊 | P2 |
| 無需新工具 | doc CLI 已足夠，重點是資料完整性 | — |

### 實作優先級建議

**Phase 1（P1，建議 v0.18.x）**：
1. ticket track list 支援 --version all
2. ticket track search --ref 跨維度查詢
3. spec frontmatter 補齊

**Phase 2（P2，建議 v0.19.x+）**：
4. ticket track search --file 反向查詢
5. ticket track decisions 技術決策摘要
6. doc nav 引用完善

**結論**：現有 Ticket + doc 工具已覆蓋 ~80% 查詢需求。主要缺口是跨維度查詢（UC/Spec→Ticket 和跨版本 list），可透過 2-3 個新子命令解決，無需引入新的儲存層。
