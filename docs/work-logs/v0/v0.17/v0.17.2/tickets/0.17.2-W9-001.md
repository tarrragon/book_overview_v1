## Problem Analysis

### 問題根因

**間歇性 hook error 的直接原因已被 IMP-048（W4-004/W5-001）修復**：

1. Hook 的 json.load(sys.stdin) 遇到含控制字元的 tool_output 時拋出 JSONDecodeError
2. 異常被 run_hook_safely 捕獲，_log_exception 無條件寫入 stderr
3. Claude Code 偵測 stderr 輸出，顯示 'hook error'

**IMP-048 修復內容**：
- 所有 Hook 的 json.load 加入 try-except 保護
- StreamHandler level 從 WARNING 提升為 CRITICAL
- 只有真正的未處理異常才寫入 stderr

**race condition 假說驗證結果：不成立**

| 測試 | 結果 |
|------|------|
| 12 hooks 並行（正常 input） | 0.3s，無 stderr |
| 12 hooks 並行（25KB input） | 0.3s，無 stderr |
| 5 輪壓力測試 | 全部通過 |
| 冷快取（刪 __pycache__） | 無影響 |
| Hook 日誌 CRITICAL 搜尋 | 無未處理異常記錄 |

### 影響範圍

12 個 PostToolUse:Bash hooks 每次 Bash 呼叫都並行啟動。

## 分析結論與精簡方案

### 1. Hook Error 頻率與並行 Bash 呼叫的關聯

**結論：間歇性 hook error 根因是 IMP-048（已修復），非平行執行 race condition。**

Claude Code hook runner 對每個事件平行啟動所有匹配 hooks，stdin JSON 獨立傳遞給每個進程。12 個 hooks 並行啟動約 0.3s，遠低於任何 timeout 設定。本地 5 輪壓力測試（含冷快取、大 input）均無法重現。

### 2. 可合併的 PostToolUse:Bash Hooks

| 合併組 | 當前 Hooks | 合併為 | 節省 |
|--------|-----------|--------|------|
| A: git commit | changelog-update, commit-handoff, post-commit-fetch | post-git-commit-hook.py | 3→1 |
| B: 測試 | test-timeout-post, pre-fix-evaluation | post-test-hook.py | 2→1 |
| C: CLI 錯誤 | cli-failure-help-reminder, skill-cli-error-feedback | cli-error-feedback-hook.py | 2→1 |
| D: ticket complete | session-context-guard, post-ticket-complete-checkpoint | post-ticket-lifecycle-hook.py | 2→1 |

不合併：handoff-cleanup（獨立 domain）、worktree-merge-reminder（獨立觸發條件）、parallel-dispatch-verification（獨立 domain）

### 3. 精簡方案

**方案：從 12 個合併為 7 個（節省 5 個進程）**

合併原則：
- 同一觸發條件（如 git commit）的 hooks 合併為一個 dispatcher
- 每個合併後的 hook 內部按 if/elif 分流到各子邏輯
- 保留獨立 domain 的 hooks 不合併

效益：
- 每次 Bash 呼叫從啟動 12 個進程降為 7 個
- 減少 ~40% 進程開銷
- 降低未來新增 hook 時的並行風險

優先級：P2（當前已穩定，合併是最佳化而非修復）
