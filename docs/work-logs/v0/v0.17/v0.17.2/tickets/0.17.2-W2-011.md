## 初步分析（2026-04-05 session）

### 觀察到的兩種截斷模式

**模式 A（IMP-047）**：回合全花在 Read/Grep，Write = 0
- 原因：agent 傾向先完整理解 codebase，大型專案 50+ 檔案耗盡回合
- 已有解法：prompt 直接提供程式碼（phase3b-dispatch-guide.md）

**模式 B（本 Ticket 要分析的）**：分析完成但 Write 被截斷
- SA agent：51 行 output、21 次 tool call、最後產出 6800 字 text
- 最後回合：先輸出 text（報告），接著要 Write 但未執行
- 報告內容完整，存在 agent output JSON 中但未寫入檔案

### 初步推測的截斷原因

1. **Token 上限**：subagent 可能有 output token 上限，6800 字 text + Write tool call 超出
2. **Turn 上限**：21 次 tool call 可能接近上限（settings.json 未找到明確 maxTurns 配置）
3. **組合上限**：最後回合 text + tool_use 的總 token 超出單次回應限制

### 需要進一步驗證

1. 檢查 Claude Code subagent 的實際 turn/token 上限（可能在 CLI 原始碼中而非 settings.json）
2. 比較模式 A 和 B 的 token 消耗量（不只是 tool call 次數）
3. 測試「Write-first 策略」（prompt 指示 agent 先寫檔再輸出摘要）是否有效
