# Execution Log

## Task Summary

修改 Context Bundle 規範和派發指南，context 存 Ticket 不存 prompt

---

## Problem Analysis

### 根因

phase3b-prompt-template.md L14 寫著「寫入 Ticket 的 Context Bundle 區段（或直接嵌入 prompt）」。這個「或直接嵌入 prompt」給了 PM 合法捷徑，繞過 Ticket 持久化。

context-bundle-spec.md L3/L9/L46 已正確定義「context 存 Ticket」原則：
- L3: PM 在派發前將資訊寫入 Ticket
- L9: Context Bundle 是 Ticket 執行日誌中的一個區段
- L46: 派發 prompt 只需 Ticket 路徑 + 動作指令

但 phase3b-prompt-template.md 的「或」字開了後門。

### 影響的文件清單

1. .claude/references/phase3b-prompt-template.md L14 — 含「或直接嵌入 prompt」後門
2. .claude/references/phase3b-prompt-template.md L133 — 禁止表只說不要求讀文件，沒說不要嵌入 prompt
3. .claude/pm-rules/context-bundle-spec.md — 規範正確但缺乏強制語氣

### 相關 Error Pattern

PC-040: Context 存 Prompt 而非 Ticket
## 問題分析

### 1. 盤點：現有規則中 context 傳遞的位置

共 4 個文件涉及 context 傳遞指引，其中 1 個存在矛盾措辭：

| 文件 | 立場 | 問題 |
|------|------|------|
| context-bundle-spec.md | context 存 Ticket（明確） | 無問題，核心規範正確 |
| phase3b-dispatch-guide.md | Context Bundle 優先（明確） | 無問題 |
| two-stage-dispatch.md | 程式碼放 Ticket，prompt 不含程式碼（明確） | 無問題 |
| phase3b-prompt-template.md (references/) | 矛盾措辭 | L14 寫「寫入 Ticket 的 Context Bundle 區段（或直接嵌入 prompt）」——給了 PM 繞過 Ticket 的許可 |

### 2. 根因：為什麼 PM 仍把 context 寫進 prompt？

PC-040 的根因不是規範缺失，而是規範存在矛盾出口：

1. phase3b-prompt-template.md 的「或直接嵌入 prompt」是唯一允許 prompt 存 context 的措辭。PM 在時間壓力下傾向走阻力最小路徑。
2. context-bundle-spec.md 沒有明確的反面禁止：雖然定義了 context 存 Ticket，但「禁止行為」表中沒有「禁止把 context 寫在 prompt 中」這一條。
3. phase3b-prompt-template.md 的模板格式本身就像是 prompt 模板：標題是「Phase 3b 代理人派發 Prompt 模板」，模板內容是給代理人看的 markdown，PM 自然把它當作 prompt 來寫。

### 3. How.strategy 欄位現況

Atomic Ticket 方法論中 How.strategy 的定義是「TDD 循環」這種抽象描述，未被設計為 context 載體。Context Bundle 規範選擇用 Ticket 的 Execution Log 區段而非 5W1H 欄位。這個設計是正確的——5W1H 是 Ticket 建立時的靜態資訊，Context Bundle 是階段轉換時的動態資訊。

### 4. 現有規範的整體評估

context-bundle-spec.md v2.0.0 的設計已經相當完整：通用模板 + Phase 指引的二層結構合理，1 Read 原則正確，PM 填寫流程清晰。

核心問題只有一個：phase3b-prompt-template.md 的矛盾措辭 + context-bundle-spec.md 缺少明確的反面禁止。

## 修改方案

### 修改 1：phase3b-prompt-template.md（references/）— 移除矛盾措辭

**位置**：`.claude/references/phase3b-prompt-template.md` L14

**現狀**：
> PM 在派發前將必要資訊寫入 Ticket 的 Context Bundle 區段（或直接嵌入 prompt）。

**改為**：
> PM 在派發前將必要資訊寫入 Ticket 的 Context Bundle 區段。代理人從 Ticket 讀取，不從 prompt 讀取。

**同文件標題**：建議從「Phase 3b 代理人派發 Prompt 模板」改為「Phase 3b Context Bundle 填寫模板」，避免 PM 誤解為 prompt 模板。

---

### 修改 2：context-bundle-spec.md — 新增反面禁止

**位置**：`.claude/pm-rules/context-bundle-spec.md` 禁止行為表（L68-73）

**新增一行**：

| 禁止 | 原因 |
|------|------|
| 把 context（規格摘要/程式碼/策略）寫在 Agent prompt 中 | prompt 是 ephemeral 載體，agent 失敗後 context 消失不可重用（PC-040） |

---

### 修改 3：context-bundle-spec.md — 新增 Agent Prompt 最簡模板

**位置**：`.claude/pm-rules/context-bundle-spec.md` 在「派發 prompt 只需」之後新增完整模板

**新增內容**：

```markdown
### Agent Prompt 模板（最簡版本）

所有 Phase 的 agent prompt 統一使用此格式：

    請執行 Ticket {ID}。
    Ticket 路徑：docs/work-logs/v{VERSION}/tickets/{ID}.md
    Context Bundle 已準備在 Ticket Execution Log 中，讀取後開始。
    完成後更新 Ticket execution log。

prompt 中禁止包含：規格摘要、程式碼範例、實作策略。這些資訊必須在 Ticket 的 Context Bundle 中。
```

---

### 修改 4：phase3b-dispatch-guide.md — 與 prompt 模板保持一致

**位置**：`.claude/pm-rules/phase3b-dispatch-guide.md` L24-28（簡單任務 Prompt 要點）

**現狀**：
> - 如有程式碼，放在 Ticket/設計文件中，prompt 只提供 Ticket 路徑

**改為**（措辭更明確）：
> - 程式碼和策略一律放在 Ticket Context Bundle 中，prompt 只提供 Ticket 路徑

### Context Bundle 存放結構總結

| Context 類型 | 存放載體 | 欄位/區段 |
|-------------|---------|----------|
| 需求摘要 | Ticket Execution Log | Context Bundle > 需求摘要 |
| API 簽名 | Ticket Execution Log | Context Bundle > API 簽名 |
| 實作策略/程式碼 | Ticket Execution Log | Context Bundle > 相關檔案 + inline |
| 關鍵檔案位置 | Ticket 5W1H | Where.files |
| 驗收條件 | Ticket 5W1H | acceptance |
| Phase 完成摘要 | Ticket Execution Log | Phase N 完成摘要 |

**不變的設計**：5W1H 是建立時的靜態資訊，Context Bundle 是階段轉換時的動態資訊，兩者共存在同一個 Ticket 文件中。

在 L46 之後新增強制規範段：

[強制] prompt 長度上限：Agent prompt 不得超過 30 行。超過表示 context 未正確存入 Ticket。
