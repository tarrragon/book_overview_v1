## Problem Analysis

現有流程缺口盤點：(1) claim 階段只有阻塞依賴檢查和建議性提醒（讀文件/理解 AC），無強制更新 5W1H 待定義欄位的規則；(2) 執行階段只強制 incident-responder 和執行日誌更新，無階段轉換時的中間欄位更新要求；(3) ticket-lifecycle.md 認領階段無「檢查並填寫待定義欄位」步驟；(4) CLI 的 claim 命令 checklist 有提到讀文件和理解 AC，但無「更新待定義欄位」項；(5) 根因：Ticket 建立時 PM 可能不知道完整的 when/how，留「待定義」是合理的，但 claim 後已有足夠資訊卻無規則要求更新。

## Solution

<!-- To be filled by executing agent -->

---

改進方案（三層）：

【第 1 層：規則更新 — ticket-lifecycle.md 認領階段】
認領階段新增強制步驟「5W1H 完善檢查」：
- claim 後，檢查 when/where/how 是否為「待定義」
- 若有待定義欄位，必須先用 set-when/set-where/set-how 更新再開始執行
- 提示強度：強制（不是建議）

認領階段 checklist 更新為：
1. [強制] 阻塞依賴檢查
2. [強制] 5W1H 待定義欄位補全（set-when/set-where/set-how）
3. [建議] 設計文件閱讀
4. [建議] 驗收條件理解
5. [建議] error-patterns 查詢

【第 2 層：規則更新 — 執行階段中間更新點】
執行階段新增「階段轉換日誌」要求：
- 分析型任務：分析完成後 append-log Problem Analysis，再開始修改
- 實作型任務：每完成一個 AC 後 append-log Solution（進度記錄）
- 提示強度：強制

執行日誌不再是「完成前才補填」，而是「轉階段時即時填寫」。

【第 3 層：CLI 改善（可選 Hook）】
ticket track claim 命令 output 中的 checklist 新增：
- [ ] 5W1H 待定義欄位已補全（when/where/how）
- 可選：claim 後自動檢查待定義欄位並輸出警告（Hook 或 CLI 內建）

【不建議的方案】
- Hook 強制阻擋 claim（過度自動化，PM 有時需要快速 claim 後再補）
- 自動填寫欄位（PM 應該手動思考並填寫，這是確認理解的過程）
