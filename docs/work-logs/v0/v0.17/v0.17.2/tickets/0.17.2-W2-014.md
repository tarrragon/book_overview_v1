# Execution Log

## Task Summary

分析 Agent 派發/回應自動記錄機制 — 失敗時可追溯

---

## Problem Analysis

### 問題根因

Agent 派發/回應的追溯完全依賴 Claude Code 的 conversation JSON（session output file），缺乏結構化的獨立記錄機制。

### 現有 Hook 基礎設施

**PostToolUse Agent Hook stdin 可用欄位**：
- `tool_input`: 包含 `prompt`（派發 prompt）、`description`（描述）、`isolation`（隔離模式）等
- `tool_response`: 包含 agent 回傳的純文字結果

**現有 PostToolUse Agent hooks（2 個）**：
- `dispatch-count-guard-hook.py`: 只讀取 tool_input 做關鍵字偵測，不讀 tool_response
- `agent-commit-verification-hook.py`: 只讀取 tool_input.description + git status，不讀 tool_response

**關鍵發現：目前沒有任何 hook 讀取或記錄 agent 的 tool_response。**

### 缺口清單

| 缺口 | 影響 | 嚴重度 |
|------|------|--------|
| Agent 輸出未持久化 | 失敗時無法追溯 agent 回報了什麼 | 高 |
| 派發 prompt 未持久化 | 無法對比「要求 vs 實際產出」 | 高 |
| 無成功/失敗分類 | PM 只能從上下文判斷，沒有結構化記錄 | 中 |
| 無時間戳記錄 | 無法統計 agent 執行時間、識別超時模式 | 低 |

### Claude Code Hook 系統限制

- PostToolUse 的 tool_response 對 Agent tool 是**純文字**，不是結構化 JSON
- Hook 無法攔截 agent 執行中途的行為，只能看到最終結果
- tool_response 可達 32K tokens，直接全文記錄需考慮儲存
- PreToolUse Agent hook 只能看到 tool_input，看不到結果

### 影響範圍

- 所有 Agent tool 派發（背景/前景皆受影響）
- PM 手動追溯成本高：需從 session output JSON 用 jq/python 解析

### 相關 Error Pattern

- feedback_agent_task_granularity.md — subagent 回合限制導致截斷
- feedback_subagent_code_in_prompt.md — prompt 內容不足導致失敗

## Solution

### 方案一：PostToolUse Agent 自動記錄 Hook（P1）

新增 `agent-dispatch-logger-hook.py`，掛載在 PostToolUse Agent。

**記錄格式**（JSONL，每行一筆）：

儲存位置：`.claude/logs/agent-dispatch.jsonl`

```json
{
  "timestamp": "2026-04-06T14:30:00",
  "ticket_id": "0.17.2-W2-014",
  "agent_description": "sage-test-architect: Phase 2 測試設計",
  "agent_type": "sage-test-architect",
  "isolation": "worktree",
  "prompt_preview": "（前 500 字元）",
  "response_preview": "（前 500 字元）",
  "success_indicators": {
    "has_commit": true,
    "has_error_keywords": false,
    "response_length": 4521
  }
}
```

**實作要點**：
1. 使用現有 hook_utils 的 extract_tool_input / extract_tool_response API
2. 從 prompt 正則提取 ticket_id
3. 成功/失敗啟發式判斷：搜尋錯誤關鍵字 + response 長度檢查
4. 行為：不阻擋（exit 0），僅記錄
5. 超過 1MB 時自動 rotate
6. 效能：寫入 < 10ms，不影響主流程

### 方案二：Agent 失敗標準除錯 SOP（P0）

**Step 1 — 查看記錄**（方案一實作後）
- 查閱 `.claude/logs/agent-dispatch.jsonl` 最近記錄

**Step 2 — 分類失敗原因**

| 失敗模式 | 識別特徵 | 處理方式 |
|---------|---------|---------|
| Tool calls 耗盡 | response 含 "exhausted" 或長度異常短 | 拆分任務 |
| Worktree 衝突 | response 含 "merge conflict" | 手動解決衝突 |
| 程式碼找不到 | response 含 "file not found" | 檢查 prompt 路徑 |
| 測試失敗 | response 含 "test failed" | 派發 incident-responder |
| 靜默失敗 | 有 commit 但內容不符 | 對比 prompt 和 diff |

**Step 3 — 修復後重新派發**
- 調整 prompt 後重新派發
- 拆分為更小任務
- 改為主線程在 feature 分支直接操作

**目前可用的除錯手段（方案一實作前）**：
1. 閱讀 conversation 中 Agent tool 的回傳文字
2. `git log --all --oneline -10` 檢查 commit
3. `git worktree list` 檢查 worktree 狀態
4. 檢查 agent-commit-verification-hook 日誌

### 方案三：Ticket 系統整合（P2，可延後）

在 ticket track complete 時自動將 agent 記錄摘要附加到 execution log。依賴方案一。
