## Problem Analysis

### 現狀分析

Hook stdin JSON 解析存在三個層面的問題：

**問題 1：兩套並存的解析入口（27 + 34 = 61 個 hook）**
- 27 個 hook 直接用 json.load(sys.stdin)（無統一保護）
- 34 個 hook 用 hook_utils.read_json_from_stdin() 或 lib.common_functions.read_json_from_stdin()

**問題 2：read_json_from_stdin 用 logger.error() 記錄 JSON 失敗**
- hook_io.py:169 — logger.error("JSON 解析錯誤: ...")
- common_functions.py:151 — 已修正為 stdout（但不一致）
- logger.error() 通過 StreamHandler 寫入 stderr → 觸發 hook error

**問題 3：StreamHandler 指向 stderr 且 level 過低**
- hook_logging.py 的 _create_stream_handler 建立 stderr StreamHandler
- 原 level=WARNING，已臨時修改為 CRITICAL（治標）
- 但這隱藏了所有 WARNING/ERROR 級別的 stderr 輸出，可能掩蓋真正的問題

## 根治方案

### 方案設計：三層防護

**第 1 層：統一 stdin 解析入口（根治）**

將 hook_io.py 的 read_json_from_stdin() 作為唯一標準入口。修改要點：
1. JSON 解析失敗從 logger.error() 改為 logger.info()（已預期的正常情況，不應觸發 stderr）
2. 所有 27 個直接用 json.load(sys.stdin) 的 hook 遷移到 read_json_from_stdin()
3. 淘汰 common_functions.py 的 read_json_from_stdin（合併到 hook_io.py）

**第 2 層：StreamHandler 設計修正（防禦）**

hook_logging.py 的 StreamHandler 設計調整：
- 保持 STREAM_HANDLER_LEVEL_NORMAL = CRITICAL（W4-004 的修復是正確方向）
- 原因：Claude Code 會將任何 stderr 輸出顯示為 "hook error"，因此 stderr 應僅限於真正的未處理異常（CRITICAL）
- 已處理的錯誤（ERROR/WARNING）只需寫入日誌檔案即可追蹤

**第 3 層：Hook 開發規範（預防）**

更新 basil-hook-architect 的知識庫和 Hook 開發模板：
1. 新 hook 模板必須使用 read_json_from_stdin()
2. 禁止直接 json.load(sys.stdin)
3. 禁止在已處理的錯誤路徑中使用 logger.error()/warning()（會被 StreamHandler 輸出到 stderr）
4. Hook 的錯誤處理決策樹：
   - 未預期異常 → logger.critical()（觸發 stderr + hook error）
   - 已預期但不正常 → logger.info() + 寫入日誌檔案
   - 正常跳過 → logger.debug()

### 立即可做的最小修復（P0）

hook_io.py:169 的 logger.error() 改為 logger.info()。這是唯一需要立即修復的程式碼，因為：
- StreamHandler level 已是 CRITICAL，logger.error 不會觸發 stderr
- 但 CRITICAL level 是治標（掩蓋），logger.info 是治本（正確語義）
- 未來若 StreamHandler level 調回 WARNING，logger.info 仍安全

### P1 遷移策略

27 個需遷移的 hook 可拆為多個子 Ticket，按目錄分組：
- hook_utils 系列（已使用 read_json_from_stdin 的不需改）
- lib 系列（需從 common_functions 遷移到 hook_io）
- 獨立 hook（test-timeout-*, pre-test-hook, changelog-update-hook, post-commit-fetch-hook 等）
