## Problem Analysis

### 問題根因

**問題場景**：Wave N 所有 Ticket 完成後，下一步是 Wave N+1 的 pending Tickets。PM 需要「從一個 session 的工作成果，引導下一個 session 接手 pending Ticket」。

**現有機制分析**：

handoff 設計為「**任務鏈內**的 context 交接工具」：
- completed Ticket → handoff 到兄弟/子（同任務鏈內）
- in_progress Ticket → context-refresh（同一 Ticket 跨 session）
- **pending Ticket 不可 handoff（設計正確）**

跨 Wave 轉換的現有流程（Checkpoint 2 情境 C）：
1. Wave 完成 → /parallel-evaluation 審查
2. 有其他 Wave pending → AskUserQuestion #3a
3. 用戶選擇 → /clear → 新 session → /ticket → 選擇 pending ticket → claim

**實際缺口**：不是「pending 無法 handoff」的問題，而是**跨 Wave context 傳遞**的問題。新 session 開始時對前一 Wave 的決策背景一無所知。

## 跨 Wave Handoff 流程設計

### 分析結論

handoff 限制 pending Ticket 的設計是**正確的**。handoff 是任務鏈內的 context 交接，不應擴展為通用任務路由。

**真正的問題**是跨 Wave context 傳遞。有三個方案：

### 方案比較

| 方案 | 說明 | 複雜度 | 建議 |
|------|------|--------|------|
| A: handoff --next-wave | 新增 handoff 到 pending ticket 的能力 | 高（破壞 handoff 語義） | **不建議** |
| B: Wave 摘要自動生成 | Wave 完成時自動產出 wave-summary.md | 中 | **建議** |
| C: 維持現狀 + worklog 強化 | 依靠 worklog 進度追蹤傳遞 context | 低 | **建議（短期）** |

### 建議方案：C（短期）+ B（中期）

**短期（C）**：不修改 handoff 機制。跨 Wave context 依靠：
1. Worklog 進度追蹤（已有，每次 commit 更新）
2. Ticket 的 5W1H 欄位（新 session 透過 ticket track query 取得）
3. /ticket 裸指令的待辦檢查（已有，SessionStart 提醒）

**中期（B）**：Wave 完成時自動產出摘要
1. 觸發時機：Checkpoint 2 情境 C（Wave 全部完成）
2. 產出內容：
   - 本 Wave 已完成的 Ticket 列表和關鍵決策
   - 發現的技術債和 pending Ticket
   - 下一 Wave 的建議優先級
3. 存放位置：docs/work-logs/v{VERSION}/wave-{N}-summary.md
4. 新 session 開始時 /ticket 自動提示最新的 wave summary

### 不建議的方案

**方案 A 不建議的理由**：
- 破壞 handoff 的核心語義（任務鏈內 context 交接）
- pending Ticket 沒有執行 context 可交接（尚未開始）
- 會讓 handoff 承擔過多職責（違反 SRP）

### 實作優先級

| 優先級 | 項目 | 版本 |
|--------|------|------|
| P2 | 維持現狀，依靠 worklog + /ticket | 當前（無需修改） |
| P2 | Wave 摘要自動生成 | v0.18.x+ |

### 流程規則更新建議

在 completion-checkpoint-rules.md 的情境 C 中，補充明確說明：
- 跨 Wave 不使用 handoff，使用 /clear + /ticket 重新選擇
- Worklog 進度追蹤是跨 session context 的主要載體
