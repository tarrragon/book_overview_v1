修改 分析→執行 Ticket 轉換流程新增完整性檢查

## Problem Analysis

### 缺口位置

decision-tree.md L370-411「ANA Ticket 結論轉化檢查」

現有檢查（L385）：children 或 spawned_tickets 非空?
現有規則（L410）：任一滿足即通過

問題：這是存在性檢查（有沒有後續 Ticket），不是完整性檢查（後續 Ticket 是否覆蓋分析結論的所有修改項）。

### 具體案例

W3-003 Solution 列出 4 項修改。PM 建立 W3-004 時只含 3 項。decision-tree 的 ANA 檢查只看 spawned_tickets 非空（W3-004 存在即通過），不驗證 W3-004 的 AC 是否覆蓋了 W3-003 Solution 的全部項目。

### 第二個缺口：背景代理人結果合併

decision-tree 中沒有任何步驟處理「背景代理人延遲完成」的結果合併。當 PM 和背景代理人同時分析同一問題時，PM 會先完成並建立後續 Ticket，代理人的額外發現被忽略。

### 相關 Error Pattern

- PC-041: 分析結論落地執行不完整
- PC-017: ANA 完成時結論未轉化（已有檢查但不夠嚴格）
## 流程缺口分析

### 1. ANA 完成後的現有流程路徑

decision-tree.md 第七層（L370-416）已有 ANA 結論轉化檢查：
- L384: Ticket type == ANA? -> 確認分析結論已轉化為 Ticket
- L385-388: 檢查 children 或 spawned_tickets 非空
- 通過條件：至少有一個後續 Ticket 追蹤分析結論

**問題：這個檢查只驗證「有沒有建立後續 Ticket」（數量 >= 1），不驗證「後續 Ticket 是否完整覆蓋分析結論的所有項目」。**

PC-041 場景：PM 建立了 W3-004（children 非空，通過檢查），但 W3-004 只覆蓋 3/4 項修改。現有機制無法攔截這種「部分遺漏」。

### 2. 缺口 A：第七層缺少「逐項對照」步驟

**位置**：decision-tree.md L384-411（ANA 結論轉化規則）

現有檢查是二元的（有/無後續 Ticket），缺少「後續 Ticket 的驗收條件是否 1:1 覆蓋分析結論所有修改項」的驗證步驟。

具體缺失：在 L388「繼續標準完成流程」之前，應有一步「逐項對照 Solution 中的修改項目，確認每項都有對應的後續 Ticket 驗收條件」。

### 3. 缺口 B：無背景代理人結果合併時機

**位置**：decision-tree.md 第七層和第八層之間，以及 ticket-lifecycle.md 完成階段

現有流程假設分析結論在 ANA Ticket 的 execution log 中已完整。但 PC-041 場景中，saffron 代理人的分析延遲完成，其結果未被合併到分析結論中。

**缺失的流程節點**：ANA Ticket complete 前，應檢查「是否有背景代理人的分析在同一問題上進行中？」如果有，應等待其完成並合併結論後再建立執行 Ticket。

### 4. 缺口 C：執行 Ticket 建立時缺少溯源驗證

**位置**：ticket-lifecycle.md 建立階段（以及 plan-to-ticket-flow.md）

當 PM 從 ANA 結論建立 IMP/DOC 執行 Ticket 時，現有流程沒有要求「執行 Ticket 的驗收條件必須 1:1 對應分析結論的修改項目」。PM 是手動撰寫驗收條件，容易遺漏。

**注意**：建立後審核（acceptance-auditor + system-analyst）只檢查 Ticket 自身品質（4V 合規、設計合理性），不會回溯比對來源 ANA Ticket 的結論完整性。

## 具體修改方案

### 修改 1：decision-tree.md 第七層 — 強化 ANA 結論轉化檢查

**檔案**：`.claude/pm-rules/decision-tree.md`
**位置**：L384-411（ANA 結論轉化規則區塊）

**現狀**（L384-388）：
```
Ticket type == ANA?
    +-- 是 -> 確認分析結論已轉化為 Ticket
    |         -> children 或 spawned_tickets 非空?
    |           +-- 是 -> 繼續標準完成流程
    |           +-- 否 -> 建立修復+防護 Ticket 後再繼續
```

**修改為**：
```
Ticket type == ANA?
    +-- 是 -> [強制] 結論完整性交叉驗證
    |         Step 1: 列出 Solution/execution log 中的所有修改項目（逐項編號）
    |         Step 2: 檢查是否有背景代理人的分析仍在進行中
    |           +-- 是 -> 等待完成，合併結論後再繼續
    |           +-- 否 -> 繼續
    |         Step 3: 逐項確認每個修改項目都有對應的後續 Ticket 驗收條件
    |           +-- 全部覆蓋 -> 繼續標準完成流程
    |           +-- 有遺漏 -> 補建/補充 Ticket 後再繼續
    |         -> children 或 spawned_tickets 非空?（原有檢查保留為最後防線）
```

同時更新 L403-411 的「ANA 結論轉化規則」表格，新增：

| 檢查項 | 說明 |
|--------|------|
| 修改項目逐項列出 | 從 Solution 提取，編號列出 |
| 背景代理人結論合併 | 等待進行中的背景分析完成 |
| 1:1 覆蓋驗證 | 每個修改項目對應一條後續 Ticket 驗收條件 |

### 修改 2：ticket-lifecycle.md — 新增「ANA 衍生 Ticket 建立規範」

**檔案**：`.claude/pm-rules/ticket-lifecycle.md`
**位置**：在「Ticket 建立強制規則」區塊（L42-48）後新增子章節

**新增內容**：

```markdown
### ANA 衍生執行 Ticket 建立規範（強制）

> 來源：PC-041 — ANA 結論 4 項修改，執行 Ticket 只覆蓋 3 項

從 ANA Ticket 建立執行 Ticket 時，驗收條件必須 1:1 對應分析結論的修改項目：

| 步驟 | 動作 |
|------|------|
| 1 | 開啟 ANA Ticket 的 execution log，列出 Solution 中所有修改項目 |
| 2 | 檢查背景代理人的分析輸出，合併額外發現 |
| 3 | 為每個修改項目撰寫一條驗收條件（1:1 對應） |
| 4 | 驗收條件中註明來源：「(來自 {ANA Ticket ID} 第 N 項)」 |

禁止行為：
- 禁止憑記憶撰寫驗收條件，必須開啟 ANA Ticket 逐項對照
- 禁止合併多個修改項目為一條驗收條件
```

### 修改 3：建立後審核增加溯源檢查

**檔案**：`.claude/pm-rules/ticket-lifecycle.md`
**位置**：建立後強制審核流程（L52-95）的審核內容表格

**現狀**（L62-66）：
| 審核者 | 面向 | 檢查項目 |
|--------|------|---------|
| acceptance-auditor | 品質 | 驗收條件 4V 合規、where.files 完整性、數字正確性 |
| system-analyst | 設計 | 設計合理性、重複實作檢查、範圍適當性 |

**修改為**：
| 審核者 | 面向 | 檢查項目 |
|--------|------|---------|
| acceptance-auditor | 品質 | 驗收條件 4V 合規、where.files 完整性、數字正確性、**ANA 衍生時：驗收條件 1:1 覆蓋分析結論** |
| system-analyst | 設計 | 設計合理性、重複實作檢查、範圍適當性 |

### 修改優先級

| 修改 | 優先級 | 理由 |
|------|--------|------|
| 修改 1（decision-tree 第七層） | 高 | 直接攔截 PC-041 場景，最小成本最大收益 |

## 修改方案

### 修改 1: decision-tree.md L385 強化 ANA 完整性檢查

現有（L385-387）：
children 或 spawned_tickets 非空? → 是 → 繼續

改為：
children 或 spawned_tickets 非空?

       → 是 → 繼續
       → 否 → 補充遺漏項到現有 Ticket 或建立新 Ticket

### 修改 2: decision-tree.md L410 結論轉化規則表新增一行

新增：
| AC 覆蓋率 | 後續 Ticket 的 AC 合集必須覆蓋 ANA Solution 的每一項修改（非存在性，而是完整性） |

### 修改 3: decision-tree.md 第八層新增背景代理人合併步驟

在 Checkpoint 1（commit 後）和 Checkpoint 2（收尾路由）之間，新增：

有未處理的背景代理人通知?
  → 是 → Read agent output，對比已建立的後續 Ticket
       → 有額外發現 → 補充到現有 Ticket AC 或建立新 Ticket
       → 無額外發現 → 繼續
  → 否 → 繼續

- ticket CLI — check-acceptance 只驗自身 AC，不負責跨 Ticket 對照
