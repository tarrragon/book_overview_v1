## Task Summary

分析 代理人派工粒度與 Ticket 中心化工作流改善

## Problem Analysis

### 問題根因

W2-005 執行過程中四次代理人派發失敗，根因歸納為三類：

**1. 任務粒度超出代理人 context 容量**

單一代理人任務包含「研究 codebase → 理解設計規格 → 撰寫 276 行測試 → 執行驗證 → commit」完整流程。代理人的回合數（約 15-20 輪）不足以完成全部步驟。

失敗模式分析：
- sage 第 1 次：探索目錄結構耗盡 context（研究階段過長）
- sage 第 2 次：prompt 含完整程式碼（~200 行），讀完規格後 context 不足以執行 Write
- general-purpose 第 1 次：worktree 中未產出任何變更（IMP-047 重現）
- general-purpose 第 2 次：做了充分研究後說「Let me write」然後結束

**2. 進度無法跨回合保存**

代理人在探索/研究階段取得的所有理解（目錄結構、測試風格、source code 邏輯），在 context 耗盡後完全遺失。沒有機制讓代理人將中間產物（如研究筆記、程式碼草稿）持久化到 Ticket。

**3. prompt 策略的兩難困境**

- 含完整程式碼的 prompt：佔用 context 預算，留給代理人執行 Write/Test 的空間不足
- 不含程式碼的 prompt：代理人需自行探索，探索本身就耗盡 context

### 影響範圍

- 所有需要代理人執行「研究 + 實作」複合任務的場景
- 特別影響 Phase 2（測試設計）和 Phase 3b（實作）的代理人派發
- 影響檔案：.claude/pm-rules/task-splitting.md, .claude/pm-rules/parallel-dispatch.md, .claude/error-patterns/implementation/IMP-047-*.md

### 相關 Error Pattern

- IMP-047（Worktree Subagent 只讀不寫）：已記錄但解決方案不完整
- IMP-047 的「提供完整程式碼」方案在 W2-005 中失敗（prompt 太長仍耗盡 context）

### 與現有規範的衝突

1. **atomic-ticket-methodology.md** 明確禁止以「程式碼行數」作為拆分依據（「行數是實作細節，單一職責可能需要 200 行」）。但現實中 200+ 行的寫入任務確實超出代理人能力。這不是 SRP 的問題，而是執行能力的物理限制。

2. **IMP-047 的方案與 context 限制矛盾**：IMP-047 建議「prompt 必須包含完整程式碼或 80% 骨架」，但當程式碼 > 150 行時，prompt 本身就佔用大量 context，導致代理人沒有足夠空間執行操作。

3. **task-splitting.md** 的拆分觸發條件（變數數 > 5、檔案數 > 5 等）都是認知負擔維度，缺少「代理人執行能力」維度的考量。
## 根因分析

### 直接原因
代理人（sage/general-purpose）在 worktree 中執行「Phase 2 測試寫入」任務時，4 次派發全部因 context/回合耗盡而未完成寫入操作。

### 根本原因
1. **任務粒度不匹配代理人能力**：單一任務包含「讀取設計文件 + 理解現有程式碼風格 + 寫入 276 行測試 + 執行驗證 + commit」5 個步驟，每步都需要大量 context
2. **探索階段無法省略**：代理人必須讀取設計文件、現有測試風格、source code 才能寫出正確的測試，但這些探索本身就佔用 50%+ context
3. **IMP-047 的矛盾**：IMP-047 要求 prompt 含完整程式碼，但完整程式碼（~200 行）佔用大量 prompt context，反而加速 context 耗盡
4. **進度不可保存**：代理人的探索發現（風格理解、檔案結構確認）沒有中間保存機制，耗盡後一切歸零

### 影響範圍
- 所有需要「探索 + 大量寫入」的代理人任務都有此風險
- 特別是 Phase 2（測試寫入）和 Phase 3b（實作寫入）任務

## 改善方案

### 方案一：新增「代理人執行能力」拆分維度

**問題**：現有 task-splitting.md 只考量認知負擔，未考量代理人的物理執行限制。

**建議**：在 task-splitting.md 新增「策略 8：按代理人執行容量」，加入以下拆分觸發條件：

| 條件 | 閾值 | 理由 |
|------|------|------|
| 需寫入的程式碼行數 | > 150 行 | prompt 含程式碼時佔用過多 context |
| 需讀取的參考檔案數 | > 3 個 | 每個 Read 消耗 1-2 輪 |
| 任務步驟數（研究+寫入+測試） | > 3 步 | 每步需 3-5 輪，15 輪上限 |

**重要澄清**：這不違反 atomic-ticket-methodology 的「禁止以行數拆分」原則。SRP 仍是 Ticket 設計的標準，但**派發時**需要根據執行能力將一個 Atomic Ticket 拆為多個**子派發**（同一 Ticket 的多次代理人呼叫），而非拆為多個 Ticket。

**具體做法**：

```
Ticket: 撰寫 SearchEngine tag 搜尋測試（SRP 正確，是一個 Ticket）
  ├── 派發 1（研究型）：sage 讀取規格 + source code，產出測試骨架寫入 Ticket
  ├── 派發 2（實作型）：thyme 根據 Ticket 中的骨架寫入檔案
  └── 派發 3（驗證型）：thyme 執行測試 + 修復
```

### 方案二：Ticket 中心化進度記錄

**問題**：代理人 context 耗盡後，所有探索成果遺失。

**建議**：新增規則「代理人進度持久化協議」：

1. **探索型代理人（sage/saffron）** 的產出必須寫入 Ticket 執行日誌或獨立文件，不得只存在於 context 中
2. **每 5 輪操作後**，代理人應將當前進度摘要寫入 Ticket（透過 append-log 或 Write）
3. **prompt 範本加入指令**：「每完成一個子步驟，立即將結果寫入 [檔案路徑]，確保 context 耗盡時工作不會遺失」

**具體的 Ticket 中間產物格式**：

```yaml
# 在 Ticket 執行日誌中記錄
## 中間產物

### 研究摘要（派發 1 產出）
- 測試風格：describe/it 結構，使用 jest.fn() mock
- 相關 source：search-engine.js L45-L89 _matchesSearchCriteria()
- 測試骨架：見 附件 A

### 程式碼草稿（派發 1 產出）
（完整程式碼或骨架，供派發 2 使用）
```

### 方案三：修正 IMP-047 的防護措施

**問題**：IMP-047 的「提供完整程式碼」方案在大型程式碼（> 150 行）場景下失效。

**建議更新 IMP-047**：

將防護措施從「prompt 包含完整程式碼」改為分級策略：

| 程式碼規模 | 策略 | 說明 |
|-----------|------|------|
| < 80 行 | prompt 內嵌完整程式碼 | 原有方案，仍然有效 |
| 80-150 行 | prompt 內嵌 + 「不要研究，直接寫」指令 | 邊界情況，加強指令 |
| > 150 行 | 拆為多次派發（研究 → 寫入 → 驗證） | 新增方案 |
| > 300 行 | 拆為多次派發 + 按功能區塊分段寫入 | 每次寫入 < 150 行 |

### 方案四：派發策略改善

**探索型 vs 實作型代理人職責邊界**：

| 角色 | 職責 | 產出 | context 使用模式 |
|------|------|------|----------------|
| 探索型（sage/saffron） | 讀取、分析、設計 | 文件/規格/骨架（寫入 Ticket） | 高讀取、低寫入 |
| 實作型（thyme/parsley） | 寫入、測試、修復 | 程式碼檔案 | 低讀取、高寫入 |

**禁止混合**：不得在同一次派發中要求代理人同時執行大量研究和大量寫入。

**Prompt 內容規範**：

| 項目 | 應包含 | 不應包含 |
|------|--------|---------|
| 實作型派發 | 完整程式碼（< 150 行）或精確檔案路徑 + 程式碼片段 | 「請先閱讀 X 理解 Y」的研究指令 |
| 探索型派發 | 明確的分析問題 + 產出格式要求 | 程式碼實作要求 |
| 所有派發 | 產出位置（寫入哪個檔案/Ticket 區段） | 模糊的「請完成此任務」 |

**Worktree vs Feature 分支選擇標準**：

| 條件 | 使用 worktree | 使用 feature 分支（PM 直接） |
|------|-------------|---------------------------|
| 任務 < 150 行寫入 | 是 | 否 |
| 任務 > 150 行或已失敗 2+ 次 | 否 | 是（PM fallback） |
| 探索/分析任務（無寫入） | 否（不需 worktree） | 不適用 |

## 總結

核心改善：**將「SRP 拆分」和「執行能力拆分」分離為兩個獨立維度**。

- SRP 決定 Ticket 數量（一個職責 = 一個 Ticket）
- 執行能力決定派發次數（一個 Ticket 可能需要多次派發）

這解決了 atomic-ticket-methodology 與代理人物理限制之間的矛盾：不需要為了適應代理人能力而違反 SRP 拆分原則，而是在同一個 Ticket 內進行多階段派發。
## 改善方案

### 方案 1：任務拆分閾值規則（新增規則）

**寫入量閾值**：代理人單次任務寫入量不得超過 150 行。超過時必須拆分為多個子任務。

| 寫入量 | 策略 |
|--------|------|
| < 50 行 | 單一代理人完成（含探索） |
| 50-150 行 | 單一代理人，但 prompt 必須包含精確的 Edit 指令（old_string/new_string） |
| > 150 行 | 必須拆分：(1) 設計子任務（產出精確的程式碼片段到 Ticket）(2) 寫入子任務（從 Ticket 讀取程式碼並注入） |

### 方案 2：IMP-047 修正 — 從「prompt 含完整程式碼」改為「Ticket 含完整程式碼」

- **現行**：prompt 中包含完整程式碼 → 佔用代理人 context
- **改為**：完整程式碼寫入 Ticket 執行日誌或設計文件 → 代理人從 Ticket 讀取 → 不佔用 prompt context
- **好處**：程式碼持久化在 Ticket 中，即使代理人失敗也不遺失

### 方案 3：二階段派發模式（設計 → 注入）

Stage 1 — 設計代理人（sage）：
- 產出：測試設計文件 + 完整可直接貼入的程式碼片段
- 程式碼片段寫入 Ticket 的 Solution 區段或獨立設計文件
- 不需要寫入任何 src/tests 檔案

Stage 2 — 注入代理人（thyme/general-purpose）：
- 輸入：Ticket 路徑（含程式碼片段位置）+ 目標檔案路徑 + 注入位置
- 操作：Read Ticket → Read 目標檔案尾部 → Edit 注入 → 執行測試 → commit
- 探索量極小（只需確認注入位置），context 充裕

### 方案 4：代理人進度檢查點

代理人完成每個步驟後，必須將中間結果寫入 Ticket：
- 完成探索 → append-log 記錄發現（檔案結構、風格要點）
- 完成設計 → append-log 記錄程式碼片段
- 完成寫入 → append-log 記錄變更檔案
- 完成驗證 → append-log 記錄測試結果
