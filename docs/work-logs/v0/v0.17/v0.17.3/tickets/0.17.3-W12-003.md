## Problem Analysis

### 用戶設計方向 — 「不做清單」+ 提案系統的角色

**核心概念：花在某件事的那一小時，就沒辦法做其他的事了**

引用：柯林斯的「不要繼續做下去的清單」（stop-doing list）— 不是效率問題，是取捨問題。假裝一心多用是天真的。

**提案系統的本質**：
- 不是「待辦清單」（那會讓人焦慮要全部做完）
- 而是「安全停放區」— 想法被記錄，不會遺失，但明確標示「現在不做」
- 讓 PM 安心地把想法放下，專注在核心優先事項

**對應本專案已有的系統**：

| 系統 | 角色 | 對應概念 |
|------|------|---------|
| 提案系統（docs/proposals/） | 安全停放區：記錄想法，不代表要做 | 柯林斯的 stop-doing list |
| Ticket 系統（pending） | 承諾清單：已決定要做，排入排程 | 核心優先事項的執行計畫 |
| 版本里程碑（CLAUDE.md） | 核心優先事項：這個版本的目標 | 英特普拉思特的「病人是客戶」 |

**W12-003 Skill 應該串接這些系統**：

```
遇到新問題/想法
    |
    v
[錨點] 影響核心客戶/核心目標嗎？
    +── 是 → 建 Ticket → 進入 WRAP 決策
    +── 否但有價值 → 建提案（docs/proposals/）→ 回到核心任務
    +── 否且低價值 → 記錄到 todolist → 回到核心任務
    +── 純粹救火 → 建 Ticket + 標記 P2 → 回到核心任務
```

**關鍵心態轉變**：
- 從「看到問題就要解決」→「看到問題先分類，只解決影響核心目標的」
- 不是忽略問題，是**把問題放在正確的系統中，用正確的時機處理**
- 提案系統讓 PM 不必擔心「失去想法」，可以安心說「現在不做」

### 用戶設計方向 — 救火排擠了重要的事

**核心概念：需要救火的事情太多了，顧不上應該優先做的事**

這正是本 session 的完美寫照：
- 核心任務：W2 design-system 重構 → W3 文件+測試
- 救火事件：hook error → 三輪修改 → 回退 → 社群搜尋 → dispatch tracking → Hook 改善
- 結果：W3 完全沒動到

**Skill 的終極設計目標**：
不是「教 PM 如何做好決策」，而是**在 PM 被救火吸走的那個瞬間攔住他**。

綁定機制的觸發信號（PM 正在被救火吸走的徵兆）：
1. 連續 2+ 個 Ticket 不在當前版本的核心 Wave 中
2. 新建的 Ticket 優先級低於正在處理的
3. 花超過 30 分鐘在一個「發現」的問題上（非原定計畫）
4. 連續失敗 2 次修改同一個問題

這些信號出現時，Skill/Hook 應該提醒：
「你在最該優先處理的問題上毫無作為。需要救火的事情太多了，顧不上應該優先做的事。回到核心任務。」
### 用戶設計方向 — 確立優先事項 vs 遵守優先事項是兩回事

**核心概念：知道什麼重要，和實際做什麼重要的事，是完全不同的兩件事**

引用：龐德斯研究 — 企業主管列出 5-8 個最重要問題，但上週實際做的事，跟那些重要問題完全無關。

**三種失敗模式**：

| 模式 | 行為 | 結果 |
|------|------|------|
| 逃避型 | 不確立優先事項，遇到衝突就迴避 | 決策混亂，隨情境擺盪 |
| 命令型 | 自負地直接下決定，不說明理由 | 團隊不理解，無法自主判斷 |
| 隨風型 | 看當時該討好誰就偏向誰 | 優先事項名存實亡 |

**對 PM 的直接對照**：
- 本 session PM 知道 v0.17.3 的核心目標是 design-system 重構
- 但上週（本 session）實際做的事：花大量時間在 hook error（W10-002）
- 這就是龐德斯研究的活生生案例

**Skill 需要解決兩個層次的問題**：

1. **確立**（Establish）：幫 PM 在 Ticket/版本開始時明確核心優先事項
2. **綁定**（Bind）：在執行過程中，持續檢查 PM 的行為是否對齊核心優先事項

**綁定機制的設計**：
- 不能只靠 PM 自律（龐德斯研究證明這不可靠）
- 需要外部提醒機制：Hook 或 Skill 在關鍵時刻問「你現在做的事服務於核心優先事項嗎？」
- 關鍵時刻：(1) 花超過 30 分鐘在同一個問題上 (2) 準備修改核心系統 (3) 連續失敗 2 次以上

### 用戶設計方向 — 每個 Ticket/提案都需要錨點確認

**核心概念：不只是版本級別，每個 Ticket 開始前都要確認核心目的**

用戶指出的關鍵問題對：

| 問題對 | 左 | 右 |
|--------|---|---|
| 交付策略 | 快速交付 | 建立穩定可擴充架構 |
| 問題性質 | 找 Bug | 處理系統設計疏失 |
| 優先維度 | 使用者 UX 體驗 | 執行效能 |

**這些不是二選一，而是「在這個 Ticket 的 context 下，哪個權重更高？」**

**應用時機**：
1. **Ticket 認領時**：「這個 Ticket 的核心目的是什麼？」
2. **開發途中**：想法可能改變，隨時可以再確認
3. **不確定時**：必須問用戶，不要自己假設

**對 PM 流程的具體改變**：

在 Ticket claim 後、派發代理人前，強制加入錨點確認步驟：

```
ticket track claim <id>
    |
    v
[錨點確認] AskUserQuestion:
  「這個 Ticket 的核心目的是？」
  - 快速交付（最小修改，盡快完成）
  - 穩定架構（花時間做對，建立基礎）
  - Bug 修復（找到根因，最小修復）
  - 設計改善（系統性重構）
  - UX 優先（使用者體驗為主）
  - 效能優先（執行效率為主）
    |
    v
[記錄到 Ticket] ticket track set-why <id> 'core purpose: ...'
    |
    v
開始分析和派發
```

**開發途中的重新確認**：
- 代理人回合耗盡時 → 重新確認：「我們要簡化 scope 快速交付，還是維持完整度？」
- 發現額外問題時 → 重新確認：「要一起處理還是建 Ticket 延後？」
- 方案需要大幅修改時 → 重新確認：「核心目的有沒有改變？」

### 用戶設計方向 — 一句話終結所有爭論：「誰是我們的客戶？」

**核心概念：確立核心優先事項後，所有困難的決定都有了錨點**

引用：「你認為志願外科醫生是我們的客戶。但我認為，病人才是我們的客戶。」
→ 確立後，所有決策都靠一句話解決：「怎麼做對我們的病患最有利？」
→ 海耶思：「這個立場的確定，改變了所有的事。」

**對本專案的核心問題**：
- 「誰是我們的客戶？」→ **使用 Readmoo 的讀者**
- 不是 PM、不是代理人、不是 Hook 系統、不是開發流程
- 當 hook error 修復（服務開發流程）vs 產品功能開發（服務讀者）衝突時 → 讀者優先

**對 Skill 設計的終極啟示**：

WRAP 框架的最頂層不是「擴增選項」，而是**先確認錨點**：

```
決策發生
    |
    v
[錨點] 「這個決策最終服務於誰？」
    |
    v
[閘門] 「這個問題影響核心客戶嗎？」
    +── 否 → 建 Ticket 延後
    +── 是 → 進入 WRAP 流程
        W: 擴增選項
        R: 實境檢驗
        A: 拉開距離
```

**完整 Skill 結構雛形**：

### 用戶設計方向 — 核心優先事項的排序決定一切

**核心概念：當兩個核心優先事項衝突時，排序就是決策**

引用：英特普拉思特的兩個使命 —
1. 免費為開發中國家的人民做重建手術（短期：幫病患動手術）
2. 協助當地醫護建立醫療自主能力（長期：讓自己不必再做這件事）

當「讓志工醫生開心」（帶眷屬、帶住院醫師）與「訓練當地醫生」（把機會讓給當地人）衝突時，只有排序才能終結爭論。新團隊選擇「訓練當地醫生」為第一優先 → 眷屬和住院醫師的問題就自動有了答案。

**對 PM/專案的啟示**：

本專案也有多重優先事項：
1. 產品功能開發（v0.17.3 Tag-based Model）
2. 開發流程改善（Hook 系統、決策樹、PM 規則）
3. 程式碼品質（design-system 統一、DRY）
4. 工具維護（hook error 修復、CLI 相容性）

本 session 的優先事項衝突：
- 產品開發（W2/W3 Ticket）vs 工具維護（W10-002 hook error）
- PM 被 #4 吸引，花了大量時間，影響了 #1

**Skill 設計啟示**：
框架需要在最前面就強制釐清核心優先事項排序：

| 問題 | 目的 |
|------|------|
| 「當前版本的核心目標是什麼？」 | 建立錨點 |
| 「這個決策服務於哪個優先事項？」 | 對齊 |
| 「如果優先事項衝突，哪個排第一？」 | 排序 |
| 「投入這個問題的時間，會擠壓哪個更重要的目標？」 | 機會成本 |

排序確定後，低優先事項的決策自動有了答案 — 記錄 Ticket 延後，不投入時間。

### 用戶設計方向 — 核心優先事項如何終結無盡爭論

**核心概念：沒有核心優先事項的決策，會陷入無止盡的討論**

引用：微笑列車董事會為了「出任務能不能帶眷屬」連續討論了 12 小時，沒有結論。看似簡單的問題，背後是核心優先事項的衝突：醫生的工作滿意度 vs 手術品質 vs 當地醫護的學習機會。

**沒有核心優先事項時的症狀**：
- 爭論沒完沒了，每個立場都有道理
- 討論持續數小時但無法收斂
- 問題「看似簡單」但決定不了

**有核心優先事項時的效果**：
- 用優先事項當過濾器：「這個選項是否服務於我們最重要的目標？」
- 自動排除不符合的選項
- 決策有錨點，不會漂移

**對 PM 的意義**：
- W10-002 的 hook error 討論就是一個「看似簡單但陷入連續排查」的案例
- 如果一開始就問「修復 hook error 是否服務於 v0.17.3 的核心目標（design-system 重構）？」→ 否 → 記錄 Ticket 延後 → 省下數小時
- 本 session 在 hook error 上花的時間（W10-002 三輪修改 + 回退 + 社群搜尋）遠超過它的重要性

**Skill 設計啟示**：
在進入實境檢驗之前，插入一道閘門：

```
準備投入分析/修改時
    |
    v
「這個問題影響核心優先事項嗎？」
    +── 是 → 進入完整 WRAP 流程
    +── 否 → 建 Ticket 記錄 → 回到核心任務
    +── 不確定 → 先花 5 分鐘試水溫（Ooch）→ 再判斷
```

**這道閘門防止 PM 被低優先問題吸進去**，就像董事會被「帶眷屬」吸進 12 小時討論一樣。

### 用戶設計方向 — 核心優先事項（Core Priorities）：決策必須對齊長期目標

**核心概念：選項拉扯的焦慮，是核心優先事項衝突的徵兆**

引用：難以抉擇的原因不是選項不好，而是長期目標之間在互相競爭。決策必須問：哪個選項最符合核心優先事項？

**A — Attain Distance（拉開距離）完整內容**：
- 機會成本顯性化（DVD 實驗）
- 核心優先事項對齊：決策是否服務於長期目標？

**對 PM 的意義**：
- W10-002 中 PM 花了大量時間排查 hook error → 但這服務於什麼核心優先事項？
- 本版本（v0.17.3）的核心目標是「Tag-based Book Model 重構」和「design-system 建立」
- hook error 是 CLI 顯示問題（低優先），不影響功能 → 不應該花 3 輪修改
- 如果在第一時間問「這個問題影響核心優先事項嗎？」→ 答案是「不影響」→ 記錄後繼續做 W3

**Skill 設計啟示**：
在做選項評估時，加入一個「核心優先事項對齊檢查」：
- 「這個決策服務於哪個長期目標？」
- 「如果不處理這個問題，核心優先事項會受影響嗎？」
- 如果不影響 → 記錄到 Ticket 延後處理，不投入時間
- 如果影響 → 值得投入，進入完整的實境檢驗

**這也解釋了為什麼 PM 會被 hook error 吸引**：
- 短期情緒：「終端機顯示 error 讓我不舒服」→ 立刻想解決
- 長期目標：「v0.17.3 的 design-system 重構」→ 被 hook error 排擠掉
- 核心優先事項檢查能幫助 PM 抵抗短期情緒的干擾

### 用戶設計方向 — 試水溫（Ooch）：能測試就不要預測

**核心概念：如果可以實地測試，何必預測？能知道時，何必猜想？**

引用：在全力投入前，先讓選項去兜個圈子。

**對抗確認偏誤的三道防線**：

| 防線 | 方法 | 對應框架 |
|------|------|---------|
| 1. 駁斥性問題 | 竭盡全力問「什麼事證能推翻這個假設？」 | 實境檢驗 |
| 2. 正確的訊息種類 | 向外 Zoom Out（基本率）+ 向內 Zoom In（實際情況） | Zoom Out + Zoom In |
| 3. 試水溫 | 小規模實際測試，不全力投入 | **新增：Ooch** |

**試水溫在 PM 決策中的應用**：

| 場景 | 全力投入（錯誤） | 試水溫（正確） |
|------|----------------|---------------|
| W10-002 hook error | 直接改 run_hook_safely 核心 | 先手動測試一個 hook 看 exit code |
| W11-002 語意重命名 | 一次派發改全部 5 個檔案 | 先改 colors.js 一個檔案，確認消費端不壞 |
| 新功能設計 | 完整實作後才測試 | 先寫最小 POC 驗證核心假設 |

**對 W10-002 的具體反思**：
- PM 假設「exit code 1 導致 hook error」→ 直接改了 run_hook_safely
- 試水溫做法：先在一個 hook 上手動測試 exit code 是否為 1 → 發現都是 0 → 假設立刻被推翻 → 不需要修改任何程式碼
- 試水溫的成本：5 分鐘
- 全力投入的成本：2 次修改 + 1 次回退 + 問題惡化

**Skill 設計啟示**：
在實境檢驗階段，每個選項都要問：
- 「有沒有辦法用最小成本驗證這個假設？」
- 「能不能在 5 分鐘內得到初步答案？」
- 如果能 → 先試水溫再決策
- 如果不能 → 才進入完整的分析流程

**W12-003 設計方向完整框架（整合《零偏見決斷法》WRAP 模型）**：

**W — Widen Your Options（擴增選項）**
- 不要只有一個選項（青少年派對問題）
- 向外求解：「誰解決過？」（沃爾頓）
- 爬梯子：從本專案 → 社群 → 跨領域（校長餐廳）
- 選項消失測試：「以上全部不能選？」（精靈哥哥）
- 假選項偵測：實質差異？（季辛吉陷阱）

### 用戶設計方向 — 專家的預測不可靠，但基本率可靠（特洛克研究）

**核心概念：82,361 項預測驗證 — 專家的預測不如簡單的基本率外推**

引用：特洛克（Philip Tetlock）的研究 — 蒐集專家 20 年的預測資料，發現即便是最準的專家，也不如「粗略外推運算法」（假設過去的趨勢會繼續）。

**再次強化的原則**：
- 相信基本率，不相信預測
- 「過去三年成長率 2.8%，那未來就預測 2.8%」— 這種「看似無聊」的方法勝過專家直覺

**對 AI/PM 的意義**：
- AI 被問「這個修改會成功嗎？」→ 這是預測 → AI 傾向樂觀回答 → 不可靠
- AI 被問「類似的修改在過去的案例中，成功率是多少？」→ 這是基本率 → 可靠
- PM 自己的直覺（「我覺得是 exit code 問題」）也是預測 → 不如查基本率

**Skill 設計的核心原則**：
框架的每一步都應該引導 PM/AI 遠離預測、靠近基本率：
- 不問「會不會成功？」→ 問「過去成功率是多少？」
- 不問「這是不是根因？」→ 問「這類症狀的常見根因分佈是什麼？」
- 不問「我的方案好嗎？」→ 問「類似方案在其他場景的結果是什麼？」

### 用戶設計方向 — 大範圍觀照 + 近距離檢視（Zoom Out + Zoom In）

**核心概念：基本率建立基準，近距離檢視觸動直覺。兩者缺一不可。**

引用：羅斯福的訊息策略 —
- 大範圍觀照（Zoom Out）：民意調查、信件統計分析、基本率簡報
- 近距離檢視（Zoom In）：親自讀信、依蓮娜的現場訪視、跳過主管直接找基層
- 多源交叉驗證：官僚體系外的訊息網絡，避免單一來源被污染

**羅斯福的訊息品質策略對 PM 的啟示**：

| 羅斯福做法 | PM 對應做法 |
|-----------|-----------|
| 不完全依賴國務卿 | 不完全依賴代理人的自我報告 |
| 聘請獨立聯絡官 | 用 Hook 系統獨立追蹤（dispatch-active.json） |
| 依蓮娜未通知訪視 | git diff --stat 客觀驗證（不問代理人做了什麼，看實際改了什麼） |
| 跳過主管找基層 | 直接讀 source code，不只看代理人摘要 |
| 信件統計（基本率）+ 親讀信件（直覺） | 基本率（社群 Issue 數量）+ 近距離（讀具體 Issue 的描述和 workaround） |

**Zoom Out + Zoom In 在決策框架中的位置**：

**Zoom Out（大範圍觀照）**：
- 搜尋社群基本率：「多少人遇到同樣問題？」「常見解法分佈？」
- AI 內建知識的基本率：「這類問題通常怎麼解決？」
- 統計性判斷：「做了這決定，合理預期會發生什麼？」

**Zoom In（近距離檢視）**：
- 讀具體案例：不只看 Issue 標題，讀內文和 workaround
- 讀 source code：不只看代理人報告，實際 git diff
- 小規模實驗：先用最簡單的方式驗證假設

**對 W10-002 的反思**：
- PM 缺少 Zoom Out：沒搜尋社群就開始修改
- PM 也缺少 Zoom In：即使搜尋了，如果只看 Issue 標題「hook error」不讀內文，可能仍然錯判
- 正確做法：先 Zoom Out（搜尋 5 個 Issues）→ 再 Zoom In（讀其中 2-3 個的完整描述和回覆）→ 得到高品質判斷

### 用戶設計方向 — 謙虛面對基本率（尊重前人經驗）

**核心概念：忽略別人的經驗不是勇敢，是自大**

引用：「我絕對不會讓這樣的分析數字阻擋我相信該做的事。」這種說法才是真正的自大。意思是，我跟其他人是不一樣的。我真的不同，我比他們更厲害。

**謙虛的問法**：「如果做了這樣的決定，我可以合理預期會發生什麼？」

**對 PM 的深層意義**：
- W10-002 中 PM 的行為本質是：「其他人在 GitHub 上報告的 hook error 跟我的不一樣，我的問題一定是 Hook 程式碼造成的」
- 事實是：5+ 個人報告了完全相同的症狀
- PM 選擇「自己修」而非「查看前人經驗」— 不是技術判斷，是認知偏誤

**框架的完整流程（整合所有概念）**：

1. **擴增選項** — 不要只有一個選項
   - 向外求解：「誰解決過同樣的問題？」
   - 爬梯子：從本專案 → 社群 → 跨領域
   - 選項消失測試：「如果以上都不能選？」
   - 假選項偵測：選項是否有實質差異？

2. **實境檢驗** — 不要未驗證就行動
   - 基本率問題：「這類問題通常怎麼解決？成功率？」
   - 事證問題：「需要什麼事證才能證明可行？」
   - 謙虛問法：「如果做了這決定，合理預期會發生什麼？」

3. **機會成本顯性化** — 不要忽略不選的代價
   - 每個選項旁邊列出：放棄什麼？風險？不選其他的代價？

4. **決策後** — 得到答案後才轉向執行
   - 先有結論，再想辦法拉高成功機會
   - 不是邊做邊想

### 用戶設計方向 — 問對問題：基本率 vs 預測

**核心概念：向專家問基本率（過去和現在），不問預測（未來）**

引用：智財權律師案例 —
- 好問題：「類似案子的重要變數是什麼？」「多少比率在開庭前和解？」「原告勝訴比例？」→ 得到可信的基本率資料
- 壞問題：「你覺得我可以打贏嗎？」→ 得到過度樂觀的預測

**對 AI 的意義**：
AI 模型本質上就是基本率的巨大資料庫 — 訓練資料中包含了大量案例的統計特徵。AI 擅長回答「這類問題通常怎麼解決？」「常見的失敗模式有哪些？」而非「這次修改會不會成功？」

**PM 應該問 AI 的正確問題**：

| 壞問題（預測） | 好問題（基本率） |
|---------------|----------------|
| 「修改 exit code 能解決嗎？」 | 「CLI 工具 hook error 的常見根因有哪些？各自佔比多少？」 |
| 「這個方案可行嗎？」 | 「類似的重構在其他專案中，常見的失敗原因是什麼？」 |
| 「代理人會完成任務嗎？」 | 「代理人回合耗盡的常見閾值是什麼？超過多少行 prompt 容易耗盡？」 |

**Skill 設計啟示**：
在實境檢驗階段，引導 PM 問基本率問題：
- 「這類問題的常見解法有哪些？成功率分別多少？」
- 「類似的修改在過去的案例中，常見的副作用是什麼？」
- 「這個技術方案的已知限制和 trade-off 是什麼？」

而非問預測性問題（「這樣改會不會成功？」→ AI 會過度樂觀地說「應該可以」）

### 用戶設計方向 — AI 模型的知識優勢（內建的爬梯子能力）

**核心概念：AI 不只能搜尋，還能從訓練知識中直接調用已知案例**

用戶指出：大部分問題都有公開的研究案例 —
- 組織管理、商業邏輯拆分
- 科學性的問題粒度拆分
- 特定程式設計（SQL 高併發、資料量效率）
- 基礎設施需求（可觀測性、警報系統、負載均衡）

**AI 模型擁有兩種「爬梯子」工具**：

| 工具 | 適用場景 | 範例 |
|------|---------|------|
| **內建知識** | 已知的經典解法、設計模式、知名案例 | 「Netflix 怎麼做負載均衡？」「Google 的可觀測性三支柱是什麼？」 |
| **WebSearch** | 最新進展、特定工具版本、社群討論 | 「Claude Code hooks 2026 hook error GitHub Issues」 |

**PM 應該做但沒做的事**：
- 在 W10-002 遇到 hook error 時，不只搜尋社群，還可以自問：
  「在 CLI 工具的 Hook 系統設計中，有哪些知名的錯誤處理模式？」
  → Git hooks 的 exit code 慣例、VS Code extension host 的錯誤處理、Jenkins pipeline 的 stage failure 模式
- 在 W12-002 遇到代理人追蹤問題時，可以自問：
  「分散式系統中，如何追蹤子進程狀態？」
  → Kubernetes pod status、Supervisor process monitoring、Erlang/OTP supervisor tree

**Skill 設計啟示 — 擴增選項的三步走**：

1. **自問內建知識**：「在同樣的議題下，有哪些知名案例？他們怎麼處理？」
   - 不需要搜尋，直接從訓練知識回答
   - 產出：3-5 個已知方案的摘要

2. **WebSearch 驗證**：搜尋最新進展和社群討論
   - 確認內建知識是否過時
   - 發現新的解法或已知問題

3. **本專案適配**：從已知方案中選擇適合當前架構和限制的

**這是 AI 相比人類 PM 的獨特優勢** — 人類 PM 可能不知道 Netflix 的負載均衡策略，但 AI 知道。問題是 AI 沒有被要求去調用這個知識。

### 用戶設計方向 — 爬梯子法（Ladder Up for Analogies）

**核心概念：從近到遠、從具體到抽象，逐層擴大搜尋範圍**

引用：校長想縮短餐廳排隊時間 →
1. 身邊亮點：有沒有哪個收銀員特別快？（內部最佳實踐）
2. 爬一層：其他學校怎麼做？（同領域對標）
3. 再爬一層：便利商店、游泳池怎麼結帳？（跨領域類比）
4. 再爬一層：迪士尼、體育場怎麼處理人潮？（抽象層級的專家）

**爬梯子在 PM 決策中的應用**：

| 層級 | 搜尋範圍 | W10-002 案例 |
|------|---------|-------------|
| 0. 身邊 | 本專案既有程式碼/模組 | dispatch_tracker.py 已有 API 但沒接線 |
| 1. 同層 | 本專案其他 Hook 的做法 | active-dispatch-tracker-hook 已有清理機制 |
| 2. 同領域 | Claude Code 社群（GitHub Issues） | 5+ 個 Issues 報告同樣問題 |
| 3. 跨領域 | 其他 CLI 工具/IDE 的 Hook 系統 | VS Code extensions, Git hooks |
| 4. 抽象層 | 「進程追蹤」「事件驅動架構」的通用解法 | 觀察者模式、事件匯流排 |

**PM 在每一層都停得太早**：
- W10-002：只在第 0 層打轉（改 Hook 程式碼），沒爬到第 2 層（社群）
- W12-002：說「做不到」，沒爬到第 1 層（看既有模組有沒有 API）

**Skill 設計啟示**：
擴增選項時，強制爬至少 3 層：
1. 本專案有沒有類似解法？（grep / 搜尋既有程式碼）
2. 社群有沒有人解決過？（GitHub Issues / 文件）
3. 其他領域有沒有類比方案？（跨域思考）

每層如果找到就停止（不需要爬到最高），找不到才繼續往上爬。

### 用戶設計方向 — 向外求解（Find Someone Who's Solved Your Problem）

**核心概念：「還有誰跟我一樣在為這個問題傷腦筋？我可以從他們那裡學到什麼？」**

引用：沃爾頓（Walmart 創辦人）找到聰明解決方案的方法。

**這正是 W10-002 的教訓**：
- PM 花了 3 輪修改試圖解決 hook error 問題
- 如果在第一輪就問「還有誰遇到同樣的問題？」→ 搜尋 GitHub Issues → 10 分鐘找到答案
- 已有 5+ 個 Issues 報告同樣症狀（#34713, #34859, #10936...）

**在決策框架中的位置**：
- 這是第一階段「擴增選項」的具體手法之一
- 擴增選項不只是自己想，還要向外看：社群、文件、類似專案
- 「自己從零開始想」vs「找到已經解決的人」— 後者幾乎永遠更有效率

**Skill 設計啟示**：
在擴增選項階段，強制加入一個檢查步驟：
- 「這個問題，社群/文件中有人遇過嗎？」
- 搜尋關鍵字：GitHub Issues, Stack Overflow, 官方文件
- 如果找到 → 可能根本不需要自己實作

### 用戶設計方向 — 季辛吉陷阱（官僚假選項）

**核心概念：三個選項中只有一個是真的，其他兩個是用來襯托的**

引用：季辛吉回憶錄 — 國務院給尼克森三個選項：發動核戰 / 維持現狀 / 投降。看似有選擇，實際上只有「維持現狀」可選。決策者自始至終都陷在偏狹的框架裡。

**對應 PM 的問題**：
- PM 自己就是那個「幕僚」，也是那個「決策者」
- PM 給自己的三個選項（改 exit code / 改 stderr / 改 stdout）本質上都是「修改 Hook」
- 就像「核戰 / 現狀 / 投降」本質上都是「對歐洲的軍事政策」
- 真正需要的選項（搜尋社群 / 接受 CLI 限制 / 改變流程設計）完全不在框架內

**Skill 設計啟示 — 假選項偵測規則**：

對 PM 列出的選項進行品質檢查：

| 檢查 | 問題 | 不通過的信號 |
|------|------|-------------|
| 根因多元性 | 選項是否基於不同的根因假設？ | 全部指向同一個根因 |
| 途徑多元性 | 選項是否代表不同的解決途徑？ | 全部是同一途徑的參數微調 |
| 利害關係人 | 選項是否只考慮一方觀點？ | 沒有「不做」或「問別人」的選項 |
| 框架測試 | 能否用一句話概括所有選項？ | 可以 → 框架太窄（如「都是修改 Hook」）|

**如果檢查不通過 → 強制回到第一階段重新擴增選項。**
### 用戶設計方向 — 避免假選項（Sham Options）

**核心概念：選項必須有實質差異，避免「分頭削弱」和「假選項」**

引用：廣告設計案例 — 如果三種設計稿只是字級大小的差異（11 級 vs 12 級），那不是真正的「分頭並進」，是「分頭削弱」（multiweaking）。假選項的作用只是讓真選項看起來順眼一點。

**品質檢查規則**：
- 擴增出的選項必須有**不同實質意義**（不同的途徑、不同的 trade-off）
- 如果選項之間只是參數差異（改 exit code vs 改 stderr vs 改 stdout 輸出格式），那是同一個途徑的微調，不是真正的多選項
- 真正的多選項範例：(1) 修改 Hook (2) 搜尋社群確認是否 CLI bug (3) 改變流程不依賴 Hook

**假選項辨識**：
- 三個選項是否都指向同一個根因假設？→ 假選項
- 如果移除一個選項，其他選項的評估會改變嗎？→ 不會則是假選項
- 選項之間的 trade-off 是否真正不同？→ 相同則是假選項

（用戶將繼續補充）
### 用戶設計方向 — 選項消失測試（Vanishing Options Test）

**核心概念：如果目前考慮的所有選項都不能選，你還可以怎麼做？**

出處：《零偏見決斷法》（Decisive）- Chip Heath & Dan Heath

**四階段決策框架（完整）**：

| 階段 | 核心問句 | 防止什麼 |
|------|---------|---------|
| 1. 擴增選項 | 「還有什麼其他方式？」 | 聚光燈只照一個選項 |
| 2. 實境檢驗 | 「需要什麼事證才能證明可行？」 | 未驗證就行動 |
| 3. 機會成本顯性化 | 「不選的代價是什麼？」 | 忽略隱性成本 |
| 4. 選項消失測試 | 「如果以上全部不能選，還能怎麼做？」 | 被既有選項框住 |

**對應 PM 的問題**：
- W10-002 中 PM 的選項都是「怎麼修改 Hook」（改 exit code / 改 stderr / 改 stdout）
- 如果這三個選項全部消失，PM 還能怎麼做？
- 答案：搜尋社群看看這是不是 CLI bug → 10 分鐘找到答案
- 「搜尋社群」完全在 PM 的選項框架之外，因為聚光燈只照著「修改 Hook」

**設計啟示**：
- Skill 在所有選項列出後，最後強制問一句：「如果以上選項全部不可用，你會怎麼做？」
- 這個問題迫使思維跳出當前框架，可能發現完全不同維度的解法

（書籍來源：《零偏見決斷法》 https://readmoo.com/book/210321101000101）
### 用戶設計方向 — 機會成本顯性化

**核心概念：機會成本必須被明確說出來，否則會被忽略**

引用：DVD 實驗 — 只加上「不買，留下 14.99 美元買別的東西」這句看似多餘的提示，不買的人就從 25% 增加到 45%。

**關鍵洞察**：
- 機會成本是「顯而易見」的事 — 不買 DVD 當然可以拿錢買別的
- 但如果沒有**明確列出**，人就是會忽略
- 這不是智力問題，是認知結構問題 — 聚光燈不會自動照到「不選的代價」

**對應 PM 的問題**：
- PM 決定修改 run_hook_safely 時，沒有明確列出「不修改的選項是什麼？」
- 如果當時被提示「不修改 hook，把時間用來搜尋社群，可能 10 分鐘就找到答案」
- 這句「看似多餘」的提示可能改變決策

**設計啟示**：
- 規則/Skill 在每個選項旁邊必須**強制列出機會成本**
- 不能假設 PM 會自動考慮「不選的代價」
- 格式類似：「選項 A：修改 hook | 機會成本：花 2 小時可能白做，這段時間可以搜尋社群或做其他 Ticket」

### 用戶設計方向 — 第二階段（待補充）

**核心概念：實境檢驗（Reality-Test Your Assumptions）**

引用：馬丁的問句「需要什麼樣的事證，才能讓這個選項是確實可行的答案？」改變了討論框架 — 把對立的雙方變成合作的夥伴。

**兩階段決策框架**：

**第一階段：擴增選項（Widen Your Options）**
- 不要預設只有一種做法
- 不要只問「要不要做」「能不能做」
- 讓聚光燈移動，看到更多可能性

**第二階段：實境檢驗（Reality-Test）**
- 對每個選項問：「需要什麼事證才能證明這個方法可行？」
- 找出每個選項的優點相對於其他方案
- 同時評估每個選項的：
  - **機會成本**：選這個方案要放棄什麼？
  - **風險**：這個方案可能失敗的條件是什麼？
  - **不選的代價**：如果不選其他選項，會付出什麼代價？

**對應 PM 的問題**：
- PM 在 W10-002 中對 hook error 問題跳過了實境檢驗
  - 假設 1（exit code）→ 沒找事證就修改
  - 假設 2（stderr）→ 沒找事證就修改  
  - 假設 3（空 stdout）→ 沒找事證就修改
  - 如果先問「需要什麼事證來證明是 exit code 問題？」→ 手動測試 exit code 都是 0 → 排除
  - 如果先問「不搜尋社群的代價是什麼？」→ 可能花大量時間修改 CLI bug

**核心轉變**：
從「我覺得是這個原因 → 直接修改」
到「需要什麼事證才能確認是這個原因？→ 先驗證再行動」
