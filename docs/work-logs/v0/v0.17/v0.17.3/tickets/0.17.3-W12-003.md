## Problem Analysis

### 用戶設計方向 — 季辛吉陷阱（官僚假選項）

**核心概念：三個選項中只有一個是真的，其他兩個是用來襯托的**

引用：季辛吉回憶錄 — 國務院給尼克森三個選項：發動核戰 / 維持現狀 / 投降。看似有選擇，實際上只有「維持現狀」可選。決策者自始至終都陷在偏狹的框架裡。

**對應 PM 的問題**：
- PM 自己就是那個「幕僚」，也是那個「決策者」
- PM 給自己的三個選項（改 exit code / 改 stderr / 改 stdout）本質上都是「修改 Hook」
- 就像「核戰 / 現狀 / 投降」本質上都是「對歐洲的軍事政策」
- 真正需要的選項（搜尋社群 / 接受 CLI 限制 / 改變流程設計）完全不在框架內

**Skill 設計啟示 — 假選項偵測規則**：

對 PM 列出的選項進行品質檢查：

| 檢查 | 問題 | 不通過的信號 |
|------|------|-------------|
| 根因多元性 | 選項是否基於不同的根因假設？ | 全部指向同一個根因 |
| 途徑多元性 | 選項是否代表不同的解決途徑？ | 全部是同一途徑的參數微調 |
| 利害關係人 | 選項是否只考慮一方觀點？ | 沒有「不做」或「問別人」的選項 |
| 框架測試 | 能否用一句話概括所有選項？ | 可以 → 框架太窄（如「都是修改 Hook」）|

**如果檢查不通過 → 強制回到第一階段重新擴增選項。**
### 用戶設計方向 — 避免假選項（Sham Options）

**核心概念：選項必須有實質差異，避免「分頭削弱」和「假選項」**

引用：廣告設計案例 — 如果三種設計稿只是字級大小的差異（11 級 vs 12 級），那不是真正的「分頭並進」，是「分頭削弱」（multiweaking）。假選項的作用只是讓真選項看起來順眼一點。

**品質檢查規則**：
- 擴增出的選項必須有**不同實質意義**（不同的途徑、不同的 trade-off）
- 如果選項之間只是參數差異（改 exit code vs 改 stderr vs 改 stdout 輸出格式），那是同一個途徑的微調，不是真正的多選項
- 真正的多選項範例：(1) 修改 Hook (2) 搜尋社群確認是否 CLI bug (3) 改變流程不依賴 Hook

**假選項辨識**：
- 三個選項是否都指向同一個根因假設？→ 假選項
- 如果移除一個選項，其他選項的評估會改變嗎？→ 不會則是假選項
- 選項之間的 trade-off 是否真正不同？→ 相同則是假選項

### 用戶設計方向 — 選項消失測試（Vanishing Options Test）

**核心概念：如果目前考慮的所有選項都不能選，你還可以怎麼做？**

出處：《零偏見決斷法》（Decisive）- Chip Heath & Dan Heath

**四階段決策框架（完整）**：

| 階段 | 核心問句 | 防止什麼 |
|------|---------|---------|
| 1. 擴增選項 | 「還有什麼其他方式？」 | 聚光燈只照一個選項 |
| 2. 實境檢驗 | 「需要什麼事證才能證明可行？」 | 未驗證就行動 |
| 3. 機會成本顯性化 | 「不選的代價是什麼？」 | 忽略隱性成本 |
| 4. 選項消失測試 | 「如果以上全部不能選，還能怎麼做？」 | 被既有選項框住 |

**對應 PM 的問題**：
- W10-002 中 PM 的選項都是「怎麼修改 Hook」（改 exit code / 改 stderr / 改 stdout）
- 如果這三個選項全部消失，PM 還能怎麼做？
- 答案：搜尋社群看看這是不是 CLI bug → 10 分鐘找到答案
- 「搜尋社群」完全在 PM 的選項框架之外，因為聚光燈只照著「修改 Hook」

**設計啟示**：
- Skill 在所有選項列出後，最後強制問一句：「如果以上選項全部不可用，你會怎麼做？」
- 這個問題迫使思維跳出當前框架，可能發現完全不同維度的解法

（書籍來源：《零偏見決斷法》 https://readmoo.com/book/210321101000101）
### 用戶設計方向 — 機會成本顯性化

**核心概念：機會成本必須被明確說出來，否則會被忽略**

引用：DVD 實驗 — 只加上「不買，留下 14.99 美元買別的東西」這句看似多餘的提示，不買的人就從 25% 增加到 45%。

**關鍵洞察**：
- 機會成本是「顯而易見」的事 — 不買 DVD 當然可以拿錢買別的
- 但如果沒有**明確列出**，人就是會忽略
- 這不是智力問題，是認知結構問題 — 聚光燈不會自動照到「不選的代價」

**對應 PM 的問題**：
- PM 決定修改 run_hook_safely 時，沒有明確列出「不修改的選項是什麼？」
- 如果當時被提示「不修改 hook，把時間用來搜尋社群，可能 10 分鐘就找到答案」
- 這句「看似多餘」的提示可能改變決策

**設計啟示**：
- 規則/Skill 在每個選項旁邊必須**強制列出機會成本**
- 不能假設 PM 會自動考慮「不選的代價」
- 格式類似：「選項 A：修改 hook | 機會成本：花 2 小時可能白做，這段時間可以搜尋社群或做其他 Ticket」

### 用戶設計方向 — 第二階段（待補充）

**核心概念：實境檢驗（Reality-Test Your Assumptions）**

引用：馬丁的問句「需要什麼樣的事證，才能讓這個選項是確實可行的答案？」改變了討論框架 — 把對立的雙方變成合作的夥伴。

**兩階段決策框架**：

**第一階段：擴增選項（Widen Your Options）**
- 不要預設只有一種做法
- 不要只問「要不要做」「能不能做」
- 讓聚光燈移動，看到更多可能性

**第二階段：實境檢驗（Reality-Test）**
- 對每個選項問：「需要什麼事證才能證明這個方法可行？」
- 找出每個選項的優點相對於其他方案
- 同時評估每個選項的：
  - **機會成本**：選這個方案要放棄什麼？
  - **風險**：這個方案可能失敗的條件是什麼？
  - **不選的代價**：如果不選其他選項，會付出什麼代價？

**對應 PM 的問題**：
- PM 在 W10-002 中對 hook error 問題跳過了實境檢驗
  - 假設 1（exit code）→ 沒找事證就修改
  - 假設 2（stderr）→ 沒找事證就修改  
  - 假設 3（空 stdout）→ 沒找事證就修改
  - 如果先問「需要什麼事證來證明是 exit code 問題？」→ 手動測試 exit code 都是 0 → 排除
  - 如果先問「不搜尋社群的代價是什麼？」→ 可能花大量時間修改 CLI bug

**核心轉變**：
從「我覺得是這個原因 → 直接修改」
到「需要什麼事證才能確認是這個原因？→ 先驗證再行動」
