## Problem Analysis

### 用戶設計方向 — 第二階段（待補充）

**核心概念：實境檢驗（Reality-Test Your Assumptions）**

引用：馬丁的問句「需要什麼樣的事證，才能讓這個選項是確實可行的答案？」改變了討論框架 — 把對立的雙方變成合作的夥伴。

**兩階段決策框架**：

**第一階段：擴增選項（Widen Your Options）**
- 不要預設只有一種做法
- 不要只問「要不要做」「能不能做」
- 讓聚光燈移動，看到更多可能性

**第二階段：實境檢驗（Reality-Test）**
- 對每個選項問：「需要什麼事證才能證明這個方法可行？」
- 找出每個選項的優點相對於其他方案
- 同時評估每個選項的：
  - **機會成本**：選這個方案要放棄什麼？
  - **風險**：這個方案可能失敗的條件是什麼？
  - **不選的代價**：如果不選其他選項，會付出什麼代價？

**對應 PM 的問題**：
- PM 在 W10-002 中對 hook error 問題跳過了實境檢驗
  - 假設 1（exit code）→ 沒找事證就修改
  - 假設 2（stderr）→ 沒找事證就修改  
  - 假設 3（空 stdout）→ 沒找事證就修改
  - 如果先問「需要什麼事證來證明是 exit code 問題？」→ 手動測試 exit code 都是 0 → 排除
  - 如果先問「不搜尋社群的代價是什麼？」→ 可能花大量時間修改 CLI bug

**核心轉變**：
從「我覺得是這個原因 → 直接修改」
到「需要什麼事證才能確認是這個原因？→ 先驗證再行動」
