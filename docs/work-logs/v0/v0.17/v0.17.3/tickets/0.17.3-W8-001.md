## Problem Analysis

## 全節點審查結果

### 一、decision-tree.md（路由索引）— 3 項發現

**F-01 [中] 明確性檢查定位模糊**
L30 的流程圖將「明確性 + 類型判斷」合併為一個步驟，但實際上明確性檢查（第零層）應在類型判斷（第一層）之前獨立發生。目前明確性檢查的內容被分散到 question-routing.md 和 command-routing.md 中，各有不同的觸發條件（問題：不確定性詞彙；命令：複雜需求），這是合理的按分支特化，但索引圖中的描述容易誤導讀者以為是同一步驟。

**F-02 [低] 執行/完成的流程圖暗示序列關係**
L40-44 將「執行 Domain」和「完成 Domain」畫成線性序列，但實際上它們是不同觸發條件（執行中發現 vs 任務完成），不是 A 完成後才到 B。

**F-03 [中] 缺少 Ticket 進度更新時機的決策節點**
整棵決策樹沒有定義「PM/代理人何時應該更新 Ticket 進度（append-log、phase update）」。context-bundle-spec.md 有代理人中間更新規範（W15-001），但基礎決策樹未將「Ticket 進度更新」作為決策節點。這導致本次 W7-001 就發生了未即時更新 Ticket 的問題。

---

### 二、dispatch-gate.md（派發閘門）— 2 項發現

**F-04 [中] 包含了不屬於閘門的路由邏輯**
L54-56「主線程必須親自處理?」是一個路由決策（決定誰來做），不是派發前檢查（決定能不能派）。派發閘門的職責應純粹是：複雜度 OK 嗎？Context Bundle 準備好了嗎？可以並行嗎？「由誰執行」屬於分工路由，應回到主索引或在 command-routing 中處理。

**F-05 [低] 缺少 PM 前台任務（ANA/DOC）跳過閘門的說明**
PM 自己做的任務不需要通過 Context Bundle 關卡（沒有代理人需要接收 Bundle），但閘門中未明確說明此豁免。

---

### 三、question-routing.md（問題路由）— 2 項發現

**F-06 [高] 「所有查詢都應派發代理人」與 PM 前台分工矛盾**
L32 寫「所有查詢都應派發代理人，主線程禁止直接執行查詢命令」，但 command-routing.md L52-53 定義「ANA/DOC 由 PM 前台執行」。簡單的 Ticket 進度查詢（如 ticket track snapshot）PM 應該直接做，不需要派發 Explore agent。這條規則過時，需區分「資料收集型查詢」（派發）和「狀態查詢」（PM 直接）。

**F-07 [低] 問題中涉及的檔案類型派發章節空洞**
L47-51 只有一個指向 agent-path-registry.md 的引用，沒有實質內容。要麼補充具體情境說明，要麼移除此空章節。

### 六、agent-path-registry.md（代理人路徑表）— 2 項發現

**F-09 [高] 路徑表與本專案實際結構不符**
本專案是 JavaScript Chrome Extension（程式碼在 src/），但路徑表包含 Flutter 路徑（ui/lib/**/*.dart）和 Go 路徑（server/**/*.go）。本專案實際需要的是：
- thyme-extension-engineer（CLAUDE.md 4 節定義）操作 src/**/*.js — 但路徑表中完全沒有此代理人
- .md 路徑的主線程編輯權限是正確的

原因：.claude/ 配置透過 sync-push/sync-pull 跨專案共用，路徑表是通用定義而非專案特定。但作為 Source of Truth，至少應包含本專案的代理人。

**F-10 [中] 缺少 thyme-extension-engineer 的路徑權限**
CLAUDE.md 第 4 節明確指定本專案的實作代理人是 thyme-extension-engineer，但 agent-path-registry.md 沒有列出此代理人的可編輯路徑（src/**/*.js）。

## 職責分工分析 + 修正建議

### 基礎決策樹 vs Ticket Skill 決策樹 職責邊界

| 層級 | 基礎決策樹（pm-rules/） | Ticket Skill 決策樹（workflow-*.md） |
|------|----------------------|-------------------------------------|
| 定義 | **行為選項** + 觸發條件 | **行為細節** + 執行步驟 |
| 回答 | 「應該做什麼？由誰做？」 | 「具體怎麼做？用什麼命令？」 |
| 範例 | 「任務完成 → Checkpoint 循環」 | 「complete 的 4 步驟 CLI 驗證」 |
| 範例 | 「ANA 類型 → PM 前台」 | 「claim 的具體 CLI 語法」 |
| 範例 | 「進度變化時 → 更新 Ticket」 | 「append-log --section 的有效值」 |

**當前問題**：基礎決策樹缺少「Ticket 進度更新時機」這個行為選項節點（F-03, F-11），導致 PM 不知道何時該更新。Ticket Skill 定義了具體命令但無人觸發。
