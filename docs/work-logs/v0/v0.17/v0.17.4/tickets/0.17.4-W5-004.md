## Problem Analysis

### 根因分析

**事實**：W4-001.3 用了兩次 WRAP（完整+快速），兩次都錨定在「tool call 閾值數字」，未能質疑「閾值校準」這個問題定義本身是否正確。直到用戶指出拆分核心是 SRP 品質。

**根因**：WRAP 框架缺少「問題定義檢驗」環節。
1. 錨點確認只問「影響決策品質嗎？」→ PM 回答「是」就進入分析，但未質疑問題定義
2. W 階段的品質檢查（選項消失/假選項/框架測試）都在解法空間內，無法跳出問題框架
3. R 階段的基本率搜尋被錯誤的問題定義限制

**類比**：這等同於 WRAP 書中「季辛吉陷阱」的上層版本 — 選項多元性不足是季辛吉陷阱，問題定義不足是「超級季辛吉陷阱」。

### 改善方案

在 WRAP 的 W 階段品質檢查中新增第 4 項「問題重新定義測試」：

現有 3 項：
1. 選項消失測試
2. 假選項偵測
3. 框架測試

新增：
4. **問題重新定義測試**：「如果這個問題本身是錯的，真正應該問的問題是什麼？」
   - 能回答 → 評估是否需要切換問題定義
   - 回答不出 → 繼續原問題
