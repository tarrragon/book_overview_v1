### Decision Questions 設計（含修正）

#### 組 1：中間步驟測試（觸發條件：功能涉及多步驟流程時）
Q7: 測試驗證的是最終結果還是中間過程？
→ 行動路徑：若只測最終結果 → 識別中間步驟清單，每個至少一個獨立測試
Q8: 新建的私有方法是否被至少一個測試的斷言直接覆蓋（透過公開入口呼叫）？
Q9: 如果實作讀錯了欄位名，這份測試資料會不會碰巧讓測試通過？（最佳範例，保持原樣）

正面範例（v0.17.0 P0-1）：
  _buildTitleAuthorKey 讀取 book.author 但 v2 是 authors → 測試資料同時有 author 和 authors 導致碰巧通過
  防護：測試資料只保留 v2 欄位 authors，移除 author → 測試會正確失敗
反面範例：
  getBookById(id) 內部只調 storage.get → 不需要中間步驟測試，直接測最終結果即可

#### 組 2：非功能性測試（觸發條件：Phase 1 規格含非功能性需求時）
Q10: 列出規格中定義的每個 error handling 路徑，對照測試清單標記覆蓋狀態
Q_new3: expect 語句驗證的是值本身（equals）還是存在性（isNotNull）？逐一列舉並確認
Q_new4: Phase 1 定義的每個異常處理路徑，是否都有對應的失敗測試？列出對照表

正面範例（v0.17.0 P1-3）：
  規格（observability-rules.md）要求「catch 區塊必須有日誌」→ 但 W2 測試未驗證 catch 區塊是否調用 logger
  防護：Phase 2 增加 mock logger 驗證 call count
非功能性需求測試推導表（新增）：
  | Phase 1 需求 | Phase 2 測試 | 驗證方式 |
  | 異常時記錄日誌 | catch 區塊日誌 | Mock logger，驗證 call count 和參數 |
  | 日誌含堆疊追蹤 | 日誌訊息結構 | 驗證日誌包含 error.stack |

#### 組 3：資料純淨性（觸發條件：涉及 schema 版本遷移時）
Q11: 測試資料工廠是否只使用目標版本的欄位？有無殘留舊版？

正面範例（v0.17.0 P0-1）：
  createBookV2() 工廠同時包含 author（v1）和 authors（v2）→ 殘留 v1 欄位掩蓋了 bug
  修正後：createBookV2() 只包含 authors，完全移除 author

### 中間步驟時序問題補充（來自防護有效性審查）
Phase 2 時新建的私有方法可能還不存在（Phase 3 才建立）。因此：
- Phase 2 設計「方法契約測試」：根據規格定義的資料結構，預定義中間方法的輸入輸出
- Phase 3 實作時，新建方法必須符合 Phase 2 預定義的契約
