# Execution Log

## Task Summary

調查 Agent worktree 分支基於舊 commit 導致無法合併和測試不實際

---

## Problem Analysis

### 問題根因

**根因 1：Claude Code worktree 建立基於當時的 main HEAD，但不會自動更新**

Claude Code 的 `isolation: "worktree"` 機制在派發代理人時，使用 `git worktree add` 基於**當時的 main HEAD** 建立分支。一旦建立後，worktree 分支的基底 commit 就固定了。

本次事件中，11 個 worktree 的基底情況如下：

| 基底 commit | worktree 數量 | 說明 |
|------------|---------------|------|
| `ec25000`（W1 phase 最後一個 commit） | 8 個 | W1~W3 期間建立，從未清理 |
| `6dfeffa`（W5-001 修復 commit） | 1 個（agent-af3a4da1） | W5 期間建立 |
| `4c1a80b`（auto-commit） | 1 個（agent-a0ddc09d，本 worktree） | W5 最新，但基底仍是 ec25000 |

main HEAD 已推進到 `79fbe99`，距離 `ec25000` 有 22 個 commit。8 個 worktree 的分支基底停留在 W1 結束時。

**根因 2：舊 worktree 未清理，Claude Code 可能重用舊 worktree**

`.claude/worktrees/` 目錄累積了 11 個 worktree，其中 8 個基於同一個過時的 commit。沒有自動清理機制（W4-006 已確認此問題）。Claude Code 的 worktree 機制以 agent ID（UUID 後 8 碼）命名，每次派發理論上建立新的 worktree。但舊 worktree 殘留佔用磁碟空間且造成 `git worktree list` 輸出混亂。

**根因 3：代理人在舊程式碼基底上工作，測試結果不反映真實狀態**

W5-004 代理人被派發到 worktree 中，其工作目錄的程式碼停留在 `ec25000`。代理人在該環境中新增了測試檔案（261 行邊界條件測試），但：
- 這些測試是針對舊版 `TagStorageAdapter`（W1 時期的版本）
- main 上的 `chrome-storage-tag-operations.test.js` 已經在 W3 Phase 3b 被重寫為 973 行的完整測試
- 代理人報告「8 個測試通過」是真實的，但測試的對象是舊版程式碼

**根因 4：auto-commit hook 保留了工作但無法合併**

`worktree-auto-commit-hook.py`（W4-006 建立）在代理人 session 結束時成功觸發，將未提交變更自動 commit 為 `4c1a80b`。Hook 運作正常，但產出的 commit 基於 `ec25000`，與 main 差距 22 個 commit。嘗試 `git merge` 會 revert 大量已完成的工作。

### 影響範圍

| 影響項目 | 說明 |
|---------|------|
| W5-004 Ticket | 代理人的工作（261 行測試）無法合併到 main，需要在 main 上重做 |
| `.claude/worktrees/` | 11 個 worktree 佔用磁碟空間，8 個已完全過時 |
| 代理人測試信任度 | 代理人報告測試通過，但基於舊程式碼的測試結果無法反映 main 的真實狀態 |

## Solution

### 修正建議

**1. 清理過時 worktree（立即執行，P0）**

```bash
# 移除所有基於 ec25000 的過時 worktree
git worktree remove .claude/worktrees/agent-a4dce5ba --force
git worktree remove .claude/worktrees/agent-a5935a88 --force
git worktree remove .claude/worktrees/agent-a5acf03a --force
git worktree remove .claude/worktrees/agent-a66ef66e --force
git worktree remove .claude/worktrees/agent-a96180bb --force
git worktree remove .claude/worktrees/agent-a9fef8e7 --force
git worktree remove .claude/worktrees/agent-aa71aa50 --force
git worktree remove .claude/worktrees/agent-aac29a94 --force
git worktree remove .claude/worktrees/agent-abe2338a --force
# 保留 agent-af3a4da1 和 agent-a0ddc09d 做後續分析

# 清理已移除 worktree 的分支
git branch -D worktree-agent-a4dce5ba
# ... 對每個移除的 worktree 重複
```

**2. 派發前 rebase worktree（流程改善，P1）**

在派發代理人到 worktree 時，應在代理人指令中加入 rebase 步驟：
```bash
# 代理人在 worktree 中開始工作前
git rebase main
```
或者改用 `git worktree add` 時明確指定基底：
```bash
git worktree add .claude/worktrees/agent-xxx -b worktree-agent-xxx main
```
（Claude Code 內建 worktree 機制是否支援此選項需進一步確認）

**3. 定期 worktree GC（防護措施，P2）**

建議在 `SessionStart` hook 中加入 worktree 過期檢查：
- 列出所有 worktree 的建立時間（用目錄 mtime）
- 超過 N 天（如 3 天）且分支已合併的 worktree 自動移除
- 超過 N 天且分支未合併的 worktree 發出 WARNING

**4. 代理人結果驗證（流程改善，P1）**

PM 在代理人完成後，應驗證 worktree 分支基底：
```bash
# 檢查 worktree 分支與 main 的距離
git log --oneline main..worktree-agent-xxx | wc -l
git log --oneline worktree-agent-xxx..main | wc -l
```
如果 worktree 落後 main 超過 5 個 commit，代理人的工作結果應視為不可信。

### 新 Error Pattern

本問題歸類為新的錯誤模式 **PC-036: Worktree 分支基底過舊導致代理人工作無效**
