## 最終執行策略（三視角終審 + 範例擴充機制）

### Decision Questions 設計（含修正）

#### 組 1：跨模組共用策略（觸發條件：規格涉及 2+ 模組時）
Q1: 多個模組是否有相似的驗證邏輯/資料結構/狀態轉換？
Q2: 若有，選擇共用或各自實作，填寫：(a) 相似度（行數/邏輯步驟）(b) 未來變更是否同步 (c) 若各自實作，重複行數估計
→ 行動路徑：相似度 > 50% 且未來同步變更 → 定義共用元件

正面範例（v0.17.0）：
  BookSchemaV2 和 TagSchema 都驗證字串長度和格式 → 應共用驗證引擎
反面範例：
  BookSchema 驗證 title（1-200字）、BlogSchema 驗證 content（任意長度含 Markdown）→ 語義不同，不應共用

#### 組 2：ID 欄位設計（觸發條件：資料結構含 ID 欄位時）
Q3: ID 生成涉及時序嗎？快速連續操作（< 1ms）是否可能碰撞？
Q4: 唯一性策略選擇：timestamp+random / UUID / 序列號 / 其他
→ 行動路徑：碰撞可能 → 必須加隨機後綴或用 UUID

正面範例（v0.17.0）：
  cat_{Date.now()} 在批量建立 tag 時碰撞 → 應改為 cat_{timestamp}_{random6}
邊界範例：
  單次操作且使用者手動觸發 → timestamp 足夠，無需 UUID

#### 組 3：非功能性需求（觸發條件：所有功能規格）
Q5: 出錯時操作者如何知道？選擇日誌/通知/靜默 → 若日誌，定義等級和包含欄位
Q6a: 本功能最慢的操作是什麼？預期延遲多少？（拆分自原 Q6）
Q6b: 寫入操作中途失敗時，已寫入的資料如何處理？回滾/孤立/重試？（拆分自原 Q6）

正面範例（v0.17.0）：
  tag-storage-adapter 所有 catch 區塊零日誌 → 規格應定義「CRUD 操作的 catch 必須記錄 warning 日誌」
反面範例：
  純讀取操作的快取 miss → 不需要日誌，靜默即可

#### 補充：欄位命名一致性（觸發條件：涉及 schema 版本遷移時）
Q_new1: 哪些欄位名稱在不同版本間有變化？列出映射表（如 author → authors）

正面範例（v0.17.0）：
  v1 用 author（字串），v2 用 authors（陣列）→ 規格應有明確映射表，防止實作混用
