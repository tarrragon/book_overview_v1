邊界範例（v0.17.0 P2-4）：
  v1-to-v2.js 和 tag-storage-adapter.js 都硬編碼 "3.0.0" → 應由 BookSchemaV2 匯出 SCHEMA_VERSION

改為「執行清單」而非「檢查清單」：
- [ ] 執行 npm run lint 並確認 0 errors
- [ ] 執行對應的測試套件並確認全部通過
- [ ] 新建私有方法：參數名稱是否對應規格中的欄位定義？
Q_new2: 新增模組的每個 import 是否都在程式碼中被使用？列舉確認

  防護：完成前執行 npm run lint → ESLint no-unused-vars 攔截

  _buildTitleAuthorKey 讀取 book.author 但規格定義 authors → 參數未對應規格
  防護：自我驗證清單「參數對應規格欄位」→ 發現不一致

### Phase 4b：流程改善回饋環（新增章節，寫入 phase4-refactor.md）
```
Phase 4 審查發現

  ↓
改善 Ticket 建立（修改 Phase reference，不修改 SKILL.md）
  ↓

關鍵原則：先改善流程再修復程式碼。修復本身也是改善流程的驗證。

```

### 範例擴充機制（寫入 phase4-refactor.md 的流程改善章節）
Phase 4 檢討完成後的標準動作：
1. 根因分析 → 歸類到對應 Phase 的 reference
2. 為對應的 decision question 追加新範例
3. 若發現全新的盲區（現有問題未覆蓋）→ 建立新的 decision question
4. 格式：[v{版本號}] {案例描述}
