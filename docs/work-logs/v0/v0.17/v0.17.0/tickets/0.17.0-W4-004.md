## Phase 4 問題根因分析

### 分析方法
逐一比對每個 P0/P1 發現，追溯至規格文件（W1）、測試設計（W2）或實作品質（W3），判斷根因歸屬。

### 各發現根因

#### P0-1: book.author vs authors（去重靜默失效）
- 根因分類: 測試盲點
- 說明: 規格明確定義 authors（陣列），W2-004 測試資料工廠也用正確欄位。但測試只驗證「合併結果是否正確」，未測試「key 生成函式」這個中間步驟。_buildTitleAuthorKey 是新寫的私有方法，讀取 book.author（單數）而非 book.authors（複數），因測試資料同時殘留 author 和 authors，測試碰巧通過。
- 防護: Phase 2 應增加「中間步驟獨立單元測試」，不只測試最終結果。

#### P0-2: Dead import bare specifier
- 根因分類: 實作品質
- 說明: 代理人建立新模組時預設加了 require('src/core/errors/ErrorCodes') 但未使用。非規格或測試問題。
- 防護: ESLint no-unused-vars 應攔截。確認 lint 是否在 CI 中強制執行。

#### P1-1/2: 驗證框架重複 + 狀態轉換重複
- 根因分類: 規格盲點
- 說明: W1 規格分別定義了 Book Schema、Tag Schema、Migration 的規則，但未指出驗證邏輯結構相同應共用。每個 W3 代理人在獨立 worktree 各自實作，自然產生重複。
- 防護: Phase 1 規格應增加「跨模組共用策略」章節。Phase 3a 策略規劃應識別共用元件。

#### P1-3: 零日誌可觀測性
- 根因分類: 測試盲點
- 說明: W2 測試只驗證功能正確性（CRUD 成功/失敗），未驗證 catch 區塊是否記錄日誌。可觀測性規則存在於 rules 中但未轉化為測試案例。
- 防護: Phase 2 測試設計應包含「非功能性需求測試」檢查項（日誌、錯誤處理路徑、可觀測性）。

#### P1-4: Date.now() ID 碰撞
- 根因分類: 規格盲點
- 說明: 規格定義了 ID 格式 cat_{timestamp} 但未指出碰撞風險或唯一性保證機制。代理人照規格實作。
- 防護: Phase 1 規格中 ID 生成相關定義應包含「唯一性保證」要求。

### 根因統計

|---------|--------|------|--------|

| 規格盲點 | 2 | 40% | P1-1/2 缺少跨模組架構指引、P1-4 缺少碰撞防護要求 |

### 改善方向

#### 規格面（Phase 1 強化）
1. 當多個模組有相似結構時，規格應明確指出「共用策略」（共用驗證引擎、共用常數等）
2. ID 生成規格應包含唯一性保證要求，不只定義格式
3. 非功能性需求（可觀測性、錯誤處理）應在規格中有專門章節

#### 測試面（Phase 2 強化）
1. 測試設計應涵蓋「中間步驟」而非只測最終結果（尤其是新建的私有方法）
2. 增加非功能性需求測試項：catch 區塊日誌驗證、錯誤處理路徑完整性
3. 測試資料工廠應嚴格只使用 v2 欄位，不殘留 v1 欄位（防止碰巧通過）

#### 流程面（Phase 3a 強化）
1. 並行派發前的策略規劃應識別跨代理人的共用元件，先建立共用模組再派發
