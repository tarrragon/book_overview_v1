## 🎯 重構動機與目標

### 當前架構的具體問題

**基於實作檔案分析發現的問題**：

1. **錯誤處理不一致**：
   - `popup-event-controller.js` 第 147-151 行：使用原生 Error 物件而非專案 ErrorCodes 體系
   - `popup.js` 第 531-534 行：混用 ErrorCodes 與原生 Error
   - 違反專案 `📋 ESLint 錯誤處理規則` 強制要求

2. **Logger 使用不規範**：
   - `popup-event-controller.js` 第 183 行：使用 `Logger.error` 的異常註解方式
   - `popup.js` 第 840 行：同樣使用 ESLint 停用註解，違反專案 Logger 規範
   - 應使用專案標準 Logger 調用模式

3. **函式複雜度過高**：
   - `handleExportData` (第 595-644 行)：50行程式碼，違反「五事件評估準則」
   - `cancelExtraction` (第 453-503 行)：51行程式碼，職責過於臃腫
   - `checkBackgroundStatus` (第 526-595 行)：70行程式碼，需要拆分

4. **事件處理模式不統一**：
   - `event-manager.js` 混用不同的事件綁定策略
   - 缺乏統一的錯誤處理包裝器
   - 重試機制實作過於複雜

### 重構後期望達成的狀態

1. **錯誤處理完全統一**：
   - 100% 使用 ErrorCodes 體系，移除所有原生 Error 物件
   - 統一 IIFE 錯誤創建模式，符合專案規範

2. **Logger 使用規範化**：
   - 移除所有 ESLint 停用註解
   - 使用專案標準 Logger 調用模式

3. **函式單一責任**：
   - 所有函式控制在 30 行以內
   - 每個函式只負責一個明確的功能
   - 清晰的語意化命名

4. **事件處理模式統一**：
   - 建立統一的事件包裝器
   - 簡化重試機制
   - 提高程式碼可測試性

### 這個重構如何解決核心問題

**系統性解決方案**：
- **品質合規**：確保符合專案所有程式碼品質規範
- **架構統一**：與專案事件驅動架構完全一致
- **可測試性**：拆分複雜函式，提高單元測試覆蓋率
- **可維護性**：清晰的職責分離和語意化命名
