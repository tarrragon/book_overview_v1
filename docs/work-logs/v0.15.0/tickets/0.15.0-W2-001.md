# 0.15.0-W2-001: 多視角分析測試失敗模式與系統設計檢討

| 欄位 | 值 |
|------|------|
| **優先級** | P1 |
| **類型** | analysis |
| **狀態** | open |
| **建立日期** | 2026-03-26 |

## 背景

v0.15.0-W1-002 修復了 512 個測試失敗，過程中發現多種系統性問題。
這些失敗不是隨機的，而是反映了底層設計缺陷。需要系統性分析出錯模式，
作為後續系統設計檢討的依據。

## 決策樹

```
分析測試失敗模式
├── Sub-1: 多視角錯誤模式分類與統整
│   ├── 源碼視角：哪些源碼設計導致測試大量失敗？
│   ├── 測試視角：測試本身的設計問題是什麼？
│   ├── 架構視角：哪些架構決策產生了系統性問題？
│   └── 產出：錯誤模式分類報告
│
├── Sub-2: 根因追溯與影響範圍評估
│   ├── 每種模式的根本原因
│   ├── 影響範圍（檔案數、測試數、模組數）
│   └── 產出：根因分析報告
│
├── Sub-3: 系統設計檢討建議
│   ├── 基於錯誤模式提出設計改善方向
│   ├── 優先級排序
│   └── 產出：設計檢討建議書
│
└── Sub-4: 執行計畫制定
    ├── 將建議轉化為具體 ticket
    ├── 依賴關係和執行順序
    └── 產出：後續 ticket 清單
```

## 子任務

### Sub-1: 多視角錯誤模式分類與統整
- [ ] **源碼視角分析** — 哪些源碼 API 變更/設計導致測試失敗？
- [ ] **測試視角分析** — 測試本身有哪些共通的設計問題？
- [ ] **架構視角分析** — 哪些架構決策產生了系統性問題？
- [ ] **產出錯誤模式分類報告**

### Sub-2: 根因追溯與影響範圍評估
- [ ] 追溯每種模式的根本原因（何時引入、為何未被發現）
- [ ] 量化每種模式的影響範圍
- [ ] 產出根因分析報告

### Sub-3: 系統設計檢討建議
- [ ] 基於分析結果提出設計改善方向
- [ ] 評估改善的成本與效益
- [ ] 產出設計檢討建議書

### Sub-4: 執行計畫制定
- [ ] 將建議轉化為具體可執行的 ticket
- [ ] 定義依賴關係和執行順序
- [ ] 產出後續 ticket 清單

## 已知的錯誤模式（初步，待深入分析）

來自 W1-002 修復過程的觀察：

| 模式 | 影響估計 | 初步描述 |
|------|---------|---------|
| ErrorCodes 缺失 | 117 個代碼 | 定義 51 個但使用 168 個 |
| .toMatchObject 誤用 | ~30 處 | 對函式呼叫用物件匹配器 |
| Mock 對象不完整 | ~200 測試 | chromeBridge、Logger 等 |
| 效能閾值過嚴 | ~50 測試 | GC/系統負載導致不穩定 |
| API 架構不匹配 | ~34 測試 | class vs factory 假設錯誤 |
| 過時測試套件 | 15 個 skip | 架構假設完全過時 |
| Logger 靜態/實例混用 | ~211 處 | 源碼用法不一致 |
| StandardError 殘留 | 24+ 檔案 | 已棄用但仍被依賴 |
