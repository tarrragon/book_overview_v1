# Execution Log

## Task Summary

設計提案驅動的需求追蹤機制，解決 spec/usecase/worklog 之間的割裂問題。

---

## Problem Analysis

### 問題根因

現有文件架構存在三個核心斷裂：

1. **spec/usecase 與 worklog 割裂**
   - spec.md 和 usecase 資料夾記錄需求規格
   - worklog 記錄開發過程
   - 兩者之間沒有追蹤連結，無法確認哪些需求已實作並驗證

2. **需求來源無統一入口**
   - 原始需求記載在 spec 上
   - 開發中產生的需求（臨時發現、架構調整、bug 衍生）沒有固定記錄位置
   - 隨著開發推進，spec 逐漸失去參考價值，但新需求也無處可尋

3. **worklog 補完而非即時更新**
   - worklog 與 ticket 沒有連動
   - 實際做法是版本推進時才根據 ticket 內容補完 worklog
   - 違反 worklog「即時記錄開發過程」的設計初衷

### 影響範圍

- `docs/app-requirements-spec.md` - 需求規格書
- `docs/app-use-cases.md` - 用例說明
- `docs/work-logs/` - 所有版本工作日誌
- `docs/work-logs/v*/tickets/` - Ticket 文件
- `.claude/pm-rules/` - PM 流程規則

### 設計靈感來源

高見龍提出的提案驅動架構：

```
需求出現（任何來源）
    |
    v
[提案階段] - 建立 Proposal 文件
    |
    v
[討論階段] - 細化提案內容、確認實作方向和驗證方式
    |
    v
[轉化階段] - 提案 -> 正式 spec/usecase 文件
    |          同時建立 worklog 條目 + 開立 ticket
    v
[實作階段] - 隨實作過程即時更新 worklog 進度
    |
    v
[驗收階段] - 驗證通過 -> 勾選 checklist -> 提案完成
```

### docs/ 四大資料夾架構（需進一步評估）

docs/ 底下應建立四個獨立資料夾，各自服務不同階段的不同角色：

```
docs/
├── proposals/    # 提案 — 開發中的需求參考
├── spec/         # 規格 — 日後維護的參考
├── usecases/     # 用例 — 設計測試及驗收的參考來源
└── work-logs/    # 工作日誌 + tickets — 開發進度掌控
```

| 資料夾 | 階段定位 | 服務對象 | 說明 |
|--------|---------|---------|------|
| proposals/ | 開發期 | 開發者 | 需求的統一入口，對開發有意義但對後人維護無直接意義 |
| spec/ | 維護期 | 維護者 | 提案確認後轉化的正式規格，長期參考文件 |
| usecases/ | 設計/驗收期 | 測試設計者、驗收者 | 設計測試案例和驗收標準的來源 |
| work-logs/ | 開發期 | PM、開發者 | 含 tickets/，掌控開發進度與實作過程 |

### 與現有 todolist 機制的關係（需進一步評估）

- **todolist** — 版本進度控管，記錄臨時性功能或技術債需求
- **提案流程** — 正規需求的入口。todolist 上的項目若要成為正規需求，須走提案流程
- 兩者互補而非替代：todolist 管「做什麼」的短期清單，提案管「為什麼做、怎麼驗收」的完整生命週期

### Checklist 定位與查詢機制（需進一步評估）

提案的 checklist 需要評估放在四層架構的哪一層。核心需求：

- **快速查詢** — 不需要逐一打開文件就能掌握專案進度與開發需求
- **追蹤用途** — 確認每個提案/需求的實作狀態（未開始/進行中/已驗收）

待評估的可能方案：

| 方案 | 位置 | 優點 | 缺點 |
|------|------|------|------|
| A. 放在 proposals/ 各提案內 | proposals/{name}.md | 提案與追蹤合一 | 需逐檔查看，無全域視角 |
| B. 獨立索引檔 | docs/ 根目錄（如 tracking.md） | 一檔掌握全貌 | 與提案分離，可能不同步 |
| C. 結構化資料檔 | docs/tracking.yaml 或類似 | 可程式化查詢、腳本處理 | 需工具支援 |
| D. 混合方案 | 提案內有 checklist + 索引檔彙總 | 兼顧細節與全局 | 維護兩處，同步成本 |

關鍵設計問題：

1. checklist 粒度 — 追蹤到「提案」層級還是「提案內的每個需求項」？
2. 查詢方式 — 純文件翻閱？CLI 指令？腳本自動彙總？
3. 同步機制 — 若 checklist 與實際文件分離，如何確保一致性？

### 核心設計要點

1. **提案（Proposal）作為需求統一入口** — 無論需求來自原始規劃或開發中發現，都先建立提案
2. **提案到 spec 的轉化追蹤** — 記錄需求如何從模糊概念變成具體規格
3. **spec/usecase 的實作 checklist** — 每個需求項目都有追蹤是否已實作並驗證
4. **需求確定即開立 ticket** — 提案轉化為 spec 的同時，記錄到對應版本 worklog 並開立 ticket
5. **worklog 即時更新** — 隨 ticket 執行過程同步更新，而非版本結束後補完
6. **四資料夾各司其職** — 提案是規格文件的一部分但有獨立生命週期，不與 spec 混放
