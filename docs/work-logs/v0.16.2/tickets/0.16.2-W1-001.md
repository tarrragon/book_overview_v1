## 現況分析

### 1. UC 編號修正

Ticket 標題和原始碼註解引用 UC-05，但 UC-05 實際是「雙模式書庫展示系統」。

PROP-002 為對應提案（status: discussing）。

### 2. 既有同步服務架構（6 個模組）

| 模組 | 檔案 | 行數 | 實作狀態 |
|------|------|------|---------|
| CrossDeviceSyncService | cross-device-sync-service.js | ~340 | 骨架完整，整合協調 |

### 3. SyncStrategyProcessor 評估（關鍵缺陷）

**問題**：三個策略方法（priority-based, timestamp-based, conflict-resolution）全部只回傳模擬結果。

缺失的實作：
- priority-based：未實作優先級比較邏輯（應根據資料類型/變更頻率排序同步順序）
- timestamp-based：未實作時間戳比對（應比較 local vs remote 的 updated_at）
- conflict-resolution：未實作實際合併邏輯（應整合 SyncConflictResolver）

### 4. RetryCoordinator 評估（可用但需增強）

已有功能：指數退避、jitter、錯誤分類（NETWORK/DATA_CONFLICT/SERVER_ERROR/AUTHORIZATION）、最大重試限制

缺失功能：
- 斷路器（Circuit Breaker）：連續失敗時應暫停重試避免雪崩
- 重試狀態持久化：Service Worker 被終止後重試狀態丟失
- 自適應策略：未根據歷史成功率調整重試參數

### 5. 45 個 E2E 測試基準

位於 tests/e2e/workflows/cross-device-sync.test.js（2,181 行），5 大類：
- A. 完整同步流程（8 個）
- B. 資料完整性驗證（10 個）
- C. 效能與可擴展性（8 個）
- D. 錯誤處理與恢復（12 個）

這些測試使用 mock 實作通過，可作為未來實作的驗收基準。

### 6. Chrome Storage Sync API 整合評估

PROP-002 範圍包含 Chrome Storage Sync API 整合。關鍵限制：
- chrome.storage.sync 空間限制：100KB 總量，8KB 每個 key

- 書籍資料同步需要：匯出/匯入機制（已有 ExportService/ImportService）

## 實作規劃建議

### 同步策略：雙層架構

**第一層：Chrome Storage Sync（輕量元資料）**
- 同步內容：使用者設定、最後同步時間戳、書庫版本號
- 技術：chrome.storage.sync API（100KB 限制內）
- 優點：Chrome 帳號自動同步，零伺服器成本

**第二層：匯出/匯入機制（完整書庫資料）**
- 同步內容：完整書籍清單、分類、標註等
- 技術：既有 ExportService/ImportService + 使用者手動觸發
- 格式：JSON 匯出，含版本標記和增量差異

### 建議實作順序（4 個 IMP Ticket）

**IMP-1：SyncStrategyProcessor 實作**（P0，約 2-3 天）
- 實作 priority-based 策略：按資料類型（設定 > 書籍 > 標註）排序
- 實作 timestamp-based 策略：比對 local/remote updated_at
- 整合 SyncConflictResolver 到 conflict-resolution 策略
- 驗收：現有單元測試從 mock 切換為真實邏輯後仍通過

**IMP-2：RetryCoordinator 增強**（P1，約 1-2 天）
- 新增斷路器模式（連續失敗 N 次後暫停）
- 新增 chrome.storage.local 持久化重試狀態
- 驗收：Service Worker 重啟後能恢復未完成的重試

**IMP-3：Chrome Storage Sync 整合**（P1，約 2 天）
- 新增 SyncMetadataManager：管理 storage.sync 中的元資料
- 監聽 chrome.storage.onChanged 事件觸發同步流程
- 處理 QUOTA_BYTES_PER_ITEM 限制（8KB/key）
- 驗收：兩個 Chrome 瀏覽器實例間設定自動同步

**IMP-4：端到端同步流程整合**（P2，約 2-3 天）
- 串接所有服務：Orchestrator → Strategy → Progress → Conflict
- 實作 UC-07 主要成功場景和 5 個替代流程
- 驗收：45 個 E2E 測試從 mock 切換為真實服務後通過

### 風險與限制

| 風險 | 影響 | 緩解方案 |
|------|------|---------|
| storage.sync 100KB 限制 | 無法同步完整書庫 | 只同步元資料，書庫用匯出/匯入 |
| Service Worker 生命週期 | 長時間同步可能被中斷 | 斷點續傳 + 狀態持久化（IMP-2） |
| 衝突解決複雜度 | 使用者難以理解衝突 | Last-Write-Wins 為預設，保守策略優先 |
| 雲端同步不在範圍 | 使用者期望自動同步 | 明確在 UI 說明「同裝置/同帳號」限制 |

### 不在本版本的項目（v2.0+ 延後）

- 雲端伺服器同步
- 跨瀏覽器同步（Firefox/Edge）
- 自動備份恢復（PROP-003）
