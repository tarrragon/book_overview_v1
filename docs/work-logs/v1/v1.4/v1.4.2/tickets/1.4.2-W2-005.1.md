### 修復內容（src/content/adapters/readmoo-adapter.js）

**F1（核心）—— 提取前整頁捲過全部書籍觸發封面真圖替換**

新增 `_scrollThroughAllItemsForCovers(coverScrollOptions)` async 方法：以整頁
`window.scrollTo` 從頂（0）漸進捲到底，每段捲動量為 viewport 高度的
`COVER_SCROLL_SEGMENT_RATIO`（0.8）倍（保留重疊避免遺漏），每段後等待
`COVER_SCROLL_SEGMENT_WAIT_MS`（300ms）讓進入 viewport 的封面載入真圖。
停止條件：捲到底（scrollY >= scrollHeight - viewportHeight）/ scrollY 停滯不增
（容器拒絕捲動，防空轉）/ 達 `COVER_SCROLL_MAX_SEGMENTS`（200）上限。
永遠 resolve（window 不可用 / scrollTo 例外時降級回傳，不阻斷提取）。

在 `loadAllBooksLazy` 兩個「item 已全載入」出口（`already_complete` 與主迴圈
結束）於 `finalize` 前呼叫此方法，捲過後重新 `_measureBooks` 補入捲動中新渲染
的 item（unique Set 天然去重）。`container_not_found` / `error` 路徑不呼叫
（無 item 可觸發封面）。

**根因**：Readmoo 封面是 viewport 可見性驅動替換（非標準 lazy-load）——viewport
外 `.library-item` 的 cover-img src 為 placeholder（`/images/openbook.png`），
進入過 viewport 後 React 才換真實 CDN URL。即使 965 本 item 全載入，提取當下
viewport 外約 90 本封面仍是 placeholder（實機實驗 5）。

**F3（防禦）—— _scrollStep 補整頁捲動**

新增 `_scrollWindowToBottom()`，於 `_scrollStep` 動作 1 後呼叫整頁
`window.scrollTo(0, scrollHeight)`。findScrollContainer 實機恆先命中
`#react-container`（不可捲，scrollHeight==clientHeight，實驗 3），原僅靠
`container.scrollTop` 無效；整頁捲動使虛擬捲動觸發路徑對任一策略皆有效。

**輔助**：新增 `getWindow()` helper（SW 無 window 退回 globalThis）、
`_waitMs(ms)` helper（使每段等待可被 fake timer 推進 / 測試注入）、
`_readScrollY(win, scrollingEl)` 容錯 helper、`COVER_SCROLL_FAILED` 訊息鍵。

**F2**：已於父 ticket 實驗 6 驗證（新 build 提取 965 本，W1-040 已生效），
本 IMP 確保用新 `build:dev` 即可，未改書數邏輯。

### 實機驗證鐵證（PM 接手，2026-06-24）

reload_extension 載入新 build 後三次量測，root cause 鎖定：

| 量測 | 起點 DOM | storage cover 結果 |
|------|---------|-------------------|
| 首次提取（reload 回 96 本） | placeholder 78 | httpCover 309 / **openbook 656** / empty 0（未生效，與修復前同） |
| 提取後查 library DOM | — | **placeholder 0**（_scrollThroughAllItemsForCovers 捲動確實執行，scrollY 到底） |
| 第二次提取（不 reload，DOM 已全真圖） | placeholder 0 | **httpCover 965 / openbook 0 / empty 0**（完全歸零） |

### Root cause（確認）
F1 方向正確（整頁捲動有效，DOM 全變真圖），parseBookElement cover 邏輯正確（DOM 真圖時讀到 965 真圖）。缺陷在**等待策略**：`_scrollThroughAllItemsForCovers` 用 window.scrollTo 瞬間設定 scrollY 到底，但 React 把 placeholder `/images/openbook.png` 換成真實 CDN src 是**異步 render**；每段僅固定等 300ms 且捲完無收斂條件，提取（getBookElements + parseBookElement）緊接著讀取時真圖尚未 render → 讀到 656 placeholder（稍後 render 完成 DOM 才變真圖，故事後查 DOM 為 0）。

### 修復方向（精準）
`_scrollThroughAllItemsForCovers` 捲動後需**等待真圖實際 render 完成**再返回，而非固定短延遲。建議收斂條件：每段捲動後輪詢「該段/全頁 .cover-img src 含 openbook 的數量歸零」或 img.complete，加總體 timeout 上限。驗證 acceptance：首次提取（DOM 從初始）即達 openbook placeholder 0 / httpCover 965。
