## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
Parent 1.4.0-W2-013 的多視角審查（linux/Plan）對治本層提出兩個未驗證假設，gate 住 S2/S4/S5 的 enforcement 形態決策：
- U1：是否存在「合法的使用者自訂 local hook」場景？若存在，sync 端 enforcement 需可靠區分框架 vs 使用者 hook，否則 false positive 誤擋使用者配置。
- U2：CC settings 是否支援 settings.local.json 對 settings.json（base）已註冊 hook 的「停用 / null 覆寫」？此為審查提出的結構方向 A（local 只存停用覆寫、不存框架 hook 正本）的技術前提。
本 spike 以本專案 git 史 + sync_exclude_manifest 靜態分析（U1）+ CC 官方文件（U2）驗證兩假設，輸出方向 A/B 可行性供 parent S2/S4/S5 定向。

## 重現實驗結果
### 實驗方法

U1：grep/git log 掃描本專案 settings.local.json 歷史、analyze sync_exclude_manifest.py 的區分邏輯、評估框架 vs 使用者 hook 的區別依據
U2：WebFetch code.claude.com/docs 官方文件，查證 settings.local.json 對 settings.json 的 hooks 合併規則與 null/false 停用語法支援

### 實驗執行

**U1：本專案 settings.local.json 掃描**

1. git log 歷史檢查：發現歷史上曾含 hooks 區塊（handoff-auto-resume-stop-hook、PreToolUse:Agent），已透過 PC-148 清理
2. 當前狀態：settings.local.json 無 hooks 區塊
3. sync_exclude_manifest 分析：
   - settings.local.json 本身歸 LOCAL_ONLY_PATTERNS（B 型），不 sync
   - hooks/ 目錄**不在黑名單**（should_exclude('hooks/xxx.py') → False）
   - → hooks/ 下所有檔（框架或使用者）都會 sync 同步
4. 區分依據：唯一邊界 = 註冊位置（settings.json vs settings.local.json）

**U2：Claude Code 官方文件驗證**

1. Settings 優先級：User < Project < Local < Cmd < Managed
2. Hooks 合併規則：**跨作用域累積**（非覆蓋）
3. Null 停用機制：
   - 無 `null` / `false` / `override` 支援（靜態分析 + WebFetch 確認）
   - 唯一機制：`disableAllHooks: true`（全局）

### 實驗發現

**F1：框架 hook 進 local 歷史（已修復）**
- handoff-auto-resume-stop-hook 曾雙重註冊（PC-148），已刪 local 副本
- 當前無內容

**F2：sync 無法區分框架 vs 使用者 hook**
- 框架與使用者 hook 檔都在 sync 範圍（hooks/ 無黑名單）
- 區分邊界只能靠註冊位置，不夠可靠

**F3：CC hooks 無細粒度停用**
- 合併非覆蓋，無 null 機制
- 全局 `disableAllHooks` 無法精細控制

**驗證結論：**
- U1 ✅：存在但已清理；架構應禁止（以註冊位置為界）
- U2 ❌：不支援 null/override 停用 → 方向 A 不可行

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 兩個假設的驗證結論

| 假設 | 結果 | 依據 |
|------|------|------|
| **U1：存在合法使用者 local hook 場景** | ✅ 存在，但已清理；架構上應禁止 | git log（F1）+ 現狀掃描（F2） |
| **U2：CC 支援 local 停用 base hook** | ❌ **不支援** | CC 官方文件 + 合併規則分析（F3） |

### 對方向 A/B 可行性的影響

**方向 A（local 只存停用覆寫、不存框架 hook 正本）**

評估：❌ **技術上不可行**
- 前提：local 能以 `null`/override 停用 project-level hooks
- 事實：CC 不支援細粒度停用（F3），只有全局 `disableAllHooks`
- 結論：無法在 local 層級精細控制框架 hook，此方向不可行

**方向 B（hook 真值移出 settings、改由 sync 範圍內 manifest + loader 驅動）**

評估：✅ **可行**（但成本高）
- 邏輯：hook 註冊從 settings.json/local.json 移至 `.claude/hooks-manifest.yaml`（sync 範圍內）
- 優勢：sync-pull 能同時更新 manifest 與檔案，避免幽靈 hook
- 缺點：引入動態 hook 載入機制，架構複雜度上升

**原 S2：PreToolUse Edit guard**

評估：⚠️ **可執行但有盲點**
- 優勢：簡單快速，防止新增框架 hook 進 local
- 盲點：無法回收 local 存量；只擋 Edit，不擋從遠端 sync 流入（per-project 無法統一執行）

### 實踐建議

基於 U1/U2 驗證結果，parent ticket 修訂方向的微調：

1. **S1（去重）**：保留；消滅當前實證地雷（handoff-auto-resume 已去重）
2. **S2（enforcement 移至 sync 端）**：技術可行，但須補加細節：
   - sync-pull 偵測並拒絕框架 hook 留在 local 的機制（比對 command 路徑與框架 hook 清單）
   - 需驗證區分的精準性（防止 false positive 阻擋使用者正當 local 配置）
3. **S4（prune）**：保留；低風險（只刪幽靈 entry，無 false positive）
4. **S5（migration manifest）**：延後；待方向 B 評估必要性後決策

### 對 parent 1.4.0-W2-013 審查結論的驗證

修訂審查的「治本層定位錯誤」（linux 視角）：
