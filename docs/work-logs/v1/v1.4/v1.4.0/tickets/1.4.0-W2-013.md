## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
（PM 前台框架，供下個 session 走 WRAP 的起點，非結論）

### 根本張力
框架 hook 註冊落在 settings.local.json（per-project、不進 git、sync-pull 刻意不碰）。框架版本遷移（2.0 hook 搬移/刪除）時 sync-pull 能更新 settings.json，卻無法觸及 settings.local.json → stale 註冊成幽靈 hook → PreToolUse hook error 阻擋所有 Agent 派發（非中止 session 但實質癱瘓派發）。

### 本專案具體發現（休眠地雷）
- handoff-auto-resume-stop-hook（ticket skill = 框架資產）在 settings.json 與 settings.local.json **雙重註冊**（PC-148 反模式）。
- 現在路徑有效所以無症狀；但框架若再遷移該 hook，sync 更新 settings.json、local 不動 → 變幽靈。
- 即「目前無 #11 症狀 ≠ 無 #11 風險」。

### 三層可組合設計空間（WRAP W 步至少從此擴增）
1. 預防（治本）：強制原則「框架 hook 註冊只准在 settings.json」，settings.local.json 僅 permissions/env/mcp/outputStyle。加 check 偵測 local 內 hooks 區塊並擋。對應 #11 建議 3 + PC-148。
2. 修復（安全網）：SessionStart HookCheck 已偵測幽靈（WARNING），擴充為自動 prune settings.local.json 中指向不存在檔的 entry（或一鍵修復命令）。對應 #11 建議 1。風險：自動改用戶 settings 須只刪「檔案不存在」entry + 完整 log。
3. 遷移工具（一般化）：框架 breaking hook 搬移時附 migration manifest（舊→新路徑 map）+ post-sync remediation step（沿用 W2-004 verify_hook_imports 的 post-sync hook 模式），同時改寫 settings.json 與 settings.local.json。直接回應「版本遷移攜帶 settings 變更」需求。

### WRAP 須補的視角
- R reality-test：方案是否真能繞過「sync 不碰 local」的設計前提？自動改 local 是否破壞 local 的「使用者主權」語意？
- A：預防層需要既有 consumer 一次性遷移既有 local hook 註冊的成本。
- P premortem：自動 prune 誤刪使用者刻意的 local 註冊（若存在合法 local hook 場景）。

### 關聯
framework issue #11（tarrragon/claude）；PC-148（hook 雙重註冊）；W2-004（post-sync hook 模式可複用於遷移工具層）。
