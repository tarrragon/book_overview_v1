# Execution Log

## Task Summary

分析 框架版本遷移需改 consumer settings（含 settings.local.json 幽靈 hook）的應對機制（framework issue #11）

---

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
（PM 前台框架，供下個 session 走 WRAP 的起點，非結論）

### 根本張力
框架 hook 註冊落在 settings.local.json（per-project、不進 git、sync-pull 刻意不碰）。框架版本遷移（2.0 hook 搬移/刪除）時 sync-pull 能更新 settings.json，卻無法觸及 settings.local.json → stale 註冊成幽靈 hook → PreToolUse hook error 阻擋所有 Agent 派發（非中止 session 但實質癱瘓派發）。

### 本專案具體發現（休眠地雷）
- handoff-auto-resume-stop-hook（ticket skill = 框架資產）在 settings.json 與 settings.local.json **雙重註冊**（PC-148 反模式）。
- 現在路徑有效所以無症狀；但框架若再遷移該 hook，sync 更新 settings.json、local 不動 → 變幽靈。
- 即「目前無 #11 症狀 ≠ 無 #11 風險」。

### 三層可組合設計空間（WRAP W 步至少從此擴增）
1. 預防（治本）：強制原則「框架 hook 註冊只准在 settings.json」，settings.local.json 僅 permissions/env/mcp/outputStyle。加 check 偵測 local 內 hooks 區塊並擋。對應 #11 建議 3 + PC-148。
2. 修復（安全網）：SessionStart HookCheck 已偵測幽靈（WARNING），擴充為自動 prune settings.local.json 中指向不存在檔的 entry（或一鍵修復命令）。對應 #11 建議 1。風險：自動改用戶 settings 須只刪「檔案不存在」entry + 完整 log。
3. 遷移工具（一般化）：框架 breaking hook 搬移時附 migration manifest（舊→新路徑 map）+ post-sync remediation step（沿用 W2-004 verify_hook_imports 的 post-sync hook 模式），同時改寫 settings.json 與 settings.local.json。直接回應「版本遷移攜帶 settings 變更」需求。

### WRAP 須補的視角
- R reality-test：方案是否真能繞過「sync 不碰 local」的設計前提？自動改 local 是否破壞 local 的「使用者主權」語意？
- A：預防層需要既有 consumer 一次性遷移既有 local hook 註冊的成本。
- P premortem：自動 prune 誤刪使用者刻意的 local 註冊（若存在合法 local hook 場景）。

### 關聯
framework issue #11（tarrragon/claude）；PC-148（hook 雙重註冊）；W2-004（post-sync hook 模式可複用於遷移工具層）。

## Solution

### WRAP 分析（完整模式，PM 前台）

### W — 擴增選項（涵蓋新增工具 / 改造既有 / 零工具三類）

| 方向 | 性質 | 做法 |
|------|------|------|
| 1 預防（治本） | 改造既有 | 把現有 SessionStart WARNING 升級為 PreToolUse guard：偵測 Edit/Write 對 settings.local.json 注入「框架 hook」即 block。零新檔，擴充既有 hook-completeness lib。 |
| 2 修復（安全網） | 改造既有 | 既有 find_phantom WARNING 保留，加一鍵 prune 命令（opt-in，只刪「command 指向不存在檔」的 entry）。 |
| 3 遷移工具（一般化） | 新增工具 | 框架 breaking hook 搬移附 migration manifest（舊→新路徑 map）+ post-sync remediation step（複用 W2-004 post-sync hook 模式），同時改寫 settings.json 與 settings.local.json。 |
| 4 純原則（零工具） | 零工具 | 僅強化規則「框架 hook 只准註冊 settings.json」，靠既有 WARNING + 人工修。 |

反向思考（Consider the Opposite）：票框架是「版本遷移如何攜帶 settings 變更」（修復導向）。反面問題是「為何框架 hook 註冊會落在 local」（預防導向）。若預防成立（local 永不含框架 hook），則「讓 sync 能改 local」的需求大半消失。→ 重新定義問題：治本在「阻止框架 hook 進 local」，而非「讓 sync 能改 local」。

### R — 現實檢驗
- 基本率：跨 per-project 檔案的框架遷移常見解法 = (a) 偵測+人工修 (b) migration script (c) 強制單一來源。三類本案皆有對應方向。
- 張力檢驗：方向 2/3「自動改 local」與 sync「刻意不碰 local（使用者主權）」設計前提衝突。方向 1/4 不改 local 既有內容、只擋「框架 hook 進 local」，不違反主權——前提是能區分框架 vs 使用者 hook。
- 關鍵未知（對應 U1）：區分依據 = hook command 路徑是否落在 sync 同步範圍（.claude/skills/ 框架 skill 或框架 hook 清單）。使用者自訂 hook 不在 sync 範圍 → 放行。
- Steelman：方向 3 最徹底回應票面需求，但成本高（每次 breaking 都寫 manifest）且為未來假想場景；方向 1 預防 + 既有偵測已覆蓋當前實證地雷（F1）。

### A — 拉開距離 / 機會成本
- 方向 1：低成本（擴充既有 hook）、高槓桿（從根斷絕），需既有 consumer 一次性遷移既有 local 註冊（本專案 handoff-auto-resume）。
- 方向 2：中成本，安全網價值，premortem 風險為誤刪。
- 方向 3：高成本，服務假想未來場景，機會成本高。
- 悖論檢查（家長主義）：prune/block 改動使用者 settings 須 opt-in + 只動「檔案不存在」或「框架 hook」entry + 完整 log + 預留覆蓋；不自動改使用者刻意的 local hook。
- 優先排序：治本（預防）> 安全網（prune）> 假想（migration）。

### P — 準備好犯錯（premortem）
- 方向 1 失敗主因：(a) 無法可靠區分框架 vs 使用者 hook → 誤擋使用者合法 local hook；(b) 既有 consumer 的 handoff-auto-resume 未先遷移就上 guard → 自我阻擋。
- 緩解：上 guard 前先遷移本專案 local 註冊（settings.local.json 已有 settings.json 副本，去重安全）；guard 只擋 sync 範圍框架 hook，使用者 hook 放行；先 WARNING 觀察期再轉 block（漸進回退）。

### 方向結論（三層取捨）
- 治本層【採】：升級既有偵測為 PreToolUse block（區分框架 hook），並先遷移本專案既有 local 註冊去重。
- 安全網層【採】：保留既有 find_phantom WARNING，加 opt-in 一鍵 prune 命令（只刪幽靈 entry）。
- 遷移工具層【延後/trigger-bound】：migration manifest 服務假想未來場景，建 trigger-bound ticket，待真實 breaking hook 遷移事件觸發。

回報 framework issue #11：採「預防（治本）+ 偵測安全網」雙層，對應 #11 建議 3（單一來源）+ 建議 1（自動偵測/修復）；建議 2（migration 工具）延後至真實遷移觸發。

### 多視角審查修訂（parallel-evaluation 情境 F：thyme / Plan / linux，--skip-basil）

三視角高度收斂，方向結論的「保守雙層」可執行性未翻盤，但治本層定位被實質修正。審查全文：scratchpad/review-{thyme,plan,linux}.md。

### 收斂發現
1. 【治本層定位錯誤｜linux + Plan】框架 hook 進 local 的唯一自動化來源是 sync/安裝流程，非手動 Edit。S2 PreToolUse Edit guard 攔錯攻擊面（premortem 自承「consumer 未遷移就上 guard → 自我阻擋」），且 guard 只擋未來寫入、不回收存量、per-project 無法覆蓋框架層所有 consumer。enforcement 應移至 sync/安裝端（拒絕把框架 hook 留在 local）。
2. 【U1 未驗證即採用｜三視角一致】S2/S4 地基假設「能區分框架 vs 使用者 hook」未證實。thyme：判定可用 `sync_exclude_manifest.should_exclude` 程式化，但「在 sync 範圍」≠「框架資產」——使用者放在 .claude/hooks/ 的自訂 hook 同樣 should_exclude==False，會被誤判 block（false positive）。須先答 U1 再決定 S2/S4 是否存在。
3. 【設計空間未窮盡｜Plan】W 四方向為「補救強度」一維光譜，漏正交方向 A（local 只存停用/覆寫、不存框架 hook 正本，sync 改 settings.json 即生效）、B（hook 真值移出 settings 改由 sync 範圍內 manifest + loader 驅動）。selective-sync 子集方向應顯式否決（違反使用者主權前提）而非隱性排除。新增最大 reality-test 缺口 U2：CC settings 是否支援 local 對 base hook 的停用/null 覆寫語意（方向 A 前置）。
4. 【S5 觸發點太晚｜Plan】issue #11 場景正是「遷移當下 local 已碰不到」，等遷移事件後才建工具救不了該次。觸發點應前移為「框架宣告遷移前（pre-migration gate）」，理由改為「待 A/B 結構決策後評估必要性」。F1 是已上膛的遷移債，非純假想。

### 共識安全部分（未翻盤）
- S1（去重）= 真正的 fix，消滅當前唯一實證地雷（F1），安全（settings.json 已有副本）。
- S3（DOC 原則固化）剛好，但方向改指向「sync/安裝端 SSOT」而非 Edit guard。
- 既有 SessionStart WARNING（find_local_hook_registrations / find_phantom_registrations）保留為免費安全網。
- S4 prune 安全（只判「檔在/不在」，繞過 U1），但應併入既有 hook-completeness `--fix` flag，不開新命令。

### 修訂後方向結論
- 治本層【修訂】：enforcement 移至 sync/安裝端（sync-pull 偵測並拒絕框架 hook 留在 local），取代原 S2 PreToolUse Edit guard。
- 安全網層【保留】：既有 WARNING + S4 prune 併入 hook-completeness `--fix`（opt-in，只刪幽靈 entry）。
- 結構治本層【新增/reality-test gate】：方向 A/B 須先驗證 U2（CC 是否支援 local 停用覆寫 / 動態 hook 載入）才能評估。
- 遷移工具層【延後】：保留 trigger-bound，觸發點前移為 pre-migration gate，理由改「待 A/B 結構決策後評估」。
