## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

CC 2.1.197 發布 Claude Sonnet 5 並設為預設模型（原生 1M context，促銷價 $2/$10 per Mtok 至 2026-08-31）。本專案 31 個 agent 定義的 `model:` 設定使用別名（sonnet/haiku）、inherit 或版本釘選，預設模型變更後需確認各設定的實際解析結果是否仍符合分工設計。

### 影響範圍

`.claude/agents/*.md` 共 31 檔的 `model:` frontmatter：
- `sonnet` 別名 12 個（thyme 系列、parsley、fennel、cinnamon、basil-hook/event、coriander、sumac、impeccable、john-carmack[DEPRECATED]）
- `inherit` 13 個（rosemary、saffron 除外的審查/分析類多數）
- `haiku` 3 個（mint、memory-network-builder[DEPRECATED]、template）
- `claude-opus-4-6[1m]` 釘選 2 個（saffron-system-analyst、bay-quality-auditor）

### 相關 Error Pattern

無直接對應。相關 memory 教訓：feedback_agent_capability_requires_experiment（代理人 model 變更前必先實驗驗證；邏輯降級必回滾）。

## 重現實驗結果

### 實驗方法

盤點式驗證：`grep -H "^model:" .claude/agents/*.md | sort | uniq -c` 取得全量 model 設定分布，對照 CC 2.1.197 release note 的預設模型變更行為。

### 實驗執行

- grep 結果：sonnet 別名 12 / inherit 13 / haiku 3 / claude-opus-4-6[1m] 釘選 2 / 無 model 欄位 2（元文件）
- 別名解析行為（release notes 佐證）：`sonnet` 別名跟隨平台最新 Sonnet（2.1.197 起為 Sonnet 5）；`inherit` 跟隨主線程 session 模型
- 釘選行為：版本釘選不受預設變更影響（2.1.153 `/model` 行為、2.1.174 釘選標籤修復佐證）

### 實驗發現

- 已驗證事實：12 個 sonnet 別名 agent 自動升級至 Sonnet 5，屬平台行為，無需框架動作
- 已驗證事實：13 個 inherit agent 跟隨主線程模型（用戶已切換 Fable 5），亦無需動作
- 已驗證事實：2 個 opus-4-6[1m] 釘選（saffron、bay-quality-auditor）不受影響，但釘選版本已落後（Opus 4.8 / Fable 5 已發布）
- 未驗證假設：`opus` 別名是否保留 1M context（原釘選的 `[1m]` 後綴是刻意的 1M 保證）——未經實測，是本次不升級釘選的關鍵理由

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### WRAP 分析

### W（擴增選項）

1. 全面維持現狀（別名自動升級 + 釘選不動）（採用）
2. 釘選升級至 `opus` 別名（跟隨最新 Opus）
3. 釘選改 `claude-opus-4-8`（換新釘選）
4. 釘選改 `inherit`（跟隨主線程 Fable 5）

### R（現實檢驗）

- 別名升級是平台自動行為，選項 1 對 27 個非釘選 agent 而言是唯一選項（無動作可做）
- 釘選升級三方案（2/3/4）皆為模型變更，依 memory 教訓需實驗驗證後才可執行；`opus` 別名是否保留 1M context 未實測，直接改動有失去 1M 保證的風險
- Sonnet 5 促銷價（至 8/31）降低 sonnet 別名 agent 的執行成本，自動受益

### A（拉開距離）

- 維持現狀的成本：saffron/bay 用 4-6 而非 4.8/Fable，審查品質可能略遜於最新模型，但 4-6[1m] 仍是可用的高階模型，差距為漸進非斷崖
- 貿然升級的成本：若 `opus` 別名不含 1M，saffron（系統分析需大 context）能力直接受損且難察覺

### P（絆腳索）

- 絆腳索 1：saffron 或 bay 產出出現 context 不足症狀（截斷、遺漏大範圍檔案）→ 重評釘選
- 絆腳索 2：CC 宣布 opus-4-6 退役（比照 2.1.68 Opus 4/4.1 移除前例）→ 強制升級，屆時先實驗驗證 `opus` 別名 1M 行為

### 結論

**維持現狀（已決策，狀態 a）**。12 個 sonnet 別名自動升級 Sonnet 5、13 個 inherit 跟隨主線程，皆屬平台行為無需動作；2 個 opus-4-6[1m] 釘選保留，因 `opus` 別名的 1M context 行為未經驗證，依 memory 教訓（model 變更需實驗驗證）不做未驗證變更。未來升級由絆腳索觸發，屆時另建 ticket。
