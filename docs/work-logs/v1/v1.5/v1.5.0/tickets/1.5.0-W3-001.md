### Issue 標題

行為記憶三層治理：error-patterns 去重死角、cbm 對 .claude/ 不索引、CC 原生 Auto Memory 未納管

### Issue 內文（存成 body-file）

### 摘要

專案現有三層「行為記憶」並存，但職責邊界未顯式界定：
1. error-patterns/（PC/IMP，手刻 + hook 路由 + grep 檢索）
2. codebase-memory-mcp（cbm，向量+BM25 混合索引，已安裝）
3. Claude Code 原生 Auto Memory（模型自主寫入 MEMORY.md，2026 年中推出）

核心待解三件事：error-patterns 的語意去重死角；本可用的向量引擎 cbm 被上游 hardcoded skip 擋在 .claude/ 外（W1-005 監測中）；CC 原生 Auto Memory 引入一條未受治理的第二知識流。

### 前因（為何浮現）

- error-patterns/ 已累積 341 篇（PC 197 + IMP 87），跨 session、跨代理人、跨中英書寫。
- 去重機制 = /error-pattern query「建之前先查詢」= grep-class 字串比對。
- 同根異名漂移實例：confabulation 根因家族橫跨 confabulated / fabrication / conflate / 幻覺 / 虛構 / 腦補（PC-166、PC-147、PC-111、PC-V1-003、PC-V1-010 等）。grep 對這些彼此零字串重疊。

### 三層現況

| 層 | 寫入者 | 檢索 | 確定性 | 去重 / 審查 |
|----|--------|------|--------|-------------|
| error-patterns（PC/IMP） | 人 / 代理人，經 git review | grep + 人工路由 | 高（hook 強制執法） | grep 查重 + PR 審查 |
| cbm | 自動索引 | search_graph 語意（向量+BM25 11-signal） | 中 | 無，且對 .claude/ 不索引 |
| CC 原生 Auto Memory | 模型自主 | 注入 + 懶載 | 低（模型自判何時讀） | 完全沒有 |

### 核心問題

1. error-patterns 去重死角（grep 漏接同根異名）：撰寫新 PC 前 query 是 grep-class。當新發現用詞與既有 PC 零字串重疊（如查「編造」對 confabulated / fabrication），grep 回 no matches，模型據此判定「未記錄過」並建新 PC。這是 silent failure，後果是近似重複 PC，使同一教訓碎裂進兩篇，且 hook 路由只指向其一，另一篇防護失聯——比完全不去重更糟，因為看起來像盡責做完了。為何是唯一死角：系統其餘部分靠「作者撰寫當下預先連結」（stub→reference）繞過搜尋；去重是唯一「關係在撰寫當下還不存在」的場景，沒有前人路由可循。

2. 向量引擎已有，卻被上游牆隔在 .claude/ 外：cbm 即向量+BM25 混合引擎（11-signal），已安裝在跑。但 cbm v0.6.1 對 .claude/ 存在 hardcoded skip（W6-001.4 六步實驗證實，include_hidden + force_reindex + 清空 exclude 全部蓋不掉）。屬上游行為，已由 W1-005 監測上游修復。

3. CC 原生 Auto Memory 未納管：非確定性、無去重、無 review gate、不在 hook 路由網內。技術上與 hook + error-patterns 不衝突（hook 永遠凌駕 context memory），但它是未受治理的第二知識流，會用模型措辭再生未審近似條目，放大第 1 點碰撞面積，侵蝕 curation 紀律。（查證：此專案 ~/.claude/.../memory/ 目前未落地 MEMORY.md，故仍是 setting 級可主動決定的開關。）

### 結論

問題不是「要不要加向量」——向量引擎已具備，瓶頸是 cbm 對 .claude/ 的上游 skip。乾淨路徑是解封 cbm，而非新建第四系統。三層需顯式界定職責邊界：error-patterns + hook 維持唯一 canonical 受執法脊椎；cbm 作為語意去重路徑；Auto Memory 必須主動決定納管範圍。

### 要面對的決定 / 選項（彼此獨立，可組合）

- 選項 A — 同義詞 map（過渡止血，W1-005 解封前）：對最高頻根因家族維護小對照表，把 /error-pattern query 從精確 grep 升級成 multi-term OR。零新系統。
- 選項 B — cbm 解封路徑：推 / 等 W1-005 上游修復；解封後把 .claude/error-patterns/ 註冊成 cbm project，query 升級語意召回。
- 選項 C — Auto Memory 治理邊界：決定 CC 原生 Auto Memory 關閉或限縮至「用完即丟的便利知識」，明文禁止它成為第二個 error-pattern 家。
