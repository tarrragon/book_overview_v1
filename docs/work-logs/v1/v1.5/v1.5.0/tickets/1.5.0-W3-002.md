### 背景

用戶在框架外的 blog 專案使用 Claude Code 原生 Workflow tool 處理 831 個 markdown lint warning（MD036/MD040/front-matter/POS/REF），體驗到 Workflow 的批量平行處理優勢後，要求評估是否可融入本框架。

### Workflow tool 核心能力

| 能力 | 說明 |
|------|------|
| `pipeline(items, stage1, stage2, ...)` | 每項獨立流經所有階段，無 barrier |
| `parallel(thunks)` | 並行執行，全部完成才返回（barrier） |
| `phase(title)` | 進度分組，`/workflows` 即時可視 |
| `agent(prompt, opts)` | 派發子代理人，支援 `schema`（結構化輸出）、`agentType`（自定義代理人型別）、`isolation: 'worktree'` |
| `budget.remaining()` | Token 預算感知，動態控制深度 |
| Resume | 相同 script+args 可 resume，已完成 agent 走快取 |
| 並行上限 | min(16, cores-2) 自動排隊，單 workflow 最多 1000 agent |

**關鍵橋接點**：`opts.agentType` 可指定框架自定義 agent type（如 `mint-format-specialist`、`thyme-python-developer`），保留 agent 專業知識同時享受 Workflow 編排能力。

### 初步比對（四機制）

| 維度 | Workflow | Agent 單派發 | bulk-evaluate | parallel-evaluation |
|------|---------|-------------|---------------|-------------------|
| 控制流 | 確定性腳本（JS） | PM 模型驅動 | PM 模型驅動 | PM 模型驅動 |
| 並行原語 | pipeline/parallel | 多 Agent() 同訊息 | 多 Agent() 同訊息 | 多 Agent() 同訊息 |
| 結果回收 | 程式化（JSON schema） | PM 讀 transcript | 寫入子 ticket | PM 彙整 |
| Ticket 追蹤 | 無 | 有 | 有（子 ticket） | 有（findings→ticket） |
| Agent 型別 | 匿名/agentType 指定 | 框架定義 | 框架定義 | 固定三人組+常駐 |
| Resume | 有 | 無 | 無 | 無 |
| Budget 感知 | 有 | 無 | 有（context 公式） | 有（token 預算） |

### 初步識別的場景邊界

### 最適用場景
1. 批量機械修（lint fix、format、front-matter 補全）— 同質處理、無需判斷
2. 批量掃描+聚合（dead code、i18n 缺漏）— fan-out + dedup + 一次性建 ticket
3. 多視角審查 Find 階段 — parallel + schema 直接收結構化 findings
4. Discovery/Audit sweep — loop-until-dry + budget 感知

### 不適用場景
1. TDD Phase 流程 — 每階段需 PM 判斷、驗收
2. 單一複雜 ticket 實作 — 需完整 context bundle + agent 知識
3. 需要 ticket lifecycle 的任務 — Workflow 無 ticket 追蹤
4. 需 PM 中途介入決策 — Workflow 是確定性腳本，無 AskUser

### WRAP 三問

**W（候選方案）**：
- A: 全面整合 — Workflow 成為框架內 dispatch 的一級選項，PM 決策樹新增路由
- B: 有限整合 — 只在特定 Skill 內部使用（parallel-evaluation Phase 2、bulk-evaluate 輕量版）
- C: 不整合 — 保持現狀，Workflow 作為框架外的獨立工具使用

**A（機會成本）**：整合工作會佔用框架維護資源，但不擠壓產品開發（v1.5.0 已完成）

**P（最可能失敗原因）**：Workflow 的確定性腳本與 PM 模型驅動決策的邊界劃分不清，導致引入後反而增加 PM 認知負擔（「何時用 Workflow vs 何時用 Agent」的決策成本）
