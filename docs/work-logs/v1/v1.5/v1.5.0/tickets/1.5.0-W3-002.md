用戶在框架外的 blog 專案使用 Claude Code 原生 Workflow tool 處理 831 個 markdown lint warning（MD036/MD040/front-matter/POS/REF），體驗到 Workflow 的批量平行處理優勢後，要求評估是否可融入本框架。

### Workflow tool 核心能力

| 能力 | 說明 |
|------|------|
| `pipeline(items, stage1, stage2, ...)` | 每項獨立流經所有階段，無 barrier |
| `parallel(thunks)` | 並行執行，全部完成才返回（barrier） |
| `phase(title)` | 進度分組，`/workflows` 即時可視 |
| `agent(prompt, opts)` | 支援 `schema`（結構化輸出）、`agentType`（自定義代理人型別）、`isolation: 'worktree'` |
| `budget.remaining()` | Token 預算感知，動態控制深度 |
| Resume | 相同 script+args 可 resume，已完成 agent 走快取 |

關鍵橋接點：`opts.agentType` 可指定框架自定義 agent type（如 `mint-format-specialist`）。

## 重現實驗結果
### 實驗方法

文件分析實驗：比對 Workflow tool 系統指令規格（Claude Code runtime 注入）與框架四機制（Agent 單派發 / bulk-evaluate / parallel-evaluation / Workflow）的能力矩陣，驗證三項假說：
- H1: agentType 可橋接框架定義 agent
- H2: Workflow 填補框架未覆蓋的能力缺口
- H3: 四機制的適用場景存在清晰邊界

### 實驗執行

**H1 驗證（agentType 橋接）**：
- 系統指令明確記載 `agent(prompt, {agentType: 'code-reviewer'})` 可指定自定義 agent type
- 說明文字：「opts.agentType uses a custom subagent type (e.g. 'Explore', 'code-reviewer') instead of the default workflow subagent — resolved from the same registry as the Agent tool」
- 框架 agent 定義（.claude/agents/*.md，共 33 個）透過 system-reminder 的 Available agent types 列表曝光給 runtime
- 結論：**H1 已驗證**。agentType 可使用框架定義的 agent，如 `linux`、`basil-writing-critic`、`thyme-python-developer` 等
- 限制：Workflow subagent 不繼承 PM context，不執行 ticket 操作（claim/complete），agent 定義中的 ticket 生命週期責任在 Workflow 內不適用

**H2 驗證（能力缺口）**：
框架現有三機制的能力限制：
1. Agent 單派發：每次派發消耗 PM context（prompt 撰寫 + 結果閱讀），N 個派發 = N 倍 PM context 壓力
2. bulk-evaluate：透過 ticket 卸載 context，但每個子任務必須建 ticket（overhead）、結果不回到主線程（無法程式化聚合）
3. parallel-evaluation：固定三人組 + 常駐委員，適用「N 視角 x 1 標的」，不適用「1 標準 x N 項目」

Workflow 填補的缺口：
- **確定性控制流**：script 決定 fan-out / barrier / 停止條件，不消耗 PM 推理 token
- **結構化輸出聚合**：schema option 強制 subagent 回傳 validated JSON，caller 可程式化 filter/map/reduce
- **Budget 感知動態深度**：`budget.remaining()` 可在 runtime 根據剩餘 token 動態調整覆蓋範圍
- **Resume**：相同 script+args 可 resume，已完成 agent 走快取（框架無此能力）
- **Pipeline 無 barrier**：pipeline 中 item A 可在 stage 3 而 item B 仍在 stage 1，wall-clock = 最慢單項鏈（非最慢階段之和）
- 結論：**H2 已驗證**。Workflow 填補「確定性批量編排 + 結構化聚合 + 動態深度」三個框架空白

**H3 驗證（場景邊界）**：
分析四機制各自的不可替代性（uniqueness axis）：
- Agent 單派發：PM 判斷介入、ticket 生命週期追蹤、跨 session handoff → 不可被 Workflow 取代
- bulk-evaluate：結論需落地到 ticket 作為持久化追蹤載體 → Workflow 結果是 ephemeral JSON，不可取代
- parallel-evaluation：常駐委員機制（linux/basil）+ Worth-It Filter + 衝突處理策略 → 領域知識密集，不適合 script 化
- Workflow：確定性批量 + 結構化聚合 + budget 感知 → 上述三機制皆無此能力
- 結論：**H3 已驗證**。四機制的 uniqueness axis 不重疊，場景邊界清晰

### 實驗發現

**已驗證事實**：
1. agentType 橋接可用，框架 agent 可在 Workflow 內作為 subagent 執行
2. Workflow 填補「確定性批量 + 結構化聚合 + 動態深度」三個框架空白
3. 四機制的適用場景邊界清晰，uniqueness axis 不重疊

**仍未驗證的假設**：
1. agentType subagent 是否繼承 AGENT_PRELOAD.md 的規則（需實機驗證）
2. Workflow resume 在跨 session 場景的穩定性（同 session 內有效，跨 session 未驗證）
3. opts.isolation='worktree' 在框架 worktree 管理機制下是否衝突（Workflow worktree 是自動清理的臨時目錄，非框架 `/worktree create` 管理的持久 worktree）

### 結論：有限整合（方案 B）

採「有限整合」策略——Workflow 作為框架的第四機制，與 Agent/bulk-evaluate/parallel-evaluation 平行存在，由 PM 決策樹路由選擇。不重構現有三機制，不嘗試用 Workflow 取代它們。

### AC1: 四機制能力比對表

| 維度 | Workflow | Agent 單派發 | bulk-evaluate | parallel-evaluation |
|------|---------|-------------|---------------|-------------------|
| 控制流 | 確定性 script | PM 模型驅動 | PM 模型驅動 | PM 模型驅動 |
| 結果回收 | 結構化 JSON schema | PM 讀 transcript | 寫入子 ticket | PM 彙整報告 |
| Ticket 追蹤 | 無 | 有 | 有（子 ticket） | 有 |
| Agent 型別 | agentType 橋接 | 框架定義 | 框架定義 | 固定三人組+常駐 |
| Resume | 有（script+args 快取） | 無 | 無 | 無 |
| PM context 消耗 | 零（script 自治） | 高（每次 prompt+result） | 低（只看統計） | 中（彙整報告） |
| Budget 感知 | 有（runtime 動態） | 無 | 無 | 無 |
| Pipeline 無 barrier | 有 | N/A | N/A | N/A |
| 跨 session 持久化 | 無（ephemeral） | 有（ticket） | 有（ticket） | 有（ticket） |
| 適用項目數 | 數十～數百 | 1-3 | 5-30 | 3-5 視角 |
| Uniqueness axis | 確定性批量+聚合+動態深度 | PM 判斷+ticket 追蹤 | Context 卸載+持久化 | 多視角+衝突處理 |

### AC2: 場景邊界

**Workflow 最適用場景**：
1. **同質批量處理**：N 個檔案做相同操作（lint 修復、格式轉換、批量遷移）
2. **多階段 pipeline**：每項目流經 find → verify → fix 多階段，無需 PM 介入
3. **對抗式驗證**：N 個發現各由 3 個獨立 skeptic 驗證，多數決過濾 false positive
4. **發現型掃描**：未知數量的目標，loop-until-dry 直到連續 K 輪無新發現
5. **Budget 驅動深度**：用戶指定 token 預算，動態調整覆蓋範圍

**Workflow 不適用場景**：
1. **需跨 session 追蹤**：結果需持久化為 ticket、handoff 給下 session → 用 Agent/bulk-evaluate
2. **需 PM 判斷介入**：步驟之間需 PM 根據上步結果做路由決策 → 用 Agent 單派發
3. **需領域專家衝突仲裁**：多視角發現需 Worth-It Filter + linux 品味否決 → 用 parallel-evaluation
4. **異質任務**：每個子任務需不同 agent type 和不同 prompt 結構 → 用 Agent 單派發
5. **修改共享狀態**：多 agent 同時修改相同檔案 → 需 worktree 隔離或序列化，Workflow 的自動 worktree 與框架 worktree 管理可能衝突

### AC3: agentType 橋接方案

**設計**：Workflow script 透過 `opts.agentType` 指定框架 agent type，使 Workflow subagent 繼承框架 agent 的系統提示（身份、允許產出、禁止行為）。

**用法**：
```javascript
// 使用框架 agent type
const result = await agent('審查此檔案的程式碼品質', {
  agentType: 'linux',
  schema: REVIEW_SCHEMA
})

// pipeline + agentType
const results = await pipeline(
  files,
  f => agent(`審查 ${f}`, {agentType: 'linux', phase: 'Review', schema: FINDINGS}),
  (review, f) => parallel(review.findings.map(finding => () =>
    agent(`驗證: ${finding.desc}`, {agentType: 'bay-quality-auditor', phase: 'Verify', schema: VERDICT})
  ))
)
```

**限制與注意事項**：
- Workflow subagent 不執行 ticket 操作（claim/complete/append-log），agent 定義中的 ticket 生命週期責任不適用
- Workflow subagent 不繼承 PM context，prompt 必須自包含
- agentType subagent 是否繼承 AGENT_PRELOAD.md 規則未經實機驗證（待後續 spike）
- Workflow 的 worktree 隔離（`isolation: 'worktree'`）是自動清理的臨時目錄，與框架 `/worktree create` 管理的持久 worktree 無關

### AC4: 融入建議（哪些 Skill 可改用 Workflow 增強）

| Skill | 改用 Workflow 的增強方向 | 價值 | 優先級 |
|-------|----------------------|------|--------|
| parallel-evaluation | Phase 2 掃描改用 Workflow pipeline，每個視角完成後立即進入 verify 階段（無 barrier） | 減少 wall-clock + PM context 消耗 | P1 |
| bulk-evaluate | 當結果不需 ticket 持久化時，改用 Workflow 取代 N 個子 ticket 建立 | 減少 ticket overhead | P2 |
| code-review（/code-review） | 已有 ultra 模式使用 Workflow | 已實現 | — |
| multi-round-review | 多輪 reviewer audit 可用 Workflow pipeline 編排 round 1 → round 2 → round 3 | 確定性 round 排程 | P2 |
| i18n-checker | 全量掃描 + 批量替換可用 Workflow pipeline | 批量處理效率 | P3 |

### AC5: PM 決策樹路由建議

```
任務需要多 agent 協作？
├── 否 → Agent 單派發
└── 是 →
    ├── 結果需跨 session 持久化（ticket 追蹤）？
    │   ├── 是 → 需多視角衝突仲裁？
    │   │   ├── 是 → parallel-evaluation
    │   │   └── 否 → bulk-evaluate
    │   └── 否 → 控制流確定性（相同操作 x N 項目）？
    │       ├── 是 → Workflow
    │       └── 否 → Agent 單派發（PM 逐步決策）
    └── 特殊場景：
        ├── 用戶說「ultracode」→ Workflow（系統自動啟用）
        ├── 用戶說「use a workflow」→ Workflow
        ├── 用戶指定 +Nk token 預算 → Workflow（budget 感知）
        └── 對抗式驗證（adversarial verify）→ Workflow
```
