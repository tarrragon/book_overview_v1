## Context Bundle

### 現狀分析

content-modular.js 目前硬編碼 Readmoo adapter：
- L35: `const createReadmooAdapter = require('./adapters/readmoo-adapter')` — 直接引用
- L71: `if (!pageStatus.isReadmooPage)` — 硬編碼書城判斷
- L86: `readmooAdapter = createReadmooAdapter()` — 硬編碼建立
- L199, L359, L379: 多處引用 `readmooAdapter` 和 `pageStatus.isReadmooPage`

PlatformRegistry 已存在（W2-006 完成）：`src/content/platform/platform-registry.js`
- `detect(url)` 回傳 `{ config, createAdapter }` 或 null
- `config.name` 為書城識別碼（如 'readmoo'）

### 改動策略

1. 移除 L35 的 `createReadmooAdapter` 直接引用
2. 新增 `const PlatformRegistry = require('./platform/platform-registry')`
3. `initializeContentScript()` 中用 `PlatformRegistry.detect(window.location.href)` 取代 `isReadmooPage` 判斷
4. adapter 改為 `platform.createAdapter()` 動態建立
5. 變數名改為通用名（`platformAdapter` 取代 `readmooAdapter`）
6. `reportReadyStatus()` / `getPageStatus()` / `getHealthStatus()` 中的 `isReadmooPage` 改為 `isSupportedPage`

### 注意事項

- `bookDataExtractor.setReadmooAdapter(readmooAdapter)` 方法名可能需要改為更通用的名稱，但這可能涉及 book-data-extractor.js 的改動。若改名影響範圍過大，可先保留方法名、只改傳入的變數。
- globalThis 上的變數名（`globalThis.readmooAdapter`）若有測試依賴需同步更新
- chrome.runtime.onMessage listener 必須保持在頂層同步註冊（Manifest V3 限制）
