## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

現有架構為單書城（Readmoo）設計，多處硬編碼阻礙多書城擴展：

1. **content-modular.js 直接 import readmoo-adapter** — 無書城選擇/路由機制
2. **page-detector.js 硬編碼 `readmoo.com`** — 無法識別其他書城頁面
3. **manifest.json content_scripts.matches 只有 readmoo.com** — 其他書城頁面不注入 content script
4. **BookSchemaV2 platform 固定為 'READMOO'** — 資料模型無法區分不同書城來源
5. **PlatformAdapterInterface 已存在但未被繼承** — readmoo-adapter 採 duck typing，介面契約未強制

### 影響範圍

| 層級 | 檔案 | 改動類型 |
|------|------|---------|
| 入口 | manifest.json | 擴增 content_scripts matches |
| 路由 | content-modular.js / page-detector.js | 書城偵測 + adapter 動態選擇 |
| 介面 | platform-adapter-interface.js | 介面擴充（新增書城識別方法） |
| 資料模型 | BookSchemaV2.js | platform 欄位改為動態值 |
| 適配器 | readmoo-adapter.js | 拆分通用/專屬邏輯 |
| 新增 | 每個書城一個 adapter + bookstore doc | 新檔案 |

### 相關 Error Pattern

無直接相關 error pattern。

## 重現實驗結果
### 實驗方法

以程式碼閱讀為實驗方法，追蹤從 manifest.json → content-modular.js → readmoo-adapter.js → BookSchemaV2.js 的完整資料流，標記所有硬編碼 Readmoo 的接合點。

### 實驗執行

1. `manifest.json` content_scripts.matches：僅 `*://*.readmoo.com/*` — 其他書城頁面不會注入 content script
2. `content-modular.js:35`：`const createReadmooAdapter = require('./adapters/readmoo-adapter')` — 直接引用，無路由
3. `page-detector.js:55`：`location.hostname.includes('readmoo.com')` — 硬編碼域名
4. `readmoo-adapter.js:1487`：`source: 'readmoo'` — 書籍資料硬編碼來源
5. `BookSchemaV2.js:47`：`platform: 'READMOO'` — Schema 級別硬編碼

已驗證的事實：
- PlatformAdapterInterface 已定義 13 個抽象方法（getPageType / isExtractablePage / checkPageReady / isValidDomain / getBookElements / getBookCount / findBookContainer / parseBookElement / extractAllBooks / extractBookData / sanitizeData / getStats / reset）
- readmoo-adapter.js 未繼承 PlatformAdapterInterface，但實作了大部分同名方法（duck typing）
- readmoo-adapter.js 2366 行，混合通用捲動載入邏輯與 Readmoo DOM 選擇器

### 實驗發現

**已驗證事實**：5 個硬編碼接合點確認；PlatformAdapterInterface 介面已存在可複用。
**仍未驗證假設**：其他書城（博客來、Kobo 等）的 DOM 結構差異程度需各書城勘查 ticket 確認（W2-002 等已建）。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 多書城共用架構設計

### 架構總覽

```
manifest.json (content_scripts.matches: 所有書城 URL)
    ↓
content-modular.js
    ↓
PlatformRegistry.detect(url)  ←── 書城偵測路由（新增）
    ↓
PlatformAdapterInterface      ←── 抽象介面（既有，擴充）
    ├── ReadmooAdapter         ←── Readmoo 專屬（重構自既有）
    ├── BooksComTwAdapter      ←── 博客來專屬（v1.5.0 新增）
    ├── KoboAdapter            ←── Kobo 專屬（v1.6.0 新增）
    └── ...                    ←── 後續書城
    ↓
BookSchemaV2 { source: adapter.name }  ←── 資料模型（platform 動態化）
```

### 1. PlatformRegistry — 書城偵測與路由

**新增檔案**：`src/content/platform/platform-registry.js`

**職責**：依 URL 偵測當前書城，回傳對應的 adapter factory。

```javascript
// platform-registry.js 骨架
const PLATFORM_CONFIGS = [
  {
    name: 'readmoo',
    displayName: 'Readmoo 讀墨',
    matchPatterns: ['*://*.readmoo.com/*', '*://readmoo.com/*'],
    hostnames: ['readmoo.com', 'read.readmoo.com'],
    adapterFactory: () => require('../adapters/readmoo-adapter'),
    libraryUrl: 'https://read.readmoo.com/#/library'
  },
  // 後續書城在此註冊
]

function detect(url) → { config, createAdapter } | null
function getAllMatchPatterns() → string[]  // 供 manifest.json 聚合
function getRegisteredPlatforms() → PlatformConfig[]
```

**設計決策**：
- 靜態配置物件陣列（非 class 註冊模式）：書城數量有限（< 15），靜態配置比動態註冊更直覺、可 grep
- `adapterFactory` 為 lazy require：只載入偵測到的書城 adapter，不預載全部
- `matchPatterns` 與 `hostnames` 分離：前者供 manifest.json 生成，後者供 runtime 偵測

### 2. PlatformAdapterInterface — 介面擴充

**既有檔案**：`src/content/platform/platform-adapter-interface.js`

**擴充方法**（新增至介面契約）：

| 方法 | 回傳 | 用途 |
|------|------|------|
| `getPlatformName()` | string | 書城識別名（如 'readmoo'） |
| `getLibraryUrl()` | string | 書庫頁 URL |
| `requiresLogin()` | boolean | 書庫頁是否需要登入 |
| `getLoginCheckSelector()` | string/null | 登入狀態偵測 DOM selector |

**保留既有方法**（13 個抽象方法不變）：所有書城適配器皆須實作。

**新增通用邏輯上提**：將 readmoo-adapter.js 中的通用捲動載入邏輯（`loadAllBooksLazy`、`waitForBookElements`、`waitForRenderSettle`）上提為 interface 的預設實作（protected helper），各書城 adapter 可選擇沿用或覆寫。

### 3. 書城適配器結構

**每個書城 adapter 的標準結構**：

```
src/content/adapters/
├── readmoo-adapter.js      # 既有，重構為符合新介面
├── books-com-tw-adapter.js # v1.5.0 新增
├── kobo-adapter.js         # v1.6.0 新增
└── ...
```

**每個 adapter 必須提供**：

| 項目 | 說明 |
|------|------|
| `name` | 書城識別名（對應 BookSchemaV2.source） |
| `SELECTORS` | 書城專屬 DOM 選擇器配置 |
| `parseBookElement(element)` | 書城專屬 DOM 解析邏輯 |
| `isValidDomain(url)` | 書城域名驗證 |
| `getPageType()` | 書城頁面類型識別 |
| `extractBookId(element)` | 書城專屬書籍 ID 提取 |

**readmoo-adapter.js 重構方向**：
- 保留 Readmoo 專屬的 DOM 選擇器和解析邏輯
- 上提 `loadAllBooksLazy` / `_scrollStep` / `waitForRenderSettle` 至通用層（多數書城都有 lazy load）
- 上提 `isUnsafeUrl` / `sanitizeText` 至通用工具
- 目標：readmoo-adapter.js 從 2366 行降至 ~800 行（專屬邏輯）

### 4. 資料模型擴充

**BookSchemaV2.js 變更**：

```javascript
// 移除硬編碼
- platform: 'READMOO',
+ // platform 改為欄位定義，非 schema 級常數

// source 欄位改為必填，值由 adapter 提供
- source: { type: 'string', required: false, auto: true, default: 'readmoo' }
+ source: { type: 'string', required: true, enum: PLATFORM_NAMES }
```

**PLATFORM_NAMES**：由 PlatformRegistry 導出的書城名稱列表，確保 source 值受控。

**Storage key 隔離策略**：
- 方案 A（推薦）：同一 storage key，source 欄位區分 — 統一搜尋/匯出，跨書城去重靠 title+author
- 方案 B：每書城獨立 storage key（`books_readmoo`, `books_kobo`）— 隔離乾淨但搜尋需聚合

**推薦方案 A**：使用者期望「我的所有電子書」統一管理，非「每個書城分開管理」。

### 5. content-modular.js 改動

```javascript
// 現有
const createReadmooAdapter = require('./adapters/readmoo-adapter')

// 改為
const PlatformRegistry = require('./platform/platform-registry')

async function initializeContentScript() {
  const platform = PlatformRegistry.detect(window.location.href)
  if (!platform) return  // 非支援書城

  const adapter = platform.createAdapter()
  // 後續流程不變，adapter 是多態的
}
```

### 6. manifest.json 改動

```json
"content_scripts": [{
  "matches": [
    "*://*.readmoo.com/*",
    "*://*.books.com.tw/*",
    "*://*.kobo.com/*",
    // ...由 PlatformRegistry.getAllMatchPatterns() 聚合
  ],
  "js": ["src/content/content-modular.js"]
}]
```

**注意**：manifest.json 是靜態檔，無法 runtime 生成。每新增書城需同步更新 manifest.json 的 matches 和 PlatformRegistry 的配置。建議在 build script 加驗證步驟，確認兩處一致。

### 7. page-detector.js 改動

```javascript
// 現有
isReadmooPage = location.hostname.includes('readmoo.com')

// 改為
const platform = PlatformRegistry.detect(location.href)
isSupportedPage = !!platform
platformName = platform?.config.name || 'unknown'
```
