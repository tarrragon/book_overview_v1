## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

現有架構為單書城（Readmoo）設計，多處硬編碼阻礙多書城擴展：

1. **content-modular.js 直接 import readmoo-adapter** — 無書城選擇/路由機制
2. **page-detector.js 硬編碼 `readmoo.com`** — 無法識別其他書城頁面
3. **manifest.json content_scripts.matches 只有 readmoo.com** — 其他書城頁面不注入 content script
4. **BookSchemaV2 platform 固定為 'READMOO'** — 資料模型無法區分不同書城來源
5. **PlatformAdapterInterface 已存在但未被繼承** — readmoo-adapter 採 duck typing，介面契約未強制

### 影響範圍

| 層級 | 檔案 | 改動類型 |
|------|------|---------|
| 入口 | manifest.json | 擴增 content_scripts matches |
| 路由 | content-modular.js / page-detector.js | 書城偵測 + adapter 動態選擇 |
| 介面 | platform-adapter-interface.js | 介面擴充（新增書城識別方法） |
| 資料模型 | BookSchemaV2.js | platform 欄位改為動態值 |
| 適配器 | readmoo-adapter.js | 拆分通用/專屬邏輯 |
| 新增 | 每個書城一個 adapter + bookstore doc | 新檔案 |

## 重現實驗結果
### 實驗方法

以程式碼閱讀為實驗方法，追蹤從 manifest.json → content-modular.js → readmoo-adapter.js → BookSchemaV2.js 的完整資料流，標記所有硬編碼 Readmoo 的接合點。

### 實驗執行

1. `manifest.json` content_scripts.matches：僅 `*://*.readmoo.com/*` — 其他書城頁面不會注入 content script
2. `content-modular.js:35`：`const createReadmooAdapter = require('./adapters/readmoo-adapter')` — 直接引用，無路由
3. `page-detector.js:55`：`location.hostname.includes('readmoo.com')` — 硬編碼域名
4. `readmoo-adapter.js:1487`：`source: 'readmoo'` — 書籍資料硬編碼來源
5. `BookSchemaV2.js:47`：`platform: 'READMOO'` — Schema 級別硬編碼

已驗證的事實：
- PlatformAdapterInterface 已定義 13 個抽象方法（getPageType / isExtractablePage / checkPageReady / isValidDomain / getBookElements / getBookCount / findBookContainer / parseBookElement / extractAllBooks / extractBookData / sanitizeData / getStats / reset）
- readmoo-adapter.js 未繼承 PlatformAdapterInterface，但實作了大部分同名方法（duck typing）
- readmoo-adapter.js 2366 行，混合通用捲動載入邏輯與 Readmoo DOM 選擇器

### 實驗發現

**已驗證事實**：5 個硬編碼接合點確認；PlatformAdapterInterface 介面已存在可複用。
**仍未驗證假設**：其他書城（博客來、Kobo 等）的 DOM 結構差異程度需各書城勘查 ticket 確認（W2-002 等已建）。
