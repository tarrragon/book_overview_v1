## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 假設

parallel-evaluation 的 Phase 2（並行視角掃描）是 PM 模型驅動的 barrier——所有視角 agent 必須全部完成才進入 Phase 3 彙整。假設改用 Workflow pipeline 可使每個視角掃描完成後立即進入驗證階段，減少 wall-clock。

### WRAP

W：(A) Workflow pipeline 取代 Phase 2 barrier，(B) 保持現狀但改善 Phase 3 自動化，(C) 混合方案（pipeline 內嵌 parallel verify）
A：不擠壓產品開發
P：parallel-evaluation 的 Phase 3 需跨視角彙整（Worth-It Filter + 衝突處理），這是合法 barrier 需求，pipeline 無法完全取代

### 現有流程分析

parallel-evaluation 三階段：
- Phase 1：PM 前台收集標的（確定掃描範圍）
- Phase 2：PM 派發 3+2 agents 並行掃描（barrier：全部完成才進 Phase 3）
- Phase 3：PM 彙整 + Worth-It Filter + 建 Ticket + 錯誤模式記錄

Phase 2 的 barrier 是必要的嗎？
- 跨視角衝突處理需要所有視角結果 → 是合法 barrier
- Worth-It Filter 需全覽才能分類 → 是合法 barrier
- 但每個視角內部的「掃描 → 驗證發現」可以 pipeline 化 → 視角間不需等待

### 核心問題

不是「要不要用 Workflow 取代 parallel-evaluation」，而是「Workflow 可以改善 parallel-evaluation 的哪個環節」。

## 重現實驗結果
### 實驗方法

結構分析：對 parallel-evaluation 現有 7 個情境（A-G）的流程拆解，識別哪些步驟可 pipeline 化、哪些必須保留 barrier。

### 實驗執行

**Phase 2 barrier 分析**：

| 情境 | 視角數（含常駐） | 每視角產出 | 跨視角需求 |
|------|----------------|----------|----------|
| A 程式碼審查 | 3+2=5 | findings 清單 | 去重（同位置不同視角可能重複） |
| B 重構評估 | 3+2=5 | 重構建議 | 衝突（加法 vs 減法方向） |
| C 架構評估 | 3+2=5 | 架構建議 | 衝突（linux 品味否決權） |
| F 結論審查 | 3+2=5 | 審查意見 | 衝突（彼此矛盾需仲裁） |

**結論**：Phase 3 的跨視角彙整（去重 + 衝突處理）是 parallel-evaluation 的核心設計——它需要 ALL results at once，這是 Workflow 文件明確定義的「barrier 合法場景」之一（「Dedup/merge across the full result set before expensive downstream work」）。

**可改善環節**：

1. **Phase 2 內部**（每個視角的掃描）：如果標的有多個檔案，可在每個視角內用 pipeline 化——掃描完一個檔案立即出 findings，不需等全部檔案掃完
2. **Phase 2 → Phase 3 之間**：Phase 2 用 parallel()（barrier）收集所有視角結果，Phase 3 的 Worth-It Filter 和衝突處理可程式化（schema + JavaScript 邏輯）
3. **Phase 3 後**：findings 驗證（adversarial verify）可用 pipeline 化——每個 finding 獨立驗證，不需等其他 finding

### 實驗發現

**已驗證**：
1. Phase 3 跨視角彙整是合法 barrier，不可改為 pipeline
2. Phase 2 內部（多檔案掃描）和 Phase 3 後（findings 驗證）可 pipeline 化
3. 最大改善在「掃描 → barrier → filter → pipeline verify」四階段 Workflow

**仍未驗證**：
1. 常駐委員（linux/basil）在 Workflow 中的角色——是每個 phase 都加入，還是只在 barrier 後的 filter 階段加入？
2. Worth-It Filter 的程式化程度——目前靠 PM 人工判斷「改善幅度 x 風險」，能否用 schema 量化？

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 結論：混合方案（方案 C）

parallel-evaluation 不應被 Workflow 取代，但其 Phase 2 和 Phase 3 後可各自用 Workflow 增強。核心跨視角彙整（barrier）保留。

### AC1: Phase 2 改用 Workflow pipeline 的可行性

**可行但受限**。Phase 2 的 barrier（所有視角完成後再彙整）是合法的——跨視角去重和衝突處理需要全覽。Workflow 可改善的是：
- Phase 2 → 3：從 PM 模型驅動（每視角一個 Agent 派發 + PM 讀取結果 + PM 手動彙整）改為 Workflow parallel()（barrier）+ schema 結構化回收
- 效果：PM context 消耗從 N*2（N 個 prompt + N 個 result 閱讀）降為 1（只看 Workflow 最終結果）

### AC2: POC script 骨架

```javascript
export const meta = {
  name: 'parallel-eval-enhanced',
  description: 'Parallel evaluation with Workflow orchestration',
  phases: [
    { title: 'Scan', detail: 'Multi-perspective scan with agentType' },
    { title: 'Dedup', detail: 'Cross-perspective dedup (barrier)' },
    { title: 'Verify', detail: 'Adversarial verify per finding' }
  ]
}

const FINDINGS_SCHEMA = {
  type: 'object',
  properties: {
    perspective: { type: 'string' },
    findings: {
      type: 'array',
      items: {
        type: 'object',
        properties: {
          file: { type: 'string' },
          line: { type: 'number' },
          severity: { enum: ['high', 'medium', 'low'] },
          description: { type: 'string' },
          suggestion: { type: 'string' }
        },
        required: ['file', 'severity', 'description']
      }
    }
  },
  required: ['perspective', 'findings']
}

const VERDICT_SCHEMA = {
  type: 'object',
  properties: {
    is_real: { type: 'boolean' },
    reason: { type: 'string' },
    severity_adjustment: { enum: ['keep', 'upgrade', 'downgrade'] }
  },
  required: ['is_real', 'reason']
}

// args: { targets: string[], perspectives: [{key, prompt}], agentTypes: {scan, verify} }

phase('Scan')
// Phase 2: parallel barrier — 所有視角同時掃描，全部完成才往下
const scanResults = await parallel(
  args.perspectives.map(p => () =>
    agent(p.prompt, {
      label: `scan:${p.key}`,
      agentType: args.agentTypes.scan,
      schema: FINDINGS_SCHEMA
    })
  )
)

phase('Dedup')
// Phase 3 前半：跨視角去重（JavaScript 程式邏輯，零 agent 消耗）
const allFindings = scanResults.filter(Boolean).flatMap(r => r.findings)
const deduped = allFindings.filter((f, i, arr) =>
  arr.findIndex(x => x.file === f.file && x.description === f.description) === i
)
log(`${allFindings.length} findings → ${deduped.length} after dedup`)

phase('Verify')
// Phase 3 後半：pipeline 驗證（每個 finding 獨立，無 barrier）
const verified = await parallel(
  deduped.map(f => () =>
    agent(`Adversarially verify: ${f.description} in ${f.file}`, {
      label: `verify:${f.file}`,
      agentType: args.agentTypes.verify,
      schema: VERDICT_SCHEMA
    }).then(v => ({ ...f, verdict: v }))
  )
)

const confirmed = verified.filter(Boolean).filter(f => f.verdict && f.verdict.is_real)
return { total_scanned: allFindings.length, deduped: deduped.length, confirmed: confirmed.length, findings: confirmed }
```

### AC3: 常駐委員機制在 Workflow 內的實現方式

| 方案 | 說明 | 評估 |
|------|------|------|
| A. agentType 指定常駐 | Scan phase 用 `agentType: 'linux'` 和 `agentType: 'basil-writing-critic'` 作為固定視角 | 可行；W4-001 已驗證 agentType 完整繼承 |
| B. Verify phase 用常駐 | linux 只在 Verify 階段作為 skeptic，basil 只在文字品質 finding 驗證時加入 | 更精確但增加 script 複雜度 |
| C. 混合：Scan 全員 + Verify 專家 | Scan 時 linux/basil 與其他視角並行掃描；Verify 時 linux 作為通用 skeptic | 推薦：保留原有三人組+常駐設計 |

推薦方案 C：perspectives 陣列中固定包含 linux 和 basil-writing-critic 視角（Scan 全員），Verify 時 linux 作為通用 adversarial verifier。

### 價值評估

| 維度 | 現有（PM 驅動） | Workflow 增強 |
|------|---------------|-------------|
| PM context 消耗 | 高（5 agent prompt + 5 result + 手動彙整） | 零（script 自治） |
| 去重 | PM 手動比對 | JavaScript filter（確定性、零 token） |
| 驗證 | 無（直接進 Worth-It Filter） | adversarial verify 過濾 false positive |
| wall-clock | barrier 後序列處理 | barrier 後 pipeline verify 並行 |
| 結構化輸出 | PM 讀 transcript 摘要 | JSON schema validated |

### 風險

1. **Worth-It Filter 程式化困難**：「改善幅度 x 風險」的判斷目前靠 PM 領域知識，schema 化可能丟失 nuance → 建議保留 PM 人工 Worth-It Filter 作為 Workflow 後的第四步
2. **常駐委員衝突處理**：linux 品味否決權（Garbage = 否決）目前靠 PM 仲裁，script 化需定義明確規則 → 建議第一版不 script 化衝突處理
