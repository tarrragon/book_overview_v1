# Execution Log

## Task Summary

確認 CC 現行是否仍有名為 Task 的工具；若無，評估 Task matcher 掛載的 4 個 hook（agent-ticket-validation/agent-prompt-length-guard/task-dispatch-readiness/askuserquestion-reminder）應否移至 Agent matcher

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

原始假設（W5-001.1 提出）：CC 2.1.194 將 hook matcher 由子串匹配改為 exact-match，subagent 派發工具已改名 `Agent`，故 settings.json 掛在 `Task` matcher 的 hook 可能靜默失效（dead matcher）。

實驗推翻此假設。真實根因分為兩層：

1. **matcher 層：`Task` matcher 並非 dead。** 當前 CC 版本仍將 subagent 派發（工具表面名 `Agent`）同時投遞給 `Task` 與 `Agent` 兩個 matcher。今日 11 次 Agent 派發，只註冊在 `Task` matcher 下（未註冊於 `Agent` 或 `*`）的 hook 全部觸發 11 次。故 `Task` 是 `Agent` 派發的有效別名 matcher。

2. **hook 內部 tool_name 守衛層：2 個 hook 功能性死亡（非 matcher 造成）。** `task-dispatch-readiness-check.py`（line 204 `if tool_name != "Task": sys.exit(0)`）與 `askuserquestion-reminder-hook.py`（line 66 `if tool_name != "Task"`）內部以字面 `"Task"` 過濾；但 matcher 投遞進來的 tool_name 是 `"Agent"`，故兩者每次都在實質邏輯前早退。matcher 活著，hook 的內部字串守衛卻是 stale——這才是真正失效的元件。

對照組（未受影響）：`agent-ticket-validation-hook.py` 無 tool_name 守衛，正常處理 Agent 派發；`agent-prompt-length-guard-hook.py` 守衛為 `if tool_name not in ("Agent","Task")`，正確接受 Agent。

### 影響範圍

| Hook | matcher | 內部 tool_name 守衛 | 今日觸發 | 實質功能狀態 |
|------|---------|---------------------|---------|-------------|
| agent-ticket-validation-hook | Task（僅） | 無 | 11 次 ALLOWED | 正常 |
| agent-prompt-length-guard-hook | Task + Agent | `not in ("Agent","Task")` | 11 次（非 22，CC 對跨 matcher 同命令去重，單次執行） | 正常，但 Task 註冊為冗餘 |
| task-dispatch-readiness-check | Task（僅） | `!= "Task"` 早退 | 0 log／無 log 目錄 | **功能性死亡**（代理人分派正確性檢查從未實際執行） |
| askuserquestion-reminder-hook | Task（僅） | `!= "Task"` skip | 11 次全部「非 Task 工具: Agent，跳過」 | **功能性死亡**（多任務派發提醒從未觸發） |

受影響檔案：`.claude/settings.json`（matcher 註冊）、`.claude/hooks/task-dispatch-readiness-check.py`、`.claude/hooks/askuserquestion-reminder-hook.py`。

潛在風險（W5-001.1 的 breaking change 疑慮以反方向存在）：目前 3 個 Task-only hook 完全依賴「CC 把 Agent 投遞給 Task matcher」這個未文件化的別名行為。若未來 CC 版本移除該別名（真正改為嚴格 exact-match，只有 tool=`Task` 才進 `Task` matcher），3 個 hook 會同時徹底失聯。

### 相關 Error Pattern

- 概念對應 quality-baseline 規則 1 邊界「測試綠燈不等於 Runtime 正確」的同型陷阱：settings.json 語法合法 + hook 檔存在 + matcher 會觸發，三者皆綠，但 hook 內部字串守衛使實質邏輯靜默失效（PC-165 false-positive 家族）。
- 建議 PM 評估是否升級為新 error-pattern：「hook 內部 tool_name 字面守衛與 matcher 投遞的工具名漂移」。

固定值證據：

- **settings.json 註冊**：`agent-ticket-validation` / `task-dispatch-readiness` / `askuserquestion-reminder` 僅註冊於 `PreToolUse` matcher `Task`（非 `Agent`、非 `*`）；`agent-prompt-length-guard` 同時註冊於 `Task` 與 `Agent`。
- **今日 log 檔數**：agent-ticket-validation=11、agent-prompt-length-guard=11、askuserquestion-reminder=11、agent-dispatch-validation（Agent-only 對照）=11、task-dispatch-readiness=0（且無 log 目錄）。
- **log 內文 tool_name**：askuserquestion-reminder 今日 11 次全部記錄 `非 Task 工具: Agent，跳過`（13:37–15:10）；agent-ticket-validation 記錄 `代理人類型 'general-purpose'` + `Task 派發驗證: is_valid=True` 並 ALLOWED。
- **原始碼守衛**：task-dispatch-readiness-check.py:204 `if tool_name != "Task": sys.exit(0)`（在任何實質邏輯與檔案 log 之前）；askuserquestion-reminder-hook.py:66 `if tool_name != "Task"`；agent-prompt-length-guard-hook.py:94 `if tool_name not in ("Agent","Task")`；agent-ticket-validation-hook.py 無 tool_name 守衛。

### 實驗發現

已驗證的事實：

1. `Task` matcher 有效，當前 CC 版本將 `Agent` 派發同時投遞給 `Task` 與 `Agent` matcher（Task-only hook 今日觸發 11 次即證）。原始「dead matcher」假設被推翻。
2. CC 對「同一命令跨兩個 matcher 都命中同一 tool call」會去重、單次執行（agent-prompt-length-guard 雙註冊卻只 11 檔而非 22 檔）。故不存在雙重執行問題。
3. task-dispatch-readiness-check 與 askuserquestion-reminder 因內部 `!= "Task"` 字面守衛，每次派發皆在實質邏輯前早退——功能性死亡，但成因是 hook 內部守衛 stale，非 matcher。
4. task-dispatch-readiness 的 0 log／無 log 目錄，佐證其早退發生在任何 log 寫入之前（line 204 早於 `log_warning_to_file`）。

仍未驗證的假設：CC 未來版本是否會移除「Agent 投遞給 Task matcher」的別名行為（屬前瞻風險，非當前事實）。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 判定

`Task` matcher **不是 dead**。它是 `Agent` 派發的有效別名 matcher。真正失效的是 2 個 hook 的內部 `tool_name != "Task"` 字面守衛（功能性死亡）。

### 修復建議（分兩部分，僅列建議，settings.json 與 hook 修改由 PM 決策後派發）

**Part 1 — 修內部守衛（真正的 bug，救活 2 個死 hook；高優先，當前 CC 即受益）**

| 檔案 | 現況 | 建議改為 |
|------|------|---------|
| task-dispatch-readiness-check.py:204 | `if tool_name != "Task": sys.exit(0)` | `if tool_name not in ("Agent","Task"): sys.exit(0)` |
| askuserquestion-reminder-hook.py:66 | `if tool_name != "Task"` | `if tool_name not in ("Agent","Task")` |

對齊 agent-prompt-length-guard 已採用的雙名接受模式。

**Part 2 — matcher 收斂（去冗餘 + 防未來 CC 別名行為消失；中優先）**

- 將 3 個 Task-only hook（agent-ticket-validation、task-dispatch-readiness、askuserquestion-reminder）移至 `Agent` matcher；移除 agent-prompt-length-guard 在 `Task` 下的重複註冊；刪除 `Task` matcher 區塊。
- Why：`Agent` matcher 已捕捉每一次派發，`Task` 區塊純屬依賴未文件化別名行為的冗餘；收斂為單一 matcher 使意圖顯性，並消除「CC 未來移除 Task→Agent 別名時 3 hook 同時失聯」的風險。
- 依賴關係：Part 2 移動 hook 後，tool_name 只會是 `Agent`，故 **Part 2 必須與 Part 1 綁定**（否則 2 個 hook 仍會早退）。Part 1 可獨立先行。

淘汰判定：無 hook 應被淘汰。4 個皆有職責，2 個死 hook 是待修復非待刪除。
