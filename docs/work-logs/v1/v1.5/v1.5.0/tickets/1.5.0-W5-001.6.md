### 問題根因

CC 2.1.184 新增 `TaskCreated` hook event（TaskCreate 工具建立 task 時觸發）。W5-001 主評估將其列為潛在的 dispatch 可觀測性增強候選，需判定是否值得在框架新增此 hook。

### 影響範圍

- `.claude/settings.json` hooks 區塊（若採用需新增註冊）
- `.claude/hooks/dispatch-record-hook.py`（既有 dispatch 追蹤，掛 Agent matcher）

### 相關 Error Pattern

無直接對應。相關規範：PC-104（平台自動受益判定模式）。

## 重現實驗結果

### 實驗方法

事實盤點：(1) grep settings.json 確認是否已有 TaskCreate/TaskUpdate 相關 matcher；(2) 確認既有 dispatch 追蹤機制的掛載點；(3) 對照 TaskCreated 觸發語意（TaskCreate 工具 = CC 內建 task list，非 Agent 派發）。

### 實驗執行

- `grep -n "TaskCreate\|TaskUpdate" .claude/settings.json`：零命中，框架未使用 CC task list 相關 hook
- 既有 dispatch 追蹤：`dispatch-record-hook.py` 掛 `Agent` matcher（W5-001.1 掃描確認），代理人派發已有記錄點
- TaskCreated 語意：觸發於 TaskCreate 工具（CC 內建 task/todo 清單），與 Agent tool 派發是不同事件流

### 實驗發現

- 已驗證事實：框架的任務追蹤走自有 ticket 系統（docs/work-logs + ticket CLI），不依賴 CC task list
- 已驗證事實：代理人派發可觀測性已由 Agent matcher 上的 dispatch-record / agent-dispatch-validation 等 hook 覆蓋
- 已驗證事實：TaskCreated 監聽的是 CC task list 事件，與框架 dispatch 追蹤的事件流（Agent tool）不重疊——採用它不會增強 dispatch 可觀測性，只會監聽一個框架不使用的系統

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### WRAP 分析

### W（擴增選項）

1. 不採用 TaskCreated hook（採用）
2. 新增 TaskCreated hook 記錄 CC task list 事件
3. 改造框架 dispatch 流程改用 CC task list（大改）

### R（現實檢驗）

- TaskCreated 觸發於 TaskCreate 工具（CC 內建 task list），框架任務追蹤走自有 ticket 系統，兩者事件流不重疊
- dispatch 可觀測性缺口（W5-001 主評估的原始動機）實際由 Agent matcher hooks 覆蓋，TaskCreated 無法填補該缺口
- 選項 3 違反框架資產穩定性（ticket 系統是核心資產），成本遠超收益

### A（拉開距離）

- 不採用的成本：零——監聽框架不使用的系統沒有資訊價值
- 採用的成本：hook 註冊 + 維護負擔，換取零價值事件流

### P（絆腳索）

- 絆腳索：框架未來若開始使用 CC task list 追蹤背景代理人（如 TaskCreate 被 harness 自動觸發的場景增加）→ 重評本結論

### 結論

**不採用（已決策，狀態 a）**。TaskCreated 監聽的 CC task list 與框架自有 ticket 系統是不同事件流，dispatch 可觀測性已由 Agent matcher hooks 覆蓋，新增此 hook 為零價值監聽。

### multi_view_status

multi_view_status: skipped
reason: 單一 hook event 的採用判定，事實盤點（grep 零命中 + 事件流不重疊）已足以支撐結論，無方案取捨空間

### 後續 Ticket 判定

無需建 children：結論為不採用，零落地動作。重評條件已綁定絆腳索（框架開始使用 CC task list），符合 decision-trigger-binding 狀態 (a)。
