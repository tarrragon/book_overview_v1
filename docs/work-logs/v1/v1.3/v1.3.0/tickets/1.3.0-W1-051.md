## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

框架 skill 與專案 skill 混在同一 repo（tarrragon/claude），導致：
1. **sync 方向混亂**：push 會將專案專屬內容（如 project-integration/）推到公開框架 repo
2. **公開上架障礙**：想公開某個 skill 時，它被嵌在一個含 50+ skills 的 monorepo 內，無法獨立安裝
3. **分類模糊**：通用 skill（如 wrap-decision）與平台專屬 skill（如 provider-architecture for Flutter）混合管理

### 影響範圍

- `.claude/skills/`（50 個 framework skills + 2 個 local-only）
- `.claude/scripts/sync-claude-push.py`、`sync-claude-pull.py`（sync 機制）
- `.claude/hooks/lib/sync_exclude_manifest.py`（排除清單）
- 所有 consumer 專案（目前已知：book_overview_v1, unipos 等）

### 現況盤點

**框架 repo（tarrragon/claude）結構**：
- 扁平結構：`skills/`、`hooks/`、`rules/` 等在根目錄（無 `.claude/` 包裝）
- 含 50 個 skills，全部同步到所有 consumer 專案
- 7 個 skill 有 `pyproject.toml`（CLI 工具型：ticket, doc, version-release, project-init, mermaid-ascii, worktree, branch-worktree-guardian）

**Consumer 專案 vs 框架 repo 差異**：
- 本地多 2 個 skills：`learned`（本地學習記錄）、`tdd-phase1-split`（缺 SKILL.md）
- 本地在 `.claude/skills/` 下，框架在 `skills/` 下

### 相關 Error Pattern

- W1-050：project-integration/ 未被 sync 排除會跨專案散播
- W1-002 ANA：F 案外移方案分析（本 ticket 的前身）

### 架構方案分析

### 方案 1：Individual Skill Repos（每 skill 一個 repo）

```
tarrragon/claude-skill-wrap-decision/
├── SKILL.md
├── references/
│   └── (通用 references)
└── hooks/  (skill-specific hooks, if any)
```

| 維度 | 評估 |
|------|------|
| 獨立性 | 最高——每個 skill 獨立版本、獨立 issue tracker |
| 維護成本 | 極高——24+ repos 需要同步更新（CI、PR review、版本管理） |
| 安裝體驗 | 每個 skill 一次 git clone 到 .claude/skills/ |
| sync 相容性 | 完全不相容——需重寫 sync 機制為 git subtree/submodule |
| 適用時機 | Anthropic 推出官方 skill registry 後 |

**結論**：過度工程化。50 個 repos 的維護負擔遠超收益。

### 方案 2：Skills-only Monorepo（skills 獨立 repo，框架另一個）

```
tarrragon/claude-skills/           # 純 skills repo
├── wrap-decision/
│   ├── SKILL.md
│   └── references/
├── compositional-writing/
│   └── ...
└── ...

tarrragon/claude/                  # 框架 infra repo（hooks, rules, agents, pm-rules）
├── hooks/
├── rules/
├── agents/
└── ...
```

| 維度 | 評估 |
|------|------|
| 獨立性 | 中——skills 與框架 infra 分離，但 skills 仍綁在一起 |
| 維護成本 | 中——2 repos，但跨 repo 引用需管理 |
| 安裝體驗 | 可選擇性 clone skills repo 或只 clone 框架 |
| sync 相容性 | 需改寫 sync 機制為雙 repo push/pull |
| 風險 | 打破現有 sync 流程，需大量改寫 sync-push/pull |

**結論**：收益不明顯。Skills 與框架 infra 互有引用（skill description 引用 rules、agents 引用 skills），強行分離造成 dangling references。

### 方案 3：框架 Monorepo + Push-direction 排除（最小變更）

```
tarrragon/claude/                  # 現有結構不變
├── skills/
│   ├── wrap-decision/
│   │   ├── SKILL.md
│   │   └── references/
│   │       ├── core-concepts.md        # 推到框架 repo
│   │       └── project-integration/    # push 排除，不推到框架 repo
│   └── ...
├── hooks/
└── ...
```

| 維度 | 評估 |
|------|------|
| 獨立性 | 低——skills 仍在框架 monorepo 內 |
| 維護成本 | 最低——改一行排除設定 |
| 安裝體驗 | 不變——consumer sync-pull 拿到所有 skills |
| sync 相容性 | 完全相容——只加 push 排除 |
| 解決的問題 | 僅解決 project-integration/ 外洩，不解決獨立上架 |

**結論**：解決眼前問題（W1-050），但不解決獨立上架需求。

### 方案 4（推薦）：漸進式混合——框架 Monorepo + 公開 Skill 按需抽取

**Phase 1**（v1.3.0，立即）：Push 排除 + 落地層分離
- sync-push 排除 `*/project-integration/`（解除 W1-050）
- 框架 monorepo 結構不變

**Phase 2**（v1.0.0 前置，按需）：公開候選 Skill 抽取為獨立 repo
- 挑選 3-5 個最通用的 skill（wrap-decision、compositional-writing、tdd 等）
- 用腳本從框架 repo 抽取通用層（排除 project-integration/）到獨立 repo
- 獨立 repo 結構：
  ```
  tarrragon/claude-skill-wrap-decision/
  ├── SKILL.md
  ├── references/
  │   └── (通用 references only)
  └── README.md (安裝說明：git clone 到 .claude/skills/)
  ```
- 框架 repo 保留完整 skill（含 project-integration/），作為 SSOT
- 獨立 repo 由腳本從框架 repo 單向生成（非雙向 sync）

**Phase 3**（Anthropic marketplace 出現後）：適配官方分發格式
- 等 Anthropic 公布 skill 分發標準後，將獨立 repo 適配其格式
- 若格式為目錄型（現行），直接可用
- 若格式為 package 型（npm/pip），在獨立 repo 加 manifest

| 維度 | 方案 4 評估 |
|------|------------|
| 獨立性 | 漸進提升——Phase 1 最小改動，Phase 2 按需抽取 |
| 維護成本 | 低——框架 repo 是 SSOT，獨立 repo 由腳本單向生成 |
| 安裝體驗 | 改善——想用單一 skill 的人直接 clone 獨立 repo |
| sync 相容性 | 完全相容——框架 repo 結構不變 |
| 風險 | 最低——每個 Phase 獨立，不做也不影響 |

### 框架 repo vs Skill repo 分界定義

| 保留在框架 repo | 搬到獨立 skill repo |
|----------------|-------------------|
| 所有 50 skills（完整版含 project-integration/） | 通用層 snapshot（排除 project-integration/） |
| hooks/、rules/、agents/、pm-rules/ | 不含 |
| sync 腳本、設定檔 | 不含 |
| Skill 的 project-integration/ | 不含（留在框架 repo） |

**SSOT 原則**：框架 repo 是所有 skill 的 SSOT。獨立 repo 是「唯讀 mirror」，由腳本從框架 repo 抽取生成。修改只在框架 repo 進行，再重新生成獨立 repo。
