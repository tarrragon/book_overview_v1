## 重現實驗結果
### 實驗方法

盤點框架 hooks/ 目錄中引用 skills/ 的所有 hook，分析其依賴關係。

### 實驗執行

- `find .claude/skills/*/hooks/` 列出已在 skill 目錄的 33 個 hooks
- `grep -rl "skills/" .claude/hooks/*.py` 找出框架層引用 skill 的 16 個 hooks
- 逐一分析依賴的 skill 數量（單一 vs 多個）

### 實驗發現

**已驗證事實**：
1. 33 個 hook 已在 skill 目錄（主要是 ticket 和 worktree skill）
2. 16 個框架 hook 引用特定 skill，其中 8 個只依賴單一 skill（可搬移），8 個依賴多個 skill（保留框架層）
3. 77 個純框架 hook 不引用任何 skill（不需搬移）

**仍未驗證假設**：
1. 搬移 hook 後 settings.json 的路徑更新是否會破壞其他 consumer 的 hook 註冊
2. 跨 skill hook（保留框架層的 8 個）在 skill 獨立安裝時是否需要條件執行（skill 不存在時 graceful skip）

## Solution

### AC 1：應搬進 skill 的 hooks（8 個）

| 框架 hook | → 目標 skill |
|-----------|------------|
| agent-dispatch-validation-hook.py | ticket |
| cli-error-feedback-hook.py | ticket |
| skill-cli-error-feedback-hook.py | ticket |
| comment-qa-hook.py | compositional-writing |

| layer-boundary-validator-hook.py | tdd |
| wrap-skill-yaml-consistency-hook.py | wrap-decision |
| version-release-guard-hook.py | version-release |

### AC 2：需標注 skill 依賴的 rules/agents

待後續盤點（需掃描 rules/ 和 agents/ 中引用特定 skill 的檔案）。此 ANA 先聚焦 hook 搬移盤點。

### AC 3：Skill 自包含標準

```
<skill-name>/
├── SKILL.md                # 必要：skill 定義
├── references/             # 可選：深度內容
├── hooks/                  # 可選：skill 專屬 hooks
│   └── <hook-name>.py
├── pyproject.toml          # 可選：CLI 工具型 skill
└── <package>/              # 可選：Python package
```

自包含判準：clone skill 目錄到任何專案的 `.claude/skills/`，skill 即可運作。hook 在 `skills/<name>/hooks/` 下，settings.json 路徑用 `$CLAUDE_PROJECT_DIR/.claude/skills/<name>/hooks/<hook>.py`。

### AC 4：Skills repo 擴充為全部框架 skills

從 24 個（B 類）→ 全部 50 個框架 skills。排除條件僅剩「專案私有 skill」（sync-skills.yaml private 清單）。

### AC 5：settings.json hook 路徑影響

搬移 hook 後，settings.json 中的路徑需更新。但 settings.json 是**專案層級**設定（每個 consumer 各自維護），不隨 sync 推送。

解法：sync-pull 完成後，由 `hook-completeness-check` SessionStart hook 自動偵測未註冊的 skill hooks 並提示用戶更新 settings.json。
