## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 現況分析

**問題根因**：popup.html 中 UI 元素（按鈕、結果卡片、匯入結果/錯誤卡片）直接以 HTML 定義，缺乏模組化。匯入結果卡片和匯入錯誤卡片結構相似但各自獨立定義，違反 DRY。

**現有元件架構**：
- `popup-ui-components.js`：元件「管理器」，查找既有 DOM 元素並操作狀態（show/hide/updateStatus），不建立 DOM
- `components/import-panel.js`：匯入面板控制器，接收外部傳入的 DOM 元素引用
- `popup.html`：所有 UI 結構硬編碼（6 個按鈕、5 個卡片區塊）

**影響範圍**：
- `src/popup/popup.html` — 按鈕和卡片 HTML 結構
- `src/popup/popup-ui-components.js` — DOM 元素引用初始化
- `src/popup/components/import-panel.js` — 匯入面板使用的 DOM 元素
- `src/popup/popup.js` — 初始化和事件綁定
- `tests/unit/popup/import-panel.test.js` — 12 個匯入面板測試
- `tests/unit/popup/popup-ui-components.test.js` — UI 元件測試

### WRAP 三問

**W（Widen）— 候選方案**：
1. **元件工廠函式**（選擇）：建立 `createButton()` / `createCard()` 工廠函式，popup.html 保留骨架，動態渲染內容區域
2. **Web Components**：使用 Custom Elements API。過重，Chrome Extension popup 不需要 Shadow DOM 隔離

**A（Attain distance）— 機會成本**：佔用 W2 容量，延後 W2 其他工作。但本 ticket 是 P2 且無下游 blocker，影響可控。

**P（Prepare to be wrong）— 最可能失敗原因**：重構後既有 12 個 import-panel 測試或 popup-ui-components 測試因 DOM 結構變更而 break。防護：漸進式重構，先建元件再逐步替換 HTML，每步確認測試全綠。
