## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-002 Phase 1 規格設計 src/import/json-importer.js 時，未充分盤點既有 src/overview/import/ 匯入系統。既有系統已涵蓋 JSON 解析、四來源格式偵測、版本轉換、書籍驗證、合併持久化等核心功能。W1-002 的 parseAndValidate() 與 ContentParser.parse() + detectInterchangeSource() 高度重疊。

SA Phase 0 審查報告只列出零散函式可複用，未識別到 ContentParser 類別本身已是完整的 parse+validate+extract 管線。

### 影響範圍

- src/overview/import/content-parser.js — 核心解析 SSOT
- src/overview/import/file-validator.js — 檔案驗證 SSOT
- src/overview/import/file-reader.js — File API 讀取 SSOT
- src/overview/book-file-importer.js — Orchestrator
- src/overview/import-flow-controller.js — overview 專屬 UI 流程
- W1-002 規格 src/import/json-importer.js — 應取消新建

### 相關 Error Pattern

SA Phase 0 重複實作檢查粒度不足。尚無對應 PC。

## 重現實驗結果
### 實驗方法

比對 W1-002 Phase 1 規格的三個公開函式 vs 既有 src/overview/import/ 系統的等價能力。

### 實驗執行

#### 對照 1：parseAndValidate() vs ContentParser.parse()

W1-002 的 JSON.parse → ContentParser._parseJSONContent() 已有。
W1-002 的 detectInterchangeSource → ContentParser._extractBooksFromData() 內部已呼叫。
W1-002 的 BOM 移除 → ContentParser._removeBOM() 已有。
W1-002 的書籍驗證 → ContentParser._filterValidBooks() 已有。
metadata 提取 → 新需求，既有系統無。
bookCount 直接回傳 → 新需求。

結論：8 個子步驟中 6 個已有等價，僅 metadata 提取和 bookCount 是新增。

#### 對照 2：checkStaleness() — 全新功能，既有系統無對應

#### 對照 3：executeImport() vs ImportFlowController.execute()

步驟 1(parse) → ContentParser.parse() 已有。
步驟 2(staleness) → 新需求。
步驟 3(來源分流) → ContentParser._extractBooksFromData() 已有。
步驟 4(合併持久化) → tagStorageAdapter.mergeAllData/replaceAllData 已有。
步驟 5(合併摘要) → 新需求。
步驟 6(last_imported_at) → 新需求。

結論：6 個步驟中 3 個已有等價，3 個是新增。

### 實驗發現

已驗證事實：
1. ContentParser 已是完整的 parse+validate+extract 管線，支援 JSON 四來源和 CSV
2. BookFileImporter 是 ContentParser+FileValidator+FileContentReader 的 orchestrator
3. ImportFlowController 綁定 overview DOM modal，popup 不可複用此層
4. popup 目前完全無匯入功能引用
5. src/import/ 目錄不存在

核心判斷：
- 解析/驗證/轉換/持久化層（ContentParser/FileValidator/BookFileImporter/mergeAllData）是 context-free 的，popup 可直接複用
- UI 流程層（ImportFlowController 的 modal DOM 操作）是 context-bound 的，popup 不可複用

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 結論：增量擴充既有系統（方案 B），不新建平行系統

### 三方案比較

| 方案 | 描述 | 優勢 | 劣勢 |
|------|------|------|------|
| A. 新建 src/import/json-importer.js | W1-002 原規格 | 無依賴衝突 | 重複 ContentParser 80% 功能；雙匯入系統維護成本高 |
| B. 擴充既有系統 | popup 複用 BookFileImporter + 新增 staleness/metadata 層 | DRY；修 bug 一處生效 | 需跨目錄引用 popup->overview/import |
| C. 抽取共用 src/import-core/ | 把 ContentParser 等移至共用目錄 | 消除跨目錄引用 | 重構範圍大，本版本不必要 |

選擇方案 B 理由：
1. 跨目錄引用在 Chrome Extension esbuild bundle 環境下無成本
2. 方案 C 的重構可延後至 v2.0 多書城時
3. 方案 A 違反 DRY，format-version-detector 修 bug 需改兩處

### W1-002 規格修訂方向

#### 保留（新建）

| 新功能 | 位置建議 | 理由 |
|--------|---------|------|
| staleness check | popup 控制器層或獨立 helper | 純新功能，與解析/合併正交 |
| metadata 提取 | 擴充 ContentParser 或獨立 helper | canonical 格式的 metadata 既有系統缺失 |
| 合併摘要計算 | 擴充 mergeAllData 回傳值或外部包裝 | 既有 mergeAllData 只回傳 success/counts |
| last_imported_at 持久化 | popup 控制器或 sync-metadata-manager | 本機行為記錄 |
| popup UI | src/popup/ | popup 專屬 |

#### 取消（改為複用既有）

| W1-002 原規格 | 改用 | 理由 |
|--------------|------|------|
| json-importer 的 JSON 解析 | BookFileImporter.parseContent() | 完全等價 |
| json-importer 的 detectInterchangeSource | ContentParser 內部已呼叫 | 完全等價 |
| json-importer 的書籍驗證 | ContentParser._filterValidBooks() | 完全等價 |
| json-importer 的來源分流轉換 | ContentParser._extractBooksFromData() | 完全等價 |
| json-importer 的 mergeAllData 呼叫 | tagStorageAdapter.mergeAllData | 完全等價 |

#### 修訂後模組架構

popup 匯入控制器 (新建，簡化版)
  +-- BookFileImporter (複用 src/overview/)
  |     +-- FileValidator.validate(file)
  |     +-- FileContentReader.read(file) -> ImportResult
  |     +-- ContentParser.parse(content, json) -> ImportResult
  +-- StalenessChecker (新建)
  |     +-- checkStaleness(exportedAt) -> isStale
  +-- tagStorageAdapter.mergeAllData (複用)
  +-- last_imported_at 持久化 (新建)

### 防舊蓋新邏輯放置位置

建議放在 popup 匯入控制器層，理由：
1. staleness check 需讀 chrome.storage.local（async I/O），ContentParser 是純函式——加入破壞純度
2. staleness check 是業務流程決策，不是資料解析邏輯
3. overview 匯入目前無 staleness 需求
4. 未來 overview 若也需要，可將 StalenessChecker 提取為共用模組
