## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### Context Bundle

**目標**：將 popup-ui-components.js 的 initializeElements() 從 getElementById 查找靜態 HTML 改為使用 ui-factory 動態建立 DOM 元素，並配套改寫測試。

**現況分析**：
- `popup-ui-components.js` 的 `initializeElements()` 用 `getElementById` 查找 popup.html 中硬編碼的 DOM 元素
- `popup-ui-components.test.js` 的 `beforeEach` 建立靜態 JSDOM HTML（複製 popup.html 結構），28 個 test 全依賴此契約
- W2-003.2 已讓匯入卡片改用 ui-factory 動態建立（popup.js 的 buildImportPanelElements）

**重構策略**：
1. `popup-ui-components.js`：initializeElements() 改為 import ui-factory 函式，呼叫 createStatusIndicator / createProgressSection / createResultsSection / createErrorSection 建立元素並掛載
2. 保持所有公開 API 不變（updateStatus / showProgress / updateProgress / showResults / showError / hideProgress / hideResults / hideError / resetAll / updateUI / setErrorHandlers / setAccessibilityLabels）
3. 向後相容層：若 DOM 中已有元素（getElementById 找到），優先使用；找不到才呼叫 ui-factory 建立（漸進式遷移，避免 popup.html 還有殘留引用時 break）

**測試改寫策略**：
- beforeEach 不再建立靜態 HTML，改為提供空容器 + 讓 PopupUIComponents 自行用 ui-factory 建立
- 或：mock ui-factory 讓測試控制產出的 DOM 結構
- 斷言邏輯不變（驗證 updateStatus/showProgress 等行為正確性）

**技術約束**：
- ui-factory.js 是 CommonJS 匯出，popup-ui-components.js 也是 CommonJS，可直接 require
- 公開 API 不可 breaking change
- popup.html 中仍有 progressContainer / resultsContainer / errorContainer 的 HTML（W2-003.2 只移除了匯入卡片），需考慮漸進式過渡

**參考檔案**：
- `src/popup/components/ui-factory.js`：元件工廠（6 函式）
- `src/popup/popup-ui-components.js`：現有管理器（489 行）
- `tests/unit/popup/popup-ui-components.test.js`：現有 28 測試
- `src/popup/popup.html`：現有 HTML 結構（含殘留的 status/progress/results/error 區塊）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 實作

1. `popup-ui-components.js`：
   - 頂層 `require('src/popup/components/ui-factory')`（與 popup.js 一致，esbuild `src/` alias + Jest moduleNameMapper 皆可解析）。
   - `initializeElements()` 拆為四個 resolver（`_resolveStatusElements` / `_resolveProgressElements` / `_resolveResultsElements` / `_resolveErrorElements`）。
   - 漸進式策略：各 resolver 先 `getElementById` 找靜態 HTML，存在則沿用（production popup.html 仍含這些區塊，零 breaking change）；不存在才以 ui-factory 動態建立、掛載到 `document.body`，並為內部元素補上既有契約 id（statusDot / progressBar / extractedBookCount 等），使既有公開 API 與 `setAccessibilityLabels` 邏輯維持有效。
   - 公開 API（updateStatus / showProgress / updateProgress / showResults / showError / hide* / resetAll / updateUI / setErrorHandlers / setAccessibilityLabels）全數不變。

2. `popup-ui-components.test.js`：
   - `beforeEach` 移除靜態 HTML 契約，改為空 body，迫使 PopupUIComponents 走 ui-factory 建立路徑。
   - 改用 Jest jsdom 環境的全域 `document`（取代另建獨立 JSDOM 實例），確保 ui-factory（以全域 document 建立）與建構子傳入 document 同一 realm，避免 cross-realm appendChild 失敗。
   - 斷言邏輯不變（仍以 getElementById 取得元素驗證行為正確性）。

### 向後相容性

公開 API 簽章與行為完全保留；production 透過 getElementById-first 沿用 popup.html 靜態元素，無 breaking change。
