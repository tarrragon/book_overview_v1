### 1. 系統一致性檢查

**命名規範**：ticket 規劃 `src/sync/json-importer.js`，建議改為 `src/import/json-importer.js`（匯入是單向操作，sync 語意過重；且與 `src/export/` 形成對等目錄）。函式命名（`importBooks` / `validateSyncMeta`）符合既有 camelCase 慣例。事件命名應對齊 `src/export/export-events.js` 的模式。

**架構模式**：匯入流程應透過 EventBus 發送進度/完成/錯誤事件。匯入為單次操作（非並發管理），不需要對等的 Manager 層，直接模組即可。格式偵測必須複用 `format-version-detector.js` 的 `detectInterchangeSource()`。

**依賴方向**：json-importer → format-version-detector / book-interchange-v1-adapter / tag-storage-adapter，方向皆正確（domain 內 + domain→storage）。popup → json-importer 為 presentation → domain，正確。

### 2. 重複實作檢查

**可直接複用**：
- `detectInterchangeSource()` — 四來源偵測
- `mapCanonicalToV1Book()` — canonical → V1 v2 read 方向映射（含多值收斂 + passthrough）
- `convertAppLegacyToV2Book()` — APP legacy 止血
- `mergeAllData()` — 三 key 原子合併 + rollback（含 `computeMergeResult` 純函式：id LWW + tagIds 聯集 + category/tag 同名收斂）
- `book-validation-helpers.js` — 驗證工具

**需新建**：
1. File API 讀取 + JSON parse + metadata 驗證
2. 防舊蓋新邏輯（exported_at vs last_imported_at）
3. 匯入 UI（popup 檔案選取按鈕 + 結果摘要）
4. 匯入結果摘要包裝（mergeAllData 回傳 counts，需加「新增/更新/不變」分類）

**結論**：重複造輪子風險低。核心合併邏輯已完整實作。新建工作集中在膠水層。

### 3. sync_meta 與 spec metadata 衝突

PROP-012 §2.3 sync_meta 使用 snake_case（`exported_at`），book-interchange-v1 spec §3 metadata 使用 camelCase（`exportedAt`）。兩者描述同一 JSON 匯出檔案。

**建議**：以 spec v3.0.0 的 `metadata` 為 SSOT。`source_device` / `last_sync_summary` 若需要，擴充 spec 的 metadata 而非建立平行 sync_meta 物件。

### 4. Chrome Extension Popup 限制

- `<input type="file">` 在 popup 中可用
- Drag-and-drop 不建議實作：popup 寬度僅 350px，拖放過程 popup 可能因失焦而關閉
- 建議簡化為檔案選取按鈕

### 5. 系統衝突檢查

| checklist | 判定 | 說明 |
|-----------|------|------|
| 5.1 跨模組依賴衝突 | 無衝突 | 匯入模組為 export/storage 公開 API 的純消費者，不修改其介面 |
| 5.2 既有 UC 變動影響 | 無衝突 | 匯入為全新功能，不修改 UC01-UC07 |
| 5.3 跨版本相容性 | 無衝突 | 消費既有 canonical v3.0.0，last_imported_at 為新增 key 向後相容 |

### 6. 審查結論：通過（有 4 項建議）
