## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

popup `checkBackgroundStatus()` 對 Service Worker 採**單次** `chrome.runtime.sendMessage(GET_STATUS)` + 固定 2 秒 `Promise.race` timeout，失敗後 catch 直接 `updateStatus('離線')` 並 `return`，無重試、無過渡態。

SW cold-start 受系統 I/O 影響（磁碟壓力/高負載/低階機器），可能 > 2 秒。此時 popup 永久顯示「離線」且不自動恢復（需手動重開 popup）。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `src/popup/popup.js` | `checkBackgroundStatus()` L695-723、`createBackgroundTimeoutPromise()` L617-629、`initialize()` L1100-1116 |
| `src/background/background.js` | GET_STATUS handler（需確認 init 期間行為） |

### 現有程式碼關鍵路徑

1. `initialize()` 步驟 6 呼叫 `checkBackgroundStatus()`
2. `checkBackgroundStatus()` 用 `Promise.race([sendMessage, 2秒timeout])`
3. 失敗 → catch → `updateStatus('離線')` → `return false`
4. `initialize()` 收到 `false` → `errorHandler.handleInitializationError()` → **流程終止，永不重試**

### WRAP 三問

- **W（Widen）**：(1) A+C 方案（重試+即時 baseline）(2) 純延長 timeout 到 5 秒 (3) SW wake-up 預觸發（popup open 事件先 ping 再等）
- **A（Attain distance）**：P2 優先級，不擠壓 P0/P1 任務；1.1.0 已無更高優先級待處理
- **P（Prepare to be wrong）**：最可能失敗原因：重試邏輯在測試環境模擬 cold-start > 2 秒較困難（Jest mock timer vs real async）；防護：Phase 2 先設計 mock 策略
