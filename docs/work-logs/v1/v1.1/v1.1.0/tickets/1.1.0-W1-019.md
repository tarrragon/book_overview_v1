## Task Summary

強化 popup↔SW cold-start 握手 robustness（單次無重試+固定2秒 → 重試/init期間即時baseline，消除極端I/O下永久離線）

## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

popup `checkBackgroundStatus()` 對 Service Worker 採**單次** `chrome.runtime.sendMessage(GET_STATUS)` + 固定 2 秒 `Promise.race` timeout，失敗後 catch 直接 `updateStatus('離線')` 並 `return`，無重試、無過渡態。

SW cold-start 受系統 I/O 影響（磁碟壓力/高負載/低階機器），可能 > 2 秒。此時 popup 永久顯示「離線」且不自動恢復（需手動重開 popup）。

### 影響範圍

| 檔案 | 影響 |
|------|------|
| `src/popup/popup.js` | `checkBackgroundStatus()` L695-723、`createBackgroundTimeoutPromise()` L617-629、`initialize()` L1100-1116 |
| `src/background/background.js` | GET_STATUS handler（需確認 init 期間行為） |

### 現有程式碼關鍵路徑

1. `initialize()` 步驟 6 呼叫 `checkBackgroundStatus()`
2. `checkBackgroundStatus()` 用 `Promise.race([sendMessage, 2秒timeout])`
3. 失敗 → catch → `updateStatus('離線')` → `return false`
4. `initialize()` 收到 `false` → `errorHandler.handleInitializationError()` → **流程終止，永不重試**

### WRAP 三問

- **W（Widen）**：(1) A+C 方案（重試+即時 baseline）(2) 純延長 timeout 到 5 秒 (3) SW wake-up 預觸發（popup open 事件先 ping 再等）
- **A（Attain distance）**：P2 優先級，不擠壓 P0/P1 任務；1.1.0 已無更高優先級待處理
- **P（Prepare to be wrong）**：最可能失敗原因：重試邏輯在測試環境模擬 cold-start > 2 秒較困難（Jest mock timer vs real async）；防護：Phase 2 先設計 mock 策略

### Phase 3b 首次派發根因分析（PM 診斷）

10 個測試全部 30 秒 timeout（hang），非斷言失敗。根因：

1. **microtask flush 不足**：`attemptGetStatus` 內部用 `Promise.race([msgPromise, timeoutPromise]).then().catch()` — 當 `jest.advanceTimersByTime` 推進 setTimeout 時，timeout reject 觸發，但 `.catch()` callback 是 microtask，需要額外 `await Promise.resolve()` 才能讓 callback 執行並呼叫外層的 resolve。測試只做了 1-2 次 flush，不夠走完 callback → retryHandshake → setTimeout → advanceTimersByTime → callback 的完整鏈。

2. **`isFinalStatus` 未宣告**：`checkBackgroundStatus` 中使用了 `isFinalStatus = true` 但此變數未在函式外宣告，可能導致隱式全域變數或 strict mode 下拋錯。

**修正方向**：不改測試（Phase 2 RED 測試是規格）。實作需配合測試的 fake timer + microtask flush 模式。兩種可行路徑：
- (A) 改回 async/await 風格（與測試的 advanceTimersByTime + await Promise.resolve 模式匹配）
- (B) 保持 callback 風格但在每個 timer callback 中確保只用 macrotask（純 setTimeout 鏈，不混 Promise）

推薦 (A)：與測試預期一致、程式碼可讀性更好。

#### Stage 5：決策

**選定方案：A+C 組合**（加權總分 A=7.45 + C=7.90，互補覆蓋）

**理由**：
1. C 方案從根源解決：init 期間 popup 發 GET_STATUS 能立即得到 baseline 回應（mode: initializing），popup 顯示「初始化中」過渡態而非「離線」
2. A 方案補強：popup 收到 initializing 後啟動輪詢重試，直到收到 mode: normal
3. B 方案（延長 timeout）不採納：只延後問題，5 秒 timeout 在極端 I/O 下仍可能被超越
4. D 方案不採納：乾淨環境 cold-start 已小於 200ms，最佳化無意義

**關鍵發現**：background.js 頂層 listener（L257）已有 init 期間回應 GET_SYSTEM_STATUS 的能力，但 popup 發的是 GET_STATUS（不同訊息類型）。init 期間 GET_STATUS 被 bufferMessageDuringInit 緩衝，popup 2 秒 timeout 到期前不會收到回應。C 方案核心改動是讓頂層 listener 同時攔截 GET_STATUS 提供 baseline。

### 功能規格（A+C 組合方案）

#### Purpose

消除 popup 與 SW cold-start 握手在極端 I/O 下的永久離線問題。popup 在 SW init 期間能收到 initializing baseline 並顯示過渡態，同時透過重試機制等待 SW 就緒後自動恢復為「線上」。

#### 1. Background 側：頂層 listener 增加 GET_STATUS 處理（C 方案核心）

**現狀**：頂層 listener（background.js L257）只認 GET_SYSTEM_STATUS，GET_STATUS 在 init 期間被 bufferMessageDuringInit 緩衝（popup 2 秒 timeout 到期前不會收到回應）。

**改動**：頂層 listener 在 init 期間（isInitialized 為 false 且非 emergencyMode）增加 GET_STATUS 的 baseline 回應。

輸入：`{ type: 'GET_STATUS' }`

輸出：
```json
{
  "success": true,
  "isEnabled": true,
  "serviceWorkerActive": true,
  "initializing": true,
  "mode": "initializing",
  "timestamp": "(Date.now())"
}
```

**設計考量**：
- 回應格式與 MessageRouter.handleGetStatusMessage 相容（success / isEnabled / serviceWorkerActive），popup 現有 response.success 判斷不需改動
- 新增 initializing: true 和 mode: initializing 欄位，popup 可據此區分「就緒」與「初始化中」
- GET_STATUS baseline 回應後 return true，不緩衝（與 GET_SYSTEM_STATUS 處理一致）
- init 完成後（isInitialized 為 true），頂層 listener 回傳 undefined 不攔截，GET_STATUS 由 MessageRouter 正常路由

#### 2. Popup 側：checkBackgroundStatus 增加 initializing 狀態處理 + 重試（A+C 核心）

**現狀**：
- checkBackgroundStatus() 單次嘗試，成功/失敗二元
- 失敗時 updateStatus('離線') 並 return false，initialize 流程終止

**改動**：

函式簽名不變：checkBackgroundStatus() 回傳 Promise(boolean)

新增內部行為：
1. 第一次嘗試：sendMessage GET_STATUS + Promise.race 2 秒 timeout
2. 若收到 response.success 為 true：
   - 若 response.initializing 為 true：updateStatus('初始化中') 並進入重試迴圈
   - 若 response.initializing 非 true：handleSuccessfulBackgroundResponse(response) 並 return true（現有邏輯）
3. 若 timeout 或 catch error：進入重試迴圈

重試迴圈規格：
- MAX_RETRY_ATTEMPTS = 3
- 退避策略：指數退避（1 秒、2 秒、4 秒）
- 每次重試：sendMessage GET_STATUS + Promise.race 2 秒 timeout
  - 收到 success 且非 initializing：成功，return true
  - 收到 success 且 initializing：更新狀態「初始化中」，繼續重試
  - timeout / error：繼續重試
- 全部重試失敗：updateStatus('離線') 並 return false

**常數定義**：
```javascript
const HANDSHAKE_CONFIG = {
  TIMEOUT_MS: 2000,
  MAX_RETRY_ATTEMPTS: 3,
  INITIAL_RETRY_DELAY_MS: 1000,
  RETRY_BACKOFF_MULTIPLIER: 2
}
```

#### 3. Popup 側：新增「初始化中」過渡態顯示

收到 initializing 回應時：
```
updateStatus('初始化中', '背景服務正在啟動...', '系統初始化中，請稍候', STATUS_TYPES.LOADING)
```

若 STATUS_TYPES 無 LOADING，使用 STATUS_TYPES.WARNING 或由 Phase 3b 確認 UI 層。

#### 狀態轉換圖

```
popup 開啟
  |
  v
[GET_STATUS 發送至 SW]
  |
  +-- SW 已就緒 --> response.success 且非 initializing --> 顯示「線上」, return true
  |
  +-- SW init 中 --> response.success 且 initializing --> 顯示「初始化中」
  |   |
  |   v
  |   [重試 1: 延遲 1 秒]
  |   +-- SW 就緒 --> 顯示「線上」, return true
  |   +-- 仍 init / timeout
  |       |
  |       v
  |       [重試 2: 延遲 2 秒]
  |       +-- SW 就緒 --> 顯示「線上」, return true
  |       +-- 仍 init / timeout
  |           |
  |           v
  |           [重試 3: 延遲 4 秒]
  |           +-- SW 就緒 --> 顯示「線上」, return true
  |           +-- 全部失敗 --> 顯示「離線」, return false
  |
  +-- 首次 timeout / error --> 顯示「初始化中」--> 進入重試迴圈（同上）
```

最大等待時間：2 + 1 + 2 + 2 + 2 + 4 + 2 = 15 秒（首次 2s timeout + 3 次重試各含延遲+timeout）。

#### 邊界條件

| 條件 | 預期行為 |
|------|---------|
| SW cold-start 小於 2 秒（正常場景） | 首次嘗試即成功，無重試，行為與現有完全一致 |
| SW cold-start 2-4 秒 | 首次 timeout 後重試 1 成功 |
| SW cold-start 4-8 秒 | 重試 1-2 可能失敗，重試 2-3 成功 |
| SW cold-start 超過 15 秒 | 全部重試失敗，顯示「離線」（合理降級） |
| SW emergency 模式 | 頂層 listener emergency 分支回 success:false + 受限提示，popup 進入重試後離線 |
| SW 完全無回應（擴展未載入） | 每次 timeout，3 次重試後離線（行為一致但延遲更長） |
| init 期間 popup 多次開啟 | 每個 popup 獨立重試，互不影響 |
| 重試期間用戶關閉 popup | Promise 中斷，無副作用 |

### 行為場景（Given-When-Then）

場景 1: 正常握手（happy path，無變更）
- Given: SW 已完成初始化（isInitialized 為 true）
- When: popup 開啟並發送 GET_STATUS
- Then: SW 由 MessageRouter 正常回應 success:true，popup 顯示「線上」

場景 2: SW init 中但快速就緒
- Given: SW 正在初始化（isInitialized 為 false）
- When: popup 開啟並發送 GET_STATUS
- Then: SW 頂層 listener 回 initializing baseline，popup 顯示「初始化中」
- When: 1 秒後 popup 重試 GET_STATUS，此時 SW 已就緒
- Then: SW 由 MessageRouter 正常回應，popup 更新為「線上」

場景 3: SW init 極慢（超過 2 秒）
- Given: SW cold-start 需 5 秒（極端 I/O）
- When: popup 開啟，首次 GET_STATUS timeout
- Then: popup 顯示「初始化中」
- When: 重試 1（1 秒後）、重試 2（2 秒後）仍在 init
- Then: 重試 3（4 秒後）SW 已就緒，popup 更新為「線上」

場景 4: SW 完全無回應
- Given: SW 未載入或擴展已失效
- When: popup 發送 GET_STATUS，首次 timeout
- Then: popup 顯示「初始化中」並重試 3 次
- When: 全部重試 timeout
- Then: popup 顯示「離線」並觸發 handleInitializationError

場景 5: SW emergency 模式
- Given: SW 進入 emergency 模式
- When: popup 發送 GET_STATUS
- Then: SW 頂層 listener emergency 分支回 success:false + 受限提示
- Then: popup catch 到非成功回應，進入重試後全部失敗，顯示「離線」

### 跨模組共用決策

Q1: GET_STATUS 與 GET_SYSTEM_STATUS 訊息類型不統一。GET_STATUS 是 popup 到 SW 的握手訊息（由 MessageRouter 路由）；GET_SYSTEM_STATUS 是系統狀態查詢（由頂層 listener 處理）。兩者語意不同，但在 init 期間需讓頂層 listener 同時攔截 GET_STATUS 提供 baseline。

Q2: initializing 回應格式需跨模組相容。background 的 initializing 回應需與 MessageRouter.handleGetStatusMessage 的回應格式相容（success / isEnabled / serviceWorkerActive），popup 側新增 initializing 欄位判斷。

### 非功能性需求

Q5（效能）：正常場景無額外延遲；最大等待時間 15 秒（3 次重試全失敗）；無新 dependency。

Q6（相容性）：向後相容現有 GET_STATUS 訊息協議；SW 就緒後行為與現有完全一致；不影響 init 期間的訊息緩衝機制（W2-006.1）。

#### 四核心問題

**1. 重構動機：是否需要重構？**
