### 問題根因

W14-032 ANA 將 hook effort 感知遷移分為 A 類（strict-validator，高 ROI）和 B 類（observability，medium ROI）。A 類已由 W14-034/036/037 完成遷移（共 16 個 hook）。本 ticket 需量測 A 類落地效果，再決定 B 類（約 15 個 hook）是否值得遷移。

### 影響範圍

`.claude/hooks/` 目錄下所有 hook 檔案

### WRAP 三問 + R

- **W**：(1) 量測後決策（本方案）(2) 直接遷移全部 B 類（無數據支撐，風險高）
- **R**：需實際量測 A 類遷移前後 token 差異——查 hook-logs 或設計量測實驗
- **A**：P3 優先級，不擠壓任何高優先任務
- **P**：最可能問題：hook-logs 無足夠歷史數據做前後對比；防護：用當前 session 即時量測替代歷史對比

## 重現實驗結果
### 實驗方法

對 A 類（已遷移 effort 感知）和 B 類（未遷移）hook 進行靜態分析：

### 實驗發現

已驗證事實：
1. A 類 9 個 hook 已成功遷移 effort 感知，low effort 時短路放行
2. B 類 16 個候選 hook 均無 effort 感知（effort_aware: 0）
3. B 類 token 成本以 parallel-dispatch-verification（12 呼叫）和 ticket-skill-sync-check（10 呼叫）最高
4. B 類 hook 觸發頻率低於 A 類（多為 PostToolUse/Stop event，非每 turn 觸發）
5. B 類多為 observability/reminder 類——low effort 放行不影響正確性，但會損失觀察力

仍未驗證假設：
- 實際 low effort 使用頻率（決定節省是否有意義）
- hook stderr 輸出的實際 token 數（靜態估算 vs 動態量測可能有差異）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 結論：不遷移 B 類 medium ROI 子集

**建議：不遷移**

#### 理由

1. **節省量不顯著**：B 類每 session 估算節省 5000-15000 tokens，佔 session 總消耗（~500K-1M tokens）的 1-3%。與 A 類的 7500-18750 tokens 接近，但 B 類觸發頻率更低（多為 PostToolUse/Stop event），實際節省偏向下限。

2. **觀察力損失風險**：B 類 hook 的核心價值是 observability（派發驗證、並行衝突偵測、commit 規範檢查）。low effort 放行等於放棄這些觀察力。A 類是 strict-validator（阻擋性質），放行只是降低誤報；B 類是 reminder/observability（提醒性質），放行等於靜默掉提醒。

3. **遷移成本 vs 收益不對稱**：16 個 hook 遷移需 ~225 行改動 + 測試 + 審查 = 1 個 IMP ticket。收益（1-3% token 節省）不足以償還成本。

4. **low effort 使用頻率不明**：目前無 telemetry 追蹤 low effort 使用率。若 low effort 佔 session 的 10%，實際節省再打九折。

#### 與 W14-032 原始評估一致性

W14-032 原始評估：「B 類 low ROI 子集單 hook token 50-300，遷移成本高於收益」。本次量測驗證此評估正確——medium ROI 子集平均每 hook ~228 tokens/觸發，加上觸發頻率低，整體 ROI 不足以支撐遷移。

#### 處置

| 項目 | 決策 |
|------|------|
| B 類 medium ROI 16 個 hook | 不遷移 effort 感知 |
| 不 spawn IMP-B | 無需建 ticket |
| 顯性豁免 | 與 W14-032 的「類別 C + B low ROI 不遷移」合併，B medium ROI 亦不遷移 |

無需建後續 IMP ticket。
