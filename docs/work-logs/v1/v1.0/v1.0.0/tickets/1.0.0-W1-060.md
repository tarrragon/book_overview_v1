### AC-1：受眾錯配 Hook 盤點清單

### 分類框架

Hook 輸出注入 LLM context 的通道有三種：

| 通道 | 機制 | 受眾 |
|------|------|------|
| hookSpecificOutput.additionalContext | PostToolUse/SubagentStop JSON stdout | 觸發該 event 的 LLM（PM 或 subagent） |
| systemMessage | Stop JSON stdout top-level 欄位 | 觸發 Stop 的 LLM（PM 或 subagent） |
| stderr | 不注入 LLM context，僅寫入日誌/終端 | 人類（用戶終端可見） |

**關鍵事實**：Stop event 在 subagent 結束時亦觸發（subagent 也有自己的 Stop lifecycle），因此 Stop hook 的 systemMessage 會注入 subagent context，除非 hook 自行偵測 subagent 環境並跳過。

### 受眾錯配清單（需修復）

**出口方向（Stop hook 訊息誤注入 subagent context）**：

目前 Stop hook 中注入 context 的 2 個 hook（handoff-auto-resume-stop / stop-worklog-handoff-sync-check）已在 W1-024/W1-030/W1-044 修復完畢，具備 subagent 偵測。**無殘留出口污染 hook**。

但出口吞沒問題仍存在：即使 Stop hook 不注入 systemMessage，Stop event 本身的 lifecycle 處理（多個 hook 的 JSON stdout 合併）仍佔用 subagent 最後回合的 token 預算，壓縮 final-message 空間。此為 CC runtime 層級問題，非單一 hook 可解。

**入口方向（PostToolUse hook 缺 subagent 偵測，訊息誤注入 subagent context）**：

| Hook | 訊息內容 | 受眾意圖 | 污染嚴重度 |
|------|---------|---------|-----------|
| commit-msg-layer2-marker-check（L232） | commit 訊息格式提醒 | PM-only | 低（資訊性，不含指令） |
| post-test-hook（L477） | 測試結果分析與建議 | PM-only | 中（可能含「建議重跑」等動作提示） |
| needs-context-listener-hook（L97） | NeedsContext 偵測結果 | PM-only | 中（含 ticket 操作提示） |
| ticket-skill-sync-check-hook（L281） | ticket CLI 行為同步提醒 | PM-only | 低（資訊性） |

### AC-2：雙向污染機制分析

### 出口方向：subagent final-message 吞沒

**機制**：

1. Subagent 完成最後一個工具呼叫後，CC runtime 觸發 Stop event
2. 註冊在 Stop 的 7 個 hook 依序執行，各自可輸出 JSON stdout
3. 若 hook 輸出含 systemMessage，該文字注入 subagent 的最後 context
4. Subagent 基於最後 context 生成 final-message（回報 PM 的結論）
5. 注入的 systemMessage 佔用 token 預算，壓縮 final-message 可用空間

**本案證據**：五委員審查輪 5/5 報告本體被擠掉（ticket why），PM 需 transcript 手術提取。

**已有防護**：W1-044 / W1-030 讓兩個關鍵 Stop hook 偵測 subagent 並跳過注入。但吞沒的根因不僅是 systemMessage 注入——Stop lifecycle 本身的 hook 執行開銷和輸出合併也佔 token。

**殘留風險**：中等。目前 Stop hook 不再注入 systemMessage 到 subagent，但 CC runtime 層的 Stop event 處理開銷仍存在。若未來新增 Stop hook 且含 systemMessage 且忘記加 subagent 偵測，問題會重現。

### 入口方向：PM-only 訊息誤導 subagent 行為

**機制**：

1. Subagent 執行 Bash 工具（git commit / npm test 等）
2. PostToolUse:Bash hook 觸發，其中 4 個 hook 無 subagent 偵測
3. hook 輸出的 additionalContext 注入 subagent context
4. Subagent 讀到「PM 立即動作」「建議 git merge」等指令性文字
5. 唯讀型 subagent（如 Explore 委員）被誘導執行寫入操作

**本案證據**：唯讀審查委員（Explore）被注入訊息中「給 PM 的建議動作」誘導，實際執行 git merge 越界（ticket why，WRITE-CMD 證據）。

**根因**：hook 開發時未考慮 subagent 會觸發同樣的 PostToolUse event。`is_subagent_environment()` helper 已存在（hook_utils/hook_io.py L451），但 4 個 hook 未使用。

**嚴重度**：高。指令性訊息直接誘導 subagent 越界——資訊性訊息（如 commit 格式提醒）風險低，但動作性訊息（如「建議 git merge」）風險高。agent-commit-verification 的 PM Action Summary 含 `git merge` / `cd` / `npm test` 明確指令，是主要誘導源。但此 hook 在 SubagentStop（僅 PM 端）觸發，非 PostToolUse，故此處入口污染的 4 個 hook 中 needs-context-listener 和 post-test-hook 的動作性提示為較高風險。

### PC-183: Hook 注入訊息受眾錯配（PM-only 訊息注入 Subagent Context）

**症狀**：
- Subagent 執行後回報訊息中混入「給 PM 的建議動作」（如「建議 git merge」「確認工作目錄」）
- 唯讀型 subagent（Explore / 審查委員）執行寫入操作（git merge / git checkout）
- Subagent final-message 被 Stop hook 訊息擠壓，報告本體遺失
- PM 需從 transcript 手動提取 subagent 報告

**根因**：
Hook 輸出透過 additionalContext / systemMessage 注入 LLM context，但未區分受眾：

1. **PostToolUse hook 缺 subagent 偵測**：PM-only 訊息（commit 格式提醒、測試結果分析、ticket sync 提醒）注入 subagent context，subagent 將其視為自身任務指令
2. **Stop hook systemMessage 在 subagent 觸發**：Stop event 在 subagent 結束時亦觸發，systemMessage 注入 subagent 最後 context 而非 PM（已由 W1-044/W1-030 修復兩個關鍵 hook，但新 hook 可能重蹈覆轍）
3. **hook 輸出無受眾標記（audience marker）機制**：CC runtime 層不支援「此訊息僅限 PM」的語義，所有 additionalContext 對觸發方的 LLM 一視同仁

**案例**：
- 五委員審查輪 5/5 報告本體被 Stop hook 訊息擠掉（出口方向）
- Explore 唯讀委員被注入「git merge」建議後實際執行 merge（入口方向）

**防護**：

A. **短期（IMP）**：4 個缺 subagent 偵測的 PostToolUse:Bash hook 加入 `is_subagent_environment()` 早期跳過

B. **中期（IMP）**：建立 hook 開發 checklist 規則——任何輸出 additionalContext / systemMessage 的 hook 必須評估 subagent 受眾適切性，新 hook PR 審查強制項

C. **長期（PROP）**：評估 CC runtime 層級的受眾標記機制（audience marker），或 AGENT_PRELOAD 教 subagent 忽略特定前綴的注入訊息
