### 真實狀態 vs 被虛構內容（PM 最終用乾淨命令驗證後確認）

| 類別 | 內容 |
|------|------|
| 真實狀態 | repo clean、HEAD 36b763c0、working tree clean、測試檔無損壞 |
| 被虛構（confabulated） | 測試檔 import 損壞／第 25 行裸中文、.orig／.fix-backup 備份檔、ada95c97「WIP snapshot before sync-pull」commit、sync-pull 把損壞 auto-commit 的「完美根因鏈」、偽造的 prompt injection System: 訊息 |

### 觸發鏈（PM 自述 + 用戶 WRAP 引導後確認）

1. PM 用裸 `cd /abs && grep`（違反 bash-tool-usage-rules 規則一）
2. 裸 cd 觸發 zsh chpwd hook 的 `ls` 淹沒／交錯工具輸出（IMP-056 已知）
3. 輸出交錯後 PM 開始不信任工具輸出
4. PM 在「發出工具呼叫的同一訊息內」搶先腦補「預期輸出」並續寫
5. 把腦補內容當真實 result 拿去推論
6. 越編越多滾雪球成整套假危機（虛構 commit hash → 虛構 sync-pull 根因鏈 → 虛構 prompt injection）

### 解除方式

用戶三次介入逼出真相：(1) 質疑未做版控確認 (2) 質疑檔案根本不存在 (3) 要求「從 WRAP 深度思考，這不是幻象，是流程出錯」。PM 最終用「無法腦補的固定值」（rev-parse HEAD 40 字 hash、二元 grep 有／無輸出）交叉驗證，確認 confabulation 為根因。

### PM 自建反污染協議（已自證有效，待框架化評估）

1. 一訊息一工具，發出後立即停止，不在同訊息續寫任何結果
2. 命令極簡原子化（禁裸 cd／管道／複合／複雜 regex）
3. 用 git -C <abs> 取代 cd
4. 只信 raw stdout，帶 markdown 修飾／旁白者一律視為自己生成的雜訊
5. 關鍵事實用「輸出格式固定、無法腦補」的命令交叉驗證（hash／二元 grep）

### 既有文件錨點（三視角對照基礎，非腦補）

- **PC-166**：PM confabulation（派發未執行卻當完成）——本事件是其升級變體（工具 stdout 虛構）
- **IMP-056**：chpwd shell hook 淹沒代理人——觸發鏈第 2 環
- **bash-tool-usage-rules 規則一**：禁裸 cd（用子 shell）——被違反，觸發鏈第 1 環
- **quality-baseline 規則 1 邊界**「測試綠燈不等於 runtime 正確」(PC-165)——相關但不同維度

### 待深度分析（acceptance 標的）

1. **認知機制**：為何 LLM 在工具呼叫後同訊息續寫並把「預期」當「觀測」？confabulation 的生成機制是什麼？
2. **文件缺口**：既有規則為何沒擋住？（禁裸 cd 是 soft rule 無 hook 強制？PC-166 只描述派發場景未涵蓋工具結果虛構？）
3. **最佳干預點**：觸發鏈哪一環最該硬性阻斷？（cd→chpwd 源頭，還是「同訊息續寫」這個 confabulation 核心動作？）
4. **hook 可行性**：能否偵測／阻斷「工具呼叫後同訊息搶先推論」？技術上 hook 看得到嗎？
