## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 方案選擇：Option C（移除死碼 module.exports，雙路徑改由單一 `export {}` 服務）

#### 根因
Logger.js 與 MessageDictionary.js 同時有頂層 ESM `export {}` 與 `if (module.exports) {...}` CJS 區塊。esbuild 因頂層 `export` 判定整檔為 ESM，對 `module.exports` 發出 commonjs-variable-in-esm warning（baseline=8：兩檔各 1 處 × 4 bundle 入口）。

#### 關鍵調查：module.exports 在兩條消費路徑皆為死碼
逐一驗證兩個模組系統的實際解析路徑：

| 消費路徑 | 解析來源 | module.exports 是否被用 |
|---------|---------|----------------------|
| Jest（CJS require） | Babel（.babelrc modules:'commonjs'）將 `export {}` 轉為 exports.X | 否（顯式 module.exports 冗餘，Babel 已產生 CJS 具名匯出） |
| esbuild bundle 內 require() 消費者 | esbuild `__toCommonJS(X_exports)` interop（bundle line 2149/2150/3788） | 否（走 ESM export，非 module.exports） |
| esbuild bundle 內 import 消費者（background.js） | esbuild ESM `export` | 否 |
| classic script window 掛載 | `else if (window)` 分支 | 否（window 分支獨立保留） |

結論：`module.exports` 在 bundle IIFE 中 `typeof module` 永為 falsy（死碼），在 Jest 中被 Babel 轉譯結果取代（冗餘）。移除安全。

#### 變更內容
兩檔皆將 `if (typeof module !== 'undefined' && module.exports) { module.exports = {...} } else if (typeof window !== 'undefined') { window.X = ... }` 改為 `if (typeof window !== 'undefined') { window.X = ... }`，保留頂層 `export {}` 不變。window 全域掛載（popup.js fallback chain 依賴）完整保留。加註說明三明示（Why/兩路徑機制/移除依據）。

#### 雙路徑 consumer 清單（src）
- Logger.js：44 require + 1 import（background.js ESM）
- MessageDictionary.js：10 require + 1 import（background.js ESM）
- tests：Logger 10 require、MessageDictionary 10 require（皆 CJS，經 Babel）
- 0 個 plain-node（非 Babel 非 esbuild）consumer（scripts/ 無引用），故移除 module.exports 無破壞風險。

#### bundle 健康驗證（PC-165 類 runtime 風險防護）
- bundle 內 `__toCommonJS(Logger_exports)` / `__toCommonJS(MessageDictionary_exports)` interop 仍在（require 消費者照常解析）
- bundle 內已無這兩模組的殘留 module.exports 死碼
- bundle 內 window.Logger（popup line 1060）/ window.MessageDictionary（line 654）掛載保留

#### 為何不選 Option A（logOverride silent）
logOverride 全域抑制 warning 類別，會遮蔽未來其他檔意外引入的真實 CJS-in-ESM 問題；Option C 從根本移除死碼，無此副作用。Option B（guard module.exports）無效（esbuild 對 ESM 檔內任何 module.exports 皆 warn，不論 guard）。

### 採用方案：移除 module.exports 死碼分支（方案 A 變體），保留 export 與 window 全域掛載

#### 根因確認
兩檔同時含 `module.exports = {...}`（CJS）與 `export { ... }`（ESM）。esbuild 因偵測到 top-level `export` 將檔案判定為 ECMAScript module，於是對 `module.exports` 發出 commonjs-variable-in-esm warning（module 在 ESM 被當 global，may not work as expected）。warning 非致命，但屬技術債。

#### 雙模式 consumer 盤點（兩路徑皆驗證）
- Logger.js：src 內 CJS require 44 處 + ESM import 1 處（background.js 入口）。tests 全走 `const { Logger } = require('...')`。
- MessageDictionary.js：src 內 CJS require 10 處 + ESM import 1 處（background.js）。tests 全走 require 解構。
- popup fallback chain（src/popup/popup.js 等）依賴 `window.Logger` / `window.MessageDictionary` 作為 require 失敗時的最後防線（classic <script> 過渡期路徑）。

#### 關鍵驗證：為何移除 module.exports 不破 CJS 路徑
以專案 babel.config.js（modules:'commonjs'）實機轉換驗證：
- Babel 將 `export { Logger, LOG_LEVELS }` 編譯為 `exports.Logger = ... / exports.LOG_LEVELS = ...`（並設 __esModule:true），完整滿足 tests 的 `const { Logger } = require('...')` 解構。
- 原 `module.exports = {...}` 在 Babel 轉換後反而會「覆蓋」上述 exports 物件（含 __esModule），屬冗餘且有害的 dead code。
- 因此 module.exports 在 Jest（babel CJS）與 esbuild bundle（ESM + __toCommonJS interop）兩條路徑皆為死碼。

#### 實際改動（src/core/ 兩檔）
將原本：
```
if (typeof module !== 'undefined' && module.exports) {
  module.exports = { ... }
} else if (typeof window !== 'undefined') {
  window.X = X
}
export { ... }
```
改為（移除 module.exports 分支，window 掛載獨立保留，export 不變）：
```
if (typeof window !== 'undefined') {
  window.X = X
}
export { ... }
```
- Logger.js：src/core/logging/Logger.js（移除 module.exports，保留 window.Logger / window.LOG_LEVELS）
- MessageDictionary.js：src/core/messages/MessageDictionary.js（移除 module.exports，保留 window.MessageDictionary / window.GlobalMessages）
- 兩檔均加入註解說明 CJS（babel）/ESM（esbuild）兩路徑來源與移除理由（1.0.0-W2-004）。

#### 為何不選方案 B（esbuild 視為 CJS）
方案 B 需對個別檔案改 bundle 設定，複雜且需維護白名單；方案 A 變體只需移除確認過的死碼，零設定變更、零 consumer 變更，且消除 warning 的同時保留全部三條消費路徑（Jest CJS / esbuild ESM / popup window fallback）。

#### 不在本 ticket 範圍
- 其他模組的雙模式寫法（本 ticket 僅這兩檔產生 warning，baseline=8 全來自此二檔）。
- esbuild bundle 設定變更（未動 scripts/build.js）。
