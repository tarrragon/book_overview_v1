## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

append-log 的 status precondition（W3-044，`lib/precondition.py` 的 `require_in_progress`）要求 status=in_progress 才可寫入 body，但 PC-040 / PC-100 要求 PM 於 ticket create 後（status=pending）立即將派發 context 寫入 Problem Analysis / Context Bundle。兩條規則在「派發前 bookkeeping」場景結構性衝突，PM 唯一出路是 `--force` 逃生閥——2026-06-11 單日 4 次，逃生閥常態化稀釋警示價值並累積 audit 噪音（每次需於 review 說明理由）。

### 影響範圍

- `.claude/skills/ticket/ticket_system/lib/precondition.py`（status 判定主邏輯）
- `.claude/skills/ticket/ticket_system/commands/track_acceptance.py`（append-log 呼叫端）
- 測試：`tests/test_require_in_progress.py`、`tests/test_cli_append_log_precondition.py`、`tests/test_append_log_section_autocreate.py`
- 文件：`SKILL.md`、`references/track-command.md`、`.claude/pm-rules/context-bundle-spec.md`

### 相關 Error Pattern

PC-040（context 存 ticket 不存 prompt）、PC-100（create 後立即寫派發 context）為本摩擦的上游規則來源；W1-025（commit fa647243）已解「缺失章節被迫 Edit」的一半摩擦，本 ticket 解 status precondition 的另一半。

### 簡化 WRAP 三問

- W（其他做法）：(1) `--bookkeeping` 旗標——仍需記旗標，摩擦未除盡且擴 CLI 表面；(2) create 時帶入 PCB——不涵蓋 create 後的多次補寫；(3) 放寬 pending 對派發前章節（採用）。
- A（機會成本）：佔用 W1 容量一個 P2 槽位；但每日多次 --force 的審計成本持續累積，淨值為正。
- P（最可能失敗）：放寬範圍過寬破壞 W3-044「未認領不得寫執行紀錄」防護——以 PRE_DISPATCH_SECTIONS 白名單（僅 2 章節）+ 回歸測試防護。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 採用方案：方案 3（放寬 pending 對派發前章節的 append）

三候選比較（how.strategy Phase 3a 評估）：

| 方案 | 摩擦消除 | 缺點 | 結論 |
|------|---------|------|------|
| (1) `--bookkeeping` 旗標 | 部分 | PM 仍需記住旗標（摩擦轉移非消除）；擴大 CLI 表面；旗標誤用於非派發前章節需另設防護 | 否決 |
| (2) create 時帶入 PCB 內容 | 部分 | 僅涵蓋 create 當下，create 後的多次補寫（常見：分析逐步累積）仍卡 precondition | 否決 |
| (3) pending 允許派發前章節直寫 | 完整 | 放寬點需嚴格限定 | 採用 |

**決策理由**：方案 3 是唯一「零額外操作」路徑——PM 的合法 bookkeeping 流程（create → append-log PCB → 派發）直接可用，符合「正向錨點」設計（保持合法流程零摩擦，而非提供逃生通道）。放寬範圍以 `PRE_DISPATCH_SECTIONS = frozenset({"Problem Analysis", "Context Bundle"})` 白名單嚴格限定，執行產出章節（Solution / Test Results / Execution Log 等）維持 in_progress 限制，保留 W3-044「未認領不得寫執行紀錄」協議防護。

### 實作細節

1. `lib/precondition.py`：
   - 新增 `PRE_DISPATCH_SECTIONS` 常數（含 Why 註解）
   - `require_in_progress` 新增 keyword-only 參數 `allow_pending: bool = False`；僅在 `status == pending` 分支生效（blocked / closed / 未知 status 不受影響）
   - force 判定先於 status 判定（既有順序不變）→ `--force` 行為與 hook-logs audit 完全不變（acceptance #2）
2. `commands/track_acceptance.py`：append-log 呼叫端傳入 `allow_pending=args.section in PRE_DISPATCH_SECTIONS`；section 無效（不在白名單）時 `in` 判定為 False，聚合回報邏輯（W1-025）不變
3. 與 W1-025 的相容性：pending + 缺失 Context Bundle → status 放行 → 自動補建照常執行（Case 7b 測試覆蓋）；無任何 W1-025 行為回退

### 測試契約調整說明
