### 問題根因（PM 第一手觀察，2026-06-07 sync-pull session；根因同源性待 claim 後 WRAP 驗證 AC-4）

本 ANA 由實際 sync-pull 事件觸發，PM 親歷處理三議題（commit 47e54652），以下為第一手觀察素材：

**初步根因假設（待 WRAP 驗證）**：三議題可能同源於 `sync-claude-pull.py` 的「單向全量覆蓋」設計——將獨立 repo 內容完整覆蓋本地 `.claude/`，缺乏 (a) 衝突偵測；(b) 本專案演化方向感知（外移決策、結構重組、特有防護的識別）。

- **議題一（同步衝突解決）**：sync-pull 無 3-way merge。獨立 repo 與本專案分叉時（base SHA `e4d8d2ceb188`），衝突以「獨立 repo 全勝」靜默解決，本地演化被覆蓋而無提示。
- **議題二（多餘 hook）**：獨立 repo 採扁平 `.claude/hooks/` 結構，本專案已重組為 skill 自包含 `skills/{ticket,worktree,wrap-decision}/hooks/`。pull 將 27 個 hook 灌入 `.claude/hooks/` 頂層，與 skills/ 版本同名但內容不同，且 settings.json 未註冊 → 孤兒污染。根因：兩 repo hook 組織結構分叉，sync 無結構對齊機制。
- **議題三（pull 覆蓋進度）**：本專案演化內容被獨立 repo 舊版回灌/刪除——(a) 反方向回流：`project-specific-conventions.md`（已外移 docs/）、`project-integration/*.md`（W1-002 計畫外移）被拉回；(b) 特有防護被刪：PC-177、malformed-tool-call-detector-hook、context-depth-warning-hook（含 settings 取消註冊）。根因：sync 無法識別「本專案比獨立 repo 新」，盲目以舊覆新。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 根因定位（Reality Test 驗證結論，推翻預填假設）

預填假設「sync-pull 單向全量覆蓋、無 3-way merge」**經讀碼推翻**：`sync-claude-pull.py` 實際具備完整 3-way merge（`three_way_merge_file` / `_git_merge_three_files` / `compute_upstream_delta` / `is_base_reachable`）、preserve 機制（`load_preserve_list`）、stale 清理（`cleanup_stale_files`）。

**真根因（單一，三議題同源 — AC-4 答案：是，但比預填精確）**：

```
.sync-state.json 缺 last_synced_base_sha（只有 last_push_hash，2026-05-29 push 記錄）
  → read_base_sha() 回 None
    → should_use_full_overlay() 回 True（base_sha is None → return True）
      → 走 full overlay fallback，3-way merge 機制形同未啟用
        ├─ sync_directory() 把獨立 repo 全部檔案 shutil.copy2 覆蓋本地
        │    ├─ 議題一：本地修改被靜默覆蓋（無 merge、無衝突提示）
        │    └─ 議題二：獨立 repo 扁平 .claude/hooks/ 27 檔複製進來，與本專案
        │              skill 自包含 skills/*/hooks/ 同名共存 → 孤兒重複
        └─ cleanup_stale_files() 刪「本地有 + 獨立 repo 無 + 不在 preserve」
             └─ 議題三：PC-177 / malformed-tool-call-detector / context-depth-warning
                       不在 sync-preserve.yaml（僅 4 項）→ item.unlink() 刪除
```

**雙重缺口本質**：
1. **base SHA 初始化缺口**：本專案有 push 過但本地視角從未成功 pull 寫入 base SHA → 首次 3-way pull 必退化 full overlay
2. **preserve 清單涵蓋缺口**：sync-preserve.yaml 手動維護（僅 4 項），本專案持續演化的特有防護未同步登錄 → full overlay 的 stale cleanup 誤刪

### WRAP 四階段

- **W（Widen）候選**：(1) base SHA 初始化修復；(2) preserve 清單擴充；(3) cleanup 改 git 追蹤狀態感知（本地獨有追蹤檔不刪只警告）；(4) full overlay 時 stale cleanup 保守化 + 報告；(5) hook 結構分叉統一；(6) sync 前 dry-run 預覽 + 確認
- **R（Reality Test）**：已完成讀碼驗證（見上）。關鍵推翻：機制存在，問題在初始化與清單缺口
- **A（Attain distance）機會成本**：v1.0.0 主線已 blocked 跨 repo，修 sync 不擠壓發布；sync 為跨專案共用腳本，修復惠及所有專案（高槓桿）
- **P（Prepare to be wrong）**：(a) base SHA「自癒」假設脆弱——這次已寫入 e4d8d2ceb188，下次若獨立 repo force-push 使 base 不可達又退化；(b) preserve 手動清單治標——違反「不依賴人類記憶」，本專案每加防護都要記得登錄；(c) 深層風險：本專案演化超前獨立 repo 的場景，本應 push 不該 pull

### 三議題方案對照

| 議題 | 短期止血 | 根本解 |
|------|---------|--------|
| 一（衝突靜默） | base SHA 已自動寫入，下次 pull 走 3-way merge | full overlay fallback 時加 dry-run 預覽 + 確認（will-overwrite 清單） |
| 二（孤兒 hook） | 本次已移除 27 孤兒（commit 47e54652） | 決策 hook 結構統一方向（skill 自包含 vs 扁平），sync 加結構對齊或一端遷移 |
| 三（覆蓋進度） | 補 sync-preserve.yaml + 本次已還原（47e54652） | cleanup_stale_files 改 git 追蹤狀態感知：本地獨有 git 追蹤檔不靜默刪，改警告 + .sync-conflicts 暫存 |
