## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

PM 前台診斷的「孤兒 token 漏網」現象屬實，但**根因比 Context Bundle 假說更深一層**，且方向不同。

PM 假說（待驗證）：單次 parse-failure 路徑下，harness 把壞掉的 `<invoke>` 抽離成 sibling `tool_use` block，孤兒 token（court）殘留在 `text` block 中，故 hook 讀到「以孤兒 token 結尾的 text」但 sig 4 因 `<invoke>` 已消失而不命中。

經 transcript 結構普查（見「重現實驗結果」）**否證**此假說的核心前提：在本 harness 下，**assistant 訊息從不把 `text` block 與 `tool_use` block 並存於同一筆持久化 entry**（census：`text+tooluse=0`），且**沒有任何持久化 assistant 訊息以孤兒短 token 結尾**（census：`text_only_ends_short=0`、`text+tooluse_text_ends_short=0`）。

真實根因（更強結論）：單次 parse-failure 漏網路徑下，孤兒 token 前綴**根本不會成為 Stop hook 可讀的 assistant text block**。harness 一旦成功把 `<invoke>` 解析/抽離為 tool_use 嘗試，該回合即以 `tool_use`-only 訊息持久化，孤兒前綴隨之被併入該 tool 回合或丟棄——`last_assistant_text()` 取到的「最後一則 assistant text」是**更早的、乾淨的** text 訊息，不含孤兒 token。因此這條路徑在 transcript 層**零 hook-visible artifact**，無論加什麼 signature 都掃不到（沒有可掃的文字）。

對照命中型（double-failure）：census `invoke_literal_in_text=16`，這些是 `<invoke>` 完整字面留在 text block 的訊息——hook 正常命中（sig 1/3/4/5），與設計一致。

### 影響範圍

- `.claude/hooks/malformed-tool-call-detector-hook.py`（Stop event hook，調查對象，本票不直接改）
- 安全網可靠度：單次 parse-failure 漏網路徑無法由 Stop hook 涵蓋（架構性缺口，非 signature 覆蓋缺口）。double-failure 路徑覆蓋完整，無削弱。

### 相關 Error Pattern

無直接對應 PC。本 hook 的設計脈絡屬 IMP-055（hook stdout 結構）、IMP-048（stderr 觸發 hook error）家族，但本票為覆蓋缺口調查，非既有 PC 復發。docstring YAGNI 段（line 41-44）的「孤兒/半截標記特異性低、誤報高」論證經本調查**獲得補強證據**（見 Solution）。

## 重現實驗結果

### 實驗方法

對本票 PM session transcript（`fe7303c7-1003-4ecc-bf31-6711de6c249b.jsonl`，本 session live 重現孤兒 court 的同一檔）做 content-block 結構普查，不讀全檔，以 python 解析每筆 JSONL：

1. 過濾 `role==assistant` 訊息，統計 content block type 組合（text-only / text+tool_use / tool_use-only）。
2. 對每筆 assistant 訊息取 text block 串接後的「最後一個非空行」，判斷是否為短 alpha token（孤兒形態）。
3. 統計 `<invoke>` 字面殘留於 text block 的筆數（命中型對照）。
4. 定位 transcript 中 `court` 出現位置，逐筆判斷其所屬 role 與 content 結構。

對照 hook `last_assistant_text()` 邏輯（只收 `type==text` block）：驗證漏網型訊息是否真有「以孤兒 token 結尾的 assistant text block」供 hook 掃描。

### 實驗執行

census 結果（同一 PM session，121 筆 assistant 訊息）：

| 指標 | 數值 | 意義 |
|------|------|------|
| assistant_text_only | 62 | 純文字訊息 |
| assistant_tooluse_only | 59 | 純 tool_use 訊息 |
| assistant_text+tooluse | **0** | 文字與 tool_use **從不並存**於同一 entry |
| text_only_ends_short | **0** | 無純文字訊息以孤兒短 token 結尾 |
| text+tooluse_text_ends_short | **0** | （並存本就 0） |
| invoke_literal_in_text | 16 | `<invoke>` 字面殘留 text（命中型，hook 正常攔截）|

`court` 定位：

- line 699 = `role=user`，內容 `court\n<invoke name="Bash">...`——這是 **PM 親自 inline 輸入的測試/示範**（`echo test`），非 assistant 漏網產出。
- 其餘 court 出現（line 715/919 等 assistant 訊息）皆為**事後討論**（court 包在反引號內，如「根因是我在 function call 區塊前生成了游離 token（\`court\`）」），非孤兒結尾。
- 無任何 `role=assistant` 訊息以裸 `court` 結尾。

### 實驗發現

**已驗證的事實**：

1. 本 harness 下 assistant 訊息的 text block 與 tool_use block 互斥持久化（`text+tooluse=0`）。PM 假說「孤兒殘留 text + invoke 抽至 sibling tool_use」的**前提結構不存在**。
2. 單次 parse-failure 漏網路徑在 transcript 留下**零 hook-visible 孤兒 text artifact**（`*_ends_short=0`）。hook 的 `last_assistant_text()` 取到的是更早的乾淨 text 訊息，無孤兒 token 可掃。
3. double-failure 命中路徑覆蓋完整（`invoke_literal_in_text=16`，sig 1/3/4/5 正常攔截），無削弱。

**修正後的根因**：漏網不是「signature 沒涵蓋孤兒結尾形態」，而是「漏網路徑在 transcript 層無孤兒文字可供任何 signature 掃描」。這是 **Stop-hook 機制的架構邊界**（依賴持久化 assistant text），非 regex 覆蓋缺口。新增孤兒 signature 無法攔到不存在的文字。

**仍未驗證的假設（誠實標註）**：

- 本結論基於**單一 session** 的 transcript 結構。不同 harness 版本／不同 parse-failure 變體是否可能持久化「孤兒結尾 text + tool_use-only 後續」的兩段式 entry，本調查未跨 session 驗證。Solution 據此選擇**保守不實作新 signature**，並把跨 session 結構驗證列為 IMP 落地的前置 gate（避免基於單樣本疊高誤報 regex）。

### PM 前台診斷（basil 從此出發，驗證 + 設計，非重新推導）

### 缺口機制（已精確定位）
hook 是 Stop event，從 transcript 取「最後一則 assistant 訊息的 text block」掃 5 個 signature。關鍵 signature 4 = 正則 `(?m)^[ \t]*[A-Za-z]{1,12}[ \t]*\n[ \t]*<\s*invoke\b`（游離 token 換行接 `<invoke`）。

能否命中取決於 harness 是否把壞掉的 invoke 留在 text block：

| harness 路徑 | text block 殘留 | hook |
|------------|----------------|------|
| double-failure（inline retry 也失敗，放棄） | `court\n<invoke...</invoke>` 完整字面 | 命中（sig 4/5）|
| 單次 parse-failure，當 tool_use 嘗試抽離、回合交還用戶 | 只剩孤兒 `court`，`<invoke>` 已被抽走 | **不命中**（sig 4 需後接 `<invoke`，已消失）|

本 session 反覆重現後者：訊息以孤兒短 token（court）結尾、`<invoke>` 被 harness 抽離 → 無任一 signature 涵蓋「訊息以孤兒短 token 結尾」→ Stop 放行交還用戶。

### 經驗證據（acceptance 1）
本 session transcript JSONL 在 `/Users/tarragon/.claude/projects/-Users-tarragon-Projects-book-overview-v1/`（取 mtime 最新一檔，勿讀全檔以免 context 溢出）。用 grep/jq 取樣比對兩類 assistant 訊息：
- hook 命中型：text block 含完整 `<invoke ...></invoke>` 字面（前綴遺失版）。
- hook 漏網型：text block 以孤兒短 token（如 court / count）結尾、無後接 `<invoke`。
驗證「text block stripping 假說」：漏網型的 tool_use 區塊是否在另一個 content block（type=tool_use）而非 text，故 text 只剩孤兒 token。
