## Context Bundle


### PM 前台診斷（basil 從此出發，驗證 + 設計，非重新推導）

### 缺口機制（已精確定位）
hook 是 Stop event，從 transcript 取「最後一則 assistant 訊息的 text block」掃 5 個 signature。關鍵 signature 4 = 正則 `(?m)^[ \t]*[A-Za-z]{1,12}[ \t]*\n[ \t]*<\s*invoke\b`（游離 token 換行接 `<invoke`）。

能否命中取決於 harness 是否把壞掉的 invoke 留在 text block：

| harness 路徑 | text block 殘留 | hook |
|------------|----------------|------|
| double-failure（inline retry 也失敗，放棄） | `court\n<invoke...</invoke>` 完整字面 | 命中（sig 4/5）|
| 單次 parse-failure，當 tool_use 嘗試抽離、回合交還用戶 | 只剩孤兒 `court`，`<invoke>` 已被抽走 | **不命中**（sig 4 需後接 `<invoke`，已消失）|

本 session 反覆重現後者：訊息以孤兒短 token（court）結尾、`<invoke>` 被 harness 抽離 → 無任一 signature 涵蓋「訊息以孤兒短 token 結尾」→ Stop 放行交還用戶。

### 經驗證據（acceptance 1）
本 session transcript JSONL 在 `/Users/tarragon/.claude/projects/-Users-tarragon-Projects-book-overview-v1/`（取 mtime 最新一檔，勿讀全檔以免 context 溢出）。用 grep/jq 取樣比對兩類 assistant 訊息：
- hook 命中型：text block 含完整 `<invoke ...></invoke>` 字面（前綴遺失版）。
- hook 漏網型：text block 以孤兒短 token（如 court / count）結尾、無後接 `<invoke`。
驗證「text block stripping 假說」：漏網型的 tool_use 區塊是否在另一個 content block（type=tool_use）而非 text，故 text 只剩孤兒 token。

### 設計約束（acceptance 2-3）
- 新 signature 候選：訊息「最後一個非空文字行」為短 alpha token（長度 <= 12、無標點），且該訊息 Stop event 對應「無成功 tool_use」或「有 parse-failure 記號」。需 Stop event 能取得 tool_use 狀態的關聯判據（查 transcript 末筆是否有 tool_use block）。
- FP 風險：docstring YAGNI 段明列「孤兒/半截標記特異性低、誤報高」為不做理由。新方案必須附 FP 評估——正常訊息以短詞結尾（如「完成」「OK」「done」）極常見，純孤兒 token 判據會大量誤報。關鍵在「孤兒 token + 無 tool_use + 該 token 像 function_calls 開頭的腐化」三條件 AND 收斂特異性。
- 回歸：現有 5 signature 真陽攔截力（本 session 多次成功攔截）不可被削弱。
- 對齊 EXEMPT_MARKER 機制（meta 討論豁免）。
