### Phase 1 功能規格（確定性 broken-link CLI scanner）

設計者：lavender-interface-designer | TDD Phase 1 | 目標：以確定性 Python CLI 取代 LLM 手動 SKILL 作為跨框架完成 gate

### 0. 框架內建機制驗證（ARCH-010）

無框架內建 broken-link 偵測可用（既有 SKILL 為 LLM Glob+Grep 流程，非確定性）。Python 標準庫 `pathlib` + `re` 足以完成路徑 resolve 與 exists 檢查，不需外部依賴。結論：自建單一掃描器 CLI，純標準庫實作。

### 1. CLI 介面

**腳本位置**：`.claude/skills/broken-link-check/scan_links.py`（可執行 `python3 scan_links.py [OPTIONS]`）

| 維度 | 規格 |
|------|------|
| 輸入：repo root | 位置參數 `[REPO_ROOT]`，預設為 cwd；掃描 `<root>/.claude/**/*.md` |
| 輸入：旋鈕 flag | `--include-code-block` / `--include-migration-backups` / `--include-placeholder`（皆預設 OFF＝排除；給出即覆寫為「納入」） |
| 輸入：輸出格式 | `--format {text,json}`，預設 `text` |
| 輸出：摘要 | broken 總數 + refs 總數 + 各分類計數 |
| 輸出：清單 | 每筆含 `source_file:line` / `raw_ref` / `resolved_path` / `category` |
| exit code | `0`＝零 broken；`1`＝偵測到 broken（gate fail）；`2`＝執行錯誤（root 不存在、I/O 失敗） |

**exit code 語意設計**：`1` 專供 gate 用途（broken>0 即非零，可直接接 `&&` 或 CI），與 `2`（工具自身錯誤）區隔，避免下游把「掃描失敗」誤判為「無 broken」。

### 2. 三排除旋鈕固定預設規則

確定性權威 baseline＝**curated + 排除 code block + 排除 placeholder**（對齊 ANA 重現實驗 164 基線）。三旋鈕預設皆為「排除」，可由 flag 顯式覆寫納入（覆寫用於 triage/debug，非 gate 預設）。

| 旋鈕 | 預設 | 固定規則 | 覆寫 flag |
|------|------|---------|-----------|
| code block | 排除 | 跳過 ``` fenced 區塊內的路徑引用（含開合配對追蹤；奇數 fence 視為未閉合，到檔尾） | `--include-code-block` |
| migration-backups | 排除 | resolved_path 落在 `migration-backups/` 或 `hook-logs/` 下者不計 broken | `--include-migration-backups` |
| placeholder 範例路徑 | 排除 | 比對固定 placeholder 樣式集（`path/file.md`、`.claude/path/file.md`、`./path/file.md`、`../path/file.md` 等 SKILL 表格自身範例），命中歸 `placeholder` 分類不計 broken | `--include-placeholder` |

掃描範圍固定排除 `.claude/hook-logs/`（與既有 SKILL 一致）。

### 3. 輸出格式（供下游 W8-034 消費）

`--format json` 產出結構（穩定 schema，W8-034 清理直接消費）：

```
{
  "baseline": 164,                       # 預設旋鈕下的權威計數
  "scanned_files": <int>,
  "total_refs": <int>,
  "broken_count": <int>,

**分類欄位**：`category` 區分 broken / placeholder / excluded_*；`subcategory` 為 best-effort triage 提示（如 `dir-reorg`），供 W8-034 分組清理。text 格式為人類可讀對應視圖（摘要 + 分組清單）。

### 4. broken-link-check skill 路由策略（擇一）

**採方案 A：SKILL 改為呼叫 CLI，並標註 CLI 為權威 gate 來源。**

| 考量 | 方案 A（SKILL 呼叫 CLI＋標權威） | 方案 B（僅標權威，SKILL 保留手動流程） |
|------|-------------------------------|------------------------------------|
| 確定性 | CLI 計數即唯一真相，消除 LLM 浮動 | LLM 手動流程仍可能浮動，與 CLI 雙軌易再分歧 |

### 6. SOLID 拆分分析

單一掃描器模組，職責＝「掃描 .claude/ md → 抽引用 → resolve → 分類 → 輸出」。內部可分純函式（無 I/O，易測）：

| 函式 | 職責 | 純度 |
|------|------|------|
| `extract_refs(text)` | 從單檔內容抽 4 種前綴引用（含 code-block 區段標記） | 純 |
| `classify_ref(raw, resolved, knobs)` | 判 broken/placeholder/excluded | 純 |
| `resolve_path(raw, source_file, root)` | 引用→實際路徑 | 純 |
| `scan(root, knobs)` | 編排 I/O＋彙總 | 含 I/O |

結論：**不拆分成多模組**。單檔 CLI（SRP 以函式層級滿足，非檔案層級）；強拆增引用維護成本而無收益。純函式邊界即 Phase 2 主要測試切點。

### 7. Decision Questions

#### A. 模組結構與介面契約（對齊測試 import/呼叫簽名）

#### B. extract_refs — fence 追蹤 + 引用抽取 + 行號（規格 §1 旋鈕 + GWT #9）

```
PLACEHOLDER_PREFIXES = ["@.claude/", ".claude/", "./", "../"]   # 4 種前綴
REF_REGEX = 編譯 pattern 比對「以上述前綴開頭、非空白序列、以 .md 結尾」的 token
            # 約：(@\.claude/|\.claude/|\.\./|\./)[^\s)\]]*?\.md
            # 排除 http(s):// 與 # anchor：前綴清單不含 http/# 即天然排除

function extract_refs(text):
    refs = []
    in_fence = False                      # code block 開合狀態
    for line_no, line in enumerate(split_lines(text), start=1):
        if line.lstrip() starts with "```":   # fence 標記行
            in_fence = not in_fence            # 開合配對：每遇一次翻轉
            continue                           # fence 標記行本身不抽引用
        for match in REF_REGEX.finditer(line):
            raw = match.group()
            # 防禦：再次確認非 http / 非 anchor（前綴已保證，但雙保險）
            refs.append({"raw_ref": raw, "line": line_no, "in_code_block": in_fence})
    return refs
```

**未閉合 fence（奇數 ```）到 EOF**（測試 `test_unclosed_fence_extends_to_eof`）：上述逐行翻轉狀態天然滿足——開了未閉合，`in_fence` 維持 True 直到迴圈結束，其後所有引用 `in_code_block=True`。**無需檔尾補償邏輯**，狀態機自然涵蓋。

**http/anchor 排除**（測試 `test_ignores_http_and_anchor`）：`https://example.com/x.md` 與 `#section.md` 因前綴不在 4 種清單中，regex 不命中 → 回傳空。

#### C. resolve_path — 兩類解析規則（規格 §1 + TestResolvePath 4 案例）

| 引用前綴 | 解析基準 | 規則 |
|---------|---------|------|
| `@.claude/X` | repo root | 去掉 `@` → `root / ".claude/X"`（即 `root/` 拼接 `@` 之後的相對段） |
| `.claude/X` | repo root | `root / ".claude/X"` |
| `./X` | source_file **目錄** | `source_file.parent / "X"` |
| `../X` | source_file **目錄** | `source_file.parent / "../X"` 再 normalize |

```
function resolve_path(raw, source_file, root):
    if raw starts with "@":
        rel = raw[1:]                          # 去 @ → ".claude/x.md"
        return str((root / rel))               # 對齊 test_at_prefix
    if raw starts with ".claude/":
        return str((root / raw))               # 對齊 test_bare_claude（不論 source 深度都相對 root）
    # ./ 與 ../：相對 source_file 所在目錄
    resolved = (source_file.parent / raw)
    return str(os_normpath(resolved))          # normalize 消解 ../ 與 ./
```

> normalize 選擇：用 `os.path.normpath` 或 `Path(...).resolve()` 皆可消解 `..`；測試以 `Path(result) == 期望路徑` 比對，期望值為消解後路徑（如 `../sibling.md` from `.claude/sub/src.md` → `.claude/sibling.md`）。**避免 `.resolve()` 觸碰檔案系統 symlink 造成 tmp_path 不一致**，優先 `os.path.normpath` 純字串消解；回傳 str 後測試再包 `Path()` 比對。

#### D. classify_ref — placeholder 樣式集 + 判定順序（TestClassifyRef 5 案例）

固定 placeholder 樣式集（規格 §2，SKILL 表格自身範例）：

```
PLACEHOLDER_SAMPLES = {
    "path/file.md", ".claude/path/file.md", "./path/file.md", "../path/file.md",
}   # 以 raw_ref 完全比對；可視需要加 "path/to/file.md" 等同構樣式
```

**判定順序（短路，順序即優先級）**——測試案例反推的正確序：

```
function classify_ref(raw, resolved, knobs, exists):
    # 1. placeholder 最優先（即使 exists=False 也不算 broken）
    if raw in PLACEHOLDER_SAMPLES and not knobs["include_placeholder"]:
        return "placeholder"
    # 2. backup 路徑：resolved 含 migration-backups/ 或 hook-logs/
    if ("migration-backups/" in resolved or "hook-logs/" in resolved):
        if not knobs["include_migration_backups"]:
            return "excluded_backup"
        # 旋鈕開啟 → 落到 broken 判定（test_backup_counted_when_knob_on：exists=False → broken）
    # 3. exists 檢查：存在則非 broken
    if exists:
        return "ok"            # 任何非 "broken" 字串皆通過 test_existing_path_not_broken
    # 4. 其餘不存在 → broken
    return "broken"
```

> 順序依據：`test_placeholder_pattern_classified_placeholder`（placeholder 先於 broken）、`test_backup_path_excluded_by_default`（backup 先於 broken）、`test_backup_counted_when_knob_on`（旋鈕開時 backup 落 broken）、`test_existing_path_not_broken`（exists 短路）。**code block 排除不在此函式**——由 scan() 依 `in_code_block` flag 前置過濾（見 E），classify_ref 不收 in_code_block 參數（對齊測試簽名）。

#### E. scan() — I/O 編排 + 旋鈕過濾 + 穩定排序（TestScanIntegration + TestDeterminism）

```
function scan(root, knobs=None):
    knobs = knobs or DEFAULT_KNOBS
    root = Path(root)
    md_files = sorted(root.glob(".claude/**/*.md"))   # 排序確定性起點
    md_files = [f for f in md_files if "hook-logs/" not in str(f)]  # 固定排除掃描範圍
    categories = {"broken":0, "placeholder":0, "excluded_code_block":0, "excluded_backup":0}
    broken_entries = []
    total_refs = 0; scanned = 0
    for f in md_files:
        try:
            text = f.read_text(encoding="utf-8")
        except (OSError, UnicodeDecodeError) as e:
            sys.stderr.write(f"[WARN] cannot read {f}: {e}\n")   # Q5 雙通道，continue
            continue
        scanned += 1
        for ref in extract_refs(text):
            total_refs += 1
            # code block 旋鈕過濾（在 classify 之前）
            if ref["in_code_block"] and not knobs["include_code_block"]:
                categories["excluded_code_block"] += 1
                continue
            resolved = resolve_path(ref["raw_ref"], f, root)
            exists = Path(resolved).exists()
            cat = classify_ref(ref["raw_ref"], resolved, knobs, exists=exists)
            if cat in categories: categories[cat] += 1
            if cat == "broken":
                broken_entries.append({
                    "source_file": rel_to_root(f, root),   # 相對 root 字串，確定性
                    "line": ref["line"], "raw_ref": ref["raw_ref"],
                    "resolved_path": rel_to_root(resolved, root),
                    "category": "broken", "subcategory": None,   # best-effort triage 留 None
                })
    # 穩定排序鍵：source_file → line（GWT #3 確定性關鍵）
    broken_entries.sort(key=lambda e: (e["source_file"], e["line"]))
    return {
        "baseline": categories["broken"], "scanned_files": scanned,
        "total_refs": total_refs, "broken_count": len(broken_entries),
        "categories": categories, "broken": broken_entries,
    }
```

> `source_file` 用相對 root 路徑（測試僅 `assert "broken.md" in src`，相對/絕對皆過；但相對路徑在不同機器/cwd 確定性更佳，且對齊規格 §3 schema 範例 `.claude/...`）。
> 排序是 GWT #3 與 `test_broken_list_sorted_by_source_then_line` 的唯一保證點：`md_files` 先 sort + `broken_entries` 最後 sort，雙重確保 dict/glob 順序不洩漏。`categories` 用固定 key 的 dict（插入序固定）→ json 序列化確定。
