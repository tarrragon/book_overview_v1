## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

兩專案共用 flat `PC-NNN` 命名空間 + full-overlay sync，但**無共享編號分配權威**。
兩專案從相同 base 獨立累加新編號 → 併發分配必然碰撞。詳見「重現實驗結果」。

### 影響範圍

- 直接：`.claude/error-patterns/process-compliance/`（兩專案，各 182 檔）
- 間接：所有引用 `PC-NNN` 的位置——rules/、methodologies/、各 PC 的 `related:` 欄、
  memory（feedback_error_pattern_conflict_sync 等）、hook 訊息。重編號 = 引用 churn。
- 共享 repo（tarrragon/claude.git）知識庫完整性：同號異義使「PC-177」失去唯一指涉。

### 相關 Error Pattern

| PC | 關聯 |
|----|------|
| PC-122 error-pattern-conflict-not-synced | 新 PC 推翻舊 PC 須同步 deprecated；本案是「同號不同義」的併發版 |
| PC-180 dual-project-sync-scope-conflation | 明確將編號碰撞 defer 給本 ticket；提供 sync-preserve 本地保留軸 |
| PC-175 framework-sync-carries-source-project-type-assets | 同屬 sync 邊界家族 |
| PC-109 claude-runtime-state-missing-sync-exclusion | sync 邊界判定 |

**關鍵釐清**：preserve 機制（PC-180）只解「本地檔不被覆蓋/刪除」，**無法預防**
同號異義檔被 sync 新增（不同 slug 故不互蓋，會並存成 intra-dir 重號）。
故編號碰撞需獨立的「分配權威」解法，非靠 preserve。

## 重現實驗結果
### 實驗方法

比對 V1（book_overview_v1）與 APP（book_overview_app）兩專案的
`.claude/error-patterns/process-compliance/` 目錄，因 `.claude/` 經
`tarrragon/claude.git` full-overlay sync 共享。指令：
- `ls <dir> | grep '^PC-[0-9]'` 取兩專案檔名集合
- `comm` / `diff` 比對差異
- `sed -E 's/^(PC-[0-9]+)-.*/\1/' | uniq -c` 找同號多檔
- `sed -E 's/^PC-[0-9]+-//' | uniq -c` 找重複 slug 不同號
- 查 V1 `sync-preserve.yaml`、PC-180、APP backlog（worklog grep + ticket list）

### 實驗執行

- V1 PC 檔數 = 182；APP PC 檔數 = 182
- 跨專案檔名集合 diff：唯二差異 = 同一教訓在兩專案編號不同
- APP backlog（worklog + ticket list）grep「編號碰撞/numbering/PC 編號」= 0 命中
- APP W1-019/W1-020 為不同主題（提案流程、git Write 陷阱），非本碰撞

### 實驗發現（已驗證事實）

**碰撞分類（完整盤點，AND 邏輯確認）：**

| 類別 | 描述 | 實況 | 嚴重度 |
|------|------|------|--------|
| A 跨專案同號異義 | 同一教訓兩專案不同號 | defensive-rule: V1=PC-181 / APP=PC-177；ui-test-green: V1=PC-182 / APP=PC-178 | 高（sync 核心威脅） |
| B APP intra-dir 重號 | sync 帶入 V1 檔後 APP 出現同號雙檔 | APP PC-177×2(defensive+malformed)、PC-178×2(ticket-body+ui-test) | 高（APP 已 manifest） |
| C legacy 一致性重號 | 兩專案皆有、內容一致的歷史重號 | PC-010/018/019/020/030/105/165 各 2 檔，兩專案完全相同 | 低（非 sync 新威脅） |
| D 重複 slug 不同號 | 同教訓兩個編號 | auq-dispatch-...-readiness-check 出現於 PC-165 與 PC-171（兩專案皆有） | 中 |

**根因（已驗證）**：兩專案從相同 base（PC-001..176 完全相同）獨立累加新 PC，
無共享分配權威（allocation authority）→ 併發分配同號。flat PC-NNN 命名空間
+ full-overlay sync 使同號異義檔在 pull 後共存（檔名 slug 不同故不互蓋）。

**已採行動（V1 端）**：V1 已將 defensive-rule 重編為 PC-181、ui-test-green 重編為
PC-182（避開 APP 的 177/178），並把 `PC-177-malformed` 列入 `sync-preserve.yaml`
本地保留。APP 端尚未處理，仍存同號雙檔。

**關聯既有決策**：PC-180 已將「編號/架構碰撞」明確 defer 給本 ticket（W1-019），
且 error-patterns 在 sync 共享範圍內（sync-claude-push.py 無 EXCLUDE 命中）。

sync），少數才是專案/語言專屬。碰撞源於「通用教訓在不同專案併發發現並各自編號」。
