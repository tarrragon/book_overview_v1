### 一、五類專案版本源讀取策略設計

**策略模式**：將版本源讀取從硬編碼改為策略注入。現有基礎設施 `extract_version_from_file()` 已支援 yaml/json/toml，只需擴展 `VERSION_FILE_CANDIDATES` 和版本源選擇邏輯。

| 專案類型 | 版本源檔案 | parser | 讀取方式 | bump 方式 |
|---------|-----------|--------|---------|----------|
| chrome-ext | `package.json` + `manifest.json` | json | `data["version"]` | `bump_json_version()`（已有） |
| flutter | `pubspec.yaml` | yaml | `version: X.Y.Z+build`（regex L256-259 已有） | 新增 `bump_yaml_version()` |
| go | 無版本檔 | git-tag | `git describe --tags --abbrev=0` | `git tag v{version}`（已有流程） |
| php | `composer.json` | json | `data["version"]`（可選欄位） | `bump_json_version()`（已有） |
| monorepo | 子專案各自 | 依子專案類型 | 遞迴各子專案版本源 | 依子專案分別 bump |

**擴展 `VERSION_FILE_CANDIDATES`**：

```python
VERSION_FILE_CANDIDATES = [
    ("pubspec.yaml", "yaml"),       # Flutter
    ("package.json", "json"),       # NPM / Chrome Extension
    ("manifest.json", "json"),      # Chrome Extension
    ("composer.json", "json"),      # PHP
    ("pyproject.toml", "toml"),     # Python
]
```

**版本源選擇邏輯**（新增函式 `resolve_version_source`）：

1. 若 `.version-release.yaml` 存在且定義 `version_source.primary` → 使用指定檔案
2. 若無 config → 依 `VERSION_FILE_CANDIDATES` 順序掃描，第一個存在的檔案即為版本源
3. 若所有候選檔案都不存在 → fallback 到 git tag（Go 專案路徑）

**Go 專案特殊處理**：Go 生態不用版本檔，版本由 git tag 決定。`detect_version()` L280-325 已有 git tag fallback（步驟 3），只需在版本源解析邏輯中將 `go.mod` 存在 + 無其他版本檔視為 "go-tag" 策略。

### 二、`.version-release.yaml` schema 設計

**設計原則**：
- 遞增設計：現有三欄位（`release_workflow`/`tag_format`/`worklog_path_pattern`）保留不動
- 新增 `project_type` + `version_source` 區塊
- `versions`/`sync_rules` 等既有深層 schema 保留向後相容

```yaml
# === 新增欄位（與現有三欄位平級） ===

# 專案類型（影響版本偵測與 bump 策略）
# 可選值：chrome-ext | flutter | go | php | python | monorepo
# 省略時由自動偵測判定
project_type: chrome-ext

# 版本源配置
version_source:
  # 主版本源檔案（必填若不依賴自動偵測）
  primary: package.json
  # 版本源 parser 類型（json | yaml | toml | git-tag）
  # 省略時由副檔名推斷
  parser: json
  # 版本 key（json/yaml/toml 用）
  key: version
  # 同步目標（版本 bump 時一併更新的檔案清單）
  sync_targets:
    - path: manifest.json
      parser: json
      key: version

# monorepo 子專案配置（僅 project_type: monorepo 時使用）
subprojects:
  ui:
    path: ui/
    version_source:
      primary: pubspec.yaml
      parser: yaml
  server:
    path: server/
    version_source:
      primary: null  # go-tag 策略
      parser: git-tag

# === 現有欄位保留不動 ===
release_workflow: trunk
tag_format: "v{version}"
worklog_path_pattern: "docs/work-logs/v{major}/v{major_minor}/v{version}"
```

**schema 演進策略**：
- 所有新欄位可選（`project_type`/`version_source`/`subprojects`）
- 缺省時 `load_version_release_config()` 的 DEFAULT 補值邏輯已有（L732-742），只需擴展 DEFAULT dict
- 現有 15 consumer 無需立即建立 config 檔（自動偵測 fallback 覆蓋）

### 三、自動偵測 fallback 設計

**觸發條件**：`.version-release.yaml` 不存在或缺少 `project_type` 欄位時啟動自動偵測。

**偵測演算法**：

```
掃描根目錄檔案 →
  pubspec.yaml 存在？ → flutter
  go.mod 存在？ → go
  package.json + manifest.json 皆存在？ → chrome-ext
  composer.json 存在？ → php
  package.json 存在（無 manifest.json）？ → npm
  pyproject.toml 存在？ → python
  以上皆無 + 子目錄含版本檔？ → monorepo
  全無？ → unknown（stderr 警告 + 使用 git tag fallback）
```

**monorepo 偵測**：根目錄無版本檔但子目錄（depth=1）存在版本檔 → 提示為 monorepo 並列出偵測到的子專案。

**stderr 確認訊息**：偵測結果印到 stderr（不污染 stdout 資料流）：
```
[INFO] 自動偵測專案類型：flutter（根據 pubspec.yaml）
[INFO] 若偵測不正確，請建立 .version-release.yaml 指定 project_type
```
