## Problem Analysis

### 問題根因

兩個 SubagentStop hook（subagent-stop-dispatch-cleanup / agent-commit-verification）存在三層缺陷共振（parent 1.0.0-W1-055 ANA 已查明，confidence 0.95）：

1. **通道語意錯置（主因）**：0.19.1-W1-046 將輸出自 top-level `systemMessage`（純顯示）改為 `hookSpecificOutput.additionalContext`。W1-055 活體確證該欄位投遞對象是「停止中的 subagent」——注入其對話並令其繼續 → 再停止 → 事件再觸發（自激迴圈，H1）。
2. **缺斷路器**：CC hook 規格為 Stop/SubagentStop 提供 `stop_hook_active` 防迴圈欄位，兩 hook 均未檢查。
3. **缺去重**：每次事件全量重掃並無條件重播相同訊息——[WAIT] 廣播（remaining>0 即輸出）與 hook error 5 分鐘視窗摘要（W1-074/075/076 重播症狀）。

### 影響範圍

- `.claude/hooks/subagent-stop-dispatch-cleanup-hook.py`（[WAIT]/[OK] 廣播）
- `.claude/hooks/agent-commit-verification-hook.py`（commit 警告 + hook error 摘要）
- 症狀：同一 agent_id 事件 5-20 次重複、PM 主線程 11+ 次重複喚醒、每次喚醒 context 重載成本

### 相關 Error Pattern

- PC-148（雙重註冊）已被 parent ANA 排除（對 SubagentStop）；W1-040.1 孤兒 waiter 為同症狀不同觸發源

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復實作（commit f32f71aa）

#### 修復 1：stop_hook_active 斷路器（兩 hook）

input_data 含 `stop_hook_active=true`（runtime 因 stop hook 而繼續）時靜默 exit 0——不清理、不掃描、不輸出。位置：agent_id 檢查之後、任何副作用之前。即使迴圈被點火，第二次事件即被斷開。

#### 修復 2a：[WAIT] 廣播 dedup（dispatch-cleanup hook）

- key = `sha256(agent_id + "|" + wait_message)`；wait_message 含 remaining 數量與描述清單，remaining 變化（agent 增減）即新 key 重新播報
- state 檔：`.claude/hook-logs/subagent-stop-dispatch-cleanup/wait-broadcast-dedup.json`（hook-logs 已 gitignore），TTL = 600 秒（`WAIT_BROADCAST_DEDUP_TTL_SECONDS`，覆蓋觀測叢集 5-15 秒間隔 × 鏈長 5-20）
- 載入時剪除過期 entry，state 不無限成長；讀寫失敗 fail-open 照常播報（quality-baseline 規則 4：寧重複勿吞通知）

#### 修復 2b：hook error fingerprint 去重（commit-verification hook）

- fingerprint = `sha256(json.dumps(sorted([(hook_name, count), ...])))`——不含 agent_id，跨事件相同錯誤集合即去重（W1-074/075/076 症狀）；任一 hook 計數成長或新錯誤 → fingerprint 變化 → 立即播報
- TTL = 掃描視窗（5 分）× 2 = 600 秒（`HOOK_ERROR_DEDUP_TTL_SECONDS`）
- `check_and_record_broadcast` 在兩 hook 刻意重複實作：hook 檔名含 hyphen 無法互相 import，且本票約束僅允許動兩 hook 本體（抽共用 lib 屬範圍外決策）

#### 修復 4：通道語意取捨評估（採回退 systemMessage）

| 候選 | 評估 |
|------|------|
| 保留 additionalContext + 僅靠斷路器 | 斷路器只擋第二次事件；首次注入仍令垂死 subagent 多跑一回合（5-15 秒 + token 浪費），且訊息落在 subagent context 而非 PM 顯示——錯通道 |
| 回退 top-level systemMessage（採用） | 兩 hook 意圖均為「通知 PM 主線程」；systemMessage 為純顯示通道、2026-06-05 前已知良好狀態（每 agent_id 恆 1 次事件）。H1 第一人稱活體確證（confidence 0.95）構成回退的充分證據，符合派發指示「若證據支持回退可一併實作」 |

實作：兩 hook 直接輸出 `{"systemMessage": context}`，移除 `build_subagent_stop_output` import。helper 本體與其測試（hook_utils/hook_io.py、test_subagent_stop_output_helper.py）不動——屬約束範圍外，且保留供未來真正需要注入語意的 hook 使用。斷路器在回退後仍保留（defense in depth，防未來通道再切換時迴圈復發）。

#### 測試（22 項）

- test_subagent_stop_dispatch_cleanup_hook.py（11）：systemMessage schema、[OK]/[WAIT] 內容、斷路器靜默 + false 不誤傷、dedup 二次事件靜默 / remaining 變化重播 / 跨 agent_id 各播一次 / TTL 過期重播 / 損毀 state fail-open
- test_agent_commit_verification_hook_dedup.py（7，新檔）：斷路器靜默（含不執行 git 掃描）+ false 不誤傷、fingerprint 二次去重 / 集合變化重播 / TTL 過期、fingerprint 排序穩定性與內容敏感性
- test_agent_commit_verification_hook_schema.py（4）：通道斷言更新為 systemMessage，移除版本分支 fallback 測試
- 測試 mock 衛生：dedup state 檔 getter 一律 monkeypatch 至 tmp_path，避免污染真實 repo hook-logs
