## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`.claude/` root 累積大量早於 rules/methodologies/skills 結構的歷史遺留 `.md`，這些檔不落 `knowledge-carrier-allocation` 載體地圖任何一行（地圖明列 root 遺留檔與 templates 為「未分類，依二軸重分配」）。內容形態橫跨清單/範例/流程/參考/報告，但都堆在 root，無預算約束也無載體歸屬。templates/ 則是「寫一次被實例化多次」的去處——內容規範若不傳導到模板，新 agent/ticket 從模板長出舊形態（W8-016 執法的對手即此自我複製機制）。

### 影響範圍

`.claude/` root 共 23 個 `.md`（總計約 14400 行，最大 code-smell-checklist 4484 行），其中 `tdd-collaboration-flow.md`（1841 行）與 `methodologies/tdd-collaboration-flow.md` 同名且內容**已分歧**（divergent duplicate）。`templates/` 11 檔（7 `.md`），其中 agent-template.md 三區塊僅 1/3（缺完整「允許產出/禁止行為/適用情境」），與 `agent-definition-standard` 不同步。

### 相關 Error Pattern

- 與 W8-009/W8-010 同源：`knowledge-carrier-allocation` 載體錯置；本票補地圖未涵蓋的 root + templates 兩類。
- templates 同步缺口直接餵養 W8-016（執法 hook 的 templates 同步檢查標的）；agent-template 三區塊缺口即 W8-016 的 motivating case。
- root tdd-collaboration-flow 分歧副本與 W8-018（TDD 家族整併）重疊，須協調 SSOT。

### 實驗發現

**已驗證事實**：

1. root 遺留檔形態混雜，多數應重分配到 references/skill/rules，少數為框架 legit 資產（README/CHANGELOG/sync 文件/terminology）。
2. `tdd-collaboration-flow.md` root 與 methodologies/ 雙版本分歧——SSOT 未定，36 refs 指向 root（但不在 CLAUDE.md/rules/tdd-skill 自動載入鏈），須與 W8-018 協調收斂。
3. 引用次數與搬移風險成正比：高引用檔（README/CHANGELOG/agent-collaboration/terminology）搬移=大量連結手術；低引用檔（code-smell-checklist 等）較安全。
4. agent-template.md 三區塊不完整（1/3），證實「模板不同步 → 新實例長出舊形態」的傳導路徑，直接餵養 W8-016 執法需求。
5. TECH-DEBT-REMINDER-DELIVERY-REPORT.md 為交付報告（非框架知識），屬 docs/work-logs 或刪除。

**仍為假設（執行階段需逐檔確認）**：各檔精確目的地需逐檔讀內容確認；高引用檔搬移的連結更新範圍需 grep 全清單；tdd-collaboration-flow SSOT 方向（保留 methodologies 或 root）待與 W8-018 一併決策。
