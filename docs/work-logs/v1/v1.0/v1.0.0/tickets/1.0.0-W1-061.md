## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 分析對象

`.claude/skills/ticket/tests/test_checkpoint_state/test_performance.py` 中的 `test_E1_checkpoint_state_performance_median_under_100ms`。

### 斷言設計

測試在真實 tmp git repo + mock ticket CLI 環境下執行 `checkpoint_state()` 30 次，計算 median 和 p95，以硬門檻斷言：

- `assert median < 100.0`（AC6 硬 AC）
- `assert p95 < 150.0`（sage E1 目標）

兩者皆為**絕對計時門檻**，使用 `time.perf_counter()` 差值作為斷言對象。

### 問題識別

1. **負載敏感性**：計時值受 CPU 負載、GC、IO 排隊影響。ticket why 記載「5 背景 agent 並行」高負載下全套件 1 RED / 1 GREEN，standalone 5/5 GREEN——典型的環境依賴 flaky 模式。
2. **規則覆蓋缺口**：`test-assertion-design-rules.md` 規則 1-4 以 Jest/npm test 為標的（`tests/unit/`、`tests/integration/`、`tests/perf/`），未涵蓋 `.claude/skills/ticket/tests/` 下的 Python pytest 計時斷言。原則相同（計時硬門檻不入主套件）但無具體條款。

### 影響範圍

- 受影響測試：僅此 1 個 test case（grep 確認 `test_performance.py` 唯一計時斷言檔）
- 受影響指令：`npm run test:hooks`（透過 `uv run pytest tests/`）
- 不影響：`npm test`（Jest 主套件）

## 重現實驗結果
### 實驗方法

1. Standalone 取樣：`(cd .claude/skills/ticket && uv run --project . pytest tests/test_checkpoint_state/test_performance.py -k "E1_checkpoint" -v)` 執行 7 次
2. Full-suite 取樣：`(cd .claude/skills/ticket && uv run --project . pytest tests/ -v)` 執行 4 次（含 1845 個測試）
3. 環境：開發機低負載（非 5 agent 並行場景）

**歷史紀錄（ticket why）**：

- Full-suite 高負載（5 背景 agent 並行）：GREEN 1 / RED 1
- Standalone：GREEN 5 / RED 0

### 實驗發現

1. **低負載下穩定通過**：standalone 7/7 + full-suite 4/4 = 11/11 GREEN
2. **高負載下曾失敗**：ticket 記載 5 agent 並行時 full-suite 1/2 RED——50% fail rate
3. **結論**：此測試在低負載下穩定，在高負載下 flaky。根因是計時硬門檻對 CPU 競爭敏感。本次取樣環境為低負載，與歷史高負載 RED 不矛盾（PC-168：少量 GREEN 不推導 always stable）。
4. **thyme 先前實測數據**（測試檔 docstring）：5 subprocess 序列 median 52ms / p95 84ms——留有 48ms / 66ms 餘量，但高負載可壓縮至門檻邊緣。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 處置方案評估

| 方案 | 描述 | 優點 | 缺點 |
|------|------|------|------|
| A: pytest -m perf 標記隔離 | 加 `@pytest.mark.perf`，預設套件排除（`-m "not perf"`） | 保留計時驗證能力；主套件穩定；與 Jest tests/perf/ 原則對齊 | 需改 pyproject.toml 設定 + 新增 npm script |
| B: 放寬門檻 | median 100→500ms, p95 150→800ms | 改動最小 | 門檻過寬失去退化偵測意義；本質仍是計時硬門檻 |
| C: 改非計時斷言 | 驗證 CheckpointState 結構正確性，移除計時 | 完全消除 flaky | 失去效能退化防護 |

### 選定方案：A（pytest -m perf 標記隔離）

**理由**：

1. **原則一致性**：與 test-assertion-design-rules.md 規則 1-2 的設計原則完全對齊——計時硬門檻不入主套件，效能測試獨立執行。Jest 用 `tests/perf/` 物理拆檔，Python 用 `pytest.mark.perf` 邏輯標記是等效手段。
2. **保留價值**：此測試確實有效能退化偵測價值（thyme baseline median 52ms），只是不應在主套件高負載下作為 pass-fail 門檻。隔離後可透過 `npm run test:hooks:perf` 獨立執行，提供可控環境下的效能基準。
3. **方案 B 否決理由**：放寬門檻治標不治本，且 500ms 門檻下幾乎不可能觸發失敗，失去存在意義。
4. **方案 C 否決理由**：功能正確性測試已由 Group A-D 覆蓋，但完全移除計時會失去唯一的效能退化防護。

### 實作規劃（由 PM 接手建立 IMP ticket）

1. `test_performance.py` 的 `test_E1_*` 加 `@pytest.mark.perf` 裝飾器
2. `pyproject.toml` 的 `[tool.pytest.ini_options]` 加 `markers = ["perf: 效能測試（獨立於預設套件）"]` 及 `addopts = "-m 'not perf'"`（預設排除 perf 標記）
3. `package.json` 新增 `test:hooks:perf` script：`uv run --project .claude/hooks pytest .claude/skills/ticket/tests/ -m perf -v`
4. 測試檔頭加效能測試聲明（仿 Jest tests/perf/ 標準）

### test-assertion-design 規則 Python 覆蓋缺口結論

**結論：不需要補 Python 專屬章節**。

**理由**：現行 test-assertion-design-rules.md 的四條規則（禁計時硬門檻入主套件、效能測試獨立執行、toBeCloseTo 精度、快取驗證方式）是**語言無關原則**。Python pytest 的具體落地手段（`@pytest.mark.perf` + `addopts`）與 Jest 的 `tests/perf/` 物理拆檔是等效實作，不需要獨立章節。本 ticket 的 IMP follow-up 完成後，此原則已在 Python 套件具體落地，無需額外規則擴充。

若未來 Python 測試套件規模擴大（>10 個效能測試），可考慮補充 Python 專屬「效能測試放置與標記」章節，但目前 1 個測試不構成建章節的充分理由。
