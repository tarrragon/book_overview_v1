## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 修改摘要

1. **`cmd_start_version` 整合 project_type 偵測**：在 Step 2（重複檢查）前插入 `detect_project_type` + `resolve_version_source` + `_resolve_worklog_pattern` 呼叫，確保後續 Step 4/5 使用正確策略。

2. **Step 5 bump 策略動態分派**：以 `bump_dispatch` dict（json→`bump_json_version`、yaml→`bump_yaml_version`、toml→`bump_toml_version`）取代硬編碼 `package.json`/`manifest.json` 迴圈。git-tag 策略（Go 專案）跳過 bump。Chrome Extension 專案自動同步 bump `manifest.json`（透過 `sync_targets` 或 `chrome-ext` 類型自動推斷）。

3. **worklog 路徑依 project_type 決定預設值**：新增 `_resolve_worklog_pattern()` + `_load_raw_version_release_config()`，優先使用 `.version-release.yaml` 明確設定，未設定時 flutter→nested-3、其餘→flat。

4. **`create_worklog_structure` 參數化**：新增 `worklog_path_pattern` 參數，內部改用 `resolve_worklog_dir()` 解析路徑（取代硬編碼），nested 判定改為動態（parent 目錄名稱是否 startswith `v`）。

5. **Step 2 重複檢查路徑動態化**：改用 `resolve_worklog_dir()` 檢查 worklog 目錄是否已存在。
