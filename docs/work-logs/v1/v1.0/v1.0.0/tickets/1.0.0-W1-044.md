## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### Spike 結論：Stop hook stdin 的 subagent 偵測手段

**方法**：靜態分析 CC hook input schema（權威來源 .claude/references/hook-architect-technical-reference.md L85-101）+ 既有 hook 程式碼證據（subagent-stop-dispatch-cleanup-hook.py L59 依賴 agent_id）+ settings.json 註冊清單核對。無法在 subagent 回合內觸發真實 PM Stop event，故採文件 schema + 程式碼證據而非 transcript heuristic（符合 ticket 防護：禁不可靠 heuristic）。

**核心發現**：

| Event | stdin 欄位 | agent_id/agent_type | 觸發時機 |
|-------|-----------|--------------------|---------|
| Stop | session_id, transcript_path, stop_hook_active, background_tasks, session_crons | 無 | 主線程停止（schema 明示「非代理人結束」）|
| SubagentStop | + agent_id, agent_type, agent_transcript_path | 有 | 代理人真正完成 |

**結論 1（可靠偵測手段）**：documented schema 中 agent_id / agent_type 為 subagent context 的權威判別欄位（subagent-stop-dispatch-cleanup-hook.py 已實證依賴 agent_id 做精準清理）。本 IMP 採此欄位偵測，不用 transcript 長度等不可靠 heuristic。

**結論 2（事件層校正 W1-030 假設）**：經 settings.json 核對，handoff-auto-resume-stop-hook 與 stop-worklog-handoff-sync-check-hook **僅註冊於 Stop event，未註冊 SubagentStop**。CC 只在主線程停止時路由 Stop-registered hooks。W1-030 靜態分析正確指出這 2 hook 有 stdout 注入能力且零 subagent 偵測，但「subagent 完成觸發這些 Stop hook」的事件層假設需校正——subagent 完成走 SubagentStop（另 2 個 hook），不走 Stop。

**結論 3（防禦性實作仍有價值）**：縱使事件層目前隔離，加入 agent_id/agent_type 偵測仍屬正確防禦——(a) 若未來 CC 版本變更路由、或 PM 誤將本 2 hook 加註 SubagentStop，偵測即生效；(b) 與 subagent-stop-dispatch-cleanup-hook 的 agent_id 依賴形成一致的 subagent context 判別慣例。subagent context 時回 {"suppressOutput": true} 跳過注入。
