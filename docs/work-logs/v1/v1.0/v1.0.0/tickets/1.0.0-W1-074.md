**實際 hook 位置**（原 where.files 已 stale，已修正）：

| 項目 | 值 |
|------|------|

| 既有測試 | `.claude/hooks/tests/test_ticket_quality_gate_type_aware.py`、`test_ticket_quality_gate_path_filter.py` |
| settings.json 註冊 | PostToolUse / matcher `Write`（單一註冊點，settings.json:555 附近） |

**根因定位**（PM 已讀原始碼確認）：

1. `validate_input_format`（約 L94-99）無條件以 `validate_tool_input(tool_input, logger, ("file_path", "content"))` 驗證——這是 Write 工具的 schema。Edit 工具 tool_input 形狀為 `{file_path, old_string, new_string}`、MultiEdit 為 `{file_path, edits}`，皆無 `content` 欄位 → ERROR「tool_input 缺少必要欄位: content」+ graceful exit。
2. tool_name 分流（`should_trigger_check` 的 `allowed_tools` 檢查，約 L126-129）在 schema 驗證**之後**才執行——順序顛倒是缺陷本體。
3. 下游 `tool_input["content"]` 直接取值點：約 L160（結構標記檢查）、L558-559（main 流程）。Edit 場景無法從 old/new string 重建全文；本 hook 為 PostToolUse，檔案已寫入磁碟，可改為 tool-aware 內容取得（Edit/MultiEdit 時讀 file_path 磁碟內容）。

**設計意圖 vs 註冊不一致（需實作時一併決策）**：

hook 內 config 預設 `allowed_tools = ["Write", "Edit", "MultiEdit"]`（設計意圖支援三工具），但 settings.json matcher 僅 `Write`。開放問題：matcher='Write' 下 Edit 輸入為何能到達 hook（W1-071/075/076 三 session 實證 Edit 觸發 ERROR）——實作前先以 log 重現驗證 matcher 語意，據此決定修法：(a) 驗證分流依 tool_name 各自 schema（Write 要 content；Edit 要 old_string/new_string；MultiEdit 要 edits），或 (b) 若確認 matcher 永遠只送 Write，則收斂 allowed_tools 並在錯誤訊息補 tool_name。兩案皆須讓錯誤日誌含 tool_name（現行 ERROR 無法歸因工具）。

**重現基準**：`.claude/hook-logs/ticket-quality-gate/ticket-quality-gate-20260611-162322.log` 末段——「輸入驗證通過(DEBUG) → ERROR tool_input 缺少必要欄位: content」序列。
