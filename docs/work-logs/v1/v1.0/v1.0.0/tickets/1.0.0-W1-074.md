## Solution

### 開放問題驗證結果（matcher='Write' 下 Edit 輸入為何到達 hook）

結論：**生產環境從未有 Edit 輸入到達本 hook**，matcher='Write' 為精確比對且運作正常。ERROR 的真實觸發源是 `.claude/hooks/tests/test_effort_awareness.py` L171-178（`TestTicketQualityGateEffort.test_medium_effort_processes`）的合成輸入 `{"tool_name": "Read", "tool_input": {"file_path": "x.py"}, "effort": {"level": "medium"}}`。

驗證證據鏈：

| 證據 | 內容 |
|------|------|
| 錯誤欄位序 | `validate_tool_input` 依序檢查並回報第一個缺欄位；ERROR 為 `content` 而非 `file_path`，表示輸入有 file_path 無 content（Read/Edit 形狀皆符合） |
| 同秒並行 hook 日誌 | 16:23:22 同一瞬間 ticket-frontmatter-validator 記錄「非 Edit/Write 工具: Read」、phase4 記錄合成命令 `echo hi`、creation-acceptance-gate 記錄 effort 三級輪替——皆為 test_effort_awareness.py 各 test case 的 payload 簽名 |
| effort 雙連發模式 | 三次 ERROR（14:53:35 / 16:08:12 / 16:23:22）皆呈「effort=low 短路 + effort=medium ERROR」同秒成對，對應 `test_low_effort_short_circuits` + `test_medium_effort_processes` 相鄰執行 |
| transcript 時間軸 | subagent transcript（agent-a31cb3af2d3225c38）顯示 08:23:12 UTC 發出 pytest/Bash 呼叫，ERROR 落在 08:23:22 UTC 執行視窗內；該時間點無任何 Edit tool_use 事件 |
| 修復後重放 | 以重現基準 payload 直灌 hook：INFO「工具 Read 不適用 schema 驗證」→「工具類型不符: Read」靜默跳過，exit 0，無 ERROR |

### 修法決策：採 (a) tool-aware schema 分流

理由：

1. **缺陷本體仍真實存在**：schema 驗證先於 tool_name 分流的順序顛倒，使任何非 Write 輸入（今日為測試的 Read、未來若 matcher 擴註冊則為 Edit/MultiEdit）都會誤撞 Write schema 產生 ERROR 噪音。
2. **(b) 收斂 allowed_tools 無法修復順序顛倒**：非 allowed 工具仍會先撞 Write schema 才到分流檢查，且與 `quality_rules.yaml` 的 `allowed_tools: [Write, Edit, MultiEdit]` 設計意圖矛盾。
3. (a) 同時消除測試噪音與支援設計意圖的三工具。

### 實作內容（`.claude/skills/ticket/hooks/ticket-quality-gate-hook.py`）

| 修改點 | 內容 |
|--------|------|
| 新增 `TOOL_REQUIRED_FIELDS` 常數 | Write→(file_path, content)；Edit→(file_path, old_string, new_string)；MultiEdit→(file_path, edits) |
| `validate_input` 改 tool-aware | 先依 tool_name 取對應 schema 再驗證；非三工具（如 Read）不套 schema、交由 `should_trigger_check` 的 allowed_tools 靜默分流；schema 失敗時 ERROR 含 tool_name |
| 新增 `resolve_ticket_content` | Write 取 tool_input.content；Edit/MultiEdit 因僅有差分，改讀 file_path 磁碟內容（PostToolUse 檔案已落盤）；讀檔失敗回 None 由呼叫端跳過 |
| `should_trigger_check` 條件 4 | 原 `tool_input["content"]` 直取改為 `resolve_ticket_content`；None 時 INFO 跳過 |
| `main` 步驟 3/5 | 步驟 3 失敗日誌補 `tool_name=`；步驟 5 內容取得改 tool-aware，None 時跳過 |
