## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 來源
W8-036 鏈（sync-pull full overlay 復活根因修復）完成後，用戶提出系統性顧慮：v1/app 僅為 N 個共用 claude.git 上游之消費者中的兩個，其他久未 pull 的專案如何清理舊孤兒檔？是否需在 pull 加專門流程？PM 已完成初步 grounding（讀 sync-claude-pull.py 1903 行 + W8-036.3/.4 結論）。

### 現行機制覆蓋（已 grounding）

| 情境 | 現行處理 | 狀態 |
|------|---------|------|
| 定期 pull（base 可達）+ 孤兒未本地修改 | delta D 自動跟刪（three_way_merge_file L575-576 → None）| 已覆蓋 |
| 久未 pull（base 仍可達）| 下次 delta 含累積刪除自動清（app 實證 base 較舊仍清 16）| 已覆蓋 |
| fresh clone / 遺失 state（base 不可達）| W8-036.4 git-delete-aware full-overlay 用本地 git 史防復活 + preview 顯示 | 已覆蓋 |
| 前次 pull 衝突殘留 | W1-084 .sync-conflicts/ 殘留警告（detect_conflict_residue）| 已覆蓋 |

### 確認缺口

- **缺口 1（已驗證，L573-577）**：`three_way_merge_file` 對「上游刪除（upstream_content is None）+ 本地修改過（local != base）」回 `(local_content, False)`——conflict=False 靜默保留，pull 不通報。任何客製過後被上游刪除之框架檔的專案 sync 後孤兒靜默殘留。= W8-036.2 v1 那 3 個殘留（document-responsibilities/eslint-error-handling-rules/test-design-antipatterns），PM 手動比對 delta vs git status 才發現。
- **缺口 2**：無主動孤兒稽核——專案無法一鍵列「本地 .claude/ 有但上游 HEAD 無」之檔，尤其早於 base 不在 delta 窗者。
- **缺口 3（系統性/工作流）**：刪除跨 N 專案傳播仰賴刪除先到上游（sync-push）。消費者本地刪框架檔卻不 push（W10-049 根因）則其他專案永不知。屬工作流紀律，非機制 bug。

### PM 初步建議方向（待 WRAP 驗證，含 toolification-bias 防護）

核心刪除傳播已運作（delta + git-delete-aware），不需重的「專門流程」（對照 W8-036.3 已拒 tombstone 工具）。傾向**改造既有**：
1. 最小高槓桿：L577 分支命中時記錄 + pull 結尾通報「上游已刪但本地分歧保留」清單（鏡像 .sync-conflicts 殘留警告）。補缺口 1。
2. 選配：孤兒稽核報告。補缺口 2（需過 toolification-bias）。
3. 零工具：工作流紀律明文（框架編輯走上游不走消費者）。補缺口 3 根因，查 framework-asset-separation 是否已涵蓋。

待 WRAP 完整評估三類方案性質、重現缺口 1、查既有規則涵蓋後定案 + spawn 落地。

## 重現實驗結果
### 實驗方法
讀 sync-claude-pull.py（1903 行）逐路徑追蹤刪除處理；以實際呼叫 three_way_merge_file 重現缺口 1；grep 既有規則查 Gap 3 涵蓋。

### 實驗執行與發現

**缺口 1 重現（airtight，呼叫實際函式）**：
- 情境「上游刪除（upstream_path=None）+ 本地修改過（local != base）」→ `three_way_merge_file` 回 `(merged=非None, conflict=False)`。merged 非 None = 保留本地；conflict=False = 不進 .sync-conflicts、pull 不通報 → **靜默殘留**。
- 對照「上游刪除 + 本地未改（local==base）」→ 回 `(None, False)` = 自動跟刪。
- 程式碼定位：three_way_merge_file L573-577（`if upstream_content is None: if local==base: return None,False; return local_content,False`）。
- 實證佐證：W8-036.2 v1 那 3 個殘留（document-responsibilities/eslint-error-handling-rules/test-design-antipatterns）走此路徑，sync-pull 輸出僅報 agent-collaboration 1 衝突，未提這 3 個 → 靜默確認。

**現行機制各路徑覆蓋（已驗證）**：
- delta D + 本地未改 → 自動跟刪（L575-576）✓
- 久未 pull（base 可達）→ delta 含累積刪除自動清（app 實證 base 60872c40 較舊仍清 16）✓
- fresh clone / base 不可達 → W8-036.4 _is_intentionally_deleted git-delete-aware full-overlay ✓
- 前次 pull 衝突殘留 → W1-084 detect_conflict_residue 警告 ✓

**Gap 3 既有涵蓋（grep 實證）**：
- framework-asset-separation.md grep「上游/消費者/push」→ 0 命中（未述編輯紀律）。
- sync-push.md L58-69 **已涵蓋刪除傳播**：`clean_stale_files` 為 `--clean` opt-in（預設關），「刪除 tracked .claude/ 檔 → push 帶 --clean」。W10-049 根因 = 刪除未帶 --clean push。→ Gap 3 部分已涵蓋，殘留為「opt-in 易忘 + framework-asset-separation 未連結此紀律」。

**Gap 2（孤兒稽核）**：現無「列本地有上游無之檔」報告；delta 路徑只涵蓋 base..HEAD 窗內。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題重框
不需重的「專門 stale-cleanup 流程」——核心刪除傳播已運作（delta D 自動跟刪 + W8-036.4 git-delete-aware fallback + sync-push --clean 跨專案傳播）。真正缺口是**可觀測性**（缺口 1 靜默殘留無通報、缺口 2 無孤兒稽核）+ **紀律連結**（缺口 3 --clean opt-in 易忘）。正解為改造既有，非新增工具（對照 W8-036.3 已拒 tombstone）。

### 三類方案性質涵蓋（toolification-bias 防護）

| 缺口 | 零工具 | 改造既有（採用）| 新增工具（不採）|
|------|-------|---------------|---------------|
| 1 靜默殘留 | 文件提醒人工比對 delta（不可靠）| **apply_upstream_delta 收集「上游刪+本地分歧保留」清單，pull 結尾通報（鏡像 W1-084 .sync-conflicts 殘留警告）** | 新建 residual-ledger（過度）|
| 2 孤兒稽核 | 教用戶手跑 git diff（不便）| **sync-pull 加 --audit 旗標，重用既有 _walk_upstream/diff 邏輯列「本地有上游無」之檔** | 新建獨立 audit skill（過度）|
| 3 紀律易忘 | DOC：framework-asset-separation 補「框架編輯含刪除走上游 + 刪除須 --clean push」連結 sync-push | **sync-push 偵測 tracked .claude/ 刪除但未帶 --clean → 結尾警告提醒**（soft，不強制改預設避免誤刪風險）| 強制 --clean 預設（風險：誤刪，opt-in 為刻意安全設計）|

### Attain Distance（機會成本/優先序）
缺口 1 是最高槓桿（靜默 = 最危險，所有客製專案受影響，修復成本最低，鏡像現成模式）→ P1。缺口 3 soft 提醒防根因復發 → P2。缺口 2 為週期性衛生 nice-to-have，delta+git-delete-aware 已覆蓋主路徑 → P2 較低，可評估是否值得。

### Premortem（P）
- R1 缺口 1 通報誤報：本地「合理保留」的客製檔（非孤兒）也會被列入「上游已刪」清單。緩解：措辭為「上游已刪，本地保留——若為孤兒請手動移除，若為刻意客製可忽略」，不自動刪、不阻擋。
- R2 缺口 3 改預設 --clean 誤刪：故不改預設，僅 soft 警告（保留 opt-in 安全設計）。
- R3 缺口 2 過度設計：先評估 --audit 是否 < 30 行重用既有邏輯；若需大改則 defer backlog。

### Spawn 落地規劃

| child | type | priority | 動作 | 關鍵約束 |
|-------|------|---------|------|---------|
| 缺口 1 靜默殘留通報 | IMP | P1 | apply_upstream_delta 收集 upstream-deleted+local-diverged 清單 + pull 結尾通報；含 reproduction test | 鏡像 W1-084 模式；措辭非阻擋（R1）|
| 缺口 3 紀律 | IMP+DOC | P2 | sync-push soft 警告偵測 tracked 刪除未帶 --clean + framework-asset-separation 補編輯紀律連結 | 不改 --clean 預設（R2）；DOC 派 basil 審文字 |
| 缺口 2 孤兒稽核 | IMP | P2 | sync-pull --audit 列本地有上游無之檔；先評估 < 30 行重用可行性，否則 defer | toolification-bias（R3）|
