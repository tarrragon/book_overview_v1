### 四處 git subprocess 封裝盤點對照表

| 封裝 | 簽名 | check | timeout | text | 例外處理 | 用途 |
|------|------|-------|---------|------|---------|------|
| `git_utils._run_git(cwd, *args)` | cwd-first | False | **無** | True | 呼叫端判 returncode | add/commit/rev-parse/diff |
| `lifecycle._auto_stage_git_add(paths)` | paths only | False | **無** | 預設(bytes) | 無 | 僅 git add |
| `track_snapshot._get_git_branch()` | 無參數 | 預設(False) | 5s | True | try/except 回 "unknown" | 僅 git branch |
| `checkpoint_state._run_subprocess(argv, cwd, timeout)` | argv+cwd+timeout | **True** | 參數 | True | 呼叫端 try/except | status/query/worktree |

### 差異分析

**形狀各異的根因是 check 語意差異（非單純疏漏）**：

- `git_utils` / `lifecycle`：`check=False`（graceful degrade 語意——呼叫端判 returncode，失敗不拋例外，body/操作保留）。這是 auto-commit/auto-stage 的**設計必要**（degrade 不中斷主流程）。
- `checkpoint_state`：`check=True`（拋例外語意——失敗即 raise，由 SAFE_CALL 捕獲 fallback）。
- 兩種 check 語意服務不同模組需求，**強制統一到單一 check 值會破壞其中一方**。

**確認的問題**：
- TD-2（真問題）：`git_utils._run_git` 與 `lifecycle._auto_stage_git_add` **無 timeout**，git hang（等認證/lock）會無限等待；另兩處有 timeout（track_snapshot=5s / checkpoint=參數）。
- TD-3（小瑕疵）：track_acceptance lazy import「避循環依賴」註解不精確（git_utils 僅 import subprocess+pathlib，無循環；真實理由是測試 patch + degrade 邊界）。

**薄封裝各自存在的合理性**：每處薄封裝便於 patch 局部 symbol（避免污染全域 subprocess.run），是測試隔離設計（W11-035 / W7-001 註解明示）。統一到單一 runner 會使 patch 點單一化，破壞各模組獨立 patch 能力。
