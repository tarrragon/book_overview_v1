### 問題根因（PM 前台分析 2026-06-11）

三層缺陷鏈共同造成靜默撞號風險：

| 層 | 位置 | 缺陷 |
|----|------|------|
| 掃描層 | `ticket_system/lib/ticket_builder.py:347` `get_next_seq` | 本地 glob + main ref（B3）兩來源同時掃空時回傳 max_seq+1 = 1，降級靜默無警告；環境依賴（worktree stale base / git 逾時 / root 解析偏差）可使兩來源皆空 |
| 配號層 | `ticket_system/commands/create.py:744` `_resolve_ticket_id` | 算出 ticket_id 後僅驗格式（validate_ticket_id），無「該 ID 是否已存在」檢查 |
| 落盤層 | `ticket_system/lib/parser.py:284` `save_ticket` | 無條件寫入，ID 撞號時靜默覆寫既有 ticket md |

重現驗證：main repo cwd 下 `get_next_seq('1.0.0', 1)` 回傳 50（正確），確認掃描邏輯本身正常；W1-039 事件的回傳 1 是兩來源同時掃空的環境態。事件中「list 短暫顯示新 title 後被 cleanup 還原、世界平面無新檔」與覆寫後被還原的行為吻合。

### 影響範圍

- `ticket_system/lib/ticket_builder.py`（get_next_seq 降級可觀測性）
- `ticket_system/commands/create.py`（_resolve_ticket_id collision guard）
- `ticket_system/tests/`（新增 collision 測試）

### 相關 Error Pattern

PC-152 collision 家族；quality-baseline 規則 4（降級不可靜默）。

### 修復設計（指導 Phase 3b）

1. **collision guard（核心）**：`_resolve_ticket_id` 算出 ID 後檢查 `get_ticket_path(version, ticket_id)` 是否已存在：
   - auto-seq 模式：存在則遞增 seq 直到找到可用值（防兩來源掃空時撞號）
   - 顯式 `--seq` 模式：存在則以 ErrorEnvelope 報錯退出（不覆寫、不自動跳號，尊重用戶顯式意圖）
2. **降級可觀測**：`get_next_seq` 兩來源皆空且 main ref 掃描為 None（降級）時，輸出 stderr warning（規則 4 雙通道）
3. **落盤後驗證**：create 流程 save_ticket 後驗證檔案存在於預期路徑，失敗時報錯
4. 不改 `save_ticket` 本身語意（update/complete 等既有呼叫端依賴覆寫行為），guard 放 create 層
