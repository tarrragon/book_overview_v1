### acceptance 1/2/4：runtime 排除缺口補強（已完成）

**檔案變更**：
- `.claude/hooks/lib/sync_exclude_manifest.py`
- `.claude/hooks/tests/test_sync_exclude_manifest.py`（新增）
- `.gitignore`（無需改動，4 pattern 既已涵蓋；測試交叉驗證確認）

**4 pattern 分流設計**（關鍵：避免 false positive 誤殺 live 內容）：

| pattern | 放置集合 | 原因 |
|---------|---------|------|
| `scheduled_tasks.lock` | `LOCAL_ONLY_PATTERNS`（part-level） | 名稱具名足夠，無誤殺風險 |
| `dispatch-active.lock` | `LOCAL_ONLY_PATTERNS`（part-level） | 同上（.json 已排除，.lock 補齊） |
| `logs` | `LOCAL_ONLY_ROOT_DIRS`（root-anchored） | 通用名，part-level 會誤殺 skills/*/logs/ |
| `state` | `LOCAL_ONLY_ROOT_DIRS`（root-anchored） | 通用名，part-level 會誤殺 skills/*/state/ |

**新增 root-anchored 機制（LOCAL_ONLY_ROOT_DIRS）**：
- 動機：實作初版用裸名 `state` 進 part-level 黑名單時，should_exclude 對 `skills/cc-release-impact-review/state/last-reviewed.md`（skill 維護的去重狀態，**live tracked 真檔**）回傳 True，會停止其同步並使 --clean 誤刪。此即 memory「stale 誤刪本專案特有防護」風險。
- 修復：`should_exclude` 新增第 5 規則——僅當「相對 claude_dir 路徑第一段」命中 LOCAL_ONLY_ROOT_DIRS 才排除。根層 `state/`、`logs/` 命中；巢狀 skill 內同名目錄不命中。
- `GITIGNORE_EXPECTED = LOCAL_ONLY_PATTERNS | LOCAL_ONLY_ROOT_DIRS`，session-start 交叉驗證 hook 涵蓋新 root dirs。

### acceptance 3：遠端 excluded-but-polluted 清除方案（設計 + 本地 clone 驗證，待 PM gate 執行）

**待刪遠端清單（本地 clone git -C 驗證，7 檔，全部 local tracked=NONE = 安全刪）**：
```
dispatch-active.json
dispatch-active.lock
hook-state/wrap-tripwire-state.json
logs/agent-dispatch.jsonl
logs/pm-automation-metrics.jsonl
scheduled_tasks.lock
state/last-session-start.marker
```

**重要邊界**：`skills/cc-release-impact-review/state/last-reviewed.md` 雖含 `state/` 段但**不在刪除清單**（root-anchored 設計正確排除），為 skill live tracked 內容，須保留同步。

**清除方案（PM gate，禁本代理人執行）**：
1. 在隔離 clone（`git -C "$TMP"`，禁裸 cd 防 chpwd 污染）執行：
   `git -C "$TMP" rm --cached <上述 7 檔>`
2. commit：`git -C "$TMP" commit -m "chore(W1-018.2): 清除遠端 excluded-but-polluted runtime 殘留（7 檔）"`
3. push：`git -C "$TMP" push origin <branch>`
4. 驗證：重新 clone 確認 7 檔已不在 ls-files

**為何 --clean 無法清這些**：dispatch-active.json / hook-state 早已在 should_exclude 命中 → --clean 遍歷時被跳過 → 永久殘留遠端。logs/state/lock 在本 ticket 補入 exclude 後同樣會被 --clean 跳過，故須一次性手動 git rm --cached（須在補 exclude 後執行，否則順序錯亂——本 ticket 補 exclude 已完成，清除即可進行）。
