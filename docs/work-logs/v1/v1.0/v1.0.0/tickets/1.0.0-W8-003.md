### 問題根因

W7 鏈（W7-001 auto-commit / 方案 C hook / auto-stage / 人為補充）全部接受同一未質疑前提：**「ticket body 必須以 git-tracked .md 檔案作為權威狀態」**。

直接根因（W1-017 ANA 已驗證）：ticket body 經 append-log 寫入 working tree 後若未即時 commit，會被三種 git 還原操作覆蓋——`git checkout -- <file>` / `git reset --hard` / `git stash`，body 退回 create commit 的 placeholder。

更深層根因（本 ANA 質疑點）：將「權威資料」與「git-tracked 檔案」綁定，使 body 暴露於 git 還原語意。W7-001 的 auto-commit 是在脆弱架構上加即時 commit 縮短暴露窗口，**未移除暴露本身**。反向解：資料層（authoritative state）與表現層（git-tracked md，渲染產物）分離，body 權威狀態存於不受 git 還原影響的獨立 store。

### 影響範圍

- `.claude/skills/ticket/ticket_system/lib/parser.py`：`parse_frontmatter` / `load_ticket` / `save_ticket`（資料層核心讀寫，每次以整檔 read_text/write_text round-trip md）
- `.claude/skills/ticket/ticket_system/lib/ticket_loader.py`：統一載入入口
- `.claude/skills/ticket/ticket_system/lib/worklog_appender.py` + `git_utils.py`：append-log + auto-commit
- 全系統 46 處 read/write ticket md 站點（grep `read_text|.open(|load_ticket|frontmatter`）
- 1926 個既有 ticket md 檔（遷移規模 baseline）
- hook 層：acceptance-gate / branch-verify 等多個 hook 直接讀 ticket md frontmatter/body
- sync 機制：實測 sync 範圍僅 `.claude/`（README-subtree-sync.md），ticket md 位於 `docs/work-logs/` **不在跨專案 sync 範圍**——ticket 資料為專案本地（重要：降低遷移複雜度，無跨專案 schema 對齊問題）

### WRAP 決策框架

### R — Reality Test（真根因驗證）

當前接受根因假設：「ticket body 易遺失」根因是「未即時 commit」（W7 框架）。
反向質疑後修正根因：真根因是「權威狀態與 git-tracked 檔案耦合」，使 body 落入 git 還原語意管轄。auto-commit 治標（縮窗口），資料層/表現層分離治本（移除暴露）。事實依據見「重現實驗結果」§實驗發現（代碼分析，非運行重現——根因本身已 W1-017 重現過）。

### W — Widen（擴增選項，含目前方案）

| 方案 | 描述 | 權威狀態載體 | git-tracked md 角色 |
|------|------|------------|-------------------|
| A（現狀，W7-001） | git-tracked md = 權威，append-log auto-commit | .md 檔（git） | 權威本身 |
| B（checkpoint_state 模式推廣） | body 權威存獨立非 git-tracked JSON/state store（如 `.claude/ticket-state/<id>.json`），md 降為渲染產物 | 獨立 JSON state | 渲染產物（可重生） |
| C（SQLite） | body + frontmatter 存 SQLite DB（單檔 `.claude/tickets.db`），md 為 export view | SQLite DB | 渲染產物（export） |
| D（hybrid：frontmatter git-tracked + body 獨立 state） | metadata（id/status/relations）續存 git-tracked md frontmatter（需 diff/blame/PR review），body 文字存獨立 store | 分裂：frontmatter=git、body=state | frontmatter 權威 + body 渲染 |
| E（不改架構，強化 A） | 維持 git-tracked md，僅以 hook + 文件強化「寫後立即 commit」紀律 | .md 檔（git） | 權威本身 |

### A — Attain distance（機會成本 / 反向思考）

反向思考「git-tracked md 作為權威」的**價值**（為何 W7 從未質疑）：
1. **可審計性**：git blame/log 提供 body 變更歷史與責任歸屬——獨立 state（B/C）會喪失或需自建 audit。
2. **PR review 可見性**：md diff 可在 PR 中 review；SQLite（C）二進位不可 diff。
3. **零外部依賴**：純檔案，無 DB engine、無 schema migration runtime——B/C 引入新依賴與失效模式。
4. **災難復原**：git 即備份；獨立 state 需另設備份。
5. **既有 1926 檔 + 46 站點 + 多 hook 直讀 md**：全屬「md = 權威」假設，遷移為大範圍 breaking change。

機會成本：投入 B/C/D 遷移會擠壓 v1.0.0 發佈收尾（雖受 APP 端阻塞）與其他 wave；且在缺 v2.0.0 context 下可能做出與未來 schema 不符的設計。

關鍵 reframe：W7 的痛點（git 還原覆蓋）本質是**未 commit 窗口**問題，W7-001 auto-commit 已將窗口縮至「append-log 返回前」近乎零。痛點實際嚴重度在 auto-commit 上線後已大幅下降——這削弱了「需架構級分離」的急迫性。

### P — Prepare to be wrong（行前預想失敗點）

最可能失敗原因（12 小時後回看）：
1. 在缺 v2.0.0 多書城 context 下倉促選 B/C，導致設計與 v2.0.0 實際需求（若有跨專案 ticket 共享 / 新 schema）不符，遷移白做。
2. 低估 1926 檔 + 46 站點 + hook 直讀的遷移波及面，B/C 遷移引入大量回歸。
3. 喪失 git audit/PR review 能力後，ticket 變更可追溯性下降，違反框架可觀測性原則（quality-baseline 規則 4 精神）。
防護：本 ANA 不在當前 context 下拍板 B/C 全面遷移；以版本錨點 trigger 綁定 v2.0.0 重評（decision-trigger-binding 規則 2）。

### 遷移成本評估（各方案）

> 假設與限制：缺 v2.0.0 多書城 context，以下成本為**架構推估非 PoC 實測**；若 v2.0.0 引入跨專案 ticket 共享或 schema 變更，B/C/D 成本與價值需重評。ticket 資料專案本地（非跨專案 sync），故無跨專案 schema 對齊負擔（降低 B/C/D 複雜度）。

| 方案 | CLI 影響 | hook 影響 | sync 影響 | worktree merge 影響 | 遷移成本（推估） |
|------|---------|----------|----------|-------------------|---------------|
| A 現狀 | 無 | 無 | 無 | merge conflict 偶發（已知，git checkout --theirs 採 worktree SSOT） | 0（已上線） |
| B 獨立 state | 46 站點改走 state API + md 渲染層；中大 | 多 hook 改讀 state；中 | state 若不入 git 則不隨 git 流動；worktree 跨進程 state 一致性新問題 | **解決 merge conflict**（md 為渲染產物可重生）；但 state 跨 worktree 同步成新問題 | 大（46 站點 + hook + 1926 檔遷移腳本 + state store 設計） |
| C SQLite | 46 站點改走 DB layer；大 | hook 改查 DB；中大 | DB 單檔可入 git 但二進位 merge 不可行；或不入 git 則同 B | DB 單檔並行寫鎖（worktree 多進程）成新問題；md export 可重生 | 大（同 B + DB engine + schema migration + 並行寫鎖設計） |
| D hybrid | frontmatter 站點不變、body 站點改 state；中 | frontmatter hook 不變、body hook 改；中 | frontmatter 續隨 git；body state 同 B 問題但範圍小 | frontmatter merge 同現狀；body 為渲染產物減少衝突 | 中（範圍小於 B/C，保留 git audit on metadata） |
| E 強化現狀 | 無 | 微（強化 commit 紀律 hook） | 無 | 同 A | 極小 |

### 結論（明確）

**採方向：E（短期保持強化現狀）+ 綁 v2.0.0 重評 D（中期 hybrid 為最優候選）；否決當前全面遷移 B/C。**

理由：
1. **否決當前 B/C 全面遷移**：W7-001 auto-commit 已將 git 還原窗口縮至近零，痛點急迫性大幅下降；B/C 大範圍 breaking（46 站點 + 1926 檔 + 多 hook）且喪失 git audit/PR review，機會成本高於收益。在缺 v2.0.0 context 下倉促遷移有設計白做風險（P 失敗點 1）。
2. **短期採 E**：維持 git-tracked md 權威 + 既有 auto-commit；痛點已被 W7-001 充分緩解，無需架構級變更。
3. **中期最優候選為 D（hybrid）而非 B/C**：D 保留 frontmatter 的 git audit/PR review/diff（metadata 最需要可追溯），僅將 body 文字降為渲染產物——以最小範圍換取「body 不受 git 還原」的核心收益，避開 B/C 喪失 audit 與 SQLite 二進位 merge/並行寫鎖問題。但 D 仍引入「body state 跨 worktree 一致性」新問題，須在 v2.0.0 有完整 context 時 PoC 驗證。
4. **trigger 綁定**：v2.0.0 多書城架構窗口開啟時重評 D（版本錨點 trigger，decision-trigger-binding 規則 2）；屆時若 v2.0.0 引入跨專案 ticket 共享需求，C（SQLite）可能因集中式查詢優勢重新進入考量。
