## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

`execute_set_field`（`.claude/skills/ticket/ticket_system/commands/fields.py`）透過 `DICT_FIELD_SUBKEY` 映射表將 set-where 的 positional value 僅寫入 `where.layer` 子欄位（W10-086 修復的設計），`where.files` 無任何 CLI 寫入路徑，保留 stale 值。

### 影響範圍

- 生產端：`ticket_system/commands/fields.py` `execute_set_field`（set-where 路徑）
- 消費端契約：`.claude/hooks/agent-dispatch-validation-hook.py` `_merge_ticket_scope` 讀 `where.files` 作 scope source of truth；L3 純 .claude/ 覆蓋要求 `(has_main=True, has_ext=False, has_other=False)`，stale files 使 L3 失效 → 派發誤擋（W1-061.1 三次派發失敗實證）
- 訊息層：`lib/command_lifecycle_messages.py`（新增同步輸出訊息）、`lib/command_tracking_messages.py`（HELP_SET_WHERE 更新）

### 相關 Error Pattern

無直接對應 PC。前置歷史：0.19.0-W3-072 ANA（set-where 子欄位 flag 評估，當時採 baseline-gated 維持現狀）、1.1.0-W1-010（監測 ticket）；本次 W1-061.1 實際誤擋事件使 PM 直接建本 IMP。

### 消費契約關鍵發現（影響設計）

dispatch hook `_classify_paths` 將任何不含 `.claude/` 的非路徑字串分類為 `has_other=True`。若 set-where 無條件把輸入值塞進 files（如描述性 layer "Framework Rules"），會破壞 L3 純 .claude/ 分類——產生與本修復同類的誤擋回歸。故同步必須路徑感知。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復設計（commit 62a2d931）

採**路徑感知同步**（無新增旗標）：set-where positional value 經 `_parse_where_path_entries` 解析——所有逗號分隔項目皆含 `/` 時視為路徑型輸入，同步覆寫 `where.files`（逐項 strip）；任一項目不含 `/`（描述性輸入如 "Domain Layer"）則僅更新 layer、files 保留。

變更檔案：

| 檔案 | 變更 |
|------|------|
| `ticket_system/commands/fields.py` | 新增 `_parse_where_path_entries` / `_sync_where_files` helper；`execute_set_field` dict 分支對 where 欄位觸發同步；stdout 輸出同步清單（可觀測性） |
| `lib/command_lifecycle_messages.py` | 新增 `FieldsMessages.WHERE_FILES_SYNCED` 訊息常數 |
| `lib/command_tracking_messages.py` | `HELP_SET_WHERE` 補述同步行為 |
| `tests/test_set_where_files_sync.py` | 新增 11 測試（5 單元 + 6 整合/回歸） |
| `tests/test_track_fields.py` | `test_set_where_success` 期待值更新至新契約 + 新增描述性輸入分支測試 |

### --files 旗標評估決策：不新增

1. **簡單方案優先**（派發指示）：路徑感知同步零學習成本，PM 既有 `set-where <id> "<paths>"` 工作流直接修復，無需記住新旗標（旗標方案下 PM 忘加 --files 會重現 stale 問題）。
2. **消費契約安全**：無條件同步會讓描述性 layer 值污染 files（dispatch hook `_classify_paths` 對非路徑項目回 `has_other=True`，破壞 L3 純 .claude/ 分類）；路徑感知判定（所有項目含 `/`）以零旗標達成兩側正確。
3. **歷史對齊**：W3-072 ANA 方案 A'（--files 旗標）當時因 baseline 25% 使用率被 G（維持現狀）擱置；本次有 W1-061.1 實證痛點但痛點本質是「同步缺失」而非「旗標缺失」，最小修復直接命中根因。
4. 若未來需要「files 與 layer 各自獨立設值」需求，再依 1.1.0-W1-010 監測 ticket 累積案例重評 A'。

### 路徑判定邊界（已知限制，文件化）

- 判定規則：項目含 `/` 即路徑。repo 根層級檔案（如 `build.js`）不含 `/` 不會觸發同步——可用 `./build.js` 形式或 Edit frontmatter；不採副檔名 regex 因 `v0.19.0` 等版本字串會誤判。
- 混合輸入（"Domain Layer, .claude/x.py"）整體視為 layer 描述、不同步，避免歧義。

### 既有測試契約更新說明

`test_track_fields.py::test_set_where_success` 原斷言「路徑型輸入後 files 保留 stale 值」——該斷言編碼的正是本 ticket 修復的 bug 行為（非本 ticket 的 RED 規格），故隨行為變更同步更新期待值並加註 W1-078 理由。W10-086 防壓扁契約（dict 結構保留）不受影響：`test_fields_dict_field_bug.py` 全綠（場景 2 描述性輸入 "Domain Layer" 走非同步分支，files 維持重建預設 `[]`）。
