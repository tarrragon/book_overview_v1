### V1↔APP 兩端 round-trip 驗證計畫

**目標**：驗證 V1 匯出 canonical JSON → APP 匯入 → APP 匯出 → V1 匯入，全鏈無損（書名、作者、tags、readingStatus、_passthrough 保留）。

**前置條件**（全部須滿足才能啟動驗證）：
1. APP v0.32.0 W1 completed（tag model + DB 建立）
2. APP v0.32.0 W2 completed（interchange 匯入匯出實作）
3. APP interchange 實作對齊 canonical v3.0.0 spec（format="book-interchange-v1", formatVersion="3.0.0"）

**驗證步驟**：

| 步驟 | 動作 | 驗證點 |
|------|------|--------|
| 1 | V1 匯出 canonical JSON（已有測試 fixture） | format="book-interchange-v1", formatVersion="3.0.0" |
| 2 | APP 匯入步驟 1 的 JSON | APP 內部 tag model 正確建立（書名、多值 author、readingStatus mapping） |
| 3 | APP 匯出 canonical JSON | 輸出 JSON 符合 v3.0.0 spec，含 tagTree + metadata |
| 4 | V1 匯入步驟 3 的 JSON | V1 內部 book model 還原，與步驟 1 原始資料比對無損 |
| 5 | Diff 比對步驟 1 vs 步驟 4 | 必要欄位完全相同；_passthrough 保留；extensions 可新增但不可丟失 |

**成功標準**：
- SC-1：步驟 5 diff = 0（必要欄位無損）
- SC-2：readingStatus 六態（want_to_read/reading/finished/abandoned/on_hold/not_set）雙向映射正確
- SC-3：多值 metadata（多 author、多 ISBN）round-trip 無損
- SC-4：_passthrough 欄位（V1 不認識的 APP 專屬 tag）保留不丟棄
- SC-5：formatVersion semver 檢查通過（APP 能 parse V1 輸出的版本號）
