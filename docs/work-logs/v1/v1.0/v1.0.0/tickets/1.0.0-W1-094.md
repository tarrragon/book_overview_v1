### 交付（實際檔案在 .claude/skills/framework-issue/，非原 where.files 的 sync-pull）
1. /framework-issue 新增 fix-status 命令：
   - `fix-status <issue-ref>`（view）：讀 issue body 的修復矩陣顯示哪些 consumer 修了。
   - `fix-status <issue-ref> --mark-fixed`（update）：標記「本 consumer」為已修，回寫 issue body。
2. 矩陣落點：issue body 內以標記區段管理（如 <!-- fix-matrix -->...<!-- /fix-matrix -->），內含 markdown 表格 `| consumer | status |`。read=gh issue view --json body 解析；write=更新該區段後 gh issue edit --body-file 回寫。
3. consumer 自我識別：沿用 .claude/error-patterns/_project-registry.yaml + git toplevel basename（同 error-pattern allocator 的 identify_project_code），勿要求手動傳 consumer 名。
4. 沿用 scripts/gh_common.py preflight 降級；gh 不可用清楚提示不 crash。
5. 測試：view 解析矩陣 / mark-fixed 更新本 consumer / 矩陣不存在時初始化 / 降級，至少 4 個（mock gh，勿真打 API）。

### 交付
1. `scripts/fix_status.py`（新增）：
   - `fix-status <issue-ref>`（view）：`gh issue view --json body` 取 body，解析 `<!-- fix-matrix -->...<!-- /fix-matrix -->` 區段內 `| consumer | status |` 表格，以表格印出各 consumer 修復狀態；矩陣不存在時提示用 --mark-fixed 初始化。
   - `fix-status <issue-ref> --mark-fixed`：把本 consumer 列標為 fixed，矩陣存在則整段替換、不存在則初始化 append；以 tempfile 走 `gh issue edit --body-file` 回寫。
   - consumer 自我識別：`get_git_toplevel()` + `identify_consumer()` 反查 `_project-registry.yaml`（同 allocator identify_project_code 邏輯），不接受手動傳入；未登錄降級報錯。
   - 沿用 gh_common.preflight()/run_gh()/emit_degraded() 降級；fetch_body 與 edit 皆走 subprocess（mock 攔截點），不真打 API。
2. SKILL.md：Commands 表加 fix-status 列、範圍註記改「四命令齊備」、新增 fix-status Usage 段（含矩陣 SSOT 說明與 consumer 自我識別）。
3. 測試：append 4 個於既有 test_framework_issue.py（view 解析 / mark-fixed 更新本 consumer / 矩陣初始化 / gh 未裝降級），全 mock gh + git toplevel。
