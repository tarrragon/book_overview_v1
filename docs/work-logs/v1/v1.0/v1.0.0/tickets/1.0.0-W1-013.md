## Problem Analysis

### 評估標的：功能字串 vs docstring/comment 的本質區別

W1-011 將框架 ticket ID 引用分為文檔論述（已處理）與「功能字串」（運行時輸出給用戶的訊息）兩類。本 ANA 評估後者。

| 載體 | 讀者 | sync 後影響 |
|------|------|-----------|
| docstring / comment | 讀程式碼的開發者 | 歷史錨點型可保留（W1-010 判準）；開發者理解「此設計源於哪個 ticket」 |
| 功能字串（print/msg/f-string） | 運行時所有用戶（含其他專案） | 引導用戶查不存在的 ticket，誤導面比 comment 廣 |

### 盤點結果

| 類型 | 數量 | 範例 | 性質 |

| 佔位符示範 ID | ~15 | `print("ticket track claim 0.31.0-W4-001")`、doctest `>>> validate("0.31.0-W3-001")` | 格式/usage 示範，非真實引用，規則 8 教學示範豁免（保留） |

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 決策結論

6 處真實 ticket ID 功能字串應清理（移除 ticket ID 或抽象化為功能描述/檔案路徑）；佔位符示範 ID（約 15）保留。

理由：功能字串是運行時輸出給所有用戶（含其他專案），sync 後引導查不存在的 ticket，誤導面比 comment 廣（comment 僅給讀碼開發者）；6 處均通過移除測試（移除 ticket ID 後訊息主旨可理解）。此結論與 W1-010 判準一致——功能字串無「設計脈絡記錄」價值（運行時用戶要 actionable 提示，非設計溯源）。

### WRAP

- W（Widen）：(a) 全清理 6 處 ← 採用；(b) 全保留（溯源價值）← 否決（運行時用戶要 actionable 提示非設計溯源）；(c) 改指向框架路徑 ← 部分採用（有對應框架資產者改路徑，如 quality-baseline.md）
- R（Reality Test）：見重現實驗結果（6 處確認 + 移除測試通過）
- A（Attain distance）：清理成本低（6 處字串）但需同步測試斷言；機會成本低
- P（Prepare to be wrong）：最可能失敗——改字串致測試斷言紅燈。防護：spawn IMP 明示「改字串 + grep tests/ 斷言同步」
