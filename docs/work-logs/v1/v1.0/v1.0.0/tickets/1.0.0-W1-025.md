## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

append-log 的「目標章節不存在 → SECTION_NOT_FOUND 失敗」摩擦有兩層成因：

1. **模板缺口**：`create_ticket_body`（ticket_builder.py）的 IMP 模板僅預生成 7 個 Schema 章節（Task Summary / Problem Analysis / Solution / Test Results / NeedsContext / Exit Status / Completion Info），未含 `## Context Bundle`。PM 派發前依 PC-040 寫 Context Bundle 時必然命中 SECTION_NOT_FOUND，被迫繞道 Edit 手動插入章節（W1-023.4.1 派發實證）。
2. **fail-fast 三輪往返**：前置檢核（status precondition / section 白名單 / 章節存在性）逐項 return，呼叫端修一項重跑一次才看到下一項失敗（W1-024 complete 前複審 adversarial 維度發現）。

### 影響範圍

- `.claude/skills/ticket/ticket_system/commands/track_acceptance.py`（`_execute_append_log_locked`）
- `.claude/skills/ticket/ticket_system/lib/ticket_builder.py`（SCHEMA_H2_SECTIONS 周邊）
- `.claude/skills/ticket/ticket_system/lib/command_tracking_messages.py`（VALID_SECTIONS / 新訊息常數）
- 既有測試 2 檔（tests/test_track_acceptance.py、tests/test_error_channel_integration.py）斷言舊行為，需對齊

### 相關 Error Pattern

- PC-040（context 存 ticket 不存 prompt）：本摩擦的上游動機——PM 必須能順暢寫 Context Bundle 到 ticket
- PC-087（禁 /tmp 中介）：heredoc 直傳依賴 append-log 可用性，章節缺失迫使繞道
- 記憶項「append-log 不接受 Completion Info」已過時：W3-099 已完成 10 章節白名單對齊，本票以 SSOT guard 測試固化（見 Solution）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 方案決策：採方案 B（CLI 自動補建），不採方案 A（模板預生成）

| 比較維度 | 方案 A：IMP 模板預生成空 Context Bundle | 方案 B：append-log 自動補建缺失 Schema 章節（採用） |
|---------|--------------------------------------|------------------------------------------------|
| 既有 ticket 治癒 | 否（只影響未來 create） | 是（任何已存在 ticket 直接受益） |
| 覆蓋範圍 | 僅 Context Bundle 單一章節 | 全部 9 個 SCHEMA_H2_SECTIONS（含 ANA 以外 type 的重現實驗結果等邊角） |
| 模板噪音 | 所有 IMP 多一個多數情況用不到的 placeholder 章節 | 章節按需 materialize，模板保持精簡 |
| 風險邊界 | 無 | 以 SCHEMA_H2_SECTIONS 白名單 + canonical 順序插入收束，無任意章節建立 |

W1-023.4.1 的摩擦場景是「已建立的 pending IMP」，方案 A 對其無效；方案 B 為嚴格超集，故不採組合。

### 實作內容

1. **`insert_missing_schema_section`**（ticket_builder.py 新函式）：缺失章節依 SCHEMA_H2_SECTIONS canonical 順序插入（錨點 = 第一個順序更晚的既有 Schema H2；無錨點則補於 body 末尾），含 type-aware Schema 標註與首筆內容一次寫入。非 Schema 章節回傳 None 由呼叫端走既有錯誤路徑。
2. **append-log 前置檢核聚合**（track_acceptance.py）：status / section 白名單 / 章節存在性三項一次評估、一次列全。輸出通道不變（status → stderr；section 類 → stdout）；exit code 契約不變（status 失敗 = 2，其餘 = 1）。status precondition 語意未動（in_progress 限制的調整屬另案範圍）。
3. **章節存在性語意調整**：缺失僅在非 Schema 章節（Execution Log H2）時構成錯誤；Schema 章節缺失改走寫入階段自動補建，stdout 提示 `SECTION_AUTO_CREATED_FORMAT`。
4. **白名單對齊（W1-024 A4 併入）**：確認 VALID_SECTIONS 已由 W3-099 完成 10 章節對齊（含 Completion Info），本票新增 SSOT guard 測試 `test_valid_sections_covers_all_schema_sections`（VALID_SECTIONS == SCHEMA_H2_SECTIONS ∪ {Execution Log}）防回歸。
5. **既有測試對齊**：3 個斷言「Schema 章節缺失 → rc 1」的測試改斷言新行為，誤匹配防護意圖保留（`## Solutions` / `## Solution alt` 不被汙染 + 新內容落於獨立補建章節）；error-channel scenario 3 改用非 Schema 章節驗證 SECTION_NOT_FOUND 通道。

### 不在本 ticket 範圍

- status precondition（pending 不可 append-log）的放寬：另案接續處理。本票完成後，「PM 一步寫 Context Bundle 到 pending IMP」的章節存在性障礙已消除，status 障礙待該案
- H2→H3 自動降級（W1-068）不受影響，自動補建路徑同樣套用（有測試覆蓋）
