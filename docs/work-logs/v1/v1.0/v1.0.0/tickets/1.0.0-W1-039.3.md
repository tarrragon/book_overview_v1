## 重現實驗結果

### 實驗方法

以三組第一手指令實證 hook stdin / transcript 識別欄位可得性（不憑假設，PC-143）：
1. `grep` 既有 PreToolUse hook 對 stdin payload 欄位的消費點（session_id / transcript_path / hook_event_name）。
2. 讀 `.claude/hooks/hook-logs/context-depth-warning.log` 取真實 session_id UUID 樣本。
3. python3 掃描 transcript jsonl top-level key union，定位 promptId / cwd 並取真實樣本。

### 實驗執行

1. **stdin 欄位消費點**：context-depth-warning-hook.py:154-155 讀 `transcript_path` + `session_id`；dispatch-count-guard-hook.py:331 讀 `hook_event_name`；branch-verify-hook.py:204-205 讀 `tool_name`/`tool_input`。framework-rule-edit-skill-trigger-hook.py 以 session_id 建 per-session cache。
2. **真實 log 佐證**：context-depth-warning.log 含 `session='55709aa6-7726-4838-8b08-1c410e3b48b2'` 等真實 UUID 逐筆記錄（2026-06-09 多筆）。
3. **transcript key union**（200 筆）：含 `promptId`、`sessionId`、`cwd`、`requestId` 等。promptId 真實樣本 `c1b8938b-c789-4360-a34e-9e43fff16bcc`（UUID）；同記錄 sessionId `8dfa33d4-...`、cwd `/Users/mac-eric/project/book_overview_v1`。
4. **缺欄確認**：全 codebase 無 hook 從 stdin 讀 promptId 或 cwd（cwd 一律 Path.cwd()）。

### 實驗發現

**已驗證事實**：
- sessionId、transcript_path、hook_event_name 由 stdin 直接可得（既有 hook 消費 + 真實 log 佐證）。
- promptId、cwd 須經 transcript_path 二次讀取派生（transcript 含此欄，stdin 不含）。
- 識別三元組 (sessionId, promptId, cwd) 可完整組出。

**關鍵反證（推翻原 how 假設）**：W1-039 ghost branch 為同一 runtime 內 turn fork/resume-replay 的兩條流，共享同 pid/sessionId/promptId（鑑識三欄全同），故 (sessionId+promptId+pid) lease 在真兇場景不可區分，PreToolUse lease 對 M2/M3 失效（對 M5 真並行有效）。

**候選方案**（詳見 Solution）：A. ticket CLI 冪等鎖（推薦）、B. git pre-commit 指紋去重、C. PostToolUse 審計 lease。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### Acceptance 1：hook 可得識別欄位實證（PC-143 不憑假設）

**stdin payload 直接可得**（既有 hook 已消費，非推論）：

| 欄位 | 來源 hook 實證 | 真實值樣本 |
|------|--------------|-----------|
| `session_id` | context-depth-warning-hook.py:155、framework-rule-edit-skill-trigger-hook.py（per-session cache） | hook log 實錄 `55709aa6-7726-4838-8b08-1c410e3b48b2`（UUID） |
| `transcript_path` | context-depth-warning-hook.py:154、auq-option-pattern-detector-hook.py:309 | `~/.claude/projects/<slug>/<sessionId>.jsonl` |
| `hook_event_name` | dispatch-count-guard-hook.py:331、file-ownership-guard-hook.py:294 | `"PreToolUse"` |
| `tool_name`/`tool_input` | branch-verify-hook.py:204-205 等 | `"Bash"` + `{command}` |

證據強度：`context-depth-warning.log` 含真實 UUID session_id 逐筆記錄，證明欄位真實送達。

**stdin 不直接含、須間接取得**：

| 欄位 | 取得方式 |
|------|---------|
| `promptId` | 讀 transcript_path 末筆含 promptId 記錄（transcript top-level key 實證存在，樣本 `c1b8938b-...`）。注意：與 W1-039 ANA 的 `44875861` 非同物件（後者為 meta.json toolUseId 截斷） |
| `cwd` | transcript 含 cwd 欄；或 hook 自身 Path.cwd()（全 hook 皆此法） |
| `pid` | os.getpid()——但見 Acceptance 2 致命限制 |

結論：**識別三元組 (sessionId, promptId, cwd) 全部可組出**；sessionId/transcript_path 直接得，promptId 經 transcript 二次讀取派生。

### Acceptance 2：lease 方案可行性結論

**致命限制：pid 對 W1-039 真兇場景無鑑別力。** W1-039 ghost branch（M2）為同一 runtime 內 turn fork / resume-replay 的兩條執行流，非兩個 OS 進程——共享同一 pid（os.getpid 同值）、同 sessionId、同 promptId（鑑識實證三欄全同）。故原 how 假設的 (sessionId+promptId+pid) lease **在真兇場景三欄全同、hook 視角不可區分，設計失效**。

**退而求其次方案（推薦序）**：

| 方案 | 機制 | 能擋 W1-039 | 評估 |
|------|------|------------|------|
| A. ticket CLI 寫入冪等鎖（推薦） | create/complete/append-log 寫入前對「同 ticketId + 同語意 op-hash」N 秒窗口查 lease，重複拒絕 | 部分（擋重複 spawn/complete，正是主危害） | 不依賴 pid，以操作語意指紋為 key，最貼合帳務危害層 |
| B. git pre-commit 訊息指紋去重 | 比對近 N 秒近重複 commit message（5df3d5f4 vs 7c21ca64） | 部分 | 與 IMP-046 同層可加掛 |
| C. PostToolUse 審計 lease（不互鎖僅觀測） | 寫入後記錄 (sessionId,promptId,op,ts) append-only，同 promptId 短窗 ≥2 同語意寫入則 WARN | 偵測不阻擋 | 零誤殺，並供 W1-041 補證 |

**PreToolUse hook lease 裁決**：欄位可得但兩執行體三欄同值 → 對 M2/M3 無效；對 M5 真並行（不同 sessionId）有效。屬**場景錯配非全盤否定**，保留為 M5 未來選項。

**推薦落地組合**：方案 A（主，擋核心危害）+ 方案 C（輔，零誤殺偵測 + W1-041 補證）；PreToolUse lease 暫不建。落地須另開 IMP ticket（本票純評估）。

### Acceptance 3：與 IMP-046（index.lock 防護）邊界釐清

| 維度 | IMP-046 | 本 lease 評估 |
|------|---------|--------------|
| 防護對象 | git 寫入競爭（同瞬間搶 index.lock） | 並發執行體重複帳務（跨時間窗） |
| 觸發層 | PreToolUse(Bash git)，清理 stale lock | ticket CLI 寫入層（A）/ PostToolUse 審計（C） |
| 偵測粒度 | 檔案鎖存在性 + 時齡（>5s） | 操作語意指紋 + 時間窗 |

**互補不重疊**：IMP-046 防物理碰撞（同一瞬間）；W1-039 雙執行體 commit 相隔數分鐘（03:20:59 vs 03:27:10），IMP-046 結構性看不到。lease 方案 A/B 防邏輯重複帳務（跨時間窗），正補此洞。

### 後續落地（spawn 規劃）

本票為 ANA spike 純評估，未部署。建議落地動作須另開 ticket：

| 項目 | 類型 | 建議 |
|------|------|------|
| 方案 A：ticket CLI 冪等鎖 | IMP | 推薦優先落地（擋核心危害） |
| 方案 C：PostToolUse 審計 lease | IMP | 與 A 並行，零誤殺 + 供 W1-041 補證 |

> 由 PM 裁決是否轉 IMP；本 spike 不自建 ticket（避免 ANA 越界，依 quality-baseline 規則 5 由 PM 統一規劃 spawn）。
