### 問題根因

`lib/blocker_resolution.py` 的 `is_fully_unblocked()`（W1-009 round-trip 自 APP W8-043 推回 V1）目前是孤兒檔——除自身外無任何模組 import。V1 仍存在兩處未接線的重複/缺陷邏輯：

1. **lifecycle.py 內重複定義**（`commands/lifecycle.py:1071-1115`）：自帶一份 `_is_fully_unblocked`，與 lib 版本邏輯完全相同（含 docstring），屬 DRY 違反。由兩處 delegate：`_check_unblocked_tickets`（:1130, `include_closed_as_resolved=False`）與 `_can_cascade_unblock`（:1501, `include_closed_as_resolved=True`）。
2. **runqueue 未採用，仍用字面判定**（`commands/track_runqueue.py:92-97`）：`_is_unblocked_pending` 以 `len(blockedBy) == 0` 字面判定 ready，正是 `blocker_resolution.py` docstring 所述 W8-042 缺陷——blocker 已 completed 但 `blockedBy` 欄位未清理時，ticket 被誤判為 blocked，PM 在 runqueue 看不到實際可接手 ticket。

### 影響範圍

| 檔案 | 變更性質 | 風險 |
|------|---------|------|
| `commands/lifecycle.py` | 純 DRY：import lib 版本、移除 :1071-1115 重複 fn、改 :1130/:1501 兩呼叫點 | 低（邏輯等價替換） |
| `commands/track_runqueue.py` | 行為修正：`_is_unblocked_pending` 改用 `is_fully_unblocked(t, ticket_map, include_closed_as_resolved=True)` | 中（runqueue ready 語義變更，需傳 ticket_map） |
| `lib/blocker_resolution.py` | 不修改（已存在，作為唯一 SSOT） | — |

**runqueue 接線細節**：`_is_unblocked_pending` / `_is_listable` 目前只收單一 ticket，需擴充簽名傳入 `ticket_map`。呼叫點 `track_runqueue.py:309`（`runnable = [t for t in tickets if _is_listable(t)]`）的 `tickets` 即全集，可在該處建 `ticket_map = {t.get("id"): t for t in tickets if t.get("id")}`（:370 已有相同建法可參照）。`include_closed_as_resolved` 取值需經 sage 測試設計確認：scheduler 場景應與 cascade 一致採 `True`（completed+closed 皆解除），但保守起見可由測試案例驅動最終決定。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 變更摘要

整合 `lib/blocker_resolution.py` 的 `is_fully_unblocked`（W1-009 round-trip SSOT，原為孤兒檔）至 lifecycle 與 runqueue，消除 DRY 違反並修正 runqueue ready 字面判定缺陷（W8-042）。

| 檔案 | 變更 |
|------|------|
| `lib/blocker_resolution.py` | 不修改（唯一 SSOT） |
| `commands/lifecycle.py` | import `is_fully_unblocked`；移除本地重複 `_is_fully_unblocked`（原 :1071-1115，與 lib 版完全相同）；改寫兩呼叫點：`_check_unblocked_tickets`（`include_closed_as_resolved=False`，保留 conservative 建議列表語義）、`_can_cascade_unblock`（`True`）；docstring 引用名同步更新 |
| `commands/track_runqueue.py` | import `is_fully_unblocked`；`_is_unblocked_pending` 從字面 `len(blockedBy)==0` 改用 `is_fully_unblocked(..., include_closed_as_resolved=True)`，新增 `ticket_map` 參數（None 時退回字面，向後相容）；`_is_listable` 同步擴充簽名；`_render_list`（:309）建 `ticket_map` 傳入 |
| `commands/track_dashboard.py` | `load_top_ready` 呼叫 `_is_unblocked_pending(t)` 改傳 `ticket_map`（影響範圍分析發現的第三呼叫端，IMP-003 防護） |

### 設計決策

- `_is_unblocked_pending` / `_is_listable` 新增 `ticket_map` 參數採 Optional 預設 None：`None` 時退回原字面判定，確保未提供全集的潛在呼叫端不破壞；runqueue/dashboard 兩個真實呼叫點皆顯式建 ticket_map 傳入，啟用正確的 AND 語義。
- runqueue 採 `include_closed_as_resolved=True`：與 cascade unblock / scheduler 場景一致（completed 與 closed blocker 皆視為已解除）。

### 影響範圍分析（IMP-003 作用域變更防護）

`grep -rn "_is_unblocked_pending\|_is_listable"` 發現除 runqueue 內部呼叫外，`track_dashboard.py:99` 亦呼叫 `_is_unblocked_pending`。簽名擴充後已同步更新該呼叫端傳入 ticket_map，無遺漏。
