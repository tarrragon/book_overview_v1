#### 因果鏈（主要根因模型）

1. **2026-06-05 10:00 commit 2c094187**（0.19.1-W1-046）：兩個 SubagentStop hook 的輸出從 top-level `systemMessage`（純顯示）改為 `hookSpecificOutput.additionalContext`（注入對話），gate 條件 CC >= 2.1.163（當前 runtime 實測 2.1.173）。
2. CC 對 Stop/SubagentStop 的 additionalContext 語意是「把 context 注入正在停止的 agent 對話並令其繼續」（此即該欄位設計目的）。subagent 收到注入的 [WAIT]/commit 警告文字 → 做一個短回合（5-15 秒）→ 再次停止 → SubagentStop 再次觸發。
3. 每次再觸發時：dispatch-cleanup 發現 agent_id 已被首次事件清掉（log 出現「無匹配記錄（精準和 FIFO 兩路徑皆失敗）」WARNING）→ 但 dispatch-active.json 仍有其他執行中 agent（W1-052 場景的 general-purpose）→ remaining>0 → 再次輸出 [WAIT] additionalContext → 迴圈自我維持。
4. commit-verification hook 同步每事件重掃 git status + 5 分鐘 hook-error 視窗（HOOK_ERROR_SCAN_MINUTES=5），每次重播逐字相同的警告——解釋「三次零差異通知」與 W1-074/075/076 的 ERROR 重複播報第二症狀（同根因）。
5. 每次迴圈迭代的 additionalContext 同時喚醒 PM 主線程一次 → 11+ 次重複喚醒。
6. 迴圈終止條件推測為 parent 收割 agent 結果或 runtime 上限；觀測鏈長 5-17 次。

CC hook 規格已預見此迴圈：Stop/SubagentStop input 含 `stop_hook_active` 欄位（runtime 因 stop hook 而繼續時為 true），hook 應檢查以避免無限迴圈。**兩個 hook 皆未檢查此欄位**——這是迴圈未被斷路的直接缺口。

#### 為何 6/5 前無此問題

systemMessage 是純顯示通道（不注入對話、不引發 agent 繼續），故 6/4-6/5 每 agent_id 僅 1 次事件。改用 additionalContext 後通道語意改變（顯示 → 注入+繼續），迴圈條件成立。

#### 附帶發現（quality-baseline 規則 5，需 PM 建票追蹤）

1. **PC-148 模式真實存在於 Stop 事件**：`handoff-auto-resume-stop-hook.py` 同時註冊於 settings.json Stop 與 settings.local.json Stop（settings.local.json:196-207），每次主線程 Stop 執行兩次。與本票 SubagentStop 重複喚醒無關，但屬獨立雙重註冊缺陷。
2. **路徑首字元截斷 bug 根因確認**：agent-commit-verification-hook.py:159 `result.stdout.strip()` 會剝除 porcelain 第一行的前導空白（worktree-modified 格式為「空格M空格filename」），隨後 line[3:] 多切一字元。兩筆觀測樣本完全吻合：docs/ → ocs/（掉 d）、.claude/ → claude/（掉 .）。僅影響 stdout 第一行且該行以空白開頭時，故非每次出現。
3. **stale 快照誤報**：commit-verification 在 SubagentStop 當下讀 git status，與 append-log auto-commit 存在 race；加上迴圈重播放大，乾淨工作區仍收到舊警告。

### 實驗發現

**已驗證的事實**：

1. SubagentStop 對同一 agent_id 重複觸發 5-20 次（log 檔案級證據，每檔一事件）
2. 重複模式起始日（6/6）與 additionalContext 切換 commit（6/5 10:00）+ 後續 session 啟動時間精確吻合；切換前每 agent_id 恆為 1 次
3. [WAIT] 重播是 hook 無條件廣播（remaining>0 即輸出）疊加事件重複的結果；兩個 SubagentStop hook 皆無 dedup、皆未檢查 CC 規格中防迴圈的 `stop_hook_active` 輸入欄位
4. PC-148 雙重註冊假說對 SubagentStop 不成立（但在 Stop 事件上成立，見 Problem Analysis 附帶發現 1）
5. 路徑首字元截斷 = stdout.strip() off-by-one（程式碼級確認，兩觀測樣本 100% 吻合）

**仍未驗證的假設**（需 IMP 以受控實驗確認）：

- H1（主要假說）：additionalContext 注入導致 subagent 繼續→再停止→事件再觸發的自激迴圈。時間相關性 + 通道語意（systemMessage 純顯示 vs additionalContext 注入對話）+ 鏈波形（收尾期短叢集而非全程連續）均支持，但未做隔離實驗（如：hook 檢查 stop_hook_active 為 true 時靜默，觀察迴圈是否消失）
- H2（替代假說，較弱）：CC 2.1.163+ 自身改變 SubagentStop 觸發語意（每回合觸發而非僅最終停止）。與「事件僅在收尾期短叢集出現、鏈長 5-20 而非數百（agent 工具呼叫數）」不符，但因 runtime 升級與 hook 切換同期發生，無法純靠時間軸排除
