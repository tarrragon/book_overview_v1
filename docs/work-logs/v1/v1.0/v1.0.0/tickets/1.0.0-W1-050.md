## Problem Analysis

### 問題根因（PM 前台分析 2026-06-11）

lock 機制位於 `ticket_system/lib/file_lock.py:76`：以 `{target_path}.lock` 形式在目標檔旁建 lock 檔（fcntl 設計，不刪檔避免 unlink race，W14-042 文件化）。測試若未 patch project root / tickets dir，`get_ticket_path` 解析到真實 `docs/work-logs/`，lock 檔即落在真實 tickets 目錄——v0.18.0 tickets/ 的 `dummy.md.lock`、`fake-id-for-test.md.lock`、`P0.md.lock`、`T-1.md.lock` 等測試命名 lock 即此來源證據。

與 W1-053（單 namespace mock 致降級分支探測真實 repo）同根因家族：**測試對 project root 的隔離不完整**。本 ticket 處理 lock 寫入面，W1-053 處理讀取探測面。

### 影響範圍

- 來源候選：grep dummy/fake-id-for-test/nonexistent/T-1 命中 10+ 測試檔（含 conftest.py），需逐一判定哪些實際走到 file_lock 且未隔離（PC-162：執行者應獨立重新驗證）
- 殘留：`find docs/work-logs -name "*.lock"` 現存 278 個（0-byte、已 gitignore，rm 無 git 影響）

### 相關 Error Pattern

W1-018.1（mock 路徑函式須回真 tmp，632 .lock 事件）、PC-162（接手者獨立驗證）。

### 修復方向（執行者評估兩案）

- 案 A（逐測試修）：定位實際產生真實路徑 lock 的測試，fixture 改導向 tmp_path（仿 test_create_auto_seq_collision.py 的 _patch_project_root 雙 namespace 模式）
- 案 B（治本評估）：conftest.py 加 autouse fixture 預設 patch project root 到 tmp_path，個別需要真實 repo 的測試顯式 opt-out——若現有測試大量依賴真實 repo 則成本過高，記錄評估結論即可（採或不採皆需顯性理由寫入 Solution）
- 清掃：`find docs/work-logs -name "*.lock" -delete`（限定 *.lock）
- 驗證：全套件跑完後 `find docs/work-logs -name "*.lock" | wc -l` 為 0，並記錄測試總數防 skip 假陽性

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 實證定位（PC-162 獨立驗證）

清掃全部 lock 後跑全套件，find 新增 lock 反推來源：23 個 lock 落在真實 docs/work-logs/（v0.18.0 / v0.31.0 / v0.1.0 三版本 tickets/），含 dummy.md.lock、fake-id-for-test.md.lock、0.18.0-W17-998.md.lock 等測試命名 lock。grep dummy/fake-id 命中 40+ 檔不可靠（多為無關字串匹配），改以「跑單檔後 find 新增 .lock」實證。

根因：paths.get_project_root() 第一優先讀 CLAUDE_PROJECT_DIR，未設時 fallback git rev-parse --show-toplevel 解析到真實 repo root。測試呼叫 file_lock(get_ticket_path(version, id)) 但未 patch 路徑解析時，lock 落在真實 tickets/（file_lock W14-042 設計不刪 lock 檔，故殘留）。既有 session fixture _assert_no_repo_pollution 只檢查 v0/ 新增目錄，lock 落在既有目錄內不觸發，故污染未被攔截。

### 案 A vs 案 B 評估

採案 B（conftest autouse patch + opt-out）。理由：

- 污染分散於多個測試檔（race 測試、create 測試、fields 測試等），案 A（逐測試修 fixture）需修 10+ 處且未來新測試易再犯，治標不治本。
- 案 B 利用 get_project_root() 既有第一優先 CLAUDE_PROJECT_DIR，patch 該環境變數即重導所有路徑解析至 tmp，最小侵入；已有 13 個測試自設 CLAUDE_PROJECT_DIR，證明此為既定隔離慣例。
- opt-out 自然達成：用 patch.dict(os.environ, {}, clear=True) 或顯式 setenv 的測試（如 test_paths_get_project_root）後注入勝出，autouse 不破壞。

未採案 A 的顯性理由：逐測試修無法防範未來新測試重蹈覆轍，且工作量與遺漏風險均高於 conftest 集中防護。

### 實作

1. 兩 conftest.py（tests/ + ticket_system/tests/）加 autouse _isolate_project_root：建含 docs/work-logs + .claude/tickets + CLAUDE.md 的最小 tmp repo，setenv CLAUDE_PROJECT_DIR。
2. 兩 conftest.py 加 real_repo_root opt-out fixture：還原真實 repo root，供需真實版本偵測（version=None → 讀 todolist/掃 work-logs）的測試使用。
3. 處理 16 個受影響測試：
   - test_create_error_envelope.py（8）/ test_create_ux_merged_validation.py（2）：module autouse 引用 real_repo_root（走 create early-exit 錯誤路徑，不抵達 file_lock）。
   - test_error_channel_integration.py scenario_4：加 real_repo_root（subprocess 繼承 os.environ）。
   - test_error_channel_integration.py scenario_1：改用 tmp 建 ticket + CLAUDE_PROJECT_DIR 導向（原用真實 ticket，append-log 的 file_lock 包圍 section 驗證會在真實目錄留 lock）。
   - test_fields_extended.py ticket_env / test_fields_dict_field_bug.py：chdir 後顯式 setenv CLAUDE_PROJECT_DIR 回各自 tmp。

未改 file_lock.py 的「不刪 lock」設計（W14-042 unlink race 取捨）。
