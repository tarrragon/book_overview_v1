## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因（PM 前台分析 2026-06-11）

lock 機制位於 `ticket_system/lib/file_lock.py:76`：以 `{target_path}.lock` 形式在目標檔旁建 lock 檔（fcntl 設計，不刪檔避免 unlink race，W14-042 文件化）。測試若未 patch project root / tickets dir，`get_ticket_path` 解析到真實 `docs/work-logs/`，lock 檔即落在真實 tickets 目錄——v0.18.0 tickets/ 的 `dummy.md.lock`、`fake-id-for-test.md.lock`、`P0.md.lock`、`T-1.md.lock` 等測試命名 lock 即此來源證據。

與 W1-053（單 namespace mock 致降級分支探測真實 repo）同根因家族：**測試對 project root 的隔離不完整**。本 ticket 處理 lock 寫入面，W1-053 處理讀取探測面。

### 影響範圍

- 來源候選：grep dummy/fake-id-for-test/nonexistent/T-1 命中 10+ 測試檔（含 conftest.py），需逐一判定哪些實際走到 file_lock 且未隔離（PC-162：執行者應獨立重新驗證）
- 殘留：`find docs/work-logs -name "*.lock"` 現存 278 個（0-byte、已 gitignore，rm 無 git 影響）

### 相關 Error Pattern

W1-018.1（mock 路徑函式須回真 tmp，632 .lock 事件）、PC-162（接手者獨立驗證）。

### 修復方向（執行者評估兩案）

- 案 A（逐測試修）：定位實際產生真實路徑 lock 的測試，fixture 改導向 tmp_path（仿 test_create_auto_seq_collision.py 的 _patch_project_root 雙 namespace 模式）
- 案 B（治本評估）：conftest.py 加 autouse fixture 預設 patch project root 到 tmp_path，個別需要真實 repo 的測試顯式 opt-out——若現有測試大量依賴真實 repo 則成本過高，記錄評估結論即可（採或不採皆需顯性理由寫入 Solution）
- 清掃：`find docs/work-logs -name "*.lock" -delete`（限定 *.lock）
- 驗證：全套件跑完後 `find docs/work-logs -name "*.lock" | wc -l` 為 0，並記錄測試總數防 skip 假陽性
