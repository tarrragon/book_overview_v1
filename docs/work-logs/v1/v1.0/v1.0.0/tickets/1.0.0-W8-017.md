### 根因定位

`.claude/skills/ticket/ticket_system/lib/file_lock.py` 的 `file_lock(target_path)` context manager（L37-87）：
- L77-79：建立 `{target}.lock`、`open(lock_path, "w")`
- L81：`fcntl.flock(LOCK_EX)` 取得
- L83-87 finally：`flock(LOCK_UN)` 釋放 + `fd.close()`，**但從不 `lock_path.unlink()`** → 0-byte 孤兒殘留

docstring L56 原刻意設計留檔（「殘留 lock file 不影響後續 reuse，已加入 .gitignore」）。本票要消除其認知負擔副作用（PM 誤判 active 操作）。

### 關鍵約束：禁止天真 inline unlink（race 風險）

> **acceptance #1「正常+異常路徑皆 cleanup」不可解讀為在 finally 內 `lock_path.unlink()`。**

inline unlink 會重新引入經典 flock-unlink TOCTOU race（docstring L67 提的 W14-005 race）：process A 釋放並 unlink 該 inode 時，process B 可能已 open 同 path 等鎖，A 刪檔後 process C 建新 inode 同名檔並取鎖 → A/C 同時持鎖。**現有 race 測試（`test_lifecycle_race.py`、`test_save_ticket_race.py`、`test_precondition_lock_safety.py` 等）必須維持全綠**，不可為清 lock 而破壞並行正確性。

### 建議安全修法（thyme 評估定案）

- **非阻塞 reap 模式（推薦）**：在 CLI process 結束點（或安全收尾處）掃 tickets dir 的 `*.md.lock`，對每個試 `flock(LOCK_EX | LOCK_NB)`；搶到（非阻塞成功 = 無人持有 = stale）才在持鎖狀態下 unlink，再釋放。安全收割 stale lock 不誤刪 active。
- 替代：獨立 reap helper / hook，於 session 安全時機呼叫。

> 若新增 `ticket track clean-locks` 類**子命令**（命令語意變更），須依 `.claude/rules/core/ticket-skill-sync-check.md` 同步 SKILL.md / pm-rules（純內部 reap 無命令變更則免）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 根因

`lib/file_lock.py` 的 `file_lock()` context manager 釋鎖（L83-87 finally：flock LOCK_UN + fd.close）後刻意保留 sentinel `{ticket}.md.lock`（docstring L54-57 設計留檔），每次 load-modify-save 序列殘留一個 0-byte lock，長期累積 54 個 stale lock，造成 PM 誤判有 active 操作。

### 修法（非阻塞 reap，不引入 W14-005 race）

不採天真 finally inline unlink（會重新引入 flock-unlink TOCTOU race）。新增 `reap_stale_locks(directory)`：遞迴 rglob `*.md.lock`，對每個試 `flock(LOCK_EX | LOCK_NB)`：
- 搶到（非阻塞成功 = 無人持鎖 = stale）→ 在持鎖狀態下 unlink，再 LOCK_UN。
- 搶不到（OSError/BlockingIOError = active）→ 跳過，絕不誤刪。

非阻塞特性保證不卡在 active lock 上；active lock 永不被收割，故既有 race 防護（W14-005）不受破壞。

### reap 設計與接線

- 收割點：`scripts/ticket.py` main() 命令執行完成後（安全收尾處），透過 `_reap_stale_locks_quietly()` 對 `get_project_root()/WORK_LOGS_DIR` 收割；任何例外靜默跳過，不影響已決定的命令返回碼。
- graceful：無 fcntl 平台 / 目錄不存在皆回傳 0，不丟例外。
- 只收割 `*.md.lock`，不動 `.ticket-create.lock` 等其他 lock。
- 純內部 reap，無新增子命令 → 依 ticket-skill-sync-check 免同步 SKILL.md/pm-rules。
