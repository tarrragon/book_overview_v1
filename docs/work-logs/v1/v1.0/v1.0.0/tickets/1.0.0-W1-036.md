## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

extract_pattern_id 的 SSOT regex `\b(?:CAT)-(?:[A-Z0-9]+-)?\d+\b` 允許 optional 前綴段含數字（因專案碼如 V1/C2C 本身含數字）。對 `PC-099-3-layer-defense.md` 會過度匹配為 `PC-099-3`（將 `099` 當前綴段、`3` 當 NNN）。hook 的 `is_flat_id` 僅以 `split('-')` 段數判定：3 段即視為前綴號放行 → 凍結 gate 被繞過。

實證（uv run）：`extract_pattern_id('PC-099-3-layer-defense.md') -> PC-099-3`，decide() 回 allow。

### 影響範圍

- `.claude/skills/error-pattern/hooks/error-pattern-flat-gate-hook.py`（is_flat_id 判定）
- 任何「數字開頭描述段」的 flat 號 error-pattern 檔名可手動新建而不被擋。
- 另：既有測試僅覆蓋 decide() 純函式，缺 subprocess 層 main/stdin/emit 接線測試。

### 相關 Error Pattern

ARCH-020（SSOT regex 複用）、IMP-055（hookSpecificOutput shape）、quality-baseline 規則 4（deny 雙通道）。修法選擇 hook 層而非改 SSOT pattern_id.py：後者為共用 lib（error_pattern_attribution / allocator 等多 caller），改動影響面大且 regex 回溯語意正確，問題在 hook 的段數判定過於樸素（PC-135 caller sync 風險規避）。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

於 hook 層強化 `is_flat_id`（不動 SSOT pattern_id.py）：

- 2 段 ID（PC-099）-> flat（原行為保留）。
- 3 段且**中段與末段皆純數字**（PC-099-3，來自 split 後 `['PC','099','3']`）-> 判 flat。依據 `_project-registry.yaml`：專案碼為「短大寫英數，語意可辨識」，所有已註冊碼（V1/APP/SCLK/CCS/C2C）皆含字母，故純數字中段不可能是合法專案碼，只能是 flat 號描述段數字被 SSOT regex 過度吸入。
- 真實前綴號（PC-V1-001 -> `['PC','V1','001']`，中段 V1 非純數字）不受影響，仍判前綴放行。

### 設計決策

採 `parts[1].isdigit() and parts[2].isdigit()` 雙條件，而非僅 `parts[1].isdigit()`，避免理論上 `PC-099-V1` 之類（中段數字末段含字母）的邊界誤判；對齊 SSOT regex 實際只會吐出末段為 \d+ 的 ID。

### 不在本 ticket 範圍

- pattern_id.py SSOT regex 本身未改（caller 多、回溯語意正確）。
- bash-edit-guard-hook（並行 agent 處理中）未觸碰。
