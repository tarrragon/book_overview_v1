## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因
父 ANA 1.0.0-W8-036 確認：sync-pull full-overlay 復活已刪上游孤兒。本子任務評估「機制側是否需防護」+「7 未註冊 hook 性質」。根因鏈：`.sync-state.json` 為 local-only（gitignored，.gitignore:75）→ fresh clone 缺 base SHA → `should_use_full_overlay` 回 True → `sync_directory` 複製全部上游檔，復活 local-absent 的上游檔；且 dry-run preview 不列復活（靜默）。

### 影響範圍
- 機制面：`.claude/scripts/sync-claude-pull.py` full-overlay 路徑。任何 fresh clone / state 遺失皆觸發復活，每次新機器/重 clone 復發。
- hook 面：7 個 hook（changelog-update / cli-failure-help-reminder / commit-handoff / post-commit-fetch / session-context-guard / skill-cli-error-feedback / test-timeout）在上游 SSOT settings.json 自身即未註冊。

### 相關 Error Pattern / Memory
命中 memory「sync-pull full overlay 風險」（W1-003）。本評估推翻了父 ANA Solution 中「tombstone 預期不採」的初判（因 fresh-clone 漏洞為真）。

## 重現實驗結果
### 實驗方法
讀 sync-claude-pull.py main() 同步策略分支（L1539-1618）確認 full-overlay 觸發條件與復活靜默性；查 .sync-state.json git 追蹤狀態決定 fresh-clone 暴露；查 7 hook 在上游/下游 settings.json 註冊狀態做分類。

### 實驗執行 + 發現（已驗證事實）
1. **full-overlay 觸發**（L1540-1544）：`base_sha is None`（無 .sync-state.json）或 `not base_reachable`（force-push/GC）。同步成功後 `write_base_sha`(L1617) 重寫 base SHA → 「無 base SHA」self-heal，僅首次/state 遺失窗口暴露。
2. **復活靜默性**：full-overlay 走 `sync_directory`(L1577) 複製全部上游檔；dry-run preview(L1557 `preview_overlay_changes`) 只列 will_overwrite（雙方都有）+ will_delete（local 獨有），**不列 resurrect（上游有 local 無的新增）** → 復活在 preview 中不可見。
3. **[決定性] .sync-state.json 為 local-only**：`git ls-files --error-unmatch` 不匹配（未追蹤）+ `.gitignore:75` 明確忽略 + 腳本註解「B - Local-only settings：.sync-state.json」。故 **fresh clone 必缺 base SHA → 首次 sync 必走 full-overlay → 復活全部上游孤兒**。漏洞真實且每次新 clone 復發。
4. **7 hook 註冊普查**：7 個（changelog-update / cli-failure-help-reminder / commit-handoff / post-commit-fetch / session-context-guard / skill-cli-error-feedback / test-timeout）在**上游 settings.json [0]、v1 settings.json [0]、settings.local.json [0]** 皆未註冊。

### 結論：已驗證 vs 假設
- [事實] M0（現狀）不足：.sync-state.json local-only → fresh clone full-overlay 復活復發。我先前「tombstone 預期不採」預測**被推翻**。
- [事實] 7 hook 在上游 SSOT 自身即未註冊 → 屬 dead/orphan hook（非登記缺口），與 16 root .md 同類。
- [假設] M2（git-delete-aware overlay）在 fresh clone 可用，因 W10-049 刪除 commit 隨 repo clone 而存在於本地 git 史——需實作時驗證 git log 查詢可靠性與「刪後又 re-add」邊界。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 結論

### acceptance #1：full-overlay fallback 需機制側防護（YES，附理由）

**結論：需要**。理由：`.sync-state.json` 為 local-only（gitignored），fresh clone 必缺 base SHA → 首次 sync 必走 full-overlay → 復活上游孤兒。此漏洞每次新 clone / 新機器 / state 遺失都復發，且 dry-run preview 不顯示復活（靜默）。僅靠上游清理（child 1）無法獨力防護「上游未來再被加入孤兒」或「上游清理尚未完成期間的 fresh clone」。

### 方案評估（W：性質三類涵蓋 + toolification-bias）

| 方案 | 性質 | 評估 |
|------|------|------|
| M0 現狀倚賴 .sync-state.json | 零工具 | **不足**：local-only → fresh clone 暴露 |
| M1 tombstone manifest（committed sync-tombstones.yaml，full-overlay 跳過） | 新增工具 | 可行、survives fresh clone；但新增 manifest 需維護、drift 風險 |
| M2 git-delete-aware full-overlay（overlay 前查 local git 史：currently-absent 且最後 git 事件為刪除則不復活） | 改造既有 | **建議**：以 committed git 史為刪除 SSOT（無 manifest drift）、survives fresh clone（刪除 commit 隨 repo clone）、鏡像既有 `cleanup_stale_files` 的 git-tracking-aware 設計 |
| M3 上游結構 reconciliation | 改造既有（大） | child 1 範疇；治本但大，且不防上游未來再污染 |
| M4 base SHA 改 committed/tracked | 改造既有 | 與 local-only 設計理念衝突（跨機器 merge 衝突/staleness），不建議 |

**建議採 M2**（改造既有，非新增工具——通過 toolification-bias 檢查），與 M3（上游清理）互補：M2 防 fresh-clone 復活 + 未來上游再污染；M3 治本清上游。M2 邊界須處理「刪後又 re-add」（如本案 12 孤兒已 re-add，M2 須以 currently-absent + 最後事件為刪除 為條件，不誤判已 re-add 檔）。

### acceptance #2：7 未註冊 hook 分類

7 個 hook 在**上游 SSOT settings.json 自身即未註冊**（非「上游有下游缺」的登記缺口），且 v1 settings/local 亦未註冊 → 分類為 **dead/orphan hook**（與 16 root .md 同類，上游 lingering dead 內容）。處置：併入上游清理（child 1 / M3 reconciliation）一併移除，或若判定仍需用則補上游 settings.json 註冊——預設傾向移除（無任何 caller）。逐一細查每個 hook 的 git 引入史與設計意圖屬清理 IMP 範疇。

### Spawn 落地
- M2 實作 → 建議建一 IMP（grandchild，depth 3 合法）實作 git-delete-aware full-overlay + 復活 preview 顯示 + 「刪後 re-add」邊界測試。本 ANA 不實作（analysis-only）。
- 7 hook 移除 → 併入下游/上游清理範疇。
