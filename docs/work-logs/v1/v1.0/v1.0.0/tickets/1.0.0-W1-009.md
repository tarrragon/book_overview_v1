### 決定性推論

既有 clean_stale_files（--clean）理論上完全能清這 27 孤兒：遠端 hooks/ticket-*.py 在本地 staging 不存在 + 不被 should_exclude → 必然被 unlink + push deletion。

孤兒殘留真因：clean_mode 是 opt-in（L1034），從沒跑過 sync-push --clean 或最近 push 未帶 --clean——非「機制缺失」。

### 對 W1-009 acceptance 的影響

### 執行期重大發現：--clean 實際範圍 = 751 檔（非 W1-016 預估 27），且初次預覽曾遭 chpwd 污染誤報

**方法論教訓（chpwd 污染）**：初次孤兒預覽用 `(cd "$TMP" && git ls-files | sort) > file` 比對，zsh chpwd hook 在 cd 時自動 `ls` 的輸出混入 stdout，使排序檔非完全排序 → `comm` 產生假數據（誤報 2610 孤兒，曾誤判為「災難性 repo-wide 刪除」）。改用 `git -C "$TMP" ls-files`（不 cd、不觸發 chpwd）後得正確值。對應 bash-tool-usage-rules 規則一 + memory[chpwd_floods_agent]。

**正確量測（git -C，排除 gitlink）**：

| 本地 tracked | 1838 |
| 交集 | 1830（本地 99.6% 在遠端） |
| 真實孤兒（遠端有 + 本地 tracked 無） | 751 |

**751 孤兒組成**：

| 類別 | 數量 | 性質 |

| skills/ticket/MagicMock/get_ticket_path()/*.lock | 632 | 測試隔離 bug 垃圾（MagicMock 物件路徑被當真實目錄建立；本地 filesystem 有但未 git-tracked，遠端殘留） |

**對 W1-016 方案 A 前提的修正**：
