| 整合面 | 結論 |
|--------|------|
| 嵌套層並行數計算口徑 | `.claude/` 並行限制（worktree 模式 <= 2）**跨層累計**：runtime deny 不區分 dispatch 層級（W1-056.4 實證 hook/runtime 嵌套層行為一致）。常態下 D2 敏感操作條件已禁止嵌套層 descend `.claude/` 寫入類子任務，計數收斂於 L0——PM dispatch-plan 即全系統並行帳本；用戶授權豁免時 descend 方須在 child Problem Analysis 載明額度佔用 |
| 嵌套 descend staging 責任歸屬 | PC-092 精準 staging 跨層不變；descend 方承擔三項：child where.files 互斥設計、staging 政策寫入 child Problem Analysis（D1：context 進 ticket 不進 prompt）、descend 前先 commit 自身中間產物。child agent 承擔執行層精準 staging。根因延伸：嵌套後「他人變更」包含父層自身未 commit 產物 |
| worktree 模式與嵌套相容性 | 核心不變量：跨層資訊一律經 ticket（D1），禁止依賴父層 working copy 未 commit 中間檔案——將 worktree 行為差異隔離在檔案層。三情境表：父層 worktree 需先 commit + merge 回 main 才對 child 可見；`.claude/` child 受 D2 禁止 + ARCH-015 跨層不變；主 repo cwd 父層 + src/ child 同單層規則。嵌套層 worktree 行為無受控資料，依規則 1.5 標註擴充性說明 + 出現需求時建 ANA ticket 對照實驗 |
