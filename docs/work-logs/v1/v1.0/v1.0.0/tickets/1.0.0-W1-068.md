## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

`uv-tool-staleness-check-hook.py` 的 SKILLS registry 對所有 entry 套用單一「目錄 + rglob」比對模型，但有兩個 entry 不符此模型：

1. **branch-worktree-guardian**：非 uv tool（無 CLI entry、不在 `uv tool list`）。它被放進 staleness registry，每次 session-start 都嘗試定位 installed module dir，因找不到而誤報。
2. **version-release**：是真 uv tool，但為**單檔模組**。source 為 `scripts/version_release.py`（單一 .py 檔），installed 為 site-packages 下的 `version_release.py`（單一 .py 檔）。registry 原本設 `module_subpath=".claude/skills/version-release/version_release"`（一個不存在的目錄），故 `source_dir.exists()` 為 False，回傳 `ERROR: source dir missing`。

### 影響範圍

- `.claude/hooks/uv-tool-staleness-check-hook.py`（registry + check 邏輯）
- `.claude/hooks/tests/test_uv_tool_staleness_check_hook.py`（測試）

兩誤報持續出現於每次 session-start additionalContext，侵蝕 staleness 警示信任度（與 PC-074 false positive 信任侵蝕同構）。

### 相關 Error Pattern

PC-074（false positive 信任侵蝕）— 誤報讓真 stale 警示失去可信度。source ANA 為 1.0.0-W1-066。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

採 source ANA 1.0.0-W1-066 候選 A（候選 B「pyproject.toml build target 推導」明確排除，如需另開 ANA）。

| entry | 修復 |
|-------|------|
| branch-worktree-guardian | 自 SKILLS registry 移除（registry 從 7 entry 減為 6）。docstring 補註說明移除理由。 |
| version-release | SkillEntry 新增 `single_file: bool = False` 旗標（預設 False，向後相容目錄模組）。設 `single_file=True`，`module_subpath` 改指向單檔 source `.claude/skills/version-release/scripts/version_release.py`，`package_dir_name` 改為 installed 單檔名 `version_release.py`。 |

### 設計決策

- `check_single_skill` 新增前置分支：`if skill.single_file: return _check_single_file_skill(...)`，目錄模組原路徑完全不變（最小侵入）。
- `_check_single_file_skill`：source 端驗 `is_file()` 後算單檔 SHA256；installed 端複用既有 `find_installed_module_dir(cli, ".", logger)` 取得 site-packages（`package_dir_name="."` 使 `site_packages / "."` 解析回 site-packages 本身），再拼 `package_dir_name` 定位單檔，算 SHA256 後單值比對。
- 不修改 `lib/uv_tool_utils.py`（符合 ticket 約束「只改 hook + tests」），單檔 hash 用 hook 內新增的 `_compute_single_file_hash`。
- `format_results` 的「全部同步」訊息用 `len(SKILLS)` 動態計數，自動適配 6。

### 自檢結果

- 認知負擔：`_check_single_file_skill` 變數數/分支數低（< 6），單一職責。
- 規則 4（異常可觀測性）：單檔 hash 失敗走 `logger.error` 雙通道後回傳 ERROR，不靜默。
- 手動執行 hook（真實環境）：6 個 entry 全數正確分類，無 `source dir missing` / `source file missing` ERROR。version-release 顯示 OUTDATED 為**真陽性**（source Jun 8 sha 27918ac… vs installed Jun 5 sha 4de976b…，source 較新），證明單檔模組 stale 偵測正確。

### 不在本 ticket 範圍

- 候選 B（pyproject.toml build target 推導）
- `lib/uv_tool_utils.py` 重構
- version-release 的實際 reinstall（OUTDATED 為真實狀態，非本票修復標的）
