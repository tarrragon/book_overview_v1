## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

`computeMergeResult` 階段 3（book tagIds 重映射 + 同 id 聯集，src/storage/adapters/tag-storage-adapter.js line 1674-1709）對「同 id 書」採盲覆蓋：

```js
// line 1696-1704（修改前）
if (localBookById.has(impBook.id)) {
  const localBook = localBookById.get(impBook.id)
  let unionTagIds = [...new Set([...(localBook.tagIds || []), ...remappedTagIds])]
  // ...截斷
  localBookById.set(impBook.id, { ...localBook, ...impBook, tagIds: unionTagIds })
}
```

`{ ...localBook, ...impBook }` 使匯入書「所有欄位」（含 progress / readingStatus / isManualStatus）一律覆蓋本地，無書級 updatedAt LWW。category 已有 `resolveCategoryConflictByLWW`（line 1455）但書級缺對應機制。

### 影響範圍

- 檔案：src/storage/adapters/tag-storage-adapter.js（單檔，符合 ticket where.files）
- 場景（UC-05 跨裝置閱讀進度同步）：裝置 B 進度較新（updatedAt 較新），但匯入裝置 A 較舊檔案時，B 的 progress 被舊值覆蓋。
- 書級 merge 邏輯確認在本檔 line 1569-1729（computeMergeResult 階段 3），不在別處 src 檔，無需擴大範圍。

### Book schema 進度相關欄位（BookSchemaV2.js）

| 欄位 | 型別 | 說明 |
|------|------|------|
| progress | number 0-100 | 閱讀進度百分比 |
| readingStatus | string enum | 閱讀狀態（progress 連動） |
| isManualStatus | boolean | 是否手動設定狀態 |
| updatedAt | string (ISO) | LWW 時間基準（auto 欄位） |

### 設計決策：欄位群分軌

book merge 拆兩軌，避免破壞既有 additive 語意：
1. **tagIds**：維持聯集（additive，永不遺失 tag）— 不受 LWW 影響
2. **其餘欄位（progress/readingStatus/title 等純量）**：採書級 updatedAt LWW

LWW 規則完全比照 resolveCategoryConflictByLWW（line 1455-1470）：
- 兩方皆有有效時間 → 較新勝，平手取 incoming（最後寫入優先）
- 僅一方有時間 → 有時間者勝
- 皆無時間 → 退回 incoming（向後相容既有「匯入覆蓋」語意）

### 向後相容驗證

既有 TC-C2「同 id 其餘欄位以匯入覆蓋」（merge-all-data-compute.test.js line 121-140）兩書皆無 updatedAt，依「皆無時間→incoming 勝」規則，progress=50 仍覆蓋本地 0，TC-C2 不破。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

src/storage/adapters/tag-storage-adapter.js 兩處變更：

1. **新增 `resolveBookConflictByLWW(localBook, incomingBook)`**（接於 resolveCategoryConflictByLWW 之後）：
   - 純函式，純量欄位以 updatedAt LWW 收斂，規則完全比照 category 版本（較新勝 / 平手取 incoming / 僅一方有時間者勝 / 皆無時間退回 incoming）。
   - 只決定純量欄位勝出方；tagIds 的 additive 聯集語意留給 caller 後續覆寫。

2. **修改 computeMergeResult 階段 3 同 id 書合併**（原 `{ ...localBook, ...impBook, tagIds: unionTagIds }` 盲覆蓋）：
   - 改為 `const lwwWinner = resolveBookConflictByLWW(localBook, { ...impBook, tagIds: remappedTagIds })`
   - 再 `{ ...lwwWinner, tagIds: unionTagIds }`，使 tagIds 一律為聯集，不受 LWW 勝負影響。

### 設計決策：兩軌分離

| 軸 | 語意 | 理由 |
|----|------|------|
| tagIds | 聯集（additive） | 永不遺失任一裝置的 tag，與既有 W1-047.4 行為一致 |
| 純量欄位（progress/readingStatus/title 等） | updatedAt LWW | 跨裝置進度同步收斂至最新，解 UC-05 |

不合併為單一 generic LWW helper：book 有 tagIds 聯集特例，category 無；分立兩函式使各自領域規則清晰，避免遮蔽特例。

### 不在本 ticket 範圍

- 書級 merge 邏輯確認在本檔（tag-storage-adapter.js）內，未擴大至別處 src 檔。
- updatedAt 的產生 / 寫入時機（提取、手動改進度）屬既有機制，本 ticket 僅消費該欄位做衝突解析。
