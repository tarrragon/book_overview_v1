## Problem Analysis

### 問題根因

auto-commit graceful degrade 後 ticket body 退回「僅存於 working tree」的脆弱狀態（W1-017 原始風險）。degrade 觸發時唯一通知管道是 stderr，但自動序列（sync-push / worktree merge / 後續 append-log）不讀 stderr —— 等於無人接住。

### 影響範圍

- `.claude/skills/ticket/ticket_system/lib/git_utils.py` — auto-commit 實作
- `.claude/skills/ticket/ticket_system/commands/track_acceptance.py:757-772` — 呼叫端 graceful degrade 處理

### 相關 Error Pattern

- PC-148（hook 雙重註冊 shebang bypass）：類似「靜默失敗」風險但不同層級
- W7-001 否決方案 C（warning hook）的論證同樣適用本問題

### Degrade 觸發條件盤點

| 觸發條件 | return/exception | 發生情境 | 頻率估計 |
|---------|-----------------|---------|---------|
| index.lock 並發 | `git_failed`（git add/commit returncode≠0）或 `TimeoutExpired` exception | 並行 agent commit / PM commit / pre-commit hook 長時間鎖定 | 中（並行派發常見） |
| 非 git repo | `not_git_repo` | ticket CLI 於非 repo 環境執行（極罕見） | 極低 |
| git 未安裝 | OSError exception | 環境異常 | 極低 |
| commit timeout（30s） | `TimeoutExpired` exception | pre-commit hook 卡死 / 網路 credential helper hang | 低 |
| git add 失敗 | `git_failed` | 權限問題 / 磁碟滿 / 檔案被鎖 | 極低 |

**核心洞察**：唯一中頻觸發條件是 index.lock 並發（並行 agent 場景），其他均為極端邊緣。

### 實際後果鏈

1. append-log 寫入 → save_ticket 成功（body 在 working tree）
2. auto-commit degrade → stderr 警告
3. 後續若發生 `git checkout -- <file>` / `git reset --hard` / `git stash` → body 遺失
4. 但步驟 3 需要**人或 agent 主動執行破壞性 git 操作**才觸發

**關鍵事實**：degrade 後 body 不會**自動**遺失；需要額外破壞性操作才遺失。下一次 append-log 呼叫會再嘗試 auto-commit（含累積的所有 body 變更），形成自然重試。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 評估

#### W（Widen）— 候選方案

| 方案 | 描述 | 複雜度 | 防護範圍 |
|------|------|--------|---------|
| A. index.lock retry | degrade 時等 1-3s 重試 auto-commit（鎖釋放後成功） | 低 | 僅 index.lock 並發 |
| B. 獨立 state 備份 | degrade 後將 body 複製到 `.ticket-body-backup/` | 中 | 全部 degrade 情境 |
| C. degrade 計數告警 | 累計 N 次 degrade 後在 stdout 輸出 WARNING | 低 | 可觀測性，非防護 |
| D. 阻擋後續自動序列 | degrade 後寫 flag 檔，sync-push/merge 檢查 flag 阻擋 | 高 | 防止未 commit body 被 push/merge 覆蓋 |
| E. 不做（接受現狀） | 依賴自然重試（下次 append-log 會重新 commit 累積變更） | 零 | 自然重試覆蓋大部分情境 |

#### R（Reality Test）— 真實風險評估

**已驗證事實**：
1. degrade 後 body 仍在 working tree（save_ticket 已完成，exit 0）
2. 下一次 append-log 自動重試 commit（含累積變更）— 自然修復
3. body 遺失需要額外破壞性 git 操作（checkout/reset --hard/stash）
4. 並行派發時 index.lock 是中頻事件（memory: feedback_parallel_agent_commit_race）

**風險窗口定義**：從 degrade 發生到下次 append-log 成功 commit 的時間段。此窗口內若有破壞性 git 操作，body 遺失。

**窗口實際大小**：典型 agent 工作流中，append-log 頻率高（每個 section 填寫都觸發），窗口通常 < 5 分鐘。破壞性操作（reset --hard / checkout --）在此窗口內發生的機率極低。

#### A（Attain distance）— 機會成本

- 方案 A（retry）：實作成本 ~30 行，但 index.lock 持續時間不確定（1s-30s+），retry 不保證成功
- 方案 B（備份）：引入新檔案管理複雜度，需清理機制，且備份本身也可能被 git clean 刪除
- 方案 D（阻擋序列）：侵入性高，sync-push/merge 不在 ticket_system 管轄範圍，跨模組耦合
- 方案 E（不做）：零成本，接受極小殘餘風險

#### P（Prepare to be wrong）— 若不做會怎樣

最壞情境：index.lock 並發 degrade → agent 隨後執行 `git stash`（worktree 操作常見）→ body 遺失。

**但**：W7-001 memory 記載 worktree agent 在自己 worktree commit，不影響 main working tree；PM stash 操作極少。實際觸發完整遺失鏈的機率 < 1%。

### 結論

**採方案 A（index.lock retry）+ 方案 E（接受殘餘風險）的混合**：

1. **index.lock retry**（值得做）：在 `_run_git` 或 `_auto_commit_ticket_md` 層加入 1 次 retry（sleep 1s），成本極低（~15 行），覆蓋最常見的 degrade 觸發條件（並行 index.lock 通常持續 < 3s）
2. **不做其他方案**（接受殘餘風險）：方案 B/C/D 的成本收益比不合理——自然重試機制（下次 append-log）已覆蓋 >99% 情境，破壞性操作在短窗口內發生的真實機率極低

**Why**：index.lock 是唯一中頻 degrade 觸發源；一次 retry 可將 degrade 率從「並行場景 ~10%」降至「< 1%」，ROI 最高。

**Consequence**：若不加 retry，並行派發場景每 ~10 次 append-log 有 1 次 degrade（可觀察但無害——自然重試兜底）。加 retry 後幾乎消除此噪音。

**Action**：spawn IMP ticket 實作 index.lock retry（git_utils.py `_auto_commit_ticket_md` 在 git_failed 且 stderr 含 "index.lock" 時 sleep 1s + 重試一次）。

### Spawn 落地確認

需 spawn 1 個 IMP ticket：在 `_auto_commit_ticket_md` 加 index.lock retry 邏輯。
