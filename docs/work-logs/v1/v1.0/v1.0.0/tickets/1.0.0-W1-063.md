## Problem Analysis

### 問題根因

ticket create 的 ID 分配流程為「掃描 tickets 目錄取最大序號 N（get_next_seq / get_next_child_seq）→ 配號 N+1 → save_ticket 落盤」，掃描與落盤之間無任何跨 process 互斥機制。並行 create（同 session background bash 或跨 session 兩個 terminal）會同讀相同 max seq、配出同一 ID，後落盤者靜默覆寫前者（IMP-072；2026-06-11 單日 2 次跨 session 撞號：W1-057/W1-059）。

### 影響範圍

- `.claude/skills/ticket/ticket_system/commands/create.py`：execute 的 Step 1（`_resolve_ticket_id_and_wave`，含 auto-seq 與 child-seq 兩路徑）至 Step 3（`_persist_and_report` → `_build_and_save_ticket`）整段臨界區
- `.claude/skills/ticket/ticket_system/commands/bulk_create.py`：批次配號迴圈共用 `get_next_seq` / `resolve_available_seq`，與並行 create 之間存在同型撞號窗口
- `.claude/skills/ticket/ticket_system/lib/file_lock.py`：既有 per-ticket-file lock（W14-042）保護單檔 load-modify-save，不覆蓋「目錄級配號」互斥

### 相關 Error Pattern

- IMP-072（本 ticket 落地其方案 A）：ticket create 並行 ID 分配 race condition
- IMP-061：撞號後遷移處置遺失 frontmatter 關聯欄位（本修復消除撞號源頭後不再觸發）
- W14-042 / W14-005：既有 file_lock 慣例（fcntl.flock advisory lock），本修復複用同一機制
