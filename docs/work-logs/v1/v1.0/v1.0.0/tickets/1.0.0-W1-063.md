## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

ticket create 的 ID 分配流程為「掃描 tickets 目錄取最大序號 N（get_next_seq / get_next_child_seq）→ 配號 N+1 → save_ticket 落盤」，掃描與落盤之間無任何跨 process 互斥機制。並行 create（同 session background bash 或跨 session 兩個 terminal）會同讀相同 max seq、配出同一 ID，後落盤者靜默覆寫前者（IMP-072；2026-06-11 單日 2 次跨 session 撞號：W1-057/W1-059）。

### 影響範圍

- `.claude/skills/ticket/ticket_system/commands/create.py`：execute 的 Step 1（`_resolve_ticket_id_and_wave`，含 auto-seq 與 child-seq 兩路徑）至 Step 3（`_persist_and_report` → `_build_and_save_ticket`）整段臨界區
- `.claude/skills/ticket/ticket_system/commands/bulk_create.py`：批次配號迴圈共用 `get_next_seq` / `resolve_available_seq`，與並行 create 之間存在同型撞號窗口
- `.claude/skills/ticket/ticket_system/lib/file_lock.py`：既有 per-ticket-file lock（W14-042）保護單檔 load-modify-save，不覆蓋「目錄級配號」互斥

### 相關 Error Pattern

- IMP-072（本 ticket 落地其方案 A）：ticket create 並行 ID 分配 race condition
- IMP-061：撞號後遷移處置遺失 frontmatter 關聯欄位（本修復消除撞號源頭後不再觸發）
- W14-042 / W14-005：既有 file_lock 慣例（fcntl.flock advisory lock），本修復複用同一機制

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復設計（IMP-072 方案 A）

commit 61873955，三檔修改 + 一檔新增：

1. `lib/file_lock.py` 新增 `create_id_allocation_lock(tickets_dir)`：目錄級 fcntl.LOCK_EX blocking lock。
   - lock file = `{tickets_dir}/.ticket-create.lock`（`*.lock` 已在 .gitignore；crash 後 OS 自動回收 fd 釋鎖）
   - 與既有 `file_lock` 的差異：file_lock 是 per-ticket-file（保護單檔 load-modify-save）且無 fcntl 平台 raise NotImplementedError；本 lock 是 per-tickets-dir（同版本目錄所有 create 互斥）且 graceful degradation——無 fcntl 平台 / lock file 開啟失敗 / flock 失敗三類情況均 stderr warn（quality-baseline 規則 4）+ 無鎖續行 + 附「並行期間請序列執行」建議，不阻斷單 process create（acceptance 2）
2. `commands/create.py` execute：Step 1（ID 分配）至 Step 3（落盤）包進 `with create_id_allocation_lock(get_tickets_dir(version))`；Step 4（Context Bundle 抽取）在鎖外
3. `commands/bulk_create.py` execute：非 dry_run 的 `_create_batch_tickets` 呼叫共用同一把目錄級 lock（批次配號與並行 create 同型撞號窗口）；dry_run 不寫檔不持鎖

### 設計決策

- 「衝突時自動取下一可用 ID 並輸出提示」：由既有 `get_next_seq` 內聚 collision guard（W1-051 `resolve_available_seq` + stderr 警告）+ 本 lock 序列化共同滿足；不在 caller 層重新加 .exists() 探測 while-loop（W1-051 已明確移除該模式，避免特例外洩）
- lock 範圍涵蓋 Step 1.5/2（驗證、config 轉換）：皆為 in-process 快速操作，持鎖成本可忽略，換取結構單純（不重排 execute 流程）
- 與 `file_lock`（parent/source ticket 更新用）無巢狀 self-block 風險：lock file 路徑不同

### 不在本 ticket 範圍

- append-log 相關程式（W1-025/W1-058 範圍，prompt 約束）未觸碰
