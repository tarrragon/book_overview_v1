## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

broken-link-check 是 **LLM 手動執行的 SKILL**（Glob + Grep + 逐條判讀），非確定性 CLI。SKILL Step 2 的排除規則（「程式碼區塊內的範例路徑」「僅含目錄名」「動態路徑」）需要 LLM 逐次主觀判讀。兩次執行（W8-019.2 報 258、.3 報 155）對下列旋鈕判讀不同即產生大幅計數差：

1. 是否排除 code block 內引用（量測 ±75-80）。
2. 是否掃描 migration-backups/（動態備份目錄，量測 ±31-55）。
3. 是否排除文件中的 placeholder/格式範例路徑（如 `.claude/path/file.md`，SKILL 自身表格即用此為範例）。

這三個判讀旋鈕的組合，使同一份 repo 的 broken 計數在 164~270 區間浮動，恰好夾住 155 與 258。

### 影響範圍

- broken-link-check 作為「跨框架完成 gate」（delta=0 才算未新增失效）的可靠度被動搖：baseline 非確定性，delta 比較失去固定參考點。
- 既有 .claude/ 確有上百真實失效連結（確定性 baseline 164），屬 framework 文件債，需 triage 與清理（落地 W8-034）。

### 相關 Error Pattern

PC-168（flaky baseline 幸運連勝）精神相關：非確定性量測作 baseline，基於假 baseline 的 gate 決策鏈會不穩。本案為「量測工具本身非確定性」的同類風險。
