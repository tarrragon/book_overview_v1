## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

broken-link-check 是 **LLM 手動執行的 SKILL**（Glob + Grep + 逐條判讀），非確定性 CLI。SKILL Step 2 的排除規則（「程式碼區塊內的範例路徑」「僅含目錄名」「動態路徑」）需要 LLM 逐次主觀判讀。兩次執行（W8-019.2 報 258、.3 報 155）對下列旋鈕判讀不同即產生大幅計數差：

1. 是否排除 code block 內引用（量測 ±75-80）。
2. 是否掃描 migration-backups/（動態備份目錄，量測 ±31-55）。
3. 是否排除文件中的 placeholder/格式範例路徑（如 `.claude/path/file.md`，SKILL 自身表格即用此為範例）。

這三個判讀旋鈕的組合，使同一份 repo 的 broken 計數在 164~270 區間浮動，恰好夾住 155 與 258。

### 影響範圍

- broken-link-check 作為「跨框架完成 gate」（delta=0 才算未新增失效）的可靠度被動搖：baseline 非確定性，delta 比較失去固定參考點。
- 既有 .claude/ 確有上百真實失效連結（確定性 baseline 164），屬 framework 文件債，需 triage 與清理（落地 W8-034）。

### 相關 Error Pattern

PC-168（flaky baseline 幸運連勝）精神相關：非確定性量測作 baseline，基於假 baseline 的 gate 決策鏈會不穩。本案為「量測工具本身非確定性」的同類風險。

## 重現實驗結果
### 實驗方法

撰寫確定性 Python 掃描器（透過 python3 stdin 執行，不落檔），對 .claude/**/*.md（排除 hook-logs）抓 SKILL 定義的 4 種路徑前綴引用（@.claude/、.claude/、./、../ 結尾 .md），逐條 resolve + exists 檢查。以 scope（full vs curated 排除 migration-backups）× excl_code（是否排除 code block 內）四組合對照計數，驗證 155/258 差異是否源於排除規則判讀。

### 實驗執行

四組合計數結果（同一份 repo，同一次掃描）：

| scope | 排除 code block | refs | broken |
|-------|----------------|------|--------|
| full（含 migration-backups） | 否 | 2946 | 270 |
| full | 是 | 2785 | 195 |
| curated（排除 migration-backups） | 否 | 2615 | 239 |
| curated | 是 | 2454 | 164 |

triage 分類（curated + 排除 code block，164 條）：

| 類別 | 數量 | 性質 |
|------|------|------|
| rules/guides/ legacy 路徑 | 33 | 真實斷鏈（目錄重組 guides/→pm-rules/core/） |
| guidelines//architecture/ legacy | 11 | 真實斷鏈（早期目錄已移除） |
| 相對 ../../ 越界 | 8 | 真實斷鏈（相對層級錯算） |
| CHANGELOG/work-logs 歷史 | 2 | 歷史引用（可豁免/標註） |
| 其他 | 110 | 混合：真實斷鏈（askuserquestion-rules core/ vs pm-rules/ x11、verification-framework x4 等）+ placeholder 範例路徑（.claude/path/file.md、./path/file.md，SKILL 自身格式範例，屬 false positive） |

### 實驗發現

已驗證事實：
- 155/258 差異根因 = SKILL 為 LLM 手動執行，三個排除旋鈕（code block ±75、migration-backups ±31-55、placeholder 範例判讀）的主觀判讀組合使計數在 164~270 浮動，夾住 155 與 258。155 ≈ curated+排除code(164)；258 ≈ full+含code(270)/curated+含code(239)。
- 確定性 baseline（curated + 排除 code block）= 164，但其中 110「其他」含 placeholder 範例 false positive，真實斷鏈（扣除明確 placeholder）約 140±。
- 真實斷鏈主來源為歷史目錄重組未同步引用（rules/guides/ 33 + guidelines/architecture/ 11 + askuserquestion-rules/verification-framework 等）。

仍未驗證假設：精確「真實斷鏈」數需排除 placeholder 範例後人工覆核（落地 W8-034 清理時逐條確認）；本 ANA 結論不依賴精確值，依賴的是「非確定性根因」已驗證。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 決策摘要

- W（Widen）：gate 確定性方案 — (A) 建確定性 CLI 取代 LLM 手動 skill；(B) 維持 LLM skill 但固化排除規則為更嚴格 checklist；(C) 放棄 broken-link 作 gate 改人工抽查。
- R（Reality Test）：見「重現實驗結果」——四組合計數 164~270 夾住 155/258，根因為 LLM 判讀旋鈕，已驗證。
- A（Attain distance）：(A) 一次性實作成本，長期換得可重現 gate；(B) 仍受 LLM 判讀漂移；(C) 失去自動化價值。選 (A)。
- P（Prepare to be wrong）：最可能失敗 = CLI 排除規則仍漏某類 placeholder 致 false positive。防護：以重現實驗的分類清單為測試案例，CLI 須對 placeholder 範例路徑（path/file.md 樣式）顯式排除。

### 結論（三 acceptance）

1. acceptance #1：權威確定性 baseline = 164（curated + 排除 code block）。155/258 差異根因 = broken-link-check 為 LLM 手動執行，三排除旋鈕（code block ±75、migration-backups ±31-55、placeholder 範例判讀）主觀組合使計數浮動於 164~270。已驗證。
2. acceptance #2：triage 完成（見重現實驗分類表）。主來源為歷史目錄重組未同步引用（rules/guides/ 33、guidelines/architecture/ 11、askuserquestion-rules/verification-framework 等）+ placeholder 範例 false positive。清理落地於既有 W8-034（已建雙向 relatedTo，本 ANA triage 作其輸入）。
3. acceptance #3：判定 gate 確定性**需修**。採方案 A：建確定性 CLI scanner 取代 LLM 手動 skill 作 gate。落地於 child 1.0.0-W8-030.1（IMP）。
