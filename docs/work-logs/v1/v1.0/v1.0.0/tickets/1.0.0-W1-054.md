### 四項衛生債收斂結果

#### (1) conftest DRY 收斂 — **採收斂**（兩擇一顯性結論）

**實證裁決**：可行，已收斂。

- 關鍵事實：`pyproject.toml` 的 `[tool.pytest.ini_options]` 位於 skill root（`.claude/skills/ticket/`），故 pytest `rootdir` = skill root，且 `testpaths = ["tests", "ticket_system/tests"]` 使兩測試樹共用單一 pytest session。
- 推論：skill-root `conftest.py` 對「在 rootdir 下任一 testpath 收集的測試」皆生效，兩樹自然共享，**無需 cross-rootdir 技巧**（linux 視角的 DRY 收斂在此 layout 下成立；coupling 視角擔憂的「跨 rootdir 必要重複」前提不存在——兩樹同屬一個 rootdir）。
- 落地：新建 `.claude/skills/ticket/conftest.py`，承載 `_isolate_project_root`（autouse）+ `real_repo_root`（opt-out）單一副本；兩子 conftest 移除逐字重複的副本（保留指向 skill-root 的說明註解）。
- 全套件 2389 passed 驗證 autouse 對兩樹皆生效、opt-out 仍解析。

**仍保留於各子 conftest 的 fixture（合理分裂，不上提）**：
- `tests/conftest.py`：`_assert_no_repo_pollution`（依賴 `parents[4]` 相對路徑與該檔位置耦合）、ticket data fixtures。
- `ticket_system/tests/conftest.py`：`_isolate_hook_logs_dir`、`_mock_track_snapshot_filesystem_scan`、precondition fixtures（僅該樹消費）。

#### (2) opt-out 統一為簽名注入 — `_use_real_repo_root` indirection 移除

- `test_create_error_envelope.py` / `test_create_ux_merged_validation.py` 原以 `@pytest.fixture(autouse=True) def _use_real_repo_root(real_repo_root): yield` 中介 fixture 間接觸發 opt-out。
- 改為各測試函式簽名直接注入 `real_repo_root` 參數（含 capsys 的測試改為 `(real_repo_root, capsys)`）。移除中介 autouse fixture。
- `test_create_error_envelope.py` 連帶移除已不再使用的 `import pytest`。
- chdir 組的 setenv 分裂未涉及本檔，依 strategy 保留。

#### (3) test_ticket_builder.py 雙 namespace patch — 補齊

- 根因（W1-053 殘餘地雷）：`TestGetNextSeq` 四測試僅 patch `ticket_builder.get_tickets_dir`（glob 來源），未處理 `get_next_seq` 第二來源 `list_ticket_files_from_main`（內部呼叫 `get_project_root()`）。glob 目錄（測試 tmp）與 main-ref 探測目錄（autouse `_isolate_project_root` 的另一 tmp）分屬兩個不同 tmp，測試僅因 main ref 在無 git 預設 tmp 降級回 None 而恰巧通過，非自洽隔離。
- 修復：新增 helper `_patch_get_next_seq_env(monkeypatch, tickets_dir)`，同時 patch `get_tickets_dir` → tickets_dir 與 `list_ticket_files_from_main` → 回**空 list `[]`**（有效空 main ref，非 None 降級訊號）。如此 get_next_seq 不進入降級 guard 分支（該分支才會經 resolve_available_seq → get_project_root 探測 autouse tmp），直接回 `max+1`，glob 成為唯一自洽來源。
- 四個 TestGetNextSeq 測試改用此 helper，斷言不再依賴環境耦合的降級巧合。
- 設計參照 `test_create_auto_seq_collision.py` 的 `_patch_project_root` 雙 namespace 手法。

#### (4) .claude/tickets 冗餘目錄 — **移除**（結論）

- 實證：`get_tickets_dir` / lock 路徑解析落點為 `docs/work-logs/...`，非 `.claude/tickets`；grep 確認無任一測試在 autouse 預設環境下消費 `.claude/tickets`。
- 落地：skill-root `_isolate_project_root` 不再 mkdir `.claude/tickets`（原兩份副本皆有此無消費者的 mkdir）。全套件綠驗證移除無回歸。

#### (5) git-init fixture 變體（main-ref 成功路徑）— **不採**（顯性結論）

- 結論：不新增 git-init fixture。
- 理由：本票 mock 一致性修復採「`list_ticket_files_from_main` → `[]`」顯式 mock 路徑，使 get_next_seq 在單一 tmp 上自洽，**無需**真實 git repo 來驗證 main-ref 成功路徑。引入 git-init fixture（`git init` + commit ticket 檔 + 切 main）會帶來 subprocess git 開銷與環境敏感性（CI git 版本/設定差異），成本大於收益。main-ref 成功路徑的真實覆蓋已由既有 `test_create_auto_seq_collision.py` 等以 mock 方式涵蓋；本票範圍限測試基礎設施衛生收斂，不擴張為新增 git 整合 fixture。
