## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

ANA 衍生票關係模型有兩條權威指引存在張力，且都未涵蓋「盤點/規劃型 ANA」這個邊界類型：

1. **PC-091（process rule，2026-05-03 升格唯一權威）**：ANA 落地 IMP/DOC **一律 children（`--parent`）**，spawned 對 ANA 落地已 deprecated。觸發案例 W5-009 是「診斷→修復」型 ANA（結論的價值需落地修復才實現）。
2. **field-semantics.md（欄位語意權威）**：`children` 業務語意=「必須一起交付完整功能」（永遠阻擋父 complete）；`spawned_tickets`=「衍生副產品，獨立排程」；且 **ANA 類型 source 的 spawned 阻擋為「過渡狀態，IMP 收斂後將移除」**——方向是讓 ANA 衍生票不阻擋父完成。
3. **CLI affordances**：`--source-ticket`（建 spawned）、`complete --yes-spawned`（ANA 帶非 terminal spawned 仍可完成）、`stuck-anas`（監測 ANA+in_progress+spawned）——框架已備「ANA 完成 + spawned follow-up」的完整機制。

**邊界缺口**：盤點型 ANA（如 W8-009/010/015，acceptance=產出分類處置計畫 + spawn 執行票，明示「不執行搬移」）既非純研究性（會 spawn follow-up），也不完全是 PC-091 的「診斷→修復」防護性（其交付物=計畫，非修復本身）。現行 binary（研究性/防護性）無法乾淨歸類，導致每次盤點 ANA 收尾都要臨場判斷 + 反覆修正。

### 影響範圍

- 規則層：`PC-091`、`ticket-lifecycle.md`（研究性/防護性 + 血緣選擇章）、`field-semantics.md`、PM memory `feedback_ana_followup_as_children`。
- 既有產物：本 session W8-009（dotted 編號 009.1/.2/.3 手動轉 spawned）、W8-010/015（sibling 編號 W8-018~024 經 --source-ticket）兩種編號並存的不一致。
- 未來：每張盤點/規劃型 ANA 的收尾路徑。

### 相關 Error Pattern

- PC-091（ANA 落地血緣選擇，本票將釐清其適用邊界，非推翻）
- PC-073（ANA 衍生用 spawned，已被 PC-091 deprecated；本票部分恢復其在「研究/規劃型 ANA」的適用）

### 實驗發現

**已驗證事實**：

1. PC-091 的「一律 children」是針對**診斷→修復型** ANA（W5-009），其前提是「ANA 價值需落地才實現」。
2. field-semantics 的 spawned 語意（衍生副產品/獨立排程）+ ANA spawned 阻擋「過渡將移除」+ --yes-spawned/stuck-anas 機制，共同支持**「ANA 完成 + spawned 獨立排程 follow-up」是框架認可的模式**（W8-013 即此模式的正確實例）。
3. 兩者非矛盾，而是**適用於不同 ANA 子類型**：PC-091 治診斷→修復；field-semantics 的 spawned 治研究/規劃型的獨立排程 follow-up。
4. 盤點型 ANA（W8-009/010/015）的 acceptance=產出計畫 + spawn 執行票，交付物是**計畫**，執行明示延後且獨立排程——符合 field-semantics spawned 語意，**非** PC-091 的診斷→修復情境。

**bias 校正**：我先前（讀 PC-091 後）判定「本 session 違反規則」是**過度修正**——PC-091 未涵蓋規劃型 ANA；field-semantics 反而支持 spawned。但反向 steelman（見 Solution）顯示 children 模型對「只診斷不開藥」風險的防護仍有價值，不可全盤否定 A。

**反向思考（Consider the Opposite）**：若「盤點 ANA 不該 complete 直到清理做完」為真會怎樣？→ 真正該問的不是「children vs spawned」，而是**「盤點 ANA 的交付物是『計畫』還是『清理後的結果』？」**。交付物若是計畫→C；若是結果→A。本 session 三票 acceptance 明示交付物=計畫（「不執行搬移」）。

**框架測試**：四選項無法一句話概括（A 跨 round 開、B 改規則、C 重分類、D 拆結構），非偽擴增。**三類涵蓋**：含零工具（A/C）+ 改造既有（B/D），抵抗工具化偏誤。

### 最終決策（用戶定案）：Option A（嚴守 children），本 session 立即執行

**模型**：ANA 落地一律 children（與 PC-091 現行一致）。盤點/規劃型 ANA（產出計畫但問題需後續清理才解決）**仍是防護性**——交付物是計畫不代表問題已解決，清理＝其 children，ANA 保持 in_progress 直到 children 完成。**本 session W8-009 把盤點誤判為「可用 spawned 完成」是核心教訓**。
