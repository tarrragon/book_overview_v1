## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

W1-019 採 Model 1（凍結 flat base + 來源前綴）作為**預防**機制，W1-019.4 完成 **V1 本地** remediation。但 flat base 碰撞已存在於所有同步專案（發散狀態），V1 的清理不會自動傳播為各專案的清理：sync 是檔案 overlay，不會替各專案刪除其本地舊號檔。因此需要一套「各專案自行 pass」的 remediation 策略 + 偵測工具，而非中央一次性修復。

### 影響範圍

- 5 個同步專案的 `.claude/error-patterns/`（見重現實驗結果，發散 63~309 檔）

- sync-push 傳播序列（框架就位 → push → 各專案 pull → 各專案 remediation）

### 相關 Error Pattern

- PC-180（dual-project-sync-scope-conflation）：本地層 vs 共享層決策分離；本策略明示「V1 改框架 + 各專案改本地」雙層
- PC-122（error-pattern-conflict-not-synced）：跨專案同步版本衝突
- PC-162：接手獨立驗證——本 ticket detect 已實測推翻「APP 同 V1」假設（APP 多 3 組未處理碰撞）

### detect 腳本設計（acceptance #2）

prototype `/tmp/pc_detect.py` 已驗證可行（5 專案實測，見重現實驗結果）。正式化規格：

- **落點**：`.claude/scripts/detect_pc_collision.py`（隨 `.claude/` sync，各專案本地可跑）
- **輸入**：專案 `.claude/error-patterns/` 全 category rglob
- **偵測軸**：
  1. 同號異義：`(CAT, prefix, NNN)` → 2+ 不同 slug（檔名層，已驗證）
  2. 同 slug 異號：`(CAT, slug)` → 2+ 編號（重複候選）
  3. **異號同義（內容層，prototype 未涵蓋，待 IMP 補）**：複用 `sync_exclude_manifest.py` 的 `compute_content_hash`，跨前綴找雷同內容 → dedup/canonical 升格候選
- **輸出**：可執行清單（分三類），exit code 0=無碰撞 / 1=有碰撞（CI gate 友善）
- **整合**：建議掛 sync-pull 後自動跑（提示而非阻擋，呼應 methodology「偵測而非阻擋」）
