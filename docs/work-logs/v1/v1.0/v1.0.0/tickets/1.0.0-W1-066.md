## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

uv-tool-staleness-check-hook.py 的 `SKILLS` registry（行 52-89）對 7 個 skill 寫死 `module_subpath`（source 端 hash 計算來源目錄）與 `package_dir_name`（installed 端子目錄名）。`check_single_skill`（行 110-113）以 `project_root / skill.module_subpath` 推導 source dir，若該目錄不存在即回 `ERROR / "source dir missing"`。

兩例誤報的真正原因是 **registry 路徑佈局假設與實際 skill 佈局不符**（hook 邏輯本身正確，registry 過時/錯誤）：

| skill | hook 假設 module_subpath | 實際佈局 | 是否真為 uv tool |
|-------|------------------------|---------|-----------------|
| version-release | `.claude/skills/version-release/version_release/`（目錄） | source 在 `scripts/version_release.py`（單檔）；installed 在 site-packages `version_release.py`（單檔，非目錄） | 是（CLI `version-release` 已安裝） |
| branch-worktree-guardian | `.claude/skills/branch-worktree-guardian/branch_worktree_guardian/`（目錄） | source 在 `scripts/`；無 `branch_worktree_guardian/` 目錄 | **否**（pyproject 無 `[project.scripts]`，`uv tool list` 無此項，`which` 找不到 CLI） |

驗證（皆以 raw 指令固定值確認）：
- `get_project_root()` = `/Users/mac-eric/project/book_overview_v1`（正確）
- 7 skill module_subpath 存在性掃描：5 EXISTS（ticket / doc / mermaid-ascii / worktree / project-init）+ 2 MISSING（version-release / branch-worktree-guardian），恰與 ticket 描述「誤報 2 例」吻合。
- `uv tool list`：doc-system / mermaid-ascii / project-init / ticket-system / version-release / worktree-skill（6 項，**無 branch-worktree-guardian**）。
- `which version-release` = `/Users/mac-eric/.local/bin/version-release`（存在）；`which branch-worktree-guardian` = not found。
- version-release pyproject：`[tool.hatch.build.targets.wheel] sources=["scripts"] only-include=["scripts/version_release.py"]` → 將 `scripts/version_release.py` 映射為 installed module `version_release.py`（單檔）。
- installed module 實體：`.../uv/tools/version-release/lib/python3.12/site-packages/version_release.py`（單檔，非目錄）。

### 影響範圍

- 檔案：`.claude/hooks/uv-tool-staleness-check-hook.py`（registry 行 58-67 version-release entry、行 78-83 branch-worktree-guardian entry）。
- 功能：SessionStart staleness 提示對此 2 skill 永遠回報 `[ERROR] source dir missing`，污染提示訊息、侵蝕真實 stale 警示信任度（與 why 欄位指出的 PC-074 false positive 信任侵蝕同構）。
- 連帶：即便修正 source path，version-release 為**單檔模組**，hook 現有 source/installed 皆用「目錄 + rglob」模型（`module_subpath` 目錄、`find_installed_module_dir` 找目錄、`_has_any_py` rglob），無法套用於單檔 skill。

### 相關 Error Pattern

PC-074（false positive 信任侵蝕）為 why 欄位明示的同構模式（誤報使真警示失去信任）。staleness 誤報本質屬「registry 靜態假設與實況漂移」，與 PC-145（stale CLI install 診斷）/ PC-175（框架 sync 攜帶不符當前佈局的資產）家族相近——registry 對 skill 佈局的硬編碼假設未隨佈局演進更新。

## 重現實驗結果
### 實驗方法

唯讀重現 hook 判定路徑（不執行 SessionStart，避免副作用），逐項以固定值指令驗證：
1. 重現 `get_project_root()` 推導值。
2. 掃描 7 skill 的 `module_subpath` 目錄存在性，比對 hook 的 ERROR 判定點（行 110-113）。
3. 對 2 個 MISSING 例讀 pyproject.toml 與 scripts/ 佈局，確認真實 source module 位置。
4. `uv tool list` / `which` 確認 installed registry，區分「真 installed 但路徑假設錯」vs「根本未 install」。
5. `find_installed_module_dir` 重現 installed 端推導，確認 installed 結構（單檔 vs 目錄）。

```

          worktree/scripts, project-init/project_init
```

### 實驗發現

已驗證事實：
1. 根因 = **SKILLS registry 路徑佈局假設錯誤**（hook 邏輯與 get_project_root 推導皆正確；非 installed registry 過時）。三選一中明確排除「hook 邏輯 bug」「installed registry 過時（version-release 確已正確安裝）」，定位為「路徑佈局假設不符」。
2. 兩例性質不同：
   - version-release：**真 uv tool**，但 source 為單檔 `scripts/version_release.py`、installed 為單檔 `version_release.py`；hook 的「目錄 + rglob hash」模型不適用。
   - branch-worktree-guardian：**根本非 uv tool**（無 CLI entry、不在 uv tool list），被誤列入 SKILLS registry。
3. 5 個正常 skill 的 `module_subpath` 假設與實況一致，故僅這 2 例誤報。

仍未驗證（留待修復決策）：branch-worktree-guardian 是否原本就不該被 staleness hook 監控（最可能），或曾規劃為 uv tool 但未落地。
