## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 來源：v1.0.0 發佈就緒度盤點（打包，2026-06-07）。PM 派發前預寫 context（PC-040）。P3 低優先收尾票。

### 問題事實
- dist/ 殘留多版本 ZIP：目前有 readmoo-book-extractor-v0.18.0.zip（5/21）+ v0.20.0.zip。package.js 只清「同名」ZIP，不清舊版本 → 累積。分發時可能誤拿舊檔。
- build/production 未壓縮約 5.39 MB：esbuild bundle 只覆蓋 4 入口（background/content/popup/overview），其餘 src/ 仍逐檔複製進 build/，屬冗餘（不影響 load unpacked，ZIP 壓縮後僅 1.36 MB）。
- 註：dist/ 與 build/ 皆 gitignored，故本票 commit 只會動 scripts/（+ ticket md），不含 ZIP/build 產物。

### 修復方向
1. package.js：打包前清理舊版本 readmoo-book-extractor-v*.zip（只保留當前版本），避免累積。
2. 評估 build/production 散檔冗餘：build.js 是否應停止逐檔複製未 bundle 的 src/（只放 4 bundle + assets + manifest），或維持現狀（設計取捨）。記錄結論於 Solution（acceptance 為「評估並記錄」，可結論維持現狀+理由，不強制改 build.js）。

### 驗證
- npm run package 後 dist/ 只含當前版本 ZIP（舊版被清）
- 若改動 build.js 散檔行為：npm run build:prod + validate:build:prod 通過、load unpacked 仍可用、npm test 全綠
- 若散檔評估結論為維持現狀：Solution 記錄理由即可，不需改 build.js
- build 不得改 worklog（W2-003 已修，仍防範）；工作區最終乾淨

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 變更摘要（scripts/package.js）

新增 `cleanStaleZips(currentVersion)` 函式並在 `packageExtension()` 內 `createZip` 前呼叫：
- 打包前掃描 dist/，刪除所有符合 `readmoo-book-extractor-v*.zip` 命名規則但版本不等於當前版本的 ZIP。
- 當前版本 ZIP 不刪（交由 `createZip` 以「移除同名再重壓」處理），非本工具命名的檔案不動（避免誤刪 dist/ 其他來源檔案）。
- 用 `/^readmoo-book-extractor-v.+\.zip$/` 比對命名規則，刪除時印出 `[CLEAN] 移除舊版本 ZIP：dist/<name>` 滿足可觀測性。
- 同步更新檔頭設計決策註解新增一條「打包前清理 dist/ 內舊版本 ZIP」。

### 散檔冗餘評估結論：維持現狀（不改 build.js 散檔複製行為）

決策：**維持現狀**。理由如下。

事實盤點：
- build.js `filesToCopy = ['manifest.json', 'src/', 'assets/']`，逐檔複製整個 src/（253 個非測試 .js）進 build/production/。
- esbuild 僅 bundle 4 入口（background.js / content-modular.js / popup.js / overview.js），於 build/ 內覆蓋自身，依賴鏈已 inline。
- build/production/ 約 5.9 MB（多數為被 inline 但仍逐檔複製的散檔冗餘），但 ZIP 壓縮後僅 1.36 MB，遠低於 package.js 的 5 MB 上限。

不宜激進精簡（只放 4 bundle）的關鍵風險：
- popup.html 以普通 `<script src>` 載入 **popup-ui-components.js**（非 bundle 入口、無 require/import，設計為獨立 `<script>` 載入），屬真實 runtime 依賴，若只保留 4 bundle 會破壞 popup。
- overview.html 以 `<link>` 引用 `../core/design-system/design-system.css`（位於 src/core/design-system/），CSS 為 runtime 依賴；overview.css 亦同。
- 完整 runtime 表面 = manifest.json + 4 bundle + popup-ui-components.js + popup/overview 的 .html/.css + src/core/design-system/*.css + assets/。精準裁剪需逐檔辨識 HTML/CSS/動態載入依賴，屬高風險重構，且收益有限（壓縮後體積已合格）。

權衡：
- 收益：build/production/ 未壓縮體積可降，但對 load unpacked 與 ZIP 分發皆無實質影響（ZIP 1.36 MB << 5 MB）。
- 成本/風險：精準散檔白名單需維護 HTML/CSS/動態依賴清單，任何遺漏即造成 runtime 靜默缺檔（CSS 404 / popup 元件未載入）。屬不對稱風險。

結論：散檔冗餘為「無害冗餘」（壓縮後不超限、不影響 runtime），不值得以高風險重構換取。維持 build.js 現狀；若未來壓縮後 ZIP 逼近 5 MB 上限，再以獨立 ticket 設計精準散檔白名單。本票 acceptance「評估並記錄」即以此結論滿足。
