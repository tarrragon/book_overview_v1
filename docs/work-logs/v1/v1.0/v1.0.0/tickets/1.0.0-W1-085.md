## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

兩個獨立的框架跨專案治理缺陷在 app 專案 sync-pull 時暴露：

1. **識別符溯源失效**：框架 code 嵌入 project-local ticket ID（如 W9-001），透過 `sync-claude-push` / `sync-claude-pull` 雙向 overlay 同步到其他 consumer 專案後，該 ID 在目標專案不存在（斷源）或指向不同任務（偽源）。
2. **零審查直推爆炸半徑**：不完整重構未經任何強制驗證即直推所有 consumer，單點錯誤可同時使所有下游專案崩潰，無版本緩衝。

### 影響範圍

- `.claude/scripts/sync-claude-push.py`、`.claude/scripts/sync-claude-pull.py`（同步機制本身）
- `.claude/references/reference-stability-rules.md`（規則 8 識別符穩定性，現況評估發現已涵蓋 code 註解，真缺口在強制層）
- 所有透過 sync-pull 接收框架配置的 consumer 專案（下游爆炸半徑）

### 相關 Error Pattern

- **ARCH-V1-002**：雙向 overlay sync 製造重複 top-level 定義（本次意外揪出 `file_lock.py` 三組重複函式活 bug）
- **PC-V1-009**：機械缺陷誤診為流程缺陷（原提案誤把 import smoke test 可解的機械問題當成需人工 PR 審查的流程問題）

---

## 重現實驗結果

### 實驗方法

ANA 本質為治理策略評估，採三條獨立驗證軸：

1. **parallel-evaluation 五委員多視角審**（linux + Evidence + Alternatives + Scope + thyme），審查原提案「cherry-pick branch+PR review gate」。
2. **grep 量化 naive regex 強制 hook 的可行性**：掃描 ticket ID pattern 在 `.claude/` 全域命中數與 `source_ticket` 誤殺數。
3. **grep 獨立驗證 `file_lock.py` 重複函式定義**（PM 親自重跑，不採信報告數字，PC-162）。

### 實驗執行

- 五委員以 3 個獨立視角分別評估原方案，linux 行使品味否決權。
- grep 掃描 ticket ID pattern：全域命中 8686 處，其中 322 處為 `source_ticket` 合法引用（會被 naive regex 誤殺）。
- grep `file_lock.py` 函式定義：確認三組 top-level 重複定義（死碼 shadow）。

### 實驗發現

**已驗證的事實**：

- 規則 8 已涵蓋 code 註解中的識別符引用，真缺口是「無強制層」而非「規則未涵蓋」。
- naive regex hard hook 不可行：8686 處命中、322 處 `source_ticket` 誤殺，誤報率過高會被繞過而失效。
- W9-001 屬機械缺陷，可由 import smoke test 消除，無需人工 PR（Alternatives / linux / thyme 三視角一致）。
- `file_lock.py` 三組重複定義為真實活 bug（已 spawn W1-086 P0 修復，現已 completed）。

**結論**：否決原 branch+PR review gate 方案（franken-design、單人 repo 空殼、破壞 pull 錨點），改採 import smoke test 閘門（C1）+ 下游版本 pin（C3）+ 識別符衛生降級為文件慣例（B3/B1）。
