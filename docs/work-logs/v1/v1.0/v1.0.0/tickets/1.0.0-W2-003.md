## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 來源：v1.0.0 發佈就緒度盤點 + 本 session 親驗（2026-06-07）。PM 派發前預寫 context（PC-040）。

### 問題事實（本 session 已實證）
- scripts/build.js 的 syncWorklogStatus 在執行 build 時會改寫對應版本 worklog 的「**狀態**」標記為「開發中」。
- 本 session 執行 npm run build:prod（package.json=0.20.0）時，實際改動了 docs/work-logs/v0/v0.20/v0.20.0/v0.20.0-main.md（已釋出版本的 worklog），PM 須手動 git checkout 還原。
- 後果：build 是概念上的唯讀產出操作，卻會靜默 mutate git 追蹤的 worklog；打包者若未察覺會誤 commit，污染 commit 邊界（W2-001 驗收時即觸發一次）。

### 修復方向（擇一，agent 評估後決定並記錄理由）
- 方案 A：移除 build 流程中的 syncWorklogStatus 自動改寫（worklog 狀態改由 ticket/版本流程顯式管理）。
- 方案 B：將 syncWorklogStatus 限定在顯式 release/版本啟動模式才觸發，一般 build:dev/build:prod 不改 worklog。
- 不論哪案，須確保 build 後 git status 不含非預期的 worklog 變更。

### 驗證
- 連續執行 npm run build:prod 兩次後，git status 不出現任何 worklog（v*-main.md）變更
- npm run build:dev 同樣不改 worklog
- npm test 全綠（build.js 若有對應測試需同步）

### 分支
已在 feat/1.0.0-W2-003-build-worklog（PM 於 main 建立）。git add 僅限 scripts/ + 本 ticket md。commit 於 feat 分支，完成後 PM 驗證合併。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 方案選擇：方案 A（移除 build 流程中的 syncWorklogStatus 自動改寫）

**選擇理由**：

| 評估維度 | 方案 A（移除自動改寫） | 方案 B（限 release 模式才觸發） |
|---------|---------------------|--------------------------------|
| 副作用根除 | 徹底——build:dev/build:prod 皆不改 worklog | 部分——若 release 模式仍走 build 路徑則問題重現 |
| 職責歸屬 | 清晰——worklog 狀態生命週期單一交給 version-release skill | 模糊——build 與 release 兩處都可能寫 worklog 標記，易漂移 |
| 概念正確性 | build 回歸「唯讀產出」語意 | build 仍帶寫入副作用，違反 build 唯讀直覺 |

**決定性事實**：version-release skill 的 `version_release.py` 已顯式管理 worklog 狀態（start 子命令寫「進行中」、release 子命令標「已完成」）。build.js 的 syncWorklogStatus 是與之重疊的「自動補救路徑」，移除後責任不落空，反而消除 build vs release 兩處搶寫 worklog 標記的漂移風險。故採方案 A。

### 變更內容

1. **scripts/build.js（build() 主流程）**：
   - 移除 `MODE === 'production' && !SKIP_VERSION_CHECK` 區塊內對 `syncWorklogStatus(...)` 的呼叫與其 log 處理（原 line 546-558）。
   - 保留唯讀的 `validateVersionAlignment(...)` 斷言（確認 package.json 版號對應 worklog 目錄存在，不改寫任何檔案），維持「ZIP 帶錯版號」的 fail-fast 防護。
   - 更新 SKIP_VERSION_CHECK 分支的 warning 文字（移除「+ worklog sync」字樣）。
   - 更新區塊註解，記錄 1.0.0-W2-003 移除寫入副作用的 Why（build 概念上唯讀、避免靜默 mutate git 追蹤 worklog、worklog 狀態改由 version-release 顯式管理）。
   - `syncWorklogStatus` 純函式定義與 module.exports 匯出皆保留（仍是合法工具，僅不再由 build() 自動觸發），故既有純函式單元測試不受影響。

2. **tests/unit/scripts/build-version-check.test.js**：
   - 更新「--skip-version-check flag 行為宣告」describe 的過時註解（原述「略過 validate + sync 兩階段」改為「只執行唯讀 validateVersionAlignment，不再自動呼叫 syncWorklogStatus」）。純屬註解校正，無斷言變更，syncWorklogStatus 純函式測試全數保留並通過。

### 不在本 ticket 範圍

- syncWorklogStatus 純函式本身的刪除（保留作匯出工具，避免破壞既有測試與潛在 version-release 整合點）。
- version-release skill 對 worklog 狀態管理的擴充（已自行管理，無需本 ticket 介入）。
