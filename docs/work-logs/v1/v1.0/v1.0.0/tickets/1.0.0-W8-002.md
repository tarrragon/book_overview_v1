## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

auto-commit（W7-001）僅在 `track_acceptance.py:execute_append_log` 內呼叫 `_auto_commit_ticket_md`。所有其他修改 ticket md 的路徑（ticket CLI 的 claim/complete/release/set-status/metadata-sync 內部 `save_ticket` 呼叫，以及 PM/agent 手動 Edit）均不觸發 auto-commit，body 停留 working tree 直到下一個人工或流程 commit。

### 影響範圍

| 修改路徑 | auto-commit 覆蓋 | 暴露窗口 |
|---------|-----------------|---------|
| `ticket track append-log` | 有（W7-001） | 無（即時 commit） |
| `ticket track check-acceptance --all`（勾選 AC） | 無 | 至 PM/agent 下次 commit |
| `ticket track claim`（metadata sync + Context Bundle） | 無 | 同上 |
| `ticket track complete`（metadata sync） | 無 | 同上（但 complete 後通常緊接其他 commit） |
| `ticket track set-status / release` | 無 | 同上 |
| PM/agent 手動 Edit ticket md（body 內容補寫、AC 手動勾選等） | 無 | 至下次 commit — **最長暴露窗口** |

### 相關 Error Pattern

- PC-178（ticket body 未 commit 被 git 還原覆蓋 + 防護演進）
- PC-019（worktree merge state loss）

### 暴露窗口實際風險評估

手動 Edit 的暴露窗口**在實務上通常很短**：PM/agent 手動 Edit 後幾乎一定會接 append-log（觸發 auto-commit）或 commit 產品碼（夾帶 ticket md）。**高風險場景**：
1. PM 手動 Edit ticket md 後立即 `/clear` 或 session 意外中斷
2. Agent worktree 內手動 Edit ticket md 但因 API 錯誤中斷（W2-009.3 模式）
3. PM 手動 Edit 後下一步觸發 `git checkout`/`git stash`（sync-push 等自動序列）

## 重現實驗結果

### 實驗方法

程式碼路徑追蹤：盤點所有 ticket md 修改路徑，確認哪些走 auto-commit、哪些不走。本 ticket 為評估型 ANA（評估防護覆蓋範圍），非 bug 重現型。

### 實驗執行

1. grep `_auto_commit_ticket_md` 呼叫點 → 僅 `track_acceptance.py:execute_append_log`
2. grep `save_ticket` 呼叫點 → claim/complete/release/set-status/metadata-sync 均有
3. 確認上述 save_ticket 呼叫後無 auto-commit
4. 歸納手動 Edit 路徑為最長暴露窗口

### 實驗發現

**已驗證事實**：
- auto-commit 僅覆蓋 append-log 路徑（1 處呼叫）
- claim/complete/release/set-status 均走 save_ticket 無 auto-commit（5+ 處）
- PM/agent 手動 Edit 完全無防護

**仍未驗證的假設**：
- 手動 Edit 後實際遺失頻率（歷史僅 W1-017 一例，auto-commit 前）

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 四階段

#### W（Widen）— 候選方案

| # | 方案 | 機制 | 覆蓋範圍 |
|---|------|------|---------|
| 1 | 現狀維持 + PC-178 文件提示 | 僅靠人為意識 | 人為自律 |
| 2 | 復活方案 C（PreToolUse hook 偵測 git 還原 + modified ticket md 警告） | Hook 攔截 git checkout/reset/stash Bash 命令 | git 還原時有 modified ticket md → 警告 |
| 3 | 擴大 auto-commit 覆蓋（lifecycle save_ticket 後也 auto-commit） | 程式碼修改 | 所有 CLI 路徑（但不含手動 Edit） |
| 4 | 輕量「未 commit ticket md 偵測」PostToolUse hook | commit/session-end 時偵測 modified ticket md → 警告 | 所有修改路徑（含手動 Edit） |
| 5 | 方案 3+4 組合 | CLI 路徑 auto-commit + 手動 Edit hook 偵測 | 全覆蓋 |

#### R（Reality-test）— 風險量化

**歷史案例頻率**：W1-017 是唯一已確認的 ticket body 遺失案例。auto-commit 根因解後，append-log 路徑（佔 ticket md 修改的 ~80%）已完全防護。殘留暴露路徑（手動 Edit + CLI metadata 操作）的**遺失概率極低**：需同時滿足 (a) 手動 Edit 後 (b) 未接 append-log 或 commit (c) 觸發 git 還原。

**自動序列風險**：sync-push / worktree merge 等自動序列在 ticket md 有 modified 變更時仍可能觸發 checkout/stash，但這些序列通常在 session 邊界執行（/clear 後），而非 Edit 後即刻觸發。

**碎 commit 代價（方案 3）**：每次 claim/complete/set-status 增一個 chore commit。ticket CLI 操作頻率高（一個 session 約 5-15 次 save_ticket），碎 commit 量會從目前的 2-4/ticket 暴增至 10-20/ticket，git log 噪音不可接受。

#### A（Attain distance）— 機會成本

- 方案 2/4（hook）：開發+測試約 1-2 小時；維護成本低
- 方案 3：改動 lifecycle.py 多處，碎 commit 噪音高
- 方案 1：零成本但接受殘留風險

#### P（Prepare to be wrong）— 失敗預想

- 方案 2 最可能失敗原因：Bash 命令正則誤報（git diff 被誤判為 git 還原）或漏報（管道/alias 繞過）
- 方案 4 最可能失敗原因：PostToolUse 觸發點不對（CC runtime 可能無 session-end hook）

### 結論

**採方案 4（輕量未 commit ticket md 偵測 hook）+ 方案 1（PC-178 文件提示維持）**

理由：
1. 方案 C（W7-001 刪除的 PreToolUse 攔截 git 還原）定位為**攻擊面攔截**——只防 git 還原時刻。方案 4 定位為**狀態偵測**——偵測「ticket md 已修改但未 commit」的危險狀態本身，覆蓋面更廣（不限 git 還原觸發，也防 session 中斷遺失）。
2. 方案 3（擴大 auto-commit）碎 commit 代價不可接受。
3. 方案 2（復活方案 C）誤報/漏報風險高且只覆蓋 git 還原場景。
4. 殘留風險極低（歷史 1 例 + 主路徑已防護），輕量 hook 警告（非阻擋）即為適當防護等級。

### 虛假安全感二階風險評估

**風險描述**：auto-commit 覆蓋 append-log 路徑後，PM/agent 形成「ticket md 已自動保護」心智模型，降低對手動 Edit 後主動 commit 的警覺。

**緩解措施**：
1. **PC-178 已記錄**手動 Edit 為補充防護場景（W7-002 DOC 產出）
2. **方案 4 hook 警告**：在適當時機（如 commit 前或 session-end 類觸發）偵測 modified ticket md 並輸出提示，打破「已自動保護」假設
3. **memory 已更新**（`append_log_auto_commit_workflow`）：明示 auto-commit 僅覆蓋 append-log，手動 Edit 需自行 commit

**結論**：虛假安全感風險存在但可控——PC-178 文件 + hook 可見性提示形成雙層緩解。hook 不阻擋（exit 0 + 警告），避免過度防護干擾流程。

### Spawn 規劃

| # | 類型 | 描述 | 條件 |
|---|------|------|------|
| 1 | IMP | 實作輕量「未 commit ticket md 偵測」hook（PostToolUse 或 PreToolUse Bash commit 時觸發，偵測 modified ticket md → stderr 警告） | 建議但非必要（P2）|

**Spawn 落地確認**：SA 無 ticket create 權限，PM 需接手建立 spawn IMP ticket。
