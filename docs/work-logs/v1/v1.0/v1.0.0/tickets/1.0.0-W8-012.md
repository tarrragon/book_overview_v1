## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

blog 是 multi-round-review 與 compositional-writing 兩 skill 的上游 SSOT。blog 端 SKILL.md 以「co-located 環境」假設撰寫——相對路徑連結指向同目錄姊妹 skill（case-first-module-workflow），並直接引用 blog 專屬規範檔（AGENTS.md / markdown-writing-spec）與 CLI（mdtools）。回流到缺對應資產的專案（book_overview_v1 未複製 case-first-module-workflow）後，相對連結成 dangling reference，專屬識別符成其他 reviewer 找不到的對象。

### 影響範圍

- `.claude/skills/multi-round-review/SKILL.md`：L142 case-first 死連結（broken-link-check 持續告警）、L39 / L128 blog 專屬識別符。
- 跨專案：任何透過 blog 回流取得此 skill 但未同步取得 case-first / AGENTS.md / mdtools 的專案皆受影響。

### 相關 Error Pattern

同 hook-flutter-preset-assumption 模式（feedback memory）：上游為特定環境寫的預設假設，sync 後在缺對應前提的專案失效。亦呼應 framework-no-ticket-references 原則（框架資產應為最廣相容形態，不綁特定專案/環境識別符）。

## 重現實驗結果
### 實驗方法

於 book_overview_v1 與 blog 兩端，以 ls / grep 驗證三項事實：(1) case-first-module-workflow skill 是否存在於 book_overview_v1；(2) multi-round-review SKILL.md 的 case-first 引用形態；(3) AGENTS.md / markdown-writing-spec / mdtools 識別符位置。並檢查 broken-link-check 是否具白名單機制。

### 實驗執行

- `ls -d .claude/skills/case-first-module-workflow`（book）→ No such file or directory
- `grep -n case-first-module-workflow`（book SKILL.md）→ L142 為 markdown link `[...](../case-first-module-workflow/SKILL.md)`
- blog 端 `find` → blog 同時擁有 multi-round-review 與 case-first-module-workflow 兩 skill（故 blog 內該連結有效）
- `grep AGENTS.md|markdown-writing-spec|mdtools`（book）→ L39（寫作規範 audit）、L128（mdtools lint/cards/fmt）
- broken-link-check skill 目錄僅含 SKILL.md，無白名單機制

### 實驗發現

已驗證事實：
1. case-first 死連結為真——book 無該 skill，相對路徑解析失敗，broken-link-check 持續告警。
2. 問題本質為「上游（blog）為自身 co-located 環境寫的相對連結與專屬識別符，回流到缺對應資產的專案後成 dangling reference」，同 hook-flutter-preset-assumption 模式。
3. 死連結與識別符修復載體不同：識別符在回流檔內（改本地會被覆蓋，須上游處置）；死連結可由純文字化在上游一次根治。
4. broken-link-check 無白名單機制——選項 (c) 需先新建功能，成本高於預期。
5. L143 的 compositional-writing 連結在 book 有效（該 skill 存在），無需處理。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 決策：(b) 上游防衛性改寫 + 重新回流（用戶 AskUserQuestion 選定）

WRAP 候選方案比較：
- (a) 複製 case-first skill 進 book + blog 識別符通用化：死連結抗回流，但複製了 book 未採用的 skill，徒增維護面。
- (b) 上游防衛性改寫【採用】：在 blog SSOT 一次根治，符合「上游修、不本地補丁」原則；同步改 book 端使兩端一致 → 下次回流冪等，中和「本地改被覆蓋」風險。
- (c) 接受現狀 + 白名單：broken-link-check 無白名單機制，需先建功能，即時成本反而最高。

### 執行內容（blog SSOT + book 兩端同步，共 6 處）

1. L39 `AGENTS.md / markdown-writing-spec` → `專案寫作規範（如 AGENTS.md / markdown-writing-spec）`（標示為示例，其他專案可映射自身規範）
2. L128 `mdtools lint / cards / fmt` → `專案 markdown 工具鏈（如 mdtools lint / cards / fmt）`
3. case-first：markdown link `[...](../case-first-module-workflow/SKILL.md)` → 純 code-span `case-first-module-workflow（若專案已採用此 skill）`。降為純文字是唯一能讓 broken-link-check 在「有/無該 skill」的專案皆通過的形態（連結路徑檢查與周圍措辭無關）。

### 設計取捨

- blog 損失一個可點連結，換取跨專案可攜性——符合 SSOT 應為最廣相容形態的原則。
- compositional-writing 連結（L143）保留 markdown link：book 與 blog 皆有該 skill，連結處處有效。

### 驗證

book 端 `grep "](../" SKILL.md` 僅剩 compositional-writing（有效）；case-first 已無 markdown link → broken-link-check 不再告警。
