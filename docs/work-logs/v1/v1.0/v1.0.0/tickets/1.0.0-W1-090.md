## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 三軸設計結論（前台 ANA，佐證：error-pattern-numbering-methodology 結構閱讀 + W1-085 五委員審查）

核心：issue 錨定不是新機制，是補強既有「升格」步驟的弱點。現行方法論已分層：捕獲(staging 前綴)/dedup detect(content-hash)/升格(canonical alias)。

### 三軸職責劃分 + issue 介入時機
| 軸 | 階段 | issue 角色 | 摩擦容忍 |
|----|------|-----------|---------|
| A 捕獲 | local-first 前綴、可離線 | 不介入 | 零容忍 |
| B 去重/身份 | sync 後 content-hash detect → 人工裁決 | 升格時建 issue 作 canonical key | 低頻可事後 |
| C 修復追蹤 | 跨 consumer 修復狀態 | 同一 issue 承載 | 可事後 |

### 設計定案
issue # 作「升格時」賦予的 canonical 身份（可選 canonical_issue frontmatter 欄），非捕獲前置關卡、非主鍵。三論證：
1. 升格本就低頻刻意（方法論原文），建 issue 成本可吸收，不傷捕獲零摩擦。
2. issue #（人工已確認同一）強於 content-hash（機器猜雷同）；detect 撈候選→人工確認→建 issue 釘身份，兩者流水線前後段非替代。
3. 方法論已否決 content-hash/UUID 作主鍵（無可讀性）；同理 issue # 只當升格產出錨點。

一個 framework issue = B canonical 身份 + W1-089 provenance 錨點 + C 修復追蹤。

### 阻塞前提（待根決策）
W1-090 與 W1-088 共用同一根決策：框架 repo 是否從「純 overlay sync 程式碼倉」升級為「有 issue tracker + release/版本」的正式上游。此根決策 yes 才能 spawn W1-090 子項。故 W1-090 暫 in_progress 待根決策。

### 展開設計（深化：issue 的真正增量 + 配號 crux 選型）

### 1. 精修——issue 不是 error-pattern 的替代身份
佐證（方法論 L61-84 + grep 確認）：error-pattern **flat-base 升格已是 canonical 身份層**（升格=於凍結 base 賦 canonical alias）。故：
- 軸 B（error-pattern 去重身份）：已由 flat-base 解決，**issue 對此冗餘**。
- 軸 C（跨 consumer 修復追蹤）：flat-base 號只是身份、**無狀態**；追蹤「哪些 consumer 修了」需有 body/state 的物件 → issue 真正用武之地。
- provenance（W1-089）：適用任意 code/doc 變更（非僅教訓）→ issue 是比 error-pattern 號更通用的錨點。
結論：issue = 「跨 consumer 狀態追蹤 + 通用 provenance」的新物件；error-pattern 可選 link（canonical_issue 欄），非被 issue 取代身份。

### 2. 配號碰撞 crux → 技術選型
- file-based issue registry：各專案離線分配必撞號，重演 error-pattern「前綴+升格」已費力解決的問題——不該再造。
- GitHub Issues：單一框架 repo、server 端原子分配 → 全域唯一無碰撞；網路依賴只在「升格/escalation」低頻刻意動作付出（同捕獲零摩擦原則：成本放低頻收斂點）。
- 選型：canonical 錨點用 GitHub Issues on tarrragon/claude.git。

### 3. canonical_issue frontmatter 欄
可選欄；值如 'tarrragon/claude#42'。捕獲時不填（零摩擦），升格/需跨 consumer 追蹤時 stamp。

### 4. 一個 issue 三用
- B（可選 link）：error-pattern canonical_issue 指向它。
- C：issue body 維護 consumer 修復矩陣（V1 done / APP pending）。
- provenance：code/doc 變更 commit trailer / 註解引 'claude#42' 取代 project ticket ID（W1-089 落地）。

### 5. 根決策（informed）
root-yes 的價值主要在 **軸 C + 通用 provenance**（軸 B 已由 flat-base 覆蓋）。
root-no：軸 B 照常；軸 C 缺有狀態中央物件 → 各專案 worklog 碎片化；provenance 退回自足 WHY+frontmatter（W1-089 file-based）。
故根決策實質 = 「需不需要跨 consumer 修復狀態追蹤 + 通用 provenance，值不值得引入 GitHub Issues 網路依賴」。
