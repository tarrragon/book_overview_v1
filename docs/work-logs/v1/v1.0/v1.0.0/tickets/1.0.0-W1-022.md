### Prototype 邏輯（W1-019.5 已驗證，正式化基礎）

檔名解析 regex：`^([A-Z]+)(?:-([A-Z0-9]+))?-(\d+)-(.+)\.md$`（group: CAT / PROJ前綴 / NNN / slug，同時支援 flat 與前綴格式）。掃描 `.claude/error-patterns/` 全 category rglob，建兩個索引：

- 軸 1 同號異義：`by_num[(CAT, prefix, NNN)] = [slug...]`，len(set) > 1 即碰撞
- 軸 2 同 slug 異號：`by_slug[(CAT, slug)] = [編號...]`，len(set) > 1 即重複候選
- 軸 3 異號同義（**prototype 未涵蓋，本 ticket 新增**）：per-file 內容雜湊，跨不同編號找雷同內容

### 設計關鍵（避免踩坑）

| 項目 | 說明 |

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 實作摘要

正式化 W1-019.5 prototype（`/tmp/pc_detect.py`）為共享腳本 `.claude/scripts/detect_pc_collision.py`，補齊第三偵測軸。

### 三偵測軸

| 軸 | 名稱 | 判定 | 索引 |
|----|------|------|------|
| 1 | 同號異義 | 同一 `(CAT, prefix, NNN)` 對應 2+ 不同 slug | `by_num` |
| 2 | 同 slug 異號 | 同一 `(CAT, slug)` 出現於 2+ 不同編號 | `by_slug` |
| 3 | 異號同義（本 ticket 新增） | per-file 內容雜湊（正規化首行去編號）跨不同編號雷同 | `by_content` |

### 設計決策

- **content-hash 軸不複用 compute_content_hash**：`sync_exclude_manifest.compute_content_hash` 是整目錄雜湊（單一目錄指紋），不適合 per-file 兩兩比對。軸 3 改用 `compute_file_content_hash`（per-file `hashlib.sha256`），且雜湊前以 `_normalize_first_line` 剝除首行編號字面（`_NUMBER_TOKEN_RE`），才能偵測「僅編號不同、內容相同」。
- **正規化僅限首行**：id/標題慣例承載編號，僅剝首行避免誤判正文不同的檔案為同義（`test_axis3_normalization_strips_only_number_not_real_diff` 守護）。
- **pure function 拆分**：`parse_filename` / `compute_file_content_hash` / `scan` / `has_collision` / `format_report` / `main`，便於 importlib 單元測試。
- **exit code**：0=無碰撞 / 1=有碰撞 / 2=目錄不存在（CI / sync-pull gate 友善）。
- **檔名 regex** 同時支援 flat（`PC-010-slug.md`）與前綴（`PC-V1-001-slug.md`）格式。

### 不在本 ticket 範圍

- 各專案實際 remediation（屬各專案 session 執行，W1-019.5 handoff 規劃）。
- 自動重編號（由既有 sync-claude-pull.py resolve_pc_collision 負責，本腳本只偵測不修改）。
