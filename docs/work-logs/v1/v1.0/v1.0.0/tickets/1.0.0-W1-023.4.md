## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

`bash-tool-usage-rules` 規則一禁裸 cd（裸 cd 觸發 zsh chpwd hook 的 ls 淹沒，為 confabulation 觸發鏈第 1 環）。但此規則屬 soft rule（規則知識），無執行層強制。母 ANA 1.0.0-W1-023 已確認 soft rule 對「當下行為」無強制力——知識影響輸出分佈，不保證單次合規。

### 影響範圍

- `.claude/hooks/`：hook 系統執行層（是否新增/擴充 hook 強制）
- `.claude/rules/core/bash-tool-usage-rules.md`：規則層（既有 soft rule）

決策分歧：hookdesign 加法視角（soft rule 失效需 hook 強制）vs taste 減法視角（零新 hook，避免過度工程）。本 ANA 以 dogfooding 活體自證裁決，由用戶定奪最終路徑。

### 相關 Error Pattern

- IMP-056：chpwd shell hook 自動 ls 淹沒工具輸出（裸 cd 的可觀測信號）
- PC-166：confabulation 幻覺工具結果（裸 cd 為觸發鏈第 1 環：輸出淹沒 → result 邊界模糊 → 同訊息腦補）
- `tool-output-trust-rules` 規則 4：git -C 取代裸 cd（觸發鏈源頭阻斷）

## 重現實驗結果
### Dogfooding 實驗：本 session PM 裸 cd 行為觀察（活體自證）

實驗對象：本 session PM（正在撰寫 confabulation 防護 + 禁裸 cd 規則 + 固化 git -C 取代裸 cd）
觀察方法：工具輸出中 chpwd ls 淹沒（裸 cd 的可觀測信號，IMP-056）

| 時點 | 操作 | 裸 cd 形式 | chpwd ls 淹沒 |
|------|------|-----------|--------------|
| W1-021 smoke test | chmod + smoke test | `cd /Users/.../book_overview_v1` | 是（大量 ls 列表淹沒工具輸出）|
| W1-023.1 append Solution | append-log | `cd ... 2>/dev/null` | 是（大量 ls 列表淹沒工具輸出）|

**結果**：N≥2 次裸 cd 違規，發生在「PM 已讀規則 + 正在寫禁裸 cd 規則」條件下。

**結論**：soft rule（規則知識）不阻止當下違反——「規則知識 ≠ 行為合規」（與 document-writing-style「理解規範不代表輸出遵守」同精神）。支持建 warn hook 在「當下」提示。

**限制**：N=2 為下界（可能有未注意的裸 cd）；精確統計需 Bash 命令歷史（不在 git，無法事後考古）。但 N≥2 已足證 soft rule 失效（單次違反即證知識不足以阻止當下行為）。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 決策：建 warn 級裸 cd guard hook（用戶定奪）

### Dogfooding 證據（活體自證為最強）

本 session PM 讀過 bash-rules 規則一（禁裸 cd）+ 正在固化 tool-output-trust 規則 4（git -C）+ 整個任務寫 confabulation 防護，**仍裸 cd 至少 2 次觸發 chpwd ls 淹沒**。連寫規則的 PM 都違反 → soft rule 對「當下行為」無效（鐵證）。

**測量限制**：「跨 session 裸 cd 違規數」無 git 痕跡無法統計（裸 cd 是 Bash 命令不進 git），acceptance [0] 此部分合理漂移，用本 session 活體自證替代。W1-023.2 規則強化剛落地，殘留率需後續觀察；本 session（強化前基線）已證 soft rule 無效。

### 決策權衡與結論

| 視角 | 主張 | dogfooding 對照 |
|------|------|----------------|
| hookdesign（加法）| 裸 cd 唯一可硬 hook 點，soft rule 失效需 hook | 活體自證支持 |
| taste（減法）| 零新 hook | 未否決裸 cd hook（否決的是 hook 偵測續寫/一訊息一工具/禁管道），偏好被 dogfooding 推翻 |

**結論：建 warn 級裸 cd guard hook**。理由：dogfooding 鐵證 soft rule 失效 + warn 不 deny 不傷工作流（裸 cd 本就該避免，提示改 git -C）+ taste 未否決此 hook。

### Hook 設計要點（供 IMP 落地）

- 類型：PreToolUse Bash hook（UV single-file，含 shebang + chmod +x，PC-086）
- 偵測：`tool_input.command` 含裸 cd（`^cd ` 或 `&& cd ` / `; cd `），**排除**子 shell `(cd ...)`、`git -C`、`uv -d`、絕對路徑還原 `cd /project/root`
- 行為：**warn**（permission_decision="allow" + reason 提示改 git -C / 子 shell），**不 deny**（不傷工作流）
- 可觀測性：reason 寫 stderr（規則 4）
- 註冊：settings.json PreToolUse matcher Bash（只註冊一處，PC-148）
- 範例參照：error-pattern flat gate hook（PreToolUse + 模式 filter + emit_hook_output）

執行 tool-discovery 五問之「既有模組有 API 但沒接線嗎」，盤點 11 個 PreToolUse Bash hooks，結論：`bash-edit-guard-hook.py` 為理想宿主。

**權衡**：bash-edit-guard 現職責為「編輯操作警告」，加入裸 cd 會將範疇輕度擴展為「Bash 命令衛生警告」（edit-specific → command hygiene）。
