### 問題根因

scan_links.py 預設計數 237 中約 99 筆（42%）為 false positive，根因為兩處排除規則覆蓋面不足，與 ANA W8-030 curated 語意不符：

1. **backup 來源端排除不對稱**：`classify_ref` 僅以 `resolved` target 端判斷 `migration-backups/`，未檢查 source 檔本身是否在 `.claude/migration-backups/`。162 個 backup md 在掃描範圍內，其內部引用（指向已搬移檔）貢獻 30 筆 broken。ANA curated 原意為「source scope 排除 migration-backups」，故須對稱排除來源端。

2. **placeholder 樣式集過窄**：`PLACEHOLDER_SAMPLES` 為 4 項 exact-match，僅命中 SKILL 表格固定範例。漏掉文件中大量樣式型示意路徑：glob 萬用字元（`*` / `**`）、角括號佔位（`<檔名>`）、大括號模板（`{language}`）、xxx/TEST 哨兵 token。

### 影響範圍

- `.claude/skills/broken-link-check/scan_links.py`（`classify_ref` + `scan` + 新增 `is_placeholder_pattern`）
- `.claude/skills/broken-link-check/tests/test_scan_links.py`（新增 14 測試）
- 下游影響：237 作為 W8-034 清理權威 baseline 不可信；修正後 139 為對齊 ANA 的權威斷鏈計數。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

evidence-driven bugfix（先補 failing 測試 → 修正 → 全綠 → 重跑 live 對照）。

### 缺陷 1：backup 來源端對稱排除

在 `scan()` 逐檔迴圈加來源端判斷：source 檔路徑含 `migration-backups/` 且 `include_migration_backups` 旋鈕關閉時，整檔引用歸 `excluded_backup`（continue）。與 `classify_ref` 既有 target 端排除對稱；旋鈕開啟時仍計入（測試 `test_backup_source_counted_when_knob_on` 守護）。

### 缺陷 2：placeholder 樣式偵測

新增純函式 `is_placeholder_pattern(raw)`，以 5 條 regex 偵測樣式型 placeholder：
- `_GLOB_PLACEHOLDER` `[*?]`、`_ANGLE_PLACEHOLDER` `<[^>]*>`、`_BRACE_PLACEHOLDER` `\{[^}]*\}`
- `_XXX_TOKEN`（`xxx` 段，不分大小寫）、`_TEST_TOKEN`（UPPERCASE `TEST(_[A-Z0-9]+)*` 哨兵）

`classify_ref` 改為 `raw in PLACEHOLDER_SAMPLES or is_placeholder_pattern(raw)`，保留固定樣本集向後相容。token 偵測刻意限 UPPERCASE TEST，避免誤排含小寫 test 的真實檔名（反例測試守護）。

### 契約相容性

未修改既有 30 測試任一斷言；新增 14 測試（10 參數化 placeholder + 1 反例守護 + 1 placeholder 旋鈕 + 2 backup-source）。既有 synthetic fixture 與 GWT 場景全數沿用，無契約破壞。

無需建 ticket：本 ticket 即 W8-030.1 衍生 follow-up，修正自閉環；139 權威計數供既有 W8-034 消費（非新增 spawn）。
