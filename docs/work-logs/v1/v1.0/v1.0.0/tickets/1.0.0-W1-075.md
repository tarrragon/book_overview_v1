## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

PC-V1-004 防護 C 方案 D（W1-073 ANA 結論，用戶 2026-06-11 確認採 D+A）。30 個注入 hook 中 10 個暴露於 subagent 無偵測；`emit_hook_output()` 已存在於 hook_io.py 但 adoption 僅 6/48（12.5%），點狀防護無法防新 hook 重蹈。

### 影響範圍（10 個 gap hook 清單，來自 W1-073 重現實驗）

| Hook | 事件類型 | 暴露風險 |
|------|---------|---------|
| active-dispatch-tracker-hook | PostToolUse | 中 |
| cli-failure-help-reminder-hook | PostToolUse | 低 |
| comment-qa-hook | PostToolUse | 低 |
| dispatch-count-guard-hook | PostToolUse, UserPromptSubmit | 中 |
| file-ownership-guard-hook | PreToolUse | 低 |
| pre-test-hook | PreToolUse | 低 |
| session-context-guard-hook | PostToolUse | 低 |
| skill-cli-error-feedback-hook | PostToolUse | 低 |
| utf8-integrity-check-hook | PostToolUse | 低 |
| worklog-format-check-hook | PostToolUse | 中 |

### 設計約束

- `audience` 參數枚舉 `pm_only` / `all`，預設值選擇需向後相容（既有 6 個 emit_hook_output 呼叫不破壞——Never break userspace）
- 內部以 `is_subagent_environment(input_data)` 過濾：subagent 觸發且 audience=pm_only 時不輸出（或輸出空殼 + suppressOutput，依既有 W1-071 跳過慣例）
- Stop event 無 agent_id（CC runtime 硬約束）：emit_hook_output 對 Stop 無法偵測，該盲區由 W1-076（[PM-ONLY] 前綴規則）補；本票實作時 audience=pm_only 的輸出可預留前綴掛點但不實作前綴邏輯（契約歸 W1-076，已設 blockedBy 序列）
- 與 ARCH-V1-001 對齊：不變量驗證下沉共同瓶頸（emit_hook_output），不只修 10 個個案

### 相關 Error Pattern

PC-V1-004（防護 C）、ARCH-V1-001（單點強制）、ARCH-020（禁平行實作，過濾邏輯必須 import hook_io 統一 helper）

### Force 理由

PM 派發前 PCB 寫入（PC-040 context 先入 ticket；pending 階段 bookkeeping 路徑摩擦由 1.0.0-W1-058 追蹤中）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 實作設計

**hook_io.py — emit_hook_output 擴充（過濾邏輯唯一實作點，ARCH-020 / ARCH-V1-001）**

| 設計點 | 決策 |
|--------|------|
| 新參數 | keyword-only `audience="all"` + `input_data=None`，預設值向後相容既有 6 個呼叫點（Never break userspace） |
| 過濾語意 | `audience="pm_only"` 且 `is_subagent_environment(input_data)` 為 True 時丟棄 `additional_context`，輸出退化為無訊息基本結構（沿用 W1-071 跳過慣例） |
| permission 欄位 | 不過濾（功能性決策非資訊性訊息；pre-test-hook 的 allow 決策在 subagent 觸發時仍生效） |
| 非法 audience | raise ValueError（開發期錯誤測試階段即暴露，避免拼字錯誤造成 PM-only 訊息靜默洩漏） |
| W1-076 掛點 | 函式內留註解標記前綴插入點，本票不實作前綴邏輯（契約屬 1.0.0-W1-076） |
| Stop event 盲區 | docstring 明示：Stop 無 agent_id（CC runtime 硬約束），由 1.0.0-W1-076 前綴規則補位 |

**10 gap hook 遷移**（訊息路徑改走 `emit_hook_output(..., audience="pm_only", input_data=...)`，hook 內零複製判斷）：

| Hook | 遷移點 |
|------|--------|
| active-dispatch-tracker-hook | housekeeping 訊息（PostToolUse） |
| cli-failure-help-reminder-hook | PC-005 提醒（PostToolUse） |
| comment-qa-hook | 品質報告 + 兩個 except 錯誤訊息（main 前置 `input_data = None` 供 except 區塊安全傳入） |
| dispatch-count-guard-hook | 派發計數警告（UserPromptSubmit；無 agent_id 時 fallback 為 PM 行為，安全） |
| file-ownership-guard-hook | `_generate_output` 重構為 `_emit_output`（dict 回傳 + print 改為統一出口直接輸出） |
| pre-test-hook | 環境警告 + allow 決策合併輸出（順手移除 unused `import json` / `import os`） |
| session-context-guard-hook | handoff 提醒（PostToolUse） |
| skill-cli-error-feedback-hook | 系統功能缺失回饋 + SKILL 引導缺陷回饋兩路徑 |
| utf8-integrity-check-hook | 既有 emit_hook_output 呼叫補 audience + input_data |
| worklog-format-check.py | 格式提醒（W1-073 表中名為 worklog-format-check-hook，實際檔名無 -hook 後綴） |

### 範圍決策

- 各 hook 的無訊息 DEFAULT_OUTPUT 早退路徑保持原樣：不攜帶訊息即無受眾 gap，最小化回歸面
- dispatch-count-guard except 路徑的 suppressOutput 形狀不屬 hookSpecificOutput 協定，不在 emit_hook_output 支援範圍，保留
- PEP 723 deps 無變更：emit_hook_output 屬本地 hook_utils，非外部套件（PC-124 不適用）

### Phase 4 評估報告

- **重構需要**: 否（emit_hook_output 認知負擔指數約 6：參數分支 + 1 巢狀層 + 過濾單點；10 hook 遷移為等價替換）
- **技術債務**: 無新增；其餘 38 個非 gap hook 不遷移屬 1.0.0-W1-073 已決策範圍（ROI 差，Spawn 規劃已明記豁免）
- **程式碼品質**: A（DRY 達成——過濾判斷全專案唯一實作點；負向防護有正向錨點——ValueError 訊息直接列出合法枚舉）
