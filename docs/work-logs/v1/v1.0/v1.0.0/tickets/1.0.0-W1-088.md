## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

現行 `sync-pull` 永遠以 remote/main HEAD 為 target，per-consumer 僅記 `last_synced_base_sha` 作三方合併 base，push 不打 tag、已算出的 VERSION 被 pull 忽略。直接後果：任何壞掉的 main HEAD 在下次 pull 時即傳染全部 13 個 consumer，無版本緩衝，崩潰瞬時且全量。

### 影響範圍

- `.claude/scripts/sync-claude-push.py`（需新增 git tagging）
- `.claude/scripts/sync-claude-pull.py`（target ref 解析需支援 pinned tag vs HEAD、新增 `--bump`）
- consumer `.sync-state.json`（新增 `pinned_version` 欄）
- 全部 13 個下游 consumer 的傳播時序（pin 後改為刻意 bump）

### 相關 Error Pattern

- 衍生自 source ANA `1.0.0-W1-085` 的 C3 方向（爆炸半徑缺口）；與 `1.0.0-W1-087`（C1 import smoke test 生產閘）組防禦縱深。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 展開設計（版本 pin 拆爆炸半徑 C3）

### 現行機制（爆炸半徑根源，已讀碼確認）
sync-pull target 永遠是 remote/main HEAD；per-consumer 只記 last_synced_base_sha 作三方合併 base。push 不打 tag、VERSION（v1.48.5）已算但被 pull 忽略。→ 壞 HEAD 下次 pull 即傳染全 consumer。

### Pin 設計（option A：tag-based，最小正確）
1. push 對每版打 git tag（VERSION 已算，僅加 tagging，低成本）。
2. consumer .sync-state.json 加 pinned_version 欄（如 v1.48.5 或 latest）。
3. sync-pull target 改為 pinned tag commit（非 HEAD）；delta = base→pinned-tag。
4. bump：顯式 sync-pull --bump [version] 更新 pin 至 latest/指定版，重跑合併。
5. 預設 latest（=現行 HEAD 行為，向後相容）；pin 特定版為 opt-in。

### 爆炸半徑效果
壞 change push 到 HEAD（tag v1.48.6）不會碰到 pin 在 v1.48.5 的 consumer，直到其 --bump。consumer 刻意 bump、可查 CHANGELOG，壞版可拒升。「全 consumer 瞬時」→「選擇 bump 到壞版者」。

### 與 W1-087(C1 smoke test) 防禦縱深
C1=防壞 change 被 push（生產閘）；C3=壞 change 漏過時限爆炸半徑（分發閘）。縱深：罕見壞 push × 刻意更新 = 幾乎不會瞬時炸 consumer。

### 成本/取捨
- 傳播變慢：改進只在 bump 時到 pin 住的 consumer（刻意非自動）。單人框架若要快傳播是摩擦——latest pin=現行行為 緩解（pin 為 opt-in）。
- sync-pull 複雜度：target ref 解析（tag vs HEAD）+ bump 命令 + base 錨點互動，中等。
- live-edit 不受影響：pin 只管「消費端 pull 哪版」，push/熱改照常。

### 遷移（漸進、向後相容）
P1 push 開始 tagging（consumer 零改）→ P2 sync-pull 支援 pinned_version + --bump（預設 latest=現行）→ P3 consumer 按需 opt-in pin 特定版。

### 根決策關鍵refinement
W1-088(tagged-release+pin) 與 W1-090(GitHub Issues) 是**可分離的兩個升級**，成本剖面差很大：
- W1-088 只需 git tags + pin-aware pull——純 git、離線可、不離開現有 overlay 模型。**較輕、較易採納**。
- W1-090 需 GitHub Issues——平台特性、網路依賴。
故「根決策」非單一「升不升級正式上游」，而是兩個獨立定價的選擇。
