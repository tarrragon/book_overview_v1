## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-042 修復把 collision guard 寫在 `create.py` `_resolve_ticket_id_and_wave` 的 caller 層 while-loop，產生兩個結構問題（Phase 4 三視角收斂）：

1. **L1（linux）**：`get_next_seq` 契約是「下一個可用序號」，「可用」本應隱含「不撞號」。現狀是 get_next_seq 回可能撞號的候選值、caller 自行兜底——特例外洩給每個未來 caller，漏接即回歸。
2. **R1（thyme/Explore）**：`bulk_create.py:212-230` 以 `wave_seq_map[wave] = get_next_seq()` 為起點、後續 `+= 1`、最後 `save_ticket`（line 276 無條件覆寫）。完全沒有 guard，降級撞號時會連鎖覆寫多票（比單票 create 更嚴重）。

### 設計指引（linux caveat，重要）

`get_next_seq` 正常路徑是純掃描（glob + parse max），手上已有 `set(local_stems) | set(main_stems)`，`max+1` 對該集合天然不撞。真正會撞的只有「兩來源同時掃空但檔案其實存在」的降級情境。因此：

- guard 應緊貼降級偵測分支（與既有 stderr warning 同居），**不要**把正常路徑改成逐一 `.exists()` 探測
- 降級分支內用檔案系統探測推進至真正可用 seq，使回傳值保證可用
- create.py 的 while-loop 隨之移除（caller 端只剩 `seq = get_next_seq(...)`）

### bulk_create 稀疏佔用風險（WRAP P 預想）

即使 get_next_seq 保證起始值可用，bulk 的批次 `+= 1` 在稀疏佔用下仍可撞（例：起始 051 可用，但 052 已被既有票佔用）。修復需讓 bulk 的每個配號都有可用性保證（逐票走 helper 或批次內逐一檢查），測試必含稀疏佔用案例。

### 影響範圍

- `lib/ticket_builder.py`（get_next_seq 降級分支 guard）
- `commands/create.py`（移除 caller while-loop）
- `commands/bulk_create.py`（批次配號可用性保證）
- `tests/test_create_auto_seq_collision.py`（既有 8 案例需隨 guard 位置遷移調整 + 補稀疏佔用/bulk 案例 + T1 docstring 案例數修正 + .yaml/child_seq 缺口補測或標註 .md-only）

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 重構摘要

將 collision 可用性保證內聚至單一權威點，消除 W1-042 留下的 caller 層特例外洩。

| 變更 | 檔案 | 說明 |
|------|------|------|
| 抽 resolve_available_seq(version, wave, start_seq) | lib/ticket_builder.py | 從 start_seq 遞增至第一個 get_ticket_path(...).exists() 為 False 的序號。判定用 get_ticket_path——該函式對 .md / .yaml 皆探測，故同時覆蓋兩種落盤格式撞號（解 .yaml 缺口） |
| get_next_seq 降級分支內聚 guard | lib/ticket_builder.py | 正常路徑保持純掃描 max+1（linux caveat：不做逐一 .exists()）；僅當降級偵測（兩來源掃空且 main ref 回 None）成立時，以 resolve_available_seq 推進至可用 seq。回傳值內部保證可用，warning 文案標明推進後序號 |
| 移除 caller while-loop | commands/create.py | _resolve_ticket_id_and_wave auto-seq 分支只剩 seq = get_next_seq(...)，不再兜底（消除 L1） |
| bulk 批次配號可用性保證 | commands/bulk_create.py | 每個配號經 resolve_available_seq（起點 = get_next_seq 或前一 seq+1），消除稀疏佔用撞號（例：起始 051 可用但 052 已佔用）與降級連鎖覆寫（消除 R1）。dry_run 亦走 guard 使預演號與正式一致 |

### linux caveat 遵循

guard 緊貼 get_next_seq 降級偵測分支（與既有 stderr warning 同居），正常路徑（任一來源有效）的 max+1 對已掃描集合天然不撞，未改成逐一探測。bulk 因批次 += 1 在稀疏佔用下仍可撞，故每筆走 guard（與 caveat 不衝突——bulk 的撞號不源於降級而源於稀疏佔用）。

### R2（子任務 child_seq）顯性結論：不納入（標註豁免理由）

get_next_child_seq 取 max(children 欄位, 檔案系統掃描) + 1，且 _scan_child_files_max_seq 已對本地 glob 與 main ref 取聯集。其不存在「兩來源同時掃空誤回 1」的降級路徑——children 欄位為 SSOT、檔案掃描為 fallback，兩者皆空即代表真無子票（回 1 正確）。無 main-ref-only 降級回退風險，撞號面遠低於根任務 auto-seq。故本 ticket 不將 child_seq 納入 guard 改造，標註低風險豁免（與 ticket Problem Analysis R2 一致）。
