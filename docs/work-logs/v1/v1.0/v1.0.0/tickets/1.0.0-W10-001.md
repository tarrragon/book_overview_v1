## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

CC 2.1.178 新增 `Tool(param:value)` 權限語法，可在 settings.json permissions 中根據工具的輸入參數做比對（支援 `*` 萬用字元），例如 `Agent(model:opus)` 可阻擋使用 Opus model 的 Agent 派發。需評估此能力對框架派發控管的採用價值。

### 影響範圍

- `.claude/settings.json` permissions（project 層級）
- `.claude/settings.local.json` permissions（local 層級）
- 框架代理人派發體系（50+ agent types）

### 相關 Error Pattern

無直接相關。間接相關：PC-042（subagent tool call 限制）、PC-088（工具選擇偏誤）。

### 現狀盤點

**現有 permissions**：project settings `allow: [Edit, Write]`；local settings 65 條 allow + 1 條 ask。無任何 Agent 相關 permission 規則。

**現有派發控管**：Agent 定義 frontmatter model 欄位 / PM 決策樹 / dispatch-prompt-check hook / branch-verify hook / disallowedTools。

---

## 重現實驗結果

### 實驗方法

盤點現有 permissions 結構，評估 Tool(param:value) 可填補的控管缺口。

### 實驗執行

1. `python3` 解析三層 settings permissions → 無 Agent 規則
2. 列出全部 37 條參數化規則 → 全為 `Bash(cmd:*)` / `Skill(name)` / `WebFetch(domain:*)` 格式
3. 審查框架派發控管機制（agent 定義 / hooks / decision tree）

### 實驗發現

**已驗證事實**：
- 框架目前無 Agent permission 規則，所有 Agent 派發走 auto mode 預設行為
- `Tool(param:value)` 語法可用於：`Agent(model:opus)` / `Agent(subagent_type:fork)` / `Agent(isolation:worktree)` 等
- 現有控管主要靠 PM prompt 設計 + hooks，非 permission 層

**評估結論**：
- 可能用例 1：`Agent(model:opus)` 阻擋非必要 Opus 派發（成本控管）— 但框架已透過 agent 定義 frontmatter 控制 model
- 可能用例 2：限制 `isolation:worktree` 只在特定場景 — 但 ARCH-015 已在 pm-role 規則層控管
- 可能用例 3：`ask` 層級的 Agent 規則要求確認 — 增加互動摩擦，與 auto mode 目標衝突

---

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->

### 結論：延後採用（記錄能力，不立即落地）

**決策**：不採用。Tool(param:value) 權限語法是有價值的平台能力，但本專案目前無未被覆蓋的控管缺口需要它填補。

**理由**：
1. **既有控管已足夠**：Agent model 由 agent 定義 frontmatter 控制；worktree 由 ARCH-015 規則控管；派發決策由 PM 決策樹 + hooks 雙層防護
2. **Permission 層是最後手段**：Permission 規則是粗粒度的「允許/詢問/拒絕」，缺乏 hook 的「條件判斷 + 回饋訊息」能力。框架的 hooks 已提供更精細的控管
3. **無觸發場景**：目前無「代理人繞過 PM 使用不當 model」的 error pattern 或 incident

**未來觸發條件**：若出現以下情境，重新評估此決策：
- 代理人自行切換 model 繞過 agent 定義（建 incident ticket 追蹤）
- 需要在 CI/headless 環境強制 model 限制（hooks 不夠，需 permission 層）
- 成本控管需求（限制 Opus 用量）
