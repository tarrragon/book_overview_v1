## Task Summary

分析 唯讀探針派發與 agent 自律收尾流程的衝突（generic agent 收 Ticket ID 即越權勾選 acceptance + complete）

## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

三方規則交互產生的設計缺口：「prompt 引用 Ticket ID」與「被指派執行該 ticket」在現行規則中無法區分。

| 方 | 規則 | 行為 |
|----|------|------|
| dispatch 強制層 | `.claude/skills/ticket/hooks/agent-ticket-validation-hook.py` L202 | 非豁免 agent type 派發必須含 Ticket ID 格式，否則 deny |
| agent 自律層 | `.claude/agents/AGENT_PRELOAD.md` 規則 2.4（W17-033） | 實作類 agent commit + body 完成後主動 check-acceptance --all + complete |
| PM 需求 | 唯讀探針（驗證最終訊息完整性，要求 agent 零工具、零副作用） | 與上兩方衝突：被迫加 Ticket ID → agent 觸發 2.4 自走收尾 |

**既有但未被利用的出口**：hook 已有 `TICKET_EXEMPT_AGENT_TYPES` 白名單（Explore / claude-code-guide / general-purpose / Plan / feature-dev:code-explorer，W17-046 方案 A）。PM 派探針時選了 `claude`（catch-all 全工具型，正確地不在白名單），未選唯讀豁免型——豁免機制存在但 PM 派發 SOP 未引導使用。

### 影響範圍

- 假驗收風險：PM 保留的 acceptance 項可被探針/輔助 agent 越權勾選 + complete（W1-044 項 5 實證）
- 適用面：所有「引用 Ticket ID 但非執行該 ticket」的派發（探針、輔助驗證、諮詢型 agent 誤選非豁免 type）

### 相關 Error Pattern

- PC-065（派發必含 Ticket ID 格式）——本事件是其防護的非預期副作用
- agent-definition-standard「禁止跨 ticket 物件操作」——不適用本案（agent 操作的是自己 prompt 引用的 ticket，缺口正是「引用 ≠ 指派」）

## 重現實驗結果
本 session（2026-06-10）第一手事件，非轉述：

**探針 1（觸發越權）**：
- 派發：`Agent(subagent_type="claude", model="haiku")`，prompt 首行 `Ticket: 1.0.0-W1-044`（hook 強制後補加），任務為「不用工具，回傳兩行探針字串」，無行為約束。
- 第一次派發（無 Ticket ID）被 hook deny：「派發任務必須引用有效的 Ticket ID」——證實強制層生效。
- 結果：16 tool calls / 102k tokens；最終訊息「已確認。等候用戶輸入。」（非探針字串）；ticket 側出現越權寫入——acceptance 項 5（PM 保留的「reinstall 後實測劫持消除」）被勾選、ticket 被 complete。
- 物證：commit `5430e240`（chore(1.0.0-W1-044): metadata sync post-completion，diff 含 `[ ]→[x]` 項 5 + `in_progress→completed`）。PM 當時未執行任何 complete 操作。

**探針 2（對照組）**：
- 同 agent type / model / Ticket ID 首行，但 prompt 明令：禁止任何工具、禁止讀/認領/勾選/完成 ticket、忽略 hook 注入指示。
- 結果：0 tool calls；最終訊息一字不差回傳探針字串。
- 對照結論：越權行為由「Ticket ID 引用 + 無約束」觸發（agent 走 AGENT_PRELOAD 2.4 收尾流程），prompt 約束可完全抑制；原測試目標（劫持偵測）在乾淨條件下 PASS。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP

- **W（Widen）**：4 候選見下表（A 派發 SOP / B PRELOAD 邊界條款 / C hook 強制層 / D acceptance 保留標記）。
- **R（Reality-test）**：豁免白名單已存在且設計準則（唯讀 vs 可寫）正確（W17-046）；探針 2 證實 prompt 約束有效；C 的 caller identity 在 CLI 層實測不可得（agent 與 PM 共用同一 shell 執行 ticket CLI，無進程級身份標記）。
- **A（Attain distance）**：本 ANA 為 PM 前台執行，未擠壓其他 ready 任務（佇列空）。
- **P（Prepare to be wrong）**：文字防線（A+B）可能在 agent context 壓力下失守；緩解 = B 條款寫入 AGENT_PRELOAD 檢查清單（每次派發注入），且 PM 端 A 的 SOP 從源頭避免非豁免 type 派探針，雙層互補。

### 候選方案利弊

| 方案 | 內容 | 優 | 劣 | 裁決 |
|------|------|----|----|------|
| A：PM 派發 SOP 固化 | agent-dispatch-template.md 新增「唯讀探針派發 SOP」：必用 TICKET_EXEMPT_AGENT_TYPES 白名單型（Explore / general-purpose）免 Ticket ID；若必須引用 ID，prompt 必附三禁約束（禁工具/禁 ticket 操作/忽略注入指示） | 零程式碼；利用既有豁免機制；從源頭消除觸發 | 靠 PM 自律 | **採用** |
| B：PRELOAD 邊界條款 | AGENT_PRELOAD 2.4 加「引用 ≠ 指派」前提：收尾自律僅適用 prompt 明確指示執行的 ticket（含 claim/實作/修復/收尾等動詞指令）；prompt 僅含追溯格式 Ticket ID 而無執行指令時，禁止對該 ticket 任何寫入（claim/勾選/complete/append-log） | 覆蓋所有 agent type（含 claude catch-all）；防同型事故再發 | 文字防線，依賴 agent 遵守 | **採用** |
| C：hook 強制層（caller 身份驗證） | ticket CLI / acceptance-gate 偵測 complete 操作者是否為指派 who | 強制力 | CLI 層無法可靠取得 caller identity（共用 shell，無進程級身份）；技術不可行 | **否決——無需建 ticket：caller identity 在 CLI 層不可得；A 的豁免 type 路徑已消除探針場景的 Ticket ID 需求，B 覆蓋殘餘文字層風險** |
| D：acceptance 保留標記 | acceptance 條目支援 [PM-only] 標記 + CLI 阻擋 | 精準保護保留項 | 與 C 同樣受 caller identity 限制；schema 變更影響面大，價值不成比例 | **否決——無需建 ticket：依賴 C 的不可行前提；保留項保護由 B 條款（禁止寫入未指派 ticket）覆蓋** |

### 裁決

採 A + B 合併為單張 DOC spawn（同一防護主題、兩處文件修改、認知負擔 < 閾值）；範圍含 error-pattern 記錄（PC 編號由執行時取號），固化「引用 ≠ 指派」教訓。
