### Phase 1 設計決策（待 parallel-evaluation 審查）

#### 方案 A：append-log 條件式 auto-stage 觸發條件評估

git 語義前提：`git checkout -- <file>` 用 **index（staged）版本** 覆蓋 working tree。故 append-log 後若已 `git add`，checkout 還原時 index 版本即新內容，body 不丟。方案 A 對最常見的 `git checkout --` 有效；`reset --hard`（清 index）/`stash` 缺口由方案 C 覆蓋。

| 選項 | 觸發條件 | 防護覆蓋 | 副作用 | 評估 |
|------|---------|---------|--------|------|
| 1 無條件 | 每次 append-log 都 `git add` ticket md | 最大 | 改變 untracked 檔狀態 | 否 |
| 2 僅 tracked（推薦） | ticket md 已 tracked 才 stage | 高（create 已 commit，ticket md 恆 tracked） | 精確路徑 add，僅 stage 該 md，副作用小 | **採** |
| 3 tracked + 無其他 staged（ANA 原建議） | 再加「index 無其他 staged」 | 低（執行中 PM/agent 常有 staged → 不防護） | 防護覆蓋驟降 | 否 |

**推薦選項 2 理由**：`_auto_stage_git_add` 為精確路徑 add（只 stage 該 ticket md，無 ./、-A、--all），副作用本就侷限；選項 3 的「無其他 staged」條件會讓執行中（PM/agent 已 stage 任何檔案時）完全失去防護，得不償失。worktree 並行 conflict（feedback_auto_stage_worktree_conflict）發生於 merge 階段，非 append-log auto-stage 本身；且方案 A 在單一 working tree 內 stage 不跨 worktree。

**實作要點**：複用 `_auto_stage_git_add`（抽到共用 lib 避免 track_acceptance → lifecycle 循環依賴）+ 仿 `_auto_stage_completion_files` 的 graceful degrade（git add 失敗 stderr 警告、不中斷 append-log）。

#### 方案 C：git 還原攔截 PreToolUse hook 規格

| 項目 | 規格 |
|------|------|
| Hook 類型 | PreToolUse，matcher `Bash` |
| 偵測命令 | `git checkout`（含 `-- <path>`）/ `git restore` / `git reset --hard` / `git stash`（push/save） |
| 觸發條件 | 工作區存在 modified（未 commit）的 ticket md（`docs/work-logs/**/tickets/*.md`） |
| 行為 | **警告（非阻擋）**，stdout 列出有未 commit 變更的 ticket md + 建議先 commit；exit 0 放行 |
| 誤報防護 | 僅當「git 還原命令」AND「有 modified ticket md」同時成立才警告 |

#### 驗收標準（Phase 1 → Phase 2 RED 測試依據）

1. append-log 對已 tracked ticket md 寫入後，該 md 進入 git staged 狀態
2. append-log auto-stage 失敗（git 不可用）時 append-log 仍成功（exit 0）+ stderr 警告
3. PreToolUse hook 偵測 git 還原命令 + 存在 modified ticket md → 警告且放行（exit 0）
4. PreToolUse hook 在無 modified ticket md 時不警告（無誤報）
5. 行為變更過 ticket-skill-sync-check（append-log 副作用變更）

### Phase 1 多視角審查結論 + 方向轉變（auto-stage → auto-commit）

原 Phase 1 設計（auto-stage 選項 2 + 方案 C hook）經情境 F 三視角審查（--skip-basil，標的為半結構化設計決策）：

| 視角 | 判定 | 核心 |
|------|------|------|
| general-purpose（結論） | 4/4 通過 | 推翻 ANA 選項 3→2 證據充分；auto-stage 路線可行 |
| thyme（實作可行性） | 可進 Phase 2 | git 語義實機驗證確認（checkout-- 用 index 還原；reset --hard 清 index；stash 含 staged 可 pop 救回）；方案 C hook 解析有誤報盲點需修 |
| linux（品味/複雜度） | **需修正** | **F1 第四選項 auto-commit**：body 進 commit 歷史則三種還原全失效，方案 C 可刪；批評 auto-stage+warning 為「兩個半解」，warning 對自動序列（不讀 stdout）防護近零 |

### WRAP 收斂 + 用戶決策

linux 洞察切中要害：auto-stage 只實防 checkout（3 種還原之一），reset --hard/stash 靠方案 C 警告，但自動序列（sync-push/worktree merge）不讀 stdout 等於沒接住。reality-test：auto-commit 每 ticket 增 4-6 個 chore commit（碎 commit 代價）。

**用戶決策（2026-06-08）：採 auto-commit 根因解**。理由：三種還原全實防 + 刪除方案 C 複雜度；碎 commit 對 ticket md（chore 類）可接受。此決策二次推翻 ANA（原否決方案 B auto-commit），由用戶 active consent 確認。

### auto-commit 專屬設計考量（Phase 2 RED 須涵蓋）

1. **空 commit 防護**：body 無實際變更時 `git commit` 應 graceful skip（不產生空 commit、不報錯）
2. **graceful degrade**：git 不可用 / 非 git repo / index.lock 競爭（memory feedback_git_index_lock）→ commit 失敗時 append-log 仍 exit 0 + stderr 警告，body 保留 working tree（下次操作或手動 commit 持久化）
3. **pre-commit hook 開銷**：ticket md 非 JS，husky lint-staged 無匹配（已實證 "could not find staged files"）；append-log 頻繁觸發 pre-commit 是否用 `--no-verify` 待 Phase 3a 實作策略權衡（傾向不繞過，接受輕微開銷以維持 hook 把關原則）
4. **worktree 場景**：agent 在 worktree append-log auto-commit → commit 進 worktree 分支，complete merge 帶回 main。實際比現狀更安全（現狀 worktree 未 commit body merge 時遺失，正是本 ticket 要解決的）

### 更新驗收標準（Phase 2 RED 依據）

1. append-log 寫入後產生 commit，git log 可見該 ticket md commit（body 進歷史）
2. auto-commit 後執行 `git checkout HEAD~ -- <md>` 再 checkout 回，或 `reset --hard` 後 body 仍可從 commit 取回（驗證根因解生效）
3. body 無變更時 append-log 不產生空 commit（graceful skip）
4. git 不可用 / index.lock 時 append-log 仍成功 exit 0 + stderr 警告
5. commit message 格式 `chore(<id>): append-log <section>`
6. 行為變更過 ticket-skill-sync-check
