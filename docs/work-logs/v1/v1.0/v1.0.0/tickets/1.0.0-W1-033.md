## Problem Analysis
<!-- Schema[IMP/Problem Analysis]: 選填（小型 IMP 可留 placeholder；大型 IMP 建議填寫）（.claude/pm-rules/ticket-body-schema.md） -->

### 問題根因

hook 日誌目錄由 `hook_utils/hook_logging.py` 的 `setup_hook_logging` / `save_check_log`
解析，路徑為 `get_project_root() / .claude / hook-logs / <hook_name>`。
`get_project_root()`（`hook_utils/hook_base.py`）優先讀取 `CLAUDE_PROJECT_DIR`，
其次 `git rev-parse --show-toplevel`。

pytest 執行時 `CLAUDE_PROJECT_DIR` 未設定，故 fallback 至 git toplevel = 真實 repo。
測試（如 `test_uv_tool_staleness_check_hook.py::test_hook_exits_0_on_global_failure`
以 `monkeypatch` 讓 `get_project_root` 拋 RuntimeError、T11 以 IOError side_effect）
觸發 hook 主程式 `logger.critical(...)` + `run_hook_safely` 寫入 traceback。

關鍵點：測試 monkeypatch 的是 **hook 模組自身** import 的 `get_project_root`，
但 `setup_hook_logging` 用的是 **`hook_logging` 模組** 內綁定的 `get_project_root`，
不受 hook 模組 mock 影響 → 日誌仍寫入真實 `.claude/hook-logs/`，含 ERROR/CRITICAL/
Traceback，污染 hook system health check 信號（quality-baseline 規則 4）。

### 影響範圍

- 直接污染：`.claude/hook-logs/<hook_name>/` 下產生測試噪音 .log（含 traceback）。
- 受影響測試：所有經 `setup_hook_logging` / `save_check_log` 且觸發 logger 寫入的 hook 測試；
  最明顯為 `test_uv_tool_staleness_check_hook.py`。
- 修復檔案：僅 `.claude/hooks/tests/conftest.py`（conftest 層 autouse fixture）。

### 相關 Error Pattern

無直接對應 PC；屬測試隔離缺口。與 IMP-064（local re-import 遮蔽 module-level mock）
同家族（mock 綁定層級錯位）：本案是「測試 mock hook 模組層 get_project_root，
但日誌走 hook_logging 模組層綁定」導致 mock 未覆蓋日誌路徑解析。

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### 修復摘要

於 `.claude/hooks/tests/conftest.py` 新增 **autouse function-scoped fixture**
`isolate_hook_logs`，僅 monkeypatch `hook_utils.hook_logging` 模組層綁定的
`get_project_root`，將日誌根目錄導向每個測試專屬的 `tmp_path / "hook_log_isolation"`。

```python
@pytest.fixture(autouse=True)
def isolate_hook_logs(tmp_path, monkeypatch):
    import hook_utils.hook_logging as _hl
    log_root = tmp_path / "hook_log_isolation"
    log_root.mkdir(parents=True, exist_ok=True)
    _original = _hl.get_project_root
    _production_root = Path(__file__).resolve().parents[3]
    def _isolated():
        resolved = _original()
        try:
            is_prod = resolved.resolve() == _production_root
        except OSError:
            is_prod = False
        return log_root if is_prod else resolved
    monkeypatch.setattr(_hl, "get_project_root", _isolated)
    yield log_root
```

### 設計決策（為何 conditional 而非無條件重導向）

| 方案 | 結果 |
|------|------|
| (A) autouse setenv `CLAUDE_PROJECT_DIR=tmp` | 全域改變 `get_project_root`，誤傷 15 個依賴真實 repo / 相對路徑 / `Path.cwd` glob 的測試（路徑分類、handoff cache、cross-repo） |
| (B) 無條件 patch `hook_logging.get_project_root → tmp` | 誤傷 3 個 `test_hook_utils` fallback 測試（delenv + chdir 驗證 CLAUDE.md 搜尋 / cwd fallback 的日誌落點） |
| (C) **僅當解析結果 == production repo 時才重導向**（採用） | 精準隔離污染路徑；測試已自行隔離（env / chdir 到 tmp）時不介入，既有日誌路徑斷言全保留 |

採方案 C：唯有「會污染 production hook-logs」的解析結果才改寫，其餘沿用測試自身設定。

### 隔離 vs 抑制

fix 是**隔離**（logs 改寫到 tmp_path）非**抑制**：`setup_hook_logging` 仍正常建立
FileHandler 並寫入，測試對日誌寫入行為的驗證能力保留（實證見 Test Results）。

### 不在本 ticket 範圍

- 未修改任何 production hook 行為 / `hook_utils/` 原始碼（約束遵守）。
- 未改動 `lib/`（conftest 層即可解決，無需動 log path 解析原始碼）。
