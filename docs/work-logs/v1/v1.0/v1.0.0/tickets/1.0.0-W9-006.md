## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題本質
ticket 工具內部 scanner 與 loader 對「同版本目錄結構」解析不一致：版本掃描器找得到扁平 docs/work-logs/v{version}/tickets/ 並回報該版本存在，但 get_tickets_dir（W14-052 後）對可解析 major.minor 的版本一律回三層階層路徑，且不檢查目錄存在性。兩者對不上 → track list 報「0/0 完成、無 Tickets」但檔案實際存在於扁平位置。

### 觸發條件
專案以扁平結構持有既有 ticket（如 wotel Windows 專案 docs/work-logs/v0.1.0/tickets/，3 個 ticket）。本專案（book_overview_v1）用三層結構故不觸發。

### 約束
W14-052（commit 3e1fbf1f）刻意移除 get_tickets_dir 的存在性檢查，使新建 ticket 一律三層，避免「未存在主版本 fallback 至 flat」造成跨專案殘留。任何修復不得反轉此不變式——新版本仍須建在三層。

### 分析方法
讀 W14-052 commit 實際 diff 確認其被移除的舊行為與真實意圖（reality-test），對照 issue #1 問題4 的需求差異（讀既有 flat vs 新建三層），於 Solution 以 WRAP 擴增 A/B/C 選項並評估。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### Solution（WRAP 決策：扁平 vs 階層 ticket 結構對齊）

### 問題
scanner 找得到扁平 docs/work-logs/v0.1.0/tickets/ 並回報版本，但 get_tickets_dir（W14-052 後）對可解析版本一律回三層 docs/work-logs/v0/v0.1/v0.1.0/tickets/ 且不檢查存在性 → track list 報 0 張但檔案存在（issue #1 問題4，於 wotel Windows 專案實測）。

### W（Widen options）
- A 原始：get_tickets_dir hierarchical 不存在時 fallback flat（直接反轉 W14-052，新版本會建在 flat 造殘留）。
- A-refined：hierarchical 存在用之；否則 flat 實際存在才回 flat（讀既有）；否則 default hierarchical（新版本仍三層）。
- B：scanner 只認 hierarchical + 新增官方「flat→hierarchical 同版本就地遷移」命令（既有 migrate 改 ID、version-shift 換版號皆無法同版本就地搬）。
- C：拆分 read（雙結構解析）vs create（一律三層）兩函式，所有 reader 改呼叫 resolver。

### R（Reality-test）
讀 W14-052 commit 3e1fbf1f 實際 diff：被移除的舊邏輯在 hierarchical 各層皆不存在時 fallback flat，使「新版本」建在 flat 造跨專案殘留。W14-052 真正關切是「新 ticket 建在三層」，非「禁止讀既有 flat」。故 A-refined 不違反 W14-052 不變式——其只在 flat 實際存在時讀 flat，新版本（flat 不存在）仍 default 三層。既有 4 個 W14-052 不變式測試（v2/v10.3.1/v0.18/v-prefix 皆 flat 不存在）在 A-refined 下全綠（已實證）。

### A（Attain distance）/ P（Prepare to be wrong）
依價值/相容性而非實作便利選擇：A-refined 零遷移摩擦 + 向後相容 + 保不變式，優於 B（需新命令 + 強制既有專案遷移）與 C（侵入所有 reader）。可能錯點：混合結構專案（v0.1.0 flat + v0.2.0 三層）並存——可接受（各版本自洽），未來如需統一以 B 的遷移命令補（非本問題阻斷項，不建追蹤 ticket）。

### 決策：Option A-refined
落地為 child IMP 1.0.0-W9-006.1（已 complete）。
