## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-016 body（Problem Analysis / 重現實驗結果 / Solution WRAP 四方案）遺失的直接原因**不是 ticket CLI bug**，而是 **git working-tree 還原**（H5）：

- ticket CLI 的 `create` 將 ticket 落盤為 `status: pending` + placeholder body 並 commit（ebd2f597）。
- 後續 `claim`（→ in_progress）、`append-log`（填 body）只修改 **working tree，未 commit**。
- 某個 git 還原操作（`git checkout -- <file>` / `git restore` / `git reset --hard` / `git stash`）將 ticket md 還原回 create commit 版本 → 同時造成 `status` 回 pending + body 回 placeholder。

### 影響範圍

- 機制層：所有「claim/append-log 後未即時 commit」的 ticket md 都暴露於此風險，非 W1-016 個案。
- 程式碼層：ticket CLI 內部路徑（claim/append-log/release/complete/cascade）**無 bug**（見「重現實驗結果」），毋需修改 CLI 邏輯本身。
- 高風險情境：sync-push/sync-pull、worktree merge（checkout --theirs/--ours）、PM 任務切換 stash、處理無關變更時的 reset/checkout。

### 相關 Error Pattern

- PC-019（worktree 派發前先 commit）— 同源：ticket md working-tree 暫態被 git 操作覆蓋。
- feedback_auto_stage_worktree_conflict — auto-stage main ticket md + worktree 並行 append-log 同檔衝突。
- feedback_worktree_commit_before_dispatch — 修改 ticket 後未 commit 就派發，stash 致狀態遺失。

## 重現實驗結果
### 實驗方法

兩階段 Reality Test（PC-063）：(A) 真實 CLI 序列排除 CLI 內部；(B) git commit 物證確認 H5。

### 實驗執行

**實驗 A — 真實 CLI 序列（獨立 process，隔離測試 ticket 1.0.0-W99-001）**

建測試 ticket → claim → append-log 寫入 marker `REPRO_MARKER` → release（in_progress→pending）→ re-claim，各階段 `grep -c REPRO_MARKER`：

| 階段 | marker 數 |
|------|-----------|
| create 後（placeholder） | 0 |
| append-log 後 | 1 |
| release（→pending）後 | 1 |
| re-claim 後 | 1 |

**實驗 B — W1-016 git commit 物證**

- create commit `ebd2f597`：`status: pending` + Solution = `<!-- To be filled by executing agent -->`（placeholder）
- complete commit `ca6789c3`：`status: completed` + Solution 完整
- git 歷史僅 2 commit（create + complete）；中間 claim/append-log/遺失/重填全在 working tree 暫態，無 commit 痕跡。

**程式碼層交叉驗證**（排除 CLI 內部）：

- `save_ticket` 用 `ticket.pop("_body","")`，但所有 caller 的 ticket dict 均來自 `load_ticket`（parser.py:246 設 `_body`）或 `list_tickets`（走 load_ticket），故 `_body` 不為空。
- `_cascade_unblock_children` 只對 `STATUS_BLOCKED` 解鎖（lifecycle.py:1873），不動 in_progress。
- 快取失效 key（save 用 `resolve_ticket_path`）== load 快取 key（`get_ticket_path`），因 `resolve_ticket_path` 回傳 `ticket["_path"]`（= load 設定值），失效正確（parser.py:364 / ticket_ops.py:130）。

### 實驗發現

**已驗證事實**：

1. 當前 ticket CLI 的 claim/append-log/release/re-claim 序列**不丟 body**（實驗 A，marker 全程保持）。
2. W1-016 的 create commit 狀態（pending + placeholder）與 W1-017 描述的「遺失後狀態」**逐字吻合**（實驗 B）。
3. CLI 內部所有 body-mutating 路徑在 code 層正確保留 `_body`（交叉驗證）。

**結論**：根因 = git working-tree 還原（H5），信心 0.85。CLI 內部 body 遺失假設（H1 快取污染 / H2 cascade 清空 / H4 快取失效 key 不匹配）**全部否證**。

**殘餘不確定性**：具體哪一個 git 操作是兇手不可考（無 reflog 證據），但機制確定——未 commit 的 ticket body 必然暴露於 git 還原風險。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 根因定性

非 CLI bug，而是「ticket body 寫入後停留於易失 working tree（未 commit），被 git 還原操作覆蓋」的協作時機問題。修復方向 = 縮短 body 暴露於 working tree 的時間窗 + 攔截危險 git 操作。

### WRAP 防護方案評估（Widen）

| 方案 | 機制 | 防護範圍 | 副作用/成本 | 採用 |
|------|------|---------|------------|------|
| A: append-log 自動 stage（git add ticket md） | 寫入後立即 `git add` | 防 `git checkout -- <file>`（最常見還原） | stage 範圍與 PM commit 邊界耦合（feedback_parallel_agent_commit_race）；不防 `reset --hard`/`stash` | 採（主，輕量） |
| B: append-log 自動 commit | 每次寫入即 commit | 完整持久 | 碎 commit 污染歷史；過重 | 否決 |
| C: Hook 攔截 git 還原操作 | PreToolUse 偵測 checkout/reset/stash + uncommitted ticket body → 警告 | 防 `reset --hard`/`stash`/`checkout` | hook 偵測 git 命令需精準匹配，誤報風險 | 採（輔，覆蓋 A 缺口） |
| D: SOP/規則固化 | PM git 還原前先 commit/stash 保護 ticket md | 全範圍（人為自律） | 無強制力 | 採（補充，PC-019 延伸） |

### 推薦方案：A + C + D 分層防護

- A 縮短時間窗（自動 stage，防最常見的 `git checkout --`）。
- C 攔截 A 防不住的 `reset --hard`/`stash`（hook 警告）。
- D 固化為規則（PC-019 延伸 + ai-communication memory），人為最後防線。

### Attain distance（機會成本）

v1.0.0 W6 發佈在即；本 bug 為 P2 framework 改善，**不阻塞發佈**。spawn ticket 設 pending 排後續 wave。

### Prepare to be wrong

最可能失敗點：方案 A 的 auto-stage 副作用（與 PM/並行 agent commit 邊界衝突）。對應防護：spawn IMP 須含 Phase 1 評估 auto-stage 是否需限縮為「僅當 ticket md 已 tracked 且無其他 staged 變更」的條件式 stage，或改採方案 C 為主、A 為可選。
