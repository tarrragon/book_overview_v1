### D1 資訊傳遞協議

**核心主張**：嵌套派發中每一層 agent 的資訊輸入與輸出必須經由 ticket 系統持久化，prompt（下行）與 final message（上行）僅作為「指向 ticket 的指標」，不作為資訊主通道。

#### 層級定義

| 概念 | 定義 |
|------|------|
| dispatch 層級 | PM 為 L0；PM 直接派發的 agent 為 L1；L1 嵌套派發的 agent 為 L2；依此類推 |
| ticket 深度 | ticket ID 的 `.` 分隔層數（`W1-056` = depth 1；`W1-056.5` = depth 2；`W1-056.5.1` = depth 3） |
| 映射規則 | dispatch 層級 N 的 agent 操作 ticket depth >= N+1 的 ticket（L0/PM 操作 depth 1+；L1 agent 操作 depth 2+；以此類推） |

#### 每層強制動作（規則層）

每一層 agent（L1 至 Lmax）在執行任務時，必須依序完成以下三個 ticket 動作。此規則層要求寫入 AGENT_PRELOAD 或嵌套派發專屬規則檔。

| 階段 | 動作 | 載體 | 現有機制 |
|------|------|------|---------|
| 進入 | `ticket track claim <id>` | frontmatter `started_at` / `who.current` | 已存在（AGENT_PRELOAD 2.4 前提一） |
| 執行中 | `ticket track append-log <id> --section "<章節>"` | body 章節（Problem Analysis / Solution / Test Results） | 已存在（AGENT_PRELOAD 2.2/2.3） |
| 結束 | (a) `ticket track complete <id>` 或 (b) NeedsContext 章節 + Exit Status 回報 | frontmatter `completed_at` + body | 已存在（AGENT_PRELOAD 2.4 + ticket-body-schema NeedsContext/Exit Status） |

**Why**：prompt 下行與 final message 上行的資訊是 session-scoped（session 結束即消失），且嵌套層間的 final message 不一定能完整傳遞到 L0（中間層可能摘要或遺漏）。ticket 的 append-log 是 auto-commit 持久化（W7-001），跨 session 可查詢，是唯一可審計通道。

**Consequence**：若允許 prompt/final message 為唯一通道，L0（PM）在嵌套執行結束後無法追蹤 L2+ agent 做了什麼——只看到 L1 的回報摘要，L2 的決策痕跡消失（資訊黑洞）。

**Action**：規則層寫入 AGENT_PRELOAD 新增章節「嵌套派發資訊協議」（D1），內容為上方三階段表。

#### 禁止 prompt/final message 為唯一通道

| 禁止模式 | 正確做法 |
|---------|---------|
| L1 agent 在 prompt 內嵌入所有 context 給 L2，不建 child ticket | L1 先 `ticket track create --parent <自身 ticket>` 建 child，context 寫入 child 的 Problem Analysis |
| L2 agent 結束後僅回傳 final message 給 L1，不寫 ticket | L2 必須 append-log Solution + complete（或 NeedsContext 停止） |
| L1 agent 以 final message 摘要 L2 結果回報 L0 | L1 在自身 ticket append-log 引用 L2 ticket ID 與結論摘要（`L2 ticket <id> 結論：...`） |

#### 強制層分工（待 .4 實證）

| 機制 | 層級 | 生效前提 |
|------|------|---------|
| agent-ticket-validation-hook（PreToolUse:Agent） | 要求 Agent 呼叫的 prompt 含 Ticket ID | 待假設 A 驗證：hook 是否在嵌套層觸發 |
| acceptance-gate-hook（complete 前 body 驗證） | 阻擋空殼 complete | 待假設 A 驗證 |
| AGENT_PRELOAD 規則 2.4 自律 | claim/append-log/complete 自律鏈 | 不依賴 hook，規則層即可生效 |

**結論**：規則層（AGENT_PRELOAD 三階段表 + 禁止模式表）不依賴 hook 即可部署；強制層（hook）依賴 .4 實驗結論決定是否自動覆蓋嵌套層。若假設 A 為否（hook 不觸發嵌套層），則強制層需替代設計（如 L1 agent 自律檢查 L2 的 ticket body 是否為空再 complete 自身）。

### D2 descend/ascend 決策表

**核心主張**：嵌套層 agent 不自行判斷「是否往下派發」——決策由 ticket 結構（認知負擔量化）與深度約束（平台 5 層上限）共同決定，寫入可判定條件表。

#### descend 條件（往下建 child 並嵌套派發）

agent 在執行 ticket 過程中，若以下**全部**條件同時成立，**應該** descend（建 child ticket 並嵌套派發 sub-agent）：

| # | 條件 | 量化基礎 | 來源 |
|---|------|---------|------|
| D-1 | 當前任務可拆分為 2+ 個獨立子任務且各自聚焦單一職責 | 功能職責 > 2 須拆分 | cognitive-load.md 速查三閾值 |
| D-2 | 子任務之間無檔案重疊（修改的檔案集合無交集） | 並行安全檢查 | parallel-dispatch.md |
| D-3 | 當前 ticket depth + 1 <= 最大允許深度（見 D3 機制） | 深度約束 | D3 層級自覺 |
| D-4 | 子任務的認知負擔指數各自 <= 10 | 認知負擔量化 | cognitive-load.md 閾值表 |
| D-5 | 子任務不涉及需要上層決策的敏感操作（架構決策、規則修改、用戶選擇） | 決策權歸屬 | pm-role.md 核心原則 |

**任一條件不成立，禁止 descend**——必須在當前層級完成，或 ascend 上報。

#### ascend 條件（停止執行、上報上一層）

agent 在執行 ticket 過程中，遇到以下**任一**條件，**必須** ascend（停止執行、寫 NeedsContext / Exit Status 回報上層）：

| # | 條件 | 既有載體 | 來源 |
|---|------|---------|------|
| A-1 | 需要上層才有的資訊（API 簽名、設計決策、規格缺口） | NeedsContext 章節（缺失項 / 觸發位置 / 影響 / 建議補料 / 重派成本） | ticket-body-schema.md |
| A-2 | 任務超出自身 agent 定義的允許產出範圍 | Exit Status `status: blocked` + `reason` | agent-definition-standard.md 區塊 1/2 |
| A-3 | 需要用戶決策（敏感規則、架構方向、影響互動模式） | Exit Status `status: needs_context` | pm-role.md 核心原則 |
| A-4 | 剩餘 tool call 預算不足以完成子任務的 claim/append-log/complete 三階段 | Exit Status `status: partial_success` | PC-042（~20 tool call 限制） |
| A-5 | 子任務的認知負擔指數 > 10 且無法進一步拆分 | NeedsContext（具體載明指數與構成） | cognitive-load.md |
| A-6 | 嵌套層已達最大深度，無法再 descend | Exit Status `status: blocked` + `reason: 深度上限` | D3 層級自覺 |

**載體選擇規則**：

| 停止原因 | 寫入 NeedsContext | 寫入 Exit Status | 兩者皆需 |
|---------|-----------------|-----------------|---------|
| 資訊缺口（A-1, A-5） | 是（詳列缺什麼） | 是（status=needs_context） | 是 |
| 職責/權限阻擋（A-2, A-3, A-6） | 否 | 是（status=blocked） | 否 |
| 資源耗盡（A-4） | 否 | 是（status=partial_success，列已完成 acceptance） | 否 |

### D3 層級自覺機制

**核心主張**：每層 agent 從 ticket ID 的深度推算自身 dispatch 層級與剩餘可用深度，無需上層在 prompt 中傳遞層級資訊。

#### ticket depth 到 dispatch 層級的映射

```
ticket_depth = ticket_id.count('.') + 1
  W1-056       → depth 1 → L0（PM 操作）
  W1-056.5     → depth 2 → L1（PM 直接派發 agent 操作）
  W1-056.5.1   → depth 3 → L2（L1 嵌套派發 agent 操作）
  W1-056.5.1.1 → depth 4 → L3
  ...
```

**agent 自身層級推算**：agent claim ticket 後，讀取 ticket ID 的 depth，得知自己處於 `L(depth - 1)` 層。

#### 深度上限約束

| 參數 | 值 | 來源 |
|------|-----|------|
| 平台最大嵌套深度 | 5 層（L0-L4） | CC 2.1.172 release notes |
| 框架最大 ticket depth | 4 層（depth 1-4，對應 L0-L3） | 保守設計：保留 1 層作為平台安全邊距 |
| 每層 agent 可用的剩餘嵌套次數 | `max_remaining = 4 - ticket_depth` | 推算公式 |

**Why**：平台 5 層上限是硬限制；框架層保留 1 層安全邊距（depth 5 = L4）避免 edge case（如平台內部消耗 1 層、或未來上限調整）。ticket depth 4（L3）為框架允許的最深操作層。

**Consequence**：若不設框架層上限，agent 可能在 depth 4 嘗試 descend 到 depth 5（L4），觸及平台硬限制後失敗，浪費 tool call 預算且無法優雅回退。

**Action**：在 D2 descend 條件 D-3 中檢查 `ticket_depth < 4`（當前 depth + 1 <= 4）。

#### 自覺實作方式

agent claim ticket 後，執行以下自檢（寫入 AGENT_PRELOAD 規則層）：

```
1. 讀取 ticket ID → 計算 depth = id.count('.') + 1
2. 推算 dispatch_level = depth - 1
3. 推算 max_remaining_descend = 4 - depth
4. 若 max_remaining_descend <= 0：本層禁止 descend，遇需拆分場景必須 ascend
5. 若 max_remaining_descend > 0：可依 D2 條件判斷是否 descend
```

**不依賴 prompt 傳遞層級的設計理由**：

| 方案 | 缺點 |
|------|------|
| 上層 prompt 傳遞 `current_level=N` | 依賴 prompt 正確性；嵌套層越深越容易遺漏；與 D1 原則（禁止 prompt 為唯一通道）衝突 |
| ticket ID depth 推算 | 零額外傳遞成本；ticket ID 是 claim 的必要輸入，不可能遺漏；世界平面事實（tool-output-trust-rules 規則 5），不可被 prompt 污染 |

**選定方案**：ticket ID depth 推算。agent 無需上層告知層級，自行從 ticket ID 得知。

#### 嵌套層 agent 的 tool 集約束（待 .4 實證決定邊界）

| 層級 | 允許工具集（初步設計） | 理由 |
|------|---------------------|------|
| L1（depth 2）| 全工具（含 Agent） | 與現行單層派發相同 |
| L2（depth 3）| 全工具（含 Agent），但 descend 前必通過 D2 條件 | 保留拆分能力但受條件約束 |
| L3（depth 4）| 禁止 Agent 工具（max_remaining = 0） | 框架深度上限，無法再 descend |

**待 .4 實證的調整**：若假設 A 證實 hook 不觸發嵌套層，L2 的 Agent 工具使用需額外規則層約束（D2 條件檢查靠 agent 自律而非 hook 攔截）。

#### D1 資訊傳遞協議（v2）

**核心主張**（不變）：嵌套派發中每一層 agent 的資訊輸入與輸出必須經由 ticket 系統持久化，prompt（下行）與 final message（上行）僅作為「指向 ticket 的指標」，不作為資訊主通道。

#### D2 descend/ascend 決策表（v2）

**核心主張**（修訂）：嵌套層 agent 不自行判斷「是否往下派發」——決策由可判定條件表驅動。**ascend 優先於 descend**：條件判斷的預設路徑為「在當前層完成」，descend 需全部條件通過才啟動。

##### descend 條件（全部 AND 才 descend）

| # | 條件 | 判定方式 | 來源 |
|---|------|---------|------|
| D-1 | 當前任務可拆分為 2+ 個獨立子任務且各自聚焦單一職責 | 機械計數：職責數 > 2 | cognitive-load.md |
| D-2 | 子任務之間無檔案重疊（並行 descend 的附加條件；序列 descend 不要求） | `where.files` 交集檢查 | parallel-dispatch.md |
| D-3 | `can_descend()` = true（見 D3 機制） | parent_id 鏈深度 < MAX_TICKET_DEPTH | D3 層級自覺 |
| D-4 | 子任務複雜度可控 | 機械計數：各子任務修改檔案 <= 5 且 acceptance 條目 <= 7 | cognitive-load.md 閾值表 |
| D-5 | 子任務不涉及需要上層決策的敏感操作 | 敏感操作清單：架構決策、規則修改、用戶選擇、`.claude/` 寫入 | pm-role.md |

**D-2 修訂說明**（P1 #1 回應）：v1 的 D-2 要求所有 descend 場景「檔案無重疊」，但序列 descend（一個完成後再派下一個）無此需求。v2 將 D-2 限縮為**並行 descend 的附加條件**——序列 descend 只需 D-1/D-3/D-4/D-5 四條通過。

**D-4 修訂說明**（P1 #2 回應）：v1 的 D-4 要求「認知負擔指數 <= 10」，但指數自評在 agent 端不可靠。v2 改為機械可數的代理指標：修改檔案 <= 5 + acceptance 條目 <= 7。

**無出口修復**（P1 #1 回應）：v1 中 「認知負擔 > 10 + 可拆分 + 檔案重疊」三向組合導致 descend 禁止、本層無法完成、ascend 為唯一出路但 ascend 語義不完全匹配。v2 透過兩處修正消除：(a) D-2 限縮後序列 descend 不受檔案重疊阻擋；(b) ascend 明文為預設路徑（不是「被禁止 descend 才 ascend」而是「預設 ascend，通過全條件才 descend」）。

##### ascend 條件（任一 OR 即 ascend）

| # | 條件 | 載體 | 來源 |
|---|------|------|------|
| A-1 | 需要上層才有的資訊（API 簽名、設計決策、規格缺口） | NeedsContext + Exit Status `needs_context` | ticket-body-schema.md |
| A-2 | 任務超出自身 agent 定義的允許產出範圍 | Exit Status `blocked` | agent-definition-standard.md |
| A-3 | 需要用戶決策（敏感規則、架構方向） | Exit Status `needs_context` | pm-role.md |
| A-5 | 子任務複雜度超標（修改檔案 > 5 或 acceptance > 7）且**本層已嘗試重構拆分仍無法降低** | NeedsContext（載明計數與已嘗試的拆分方式） | cognitive-load.md |
| A-6 | `can_descend()` = false **且**當前任務需要拆分才能完成 | Exit Status `blocked` + reason: 深度上限 | D3 |

**A-4 刪除說明**（P1 #2 回應）：v1 的 A-4（tool call 預算不足）不可判定——agent 無法可靠估計剩餘 tool call。Exit Status `partial_success` 已覆蓋事後報告（agent 在任何時刻耗盡均可寫 partial_success），無需前置條件。

#### D3 層級自覺機制（v2）

**核心主張**（重設計）：每層 agent 從 ticket 的 `parent_id` 鏈長度得知深度，以 `can_descend()` 單一判準決定是否可嵌套派發。刪除 `dispatch_level` 絕對映射概念。

##### 深度計算（parent_id 鏈計數）

```
depth(ticket) = 沿 parent_id 鏈回溯到根的邊數 + 1
  W1-056         → parent_id: null    → depth 1（根）
  W1-056.5       → parent_id: W1-056  → depth 2
  W1-056.5.1     → parent_id: W1-056.5 → depth 3
```

**Why**（P0 修正）：v1 使用 `ticket_id.count('.') + 1`，但完整 ticket ID 含版本號前綴（如 `1.0.0-W1-056.5`，有 3 個點），會算出錯誤深度。`parent_id` 鏈是世界平面 SSOT（field-semantics.md），語意準確且與 ID 字串格式解耦。

**實作方式**：agent claim 後執行 `ticket track query <id>` 讀取 `parent_id`，遞迴查詢至 `parent_id: null` 計算深度。或由 ticket CLI 提供 `ticket track depth <id>` 命令直接回傳（見 spawn 規劃）。

##### 深度上限與 can_descend()

| 參數 | 值 | 來源 |
|------|-----|------|
| MAX_TICKET_DEPTH | 3 | 保守設計：平台 5 層 - 1 層 PM - 1 層安全邊距 = 3 層 agent 可用 |
| can_descend(ticket) | depth(ticket) < MAX_TICKET_DEPTH | 單一判準 |

**上限收緊說明**（P2 Alternatives-5 + linux-F5 併入）：v1 上限為 depth 4（對應 L3），但 tool 集三層表（L1/L2/L3 差異化工具集）依賴不存在的動態工具切換機制。v2 收緊至 depth 3，待真實需求驗證後以用戶決策擴展。depth 3 已涵蓋最常見場景（PM→L1→L2 一層嵌套）。

**DRY**（linux F2 修正）：MAX_TICKET_DEPTH 常數只出現在 `can_descend()` 定義處一次；D2 條件 D-3 引用 `can_descend()` 而非重複數字。

##### agent 自檢流程（寫入 AGENT_PRELOAD）

```
1. claim ticket → ticket track query <id> → 讀取 parent_id
2. 計算 depth = 沿 parent_id 鏈計數
3. can_descend = depth < MAX_TICKET_DEPTH
4. 若需拆分子任務：
   - can_descend = true → 依 D2 條件表判斷 descend
   - can_descend = false → ascend（A-6）
5. 若不需拆分 → 在本層完成，無需考慮 depth
```

**tool 集三層表刪除說明**（P2 Alternatives-5 併入）：v1 試圖按層級差異化工具集（L1 全工具、L2 有條件、L3 禁 Agent），但 CC runtime 無動態工具切換 API，此設計不可實作。v2 統一：所有層級工具集相同，descend 控制完全由 D2 條件表 + `can_descend()` 決定。
