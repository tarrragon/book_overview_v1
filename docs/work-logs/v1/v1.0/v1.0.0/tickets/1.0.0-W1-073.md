### 問題根因

PC-V1-004 根因 3：CC runtime 對 hook 輸出（additionalContext / systemMessage）無「受眾標記」語義，所有注入訊息對觸發方 LLM 一視同仁。現行防護是逐 hook 加 `is_subagent_environment()` 早期跳過（W1-071 修 4 個）+ 開發 checklist 強制項（W1-072 已固化），但屬點狀防護：每個新 hook 都要各自記得做，漏一個就重蹈。

### 影響範圍

- `.claude/hooks/` 全體輸出注入訊息的 hook（盤點對象）
- `.claude/agents/AGENT_PRELOAD.md` — 候選方案 A 落點（教 subagent 忽略 PM-only 前綴）
- CC runtime hook schema — 候選方案 B 依賴（受眾欄位是否存在，已派 claude-code-guide 查證，結果將補入本章節）

### 相關 Error Pattern

PC-V1-004（hook-injection-audience-mismatch）防護 C 層。防護 A（W1-071 逐 hook 偵測）與防護 B（W1-072 checklist）已完成；本 ANA 評估系統性長期方案與點狀防護的取捨。

### 既有事實（接手者免重查）

- helper：`is_subagent_environment()`（`.claude/hooks/hook_utils/hook_io.py`，以 agent_id 欄位判定）——hook 輸入可區分 subagent 觸發是已驗證事實

#### 不暴露的 gap hook（session-start / UserPromptSubmit only，不在 subagent 觸發）

### 實驗發現

1. **已驗證事實**：19 個 gap hook 中 10 個實際暴露於 subagent，佔全部注入 hook 的 33%（10/30）
2. **已驗證事實**：`emit_hook_output()` 集中出口函式已存在於 hook_io.py，但僅 6/48 個 hook 使用（adoption rate 12.5%）——這是方案 C 的關鍵槓桿點
3. **已驗證事實**：Stop event 無 agent_id（CC runtime 硬約束），`is_subagent_environment()` 對 Stop hook 永遠返回 False
4. **仍未驗證假設**：AGENT_PRELOAD 加 PM-only 前綴忽略規則後 subagent 是否能穩定遵循（需 spike 驗證）

### WRAP 分析

#### W（Widen）：擴展選項

| 方案 | 描述 |
|------|------|
| A. AGENT_PRELOAD PM-only 前綴慣例 | hook 輸出加 `[PM-ONLY]` 前綴，AGENT_PRELOAD 教 subagent 忽略含此前綴的訊息 |
| B. 維持逐 hook 偵測 + checklist（現狀延續） | 每個 hook 各自呼叫 `is_subagent_environment()` 早期跳過 |
| C. emit_hook_output 統一出口層偵測 | 擴充 `emit_hook_output()` 加 `audience` 參數，函式內部自動偵測 subagent 並過濾 PM-only 訊息 |
| D. 方案 C + B 組合（推薦） | 擴充 emit_hook_output 統一出口 + 10 個 gap hook 遷移至 emit_hook_output + checklist 維持新 hook 審查 |

#### R（Reality-test）：現實測試

| 方案 | 優勢 | 劣勢 | Stop event 覆蓋 | ARCH-V1-001 對齊 |
|------|------|------|----------------|-----------------|
| A. AGENT_PRELOAD 前綴 | 零 hook 改動；跨所有事件類型生效（含 Stop） | 依賴 LLM 遵循前綴規則（機率性，非確定性）；token 浪費（PM-only 訊息仍注入 subagent context 只是教它忽略）；無法防止指令性訊息的潛意識影響 | 有（前綴不依賴 agent_id） | 低（規則層防護，非強制） |
| B. 逐 hook 偵測（現狀） | 確定性跳過；已有 16 個 hook 實作 | 點狀防護，每個新 hook 都要記得加；Stop event 永遠無法覆蓋（無 agent_id）；已有 10 個 gap hook | 無 | 低（分散式） |
| C. emit_hook_output 統一出口 | 單點強制（ARCH-V1-001）；新 hook 只要用此函式即自動受保護 | 需遷移 48 個 hook 從 json.dumps 到 emit_hook_output（大工程）；Stop event 仍受限（emit_hook_output 也依賴 input_data 的 agent_id） | 無（同 B） | 高 |
| D. C + B 組合 | 統一出口 + checklist 雙層；遷移範圍縮至 10 個 gap hook（而非全 48 個） | Stop event 仍受限；新 hook 若不用 emit_hook_output 仍可繞過 | 無（同 B/C） | 中高（漸進） |

#### A（Attain distance）：拉開距離

**核心取捨**：確定性（B/C/D：程式碼層跳過，subagent 根本收不到訊息）vs 覆蓋率（A：含 Stop event，但依賴 LLM 遵循）。

Stop event 硬約束分析：
- Stop event 無 agent_id → `is_subagent_environment()` 永遠返回 False → B/C/D 對 Stop hook 無效
- 但 Stop event 在 subagent 的影響範圍有限：subagent 結束時觸發，此時 subagent 已在生成 final-message，Stop 注入只影響最後幾個 token（W1-060 出口污染的實際機制）
- 方案 A 是唯一能覆蓋 Stop 的方案，但其防護品質最低（LLM 可能仍受前綴訊息影響）

成本效益：
- 方案 A：改動量最小（AGENT_PRELOAD 加 1 段 + hook 加前綴），但防護品質最低
- 方案 B：零改動（已是現狀），但 gap 持續存在
- 方案 C：改動量最大（48 hook 遷移），防護品質最高但 ROI 差
- 方案 D：改動量適中（10 hook + emit_hook_output 擴充），防護品質中高

#### P（Prepare to be wrong）：備案

| 方案 | 失敗模式 | 備案 |
|------|---------|------|
| A | LLM 不遵循前綴規則（已知 LLM 對 system message 的遵循率 < 100%） | 降級為 B+D 組合 |
| B | 新 hook 開發者忘記加偵測（已發生 10 次） | checklist 已有（W1-072），但依賴人工自律 |
| C | 遷移過程引入 regression | 分批遷移 + 測試覆蓋 |
| D | 10 個 gap hook 遷移後仍有新 hook 不用 emit_hook_output | checklist + CI lint 偵測（可加）|

### 結論

**推薦方案 D（emit_hook_output 統一出口 + 10 gap hook 遷移 + checklist 維持）** + **方案 A 作為 Stop event 專屬補充**。

理由：
1. 方案 D 將 10 個暴露 hook 納入統一出口，與 ARCH-V1-001（單點強制原則）一致
2. 方案 A 補充 Stop event 盲區（D/B/C 皆無法覆蓋），成本低（AGENT_PRELOAD 加 1 段）
3. 不追求全 48 hook 遷移（ROI 差），漸進式推動 emit_hook_output adoption
4. checklist（W1-072）維持作為新 hook 審查的人工防線
