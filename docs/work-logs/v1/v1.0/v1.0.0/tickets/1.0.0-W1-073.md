## Problem Analysis

### 問題根因

PC-V1-004 根因 3：CC runtime 對 hook 輸出（additionalContext / systemMessage）無「受眾標記」語義，所有注入訊息對觸發方 LLM 一視同仁。現行防護是逐 hook 加 `is_subagent_environment()` 早期跳過（W1-071 修 4 個）+ 開發 checklist 強制項（W1-072 已固化），但屬點狀防護：每個新 hook 都要各自記得做，漏一個就重蹈。

### 影響範圍

- `.claude/hooks/` 全體輸出注入訊息的 hook（盤點對象）
- `.claude/agents/AGENT_PRELOAD.md` — 候選方案 A 落點（教 subagent 忽略 PM-only 前綴）
- CC runtime hook schema — 候選方案 B 依賴（受眾欄位是否存在，已派 claude-code-guide 查證，結果將補入本章節）

### 相關 Error Pattern

PC-V1-004（hook-injection-audience-mismatch）防護 C 層。防護 A（W1-071 逐 hook 偵測）與防護 B（W1-072 checklist）已完成；本 ANA 評估系統性長期方案與點狀防護的取捨。

### 既有事實（接手者免重查）

- helper：`is_subagent_environment()`（`.claude/hooks/hook_utils/hook_io.py`，以 agent_id 欄位判定）——hook 輸入可區分 subagent 觸發是已驗證事實
- 雙向污染證據鏈：出口（5/5 委員報告被 Stop hook 訊息擠掉）+ 入口（Explore 唯讀委員被注入「git merge 建議」誘導實際執行）見 1.0.0-W1-060 Solution
- W1-072 checklist 位置：`.claude/references/hook-architect-technical-reference.md` L518「受眾評估 checklist」

### CC runtime 能力查證結果（claude-code-guide，2026-06-11）

| 查詢項 | 結論 | 證據出處 |
|--------|------|---------|
| Hook JSON 輸出 audience 欄位 | 無 | hooks.md + agent-sdk/hooks.md schema 定義 |
| PostToolUse/Stop/SubagentStop 在 subagent 內注入行為 | 部分支援（可檢測，無隔離） | Changelog v2.1.145 + Agent SDK 示例 |

**總結**：CC runtime 層級無受眾標記語義；subagent 檢測能力不完整（Stop event 無 agent_id 區分，正是 W1-060 出口污染的事件類型）。方案評估須以此為硬約束。

grep 盤點 `.claude/hooks/*.py` 全量，統計 additionalContext/systemMessage 輸出覆蓋率 vs `is_subagent_environment()` 偵測覆蓋率。


#### Gap 中實際暴露於 subagent 的 hook（PostToolUse/PreToolUse 事件，有 agent_id 可偵測）

| Hook | 事件類型 | 暴露風險 |
|------|---------|---------|
| active-dispatch-tracker-hook | PostToolUse | 中（dispatch 追蹤注入 subagent） |

| comment-qa-hook | PostToolUse | 低（註解品質提醒） |
| dispatch-count-guard-hook | PostToolUse, UserPromptSubmit | 中（派發計數） |
| file-ownership-guard-hook | PreToolUse | 低（檔案歸屬提醒） |
| pre-test-hook | PreToolUse | 低（測試前提醒） |
| session-context-guard-hook | PostToolUse | 低（session 上下文） |
| skill-cli-error-feedback-hook | PostToolUse | 低（CLI 錯誤回饋） |
| utf8-integrity-check-hook | PostToolUse | 低（UTF8 檢查） |
| worklog-format-check-hook | PostToolUse | 中（worklog 格式提醒可能誘導 subagent 寫 worklog） |

共 10 個 hook 在 subagent 觸發時會注入訊息但無偵測。

#### 不暴露的 gap hook（session-start / UserPromptSubmit only，不在 subagent 觸發）

9 個 hook（session-start-* / command-entrance-gate / project-init-env-check 等）僅在主線程事件觸發，無 subagent 暴露。

### 實驗發現

1. **已驗證事實**：19 個 gap hook 中 10 個實際暴露於 subagent，佔全部注入 hook 的 33%（10/30）
2. **已驗證事實**：`emit_hook_output()` 集中出口函式已存在於 hook_io.py，但僅 6/48 個 hook 使用（adoption rate 12.5%）——這是方案 C 的關鍵槓桿點
3. **已驗證事實**：Stop event 無 agent_id（CC runtime 硬約束），`is_subagent_environment()` 對 Stop hook 永遠返回 False
4. **仍未驗證假設**：AGENT_PRELOAD 加 PM-only 前綴忽略規則後 subagent 是否能穩定遵循（需 spike 驗證）
