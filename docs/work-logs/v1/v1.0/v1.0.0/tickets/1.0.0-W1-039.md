### 統一機制模型：記錄平面與世界平面的二相性

系統存在兩個平面，語意不對稱：

| 平面 | 內容 | 語意 |
|------|------|------|
| 世界平面 | filesystem / git / ticket 狀態 | 工具副作用 **at-least-once**（重試/分叉可重複執行，無 transaction 包裝） |
| 記錄平面 | transcript / context / 「記憶」 | 單寫者勝出 **at-most-once**（輸家分支的記錄不存在） |

兩平面間沒有一致性保證 → 必然出現雙向漂移。漂移分型如下。

> **審查後修訂（2026-06-10，依 Test Results 四視角閘門）**：原 5 列分型經 linux（M2/M3 冗餘）+ alternatives（M1/M3 negative case 不可區分）審查修訂——M3 併入 M2（同機制兩面）、M4 拆 M4a/M4b、M5 標對照組、宿主機制降級（見鑑識結果 3）。原始 5 列與審查推理見 Test Results。

| 模式 | 記錄 | 世界 | 可操作判據 | 歷史案例 | 真因層 |
|------|------|------|-----------|---------|--------|
| M1 生成幻覺 | 有（虛構） | 無 | 無 ghost 痕跡（與原 M3 negative case 不可區分） | PC-166 原始案例（同訊息續寫虛構 stdout） | 模型層 |
| M2 ghost branch（含原 M3 鏡像面） | 輸家記錄消失 | 有（留檔案痕跡）／原 M3 為執行流被棄 | 有 ghost 檔案痕跡（subagents 目錄／無主 commit／birth time）→ 執行確實發生 | 本案（7c21ca64 / W1-027~034 spawn / W1-024 complete）；原 M3：B4「以為派了沒派」 | runtime 層 |
| M4a 觀測污染 | 有但錯位 | 有 | shell hook 輸出混入工具結果 | chpwd 淹沒（IMP-056） | 觀測層 |
| M4b 訊息截斷 | 有但被替換 | 有 | 最終 text message 被 Stop hook 替換（→ W1-030） | B2／本 session agent 最終訊息劫持 | 觀測層 |
| M5 真並行（對照組，非漂移） | 各自有 | 共享 | 各執行體均有完整 transcript | 並行 agent commit race（PC-092） | 協調層 |

**不可判區間**：查無 ghost 檔案痕跡時，無法區分 M1（從未執行）與原 M3（執行流被棄、記錄一併消失），因 CC 不導出棄流 trace。此為固化邊界，補證見 spawned 1.0.0-W1-041。

### 核心洞察：為什麼「修了好幾次還再犯」

1. **M2 與 M3 是同一機制的兩面**：turn 執行體與 transcript 寫入權分離。誰搶到寫入權，誰的歷史成為「事實」；輸家的副作用成為「鬼」（M2），贏家未執行的宣稱成為「幻覺」（M3）。
2. **歷史防護鏈把 M3 當 M1 治**：tool-output-trust（禁止續寫結果 / raw stdout 判據 / 固定值驗證）針對模型生成紀律，防 M1 有效；但 M3 中模型誠實生成了動作與宣稱，是 runtime 未把該分支的執行/記錄配對保存——生成紀律修不到，故同類「幻覺」反覆出現。
3. **W1-024 B4 的裁決需推翻**：原裁決「派發確認信號已由 W1-023 鏈覆蓋」——W1-023 鏈防 M1/M4，不防 M3。
