### 統一機制模型：記錄平面與世界平面的二相性

系統存在兩個平面，語意不對稱：

| 平面 | 內容 | 語意 |
|------|------|------|
| 世界平面 | filesystem / git / ticket 狀態 | 工具副作用 **at-least-once**（重試/分叉可重複執行，無 transaction 包裝） |
| 記錄平面 | transcript / context / 「記憶」 | 單寫者勝出 **at-most-once**（輸家分支的記錄不存在） |

兩平面間沒有一致性保證 → 必然出現雙向漂移。漂移分型：

| 模式 | 記錄 | 世界 | 歷史案例 | 當時歸因 | 真因層 |
|------|------|------|---------|---------|--------|
| M1 生成幻覺 | 有（虛構） | 無 | PC-166 原始案例（同訊息續寫虛構 stdout） | 模型生成 | 模型層 |
| M2 ghost branch | 無 | 有 | 本案（7c21ca64 / W1-027~034 spawn / W1-024 complete 皆無對話記錄） | 誤判「並行 session」 | runtime 層 |
| M3 鏡像 ghost | 有（誠實生成過） | 無（執行流被棄） | B4「以為派了 basil 沒派」、歷次「執行結果不存在」 | 模型幻覺（PC-166） | runtime 層 |
| M4 觀測污染 | 有但錯位 | 有 | chpwd 淹沒（IMP-056）、B2 最終訊息劫持 | 個別機制 | 觀測層 |
| M5 真並行 | 各自有 | 共享 | 並行 agent commit race（PC-092） | 正確 | 協調層 |

### 核心洞察：為什麼「修了好幾次還再犯」

1. **M2 與 M3 是同一機制的兩面**：turn 執行體與 transcript 寫入權分離。誰搶到寫入權，誰的歷史成為「事實」；輸家的副作用成為「鬼」（M2），贏家未執行的宣稱成為「幻覺」（M3）。
2. **歷史防護鏈把 M3 當 M1 治**：tool-output-trust（禁止續寫結果 / raw stdout 判據 / 固定值驗證）針對模型生成紀律，防 M1 有效；但 M3 中模型誠實生成了動作與宣稱，是 runtime 未把該分支的執行/記錄配對保存——生成紀律修不到，故同類「幻覺」反覆出現。
3. **W1-024 B4 的裁決需推翻**：原裁決「派發確認信號已由 W1-023 鏈覆蓋」——W1-023 鏈防 M1/M4，不防 M3。
