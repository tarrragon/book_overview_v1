## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

W1-024 B2 將「agent 最終訊息被劫持」歸為 CC runtime limitation，但**未驗證劫持內容的實際來源**。settings.json 註冊 7 個專案自有 Stop hooks，其中部分會向 stdout 輸出 JSON（CC runtime 讀取 Stop hook stdout 並注入對話），可能是劫持的部分或全部來源。

### 影響範圍

- 所有被 PM 派發的 subagent（Agent() 呼叫）在 session 結束時都會觸發 7 個 Stop hooks
- 劫持導致 agent 回傳值不可靠（W1-024 實證：basil 回傳「無動作」實則完整完成）
- 現行 workaround：deliverable 寫 ticket/檔案 + PM 以 git ground truth 驗證

### 相關 Error Pattern

- PC-166（confabulation）：agent 回傳值不可信的上位問題
- tool-output-trust-rules 規則 5（記錄平面非 ground truth）

## 重現實驗結果
### 實驗方法

靜態分析法：逐一檢視 7 個 Stop hooks 的原始碼，識別哪些在正常執行（非 --self-test）時向 stdout 輸出內容（CC runtime 透過讀取 Stop hook stdout JSON 來注入對話內容）。同時檢查是否有任何 hook 偵測 subagent context。

### 實驗執行

**步驟 1：列出 7 個 Stop hooks**（來源：`.claude/settings.json`）

| # | Hook 檔案 | 簡稱 |
|---|----------|------|
| 1 | `evaluate-session.py` | evaluate-session |
| 2 | `handoff-auto-resume-stop-hook.py` | handoff-auto-resume |
| 3 | `session-experience-persistence-reminder-hook.py` | session-persistence |
| 4 | `worktree-auto-commit-hook.py` | worktree-auto-commit |
| 5 | `stop-worklog-handoff-sync-check-hook.py` | worklog-sync-check |
| 6 | `malformed-tool-call-detector-hook.py` | malformed-tool-call |
| 7 | `context-depth-warning-hook.py` | context-depth-warning |

**步驟 2：逐 hook 原始碼分析——正常執行時的 stdout 輸出**

| # | Hook | 正常執行 stdout 輸出 | 輸出內容 | 劫持潛力 |
|---|------|---------------------|---------|---------|
| 1 | evaluate-session | **無**（僅 stderr） | — | 無 |
| 2 | handoff-auto-resume | **有**：`print(json.dumps(hook_output))` | `{"decision":"block","reason":"[Handoff] 偵測到未完成的 handoff 任務...請執行恢復步驟..."}` 或 `{"suppressOutput":true}` | **高**：block + reason 會注入恢復指令到對話 |
| 3 | session-persistence | **有**：`print(json.dumps(hook_output))` | `{"suppressOutput":true}`（即使找到敏感 ticket 也 suppress） | 無（始終 suppress） |
| 4 | worktree-auto-commit | **無**（僅 subprocess capture） | — | 無 |
| 5 | worklog-sync-check | **有**：`print(json.dumps(output))` | `{"systemMessage":"...drift warning..."}` | **中**：systemMessage 會注入漂移警告 |
| 6 | malformed-tool-call | **無**（stdout 僅 --self-test） | — | 無 |
| 7 | context-depth-warning | **無**（stdout 僅 --self-test） | — | 無 |

**步驟 3：subagent context 偵測**

`grep -in 'subagent\|sub_agent\|is_agent\|agent_name' *.py` 對全部 7 個 hook 回傳空——**無任何 hook 偵測是否在 subagent 環境中執行**。

### 實驗發現

**已驗證事實**：

1. **劫持來源混合型**——CC runtime 行為 + 專案 hooks 共同構成：
   - CC runtime 負責「讀取 Stop hook stdout 並注入對話」的機制（不可修）
   - 專案 hooks 負責「產出什麼內容給 runtime 注入」（可修）

2. **兩個專案 hook 有劫持潛力**：
   - `handoff-auto-resume`（高）：`decision: "block"` + `reason` 欄位注入恢復指令（"[Handoff] 偵測到未完成的 handoff 任務..."），這是最可能覆蓋 agent 最終訊息的元兇
   - `worklog-sync-check`（中）：`systemMessage` 欄位注入漂移警告

3. **零個 hook 偵測 subagent context**：所有 7 個 hook 對 PM session 和 subagent session 的行為完全相同。handoff pending JSON 若存在，即使是 subagent stop 也會觸發 block。

4. **可修性明確**：在 handoff-auto-resume 和 worklog-sync-check 中加入 subagent context 偵測（例如檢查 stdin input_data 的 session 特徵或環境變數），subagent stop 時 `suppressOutput: true` 即可消除劫持。

**仍未驗證的假設**：

- CC runtime 對 Stop hook 的 `decision: "block"` 在 subagent context 中的具體行為（是否真的阻止 session 結束 + 覆寫回傳值，或只是注入 systemMessage）——需實際派發 agent 觀察。但靜態分析已足以確認「專案 hook 有產出劫持內容的能力且未做 subagent 豁免」。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 來源判定結論

**劫持來源 = 混合型（專案 hook 可修 + CC runtime 機制不可修）**

W1-024 B2 將劫持**全部**歸為 CC runtime limitation 是不準確的。準確分層：

| 層 | 來源 | 可修性 | 說明 |
|----|------|--------|------|
| 機制層 | CC runtime | **不可修** | runtime 設計為：Stop hook stdout → 注入對話。此機制本身不可關閉 |
| 內容層 | 專案 Stop hooks | **可修** | 專案 hook 決定注入什麼內容。目前 2 個 hook 未區分 PM/subagent context，對 subagent 注入了不適用的恢復指令和漂移警告 |

### 裁決：框架可修

**修復方向**：在 `handoff-auto-resume-stop-hook.py` 和 `stop-worklog-handoff-sync-check-hook.py` 中加入 subagent context 偵測，subagent session stop 時直接 `{"suppressOutput": true}` 跳過輸出。

**subagent 偵測方法候選**：

| 方法 | 可行性 | 說明 |
|------|--------|------|
| 檢查 Stop hook stdin `input_data` 的 session 欄位 | 需驗證 | CC runtime 傳給 Stop hook 的 stdin JSON 是否含 subagent 標記 |
| 環境變數 `CLAUDE_AGENT_*` | 需驗證 | CC runtime 可能設定 subagent 環境變數 |
| 檢查 session transcript 長度/特徵 | 低 | 不可靠，subagent 可能有長對話 |
| 檢查是否存在 parent process 特徵 | 低 | 平台相依 |

**建議 spawn 計畫**（由 PM 決定是否建立）：

1. **IMP-1**（小型）：在 handoff-auto-resume 和 worklog-sync-check 中加入 subagent 偵測 + suppressOutput。需先驗證 CC runtime Stop hook stdin 是否攜帶 subagent 標記（可在 IMP 內做 spike）。
2. **更新 W1-024 B2 定性**（DOC 或直接回寫）：從「屬已知 CC runtime limitation，無需 spawn」修正為「混合來源，專案 hook 端可修，已 spawn W1-030 驗證 + IMP 修復」。

### 與 /tmp 檔案交付 workaround 的關係

修復 hook 端劫持後，agent 最終訊息的可靠性會顯著提升（不再被恢復指令/漂移警告覆寫），但仍建議保留 deliverable 寫 ticket body 的實踐——因為即使無 hook 劫持，CC runtime 仍可能有其他注入（如 token 用量提示），且 ticket body 本身就是良好的持久化實踐。
