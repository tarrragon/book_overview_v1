## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 實作摘要

於 create.py 新增兩個驗證 helper 並 wire 進 `_parse_cli_args_to_config`：

- `_validate_where_file_token(token)`：取路徑首段（首個 '/' 之前），若含 '=' 視為 key=value 前綴髒值（src=、layer=）並 reject，回傳修正提示（剝除前綴後的建議路徑）；'=' 在中段/檔名屬合法檔名字元放行；空字串放行。
- `_validate_where_files(tokens)`：批次驗證，收集所有髒值錯誤訊息。

接線位置：`_parse_cli_args_to_config` 解析 where_files 後立即驗證，命中髒值印 INVALID_WHERE_FILE_TOKEN 錯誤並 return None（既有呼叫鏈在 execute() L1468 已處理 None → return 1，無新增流程）。

### 設計取捨

- 判定限「首段含 '='」而非「任意位置含 '='」，避免誤殺路徑中段/檔名含 '=' 的合法路徑（acceptance 2 零誤殺）。
- 採 reject（exit 1）而非 warn，因髒值會級聯成派發誤診，源頭硬擋成本最低。
- 框架碼遵循 reference-stability 規則 8：註解用通用描述，未嵌 project-local ticket ID。

## Context Bundle


### 實作規格

**動機（W1-086 派發誤診暴露）**：ticket create --where 靜默接受 'src=.claude/x' 當檔案路徑，髒值寫進 where.files 後，dispatch hook 的 _classify_paths 誤判該 token 非 .claude/（因含 'src=' 前綴），級聯成 worktree 誤強制與派發誤診。屬 PC-V1-009（機械缺陷誤診為流程缺陷）的源頭——前置驗證可從輸入端攔下整個缺陷類別。

**修改檔案**：.claude/skills/ticket/ticket_system/commands/create.py（--where 解析處）。

**策略（how.strategy）**：--where 解析後驗證每個路徑 token——含 '=' 於路徑首段（如 src=、layer=）或不像合法相對/絕對路徑者 reject 或 warn；正常路徑（.claude/x, src/y, a/b.py）100% 放行不誤殺。可選提供自動剝除已知前綴（src=/layer=）的修正提示。

**關鍵——改 CLI source 必須 reinstall（IMP-023 / W11-037）**：ticket_system 是 uv tool install 的獨立 CLI，source 與 installed 是兩份。改 create.py 後必須 `uv tool install .claude/skills/ticket --force --reinstall`，否則 CLI 仍用 stale installed 版、新驗證不生效，測試可能假綠。實作後務必 reinstall 再驗收。

**測試**：ticket_system 測試（pytest）。新增至少 3 個驗證測試：髒值 reject（src=.claude/x）+ 正常放行（.claude/x, src/y, a/b.py）+ 邊界（如路徑中段含 = 的合法情況需釐清）。

**TDD**：phase1-4，從 Phase 2 RED 起。100% 測試通過為完成前提。

**框架約束**：.claude/ 框架碼會 sync 全 consumer——禁嵌 project-local ticket ID 於 code/註解（reference-stability-rules 規則 8，本 session W1-089 已強化此慣例）；註解用通用描述。
