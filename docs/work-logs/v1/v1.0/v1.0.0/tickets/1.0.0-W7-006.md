## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 校準實作（2026-06-12, thyme-python-developer）

**WRAP 三問**：
- W（候選方案）：(a) `CHARS_PER_TOKEN = 1.3` + `int(chars / 1.3)` 除法浮點；(b) 改乘法 `TOKEN_PER_CHAR = 0.77`。採 (a)——保留既有常數名與「chars per token」語意，diff 最小且測試引用不需改名。
- A（機會成本）：本票範圍小（兩檔），不擠壓其他 Wave 7 目標。
- P（最可能失敗）：浮點化後測試硬編碼 `// 3` 斷言失效——防護為 acceptance 2 強制測試改引用常數。

**變更內容**：

1. `.claude/hooks/file-size-guardian-hook.py`
   - `CHARS_PER_TOKEN` 由 `3` 校準為 `1.3`，常數註解記錄校準依據：2026-06-12 /context Memory files 同集合實測 38.9k tokens vs 合計 chars 50.7k → 50.7k / 38.9k ≈ 1.30 chars/token（繁中為主集合），並標註原值 3 為未經實測的保守假設、低估 2.3 倍（IMP-V1-001）
   - 估算式由整數除法 `char_count // CHARS_PER_TOKEN` 改浮點 `int(char_count / CHARS_PER_TOKEN)`

2. `.claude/hooks/tests/test_file_size_guardian_auto_load_budget.py`
   - 模組 docstring 由「Token 估算（chars // 3）」更新為引用常數的浮點估算說明
   - `test_token_estimation_chars_div_3` 改名 `test_token_estimation_uses_chars_per_token_constant`，斷言由硬編碼 `300 // 3 + 90 // 3` 改為 `int(300 / hook_mod.CHARS_PER_TOKEN) + int(90 / hook_mod.CHARS_PER_TOKEN)`（引用常數，係數再調整時測試不需改）
   - 超標測試註解 chars 門檻由「> 135k chars」更新為「> 45k * CHARS_PER_TOKEN ≈ 58.5k chars」（測試資料 200k chars 在新係數下約 153k tokens，仍穩定超標，無需改資料量）

**範圍守恆**：45k 預算值未動（不在本票範圍）；state 檔差值跳增（16.9k → 39.1k）屬係數變更預期行為。
