## 重現實驗結果
### 實驗方法

依 docs/bookstores/readmoo.md §MCP E2E Checklist，用 chrome-devtools-mcp 對 live Readmoo 書庫頁（read.readmoo.com/#/library）實機重跑提取流程。環境：build:dev v0.20.0，extension ID jjojfonkallmalpccagibhbhbbepjkom，用戶手動登入 Readmoo（--isolated profile）。

### 實驗執行

- Step 1 Build：npm run build:dev 成功，manifest 0.20.0 = package.json 0.20.0。
- Step 2 登入：navigate #/library → redirect SSO 登入頁 → 用戶手動登入 → 回 library。
- Content script 注入確認：console 出現 `[DETECT] 頁面檢測: Readmoo 頁面 (library)`、App 就緒報告 isReadmooPage=true / pageType=library / extractable=true / 五模組全 true。
- 書庫規模：擁有 966 本，封存 15，借出 1 → 預期可見 950。
- DOM 漂移檢測（96 樣本 selector 命中率）：library-item-grid-view 96/96、a.reader-link[href] 96/96、img.cover-img[alt] 96/96、[id^=privacy-] 96/96、.info .title[title] 96/96、.progress.progress-simple .progress-bar 96/96、.rendition-overlay .label.rendition 96/96。首本取證：privacy-15636449 / 「進烤箱的好日子」/ cover CDN gpoitzm_210x315.jpg / reader-link dummy URL / rendition「流式」。
- Step 3 瀑布流：finalCount=950（966-15-1），19 輪後 plateau。
- Step 4-5 提取：popup 點「開始提取」→「資料提取完成」，檢測 950，storage books_count=950，950/950 有書名，950/950 reader- ID（idPrefix 全 reader），0 重複。
- Step 6 overview：總書籍數顯示 950，[data-book-id] 書卡 950，作者「— 待補」（W1-061 來源限制）。overview 實際路徑為 src/overview/overview.html（非 checklist 記載的根目錄 overview.html）。
- 去重重驗：第二次提取後 books_count 仍 950、0 重複、timestamp 更新（合併非追加）。
- Auto-load happy-path：清空 storage → hard reload（DOM 重置 96）→ 不手動捲動直接提取 → 自動載入並提取 950（950 unique），library DOM 由 96 展開至 950。

### 實驗發現

已驗證事實：
1. Readmoo 書庫頁 URL（read.readmoo.com/#/library）與 DOM 結構**未漂移**——readmoo.md 記錄的全部 selector 100% 命中。
2. 提取核心 happy-path 完整可運作：登入 → 提取 → storage → overview 顯示 → 去重，端到端正常。
3. ID 策略正確：950/950 為 reader-{privacyBookId}，與 DOM privacy-id 一一對應，0 重複。
4. loadAllBooksLazy 自動瀑布流**已落地且運作**（W1-030）——用戶不需手動捲動，點提取即自動載入全 950。readmoo.md「實作對照表」記載的 W6-021「extractAllBooks 缺 scroll-to-end，96/944 缺口」**已過時**。

仍未驗證 / 範圍外：CSV 匯出（Step 7，本次未測，屬 UC-03 範圍）；節流網路下的 timeout 行為（W6-012.9.4 既有未驗項）。
