## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

_has_gitignore_rule（onboard_checker.py line 588-617）用字面精確比對判定 .gitignore 是否含某必要規則：normalize（rstrip "/*" + rstrip "/"）後與 pattern 字串相等才認可。此設計不理解 gitignore pattern matching 語意：

- 萬用字元 **（**/hook-logs/）：normalize 後 **/hook-logs != .claude/hook-logs
- 根錨定 /（/.claude/hook-logs/）：normalize 後 /.claude/hook-logs != .claude/hook-logs
- 具體檔名（.claude/dispatch-active.json）：normalize 不去 .json 後綴
- 副檔名萬用（.claude/dispatch-active.*）：normalize 後 .claude/dispatch-active. != .claude/dispatch-active

### 影響

任何用合法但非「精確字面」寫法的專案，跑 project-init onboard/check 會被誤報缺 gitignore 規則（false positive）。本專案自身即中 2 條（hook-logs 用 **/、dispatch-active 用具體檔名）。false positive 打斷正常 onboard，且可能誘導使用者改寫已正確的 .gitignore。

### 相關
W9-007（測試漂移）、W9-008（dispatch-active 建議字面，僅修「照建議寫能通過」未修「其他合法寫法被誤判」）的延伸根因。

- tool-results：本專案 .gitignore 真無此規則（git check-ignore 不命中，true negative）
- dispatch-active：本專案用具體 .json+.lock（line 99-100），git 實際忽略，但 check 判缺（false positive）

### _has_gitignore_rule 對合法 gitignore 變體的判定（量化 false positive）

.claude/hook-logs 規則：
| 寫法 | _has_gitignore_rule | git 實際忽略 |
|------|---------------------|-------------|
| .claude/hook-logs/ | True | 是 |
| .claude/hook-logs | True | 是 |
| **/hook-logs/ | False（誤判） | 是 |
| /.claude/hook-logs/ | False（誤判） | 是 |
| .claude/hook-logs/* | True | 是 |

.claude/dispatch-active 規則：
| 寫法 | _has_gitignore_rule | git 實際忽略 |
|------|---------------------|-------------|
| .claude/dispatch-active* | True | 是 |
| .claude/dispatch-active.json | False（誤判） | 是 |
| .claude/dispatch-active.* | False（誤判） | 是 |

### 結論：4 種合法 git 語法被系統性誤判
- ** 萬用字元前綴
- / 根錨定前綴
- 具體檔名（無萬用字元的副檔名）
- .* 副檔名萬用

根因：_has_gitignore_rule（line 588-617）以「normalize(rstrip /*) 後字面精確相等」判定，不理解 gitignore pattern matching 語意（萬用字元展開、路徑錨定、檔名 vs 目錄匹配）。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### WRAP 分析

### W — Widen options（修法方向）

| 方向 | 機制 | 語意正確性 | 成本 |
|------|------|-----------|------|
| 1. git check-ignore 子進程 | 對每規則的代表目標路徑呼叫 git check-ignore，由 git 引擎判斷是否被忽略 | 100%（git 自身） | git CLI 依賴 + 無 repo fallback |
| 2. pathspec 庫 | 純 Python gitignore 解析 | 高 | 新增第三方依賴 |
| 3. 字面改良 | 擴充 normalize 容忍 **/、/錨定、具體檔名 | 治標 | 仍漏 ! 否定、[] 字元類、? 等語法 |
| 4. 混合（推薦） | 有 git repo 用 check-ignore，無則 fallback 現有字面 | 高 + 容錯 | 中等 |

### R — Reality-test

- onboard 場景目標多為 git repo（通常已 git init），git 在 PATH 為合理假設
- git check-ignore --stdin 可批次（8 規則一次子進程，效能 OK）
- check-ignore 對不存在路徑也能判定（pattern matching 不需檔案實際存在）
- 無 git repo（git init 前 onboard）→ fallback 現有 _has_gitignore_rule
- 跨平台：git CLI 行為一致

### A — Attain distance（機會成本）

字面改良（方向 3）快但脆弱：每發現新語法補一次（** 補完還有 !、[]、? ），維護債持續累積。git check-ignore（方向 1/4）一次涵蓋所有 gitignore 語法，不重造 parser（DRY，複用 git 引擎）。

### P — Prepare to be wrong

- fallback：無 git / 無 repo 降級字面匹配（graceful degrade，現有行為不破）
- 回歸防護：保留既有測試 + 新增 4 變體測試（**/、/錨定、具體檔名、.*）
- 殘風險：「規則→代表路徑」映射需正確（如 hook-logs 規則測 .claude/hook-logs/_probe）

### 推薦：方向 4（混合）

改 check_gitignore_completeness：
- 新增 helper _is_path_ignored_by_git(project_root, target_path)：呼叫 git check-ignore，回 True/False/None（無法判斷）
- 每必要規則定義代表目標路徑（hook-logs→.claude/hook-logs/_probe；dispatch-active→.claude/dispatch-active.json 等）
- 有 git 判斷用之；None（無 repo/git）fallback 現有 _has_gitignore_rule
- 既有精確匹配保留為 fallback 層，不刪
