## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 問題根因

ticket why 的前提「W1-005 收尾發現 sync-claude-pull.py 含 14+ 處 ticket ID，範圍可能涵蓋多個 scripts」**被 Reality Test 推翻為嚴重低估**：全框架 ticket ID 引用達約 7500 處（5497 .py + 1996 .md，排除 error-patterns/work-logs），跨 scripts/rules/pm-rules/references/methodologies/skills/agents/hooks 全部目錄。

但「7500 處 = 7500 違反」是錯誤推論。真根因是**規則 8 字面一刀切（框架文件禁止任何 ticket ID）與三個既有機制衝突，缺乏「引用性質」的區分判準**：

1. **與 comment-writing 方法論衝突**：方法論鼓勵 comment 記錄業務脈絡與設計決策來源；`# ARCH-020，W1-027` `# W10-092 遷移後` 這類歷史錨點是合理的設計脈絡記錄。
2. **與 PC-093-exempt `history` 類別衝突**：PC-093 已建立 `history` 豁免（允許「引用已完成歷史 / 動機脈絡」並以 ticket ID 作歷史錨點），規則 8 卻無對應豁免（grep 規則 8 對 history/設計脈絡/comment-writing 零命中）。
3. **與文件慣例衝突**：每個規則檔末尾 `**Source**: W-xxx` 來源標注是全框架慣例，規則 8 字面將其全列為違反。

### 影響範圍

| 引用類型 | 規模 | 性質 | 規則 8 字面判定 | 合理判定 |
|---------|------|------|----------------|---------|
| 歷史錨點/設計脈絡型 | ~7400 | comment 記錄「W-xxx 落地/遷移/實證」，不建立跳轉依賴 | 違反 | 應豁免（同 PC-093 history） |
| 依賴型（框架資產正文/docstring） | 66 | 「詳見/參考/依 W-xxx」建立跳轉依賴 | 違反 | 真違反，需審視清理 |
| 依賴型（tests/migration-backups） | 37 | 同上但在測試/歷史備份 | 違反 | 備份豁免、測試低優先 |
| provenance 功能字串 | 0（誤指） | sync-pull 的 provenance 是 PC 編號去重溯源功能，與 ticket ID 引用無關 | — | 本就合法 |

真依賴型 66 處分佈：skills 21 / hooks 16 / references 14 / methodologies 7 / pm-rules 4 / agents 3 / rules 2。

### 相關 Error Pattern

- PC-093（YAGNI deferred decision）的 `history` 豁免類別——規則 8 缺對應機制，為核心缺口。
- PC-098（PM 寫規則本能引用 ticket ID）——規則 8 的執行面 error-pattern，但聚焦「新寫入時防範」，未處理「既有 ~7500 處存量 + 判準缺口」。

## 重現實驗結果
### 實驗方法

grep 全 `.claude/`（排除 error-patterns / work-logs）的 ticket ID 引用（`W\d+-\d+` 與 `X.Y.Z-W` 兩格式），按「程式碼層 vs 文件層」「依賴型 vs 歷史錨點型」「框架資產 vs tests/backups」三維度分類計數，抽樣驗證分類正確性。

### 實驗執行

1. 規模盤點：`.py` 命中 5497 行、`.md` 命中 1996 行（合計約 7500），分佈跨 scripts(15 檔)/rules(14)/pm-rules(24)/references(24)/methodologies(14)/skills(301)/agents(27)/hooks(177)/commands(1)。
2. 依賴型篩選：`(詳見|參考|依|根據|來源[:：]).{0,8}ticket-id` → 103 處；排除 migration-backups + tests → 框架資產正文/docstring **66 處**。
3. 抽樣驗證：依賴型多為案例指引（`參考案例：W3-072 暴露的缺口`、`參考 W15-005 WRAP 方案 F`），非「不跳轉無法理解」的硬依賴；歷史錨點型抽樣（`# ARCH-020，W1-027`、`# W10-092 遷移後`、`W17-144 落地`）為設計脈絡標注。
4. provenance 驗證：grep `provenance` 命中 sync-claude-pull.py，讀碼確認為 PC 編號去重溯源的資料結構欄位（`provenance: dict[tuple[int,str],int]`），與 ticket ID 引用無關——ticket why 假設的「provenance 功能字串違反」不存在。
5. 規則衝突驗證：grep 規則 8（reference-stability-rules.md）對 `history`/`設計脈絡`/`comment-writing`/`PC-093` **零命中**，確認規則 8 無「依賴型 vs 歷史錨點型」區分，亦無 PC-093 history 類比豁免。

### 實驗發現

**已驗證的事實**：
- ticket why「14+ 處」前提嚴重低估，真實規模約 7500 處
- 7500 中約 7400 為歷史錨點/設計脈絡型（合理，不應清理）
- 框架資產正文/docstring 真依賴型僅 66 處（唯一可能需審視子集）
- provenance「違反」假設不成立（誤指功能欄位）
- 核心缺口 = 規則 8 缺「引用性質」判準，與 comment-writing + PC-093 history + 文件 Source 慣例衝突

**仍未驗證的假設**：66 處依賴型中實際「硬依賴」（移除 ticket ID 即無法理解）vs「案例指引」（可保留或抽象化）的比例——下放 spawn DOC 依新判準逐處審視。

## Solution
<!-- Schema[ANA/Solution]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 決策結論

**方向 = 規則 8 精緻化（區分「依賴型引用」vs「歷史錨點型引用」）+ 依新判準審視 66 處真依賴型；不做 7500 處大清理。**

ticket 原框架「盤點違反 → 大清理」是偽命題——7500 處中 ~7400 為合理設計脈絡，清理不可行且有害（損失 comment-writing 鼓勵的脈絡記錄）。真問題是規則 8 缺判準。

### WRAP 四階段

**W（Widen）候選**：
- 方向 1：大規模清理 7500 處 ← 否決（不可行 + 損失設計脈絡 + 違反 comment-writing）
- 方向 2：規則 8 精緻化（區分引用性質）+ 對齊 PC-093 history ← **採用（核心）**
- 方向 3：維持規則 8 字面、不主動清理 ← 否決（執行落差永存，新人面對 7500 處無從判斷）
- 方向 4：方向 2 + 審視 66 處真依賴型清理硬依賴 ← **採用（執行半）**

**R（Reality Test）**：已執行（見重現實驗結果）。關鍵推翻：14+ → 7500 規模 + 66 真依賴型 + provenance 誤指 + 規則 8 無 history 判準。

**A（Attain distance）機會成本**：大清理 7500 處擠壓 v1.0.0 主線且有害；規則精緻化高槓桿——一次定義救未來所有「ticket ID 該不該寫」判斷，跨專案 sync 受惠，且與 PC-093/comment-writing 收斂為一致判準。

**P（Prepare to be wrong）**：精緻化判準若模糊會再淪主觀。防護：判準須可操作——以「移除 ticket ID 後讀者是否仍能理解該句」為測試（仍能理解=歷史錨點，可保留；無法理解=硬依賴，需抽象化或內聯定義）。
