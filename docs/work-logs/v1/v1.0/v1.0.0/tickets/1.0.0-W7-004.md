## Problem Analysis
<!-- Schema[ANA/Problem Analysis]: 必填（.claude/pm-rules/ticket-body-schema.md） -->
### 背景量測（2026-06-12 /context 實測）

Memory files 合計 82.5k tokens（22 檔），200k 模型下佔 40%。前六大：MEMORY.md 12.2k、decision-trigger-binding 6.3k、test-assertion-design-rules 6.1k、document-writing-style 5.8k、bash-tool-usage-rules 5.5k、quality-baseline 5.4k。

### 根因

框架有「開放內容放 rules/、封閉內容放 references/」原則，但缺乏自動載入集合的 token 預算上限與回收機制。每次事故教訓傾向寫進自動載入層（怕忘記），file-size-guardian 只量單檔行數（300 行閾值，29 檔超標且不阻擋），無機制量測 auto-load 集合總量，成長單調遞增。

### 五個結構性問題

### 1. 三明示原則套用在錯誤的載入層（最大單一因素）

document-writing-style 強制每段論述含 Why/Consequence/Action 三層，適合按需讀取的教學文件，不適合每 session 自動載入的行為提示層。實例：bash-tool-usage-rules 規則六一條約束寫約 50 行（Why 段 + Consequence 段 + Action 表 + 三慣例表 + 調和表），約束本身兩行可說完。三明示讓每條規則約 3 倍化。

### 2. 案例考古混進規則層（規則 vs error-pattern 職責不分）

grep 實測：test-assertion-design-rules 與 decision-trigger-binding 各含 18 處 ticket/PC 編號引用。前者整檔複述 W1-017/W1-018 實證細節（2.48ms、23/23、195 suites），且已有同名 skill test-assertion-design 可按需觸發。quality-baseline 規則 1 兩個「邊界」段落複述 PC-165/PC-168 事件鏈，PC 檔案本身已存在，規則只需一行路由。

### 3. 半外移不徹底——stub 仍是半篇主文

成功範本已存在：quality-common（609 tokens 完全外移）、ticket-skill-sync-check（806 tokens 純路由 stub）。失敗案例：ai-communication-rules 規則 5 聲稱外移卻留完整觸發情境表 + 授權邊界，規則 6 整段本文全留；cognitive-load 自稱路由 stub 仍 192 行；decision-trigger-binding 的豁免 marker 位置 / frontmatter 場景 / history vs ticket-tracked 語意全是「寫 ticket 命中 hook 時」才需要的內容，卻整檔自動載入。

### 4. 自動載入判準錯置——「重要」被當成「高頻」

document-writing-style（5.8k，受眾=正在寫文件的人）、agent-definition-standard（4.6k，受眾=正在編輯 agent 定義的人）、test-assertion-design-rules（受眾=正在寫測試的人）皆屬「動作觸發時才需要」。正確判準：「PM 在不做任何特定動作時，這行會改變行為嗎？」答否即外移。另：各檔「與其他規則的邊界」表雙向重複（A 寫與 B 差異、B 又寫與 A 差異）；footer 版本歷史留多代完整描述（tool-output-trust-rules 留 1.0/1.1/1.2 三代）。

### 5. MEMORY.md 違反單行索引設計（12.2k，單檔最大）

120 條目，多數是完整教訓濃縮（含案例編號、commit hash、多子句）而非單行 pointer。大量條目已升級為 PC error-pattern 或正式規則（PC-040/PC-064/PC-087 系列已固化），依四問升級設計應於升級後刪除，實際只加不減。

### 附帶：CLAUDE.md 本身（4.9k）

pre-commit hook + CI lint 兩章節（含豁免對照表、完整事故敘事）約佔全檔 40%，屬 docs/development-setup.md 內容；CLAUDE.md 只需指令表 + 一行原則 + 路由。

### 收斂潛力估算

| 動作 | 預估節省 |
|------|---------|
| 六重災檔降為 0.5-1.5k 路由 stub（decision-trigger-binding / test-assertion-design / document-writing-style / bash-tool-usage / quality-baseline / agent-definition-standard） | ~25k |
| MEMORY.md 修剪（已升級者刪、其餘壓真單行） | ~7k |
| CLAUDE.md hook/CI 章節外移 | ~2k |
| 全檔邊界章節去雙向重複 + footer 只留最新版 | ~3-5k |

目標：82.5k → 40-45k（200k 模型佔比 40% → 約 20%）。

### 收斂時的不變量（防止瘦身傷及功能）

1. 外移內容必須有明確路由（stub 內保留觸發條件表 + 完整版路徑），不可直接刪除 substance。
2. hook 強制層引用的規則檔路徑不可失效（外移前 grep .claude/hooks/ 確認引用點）。
3. CLAUDE.md @ 引用鏈與 rules/README.md 檔數描述需同步更新（避免 W1-032 stale 模式重演）。
4. 收斂後需建立預算守門機制（如 file-size-guardian 擴充 auto-load 集合總量量測），否則回彈。

### WRAP 決策紀錄（快速+模式）

- **Step 0**：資料充足——/context 實測 + 全檔內容已檢視 + wc/grep 量化。風險低（git 可逆）。
- **W 擴增選項**（三類涵蓋）：(a) 零工具——換 1M 模型或減少刷新頻率；(b) 改造既有——文件降 stub + substance 外移 references/（主體方案）；(c) 新增工具——file-size-guardian 擴充 auto-load 總量預算量測（防回彈）。反向思考：「自動載入多 = 防錯」不成立——規則價值在禁令被讀到，82.5k 中大量是論證非禁令，且 attention 稀釋本身降低遵循率（PC-072/PC-080 token 污染先例）；外移後 substance 仍可按需讀，防錯價值不損失。
- **R 現實檢驗**：基本率——repo 內已有兩個成功外移範本（W10-137 / W10-076.1 同類工作），可行性已驗證。R.2 事實宣稱：「外移後 hook 引用不失效」未驗證 → 列為每張 spawn ticket 的強制 AC（grep .claude/hooks/ 引用點）。Ooch：建議先做 test-assertion-design-rules 單檔試水溫（已有同名 skill 承接，風險最低）。
- **A 機會成本**：不做 → 每 session 固定多耗 ~40k；做 → 6-7 張 DOC/IMP 容量。悖論自檢：收斂結論寫入 ticket（按需），不新增自動載入規則文字；守門交 hook 機器層。
- **P Premortem 三風險**：(1) 外移砍掉 hook 錨點文字 → 防護：AC 含 grep hooks 驗證；(2) stub 過瘦致行為退化 → 防護：stub 保留禁令速查表 + 路由，分批執行可觀察；(3) .claude/ sync 把舊版覆蓋回來 → 防護：收斂完成後 sync-push 固化。回退：單 commit git revert，成本低。
- **偏好暴露**：傾向「文件收斂 + 預算守門」組合。盲點：未量測外移內容的按需讀取頻率，若高頻回讀會吃回部分節省（仍優於每 session 固定載入）。
