### 問題根因

W1-019 Model 1 決策後，新 error-pattern 須採來源前綴 `<CAT>-<PROJ>-NNN`。但
`/error-pattern add` 是純文件互動流程（skill.md，無 Python 入口），無法自動：
(a) 自我識別專案代號、(b) 掃描該專案前綴空間取下一號。需建 allocator tooling
讓 add 產生正確前綴 ID（否則人工算號易錯且 flat base 與前綴空間混淆）。

### 影響範圍

| 檔案 | 變更 |

| `.claude/skills/error-pattern/lib/allocator.py` | **新建** allocator（`identify_project_code` + `allocate_pattern_id`） |
| `.claude/skills/error-pattern/skill.md` | add 流程加步驟 7「自動分配來源前綴 ID」+ 輸出命名格式 |

## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->

### allocator 設計（兩函式解耦）

| 函式 | 職責 |
|------|------|
| `identify_project_code(registry_path, repo_toplevel)` | git toplevel basename 反查 registry `dir` → `code`；未登錄 raise ValueError（防靜默錯誤前綴） |
| `allocate_pattern_id(category_prefix, claude_dir, project_code)` | glob `<cat-dir>/<CAT>-<PROJ>-*.md` 取 max+1（首次 001）；flat 凍結 base 不參與遞增 |

### 關鍵設計

- **自我識別零設定**：registry 用 `git rev-parse --show-toplevel` basename 對應 `dir` 反查 `code`，tooling 不需 project-local 設定檔。
- **前綴空間隔離**：只掃 `<CAT>-<PROJ>-` 前綴，flat（`PC-099`）與他專案（`PC-APP-*`）不影響本專案遞增（`test_flat_base_not_counted` / `test_other_project_prefix_isolated` 鎖定）。
- **複用 SSOT**：用 `lib/pattern_id.py` 的 `PATTERN_ID_RE`（W1-019.2）解析既有號，不複製 regex。
- **category 映射**：`_CATEGORY_DIRS` 對應 7 目錄（與 skill.md 編號章節一致）；未知 prefix raise。

### skill.md 整合

add 流程加步驟 7：呼叫 allocator 取前綴 ID + 明示「禁止手動 flat 號」。輸出命名改 `<CAT>-<PROJ>-NNN-<slug>.md`。

### spawn 落地（cognitive-load 拆分）

- **W1-021**（IMP，source=本 ticket）：negative gate hook（E1 補強，拒絕 flat 新增的 PreToolUse 強制層）。理由：純函式 allocator vs PreToolUse 攔截 hook 為不同職責，後者需 settings 註冊 + 對所有 Write 觸發，量級不同。

### 不在本 ticket 範圍

- negative gate hook → W1-021
- skill 完整 package 化（pyproject + 測試內遷）→ W1-001 上架範圍
- flat base remediation → W1-019.4

### Phase 4 評估報告

- **重構需要**: 否
- **技術債務**: 無
- **程式碼品質**: A

**結論**：allocator 兩函式單一職責、解耦（識別 / 分配）、複用 SSOT 不重複。命名自說明，
認知負擔低。negative gate 已依 cognitive-load 主動拆分至 W1-021，避免本 ticket 職責膨脹。
無需額外重構。
