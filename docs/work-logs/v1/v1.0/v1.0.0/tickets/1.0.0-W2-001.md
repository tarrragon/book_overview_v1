## Solution
<!-- Schema[IMP/Solution]: 選填（.claude/pm-rules/ticket-body-schema.md） -->
### 修復摘要

**問題根因**：assets/icons/icon-{16,48,128}.png 三檔副檔名為 .png，實際內容為 SVG（藍底 #4f46e5 白字 R placeholder）。Chrome MV3 manifest 的 icons / action.default_icon 欄位不支援 SVG，圖示靜默不渲染。validate-build.js 的 validateIcons() 僅檢查檔案存在（fileExists），不檢查內容格式，故 npm run validate:build:prod 全綠漏檢。

### 變更內容

1. **assets/icons/icon-{16,48,128}.png 轉真 PNG**
   - 以 macOS `qlmanage -t -s <size>` 將原 SVG 內容渲染為對應尺寸 PNG（保留原 placeholder 設計：藍底 #4f46e5 白字 R）。
   - 結果：icon-16.png 16x16、icon-48.png 48x48、icon-128.png 128x128，皆為 8-bit RGBA non-interlaced PNG。
   - 工具選擇：環境無 rsvg-convert / ImageMagick / PIL，sips 無法直接轉 SVG；qlmanage（QuickLook）可渲染 SVG 且輸出尺寸精確符合 -s 參數。

2. **scripts/validate-build.js 加入 PNG magic bytes 格式檢查**
   - 新增常數 `PNG_SIGNATURE`（89 50 4E 47 0D 0A 1A 0A）與 `isRealPng(filePath)`：以 fs.openSync + readSync 讀檔頭 8 bytes 比對 PNG signature。
   - 改寫 `validateIcons()`：原僅判 fileExists；改為「不存在 → error」「存在但檔頭非 PNG → error（提示可能 SVG 偽裝）」「真 PNG → success」。
   - 攔截邏輯：SVG 偽裝成 .png 時檔頭為 `<svg`，不符 PNG signature，validateIcons 標記 error 使整體 validate 失敗（exit 1）。

### 設計決策

- magic bytes 檢查只讀檔頭 8 bytes（fs.readSync 指定 length），不載入整檔，效能與既有 fileExists 相當。
- 維持既有錯誤處理風格（logError 累積 hasErrors，最終 process.exit(1)），不引入新依賴。

### 不在本 ticket 範圍

- 圖示視覺設計優化（仍為 placeholder）。
- Chrome 實機 load unpacked 渲染驗證：本任務於 CLI 環境，已透過 file/sips 確認真 PNG + 尺寸正確 + manifest 宣告 PNG，符合 MV3 icons 要求。
