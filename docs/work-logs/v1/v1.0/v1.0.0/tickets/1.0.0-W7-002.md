### 範圍重評（W7-001 auto-commit 根因解後）

W7-002 原規劃固化方案 D「ticket body 寫入後、git 還原操作前先 commit/stash 保護」為**主防護規則**（W1-017 ANA 設計）。但 W7-001 已採 **auto-commit 根因解**（append-log 寫入後立即 commit ticket md，body 即時進 commit 歷史，使 checkout/reset --hard/stash 三種還原全失效），方案 D 從**主防護降為補充防護**。

### 重評後範圍

| 原規劃 | 重評後 |
|--------|--------|
| 固化「git 還原前先保護」為主要防護規則 | auto-commit 已承擔主防護；本 DOC 固化**補充防護** |
| 人為 commit/stash 為最後防線 | 補充防護限兩場景：(1) auto-commit graceful degrade 失敗（非 git repo / index.lock / commit 失敗）的人為 fallback；(2) auto-commit 不涵蓋場景（手動 Edit ticket md 而非走 append-log）|

### DOC 產出調整

| acceptance | 重評後內容 |
|-----------|-----------|
| PC-019 延伸/新 PC | 記錄「ticket body 未 commit 被 git 還原覆蓋」模式 + **防護演進**（auto-commit 主防護 W7-001 + 人為補充），而非原本的「人為規則為主」 |
| 更新 memory | feedback_worktree_commit_before_dispatch 補充 auto-commit 已自動保護 append-log 路徑；關聯 [[append-log-auto-commit-workflow]] |
| 與 IMP 防護方案對齊 | 對齊 W7-001 auto-commit（非原設想的 auto-stage+hook） |

此重評不縮減 DOC 價值——補充防護規則對 auto-commit degrade 失敗與手動編輯場景仍必要，但定位需精確（避免後人誤把已被 auto-commit 根除的「主防護」當作仍需人為執行）。

### where.files `rules/core/` 重評：不修改（決策）

原規劃 where.files 列 `.claude/rules/core/`（假設方案 D 為主防護，需自動載入規則）。重評後：

- auto-commit 是**機制防護**（append-log 自動 commit），無需人為規則自動載入。
- 補充防護（degrade fallback + 手動 Edit）屬 **edge case**，記於 PC-178（按需 `/error-pattern query`）已足夠。
- 在 `rules/core/` 加規則會增加自動載入內容（PM token attention pool 負擔），收益低於成本。

**決策**：`rules/core/` 不修改，避免規則膨脹（認知負擔考量）。防護知識集中於 PC-178（按需查）+ memory（跨 session 行為）。
