### 分類邊界案例（合理但值得記錄）

| 代碼 | 目前歸類 | 備註 |
|------|---------|------|
| `QUOTA_EXCEEDED_ERROR` | storage | 合理，與 `STORAGE_QUOTA_EXCEEDED` 語意重疊（TD-3） |
| `OUT_OF_MEMORY` / `OUT_OF_MEMORY_ERROR` | storage | 合理，memory 歸 storage 可接受 |
| `POPUP_CONTEXT_NOT_FOUND` | ui | 合理，Popup 是 UI 層 |
| `INVALID_ARGUMENT` / `INVALID_API_PATH` | data-processing | 可考慮歸 validation，但現歸類不算錯 |
| `UNKNOWN_BUTTON` | system | 語意上偏 UI，但作為兜底域可接受 |
| `NOT_FOUND_ERROR` | system | 通用錯誤，歸 system 合理 |

結論：分類整體合理，無明顯歸類錯誤。

## 3. 技術債務

| ID | 優先級 | 描述 |
|----|--------|------|
| TD-1 | 低 | export.js 含 40 個代碼，可考慮拆分為 export-core / export-progress / export-notification 子域 |
| TD-2 | 低 | system.js 含 49 個代碼，作為兜底域可接受，但長期若持續膨脹應拆分 |
| TD-3 | 低 | 語意重疊代碼對：`QUOTA_EXCEEDED_ERROR` vs `STORAGE_QUOTA_EXCEEDED`、`OUT_OF_MEMORY` vs `OUT_OF_MEMORY_ERROR`、`TEMPORARY_FAILURE` vs `TEMPORARY_NETWORK_FAILURE`。建議未來版本統一，保留一方並 deprecate 另一方 |
| TD-4 | 低 | `NOTIFICATION_*` 代碼分散在 ui.js（通用通知）和 export.js（匯出通知），命名前綴相近但分屬不同域，可能造成開發者混淆 |
